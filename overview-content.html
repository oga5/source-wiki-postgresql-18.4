LLM Call Flows

  1. Module: contrib/amcheck主なフロー

    1. **セットアップ**: amcheck_lock_relation_and_check が対象をロックし、所有者権限への昇格と GUC ネストレベルの管理を行う。 2. **事前検証**: index_checkable により、対象が検証可能な状態（有効なインデックス等）かを確認。 3. **実行**: 各アクセス手法（AM）固有のコールバック関数が呼び出され、ページ走査と不変条件のチェックを実行。 4. **終了処理**: 権限の復元、GUC のロールバック、およびロックの解放。

  2. Module: contrib/auth_delay主なフロー

    1. **初期化 (_PG_init)**: GUC 変数を定義し、現在の ClientAuthentication_hook を退避させた上で、自身の auth_delay_checks をフックに登録する。 2. **認証実行 (auth_delay_checks)**:

    • 既存のフックが存在する場合は先に実行する。
    • 認証失敗が確認された場合、pg_usleep を呼び出し指定時間待機する。
  3. Module: contrib/auto_explain主なフロー

    1. **初期化 (_PG_init)**: GUC 変数の定義と、標準の Executor フックの乗っ取りを行う。 2. **開始 (explain_ExecutorStart)**: サンプリング判定を行い、対象クエリに対して統計収集（instrument_options）を有効化する。 3. **実行 (explain_ExecutorRun/Finish)**: nesting_level を管理し、実行状況を追跡する。 4. **終了 (explain_ExecutorEnd)**: 実行時間を計測し、閾値を超えている場合に ExplainPrintPlan を用いてログ出力を行う。

  4. Module: contrib/basebackup_to_shell主なフロー

    1. **初期化**: _PG_init にてGUCパラメータを定義し、バックアップターゲットとして登録。 2. **セットアップ**: shell_get_sink によりコマンド文字列の検証と構造体の初期化を実行。 3. **データ転送**: begin_archive/begin_manifest でパイプを開き、archive_contents/manifest_contents でデータを書き込み、end_archive/end_manifest でパイプを閉じて終了ステータスを検証する。

  5. Module: contrib/basic_archive主なフロー

    1. **初期化**: _PG_init にてGUC変数を登録。 2. **設定確認**: basic_archive_configured によりアーカイブ先ディレクトリの妥当性をチェック。 3. **アーカイブ処理**:

    • 宛先に同名ファイルが存在する場合、compare_files で内容を検証。
    • 一致しない場合はエラーとし、一致する場合は成功として終了。
    • 新規ファイルは一時ファイルへコピー後、durable_rename で永続化。
  6. Module: contrib/bloom主なフロー

    1. **構築/挿入**: blinsert が入力値をハッシュ化し、blutils のビット操作関数を用いて BloomTuple を生成、ページへ格納します。 2. **スキャン**: blgetbitmap が検索条件から署名を生成し、全ページを走査して一致するタプルをビットマップへ登録します。 3. **メンテナンス**: blbulkdelete が不要なタプルを削除し、空いたページを blvacuumcleanup が FSM（Free Space Map）へ解放します。

  7. Module: contrib/bool_plperl主なフロー

    1. **bool_to_plperl**: PostgreSQL の bool を取得し、Perl の真偽値定数へマッピングして返却。 2. **plperl_to_bool**: Perl の SV* を取得し、SvTRUE マクロを用いて PostgreSQL の bool へ変換して返却。

  8. Module: contrib/btree_gin主なフロー

    1. **インデックス構築**: gin_extract_value_<type> により、値を Datum として抽出します。 2. **クエリ解析**: gin_extract_query_<type> が検索条件を解析し、QueryInfo を生成します。範囲検索時には leftmostvalue を用いてスキャン開始点を決定します。 3. **スキャン実行**: gin_btree_compare_prefix がインデックス内のキーと QueryInfo 内の値を比較し、型固有の比較関数を介してスキャンの継続・終了を判定します。
