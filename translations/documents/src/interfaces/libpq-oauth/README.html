src/interfaces/libpq-oauth/README

libpq-oauthは、OAuthクライアント（RFC 8628）のデバイス認可フローを実装するオプションのモジュールです。libcurlへの依存関係を分離するため、独自の共有ライブラリとして保守されています。（Curlへの依存を望まないエンドユーザーは、このモジュールをインストールしないことを選択するだけで済みます。）

接続文字列でOAuthの使用が許可されており、サーバーから要求があり、かつlibpqクライアントが独自のカスタムOAuthフローをインストールしていない場合、libpqはdlopen()と以下のABIを使用してこのモジュールの遅延読み込みを試みます。読み込みに失敗すると、接続は失敗します。

= ロード時ABI =

このモジュールのABIは内部的な実装詳細であるため、メジャーリリースごとに変更される可能性があります。モジュール名（libpq-oauth-MAJOR）はそのことを反映しています。このモジュールは以下のシンボルをエクスポートします。

- PostgresPollingStatusType pg_fe_run_oauth_flow(PGconn *conn);
- void pg_fe_cleanup_oauth_flow(PGconn *conn);

pg_fe_run_oauth_flowとpg_fe_cleanup_oauth_flowは、それぞれconn->async_authとconn->cleanup_async_authの実装です。

現時点では、pg_fe_run_oauth_flow()はlibpqのpg_g_threadlockとlibpq_gettext()に依存しています。これらは、フローが実行される前に、以下の初期化関数を使用してlibpqによって注入される必要があります。

- void libpq_oauth_init(pgthreadlock_t threadlock,
						libpq_gettext_func gettext_impl,
						conn_errorMessage_func errmsg_impl,
						conn_oauth_client_id_func clientid_impl,
						conn_oauth_client_secret_func clientsecret_impl,
						conn_oauth_discovery_uri_func discoveryuri_impl,
						conn_oauth_issuer_id_func issuerid_impl,
						conn_oauth_scope_func scope_impl,
						conn_sasl_state_func saslstate_impl,
						set_conn_altsock_func setaltsock_impl,
						set_conn_oauth_token_func settoken_impl);

また、PGconn構造体のいくつかのメンバーへのアクセスにも依存しています。これらはマイナーバージョン間で位置が変更される可能性があるだけでなく、単一のマイナーリリース内でもオフセットが必ずしも安定しているとは限りません（例えば、conn->errorMessageはconfigure時のオプションによってオフセットが変わる可能性があります）。そのため、必要なアクセサ（conn_*という名前）とミューテータ（set_conn_*）がここで注入されます。このアプローチにより、たとえモジュールがlibpqと同時にコンパイルされていなかったとしても、標準のdlopen()パス（RPATH、LD_LIBRARY_PATH、SOキャッシュなど）を安全に検索して使用する実装モジュールを見つけることができます。これは、実行中のlibpqアプリケーションがディスクから最初に読み込まれる前にlibpq-oauthモジュールが更新されるような「ライブアップグレード」の状況において特に重要となります。

= 静的ビルド =

静的ライブラリlibpq.aは、動的読み込みを行いません。組み込みフローが有効な場合、アプリケーションは必要なシンボルを提供するためにlibpq-oauth.aと直接リンクすることが期待されます。（libpq.aとlibpq-oauth.aは同じビルドの一部である必要があります。動的モジュールとは異なり、変換用のシムは提供されません。）
