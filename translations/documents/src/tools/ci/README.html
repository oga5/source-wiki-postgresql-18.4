Postgres 継続的インテグレーション (CI)
====================================

Postgres には2種類の CI があります：

1) メインの postgres リポジトリ内のサポートされているすべてのブランチは、buildfarm を通じて継続的にテストされます。これはメインリポジトリのみを対象としているため、機能開発中には使用できません。

   詳細は https://buildfarm.postgresql.org/ を参照してください。

2) まだマージされていない開発中の作業については、一部の git ホスティングプロバイダーで CI を有効にできます。これにより、開発者はパッチがマージ（または投稿）される前に、さまざまなプラットフォームでテストを行うことができます。


個人リポジトリでの CI の設定
=======================================

現在、postgres には cirrus-ci を利用した CI サポートが含まれています。cirrus-ci は現在 GitHub でのみ利用可能です。


GitHub リポジトリで cirrus-ci を有効にする
=========================================

リポジトリで cirrus-ci を有効にするには、https://github.com/marketplace/cirrus-ci にアクセスし、「Public Repositories」を選択します。次に「Install it for free」および「Complete order」を選択します。次のページで、cirrus-ci がアクセスできるリポジトリを設定できます。該当するリポジトリを選択し、「Install」をクリックします。

https://cirrus-ci.org/guide/quick-start/ も参照してください。

リポジトリで有効にすると、そのリポジトリへの今後のコミットやプルリクエストによって、CI ビルドが自動的にトリガーされます。これらはコミット履歴や PR から確認できるほか、https://cirrus-ci.com/github/<username>/<reponame>/ の cirrus-ci UI でも確認できます。

ヒント：すべてのビルドログファイルは cirrus-ci にアップロードされます。ビルドのサマリーページで特定のタスクをクリックした後、cirrus-ci UI の「Artifacts」セクションからダウンロードできます。


CI に使用されるイメージ
==================

CI の時間を許容範囲内に保つため、ほとんどのプラットフォームでは事前に生成されたイメージを使用しています。一部のプラットフォームではコンテナを使用し、他のプラットフォームではフル VM を使用します。どちらのイメージも CI 実行とは別に生成されます。そうしないと、テスト対象の各 git リポジトリが独自のコンテナセットをビルドする必要があり、時間と容量の両面で無駄になるためです。

これらのイメージは、github.com/anarazel/pg-vm-images/ の仕様に基づいて毎日ビルドされます。


コミットメッセージによる CI の制御
=====================================

CI の動作は、コミットメッセージ内の特別なコンテンツによって制御できます。現在、以下の制御が可能です：

- ci-os-only: {(freebsd|linux|macos|mingw|netbsd|openbsd|windows)}

  指定されたオペレーティングシステムでのみ CI を実行します。これは、特定のプラットフォームのみに影響する移植性の問題に対処する場合に便利です。


CI にカスタム計算リソースを使用する
=====================================

リポジトリで大量のテストを実行する場合、cirrus-ci の無料クレジットでは不足することがあります。そのような場合、テスト実行のために別のインフラストラクチャを使用するようにリポジトリを設定できます。これを行うには、cirrus-ci Web インターフェース（https://cirrus-ci.com/github/<user or organization>）で、リポジトリの REPO_CI_CONFIG_GIT_URL 変数を設定します。この変数によって参照されるファイル（https://cirrus-ci.org/guide/programming-tasks/#fs を参照）は、.cirrus.yml で定義された関連する yaml アンカーを再定義することで、オペレーティングシステムごとのデフォルトの実行方法を上書きできます。

カスタム計算リソースは以下を使用して提供できます：
- https://cirrus-ci.org/guide/supported-computing-services/
- https://cirrus-ci.org/guide/persistent-workers/


手動タスクをデフォルトで有効にする
================================

CI クレジットの急速な消費を避けるため、一部のタスクはデフォルトでは自動的にトリガーされません。これは、カスタム計算リソースが設定されている場合など、リポジトリレベルで変更できます。

以下のリポジトリレベルの環境変数が認識されます：
- REPO_CI_AUTOMATIC_TRIGGER_TASKS - (mingw|netbsd|openbsd) のスペース区切りリスト
