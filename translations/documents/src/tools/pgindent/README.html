PostgreSQL ソースツリーの pgindent
=======================================

pgindent は、C および Perl コードのレイアウトを統一するために使用され、すべてのコミットで実行する必要があります。リリースサイクルごとに少なくとも一度は実行すべき、追加のコード整形タスクもあります。

以下のブログ記事が参考になるかもしれません:
http://adpgtech.blogspot.com/2015/05/running-pgindent-on-non-core-code-or.html


前提条件:

1) pg_bsd_indent を PATH にインストールしてください。そのソースコードは兄弟ディレクトリである src/tools/pg_bsd_indent にあります。そのディレクトリ内の README ファイルの指示を参照してください。

2) perltidy をインストールしてください。バージョン 20230309 であることを確認してください（古いバージョンや新しいバージョンではフォーマットの選択が異なるため、一貫性が求められます）。
   正しいバージョンは以下から入手できます:
   https://cpan.metacpan.org/authors/id/S/SH/SHANCOCK/
   インストールするには、Perl モジュールの通常のインストール手順に従ってください（"man perlmodinstall" で説明されています）。または、cpan がインストールされている場合は、以下で実行できるはずです:
   cpan SHANCOCK/Perl-Tidy-20230309.tar.gz
   または cpanm がインストールされている場合は、以下を使用できます:
   cpanm https://cpan.metacpan.org/authors/id/S/SH/SHANCOCK/Perl-Tidy-20230309.tar.gz


通常のコミット前にインデントを実行する方法:

1) ソースツリーのトップディレクトリに移動します。

2) C ファイルに対して pgindent を実行します:

	src/tools/pgindent/pgindent .

   ファイルでエラーが発生した場合は、"git checkout" で元のバージョンに戻し、以下のクリーンアップのヒントを参照してください。

3) "git status" を使用して、新しく作成されたファイルがないか確認します。本来あるべきではありません。（pgindent は問題が発生すると *.BAK ファイルを残し、perltidy は *.LOG ファイルを残します。）

4) pgindent がコミットに関係のない部分を変更しようとする場合は、一度停止して理由を確認してください。コミットで追加した typedef の周囲で不格好な空白の変更が行われている場合は、それらの typedef を src/tools/pgindent/typedefs.list に追加する必要があります。

5) 余裕があれば、"git diff" の出力を目視で確認し、あまりにもひどい変更がないかチェックすることをお勧めします。クリーンアップのヒントについては以下を参照してください。

6) フルテストビルドを実行します:

	make -s clean
	make -s all	# 予期しない警告やエラーがないか確認してください
	make check-world

   configure のスイッチには少なくとも --enable-tap-tests を含める必要があります。そうしないと、Perl コードの大部分が実行されません。
   ecpg の回帰テストは、ecpg の出力にコピーされるヘッダーファイルが pgindent によって更新されるため、失敗する可能性があります。その場合は、期待値ファイル (expected-files) をそれに合わせて調整してください。


リリースサイクルごとに少なくとも一度は行うこと:

1) ビルドファームから最新の typedef ファイルをダウンロードします:

	wget -O src/tools/pgindent/typedefs.list https://buildfarm.postgresql.org/cgi-bin/typedefs.pl

   この手順により、インクリメンタルに更新されたファイルと、クリーンで自動生成されたファイルとの間の差異が解消されます。
   （バックブランチのインデントを行いたい場合は、https://buildfarm.postgresql.org/cgi-bin/typedefs.pl?show_list で typedef ファイルの全リストを参照してください。）

2) 上記と同様に pgindent を実行します。

3) perltidy を使用して Perl コードをインデントします:

	src/tools/pgindent/pgperltidy .

   PATH にないバージョンの perltidy を使用したい場合は、まず PERLTIDY 環境変数を設定してその場所を指定してください。

4) ブートストラップカタログのデータファイルを再フォーマットします:

	./configure     # "make" は設定されていないツリーでは動作しません
	cd src/include/catalog
	make reformat-dat-files
	cd ../../..

5) 完了したら、使用した typedefs.list ファイルを含めてすべてを "git commit" します。

6) 新しく作成したコミットを .git-blame-ignore-revs ファイルに追加し、"git blame" がそのコミットを無視するようにします（無視ファイルの使用を選択したユーザー向け）。.git-blame-ignore-revs ファイルの先頭にある指示に従ってください。

無視ファイルの変更に対して、もう一度 "git commit" が必要になります。

失敗時や出力が不格好な場合のクリーンアップ
---------------------------------------------

特定のファイルの出力が気に入らない場合は、"git checkout" でそのファイルの変更を取り消し、必要に応じてファイルをパッチし、インデントプロセスを繰り返してください。

pgindent は、左端にないコメントブロックをリフローします。保持すべき手動フォーマットが崩れてしまう場合は、ダッシュを使用してコメントブロックを保護してください:

	/*----------
	 * ここにあるテキストは pgindent によって変更されません。
	 */

typedef 名の周囲の奇妙な間隔は、typedef リストが不完全であることを示している可能性があります。

pgindent は、typedef と名前が一致する C 関数の宣言と定義の両方を破壊します。現在のところ、最善の回避策は競合しない名前を選択することです。

pgindent は、コンパイラには正しく見えるが、全体として見ると中括弧や括弧が一致していない #if シーケンスによって混乱することがあります。通常、それは人間にとっても非常に読みにくいため、そうならないように #if テストを再構成するのがベストプラクティスです。

pgindent や perltidy が奇妙な出力を生成する場合、余分なカンマなどの小さなバグが原因であることがあります。その際は、遠慮なくクリーンアップしてください。

---------------------------------------------------------------------------

BSD indent
----------

私たちは FreeBSD の indent を標準として採用し、pg_bsd_indent と改名しました。pg_bsd_indent は FreeBSD のバージョンとわずかに異なり、主に非 BSD プラットフォームへの移植性が高められています。兄弟ディレクトリの src/tools/pg_bsd_indent にあります。

GNU indent バージョン 2.2.6 にはいくつかの問題があり、推奨されません。これらのバグは、50 万行を超えるコードを扱う際に非常に大きな問題となります。信じられない場合は、ディレクトリをコピーして、コピーに対して GNU indent を使用して pgindent を実行し、diff -r を実行してみてください。意味がわかるはずです。GNU indent は一部の処理は優れていますが、破壊も行います。詳細については以下を参照してください:

	http://archives.postgresql.org/pgsql-hackers/2003-10/msg00374.php
	http://archives.postgresql.org/pgsql-hackers/2011-04/msg01436.php

処理されるファイル
-------------------------

pgindent の実行は（ほぼ）すべての PostgreSQL の *.c および *.h ファイルを処理しますが、現在は *.y および *.l ファイル、ならびに *.y および *.l から派生した *.c および *.h ファイルを除外しています。その他の例外は exclude_file_patterns にリストされています。その理由については、そこにある注記を参照してください。

ecpg のヘッダーファイルは実行から除外されていないことに注意してください。それらの一部は ecpg の出力にそのままコピーされるため、ecpg の期待値ファイルを一致させるために更新する必要がある場合があります。

perltidy の実行は、すべての *.pl および *.pm ファイルに加え、そのように命名されていないいくつかの実行可能な Perl スクリプトを処理します。詳細については pgperltidy 内の "find" ルールを参照してください。
