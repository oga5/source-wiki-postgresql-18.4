分離テスト
==========

このディレクトリには、PostgreSQLにおける同時実行動作をテストするための一連のテストが含まれています。これらのテストでは、相互に作用する複数のトランザクションを実行する必要があり、複数の同時接続を管理しなければならないため、通常の pg_regress プログラムを使用してテストすることはできません。「isolation（分離）」という名前は、元々の動機がシリアライザブル分離レベルをテストすることであったことに由来しますが、他の種類の同時実行動作に対するテストも追加されています。

現在のビルドツリーに対してテストを実行するには、以下のように入力します。
    make check
あるいは、既存のインストール環境に対して実行するには、以下のように入力します。
    make installcheck
(これは libpq が想定するデフォルトポートでサーバーに接続します。これを制御するには、環境変数で PGPORT などを設定してください。)

インストール済みのサーバーに対して特定のテストのみを実行するには、以下のようにします。
    ./pg_isolation_regress fk-contention fk-deadlock
(利用可能なテストについては specs/ サブディレクトリを参照してください。)

一部のテストでは、サーバーの max_prepared_transactions パラメータが少なくとも 3 に設定されている必要があるため、デフォルトでは実行されません。これらをテスト実行に含めるには、サーバー設定が正しいことを確認した上で (「check」の場合の調整については TEMP_CONFIG を参照)、以下を実行してください。
    make check-prepared-txns
または
    make installcheck-prepared-txns

トランザクションが重なるテストを定義するために、カスタム構文を持つテスト仕様ファイルを使用します。これについては次節で説明します。新しいテストを追加するには、specs/ サブディレクトリに spec ファイルを配置し、expected/ サブディレクトリに期待される出力を追加し、isolation_schedule ファイルにテスト名を追加してください。

isolationtester は、libpq を使用して複数の接続を開き、spec ファイルで指定されたテストを実行するプログラムです。libpq の接続文字列で接続先のサーバーとデータベースを指定します。指定がない場合は、環境変数から導出されたデフォルト値が使用されます。

pg_isolation_regress は pg_regress に似たツールですが、テストの実行に psql ではなく isolationtester を使用します。pg_regress と同じすべてのコマンドライン引数を受け付けます。

デフォルトでは、isolationtester はテストの各ステップが完了するまで最大 360 秒（6 分）待機します。これを調整する必要がある場合は、環境変数 PG_TEST_TIMEOUT_DEFAULT に、希望するタイムアウト秒数の半分を設定してください。


テスト仕様
==========

各分離テストは、specs サブディレクトリに格納された仕様ファイルによって定義されます。テスト仕様は、いくつかの SQL「ステップ」を定義し、それらを「セッション」にグループ化し（1 つのセッションのすべてのステップは同じバックエンドで実行されます）、実行されるステップの「順列（permutation）」または順序を指定します。

テスト仕様は、以下の順序で 4 つの部分から構成されます。

setup { <SQL> }

  指定された SQL ブロックは、テスト実行前に（順列ごとに）一度だけ実行されます。テストテーブルやその他の必要なオブジェクトはここで作成してください。この部分はオプションです。必要に応じて複数の setup ブロックを許可します。各ブロックは指定された順序で個別に実行されます。（複数の setup ブロックを許可する理由は、各ブロックが単一の PQexec 送信として実行されるためであり、VACUUM のような一部のステートメントはそのようなブロック内で他のステートメントと組み合わせることができないためです。）

teardown { <SQL> }

  teardown SQL ブロックは、テスト終了後に一度だけ実行されます。これは、次の順列の準備としてクリーンアップを行うために使用します（例：setup で作成したテストテーブルの削除）。この部分はオプションです。

session <name>

  通常、spec ファイルには複数の「session」部分があります。各セッションは独自の接続で実行されます。セッション部分は、setup、teardown、および 1 つ以上の「step」の 3 つの部分で構成されます。セッションごとの setup および teardown 部分は、前述のテストごとの setup および teardown と同じ構文を持ちますが、各セッションで実行されます。例えば、setup 部分にはトランザクションを開始するための「BEGIN」コマンドを含めることができます。

  各ステップの構文は以下の通りです。

  step <name> { <SQL> }

  ここで <name> はこのステップを識別する名前であり、<SQL> はステップ内で実行される SQL ステートメント（セミコロンで区切られた複数のステートメントも可）です。ステップ名は spec ファイル全体で一意である必要があります。

permutation <step name> ...

  permutation 行は、その順序で実行されるステップのリストを指定します。permutation 行はいくつでも記述できます。permutation 行が指定されていない場合、テストプログラムは各セッションのステップの可能なすべてのインターリーブを自動的に実行します（各セッションのステップは順序通りに実行されます）。手動で指定された「permutation」行のステップリストは、必ずしも利用可能なステップの順列である必要はないことに注意してください。例えば、一部のステップを複数回繰り返したり、他のステップを除外したりすることも可能です。また、各ステップ名は、後述する括弧付きのマーカーで注釈を付けることができます。

セッション名とステップ名は、プレーンまたは二重引用符で囲まれた SQL 識別子です。標準 SQL との違いは、大文字小文字の変換が行われない点です。したがって、FOO と "FOO" は同じ名前ですが、FOO と Foo は引用符の有無にかかわらず異なる名前となります。分離テストのキーワード（「permutation」など）を名前として使用したい場合は、引用符を使用する必要があります。

# 文字はコメントの開始を意味し、行末まで続きます。（ただし、これは <SQL> ブロック内では機能しません。その場合は通常の SQL コメントの慣習を使用してください。）

<SQL> ブロック内に「}」文字を含める方法はありません。

セッションステップの各順列（spec ファイルで手動指定されたか、自動生成されたかにかかわらず）に対して、isolationtester はメインの setup 部分を実行し、次にセッションごとの setup 部分、選択されたセッションステップ、セッションごとの teardown、そしてメインの teardown スクリプトを実行します。選択された各ステップは、そのセッションに関連付けられた接続に送信されます。メインの setup および teardown スクリプトは、別の「制御」セッションで実行されます。


ブロッキングコマンドのサポート
=============================

各ステップには、さらなるアクションが取られるまでブロックされるコマンドが含まれている場合があります（多くの場合、他のセッションがブロックを解除するステップを実行するか、デッドロックを引き起こすまで）。この機能を使用するテストでは、有効な順列を手動で指定する必要があります。つまり、ブロックされたセッションがコマンドを実行することを期待しない順列です。テストがこのルールに従わない場合、isolationtester は 2 * PG_TEST_TIMEOUT_DEFAULT 秒後にそのテストをキャンセルします。キャンセルが機能しない場合、isolationtester は合計 4 * PG_TEST_TIMEOUT_DEFAULT 秒後に異常終了します。無効な順列のテストは、分離テストの実行時間が非常に長くなる可能性があり、有用なテスト目的を果たさないため、避けるべきです。

isolationtester は、pg_locks ビューで待機中として表示されているかどうかを確認することで、コマンドがブロックされたことを認識することに注意してください。したがって、重量級ロックでのブロックのみが検出されます。


競合状態への対処
============================

場合によっては、タイミングの問題により、テストスクリプトに対する isolationtester の出力が異なることがあります。これに対処する 1 つの方法は、バリアントの期待値ファイルを作成することです。これは、foo.spec のバリアントが foo_1.out、foo_2.out などと命名されるという通常の PG の慣習に従います。しかし、余分なファイルはメンテナンスの妨げになるため、この方法は推奨されません。通常は、permutation 行にリストされた一部のステップ名に特別なマーカーを適用することで、テスト出力を安定させることが可能です。

permutation エントリの一般的な形式は以下の通りです。

	<step name> [ ( <marker> [ , <marker> ... ] ) ]

各マーカーは「ブロッキング条件」を定義します。ステップは、すべてのブロッキング条件が満たされるまで完了として報告されません。可能なマーカーは以下の通りです。

	*
	<other step name>
	<other step name> notices <n>

mystep(*) のようなアスタスクマーカーは、ステップが起動された直後に、後で待機中として検出されるかどうかにかかわらず、isolationtester にそのステップを「waiting」として報告させます。これは、ステップの相対的な速度や isolationtester のステータス監視クエリに応じて、待機中として報告されることもあれば即座に完了として報告されることもあるケースを安定させるのに役立ちます。

ステップ名のみで構成されるマーカーは、そのステップが、他のステップが完了するまで完了として報告されない可能性があることを示します。これにより、2 つのクエリがどちらの順序で完了するかが見られるケースを安定させることができます。このステップは、他のステップが完了する前に「起動」される可能性があることに注意してください。（他のステップが現在の順列内で複数回使用されている場合、このステップはそれらのインスタンスのいずれかがアクティブな間は完了できません。）

"<other step name> notices <n>"（<n> は正の整数）という形式のマーカーは、このステップが、他のステップのセッションがこのステップの起動時から数えて少なくとも <n> 個の NOTICE メッセージを返すまで、完了として報告されない可能性があることを示します。これは、ステップが実際に完了する前に NOTICE メッセージを返す可能性があり、それらのメッセージを他のステップの完了と同期させる必要があるケースを安定させるのに役立ちます。

これらのマーカーは、ステップの起動ではなく、ステップの完了の報告を遅らせることしかできないことに注意してください。isolationtester は、(A) 同じセッションの先行するすべてのステップが完了し、(B) 順列内の直前のステップが完了したか、またはブロックされたとみなされた場合に、順列内の次のステップを起動します。この目的において「ブロックされたとみなされる」とは、データベースロックで待機していることが確認されたか、または完了しているがその完了報告がこれらのマーカーのいずれかによって遅延されていることを意味します。

場合によっては、ブロックされたとみなされる可能性のある別のセッションのステップが完了するまで、ステップを起動しないことが重要です。例として、セッション A のステップ s1 がセッション B のステップ s2 のキャンセルを発行している場合、s1 が完了したと確信するまで B の次のステップを起動しない方がよいでしょう。s1 がブロック可能である場合、問題が発生する可能性があります。これを防ぐ最善の方法は、セッション A に空のステップを作成し、マーカーなしで、セッション B の次のステップの直前に実行することです。空のステップは s1 が完了するまで起動できず、その結果、セッション B の次のステップは空のステップが終了するまで起動できません。
