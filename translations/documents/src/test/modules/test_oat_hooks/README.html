src/test/modules/test_oat_hooks/README

概要
====

このテストモジュール "test_oat_hooks" は、オブジェクトアクセスフック (OAT) を使用して強制アクセス制御 (MAC) を実施する方法の例です。

テスト戦略は以下の通りです。このモジュールがロードされると、さまざまなタイプのフックを登録します（下記参照）。各フックを制御するために GUC が定義されており、フックが起動するアクションを許可するか拒否するかを決定します。追加の GUC が 1 つあり、フックの冗長性を制御します。GUC はデフォルトで許可的/静的であり、ログにノイズを生成したり、回帰テスト開始までのアクティビティを拒否したりすることなくモジュールをロードできます。テストが開始されると、SET コマンドを使用してログ記録を有効にし、各フックの許可/制限動作を制御します。スーパーユーザーおよび一般ユーザーの両方の権限で、さまざまな SQL 文が実行されます。出力は期待される出力と比較され、フックが期待通りの順序で動作し、起動したことが検証されます。

ユーザーは OAT フックに対する他のシステムフックの起動順序に関心を持つ可能性があるため、ProcessUtility フックおよび ExecutorCheckPerms フックもこのモジュールによって登録され、独自のログ記録および許可/拒否動作が行われます。


SUSET テスト構成 GUC
====================

以下の構成パラメータ (GUC) は、このテストモジュールのオブジェクトアクセスタイプ (OAT)、プロセスユーティリティ、およびエグゼキュータチェック権限フックを制御します。一般的なパターンとして、各フックには、フックが呼び出される操作を許可するか拒否するかを制御する対応する GUC があります。実際の OAT フックは、「すべて許可」対「すべて拒否」よりもきめ細かい制御を提供する必要がありますが、テスト目的ではこれで十分です。

これらのフックがアクションを許可する場合でも、コア権限システムがそのアクションを拒否する場合があることに注意してください。コア権限システムに対するフックの起動順序は、アクションが拒否される前にどの NOTICE メッセージが出力されるかから推測できます。

各フックは、非スーパーユーザーによって実行されるすべての操作に対して、許可対拒否の設定を適用します。

- test_oat_hooks.deny_set_variable

  object_access_hook_str MAC 関数が構成パラメータの設定試行を拒否するかどうかを制御します。

- test_oat_hooks.deny_alter_system

  object_access_hook_str MAC 関数がシステム構成パラメータの変更試行を拒否するかどうかを制御します。

- test_oat_hooks.deny_object_access

  object_access_hook MAC 関数が呼び出されるすべての操作を拒否するかどうかを制御します。

- test_oat_hooks.deny_exec_perms

  exec_check_perms MAC 関数が呼び出されるすべての操作を拒否するかどうかを制御します。

- test_oat_hooks.deny_utility_commands

  ProcessUtility_hook 関数が呼び出されるすべての操作を拒否するかどうかを制御します。

- test_oat_hooks.audit

  各フックが各試行に対して、成功または失敗のステータスとともに NOTICE メッセージをログに記録するかどうかを制御します。この GUC をクリアまたは設定すると、新しい設定が有効になる前には表示されず後には表示される、あるいはその逆の NOTICE メッセージが生成される可能性があることに注意してください。


関数
====

このモジュールは、以下の名前でフックを登録します。

- REGRESS_object_access_hook

- REGRESS_object_access_hook_str

- REGRESS_exec_check_perms

- REGRESS_utility_command
