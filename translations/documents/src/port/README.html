libpgport
=========

libpgport は特別な動作をする必要があります。これはライブラリとアプリケーションの両方に機能を提供するためです。しかし、以下の2つの複雑な点があります。

1) ライブラリは、そのライブラリと全く同じフラグでコンパイルされたオブジェクトファイルを使用する必要があります。libpgport は同じフラグを使用しない可能性があるため、個々のライブラリ用にオブジェクトファイルを再コンパイルする必要があります。これは、リンク行から -lpgport を削除することで行われます。

        # libpgport のオブジェクトファイルを再コンパイルする必要がある
        LIBS := $(filter-out -lpgport, $(LIBS))

そして、オブジェクトファイルを再コンパイルするためのインフラストラクチャを追加します。

        OBJS= execute.o typename.o descriptor.o data.o error.o prepare.o memory.o \
                connect.o misc.o path.o exec.o \
                $(filter strlcat.o, $(LIBOBJS))

問題は、どのオブジェクトファイルを追加する必要があるかをテストする方法がないことですが、不足している関数は通常、ユーザーアプリケーションをリンクする際に判明します。

2) アプリケーションの場合、libpgport の静的ファイルが最初にリンクされるように、-lpq の前に -lpgport を使用します。これにより、アプリケーションが libpq によって「使用」されているものの、libpq からエクスポートされることを意図していないシンボルに依存することを回避できます。libpq の libpgport の使用状況は時間の経過とともに変化するため、そのような依存関係は問題となります。Windows、Linux、および macOS では、エクスポートリストを使用して libpq によってエクスポートされるシンボルを制御しています。
