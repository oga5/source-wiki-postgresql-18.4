V1関数マネージャインターフェース
---------------------------------

設計の核心は、関数検索の結果を表現するためのデータ構造と、特定の関数呼び出しに渡されるパラメータを表現するためのデータ構造です。（システムの多くの部分で同じ関数が繰り返し適用されるため、関数検索と関数呼び出しは分離しておく必要があります。検索のオーバーヘッドはクエリごとに1回支払うべきであり、タプルごとに1回支払うべきではありません。）

pg_proc内で関数が検索されると、その結果は以下のように表現されます。

typedef struct
{
    PGFunction  fn_addr;    /* 呼び出される関数またはハンドラへのポインタ */
    Oid         fn_oid;     /* 関数のOID（ハンドラがある場合でもハンドラのOIDではない） */
    short       fn_nargs;   /* 入力引数の数 (0..FUNC_MAX_ARGS) */
    bool        fn_strict;  /* 関数が「厳密(strict)」か（NULL入力 => NULL出力） */
    bool        fn_retset;  /* 関数が集合を返すか（複数回の呼び出しにわたる） */
    unsigned char fn_stats; /* track_functionsがこれより大きい場合に統計を収集する */
    void       *fn_extra;   /* ハンドラが使用するための追加領域 */
    MemoryContext fn_mcxt;  /* fn_extraを格納するためのメモリコンテキスト */
    Node       *fn_expr;    /* 呼び出しの式構文木、またはNULL */
} FmgrInfo;

通常の組み込み関数の場合、fn_addrは関数を実装するCルーチンのアドレスです。それ以外の場合は、対象の関数を含む関数クラスのハンドラのアドレスとなります。ハンドラは、関数OIDや、場合によってはfn_extraスロットを使用して、実行する特定のコードを見つけることができます。（fn_oid = InvalidOid は、まだ初期化されていないFmgrInfo構造体を示すために使用できます。関数検索コードによってFmgrInfoが最初に埋められるとき、fn_extraは常にNULLになりますが、関数ハンドラはクエリ中に同じFmgrInfoが繰り返し使用される際に自身の検索を繰り返さないように、これを設定することができます。）fn_nargsは関数が期待する引数の数、fn_strictは厳密性フラグ、fn_retsetは集合を返すかどうかを示します。これらの値はすべて、関数のpg_procエントリから取得されます。fn_statsも、この関数の呼び出しに関する実行時統計を追跡するかどうかを制御するために設定されます。

関数がSQL式の一部として呼び出される場合、fn_exprは関数呼び出しの式構文木を指します。これは、実際の引数に関する解析時の知識を抽出するために使用できます。このフィールドは実際には関数に関する情報ではなく引数に関する情報ですが、論理的にはFunctionCallInfoBaseDataに置くべきところを、FmgrInfoに保持しておく方が便利であることが判明しています。

関数の呼び出し中、以下のデータ構造が作成され、関数に渡されます。

typedef struct
{
    FmgrInfo   *flinfo;         /* この呼び出しに使用される検索情報へのポインタ */
    Node       *context;        /* 呼び出しのコンテキストに関する情報を渡す */
    Node       *resultinfo;     /* 結果に関する追加情報を渡すまたは返す */
    Oid         fncollation;    /* 関数が使用する照合順序 */
    bool        isnull;         /* 結果がNULLの場合、関数はtrueに設定しなければならない */
    short       nargs;          /* 実際に渡された引数の数 */
    NullableDatum args[];       /* 関数に渡される引数 */
} FunctionCallInfoBaseData;
typedef FunctionCallInfoBaseData* FunctionCallInfo;

flinfoは、呼び出しを行うために使用された検索情報を指します。通常の関数はこのフィールドを無視するでしょうが、関数クラスのハンドラは、呼び出されている特定の関数のOIDを見つけるためにこれを必要とします。

contextは「通常の」関数呼び出しではNULLですが、特定のコンテキストで関数が呼び出された場合には追加情報を指すことがあります。（例えば、トリガーマネージャはここで現在のトリガーイベントに関する情報を渡します。）詳細は以下の「関数呼び出しコンテキスト」を参照してください。

resultinfoは、単純なDatumの結果が期待される関数を呼び出すときはNULLです。関数がDatum以上のものを返す場合、Nodeの何らかのサブタイプを指すことがあります。（例えば、以下で説明するように、集合を返す関数を呼び出すときにresultinfoが使用されます。）contextフィールドと同様に、resultinfoは拡張のためのフックであり、fmgr自体はこのフィールドの使用を制限しません。

fncollationはパーサーによって導出された入力照合順序です。照合順序を持つ型の入力がない場合や、共通の照合順序を共有しない場合はInvalidOidとなります。これは実質的に隠された追加の引数であり、照合順序に依存する関数がその動作を決定するために使用できます。

nargsとargs[]は、関数に渡される引数を保持します。関数に渡されるすべての引数（およびその結果値）は、現在では一律にDatum型であることに注意してください。以下で説明するように、呼び出し元と呼び出し先は、特定の関数の実際の引数型に変換するために、標準のDatum変換マクロを適用する必要があります。args[i].isnullがtrueの場合、args[i].valueの値は未定義です。

渡される引数の数が呼び出し先の期待するものと一致していることを確認するのは、一般的に呼び出し元の責任です。可変個の引数を取る呼び出し先を除き、呼び出し先は通常nargsフィールドを無視し、args[]から値を取得するだけです。

isnullフィールドは呼び出し前に「false」に初期化されます。関数からの戻り時に、isnullは関数結果のNULLフラグとなります。これがtrueであれば、実際の関数の戻り値に関係なく、関数の結果はNULLです。単純な「厳密(strict)」関数は、args[i].isnullにTRUE値がある場合には呼び出されないため、isnullとargs[i].isnullの両方を無視できることに注意してください。

FunctionCallInfoは、FmgrValuesと、一連の場当たり的なパラメータ規約、グローバル変数（少なくともfmgr_pl_finfoとCurrentTriggerData）、およびその他の不格好な仕組みを置き換えるものです。

個々の関数であれ関数ハンドラであれ、呼び出し先は常に以下のシグネチャを持つ必要があります。

Datum function (FunctionCallInfo fcinfo);

これは以下のtypedefで表現されます。

typedef Datum (*PGFunction) (FunctionCallInfo fcinfo);

関数は、fcinfo->isnullを適切に設定し、Datumとして表現された結果を返す責任があります。すべての呼び出し先が現在では全く同じシグネチャを持ち、そのシグネチャで宣言された関数ポインタを通じて呼び出されるため、移植性や最適化の問題は発生しないはずです。

関数コーディング規約
---------------------------

提案されるマクロとコーディング規約は以下の通りです。

fmgrで呼び出し可能な関数の定義は、常に以下のようになります。

Datum
function_name(PG_FUNCTION_ARGS)
{
	...
}

"PG_FUNCTION_ARGS" は単に "FunctionCallInfo fcinfo" に展開されます。このマクロを使用する主な理由は、スクリプトが関数定義を見つけやすくするためです。しかし、もし将来的に呼び出し規約を変更することになった場合、このマクロが存在していると便利かもしれません。

非厳密(nonstrict)関数は、各引数がNULLかどうかをチェックする責任があり、これは PG_ARGISNULL(n) （単に "fcinfo->args[n].isnull"）で行うことができます。NULLである引数の値を取得しようとすることは避けるべきです。

厳密(strict)関数と非厳密(nonstrict)関数の両方とも、必要に応じて以下のようにNULLを返すことができます。
	PG_RETURN_NULL();
これは以下に展開されます。
	{ fcinfo->isnull = true; return (Datum) 0; }

引数値は通常、以下のようなコードで取得されます。
	int32	name = PG_GETARG_INT32(number);

float4、float8、int8の場合、PG_GETARGマクロは、それらの型が値渡し(pass-by-value)か参照渡し(pass-by-reference)かを隠蔽します。例えば、float8が参照渡しの場合、PG_GETARG_FLOAT8は以下のように展開されます。
	(* (float8 *) DatumGetPointer(fcinfo->args[number].value))
そして通常は以下のように呼び出されます。
	float8  arg = PG_GETARG_FLOAT8(0);
歴史的な理由により、浮動小数点関連のtypedefとマクロは型幅をバイト単位（4または8）で表現していますが、整数型の幅はビット単位でラベル付けすることを好んでいます。

NULL以外の値は、適切な型の PG_RETURN_XXX マクロで返されます。例えば、PG_RETURN_INT32 は以下のように展開されます。
	return Int32GetDatum(x)
PG_RETURN_FLOAT4、PG_RETURN_FLOAT8、PG_RETURN_INT64 は、必要に応じてpallocを行うことで、データ型が値渡しか参照渡しかを隠蔽します。

fmgr.h は、すべての基本データ型に対して PG_GETARG および PG_RETURN マクロを提供します。特殊なSQLデータ型（例：timestamp）を定義するモジュールやヘッダーファイルは、それらの型に対する適切なマクロを定義すべきであり、それにより型を操作する関数を標準的なスタイルで記述できるようになります。

非プリミティブデータ型（特に可変長型）の場合、データ型の参照渡しの性質を隠蔽することはあまり現実的ではないため、それらの型の PG_GETARG および PG_RETURN マクロは、DatumGetPointer/PointerGetDatum と適切な型キャスト以上のことはほとんど行いません（ただし、以下のTOASTの議論を参照）。そのような型を返す関数は、結果領域を明示的に palloc() する必要があります。そのような型の GETARG および RETURN マクロには、ポインタを生成または受け取ることを想起させるために、末尾に "_P" を付けることを推奨します。例えば、PG_GETARG_TEXT_P は "text *" を生成します。

関数が fcinfo->flinfo や FunctionCallInfo の他の補助フィールドにアクセスする必要がある場合は、そのままアクセスすべきです。これらのケースに対して構文糖衣マクロを提供することは有益ではないと考えられます。

TOAST可能なデータ型へのサポート
---------------------------------

TOAST可能なデータ型の場合、PG_GETARG マクロはTOAST解除されたデータ値を返します。TOASTされたままの値が必要なケースも稀にありますが、圧倒的多数のケースではTOAST解除された結果が求められるため、それがデフォルトとなります。TOAST解除を行わずに引数値を取得するには、PG_GETARG_RAW_VARLENA_P(n) を使用してください。

一部の関数は、入力値の変更可能なコピーを必要とします。このような場合、TOAST解除のためにデータをコピーしたのであれば、余分なコピー手順を行うのは無意味です。そのため、各TOAST可能なデータ型には、例えば PG_GETARG_TEXT_P_COPY(n) のような追加の取得マクロがあり、これは可能な限りTOAST解除ステップと組み合わせて、確実に新しいコピーを提供します。

また、PG_FREE_IF_COPY(ptr,n) マクロもあり、これは指定されたポインタがn番目の引数の元の値と異なる場合にのみ、そのポインタを pfree します。これは、n番目の引数が実際にTOAST解除された場合に、その値を解放するために使用できます。現在、コアバックエンドコードが一時的な領域を定期的に解放するため、関数の実行中にリークしたメモリは大きな問題にはならず、ほとんどの関数でこれを行う必要はありません。しかし、7.1の時点では、インデックス検索によって呼び出される関数内のメモリリークは、トランザクション終了までクリーンアップされません。したがって、pg_amop や pg_amproc にリストされている関数は、TOAST解除されたコピーをリークさせないように注意する必要があり、これらの関数はTOAST可能な入力に対して PG_FREE_IF_COPY() を使用する必要があります。

関数は、その結果値を再TOASTしようとしてはなりません。現在のメモリコンテキストで palloc された、TOASTされていない結果をそのまま返す必要があります。値が実際にタプルに格納されるとき、タプルのトースター(toaster)がTOASTが必要かどうかを決定します。

関数呼び出しコンテキスト
----------------------

呼び出し元が fcinfo->context にNULL以外のポインタを渡す場合、それは Node の何らかのサブタイプを指している必要があります。コンテキストの特定の種別は、ノード型フィールドによって示されます。（呼び出し先は、渡されているコンテキストの種類を知っていると仮定する前に、IsA() を介して常にノード型をチェックすべきです。）fmgr自体は、このフィールドの使用に対して他の制限を設けていません。

この規約の現在の使用例には以下が含まれます。* トリガー関数には struct TriggerData のインスタンスが渡され、トリガーコンテキストに関する情報が含まれます。（トリガー関数は通常の引数を受け取りません。）詳細およびトリガー関数で一般的に使用されるマクロについては、commands/trigger.h を参照してください。

* 集約関数（正確には、その遷移関数および最終関数）には、呼び出し元の Agg 計画ノードのエグゼキュータ状態ノードである struct AggState のインスタンスが渡されます。ウィンドウ関数として呼び出された場合は、struct WindowAggState のインスタンスが渡されます。これらのポインタは、fmgr.h で宣言され、src/backend/executor/nodeAgg.c のソースコード内でのみ文書化されている AggCheckCallContext() およびその関連関数を通じてのみ使用することが推奨されます。通常、これらのコンテキストノードは、遷移関数や最終関数が、スタンドアロンの SQL 関数としてではなく集約内で使用されていることを認識して実行を最適化したい場合にのみ関心対象となります。

* 真のウィンドウ関数は、struct WindowObject のインスタンスを受け取ります。（トリガー関数と同様に、通常の引数は受け取りません。）詳細については windowapi.h を参照してください。

* プロシージャには struct CallContext のインスタンスが渡され、CALL 文のコンテキストに関する情報、特に「アトミック」な実行コンテキスト内にあるかどうかの情報が含まれます。

* データ型入力関数（および将来的には他のクラスの関数）の呼び出し元の一部は、ErrorSaveContext のインスタンスを渡します。これは、呼び出し元がトランザクションを終了させる例外をスローせずに「ソフト」なエラーを処理したいことを示します。代わりに、呼び出された関数はエラーの原因に関する情報を ErrorSaveContext 構造体に格納し、ダミーの結果値を返す必要があります。詳細については、以下の「ソフトエラーの処理」を参照してください。


ソフトエラーの処理
--------------------

Postgres の標準的なエラー報告メカニズム（ereport() または elog()）は、あらゆる種類のエラー条件に使用されます。これは、ereport(ERROR) を介して例外をスローすると、例外キャッチャーが何が問題であったかについて多くの前提を置くことができないため、コストのかかるトランザクションまたはサブトランザクションの中断とクリーンアップが必要になることを意味します。完全なトランザクションクリーンアップなしで回復しても安全であることがわかっているエラーに対処するために、より軽量なメカニズムが必要な状況があります。SQL 呼び出し可能な関数は、ErrorSaveContext コンテキストメカニズムを使用してこのニーズをサポートできます。

「ソフト」エラーを報告するには、SQL 呼び出し可能な関数は、以前に ereport(ERROR, ...) を行っていた場所で、
	errsave(fcinfo->context, ...)
を呼び出す必要があります。渡された "context" が NULL であるか、ErrorSaveContext ノードではない場合、errsave は ereport(ERROR) とまったく同じように動作します。つまり、例外が longjmp を介してスローされるため、制御は戻りません。"context" が ErrorSaveContext ノードである場合、errsave の付随する報告呼び出しに含まれるエラー情報はコンテキストノードに格納され、errsave から制御が正常に戻ります。その後、関数は呼び出し元にダミー値を返す必要があります。（ダミー値としては SQL NULL が推奨されますが、呼び出し元は ErrorSaveContext ノードでエラーが報告されたことを確認すると関数の戻り値を無視することが期待されるため、何でも構いません。）

errsave() を呼び出した後に戻る以外に何もすることがない場合は、
	ereturn(fcinfo->context, dummy_value, ...)
と記述することで、errsave() を実行してから "return dummy_value" するという 1〜2 行を省略できます。

「ソフト」に報告されるエラーは、トランザクションの通常の処理を継続できることに疑いの余地がないという意味で、安全でなければなりません。この方法で処理すべきではないエラー条件には、メモリ不足、予期しない内部エラー、または簡単にクリーンアップできないものが含まれます。このようなケースは、これまで通り ereport でスローされるべきです。

データ型入力関数を例に挙げると、典型的な「ソフト」エラー条件には、入力構文エラーや範囲外の値が含まれます。入力関数は通常、単純な if テストでこのようなケースを検出し、その後の ereport 呼び出しを errsave または ereturn に簡単に変更できます。この制限のため、低レベル関数によって報告されるエラーは通常内部的なものと見なすのが妥当であるため、ErrorSaveContext ポインタを深く渡す必要は通常ありません。（この区別を別の言い方で表現すると、入力関数はすべての無効な入力条件をソフトに報告すべきですが、内部的な問題はハードエラーであるということです。）

トランザクションのクリーンアップは行われないため、errsave() の戻り後に終了する関数は、リソースのクリーンアップに対して責任を負う必要があります。呼び出し元は短命なメモリコンテキストで関数を実行しているはずであるため、palloc されたメモリのわずかなリークについて心配する必要はありません。ただし、ロック、開いているファイル、バッファピンなどのリソースは、エラー以外のコードパスと同様に、クリーンに閉じなければなりません。

エラーをトラップするために ErrorSaveContext メカニズムを使用する呼び出し元の規約については、nodes/miscnodes.h のその構造体の宣言とともに説明されています。


集合を受け取る、または返す関数
-----------------------------

関数が pg_proc で集合を返すものとしてマークされている場合、fcinfo->resultinfo が ReturnSetInfo 型のノードを指す状態で呼び出されます。集合を返そうとする関数は、resultinfo が NULL であるか、ReturnSetInfo ノードを指していない場合に、「集合結果を受け入れないコンテキストで呼び出されました」というエラーを発生させる必要があります。

現在、関数が集合結果を返すモードには、value-per-call（呼び出しごとの値）と materialize（マテリアライズ）の 2 つがあります。value-per-call モードでは、関数は呼び出されるたびに 1 つの値を返し、返す値がなくなると最終的に「完了」を報告します。materialize モードでは、関数の出力集合は Tuplestore オブジェクトにインスタンス化され、すべての値が 1 回の呼び出しで返されます。将来的に追加のモードが追加される可能性があります。

ReturnSetInfo には「allowedModes」フィールドが含まれており、これは（呼び出し元によって）呼び出し元がサポートできるモードの OR ビットマスクに設定されます。関数によって実際に使用されるモードは、別のフィールド「returnMode」で返されます。後方互換性の理由から、returnMode は value-per-call に初期化されており、関数が別のモードを使用したい場合にのみ変更する必要があります。関数は、呼び出し元がサポートするモードのいずれも使用できない場合は ereport() を行う必要があります。

value-per-call モードは次のように機能します。ReturnSetInfo には「isDone」フィールドが含まれており、これを以下のいずれかの値に設定する必要があります。

    ExprSingleResult             /* 式は集合を返さない */
    ExprMultipleResult           /* この結果は集合の要素である */
    ExprEndResult                /* 集合にはこれ以上要素がない */

（呼び出し元はこれを ExprSingleResult に初期化します）。関数が ReturnSetInfo に触れずに単に Datum を返す場合、呼び出しは終了し、単一項目の集合が返されたことになります。集合を返すには、関数は集合の各要素に対して isDone を ExprMultipleResult に設定する必要があります。すべての要素が返された後、次の呼び出しで isDone を ExprEndResult に設定し、null 結果を返す必要があります。（最初の呼び出しでこれを行うことで空の集合を返すことも可能です。）

value-per-call 関数は、完了まで実行されると想定してはなりません。エグゼキュータは、例えば LIMIT のために、呼び出しを単に停止する可能性があります。したがって、最終的な呼び出しでリソースのクリーンアップを実行しようとするのは安全ではありません。通常、メモリをクリーンアップする必要もありません。ファイル記述子などの他の種類のリソースをクリーンアップする必要がある場合は、ReturnSetInfo ノードが指す ExprContext にシャットダウンコールバック関数を登録できます。（ただし、ファイル記述子は限られたリソースであるため、呼び出しをまたいで開いたままにするのは一般的に賢明ではありません。ファイルアクセスを必要とする SRF は、Materialize モードを使用して 1 回の呼び出しで行うように記述する方が適切です。）

Materialize モードは次のように機能します。関数は（空の可能性がある）結果集合を保持する Tuplestore を作成し、それを返します。複数の呼び出しはありません。関数は、タプルの構造を示す TupleDesc も返す必要があります。Tuplestore と TupleDesc は econtext->ecxt_per_query_memory コンテキストで作成する必要があります（これは関数が呼び出されるコンテキストとは*異なる*ことに注意してください）。関数は Tuplestore と TupleDesc へのポインタを ReturnSetInfo に格納し、returnMode を設定して materialize モードであることを示し、null を返します。isDone は使用されないため、ExprSingleResult のままにしておく必要があります。

allowedModes で SFRM_Materialize_Random が設定されている場合、Tuplestore は randomAccess = true で作成する必要がありますが、そうでない場合は randomAccess = false で作成できます（作成することが推奨されます）。ValuePerCall モードと Materialize モードの両方をサポートできる呼び出し元は、優先するモードに応じて SFRM_Materialize_Preferred を設定するかどうかを決定します。

利用可能な場合、期待されるタプル記述子が ReturnSetInfo に渡されます。他のコンテキストでは expectedDesc フィールドは NULL になります。関数は expectedDesc に注意を払う必要はありませんが、特別な場合には役立つことがあります。

InitMaterializedSRF() は、Materialize モードに適した構成で Tuplestore と TupleDesc を入力し、1 回の呼び出しのために関数の ReturnSetInfo をセットアップできるヘルパー関数です。

集合を受け取る関数のサポートはありません。代わりに、関数は入力集合の各要素に対して 1 回ずつ、複数回呼び出されます。


関数ハンドラに関する注意
-----------------------------

関数クラスのハンドラは、この設計により、より簡単かつクリーンに実装できるはずです。呼び出された関数の OID は渡されたパラメータから直接到達可能であり、グローバル変数 fmgr_pl_finfo はもう必要ありません。また、fcinfo->flinfo->fn_extra を変更することで、ハンドラはルックアップ情報をキャッシュし、同じ関数が何度も呼び出されたときの繰り返しルックアップを回避できます。（fn_extra はヒントとしてのみ使用できます。呼び出し元が FmgrInfo 構造体を再利用する必要はないためです。しかし、パフォーマンスが重要なパスでは通常再利用されます。）

ハンドラが fn_extra データを保持するためにメモリを割り当てたい場合、CurrentMemoryContext で行ってはなりません。現在のコンテキストは、FmgrInfo があるコンテキストよりも寿命が短い可能性があるためです。代わりに、flinfo->fn_mcxt コンテキスト、または長寿命のキャッシュコンテキストにメモリを割り当ててください。fn_mcxt は通常、FmgrInfo 構造体が作成された時点での CurrentMemoryContext を指します。いずれにせよ、少なくとも FmgrInfo 自体と同じくらい長寿命のコンテキストである必要があります。
