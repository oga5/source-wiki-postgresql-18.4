メモリコンテキストシステムの設計概要
=====================================

背景
----------

PostgreSQLでは、メモリ割り当ての大部分を「メモリコンテキスト」内で行っています。これは通常、src/backend/utils/mmgr/aset.c で実装されている AllocSets です。多くのオーバーヘッドを伴わずにメモリ管理を成功させる鍵は、適切な寿命を持つ有用なコンテキストのセットを定義することです。

メモリコンテキストに対する基本的な操作は以下の通りです。

* コンテキストの作成

* コンテキスト内でのメモリチャンクの割り当て（標準Cライブラリの malloc() に相当）

* コンテキストの削除（その中で割り当てられたすべてのメモリの解放を含む）

* コンテキストのリセット（コンテキストオブジェクト自体は解放せず、その中で割り当てられたすべてのメモリを解放する）

* コンテキストに割り当てられたメモリ総量の照会
  （コンテキストがチャンクを割り当てるための生メモリ量であり、チャンクそのものの量ではない）

コンテキストから以前に割り当てられたメモリチャンクがある場合、それを解放したり、サイズを拡大・縮小して再割り当てしたりすることができます（標準Cライブラリの free() および realloc() ルーチンに相当）。これらの操作は、チャンクが元々割り当てられていたのと同じコンテキストに対してメモリを返却、あるいはそこからメモリを取得します。

常に CurrentMemoryContext というグローバル変数で示される「現在の」コンテキストが存在します。palloc() は暗黙的にそのコンテキスト内に領域を割り当てます。MemoryContextSwitchTo() 操作は新しい現在のコンテキストを選択します（また、呼び出し元が終了前に以前のコンテキストを復元できるように、以前のコンテキストを返します）。

malloc/free を単純に使用する場合と比較したメモリコンテキストの主な利点は、メモリコンテキストの内容全体を簡単に解放できることです。その中の個々のチャンクごとに解放を要求する必要はありません。これは、チャンクごとの簿記管理よりも高速で信頼性が高い方法です。私たちはこの事実を利用して、トランザクション終了時にクリーンアップを行います。トランザクションまたはそれより短い寿命を持つすべてのアクティブなコンテキストをリセットすることで、すべての一時的なメモリを回収できます。同様に、各クエリの終了時や、クエリ処理中の各タプル処理後にもクリーンアップを行うことができます。


palloc API と標準Cライブラリに関するいくつかの注意点
---------------------------------------------------------

palloc とその関連関数の動作は標準Cライブラリの malloc とその関連関数に似ていますが、意図的な違いもいくつかあります。動作を明確にするための注意点を以下に示します。

* メモリ不足の場合、palloc および repalloc は elog(ERROR) を介して終了します。これらは決して NULL を返さないため、そのような結果をテストする必要も有用性もありません。palloc_extended() を使用すると、MCXT_ALLOC_NO_OOM フラグを使用してその動作を上書きできます。

* palloc(0) は明示的に有効な操作です。NULL ポインタではなく、バイトを使用できない有効なチャンクを返します。ただし、そのチャンクは後で repalloc() によってサイズを大きくされる可能性があり、エラーなしで pfree() することも可能です。同様に、repalloc はサイズゼロへの再割り当てを許可します。

* pfree および repalloc は NULL ポインタを受け付けません。これは意図的なものです。（repalloc については、前述の通り repalloc は現在のメモリコンテキストに依存しないため、これが必要です。しかし、その場合、どのメモリコンテキストで割り当てを行うかを知る必要があるため、最初の割り当ては repalloc の外で行う必要があります。pfree については、この動作は主に歴史的な理由によるものであり、追加のチェックがパフォーマンスに影響を与えるためでもあります。）


現在のメモリコンテキスト
--------------------------

呼び出されるルーチンごとに適切なメモリコンテキストを常に渡すのは表記上のオーバーヘッドが大きすぎるため、常に現在のメモリコンテキスト CurrentMemoryContext という概念が存在します。これがないと、例えば copyObject ルーチンや、参照渡しデータ型を返す関数実行ルーチンにコンテキストを渡す必要が生じます。内部で一時的に領域を割り当てるが呼び出し元には返さないルーチンについても同様です。「あなたがやりたいかもしれない一時的なメモリ割り当てのために、ここにコンテキストがあります」という引数でシステム内のすべての呼び出しを煩雑にしたくはありません。

しかし、その推論の結論として、CurrentMemoryContext は可能な限り短い寿命のコンテキストを指すべきです。クエリ実行中、それは通常、各タプル後にリセットされるコンテキストを指します。永続的なメモリリークのリスクがあるため、トランザクションより長い寿命を持つコンテキストを指すのは、*非常に*限定されたコード内のみであるべきです。


pfree/repalloc は CurrentMemoryContext に依存しない
----------------------------------------------------

pfree() および repalloc() は、CurrentMemoryContext に属しているかどうかにかかわらず、どのチャンクに対しても適用できます。チャンクが属するコンテキストが、いずれの場合も操作を処理するために呼び出されます。


「親」および「子」コンテキスト
-----------------------------

すべてのコンテキストが独立していると、特にエラー発生時にそれらを追跡するのが困難になります。これは「親」および「子」コンテキストのツリーを作成することで解決されます。メモリコンテキストを作成する際、新しいコンテキストを既存のコンテキストの子として指定できます。コンテキストは複数の子を持つことができますが、親は1つだけです。このようにして、コンテキストは森（必ずしも単一のツリーではなく、複数のトップレベルコンテキストが存在する可能性があるため。ただし現在の慣習では、TopMemoryContext という1つのトップコンテキストのみが存在します）を形成します。

コンテキストを削除すると、その直接的および間接的なすべての子も削除されます。コンテキストをリセットする場合、子コンテキストも削除する方がほとんどの場合有用です。そのため MemoryContextReset() はそのように動作します。もし本当に空のコンテキストのツリーが必要な場合は、MemoryContextResetOnly() に加えて MemoryContextResetChildren() を呼び出す必要があります。

これらの機能により、リークを恐れることなく多数のコンテキストを管理できます。トランザクション終了時に削除する1つのトップレベルコンテキストのみを追跡し、作成する短い寿命のコンテキストがすべてそのコンテキストの子孫であることを確認するだけで済みます。ツリーは複数のレベルを持つことができるため、トランザクションごと、ステートメントごと、スキャンごと、タプルごとといった、ネストされたストレージの寿命を簡単に扱うことができます。部分的にしか重ならないストレージの寿命は、コンテキストの森の異なるツリーから割り当てることで処理できます（次のセクションにいくつかの例があります）。

利便性のため、「特定のコンテキストのすべての子をリセット/削除するが、そのコンテキスト自体はリセットまたは削除しない」といった操作も提供しています。


メモリコンテキストのリセット/削除コールバック
-------------------------------------

Postgres 9.5 で導入された機能により、メモリコンテキストを単なる palloc されたメモリ以上のリソース管理に使用できるようになりました。これは、メモリコンテキストに対して「リセットコールバック関数」を登録することで行われます。このような関数は、コンテキストが次にリセットまたは削除される直前に一度だけ呼び出されます。これは、コンテキスト内で割り当てられたオブジェクトに関連付けられたリソースを解放するために使用できます。考えられるユースケースは以下の通りです。
* tuplesort オブジェクトに関連付けられたオープンファイルを閉じる。
* リセットされるコンテキスト内のオブジェクトによって保持されている、長寿命のキャッシュオブジェクトの参照カウントを解放する。
* palloc されたオブジェクトに関連付けられた malloc 管理のメモリを解放する。
最後のケースは、純粋な Postgres コードとしては悪いプログラミング慣習を表しているだけです。ターゲットコンテキストまたはその子コンテキストで palloc を使用してすべての割り当てを行う方が適切です。ただし、Postgres 以外のライブラリとインターフェースするコードには非常に役立つ可能性があります。

メモリコンテキストに対して任意の数のリセットコールバックを設定できます。これらは登録の逆順で呼び出されます。また、コンテキストのツリーがリセットまたは削除される場合、子コンテキストに付加されたコールバックは、親コンテキストに付加されたコールバックの前に呼び出されます。

このための API では、呼び出し元がコールバックの状態を保持するために MemoryContextCallback メモリチャンクを提供する必要があります。通常、これは論理的に付加されるのと同じコンテキスト内に割り当てるべきであり、そうすることで使用後に自動的に解放されます。呼び出し元にこのメモリの提供を求める理由は、ほとんどの利用シナリオにおいて、呼び出し元がターゲットコンテキスト内に大きな構造体を作成するためであり、MemoryContextCallback 構造体をこの大きな構造体に含めることで、個別の palloc() 呼び出しなしで「無料で」作成できるためです。


実践におけるメモリコンテキスト
===========================

グローバルに知られているコンテキスト
-----------------------

通常、グローバル変数を介して参照される、広く知られたコンテキストがいくつかあります。システムにはいつでも多くの追加コンテキストが存在する可能性がありますが、エラー発生時にリークしないように、他のすべてのコンテキストはこれらのコンテキストの直接的または間接的な子である必要があります。

TopMemoryContext --- これはコンテキストツリーの実際のトップレベルです。他のすべてのコンテキストは、これの直接的または間接的な子です。ここで割り当てることは本質的に「malloc」と同じです。なぜなら、このコンテキストは決してリセットまたは削除されないからです。これは永遠に存続すべきもの、あるいは制御モジュールが適切なタイミングで削除を担当するもの用です。例として fd.c のオープンファイルテーブルがあります。本当に必要な場合を除いてここで割り当てることは避け、特に CurrentMemoryContext がここを指した状態で実行することは避けてください。

PostmasterContext --- これは postmaster の通常の作業コンテキストです。バックエンドが生成された後、postmaster が使用していた不要なメモリのコピーを解放するために、PostmasterContext を削除できます。
EXEC_BACKEND 以外のビルドでは、pg_hba.conf および pg_ident.conf データの postmaster のコピーが、バックエンドプロセスでの認証中に直接使用されることに注意してください。そのため、バックエンドはそれが完了するまで PostmasterContext を削除できません。（postmaster は TopMemoryContext、PostmasterContext、ErrorContext のみを持っています。残りのトップレベルコンテキストは、起動時に各バックエンドで設定されます。）

CacheMemoryContext --- relcache、catcache、および関連モジュールの永続ストレージです。これも決してリセットまたは削除されないため、TopMemoryContext と区別する必要は本来ありません。しかし、デバッグ目的で区別を維持する価値はあると思われます。（注: CacheMemoryContext は、より短い寿命を持つ子コンテキストを持っています。例えば、relcache エントリに関連付けられた補助ストレージを保持するには子コンテキストが最適です。そうすれば、信頼できる freeObject() を構築することに依存せずに、ルール構文木などを簡単に解放できます。）

MessageContext --- このコンテキストは、フロントエンドからの現在のコマンドメッセージと、現在のメッセージと同じ期間だけ存続すればよい派生ストレージを保持します（例えば、simple-Query モードでは、解析木や計画木はここに存在できます）。このコンテキストは、PostgresMain の外側のループの各サイクルの先頭でリセットされ、すべての子が削除されます。クエリ文字列は、単一のトランザクションやポータルよりも長く、あるいは短く存続する必要がある可能性があるため、これはトランザクションごとやポータルごとのコンテキストとは別に保持されます。

TopTransactionContext --- これはトップレベルトランザクションの終了まで存続するすべてを保持します。このコンテキストは、各トップレベルトランザクションサイクルの終了時にリセットされ、すべての子が削除されます。ほとんどの場合、ここで直接割り当てるのではなく、CurTransactionContext で割り当てるべきです。ここに属するのは、複数のサブトランザクション間でステータスを管理するために明示的に存在する制御情報です。注: このコンテキストはエラー発生時に即座にクリアされるわけではありません。その内容は COMMIT/ROLLBACK によってトランザクションブロックが終了するまで存続します。CurTransactionContext --- 現在のトランザクションの終了まで存続させる必要があるデータを保持します。特に、トップレベルのトランザクションコミット時に必要となります。トップレベルのトランザクション内ではこれは TopTransactionContext と同じですが、サブトランザクション内では子コンテキストを指します。サブトランザクションがアボートした場合、その CurTransactionContext はアボート処理の完了後に破棄されることを理解しておくことが重要です。しかし、コミットされたサブトランザクションの CurTransactionContext は（もちろん中間レベルのサブトランザクションがアボートしない限り）トップレベルのコミットまで保持されます。これにより、失敗したサブトランザクションのデータを必要以上に長く保持しないようにしています。この動作のため、サブトランザクションのアボート時には適切にクリーンアップを行うよう注意しなければなりません。サブトランザクションの状態は、上位のトランザクションで保持されているポインタやリストから切り離す必要があります。そうしないと、ダングリングポインタが発生し、トップレベルのコミット時にクラッシュする原因となります。ここに保持されるデータの例として、保留中の NOTIFY メッセージがあります。これはトップレベルのコミット時に送信されますが、生成元のサブトランザクションがアボートしなかった場合に限られます。

PortalContext --- これは実際には独立したコンテキストではなく、現在アクティブな実行ポータルのポータルごとのコンテキストを指すグローバル変数です。現在のポータルの実行に必要な期間だけ存続するストレージを割り当てる必要がある場合に使用できます。

ErrorContext --- この永続的なコンテキストは、エラー回復処理のために切り替えられ、回復の完了時にリセットされます。常に数 KB のメモリが利用できるように配置されています。これにより、バックエンドでメモリが不足した場合でも、エラー回復のためにメモリを確保できます。その結果、メモリ不足を FATAL エラーではなく、通常のエラー条件として扱うことが可能になります。


準備されたステートメントとポータルのためのコンテキスト
------------------------------------------------------------

準備されたステートメントオブジェクトには、関連付けられたプライベートコンテキストがあり、そこにクエリのパースツリーと計画ツリーが格納されます。これらのツリーはエグゼキュータに対して読み取り専用であるため、準備されたステートメントはこれらのツリーを再コピーすることなく何度も再利用できます。

実行ポータルオブジェクトには、ポータルがアクティブなときに PortalContext によって参照されるプライベートコンテキストがあります。DECLARE CURSOR によって作成されたポータルの場合、このプライベートコンテキストにはクエリのパースツリーと計画ツリーが含まれます（これらを保持できる他のオブジェクトが存在しないため）。準備されたステートメントから作成されたポータルは、単に準備されたステートメントのツリーを参照するだけであり、プライベートコンテキストにストレージを割り当てる必要はありません。


論理レプリケーションワーカーのコンテキスト
-------------------------------------------

ApplyContext --- 適用ワーカーの全存続期間中、永続的です。ここでも TopMemoryContext を使用することは可能ですが、メモリ使用量分析を単純化するために別のコンテキストを作成しています。

ApplyMessageContext --- 各論理レプリケーションプロトコルメッセージが処理された後にリセットされる短命なコンテキストです。


実行中の過渡的なコンテキスト
----------------------------

準備されたステートメントを作成する際、パースツリーと計画ツリーは MessageContext の子である一時的なコンテキスト内に構築されます（エラー発生時に自動的に破棄されるようにするため）。成功すると、完成した計画は準備されたステートメントのプライベートコンテキストにコピーされ、一時的なコンテキストは解放されます。これにより、実行が開始される前にプランナの一時領域を回収できます。（単純なクエリモードでは、余分なコピー手順は行わないため、プランナの一時領域はクエリの終了まで残ります。）

トップレベルのエグゼキュータルーチンや、ほとんどの「計画ノード」実行コードは、通常、ExecutorStart によって作成され ExecutorEnd によって破棄されるコンテキスト内で実行されます。このコンテキストは、ExecutorStart 中に構築された「計画状態」ツリーも保持します。これらのルーチンで割り当てられるメモリのほとんどはクエリ終了まで存続することを意図しているため、これらの目的に適しています。エグゼキュータのトップコンテキストは PortalContext の子、つまりクエリの実行を表すポータルのポータルごとのコンテキストです。

エグゼキュータにおける主なメモリ管理の考慮事項は、式評価（qual テストとターゲットリストエントリの計算の両方）でメモリリークが発生しないようにすることです。これを行うために、エグゼキュータ内で作成される各 ExprContext（式評価コンテキスト）には、それに関連付けられたプライベートメモリコンテキストがあり、その ExprContext 内で式を評価する際にはそのコンテキストに切り替えます。ExprContext を所有する計画ノードは、式評価の結果が不要になったときにプライベートコンテキストを空にリセットする責任があります。通常、リセットは計画ノード内の各タプル取得サイクルの開始時に行われます。

この設計により、各計画ノードは独自の式評価メモリコンテキストを持つことになります。これはネストされた結合を適切に処理するために必要なようです。なぜなら、外部の計画ノードは、内部ノードから次のタプルを取得する間に計算した式の結果を保持する必要があるかもしれない一方で、内部ノードはタプルを返す前に多くのタプルサイクルと多くの式を実行する可能性があるからです。内部ノードは、外部タプルサイクルごとに1回よりも頻繁に独自の式コンテキストをリセットできなければなりません。幸いなことに、メモリコンテキストは十分に安価であるため、各計画ノードに1つずつ与えても問題にはならないようです。

クエリ存続期間のコンテキストでインデックスアクセスやソートを実行することの問題は、これらの操作がデータ型固有の比較関数を呼び出すことであり、もし比較関数がメモリをリークすると、そのメモリはクエリ終了まで回収されないことです。比較関数はすべて bool または int32 を返すため、結果データに問題はありませんが、内部の一時データのリークが問題になる可能性があります。特に、TOAST 可能なデータ型を操作する比較関数は、デトーストされた入力のコピーをリークしないように注意する必要があります。これは面倒ですが、インデックスやソートのルーチンを修正するよりも比較関数を適合させる方がはるかに簡単であると思われたため、7.1 ではそのように行われました。これは btree およびハッシュインデックスにおける現状のままであるため、btree およびハッシュのサポート関数はメモリをリークしないようにする必要があります。他のほとんどのインデックス AM は、opclass サポート関数を短命なコンテキストで実行するように変更されているため、リークは問題になりません。サポート関数がはるかに複雑になる傾向があることを考えると、これは必要なことです。

集約関数のような特別なケースもあります。nodeAgg.c は、あるタプルサイクルから次のタプルサイクルへ集約遷移関数の評価結果を記憶する必要があるため、各サイクルですべてのタプルごとの状態を破棄するわけにはいきません。これを処理する最も簡単な方法は、集約ノード内に2つのタプルごとのコンテキストを持ち、それらの間でピンポンを行うことです。これにより、各タプルにおいて一方がアクティブな割り当てコンテキストとなり、もう一方が前サイクルの遷移関数によって割り当てられた結果を保持します。

アクティブな CurrentMemoryContext を切り替えるエグゼキュータルーチンは、戻る前に呼び出し元の現在のメモリコンテキストにデータをコピーする必要がある場合があります。しかし、実行サイクルの終了時ではなく「開始」時にタプルごとのコンテキストをリセットするという慣習により、その必要性は最小限に抑えられています。このルールにより、実行ノードはタプルごとのコンテキストで palloc されたタプルを返すことができ、そのタプルはノードが別のタプルを要求されるか実行終了を指示されるまで有効なままとなります。これは、テーブルスキャンレベルでの参照渡し値の状況と並行しています。スキャンノードは、その期間だけ有効であることが保証されているディスクバッファ内のタプルへの直接ポインタを返すことができるためです。

データをコピーするより一般的な理由は、タプルごとのコンテキストからクエリごとのコンテキストへ結果を転送するためです。例えば、Unique ノードは最後の重複排除されたタプル値をクエリごとのコンテキストに保存するため、コピー手順が必要になります。


複数のタイプのコンテキストを許可するためのメカニズム
--------------------------------------------------

異なる割り当てパターンを効率的に許可し、実験を行うために、異なる割り当てポリシーを持ちながら外部動作は類似している、異なるタイプのメモリコンテキストを許可しています。これを処理するために、メモリ割り当て関数には関数ポインタを介してアクセスし、すべてのコンテキストタイプがここで与えられた慣習に従うことを要求しています。

メモリコンテキストは struct MemoryContextData（memnodes.h を参照）によって表されます。この構造体はコンテキストの正確なタイプを識別し、親コンテキストや子コンテキスト、コンテキスト名など、異なるタイプの MemoryContext 間で共通の情報を含みます。

これは本質的に抽象スーパークラスであり、動作はどの MemoryContextMethods のセットを使用するかを参照する「methods」ポインタによって決定されます。特定のメモリコンテキストタイプは、これらのフィールドを最初のフィールドとして持つ派生構造体を使用します。特定のタイプのすべてのコンテキストは、mcxt.c で定義されている mcxt_methods[] 配列内の対応する要素を指すメソッドポインタを持ちます。

コンテキストからの割り当てやリセットのような操作は関連する MemoryContext をパラメータとして取りますが、free や realloc のような操作はより複雑です。これらを機能させるために、すべてのメモリコンテキストタイプは、割り当てられたチャンクの直前に、パディングなしで uint64 値を配置することを要求しています。この値の下位4ビットは、所有するコンテキストの MemoryContextMethodID に設定されます。これにより、コードは4ビットをインデックスとして使用して mcxt_methods[] 配列を検索し、使用すべき正しい MemoryContextMethods を決定できます。

あるタイプの割り当て機能が、割り当てサイズなどのチャンクに関する追加情報を必要とする場合、その情報は先行する uint64 値の残りの60ビットにエンコードするか、より多くのスペースが必要な場合は uint64 値の直前にさらに値を格納することができます。これを管理するのはコンテキスト実装側の責任です。

これを踏まえると、pfree のようなルーチンは、GetMemoryChunkMethodID() を呼び出して mcxt_methods[] 配列内の対応する MemoryContextMethods を見つけることで、どの MemoryContextMethods セットの free_p 関数を呼び出すべきかを決定できます。便宜上、MCXT_METHOD() マクロが提供されており、コードを以下のように単純化しています。

void
pfree(void *pointer)
{
	MCXT_METHOD(pointer, free_p)(pointer);
}

現在のすべてのメモリコンテキストは、memutils_memorychunk.h で定義されている MemoryChunk ヘッダータイプを使用しています。これは、uint64 ヘッダーの残りの60ビットを使用して、メモリチャンクのサイズ（または aset.c の場合はフリーリストインデックス）と、チャンクが属するブロックへの参照を取得するためにチャンクから減算する必要があるバイト数を効率的にエンコードするため、既存のすべてのコンテキストタイプに適しています。それぞれに30ビットが使用されますが、チャンクからブロックへのオフセットの最下位ビットはチャンクサイズの最上位ビットと同じビットであるため、合計で59ビットしか使用されません。この重複は、ブロックとチャンク間の相対オフセットが MAXALIGN された値であると想定され、最下位ビットが常に0であることが保証されているため可能です。これらの各フィールドに30ビット以上が必要な場合、メモリコンテキストはそれを自身で管理する必要があります。これは、指定されたチャンクに対して MemoryChunkSetHdrMaskExternal() 関数を呼び出すことで実行できます。チャンクが外部チャンクかどうかは、64ビット MemoryChunk の残りの1ビットで判断できます。

現在、各メモリコンテキストタイプは大きな割り当てを専用のブロック（常に単一のチャンクのみを含む）に格納します。これらについては、チャンクが指定されたブロック上の最初のチャンクでなければならないことがわかっているため、ブロックの検索は単純であり、ブロックは常にチャンクからの固定オフセットにあります。これらについては、ブロックが常に endptr を格納しており、それを使用してチャンクのサイズを計算できるため、チャンクサイズの検索も単純です。

aset.c の動作に対するより詳細な制御
-----------------------------------デフォルトでは、aset.cはコンテキスト内での最初の割り当て時に常に8Kのブロックを割り当て、それ以降のブロック要求ごとにそのサイズを倍増させます。これは、*大量の*データを保持する可能性のあるコンテキストにとっては適切な動作です。しかし、システム内に数十、あるいは数百もの小さなコンテキストが存在する場合、もう少し細かく調整できるようにする必要があります。

コンテキストの作成者は、初期ブロックサイズと最大ブロックサイズを指定できます。小さな値を指定することで、それほど多くのデータを保持しないと予想されるコンテキスト（例：relcacheの各リレーション用コンテキスト）でのスペースの無駄を防ぐことができます。

また、何らかの理由で追加ブロックの初期サイズとは異なる必要がある場合に備えて、最小コンテキストサイズを指定することも可能です。aset.cのコンテキストには、minContextSizeが指定されていればそのサイズのブロックが、指定されていなければinitBlockSizeのサイズのブロックが、常に少なくとも1つ含まれます。

タプルごとのコンテキストは頻繁にリセットされ、通常、タプルサイクルごとにそれほど多くのスペースを割り当てないことが予想されます。この使用パターンを低コストにするため、コンテキスト内で最初に割り当てられたブロックは、リセット時にmalloc()に返されるのではなく、単にクリアされます。これにより、mallocの頻繁な呼び出し（スラッシング）を回避します。

代替のメモリコンテキスト実装
------------------------------------------

aset.c (AllocSetContext) は、デフォルトの汎用アロケータです。これ以外に、特定の目的を持つ3つのアロケータタイプが存在します。

* slab.c (SlabContext) は、固定サイズのチャンクを割り当てるために設計されています。固定チャンクサイズは、コンテキスト作成時に指定する必要があります。新しいチャンクは最も詰まったブロックに割り当てられ、使用中のチャンクを密に配置することでメモリの断片化を防ぎます。これにより、チャンクをpfreeした際にブロック内の全チャンクが空になり、オペレーティングシステムに解放できる可能性が高まります。

* generation.c (GenerationContext) は、チャンクが同様の寿命（世代）を持つグループで割り当てられる場合、または概ねFIFO順で割り当てられる場合に最適です。pfreeされたチャンクによって残されたスペースを再利用する試みは行われません。ブロックは、その上のすべてのチャンクがpfreeされたときにオペレーティングシステムに返されます。

* bump.c (BumpContext) は、個別にpfreeやrepallocを行う必要のない、密に割り当てられたメモリチャンクを必要とするユースケースに最適です。BumpContextのチャンクにはチャンクヘッダーがないため、これらの操作はサポートされていません。チャンクヘッダーがないということは、チャンクがより密に配置されることを意味し、これは小さな割り当てを大量に行うワークロードで特に有用です。ブロックは、コンテキストがリセットまたは削除されたときにのみオペレーティングシステムに解放されます。

詳細については、対応する.cファイルのヘッダーコメントをお読みください。

メモリアカウンティング
-----------------

基本的なメモリコンテキスト操作の1つに、コンテキスト（およびその子コンテキスト）で使用されているメモリ量を特定することがあります。独自の場当たり的なメモリアカウンティングを実装している箇所が複数ありますが、これは統一されたアプローチを提供することを目的としています。場当たり的なアカウンティングソリューションは、割り当てを厳密に制御している場所や、割り当てられたチャンクのサイズを簡単に特定できる場所（例：タプルのみを扱う場所）では機能します。

メモリコンテキストに組み込まれたアカウンティングは透過的であり、適切なメモリコンテキストのサブツリー内に割り当てが行われる限り、すべての割り当てに対して透過的に機能します。

例えば集約関数を考えてみましょう。集約状態は多くの場合、任意の構造体で表現され、遷移関数から割り当てられるため、場当たり的なアカウンティングでは機能しない可能性が高いです。しかし、組み込みのアカウンティングであれば、そのようなケースも問題なく処理できます。

オーバーヘッドを最小限に抑えるため、アカウンティングは個々の割り当てチャンク単位ではなく、ブロック単位で行われます。

アカウンティングは遅延評価されます。ブロックが割り当て（または解放）された後、そのブロックを所有するコンテキストのみが更新されます。つまり、特定のコンテキストのメモリ使用量を問い合わせる際には、すべての子コンテキストを再帰的にたどる必要があります。このため、メモリアカウンティングは、（関連するサブツリー内に）メモリコンテキストが多すぎるケースでの使用を想定していません。
