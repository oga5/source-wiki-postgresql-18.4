GUC 実装ノート
==============

GUC (Grand Unified Configuration) モジュールは、複数の型（現在は boolean、enum、int、real、string）の構成変数を実装します。変数の設定はさまざまな場所から取得でき、優先順位によってどの設定が使用されるかが決定されます。


変数ごとのフック
------------------

GUC に認識されている各変数には、カスタマイズされた動作を提供するために、オプションで check_hook、assign_hook、および/または show_hook を持たせることができます。
Check フックは、変数値の妥当性チェック（GUC が行える範囲を超えたもの）を実行するため、派生設定を計算するために（それを行うために自明ではない作業が必要な場合）、またオプションでユーザーが提供した値を「正規化」するために使用されます。Assign フックは、GUC 変数が設定されたときに変更が必要な派生状態を更新するために使用されます。Show フックは、変数のデフォルトの SHOW 表示を変更するために使用されます。


check_hook が提供される場合、それは以下のシグネチャを持つ関数を指します。
	bool check_hook(datatype *newvalue, void **extra, GucSource source)
"newvalue" 引数の型は、それぞれ bool、int/enum、real、または string 変数に対して bool *、int *、double *、または char ** です。チェック関数は提案された新しい値を検証し、OK であれば true を、そうでなければ false を返す必要があります。関数はオプションで他のいくつかの処理を行うことができます。

* 不正な提案値を拒否する場合、guc.c が出力する一般的な「パラメータ FOO の値が無効です (invalid value for parameter FOO)」という警告に、追加情報を付加すると便利なことがあります。それを行うには、以下を呼び出します。
	void GUC_check_errdetail(const char *format, ...)
ここで、フォーマット文字列と追加の引数は errdetail() 引数のルールに従います。結果の文字列は guc.c のエラーレポートの DETAIL 行として出力されるため、DETAIL メッセージのメッセージスタイルガイドラインに従う必要があります。同様の方法で HINT メッセージを付加するために使用できる以下もあります。
	void GUC_check_errhint(const char *format, ...)
場合によっては、guc.c の一般的なプライマリメッセージやエラーコードを上書きすることが適切なこともあり、それは以下で行うことができます。
	void GUC_check_errcode(int sqlerrcode)
	void GUC_check_errmsg(const char *format, ...)
一般的に、check_hook は可能であれば直接エラーをスローすることを避けるべきですが、メモリ不足のような一部のコーナーケースではこれを避けるのが現実的でない場合があります。

* newvalue は参照渡しであるため、関数はそれを変更できます。
これは例えば、文字列値の綴りを正規化したり、バッファサイズを最も近いサポートされている値に丸めたり、"-1" のような特別な値を計算されたデフォルト値に置き換えたりするために使用されるかもしれません。関数が文字列値を置き換えたい場合、guc_malloc（palloc ではない）を使用して置換値を割り当て、以前の値を必ず guc_free() する必要があります。

* ユーザー名によって表されるロール OID のような派生情報は、assign フックで使用するために保存できます。これを行うには、情報のために（palloc ではなく）guc_malloc でストレージ領域を確保し、そのアドレスを *extra に返します。関連する GUC 設定が不要になったとき、guc.c は自動的にこの領域を guc_free() します。*extra は呼び出し前に NULL に初期化されるため、不要な場合は無視できます。

"source" 引数は、提案された新しい値のソースを示します。
それが >= PGC_S_INTERACTIVE の場合、対話的な割り当て（例：SET コマンド）を実行しています。しかし、source < PGC_S_INTERACTIVE の場合、postgresql.conf のような非対話的なオプションソースを読み取っています。これは、設定を許可すべきかどうかを判断するために必要な場合があります。check_hook は、何が許可されるかを判断するために、変数の現在の実際の値を確認することもあります。

check_hook は、実際に設定を変更する意図はなく、単に値を検証するためだけに呼び出されることがあることに注意してください。したがって、check_hook は割り当てが発生するという前提に基づいてアクションを起こしては*なりません*。


assign_hook が提供される場合、それは以下のシグネチャを持つ関数を指します。
	void assign_hook(datatype newvalue, void *extra)
ここで "newvalue" の型は変数の種類と一致し、"extra" は check_hook によって返された派生情報ポインタです（check_hook がない場合は常に NULL）。この関数は、変数の値を実際に設定する直前に呼び出されます（したがって、例えば値が実際には変更されていないときに作業を避けるために、実際の変数を見て古い値を確認できます）。

失敗時の結果コードを返す仕組みはないことに注意してください。assign_hook は、最も深刻な状況を除いて決して失敗してはなりません。失敗すると、例えばトランザクションの中断中に GUC 設定が適切にロールバックされない可能性があるためです。一般的に、失敗する可能性のあることはすべて check_hook で行い、その結果を "extra" 構造体で渡すようにして、assign フックがデータをどこかにコピーする以上のことをしなくて済むようにしてください。これは特にカタログ検索に当てはまります。必要な検索はすべて check_hook で行う必要があります。assign_hook はトランザクションのロールバック中に実行される可能性があり、その際、検索は安全ではないためです。

check_hook もまた、トランザクションの外で呼び出されることがあることに注意してください。これは、組み込みの "bootstrap" 値を処理するとき、postmaster のコマンドラインや環境から値が来るとき、あるいは postgresql.conf から値が来るときに発生します。したがって、check_hook で行われるカタログ検索は IsTransactionState() テストで保護されるべきであり、トランザクション内での GUC 設定の最初の使用時に派生値を計算できるようにするためのフォールバックパスが必要です。典型的な配置は、check_hook によって計算され assign_hook によってインストールされたカタログ値が、新しい設定が行われたトランザクションの残りの期間のみ使用されるようにすることです。後続の各トランザクションは、最初の使用時に値を新たに検索します。この配置は、トランザクション外で GUC 値をチェックする必要があるという問題とは無関係に、古いカタログ値の使用を防ぐのに役立ちます。


show_hook が提供される場合、それは以下のシグネチャを持つ関数を指します。
	const char *show_hook(void)
このフックにより、SHOW（および GUC 変数値を表示するための他の SQL 機能）によって表示される値の変数固有の計算が可能になります。show 関数はリエントラント（再入可能）には使用されないため、戻り値は静的バッファを指すことができます。


GUC 変数値の保存/復元
------------------------------------

構成変数の以前の値は、いくつかの特別なケースに対処するために記憶しておく必要があります。RESET（別名 SET TO DEFAULT）、トランザクション中断時の SET のロールバック、トランザクション終了時（コミットまたは中断）の SET LOCAL のロールバック、および SET オプションを持つ関数の前後での保存/復元です。RESET は、現在のセッションで SET コマンドが一度もなかった場合に有効であったであろう値を選択することとして定義されます。

これらのケースを処理するために、各変数について多くの異なる値を追跡する必要があります。主な値は以下の通りです。

* 実際の変数内容	常に現在の有効な値

* reset_val			RESET に使用する値

（各 GUC エントリには、組み込みのデフォルト値である boot_val もあります。これは InitializeGUCOptions() 中に reset_val と実際の変数に割り当てられます。boot_val は、SIGHUP 処理によって以前 postgresql.conf で指定されていた変数がそこに設定されなくなったことが判明した場合に、正しい reset_val を復元するためにも参照されます。）

主要な値に加えて、将来復元する必要がある可能性のある以前の有効な値のスタックがあります。スタックへの積み上げと取り出しは GUC の「ネストレベル」によって制御されます。これは、トランザクション外ではゼロ、トップトランザクションレベルでは 1 であり、開いているサブトランザクションや SET オプションを持つ関数呼び出しごとにインクリメントされます。GUC 変数が特定のネストレベルで最初に変更されるたびに、スタックエントリが作成されます。（注：reset_val は非トランザクション操作によってのみ変更されるため、スタックする必要はありません。）

スタックエントリには、状態、GUC 変数の以前の値、その以前の値の記憶されたソースがあり、状態によっては「マスクされた」値を持つこともあります。マスクされた値は、同じネストレベルで SET の後に SET LOCAL が発生した場合に必要になります。SET の値はマスクされますが、トランザクションコミット後に復元するために記憶しておく必要があります。

初期化中、最も優先度の高い非対話的ソースに基づいて実際の値と reset_val を設定します。これらは同じ値になります。

GUC 値に対する可能なトランザクション操作は以下の通りです。

SET オプションを持つ関数への入り口：

	以前の変数値と状態 SAVE を持つスタックエントリをプッシュし、
	その後変数を設定する。

通常の SET コマンド：

	現在のレベルのスタックエントリがない場合：
		以前の値と状態 SET を持つ新しいスタックエントリをプッシュする
	スタックエントリの状態が SAVE、SET、または LOCAL の場合：
		スタック状態を SET に変更し、保存された値は変更しない
		（ここでは以前の set アクションの影響を忘れている）
	それ以外（エントリの状態は SET+LOCAL でなければならない）：
		マスクされた値を破棄し、状態を SET に変更する
		（ここでは以前の SET と SET LOCAL の影響を忘れている）
	次に新しい値を設定する。

SET LOCAL コマンド：

	現在のレベルのスタックエントリがない場合：
		以前の値と状態 LOCAL を持つ新しいスタックエントリをプッシュする
	スタックエントリの状態が SAVE、LOCAL、または SET+LOCAL の場合：
		スタックエントリに変更なし
		（SAVE の場合、SET LOCAL は関数終了時に忘れられる）
	それ以外（エントリの状態は SET）：
		現在アクティブな値をマスクされたスロットに入れ、状態を SET+LOCAL にする
	次に新しい値を設定する。

トランザクションまたはサブトランザクションの中断：

	トップがサブトランザクションの深さ未満になるまで、スタックエントリをポップし、以前の値を復元する

トランザクションまたはサブトランザクションのコミット（関数の正常終了を含む）：

	スタックエントリのレベル >= サブトランザクションの深さの間

		エントリの状態が SAVE の場合：
			ポップし、以前の値を復元する
		レベルが 1 でエントリの状態が SET+LOCAL の場合：
			ポップし、*マスクされた*値を復元する
		レベルが 1 でエントリの状態が SET の場合：
			ポップし、古い値を破棄する
		レベルが 1 でエントリの状態が LOCAL の場合：
			ポップし、以前の値を復元する
		レベル N-1 のエントリが正確に存在しない場合：
			エントリのレベルをデクリメントし、他の状態変更はなし
		それ以外
			以下で指定されるようにレベル N-1 と N のエントリをマージする

マージされたエントリはレベル N-1 を持ち、以前の値は古い以前の値となるため、古いエントリを保持して新しいエントリを解放するのが最も簡単です。レベル N の状態 = SAVE は既に処理済みであるため、12 の可能性があります。

N-1		N

SAVE		SET		トップの以前の値を破棄し、状態を SET に設定
SAVE		LOCAL		トップの以前の値を破棄し、スタックエントリに変更なし
SAVE		SET+LOCAL	トップの以前の値を破棄し、マスクされた値をコピー、状態を S+L

SET		SET		トップの以前の値を破棄し、スタックエントリに変更なし
SET		LOCAL		トップの以前の値をマスクされた値にコピー、状態を S+L
SET		SET+LOCAL	トップの以前の値を破棄し、マスクされた値をコピー、状態を S+L

LOCAL		SET		トップの以前の値を破棄し、状態を SET に設定
LOCAL		LOCAL		トップの以前の値を破棄し、スタックエントリに変更なし
LOCAL		SET+LOCAL	トップの以前の値を破棄し、マスクされた値をコピー、状態を S+L

SET+LOCAL	SET		トップの以前の値と 2 番目のマスクされた値を破棄、状態を SET
SET+LOCAL	LOCAL		トップの以前の値を破棄し、スタックエントリに変更なし
SET+LOCAL	SET+LOCAL	トップの以前の値を破棄し、マスクされた値をコピー、状態を S+L


RESET は SET と同様に実行されますが、希望する新しい値として reset_val を使用します。（RESET LOCAL コマンドは提供していませんが、SET LOCAL TO DEFAULT は RESET LOCAL と同じ動作をします。）reset_val に関連付けられたソースも、実際の値に関連付けられるようになります。

SIGHUP を受信した場合、GUC コードは postgresql.conf 構成ファイルを再読み込みします（これはシグナルハンドラ内ではなく、メインループへの次の戻り時に発生します。トランザクション内でも実行される可能性があることに注意してください）。postgresql.conf からの新しい値は、実際の変数、reset_val、およびスタックされた実際の値に割り当てられますが、それはそれぞれの現在のソース優先度が <= PGC_S_FILE である場合に限られます。（したがって、実際の変数の対話的な値が現在異なっていても、reset_val が設定ファイルの設定を追跡することは可能です。）

check_hook、assign_hook、および show_hook ルーチンは実際の変数のみを操作し、GUC によって維持される追加の値については直接認識しません。


GUC メモリ処理
-------------------文字列変数の値は guc_malloc または guc_strdup で割り当てられます。これらは値が長期間存続するコンテキストに保持されることを保証し、通常の palloc よりもメモリ不足エラーの処理をより詳細に制御できるようにします。

文字列変数の実際の値、reset_val、boot_val、およびスタックされた値が同じストレージを指すことを許可しています。これにより、領域の解放がわずかに困難になります（解放しようとしている値が、GUC エントリや関連するスタック項目内の他のポインタと等しくないことを確認する必要があります）。主な利点は、トランザクションのコミット/アボート中に malloc を行う必要がないため、そこでメモリ不足エラーを引き起こすことがない点です。

check_hook ルーチンによって返される "extra" 構造体は、文字列値と同じ方法で管理されます。なお、"extra" 構造体は主に文字列で有用ですが、すべての型の GUC 変数でサポートされています。


GUC と NULL 文字列変数
-----------------------------

GUC 文字列変数は boot_val に NULL を持つことができます。guc.c はこれを想定通りに処理し、基盤となる C 変数に NULL を代入します。そのような変数を使用するコードや、そのフック関数は、NULL 値を扱う準備をしておく必要があります。

ただし、他の方法で GUC 文字列変数に NULL 値を代入することはできません。SET や postgresql.conf などから来る値は空文字列になることはあっても、NULL になることはありません。また、SHOW は NULL を空文字列と同じように表示します。したがって、NULL 値をユーザーから見える個別の設定として扱うことは適切ではありません。NULL の boot_val の典型的な用途は、起動後の段階で実際の値が設定される変数に対して、まだ値が設定されていないことを示すことです。

基盤となる C 変数を使用するコードが NULL 値を気にする必要がないようにしたい場合は、変数に NULL 以外の静的初期化子と NULL 以外の boot_val を指定できます。guc.c は InitializeGUCOptions の実行中に静的初期化子のポインタを boot_val のコピーで上書きしますが、変数が NULL になることはありません。
