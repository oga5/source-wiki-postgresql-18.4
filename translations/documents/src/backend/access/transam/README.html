トランザクションシステム
======================

PostgreSQLのトランザクションシステムは3層構造になっています。最下層は低レベルのトランザクションおよびサブトランザクションを実装しており、その上にメインループの制御コードが配置されています。この制御コードが、ユーザーから見えるトランザクションとセーブポイントを実装しています。

中間層のコードは、各クエリの処理前後、またはエラー検出後に postgres.c から呼び出されます。

		StartTransactionCommand
		CommitTransactionCommand
		AbortCurrentTransaction

一方、ユーザーは SQL コマンド BEGIN、COMMIT、ROLLBACK、SAVEPOINT、ROLLBACK TO、または RELEASE を発行することでシステムの状態を変更できます。トラフィックコップ（traffic cop）は、これらの呼び出しを以下のトップレベルルーチンにリダイレクトします。

		BeginTransactionBlock
		EndTransactionBlock
		UserAbortTransactionBlock
		DefineSavepoint
		RollbackToSavepoint
		ReleaseSavepoint

システムの現在の状態に応じて、これらの関数は低レベル関数を呼び出し、実際のトランザクションシステムをアクティブにします。

		StartTransaction
		CommitTransaction
		AbortTransaction
		CleanupTransaction
		StartSubTransaction
		CommitSubTransaction
		AbortSubTransaction
		CleanupSubTransaction

さらに、トランザクション内では CommandCounterIncrement が呼び出されてコマンドカウンタがインクリメントされます。これにより、以降のコマンドは同じトランザクション内の先行コマンドによる影響を「参照」できるようになります。これはトランザクションブロック内の各クエリの後に CommitTransactionCommand によって自動的に行われますが、一部のユーティリティ関数も内部的にこれを行い、一部の操作（通常はシステムカタログに対するもの）が同じユーティリティコマンド内の後続の操作から参照できるようにしています。（例えば、DefineRelation では、pg_class の行を参照してロックできるように、ヒープ作成後にこれが実行されます。）


例えば、以下のユーザーコマンドのシーケンスを考えます。

1)		BEGIN
2)		SELECT * FROM foo
3)		INSERT INTO foo VALUES (...)
4)		COMMIT

メイン処理ループでは、これは以下の関数呼び出しシーケンスになります。

     /  StartTransactionCommand;
    /       StartTransaction;
1) <    ProcessUtility;                 << BEGIN
    \       BeginTransactionBlock;
     \  CommitTransactionCommand;

    /   StartTransactionCommand;
2) /    PortalRunSelect;                << SELECT ...
   \    CommitTransactionCommand;
    \       CommandCounterIncrement;

    /   StartTransactionCommand;
3) /    ProcessQuery;                   << INSERT ...
   \    CommitTransactionCommand;
    \       CommandCounterIncrement;

     /  StartTransactionCommand;
    /   ProcessUtility;                 << COMMIT
4) <        EndTransactionBlock;
    \   CommitTransactionCommand;
     \      CommitTransaction;

この例のポイントは、StartTransactionCommand と CommitTransactionCommand が状態を認識している必要があることを示すことです。これらは BeginTransactionBlock と EndTransactionBlock の呼び出しの間で CommandCounterIncrement を呼び出すべきであり、これらの呼び出しの外側では通常の開始、コミット、またはアボート処理を行う必要があります。

さらに、「SELECT * FROM foo」がアボート条件を引き起こしたと仮定します。この場合、AbortCurrentTransaction が呼び出され、トランザクションはアボート状態になります。この状態では、トランザクション終了ステートメントまたは ROLLBACK TO <savepoint> コマンドを除き、ユーザー入力はすべて無視されます。

トランザクションのアボートは以下の2つの方法で発生します。

1) システムが内部的な原因（構文エラーなど）で停止する
2) ユーザーが ROLLBACK と入力する

これらを区別しなければならない理由は、以下の2つの状況で説明されます。

        ケース 1                                  ケース 2
        ------                                  ------
1) ユーザーが BEGIN と入力                1) ユーザーが BEGIN と入力
2) ユーザーが何かを実行                   2) ユーザーが何かを実行
3) ユーザーが結果を気に入らず             3) システムが何らかの理由でアボート
   ABORT と入力                            （構文エラーなど）

ケース 1 では、トランザクションをアボートしてデフォルト状態に戻したいと考えます。ケース 2 では、同じトランザクションブロックの一部である後続のコマンドが来る可能性があるため、COMMIT または ROLLBACK を確認するまでこれらのコマンドを無視しなければなりません。

内部的なアボートは AbortCurrentTransaction によって処理され、ユーザーによるアボートは UserAbortTransactionBlock によって処理されます。どちらも実際の作業は AbortTransaction に依存しています。唯一の違いは、AbortTransaction が作業を終えた後にどの状態に入るかです。

* AbortCurrentTransaction は TBLOCK_ABORT 状態にします。
* UserAbortTransactionBlock は TBLOCK_ABORT_END 状態にします。

低レベルのトランザクションアボート処理は2つのフェーズに分かれています。
* AbortTransaction は、トランザクションの失敗を認識した直後に実行されます。他のバックエンドを不必要に遅延させないよう、すべての共有リソース（ロックなど）を解放する必要があります。
* CleanupTransaction は、最終的にユーザーの COMMIT または ROLLBACK コマンドを確認したときに実行されます。これは後始末を行い、トランザクションから完全に脱出させます。特に、この時点まで TopTransactionContext を破棄してはなりません。

また、トランザクションがコミットされたとき、すぐにはクローズされないことに注意してください。むしろ TBLOCK_END 状態に置かれます。これは、クエリの処理終了後に CommitTransactionCommand が呼び出されたときにトランザクションをクローズする必要があることを意味します。この区別は微妙ですが重要です。なぜなら、制御がトランザクションを開いたまま xact.c コードを離れることを意味し、メインループが同じトランザクション内で処理を継続できるからです。したがって、ある意味でトランザクションのコミットも2つのフェーズで処理されます。1つ目は EndTransactionBlock で、2つ目は CommitTransactionCommand（実際に CommitTransaction が呼び出される場所）です。

xact.c 内の残りのコードは、トランザクションおよびサブトランザクションの作成と終了をサポートするルーチンです。例えば、AtStart_Memory はメイントランザクション開始時にメモリサブシステムの初期化を担当します。


サブトランザクションの処理
-----------------------

サブトランザクションは TransactionState 構造体のスタックを使用して実装されており、各構造体は親トランザクションの構造体へのポインタを持っています。新しいサブトランザクションを開くときは PushTransaction が呼び出され、親リンクが現在のトランザクションを指す新しい TransactionState が作成されます。StartSubTransaction は、新しい TransactionState を正常な値で初期化し、他のサブシステム（AtSubStart ルーチン）を適切に初期化する役割を担います。

サブトランザクションをクローズするときは、CommitSubTransaction（サブトランザクションがコミットする場合）または AbortSubTransaction と CleanupSubTransaction（アボートする場合）のいずれかを呼び出す必要があります。いずれの場合も PopTransaction が呼び出され、システムは親トランザクションに戻ります。

サブトランザクション処理に関する重要な点は、単一のユーザーコマンドに応答して複数をクローズする必要がある場合があることです。これは、セーブポイントには名前があり、最後に開かれたものではないセーブポイントであっても名前でコミットまたはロールバックできるためです。また、COMMIT または ROLLBACK コマンドはスタック全体をクローズできなければなりません。これは、ユーティリティコマンドのサブルーチンが状態スタックのすべてのエントリをコミット保留またはアボート保留としてマークし、メインループが CommitTransactionCommand に到達したときに実際の作業を行うことで処理されます。このようにする主な理由は、状態スタックのエントリをポップする際にエラーが発生した場合でも、残りのスタックエントリが終了処理に必要な手順を示し続けるためです。

ROLLBACK TO <savepoint> の場合、セーブポイント名で識別されるサブトランザクションまでのすべてをアボートし、そのサブトランザクションレベルを同じ名前で再作成します。したがって、内部的には完全に新しいサブトランザクションとなります。

他のサブシステムは「内部」サブトランザクションを開始することが許可されており、これは BeginInternalSubTransaction によって処理されます。これは、PL/pgSQL などで例外処理を実装できるようにするためです。ReleaseCurrentSubTransaction および RollbackAndReleaseCurrentSubTransaction を使用すると、サブシステムはこれらのサブトランザクションをクローズできます。これとセーブポイント/リリースパスの主な違いは、CommitTransactionCommand まで作業を延期するのではなく、各サブルーチンで完全な状態遷移を即座に実行する点です。もう1つの違いは、明示的なトランザクションブロックが確立されていない場合でも BeginInternalSubTransaction は許可されますが、DefineSavepoint は許可されない点です。


トランザクションとサブトランザクションの番号付け
----------------------------------------

トランザクションとサブトランザクションには、タプルの挿入/更新/削除など、XID を必要とする操作を最初に行うとき（またはその時点）にのみ永続的な XID が割り当てられます。ただし、XID 割り当てが必要な場所は他にもいくつかあります。サブトランザクションが XID を必要とする場合、常にその親に先に XID を割り当てます。これにより、子トランザクションの XID は親よりも大きくなるという不変条件が維持され、これは多くの場所で前提とされています。

XID のロックを取得し、pg_subtrans および PGPROC に登録する付随的なアクションは、割り当て時に行われます。

XID を持たないトランザクションであっても、ロックの保持など、さまざまな目的で識別される必要があります。この目的のために、各トップレベルトランザクションに「仮想トランザクション ID」または VXID を割り当てます。VXID は procNumber とバックエンドローカルなカウンタの2つのフィールドから構成されます。この構成により、共有メモリの競合なしにトランザクション開始時に新しい VXID を割り当てることができます。バックエンド終了後すぐに VXID が再利用されないように、バックエンド終了時に最後のローカルカウンタ値を共有メモリに保存し、バックエンド開始時に同じ PGPROC スロットの以前の値から初期化します。これらのカウンタはすべて共有メモリの再初期化時にゼロに戻りますが、VXID はディスク上のどこにも現れないため問題ありません。

内部的には、バックエンドは XID の有無にかかわらずサブトランザクションを識別する方法を必要としますが、この必要性は親トップトランザクションが存続している間だけです。そのため、SubTransactionId があります。これは各トップトランザクションの開始時にリセットされるカウンタから生成されるという点で CommandId に似ています。トップレベルトランザクション自体は SubTransactionId 1 を持ち、サブトランザクションは 2 以上の ID を持ちます。（0 は InvalidSubTransactionId として予約されています。）サブトランザクションは独自の VXID を持たず、親トップトランザクションの VXID を使用することに注意してください。


トランザクション開始、トランザクション終了、およびスナップショットのインターロック
--------------------------------------------------------------

トランザクションの開始/終了やスナップショット取得という頻繁なアクティビティに伴うオーバーヘッドとロック競合を最小限に抑えるよう努めています。残念ながら、トランザクションのコミット順序に関する一貫性を確保する必要があるため、これには何らかのインターロックが必要です。
例えば、トランザクション A の UPDATE がトランザクション B による同じ行の先行更新によってブロックされており、トランザクション B がコミット処理中にトランザクション C がスナップショットを取得する場合を考えます。トランザクション A は B がロックを解放するとすぐに完了してコミットできます。もしトランザクション C の GetSnapshotData がトランザクション B を実行中と見なすなら、トランザクション A も実行中と見なさない限り、B によって削除されたバージョンと A によって挿入されたバージョンの2つのタプルバージョンが見えてしまうことになります。これが悪いもう1つの理由は、C が（A によって挿入された行において）B による以前の変更を見てしまい、データベースの他の場所で B の変更を一切見ないことが C にとって不整合になるためです。

形式的には、正確性の要件は「スナップショット A がトランザクション X をコミット済みと見なし、トランザクション X のスナップショットのいずれかがトランザクション Y をコミット済みと見なす場合、スナップショット A はトランザクション Y をコミット済みと見なさなければならない」となります。私たちが実際に強制しているのは、コミットとロールバックの厳密なシリアライズとスナップショット取得の組み合わせです。スナップショットの取得中に、実行中のトランザクションの集合からトランザクションが抜けることを許可しません。（このルールは一貫性を保つために必要なものより強力ですが、強制するのが比較的単純であり、後述するように他のいくつかの問題の解決にも役立ちます。）この実装では、GetSnapshotDataはProcArrayLockを共有モードで取得します（これにより、複数のバックエンドが並行してスナップショットを取得できます）。しかし、ProcArrayEndTransactionは、トランザクション終了時（コミットまたはアボート）にProcGlobal->xids[]エントリをクリアする間、ProcArrayLockを排他モードで取得しなければなりません。（コンテキストスイッチを減らすため、複数のトランザクションがほぼ同時にコミットする場合、1つのバックエンドがProcArrayLockを取得し、複数のプロセスのXIDを一度にクリアします。）

ProcArrayEndTransactionは、共有変数latestCompletedXidを進める間もロックを保持します。これにより、GetSnapshotDataはスナップショットのxmaxとしてlatestCompletedXid + 1を使用できます。スナップショットが完了済みとみなす必要があるトランザクションで、このXID値以上のものは存在し得ないからです。

つまり、ルールは「latestCompletedXidを取得してからスナップショットの構築を完了するまでの間、現在実行中のトランザクションの集合からどのトランザクションも抜けてはならない」ということです。ただし、この制限はXIDを持つトランザクションにのみ適用されます。読み取り専用トランザクションは、他の誰のスナップショットにもlatestCompletedXidにも影響を与えないため、ProcArrayLockを取得せずに終了できます。

トランザクション開始自体は、これらの考慮事項と相互ロックしません。なぜなら、トランザクション開始時に即座にXIDを割り当てることはなくなったからです。しかし、XIDを割り当てることを決定したとき、GetNewTransactionIdはXidGenLockを解放する前に、新しいXIDを共有ProcArrayに格納しなければなりません。これにより、latestCompletedXid以下のすべてのトップレベルXIDが、ProcArrayに存在するか、あるいはすでに実行中でないかのいずれかであることが保証されます。（この保証はサブトランザクションXIDには適用されません。サブxid配列に空きがない可能性があるためです。その代わり、それらが存在するか、オーバーフローフラグが設定されていることを保証します。）もしバックエンドがProcGlobal->xids[]にXIDを格納する前にXidGenLockを解放すると、別のバックエンドがより後のXIDを割り当ててコミットし、その値がProcArrayで見えるようになる前にlatestCompletedXidが最初のバックエンドのXIDを追い越してしまう可能性があります。これは後述するComputeXidHorizonsを破壊することになります。

GetNewTransactionIdがProcArrayLockを取得せずにXIDをProcGlobal->xids[]（またはサブxid配列）に格納することは許可されています。これはかつてデッドロックを回避するために必要でしたが、現在はそうではありません。それでもパフォーマンス上の利点があります。そのため、XIDのフェッチ/ストアがアトミックであることに依存しています。そうでなければ、他のバックエンドが部分的に設定されたXIDを見る可能性があります。これはまた、ProcArrayのxidフィールドの読み取り側が、毎回同じ答えが得られると仮定するのではなく、一度だけ値をフェッチするように注意しなければならないことを意味します。（これを行うときは、Cコンパイラが指示通りの動作を確実に行うよう、volatile修飾されたポインタを使用してください。）

共有ProcArrayを使用するもう一つの重要な活動はComputeXidHorizonsです。これは、システム全体でアクティブなMVCCスナップショットの最も古いxminの下限を決定しなければなりません。各バックエンドは、自身のスナップショットの最小xminをMyProc->xminでアドバタイズします。現在ライブなスナップショットがない場合（例：トランザクション間である、または新しいトランザクションのスナップショットをまだ設定していない場合）は0になります。ComputeXidHorizonsは、有効なxminフィールドのMIN()をとります。これはProcArrayLockの共有ロックのみで行われるため、GetSnapshotDataを並行して実行する他のバックエンドとの間で競合状態が発生する可能性があります。xminを設定しようとしている並行バックエンドが、ComputeXidHorizonsが決定する値よりも小さいxminを計算しないことを確実にする必要があります。これを保証するために、有効なxminとともに、すべてのアクティブなXIDをMIN()計算に含めます。トランザクションが排他ProcArrayLockを取得せずに終了できないというルールにより、共有ProcArrayLockの同時保持者は、現在アクティブなXIDの同じ最小値を計算することが保証されます。共有ProcArrayLockを保持している間は、どのトランザクションも（特に最も古いものも）終了できないからです。したがって、ComputeXidHorizonsから見た最小アクティブXIDは、同時実行されるGetSnapshotDataから見たものと同じになり、過大評価を生むことはありません。アクティブなトランザクションが全くない場合、ComputeXidHorizonsはlatestCompletedXid + 1を使用します。これは、同時またはそれ以降のGetSnapshotData呼び出しによって計算される可能性のあるxminの下限です。（前述のXidGenLockの相互ロックにより、これより小さいXIDがProcArrayに出現する可能性がないことは分かっています。）

GetSnapshotDataはパフォーマンスが重要であるため、正確な最古のxmin計算は行いません（v14までは行っていました）。スナップショットの内容は他のバックエンドのxminではなく、XIDのみに依存します。バックエンドのxminはXIDよりもはるかに頻繁に変更されるため、GetSnapshotDataがxminを参照すると、不要なキャッシュラインのピンポンが大量に発生する可能性があります。その代わり、GetSnapshotDataは近似的な閾値を更新します（一つはそれより古い削除済み行がすべて削除可能であることを保証し、もう一つはそれより新しい削除済み行が削除不可能であることを決定します）。GlobalVisTest*はそれらの閾値を使用して不可視性の決定を行い、必要に応じてComputeXidHorizonsにフォールバックします。

GetSnapshotDataの2つの同時実行が自身のスナップショットに対して同じxminを計算することは確実ですが、ComputeXidHorizonsによって計算されるホライゾンにはそのような保証はないことに注意してください。これは、XIDを持たないトランザクションがMyProc->xminを非同期に（ProcArrayLockを取得せずに）クリアすることを許可しているため、ある実行が最古のxminであったものを見る一方で、別の実行は見ない可能性があるためです。閾値は有効な下限であればよいため、これは問題ありません。前述の通り、私たちはすでにxidフィールドのフェッチ/ストアがアトミックであると仮定しているため、xminについても同様に仮定しても追加のリスクはありません。

pg_xactおよびpg_subtrans
-----------------------

pg_xactおよびpg_subtransは、トランザクション関連情報の永続的（ディスク上）なストレージです。それぞれメモリ内に保持されるページ数には限りがあるため、多くの場合、実際にディスクから読み取る必要はありません。しかし、長時間実行されるトランザクションや、トランザクションを開いたままアイドル状態のバックエンドがある場合、この情報をディスクから読み書きできるようにしておく必要があります。また、これらはサーバーの再起動後も情報を永続化させます。

pg_xactは、XIDが割り当てられた各トランザクションのコミットステータスを記録します。トランザクションは、進行中、コミット済み、アボート済み、または「サブコミット済み」の状態になり得ます。この最後の状態は、実行中ではないサブトランザクションだが、親がまだその状態を更新していないことを意味します。サブトランザクションのトランザクションステータスをサブコミットに更新する必要はないため、メインのトランザクションコミットまで延期できます。トランザクションをサブコミット済みとしてマークする主な役割は、トランザクションステータスが複数のclogページにまたがる場合にアトミックなコミットプロトコルを提供することです。その結果、トランザクションステータスが複数のページにまたがる場合は常に、2フェーズコミットプロトコルを使用しなければなりません。第1フェーズはサブトランザクションをサブコミット済みとしてマークし、次にトップレベルのトランザクションとすべてのサブトランザクションをコミット済みとしてマークします（その順序で）。したがって、アボートしていないサブトランザクションは、すでに終了していても進行中として表示され、サブコミットステータスはメインのトランザクションコミット中の非常に短い一時的な状態として表示されます。サブトランザクションのアボートは、発生するとすぐにclogにマークされます。トランザクションステータスがすべて単一のCLOGページに収まる場合、中間的なサブコミット状態を経由することなく、すべてをアトミックにコミット済みとしてマークします。

セーブポイントはサブトランザクションを使用して実装されます。サブトランザクションはトランザクション内のトランザクションであり、そのコミットまたはアボートステータスは、自身がコミットしたかどうかだけでなく、親トランザクションがコミットしたかどうかにも依存します。トランザクション内に複数のセーブポイントを実装するために、無制限のトランザクションネスト深度を許可しているため、特定のサブトランザクションのコミット状態は、すべての祖先トランザクションのコミットステータスに依存します。

「サブトランザクション親」(pg_subtrans)メカニズムは、XIDを持つ各トランザクションについて、その親トランザクションのTransactionIdを記録します。この情報は、サブトランザクションにXIDが割り当てられるとすぐに保存されます。トップレベルのトランザクションには親がないため、pg_subtransエントリはデフォルト値のゼロ（InvalidTransactionId）のままです。

pg_subtransは、対象のトランザクションがまだ実行中かどうかを確認するために使用されます。トランザクションのメインXIDはProcGlobal->xids[]に記録され、PGPROC->xidにもコピーが保持されますが、サブトランザクションの任意のネストを許可しているため、すべてのXIDを共有メモリに収めることはできず、ディスクに保存する必要があります。ただし、各トランザクションについて、トランザクションツリーの一部であることがわかっているXIDの「キャッシュ」を保持しているため、キャッシュがオーバーフローしたことがわかっている場合を除き、pg_subtransを見る必要はありません。詳細についてはstorage/ipc/procarray.cを参照してください。

slru.cは、pg_xactとpg_subtransの両方をサポートするメカニズムです。これは、メモリ内バッファページのLRUポリシーを実装しています。pg_xactの上位ルーチンはtransam.cに実装されており、低位関数はclog.cにあります。pg_subtransは完全にsubtrans.cに含まれています。

先行書き込みログ（WAL）のコーディング
----------------------

WALサブシステム（コード内ではXLOGとも呼ばれます）は、クラッシュリカバリを保証するために存在します。また、ポイントインタイムリカバリや、ログシッピングによるホットスタンバイレプリケーションを提供するためにも使用できます。以下に、その設計の自明ではない側面に関するメモをいくつか示します。

先行書き込みログの基本的な前提は、ログエントリが記述するデータページの変更よりも先に、ログエントリが安定したストレージに到達しなければならないということです。これにより、ログを最後までリプレイすることで、部分的に実行されたトランザクションが存在しない一貫した状態になることが保証されます。これを保証するために、各データページ（ヒープまたはインデックス）には、そのページに影響を与える最新のXLOGレコードのLSN（ログシーケンス番号。実際にはWALファイルの位置）がマークされます。bufmgrがダーティページを書き出す前に、xlogが少なくともそのページのLSNまでディスクにフラッシュされていることを確認しなければなりません。この低レベルの相互作用により、必要になるまでXLOG I/Oを待機しないことでパフォーマンスが向上します。LSNチェックは共有バッファマネージャにのみ存在し、一時テーブルに使用されるローカルバッファマネージャには存在しません。そのため、一時テーブルに対する操作はWALログに記録してはなりません。

WALリプレイ中、ページのLSNをチェックして、現在のログエントリによって記録された変更がすでに適用されているかどうかを検出できます（ページLSNがログエントリのWAL位置以上であれば、適用済みです）。

通常、ログエントリには、ページ（または小さなページのグループ）に対する単一の増分更新をやり直すのに十分な情報のみが含まれています。これは、ファイルシステムとハードウェアがデータページの書き込みをアトミックなアクションとして実装している場合にのみ機能し、ページが破損した部分書き込み状態で残されることはありません。実際にはそれが維持できない前提であることが多いため、変更されたページの完全な再構築を可能にするための追加情報をログに記録します。チェックポイント後に特定のページに影響を与える最初のWALレコードには、ページ全体のコピーが含まれるようにし、更新をやり直す代わりにそのページコピーを復元することでリプレイを実装します。（これは、WALレコードのCRCの妥当性をチェックできるため、データストレージ自体よりも信頼性が高いです。）「チェックポイント後の最初の変更」は、ページの古いLSNが最後のチェックポイント時点のWALの終わり（RedoRecPtr）よりも前であるかどうかを確認することで検出できます。

WALログに記録されたアクションを実行するための一般的なスキーマは以下の通りです。

1. 変更対象のデータページを含む共有バッファをピン留めし、排他ロックする。2. START_CRIT_SECTION() （次の3つのステップのいずれかでエラーが発生した場合、共有バッファにはログに記録されていない変更が含まれることになり、それらがディスクに書き込まれないように保証する必要があるため、PANICを引き起こさなければなりません。当然のことながら、クリティカルセクションを開始する前に、ページ上に十分な空き領域があるかどうかなどの条件を確認しておくべきです。）

3. 共有バッファに必要な変更を適用します。

4. MarkBufferDirty() を使用して共有バッファをダーティとしてマークします。（これはWALレコードが挿入される前に行う必要があります。SyncOneBuffer() の注記を参照してください。）MarkBufferDirty() でバッファをダーティとしてマークするのは、WALレコードを書き込む場合にのみ行うべきであることに注意してください。以下の「ヒントの書き込み」を参照してください。

5. リレーションがWALログを必要とする場合、XLogBeginInsert および XLogRegister* 関数を使用してWALレコードを構築し、挿入します。（以下の「WALレコードの構築」を参照してください。）その後、返されたXLOG位置を使用してページのLSNを更新します。例えば、

		XLogBeginInsert();
		XLogRegisterBuffer(...)
		XLogRegisterData(...)
		recptr = XLogInsert(rmgr_id, info);

		PageSetLSN(dp, recptr);

6. END_CRIT_SECTION()

7. バッファのロックとピンを解除します。

複雑な変更（マルチレベルインデックスの挿入など）は、通常、一連のアトミックなアクションWALレコードによって記述される必要があります。中間状態は自己整合的である必要があり、任意の2つのアクション間でリプレイが中断された場合でも、システムが完全に機能するようにしなければなりません。例えばbtreeインデックスでは、ページ分割には新しいページの割り当てと親btreeレベルへの新しいキーの挿入が必要ですが、ロックの理由から、これは2つの別々のWALレコードによって反映される必要があります。最初のレコードをリプレイして新しいページを割り当て、タプルをそこに移動すると、キーがまだ親に挿入されていないことを示すフラグがページに設定されます。2番目のレコードをリプレイすると、そのフラグがクリアされます。この中間状態は、通常の操作中には他のバックエンドからは見えません。なぜなら、子ページへのロックが2つのアクションを通じて保持されるためですが、2番目のWALレコードを書き込む前に操作が中断された場合には見えてしまいます。検索アルゴリズムは通常通り中間状態で動作しますが、挿入処理が不完全な分割フラグが設定されたページに遭遇した場合、処理を続行する前に、親にキーを挿入して中断された分割を完了させます。


WALレコードの構築
-------------------------

WALレコードは、すべてのWALレコードタイプに共通のヘッダー、レコード固有のデータ、および変更されたデータブロックに関する情報で構成されます。各変更されたデータブロックはID番号で識別され、オプションでブロックに関連付けられたレコード固有のデータをさらに持つことができます。XLogInsert がブロックのフルページイメージを取得する必要があると判断した場合、そのブロックに関連付けられたデータは含まれません。

WALレコードを構築するためのAPIは、XLogBeginInsert、XLogRegisterBuffer、XLogRegisterData、XLogRegisterBufData、および XLogInsert の5つの関数で構成されます。まず、XLogBeginInsert() を呼び出します。次に、XLogRegister* 関数を使用して、変更されたすべてのバッファと変更をリプレイするために必要なデータを登録します。最後に、XLogInsert() を呼び出して構築されたレコードをWALに挿入します。

	XLogBeginInsert();

	/* このWALログ記録アクションの一部として変更されたバッファを登録する */
	XLogRegisterBuffer(0, lbuffer, REGBUF_STANDARD);
	XLogRegisterBuffer(1, rbuffer, REGBUF_STANDARD);

	/* WALレコードに常に含まれるデータを登録する */
	XLogRegisterData(&xlrec, SizeOfFictionalAction);

	/*
	 * バッファに関連付けられたデータを登録する。フルページイメージが
	 * 取得される場合、これはレコードに含まれない。
	 */
	XLogRegisterBufData(0, tuple->data, tuple->len);

	/* バッファに関連付けられたその他のデータ */
	XLogRegisterBufData(0, data2, len2);

	/*
	 * よし、WALレコードに含めるすべてのデータとバッファが登録された。
	 * レコードを挿入する。
	 */
	recptr = XLogInsert(RM_FOO_ID, XLOG_FOOBAR_DO_STUFF);

API関数の詳細:

void XLogBeginInsert(void)

    XLogRegisterBuffer および XLogRegisterData の前に呼び出す必要があります。

void XLogResetInsertion(void)

    WALレコード構築ワークスペースから、現在登録されているデータとバッファをクリアします。これは、すでに XLogBeginInsert() を呼び出したが、結局レコードを挿入しないことにした場合にのみ必要です。

void XLogEnsureRecordSpace(int max_block_id, int ndatas)

    通常、WALレコード構築バッファには以下の制限があります:

    * 使用できる最大ブロックIDは4（5つのブロック参照が可能）
    * 登録データの最大チャンク数は20

    これらのデフォルト制限は、ディスク上の構造を変更するほとんどのレコードタイプには十分です。より多くのデータを必要とする、またはより多くのバッファを変更する必要がある特殊なケースでは、XLogEnsureRecordSpace() を呼び出すことでこれらの制限を引き上げることができます。XLogEnsureRecordSpace() は XLogBeginInsert() の前、かつクリティカルセクションの外で呼び出す必要があります。

void XLogRegisterBuffer(uint8 block_id, Buffer buf, uint8 flags);

    XLogRegisterBuffer は、データブロックに関する情報をWALレコードに追加します。block_id は、redoルーチンでこのページ参照を識別するために使用される任意の番号です。redo時にページを再検索するために必要な情報（relfilelocator、フォーク、ブロック番号）はWALレコードに含まれます。

    XLogInsert は、前回のチェックポイント以降にバッファが初めて変更された場合、ページ内容の完全なコピーを自動的に含めます。torn-page（ページ書き込みの不整合）の危険を避けるため、アクションによって変更されたすべてのバッファを XLogRegisterBuffer で登録することが重要です。

    フラグは、バッファの内容がいつ、どのようにWALレコードに含まれるかを制御します。通常、フルページイメージは、前回のチェックポイント以降にページが変更されておらず、かつ full_page_writes=on であるかオンラインバックアップが進行中の場合にのみ取得されます。REGBUF_FORCE_IMAGE フラグを使用すると、フルページイメージを常に強制的に含めることができます。これは、例えばページの大半を書き換える操作などで、詳細を追跡する価値がない場合に便利です。torn-pageから保護する必要がない稀なケースでは、REGBUF_NO_IMAGE フラグを使用してフルページイメージの取得を抑制できます。REGBUF_WILL_INIT もフルページイメージを抑制しますが、redoルーチンは古いページ内容を見ることなく、ゼロからページを再生成しなければなりません。ページを再初期化することは、フルページイメージと同様にtorn-pageの危険から保護します。

    REGBUF_STANDARD フラグを他のフラグと一緒に指定すると、ページが標準的なページレイアウトに従っていることを示すことができます。これにより、pd_lower と pd_upper の間の領域がイメージから除外され、WALボリュームが削減されます。

    REGBUF_KEEP_DATA フラグが指定されている場合、フルページイメージが取得された場合でも、XLogRegisterBufData() で登録されたバッファごとのデータがWALレコードに含まれます。

void XLogRegisterData(const void *data, int len);

    XLogRegisterData は、任意のデータをWALレコードに含めるために使用されます。XLogRegisterData() が複数回呼び出された場合、データは追加され、1つの連続したチャンクとしてredoルーチンで利用可能になります。

void XLogRegisterBufData(uint8 block_id, const void *data, int len);

    XLogRegisterBufData は、以前に XLogRegisterBuffer() で登録された特定のバッファに関連付けられたデータを含めるために使用されます。同じブロックIDで XLogRegisterBufData() が複数回呼び出された場合、データは追加され、1つの連続したチャンクとしてredoルーチンで利用可能になります。

    挿入時にバッファのフルページイメージが取得される場合、REGBUF_KEEP_DATA フラグが使用されない限り、データはWALレコードに含まれません。


REDOルーチンの作成
----------------------

REDOルーチンは、WALレコードに含まれるデータとページ参照を使用して、ページの新しい状態を再構築します。xlogreader.c/h 内のレコードデコード関数とマクロを使用して、レコードからデータを抽出できます。

複数のページに対する変更を記述するWALレコードをリプレイする場合、同時実行中のホットスタンバイクエリが不整合な状態を見ないように、ページを適切にロックするように注意する必要があります。これに2つ以上のバッファロックを同時に保持する必要がある場合は、適切な順序でページをロックし、すべての変更が完了するまでロックを解放してはなりません。

アクションがシリアライズされていることがわかっている場合にのみ PageSetLSN/PageGetLSN() を使用しなければならないことに注意してください。リカバリ中にデータブロックを変更できるのはスタートアッププロセスのみであるため、スタートアッププロセスはシリアライズの問題を恐れることなく PageGetLSN() を実行できます。他のすべてのプロセスは、排他的バッファロック、または共有ロックとバッファヘッダーロックの両方を保持している場合、あるいはリレーションに対して AccessExclusiveLock を保持しながら共有バッファ経由ではなく直接データブロックを書き込んでいる場合にのみ、PageSet/GetLSN を呼び出さなければなりません。


ヒントの書き込み
-------------

場合によっては、先行するWALレコードを書き込まずに、データブロックに追加情報を書き込むことがあります。これは、クラッシュ後にデータが後から再構築可能であり、そのアクションが単なるパフォーマンス最適化のための手段である場合にのみ行うべきです。ヒントが書き込まれるときは、MarkBufferDirtyHint() を使用してブロックをダーティとしてマークします。

バッファがクリーンでチェックサムが使用されている場合、MarkBufferDirtyHint() は XLOG_FPI_FOR_HINT レコードを挿入し、ヒントを含むフルページイメージを確実に取得するようにします。これを行うのは、ダーティになったページを書き込む際に、ページの部分的な書き込みを避けるためです。リカバリ中はWALが書き込まれないため、リカバリ中にヒントのためにブロックをダーティにすることは単にスキップします。

WALレコードを最適化して削除することにした場合、MarkBufferDirty() の呼び出しはすべて MarkBufferDirtyHint() に置き換える必要があります。そうしないと、ページの部分的な書き込みのリスクにさらされることになります。

ヒープページ内の全可視（all-visible）ヒント（PD_ALL_VISIBLE）は、ある点では永続的な変更として扱われ、別の点ではヒントとして扱われるため、特別なケースです。これは、ヒープページに関連付けられた可視性マップ（VM）ビットが設定されている場合、ヒープページ自体にも PD_ALL_VISIBLE が設定されているという不変条件を満たさなければなりません。この不変条件を維持するため、PD_ALL_VISIBLE のクリアは常に完全に永続的な変更として扱われます。さらに、チェックサムまたは wal_log_hints が有効な場合、PD_ALL_VISIBLE の設定も、torn-pageから保護するために完全に永続的な変更として扱われます。

しかし、チェックサムも wal_log_hints も有効でない場合、変更が PD_ALL_VISIBLE だけであれば、torn-pageは問題になりません。そのため、ヒープページのフルイメージは取得されず、ヒープページのLSNも更新されません。注意：この最適化を適用する際にヒープページのLSNを更新するのは誤りです。関連するWALレコードが存在する場合であっても、そのページの後続の変更者（無関係なUPDATEなど）が、フルページイメージが不要であると誤って判断する可能性があるためです。

ファイルシステムアクションのための先行書き込みログ（WAL）
------------------------------------------

前のセクションでは、共有バッファ内のページ内容のみを変更するアクションをWALログに記録する方法を説明しました。その種のアクションでは、実際の変更を開始する前に、考えられるすべてのエラーケース（ページ上の容量不足など）を確認することが一般的に可能です。したがって、変更とそれに関連するWALログレコードの作成をクリティカルセクションでラップすることで「アトミック」にすることができます。途中で失敗する可能性は十分に低いため、もし発生したとしてもPANICで許容されます。

明らかに、このアプローチは、新しいファイルやデータベースの作成など、ログに記録されるアクション内で失敗する可能性が高い場合には機能しません。PANICは望ましくありませんし、特にアクションを実行したことを示すWALレコードをすでに書き込んだ後にPANICすることは避けなければなりません。もしそうしてしまうと、レコードのリプレイはおそらく再び失敗し、再びPANICを引き起こし、障害が回復不能になってしまいます。これは、「記述する変更の前にWALを書き込む」という通常のWALルールが機能しないことを意味しており、そのようなケースには異なる設計が必要です。

この問題を抱えるファイルシステムアクションには、いくつかの基本的なタイプがあります。それぞれへの対処方法は以下の通りです：1. 既存のテーブルへのディスクページの追加。

このアクションはWALには一切記録されません。テーブルの末尾にゼロで埋められたページを書き込むことでテーブルを拡張します。ファイルシステムが領域を割り当てたことを確実にするため、実際にこの書き込みを行う必要があります。書き込みに失敗した場合は、通常通りエラーを返すことができます。領域の割り当てが確認できたら、1つ以上の通常のWAL記録を伴うアクションを通じて、そのページを初期化しデータを書き込むことができます。ファイルを拡張してからWALエントリを書き出すまでの間にクラッシュする可能性があるため、テーブルやインデックス内にすべてゼロのページを発見した場合は、エラーではないと見なす必要があります。そのような場合、その領域を再利用のために回収するだけで済みます。

2. 新しいテーブルの作成。これにはファイルシステム上の新しいファイルが必要です。

ファイルの作成を試み、成功した場合はその旨を記したWALレコードを作成します。成功しなかった場合は、単にエラーをスローします。ファイルを作成したものの、それに関するWALをディスクに書き込む前にクラッシュする可能性があることに注意してください。この期間中にクラッシュした場合、ファイルは「孤立（orphan）」した状態でディスクに残ります。このような孤立ファイルは、データベース再起動時にpg_classにコミットされたエントリを持たないファイルを検索することでクリーンアップ可能ですが、クラッシュのフォレンジック分析に役立つデータを削除してしまう可能性があるため、現在は行われていません。孤立ファイルは無害であり、せいぜいディスク領域をわずかに浪費するだけです。新しいrelfilenumber OIDを割り当てる際にディスク上の衝突をチェックしているため、クリーンアップは実際には不要です。

3. テーブルの削除。これには失敗する可能性のあるunlink()が必要です。

ここでのアプローチは、操作を先にWALログに記録し、実際のunlink()呼び出しの失敗はエラーではなく警告として扱うことです。繰り返しになりますが、これにより孤立ファイルが残る可能性がありますが、代替案と比較すれば低コストです。DROP TABLEトランザクションをコミットした後でなければ実際にunlink()を実行できないため、エラーをスローすることはそもそも不可能です。（ファイル削除に関するWALエントリは、実際には削除トランザクションのコミットレコードの一部であることに留意してください。）

4. データベースおよびテーブルスペースの作成と削除。これにはディレクトリおよびディレクトリツリー全体の作成と削除が必要です。

これらのケースは個別のファイルの作成と同様に扱われます。つまり、まずアクションの実行を試み、成功した場合にのみWALエントリを書き込みます。当然ながら、浪費されるディスク領域の潜在的な量はより大きくなります。作成のケースでは、領域の浪費リスクを減らすため、作成に失敗した場合はディレクトリツリーの削除を再試行します。削除操作の途中で失敗するとデータベースが破損します。DROPは失敗しましたが、データの一部はすでに失われているためです。これに対してできることはほとんどありませんが、いずれにせよ、それはユーザーがもはや必要としないデータであると想定されます。

これらすべての場合において、WALリプレイが元のアクションの再実行に失敗した場合は、パニックを起こしてリカバリを中止しなければなりません。DBAは手動でクリーンアップ（例えば、ディスク領域の解放やディレクトリ権限の修正など）を行い、その後リカバリを再開する必要があります。これが、元のアクションが成功するまでWALエントリを書き込まない理由の一部です。


新しいRelFileLocatorのためのWALスキップ
--------------------------------

wal_level=minimal の下で、ROLLBACKによってunlinkされる可能性のあるrelfilenumberを変更する場合、インツリーのアクセスメソッドはその変更に対してWALを書き込みません。RelationNeedsWAL()を呼び出さずにWALを書き込むコードは、このケースをチェックする必要があります。このスキップは必須です。同じブロックに対してWALを書き込む変更がWALをスキップする変更の前に発生した場合、REDOがWALをスキップする変更を上書きする可能性があります。同じブロックに対してWALを書き込む変更がWALをスキップする変更の後に発生した場合、関連する問題が発生します。WALレコードにフルページイメージが含まれていない場合、REDOはページがレコード挿入直前の内容と一致することを期待します。WALをスキップする変更はディスクに全く到達しない可能性があり、full_page_writes=off の下でのREDOの期待に違反します。どのアクセスメソッドであっても、CommitTransaction()はコミットを記録する前に影響を受けるブロックを書き込み、fsyncします。

将来のアクセスメソッドでも同様にすることを推奨します。ただし、他の2つのアプローチも機能します。第一に、アクセスメソッドは FlushRelationBuffers() および smgrimmedsync() を呼び出すことで、特定のフォークをWALスキップからWAL書き込みへ不可逆的に移行させることができます。第二に、アクセスメソッドは永続リレーションに対して無条件にWALを書き込むことを選択できます。これらアプローチの下では、アクセスメソッドのコールバックは RelationNeedsWAL() に反応する関数を呼び出してはなりません。

これは、リプレイによって新しいrelfilenumberに格納されたバイトが変更されるWALレコードにのみ適用されます。XLOG_SMGR_CREATE のような、relfilenumberに関する他のレコードには適用されません。個々のrelfilenumberレベルで動作するため、密結合のリレーションでは RelationNeedsWAL() が異なる場合があります。「CREATE TABLE t (); BEGIN; ALTER TABLE t ADD c text; ...」を検討してください。この場合、ALTER TABLEはTOASTリレーションを追加します。TOASTリレーションはWALをスキップしますが、それを所有するテーブルはスキップしません。ALTER TABLE SET TABLESPACE はテーブルにWALをスキップさせますが、そのインデックスには影響しません。


非同期コミット
-------------------

PostgreSQL 8.3以降、非同期コミットを実行できるようになりました。つまり、コミットのWALレコードがfsyncされるのを待機しません。
synchronous_commit = off の場合、非同期コミットを実行します。コミットのLSNまで XLogFlush() を実行する代わりに、共有メモリ内のLSNを記録するだけです。その後、バックエンドは他の作業を続行します。非同期コミットの場合のみLSNを記録し、アボートの場合は記録しません。クラッシュ後の前提はトランザクションがアボートしたと見なされるため、アボートレコードをフラッシュする必要は全くありません。

リレーションを削除するトランザクションでは、リレーションがファイルシステムから削除される前にコミットレコードがディスクに書き込まれることを保証するため、常に同期コミットを強制します。また、ファイルシステムの変更など、ロールバック不可能な副作用を持つ特定のユーティリティコマンドも、ファイルシステムの変更が行われたにもかかわらずトランザクションのコミットが保証されていない期間を最小限に抑えるため、同期コミットを強制します。

walwriterは定期的に（wal_writer_delayを介して）起動するか、（非同期にコミットするバックエンドによって設定されるラッチを介して）起こされ、XLogBackgroundFlush() を実行します。これは最後に完全に埋まったWALページの場所をチェックします。もしその場所が前方に移動していれば、その時点までのすべての変更されたバッファを書き込みます。これにより、高負荷時には完全なバッファのみを書き込むようになります。アクティビティが途切れ、現在のWALページが以前と同じであれば、最新の非同期コミットのLSNを特定し、必要に応じて（つまり、現在のWALページ内にある場合）その時点まで書き込みます。前回のフラッシュから wal_writer_delay 以上の時間が経過しているか、wal_writer_flush_after ブロック以上が書き込まれている場合、WALも現在の場所までフラッシュされます。この仕組み自体が、非同期コミットレコードがトランザクション完了後、最大でも wal_writer_delay の2倍の時間でディスクに到達することを保証します。しかし、walwriterのサイクルごとに複数のWALページが埋まるような高負荷時に発行される書き込み数を最小限に抑えるため、XLogFlush が完全なバッファを「柔軟に」（つまり、循環WALバッファ領域の末尾で折り返さずに）書き込み/フラッシュすることも許可しています。これにより、最悪の場合の遅延は wal_writer_delay サイクルの3倍になります。

非同期コミットに関して考慮すべき他の微妙な点があります。第一に、CLOGの各ページについて、ページに影響を与える最新のコミットのLSNを記憶しておく必要があります。これにより、通常のリレーションページに対して行う「書き込み前にWALをフラッシュする」というルールを強制できます。そうしないと、コミットのレコードがWALレコードよりも先にディスクに到達してしまう可能性があります。繰り返しになりますが、アボートレコードはこの考慮事項に含める必要はありません。

実際には、各clogページに対して複数のLSNを格納しています。これは、可視性テスト中にトランザクションステータスのヒントビットを設定する方法に関連しています。リレーションページにトランザクションコミットのヒントビットを設定し、そのレコードがコミットのWALレコードよりも先にディスクに書き込まれるような事態は避けなければなりません。可視性テストは通常、バッファ共有ロックを保持したまま行われるため、WAL同期を保証するためにページのLSNを変更する選択肢はありません。その代わり、トランザクションに関連付けられたLSNまでWALをフラッシュしていない場合は、ヒントビットの設定を延期します。これには、フラッシュされていない各非同期コミットのLSNを追跡する必要があります。このデータをclogバッファに関連付けるのが便利です。clogページを書き込む前にWALをフラッシュするため、トランザクションのLSNを、そのコミットステータスを保持するclogページがメモリ内にある期間よりも長く記憶する必要がないことがわかっているからです。しかし、各clog位置に対してLSNを格納する単純なアプローチは魅力的ではありません。LSNは2ビットのコミットステータスフィールドよりも32倍大きく、8Kのclogバッファページごとに256Kの追加共有メモリが必要になるためです。その代わり、ページごとに少数のLSNを格納することを選択しました。各LSNは、そのページ上の連続するトランザクションID範囲内の任意のトランザクションコミットに関連付けられた最大のLSNです。これにより、トランザクションヒントビットの設定において多少の不要な遅延が発生する代償として、ストレージを節約できます。

同じキャッシュされたLSN（N）を何個のトランザクションで共有すべきでしょうか？システムのワークロードが小さな非同期コミットトランザクションのみで構成されている場合、Nをwalwriterサイクルあたりのトランザクション数と同程度にするのが妥当です。なぜなら、それがトランザクションが実際にコミット（したがってヒント可能）になる粒度だからです。最悪のケースは、同期コミットのトランザクションが、少し後にコミットされる非同期コミットのトランザクションとキャッシュされたLSNを共有する場合です。最初のトランザクションをディスクに同期させるコストを支払ったにもかかわらず、2番目のトランザクションが同期される（最大3 walwriterサイクル後）まで、その出力をヒントすることができません。これは、N（グループサイズ）を可能な限り小さく保つべきという議論になります。現時点ではグループサイズを32に設定しており、これによりLSNキャッシュ領域は（BLCKSZに関係なく）実際のclogバッファ領域と同じサイズになります。

同期コミットトランザクションと非同期コミットトランザクションを同時に実行できることは有用ですが、その安全性はすぐには明らかではないかもしれません。2つのトランザクションT1とT2があると仮定します。ログシーケンス番号（LSN）は、トランザクションコミットが記録されるWALシーケンス内のポイントであるため、LSN1とLSN2はそれらのトランザクションのコミットレコードです。T2がT1によって行われた変更を確認できる場合、T2がコミットするときには LSN2 が LSN1 に続くことが真でなければなりません。したがって、T2がコミットするとき、T1によって行われたすべての変更もWALに記録されていることが確実です。これはT1が非同期であっても同期であっても真です。その結果、非同期コミットと同期コミットが、同期コミットによって書き込まれたデータを危険にさらすことなく同時に動作することは安全です。サブトランザクションは、ディスクへの最終的な書き込みがトップレベルトランザクションのコミット時にのみ発生するため、ここでは重要ではありません。

データブロックへの変更は、データブロックのLSNのポイントまでWALがフラッシュされない限り、ディスクに到達できません。安全でないデータをディスクに書き込もうとすると、そのトランザクションおよびそれ以前のトランザクションによって書き込まれたすべてのデータの安全性を保証する書き込みがトリガーされます。データブロックとclogページはどちらもLSNによって保護されています。

一時テーブルへの変更はWALログに記録されないため、T1のコミットよりも先にディスクに到達する可能性がありますが、一時テーブルの内容はクラッシュしても残らないため、気にする必要はありません。

新しいrelfilenumberのためにWALをスキップするデータベース書き込みも安全です。これらのケースでは、T1がインターロックなしでディスクにfsyncするため、データがT1のコミットよりも先にディスクに到達することは十分にあり得ます。しかし、これらのパスはすべて、T1がコミットされるまで他のトランザクションが見ることができないデータを書き込むように設計されています。したがって、状況は通常のWALログに記録される更新と変わりません。

リカバリ中のトランザクションエミュレーション
-------------------------------------リカバリ中、私たちはトランザクションの変更を発生順にリプレイします。
このリプレイの一環として、読み取り専用バックエンドがMVCCスナップショットを取得できるように、いくつかのトランザクション動作をエミュレートします。これを行うために、リプレイ中のトランザクションに属するXIDのリストを保持し、データベースへの書き込みに関するWALレコードを記録した各トランザクションがコミットされるまで配列内に存在するようにします。詳細はprocarray.c内のコメントに記載されています。

多くの操作はWALレコードを一切書き込みません。例えば、読み取り専用トランザクションがこれに該当します。これらはリカバリ中のMVCCには影響を与えないため、発生しなかったものとして扱うことができます。サブトランザクションのコミットもWALレコードを書き込まず、影響はほとんどありません。ロックの待機者は親トランザクションの完了を待つ必要があるためです。

すべてのトランザクション動作がエミュレートされるわけではありません。例えば、ロックテーブルへのトランザクションエントリの挿入は行われず、メモリ内のトランザクションスタックも保持されません。Clog、multixact、およびcommit_tsのエントリは通常通り作成されます。Subtransはリカバリ中も維持されますが、トランザクションツリーの詳細は無視され、すべてのサブトランザクションはトップレベルのTransactionIdを直接参照します。コミットはアトミックであるため、これにより正しいロック待機動作が提供され、かつサブトランザクションのエミュレーションが大幅に簡素化されます。

リカバリにおけるロックメカニズムの詳細については、Lock rmgrコード内のコメントを参照してください。
