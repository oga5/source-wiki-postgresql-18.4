src/backend/access/spgist/README

SP-GiST は space-partitioned GiST の略称です。これは、四分木 (quadtree)、k-d 木 (k-d tree)、基数木 (radix tree/trie) といった空間分割データ構造を実装するための汎用的なインフラストラクチャを提供します。メインメモリ上に実装される場合、これらの構造は通常、ポインタでリンクされた動的に割り当てられるノードの集合として設計されます。これはディスクへの直接保存には適していません。ポインタの連鎖が非常に長くなり、ディスクアクセス回数が過剰に増えてしまうためです。対照的に、ディスクベースのデータ構造は、I/O を最小限に抑えるために高いファンアウト（分岐数）を持つべきです。課題は、ツリーのノードをディスクページにマッピングする際に、たとえ多くのノードを走査する場合であっても、検索アルゴリズムがアクセスするディスクページを少数に抑えるようにすることです。


共通構造の記述

論理的に、SP-GiST ツリーはタプルの集合であり、各タプルは内部タプル (inner tuple) またはリーフタプル (leaf tuple) のいずれかです。各内部タプルには「ノード」が含まれており、これは (ラベル, ポインタ) のペアです。ここでポインタ (ItemPointerData) は、別の内部タプル、またはリーフタプルのリストの先頭を指します。内部タプルはそれぞれ異なる数のノード（子）を持つことができます。ブランチは異なる深さを持つことができ（実際には、バランスを維持するための制御やコードは存在しません）、これはツリーが非平衡であることを意味します。ただし、リーフタプルと内部タプルを同じレベルで混在させることはできません。内部タプルのノードからのダウンリンクは、1 つの内部タプル、またはリーフタプルのリストのいずれかに繋がります。

SP-GiST コアは、内部タプルとリーフタプルが単一のインデックスページに収まることを要求します。さらに厳密には、単一の内部タプルノードから到達するリーフタプルのリストはすべて、同じインデックスページに格納されなければなりません。（このようなリストがページをまたがないように制限することでシークを減らし、リストのリンクを単純な 2 バイトの OffsetNumber として格納できるようになります。）したがって、SP-GiST インデックスの演算クラスは、1 つの内部タプルに必要なノード数が多すぎないようにし、内部タプルのプレフィックスやリーフノードのデータ値が大きくなりすぎないようにする必要があります。

内部タプルとリーフタプルは別々に格納されます。前者は「内部」ページにのみ格納され、後者は「リーフ」ページにのみ格納されます。また、ルートページには特別な制限があります。インデックス作成の初期段階で、データが 1 ページ分しかない場合、ルートページには整理されていないリーフタプルの集合が含まれます。最初のページ分割が発生した後は、ルートには正確に 1 つの内部タプルが含まれる必要があります。

検索走査アルゴリズムが内部タプルに到達すると、ツリーの深さを走査し続けるための一連のノードを選択します。リーフページに到達した場合は、リーフタプルのリストをスキャンしてクエリに一致するものを見つけます。SP-GiST は順序付き（最近傍）検索もサポートしています。つまり、スキャン中に保留中のノードを優先度付きキューに入れ、最も近いものから順に走査を行うモデルです。


挿入アルゴリズムも同様にツリーを降下しますが、各内部タプルから降下するノードを 1 つだけ選択する必要があります。挿入時には、降下する前に内部タプルを変更しなければならない場合があります。新しいノードを追加したり、挿入する値と一致するように、より一般的でないプレフィックスを得るためにタプルを「分割」したりすることがあります。新しいリーフタプルをリストに追加する必要があるにもかかわらずページに空き容量がない場合、SP-GiST は新しい内部タプルを作成し、リーフタプルを（おそらく複数のページにまたがる）一連のリストに分散させます。

内部タプルは以下で構成されます：

  オプションのプレフィックス値 - すべての後続ノードはこれと整合している必要があります。
    例：
        基数木 - プレフィックス値は共通のプレフィックス文字列
        四分木 - 重心
        k-d 木 - 1 つの座標

  ノードのリスト。ノードは (ラベル, ポインタ) のペア。
    ラベルの例：基数木における 1 文字

リーフタプルは以下で構成されます：

  リーフ値
    例：
        基数木 - 文字列の残り（ポストフィックス）
        四分木および k-d 木 - 点そのもの

  対応するヒープタプルへの ItemPointer
  リーフページ上のチェーンにおける次のリーフタプルの nextOffset 番号

  オプションの NULL ビットマスク
  オプションの INCLUDE カラム値

v14 より前のインデックスとの互換性のため、リーフタプルが NULL ビットマスクを持つのは、NULL 値が存在し（リーフ値と INCLUDE 値の間に）、かつ少なくとも 1 つの INCLUDE カラムが存在する場合のみです。リーフ値が NULL かどうかは、そのタプルが「NULL ページ」（後述）にあるかどうかから推測できるため、明示的に表現する必要はありません。しかし、標準的なタプル分解コードを使用できるように、INCLUDE 値と共に使用されるビットマスクに含めています。


NULL の扱い

SPGiST インデックス可能な演算子は厳密（NULL 入力に対しては決して成功しない）であると仮定します。それでも、テーブル全体のインデックススキャンを可能にし、「x IS NULL」を SPGiST インデックススキャンで実装できるようにするために、NULL をインデックス化することは望ましいです。しかし、SPGiST インデックスの演算クラスが NULL を処理しなくて済むようにするのが望ましいと考えられます。そのため、SPGiST インデックスのメインツリーには NULL エントリは含まれません。NULL エントリは、分離されたページセット（特に、それ専用のルートページ）を占有する別の SPGiST ツリーに格納します。NULL ツリーでの挿入と検索では、演算クラスが提供する関数は使用されず、通常のツリーにおける AllTheSame のケースと同様のハードコードされたロジックが使用されます。


挿入アルゴリズム

挿入アルゴリズムは、常にツリーを整合性のある状態に保つように設計されています。以下は簡略化された挿入アルゴリズムの仕様です（番号は以下の注釈を参照）：

  ルートページの最初のタプルから開始する (1)

  ループ：
    if (ページがリーフ) then
        if (十分な空き容量がある)
            ページに挿入して終了 (5)
        else (7)
            PickSplitFn() を呼び出す (2)
        end if
    else
        switch (chooseFn())
            case MatchNode  - 選択されたノードを降下する
            case AddNode    - ノードを追加し、chooseFn を再試行する (3, 6)
            case SplitTuple - 内部タプルをプレフィックスとポストフィックスに分割し、
                              プレフィックスタプルで chooseFn を再試行する (4, 6)
    end if

注釈：

(1) 最初は、ページがいっぱいになるまでリーフタプルをルートページに詰め込み、その後分割します。ルートがリーフページでなくなると、ルート上の空き容量を最大限に確保するために、内部タプルを 1 つだけ持つことができます。これらのルールはどちらも、ルートに対する PickSplit を可能な限り先延ばしにし、検索空間の最上位のパーティショニングを可能な限り最適にするためのものです。

(2) 現在の実装では、新しいリーフタプルのリストが 1 ページに収まる場合、picksplit を実行して新しいリーフタプルを挿入する操作を 1 回で行うことができます。四分木や k-d 木のような小さなノードを持つツリーでは常に可能ですが、基数木では別の picksplit が必要になる場合があります。

(3) ノードの追加は、内部タプルのサイズをページに収まる程度に小さく保つ必要があります。追加後、ページ上の他のタプルの影響で内部タプルが現在のページに収まらなくなることがあります。その場合、別の内部ページに移動されます（ページ管理に関する注釈を参照）。タプルを別のページに移動する際、ページ上の他のタプルの番号を変更することはできません。変更すると、それらへのダウンリンクポインタが無効になってしまうためです。これを防ぐため、SP-GiST は「プレースホルダー」タプルを残し、後でそのページに別のタプルが追加されたときに再利用できるようにします。以下の「並行処理」および「バキューム」セクションも参照してください。現在、タプルにノードを追加できるのは基数木のみです。四分木と k-d 木は、PickSplitFn() の呼び出し時に可能なすべてのノードを一度に作成します。

(4) プレフィックス値は新しい値と部分的にしか一致しない可能性があるため、SplitTuple アクションによって現在のツリーブランチを上下のセクションに分割できます。別の言い方をすれば、現在の内部タプルを「プレフィックス」部分と「ポストフィックス」部分に分割し、プレフィックス部分が新しい値と一致するようにします。基数木への挿入の例を考えます。以下の表記を使用します（タプルの ID は議論のためのものであり、実際には保存されません）：

内部タプル：{タプル ID}(プレフィックス文字列)[カンマ区切りのノードラベルリスト]
リーフタプル：{タプル ID}<値>

文字列 'www.gogo.com' を内部タプルに挿入する必要があるとします

    {1}(www.google.com/)[a, i]

文字列がプレフィックスと一致しないため、降下できません。内部タプルを 2 つのタプルに分割する必要があります：

    {2}(www.go)[o]  - プレフィックスタプル
                |
                {3}(gle.com/)[a,i] - ポストフィックスタプル

ループの次の反復で、'www.gogo.com' はプレフィックスと一致しますが、ノードラベルとは一致しないことがわかります。そのため、タプル {2} にノード [g] を追加します：

                   NIL (まだ子が存在しない)
                   |
    {2}(www.go)[o, g]
                |
                {3}(gle.com/)[a,i]

これで [g] ノードを降下できるようになり、ターゲット文字列が 'o.com' に更新されます。最後に、その文字列を持つリーフタプルを挿入します：

                  {4}<o.com>
                   |
    {2}(www.go)[o, g]
                |
                {3}(gle.com/)[a,i]

ご覧の通り、元のタプルのノード配列は変更なしでポストフィックスタプルに移動します。また、SP-GiST コアはプレフィックスタプルが古い内部タプルより大きくないことを前提としている点に注意してください。これにより、古い内部タプルの場所に直接プレフィックスタプルを格納できます。SP-GiST コアは可能であれば同じページにポストフィックスタプルを格納しようとしますが、空き容量が足りない場合は別のページを使用します（注釈 5 および 6 を参照）。現在、四分木と k-d 木はこの機能を使用していません。これらにはプレフィックスが新しい値と「整合しない」という概念がないためです。これらは PickSplitFn() の呼び出しによってのみ深さが増加します。

(5) 親のノードからのポインタが NIL ポインタである場合、アルゴリズムは格納先のリーフページを選択します。まず、ディスク領域を有効活用するために、（各バックエンドで追跡している）空き容量が最大の、最後に使用されたリーフページを使用しようとします。それが十分な大きさでない場合、アルゴリズムは新しいページを割り当てます。

(6) 内部ページの管理は、(5) で説明したリーフページの管理と非常によく似ています。

(7) 実際には、現在の実装では、リストが十分に短ければ、リーフタプルのリスト全体と新しいタプルを別のページに移動できます。これにより領域利用率は向上しますが、アルゴリズムの基本は変わりません。


並行処理

ツリーを降下中、挿入アルゴリズムは一度に 2 つのツリーレベル（親ページと子ページの両方）に対して排他ロックを保持します（ただし、親ページと子ページが同じである場合もあります。上記の注釈を参照）。異なるブランチに相互参照されたページがある場合、2 つの挿入処理間でデッドロックが発生する可能性があります。つまり、ページ M 上の内部タプルがページ N 上に子を持ち、別のブランチの内部タプルがページ N 上にありページ M 上に子を持つ場合、2 つの挿入処理がそれぞれのブランチを降下する際に、親ページのロックを保持したまま子ページのロックを取得しようとするため、デッドロックが発生する可能性があります。

現在、これに対処するため、ツリーを降下する際に条件付きでバッファをロックしています。バッファのロック取得に失敗した場合、両方のバッファを解放して挿入プロセスを再開します。これは効率的ではない可能性がありますが、より決定論的なアプローチのロックコストは非常に高いと考えられます。

これが発生するケースを減らすために、ページの「トリプルパリティ」という概念を導入しています。内部タプルが BlockNumber N のページにある場合、その子タプルは同じページ、または (N+1) mod 3 == M mod 3 となる BlockNumber M のページに配置されるべきです。このルールにより、ページ M 上のタプルはページ N 上に子を持たないことが保証されます（(M+1) mod 3 != N mod 3 であるため）。これにより、ツリー降下中に 2 つの挿入プロセスが競合する可能性が低くなります。ただし、完璧ではありません。第一に、3 つ以上の挿入プロセス間でデッドロックが発生する可能性があり、第二に、内部タプルを拡張または分割するたびにこの不変条件を維持するのは非現実的です。そのため、依然としてデッドロックを許容する必要があります。挿入時には、降下したページに十分な空きがない場合、適切な型のタプルを追加するために、追加の内部ページやリーフページに対してもロックを取得する必要があるかもしれません。しかし、どのページに追加するかは厳密には問わないため、追加のバッファを条件付きでロックすることでデッドロックを回避できます。つまり、追加のページのロック取得に失敗した場合は、別のページを試すだけです。

検索のトラバーサルアルゴリズムは比較的伝統的なものです。非リーフレベルでは、ページを共有ロックし、現在の内部タプル内で訪問が必要なノードを特定し、それらのアドレスを後で調べるためのページスタックに積みます。その後、次のスタック項目を訪問する前に現在のバッファのロックを解放します。したがって、一度にロックされるページは1つだけであり、デッドロックは発生しません。その代わり、競合状態を考慮する必要があります。指し示されたページに到達するまでに、並行する挿入処理によってターゲットの内部タプル（またはリーフタプルチェーン）が別の場所に配置されたデータに置き換えられている可能性があるからです。これに対処するため、挿入アルゴリズムが内部タプルの空でないダウンリンクを変更する際は、以前のリンク先であった下位レベルの内部タプルやリーフタプルチェーンの先頭の代わりに「リダイレクトタプル」を配置します。スキャン（挿入は除く）は、このようなリダイレクトを考慮できるように準備しておく必要があります。親レベルを既に訪問したスキャンのみがこのようなリダイレクトタプルに到達する可能性があるため、システムからすべてのアクティブなトランザクションがフラッシュされた後にリダイレクトを削除できます。


DEAD TUPLES

リーフページ上のタプルは、以下の4つの状態のいずれかになります。

SPGIST_LIVE: ヒープタプルへの通常かつ有効なポインタ。

SPGIST_REDIRECT: インデックス内の別の場所へのリンクを含むプレースホルダー。リーフタプルのチェーンを別のページに移動する必要がある場合、チェーンの先頭タプルの代わりにリダイレクトタプルが挿入されます。これが発生すると親内部タプルのダウンリンクが更新されますが、並行するスキャンは（親ページのロックを解放してから子ページのロックを試みるため）親ページから子ページへ「移動中」である可能性があります。リダイレクトポインタは、そのようなスキャンに対してどこへ行くべきかを伝える役割を果たします。リダイレクトポインタは、そのような並行スキャンが進行中である可能性がある間のみ必要です。最終的にはVACUUMによってPLACEHOLDER（デッドタプル）に変換され、置換の候補となります。（チェーンの一部であるべきではない）このようなタプルを見つけた検索処理は、リダイレクトタプルを無視して直ちに別の場所へ進むべきです。有効なダウンリンクがこのようなタプルを指すことは決してないため、このようなタプルに到達した挿入処理はエラーを発生させるべきです。

SPGIST_DEAD: タプルはデッドですが、インデックス内の他の場所にある内部タプルからリンクされているため、削除したりページ上の別のオフセットに移動したりすることはできません。（このようなタプルは、チェーン内に有効なものが何も残っていない場合を除き、チェーンの一部にはなりません。）検索処理はこのようなエントリを無視すべきです。挿入処理がこのようなタプルに到達した場合は、その場で置換する（ページ内に新しいリーフタプルを保持する十分な空きがある場合）、または新しいリーフタプルを配置する場所へのリダイレクトポインタで置換すべきです。

SPGIST_PLACEHOLDER: タプルはデッドであり、他からのリンクが存在しないことが分かっているものです。有効なタプルが削除または移動され、リダイレクトポインタで置換されなかった場合、同じページ上の後続のタプルのオフセットが変化しないようにプレースホルダーで置換されます。プレースホルダーは、ページに新しいタプルを追加する際に自由に置換でき、またVACUUMは有効なタプルオフセット範囲の末尾にあるものを削除します。検索処理と挿入処理の両方は、他からのリンクがプレースホルダータプルに導かれた場合は警告を発するべきです。

ルートページがリーフでもある場合、そのすべてのタプルはLIVE状態でなければなりません。リンクが存在せず、オフセット番号を保持する必要もないため、他の状態は不要です。

内部ページ上のタプルは、LIVE、REDIRECT、またはPLACEHOLDER状態になり得ます。REDIRECT状態はリーフページと同じ機能を持ち、内部タプルが別のページに移動された後に、並行する検索処理を正しい場所へ誘導します。期限切れのREDIRECTポインタはVACUUMによってPLACEHOLDER状態に変換され、置換の候補となります。VACUUMは未使用の内部タプルを削除しようとしないため、DEAD状態は現在発生しません。


VACUUM

VACUUM（より正確には spgbulkdelete）は、インデックス全体に対して単一のシーケンシャルスキャンを実行します。リーフページと内部ページの両方で、古いREDIRECTタプルをPLACEHOLDER状態に変換し、その後ページの末尾にあるPLACEHOLDERを削除できます（これらは有効なタプルのオフセットを保持するために必要ないため）。リーフページでは、ヒープTIDがVACUUM対象のTIDと一致するために削除が必要なタプルをスキャンします。

チェーンの先頭ではない削除可能なタプルを見つけた場合、単にPLACEHOLDERで置換し、チェーンリンクを更新してチェーンから削除できます。チェーンの先頭にある場合でも、チェーン内に少なくとも1つの有効なタプルが残っていれば、その有効なタプルを先頭タプルのオフセットに移動し、他のタプルのオフセットを保持するためにPLACEHOLDERで置換します。これにより、親内部タプルのダウンリンクが有効に保たれます。チェーン内のすべての有効なタプルを削除することになった場合は、先頭タプルをDEADタプルで置換し、残りをPLACEHOLDERで置換します。これにより、親内部タプルのダウンリンクはDEADタプルを指すことになり、前述のルールによってすべてが正常に機能します。

VACUUMは、どのタプルがチェーンの先頭であるかを事前に知りませんが、nextOffsetリンクの逆マップである先行配列を構築することで簡単に把握できます（つまり、タプルxがタプルyにリンクしていることが分かれば、predecessor[y] = x と設定します）。その結果、先行タプルを持たないものが先頭タプルとなります。

VACUUMの実行中にも挿入が発生する可能性があるため、純粋なシーケンシャルスキャンでは一部のターゲットリーフタプルの削除を見逃す可能性があります。PickSplitやMoveLeafs操作の結果として、まだ訪問していないリーフページから既に訪問済みのリーフページへ移動される可能性があるためです。ターゲットTIDを削除し損ねることは許されないため、このようなケースに対処できるようにアルゴリズムを拡張する必要があります。VACUUMスキャン開始後に作成された可能性を示すXIDを持つリーフページのリダイレクトタプルを見つけた場合、そのような移動が発生した可能性があると認識します。リダイレクト先のTIDを再チェックが必要な「保留リスト」に追加します。メインのシーケンシャルスキャンのページ間において、リストされた各TIDを訪問することで保留リストを空にします。（PickSplitによる）内部タプルを指している場合は、各ダウンリンクTIDを保留リストに追加します。リーフページを指している場合は、そのページをVACUUMします。（指し示された単一のチェーンのみをVACUUMすることもできますが、ページ全体をVACUUMする方がコードが単純化され、VACUUMが同じページを複数回修正する可能性が減ります。）並行するインデックス変更があっても保留リストの処理が無限ループに陥らないように、リストエントリは即座に削除せず、すべての保留リスト処理が完了した後にのみ削除します。代わりに、処理後にアイテムを完了済みとしてマークします。リストに既に存在するTIDを追加することは、そのアイテムが完了済みとしてマークされているかどうかにかかわらず、何もしない（no-op）操作となります。

spgbulkdeleteはインデックスの空き領域マップも更新します。

現在、spgbulkdeleteが実行された場合、spgvacuumcleanupには何もすることがありません。それ以外の場合は、リダイレクトとプレースホルダーをクリーンアップし、空き領域マップを更新し、統計情報を収集するために、空のターゲットリストでspgbulkdeleteスキャンを実行します。


最後に使用したページ管理

最後に使用したページのリストには4つのページが含まれます。1つのリーフページと、各「トリプルパリティ」グループから1つずつの計3つの内部ページです。（実際には、メインツリー用に1つ、NULLツリー用に別のリストが存在します。）このリストは呼び出しの合間にインデックスのメタページに保存されますが、WALトラフィックを減らすために更新はWALログに記録されません。メタページ上のデータが不正確であっても、いつでも新しいページを割り当てることができるため、致命的ではありません。
