GiST インデックス
=================

このディレクトリには、Postgres 用の GiST インデックスの実装が含まれています。

GiST は Generalized Search Tree（汎用探索木）の略です。これは、1995 年の記念碑的な論文 "Generalized Search Trees for Database Systems" (Joseph M. Hellerstein, Jeffrey F. Naughton, Avi Pfeffer 著) で導入されました。

    http://www.sai.msu.su/~megera/postgres/gist/papers/gist.ps
    https://dsf.berkeley.edu/papers/sigmod97-gist.pdf

そして、PostgreSQL の初期バージョンにおいて J. Hellerstein と P. Aoki によって実装されました（詳細は、バークレー校の The GiST Indexing Project http://gist.cs.berkeley.edu/ を参照してください）。「大学」のプロジェクトであったため、機能は限定的で、ほとんど使用されていませんでした。

現在の GiST の実装は以下をサポートしています：

  * 可変長キー
  * 複合キー（マルチキー）
  * 順序付き探索（最近傍探索）
  * GiST コアへの NULL セーフなインターフェースの提供
  * 同時実行制御
  * WAL ログによるリカバリサポート
  * バッファリング構築アルゴリズム
  * ソート済み構築メソッド

PostgreSQL に実装された同時実行制御のサポートは、Marcel Kornacker による論文 "Access Methods for Next-Generation Database Systems" に基づいて開発されました。

    http://www.sai.msu.su/~megera/postgres/gist/papers/concurrency/access-methods-for-next-generation.pdf.gz

GiST のバッファリング構築アルゴリズムは、Lars Arge、Klaus Hinrichs、Jan Vahrenhold、Jeffrey Scott Vitter による論文 "Efficient Bulk Operations on Dynamic R-trees" に基づいて開発されました。

    http://citeseerx.ist.psu.edu/viewdoc/download?doi=10.1.1.135.9894&rep=rep1&type=pdf

オリジナルのアルゴリズムはいくつかの点で修正されています：

* PostgreSQL の慣習に適応させる必要がありました。例えば、SEARCH アルゴリズムは大幅に変更されました。PostgreSQL では、検索関数は一度にすべてのタプルを返すのではなく、1 つのタプル（next）を返す必要があるためです。また、呼び出しの間にページロックを解放する必要があります。
* 可変長キーのサポートを追加したため、分割後にページ上のすべてのキーに対して十分な空き容量を保証することは不可能です。ユーザー定義関数 picksplit はタプルのサイズに関する情報を持たず（picksplit は 1 つのキーのみを扱う可能性がある一方で、各タプルはマルチカラムインデックスのように複数のキーを含む可能性がある）、ページに関する情報も持ちません。
* パフォーマンス上の理由から、オリジナルの INSERT アルゴリズムを修正しました。特に、現在はシングルパスのアルゴリズムになっています。
* 論文は理論的なものであったため、いくつかの詳細が省略されており、特定のいくつかの問題を解決する方法を自力で見つける必要がありました。

上記の理由により、GiST コアと PostgreSQL WAL システムの相互作用を修正しました。さらに、クラッシュ後のリカバリ時に完了していない挿入の問題に遭遇し（解決済み）、これは論文では触れられていませんでした。

探索アルゴリズム
----------------

探索コードは未訪問アイテムのキューを保持します。ここで「アイテム」とは、探索条件を満たすことが判明しているヒープタプル、または親ページのダウンリンクアイテムの検査に基づき探索条件と整合するインデックスページのことです。最初はルートページを探索して、その中の未訪問アイテムを見つけます。次に、キューからアイテムを取り出します。ヒープタプルポインタはそのまま即座に返されます。インデックスページエントリの場合はそのページが探索され、さらなるキューエントリが生成されます。

キューは、ヒープタプルアイテムを先頭に、次にインデックスページエントリが来るように順序付けが維持されます。新しく追加されたインデックスページエントリは、既存のインデックスページエントリの前に挿入されます。これによりインデックスの深さ優先探索が保証され、特に最初のいくつかのヒープタプルが可能な限り早く返されるようになります。これは、少数のタプルのみを生成すればよい LIMIT がある場合に役立ちます。

最近傍探索を実装するため、キューエントリには距離データが追加されます。ヒープタプルエントリには探索引数からの正確な距離がラベル付けされ、インデックスページエントリにはその子ノードが持ち得る最小距離がラベル付けされる必要があります。その後、キューエントリは最小距離順に取り出され、距離が同一のエントリは前述の段落で述べた通りに管理されます。

探索アルゴリズムは、インデックスページのエントリをスキャンして探索条件を満たすものをキューに入れるのに十分な時間だけ、ページをロックします。挿入は探索と並行して発生する可能性があるため、キューエントリを作成した時点（親ページを訪問中）から実際に子ページに到達してスキャンするまでの間に、インデックスの子ページが分割される可能性があります。右側の兄弟ページに移動されたエントリを見逃さないように、子ページの NSN（ノードシーケンス番号、特殊な LSN）を、訪問時に親が持っていた LSN と比較することで、分割が発生したかどうかを検出します。分割が発生していた場合、兄弟ページが即座にキューの先頭に追加され、元のページにあった場合と同じ順序でアイテムがスキャンされることが保証されます。

Postgres の通常の動作として、探索アルゴリズムはスキャン開始前に存在していたインデックスエントリを見つけることのみを保証します。スキャン中に追加されたインデックスエントリは訪問されることもあれば、されないこともあります。すべての探索が MVCC スナップショットルールを使用してスキャン開始時より新しいヒープタプルを拒否する限り、これは問題ありません。特に、これは親ページのダウンリンクキーが参照後に「拡大」された場合について心配する必要がないことを意味します。そのような拡大は、いずれにせよ返す必要のない子アイテムを追加するためのものだからです。


挿入アルゴリズム
----------------

INSERT は GiST 木が平衡を保つことを保証します。挿入するサブツリーを選択するためにユーザー定義のキーメソッド Penalty が使用され、ノード分割アルゴリズムには PickSplit メソッドが、木の特性を維持するために変更を上方に伝播させるには Union メソッドが使用されます。

タプルを挿入するには、まず挿入に適したリーフページを見つける必要があります。アルゴリズムはルートから開始し、最小の Penalty を持つパスに沿って木を降ります。各ステップで以下を行います：

1. 親を参照してからこのページが分割されていないかを確認します。もし分割されていれば、もう半分の方に挿入すべき可能性があるため、親に戻ります。
2. これがリーフノードであれば、ターゲットノードが見つかったことになります。
3. そうでなければ、Penalty を使用して新しいターゲットサブツリーを選択します。
4. ターゲットサブツリーを表すキーを確認します。挿入しようとしているキーをまだカバーしていない場合は、古いダウンリンクキーと挿入するキーの Union で置き換えます。（実際には常に Union を呼び出し、Union されたキーが既存のキーと同じであれば置き換えをスキップします）
5. ステップ 4 でキーを置き換えると、ページが分割される可能性があります。その場合は変更を上方に伝播させ、分割する必要がなかった最初の親からアルゴリズムを再開します。
6. ターゲットサブツリーまで降りて、1 に戻ります。

これはオリジナルの論文の挿入アルゴリズムとは異なります。オリジナルの論文では、まずリーフページに到達するまで木を降り、次に親のダウンリンクを調整し、最悪の場合はルートまで調整を伝播させます。しかし、私たちは木を降りる時点で新しいキーをカバーするようにダウンリンクを調整するため、リーフページに到達したときには、ページ分割が必要でダウンリンクを挿入する場合を除き、親を更新する必要はありません。これによりクラッシュリカバリが単純化されます。ページにキーを挿入した後、親を更新しなくても木は即座に自己整合性を保ちます。ページを分割して、親へのダウンリンクを挿入する前にクラッシュした場合でも、分割の右半分は左ページの rightlink（元のページを置き換えたもの）を介してアクセスできるため、木は自己整合性を保ちます。

新しいキーのために内部ページを分割する必要がある場合、リーフページに到達する前にアルゴリズムが木を上下に移動する可能性があることに注意してください。最終的には最下部に到達し、新しいタプルの挿入に進みます。

挿入先のターゲットページが見つかったら、新しいタプルのための空きがあるかを確認します。空きがあればタプルを挿入して完了です。しかし、収まらない場合はページを分割する必要があります。キーの長さが異なる場合や、一度に複数のキーが挿入される場合（下位レベルでのページ分割の結果としてダウンリンクを挿入する際に発生する可能性があります）、ページを 2 つ以上に分割する必要がある可能性があることに注意してください。ページを分割した後、親ページを更新する必要があります。新しいページのダウンリンクを挿入し、分割の左半分となった古いページのダウンリンクを、左ページに残ったタプルのみをカバーするように更新する必要があります。親へのダウンリンク挿入は再びページ分割を引き起こす可能性があり、最悪の場合はルートページまで再帰します。

gistplacetopage は挿入の 1 ステップを実行する中心的な関数です。タプルが収まれば指定されたページに挿入し、収まらなければページを分割して、分割されたページ用の新しいダウンリンクタプルを構築します。呼び出し元は、ダウンリンクタプルを挿入するために親ページに対して gistplacetopage() を呼び出す必要があります。子へのダウンリンクを保持する親ページは、親の同時分割の結果として移動している可能性があるため、gistFindCorrectParent() を使用して親ページを見つけます。

ルートページの分割は少し異なります。ルート分割時、gistplacetopage() は新しい子ページを割り当て、古いルートページを新しい子へのダウンリンクを含む新しいルートで置き換える処理を、すべて 1 つの操作で行います。

findPath は findParent のサブルーチンであり、親レベルで rightlink をたどっても正しい親ページが見つからない場合に使用されます：

findPath( stack item )
	push stack, [root, 0, 0] // ページ, LSN, 親
	while( stack )
		ptr = stack のトップ
		latch( ptr->page, S-mode )
		if ( ptr->parent->page->lsn < ptr->page->nsn )
			push stack, [ ptr->page->rightlink, 0, ptr->parent ]
		end
		for( ページ上の各タプル )
			if ( tuple->pagepointer == item->page )
				return stack
			else
				add to stack at the end [tuple->pagepointer,0, ptr]
			end
		end
		unlatch( ptr->page )
		pop stack
	end

gistFindCorrectParent は、挿入中にページの親を再検索するために使用されます。ページ分割により、木を降りた後で右側に移動している可能性があります。

findParent( stack item )
	parent = item->parent
	if ( parent->page->lsn != parent->lsn )
		while(true)
			parent->page 上で親タプルを検索し、見つかれば return
			rightlink = parent->page->rightlink
			unlatch( parent->page )
			if ( rightlink が不正 )
				break loop
			end
			parent->page = rightlink
			latch( parent->page, X-mode )
		end
		newstack = findPath( item->parent )
		スタックの一部を新しいものに置き換え
		latch( parent->page, X-mode )
		return findParent( item )
	end

pageSplit 関数は、ページ分割後に新しいページへキーをどのように分配するかを決定します：

pageSplit(page, allkeys)
	(lkeys, rkeys) = pickSplit( allkeys )
	if ( page がルート )
		lpage = 新しいページ
	else
		lpage = page
	rpage = 新しいページ
	if ( rpage に空きがない )
		newkeys = pageSplit( rpage, rkeys )
	else
		push newkeys, union(rkeys)
	end
	if ( lpage に空きがない )
		push newkeys, pageSplit( lpage, lkeys )
	else
		push newkeys, union(lkeys)
	end
	return newkeys

同時実行制御
-------------------
経験則として、複数のページのロックを同時に保持する必要がある場合、ロックは「子ページを親より先に」、かつ「同じレベルでは左から右へ」の順序で取得する必要があります。常に同じ順序でロックを取得することで、デッドロックを回避できます。

探索アルゴリズムは一度に 1 つのページのみを参照・ロックします。その結果、探索とページ分割の間に競合状態が発生します。ページ分割は 2 つのフェーズで行われます：1. ページが分割される、2. ダウンリンクが親に挿入される。探索がこれらのステップの間に親ページを参照し、ダウンリンクが挿入される前に親ページを見た場合、左側のページの rightlink をたどることで新しい右半分を見つけることができます。しかし、親でダウンリンクを見た場合は rightlink をたどってはいけません。さもないとページが 2 回訪問されてしまいます！分割は、最初は左側のページに F_FOLLOW_RIGHT フラグをマークします。スキャンがそのフラグが設定されていることを確認すると、右側のページにダウンリンクが欠落しており、そこも訪問する必要があることを認識します。分割が親へのダウンリンクを挿入する際、子ページの F_FOLLOW_RIGHT フラグをクリアし、子ページのヘッダーにある NSN フィールドを、親への挿入時の LSN と一致するように設定します。F_FOLLOW_RIGHT フラグが設定されていない場合、スキャンは子ページの NSN と親で確認した LSN を比較します。子ページの NSN が親で確認した LSN よりも大きい場合、そのスキャンはダウンリンクが挿入される前に親ページを参照したことになるため、ライトリンク（右リンク）をたどる必要があります。それ以外の場合、スキャンは親ページ内のダウンリンクを確認したことになり、通常通りそれをたどります（またはたどりました）。

通常、スキャンが F_FOLLOW_RIGHT フラグが設定されたページを見ることはありません。なぜなら、ページ分割はダウンリンクが親に挿入され、フラグが再びクリアされるまで子ページをロックし続けるからです。しかし、ページ分割の途中でクラッシュが発生し、ダウンリンクが親に挿入される前に停止した場合、F_FOLLOW_RIGHT が設定されたページがツリーに残ります。スキャンはこれを問題なく処理しますが、パフォーマンス上の理由から、最終的にはこれを修正したいと考えています。さらに重要なことに、親にダウンリンクポインタが欠落しているページを扱うことは、挿入アルゴリズムを複雑にします。そのため、挿入処理が F_FOLLOW_RIGHT が設定されたページを検出した場合、親にダウンリンクを追加することで、途中でクラッシュした分割を直ちに完了させようとします。

バッファリング構築アルゴリズム
-------------------------

バッファリングインデックス構築アルゴリズムでは、一部またはすべての内部ノードにバッファが付属しています。タプルが最上位に挿入されると、ツリーの降下はバッファに到達した時点で停止し、タプルはそのバッファにプッシュされます。バッファが一杯になると、その中のすべてのタプルが下位レベルにフラッシュされ、そこで再び下位レベルのバッファやリーフページに到達します。これにより、挿入は深さ優先ではなく幅優先に近い順序で行われるようになり、必要なランダム I/O の量が大幅に削減されます。

このアルゴリズムでは、レベルはリーフページがレベル0、内部ノードのレベルが1から順にカウントされるように番号付けされます。この番号付けにより、ルートページが分割された場合でも、ページのレベル番号が変わることはありません。

レベル                    ツリー

3                         *
                      /       \
2                *                 *
              /  |  \           /  |  \
1          *     *     *     *     *     *
          / \   / \   / \   / \   / \   / \
0        o   o o   o o   o o   o o   o o   o

* - 内部ページ
o - リーフページ

特定のレベルに属する内部ページには、バッファが関連付けられています。リーフページにはバッファはありません。どのレベルにバッファがあるかは "level step" パラメータによって制御されます。level_step の倍数であるレベル番号にはバッファがあり、それ以外にはありません。例えば、level_step = 2 の場合、レベル 2, 4, 6, ... のページにバッファがあります。level_step = 1 の場合、すべての内部ページにバッファがあります。

レベル        ツリー (level_step = 1)                ツリー (level_step = 2)

3                      *                                     *
                   /       \                             /       \
2             *(b)              *(b)                *(b)              *(b)
           /  |  \           /  |  \             /  |  \           /  |  \
1       *(b)  *(b)  *(b)  *(b)  *(b)  *(b)    *     *     *     *     *     *
       / \   / \   / \   / \   / \   / \     / \   / \   / \   / \   / \   / \
0     o   o o   o o   o o   o o   o o   o   o   o o   o o   o o   o o   o o   o

(b) - バッファ

論理的には、バッファは単なるタプルの集合です。物理的には、一時ファイルに裏打ちされたページに分割されています。各バッファは次の2つの状態のいずれかになります。
a) バッファの最後のページがメインメモリに保持されている。ノードバッファは、新しいインデックス・タプルが追加されたり、タプルが削除されたりすると、自動的にこの状態に切り替わります。
b) バッファのすべてのページがディスクにスワップアウトされている。バッファが一杯になり、フラッシュを開始すると、他のすべてのバッファはこの状態に切り替わります。

インデックス・タプルが挿入される際、その初期処理は以下のいずれかの点で終了します。
1) インデックスの深さが <= level_step の場合、リーフページ。これは、内部ページのいずれにもバッファが関連付けられていないことを意味します。
2) バッファを持つ最上位レベルページのバッファ。

新しいインデックス・タプルは、最上位のバッファレベルにあるバッファのいずれかが半分埋まるまで処理されます。バッファが半分埋まると、空にするキューに追加され、新しいタプルが処理される前に空にされます。

バッファを空にするプロセスとは、バッファ内のインデックス・タプルを下位レベルのバッファまたはリーフページに移動させることを意味します。まず、メモリを解放するために他のすべてのバッファをディスクにスワップします。次に、バッファからタプルを1つずつ取り出し、ツリーをカスケード状に下って、バッファリングされたノードの下にある次のバッファまたはリーフページへ移動させます。

バッファを空にすることには、空にしているバッファと、その下にある次のバッファレベルまたはリーフレベルとの間にある中間ページがキャッシュされるという興味深い動的特性があります。ノードの下にバッファがもうない場合、タプルが最終的に到達するリーフページもキャッシュされます。バッファがある場合は、下位の各バッファの最後のバッファページがメモリに保持されます。これは以下の図に示されています。

   空にされるバッファから               空にされるバッファから
     下位レベルバッファへ                 リーフページへ

               +(fb)                                 +(fb)
            /     \                                /     \
        +             +                        +             +
      /   \         /   \                    /   \         /   \
    *(ab)   *(ab) *(ab)   *(ab)            x       x     x       x

+    - キャッシュされた内部ページ
x    - キャッシュされたリーフページ
*    - キャッシュされていない内部ページ
(fb) - 空にされているバッファ
(ab) - 追加先のバッファ（最後のページがメモリ内にある）

インデックス構築の開始時に、level-step は、1つのバッファを空にする際に関与するすべてのページがキャッシュに収まるように選択されます。そのため、それらの各ページに一度アクセスしてキャッシュされた後は、バッファを空にする際にそれ以上の I/O は発生しません。この局所性が、バッファリングアルゴリズムの高速化の源泉です。

1つのバッファを空にすると、下位レベルのバッファが1つ以上満たされ、それらのバッファも空にする処理がトリガーされます。バッファが一杯になるたびに、それは空にするキューに追加され、現在のバッファが処理された後に空にされます。

最悪の場合でも各バッファのサイズを制限するために、バッファを空にする処理はバッファが半分埋まった時点でスケジュールされ、公称バッファサイズの 1/2 分のタプルが空になるまで継続されます。これにより、バッファを空にする処理が開始されるとき、すべての下位レベルのバッファが最大でも半分しか埋まっていないことが保証されます。すべてのタプルが同じ下位レベルのバッファにカスケードされるという最悪のケースでも、そのバッファには上位レベルのバッファから空にされたすべてのタプルを収容する十分なスペースがあります。ただし、使用されるデータ構造には厳密なサイズ制限はないため、これは近似値で構いません。一部のバッファがわずかにオーバーフローしても問題はありません。

バッファが関連付けられている内部ページが分割される場合、バッファも分割される必要があります。バッファ内のすべてのタプルがスキャンされ、ペナルティ関数を使用して各タプルがどのバッファに行くべきかを決定し、正しい兄弟バッファに再配置されます。

ヒープからのすべてのタプルの処理が完了した後も、バッファにはいくつかのインデックス・タプルが残っています。この時点で、最終的なバッファの空き処理が開始されます。すべてのバッファはトップダウンの順序で空にされます。ページ分割によって空にする最中に新しいバッファが割り当てられる可能性があるため、これは少し複雑です。しかし、新しいバッファは常にまだ完全には空にされていないバッファの兄弟となり、タプルがツリーを上方向に移動することはありません。最終的な空き処理では、特定のレベルのバッファをすべて空にするまでループし、その後次のレベルへ移動します。

ソート済み構築メソッド
-------------------

すべての入力タプルをソートし、ソート順で GiST リーフページに詰め込み、進むにつれてダウンリンクと内部ページを作成します。このメソッドは、B-tree インデックスの構築方法と同様に、ボトムアップでインデックスを構築します。

ソート済みメソッドは、すべての列の演算子クラスに "sortsupport" が定義されている場合に使用されます。それ以外の場合は、オプションのバッファリングを伴うタプルの1つずつの挿入にフォールバックします。

GiST 構築のソートには、ソート演算子クラスの良好な線形化が必要です。多次元データでは必ずしもそうとは限りません。この異常に対処するため、インデックス・タプルをバッファリングし、多次元を意識できる picksplit 関数を適用します。

一括削除アルゴリズム (VACUUM)
------------------------------

VACUUM は2つの段階で動作します。

第1段階では、インデックス全体を物理的な順序でスキャンします。同時実行中のページ分割によって移動したデッドタプルを見逃さないように、各ページの F_FOLLOW_RIGHT フラグと NSN をチェックし、ページが同時に分割されたかどうかを検出します。同時ページ分割が検出され、ページの一方がすでにスキャン済みの位置に移動していた場合、そのページを再度スキャンするために「後ろにジャンプ」します。これは B-tree VACUUM が使用するのと同じメカニズムですが、検索中のページ分割を検出するためにページ上にすでに NSN があるため、B-tree のような「バキュームサイクル ID」の概念は必要ありません。

すべてのページをスキャンする間、完全に空のリーフページにも注目します。スキャン後にそれらをツリーからリンク解除しようとします。また、すべての内部ページのブロック番号を記録します。これらは、リンク解除中に空のページの親を見つけるために必要です。

空のリーフページをツリーからリンク解除し、そのスペースを再利用できるようにします。空のページを削除するには、そのダウンリンクを親から削除する必要があります。第1段階で記憶したブロック番号を持つすべての内部ページをスキャンし、空であると記憶したページへのダウンリンクを探します。見つかった場合は、親ページと子ページのロックを取得し、子ページがまだ空であることを再確認します。その後、ダウンリンクを削除して子を削除済みとしてマークし、ロックを解放します。

内部ページが完全に空になると、挿入アルゴリズムが混乱します。そのため、内部ページが空であっても、その最後の子供は削除しません。現在、リーフページの削除のみをサポートしています。

このページ削除アルゴリズムはベストエフォート型です。第1段階の後に同時ページ分割によってダウンリンクが移動した場合、それを見つけられない可能性があります。その場合、すべての空のページを削除することはできません。これは問題ありません。頻繁に発生することは想定されておらず、次の VACUUM でクリーンアップされることが期待されます。

ページを削除した際、削除前にダウンリンクを確認した進行中の検索が、そのページに降下する可能性があります。検索はそれが削除されていることを確認して無視しますが、そのようなことが起こり得る限り、そのページを再利用することはできません。進行中の検索を「待ち切る」ために、ページが削除されると、現在の次トランザクションカウンタ値がラベル付けされます。その XID が誰からも見えなくなるまで、ページはリサイクルされません。これは必要以上に保守的ですが、シンプルに保つことにします。
