PostgreSQL用Gin
==================

Ginはjfg://networks (http://www.jfg-networks.com/) のスポンサーシップにより開発されました。

GinはGeneralized Inverted Index（汎用転置インデックス）の略であり、飲み物ではなく、魔法のランプの精（genie）のように考えるべきものです。

「汎用（Generalized）」とは、インデックスが自身で加速する操作を認識していないことを意味します。その代わりに、特定のデータ型に対して定義されたカスタム戦略を用いて動作します（PostgreSQLドキュメントの「インデックスメソッドの戦略」を参照してください）。その意味で、GinはGiSTに似ており、比較ベースの操作があらかじめ定義されているbtreeインデックスとは異なります。

転置インデックスとは、(キー, ポスティングリスト) のペアの集合を格納するインデックス構造です。ここで「ポスティングリスト」とは、そのキーが出現するヒープ行の集合のことです。（テキストドキュメントには通常、多くのキーが含まれます。）Ginインデックスの主な目的は、PostgreSQLにおける高度にスケーラブルな全文検索のサポートです。

Ginインデックスは、キー値の上に構築されたB-treeインデックスで構成されます。各キーはインデックス付けされた項目（配列の要素、tsvectorの語彙素）の要素であり、リーフページの各タプルには、項目ポインタのB-tree（ポスティングツリー）へのポインタ、またはリストが十分に小さい場合は項目ポインタの単純なリスト（ポスティングリスト）が含まれます。

注意: キー（エントリ）ツリーには削除操作はありません。その理由は、我々の経験上、大規模なコーパスにおける異なる単語の集合は非常にゆっくりとしか変化しないためです。これにより、コードと並行性アルゴリズムが大幅に簡素化されます。

PostgreSQLのコアには、一次元配列（例: integer[]、text[]）に対するGinの組み込みサポートが含まれています。以下の操作が利用可能です。

  * contains: value_array @> query_array
  * overlaps: value_array && query_array
  * is contained by: value_array <@ query_array

概要
--------

=# create index txt_idx on aa using gin(a);

機能
--------

  * 並行性
  * 先行書き込みログ (WAL)。（クラッシュからの復旧。）
  * ユーザー定義の演算子クラス (opclass)。（スキームはGiSTに似ています。）
  * 最適化されたインデックス作成（maintenance_work_memを使用してメモリ内にポスティングを蓄積します。）
  * 演算子クラスによるテキスト検索サポート
  * GUC変数を用いた、返される結果セットに対するソフト上限:
    gin_fuzzy_search_limit

Ginファジーリミット
---------------

全文検索が非常に大きな結果セットを返す状況はよくあります。ディスクからタプルを読み取ってソートするには時間がかかる可能性があるため、これは本番環境では許容されません。（検索自体は非常に高速であることに注意してください。）

このようなクエリには通常、非常に頻出する語彙素が含まれているため、結果はあまり役に立ちません。このようなクエリの実行を容易にするため、GinにはGUC変数 'gin_fuzzy_search_limit' によって決定される、返されるセットのサイズに対する設定可能なソフト上限があります。これはデフォルトで0（制限なし）に設定されています。

ゼロ以外の検索制限が設定されている場合、返されるセットは全結果セットのランダムに選択されたサブセットになります。

「ソフト」とは、クエリやシステムの乱数生成器の品質に応じて、実際に返される結果の数が指定された制限と異なる可能性があることを意味します。

経験上、'gin_fuzzy_search_limit' の値は数千（例: 5000〜20000）が適切です。これは、この数よりも少ないタプルを返すクエリに対しては、'gin_fuzzy_search_limit' は何の影響も及ぼさないことを意味します。

インデックス構造
---------------

GINインデックスがインデックス付けする「項目」は、0個以上の「キー」を含む複合値です。例えば、項目が整数配列である場合、キーは個々の整数値となります。インデックスは実際には項目そのものではなく、キー値を格納および検索します。GIN演算子クラスのpg_opclassエントリにおいて、opcintypeは項目のデータ型であり、opckeytypeはキーのデータ型です。GINは、項目が多くのキーを含み、同じキー値が多くの異なる項目に出現するケースに最適化されています。

GINインデックスには、メタページ、キーエントリのB-tree、および場合によっては「ポスティングツリー」ページが含まれます。ポスティングツリーページは、キーエントリがB-treeページに収まらないほど多くのヒープタプルポインタを取得した際のオーバーフローを保持します。さらに、高速更新機能が有効な場合、まだメインのB-treeにマージされていない「保留中」のキーエントリを保持する「リストページ」が存在する可能性があります。検索を行う際、リストページは線形にスキャンされる必要があるため、保留中のエントリが多すぎる状態になる前にメインのB-treeにマージする必要があります。保留リストの利点は、数千のエントリの一括挿入が個別の挿入よりもはるかに高速になることです。（この利点は主に、同じキーが複数の新しいヒープタプルに出現する場合に、複数の検索/挿入を行う必要がないことから得られます。）

キーエントリは名目上、他のインデックスタイプで使用されるものと同じIndexTuple形式ですが、リーフキーエントリは通常複数のヒープタプルを参照するため、重要な違いがあります。（「通常」のインデックスタプルを構築してからそれを変更するGinFormTupleを参照してください。）知っておくべき点は以下の通りです。

* 単一列インデックスでは、キーのタプルにはキーデータのみが含まれますが、複数列インデックスでは、キーのタプルには (列番号, キーデータ) のペアが含まれ、列番号はint2として格納されます。これは、異なる列で異なるキーデータ型をサポートするために必要です。タプルのこの部分は、通常のルールに従ってindex_form_tupleによって構築されます。列番号（存在する場合）がNULLになることはありませんが、キーデータはNULLになる可能性があり、その場合は通常通りNULLビットマップが存在します。（インデックスタプルの常として、NULLビットマップのサイズはINDEX_MAX_KEYSに固定されています。）

* キーデータがNULLの場合（つまり、IndexTupleHasNulls()が真の場合）、名目上のインデックスデータの直後（つまり、オフセットIndexInfoFindDataOffsetまたはIndexInfoFindDataOffset + sizeof(int2)）に、NULLエントリの「カテゴリ」を示すバイトがあります。可能なカテゴリは以下の通りです。
	1 = インデックス可能な項目から抽出された通常のNULLキー値
	2 = ゼロキーのインデックス可能な項目のプレースホルダー
	3 = NULLのインデックス可能な項目のプレースホルダー
プレースホルダーのNULLエントリがインデックスに挿入されるのは、そうしなければ空またはNULLのインデックス可能な項目に対するインデックスエントリが全く存在しなくなり、完全なインデックススキャンが実行できず、様々なコーナーケースで誤った回答が返されることになるためです。NULLエントリの異なるカテゴリは、B-treeによって異なるキーとして扱われますが、同じカテゴリのNULLエントリに対するヒープ項目ポインタは、通常のキーエントリと同様に1つのインデックスエントリにマージされます。

* B-treeリーフレベルのキーエントリにおいて、次のSHORTALIGN境界には、インデックス可能な項目がこのキーを含むヒープタプルを指す、圧縮形式（「ポスティングリストの圧縮」セクションを参照）の項目ポインタのリストがあります。これは「ポスティングリスト」と呼ばれます。

リストが大きすぎてインデックスタプルがインデックスページに収まらない場合、ItemPointerは別のポスティングページに押し出され、キーエントリ自体には何も表示されなくなります。別のページは「ポスティングツリー」（下記参照）と呼ばれます。いずれの場合も、キーに関連付けられたItemPointerはソートされた順序で簡単に読み取ることができることに注意してください。これはスキャンアルゴリズムによって依存されています。

* リーフキーエントリのインデックスタプルヘッダーフィールドは、以下のように悪用されます。

1) ポスティングリストの場合:

* ItemPointerGetBlockNumber(&itup->t_tid) には、インデックスタプルの開始位置からポスティングリストまでのオフセットが含まれます。
  アクセス用マクロ: GinGetPostingOffset(itup) / GinSetPostingOffset(itup,n)

* ItemPointerGetOffsetNumber(&itup->t_tid) には、ポスティングリスト内の要素数（ヒープ項目ポインタの数）が含まれます。
  アクセス用マクロ: GinGetNPosting(itup) / GinSetNPosting(itup,n)

* IndexTupleHasNulls(itup) が真の場合、NULLカテゴリバイトは GinGetNullCategory(itup,gs) / GinSetNullCategory(itup,gs,c) でアクセス/設定できます。

* ポスティングリストは GinGetPosting(itup) でアクセスできます。

* GinItupIsCompressed(itup) が真の場合、ポスティングリストは圧縮形式で格納されます。それ以外の場合は単なるItemPointerの配列です。新しいタプルは常に圧縮形式で格納されますが、データベースが9.3以前のバージョンから移行された場合は、非圧縮の項目が存在する可能性があります。

2) ポスティングツリーの場合:

* ItemPointerGetBlockNumber(&itup->t_tid) には、ポスティングツリーのルートのインデックスブロック番号が含まれます。
  アクセス用マクロ: GinGetPostingTree(itup) / GinSetPostingTree(itup, blkno)

* ItemPointerGetOffsetNumber(&itup->t_tid) には、ポスティングリストの場合と区別するためのマジックナンバー GIN_TREE_POSTING が含まれます（これは、それほど多くのヒープ項目ポインタがインデックスページに収まるはずがないほど大きな値です）。この値は GinSetPostingTree マクロによって自動的に挿入されます。

* IndexTupleHasNulls(itup) が真の場合、NULLカテゴリバイトは GinGetNullCategory(itup,gs) / GinSetNullCategory(itup,gs,c) でアクセス/設定できます。

* ポスティングリストは存在せず、アクセスしてはなりません。

どちらのケースが適用されるかを判断するには、マクロ GinIsPostingTree(itup) を使用してください。

いずれの場合も、itup->t_info & INDEX_SIZE_MASK にはタプルの実際の合計サイズが含まれており、INDEX_VAR_MASK および INDEX_NULL_MASK ビットは index_form_tuple によって設定される通常の意味を持ちます。

B-treeの非リーフレベルのインデックスタプルには、上記のようなオプションの列番号、キーデータ、NULLカテゴリバイトが含まれます。これらにはポスティングリストは含まれません。ItemPointerGetBlockNumber(&itup->t_tid) は次の下位B-treeレベルへのダウンリンクであり、ItemPointerGetOffsetNumber(&itup->t_tid) は InvalidOffsetNumber です。ダウンリンクを取得/設定するには、アクセス用マクロ GinGetDownlink/GinSetDownlink を使用してください。

「保留リスト」ページに出現するインデックスエントリも少し異なる動作をします。オプションの列番号、キーデータ、NULLカテゴリバイトは他のGINインデックスエントリと同様です。ただし、保留中のエントリには常に1つのヒープ項目ポインタが関連付けられており、非GINインデックスと同様に t_tid ヘッダーフィールドに格納されます。ポスティングリストはありません。さらに、保留リストを検索するコードは、特定のヒープタプルのすべてのエントリが保留リスト内で連続して出現し、列番号とキーデータの合計でソートされていると想定しています。GIN_LIST_FULLROW ページフラグビットは、特定のヒープタプルのエントリが複数の保留リストページにまたがっているかどうかを示します。GIN_LIST_FULLROW が設定されている場合、そのページには1つ以上のヒープタプルのすべてのエントリが含まれています。GIN_LIST_FULLROW がクリアされている場合、そのページには1つのヒープタプルのエントリのみが含まれており、*かつ*それらがそのタプルのすべてのエントリではありません。（したがって、すべてのエントリが1つの保留リストページに収まらないヒープタプルは、たとえその結果としてタプルの前のページや最後のページのスペースの多くが無駄になったとしても、それらのページを占有する必要があります。）

GINは、nbtreeとは異なる方法でダウンリンクとピボットキーを内部ページタプルにパックします。Lehman & Yaoはこれを以下のように定義しています。

P_0, K_1, P_1, K_2, P_2, ... , K_n, P_n, K_{n+1}

ここで P_i はダウンリンク、K_i はキーです。K_i は P_{i-1} と P_i (0 <= i <= n) の間のキー空間を分割します。K_{n+1} はハイキーです。

内部ページタプルでは、キーとダウンリンクがグループ化されています。nbtreeはキーとダウンリンクを以下のようにタプルにパックします。

(K_{n+1}, None), (-Inf, P_0), (K_1, P_1), ... , (K_n, P_n)

ここでタプルは括弧内に示されています。つまり、ハイキーは別々に格納されます。P_i は K_i とグループ化されます。P_0 は -Inf キーとグループ化されます。

GINはキーとダウンリンクを異なる方法でタプルにパックします。

(P_0, K_1), (P_1, K_2), ... , (P_n, K_{n+1})

P_i は K_{i+1} とグループ化されます。-Inf キーは不要です。

K_{n+1} キーに関して、いくつか追加の注意点があります。
1) 最右ページのエントリツリーにおいて、P_n と結合されたキーは実際には重要ではありません。ハイキーは無限大であると想定されます。
2) ポスティングツリーにおいて、P_n と結合されたキーは常に重要ではありません。最右ページ以外のハイキーは別々に格納され、GinDataPageGetRightBound() を介してアクセスされます。

ポスティングツリー
------------

ポスティングリストが大きすぎてキーエントリ内にインラインで格納できない場合、ポスティングツリーが作成されます。ポスティングツリーはB-tree構造であり、ItemPointerがキーとして使用されます。内部ポスティングツリーページは、標準のPageHeaderと他のGINページと同じ「不透明（opaque）」な構造体を使用しますが、通常のインデックス・タプルは含みません。その代わり、ページの内容はPostingItem構造体の配列になっています。各PostingItemは、子ページのブロック番号と、その子ページの右境界（ItemPointerとして）で構成されます。ページの右境界は、ページヘッダーの直後、PostingItem配列の前に格納されます。

ポスティングツリーのリーフページも標準のPageHeaderと不透明な構造体を使用し、ページの右境界はページヘッダーの直後に格納されますが、ページの内容は複数の圧縮されたポスティングリストで構成されます。圧縮されたポスティングリストは、ページヘッダーとpd_lowerの間に、互いに連続して格納されます。pd_lowerとpd_upperの間の領域は未使用であり、これによりポスティングツリーのリーフページのフルページイメージにおいて、中央の未使用領域をスキップすることができます（XLogRecDataでbuffer_std = true）。

アイテムポインタは、1つの大きなリストではなく、複数の独立した圧縮ポスティングリスト（セグメントとも呼ばれます）に格納されます。これは、特定のアイテムポインタへのランダムアクセスを高速化するためです。圧縮リスト内のアイテムを見つけるにはリストの先頭から読み取る必要がありますが、アイテムが複数のリストに分割されていれば、探しているアイテムを含むリストまでスキップし、そのセグメントのみを読み取ることができます。また、更新時も影響を受けるセグメントのみを再エンコードすれば済みます。

ポスティングリストの圧縮
------------------------

ページにできるだけ多くのアイテムポインタを収めるため、ポスティングツリーのリーフページおよびエントリツリーのリーフタプル内にインラインで格納されるポスティングリストでは、軽量な圧縮形式を使用します。アイテムポインタがソートされた順序で格納されているという事実を利用します。各アイテムポインタのブロック番号とオフセット番号を個別に格納する代わりに、前のアイテムとの差分を格納します。それ自体では大きな効果はありませんが、いわゆるvarbyteエンコーディングを使用して圧縮することを可能にします。

Varbyteエンコーディングは整数をエンコードする手法で、小さな数値は少ないスペースで済み、大きな数値はより多くのスペースを必要とします。各整数は可変長のバイト数で表現されます。varbyteエンコーディングにおける各バイトの上位ビットは、次のバイトがまだこの数値の一部であるかどうかを決定します。したがって、単一のvarbyteエンコードされた数値を読み取るには、上位ビットがセットされていないバイトが見つかるまでバイトを読み取る必要があります。

エンコード時、アイテムポインタを形成するブロック番号とオフセット番号は単一の整数に結合されます。オフセット番号は下位11ビットに格納され（ginpostinglist.cのMaxHeapTuplesPerPageBitsを参照）、ブロック番号は上位ビットに格納されます。これには合計43ビットが必要ですが、これは最大6バイトに収まります。

圧縮されたポスティングリストは、GinPostingList構造体として受け渡され、ディスクに格納されます。リスト内の最初のアイテムは通常のItemPointerDataとして非圧縮で格納され、その後にリストのバイト単位の長さ、そしてパックされたアイテムが続きます。

同時実行制御
-----------

エントリツリーと各ポスティングツリーはBツリーであり、同じレベルの兄弟ページを右リンク（right-links）で接続しています。これは通常のBツリーインデックス（Lehman & Yaoによって考案）で使用されるのと同じ構造ですが、GINツリーでは逆方向のスキャンをサポートしていないため、左リンクは必要ありません。エントリツリーのリーフには専用のハイキーはなく、代わりに最大のリーフタプルがハイキーとして機能します。これは、エントリツリーからタプルが削除されることがないため機能します。

エントリツリーおよびポスティングツリーを操作するために使用されるアルゴリズムを以下に検討します。

### リーフページの特定

GIN Bツリーで読み取りを行うためにリーフページを検索する場合、ルートページからリーフまで、ダウンリンクを使用して一度に1つのページのピンと共有ロックを取得しながら降下します。つまり、次のページのピンと共有ロックを取得する前に、前のページのピンと共有ロックを解放します。

下の図は、リーフページを見つけた後のツリーの状態を示しています。小文字はツリーページを表します。'S'はページに対する共有ロックを表します。

               a
           /   |   \
       b       c       d
     / | \     | \     | \
   eS  f   g   h   i   j   k

### 右への移動

同時実行的なページ分割はキー空間を右に移動させるため、ダウンリンクを辿った後、実際に探しているキーを含むページは、到達したページの右側にある可能性があります。その場合、探しているページが見つかるまで右リンクを辿ります。

右へ移動する間、現在のページからピンと共有ロックを解放する前に、右側の兄弟ページのピンと共有ロックを取得します。このメカニズムは、削除されたページへ移動することを防ぐために設計されました。右リンクを指すロックを保持したまま右側の兄弟ページへ移動するため、他の誰かが同時に右リンクを更新したり、右側の兄弟ページを削除したりしないことが保証されます。

下の図は、右へ移動する際に一度にロックされた2つのページを示しています。

               a
           /   |   \
       b       c       d
     / | \     | \     | \
   eS  fS  g   h   i   j   k

### 挿入

挿入に適したリーフを見つける間も、ルートからリーフまで降下し、一度に1つのページを共有ロックします。しかし、挿入中はルートや内部ページのピンを解放しません。これは、親が同時実行的な分割によって変更されないと仮定すれば、ダウンリンク挿入のためのバッファハッシュテーブルへのルックアップをいくつか節約できる可能性があります。リーフに到達したら、そのページを排他モードで再ロックします。

下の図は、排他モードでロックされ、挿入の準備が整ったリーフページを示しています。'P'と'E'はそれぞれピンと排他ロックを表します。

               aP
           /   |   \
       b       cP      d
     / | \     | \     | \
   e   f   g   hE  i   j   k

挿入によってページ分割が発生した場合、左の子をアンロックする前に親が排他モードでロックされます。そのため、挿入アルゴリズムは子から開始して親と子両方のページを同時に排他ロックできます。

下の図は、リーフページの分割後のツリー状態を示しています。'q'は分割によって生成された新しいページです。親'c'にダウンリンクが挿入されようとしています。

                  aP
            /     |   \
       b          cE      d
     / | \      / | \     | \
   e   f   g  hE  q   i   j   k

### ページ削除

バキューム（Vacuum）は、エントリツリーからタプルやページを削除することはありません。右リンクによって論理的な順序でエントリツリーのリーフを走査し、ポスティングリストから削除可能なTIDを削除します。ポスティングツリーはエントリツリーのリーフからのリンクによって処理されます。これらは2段階でバキュームされます。第1段階では、リーフから削除可能なTIDが削除されます。第1段階で少なくとも1つの空のページが検出された場合、第2段階でginScanToDelete()が空のページを削除します。

ginScanToDelete()は、ツリー全体を深さ優先探索で走査します。ツリーのルートに対する完全なクリーンアップロックから開始します。このロックは、ページを削除している間、このツリーへのすべての同時挿入を防ぎます。ただし、ルートをロックする前にルートを通過した進行中の読み取り処理がまだ存在する可能性があります。

下の図は、ページ削除アルゴリズムがツリーの最も左のリーフまで走査した後のツリー状態を示しています。

               aE
           /   |   \
       bE      c       d
     / | \     | \     | \
   eE  f   g   h   i   j   k

削除アルゴリズムは、現在調査中のパスを構成するページの左側の兄弟に対する排他ロックを保持します。したがって、現在のページが削除される場合、ダウンリンクと右リンクの両方を削除するために必要なすべてのページはすでにロックされています。これにより、同時実行的な右への移動とデッドロックを引き起こす可能性のある、右から左へのページロック順序を回避します。

ページ削除と並行して行われる検索は、削除されるページへのポインタをすでに読み取っており、まさにそれを辿ろうとしている可能性があります。ページには、左側の兄弟の右リンク経由、または親のダウンリンク経由で到達できます。

バックエンドが右リンク経由で削除されたページに到達するのを防ぐため、右への移動アルゴリズムは、右側のページのロックが取得されるまで現在のページのロックを解放しません。

ダウンリンクはより複雑です。ツリーを降下する検索は、子をロックする前に親ページのロックを解放しなければなりません。そうしないと、子ページの同時分割とデッドロックする可能性があります（ページ分割は、子ページのロックを保持したまま親をロックするため）。そのため、削除されたページを即座に再利用することはできません。代わりに、このページを参照する可能性のあるすべてのトランザクションが終了するのを待つ必要があります。対応するプロセスは、ページが削除済みとしてマークされていることを確認し、それに応じて復旧する必要があります。

下の図は、ページ削除アルゴリズムがさらにツリーを走査した後のツリー状態を示しています。現在調査中のパスは'a-c-h'です。'c'と'h'の左側の兄弟である'b'と'g'も排他ロックされています。

               aE
           /   |   \
       bE      cE      d
     / | \     | \     | \
   e   f   gE  hE  i   j   k

次の図は、ページ'h'が削除された後のツリー状態を示しています。'deleted'フラグと、それを訪れる可能性のある最新のxidでマークされています。'c'から'h'へのダウンリンクも削除されています。

               aE
           /   |   \
       bE      cE      d
     / | \       \     | \
   e   f   gE  hD  iE  j   k

しかし、同時実行的な読み取り処理が、削除前に'c'から'h'へのダウンリンクを見ている可能性は依然としてあります。その場合、この読み取り処理は'h'から右へ移動し、削除されていないページが見つかるまで続けます。ページ'h'のXidマーキングは、そのようなすべての読み取り処理が終了するまで、このページが再利用されないことを保証します。次に調査されるリーフページは'i'です。'g'は'i'の左側の兄弟となるため、ロックされたままです。

次の図は、'i'と'c'が削除された後のツリー状態を示しています。内部ページ'c'はダウンリンクがないことが判明したため削除されました。調査中のパスは'a-d-j'です。ページ'b'と'g'は、'd'と'j'の左側の兄弟としてロックされています。

               aE
           /       \
       bE      cD      dE
     / | \             | \
   e   f   gE  hD  iD  jE  k

スタンバイ側でページ削除をリプレイする際、ページの左側の兄弟、ターゲットページ、およびその親がその順序でロックされます。この順序により、同時実行的な読み取りとのデッドロックがないことが保証されます。

述語ロック（Predicate Locking）
-----------------

GINは、シリアライザブル・スナップショット分離のための述語ロックをサポートしています。
述語ロックは、スキャンが値の範囲をスキャンしたことを表します。これらは物理的なページそのものに関心があるのではなく、論理的なキー値に関心があります。ページ上の述語ロックは、現在一致するタプルがあるかどうかに関わらず、そのページに属するキー範囲をカバーします。言い換えれば、インデックスページ上の述語ロックは、インデックス・タプル間の「ギャップ」をカバーします。誤検知を最小限に抑えるため、述語ロックは可能な限り細かいレベルで取得されます。

* Bツリーインデックスと同様に、すべての挿入はリーフレベルで行われるため、リーフページのみをロックすれば十分です。

* 等価検索（部分一致検索ではない）において、キーエントリがポスティングツリーを持っている場合、そのキーエントリのみに対するロックを表すためにポスティングツリーのルートページをロックします。それ以外の場合は、エントリツリーページをロックします。一致が見つからない場合も、エントリが存在していたはずの「ギャップ」をロックするためにエントリツリーページをロックします。

* 部分一致検索では、値の間の「ギャップ」を表すために、ポスティングツリーのルートに対するロックに加えて、スキャンするすべてのエントリリーフページをロックします。

* エントリリーフページとポスティングツリーのルートに対するロックに加え、すべてのスキャンはメタページに対するロックを取得します。これは、高速更新保留リスト（fast update pending list）への挿入と連動するためです。保留リストへの挿入はツリー内のどこにでも属する可能性があるため、メタページ上のロックがそれを表します。

高速更新保留リストの連動は、fastupdate=onの場合、実質的に常にフルインデックスロックを取得することを意味し、多くの誤検知が発生する可能性があります。

互換性
-------------

TIDの圧縮は9.4で導入されました。一部のGINインデックスは、9.3以前のバージョンからのpg_upgradeにより、非圧縮形式のまま残っている可能性があります。互換性のために、古い非圧縮形式もサポートされています。これを処理するために以下のルールが使用されます。* GIN_ITUP_COMPRESSED フラグは、ポスティングリストを含むインデックスタプルをマークします。
このフラグは、ItemPointerGetBlockNumber(&itup->t_tid) の上位ビットに格納されます。
このフラグを確認するには GinItupIsCompressed(itup) を使用してください。

* 新しい形式のポスティングツリーページは、GIN_COMPRESSED フラグでマークされます。
このフラグの確認および設定には、マクロ GinPageIsCompressed(page) および GinPageSetCompressed(page) が使用されます。

* すべてのスキャン操作はポスティングリストの形式を確認し、対応するコードを使用してその内容を読み取ります。

* 非圧縮ポスティングリストを含むインデックスタプルを更新する場合、それは圧縮リストを含む新しいインデックスタプルに置き換えられます。

* 非圧縮ポスティングツリーのリーフページを更新する場合、それは圧縮されます。

* VACUUM が非圧縮ポスティングリスト内に不要な TID を見つけた場合、それらは圧縮ポスティングリストに変換されます。これは、圧縮ポスティングリストが非圧縮リストによって占有されていたスペースに収まることを前提としています。言い換えれば、不要なアイテムが削除された圧縮バージョンのページは、古い非圧縮バージョンよりも少ないスペースしか取らないと想定しています。

制限事項
-----------

  * Gin は scan->kill_prior_tuple および scan->ignore_killed_tuples を使用しません。
  * Gin は、等価マッチング、または「部分一致」機能を使用した単純な範囲マッチングによってのみエントリを検索します。
