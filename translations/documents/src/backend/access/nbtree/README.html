Btree インデックス
==================

このディレクトリには、Lehman と Yao による高並行性 B-tree 管理アルゴリズムの正確な実装が含まれています (P. Lehman and S. Yao, Efficient Locking for Concurrent Operations on B-Trees, ACM Transactions on Database Systems, Vol 6, No. 4, December 1981, pp 650-670)。また、Lanin と Shasha が記述した削除ロジックの簡略版も使用しています (V. Lanin and D. Shasha, A Symmetric Concurrent B-Tree Algorithm, Proceedings of 1986 Fall Joint Computer Conference, pp 380-389)。

基本的な Lehman & Yao アルゴリズム
--------------------------------

従来の B-tree と比較して、L&Y は各ページに右側の兄弟ページへの「右リンクポインタ」を追加します。また、各ページに「ハイキー (high key)」を追加します。これは、そのページに許可されるキーの上限です。これら 2 つの追加により、並行して発生したページ分割を検出することが可能になり、(読み取り中に単一のページが変更されないように保護することを除き) 読み取りロックを保持せずにツリーを検索できるようになります。

検索が子ページへのダウンリンクをたどる際、そのページのハイキーと検索キーを比較します。検索キーがハイキーよりも大きい場合、そのページは並行して分割されたはずであり、探しているキー範囲を含む新しいページを見つけるために右リンクをたどる必要があります。ページが複数回分割されている場合は、これを繰り返す必要があるかもしれません。

Lehman と Yao は、内部ページにおいてタプルやレコードではなく、「セパレータ」キーとダウンリンクを交互に配置することについて述べています。私たちは、ヒープタプルを指さず、ツリーのナビゲーションのみに使用されるタプルを指すために「ピボット」タプルという用語を使用します。非リーフページのすべてのタプルと、リーフページのハイキーはピボットタプルです。ピボットタプルは各ページにどのキー空間が属するかを表すためにのみ使用されるため、以前に削除され VACUUM によって無効化された非ピボットタプルからコピーされた属性値を持つことができます。ピボットタプルは、「セパレータ」キーとダウンリンク、セパレータキーのみ (つまりダウンリンク値は暗黙的に未定義)、またはダウンリンクのみ (つまりすべての属性が切り詰められている) を含むことができます。

すべての btree キーが一意であるという要件は、ヒープ TID をタイブレーカー属性として扱うことで満たされます。論理的な重複はヒープ TID の順序でソートされます。これは、Lehman と Yao が、サブツリー S のキー範囲が Ki < v <= Ki+1 (Ki と Ki+1 は親ページ内の隣接するキー) で記述されることを要求しているため必要です (Ki は v より「厳密に」小さくなければならず、これは確実に一意なキーを持つことで保証されます)。キーはそのレベル内で常に一意ですが、リーフページのハイキーは例外であり、ページ内の最後のアイテムと完全に等しくなる可能性があります。

Postgres のサフィックス切り詰め実装は、Lehman と Yao の不変条件が維持されることを確認しなければならず、ピボットタプル内の欠落/切り詰められた属性が「マイナス無限大」というセンチネル値を持つことを表します。Lehman & Yao の不変条件が実際の例でどのように機能するかが不明な場合は、後のサフィックス切り詰めに関するセクションが役立ちます。

Lehman & Yao アルゴリズムとの違い
-----------------------------------------

L&Y アルゴリズムを Postgres に組み込むために、以下の変更を行いました。

Lehman と Yao は読み取りロックを必要としませんが、ツリーページのメモリ内コピーが共有されていないと仮定しています。Postgres はメモリ内バッファをバックエンド間で共有します。その結果、レコードを検査している間に変更されないことを保証するために、btree ページに対してページレベルの読み取りロックを行います。これにより並行性は低下しますが、正しい動作が保証されます。

私たちは、挿入、削除、単純なルックアップだけでなく、インデックスの順序付き「スキャン」の概念もサポートしています。前方方向のスキャンは問題ありません。L&Y が要求する右兄弟ポインタを使用するだけです。(したがって、スキャンの開始点までツリーを降下した後は、スキャンはリーフページのみを参照し、それより高いツリーレベルを参照することはありません。) 後方方向のスキャンをサポートするために、「右兄弟」と同様に「左兄弟」リンクも格納します。(これには L&Y 分割アルゴリズムに余分なステップが追加されます。分割中のページに対する書き込みロックを保持している間、そのページの左リンクを更新するために、以前の右兄弟もロックします。そのページの書き込み者は私たちのページに対する書き込みロックの取得に関心を持てないため、これは安全です。) 後方スキャンにはもう 1 つの複雑さがあります。左リンクをたどった後、左兄弟ページが読み取る前に分割された可能性を考慮しなければなりません。そのため、右リンクが元のページと一致するページを見つけるまで右に移動する必要があります。(実際にはそれよりもさらに困難です。以下のページ削除の議論を参照してください。)

ページ読み取りロックは、スキャンがページを検査している間のみ保持されます。ロック/アンロックのトラフィックを最小限に抑えるため、インデックススキャンは常にリーフページを検索して一致するすべてのアイテムを一度に特定し、そのヒープタプル ID をバックエンドローカルストレージにコピーします。その後、インデックス内のページロックを保持せずにヒープタプル ID が処理されます。並行削除から保護するために、状況によってはリーフページへのピンを保持し続けます (下記参照)。この状態では、スキャンはピン留めされたページの前後など、「ページ間」で実質的に停止しています。アイテムは既存のページ境界を越えて移動することはないため、これは並行挿入やページ分割が存在する場合でも安全です。したがって、スキャンが見逃すべきアイテムを見逃したり、誤って同じアイテムを 2 回返したりすることはありません。スキャンは、スキャン時のページの右リンクを記憶しておく必要があります。それが右に移動すべきページだからです。現在の右リンクに右へ移動すると、ページ分割によって移動されたアイテムを再スキャンしてしまいます。また、左リンクも記憶し、スキャンが後方に移動する際にそれをたどります (ただし、これには左兄弟の並行分割を考慮するための追加の処理が必要です。以下の詳細な左移動アルゴリズムを参照してください)。

ほとんどの場合、移動先のページのピンとロックを取得しようとする前に、現在のページのロックとピンを解放します。一部の場所では、現在のページを解放する前に次のページをロックする必要があります。これは右または上に移動する場合は安全ですが、左または下に移動する場合は安全ではありません (そうしないとデッドロックの可能性が生じます)。

Lehman と Yao は、ルートページがいっぱいになり分割が必要になった場合に何が起こるかについて言及していません。私たちの実装では、ルートを他のページと同様に分割し、結果として得られる両方のページ (これらはツリーの次のレベルで兄弟になります) へのポインタを保持する新しいルートページを作成します。新しいルートページは、メタデータページのルートポインタを変更することでインストールされます (下記参照)。ルートは他の方法で特別に扱われないため、これは機能します。特に、リンクが設定されている場合、検索はリンクポインタを使用して右に移動します。したがって、検索はメタデータページが更新される前に読み取ったとしても、右兄弟に移動されたデータを見つけることができます。これは、非ルートページの分割を安全にするのと同じ論理です。ロックの考慮事項も同様です。

挿入者がツリーを再帰的に上昇し、下のレベルに挿入されたページへのリンクを挿入するために内部ページを分割する場合、降下を開始した時点のルート (より正確には、メタデータページを読み取った時点のルート) よりも上のレベルのページにアクセスする必要がある可能性があります。この場合、降下中に作成したスタックは正しいページを見つけるのに役立ちません。これが発生した場合、リンクを挿入する必要があるレベルの 1 つ上のレベルに到達するまでツリーを再降下し、必要に応じて右に移動することで正しい場所を見つけます。(通常、これにはメタデータページと新しいルートの 2 回のフェッチしかかかりませんが、原則としてルートを見てからルート分割が複数回発生している可能性があります。各ページに格納されているレベル番号によって正しいツリーレベルを特定できます。この状況は非常にまれであるため、より効率的な解決策は必要ありません。)

Lehman と Yao は、ツリーの上昇中に子ページのダウンリンクを再配置する際、右に移動する過程でロックを結合/連鎖させる必要があります。これは、Lehman と Yao が同時に 3 つのロック (子、元の親、元の親の右兄弟に対するロック) を保持しなければならない唯一のポイントです。ただし、同じレベルのページに対して内部ページのロックを結合する必要はありません。私たちは、子ページのブロック番号を 1 つ上のレベルのピボットタプルからのダウンリンクと照合しますが、Lehman と Yao は最初の降下中にたどったダウンリンクに関連付けられたセパレータキーで照合します。再配置しなければならないセパレータキーが元の親のハイキーになるケースに対処する必要はないため、右兄弟のロックを取得する前に元の親ページのロックを解放できます。Lanin と Shasha もここでロックを結合しませんが、上昇中にレベル間でロックを結合することもありません。彼らは競合を避けるために「待機して再試行」することを厭いません。彼らのアルゴリズムは楽観的であり、これは「挿入は上昇中に一度に 1 つ以上の書き込みロックを保持しない」ことを意味します。私たちは、ツリーを上昇する際に親と子のロックを保守的に結合するという Lehman と Yao のアプローチをほぼ踏襲しています。その方がはるかに単純だからです。

Lehman と Yao は固定サイズのキーを想定していますが、私たちは可変サイズのキーを扱わなければなりません。そのため、ページあたりのキーの最大数は固定されていません。収まるだけ詰め込みます。ページを分割する際、(サフィックス切り詰めも考慮されますが) ページに割り当てられるアイテム数ではなく、バイト数を均等にするように努めます。この計算には受信アイテムを含める必要があることに注意してください。そうしないと、受信アイテムが移動先の分割ページに収まらない可能性があるためです！

VACUUM 中のインデックスタプルの削除
-----------------------------------

リーフアイテムを削除する前に、ターゲットページに対する完全なクリーンアップロックを取得し、削除開始時に他のバックエンドがそのページにピンを保持していないようにします。これは btree インデックス操作自体の正確性の観点からは必須ではありません。前述の通り、インデックススキャンは論理的に「ページ間」で停止するため、場所を見失うことはありません。これを行う理由は、VACUUM と、ヒープを訪問する際の並行 TID 再利用に対処する準備ができていないインデックススキャンとの間にインターロックを提供するためです。ヒープ内の指し示されたアイテムを LP_UNUSED とマークできるのは VACUUM だけであり、これは btbulkdelete が戻った「後」にのみ発生するため、インデックススキャンが (リーフページから読み取る際に使用される) ピンをヒープの訪問が終わる「後」まで保持することで、並行 TID 再利用を防ぐことができます。インデックススキャンがリーフページの処理を完全に終了するまで、VACUUM は競合するクリーンアップロックを取得できません。

このアプローチはかなり粗いため、可能な限り避けています。実際には、ほとんどのインデックススキャンはピンを保持しないため、VACUUM をブロックしません。これらのインデックススキャンは TID 再利用を直接処理しなければなりませんが、これはより複雑であり、常に可能とは限りません。並行 TID 再利用を安全にする方法に関する後のセクションを参照してください。

日和見的なインデックスタプル削除は、排他ロックのみを保持しながら、ほぼ同じページレベルの変更を実行します。後で TID 再利用が発生する問題はないため、これは安全です。VACUUM だけが TID を再利用可能にできるからです。以下の単純な削除およびボトムアップ削除も参照してください。ピンは常に保持されているわけではなく、ピンを保持している最中であってもページが分割される可能性があるため、インデックススキャンが、ピンを保持しているページ上にはもはや存在せず、そのページの右側のどこかにあるアイテムを返す可能性があります。このシナリオにおいて、VACUUMがTIDを時期尚早に再利用可能にすることを防ぐため、btbulkdeleteはインデックス内のすべてのリーフページ（削除可能なタプルを含まないページであっても）に対してクリーンアップロックを取得する必要があります。なお、この要件は、btbulkdeleteが特定の順序でページを訪問しなければならないことを意味するものではありません。

VACUUMの線形スキャン、同時実行されるページ分割
--------------------------------------------

VACUUMは、削除可能なTIDを検索するために線形スキャンを実行し、その過程で空のページの削除を検討することでインデックスにアクセスします。これは論理的/キー空間順ではなく、物理的/ブロック順で行われます。この処理の難しい点は、同時実行されるページ分割が存在する場合に、削除可能なタプルを見逃さないようにすることです。ページ分割によって、シーケンシャルスキャンでまだ通過していないページから、すでに通過済みの番号の小さいページへタプルが簡単に移動してしまう可能性があるためです。

これを実装するために、現在のbtbulkdeleteサイクルが開始されてからページが分割されたかどうかを判断可能にする「バキュームサイクルID」メカニズムを提供します。btbulkdeleteが開始後に分割されたページを見つけ、かつ、より小さいページ番号を指す右リンクを持っている場合、一時的にシーケンシャルスキャンを中断し、代わりにそのページを訪問します。btbulkdeleteが開始されてから分割されていないページ、または外部のシーケンシャルスキャンの位置よりも上位のページに到達するまで、右リンクをたどり続け、デッドタプルをバキュームしなければなりません。その後、シーケンシャルスキャンを再開できます。これにより、すべてのタプルが確実に訪問されます。一部のタプルが2回訪問される可能性がありますが、それは不正確なインデックスのタプルカウントよりも悪い影響はありません（そもそも同時実行アクティビティがある場合、正確なカウントを保証することはできません）。なお、このテストは、偽陰性が決して発生しない限り、最近分割されたかどうかのテストにわずかな確率で偽陽性があっても機能します。これにより、各インデックスページに格納された小さなカウンタ値を使用してテストを実装することが可能になります。

VACUUM中のページ全体の削除
-----------------------------------

btreeからページ全体を削除するのは、アイテムが完全に空になった場合のみです。（部分的に埋まっているページをマージすればスペースの再利用効率は向上しますが、既存のデータアイテムを左右に移動させることは、逆方向に移動するスキャンがアイテムを見逃す可能性があるため、現実的ではないと思われます。）また、ツリーレベルの最右ページは決して削除しません（この制限により、後述するようにトラバーサルアルゴリズムが簡素化されます）。ページ削除は常に空のリーフページから始まります。内部ページは、サブツリー全体を削除する際の一部としてのみ削除可能です。これは常に、内部ページの「チェーン」と単一のリーフページで構成される「細い」サブツリーです。サブツリーの各レベルに1つのページがあり、各レベル/ページは同じキー空間をカバーしています。

リーフページの削除は2段階のプロセスです。第1段階では、ページを親から切り離し、「半死（half-dead）」状態としてマークします。親ページは、挿入分割時に親を見つけるために使用されるのと同じタイプの検索を使用して見つける必要があります。ターゲットページと親ページをロックし、ターゲットのダウンリンクを右側の兄弟を指すように変更し、古いダウンリンクを削除します。これにより、ターゲットページのキー空間は実質的にその右側の兄弟に属することになります。（左右の兄弟ページは、もしあったとしても「ハイキー（high key）」を変更する必要はないため、ハイキーを置き換えるのに十分なスペースがないという問題は発生しません。）同時に、ターゲットページを半死状態としてマークします。これにより、その後の検索はすべてそのページを無視して右（または逆方向スキャンでは左）に移動するようになります。これにより、ツリーはページ分割時と同様の状態になります。つまり、ページを指すダウンリンクはなくなりますが、兄弟とのリンクは維持されたままです。

（注：LaninとShashaはキー空間を左に移動させることを好みますが、その主張は左リンクがないことに依存しています。しかし、我々には左リンクがあるため、キー空間を右に移動させることでアルゴリズムを簡素化します。これが可能なのは、ページ分割中にツリーを昇順でたどる際、LehmanやYao/LaninやShashaとは異なり、セパレータキーでマッチングを行わないためです。挿入者が最初にツリーを降下した際に遭遇したセパレータキーと一致しないピボットタプル内でダウンリンクが再発見されたとしても問題はありません。）

親レベルでの整合性を維持するため、右側の兄弟が同じ親の子でない限り、ページのキー空間を右側の兄弟にマージすることはできません。そうでない場合、親のキー空間の割り当ても変更されてしまい、その親、さらにはツリー全体にわたって境界キーの更新が必要になる可能性があります。これをアトミックに実行することは不可能なため、このケースは禁止しています。つまり、親ノードの最右の子は、それが唯一の残りの子でない限り削除できません。唯一の子である場合は、親も削除します（下記参照）。

第2段階では、半死状態のリーフページを兄弟から切り離します。まず、ターゲットの左側の兄弟（もしあれば）、ターゲットページ自身、そしてその右側の兄弟（必ず存在します）の順にロックします。次に、兄弟のサイドリンクを更新し、ターゲットページを削除済みとしてマークします。

親ページの最後の子を削除しようとする場合、状況は少し複雑になります。第1段階では、リーフページの直近の親はそのままにしておき、代わりに祖父母から親ページへのダウンリンクを削除します。祖父母の最後の子でもある場合は、複数の子を持つ親が見つかるまで再帰的に遡り、そのページのダウンリンクを削除します。リーフページは半死状態としてマークされ、ダウンリンクが削除されたページのブロック番号が半死状態のリーフページに保存されます。これにより、それぞれ1つのダウンリンクを持つ内部ページのチェーンが残り、半死状態のリーフページへとつながり、チェーンの最上位ページを指すダウンリンクはなくなります。

チェーン内の最上位の親を見つけるために再帰的に遡る間、リーフページはロックしたままにしますが、リーフと最上位の親の間にある中間ページのロックを保持する必要はありません。上位ツリーレベルへの挿入は子ページの分割の結果としてのみ発生し、リーフをロックしている間はそれが発生することはないためです。チェーン内の内部ページは、リーフページが半死状態としてマークされ分割されることがないため、その後新しい子を獲得することもありません。

削除対象のサブツリー/チェーンの最上位へのダウンリンクを削除することで、すべての中間レベルのキー空間も1つのアトミックな操作で右側の兄弟に効果的に転送されます。同時実行中の検索が中間ページを訪問する可能性はありますが、リーフレベルの半死状態のページに到達すると右に移動します。特に、半死状態のリーフページの右側の兄弟は「いとこ」ページでなければならないため（半死状態のチェーンがリーフページの祖父母から始まる場合は「またいとこ」ページなど）、検索は半死状態のリーフページの右側のサブツリーへ移動します。

第2段階では、チェーン内の最上位ページが兄弟から切り離され、半死状態のリーフページがチェーン内の次のページを指すように更新されます。チェーン内に内部ページがなくなるまでこれが繰り返されます。最後に、半死状態のリーフページ自身が兄弟から切り離されます。

削除されたページは、他のプロセスがそれを参照しようとして待機している可能性があるため（例：親を離れたばかりの検索プロセスや、兄弟から右または左に移動しているスキャン）、すぐには再利用できません。これらのプロセスは、削除操作後もしばらくの間、削除されたページを観測できなければなりません。少なくともそこから復旧できるようにするためです（同時実行されるページ分割と同様に、右に移動することで復旧します）。検索者は、同時実行されるページ再利用について心配する必要はありません。

VACUUMが削除されたページを再利用可能にするタイミングと方法については、以下の「FSMへの削除済みページの配置」セクションを参照してください。

ページ削除と逆方向スキャン
---------------------------------

逆方向スキャンで左に移動するのは複雑です。左側の兄弟がちょうど分割された可能性（左側の兄弟から派生した最右ページを見つける必要があることを意味します）と、直前にいたページが削除され、兄弟チェーンに存在しなくなった可能性を考慮しなければならないためです。そのため、左移動アルゴリズムは以下のようになります。

0. 現在のページを「元のページ」として記憶する。
1. 元のページの左リンクをたどる（これがゼロなら終了）。
2. 現在のページが有効で、その右リンクが「元のページ」と一致すれば終了。
3. そうでなければ、右リンクが「元のページ」と一致する有効なページを探して、右に1回以上移動する。見つかれば終了。（原則としてインデックスの右端までスキャンすることも可能ですが、実際には数回試行して見つからなければ諦めるのが良さそうです。移動中に元のページの兄弟が何度も分割される可能性は低いためです。数回試しても一致するリンクが見つからない場合、元のページは削除されている可能性が高いです。）
4. 「元のページ」に戻る。まだ有効であればステップ1に戻る（削除されたという推測が間違っていたため、現在の左リンクから再開する）。削除されていれば、非削除ページが見つかるまで右に移動し（最右ページは決して削除されないため、必ず1つは存在するはずです）、それを新しい「元のページ」としてマークしてステップ1に戻る。

このアルゴリズムは正しいです。ステップ4で見つかった有効なページは、開始したページと同じ左側のキー空間境界を持つためです。したがって、最終的に終了する際には、開始した場所の左側の境界と一致する右側のキー空間境界を持つページにいることになり、アイテムを見逃したり再スキャンしたりしないことが保証されます。

ページ削除とツリーの高さ
-----------------------------

どのレベルの最右ページも削除しない（特にルートは決して削除しない）ため、ツリーの高さが減少することはありません。大規模な削除の後、ルートの下に複数の単一ページレベルを持つ「細い」ツリーになるシナリオがあり得ます。この場合でも操作は正しく行われますが、単一ページレベルを降下するサイクルが無駄になります。これに対処するため、LaninとShashaのアイデアを使用します。最も低い単一ページレベルである「高速ルート（fast root）」レベルを追跡します。メタデータページは、真のルートだけでなく、このレベルへのポインタも保持します。すべての通常の操作は、真のルートではなく高速ルートで検索を開始します。レベル上で単独のページを分割したり、レベル上の最後から2番目のページを削除したりする場合（どちらのケースも容易に検出可能）、高速ルートポインタが適切に調整されるようにする必要があります。分割の場合は親レベルへの挿入のためのアトミック更新の一部として、削除の場合は削除のためのアトミック更新の一部としてこれを行います（いずれにせよ、デッドロックのリスクを避けるため、メタページは更新時に最後にロックされるページでなければなりません）。これにより、2つの操作が同時に実行されている場合の競合状態を回避できます。

FSMへの削除済みページの配置
--------------------------------

ページの再利用はページ削除とは切り離されています。削除されたページは、それを参照している可能性のあるスキャンや検索がなくなった場合にのみ、再利用のためにFSMに配置できます。それまでは、同時実行される検索が削除を検出し、そこから復旧できるようにするための「墓石（tombstone）」として、兄弟リンクを乱さずにそのままにしておく必要があります（検索者にとって、これは同時実行されるページ分割と似ています）。この設計は、LaninとShashaが「ドレイン技術（the drain technique）」と呼ぶものの実装です。この手法は、ページ削除の時点でアクティブなスナップショットおよび登録済みスナップショットがすべてなくなるまで待機することで実装します。これは非常に強力な制約ですが、Postgres内ではシンプルに実装できます。完全にデッドとマークされた削除済みページには、次のトランザクションカウンタ値がラベル付けされます。VACUUMは、格納されたXIDが「誰からも可視」であることが保証された時点で、再利用のためにページを回収できます。副作用として、次のトランザクションがXIDを割り当てるまで、その時点までに取得されたスナップショットを待機します。また、スナップショットを持たない実行中のXIDも待機します。

PostgreSQL 14より前では、VACUUMは線形スキャン中に遭遇した「古い」削除済みページ（以前のVACUUM操作によって削除されたページ）のみをFSMに配置していました。新しく削除されたページは、それが「常に」安全ではないと見なされていたため、FSMに配置されることはありませんでした。しかし、実際にはその想定は不必要に悲観的でした。新しく削除されたページがFSMに配置しても安全な状態になるまで、それほど時間はかからないことが多いためです。削除されたページがいつFSMへの再利用として安全になるかを予測する真に原則的な方法はありません。ほぼ即座に安全になることもあれば（現在のVACUUMが完了するよりずっと前）、次のVACUUMが行われる時点でも安全ではないこともあります。再利用の安全性は、物理データ構造の一貫性（あるいは少なくとも見かけ上の一貫性）を維持できるかどうかの問題に過ぎません。VACUUMを実行しているバックエンド内の状態は、これとは無関係です。

PostgreSQL 14では、ページ削除が行われたフルインデックススキャンの終了時に、新しく削除されたページを再利用できるかどうかをVACUUMが検討する機能が追加されました。その時点で安全かどうかを確認するのは便利です。これには、VACUUMが新しく削除されたページに関するわずかな管理情報を保持する必要がありますが、コストは非常に低いです。このためにメモリ内の状態を使用することで、後で新しく削除されたページを再訪する必要がなくなります。ローカルの管理状態にあるsafexid値を使用して、遅延した方法で再利用の安全性を判断できるためです。

ページ削除操作の後にFSMの追加的な間接参照が必要となるのは、LehmanとYaoの設計によるインデックススキャンの非常に寛容なルールの自然な結果です。一般に、インデックススキャンはツリーを降下する際に、どのページに対してもロックやピンを保持する必要はありません（通常「レベル間」のインターロックと見なされるものは保持されません）。同時に、インデックススキャンは、同時実行の再利用（同時実行の削除と混同しないこと）によって、実際には無関係なページに到達することが許されてはなりません。クエリに対して誤った回答が返されるためです。再利用を遅延させる必要のない、より単純なページ削除アプローチも可能ですが、LehmanとYaoの設計と互換性があるものは見当たりません。

すでに削除されたページをFSMに配置して必要に応じて再利用しても、実際にはページの状態は変わりません。ページは、その後FSMから再利用のために取り出されたときに変更されます。削除されたページの内容は、分割操作によって上書きされます（新しい右側の兄弟ページになります）。

同時実行TID再利用の安全化
------------------------------------

VACUUM中のインデックスタプル削除に関する前述のセクションで説明したように、個々のインデックススキャンが同時実行のTID再利用を回避できるようにするロックプロトコルを実装しています。ただし、インデックススキャンは安全な場合はいつでも（ヒープを訪問する際に）リーフページのピンを解放します。ピンを早期に解放することは、VACUUMによる進行のブロックを回避できるため有用です。これは、アイドル状態のカーソルが比較的長時間停止することがあるため、カーソルによって使用されるインデックススキャンにおいて特に重要です。極端な場合、クライアントアプリケーションがアイドル状態のカーソルを数時間、あるいは数日間保持し続けることがあります。それほど長くVACUUMをブロックすることは壊滅的な結果を招く可能性があります。

バッファピンを保持しないインデックススキャンは、代わりにMVCCスナップショットを保持することで保護されます。このより限定的なインターロックはクエリへの誤った回答を防ぎますが、TIDの再利用そのものを防ぐわけではありません（ヒープへのアクセス中にリーフページのピンを保持することだけがそれを保証します）。

インデックスのみのスキャン（Index-only scans）は、参照されているTIDが再利用可能になることを許容できないため、バッファピンを解放することはできません。インデックスのみのスキャンは通常、可視性マップ（ヒープ本体ではなく）のみを参照するため、（最初はすべてに対してデッドなヒープ項目を指していたTIDについて）古いTID参照がVACUUMによってヒープ内で同時にLP_UNUSEDとマークされたことに確実には気づきません。これにより、VACUUMがその直後に可視性マップ内のヒープページ全体をall-visibleに設定することが容易に可能になってしまいます。MVCCスナップショットが通常のインデックススキャン中の問題を回避するのに十分なのは、それらがヒープ本体から詳細な可視性情報にアクセスする必要があるためです。通常のインデックススキャンは、ヒープページが一般的にall-visibleと見なされている場合でも、ヒープ内のLP_UNUSED項目（再利用可能だがまだ再利用されていない項目）を「不可視」として認識します。

kill_prior_tuple最適化（以下の単純削除で詳述）によるインデックスタプルのLP_DEAD設定も、リーフページのピンを解放するインデックススキャンにとってはより複雑です。同時実行のTID再利用の結果として、同じTIDを共有しているという理由だけで、既知のデッドなインデックスタプルに見える新しいインデックスタプルをLP_DEADとマークしないように注意しなければなりません。他のセッションが、同じリーフページ上に、同じ元のTIDを持つ、無関係な新しいインデックスタプルを挿入した可能性はわずかながらあります。この新しい無関係なインデックスタプルをLP_DEADに設定するのは完全に誤りです。

このkill_prior_tupleの競合状態は、影響を受けるインデックススキャンが、リーフページへの変更はすべて、バッファピンが保持されていない中間期間にbtbulkdeleteによって行われたと保守的に想定することで処理します。これは、ページのLSNが変更された場合に、リーフページ上のLP_DEADビットを一切設定しないことで実装されています。（これはログに記録されないインデックスでは機能しないため、現時点ではその最適化を適用していません。）

インデックス挿入の高速パス
----------------------------

インデックスキーの値が増加していく一般的な挿入ケースに対して、このバックエンドが最後に値を挿入したページが最も右側のリーフページであった場合、そのページをキャッシュすることで最適化を行います。次の挿入時には、キャッシュされたページが依然として最も右側のリーフページであり、現在の値を保持する正しい場所であるかを素早く確認できます。このような一般的なケースでは、ツリーを降下するコストを回避できます。

この最適化は、無視できないリーフの最右ページは1つしか存在し得ないという前提で機能するため、「誰からも可視」スタイルのインターロックさえ必要ありません。B-Tree内には常にそのようなページは1つしか存在し得ないため、ヒントが無効化されたことを検知できないという事態は起こり得ません。バックエンドがキャッシュしたページが無効化されたと検知されることなくページが削除・再利用される可能性はありますが、それは、かつて最右リーフページであったブロックが再び最右リーフページとして再利用された場合に限られます。

単純削除
----------------

プロセスがヒープタプルを訪問し、それがデッドで削除可能（つまり、そのプロセスだけでなく、すべてのオープンなトランザクションに対してデッド）であると判断した場合、インデックスに戻って対応するインデックスエントリを「既知のデッド」とマークし、後続のインデックススキャンがそのヒープタプルの訪問をスキップできるようにします。「既知のデッド」のマーク付けは、インデックス項目のlp_flags状態をLP_DEADに設定することで機能します。これは現在、ビットマップスキャンではなく通常のインデックススキャンでのみ行われます。通常のインデックススキャンのみがヒープとインデックスを「同期して」訪問するため、ビットマップスキャンでこれを行う便利な方法がないためです。また、LP_DEADビットは、挿入時の競合についてユニークインデックスを確認する際にも頻繁に設定されることに注意してください（これはリーフページに対する排他ロックを保持している間に行われるため、より単純です）。

インデックスタプルがLP_DEADとマークされると、実際にインデックスから即座に削除できます。インデックススキャンはページ「間」でしか停止しないため、そのような削除によってスキャンが場所を見失うことはありません。LP_DEADの設定は共有ロックのみで行えるようにしているため（ヒープタプルのヒントビットのようなものです）、物理的なタプルの削除には排他ロックが必要となるため、手順を分けています。また、各削除操作のWALレコードに対してsnapshotConflictHorizonを生成する必要があり、削除が実際に行われる際にtableamとの追加の調整が必要になります。（snapshotConflictHorizonの値は、スタンバイによるレコードのREDO中に競合を生成するために使用される場合があります。）

このようにインデックスタプルの削除を遅延させバッチ処理することで、さらなる最適化が可能になります。それは、近くにある「余分な」インデックスタプル（LP_DEADと設定されていないタプル）を、ついでに確認するコストが非常に低い場合に日和見的に確認することです（snapshotConflictHorizonを生成するためにtableamがそのテーブルブロックを訪問することがすでにわかっているため）。削除しても安全であることが判明したインデックスタプルはすべて削除されます。単純削除は、実際に削除可能であることが判明した余分なタプルが、最初からLP_DEADビットが設定されていたかのように動作します。

重複排除もページ分割を防ぐことができますが、インデックスタプルの削除が推奨されるアプローチです。ポスティングリストタプルは、ポスティングリスト内のすべてのテーブルTIDがデッドであると判明したときにのみ、LP_DEADビットを設定できることに注意してください。LP_DEADビットは削除の出発点に過ぎないため、実際にはこれは大きな問題ではありません。本当に重要なのは、ページが最終的に分割される前のどこかの時点で、テーブル内で近くにある関連するTIDを対象とした「何らかの」削除操作が行われることです。削除プロセスがポスティングリストタプルからデッドTIDのグループを詳細に削除するために必要なのはこれだけです（状況が手に負えなくなることは許されません）。

ボトムアップ削除
------------------

連続するUPDATEによるバージョンチャーンが原因で重複が発生していると疑われる場合、ページ上に存在する重複を削除しようと試みます。これは、heapamのHOTのような最適化がインデックスに対して機能しなかったことを示すエグゼキュータヒントを受け取った場合にのみ発生します。入力されたタプルは、MVCCの目的で必要な論理的に変更されていない重複である必要があり、それが当該リーフページにおける新しいインデックスタプルの主要な発生源である可能性が高いことを示唆しています。（また、ボトムアップ削除は、外部ヒントなしで検知が容易であるため、継続的なINSERTとDELETEに関連するチャーンがある場合のユニークインデックス内でもトリガーされます。）

ボトムアップ削除パスが行われるときには、単純削除はすでにページ分割を防ぐことに失敗しているでしょう（多くの場合、ページ上にLP_DEADビットが一度も設定されなかったためです）。これら2つのメカニズムの実装は密接に関連しています。各操作には同じWALレコードが使用され、どのTID/タプルが実際に削除しても安全かを判断するために同じtableamインフラストラクチャが使用されます。実装の違いは、削除を検討するTIDをどのように選択するか、そしてtableamがすべてのテーブルブロックにアクセスする前に諦めるかどうかという点だけです（ボトムアップ削除は、失敗のコストを低く抑えることで、その成功の不確実性と共存しています）。それでもなお、これら2つのメカニズムは概念レベルでは明らかに区別されます。

ボトムアップインデックス削除は完全にヒューリスティックによって駆動されます（一方、単純削除は、少なくともすでにLP_DEADとマークされているインデックスタプルを削除することが保証されています。少なくとも1つは存在しなければなりません）。削除すべきインデックスタプルを1つでも見つけられるという確証はありません。そのため、tableamと密接に協力して、支払うコストと得られる利益のバランスを保っています。これに使用するインターフェースについては、access/tableam.hで詳しく説明されています。ボトムアップインデックス削除は、バージョン駆動型の不必要なページ分割に対するバックストップ（防波堤）メカニズムと考えることができます。これは部分的に、世代別ガベージコレクションの考え方である「世代別仮説」に基づいています。これは「ほとんどのオブジェクトは若くして死ぬ」という経験的な観察結果です。nbtree内では、新しいインデックスのタプルは同じ場所に素早く出現し、その後すぐにゴミとなることがよくあります。特定のワークロードでは、比較的少数のリーフページにゴミが集中することがあります（少なくとも、ボトムアップインデックス削除がない以前のPostgreSQLバージョンではそうでした）。nbtreeにおけるボトムアップインデックス削除の設計原則（VACUUMをどのように補完するかという詳細を含む）については、doc/src/sgml/btree.sgmlの高レベルな説明を参照してください。

各ボトムアップパス内で、削除しても安全なタプルがかなりの数見つかると予想されます。もしそうでない場合、同じリーフページに対してボトムアップ削除を検討する必要はしばらくありません（通常はページが分割され、当面の問題が解決するためです）。バージョンチャーン（頻繁な更新）のみによって引き起こされる不必要なページ分割のリスクが常にあるページに対して、定期的にボトムアップ削除操作を実行することを想定しています。このメカニズムがうまく機能しているときは、常にバージョンチャーンによるページ分割の「瀬戸際」にありながら、実際には一度も発生しない状態を維持します。

重複ヒューリスティックは、かなり単純であるにもかかわらずうまく機能します。不必要なページ分割は、バージョンチャーンが真に病的なレベルにある場合にのみ発生します（理論上、少量のバージョンチャーンがページ分割を厳密に必要な時期よりも早く発生させる可能性はありますが、それはほとんど無害です）。基盤となるワークロードを理解する必要はありません。対象とする病理の一般的な性質を理解するだけで十分です。バージョンチャーンが真に病的である場合、それを特定するのは容易です。影響を受けるリーフページはかなり均質です。

WALに関する考慮事項
------------------

挿入および削除のアルゴリズム自体は、クラッシュ後のbtreeの一貫性を保証するものではありません。堅牢性を提供するために、WALリプレイに依存しています。単一のWALエントリは実質的にアトミックなアクションであり、完了に失敗した場合はログからやり直すことができます。

通常のアイテム挿入（ページ分割を強制しないもの）は、1つのページにしか影響しないため、当然ながら単一のWALエントリとなります。リーフアイテムの削除も同様です（削除によってリーフページのアイテム数がゼロになった場合、そのページは削除候補となりますが、それは別のアクションです）。

ページ分割を引き起こす挿入は、挿入レベルで発生する変更（右兄弟の左リンクの更新を含む）に対して単一のWALエントリとして記録され、続いて親レベルでの挿入（それ自体がページ分割である可能性があり、その上の追加の挿入が必要になるなど）に対して2番目のWALエントリが記録されます。

ルート分割の場合、後続のWALエントリは「挿入」エントリではなく「新しいルート」エントリとなりますが、それ以外の詳細はほぼ同じです。

分割には複数のアトミックなアクションが含まれるため、ページを分割してから新しい半分へのダウンリンクを親に挿入するまでの間にシステムがクラッシュする可能性があります。リカバリ後、新しいページのダウンリンクが欠落します。検索アルゴリズムは、左兄弟からの右リンクをたどることでページが見つかるため正しく動作しますが、ツリー内のダウンリンクが多く欠落しているとパフォーマンスが低下します。より深刻な結果として、ダウンリンクのないページが再度分割された場合、挿入アルゴリズムは親レベルでダウンリンクを挿入する場所を見つけることができなくなります。

我々のアプローチは、新しい挿入のためにツリーを検索する際に、欠落しているダウンリンクをオンザフライで作成することです。検索中に行うことも可能ですが、本来読み取り専用であるはずの操作に余分な更新を追加しないのが最善と思われます（ホットスタンバイモードでは更新自体が不可能です）。VACUUMで欠落したダウンリンクを追加するのが自然に思えますが、ダウンリンクの挿入にはページの分割が必要になる場合があり、ディスク容量が不足すると失敗する可能性があります。これはVACUUM中に発生すると問題です。そもそもVACUUMを実行する理由がディスク容量不足である可能性があり、その場合、ディスク容量不足のためにVACUUMが完了しなくなるからです。対照的に、挿入は物理ファイルの拡張を必要とする場合があります。1つだけ小さな例外があります。VACUUMは、内部ページの子を削除する際に、中断された内部ページの分割を完了させます。これにより、親アイテムを再検索するためのコードをページ分割とページ削除の両方で使用できるようになります。

欠落したダウンリンクを特定するために、ページが分割される際、左側のページに分割がまだ完了していないことを示すフラグ（INCOMPLETE_SPLIT）が立てられます。親へのダウンリンクが挿入されると、挿入と同時にフラグがアトミックにクリアされます。子ページは、親での挿入が完了し、子ページのフラグがクリアされるまでロックが保持されますが、親も分割する必要がある場合にツリーを再帰的に遡る前であれば、その直後に解放できます。これにより、不完全に分割されたページは通常の状態では見えないことが保証されます。親への挿入が何らかの理由で失敗した場合にのみ発生します。（リカバリ中にリーダーが不完全分割フラグが設定されたページを観測する可能性もあります。詳細は後の「リカバリ中のスキャン」のセクションを参照してください。）

右側のページにダウンリンクがないにもかかわらず左側のページにフラグを立てるのは、左側のページから右側のページへ右リンクをたどる際に、そのページが親へのダウンリンク挿入を必要とすることがあらかじめ分かっている方が便利だからです。

そのレベルで唯一の非ルートページを分割する場合、必要なメタページ更新（「高速ルート」リンクの）が実行され、親レベルへの挿入の一部としてログに記録されます。ルートページを分割する場合、メタページの更新は「新しいルート」アクションの一部として処理されます。

ページ削除の各ステップは個別のWALエントリとして記録されます。リーフを半死状態（half-dead）としてマークし、ダウンリンクを削除するのが1つのレコードであり、ページのリンクを解除するのが2番目のレコードです。何らかの理由でVACUUMが中断されたり、システムがクラッシュしたりしても、ツリーは検索と挿入に対して一貫性を保ちます。次のVACUUMが半死状態のリーフページを見つけ、削除を継続します。

9.4以前は、リカバリ中に不完全な分割やページ削除を追跡し、次の挿入やVACUUMで遅延して処理するのではなく、リカバリ終了時に即座に完了させていました。しかし、それによりリカバリが非常に複雑になり、クラッシュリカバリが実行された場合にのみ問題が解決されていました。不完全な分割は、親へのダウンリンク挿入中にメモリ不足やディスク容量不足のような回復可能なエラーが発生した場合にも起こり得ます。

リカバリ中のスキャン
---------------------

nbtreeインデックスは、ホットスタンバイモードでの読み取りクエリをサポートしています。すべてのアトミックなアクション/WALレコードは、ツリーをリーダーにとって一貫した状態に保つ分離された変更を行います。リーダーは、プライマリ上のリーダーが従うのと同じルールに従ってページをロックします（リーダーは「同時実行」のページ分割やページ削除から回復するために右に移動する必要がある場合があります。これはプライマリ上と同じです）。

ただし、プライマリ上の元の書き込み操作と比較して、リプレイ/スタートアッププロセスによってページがどのようにロックされるかにはいくつかの違いがあります。例外はページ分割とページ削除に関係します。ページ分割の第1フェーズと第2フェーズは、独立したアトミックなアクションであるため、リプレイ中に個別に処理されます。プライマリで行われた親ページと子ページの書き込みロックの結合を再現しようとはしません。リーダーは不完全分割フラグを気にしないため、これは安全です。プライマリ上で余分な書き込みロックを保持するのは、最初のライターが分割を完了する前に、2番目のライターが不完全分割フラグを観測できないようにするためだけです。もしプライマリ上の同時ライターが同じページ上の不完全分割フラグを観測できるようにしてしまうと、各ライターが未完了の分割を完了させようとし、親ページを破損させてしまいます。（同様に、ページ削除レコードのリプレイは、ターゲットのリーフページに対する書き込みロックを常時保持するわけではありません。プライマリのみが、削除中のページに挿入しようとする同時ライターをブロックする必要があります。）

WALリプレイは、元の実行中に取られたアプローチと一致する方法で同レベルのロックを保持します。これにより、リーダーが同レベルの不整合を観測するのを防ぎます。リカバリ中に同レベルのロックを取得する方法をもっと緩やかにすることは可能かもしれませんが（同レベルのロックを結合しなくても、ほとんどの種類のリーダーは回復のために右に移動できるため）、ここでは保守的であることを選択しています。

リカバリ中、すべてのインデックススキャンは ignore_killed_tuples = false で開始され、kill_prior_tuple が設定されることはありません。これを行う理由は、スタンバイサーバー上の最も古いxminがプライマリサーバー上の最も古いxminよりも古くなる可能性があり、タプルがスタンバイ上でまだ可視であっても LP_DEAD とマークされる可能性があるためです。タプルの LP_DEAD ビットはWALログに記録されませんが、フルページ書き込みによってスタンバイ上に現れる可能性があります。そのため、スタンバイでは常にそれらを無視しなければならず、設定する価値もありません。（LP_DEAD とマークされたタプルが最終的にプライマリで削除されると、その削除はWALログに記録されます。したがって、スタンバイで実行されるクエリは、プライマリで行われる LP_DEAD 設定の恩恵を十分に受けることができます。）

リカバリ中に開始されるスキャンについて述べていることに注意してください。リカバリ中にスキャンを開始し、リカバリ完了後の通常実行中に終了できるようにするために少し工夫をしています。これは、スタンバイが通常実行中のサーバーに状態を変化させている間も、実行中のアプリケーションを継続できるため、重要な機能です。

非MVCCスキャンから誤った結果が返されるのを防ぐために必要なインターロックは、スタンバイノードでは必要ありません。リカバリ中にVACUUMレコードをリプレイする際は完全なクリーンアップロックを取得しますが、リカバリではすべてのリーフページをロックする必要はありません（削除するアイテムがあるリーフページのみで十分です）。これは、リカバリ中のインデックスのみスキャン（index-only scan）が壊れるのを防ぐのに十分です（TIDリサイクルの安全性を確保することに関する上記のセクションを参照）。そのため、通常のインデックススキャンのみが懸念として残ります。（XXX: リカバリ中にこれが完全に不要な理由は実際には不明です。）

MVCCスナップショットの通常のインデックススキャンは、元の実行中に安全であるのと同じ理由で常に安全です。HeapTupleSatisfiesToast() はMVCCセマンティクスを使用しませんが、それはその必要がないからです。メインヒープ行が可視であれば、TOAST行も可視になります。したがって、可視（ライブ）タプルからTOASTポインタをたどる限り、対応するTOAST行も可視になるため、それらに対してMVCCを再チェックする必要はありません。

知っておくと便利なその他のこと
-----------------------------------

すべてのbtreeのページ0はメタデータページです。このページには、真のルートと現在の有効なルート（「高速」ルート）の両方のルートページの位置が格納されます。すべてのインデックス検索でメタページをフェッチするのを避けるため、メタデータ情報のコピーをインデックスの relcache エントリ（rd_amcache）にキャッシュします。キャッシュを使用することは、古くなっている可能性のあるルートページポインタをたどることを意味するため、少し注意が必要です。しかし、キャッシュされたポインタをたどるバックエンドは、真のルートに向かうときに is-root フラグをチェックするか、高速ルートに向かうときにページに兄弟がないことをチェックすることで、意図したページに到達したかどうかを十分に検証できます。最悪の場合、実際の高速ルートよりも上にある高速ルートへのキャッシュされたポインタを持っていると、ツリーのレベルを余分に下ることになる可能性があります。そのようなケースは最適化するほど頻繁には発生しないはずです。いずれにせよ、高速ルートポインタを移動させたVACUUMがインデックスの統計更新を発行すると予想されるため、relcacheのフラッシュによってキャッシュされたメタページはすぐに破棄されると考えられます。このアルゴリズムは、1ページあたり少なくとも3つのアイテム（「ハイキー」1つと実際のデータアイテム2つ）を格納できることを前提としています。そのため、ページサイズの1/3を超えるアイテムを受け入れることは安全ではありません。より大きなアイテムでも動作する場合があるかもしれませんが、そのページに他に何が配置されるかによって、後で障害を引き起こす可能性があります。

「ScanKey」データ構造は、このコード内で「検索（search）」スキャンキーと「挿入（insertion）」スキャンキーという、根本的に異なる2つの方法で使用されます。検索スキャンキーは、btreeコードの外部から btbeginscan() や btrescan() に渡されるものです。検索スキャンキー内の sk_func ポインタは、int4lt のようなブール値を返す比較関数を指します。特定のインデックス列に対してスキャンキーエントリが複数存在する場合もあれば、全く存在しない場合もあります。（キーはインデックス列の順序で現れる必要がありますが、特定の列に対する複数のキーの順序は規定されていません。）挿入スキャンキー（「BTScanInsert」データ構造）は、同様の ScanKey の配列データ構造を使用しますが、sk_func ポインタは btree 比較サポート関数（つまり、<0, =0, >0 として解釈される int4 値を返す3方向比較関数）を指します。挿入スキャンキーでは、インデックス列ごとに最大1つのエントリしか存在しません。また、スキャンを開始する場所を特定するためのルール（スキャンが「nextkey」スキャンかどうかなど）に関する他のデータも含まれます。挿入スキャンキーは btree コード内（例: _bt_mkscankey()）で構築され、スキャンの開始点を特定するため、および新しいインデックス・タプルを挿入する場所を特定するために使用されます。（注：検索スキャンキーから構築された挿入スキャンキーや、切り詰められたピボット・タプルから構築された挿入スキャンキーの場合、インデックス列よりもキーの数が少ないことがあり、これは残りのインデックス列に対する制約がないことを示しています。）スキャンの開始点を特定した後、各インデックス・エントリが順次スキャンされる際に、元の検索スキャンキーが参照され、エントリを返すかどうか、およびスキャンを停止できるかどうかが判断されます（_bt_checkkeys() を参照）。

サフィックス切り詰めに関する注記
-----------------------------

リーフページの分割中、ページハイキーに不要なサフィックス・キー属性を切り詰めます。残りの属性は、分割後の左ページの最後のインデックス・タプルが左ページに属すること、および分割後の右ページの最初のインデックス・タプルが右ページに属することを区別できなければなりません。タプルは論理的には切り詰められたキー属性を保持していますが、暗黙的に「負の無限大」という値を持つものとみなされ、ストレージ上のオーバーヘッドはありません。ハイキーはその後、新しい右ページのための親ページ内のダウンリンクとして再利用されるため、サフィックス切り詰めによってピボット・タプルは短くなります。INCLUDE インデックスは、リーフページ分割時に非キー属性が切り詰められることが保証されていますが、キー属性の通常の基準に基づいて、一部のキー属性も切り詰められる場合があります。非キー属性は B-Tree 検索にとって単なるペイロードであるため、これらは特別なケースではありません。

キー属性のサフィックス切り詰めの目的は、インデックスのファンアウトを改善することです。この手法は、Bayer と Unterauer によって最初に記述されました（R.Bayer and K.Unterauer, Prefix B-Trees, ACM Transactions on Database Systems, Vol 2, No. 1, March 1977, pp 11-26）。Postgres の実装は、彼らの論文に緩やかに基づいています。Postgres は、論文で「単純なプレフィックス B-Tree」と呼ばれているもののみを実装していることに注意してください。また、この論文では、ツリーがサフィックス・ツリーに格納される文字列のように「プレフィックス特性」を維持する単一の文字列からなるキーを持つことを前提としていることにも注意してください（以前のバイトの比較は常に後のバイトの比較よりも重要でなければならず、一般的に、文字列はそれらが断片に分割されたときに推移的な一貫性を損なわない方法で比較される必要があります）。Postgres におけるサフィックス切り詰めは現在、属性単位の粒度でしか機能しませんが、text 型のような可変長型の場合に、より小さな属性値を生成する opclass インフラストラクチャを考案することは容易です。opclass サポート関数は、リーフページ分割の各半分を正しく分離できる最小のキー値を生成できます。

リーフページの分割点を選択するための洗練された基準があります。一般的な考え方は、ページ分割の各半分のスペースのバランスに過度な影響を与えることなく、サフィックス切り詰めを効果的にすることです。リーフ分割点の選択は、分割されるページ上のアイテム間にある点の中から選択するものと考えることができます（少なくとも、受信したタプルがすでにページ上に配置されていると仮定した場合。実際にはページ上に十分なスペースがないため、仮定する必要があります）。最初の不一致属性が可能な限り早く現れる2つのインデックス・タプルの間で分割点を選択すると、可能な限り多くのサフィックス属性を切り詰める結果となります。分割の各半分間でスペースを均等にバランスさせることが通常は第一の懸念事項ですが、正確な分割点をわずかに調整するだけでも、切り詰めをはるかに効果的にすることができます。

サフィックス切り詰めは、主にピボット・タプルを小さくし、内部ページの分割を遅らせるために価値がありますが、それが効果的である理由はそれだけではありません。アライメントのためにピボット・タプルが小さくならないような切り詰めであっても、どの値がどのページに属するかを記述する際に、ピボット・タプルが必要以上に制限的になることを防ぎます。

内部ページの分割中にサフィックス切り詰めを正しく実行することは不可能ですが、内部ページを分割する際に識別的であることは依然として有用です。フィルファクター的に最適な分割点の許容範囲内で、親に挿入されるダウンリンクが最小となるような分割点が選択されます。この考え方もプレフィックス B-Tree の論文から来ています。このプロセスは、サフィックス切り詰めを効果的にするためにリーフレベルで行われることと多くの共通点があります。全体的な効果として、サフィックス切り詰めは、特にインデックスの寿命の初期において、より小さく、より識別的なピボット・タプルを生成する傾向があり、一方、内部ページの分割を偏らせることで、初期の小さなピボット・タプルがルートページに行き着くようになり、ルートページの分割を遅らせます。

論理的な重複は特別に考慮されます。分割点を選択するためのロジックは、重複が複数のページにまたがることを避けるために多大な努力を払っており、ほとんどの場合、ユーザーキーが異なる2つのタプルの間で分割点を選択することに成功し、必要であれば完全に偏った分割を受け入れます。すでに重複でいっぱいのページを分割しなければならない場合、フォールバック戦略は、重複が主にヒープ TID の昇順で挿入されると仮定します。ページは、左半分がほぼ満杯になり、右半分がほぼ空になるように分割されます。全体的な効果として、リーフページの分割は、大量の重複の挿入にうまく適応し、スペースの利用率を最大化します。また、このように大量の重複を同じリーフページに「閉じ込める」ことで、重複排除がより効率的になることにも注意してください。重複排除は、既存のポストリスト・タプルを頻繁にマージすることなく、まれに実行できます。

重複排除に関する注記
-------------------------

ストレージのオーバーヘッドを削減し、ページ分割を回避（または少なくとも遅延）するために、非一意インデックス内の非ピボット・タプルを重複排除します。一意インデックスにおける重複排除の目的はかなり異なることに注意してください。詳細は後のセクションを参照してください。重複排除は、インデックスの論理的な内容を変更せず、読み取りクエリにオーバーヘッドを追加することなく、タプルの物理的な表現を変更します。非ピボット・タプルは、ポストリスト（標準的なアイテムポインタ形式を持つヒープ TID の単純な配列）を持つ単一の物理タプルにマージされます。重複排除は常に遅延して適用され、ページ分割を実行する必要がある時点で行われます。これは、リーフページを分割するための最後の防衛線として、LP_DEAD アイテムが削除された場合にのみ発生します（ボトムアップ・インデックス削除が最後から2番目の防衛線として最初に試行される場合があります）。すべての TID がデッドであると判明している場合に限り、ポストリスト・タプルで LP_DEAD ビットを設定できます。

重複排除に対する遅延アプローチにより、ページ分割中に使用されるページスペースの計算において、ポストリストに関する特別なケースのロジックを最小限に抑えることができます。ポストリストは、リーフページ分割中に必要に応じてサフィックス切り詰めによって確実に切り詰められる追加のペイロードと考えることができます。これは、INCLUDE インデックス・タプルの非キー列と同様です。受信した新しいタプルは、一般的に重複しないプレーンなアイテムとして扱うことができます（ただし、重複する新しい/受信アイテムが実際にどのように処理されるかについては、ポストリスト分割に関するセクションを参照してください）。

ポストリストの表現は GIN が使用するポストリストとほぼ同一であるため、GIN の可変バイト符号化圧縮スキームを個々のポストリストに適用することは容易です。ポストリストの圧縮は、ページスペースの計算に関するポストリスト分割の前提を壊すため（後のセクションを参照）、圧縮を nbtree とどのように統合できるかは不明です。さらに、nbtree は一般的に多くの同時読み取りおよび書き込みを行う環境で一貫したパフォーマンスを発揮するように最適化されているため、ポストリストの圧縮は nbtree にとって魅力的なトレードオフを提供しません。また、圧縮を行うと、ポストリストからの TID のサブセットの削除が遅く複雑になり、ボトムアップ・インデックス削除に大きく依存するワークロードにとって大きな問題となります。

重複排除に対する遅延アプローチの主な目的は、ランダム更新を伴う重複排除のパフォーマンスへの影響を制限することです。同じキー値の同時追加のみの挿入であっても、個々のインデックス・タプルの挿入順序はヒープ TID の順序と完全には一致しない傾向があります。重複排除を遅延させることで、ページレベルの断片化を最小限に抑えます。

一意インデックスにおける重複排除
-------------------------------

非常に多くの場合、一意インデックスのほぼすべてのリーフページに配置できる異なる値の数は、固定されており永続的です。例えば、ID 列の主キーでは、リーフページの分割は通常、最も右側のリーフページ内への新しい論理行の挿入によってのみ引き起こされます。最も右側ではないリーフページの分割が発生した場合、その分割は既存の論理行の UPDATE に関連する挿入によってトリガーされたはずです。複数のバージョンを格納するためだけにリーフページを分割するのは誤った節約です。事実上、一時的な重複のバーストを吸収するためだけに、インデックス構造を恒久的に劣化させていることになります。

一意インデックスにおける重複排除は、これらの病的なページ分割を防ぐのに役立ちます。長期的には重複は存在しなくなるため、重複をスペース効率の良い方法で格納することが目的ではありません。むしろ、ページ分割が必要になる前に、標準的なガベージコレクション・メカニズムが実行されるまでの時間を稼いでいるのです。

一意インデックスのリーフページは、挿入（ページを分割する必要があるかもしれないもの）が、たまたまページ上に既存の重複を観測した場合にのみ、重複排除パスを実行します。これは、すべての新しい挿入が UPDATE による重複である場合にのみ重複排除がうまくいくという仮定に基づいています。これは、ページ分割を遅らせる機会を逃すことを意味するかもしれませんが、最終的な目標はリーフページの分割を「無期限に」遅らせること（つまり、完全に防ぐこと）であるため、問題ありません。いずれにせよ避けられない分割を遅らせようとしてもほとんど意味がありません。これにより、重複がほとんどまたは全くない一意インデックスで重複排除を試みるオーバーヘッドを回避できます。バージョンチャーンによって引き起こされる「不要な」ページ分割を回避することも、PostgreSQL 14 で追加されたボトムアップインデックス削除の目的です。ボトムアップインデックス削除は、現在この問題に対処するための推奨される方法です（あらゆる種類のインデックスに対応していますが、特にユニークインデックスで有効です）。それでも、重複排除がボトムアップインデックス削除を補完できる場合があります。（古いスナップショットがクリーンアップを保留しているために）削除がタプルを解放できない場合、重複排除にフォールバックすることで追加の容量を確保できます。重複排除によってページ分割を遅延させることで、同じページに対する将来のボトムアップ削除パスを成功させることが可能になります。

ポスティングリストの分割
------------------------

受信したタプルが既存のポスティングリストと重なる場合、ポスティングリストの分割が実行されます。ページ分割と同様に、ポスティングリストの分割は、新しい（受信した）アイテムが「収まらない」状況を解決すると同時に、そのアイテムを（同じアトミックなアクションの一部として）挿入します。ほぼ満杯のページに新しいアイテムを挿入する際、ポスティングリストと重なり、ポスティングリストの分割とページ分割の両方が発生する可能性があります（可能性は低いですが）。その場合でも、ポスティングリストを分割するアトミックなアクションが新しいアイテムも挿入します（ページ分割は常に新しいアイテムをその場で挿入するため）。ポスティングリストの分割を挿入と同じアトミックなアクションに含めることで、同じポスティングリストへの同時挿入によって引き起こされる問題を回避できます。ポスティングリストを変更する正確な詳細は新しいアイテムに依存し、その逆も同様であるためです。また、単一のアトミックなアクションにすることで、ポスティングリストの分割に必要な追加のWALの量を最小限に抑えることができます。元のポスティングリストタプルを明示的にWALログに記録する必要がないためです。

新しいタプルを挿入するのと同じアトミックなアクションに便乗しているにもかかわらず、ポスティングリストの分割は、挿入自体（またはページ分割自体）とは別の追加アクションと考えることができます。ポスティングリストの分割は、概念的には、既存のポスティングリストと重なる挿入を、最終的な新しいアイテムをポスティングリストのすぐ右側に追加する挿入へと「書き換える」ものです。ポスティングリストのサイズは変わらないため、ページスペースの管理コードはポスティングリストの分割をまったく気にする必要はありません。これは我々の設計の重要な利点です。ページ分割ポイントを選択するロジックは、ポスティングリストの分割を考慮する必要がない場合でも非常に繊細なためです。

新しいアイテムが最初から既存のポスティングリストと重なっていなかったという幻想を維持するために必要な追加ステップは、わずかです。受信したタプルのヒープTIDを、既存の（元々重なっていた）ポスティングリストの最も右側（最大）のヒープTIDで置き換えます。同様に、元の受信アイテムのTIDは、ポスティングリスト内の適切なオフセットに再配置されます（通常、TIDをずらして空き場所を作ります）。最後に、ページ分割を伴うポスティング分割の場合、最終的な新しいアイテムとリスト分割後のポスティングタプルの両方を持つ元のページの仮想バージョンに基づいて、新しいハイキーを生成する必要があります（ページ分割は通常、新しいアイテムや収まらないアイテムを含む仮想バージョンに対して動作します）。

このアプローチにより、受信したアイテムの挿入を完了させずにポスティングリストを分割するような「先行的な」アトミックなポスティング分割操作を考案することを回避できます。この代替設計はよりクリーンに見えるかもしれませんが、ページスペースの管理において繊細な問題を引き起こします。一般的に、受信した新しいアイテムがいずれのポスティングリストの半分とも重ならないようにポスティングリストを分割するのに十分な空きスペースがページ上にない場合があり、新しいアイテムの実際の挿入が始まる前に操作が失敗する可能性があります。結局、いずれにせよページ分割が必要なポスティングリストの分割を処理しなければならなくなります。さらに、ポスティングリストの分割中に可変の「分割ポイント」をサポートしても、全体的なスペース利用率は向上しません。

データ表現に関する注記
----------------------

L&Yで必要とされる右側の兄弟リンクは、左側の兄弟リンク、ページレベル、およびいくつかのフラグと同様に、ページの「不透明なデータ（opaque data）」領域に保持されます。ページレベルは、リーフレベルの0からルートのツリー深さマイナス1まで上向きにカウントされます。（リーフから上向きにカウントすることで、ルートを分割する際に既存のページ番号を振り直す必要がなくなります。）

Postgresのディスクブロックデータ形式（アイテムの配列）は、LehmanとYaoのディスクページの「キーとポインタが交互に並ぶ」という概念に適合しないため、いくつかの工夫が必要です。（キーとポインタが交互に並ぶという概念は、内部ページの分割において重要です。概念的には既存のピボットタプルの中間で分割され、タプルの「セパレータ」キーが左側の新しいハイキーとして左側に配置され、タプルのポインタ/ダウンリンクが右側に最初の/マイナス無限大のダウンリンクとして配置されます。）

ツリーレベルで最も右側にないページでは、「ハイキー」はページの最初のアイテムに保持され、実際のデータアイテムはアイテム2から始まります。「ハイキー」アイテムのリンク部分は使用されません。最も右側にあるページには「ハイキー」がないため（暗黙的にプラス無限大）、データアイテムは最初のアイテムから始まります。ハイキーを右ではなく左に置くのは奇妙に見えるかもしれませんが、データアイテムを追加するたびにハイキーを移動させる必要がなくなります。

リーフページでは、データアイテムは単にインデックス付けされるリレーション内のタプルへのリンク（TID）であり、関連するキー値が伴います。

非リーフページでは、データアイテムは境界キーを持つ子ページへのダウンリンクです。各データアイテム内のキーは、その子ページ上のキーに対する厳密な下限であるため、論理的にはキーはそのダウンリンクの左側にあります。ハイキー（存在する場合）は、最後のダウンリンクの上限です。そのような各ページの最初のデータアイテムには下限がありません（あるいは、必要であればマイナス無限大の下限とみなします）。比較ルーチンはそれに応じて処理する必要があります。アイテムに格納される実際のキーは無関係であり、まったく格納する必要はありません。この配置は、L&Yの非リーフページがキーよりも1つ多くのポインタを持つという事実に対応しています。サフィックス切り捨てのマイナス無限大属性も同様に動作します。
