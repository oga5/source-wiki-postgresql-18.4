ブロック範囲インデックス (BRIN)
==========================

BRINインデックスは、極めて巨大なテーブルの非常に高速なスキャンを可能にすることを目的としています。

BRINインデックスの基本的な考え方は、ヒープページの連続したグループ（ページ範囲）における要約値を追跡することです。例えば、btree演算子クラスを持つデータ型の場合は最小値と最大値、幾何データ型の場合は境界ボックスなどがこれに当たります。これらの値は、クエリの条件に応じて、テーブルスキャン中に該当するページのスキャンを回避するために使用できます。

この代償として、タプルが挿入されるたびに、各ページ範囲の保存された要約値を更新する必要があります。


アクセスメソッドの設計
--------------------

このタイプのインデックス内にはアイテムポインタが保存されないため、amgettupleインターフェースをサポートすることはできません。その代わり、amgetbitmapサポートのみを提供します。amgetbitmapルーチンは、クエリ条件に一致するページ範囲内のすべてのページを含む、非可逆（lossy）なTIDBitmapを返します。BitmapHeapScanノード内の再チェックステップで、クエリ条件に従って可視ではないタプルが除外されます。

演算子クラスは、以下のエントリを持つ必要があります。

- すべての演算子クラスで共通の汎用サポートプロシージャ (pg_amproc):
  * "opcinfo" (BRIN_PROCNUM_OPCINFO): インデックス作成またはスキャンのための構造体を初期化します。
  * "addValue" (BRIN_PROCNUM_ADDVALUE): インデックスのタプルとヒープアイテムを受け取り、ヒープアイテムの値が含まれるようにインデックスのタプルを必要に応じて変更します。
  * "consistent" (BRIN_PROCNUM_CONSISTENT): インデックスのタプルとクエリ条件を受け取り、インデックスのタプル値がクエリ条件と一致するかどうかを返します。
  * "union" (BRIN_PROCNUM_UNION): 2つのインデックスのタプルを受け取り、それらの和集合を表すように最初のタプルを修正します。
  プロシージャ番号10までは将来の拡張のために予約されています。

さらに、各演算子クラスには追加のサポート関数が必要です。
- Minmax形式の演算子クラス:
  * プロシージャ番号11～14は、その型の不等号演算子を実装する関数に使用されます。順に、未満、以下、以上、より大きい、となります。

異なる設計を使用する演算子クラスでは、異なる追加のプロシージャ番号が必要になります。

演算子クラスは、オプティマイザがクエリを実行するためにインデックスを選択できるように、演算子 (pg_amop) エントリも持つ必要があります。
- Minmax形式の演算子クラス:
  * btreeと同じ演算子 (<=, <, =, >=, >)

各インデックスのタプルは、いくつかのNULLビットと、演算子クラスで指定された値を保存します。これらは、列数の2倍の長さを持つ単一のNULLビットマスクに保存されます。汎用的なNULLビットは、各列について以下を示します。
  * bt_hasnulls: ページ範囲内にNULL値が1つでも存在するかどうか
  * bt_allnulls: ページ範囲内のすべての値がNULLであるかどうか

演算子クラスで指定される値は以下の通りです。
- Minmax形式の演算子クラス
  * 範囲内のすべてのタプルにわたる最小値
  * 範囲内のすべてのタプルにわたる最大値

addValueおよびUnionサポートプロシージャは、メモリ内のBRINタプルに保存したい値をdatumCopy()する際に注意が必要であり、古いコピーを置き換える際にはpfree()しなければならないことに注意してください。タプルから参照される値の一部は永続し、他は消滅するため、これを自動化できるような明確に定義されたメモリコンテキストの寿命は存在しません。


範囲マップ (Range Map)
--------------------

特定のページ範囲のインデックスのタプルを見つけるために、範囲マップ（略して "revmap"）と呼ばれる内部構造があります。これはページ範囲ごとに1つのTIDを保存しており、それがその範囲を要約するインデックスのタプルのアドレスとなります。マップエントリは固定サイズであるため、任意のヒープページに対する範囲マップエントリのアドレスは単純な算術計算で算出可能です。

要約されたページ範囲に新しいヒープタプルが挿入されると、既存のインデックスのタプルと新しいヒープタプルを比較します。もしヒープタプルがインデックスのタプルによって与えられた要約データの範囲外である場合（あるいは新しいヒープタプルにNULL値が含まれているのにインデックスのタプルがNULLなしを示している場合）、インデックスは新しい値で更新されます。多くの場合、インデックスのタプルをその場で更新できますが、新しいインデックスのタプルが古いものより大きく、ページ内に十分な空きがない場合は、新しい値を持つ新しいインデックスのタプルを作成する必要があります。範囲マップはそれを指すように迅速に更新でき、古いインデックスのタプルは削除されます。

範囲マップが無効なTIDを指している場合、対応するページ範囲は要約されていないとみなされます。要約されていないページにタプルが追加されても、何も起こる必要はありません。

BRINインデックスに従ってテーブルをスキャンする場合、範囲マップを順次スキャンします。これにより、ページ範囲の昇順でインデックスのタプルが得られます。クエリ条件が各インデックスのタプルと照合され、一致すれば、そのページ範囲内の各ページが出力TIDビットマップの一部として返されます。一致しない場合はスキップされます。無効なインデックスTIDを返す範囲マップエントリ、つまり要約されていないページ範囲も、TIDビットマップ内で返されます。

revmapは、メタページの直後にあるインデックスのメインフォークの最初の数ブロックに保存されます。revmapを別のページに拡張する必要があるたびに、そのページ内の既存のタプルは別のページに移動されます。

ヒープタプルは制限なくどこからでも削除できます。ヒープタプルが要約データの制約値（btree演算子クラスを持つデータ型の場合は最小値または最大値）の1つである場合、将来その範囲で要約を再実行する必要があることを認識できるように、対応するインデックスのタプルに何らかのマークを付けると有用かもしれません。これにより、要約値がより厳密になる可能性があります。

要約 (Summarization)
--------------------

インデックス作成時にテーブル全体がスキャンされます。各ページ範囲について、各インデックス列の要約値とNULLビットマップが収集され、インデックスに保存されます。テーブルの末尾にある部分的に埋まったページ範囲も要約されます。

テーブルの末尾に新しいタプルが挿入されると、末尾の部分的なページ範囲を要約するインデックスのタプルが更新される場合があります。最終的にそのページ範囲が完了すると、新しいタプルはまだ要約されていない新しいページ範囲に属することになります。これらの挿入は新しいインデックスエントリを作成せず、そのページ範囲は後で要約されるまで未要約のままとなります。

テーブルに対してVACUUMが実行されるたびに、すべての未要約のページ範囲が要約されます。このアクションは、ユーザーが brin_summarize_new_values() を介して呼び出すこともできます。これらのプロシージャはどちらも、すべての未要約範囲をスキャンし、要約タプルを作成します。これも同様に、テーブル末尾の部分的に埋まったページ範囲を含みます。

バキューム (Vacuuming)
--------------------

BRINインデックスにはヒープTIDが保存されないため、ヒープタプルが削除されたときにインデックスをスキャンする必要はありません。ヒープタプルが削除された場合、一部の要約値がより厳密になる可能性がありますが、これは最適化の機会であり、正確性の問題ではありません。既存の値から値を「減算」するよりも、影響を受けるページ範囲で要約を再実行する必要があると表現する方が単純です。これは現在実装されていません。

テーブルにBRINインデックス以外のインデックスがない場合、詳細なインデックススキャンを行う必要がない（したがってバキュームが削除対象のTIDを保存する必要がない）ため、バキュームによる maintenance_work_mem の使用量を大幅に削減できることに注意してください。ただし、主キーはbtreeでなければならないため、テーブル内の唯一のインデックスがBRINである可能性は低く、この最適化は実装していません。


オプティマイザ
--------------

オプティマイザは、列に対する演算子クラスの pg_amop エントリに基づいてインデックスを選択します。


将来の改善点
------------

* ページ範囲のサイズを可変にするか？
  現在の設計では、BRINインデックスの各「インデックスエントリ」は同じ数のページをカバーします。これに厳密な理由はありません。要約値をよりコンパクトにできるのであれば、一部のインデックスエントリがより小さなページ範囲をカバーできるようにインデックスが自己調整できるようにするのは理にかなっているかもしれません。これはインデックス自体のBRINオーバーヘッドを大きくしますが、スキャン中のページ範囲の除外をより効率的にできる可能性があります。1ページあたり1つのインデックスのタプルという極限状態では、要約値が厳密であっても、ほとんどのヒープページを読み飛ばせるにもかかわらず、インデックス自体が占有するスペースが大きくなりすぎます。逆に、テーブル全体を要約する単一のタプルという極限状態では、インデックスは非常に小さいにもかかわらず、何も除外できません。これは、範囲マップ自体をインデックスとして使用することで実現できる可能性があります。

* TIDBitmapのよりコンパクトな表現
  TIDBitmapはビットマップスキャンを表すために使用される構造体です。範囲内のページを表すためにBitmapsetを使用しているため、非可逆なページ範囲の表現は私たちの目的には最適ではありません。大きな範囲内のすべてのページを返すことになるため、代わりに開始ページ番号と終了ページ番号を使用して範囲を表す構造体を許可する方が便利かもしれません。

* より良いバキューム
  削除されたタプルについて、バキューム中にBRINインデックスにより有用な情報を渡せるようにすると便利かもしれません。つまり、コールバックが各タプルのTIDを渡す必要がないようにすることです。例えば、TIDの代わりにブロック番号を渡すコールバックが必要になるかもしれません。これは、タプルの削除が多かったブロックでいつ要約を再実行するかを判断するのに役立ちます。
