WALリソースマネージャの記述関数
==========================================

デバッグ目的のために、各WALリソースマネージャには「記述関数（description function）」、すなわちrmgrdesc関数が存在します。rmgrdesc関数はWALレコードを解析し、WALレコードの内容を人間が読み取れる形式で出力します。

すべてのリソースマネージャのrmgrdesc関数はこのディレクトリに集められています。これは、スタンドアロンのpg_waldumpプログラムでも使用されるためです。これらは外部のデバッグツールでも使用できる可能性がありますが、記述関数や出力形式は安定したAPIの一部とはみなされないことに注意してください。

rmgrdesc出力形式のガイドライン
-------------------------------------

これらのガイドラインの目的は、各rmgr間での不必要な不整合を避け、ユーザーがdesc出力文字列を過度な困難なしに解析できるようにすることです。これはAPI仕様や交換フォーマットではありません。（いずれにせよ、現時点でこれらのガイドラインに従っているのはheapamとnbtreeのdescルーチンのみです。）

レコードの記述は、JSONスタイルのキー/値オブジェクトに似ています。ただし、明示的な「文字列」型や文字列のエスケープはありません。最上位の { } ブラケットは省略する必要があります。例：

snapshotConflictHorizon: 0, flags: 0x03

レコードの記述には可変長配列を含めることができます。例：

nunused: 5, unused: [1, 2, 3, 4, 5]

ネストされたオブジェクトは { } ブラケットを使用してサポートされます。これらは通常、可変長配列内に現れます。例：

ndeleted: 0, nupdated: 1, deleted: [], updated: [{ off: 45, nptids: 1, ptids: [0] }]

基となる物理WALレコード構造体のフィールド順序を忠実に反映した順序で出力するようにしてください。解析を容易にするため、キー名は（同じネストレベル内で）一意である必要があります。配列内の要素数は、配列の前に表示するのが望ましいです。

個々のWALレコードタイプが独自の規約を作成しても構いません。例えば、Heap2のPRUNEレコードの記述では、レコードの「redirected」フィールドに対してカスタム配列形式を使用しています：

... redirected: [1->4, 5->9], dead: [10, 11], unused: [3, 7, 8]

議論の余地はありますが、descルーチンでは代わりにオブジェクト表記を使用すべきかもしれません。しかし、基となる物理データ構造に関する有用な情報を伝える場合には、カスタム形式を使用することにも価値があります。

このアドホックな形式には、「dead」や「unused」配列（これらはページオフセット番号配列の標準的なdesc規約に従っています）に使用される形式に近いという利点があります。これは、示されている「redirected」要素が単なるページオフセット番号のペアであることを示唆しています（実際にはそのように機能するためです）。

rmgrdesc_utils.cには、この形式でデータを出力するためのヘルパー関数がいくつか含まれています。
