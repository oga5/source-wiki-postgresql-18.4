オプティマイザ
=========

これらのディレクトリは、パーサーが返した Query 構造体を受け取り、エグゼキュータで使用されるプランを生成します。/plan ディレクトリは実際の出力プランを生成し、/path コードはテーブルを結合するすべての可能な方法を生成し、/prep は特殊なケースに対するさまざまな前処理ステップを処理します。/util はユーティリティ関連です。/geqo は個別の「遺伝的最適化 (genetic optimization)」プランナであり、結合ツリー空間を網羅的に検討するのではなく、半ランダムな探索を行います。（ただし、/geqo によって検討される各結合は、パスを作成するために /path に渡されるため、GEQO モードであっても、特定の結合ペアに対するすべての可能な実装パスが検討されます。）


パスと結合ペア
--------------------

計画/最適化プロセスにおいて、クエリを実行するさまざまな方法を表す「Path」ツリーを構築します。目的のリレーションを生成する最もコストの低い Path を選択し、それをエグゼキュータに渡す Plan に変換します。（Path ツリーと Plan ツリーの間にはほぼ 1 対 1 の対応関係がありますが、Path ノードは計画中に不要な情報を省略し、計画には必要だがエグゼキュータには不要な情報を含みます。）

オプティマイザは、クエリで使用される各ベースリレーションに対して RelOptInfo 構造体を構築します。ベースリレーションは、プリミティブテーブルであるか、プランナの個別の再帰呼び出しを通じて計画されるサブクエリのサブセレクトです。RelOptInfo は、計画中に検討される各結合リレーションに対しても構築されます。結合リレーションは単にベースリレーションの組み合わせです。特定のベースリレーションのセットに対しては、結合 RelOptInfo は 1 つしか存在しません。たとえば、{A B C} という結合は、A と B を先に結合してから C を追加した場合でも、B と C を先に結合してから A を追加した場合でも、同じ RelOptInfo によって表されます。これらの結合リレーションを構築するさまざまな手段は、Path として表されます。各 RelOptInfo に対して、そのリレーションのスキャンや結合を実装する妥当な方法を表す Path のリストを構築します。リレーションに対するすべての妥当な Path を検討したら、プランナのコスト見積もりに基づいて最もコストの低いものを選択します。最終的なプランは、クエリのすべてのベースリレーションを含む RelOptInfo の最もコストの低い Path から導出されます。

プリミティブテーブルリレーションの可能な Path には、通常のシーケンシャルスキャン、テーブルに存在するインデックスに対するインデックススキャン、および 1 つ以上のインデックスを使用したビットマップインデックススキャンが含まれます。関数 RTE などの特殊な RTE 型には、可能な Path が 1 つしかない場合があります。

結合は常に 2 つの RelOptInfo を使用して発生します。一方が外部（outer）、もう一方が内部（inner）となります。外部リレーションは内部リレーションの値のルックアップを駆動します。ネステッドループでは、内部リレーションの値のルックアップは、外部タプルごとに内部パスを 1 回スキャンして一致する内部行を見つけることで行われます。マージ結合では、内部行と外部行が順序付けられ、順序通りにアクセスされるため、結合全体を実行するのに必要なスキャンは 1 回だけです。内部パスと外部パスの両方が同期してスキャンされます。（マージ結合において内部と外部の間に大きな違いはありません...）ハッシュ結合では、まず内部リレーションがスキャンされてすべての行がハッシュテーブルに入力され、次に外部リレーションがスキャンされ、各行に対して結合キーをハッシュテーブルでルックアップします。

結合リレーションの Path は実際にはツリー構造であり、最上位の Path ノードが最後に適用された結合メソッドを表します。これには、2 つの入力リレーションに使用されるスキャンまたは結合メソッドを表す左および右のサブパスが含まれます。


結合ツリーの構築
----------------------

オプティマイザは、クエリを実行する方法をほぼ網羅的に探索することで、最適なクエリプランを生成します。最適な Path ツリーは、再帰的なプロセスによって見つけられます。

1) クエリ内の各ベースリレーションを取り出し、それに対する RelOptInfo 構造体を作成します。シーケンシャルスキャンやインデックススキャンなど、リレーションにアクセスする潜在的に有用な各方法を見つけ、それらの方法を表す Path を作成します。特定のリレーションに対して作成されたすべての Path は、その RelOptInfo.pathlist に配置されます。（実際には、明らかに劣る代替 Path は pathlist に入る前に破棄されます。pathlist に最終的に残るのは、リレーションの潜在的に有用な各ソート順序とパラメータ化を生成する最もコストの低い方法です。）また、このリレーションに関与するすべての結合句を含む RelOptInfo.joininfo リストを作成します。たとえば、WHERE 句 "tab1.col1 = tab2.col1" は、tab1 と tab2 の両方の joininfo リストにエントリを生成します。

クエリ内にベースリレーションが 1 つしかない場合は、完了です。それ以外の場合は、ベースリレーションをどのように結合して単一の結合リレーションにするかを決定する必要があります。

2) 通常、明示的な JOIN 句は「フラット化」され、結合するリレーションのリストのみになります。ただし、FULL OUTER JOIN 句は決してフラット化されず、join_collapse_limit や from_collapse_limit の制限によってフラット化プロセスが停止された場合、他の種類の JOIN もフラット化されない可能性があります。その結果、任意の順序で結合されるリレーションのリストを含む計画問題に直面することになります。個々のアイテムは、兄弟リレーションと結合することを検討する前に結合する必要があるサブリストである可能性があります。これらのサブ問題をボトムアップで再帰的に処理します。結合リスト構造は可能な結合順序を制約しますが、各結合での結合実装メソッド（ネステッドループ、マージ、ハッシュ）を制約するものではなく、どのリレーションが外部または内部として検討されるかを指定するものでもないことに注意してください。Path を構築する際に、これらすべての可能性を検討します。実行可能な各結合メソッドに対して Path を生成し、最もコストの低い Path を選択します。

したがって、各計画問題に対して、ベースリレーションまたはサブ結合リストごとに構築された結合リレーションのいずれかであるリレーションのリストを持つことになります。プランナが適切と判断した任意の順序で、これらのリレーションを結合できます。標準（非 GEQO）プランナは、次のようにこれを行います。

使用可能な結合句が存在する各 RelOptInfo を他の各 RelOptInfo と結合することを検討し、そのような各ペアに対して可能な各結合メソッドの Path を生成します。（結合句を持たない RelOptInfo がある場合、結合句なしのデカルト積結合を生成するしかありません。そのため、そのリレーションを他の利用可能な各リレーションと結合することを検討します。しかし、結合句が存在する場合は、利用可能な結合句を使用する結合のみを検討します。外部結合や IN/EXISTS 句によって誘発される結合順序の制限もチェックされ、それらの制限によって結合句なしの結合が強制される場合に実行可能な結合順序が見つかるようにします。）

リスト内にリレーションが 2 つしかない場合は、完了です。結合 RelOptInfo に対して最もコストの低いパスを選択するだけです。3 つ以上ある場合は、結合 RelOptInfo 同士を結合して、3 つ以上のリストアイテムを表す結合 RelOptInfo を作成する方法を検討する必要があります。

結合ツリーは「動的計画法」アルゴリズムを使用して構築されます。最初のパス（すでに説明済み）では、正確に 2 つのリストアイテムを表す結合リレーションを作成する方法を検討します。2 番目のパスでは、正確に 3 つのリストアイテムを表す結合リレーションを作成する方法を検討し、次のパスでは 4 つのアイテム、というように続きます。最後のパスでは、すべてのリストアイテムを含む最終的な結合リレーションを作成する方法を検討します。当然のことながら、この最上位レベルでは結合リレーションは 1 つしか存在できませんが、下位レベルでは複数の結合リレーションが存在する可能性があります。各レベルで、最初のレベルで説明したのと同様に、可能であれば利用可能な結合句に従う結合を使用します。

例:

    SELECT  *
    FROM    tab1, tab2, tab3, tab4
    WHERE   tab1.col = tab2.col AND
        tab2.col = tab3.col AND
        tab3.col = tab4.col

    テーブル 1, 2, 3, 4 は次のように結合されます:
    {1 2},{2 3},{3 4}
    {1 2 3},{2 3 4}
    {1 2 3 4}

    SELECT  *
    FROM    tab1, tab2, tab3, tab4
    WHERE   tab1.col = tab2.col AND
        tab1.col = tab3.col AND
        tab1.col = tab4.col

    テーブル 1, 2, 3, 4 は次のように結合されます:
    {1 2},{1 3},{1 4}
    {1 2 3},{1 3 4},{1 2 4}
    {1 2 3 4}

これらの各例において、他の結合順序の可能性は結合句の欠如により除外されます。（実際には、EquivalenceClasses を使用することで、より多くの結合順序を可能にする追加の結合句を推論できますが、ここでは例の単純さを保つためにそれを無視しています。）

左深木プラン（上位結合の外部リレーションは結合リレーションだが、内部は常に単一のリストアイテム）、右深木プラン（外部リレーションは常に単一アイテム）、および茂み状（bushy）プラン（内部と外部の両方が結合リレーションになり得る）を検討します。たとえば、{1 2 3 4} を構築する際、{1 2 3} と {4} を結合（左深木）、{4} と {1 2 3} を結合（右深木）、{1 2} と {3 4} を結合（茂み状）などを検討します。結合ツリーのスキャンコードはこれらの潜在的な結合の組み合わせを一度に 1 つずつ生成しますが、同じ結合ベースリレーションセットを生成するすべての方法は同じ RelOptInfo を共有するため、同等の結合リレーションを生成する異なる結合の組み合わせから生成されたパスは add_path() で競合します。

動的計画法のアプローチには、すぐには明らかではない重要な特性があります。それは、あるリレーションを含むリレーションのパスを構築する前に、そのリレーションのすべてのパスの構築を完了するということです。これは、上位レベルのリレーションがそれを知る必要がある前に、各リレーションの「最もコストの低いパス」を確実に特定できることを意味します。また、同じリレーションに対する別のパスの方が優れていると判断したときに、そのパスを安全に破棄できます。上位レベルの結合パスでそのパスへの参照がすでに存在している可能性を心配する必要はありません。これがないと、パスのメモリ管理ははるかに複雑になります。

最終的な結合リレーションを構築したら、それに対する最もコストの低いパス、または目的の順序を持つ最もコストの低いパス（それが他の最もコストの低いパスにソートを適用するよりも安価な場合）のいずれかを使用します。

クエリに片側外部結合（LEFT または RIGHT 結合）が含まれている場合、あるいはセミ結合やアンチ結合に変換された IN または EXISTS WHERE 句が含まれている場合、可能な結合順序の一部が不正になる可能性があります。これらは、join_is_legal がそのような「特殊な」結合のサイドリストを参照し、提案された結合が不正かどうかを確認することで除外されます。（同じ参照により、有効な結合に対してどの結合スタイルを適用すべきか、つまり JOIN_INNER、JOIN_LEFT なども確認できます。）


有効な外部結合の最適化
------------------------------

プランナによる外部結合の並べ替え処理は、以下の恒等式に基づいています。

1.	(A leftjoin B on (Pab)) innerjoin C on (Pac)
	= (A innerjoin C on (Pac)) leftjoin B on (Pab)

ここで Pac は A と C を参照する述語などです（この場合、Pac が B を参照することはあり得ません。さもなければ変換は無意味になります）。

2.	(A leftjoin B on (Pab)) leftjoin C on (Pac)
	= (A leftjoin C on (Pac)) leftjoin B on (Pab)

3.	(A leftjoin B on (Pab)) leftjoin C on (Pbc)
	= A leftjoin (B leftjoin C on (Pbc)) on (Pab)

恒等式 3 は、述語 Pbc がすべての B の NULL 行に対して失敗しなければならない（つまり、Pbc が B の少なくとも 1 つの列に対して厳密である）場合にのみ成立します。Pbc が厳密でない場合、最初の形式では C 列が非 NULL の行が生成される可能性がありますが、2 番目の形式ではそれらのエントリが NULL になる可能性があります。

RIGHT JOIN は、2 つの入力テーブルを入れ替えた後の LEFT JOIN と同等であるため、同じ恒等式が右結合にも適用されます。

うまくいかないケースの例として、内部結合を外部結合の NULL 可能側へ移動したり、そこから外へ移動したりするケースがあります。

	A leftjoin (B join C on (Pbc)) on (Pab)
	!= (A leftjoin B on (Pab)) join C on (Pbc)

SEMI 結合は少し異なる動作をします。セミ結合は、別のセミ結合、左結合、またはアンチ結合の左側に再結合したり、そこから外へ再結合したりすることはできますが、右側への再結合はできません。同様に、内部結合、左結合、またはアンチ結合は、セミ結合の左側に再結合したり、そこから外へ再結合したりすることはできますが、右側への再結合はできません。ANTI結合は、LEFT結合とほぼ同様に動作しますが、Cへの結合がアンチ結合である場合（Pbcが厳密である場合や、もう一方の結合がLEFT結合であるかアンチ結合であるかに関わらず）、identity 3は失敗します。そのため、関連する句が厳密であっても、アンチ結合をLEFT結合やアンチ結合のRHS（右辺）の内外に並べ替えることはできません。

現在のコードでは、FULL結合の並べ替えは一切試みられません。FULL結合の順序付けは、jointreeを「joinlist」表現に変換する際にFULL結合ノードを折り畳まないことで強制されます。他の種類のJOINノードは通常、結合順序の検索に完全に参加できるように折り畳まれます。不正な結合順序が生成されるのを防ぐため、プランナは内部結合以外の結合ごとにSpecialJoinInfoノードを作成し、join_is_legalがこのリストをチェックして提案された結合が妥当かどうかを判断します。

SpecialJoinInfoノードに格納されるのは、外部結合を形成するために結合の各側に必要な最小限のRelidセットです。これらは最小値であることに注意してください。OJ（外部結合）の構文上のリレーションに他のリレーションを結合することが妥当である可能性があるため、明示的な最大値はありません。identity 1および2に従い、FULL結合以外の結合はOJの左辺に自由に結合できますが、場合によっては右辺に結合できないことがあります。そのため、join_is_legalによって強制される制限は、提案された結合がRHS境界の内側または一部内側にあるリレーションを境界の外側のリレーションと結合できないというものです。ただし、その結合がidentity 3を使用してSpecialJoinInfoのRHSに結合できるLEFT結合である場合は例外です。

最小Relidセットの使用にはいくつかの落とし穴があります。例えば、以下のようなクエリを考えてみます。
	A leftjoin (B leftjoin (C innerjoin D) on (Pbcd)) on Pa
ここで、PaはB/C/Dについて全く言及していません。この場合、単純な計算では、上側のleftjoinの最小LHSは{A}、最小RHSは{C,D}となります（内部結合がleftjoinのRHSから外へ結合できないことを知っており、そのRelidをleftjoinの最小RHSに含めることでそれを強制するため）。そして、下側のleftjoinの最小LHSは{B}、最小RHSは{C,D}となります。このような情報が与えられると、join_is_legalは上側の結合を下側の結合のRHSに結合しても問題ないと判断し、クエリを以下のように変換してしまいます。
	B leftjoin (A leftjoin (C innerjoin D) on Pa) on (Pbcd)
これは全く誤った結果をもたらします。これを防ぐために、上側の結合の最小RHSにBを含めることを強制します。これはおそらく過度に制限的ですが、そのようなケースは頻繁には発生しないため、より複雑なシステムを開発する価値があるかどうかは不明です。


サブクエリの引き上げ (Pulling Up Subqueries)
-------------------------------------------

前述の通り、FROM句に現れるサブクエリは個別に計画され、外側のクエリの計画中には「ブラックボックス」として扱われます。これは、サブクエリが集約、GROUP、DISTINCTなどの機能を使用する場合に必要です。しかし、サブクエリが単純なスキャンや結合である場合、それをブラックボックスとして扱うと、計画全体の検索空間の一部として考慮する場合と比較して、質の低い計画が生成される可能性があります。そのため、計画プロセスの開始時に、プランナは単純なサブクエリを探し、それらをメインクエリのjointreeに引き上げます。

サブクエリを引き上げると、結合ツリーの最上部より下にFROMリストの結合が現れることがあります。各FROMリストは、前述の動的計画法による検索手法を使用して計画されます。

サブクエリの引き上げによって、あるFROMリストが別のFROMリストの直接の子として生成された場合、それら2つのFROMリストをマージできます。それが完了すると、サブクエリは外側のクエリの完全に不可欠な一部となり、結合ツリーの検索空間を制限することはなくなります。しかし、動的計画法による検索は考慮するFROMアイテムの数に対して指数関数的な実行時間を要するため、これにより計画時間が大幅に増加する可能性があります。そのため、マージの結果、1つのリスト内のFROMアイテムが多すぎるようになる場合は、FROMリストをマージしません。


VarsとPlaceHolderVars
---------------------

Varノードは、テーブル列参照を構文木で表現した単純なものです。しかし、外部結合が存在する場合、その概念は見た目よりも微妙です。VarをNULLにする可能性のある外部結合の「上」と「下」で、Varの値を区別する必要があります。例として以下を考えます。

	SELECT * FROM t1 LEFT JOIN t2 ON (t1.x = t2.y) WHERE foo(t2.z)

（foo()は厳密ではないと仮定します。そうでないと、left joinを単純な結合に簡約できないためです。）単純な実装では、foo(t2.z)の呼び出しをt2のスキャンまでプッシュダウンしようとするかもしれませんが、これは正しくありません。なぜなら、(a) NULL拡張された結合行に対してfoo()が実際に受け取るべき値はNULLであり、(b) foo()がfalseを返す場合、t1の行を結合から完全に除外すべきであり、NULL拡張されたt2の行と共に発行すべきではないからです。一方で、クエリが以下のようであれば、その呼び出しをスキャンレベルまでプッシュダウンすることは正しく（かつ望ましい）ことです。

	SELECT * FROM t1 LEFT JOIN t2 ON (t1.x = t2.y AND foo(t2.z))

これは、left joinのON句内の「t2.z」を、JOIN句の外側の「t2.z」とは異なる値として考慮する動機となります。前者はリレーションスキャンレベルで見られるt2.zと同一視できますが、後者はできません。

別の例は、EquivalenceClasses（後述）に関連して発生します。

	SELECT * FROM t1 LEFT JOIN t2 ON (t1.x = t2.y) WHERE t1.x = 42

この場合、EquivalenceClassメカニズムを使用して「t2.y = 42」を導出し、t2のスキャンに対する制限句として使用したいと考えます。（yが42と異なるt2の行はクエリ結果に影響を与えないため、これは機能します。）しかし、すべての結合行においてt2.yがt1.xと等しくなると結論付けるのは誤りです。この問題の解決策の一部は、ON句内の「t2.y」がt2.yのリレーションスキャンレベルの値を示すと見なすことです。結合行におけるyの値（t1.xと等しいのではなくNULLになる可能性がある）を示すとは見なしません。

そのため、Varノードには「varnullingrels」が装飾されています。これは、クエリ内のそのVarが現れる場所でVarをNULLにする可能性のある外部結合のレンジテーブルインデックスのセットです。（順序付きリストではなくセットを使用するのは、どの結合が値をNULLにしたかは重要ではなく、異なる外部結合順序を考慮する際に表現を変更する必要がないためです。）上記の例では、t1.xのすべての出現は空のvarnullingrelsを持ちます。left joinはt1をNULLにしないためです。JOIN ON句内のt2参照も空のvarnullingrelsを持ちます。しかし、JOIN句の外側では、t2を参照するすべてのVarは、JOINのレンジテーブルエントリ（RTE）のインデックスを含むvarnullingrelsを持ちます。これにより、それらはスキャンレベルで見られるt2の値とは潜在的に異なると理解されます。WHERE句内のt2.zにJOINのRTインデックスをラベル付けすることで、そのfoo(t2.z)の出現をt2のスキャンレベルまでプッシュダウンできないことを認識できます。その値はスキャンレベルでは評価できず、結合が行われた後にのみ評価できるからです。

LEFTおよびRIGHT外部結合の場合、結合のNULL許容側から来るVarのみがその結合のRTインデックスでマークされます。FULL結合の場合、両方の入力からのVarがマークされます。（このようなマーク付けでは、VarがFULL結合のどちら側から来たのかを判断することはできませんが、その情報は必要に応じて他の場所で見つけることができます。）

概念的には、空でないvarnullingrelsを持つVarは、以下のように考えることができます。
	CASE WHEN これらの外部結合のいずれかがNULL拡張行を生成した
	  THEN NULL
	  ELSE 列のスキャンレベルの値
	  END
これはあくまで概念的なものであり、実際にこのような計算が明示的に行われることはありません。完成した計画では、スキャンレベルのプランノードに現れるVarは実際のテーブル列値を表しますが、上位レベルのVarは常に下位レベルのプランノードの出力への参照です。結合ノードがNULL拡張行を発行する場合、入力から値をコピーするのではなく、関連する出力列に対して単にNULLを返します。この計算を明示的に行う必要がないため、外部結合のどちら側がNULL拡張されたかを区別する必要はありません。これはFULL結合の場合には不可欠な情報となるはずですが、不要です。

（前述の）外部結合のidentity 3は、この状況を少し複雑にします。
	A leftjoin (B leftjoin C on (Pbc)) on (Pab)
という形式では、句PbcおよびPab内のすべてのVarは空のvarnullingrelsを持ちますが、もし
	(A leftjoin B on (Pab)) leftjoin C on (Pbc)
から始めた場合、パーサはPbcのBのVarをA/B結合のRTインデックスでマークするため、この形式は最初の形式とは人工的に異なるものになります。議論のために、このマーク付けを星印で表すと、
	(A leftjoin B on (Pab)) leftjoin C on (Pb*c)
となります。これに対処するため、これらの結合を交換することが妥当であると検出したら、パーサが作成したBのVarから最初の結合のRTインデックスを削除または追加することで、そのON句のPbc形式とPb*c形式の両方を生成します。BとCを結合する計画ステップのパスを生成する際、AがすでにBに結合されているかどうかに応じて適切な形式のみを関連する結合条件として含めます。

また、identity 3により「left joinのRTインデックス」自体が少し曖昧な概念になることにも注意が必要です。各結合RTEの構文上のスコープは、パーサによって生成された形式に依存するためです。これを解決するために、left joinのアイデンティティは、その最小の右辺入力リレーションのセットによって決定されると考えます。identity 3で許可される両方の形式において、最初の結合は最小RHSをBとし、2番目の結合は最小RHSをCと識別できます。

もう1つ注意すべき点は、ネストされたJOIN句の外側に現れるCのVarは、元のパーサ入力がidentity 3の最初の形式であった場合、両方のleft joinによってNULL化されるとマークされますが、パーサ入力が2番目の形式であった場合、そのようなVarは2番目の結合によってのみNULL化されるとマークされることです。これは実際には意味論的な問題ではありません。そのようなVarはクエリの上部全体で同じようにマークされるため、すべてequal()と見なされ、これは正しいです。また、JOIN ON句内やこれらの結合の下に現れるCのVarとはequal()とは見なされません。しかし、結合リレーションの出力を表すVarを作成する際は、それらがクエリ上部の既存のVarとequal()と見なされるように、varnullingrelsが構文上の結合順序と一致する値に設定されていることを確認する必要があります。

外部結合は、サブクエリの引き上げの処理も複雑にします。以下を検討してください。

	SELECT ..., ss.x FROM tab1
	  LEFT JOIN (SELECT *, 42 AS x FROM tab2) ss ON ...

前述のようにサブクエリを引き上げたいと考えますが、最上位のSELECTリスト内の「ss.x」Varを定数42に置き換えるだけでは不十分です。それでは、NULL拡張された結合行でNULLを発行するのではなく、常に42を発行することになってしまいます。

これを解決するために、PlaceHolderVarsという概念を導入します。
PlaceHolderVarはVarに似ており、その値はリレーションスキャンレベルで発生し、その後、上位レベルの外部結合によってNULLを強制される可能性があります。そのため、PlaceHolderVarはVarと同様にNULL化リレーションIDのセットを持ちます。元の値がテーブルから来るVarとは異なり、PlaceHolderVarの元の値はクエリによって決定される式（この例では「42」）によって定義されます。そのため、PlaceHolderVarをその式を子として持つノードとして表現します。サブクエリの引き上げが、空でないvarnullingrelsを持つサブクエリ参照Varを、単なるVarではない式に置き換える必要がある場合に、PlaceHolderVarを挿入します。（置換式が引き上げられたVarである場合は、置換されたVarのvarnullingrelsをそのセットに追加するだけで済みます。また、置換されたVarが空のvarnullingrelsを持つ場合、PlaceHolderVarは不要です。値をNULLにするものは何もないため、引き上げられた式をそのまま使用して問題ありません。）完成した計画では、PlaceHolderVarは評価されるべきプランレベルで含まれる式そのものになり、上位レベルの出現は下位プランレベルの出力列へのVar参照に置き換えられます。これにより、通常のVarと同様に、外部結合で適切な場合に値がNULLになります。したがって、PlaceHolderVarがプランナの外で見られることはありません。PlaceHolderVars (PHV) は、別の点でも Vars より複雑です。
それらの元の値は、ベースレベルのリレーションスキャンではなく、結合（join）の時点で計算する必要がある場合があります。これは、引き上げられたサブクエリに結合が含まれている場合に発生する可能性があります。このため、PHV は、使用される前に計算されることを保証するために、本来存在しないはずの結合順序制約を作成することがあります。PHV の式には LATERAL 参照を含めることもでき、これについては後述する複雑な問題が加わります。


リレーションの識別と Qual 句の配置
-------------------------------------------------

WHERE または JOIN/ON から得られる qual 句は、その句で使用されるすべてのリレーションを含む最も低いスキャンレベルまたは結合レベルで適用できます。この目的のために、varnullingrels または phnullingrels にリストされている外部結合（outer join）はその句で使用されているとみなします。なぜなら、そのような Vars が NULL になったかどうかを判断するまで、qual の結果を正しく計算できないからです。

この一般原則の唯一の例外は、非退化的な外部 JOIN/ON qual（結合の非 NULL 側を参照するもの）は、たとえ NULL 側を参照していなくても、その結合より下で適用することはできないという点です。それを非 NULL 側にプッシュダウンすると、行が NULL 拡張された行として現れるのではなく、結合結果から消滅してしまう結果になります。これに対処するため、そのような qual を識別した際、その結合の最小入力 relid セットを、その qual が使用するとみなされるリレーションのセットに人為的に追加し、その結合レベルで正確に評価されるように強制します。リレーションを全く言及しない外部結合の qual についても同様です。

外部結合を実行する結合プランノードに qual 句を付与する場合、その qual 句が varnullingrels または phnullingrels セットのいずれにおいてもその外部結合を参照していない場合は「結合句（join clause）」（つまり、結合が NULL 拡張を実行する前に適用される）とみなされ、その外部結合を参照している場合は「フィルタ句（filter clause）」（NULL 拡張後に適用される）とみなされます。元々その外部結合の JOIN/ON に現れていた qual 句は、パーサーがその Vars のいずれも外部結合を参照しているとマークしないため、最初のカテゴリに分類されます。元々上位の ON 句や WHERE 句から来た qual 句は、NULL 側の Vars を参照していれば、パーサーによってそのようにマークされるため、外部結合を参照しているとみなされます。しかし、そのような qual が NULL 側の Vars を全く参照していない場合は、非 NULL 側にプッシュダウンしても問題ないため、そもそも結合ノードには付与されません。

これらのことから、プランナー内では、結合リレーションを、それらが含むベースリレーションの RT インデックスと外部結合の RT インデックスのセットによって識別することになります。そうすることで、qual 句によって使用されるリレーションのセットを結合リレーションの relid セットと直接比較し、句をどこに配置すべきかを確認できます。これらの識別セットは一意です。なぜなら、ベースリレーションの任意の集合に対して、その集合を結合する過程で実行される外部結合の有効なセットは（前述のように適用順序は異なる可能性がありますが）1つしかないからです。

SEMI 結合には RT インデックスがありません。なぜなら、それらはパーサーではなくプランナーによって作成されたアーティファクトだからです。（それらのために rangetable エントリを作成することもできますが、現時点ではその必要はないようです。）これは qual の配置には問題を引き起こしません。なぜなら、セミ結合の NULL 側は結合の上位からは参照できないため、varnullingrels や phnullingrels でそれを引用する必要がないからです。また、結合リレーションの識別にも問題を引き起こしません。セミ結合が完了したかどうかは、結合に含まれるベースリレーションのセットに暗黙的に含まれているからです。

いつものように、外部結合の同一性 3（outer join identity 3）が問題を複雑にします。もし
	(A leftjoin B on (Pab)) leftjoin C on (Pbc)
から始めた場合、パーサーはこれらの結合の上位に現れる C の Vars を B/C 結合の RT インデックスでマークします。ここで、もし
	A leftjoin (B leftjoin C on (Pbc)) on (Pab)
へと変換すると、そのような Vars のみを使用する句はプッシュダウンされ、下位の B/C 結合で（結合句ではなく）フィルタ句として適用できるように見えるかもしれません。しかし、*これは正しい答えを導かない可能性があります*。なぜなら、その句は、A/B 結合が実行されると NULL に置き換えられる C Var に対して非 NULL 値を見てしまう可能性があるからです。これに対処するため、プッシュダウンされた結合は B/C 結合の作業を完全には実行しておらず、したがってその relid セットにその外部結合の relid を含める資格はないとみなします。A/B 結合を形成する際、両方の外部結合の relid がその relid セットに追加され、その後、上位の句が正しい結合レベルで適用されます。（同一性 3 が逆方向に適用される場合には問題がないことに注意してください。2 番目の形式から始めた場合、上位の C Vars は両方の外部結合 relid でマークされるため、どちらの結合が後に適用されても、それより下位に落ちることはありません。）同様に、プッシュダウンされた結合の出力を表す Vars は、上位の結合が実行されるまで、その結合に対する nullingrel ビットを獲得しません。

qual 句の配置にはもう1つ複雑な点があります。これは、前述のように外部結合句の複数のバージョンを作成した場合（つまり、外部結合の同一性 3 で見られる同じ句の "Pbc" 形式と "Pb*c" 形式の両方がある場合）に発生します。外部結合を形成する際、冗長なバージョンの句のうち1つだけを適用したいと考えます。A/B 結合をまだ計算せずに B/C 結合を形成している場合、"Pb*c" 形式の必要な relid セットには入力に含まれていない A/B 結合 relid が含まれているため、それを拒否するのは簡単です。しかし、A/B の後に B/C を形成する場合、そのテストでは両方の形式の句が適用可能に見えてしまいます。句の "Pbc" 形式が、直接アクセスできなくなったリレーション B を参照していることに気づくために、より詳細に調べる必要があります。このようなチェックはリレーションごとの RelOptInfo.nulling_relids データを使用して実行できますが、異なる結合パスを検討するたびに何度も繰り返すのは非常にコストがかかります。これを単純かつ確実にするために、冗長な句の各バリアントバージョン（クローン）に対して "incompatible_relids" セットを計算します。クローン句は、incompatible_relids にリストされている外部結合 relid のいずれかが現在の結合より下位ですでに計算されている場合は、適用されるべきではありません。

オプティマイザは、パスによって返されるタプルの順序付けを考慮することに多くの時間を費やします。これが有用な理由は、パスのソート順序を知ることで、そのパスをマージ結合の左または右の入力として使用し、明示的なソートステップを回避できる可能性があるためです。ネステッドループやハッシュ結合は入力の順序を気にしませんが、マージ結合は適切に順序付けられた入力を必要とします。したがって、最適化プロセス中に生成されるすべてのパスには、上位レベルのマージで使用される可能性があるため、（既知の範囲で）ソート順序がマークされます。最終的なパスがすでに正しい順序になっている場合、ユーザーの ORDER BY 句を実装するために明示的なソートステップを回避することも可能です。そのため、ソート順序はトップレベルであっても重要です。grouping_planner() は、目的の順序と一致するソート順序を持つ最もコストの低いパスを探し、そのコストを、全体で最もコストの低いパスを使用して明示的なソートを行う場合のコストと比較します。

特定の RelOptInfo に対してパスを生成する際、同じかそれ以上のソート順序を持つ別の既知のパスよりもコストが高いパスは破棄します。特定のソート順序を実現する唯一の既知の方法であるパス（明示的なソートを行わない場合）は、決して破棄しません。このようにすることで、次の上位レベルでは、入力に対して保持されたパスのいずれかから選択できるため、ソートなしでマージ結合を構築する最大限の自由度が得られます。

EquivalenceClasses
------------------

クエリの qual 句を deconstruct_jointree() でスキャンする際、マージ結合可能な等価句 A = B を探します。見つかった場合、それらが等しいことを記録するために、式 A と B を含む EquivalenceClass を作成します。その後、別の等価句 B = C が見つかった場合は、既存の {A B} の EquivalenceClass に C を追加します。これには、既存の2つの EquivalenceClass のマージが必要になる場合があります。スキャンの終了時には、推移的にすべて等しいことがわかっている値の集合が得られます。したがって、値の任意のペアの比較を制限句または結合句として使用できます（もちろん、これらの値がスキャンまたは結合時に利用可能な場合）。さらに、すべての比較ではなく、そのような比較を1つだけテストすれば十分です。そのため、等価句は標準の qual 分布プロセスから削除されます。代わりに、制限句または結合句のリストを準備する際、各 EquivalenceClass が句に寄与できるかどうかを調べ、寄与できる場合は比較に適した値のペアを選択します。例えば、A のリレーションを C のリレーションと結合しようとしている場合、元のクエリには明示的に現れていなくても、句 A = C を生成できます。これにより、デカルト積結合が必要であるとして拒否されていた可能性のある結合パスを探索できる場合があります。

EquivalenceClass に擬似定数式（現在のクエリレベルの Vars や Aggs を含まず、揮発性関数も含まないもの）が含まれることがあります。この場合、結合句を動的に生成するポリシーには従わず、クラスの変数メンバーのいずれかが最初に計算できる場所で、制限句 "var = const" を動的に生成します。例えば、A = B かつ B = 42 の場合、実質的に制限句 A = 42 および B = 42 を生成し、リレーションが結合される際に結合句 A = B を明示的にテストする必要はなくなります。事実上、すべてのクラスメンバーをリレーションスキャンレベルでテストできるため、結合テストは不要になります。

EquivalenceClass の正確な技術的解釈は、そのメンバー値の複数が計算可能な任意のプランノードにおいて、値がすべて等しくない出力行はクエリ結果に影響を与えずに破棄できる、という主張です。（プランのすべてのレベルで EquivalenceClasses を強制する必要があるため、結合は子ノードによって計算可能な値の等価性を再チェックする必要はありません。）

しかし、外部結合はこの状況をかなり複雑にします。理論上は、外部結合条件に現れるマージ結合可能な等価句を EquivalenceClasses のソースとして使用できますが、深刻な困難があります。それは、得られた推論がどこでも有効とは限らないことです。例えば、以下の場合：

	SELECT * FROM a LEFT JOIN b ON (a.x = b.y AND a.x = 42);

b.y = 42 を安全に導出し、B のスキャンで使用できます。なぜなら、b.y = 42 を持たない B の行は結合結果に寄与しないからです。しかし、A のスキャンで a.x = 42 を適用することはできません。適用すると、結合結果に含まれるべき行が削除されてしまうからです。a.x = 42 を外部結合条件として適用することはできます（その場合、a.x = b.y もチェックする必要はなくなります）。ただし、これはまだ実装されていません。

関連する問題として、外部結合の下に現れる定数は、見た目ほど定数ではないという点があります。通常、"A = 1" と "B = 1" が見つかれば、A と B を同じ EquivalenceClass に入れても問題ありません。しかし、以下を検討してください。

	SELECT * FROM a
	  LEFT JOIN (SELECT * FROM b WHERE b.z = 1) b ON (a.x = b.y)
	WHERE a.x = 1;

a.x = b.z と結論付けるのは重大な誤りであるため、単一の EquivalenceClass {a.x b.z 1} を形成することはできません。

このことから、EquivalenceClasses は「結合ドメイン」（互いに内部結合されるリレーションの集合）内で適用されると考えることになります。（この目的のために、セミ結合は内部結合であるかのように扱うことができます。）トップレベルの結合ドメインがあり、クエリ内の各外部結合が、そのヌル許容側を含む新しい結合ドメインを作成します。完全結合は、各側に1つずつ、2つの結合ドメインを作成します。WHERE から生成された EquivalenceClasses はトップレベルの結合ドメインに関連付けられます。外部結合の ON 句から生成された EquivalenceClasses は、その外部結合によって作成されたドメインに関連付けられます。内部結合またはセミ結合の ON 句から生成された EquivalenceClasses は、構文的に最も深くネストされた結合ドメインに関連付けられます。

これらのドメインを定義することで、定数が同じ結合ドメイン内の句から来ている場合にのみ EquivalenceClass メンバーと一致すると考えることで、「それほど定数ではない定数」の問題を解決できます。上記の例では、{a.x 1} と {b.z 1} を別々の EquivalenceClasses として保持し、誤ってマージしないことを意味します。Vars（または Vars を含む式）については、異なる結合ドメインからの「同じ」列への参照は異なる varnullingrels を持つため、そもそも equal() ではないため、この点を心配する必要はありません。

将来的には、結合ドメインの概念により、マージ結合可能な外部結合条件を EquivalenceClasses のソースとして扱えるようになるかもしれません。その考え方は、そのようなクラスから導出された条件は、適切な結合ドメイン内にあるスキャンまたは結合でのみ強制されるというものです。ただし、詳細は見た目よりも複雑であるため、これはまだ実装されていません。

もう一つの有益な例は以下の通りです。

	SELECT *
	  FROM a LEFT JOIN
	       (SELECT * FROM b JOIN c ON b.y = c.z WHERE b.y = 10) ss
	       ON a.x = ss.y
	  ORDER BY ss.y;

EquivalenceClass {b.y c.z 10} を形成し、それによって C のスキャン中に c.z = 10 を、B のスキャン中に b.y = 10 を適用できるため、内部結合で句をチェックする必要はありません。左結合句 "a.x = ss.y"（実際には "a.x = b.y"）は等価句とは見なされないため、a.x を同じ EquivalenceClass に挿入しません。もし挿入した場合、誤って a.x = 10 と結論付けてしまいます。ただし、将来的に a.x = 10 を A のスキャンで適用しないようにできれば、それが可能になるかもしれません。原則として、EquivalenceClass が {B,C} 結合ドメイン内でのみ適用されることに注意すれば可能です。

また、ORDER BY 内の ss.y は実際には b.y*（つまり、b.y のヌル化された可能性のある形式）であるため、下位の EquivalenceClass の b.y メンバーと混同することはありません。したがって、ss.y が定数と等しいと誤って結論付けることはなく、もしそうであれば ORDER BY のためのソートは不要であると考えることになります（以下の PathKeys の議論を参照）。代わりに、b.y* のみを含む別の EquivalenceClass が存在し、これが要求されるソート順序を記述する PathKey の基礎となります。

また、このバリエーションも検討してください。

	SELECT *
	  FROM a LEFT JOIN
	       (SELECT * FROM b JOIN c ON b.y = c.z WHERE b.y = 10) ss
	       ON a.x = ss.y
	  WHERE a.x = 42;

依然として EquivalenceClass {b.y c.z 10} を形成し、さらに異なる結合ドメインに属する EquivalenceClass {a.x 42} があります。"a.x = b.y" を使用してこれらのクラスをマージすることはできません。しかし、その外部結合句を既存の EquivalenceClasses と比較し、導出句 "b.y = 42" を形成できます。これは下位の結合ドメイン内で有効な等価性として扱えます（そのドメインのどの行も b.y = 42 を持たなければ外部結合結果に寄与できないため）。これにより、下位の EquivalenceClass は {42 b.y c.z 10} となり、10 = 42 という矛盾が生じます。これにより、プランナーは B/C 結合を計算する必要がないと推論でき、結合ドメイン全体の結合結果を空に強制できます。（これは、単なる Const よりも Param 値が矛盾に関与することが多いため、実行時にチェックする必要があり、ゲーティング Result フィルタとして実装されます。結合ドメインを使用して、ゲーティング条件を配置する結合レベルを決定できます。）

恒等式 3 に従って外部結合を並べ替える際に、追加の複雑さがあります。そのような結合に対して考慮する2つの選択肢は以下の通りです。

	A leftjoin (B leftjoin C on (Pbc)) on (Pab)
	(A leftjoin B on (Pab)) leftjoin C on (Pb*c)

ここで、星印は B の Vars 上の varnullingrels マーカーを示します。Pbc がマージ結合可能な句である（または含む）場合、以下のようになります。

	A leftjoin (B leftjoin C on (b.b = c.c)) on (Pab)
	(A leftjoin B on (Pab)) leftjoin C on (b.b* = c.c)

b.b と c.c をリンクする EquivalenceClause を生成できますが、その後 b.b* と c.c もリンクしようとすると、b.b と b.b* が等しい（少なくともプランツリーの一部では）という無意味な結論に至ります。いずれにせよ、そのようなものから導出できる結論は大部分が重複しています。b.b* を含む条件はこの結合ネストの下では計算できず、計算可能な条件は b.b/c.c の組み合わせから得られるものと重複します。したがって、b.b と c.c をリンクする EquivalenceClause を生成することを選択し、"b.b* = c.c" は通常の句として扱います。

マージ結合で機能するソート順序を決定しやすくするため、各マージ結合可能な句にその左入力と右入力の EquivalenceClasses をマークします。等価句の場合、これらはもちろん同じ EquivalenceClass です。非等価のマージ結合可能な句（外部結合の修飾など）の場合、左入力と右入力に対して2つの別々の EquivalenceClasses を生成します。これにより、単一項目の等価「クラス」が作成される可能性がありますが、もちろんこれらは他の等価句が同じ式に関係していることが後に判明した場合にはマージの対象となります。

単一項目の EquivalenceClass を形成するもう一つの方法は、目的のソート順序を表す PathKey を作成する際です（下記参照）。これは、ORDER BY または GROUP BY キーがいずれの等価句にも言及されていない場合に発生します。そのようなクエリでソート順序について推論する必要があり、ソート順序の表現は EquivalenceClass に依存する PathKey であるため、EquivalenceClass を作成する必要があります。これは上記のケースとは少し異なります。なぜなら、そのような EquivalenceClass には集約関数や揮発性式が含まれる可能性があるからです。（揮発性関数を含む句は、そのトップ演算子がマージ結合可能であってもマージ結合可能とは見なされないため、揮発性式が EquivalenceClasses に入る方法はありません。集約は WHERE 内では完全に許可されていないため、マージ結合可能な句で見つかることはありません。）これは一貫した PathKey 表現を維持するための便宜的なものであり、そのような EquivalenceClass が他のものとマージされることはありません。特に、単一項目の EquivalenceClass {a.x} は a.x = a.x という主張を意味するものではないことに注意してください。これの実用的な効果は、a.x が NULL になり得るということです。EquivalenceClass は、btree opfamily OID のリストも保持しており、これがそのクラスが表す等価性が実際に何を「意味」するのかを決定します。EquivalenceClass に寄与するすべての等価句は、同じ opfamily セットに属する等価演算子を持たなければなりません。（注：ほとんどの場合、特定の等価演算子は1つのファミリーにのみ属しますが、複数のファミリーに属することも可能です。マージジョインの目的でいずれかのファミリーに属するインデックスを利用できるように、すべてのファミリーを追跡しています。）

同様の理由で、EquivalenceClass は、そのデータ型が照合順序（collation）を考慮する場合、特定の照合順序とも関連付けられます。

EquivalenceClass には「em_is_child」メンバーを含めることができます。これは、appendrel の親リレーションの Var を含むメンバーのコピーであり、等価な子リレーションの変数または式を含むように変換されたものです。これらのメンバーは EquivalenceClass の正式なメンバーでは*なく*、クラスの全体的なプロパティには一切影響しません。これらは、子リレーションの式と EquivalenceClass とのマッチングを簡素化するためだけに保持されています。EquivalenceClass に対するほとんどの操作では、子メンバーを見る必要はありません。


PathKeys
--------

PathKeys データ構造は、特定の Path によって生成されるタプルのソート順序について何が分かっているかを表します。パスの pathkeys フィールドは PathKey ノードのリストであり、n 番目の項目が結果の n 番目のソートキーを表します。各 PathKey には以下のフィールドが含まれます：

	* EquivalenceClass への参照
	* btree opfamily OID（EC 内のいずれかと一致する必要がある）
	* ソート方向（昇順または降順）
	* NULL 値の配置フラグ（先頭か末尾か）

EquivalenceClass はソート対象の値を表します。EquivalenceClass の様々なメンバーは opfamily に従って等しいと見なされるため、それらのいずれかでソートされたパスは、他のいずれのメンバーでソートされていても同様にソートされていると見なすことができます。これが、単一のメンバーではなく EquivalenceClass 全体を参照することの正当性です。

単一/ベースリレーションの RelOptInfo において、Paths はリレーションをスキャンする様々な方法と、その結果得られるタプルの順序を表します。シーケンシャルスキャン Path は NIL の pathkeys を持ち、既知の順序がないことを示します。インデックススキャンは、もしあれば、選択されたインデックスの順序を表す Path.pathkeys を持ちます。単一キーのインデックスは単一の PathKey リストを作成し、複数列のインデックスはキーとなるインデックス列ごとに1つの要素を持つリストを生成します。カバリングインデックスの INCLUDE 句で指定された非キー列は、インデックスの順序に影響を与えないため、リスト内に対応する PathKeys を持ちません。（実際には、インデックスは前方または後方にスキャンできるため、2つの可能なソート順序と、それが生成し得る2つの可能な PathKey リストが存在します。）

ビットマップスキャンは、その結果の全体的な順序について何も言えないため、NIL の pathkeys を持つことに注意してください。また、順序付けされていないタイプのインデックスに対するインデックススキャンも NIL の pathkeys を生成します。ただし、明示的なソートを行うことで常に pathkey を作成できます。Sort プランの出力に対する pathkeys は、単にソートキーフィールドと使用された順序演算子を表します。

結合を考慮すると、話はより興味深くなります。A.X = B.Y というマージ句を使用して A と B の間でマージジョインを行うとします。マージジョインの出力は X でソートされますが、同時に Y でもソートされます。これもまた、X と Y の両方を含む EquivalenceClass を参照する PathKey によって表すことができます。

さらに考えると、ネストループ結合もソートされた出力を生成できることが明らかになります。例えば、外部リレーション A と内部リレーション B の間でネストループ結合を行う場合、A に関連するすべての pathkeys は結合結果に対しても有効です。A からのタプルの順序を変更していないからです。さらに興味深いことに、A.X=B.Y という等価句があり、A.X が外部リレーション A の pathkey であった場合、B.Y が結合結果の pathkey であると断言できます。X は以前から順序付けされており、現在もそうであり、Y の結合値は X の結合値と等しいため、Y も順序付けされているはずです。これは、明示的なソートも Y 上のマージジョインも使用しなかった場合でも真です。（注：ハッシュ結合は外部リレーションの順序を保持するとは期待できません。エグゼキュータが結合を「バッチ処理」することを決定する可能性があるため、ハッシュ結合パスの pathkeys は常に NIL に設定されます。）

外部結合は、その NULL 値許容入力リレーションの順序を保持しません。順序内のランダムな位置に NULL を挿入する可能性があるためです。PathKey 表現においてこれを明示的に考慮する必要はありません。結合後の変数を表す PathKey には varnullingrel ビットが含まれ、結合前の値を表す PathKey とは等しくなくなるためです。

一般に、EquivalenceClass を pathkeys の基礎として使用することは正当化できます。なぜなら、複数の EquivalenceClass メンバーを含むリレーションをスキャンしたり、それぞれが EquivalenceClass メンバーを含む2つのリレーションを結合したりするたびに、EquivalenceClass から導出された制限句または結合句を適用するからです。これにより、EquivalenceClass にリストされている任意の2つの値が、スキャンまたは結合によって出力されるすべてのタプルにおいて実際に等しいことが保証され、したがって、タプルがそれらの値の1つによってソートされている場合、他の値によってもソートされていると見なすことができます。テスト句がマージ句として使用されるか、単に qpqual フィルタとして事後的に強制されるかは問題ではありません。

パスのソート順序に、パスの出力にまだ結合されていない変数を含む EquivalenceClass を参照する PathKey をラベル付けすることに特別な困難はないことに注意してください。そのようなエントリは（まだ）関連がないとして単に無視できます。これにより、結合計画プロセス全体を通じて同じ EquivalenceClasses を使用することが可能になります。実際、同一の PathKey オブジェクトを複数生成しないように注意することで、EquivalenceClasses と PathKeys の比較を単純なポインタ比較にまで削減できます。これは、add_path が競合するパスが同等にソートされているかどうかを判断する際に多数の PathKey 比較を行う必要があるため、非常に大きな節約になります。

Pathkeys は、潜在的な候補パスの pathkeys と簡単に比較できるため、達成したい順序を表すのにも役立ちます。そのため、SortGroupClause リストはオプティマイザ内で使用するために pathkeys リストに変換されます。

追加の改良として、正規化された pathkey リスト（ソート順序）が同じ EquivalenceClass を複数回言及しないように強制することができます。例えば、以下のすべてのケースにおいて、2番目のソート列は冗長です。なぜなら、最初のソート列に従って同じ値であるものを区別できないからです：
	SELECT ... ORDER BY x, x
	SELECT ... ORDER BY x, x DESC
	SELECT ... WHERE x = y ORDER BY x, y
ユーザーが直接 "ORDER BY x,x" と書くことはまずないでしょうが、等価クラスが考慮されると、このような冗長性はより起こりやすくなります。また、システムはマージジョインに必要なソート順序を計算する際に冗長な pathkey リストを生成することがあります。冗長性を排除することで、時間を節約し、計画を改善できます。プランナが同等の順序をより容易に認識できるようになるためです。

もう一つの興味深い特性は、基となる EquivalenceClass に定数が含まれている場合、その pathkey は完全に冗長であり、ソートする必要が全くないということです！すべての興味深い行は同じ値を含んでいるはずなので、ソートの必要はありません。ユーザーが "... WHERE x = 42 ORDER BY x" と書くことはまずないため、これは無意味に見えるかもしれませんが、特定のインデックス列がソート順序に関係ないことを認識できるようになります。例えば "... WHERE x = 42 ORDER BY y" の場合、(x,y) 上のインデックスをスキャンすると、ソートステップなしで正しく順序付けされたデータが生成されます。以前は、限られたコンテキストでこれを認識するために非常に見苦しいアドホックなコードを使用していましたが、pathkeys から定数 EC を破棄することで、これをクリーンかつ自動的に実現できるようになりました。


EquivalenceClasses と PathKeys の処理順序
-------------------------------------------------------

上記で触れたように、計画作成中の EquivalenceClasses と PathKeys の処理には特定のフェーズ順序があります。クエリの修飾句の初期スキャン中（deconstruct_jointree に続く reconsider_outer_join_clauses）、修飾句で見つかったマージ結合可能な句に基づいて EquivalenceClasses を構築します。このプロセスの終了時点で、異なる変数の等価性について知り得るすべての情報を把握しているため、その後 EquivalenceClasses がさらにマージされることはありません。その時点で EquivalenceClasses を「正規」と見なし、それらを参照する正規の PathKeys を構築することが可能です。この時点で、クエリの ORDER BY および関連する句のための PathKeys を構築します。（他の場所に出現しない順序付け式は新しい EquivalenceClasses の作成につながりますが、既存のクラスのマージにはつながらないため、正規性は失われません。）

すべての EquivalenceClasses はパス生成を開始する前に既知であるため、どのインデックスが関心対象であるかのガイドとしてそれらを使用できます。インデックスの列がいずれの EquivalenceClass にも言及されていない場合、そのインデックスのソート順序がクエリに役立つことはあり得ません。これにより、無関係なインデックスに対する create_index_paths() の処理の多くを短絡させることができます。

planner.c が query_planner の完了後にさらに EquivalenceClasses と PathKeys を構築するケースがいくつかあります。これらのケースでは、query_planner 中に考慮されなかったソート順序を表すために追加の EC/PK が必要となります。このような状況は最小限に抑えるべきです。query_planner がそのようなソート順序を生成するプランを返すことは不可能であり、明示的なソートが常に必要になることを意味するためです。現在、これは異なる順序付けを持つ複数のウィンドウ関数を含むクエリでのみ発生し、その場合はいずれにせよ追加のソートが必要になります。


パラメータ化されたパス
-------------------

WHERE A.X = B.Y のような句を使用して2つのリレーションを結合する素朴な方法は、次のようなネストループプランを生成することです：

	NestLoop
		Filter: A.X = B.Y
		-> Seq Scan on A
		-> Seq Scan on B

マージ結合やハッシュ結合を使用することでこれを改善できますが、それでも両方の入力リレーションをすべてスキャンする必要があります。A が非常に小さく B が非常に大きい場合で、B.Y にインデックスがあるなら、次のようなことを行う方がはるかに優れている可能性があります：

	NestLoop
		-> Seq Scan on A
		-> Index Scan using B_Y_IDX on B
			Index Condition: B.Y = A.X

ここでは、A からスキャンされる各行に対して、ネストループプランノードが A.X の現在の値を B のスキャンに渡すことを期待しています。これにより、インデックススキャンは A.X を呼び出しごとに定数として扱い、それをインデックスキーとして使用できます。これは B のすべてを取得することを回避できる唯一のプランタイプであり、A から来る行数が少ない場合、他のすべての考慮事項を圧倒します。（A が大きくなるにつれて、これは魅力的ではなくなり、最終的にはマージ結合やハッシュ結合が勝るようになります。そのため、何をすべきかを決定するためにすべての代替案のコストを計算する必要があります。）

パラメータ値を結合の中間層を通じて渡すことが有用な場合もあります。例えば：

	NestLoop
		-> Seq Scan on A
		Hash Join
			Join Condition: B.Y = C.W
			-> Seq Scan on B
			-> Index Scan using C_Z_IDX on C
				Index Condition: C.Z = A.X

すべての結合が単純な内部結合であれば、これは通常不要です。パラメータを提供するネストループノードの直下でパラメータが使用されるように結合を並べ替えることが可能だからです。しかし、外部結合が存在する場合、そのような結合の並べ替えは不可能な場合があります。

また、最下層のスキャンは複数の他のリレーションからのパラメータを必要とする場合があります。原則として、他のリレーションを先に結合して、すべてのパラメータが単一のネストループレベルから供給されるようにすることもできます。しかし、それらの他のリレーションに共通の結合句がない場合（スター型スキーマのクエリなどで一般的です）、プランナはそれらを直接結合することを検討しません。そのような場合、次のようなプランを作成できるようにする必要があります。NestLoop
    -> Seq Scan on SmallTable1 A
    NestLoop
        -> Seq Scan on SmallTable2 B
        -> Index Scan using XYIndex on LargeTable C
             Index Condition: C.X = A.AID and C.Y = B.BID

そのため、プランが強制的にこのような構造になるような結合順序の制約がない場合でも、A.AID を結合を通じて引き下げる（pass down）ことを検討すべきです。

バージョン 9.2 より前では、Postgres はこの種の nestloop クエリの計画と実行にアドホックな手法を使用していましたが、それらの手法では複数の結合レベルを通じてパラメータを引き渡すことはできませんでした。

このようなクエリを計画するために、現在では「パラメータ化されたパス（parameterized path）」という概念を使用しています。これは、そのパスではスキャンされないリレーションへの結合句を利用するパスのことです。上記の2番目の例では、以下のような可能性を表すパスを構築します。

	-> Index Scan using C_Z_IDX on C
		Index Condition: C.Z = A.X

このパスは、リレーション A によってパラメータ化されているとマークされます。（注：これは C に対する可能なアクセスパスの1つに過ぎません。通常のパラメータ化されていない seqscan や、他の可能性も依然として存在します。）パラメータ化のマーカーは、そのパスを B と結合することを妨げないため、結合リレーション {B C} に対して生成されるパスの1つは以下を表すことになります。

	Hash Join
		Join Condition: B.Y = C.W
		-> Seq Scan on B
		-> Index Scan using C_Z_IDX on C
			Index Condition: C.Z = A.X

このパスは依然として A によってパラメータ化されているとマークされます。{B C} を A と結合して完全な結合ツリーを形成しようとする際、このようなパスは nestloop 結合の内部側としてのみ使用でき、他の結合タイプでは無視されます。したがって、上記のようなクエリプランを表す結合パスが形成され、パラメータ化されていないスキャンから構築されたパスと通常通り競合することになります。

特定のリレーションに対するすべての通常のパスは（すべて同じ制限句のセットを適用しなければならないため）同じ行セットを生成しますが、パラメータ化されたパスは通常、より少ない行を生成します。これは、処理すべき追加の句が存在するためです。つまり、生成される行数を追加の評価指標として考慮しなければなりません。他のパスよりもコストが高くても生成行数が少ないパスは、中間的な結合レベルで作業を削減できる可能性があるため、保持しておく必要があります。（パラメータのソースと即座に結合される場合は、何も削減されません。）

コスト見積もりのルールを比較的単純に保つため、同じパラメータ化（つまり、パラメータを提供する外部リレーションのセットが同じ）を持つ特定のリレーションに対するすべてのパスは、同じ行数見積もりを持たなければならないという実装上の制限を設けています。これは、そのような各パスが、指定された外部リレーションで利用可能な「すべての」結合句を適用することを要求することで正当化されます。例えば、異なるパスがインデックス句として使用する結合句を選択するかもしれませんが、その場合でも、同じ外部リレーションから利用可能な他のすべての結合句をフィルタ条件として適用しなければならず、結果として返される行セットは一定に保たれます。この制限は完成したプランの品質を低下させることはありません。これは、移動可能な結合句を常に可能な限り低い評価レベルに押し下げるべきであるということを意味しており、いずれにせよ良いことです。この制限は、特に add_path_precheck における結合パスの事前フィルタリングをサポートするのに役立ちます。このルールがなければ、行数見積もりを計算する前にパラメータ化されたパスを拒否することはできず、事前フィルタメカニズムの価値が大幅に低下してしまいます。

計画時間を制限するために、不当に多数のパラメータ化されたパスを生成することを避けなければなりません。これを行うために、インデックススキャンに対してのみパラメータ化されたリレーションスキャンパスを生成し、さらに、適切な結合句が利用可能なインデックスに対してのみ生成するようにしています。また、結合計画には、考慮されるパラメータ化されたパスの数を制限しようとするヒューリスティックも存在します。

特に、パラメータ化された結合ステップ（下位の結合の入力にパラメータを提供する nestloop の下で発生するもの）については、マージ結合よりもハッシュ結合を優先するという意図的なポリシー決定がなされています。マージ結合を完全に無視するわけではありませんが、joinpath.c はパラメータ化された入力を持つ潜在的なマージ結合の空間を完全には探索しません。また、add_path はパラメータ化されたパスを pathkeys を持たないものとして扱うため、コストと行数のみで競合し、特別なソート順序を生成することによる優先権は得られません。これにより、明示的なソートステップなしでマージを実行するのに役立つ可能性があったパスを破棄してしまう可能性があるため、マージ結合に対してさらなるバイアスが生じます。パラメータ化されたパスは最終的に nestloop の内部で使用されなければならず、そのソート順序は重要ではないため、これらの選択はクエリの最終的な出力順序に対する要件には影響しません。単に、より低いレベルでマージ結合を使用することを難しくするだけです。その分の計画作業の節約が、これを正当化します。

同様に、パラメータ化されたパスは、add_path において起動コストが安いという理由で優先されることは通常ありません。nestloop の内部にあるときはほとんど価値がないため、その理由だけで余分なパスを保持する価値はないと考えられます。例外として、SEMI 結合または ANTI 結合の RHS リレーションに対するパラメータ化されたパスがあります。これらの場合、最初のマッチング後に内部スキャンを停止できるため、合計コストではなく起動コストが主に重要となります。


LATERAL サブクエリ
------------------

9.3 以降、FROM 内のサブクエリ（および FROM 内の関数）からの SQL 標準の LATERAL 参照をサポートしています。プランナは、LATERAL 参照を含む RTE に対してパラメータ化されたパスを生成することでこれらを実装します。そのような場合、そのリレーションに対する「すべての」パスは、少なくともその LATERAL 参照で使用されるリレーションのセットによってパラメータ化されます。（そして結果として、そのようなサブクエリを含む結合リレーションは、パラメータ化されていないパスを一切持たない可能性があります。）ただし、パラメータ化されたパスについて上記で述べた他のすべてのコメントは依然として適用されます。特に、そのような各パスは、押し下げ可能なすべての結合句を強制的に適用することが期待されており、同じパラメータ化を持つすべてのパスは同じ行数を持つことになります。

また、LATERAL サブクエリをオプティマイザによってフラット化（親クエリにプルアップ）することも許可していますが、これは、そのクエリの外部にある最も低い外部結合の外側を参照するような LATERAL 参照を JOIN/ON 句に導入しない場合に限られます。そのような句のセマンティクスは不明瞭になるためです。この制限があっても、LATERAL サブクエリのプルアップによって、構文上のスコープ外にあるリレーションへの LATERAL 参照を含む PlaceHolderVar が作成される可能性があることに注意してください。私たちは依然としてそのような PHV をその構文上の位置またはそれより下で評価しますが、プランノードの句やターゲットリストにそのような PHV が存在する場合、そのノードは LATERAL 参照を提供するリレーションに対して nestloop 結合の内部に現れる必要があります。（おそらく、その仕組みが機能するようになった今、プルアップの制限を緩和できるかもしれません。）


句に対するセキュリティレベルの制約
----------------------------------

行レベルセキュリティ（RLS）およびセキュリティバリアビューを効率的にサポートするため、句（RestrictInfo ノード）に "security_level" フィールドをマークしています。基本的な概念は、セキュリティレベルが低い句は、高い句よりも先に評価されなければならないというものです。これにより、機密データを漏洩させる可能性のある「漏洩しやすい（leaky）」句が、機密行をフィルタリングするためのセキュリティバリア句よりも先に評価されないことが保証されます。ただし、多くの句条件は「漏洩防止（leakproof）」です。つまり、それらが使用する関数がデータを漏洩させないと信頼できるということです。不必要に非効率なプランを避けるため、漏洩防止の句は、たとえ他の句よりも構文上のセキュリティレベルが高くても、セキュリティレベルの考慮によって遅延されることはありません。

RLS やセキュリティバリアビューを使用しないクエリでは、すべての句のセキュリティレベルはゼロになるため、これらの制限は機能しません。句条件の漏洩防止性をチェックする必要さえありません。

セキュリティバリア句が存在する場合、それらはセキュリティレベル 0（バリアが複数層ある場合はそれ以上）になります。クエリテキストから来る通常の句は、バリア句に使用される最高レベルより 1 つ高いセキュリティレベルを持ちます。

EquivalenceClass 処理によって新しい句が生成される場合、それらにセキュリティレベルを割り当てる必要があります。これは見た目よりも複雑です。EquivalenceClass のソース句の中で見つかった最大のレベルを使用するのが安全であるというのが最初の直感ですが、それは全く安全ではありません。なぜなら、セキュリティバリア句の望ましくない遅延を許容してしまうからです。バリア句 "t.x = t.y" とクエリ句 "t.x = constant" があり、バリア句がチェックされる前に評価してはならない別のクエリ句 "leaky_function(t.z)" があると仮定します。この場合、EC {t.x, t.y, constant} が存在し、EC 句を "t.x = constant AND t.y = constant" に置き換えることになります。（後者の条件によって t.y のインデックスを使用できる可能性があり、元の句からは決して発見できないため、この動作をあきらめたくはありません。）もしこれらの生成された句にクエリ句と同じセキュリティレベルが割り当てられると、leaky_function 句が先に評価され、バリア条件を通過しない可能性のある行のデータが leaky_function に見えてしまう可能性があります。

代わりに、EC を伴うセキュリティレベルの扱いは以下のようになります：
* 句が漏洩防止であるか、セキュリティレベルがゼロでない限り、そもそも EC のソース句として受け入れられません。
* EC 由来の句には、EC のソース句の中で見つかった最小（最大ではない）のセキュリティレベルが割り当てられます。
* EC のソース句の中で見つかった最大のセキュリティレベルが 0 より大きい場合、由来する句に対して選択される等価演算子は漏洩防止でなければなりません。そのような演算子が見つからない場合、その EC は「壊れている」とみなされ、追加の由来句なしでソース句を出力するフォールバックを行います。

これらのルールにより、信頼できない句（セキュリティレベルが 0 より大きいもの）が EC に漏洩しやすい由来句を生成させることができないことが保証されます。これにより、由来句に対して最大ではなく最小のセキュリティレベルを使用することが安全になります。これらのルールは、由来句を全く生成できないことによる質の低いプランにつながる可能性がありますが、ほとんどの btree 等価演算子は漏洩防止であるため、実際にはそのリスクは小さいです。また、レベル 0 の句に対して例外を設けることで、バリア句が存在しない場合にプランが劣化しないことを保証しています。

すべての句にセキュリティレベルが割り当てられたら、バリア句の順序制約の強制は 2 つのルールに集約されます：

* テーブルスキャンプランノードは、漏洩防止であるか、同じプランノードで実行される予定の他のどの句よりもセキュリティレベルが高くない限り、句を早期実行のために選択してはなりません（例えば、インデックススキャンでインデックス修飾子として使用するなど）。（句を早期実行のために選択してもよいかどうかを確認するには、ユーティリティ関数 restriction_is_securely_promotable() を使用してください。）

* 句リストの通常の実行は、同じセキュリティルールを満たす順序で実行しなければなりません。つまり、漏洩防止でない限り、より高いセキュリティレベルは後で評価されなければなりません。（これは createplan.c の order_qual_clauses() によって一箇所で処理されます。）

order_qual_clauses() は、漏洩防止句を正確にどう扱うかを決定するためにヒューリスティックを使用します。通常はセキュリティレベル、次にコストで句をソートしますが、明確な理由なしに句を並べ替えないよう、ソートが安定していることに注意を払います。しかし、これにより、より高いセキュリティレベルを持つ安価な句よりも、非常に高価な句が先に実行される可能性があります。安価な句が漏洩しやすい場合は選択の余地がありませんが、漏洩防止であれば先に置くことができます。私たちは、漏洩防止句のコストが cpu_operator_cost の 10 倍未満である場合に限り、それらをセキュリティレベル 0 であるかのようにソートすることを選択します。その制限は、漏洩防止であるという理由だけで高価な句を先に実行してしまうという逆の問題を緩和します。結合条件（join quals）間にセキュリティレベルの混在がある場合、結合条件の安全な処理をサポートするために追加のルールが必要になります。例えば、セキュリティレベルの低い他の条件よりも低い結合レベルで、セキュリティレベルの高い漏洩の恐れがある条件が評価されることを防ぐ必要があります。現在、セキュリティ優先の条件はRLSポリシーやセキュリティバリアビューに由来する単一テーブルの制限条件のみであり、セキュリティバリアビューのサブクエリは親クエリにフラット化されることがないため、これを考慮する必要はありません。したがって、セキュリティ優先条件の強制はテーブルスキャンレベルでのみ行われます。結合条件間のセキュリティレベルを安全に処理するための追加ルールがあれば、セキュリティバリアビューを親クエリにフラット化できるようになり、クエリ評価の必要な順序を維持しつつ、プランニングの柔軟性が向上するはずです。しかし、それは将来の課題です。


スキャン/結合後のプランニング
-----------------------

これまではスキャン/結合のプランニング、つまりSQLクエリのFROM句とWHERE句の実装についてのみ議論してきました。しかし、プランナはGROUP BY、集約、およびクエリのその他の高レベルな機能をどのように扱うかも決定しなければなりません。多くの場合、これらのステップを実行する方法は複数あり、最適化の選択肢が存在します。これらのステップは、スキャン/結合のプランニングと同様に、ステップを実行するさまざまな方法を表すPathを構築し、最もコストの低いPathを選択することで処理されます。

すべてのPathは「親」としてRelOptInfoを必要とするため、これらの上位レベルの処理ステップの出力を表すRelOptInfoを作成します。これらのRelOptInfoはほとんどがダミーですが、そのpathlistリストには各ステップで有用と見なされるすべてのPathが保持されます。現在、上位レベルのプランニング中に以下の種類の追加RelOptInfoを作成する可能性があります。

UPPERREL_SETOP		UNION/INTERSECT/EXCEPTの結果（存在する場合）
UPPERREL_PARTIAL_GROUP_AGG	部分的なグループ化/集約の結果（存在する場合）
UPPERREL_GROUP_AGG	グループ化/集約の結果（存在する場合）
UPPERREL_WINDOW		ウィンドウ関数の結果（存在する場合）
UPPERREL_PARTIAL_DISTINCT	部分的な「SELECT DISTINCT」の結果（存在する場合）
UPPERREL_DISTINCT	「SELECT DISTINCT」の結果（存在する場合）
UPPERREL_ORDERED	ORDER BYの結果（存在する場合）
UPPERREL_FINAL		残りのトップレベルアクションの結果

UPPERREL_FINALは、LockRows（SELECT FOR UPDATE）、LIMIT/OFFSET、ModifyTableといった最終的な処理ステップを表すために使用されます。これらのステップを実行する順序に柔軟性はないため、この段階をより細かく分割する必要はありません。

これらの「上位リレーション（upper relations）」は、上記に示すUPPERREL列挙値と、同じ種類の上位リレーションを複数存在させることを可能にするrelidsセットによって識別されます。同じ種類の上位リレーションが複数必要ない場合は、relidsにNULLを使用します。実際、現在relidsセットは常にNULLであるため痕跡的なものとなっていますが、将来的には変更される予定です。例えば、集合演算のプランニングにおいて、互いに競合する特定のPathグループ内で、どのリーフSELECTのサブセットが結合されたかを示すためにrelidsが必要になる可能性があります。

subquery_planner()の結果は、常にNULLのrelidsを持つUPPERREL_FINALリレーションに格納されたPathのセットとして返されます。他の種類の上位リレーションは、特定のクエリで必要な場合にのみ作成されます。


並列クエリと部分パス（Parallel Query and Partial Paths）
-------------------------------------------------------

並列クエリとは、クエリ全体またはクエリの一部で実行する必要がある作業を、並列ワーカ（parallel workers）と呼ばれる1つ以上のワーカプロセスによって実行できるように分割することを指します。並列ワーカは動的バックグラウンドワーカのサブタイプです。詳細については、src/backend/access/transam/README.parallelを参照してください。並列クエリに関する学術文献では、並列実行戦略は本質的に2つのカテゴリに分類できると示唆されています。1つはパイプライン並列処理（クエリの実行を複数のステージに分割し、各ステージを個別のプロセスが処理する）、もう1つはパーティション並列処理（データを複数のプロセス間で分割し、各プロセスがそのサブセットを処理する）です。しかし、文献によると、パイプライン並列処理による利益は、パイプラインの停滞を回避することが難しいため、非常に限定的であることが多いとされています。そのため、現在この手法を使用するクエリプランの生成は試みていません。

その代わり、基底テーブルがパーティション化されていることを必要としない、パーティション並列処理に焦点を当てています。これに必要なのは、(1)関係に含まれる少なくとも1つの基底テーブルのデータを複数のプロセスにわたって分割する方法があること、(2)各プロセスが自身のデータ部分を処理できるようにすること、そして(3)結果を収集することだけです。要件(2)と(3)は、エグゼキュータノードであるGather（またはGatherMerge）によって満たされます。これは任意の数のワーカプロセスを起動し、その単一の子プランをすべてのワーカで実行します。子プランがリーダーを忙しくさせるのに十分なデータを生成していない場合は、リーダーでも実行される可能性があります。要件(1)はテーブルスキャンノードによって処理されます。parallel_aware = trueで呼び出されると、このノードは実質的にテーブルをブロック単位でパーティション化し、そのスキャンノードが実行される各ワーカで関係のタプルの一部を返します。

非並列アクセス手法と同様に、並列プランで使用できるアクセス戦略を表すPathを構築します。これらは本質的に非並列プランで利用可能な戦略と同じですが、重要な違いがあります。Gatherノードの下で実行されるパスは、すべての結果ではなく、各ワーカでクエリ結果のサブセットのみを返します。実際に実行可能なパスを形成するには、Gatherノードの（かなり大きな）コストを考慮に入れる必要があります。この理由などから、Gatherノードの下で実行されることを意図したパス（ワーカごとに結果のサブセットのみを返すため「部分（partial）」パスと呼びます）は、通常のパスとは別に保持する必要があります（RelOptInfoのpartial_pathlistとadd_partial_path関数を参照）。

並列クエリを効果的にするための鍵の1つは、クエリの可能な限り多くの部分を並列に実行することです。したがって、一般的にはGatherステージをプランの可能な限り上部まで延期することが望ましいと考えられます。Gatherの下により多くの作業をプッシュできるケースの範囲を拡大し（かつそれらを正確にコスト計算する）、今後も継続して取り組む予定です。


パーティション単位の結合（Partitionwise joins）
-------------------------------------------

同様にパーティション化された2つのテーブル間の結合は、結合テーブルのパーティションキー間に等価結合条件が存在する場合、それらの一致するパーティション間の結合に分解できます。パーティションキー間の等価結合は、一方のパーティション化されたテーブルの特定の行に対するすべての結合相手が、もう一方のパーティション化されたテーブルの対応するパーティション内に存在しなければならないことを意味します。このため、パーティション化されたテーブル間の結合を、一致するパーティション間の結合に分解できます。結果として得られる結合は、結合対象のリレーションと同じ方法でパーティション化されるため、パーティションキー間に等価結合条件を持つ同様にパーティション化されたテーブル間のN方向結合を、一致するパーティション間のN方向結合に分解できます。この、パーティション化されたテーブル間の結合をパーティション間の結合に分解する手法を「パーティション単位の結合（partitionwise join）」と呼びます。ここでは、「パーティション化されたリレーション」という用語を、パーティション化されたテーブル、または互換性のある方法でパーティション化されたテーブル間の結合のいずれかに対して使用します。

結合対象のリレーションが完全に同じパーティション境界を持たない場合でも、高度なパーティションマッチングアルゴリズムを使用することでパーティション単位の結合を適用できます。このアルゴリズムは、両方の結合対象リレーションについて、一方の結合対象リレーションの各パーティションが、もう一方の結合対象リレーションの最大1つのパーティションとしか一致しないかどうかをチェックします。その場合、結合対象リレーション間の結合を、一致するパーティション間の結合に分解できます。その結合リレーションはパーティション化されていると見なすことができます。このアルゴリズムは、結合を計算するためのパーティション単位の結合を可能にするために、一致するパーティションのペアと、結合リレーションのパーティション境界を生成します。このアルゴリズムはpartition_bounds_merge()に実装されています。このようにパーティション化されていると見なされるN方向結合リレーションの場合、すべての結合対象リレーションのペアがパーティション単位の結合を使用できるわけではありません。例えば：

	(A leftjoin B on (Pab)) innerjoin C on (Pac)

ここでA、B、Cはパーティション化されたテーブルであり、AはBやCと比較して余分なパーティションを持っています。結合{A B}に対してパーティション単位の結合を検討する場合、Aの余分なパーティションにはNULL許容側に一致するパーティションが存在しません。これは現在のパーティション単位の結合の実装では処理できないケースです。そのため、{A B}はパーティション化されているとは見なされず、3方向結合のために検討される{A B}とCのペアはパーティション単位の結合を使用できません。一方で、{A C}とBのペアはパーティション単位の結合を使用できます。なぜなら、{A C}は余分なパーティションを除外することでパーティション化されていると見なされるからです（外部結合の並べ替えに関する恒等式1を参照）。N方向結合がパーティション単位の結合を使用できるかどうかは、両方がパーティション化されており、かつパーティション単位の結合を使用できる最初の結合対象リレーションのペアに基づいて決定されます。

パーティション化されたリレーションのパーティション化プロパティは、そのRelOptInfoに格納されます。パーティションキーのデータ型に関する情報はPartitionSchemeData構造体に格納されます。プランナは正規のパーティションスキーム（個別のPartitionSchemeDataオブジェクト）のリストを保持しており、同じパーティション化スキームを持つ任意の2つのパーティション化されたリレーションのRelOptInfoが、同じPartitionSchemeDataオブジェクトを指すようになっています。これにより、PartitionSchemeDataオブジェクトによって消費されるメモリが削減され、結合対象リレーションのパーティションスキームを比較しやすくなります。

パーティション単位の集約/グループ化（Partitionwise aggregates/grouping）
-----------------------------------------------------------------------

GROUP BY句にすべてのパーティションキーが含まれている場合、特定のグループに属するすべての行は単一のパーティションから来る必要があります。したがって、集約は各パーティションに対して完全に個別に行うことができます。そうでない場合は、各パーティションに対して部分的な集約を計算し、個々のパーティションからの結果を結合した後に最終処理を行うことができます。この、パーティション化されたリレーションに対する集約やグループ化を、そのパーティションに対する集約やグループ化に分解する手法を「パーティション単位の集約（partitionwise aggregation）」と呼びます。特にパーティションキーがGROUP BY句と一致する場合、これは通常の手法よりも大幅に高速になる可能性があります。
