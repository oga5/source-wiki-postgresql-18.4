ノード構造
==========

はじめに
--------

Postgresは「ノード」型を使用して、パースツリー、計画ツリー、およびエグゼキュータ状態ツリーを構成します。そのようなツリーに現れるすべてのオブジェクトは、ノード型として宣言されなければなりません。さらに、パース/計画/実行ノードツリーの一部ではない一部のオブジェクト型も、識別目的でNodeTagを受け取ります。これは通常、同じポインタを通じて複数のオブジェクト型を渡したいAPIに関与しているためです。

ノード構造体は、最初のフィールドがNodeTag型である、ごく普通のC構造体です。「継承」は慣習によって実現されます。最初のフィールドは、別のノード型にすることもできます。

ノード型は通常、copyObject()によるコピー、equal()による比較、outNode()によるシリアライズ、およびnodeRead()によるデシリアライズのサポートを備えています。一部のクラスのノードについては、これらのサポート関数のすべてが必要なわけではありません。例えば、エグゼキュータ状態ノードは現在、それらのいずれも必要としません。システムに関する限り、出力関数と読み込み関数は、カタログに格納されたパースツリーに現れる可能性のあるノード型と、並列ワーカに渡すためにシリアライズされる計画ツリーノードに対してのみ必要です。しかし、デバッグに非常に便利であるため、他のいくつかのノード型に対しても出力関数を提供しています。現在、そのようなカバレッジは、生のパースツリーとほとんどのプランナデータ構造に対して存在します。ただし、生のパースツリーの出力カバレッジは不完全です。特に、ユーティリティ文はほとんど完全にサポートされていません。

関連ファイル
------------

ノード構造を操作するためのユーティリティ関数は、このディレクトリにあります。一部のサポート関数はgen_node_support.plスクリプトによって自動生成され、他の関数は手動で管理されます。サポート関数の自動生成を制御するために、ノード型とノードフィールドにpg_node_attr()仕様を注釈として付けることができます。詳細はsrc/include/nodes/nodes.hのドキュメントを参照してください。


このディレクトリ内のファイル (src/backend/nodes/)

    汎用ノード操作関数:
	copyfuncs.c	- ノードツリーのコピー (*)
	equalfuncs.c	- 2つのノードツリーの比較 (*)
	outfuncs.c	- ノードツリーをテキスト表現に変換 (*)
	readfuncs.c	- テキスト表現をノードツリーに戻す (*)
	makefuncs.c	- いくつかの一般的なノード型の作成関数
	nodeFuncs.c	- その他の汎用操作関数
	queryjumblefuncs.c - クエリジャンブル用のノードツリーの計算 (*)

    (*) - これらのファイルの関数のほとんどは、gen_node_support.plによって生成され、そこで#includeされています。

    専門的な操作関数:
	bitmapset.c	- Bitmapsetのサポート
	list.c		- 汎用リストのサポート
	multibitmapset.c - List-of-Bitmapsetのサポート
	params.c	- Paramのサポート
	tidbitmap.c	- TIDBitmapのサポート
	value.c		- 値ノードのサポート

src/include/nodes/ 内のファイル

    ノード定義は主に以下に現れます:
	nodes.h		- ノードタグ(NodeTag)の定義 (*)
	primnodes.h	- プリミティブノード
	parsenodes.h	- パースツリーノード
	pathnodes.h	- パスツリーノードおよびプランナ内部構造
	plannodes.h	- 計画ツリーノード
	execnodes.h	- エグゼキュータノード
	memnodes.h	- メモリノード
	pg_list.h	- 汎用リスト

    (*) - gen_node_support.plによって生成されたファイルも#includeします。


ノードを追加する手順
--------------------

Fooというノードを定義したいとします:

1. 適切なinclude/nodes/???.hファイルに構造体定義を追加します。
   例えばPlanノードから継承する場合は、構造体定義の最初のフィールドにPlanを配置してください。(T_Fooタグは自動的に作成されます。)
2. copyfuncs.funcs.c、equalfuncs.funcs.c、outfuncs.funcs.c、queryjumblefuncs.funcs.c、およびreadfuncs.funcs.c内の生成されたサポート関数が正しく見えるか確認してください。必要に応じて属性を追加して結果を制御します。(一部のクラスのノード型では、すべてのサポート関数が必要なわけではありません。関連するノード型と同様のノード属性を使用してください。)
3. 必要に応じてnodeFuncs.c内の関数にケースを追加します。新しいノード型について教える必要がある場所は他にもたくさんあるでしょう。同様の既存のノード型を1つか2つgrepして、修正すべきすべての場所を見つけるのが最善です。
   (頻繁に作成されるノードを除き、makefuncs.cに作成関数を書く必要はありません。)
4. debug_copy_parse_plan_trees、debug_write_read_parse_plan_trees、およびdebug_raw_expression_coverage_testを使用して新しいコードをテストし、必要なすべての場所にサポートが追加されていることを確認してください (例: PG_TEST_INITDB_EXTRA_OPTS='-c debug_...=on' でテストを実行)。

新しいノード型を追加すると既存のタグに関連付けられた番号がずれるため、これを行った後はツリー全体を再コンパイルする必要があります。(--enable-dependが通常役立ちます。) ただし、番号がディスクに書き込まれることはないため、initdbを強制する必要はありません。しかし、ノード型の変更や削除は、通常、initdbを強制するカタログバージョンの変更を伴うべきです。システムカタログに格納されたシリアライズ済みノードツリーの解釈が影響を受けるためです。(例えばPlanノードのように、ノード型が格納されたパースツリーに現れない場合は、変更のためにcatversionの変更は不要です。)
