Postgres エグゼキュータ
=====================

エグゼキュータは「プランノード」のツリーを処理します。プランツリーは本質的に、タプル処理操作のデマンドプル型パイプラインです。各ノードは呼び出されると、出力シーケンスの次のタプルを生成するか、タプルがもうない場合は NULL を返します。そのノードが単純なリレーションスキャンノードではない場合、入力タプルを取得するために順番に呼び出す子ノードを持ちます。

この基本モデルに対する改良点には以下が含まれます：

* スキャン方向の選択（順方向または逆方向）。注意：これは現在、十分にサポートされていません。単純なスキャンノードでは機能しますが、結合や集約などではあまりうまく機能しません。

* ノードをリセットし、出力シーケンスを最初から再生成させるための Rescan コマンド。

* ノードの結果を変更できるパラメータ。パラメータを調整した後、そのノードおよびその上位のすべてのノードに対して Rescan コマンドを適用する必要があります。不要なノードの再スキャンを回避するための適度にインテリジェントなスキームが存在します（例えば、Sort は入力のパラメータが変更されていない場合、保存されたソート済みデータを再読み込みするだけでよいため、入力を再スキャンしません）。

SELECT の場合、最上位の結果タプルをクライアントに配信するだけで十分です。INSERT/UPDATE/DELETE/MERGE の場合、実際のテーブル変更操作は最上位の ModifyTable プランノードで行われます。クエリに RETURNING 句が含まれている場合、ModifyTable ノードは計算された RETURNING 行を出力として配信し、そうでなければ何も返しません。INSERT の処理は非常に単純です。ModifyTable より下のプランツリーから返されたタプルが、正しい結果リレーションに挿入されます。UPDATE の場合、プランツリーは更新された列の新しい値と、どのテーブル行を更新するかを識別する「ジャンク」（隠し）列を返します。ModifyTable ノードはその行をフェッチして変更されていない列の値を取得し、値を新しい行に結合して更新を適用する必要があります。（ヒープテーブルの場合、行識別ジャンク列は CTID ですが、他のテーブルタイプでは別のものが使用される可能性があります。）DELETE の場合、プランツリーはジャンク行識別列を配信するだけでよく、ModifyTable ノードはそれらの各行を訪問して削除済みとしてマークします。MERGE については後述します。

プランツリーとステートツリー
--------------------------

プランナによって配信されるプランツリーには、Plan ノード（struct Plan から派生した構造体型）のツリーが含まれています。エグゼキュータの起動中に、エグゼキュータステートノードを含む同一構造の並列ツリーを構築します。一般的に、すべてのプランノード型には対応するエグゼキュータステートノード型があります。ステートツリーの各ノードは、プランツリー内の対応するノードへのポインタと、そのノード型を実装するために必要なエグゼキュータステートデータを保持します。この配置により、エグゼキュータに関する限り、プランツリーを完全に読み取り専用にすることができます。実行中に変更されるすべてのデータはステートツリー内にあります。読み取り専用のプランツリーは、プランのキャッシュと再利用を大幅に簡素化します。

実行時のパーティションプルーニングにより一致するレコードが見つからないと判断された場合、エグゼキュータがサブプラン全体が不要であると判断すると、エグゼキュータの起動中に対応するエグゼキュータステートノードが作成されないことがあります。これは現在、Append ノードと MergeAppend ノードでのみ発生します。この場合、不要なサブプランは無視され、エグゼキュータステートのサブノード配列はプランのサブプランリストと順序が一致しなくなります。

各 Plan ノードには、ターゲットリスト、制約条件などを表すために、関連付けられた式ツリーが存在する場合があります。これらのツリーもエグゼキュータにとっては読み取り専用ですが、以下で説明するように、式評価のためのエグゼキュータステートは Plan 式のツリー形状を反映しません。むしろ、複雑な式ノード型に対してサブノードを持つ場合がありますが、式ツリーごとに ExprState ノードが 1 つだけ存在します。

全体として、これらのツリーで使用されるノードには、Plan ノード、それに対応する PlanState ノード、Expr ノード、ExprState ノードの 4 つのクラスがあります。（実際には、これら 3 つのツリーベースの表現すべてで「接着剤」として使用される List ノードもあります。）


式ツリーと ExprState ノード
--------------------------

式ツリーは、Plan ツリーとは対照的に、ステートノードの対応するツリーにミラーリングされません。代わりに、個別に実行可能な各式ツリー（例：Plan の qual や targetlist）は 1 つの ExprState ノードによって表されます。ExprState ノードには、式をコンパクトで線形な形式で評価するために必要な情報が含まれています。そのコンパクトな形式は、ExprState->steps[] 内のフラットな配列として格納されます（ExprEvalStep の配列であり、ExprEvalStep * ではありません）。

このような表現を選択する理由は以下の通りです：
- 通常、1 つの Expr 型ノードを評価するために必要な作業量は非常に小さいため、評価中にツリーウォークを実行するオーバーヘッドが無視できません。
- フラットな表現は単一関数内で非再帰的に評価できるため、スタックの深さと関数呼び出しのオーバーヘッドが削減されます。
- このような表現は、高速なインタープリタ実行とネイティブコードへのコンパイルの両方に使用できます。

式の Plan ツリー表現は、ExecInitExpr() によって ExprState ノードにコンパイルされます。可能な限り多くの複雑さは、インタープリタ版とコンパイル版の両方が複雑さに対処する必要がある実行時ではなく、ExecInitExpr()（およびヘルパー）によって処理されるべきです。実行アプローチ間で作業を重複させるだけでなく、実行時の初期化チェックは、式が評価されるたびに小さくても無視できないコストがかかります。したがって、ExecInitExpr() が、ドメイン型に適用される CHECK 制約式のセットなど、単一クエリの実行全体で変化しないと予想される情報を事前計算できるようにしています。これは、プランの無効化を必要とするイベントの数を大幅に増やすことなく、プラン時に行うことはできませんでした。（以前は、この種の情報のいくつかは式評価のたびに再チェックされていましたが、それは不必要なオーバーヘッドであると思われます。）


式の初期化
-------------------------

ExecInitExpr() および類似のルーチンでは、Expr ツリーがフラットな表現に変換されます。各 Expr ノードは、0 個、1 個、または複数の ExprEvalSteps によって表される場合があります。

各 ExprEvalStep の作業はそのオペコード（enum ExprEvalOp）によって決定され、その作業の結果を ExprEvalStep->resvalue/resnull が指す Datum 変数およびブール型の null フラグ変数に格納します。複雑な式は、いくつかのステップを連鎖させることで実行されます。例えば、「a + b」（2 つの Var 式を持つ 1 つの OpExpr）は、Var 値を取得する 2 つのステップと、+ 演算子の基盤となる関数を評価する 1 つのステップとして表されます。Var のステップは、resvalue/resnull が関数評価ステップで使用される FunctionCallInfoBaseData 構造体の適切な args[].value .isnull 要素を直接指すようにし、結果値をコピーする余分な作業を回避します。

完了した ExprState->steps 配列の最後の要素は常に EEOP_DONE_RETURN または EEOP_DONE_NO_RETURN ステップです。これにより、反復中に配列の末尾をテストする必要がなくなります。前者は式が直接値を返す場合に使用され、後者は式初期化の副作用が目的である場合（例：射影や集約遷移値の計算）に使用されます。

また、式に変数参照（ExprContext の INNER、OUTER、または SCAN タプルのユーザー列への参照）が含まれている場合、steps 配列は EEOP_*_FETCHSOME ステップで始まり、関連するタプルが分解されて必要な列が直接利用可能になることを保証します（slot_getsomeattrs() を参照）。これにより、個々の Var 取得ステップは配列ルックアップとほとんど変わらなくなります。

ExecInitExpr() の作業のほとんどは、再帰関数 ExecInitExprRec() とそのサブルーチンによって行われます。ExecInitExprRec() は 1 つの Expr ノードを実行に必要なステップにマッピングし、サブ式が必要に応じて再帰します。

各 ExecInitExprRec() 呼び出しは、そのサブ式の結果をどこに格納するかを指定する必要があります（resv/resnull パラメータ経由）。これにより、前述のサブ式を fcinfo->args[].value/isnull に直接評価するシナリオが可能になりますが、注意も必要です。ターゲットの Datum/isnull 変数は、そのターゲット Datum/isnull 変数の使用前に実行されるステップでしか結果が必要とされない場合を除き、他の ExecInitExprRec() と共有してはなりません。ExprEvalStep 表現の非再帰性により、通常はそれを保証するのは簡単です。

ExecInitExprRec() は ExprEvalPushStep() を使用して新しい操作を ExprState->steps 配列にプッシュします。ステップを連続した配列として保持するために、ExprEvalPushStep() は十分なスペースがない場合に配列全体を repalloc する必要があります。そのため、式初期化中にステップのいずれかを直接指すことは許可されていません。したがって、サブ式の resv/resnull は通常、steps 配列とは別に palloc されたストレージを指します。例えば、関数呼び出しステップの FunctionCallInfoBaseData は、ExprEvalStep 配列の一部ではなく個別に割り当てられます。完全な式の全体的な結果は、通常、ExprState ノード自体の resvalue/resnull フィールドに返されます。

ブール式などの一部のステップでは、特定のサブ式の評価をスキップできます。フラットな表現では、これは次のステップに連続して進むのではなく、後のステップにジャンプすることに相当します。このようなジャンプのターゲットは、次に実行するステップの ExprState->steps 配列内の整数インデックスで表されます。（execExprInterp.c の EEO_NEXT マクロと EEO_JUMP マクロを比較してください。）

通常、ExecInitExprRec() はジャンプステップを steps 配列にプッシュし、スキップされる可能性のあるサブ式のステップを再帰的に生成してから、戻ってサブ式のステップの既知の長さを利用してジャンプターゲットインデックスを修正する必要があります。これは execExpr.c の adjust_jumps リストによって処理されます。

ExprState を構築する最後のステップは ExecReadyExpr() を適用することであり、これにより選択された実行メソッドを使用して実行の準備が整います。


式の評価
---------------------

異なる式評価メソッドを可能にし、より良い分岐/ジャンプターゲット予測を実現するために、式は ExprState->evalfunc を呼び出すことによって評価されます（ExecEvalExpr() および関連関数経由）。

ExecReadyExpr() は、evalfunc を適切な関数に設定することで解釈メソッドを選択できます。デフォルトの実行関数である ExecInterpExpr は execExprInterp.c に実装されています。詳細についてはそのヘッダーコメントを参照してください。特に単純な式には、特殊なケースの evalfuncs が使用されます。

単純なものよりもパフォーマンスが重要ではない、より複雑な式評価ステップの多くは、式実行の高速パスの外にある個別の関数として実装されており、その実装をインタープリタ式評価とコンパイル式評価の間で共有できるようにしていることに注意してください。これは、これらのヘルパー関数自体が式ステップのディスパッチを実行することは許可されていないことを意味します。ディスパッチメソッドは呼び出し元に基づいて変化するためです。したがって、ヘルパーはサブ式の実行を呼び出すことができません。必要なすべてのサブ式の結果は、以前のステップによって計算されている必要があります。また、次の式ステップへのディスパッチは、ヘルパーから戻った後に実行する必要があります。


ターゲットリストの評価
---------------------ExecBuildProjectionInfoは、ターゲットリストを評価してExprState->resultslotに格納する効果を持つExprStateを構築します。汎用的なターゲットリスト式は、前述のように評価（結果をExprStateのresvalue/resnullフィールドに格納）し、その後EEOP_ASSIGN_TMPステップを使用して、その結果を結果スロットの適切なtts_values[]およびtts_isnull[]配列要素に移動することで実行されます。ターゲットリストのエントリが単純なVarである場合、2つのステップではなく1つのステップのみを使用する特別な高速パスステップ型（EEOP_ASSIGN_*_VAR）が存在します。


MERGE
-----

MERGEは、複数テーブル、複数アクションのコマンドです。ターゲットテーブルとソースリレーションを指定し、複数のWHEN MATCHEDおよびWHEN NOT MATCHED句を含むことができ、それぞれがUPDATE、INSERT、DELETE、またはDO NOTHINGアクションのいずれかを指定します。ターゲットテーブルはMERGEによって変更され、ソースリレーションはアクションのための追加データを提供します。各アクションは、タプルごとに評価される修飾式をオプションで指定できます。

プランナにおいて、transform_MERGE_to_joinは、ターゲットテーブルとソースリレーションの間の結合を構築し、ターゲットテーブルからの行識別用のジャンク列を含めます。MERGEコマンドにWHEN NOT MATCHED句が含まれている場合、この結合は外部結合となります。ModifyTableノードはその結合のプランツリーからタプルを取得します。取得したタプルの行識別列がNULLの場合、ソースリレーションにはターゲットテーブル内のどのタプルとも一致しないタプルが含まれているため、プランによって返されたそのタプルを基に、各WHEN NOT MATCHED句の修飾式が評価されます。式が真を返せば、その句で示されたアクションが実行され、それ以降の句は評価されません。一方、行識別列がNULLでない場合、ターゲットテーブルから一致するタプルを取得できます。各WHEN MATCHED句の修飾式は、取得したタプルとプランによって返されたタプルの両方を基に評価されます。

WHEN NOT MATCHED句が存在しない場合、プランナによって構築される結合は内部結合となり、行識別用のジャンク列は常に非NULLとなります。

WHEN MATCHEDが、並行して更新または削除された行を処理することになった場合、EvalPlanQual（後述）を使用して行の最新バージョンを検索し、再取得します。それが存在する場合、使用する一致するWHEN MATCHED句の検索が最初から開始されます。

MERGEは独自のトリガータイプを許可せず、代わりにUPDATE、DELETE、およびINSERTトリガーを起動します。行トリガーは、その行に対してアクションが実行されるたびに行ごとに起動されます。ステートメントトリガーは、対応する句に一致する行があるかどうかにかかわらず、常に起動されます。


メモリ管理
-----------------

「クエリごと」のメモリコンテキストはCreateExecutorState()の間に作成されます。エグゼキュータの呼び出し中に割り当てられたすべてのストレージは、そのコンテキストまたは子コンテキストに割り当てられます。これにより、エグゼキュータのシャットダウン時にストレージを簡単に回収できます。個別のpfreeや発生しうるストレージリークに対処する代わりに、メモリコンテキストを破棄するだけで済みます。

特に、前述のプラン状態ツリーおよび式状態ツリーは、クエリごとのメモリコンテキストに割り当てられます。

クエリ内のメモリリークを避けるため、クエリ実行中のほとんどの処理は「タプルごと」のメモリコンテキストで行われます。これは、通常タプルごとに空にリセットされるため、そのように呼ばれます。タプルごとのコンテキストは通常ExprContextに関連付けられており、一般的に各PlanStateノードは、自身のqualおよびターゲットリスト式を評価するために独自のExprContextを持っています。


クエリ処理の制御フロー
-----------------------------

これは、完全なクエリ処理の制御フローの概要です。

	CreateQueryDesc

	ExecutorStart
		CreateExecutorState
			クエリごとのコンテキストを作成
		ExecInitNodeを実行するためにクエリごとのコンテキストに切り替え
		AfterTriggerBeginQuery
		ExecInitNode --- プランツリーを再帰的にスキャン
			ExecInitNode
				子ノードを再帰的に処理
			CreateExprContext
				タプルごとのコンテキストを作成
			ExecInitExpr

	ExecutorRun
		ExecProcNode --- クエリごとのコンテキストで再帰的に呼び出される
			ExecEvalExpr --- タプルごとのコンテキストで呼び出される
			ResetExprContext --- メモリを解放するため

	ExecutorFinish
		ExecPostprocessPlan --- 未完了のModifyTableノードを実行
		AfterTriggerEndQuery

	ExecutorEnd
		ExecEndNode --- リソースを再帰的に解放
		FreeExecutorState
			クエリごとのコンテキストと子コンテキストを解放

	FreeQueryDesc

上記のコメントの通り、ExecEndNodeでメモリを解放することは実際には重要ではありません。いずれにせよFreeExecutorStateですべて消去されるためです。ただし、リレーションを閉じる、バッファピンをドロップするなどの処理には注意が必要であるため、これらのリソースを見つけるためにプラン状態ツリーをスキャンする必要があります。


エグゼキュータは、プランツリーなしで単純な式を評価するためにも使用できます（「単純」とは「集約やサブセレクトがない」ことを意味しますが、関数呼び出しの中に隠れている可能性はあります）。この場合の制御フローは以下のようになります。

	CreateExecutorState
		クエリごとのコンテキストを作成

	CreateExprContext	-- またはGetPerTupleExprContext(estate)を使用
		タプルごとのコンテキストを作成

	ExecPrepareExpr
		一時的にクエリごとのコンテキストに切り替え
		expression_plannerを通じて式を実行
		ExecInitExpr

	繰り返し実行:
		ExecEvalExprSwitchContext
			ExecEvalExpr --- タプルごとのコンテキストで呼び出される
		ResetExprContext --- メモリを解放するため

	FreeExecutorState
		クエリごとのコンテキストとExprContextを解放
		（個別のFreeExprContext呼び出しは不要）


EvalPlanQual (READ COMMITTED更新チェック)
---------------------------------------------

単純なSELECTの場合、エグゼキュータは現在のトランザクションから見えるスナップショットに従って有効なタプル（つまり、以前にコミットされたトランザクションによって挿入され、以前にコミットされたトランザクションによって削除されていないもの）のみに注意を払う必要があります。しかし、UPDATE、DELETE、およびMERGEの場合、オープン中または並行してコミットされたトランザクションによって変更されたタプルを変更または削除することは適切ではありません。SERIALIZABLE分離レベルで実行している場合、この条件が発生したときに単にエラーを発生させます。READ COMMITTED分離レベルでは、より多くの処理が必要です。

READ COMMITTEDモードの基本的な考え方は、並行トランザクションによってコミットされた変更済みタプル（必要に応じてコミットを待機した後）を取得し、クエリの修飾条件を再評価して、それらが依然として条件を満たすかどうかを確認することです。満たす場合、（UPDATEを実行している場合は）変更済みタプルから更新後のタプルを再生成し、最終的に変更済みタプルを更新/削除します。SELECT FOR UPDATE/SHAREも同様に動作しますが、そのアクションは変更済みタプルをロックし、そのバージョンのタプルに基づいて結果を返すことだけです。

このチェックを実装するために、実際には変更されたタプル（またはSELECT FOR UPDATEの場合はタプルのセット）ごとにクエリを最初から再実行します。その際、リレーションスキャンノードを調整して、現在のタプル（元のタプル、または変更済みタプルの更新されロックされたバージョン）のみを返すようにします。このクエリがタプルを返す場合、変更済みタプルは修飾条件を通過したことになります（UPDATEを実行している場合は、クエリの出力は適切に変更された更新タプルとなります）。タプルが返されない場合、変更済みタプルは修飾条件を満たさなかったため、現在の結果タプルを無視して元のクエリを続行します。

UPDATE/DELETE/MERGEでは、ターゲットリレーションのみをこのように処理する必要があります。SELECT FOR UPDATEでは、FOR UPDATEがフラグ付けされたリレーションが複数存在する可能性があるため、再チェックを実行する前に、そのような各リレーションで現在のタプルバージョンのロックを取得します。

クエリ内にロックされないリレーションが存在する可能性もあります（それらはUPDATE/DELETE/MERGEのターゲットではなく、SELECT FOR UPDATE/SHAREでロック対象として指定されてもいない）。テストクエリを再実行する際、ロックされた行と結合されたこれらのリレーションの同じ行を使用したいと考えます。通常のリレーションの場合、結合出力に行TIDを含め、そのTIDを再取得することで比較的低コストで実装できます。（再取得は高コストですが、再テストが不要な通常のケースを最適化しようとしています。）また、ValuesScanやFunctionScanのようなテーブル以外のリレーションも考慮する必要があります。これらについては、TIDに相当するものがないため、結合出力行にタプル値全体を含めることが唯一の実用的な解決策のようです。

SELECT FOR UPDATEのターゲットリストにおける集合を返す関数（SRF）は、スキャンタプルの特定のセットに対して最大で1つのタプルしか返されないことを保証するために禁止されています。そうでない場合、元のクエリが同じスキャンタプルのセットを複数回返すため、重複が発生します。同様に、UPDATEのターゲットリストでもSRFは禁止されています。そこでは、同じ行が複数回更新されるという結果になり、あまり有用ではなく、最初の更新以降の更新は何の効果も持たないためです。


非同期実行
----------------------

ForeignScanがネットワークI/Oを待機している場合など、データベースシステム外部のイベントをノードが待機している状況では、ノードが直ちにタプルを返せないが、後で返せる可能性があることを示すことが望ましいです。この状況を発見したプロセスは、単にブロックすることで対処できますが、これはすぐに進捗が得られる可能性のあるプランツリーの他の部分を実行するために費やせる時間を無駄にする可能性があります。これは、プランツリーにAppendノードが含まれている場合に特に発生しやすいです。非同期実行は、パフォーマンスを向上させるために、Appendノードの複数の部分を直列ではなく並行して実行します。

非同期実行のために、AppendノードはまずExecAsyncRequestを使用して、非同期対応の子ノードにタプルを要求する必要があります。次に、ExecAppendAsyncEventWaitを使用して非同期イベントループを実行する必要があります。最終的に、非同期要求が行われた子ノードがタプルを生成すると、AppendノードはExecAsyncResponseを介してイベントループからそれを受け取ります。現在の非同期実行の実装では、非同期対応の子ノードにタプルを要求する唯一のノードタイプはAppendであり、非同期対応である可能性がある唯一のノードタイプはForeignScanです。

通常、ExecAsyncResponseコールバックは、非同期にタプルを要求したいノードにとって必要な唯一のものです。一方、非同期対応ノードは一般的に3つのメソッドを実装する必要があります。

1. 非同期要求が行われると、ノードのExecAsyncRequestコールバックが呼び出されます。ノードはExecAsyncRequestPendingを使用して、後述するコールバックの要求が保留中であることを示す必要があります。あるいは、結果がすぐに利用可能な場合は、代わりにExecAsyncRequestDoneを使用できます。

2. イベントループがファイル記述子のイベントを待機またはポーリングしたい場合、ノードのExecAsyncConfigureWaitコールバックが呼び出され、ノードが待機したいファイル記述子のイベントを設定します。

3. ファイル記述子が準備完了になると、ノードのExecAsyncNotifyコールバックが呼び出されます。#1と同様に、別のコールバックのためにExecAsyncRequestPendingを使用するか、結果をすぐに返すためにExecAsyncRequestDoneを使用する必要があります。
