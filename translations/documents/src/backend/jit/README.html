Just-in-Timeコンパイルとは何か？
=================================

Just-in-Timeコンパイル（JIT）とは、何らかの形式で解釈されるプログラム評価をネイティブプログラムに変換するプロセスであり、それを実行時に行うものです。

例えば、`WHERE a.col = 3` のようなSQL述語を評価するために任意のSQL式を評価できる機能を使う代わりに、その式だけを処理するCPUでネイティブ実行可能な関数を生成することで、高速化を図ることが可能です。

これがahead-of-time（AOT）コンパイルではなくJITである理由は、クエリの実行時に行われるためであり、おそらく関連するタスクが何度も繰り返される場合にのみ行われるためです。PostgreSQLにおけるJITコンパイルの使われ方を考えると、解釈、AOT、JITの境界線はいくぶん曖昧です。

ネイティブプログラムに変換される「解釈されるプログラム」は、必ずしも古典的な意味でのプログラムである必要はないことに注意してください。例えば、タプルデフォーミング（tuple deforming）を、特定の種類のテーブルだけを扱うネイティブ関数にJITコンパイルすることは、タプルデフォーミングが一般的に「プログラム」として理解されていないにもかかわらず、非常に有益です。


なぜJITなのか？
==============

PostgreSQLの一部は、比較的小さなCPU集約型のコードによってボトルネックになることがよくあります。多くの場合、これは関連するコードが非常に汎用的でなければならないためです（例：任意のSQLレベルの式を、任意のテーブルに対して、任意の拡張機能がインストールされた状態で処理する）。これは多くの場合、多数の間接ジャンプや予測不可能な分岐、そして一般的に特定のタスクに対する命令数の増加につながります。例えば、データベース内の列と整数を比較する式を評価するだけでも、数百サイクルを要することになります。

ネイティブコードを生成することで、多数の間接ジャンプを直接分岐に置き換える（例：SQL演算子の実装への間接呼び出しを、その関数への直接呼び出しに置き換える）、あるいは完全に削除する（例：入力が定数であるためコンパイル時に分岐を評価して削除する）ことで、それらを取り除くことができます。同様に、多くの分岐を完全に削除することも可能です（例：入力が定数であるためコンパイル時に分岐を評価して削除する）。後者は、タプルデフォーミング中の分岐を削除するのに特に有益です。


JITの方法
==========

PostgreSQLはデフォルトで、JITを実行するためにLLVMを使用します。LLVMが選ばれた理由は、複数の大企業によって開発されており廃止される可能性が低いこと、PostgreSQLと互換性のあるライセンスを持っていること、そしてそのIR（中間表現）がClangコンパイラを使用してCから生成できるためです。


共有ライブラリの分離
--------------------

メインのPostgreSQLバイナリがLLVMに直接依存することを避けるため（これはOSのパッケージマネージャによってLLVMサポートが個別にインストールされることを妨げるため）、LLVM依存のコードはオンデマンドで読み込まれる共有ライブラリ内に配置されています。

これを行うことのさらなる利点は、JITコンパイルを提供する共有ライブラリを入れ替えることで、LLVMを使用しないJITコンパイルを比較的容易に評価できることです。

これを実現するために、JITを実行しようとするコード（例：式評価）は、jit.c内にあるLLVM非依存のラッパーを呼び出します。JITサポートを提供する共有ライブラリが読み込める場合（つまり、PostgreSQLがLLVMサポート付きでコンパイルされており、共有ライブラリがインストールされている場合）、式をJITコンパイルするタスクは共有ライブラリに引き渡されます。これは当然、JITプロバイダが読み込めない場合にjit.c内の関数が失敗することを許容する必要があります。

どの共有ライブラリが読み込まれるかはjit_provider GUCによって決定され、デフォルトは "llvmjit" です。

JITを実行するコードを共有ライブラリに隔離することは、残念ながら、コードのさまざまな部分に対してJITコンパイルを行うコードを、JITなしでそれを行うコードとは別に配置しなければならないことも意味します。例えば、execExprInterp.cのJIT版はexecutor/ではなくjit/llvm/に配置されています。


JITコンテキスト
---------------

パフォーマンスと利便性の理由から、JIT化された関数をまとめて生成および解放できるようにすることは有用です。例えば、クエリの初期化時に多数の関数を作成し、クエリ実行中にそれらを使用し、クエリの終了時にそれらすべてをまとめて解放することは非常に一般的です。

JIT化された関数の寿命はJITContextを介して管理されます。作成されたすべてのJIT化された関数が同じ寿命を持つべき作業については、そのようなコンテキストを正確に1つ作成する必要があります。例えば、実行される各クエリに対して、クエリのEState内に正確に1つのJITContextが存在します。JITContextの解放のみがプロバイダ非依存の機能に公開されます。作成はJIT実装によってオンデマンドで行われるためです。

個々の関数を別々に生成するよりも、複数の関数を一度に生成する方がコストが低く、それらをまとめて生成することで追加の最適化の機会が得られます。これを促進するために、LLVMプロバイダは関数の定義と、実行可能な形式での最適化および生成を分離しています。

現在の変更可能なモジュール（モジュールは本質的にCにおける翻訳単位のLLVM版）に関数を作成するには、以下を使用します。
  extern LLVMModuleRef llvm_mutable_module(LLVMJitContext *context);
これを使用して、LLVM APIを使い、必要なだけコードを生成できます。関数を実際に呼び出す必要があるときはいつでも、以下がその関数へのポインタを返します。
  extern void *llvm_get_function(LLVMJitContext *context, const char *funcname);

例えば、式評価の場合、この設定によりクエリ内のほとんどの関数をExecInitNode()中に生成し、関数が実際に最初に使用される時まで関数の生成を遅らせることができます。


エラーハンドリング
------------------

エラーハンドリングには2つの側面があります。第一に、生成された（LLVM IR）関数および出力された関数（mmap()されたセグメント）は、クエリの実行成功後とエラー後の両方でクリーンアップされる必要があります。これは、作成された各JITContextを現在のリソース所有者に登録し、エラー時やトランザクション終了時にそれをクリーンアップすることで行われます。リソースをより早く解放したい場合は、jit_release_context()を使用できます。

エラーハンドリングの2つ目の、あまり美しくない側面は、LLVM自体の中でのOOM（メモリ不足）ハンドリングです。上記のリソース所有者ベースのメカニズムは、ERROR発生時に生成されたコードをクリーンアップしますが、LLVM自体がメモリ不足になる可能性もあります。LLVMはデフォルトではC++の例外を*使用しません*。その割り当ては主に標準の「new」ハンドラ、およびmalloc()とmmap()の直接使用を通じて行われます。前者については「newハンドラ」が存在します：
http://en.cppreference.com/w/cpp/memory/new/set_new_handler
後者については、LLVMは失敗時に呼び出されるコールバックを提供します（残念ながら、mmap()の失敗はOOMエラーではなく致命的なものとして扱われます）。現時点で私たちが選択した方法は、LLVMを使用するコードが使用しなければならない2つの関数を用意することです：
extern void llvm_enter_fatal_on_oom(void);
extern void llvm_leave_fatal_on_oom(void);
これらはLLVMコードとやり取りする前に使用します。

libstdc++のnewエラーまたはLLVMエラーが発生すると、上記の関数によって設定されたハンドラがFATALエラーをトリガーします。外部ライブラリの内部状態を破損させるリスクなしにERRORを確実にスローすることは*できない*ため、ERRORではなくFATALを使用する必要があります。

上記のセクションの利用者はPG_TRY/CATCHブロックを使用する必要は*ありません*。ハンドラはトップレベルのsigsetjmp()レベルでリセットされます。

これらのハンドラをグローバルに設定するのではなく、比較的狭いenter/leaveで保護されたコードセクションを使用することで、PostGISのようにC++を使用する可能性のある拡張機能との負の相互作用を回避できます。LLVMのコード生成は任意のコードを実行すべきではないため、これらのハンドラを一時的に設定するだけで十分なはずです。


型同期
--------------------

「解釈される」PostgreSQLが行うタスクを実行できるコードを生成できるようにするには、コード生成が少なくともいくつかのPostgreSQL型について知っている必要があります。Cコードで手動で再作成することでLLVMに型定義を知らせることは可能ですが、それは失敗しやすく、手間がかかります。

その代わりに、JITに必要な各型を参照する小さなファイル（llvmjit_types.c）が1つあります。そのファイルはコンパイル時にビットコードに変換され、バックエンドでLLVMが初期化されるときに読み込まれます。

これは型定義を同期させるのには非常にうまく機能しますが、残念ながらIRレベルの表現はフィールド名を知らないため、オフセットを同期させることは*できません*。その代わり、必要なオフセットは元の構造体定義内で以下のようにdefineとして保持されます：
#define FIELDNO_TUPLETABLESLOT_NVALID 9
        int                     tts_nvalid;             /* # of valid values in tts_values */
これは依然として定義する必要がありますが、比較的少数のフィールドに対してのみ必要であり、構造体定義と一緒にまとめられているため、簡単に同期を保つことができます。


インライン化
------------

式をJIT化する大きな利点の1つは、呼び出された関数/演算子の本体をインライン化することで、PostgreSQLの拡張可能な関数/演算子メカニズムのオーバーヘッドを大幅に削減できることです。

インライン化のためだけに、よく使われる関数の2番目の実装を維持することは明らかに望ましくありません。その代わり、ClangコンパイラがLLVM IRを出力できるという事実を利用します。

そうすることで、2つのコピーを維持することなく、すべての演算子（例：int8eq、float8plなど）のLLVM IRを取得できます。これらのビットコードファイルはサーバーの以下にインストールされます。
  $pkglibdir/bitcode/postgres/
既存のLLVM機能（並列LTOコンパイル用）を使用して、これらにわたるインデックスが $pkglibdir/bitcode/postgres.index.bc に保存されます。

同様に、拡張機能は以下にコードをインストールできます。
  $pkglibdir/bitcode/[extension]/
これには以下が伴います。
  $pkglibdir/bitcode/[extension].index.bc

これは実際のライブラリのすぐ横に配置されます。拡張機能のインデックスは、対応する共有ライブラリ内に配置されているシンボルを検索するために使用されます。拡張機能内で使用されるシンボルは、インライン化される際、まずメインバイナリ内で検索され、次に拡張機能内で検索されます。


キャッシュ
----------

現在、生成された関数をキャッシュすることはまだできませんが、パフォーマンスの観点からは望ましいことです。問題は、生成された関数が通常、実行ごとのメモリへのポインタを含んでいることです。式評価メカニズムは、それを回避するために少し再設計する必要があります。基本的には、実行ごとのすべてのメモリは、絶対的なメモリポインタではなく、ExprStateに格納されたメモリブロックへのオフセットとして参照される必要があります。

それが解決されれば、生成されたLLVM IRをキーとするLRUキャッシュを追加することで、より高速なクエリに対しても最適化された関数を使用できるようになります。

より長期的なプロジェクトは、式コンパイルをプランナー段階に移行し、コンパイルされた式をプリペアドステートメントに関連付けられるようにすることです。

さらに高度なアプローチとしては、最初は最適化をほとんど行わずにJITを使用し、バックグラウンドで最適化バージョンを構築することが考えられます。しかし、それはまだ先の話です。


何をJITするか
===========

現在、式評価とタプルデフォーミングがJIT化されています。これらは分析クエリにおいて主要なCPUボトルネックになることが多いため選ばれましたが、有益なケースはこれらに限定されるわけではありません。

JIT化が有益であるためには、コードの一部が何よりもまずCPUのボトルネックである必要があります。しかし重要なことに、JIT化はそれによってオーバーヘッドが削除できる場合にのみ有益です。例えば、タプルデフォーミングの場合、列の数とその型に関する知識があれば、かなりの数の分岐を削除でき、式評価の場合、多くの間接ジャンプ/呼び出しを削除できます。どちらも当てはまらない場合、JIT化はリソースの無駄です。

JIT化の将来の道筋としては、タプルソート、COPYの解析/出力生成、そして将来的にはクエリのより大きな部分のコンパイルが挙げられます。


いつJITするか
===========

現在、JIT化に影響を与えるGUCがいくつかあります：- jit_above_cost = -1, 0-DBL_MAX - 合計コストがこれより高いすべてのクエリは、最適化（コストのかかる部分）なしでJITコンパイルされます。これは-O0に相当します。式やデータの変形がボトルネックになっている場合（主に動的分岐の削除により）、これだけでも通常は大幅な高速化につながります。
- jit_optimize_above_cost = -1, 0-DBL_MAX - 合計コストがこれより高いすべてのクエリは、最適化（コストのかかる部分）ありでJITコンパイルされます。
- jit_inline_above_cost = -1, 0-DBL_MAX - クエリのコストがこれより高い場合、インライン化が試行されます。

クエリの合計コストがこれらの制限を超えると、常にJITコンパイルが実行されます。

クエリのcpu_*コストが低い部分に対して個別のパスを生成するなど、代替のコスト計算モデルも考えられますが、そのオーバーヘッドに見合う効果があるかは疑わしいところです。もう一つの代替案として、個々の式が評価されると推定される回数をカウントし、それらの個々の式に対してJITコンパイルを実行する方法もあります。

実行回数に応じて個々の式をJITコンパイルするという一見明白なアプローチは、あまりうまくいかないことが判明しています。主な理由は、小さな関数を個別に多数生成することに大きなオーバーヘッドがかかるためです。また、JITコンパイルが行われるまでの時間が相対的な低速化を招き、JITコンパイルによる利得を食いつぶしてしまうためでもあります。
