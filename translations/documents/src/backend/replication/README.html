Walreceiver - libpqwalreceiver API
----------------------------------

walreceiver のトランスポート固有部分は、プライマリサーバーへの接続、WAL ファイルの受信、メッセージの送信を担当し、メインサーバーバイナリと libpq をリンクする必要を避けるために動的にロードされます。動的にロードされるモジュールは libpqwalreceiver サブディレクトリにあります。

この動的にロードされるモジュールは一連の関数を実装しており、それぞれの詳細は src/include/replication/walreceiver.h に記載されています。

この API は現時点では内部的なものと見なすべきですが、将来的に libpqwalreceiver のサードパーティによる置き換えを可能にし、WAL 受信のためのプラグ可能なメソッドを提供できるようにする可能性があります。

Walreceiver IPC
---------------

スタートアッププロセスでの WAL リプレイが、restore_command を使用して復元可能なアーカイブ済み WAL の末尾に達すると、(ストリーミングレプリケーションが設定されている場合) さらなる WAL を取得するために walreceiver プロセスを起動します。

Walreceiver は postmaster のサブプロセスであるため、スタートアッププロセスが直接フォークすることはできません。その代わり、postmaster にシグナルを送り、起動を要求します。ただしその前に、スタートアッププロセスは WalRcvData->conninfo と WalRcvData->slotname を埋め、WalRcvData->receiveStart の開始点を初期化します。

walreceiver がプライマリサーバーから WAL を受信し、それをディスク (pg_wal) に書き込んでフラッシュすると、WalRcvData->flushedUpto を更新し、WAL リプレイがどこまで進められるかをスタートアッププロセスに通知します。

Walreceiver は、新しい WAL を書き込んだかフラッシュしたとき、あるいは指定された間隔が経過したときに、レプリケーションの進捗状況に関する情報をプライマリサーバーに送信します。これは報告目的で使用されます。

Walsender IPC
-------------

シャットダウン時、postmaster は walsender プロセスを通常のバックエンドとは異なる方法で処理します。通常のバックエンドが終了するのを待ってからシャットダウンチェックポイントを書き込み、pgarch やその他の補助プロセスを終了させますが、walsender の場合はこれが望ましくありません。なぜなら、プライマリがシャットダウンされる前に、スタンバイサーバーがシャットダウンチェックポイントを含むすべての WAL を受信する必要があるからです。そのため、postmaster は walsender を pgarch プロセスのように扱い、すべての通常のバックエンドが終了し、checkpointer がシャットダウンチェックポイントを発行した後の PM_WAIT_XLOG_ARCHIVAL フェーズで終了するように指示します。

postmaster が接続を受け入れると、ハンドシェイクと認証を処理するために直ちに新しいプロセスをフォークし、そのプロセスはバックエンドになるように初期化されます。この時点では、postmaster はそのプロセスが通常のバックエンドになるのか walsender プロセスになるのかを知りません（それは接続ハンドシェイクで示されます）。そのため、postmaster が walsender プロセスを識別できるようにするための追加のシグナル伝達が必要です。

walsender プロセスが起動すると、PMSignal 配列内で自身を walsender プロセスとしてマークします。これにより、postmaster は通常のバックエンドと区別できるようになります。

postmaster が walsender を通常のバックエンドだと誤認しても大きな害はないことに注意してください。単にシャットダウンフェーズで walsender が早期に終了するだけです。walsender は、初期化が完了して PMSignal 配列に自身をマークするまでは通常のバックエンドのように見え、プロセス終了時には PMSignal スロットのマークを解除した後に同様に見えます。

各 walsender は WalSndCtl 配列からエントリを割り当て、レプリケーションの進捗状況に関する情報を追跡します。ユーザーは統計ビューを通じてこれらを監視できます。
