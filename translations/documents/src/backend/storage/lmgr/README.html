ロックの概要
===========

Postgresは4種類のプロセス間ロックを使用します。

* スピンロック (Spinlocks)。これらは*非常に*短期間のロックを意図したものです。もしロックを数十命令以上保持する場合、あるいは何らかのカーネル呼び出し（または非自明なサブルーチン呼び出し）をまたぐ場合は、スピンロックを使用してはいけません。スピンロックは主に軽量ロックのインフラストラクチャとして使用されます。これらは、利用可能であればハードウェアの原子的なテスト・アンド・セット命令を使用して実装されます。待機中のプロセスは、ロックを取得できるまでビジーループを行います。デッドロック検出やエラー時の自動解放、その他の便利な機能は一切ありません。1分程度経過してもロックを取得できない場合はタイムアウトが発生します（これは意図されたロック保持時間と比較するとほぼ永遠に等しいため、確実にエラー状態です）。

* 軽量ロック (LWLocks)。これらのロックは通常、共有メモリ内のデータ構造へのアクセスをインターロックするために使用されます。LWLocksは排他ロックモードと共有ロックモードの両方をサポートします（共有オブジェクトへの読み書きアクセスおよび読み取り専用アクセスのため）。デッドロック検出の仕組みはありませんが、LWLockマネージャはelog()によるリカバリ中に保持されているLWLocksを自動的に解放するため、LWLocksを保持したままエラーを発生させても安全です。ロックの競合がない場合、LWLockの取得や解放は非常に高速です（数十命令程度）。プロセスがLWLockを待機しなければならない場合、CPU時間を消費しないようにSysVセマフォ上でブロックされます。待機中のプロセスには、到着順にロックが与えられます。タイムアウトはありません。

* 通常ロック (Regular locks、別名ヘビーウェイトロック)。通常ロックマネージャは、テーブル駆動型のセマンティクスを持つさまざまなロックモードをサポートしており、完全なデッドロック検出とトランザクション終了時の自動解放機能を備えています。通常ロックは、すべてのユーザー主導のロック要求に使用されるべきです。

* SIReadLock述語ロック。詳細は個別のREADME-SSIファイルを参照してください。

スピンロックまたは軽量ロックのいずれかを取得すると、それらすべてのロックが解放されるまで、クエリキャンセルおよびdie()割り込みが保留されます。ただし、通常ロックにはこのような制限はありません。また、通常ロックを待機している間はクエリキャンセルおよびdie()割り込みを受け付けますが、スピンロックやLWロックを待機している間は受け付けないことにも注意してください。したがって、待機時間が数秒を超える可能性がある場合にLWロックを使用することは推奨されません。

このREADMEファイルの残りの部分では、通常ロックマネージャについて詳しく説明します。


ロックデータ構造
----------------

ロックメソッドは、全体的なロック動作を記述します。現在、DEFAULTとUSERの2つのロックメソッドがあります。

ロックモードは、ロックの種類（読み取り/書き込み、または共有/排他）を記述します。原則として、各ロックメソッドは異なる競合ルールを持つ独自のロックモードセットを持つことができますが、現在はDEFAULTメソッドとUSERメソッドは同一のロックモードセットを使用しています。詳細はsrc/include/storage/lock.hを参照してください。（ロックモードは、コードやドキュメントの場所によってはロックタイプとも呼ばれます。）

共有メモリ内にロックを記録するための主な方法は2つあります。第一のメカニズムは、2つの主要な構造体を使用します。ロック可能なオブジェクトごとのLOCK構造体と、ロックおよび要求者ごとのPROCLOCK構造体です。LOCKオブジェクトは、現在ロックが保持されているか要求されている各ロック可能オブジェクトに対して存在します。PROCLOCK構造体は、各LOCKオブジェクトに対してロックを保持または要求している各バックエンドに対して存在します。

また、バックエンドが非常に特定の特性を持つ限られた数のロックを記録するために使用できる特別な「ファストパス (fast path)」メカニズムもあります。それらはDEFAULTロックメソッドを使用しなければならず、データベースリレーション（共有リレーションではない）上のロックを表さなければならず、競合する可能性が低い「弱い」ロック（AccessShareLock、RowShareLock、またはRowExclusiveLock）でなければならず、さらにシステムが競合するロックが存在しないことを迅速に検証できなければなりません。詳細は以下の「ファストパスロッキング (Fast Path Locking)」を参照してください。

各バックエンドは、現在保持または要求している各ロック可能オブジェクトおよびロックモードに対して、共有されないLOCALLOCK構造体も維持します。共有ロック構造体は、ロック可能オブジェクト/ロックモード/バックエンドごとに単一のロック付与のみを許可します。しかし、バックエンド内部では、同じロックがトランザクション内で複数回要求および解放される可能性があり、またトランザクション的およびセッション全体の両方で保持されることもあります。内部要求カウントはLOCALLOCKに保持されるため、それらを変更するために共有データ構造にアクセスする必要はありません。

---------------------------------------------------------------------------

ロックマネージャのLOCKオブジェクトには以下が含まれます：

tag -
    共有メモリのロックハッシュテーブル内でロックをハッシュ化するために使用されるキーフィールド。tagの内容は、本質的に個々のロック可能オブジェクトを定義します。サポートされているロック可能オブジェクトの種類の詳細はinclude/storage/lock.hを参照してください。これは、常に正しいバイト数をゼロクリアすることを保証するために個別の構造体として宣言されています。コンパイラが構造体に挿入する可能性のあるアラインメントパディングバイトをゼロクリアすることは非常に重要です。そうしないと、ハッシュ計算がランダムになってしまいます。（現在、struct LOCKTAGの定義にはパディングバイトが含まれないように注意しています。）

grantMask -
    このビットマスクは、特定のロック可能オブジェクトに対して現在どのような種類のロックが保持されているかを示します。これは（ロックテーブルの競合テーブルに対して）新しいロック要求が既存の保持されているロックタイプと競合するかどうかを判断するために使用されます。競合は、grantMaskと要求されたロックタイプの競合テーブルエントリとの間のビット単位のAND演算によって決定されます。grantMaskのビットiは、granted[i] > 0の場合にのみ1になります。

waitMask -
    このビットマスクは、待機中のロックの種類を示します。waitMaskのビットiは、requested[i] > granted[i]の場合にのみ1になります。

procLocks -
    これは、ロックオブジェクトに関連付けられたすべてのPROCLOCK構造体の共有メモリキューです。付与済みおよび待機中の両方のPROCLOCKがこのリストに含まれていることに注意してください（実際、同じPROCLOCKがすでに付与されたロックを持ちつつ、さらに追加のロックを待機している可能性があります！）。

waitProcs -
    これは、別のバックエンドがこのロックを解放するまで待機（スリープ）しているバックエンドに対応する、すべてのPGPROC構造体の共有メモリキューです。プロセス構造体は、ロックが解放されたときにウェイクアップすべきかどうかを判断するために必要な情報を保持します。

nRequested -
    このロックを取得しようとした回数のカウントを保持します。このカウントには、競合のためにスリープ状態にされたプロセスによる試行も含まれます。また、例えばバックエンドプロセスが最初に読み取りを取得し、次に書き込みを取得した場合、同じバックエンドを2回カウントします。（ただし、バックエンド内での同じロック/ロックモードの複数回の取得はここでは複数回カウントされません。それらはバックエンドのLOCALLOCK構造体にのみ記録されます。）

requested -
    各タイプのロックが何回試行されたかのカウントを保持します。要素1からMAX_LOCKMODES-1のみが使用されます。これらは定義されたロックタイプの定数に対応するためです。requested[]の値を合計するとnRequestedと等しくなるはずです。

nGranted -
    このロックが何回正常に取得されたかのカウントを保持します。このカウントには、競合のために待機中の試行は含まれません。それ以外の場合、カウントルールはnRequestedと同じです。

granted -
    現在保持されている各タイプのロックが何回あるかのカウントを保持します。ここでも要素1からMAX_LOCKMODES-1のみが使用されます（0は使用されません）。また、requested[]と同様に、granted[]の値を合計するとnGrantedの値と等しくなるはずです。

常に 0 <= nGranted <= nRequested であり、各iについて 0 <= granted[i] <= requested[i] である必要があります。すべての要求カウントがゼロになると、LOCKオブジェクトは不要になり解放されます。

---------------------------------------------------------------------------

ロックマネージャのPROCLOCKオブジェクトには以下が含まれます：

tag -
    共有メモリのPROCLOCKハッシュテーブル内のエントリをハッシュ化するために使用されるキーフィールド。これは、常に正しいバイト数をゼロクリアすることを保証するために個別の構造体として宣言されています。コンパイラが構造体に挿入する可能性のあるアラインメントパディングバイトをゼロクリアすることは非常に重要です。そうしないと、ハッシュ計算がランダムになってしまいます。（現在、struct PROCLOCKTAGの定義にはパディングバイトが含まれないように注意しています。）

    tag.myLock
        このPROCLOCKが対象とする共有LOCKオブジェクトへのポインタ。

    tag.myProc
        このPROCLOCKを所有するバックエンドプロセスのPGPROCへのポインタ。

    注：ここでポインタを使用しても問題ありません。PROCLOCKはロックやprocよりも長生きすることはないためです。したがって、同じタグ値が他のタイミングで別の意味を持つ可能性があるとしても、タグは必要な期間中一意です。

holdMask -
    このPROCLOCKによって正常に取得されたロックモードのビットマスク。これはLOCKオブジェクトのgrantMaskのサブセットであり、また（PGPROCが現在このロック上の別のロックモードを待機している場合は）PGPROCオブジェクトのheldLocksマスクのサブセットである必要があります。

releaseMask -
    LockReleaseAll中に解放される予定のロックモードのビットマスク。これはholdMaskのサブセットでなければなりません。パーティションLWLockを取得せずに変更されるため、PROCLOCKを所有するバックエンド以外のバックエンドがこれを調査/変更することは安全ではないことに注意してください。

lockLink -
    同じLOCKに対するすべてのPROCLOCKオブジェクトの共有メモリキューのリストリンク。

procLink -
    同じバックエンドに対するすべてのPROCLOCKオブジェクトの共有メモリキューのリストリンク。

ロックマネージャの内部ロッキング
-----------------------------

PostgreSQL 8.2以前は、ロックマネージャが使用するすべての共有メモリデータ構造は、単一のLWLockであるLockMgrLockによって保護されていました。これらのデータ構造に関わるすべての操作は、LockMgrLockを排他的にロックする必要がありました。驚くことではありませんが、これが競合のボトルネックとなりました。競合を減らすために、ロックマネージャのデータ構造は複数の「パーティション」に分割され、それぞれが独立したLWLockによって保護されるようになりました。ほとんどの操作は、作業対象の単一パーティションをロックするだけで済みます。詳細は以下の通りです。

* 各ロックは、LOCKTAG値のハッシュに基づいていずれかのパーティションに割り当てられます。パーティションのLWLockは、そのパーティションのすべてのLOCKオブジェクトとその子であるPROCLOCKを保護すると見なされます。

* LOCKおよびPROCLOCK用の共有メモリハッシュテーブルは、異なるパーティションが異なるハッシュチェーンを使用するように構成されているため、異なるパーティション内のオブジェクトを操作する際に競合は発生しません。これは、LOCKテーブルに対するdynahash.cの「パーティション化テーブル」メカニズムによって直接サポートされています。パーティション番号がLOCKTAGのdynahashハッシュ値の下位ビットから取得されることを保証するだけで済みます。PROCLOCKでこれを機能させるには、PROCLOCKのハッシュ値が関連付けられたLOCKと同じ下位ビットを持つことを保証する必要があります。これには専用のハッシュ関数が必要です（proclock_hashを参照）。

* 以前は、各PGPROCはそれに属するPROCLOCKの単一リストを持っていました。これは現在パーティションごとのリストに分割されており、特定のPROCLOCKリストへのアクセスは関連するパーティションのLWLockによって保護できるようになっています。（このルールにより、あるバックエンドが別のバックエンドのPROCLOCKリストを操作できるようになります。これは当初は必要ありませんでしたが、ファストパスロッキングに関連して現在必要となっています。下記を参照してください。）

* PGPROCの他のロック関連フィールドは、PGPROCがロックを待機している場合にのみ重要となるため、待機中のロックのパーティションLWLockによって保護されていると見なされます。通常のロックの取得および解放については、目的のロックを含むパーティションをロックすれば十分です。デッドロックのチェックは一般的に複数のパーティションに触れる必要があるため、単純化のために、パーティション番号順にすべてのパーティションをロックするようにしています。（LWLockのデッドロックを防ぐため、一度に複数のパーティションをロックする必要があるバックエンドは、必ずパーティション番号順にロックしなければならないというルールを設けています。）典型的なケースではすべてのパーティションに触れずにデッドロックチェックを行える可能性もありますが、適切に機能しているシステムではデッドロックチェックがパフォーマンスに影響するほど頻繁に発生することはないはずであるため、これを実現しようとすることは生産的な努力とは言えません。

バックエンドの内部的な LOCALLOCK ハッシュテーブルはパーティション分割されていません。LOCALLOCK テーブルのエントリには locktag ハッシュコードのコピーを格納しており、そこからパーティション番号を計算できますが、これは純粋に速度と空間のトレードオフです。必要に応じて LOCKTAG からパーティション番号を再計算することも可能です。

Fast Path Locking
-----------------

Fast path locking は、非常に頻繁に取得・解放されるものの競合することは稀な特定の種類のロックについて、その取得・解放のオーバーヘッドを削減するために設計された特殊なメカニズムです。現在、これには以下の2つのカテゴリのロックが含まれます。

(1) 弱いリレーションロック。SELECT、INSERT、UPDATE、および DELETE は、操作対象のすべてのリレーションに加え、内部的に使用される可能性のある様々なシステムカタログに対してロックを取得する必要があります。多くの DML 操作は同じテーブルに対して同時に並行して実行できます。CLUSTER、ALTER TABLE、DROP といった DDL 操作、あるいは LOCK TABLE といった明示的なユーザー操作のみが、DML 操作によって取得される「弱い」ロック（AccessShareLock、RowShareLock、RowExclusiveLock）と競合を引き起こします。

(2) VXID ロック。すべてのトランザクションは、自身の仮想トランザクション ID に対してロックを取得します。現在、これらのロックを待機する唯一の操作は CREATE INDEX CONCURRENTLY と Hot Standby（競合が発生した場合）であるため、ほとんどの VXID ロックは、他の誰かが気にする必要なく所有者によって取得および解放されます。

主要なロックメカニズムはこのワークロードにはうまく対応できません。ロックマネージャのロックはパーティション分割されていますが、特定のリレーションの locktag は必ず1つのパーティションにのみ分類されます。そのため、多くの短いクエリが同じリレーションにアクセスしている場合、そのパーティションのロックマネージャのパーティションロックが競合のボトルネックとなります。この影響は2コアのサーバーでも測定可能であり、コア数が増えるにつれて非常に顕著になります。

このボトルネックを緩和するため、PostgreSQL 9.2 以降では、各バックエンドは主要なロックテーブルを使用する代わりに、PGPROC 構造体内の配列に共有されていないリレーションに対する限られた数のロックを記録することが許可されています。このメカニズムは、ロック取得時に競合するロックが存在しないことを確認できる場合にのみ使用できます。

このアルゴリズムの重要な点は、共有 LWLock やスピンロックを奪い合うことなく、競合する可能性のあるロックが存在しないことを検証できなければならないという点です。そうでなければ、この取り組みは単に競合のボトルネックを別の場所に移動させるだけになってしまいます。これを実現するために、1024個の整数カウンタの配列を使用します。これは実質的にロック空間を1024分割するものです。各カウンタは、そのパーティションに該当する共有されていないリレーションに対する「強い」ロック（すなわち ShareLock、ShareRowExclusiveLock、ExclusiveLock、および AccessExclusiveLock）の数を記録します。このカウンタがゼロでない場合、そのパーティション内で新しいリレーションロックを取得するために fast path メカニズムを使用してはなりません。強いロックを取得しようとするプロセスはカウンタをインクリメントし、各バックエンドの配列をスキャンして一致する fast-path ロックを探します。見つかったものは、適切なロック競合およびデッドロック検出を確実にするため、ロックを取得しようとする前に主要なロックテーブルへ転送しなければなりません。

SMP システムでは、適切なメモリ同期を保証する必要があります。ここでは、LWLock の取得がメモリシーケンスポイントとして機能するという事実に依存しています。A がストアを実行し、A と B の両方がいずれかの順序で LWLock を取得し、その後 B が同じメモリ位置でロードを実行した場合、B は A のストアを確実に認識できます。この場合、各バックエンドの fast-path ロックキューは LWLock によって保護されています。fast-path ロックを取得しようとするバックエンドは、競合する強いロックの存在を確認するために FastPathStrongRelationLocks を調べる前に、この LWLock を取得します。また、強いロックを取得しようとするバックエンドは、fast-path メカニズムを介して取得された一致する弱いロックを共有ロックテーブルに転送しなければならないため、バックエンドの fast-path キューを保護するすべての LWLock を順番に取得します。したがって、FastPathStrongRelationLocks を調べてゼロであれば、その値は真にゼロであるか、あるいは古い値であったとしても、強いロックを取得しようとするプロセスは我々が保持しているバックエンドごとの LWLock（あるいは実際には最初のバックエンドごとの LWLock でさえ）をまだ取得しておらず、我々が弱いロックを取得した際にそれを認識することになります。

Fast-path VXID ロックは FastPathStrongRelationLocks テーブルを使用しません。VXID に対して取得される最初のロックは、常に所有者による ExclusiveLock です。その後のロック取得者は、VXID が終了するのを待機する共有ロック取得者です。実際、VXID ロックが（他の方法で VXID の終了を待つのではなく）ロックマネージャを使用する唯一の理由は、デッドロック検出のためです。したがって、最初の VXID ロックは競合チェックなしで *常に* fast path を介して取得できます。その後のロック取得者は、ロックがメインのロックテーブルに転送されているかどうかを確認し、転送されていなければ転送しなければなりません。VXID を所有するバックエンドは、トランザクション終了時にメインのロックテーブル内のエントリを確実にクリーンアップするように注意する必要があります。

デッドロック検出は fast-path データ構造を調べる必要はありません。デッドロックに関与する可能性のあるロックは、事前にメインテーブルに転送されている必要があるためです。

デッドロック検出アルゴリズム
----------------------------

ユーザーのトランザクションは任意の順序でロックを要求できるため、デッドロックが発生する可能性があります。私たちは本質的にかなり標準的なデッドロック検出/解消アルゴリズムを使用していますが、Postgres の汎用的なロックモデルに対処するために多くの特別な考慮事項が必要です。

重要な設計上の考慮事項は、デッドロックがない場合には日常的な操作（ロックの付与と解放）を高速に実行し、デッドロック処理のオーバーヘッドを可能な限り回避することです。「楽観的待機」アプローチを採用しており、プロセスが目的のロックを即座に取得できない場合、デッドロックチェックを行わずにスリープ状態に入ります。ただし、DeadlockTimeout ミリ秒（通常は1秒に設定）の遅延タイマーも設定します。プロセスが目的のロックを取得する前に遅延が経過した場合、デッドロック検出/解消コードを実行します。通常、このコードはデッドロック状態ではないと判断し、プロセスはスリープに戻り、ロックが付与されるまで静かに待機します。しかし、実際にデッドロック状態が存在する場合はそれが解消され、通常は検出したプロセスのトランザクションをアボートすることで解決されます。このようにして、ロックの待機時間が DeadlockTimeout 未満である場合にはデッドロック処理のオーバーヘッドを回避しつつ、エラーが発生した場合には不当に長い検出遅延を課さないようにしています。

ロックの取得（LockAcquire および ProcSleep ルーチン）は以下のルールに従います。

1. 既存または待機中のロック要求と競合しない場合、あるいはプロセスがすでに同じ種類のロックを保持している場合（例：読み取りロックを2回取得してもペナルティはない）、ロック要求は即座に付与されます。プロセスは自分自身と競合することはないことに注意してください（例：排他ロックを保持しているときに読み取りロックを取得することは可能です）。

2. そうでない場合、プロセスはロックの待機キューに参加します。通常はキューの最後に追加されますが、例外があります。もしプロセスが、保留中の待機者の要求と競合する同じロック可能オブジェクトに対するロックをすでに保持している場合、そのプロセスは最初のそのような待機者の直前に挿入されます。（このチェックを行わなかった場合、デッドロック検出コードが競合を解決するためにキューの順序を調整しますが、ProcSleep でチェックを行い、この場合にデッドロックタイムアウトの遅延を回避することは比較的低コストです。）キューの最後より前に挿入する場合の特別なケースとして、プロセスの要求が挿入ポイントより前の既存のロックや待機中の要求と競合しない場合は、待機せずにロックを付与します。

ロックが解放されると、ロック解放ルーチン（ProcLockWakeup）はロックオブジェクトの待機キューをスキャンします。各待機者は、(a) その要求がすでに付与されているロックと競合せず、(b) その要求が先に待機している（まだ起床していない）待機者の要求と競合しない場合に起床されます。ルール (b) は、競合する要求が到着順に付与されることを保証します。デッドロックを回避するために、後の待機者が競合する前の待機者の前に割り込むことを許可しなければならないケースがありますが、これらのケースを認識するのは ProcLockWakeup の責任ではありません。代わりに、必要に応じてデッドロック検出コードが待機キューを並べ替えます。

デッドロックチェックを実行するために、様々なプロセスを指向性グラフ（待機グラフまたは WFG）のノードと見なす標準的な手法を使用します。A が B を待機している場合（すなわち、A が何らかのロックを待機しており、B が競合するロックを保持している場合）、プロセス A からプロセス B へのグラフエッジが存在します。WFG にサイクルが含まれている場合にのみ、デッドロック状態が存在します。開始点に戻るかどうかを確認するために、待機エッジに沿って外向きに検索することでサイクルを検出します。3つの可能な結果があります。

1. すべての出力パスが実行中のプロセス（出力エッジを持たない）で終了する。

2. 開始点にループバックすることでデッドロックが検出される。このようなデッドロックは、開始点のロック要求をキャンセルし、そのトランザクションでエラーを報告することで解決します。これは通常、トランザクションのアボートと、そのトランザクションが保持しているロックの解放につながります。サイクルを解消するには1つの要求をキャンセルすれば十分であり、関与しているすべてのトランザクションを強制終了する必要はありません。

3. いくつかのパスが開始点以外のノードにループバックする。これはデッドロックを示していますが、開始プロセスは関与していません。このようなデッドロックの解決は関与しているプロセスの責任であるという理由で、この状態は無視します。開始プロセスを強制終了してもデッドロックは解決しません。したがって、ケース1と3はどちらも「デッドロックなし」と報告します。

Postgres の状況は、以下の2つの理由から、標準的なデッドロック検出の議論よりも少し複雑です。

1. 待機者の要求と競合する（競合しない）ロックタイプの PROCLOCK が複数存在する可能性があるため、プロセスは複数の他のプロセスを待機している可能性があります。しかし、これは実際には困難を生じません。単に複数の出力エッジを追跡できるように準備しておくだけです。

2. あるロックの待機キューにおいてプロセス A がプロセス B の後ろにあり、それらの要求するロックが競合する場合、ProcLockWakeup は B より前に A を起床させることはないため、A は B を待機していると言わなければなりません。これにより、WFG に追加のエッジが作成されます。これらを、すでに保持されているロックによって誘発される「ハード」エッジに対して、「ソフト」エッジと呼びます。B がすでに A の要求と競合するロックを保持している場合、それらの関係はソフトエッジではなくハードエッジであることに注意してください。

「ソフト」ブロック、または待機優先ブロックは、ハードブロックと同様にデッドロックを誘発する可能性があります。しかし、関与するトランザクションをアボートせずにソフトブロックを解決できる場合があります。代わりに待機キューの順序を並べ替えることができます。この並べ替えにより、キューの順序が逆転した競合する要求を持つ2つのプロセス間のソフトエッジの方向が逆転します。新しいサイクルを作成せずにサイクルを解消する並べ替えを見つけることができれば、アボートを回避できます。そのような並べ替えの可能性をチェックすることが、このアルゴリズムの最も難しい部分です。デッドロック検出器の主力となるルーチンは FindLockCycle() であり、これには開始点となるプロセス（待機中のプロセスでなければならない）が与えられます。
このルーチンは、前述の通り「待機先（waits-for）」エッジを再帰的に外側に向かってスキャンします。
開始点を含むサイクルが見つからなければ、"false" を返します。
（前述の通り、開始点を含まないサイクルは無視できます。）
そのようなサイクルが見つかると、FindLockCycle() は "true" を返し、巻き戻しの過程でサイクルに含まれるすべての「ソフト」エッジのリストを作成します。
結果として得られたリストが空であれば、ハードデッドロックが発生しており、その構成では成功できません。しかし、リストが空でなければ、待機キューの並べ替えを通じてリストされたエッジのいずれか1つを反転させることで、そのサイクルを解消できます。このような反転によって別の場所にサイクルが作成される可能性があるため、あらゆる可能性を試す必要があるかもしれません。したがって、現在の実際の順序だけでなく、仮説的な構成（待機順序）に対しても FindLockCycle() を呼び出せるようにする必要があります。

これを処理する最も簡単な方法は、並べ替えを検討している各待機キューに対して、提案された新しいキュー順序を示すルックアサイドテーブルを持つことのようです。このテーブルは FindLockCycle によってチェックされ、ルックアサイドテーブルにエントリがある各ロックについては、実際の順序ではなく提案されたキュー順序が使用されます。

提案された新しいキュー順序は、既存のエントリの「トポロジカルソート」を行うことで構築されます。現在反転を検討している各ソフトエッジは、トポロジカルソートが強制しなければならない半順序のプロパティを作成します。デッドロックに関与していないプロセスの到着順序を不必要に壊さないよう、入力順序を可能な限り維持するソート手法を使用しなければなりません。（例えば、Knuth に示されている tsort メソッドはこれに該当しませんが、各ステップで最初の有効な候補を出力する単純な二重ループメソッドで簡単に実現できます。幸いなことに、半順序制約の数はそれほど多くないと思われるため、非常に効率的なソートアルゴリズムは必要ありません。）トポロジカルソートの失敗は、順序制約が競合していること、つまり最後に追加されたソフトエッジの反転が以前のエッジ反転と競合していることを示していることに注意してください。可能な並べ替えが機能しない場合に無限ループに陥るのを避けるため、このケースを検出する必要があります。そうしないと、反転を試みてサイクルが残っていることを確認し、そのサイクルを取り除くために反転を元に戻そうとする、といった動作を繰り返す可能性があります。トポロジカルソートの失敗は、この文脈において「反転を元に戻すこと」が正当な操作ではないことを示しています。

したがって、並べ替え手法の基本的なステップは、サイクル内のソフトエッジのリスト（FindLockCycle() によって返される）を取得し、検討中の既存の制約にトポロジカルソートの制約として各エッジの反転を順次試すことです。すべてのサイクルを一度に解消できる制約のセットがあるかどうかを調べるために、そのような制約のセットすべてを再帰的に検索します。これは不可能に思えるほど非効率に見えるかもしれませんが、実際にはデッドロックサイクルは通常非常に少なく、規模も大きくないため（存在する場合でも）、大きな問題にはなりません。したがって、組み合わせ的な非効率性は問題になりません。それに、不必要なときにトランザクションを中断するよりも、考えられるすべての回避策をチェックしたことを保証するために時間を費やす方が良いのです。

各エッジ反転の制約は、待機中のプロセス A を、両者が属する待機キュー内のブロックしているプロセス B よりも前に移動するように要求するものと見なすことができます。このアクションは、目的のソフトエッジを反転させるとともに、A と、A が追い越す他のプロセスとの間の他のソフトエッジも反転させます。他のエッジは影響を受けません（これは、必要以上にキューを並べ替えないようにするという、トポロジカルソート手法に対する制約であることに注意してください）。したがって、FindLockCycle(A) も FindLockCycle(B) もサイクルを発見しなければ、新しいデッドロックサイクルを作成していないと確信できます。前述の FindLockCycle の動作を考慮すると、これらの各検索は必要かつ十分です。なぜなら、元の開始点から始まる FindLockCycle は、元の開始点を含まず A や B を含むサイクルについては警告しないからです。

つまり、待機キューの提案された並べ替えは、1つ以上の壊れたソフトエッジ A->B によって決定され、関係する各待機キューのトポロジカルソートの出力によって完全に指定され、元の開始点および言及された各プロセス（A および B）から開始して FindLockCycle() を呼び出すことでテストされます。どのテストでもサイクルが検出されなければ、有効な構成であるため、ソート出力に従って待機キューを並べ替えることで実装できます（その後、待機者がウェイクアップ可能になった場合に備えて、各並べ替えられたキューに対して ProcLockWakeup を適用します）。いずれかのテストでソフトサイクルが検出された場合は、そのサイクル内の各ソフトリンクを順番に提案された並べ替えリストに追加することで解決を試みることができます。これは、実行可能な並べ替えが見つかるか、存在しないと判断されるまで再帰的に繰り返されます。後者の場合、外側のレベルで元の開始点トランザクションを中断することでデッドロックを解決します。

並べ替えが試行される特定の順序は、FindLockCycle() がスキャンする順序に依存するため、待機キューの実行可能な並べ替えが複数ある場合、どれが選択されるかは未定義です。より重要なのは、成功につながる可能性のあるすべてのキューの並べ替えを確実に試すことです。（例えば、A、B、C の順序で、必要な順序制約が C が A より前、かつ B が C より前である場合、まず A が C より前では機能しないことを発見し、C、A、B の順序での並べ替えを試みます。これは最終的に、B が C より前という追加制約の発見につながります。）

雑記
----

1. デッドロックチェックの非同期呼び出しによってデッドロックが見逃されることはないことが簡単に証明できます。WFG（待機先グラフ）内のデッドロックサイクルは、サイクルの最後のエッジが追加されたときに形成されます。したがって、サイクル内で最後に待機したプロセス（そのエッジの出力元）は、後で CheckDeadLock を実行したときに確実にサイクルを検出し、解決します。これは、そのエッジの追加によって複数のサイクルが作成された場合でも当てはまります。プロセスはそれらの追加サイクルに気づかずに中断するかもしれませんが、それは特に問題ではありません。デッドロックが発生する唯一の他の可能性は、デッドロック解決のための待機キューの並べ替え中ですが、そのアルゴリズムは並べ替えを実際に実行しようとする前に、新しいデッドロックを作成しないことを証明することは既に確認済みです。

2. 最後に待機したプロセスを中断することでデッドロックが解決されるとは限りません。サイクル内の以前の待機者がまだ CheckDeadLock を実行していない場合、最初に実行した者が犠牲者となります。

3. ProcLockWakeup は待機キューのすべてのメンバーを検査するため、ライブ（ウェイクアップ可能）なプロセスが見逃されることはありません（ちなみに、7.0 の実装ではそうではありませんでした）。したがって、ロックが解放された後や待機キューが並べ替えられた後に常に ProcLockWakeup が呼び出されるのであれば、ウェイクアップ可能なプロセスをウェイクアップし損ねることはありません。また、LockErrorCleanup（外部要因による待機者の中断）は、キャンセルされた待機者が他の待機者をソフトブロックしていた場合に備えて、ProcLockWakeup を実行しなければならないことにも注意してください。

4. ソフトエッジを探す際に待機キューを先頭からスキャンするように注意することで、過剰な並べ替え試行作業を最小限に抑えることができます。例えば、キューの順序が A,B,C で、C が A と B の両方とデッドロックの競合を起こしている場合、「C が B より前」という無駄な時間を費やすのではなく、「C が A より前」という制約を先に生成したいと考えます。そのため、待機キューの先頭から開始して、C から出力されるソフトエッジを探します。

5. デッドロック検出コードに必要な作業データ構造は、MaxBackends から計算されたエントリ数に制限できます。したがって、バックエンドの起動時に必要な最悪のケースのスペースを割り当てることができます。これは、実行時にワークスペースを割り当てようとするよりも安全なアプローチのように思えます。デッドロック検出器のメモリが不足するリスクは避けたいからです。そうでないと、デッドロックが検出されるという保証が全くなくなってしまいます。

6. 自動バキューム（autovacuum）のキャンセルを実装するためにデッドロック検出器を悪用しています。検出器を実行して、待機先グラフに自動バキュームワーカーが含まれていることがわかった場合、その PGPROC へのポインタを保存し、特別なリターンコードを返します（ハードデッドロックが検出された場合を除く）。呼び出し元はその後、キャンセルシグナルを送信できます。これは、自動バキュームのロック優先順位が低い（例：テーブルに対する DDL をブロックしてはならない）という原則を実装するものです。

グループロック
-------------

これらすべてが十分に複雑でないかのように、PostgreSQL は現在並列処理をサポートしています（src/backend/access/transam/README.parallel を参照）。これは、個々のプロセスではなく、関連するプロセスのグループ間で発生するデッドロックを解決する必要がある可能性があることを意味します。これは基本的なデッドロック検出アルゴリズムを大きく変えるものではありませんが、簿記をより複雑にします。

同じ並列グループ内のプロセスによって保持されるロックは、リレーション拡張ロックを除いて、競合しないものと見なすことにしました。これは、並列グループ内の2つのプロセスが同時に同じリレーションに対して自己排他的なロックを保持できること、または一方が AccessExclusiveLock を保持している間に他方が AccessShareLock を取得できることを意味します。これは危険に見えるかもしれませんし、実際そうなる場合もあります（詳細は後述）が、これを行わなければ並列クエリは自己デッドロックに非常に陥りやすくなります。例えば、リーダーが既に AccessExclusiveLock を保持しているリレーションに対する並列クエリはハングします。ワーカーが同じリレーションをロックしようとしてリーダーによってブロックされるためですが、リーダーはすべてのワーカーから完了通知を受け取るまで終了できません。結果として、検出されないデッドロックが発生します。このような問題が発生するシナリオはこれだけではありません。リーダーが AccessShareLock のみを保持し、ワーカーが AccessShareLock を求めようとした場合でも、リーダーがロックを取得しようとする時間とワーカーが取得しようとする時間の間に、他のプロセスが AccessExclusiveLock を待機してキューに入ると、同じことが起こります。この場合も、無期限のハングが発生します。

ワーカーが取得しようとするロックを予測し、並列化する前にそれらのロックが正常に取得されることを保証できるように思えるかもしれません。しかし、これを一般的な方法で機能させるのは非常に困難です。例えば、並列ワーカーのクエリプランの一部に、動的にクエリを生成する SQL 呼び出し可能な関数が含まれている場合があり、そのクエリがリーダーが AccessExclusiveLock を保持しているテーブルにヒットする可能性があります。ワーカーができることに十分な制限を課すことで、最終的にはその動作を適切に制限できる状況を作り出せるかもしれませんが、これらの制限はかなり厄介なものになり、たとえそうしたとしても、ワーカーが必要なロックを取得できるかどうかを判断するシステムは複雑で、バグが含まれる可能性があります。そのため、代わりに、ロックグループ内のロックは競合しないと判断するアプローチをとります。これにより、検出されないデッドロックの可能性は排除されますが、いくつかの問題ケースも生じます。リーダーとワーカーが、通常であれば重量級ロック機構によって防止されるような操作を同時に行おうとした場合、未定義の動作が発生する可能性があります。実際には、その危険性はわずかなものです。リーダーとワーカーは同じトランザクション、スナップショット、およびコンボCIDハッシュを共有しており、どちらもDDLを実行したり、データを書き込んだりすることはできません。したがって、一方が排他的にロックしているテーブルをもう一方が読み取ることは十分に安全です。問題が発生するのは、リーダーがバックエンド固有の状態を持つコード内の地点から並列処理を開始し、それによって別のプロセスからのテーブルアクセスが安全でなくなった場合です。例えば、SetReindexProcessingを呼び出した後、ResetReindexProcessingを呼び出す前に並列処理を開始すると、ワーカーはその状態を持たないため、破滅的な結果を招く可能性があります。同様に、GINページロックのような特定の種類の非リレーションロックでも問題が発生する可能性があります。関連する2つのプロセスが同時にGINクリーンアップを実行することは、無関係なプロセスが同じことを行うのと同様に危険です。しかし、現時点では並列モードは厳密に読み取り専用であるため、これや同様のケースのほとんどは発生しません。並列書き込みを許可するには、(1)デッドロック検出器をさらに強化して、それらの種類のロックを他の種類とは異なる方法で処理するようにするか、(2)並列ワーカーがそのようなケースに対して別の相互排他手法を使用するようにする必要があります。

グループロックは、各PGPROCにlockGroupLeader、lockGroupMembers、lockGroupLinkという3つの新しいメンバーを追加します。並列クエリに関与していないプロセスでは、PGPROCのlockGroupLeaderはNULLです。プロセスが並列ワーカーと連携したい場合、それはロックグループリーダーとなり、このフィールドを自身のPGPROCを指すように設定します。並列ワーカーが起動すると、このフィールドをリーダーに向けます。lockGroupMembersフィールドはリーダーでのみ使用され、ロックグループのメンバーPGPROC（リーダーとすべてのワーカー）のリストです。lockGroupLinkフィールドは、このリストのリストリンクです。

これら3つのフィールドはすべて、ロックマネージャーのパーティションロックによって保護されていると見なされます。特定のロックグループ内でこれらのフィールドを保護するパーティションロックは、リーダーのpgprocnoをロックマネージャーのパーティション数で割った余りによって選択されます。この珍しい配置には大きな利点があります。デッドロック検出器は、すべてのロックマネージャーロックを保持していることを知っているため、デッドロック検出器の実行中にlockGroupLeaderフィールドが変更されることはないという事実に依存できます。また、この単一のロックを保持することで、ロックグループのlockGroupMembersリストを安全に操作できます。

これらのフィールドを設定する際には追加のインターロックが必要です。なぜなら、新しく起動した並列ワーカーはリーダーのロックグループに参加しようとしますが、起動した時点でグループリーダーがまだ生存している保証がないからです。通常の場合、すべてのワーカーの後に並列リーダーが終了するように努めますが、万が一それが失敗した場合でもシステムが比較的無傷で存続できるようにしています。これはそのようなシナリオに対する予防策の1つです。リーダーは自身のPGPROCとPIDをワーカーに中継し、ワーカーは指定されたPGPROCが依然として同じPIDを持ち、かつ依然としてロックグループリーダーである場合を除き、ロックグループへの参加を失敗させます。このインターロックが失敗するほどPIDが急速に再利用されることはないと想定しています。

ユーザーロック（アドバイザリロック）
---------------------------

ユーザーロックは、通常のトランザクション境界を超えて拡張される可能性のある長期的な協調ロックとして、アプリケーション側で完全に処理されます。その目的は、誰かがアイテムに対して「作業中」であることをアプリケーションに示すことです。したがって、タプルのoidにユーザーロックをかけ、タプルを取得し、1時間作業してから更新し、ロックを解除することが可能です。ロックがアクティブな間も、他のクライアントはタプルを読み書きできますが、誰かによってアプリケーションレベルでロックされていることを認識できます。

ユーザーロックと通常のロックは完全に直交しており、互いに干渉しません。

ユーザーロックは、セッションレベルまたはトランザクションレベルのいずれかで取得できます。セッションレベルのロック要求は、トランザクション終了時に自動的に解放されることはなく、アプリケーションによって明示的に解放される必要があります。（ただし、残りのロックはセッション終了時に常に解放されます。）トランザクションレベルのユーザーロック要求は、通常のロック要求と同様に動作し、トランザクション終了時に解放されるため、明示的なロック解除は不要です。

ホットスタンバイ中のロック
--------------------------

リカバリ中に変更を行えるのはスタートアッププロセスのみであり、他のすべてのバックエンドは読み取り専用です。その結果、スタートアッププロセスは、ロックレベルがAccessExclusiveLockである場合を除き、リレーションやオブジェクトに対するロックを取得しません。

通常のバックエンドは、RowExclusiveLock以下のリレーションやオブジェクトに対してのみロックを取得できます。これにより、スタートアッププロセスによってAccessExclusiveLockが要求されない限り、それらが互いに、またはスタートアッププロセスと競合しないことが保証されます。

AccessExclusiveLockを伴うデッドロックは発生しないため、ユーザーが開始したデッドロックによってリカバリの進行が妨げられることを懸念する必要はありません。

プライマリノード上のAccessExclusiveLockはWALレコードを生成し、それがスタートアッププロセスによって適用されます。ロックは、通常の処理と同様にトランザクション終了時に解放されます。これらのロックは、元々これらのロックを取得したバックエンドのプロキシとして機能するスタートアッププロセスによって保持されます。繰り返しになりますが、これらのロックは互いに競合することはないため、スタートアッププロセス自体がデッドロックに陥ることもありません。

デッドロックは発生しませんが、通常のバックエンドの弱いロックが、スタートアッププロセスによるWALの適用を妨げる可能性があり、これは通常、長時間許容されるべきではありません。通常のバックエンドのクエリがスタートアッププロセスを長時間ブロックした場合に、そのクエリを強制的にキャンセルするメカニズムが存在します。
