非同期 & ダイレクト IO

動機

なぜ非同期 IO なのか

非同期 IO が導入されるまで、Postgres は同期 IO のコストを隠蔽するためにオペレーティングシステムに依存していました。これは多くのワークロードで驚くほど上手く機能していましたが、プリフェッチや制御されたライトバックに関しては、我々が望むような十分な性能を発揮できていませんでした。

fdatasync() のような、オペレーティングシステムがストレージのレイテンシを隠蔽できない重要な高コスト操作が存在します。これは特に WAL 書き込みにおいて重要であり、fdatasync() や O_DSYNC 書き込みを非同期に発行できる能力は、スループットを大幅に向上させる可能性があります。

なぜダイレクト / 非バッファ IO なのか

ダイレクト IO を使用したい主な理由は以下の通りです。

  • CPU 使用率の低下 / スループットの向上。特に現代のストレージにおいて、バッファ付き書き込みは、オペレーティングシステムが CPU を使用してカーネルのページキャッシュから Postgres のバッファプールへデータをコピーしなければならないというボトルネックに直面します。一方、ダイレクト IO は多くの場合、DMA を使用してストレージデバイスと Postgres のバッファキャッシュ間で直接データを移動できます。その転送が行われている間、CPU は他の作業を自由に行うことができます。
  • レイテンシの低減 - ダイレクト IO はバッファ付き IO よりも大幅にレイテンシを低くできる可能性があり、WAL 書き込みレイテンシがボトルネックとなる OLTP ワークロードにおいて効果的です。
  • オペレーティングシステムのキャッシュと Postgres の shared_buffers との間の二重バッファリングを回避します。
  • ダーティデータのライトバックのタイミングとペースをより適切に制御できます。

ダイレクト IO を使用すべきではない主な理由は以下の通りです。

  • AIO なしでは、ダイレクト IO はほとんどの目的において使用できないほど低速です。
  • AIO があっても、Postgres の多くの部分は明示的なプリフェッチを行うように修正する必要があります。
  • shared_buffers を十分に大きく設定できない状況（例：共有ハードウェア上で多くの異なる Postgres インスタンスがホストされている場合）では、バッファ付き IO を使用する場合よりもパフォーマンスが悪化することがよくあります。

AIO 使用例

多くの場合、AIO の恩恵を受けられるコードは、直接 AIO インターフェースを操作する必要はなく、より高レベルな抽象化を通じて AIO を使用できます。ヘルパーを参照してください。

この例では、バッファが共有バッファに読み込まれます。

/*
 * 操作の結果。このバックエンド内でのみアクセスされる。
 */
PgAioReturn ioret;

/*
 * AIO ハンドルを取得する。完了時に ioret に結果が格納される。
 *
 * ioret は IO が完了するか、CurrentResourceOwner が解放される（つまりエラーがスローされる）まで生存している必要があることに注意。
 */
PgAioHandle *ioh = pgaio_io_acquire(CurrentResourceOwner, &ioret);

/*
 * 下記で開始する IO を待機するために使用できる参照。この参照はローカルメモリまたは共有メモリに配置でき、どのプロセスからでも待機可能。各 IO に対して任意の数の参照を作成できる。
 */
PgAioWaitRef iow;

pgaio_io_get_wref(ioh, &iow);

/*
 * IO 完了時に共有バッファ完了コールバックが呼び出されるように手配する。このコールバックは AioHandle に関連付けられたバッファ記述子を更新し、他のバックエンドがバッファにアクセスできるようにする。
 *
 * コールバックには、バッファが無効な場合にゼロクリアするかどうかを示すなどの小さなデータを渡すことができる。
 *
 * 各ハンドルに対して複数の完了コールバックを登録できる。
 */
pgaio_io_register_callbacks(ioh, PGAIO_HCB_SHARED_BUFFER_READV, 0);

/*
 * 完了コールバックは、IO 完了時にどのバッファを更新すべきかを知る必要がある。AIO サブシステムはバッファについて知らないため、この情報を AioHandle に関連付けて、上記の完了コールバックで使用できるようにする必要がある。
 *
 * この例では単一のバッファのみを読み込んでいるため、1 を指定する。
 */
pgaio_io_set_handle_data_32(ioh, (uint32 *) &buffer, 1);

/*
 * AIO ハンドルを下位レベルの関数に渡す。バッファレベルで操作する場合、IO が正確にどのように実行されるかは不明であり、それはストレージマネージャ実装の責任である。
 *
 * 例: md.c はブロック番号をセグメント内のオフセットに変換する必要がある。
 *
 * IO ハンドルが smgrstartreadv() に引き渡された後は、それ以上使用してはならない。IO が smgrstartreadv() の下で即座に実行され、ハンドルが別の IO に再利用される可能性があるためである。
 *
 * 効率的に複数の IO を発行するには、呼び出し元は複数の IO を開始する前に pgaio_enter_batchmode() を呼び出し、pgaio_exit_batchmode() でそのバッチを終了できる。未送信の IO がある間は注意が必要である。別のバックエンドが未送信の IO のいずれかを待機する必要がある可能性があるためである。もしこのバックエンドが他のバックエンドを待機しなければならなくなった場合、検出されないデッドロックに陥る。詳細は pgaio_enter_batchmode() を参照のこと。
 *
 * バッチモード中であっても、未送信 IO の制限に達したことなどが原因で IO が即座に送信され、smgrstartreadv() が戻る前に完了する可能性があることに注意。
 */
void *page = BufferGetBlock(buffer);
smgrstartreadv(ioh, operation->smgr, forknum, blkno,
               &page, 1);

/*
 * AIO の恩恵を受けるには、IO の完了を待つ前に、他の IO の送信を含む他の作業を行うことが有益である。そうでなければ、同期的なブロッキング IO を使用すればよかったことになる。
 */
perform_other_work();

/*
 * 他の作業を行ったので、続行するために IO 操作が完了している必要がある。
 */
pgaio_wref_wait(&iow);

/*
 * この時点で IO は完了している。ただし、成功したか失敗したかはまだ不明である。バッファの状態は更新されており、これにより他のバックエンドがバッファを使用できる（IO が成功した場合）か、IO を再試行できる（失敗した場合）。
 *
 * IO が失敗した場合、LOG メッセージが出力される可能性があるが、ERROR は発生していないことに注意。これは重要である。この IO を待機している別のバックエンドが ERROR を見てはならないためである。
 *
 * 操作が成功したかどうかを確認し、ERROR を発生させるか、あるいはより適切な LOG を出力するかについては、pgaio_io_acquire() に渡した PgAioReturn を使用する。
 */
if (ioret.result.status == PGAIO_RS_ERROR)
    pgaio_result_report(ioret.result, &ioret.target_data, ERROR);

/*
 * IO は完全に成功しただけでなく、a) 部分的に完了したか、b) 警告付きで成功した（例: zero_damaged_pages による）可能性がある。例えば一度に多くのブロックを読み込もうとした場合、最初の数ブロックのみ読み込みに成功した可能性がある。
 *
 * IO が部分的に成功し、このバックエンドがすべてのブロックの完了を必要とする場合、このバックエンドは残りのバッファに対して IO を再発行する必要がある。AIO サブシステムはこの再試行を透過的に処理できない。
 *
 * この例はすでに長いため、単一ブロックのみを読み込んでいる場合、部分的な読み込みや警告があればエラー終了することにする。
 */
if (ioret.result.status != PGAIO_RS_OK)
    pgaio_result_report(ioret.result, &ioret.target_data, ERROR);

/*
 * IO は成功したため、バッファを使用できる。
 */

設計基準と動機

AIO によるデッドロックと飢餓の危険性

AIO を素朴な方法で使用すると、AIO のソース/ターゲットが Postgres の shared_buffers 内のページのような共有リソースである環境では、簡単にデッドロックを引き起こす可能性がある。

あるバックエンドがテーブルの先読みを実行し、現在の「スキャン位置」より先の複数のバッファに対して IO を開始すると仮定する。もしそのバックエンドがブロックする操作を実行したり、単に低速であったりすると、非同期に開始された読み込みの IO 完了が処理されない可能性がある。

この AIO 実装では、AIO メソッドがシステム内のどのバックエンドでも AIO 完了を処理できるようにする（例: io_uring）、あるいは発行元のバックエンドがブロックされている場合でも AIO 処理が行われることを保証する（例: ワーカーモード。完了処理を AIO ワーカーにオフロードする）ことを要求することで、この問題を解決している。

IO はクリティカルセクション内で開始できる

WAL 書き込みに AIO を使用すると、WAL ログのオーバーヘッドを大幅に削減できる。

  • AIO を使用すると WAL 書き込みを積極的に開始できるため、待機が必要になる前に完了できる。
  • AIO を使用すると、複数の WAL フラッシュを同時に進行させることができる。
  • AIO を使用すると O_DIRECT + O_DSYNC を使用するのが現実的になる。これにより、一部の OS やストレージハードウェアにおいてストレージへのラウンドトリップ回数を削減できる（バッファ付き IO や O_DSYNC なしのダイレクト IO は、書き込みを発行し、その完了後にキャッシュフラッシュを発行する必要があるが、O_DIRECT + O_DSYNC は単一の Force Unit Access (FUA) 書き込みを使用できる）。

クリティカルセクション内で IO を実行できる必要があることは、AIO サブシステムに重大な設計上の影響を与える。主に、IO の完了（前述のセクションを参照）を、たとえその IO 自体がクリティカルセクション内で発行されたものでなくても、クリティカルセクション内で可能にする必要があるためである。例えば、バックエンドが最初に共有バッファからの複数の書き込みを開始し、次に WAL のフラッシュを開始する場合を考える。同時に進行できる IO の量には限りがあるため、WAL フラッシュのための IO を開始するには、先に開始された IO を完了させる必要がある場合がある。

AIO の状態は共有メモリに存在する必要がある

Postgres はプロセスモデルを使用しており、AIO はどのバックエンドからでも完了可能である必要があるため、AIO サブシステムの多くの状態は共有メモリに存在する必要がある。

EXEC_BACKEND ビルドでは、ASLR のため、バックエンドの実行可能コードやその他のプロセスローカルな状態は、必ずしも各プロセスで同じアドレスにマッピングされるとは限らない。つまり、共有メモリにはコールバックへのポインタを含めることができない。

AIO サブシステムの設計

AIO メソッド

移植性とパフォーマンスを達成するために、複数の AIO 実行メソッドが実装されており、将来的には他のメソッドの追加も検討に値する。

同期モード

io_method=sync は実際には AIO を実行しないが、同期 IO を実行しながら AIO API を使用することを可能にする。これはデバッグに役立つ。同期モードのコードは、ワーカーモードなどがワーカーで実行できない IO を実行するためのフォールバックとしても使用される。

ワーカー

io_method=worker は Postgres が動作するすべてのプラットフォームで利用可能であり、発行プロセスから見て、IO を同期的に実行する複数のワーカープロセスのいずれかに IO をディスパッチすることで非同期 IO を実装する。

io_uring

io_method=io_uring は Linux 5.1 以降で利用可能である。ワーカーモードとは対照的に、プロセス内からすべての IO をディスパッチするため、コンテキストスイッチの頻度とレイテンシを低減する。

AIO ハンドル

Postgres の AIO 抽象化における中心的な API コンポーネントは AIO ハンドルである。IO を実行するには、まず IO ハンドルを取得し (pgaio_io_acquire())、次にそれを「定義」する、つまり IO 操作をハンドルに関連付ける必要がある。

多くの場合、AIO ハンドルは高レベルで取得され、完全に定義されるために低レベルへ渡される。例えば、共有バッファへの/からの IO の場合、bufmgr.c ルーチンがハンドルを取得し、それが smgr.c、md.c を経由して fd.c で最終的に完全に定義される。

操作を定義するために最下層で使用される関数は pgaio_io_start_*() である。

IO ハンドルの取得は 常に成功しなければならず、AIO ハンドルの数は 制限されなければならない ため、AIO ハンドルは完了次第再利用できる。当然、コードは IO 完了に対応できる必要がある。状態は AIO 完了コールバック を使用して更新でき、発行元のバックエンドは AIO 結果 で説明されているように、IO の結果を受け取るためにバックエンドローカル変数を提供できる。IO は AIO 参照 を使用して、発行元および他のバックエンドの両方から待機できる。

AIO ハンドルは pgaio_io_acquire() を呼び出した直後には実行可能ではなく、また pgaio_io_acquire() は（PANIC を除き）常に成功する必要があるため、IO が定義される（潜在的に間接的に pgaio_io_start_*() が呼び出されることによる）ことなく取得できる（つまり pgaio_io_acquire() によって返される）AIO ハンドルは 1 つだけである。そうでなければ、バックエンドは IO の完了を待つ能力なしにすべての AIO ハンドルを使い果たし、自らデッドロックを引き起こす可能性がある。AIO Handleが不要になった場合（例：競合するロックを保持する前にハンドルが取得された場合など）は、pgaio_io_release() を使用して定義せずに解放することができます。

AIOコールバック

一般的に、IOの完了に対して複数のレイヤーが反応する必要があります。例えば読み取りの場合、md.cはIOが完全に失敗したか、必要な長さより短かったかを確認する必要があり、bufmgr.cはページが有効に見えるかを確認し、さらにバッファの状態を更新するためにBufferDescを更新する必要があります。

複数のレイヤーやサブシステムがIO完了に反応する必要があるという事実は、いくつかの課題をもたらします。

  • 上位レイヤーは下位レイヤーの詳細を知るべきではありません。例えば、bufmgr.cはIOがmd.cを経由することを前提とすべきではありません。そのため、上位レベルでは下位レイヤーが何をエラーとみなすかを知ることができません。
  • 下位レイヤーは上位レイヤーについて知るべきではありません。例えば、smgr APIは共有バッファを経由する場合にも、共有バッファをバイパスする場合にも使用されます。これは、例えばmd.cがチェックサムを検証する立場にないことを意味します。
  • レイヤーのあらゆる組み合わせに対してAIOサブシステム内にコードを持つことは、多大な重複を招きます。

この「解決策」は、ハンドルに複数の完了コールバックを関連付ける機能です。例えば、bufmgr.cはBufferDescの状態を更新しページを検証するためのコールバックを持ち、md.cはIO操作が成功したかを確認するための別のコールバックを持つことができます。

前述の通り、共有メモリには現在関数ポインタを含めることができません。そのため、完了コールバックは関数ポインタによって直接識別されるのではなく、ID (PgAioHandleCallbackID) によって識別されます。これによる大きな付加的利点は、コールバックを非常に少ないメモリ量（現在は1バイト）で識別できることです。

完了に加えて、AIOコールバックはIOを「ステージング（準備）」するためにも呼び出されます。これは例えば、AIOサブシステムがバッファを参照していることを考慮してバッファの参照カウントを増やすために使用されます。これは、IOが進行中に発行元のバックエンドがエラーになり、自身のピンを解放してしまうケースを処理するために必要です。

前述の通り、IO完了はクリティカルセクション内で安全に実行される必要があります。IOを発行したバックエンドが失敗時にエラーを出せるようにするため、AIO結果を使用できます。

AIOターゲット

上記の完了コールバックに加え、各AIO Handleは正確に1つの「ターゲット」を持ちます。各ターゲットはAIO Handle内にそのターゲット固有の情報を持つための領域を持ち、基盤となるファイルを再オープンするためのコールバック（ワーカーモードで必要）や、IO操作を記述するためのコールバック（デバッグログやエラーメッセージで使用）を提供できます。

つまり、AIOの2つの異なる用途が、操作対象のファイルの同一性を同じ方法で記述できる場合、同じターゲットを使用するのが合理的です。例えば、異なるsmgr実装はRelFileLocator、ForkNumber、BlockNumberでIOを記述できるため、ターゲットを共有できます。対照的に、WALファイルのIOはTimeLineIDとXLogRecPtrで記述されるため、smgrとWALで同じターゲットを使用することは合理的ではありません。

AIO待機参照

前述の通り、AIO Handleは完了後すぐに再利用されるため、IOの完了を待機するために使用することはできません。待機はAIO待機参照を使用して有効になります。これにはAIO Handleを識別するだけでなく、ハンドルの「世代（generation）」も含まれます。

AIO Handleへの参照は pgaio_io_get_wref() を使用して取得でき、その後 pgaio_wref_wait() を使用して待機できます。

AIO結果

AIO完了コールバックはクリティカルセクション内で実行され、任意のバックエンドによって実行される可能性があるため、完了コールバックを使用して、例えばIOをトリガーしたクエリをERRORで終了させることはできません。

失敗したIOに対応できるようにするため、発行元のバックエンドはバックエンドローカルメモリ内の PgAioReturn へのポインタを渡すことができます。AIO Handleが再利用される前に、PgAioReturn はIOに関する情報で満たされます。これには、IOが成功したかどうか（PgAioResultStatus の値として）や、失敗した場合にエラーを発生させるために十分な情報（pgaio_result_report() を介して、エラー詳細は PgAioResult にエンコードされる）が含まれます。

AIOエラー

共有完了コールバックがエラーの詳細を ErrorData としてエンコードし、後で発生させられるようにすると非常に便利です。残念ながら、そうするにはメモリの割り当てが必要になります。elog.cは（ある程度）メッセージのログ記録がメモリ不足にならないことを保証できますが、それはログ記録中のメッセージ数が非常に限られている場合にのみ機能します。AIOでは、大量のAIOが同時に発行され、失敗する可能性があります。

（共有メモリ内に！）潜在的に大量のメモリを事前割り当てする必要を避けるため、完了コールバックはエラーを、エラーメッセージに変換可能なよりコンパクトな形式でエンコードしなければなりません。

ヘルパー

低レベルのAIO APIを使用すると、ツリー全体で複雑になりすぎます。AIOのほとんどの用途は、再利用可能な高レベルのヘルパーを介して行われるべきです。

リードストリーム

AIOの一般的で非常に有益な用途は、読み取り対象の場所が事前に多数判明している読み取りです。例えば、シーケンシャルスキャンでは、読み取る必要のあるブロックのセットは、現在の位置を知り、バッファマッピングテーブルを確認するだけで決定できます。

リードストリームインターフェースを使用すると、このようなユースケースでAIOを比較的簡単に使用できます。
