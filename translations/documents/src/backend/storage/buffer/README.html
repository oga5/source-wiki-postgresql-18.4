共有バッファアクセスルールに関する注記
======================================

共有ディスクバッファには、参照カウント（別名：ピンカウント）とバッファコンテンツロックという、2つの独立したアクセス制御メカニズムが存在します。（実際には、第3レベルのアクセス制御も存在します。リレーションに属するページにアクセスするには、その前にリレーションに対して適切な種類のロックを保持していなければなりません。リレーションレベルのロックについてはここでは説明しません。）

ピン（Pins）：バッファに対して何らかの操作を行うには、その前にバッファの「ピンを保持（hold a pin on）」（参照カウントをインクリメント）しなければなりません。ピンされていないバッファは、いつでも別のページのために再利用・回収される可能性があるため、それに触れることは安全ではありません。通常、ピンは ReadBuffer を介して取得され、ReleaseBuffer を介して解放されます。単一のバックエンドが同一ページを同時に複数回ピンすることは問題なく、実際によく行われます。バッファマネージャはこれを効率的に処理します。ピンを長時間保持することは問題ないとされています。例えば、シーケンシャルスキャンでは、ページ上のすべてのタプルを処理し終えるまで現在のページのピンを保持します。スキャンが結合（join）の外側スキャンである場合、これはかなりの時間になる可能性があります。同様に、Bツリーインデックススキャンも現在のインデックスページのピンを保持し続けることがあります。通常の操作はページのピンカウントがゼロになるのを待つことはないため、これは問題ありません。（このような待機が必要になる可能性のある処理は、代わりにリレーションレベルのロックを取得するまで待機することで処理されます。そのため、最初にロックを保持しておくことが推奨されます。）ただし、ピンをトランザクション境界を越えて保持することはできません。

バッファコンテンツロック（Buffer content locks）：バッファロックには共有（shared）と排他（exclusive）の2種類があり、期待通りに動作します。複数のバックエンドが同じバッファに対して共有ロックを保持できますが、排他ロックは他の誰かが共有ロックまたは排他ロックを保持することを防ぎます。（これらは、READロックおよびWRITEロックとも呼ばれます。）これらのロックは短期間の使用を意図しており、長時間保持すべきではありません。バッファロックは LockBuffer() によって取得および解放されます。単一のバックエンドが同じバッファに対して複数のロックを取得しようとしても機能しません。バッファをロックしようとする前に、必ずバッファをピンしなければなりません。

バッファアクセスルール：

1. タプルをスキャンするためにページにアクセスするには、ピンを保持した上で、共有または排他コンテンツロックのいずれかが必要です。共有バッファ内のタプルのコミット状態（XIDおよびステータスビット）を調べる場合も同様に、ピンを保持した上で共有または排他ロックが必要です。

2. タプルが対象（現在のトランザクションから可視）であると判断したら、コンテンツロックを解放しても構いません。ただし、バッファピンを保持している限り、タプルのデータへのアクセスは継続できます。これはヒープスキャンで一般的に行われる手法です。heap_fetch が返すタプルには、共有バッファ内のタプルデータへのポインタが含まれているためです。したがって、ピンが保持されている間はタプルが消滅することはありません（ルール#5を参照）。状態が変化する可能性はありますが、可視性の初期判定を行った後は、それは問題にならないと想定されています。

3. タプルを追加する場合、または既存のタプルの xmin/xmax フィールドを変更する場合は、対象のバッファに対してピンと排他コンテンツロックを保持しなければなりません。これにより、他の誰かが可視性チェックを行っている最中に、タプルが部分的に更新された状態を見ることを防ぎます。

4. バッファに対して共有ロックとピンのみを保持した状態で、タプルのコミットステータスビット（つまり、HEAP_XMIN_COMMITTED、HEAP_XMIN_INVALID、HEAP_XMAX_COMMITTED、または HEAP_XMAX_INVALID の値を t_infomask に OR する）を更新することは問題ないとされています。これは、ほぼ同時にタプルを見ている別のバックエンドも同じビットをフィールドに OR するため、更新の競合リスクがほとんど、あるいは全くないためです。さらに、万が一競合が発生したとしても、それはビット更新が一つ失われ、後で再実行する必要があるという程度の意味しか持ちません。これら4つのビットは単なるヒント（pg_xact 内のトランザクション状態検索結果をキャッシュするもの）であるため、競合する更新によってゼロにリセットされたとしても大きな害はありません。ただし、HEAP_XMIN_INVALID と HEAP_XMIN_COMMITTED の両方を設定することでタプルがフリーズされる点に注意してください。これは重要な更新であるため、排他バッファロックが必要です（また、WALログへの記録も必須です）。

5. タプルを物理的に削除したり、ページ上の空き領域を圧縮したりするには、ピンと排他ロックを保持した上で、排他ロックを保持している間にバッファの共有参照カウントが1である（つまり、他のバックエンドがピンを保持していない）ことを確認しなければなりません。これらの条件が満たされていれば、排他ロックが解放されるまで他のバックエンドはページスキャンを実行できず、また他のバックエンドが再調査を期待するような既存タプルへの参照を保持することもできません。なお、クリーンアップを実行している間に別のバックエンドがバッファをピン（参照カウントをインクリメント）する可能性はありますが、そのバックエンドは共有または排他コンテンツロックを取得するまで、実際にページを調査することはできません。

ルール#5で必要なロックの取得は、bufmgr ルーチンの LockBufferForCleanup() または ConditionalLockBufferForCleanup() によって行われます。これらはまず排他ロックを取得し、次に共有ピンカウントが現在1であるかを確認します。もし1でなければ、ConditionalLockBufferForCleanup() は排他ロックを解放して false を返します。一方、LockBufferForCleanup() は排他ロックを解放し（呼び出し元のピンは解放しません）、別のバックエンドからシグナルが送られるまで待機し、その後再試行します。シグナルは UnpinBuffer が共有ピンカウントを1に減らした時に発生します。上記のように、この操作はロックを取得するまでかなりの時間を待機する可能性がありますが、同時実行される VACUUM にとっては大きな問題ではありません。現在の実装では、特定の共有バッファに対してピンカウントが1になるのを待機できるのは1プロセスのみです。これは、単一のリレーションに対して複数の VACUUM を同時に許可していないため、VACUUM の用途としては十分です。リカバリや VACUUM 以外でクリーンアップロックを取得したい場合は、条件付きの関数バリアントを使用しなければなりません。

バッファマネージャの内部ロック
---------------------------------

PostgreSQL 8.1以前では、共有バッファマネージャ自体のすべての操作は、システム全体で単一のロックである BufMgrLock によって保護されていましたが、これは当然ながら競合の原因となっていました。新しいロック方式では、一般的なコードパスでシステム全体に及ぶ排他ロックを取得することを避けています。仕組みは以下の通りです。

* BufMappingLock というシステム全体の LWLock が存在し、バッファタグ（ページ識別子）からバッファへのマッピングを概念的に保護します。（物理的には、buf_table.c が管理するハッシュテーブルを保護していると考えることができます。）タグに対応するバッファが存在するかどうかを調べるには、BufMappingLock の共有ロックを取得すれば十分です。なお、BufMappingLock を解放する前に、見つかったバッファ（もしあれば）をピンしなければならないことに注意してください。バッファのページ割り当てを変更するには、BufMappingLock の排他ロックを保持しなければなりません。このロックは、バッファのヘッダーフィールドの調整および buf_table ハッシュテーブルの変更の間、保持されなければなりません。排他ロックを必要とする唯一の一般的な操作は、共有バッファにまだ存在しないページを読み込むことですが、これには少なくともカーネル呼び出しが必要であり、通常は I/O 待ちが発生するため、いずれにせよ低速です。

* PG 8.2以降、BufMappingLock は NUM_BUFFER_PARTITIONS 個の個別のロックに分割され、それぞれがバッファタグ空間の一部を保護しています。これにより、通常のコードパスにおける競合がさらに削減されます。特定のバッファタグが属するパーティションは、タグのハッシュ値の下位ビットから決定されます。上記のルールは各パーティションに独立して適用されます。一度に複数のパーティションをロックする必要がある場合は、デッドロックのリスクを避けるために、パーティション番号の順序でロックしなければなりません。

* buffer_strategy_lock という個別のシステム全体のスピンロックが、バッファの空きリストにアクセスしたり、置換対象のバッファを選択したりする操作に対して相互排他を提供します。効率化のため、軽量ロックではなくスピンロックが使用されています。buffer_strategy_lock を保持している間は、他のいかなる種類のロックも取得してはなりません。これは、複数のバックエンドでバッファ置換を妥当な並行性で実行するために不可欠です。

* 各バッファヘッダーにはスピンロックが含まれており、そのバッファヘッダーのフィールドを調査または変更する際には、このロックを取得しなければなりません。これにより、ReleaseBuffer のような操作が、システム全体のロックを取得することなくローカルな状態変更を行えるようになります。ロックを数命令以上保持する必要があるケースは存在しないため、LWLock ではなくスピンロックを使用しています。

バッファヘッダーのスピンロックは、バッファ内に保持されているデータへのアクセスを制御するものではないことに注意してください。各バッファヘッダーには「バッファコンテンツロック」という LWLock も含まれており、これがバッファ内のデータにアクセスする権利を表します。これは上記のルールに従って使用されます。

* BM_IO_IN_PROGRESS フラグは一種のロックとして機能し、バッファに対する I/O の完了を待機するために使用されます（バージョン14以前では、バッファごとの LWLock が伴っていました）。読み込みまたは書き込みを開始するプロセスがこのフラグをセットします。I/O が完了すると（I/O を開始したプロセスか、別のプロセスかに関わらず）、フラグが解除され、バッファの条件変数がシグナルされます。I/O 完了を待機する必要があるプロセスは、BufferDesc->io_wref を使用して非同期 I/O を待機し、バッファの条件変数でスリープすることで BM_IO_IN_PROGRESS が解除されるのを待つことができます。

通常のバッファ置換戦略
----------------------------------

置換の有力な候補となるバッファの「空きリスト」が存在します。特に、完全に空（有効なページを含まない）のバッファは常にこのリストに含まれます。ページがすぐに必要になる可能性が低いと判断された場合にバッファをこのリストに投入することも可能ですが、現在のアルゴリズムではそれは行われません。リストはバッファヘッダー内のフィールドを使用して単方向連結されており、先頭と末尾のポインタはグローバル変数で保持されます。（注：リストのリンクはバッファヘッダー内にありますが、これらはバッファヘッダーのスピンロックではなく、buffer_strategy_lock によって保護されているとみなされます。）空きバッファがない場合に再利用する犠牲バッファを選択するために、単純なクロックスイープアルゴリズムを使用しており、これにより一般的な操作中にシステム全体のロックを取得する必要がなくなります。仕組みは以下の通りです。

各バッファヘッダーには使用カウンタが含まれており、バッファがピンされるたびに（小さな上限値まで）インクリメントされます。（これにはバッファヘッダーのスピンロックのみが必要であり、参照カウントをインクリメントするためにいずれにせよ取得する必要があるため、コストはほぼゼロです。）

「クロックハンド」は nextVictimBuffer というバッファインデックスであり、すべての利用可能なバッファを循環的に移動します。nextVictimBuffer は buffer_strategy_lock によって保護されています。

犠牲バッファを取得する必要があるプロセスのアルゴリズムは以下の通りです。

1. buffer_strategy_lock を取得する。

2. バッファ空きリストが空でなければ、その先頭バッファを取り出す。buffer_strategy_lock を解放する。バッファがピンされているか、使用カウンタがゼロでなければ、そのバッファは使用できないため、無視してステップ1に戻る。そうでなければ、バッファをピンして返す。

3. バッファ空きリストが空であれば、nextVictimBuffer が指すバッファを選択し、次回のために nextVictimBuffer を循環的に進める。buffer_strategy_lock を解放する。

4. 選択されたバッファがピンされているか、使用カウンタがゼロでなければ、そのバッファは使用できない。使用カウンタを（ゼロでなければ）デクリメントし、buffer_strategy_lock を再取得して、ステップ3に戻り次のバッファを調査する。

5. 選択されたバッファをピンして返す。

（選択されたバッファがダーティである場合、再利用する前に書き出す必要があることに注意してください。その間に他の誰かがバッファをピンした場合は、諦めて別のバッファを試す必要があります。ただし、これは基本的な「犠牲バッファ選択」アルゴリズムの関心事ではありません。）

バッファリング置換戦略
---------------------------------VACUUMや大規模なシーケンシャルスキャンなど、一度に大量のページにアクセスする必要があるクエリを実行する場合、異なる戦略が使用されます。このようなスキャンによってのみ参照されたページは、すぐに再び必要になる可能性は低いため、通常のクロックスイープアルゴリズムを実行してバッファキャッシュ全体を追い出す代わりに、通常のクロックスイープアルゴリズムを使用して小さなバッファリングを割り当て、そのバッファをスキャン全体で再利用します。これはまた、そのようなステートメントによって引き起こされる書き込みトラフィックの多くが、他のプロセスに押し付けられるのではなく、バックエンド自体によって実行されることを意味します。

シーケンシャルスキャンでは、256KBのリングが使用されます。これはL2キャッシュに収まるほど小さいため、OSキャッシュから共有バッファキャッシュへのページ転送が効率的になります。これより少なくても十分な場合が多いですが、リングはスキャン内で同時にピン留めされるすべてのページを収容できる大きさである必要があります。256KBあれば、他のバックエンドが同期シーケンシャルスキャンに参加するための小さなキャッシュトレイルを残すのにも十分なはずです。リングバッファがダーティになり、そのLSNが更新された場合、通常はバッファを再利用する前にWALを書き込み、フラッシュする必要があります。この場合、代わりにリングからバッファを破棄し、（後で）通常のクロックスイープアルゴリズムを使用して置換対象を選択します。したがって、この戦略は読み取り専用（またはせいぜいヒントビットを更新する程度）のスキャンに最適です。バルクUPDATEやDELETEのようにスキャン内のすべてのページを変更するスキャンでは、リング内のバッファは常にダーティになり、リング戦略は実質的に通常の戦略に低下します。

VACUUMはシーケンシャルスキャンと同様のリングを使用しますが、このリングのサイズはvacuum_buffer_usage_limit GUCによって制御されます。ダーティページはリングから削除されません。その代わり、バッファの再利用を可能にするために必要に応じてWALがフラッシュされます。8.3でバッファリング戦略が導入される前は、VACUUMのバッファはフリーリストに送られていました。これは実質的に1バッファのバッファリングであり、過剰なWALフラッシュを引き起こしていました。

バルク書き込みはVACUUMと同様に機能します。現在、これはCOPY INとCREATE TABLE AS SELECTにのみ適用されます。（シーケンシャルスキャンのUPDATEやDELETEにバルク書き込み戦略を使用させることは興味深いでしょうか？）バルク書き込みには16MBのリングサイズを使用します（ただし、shared_buffersの1/8を超えないようにします）。サイズが小さいと、WALフラッシュのためにCOPYが頻繁にブロックされることがわかっています。バックグラウンドVACUUMが自身のWALフラッシュによって遅くなることは許容されますが、COPYがその影響を受けることは望ましくないため、バッファ領域を少し多めに使用できるようにしています。

バックグラウンドライターの処理
------------------------------

バックグラウンドライターは、すぐに再利用される可能性が高いページを書き出すように設計されており、それによってアクティブなバックエンドから書き込み作業をオフロードします。これを行うために、nextVictimBufferの現在の位置から（これを変更せずに！）円状に前方へスキャンし、ダーティであり、ピン留めされておらず、使用カウントが正でないバッファを探します。そのようなバッファを見つけると、ピン留めし、書き込み、解放します。

nextVictimBufferの読み取りがアトミックな操作であると仮定できる場合、ライターは書き込むバッファを探すためにbuffer_strategy_lockを取得する必要さえありません。各バッファヘッダーをダーティビットを確認するのに十分な時間だけスピンロックするだけで済みます。その仮定がない場合でも、ライターはバッファをスキャンしている間ではなく、変数の値を読み取るのに十分な時間だけロックを取得すれば済みます。（これは、PG 8.0と比較して、ライターの競合コストにおける非常に大幅な改善です。）

バックグラウンドライターは、バッファを書き出す間、そのバッファに対して共有コンテンツロックを取得します（バッファの内容をディスクにフラッシュする他のすべてのプロセスも同様に行う必要があります）。これにより、ディスクに転送されるページイメージが適度に一貫していることが保証されます。ヒントビットの更新を1つか2つ見逃す可能性がありますが、バッファアクセスルールの下で言及された理由と同じ理由で、これは問題ではありません。

8.4以降、バックグラウンドライターは、実行すべき潜在的に拡張されたリカバリがある場合、リカバリモード中に開始されます。書き込むチェックポイントが技術的には再開ポイント（restartpoint）であることを除けば、通常の処理と同じサービスを実行します。
