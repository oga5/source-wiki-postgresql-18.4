ストレージマネージャ
================

オリジナルのBerkeley Postgresシステムには複数のストレージマネージャが存在していましたが、現在残っているのは「磁気ディスク」マネージャのみです。（BerkeleyにはSony WORM光ディスクジュークボックスや永続メインメモリ用のマネージャもありましたが、これらは外部にリリースされたPostgresやPostgreSQLのどのバージョンでもサポートされていませんでした。）「磁気ディスク」マネージャという名称は、実際にはオペレーティングシステムが標準的なファイルシステム操作を提供するあらゆる種類のデバイスをサポートしているため、非常に誤解を招く名称となっています。今日では、関心のあるほとんどすべてのデバイスがこれに該当します。しかし、将来的に他の種類のストレージマネージャを再導入したいと考える人が現れる可能性に備え、ストレージマネージャの切り替えという概念は維持しています。ストレージアクセス操作はC言語の関数呼び出しを1回余分に行うよりもはるかにコストがかかるため、切り替えレイヤーを削除しても目に見えるような節約にはなりません。

Berkeley Postgresでは、各リレーションにはそれを使用するストレージマネージャのIDがタグ付けされていました。これは廃止されました。もし将来的にシステムカタログに複数のストレージマネージャを再導入することがあれば、ストレージマネージャをテーブルスペースに関連付ける方が合理的でしょう。

このディレクトリ内のファイルとその内容は以下の通りです。

    smgr.c	ストレージマネージャの切り替えディスパッチコード。このファイルのルーチンは、上位レベルのコードから要求されたストレージアクセスを実行するために、適切なストレージマネージャを呼び出します。smgr.cはファイルハンドルキャッシュ（SMgrRelationテーブル）の管理も行います。

    md.c	「磁気ディスク」ストレージマネージャ。実際にはカーネルのファイルシステム操作へのインターフェースに過ぎません。

md.cはsrc/backend/storage/file/fd.cに依存していることに注意してください。


リレーションフォーク
================

8.4以降、単一のsmgrリレーションは、リレーションフォークと呼ばれる複数の物理ファイルで構成できるようになりました。これにより、空き領域情報などの追加メタデータを別のフォークに格納できるようになります。これらのフォークは、メインデータファイルとは独立して拡張や切り詰めが可能でありながら、システムカタログ上ではすべて単一の物理リレーションとして扱うことができます。

メインフォーク（フォーク番号0、またはMAIN_FORKNUM）は常に存在すると想定されています。フォーク番号はsrc/include/common/relpath.hで割り当てられています。smgr.cおよびmd.cの関数は、relfilelocatorとブロック番号に加え、アクセスしたいリレーションフォークを特定するための追加のフォーク番号引数を取ります。ほとんどのコードはメインフォークへのアクセスを必要とするため、利便性を考慮して、バッファマネージャにはMAIN_FORKNUMにアクセスするReadBufferのショートカット版が用意されています。
