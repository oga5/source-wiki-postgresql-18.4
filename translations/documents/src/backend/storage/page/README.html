チェックサム
---------

データページのチェックサムは、I/Oシステムによる破損を検出するように設計されています。
大規模サーバーファームに関する研究（http://www.cs.toronto.edu/~bianca/papers/sigmetrics09.pdf、2010/12/22の-hackersメーリングリストで議論）によると、修正不可能なメモリエラーの発生率は非常に低いと測定されているため、バッファをメモリエラーから保護することはしていません。

現在の実装では、initdb時にシステム全体で有効にするか、オフラインのクラスタに対してpg_checksumsツールを使用する必要があります。

データページのチェックサムは常に有効なわけではありません！！
チェックサムは、ページが共有プールから出る際に有効となり、その後I/Oの結果として共有プールに再入する際にチェックされます。
共有プール内のバッファのチェックサムは、バッファをフラッシュする直前に設定されます。その結果、データ変更やヒントビットの更新のためにページを変更すると、暗黙的にページのチェックサムは無効になります。つまり、共有バッファ内の多くの、あるいはほとんどのページは無効なページチェックサムを持っている可能性があるため、pd_checksumフィールドの解釈には注意してください。

つまり、WALログに記録されたページへの変更はページチェックサムを更新しないため、フルページイメージは有効なチェックサムを持っていない可能性があります。しかし、それらのページイメージはWAL CRCによって保護されているため、このメカニズムとは別に検証されます。WALリプレイは、フルページイメージのチェックサムをテストすべきではありません。

これを理解する最善の方法は、WAL CRCはWALストリームに入るレコードを保護し、データページ検証は共有バッファプールに入るブロックを保護するものだと考えることです。これらは目的は似ていますが、完全に別個のものです。これらを合わせることで、PostgreSQLが管理するメモリに再入するデータの誤りを確実に検出できるようになります。また、WALチェックサムは32ビットCRCですが、ページチェックサムは16ビットであることにも注意してください。

データブロックの書き込みが失敗した場合、書き込みによってページが破損（torn page）する可能性があります。これを防ぐのがWALに保存されるフルページ書き込みです。ページが既にダーティな状態でヒントビットを設定することは問題ありません。なぜなら、前回のチェックポイント以降にフルページ書き込みが既に実行されているはずだからです。クリーンなページに対してヒントビットを設定すると、ページ破損が発生する可能性があります。通常、ヒントビットは単なるヒントであるため問題にはなりませんが、ページにチェックサムがある場合、ビットがいくつか失われるとチェックサムが無効になってしまいます。そのため、full_page_writes = on かつチェックサムが有効な場合、WALにフルページイメージを記録するために、専用のWALレコードを書き込む必要があります。ヒントビットの更新はMarkBufferDirtyHint()を使用して保護されるべきであり、この関数が必要に応じてフルページイメージの書き込みを担当します。

ページチェックサムを書き込む際、標準的なページの中央にあるホール（空洞）を形成する、本来ゼロであるべきバイトも含めていることに注意してください。したがって、ストレージからブロックを読み戻す際、ホールがすべてゼロであることを暗黙的にチェックしています。これを行うのは、実際にデータが破壊されていなくても、データを破壊する可能性のあるエラーを確実に発見するためです。WALに保存されたフルページイメージは、ホールがすべてゼロであるかのチェックは行いません。ホール内のデータは単にスキップされ、バックアップブロックが再適用される際に再ゼロ化されます。これを行う理由は、WALの失敗は致命的なエラーでありリカバリを妨げるものですが、通常のデータブロックでのチェックサム失敗はハードエラーではあるものの、ユーザーにとっては非常に深刻な事態であっても、サーバーにとっては必ずしも致命的ではないためです。

リカバリ中は新しいWALレコードを書き込めないため、チェックサムが有効な場合、リカバリ中に設定されるヒントビットは、バッファが既にダーティでない限り、ページをダーティにしてはいけません。ホットスタンバイモードのシステムではヒントビットの設定が有効に働く可能性がありますが、チェックサムが有効な場合、ヒントビットを設定した後にページをダーティにすることはできません（ページ破損のリスクがあるため）。そのため、ヒントビットの更新を含むフルページイメージがプライマリから送られてくるのを待つ必要があります。
