src/timezone/README

これは、以下のIANAタイムゾーンライブラリをPostgreSQL向けに適合させたものです。

	https://www.iana.org/time-zones

タイムゾーンデータおよびライブラリソースコードの最新バージョンは、そのページから直接入手可能です。他のアーカイブ形式には tzdata.zi が含まれていないため、マージ済みのファイル tzdb-NNNNX.tar.lz を取得するのが最適です。過去のバージョンやリリースのアナウンスは、サイト内の他の場所で見つけることができます。

世界の一部地域ではタイムゾーンのルールが頻繁に変更されるため、PostgreSQLの各リリースの前にデータファイルを更新するよう努めるべきです。コードはそれほど頻繁に更新する必要はありませんが、データファイルの解釈に影響を与える可能性のある変更は追跡しなければなりません。


タイムゾーンデータ
================

タイムゾーンのソースデータは、src/timezone/data/ 配下にそのままの形で配布しています。現在、tarballのサイズ削減およびgitリポジトリ内での変更履歴（churn）を抑えるため、省略された単一ファイル形式の「tzdata.zi」のみを配布しています。このファイルを zic に渡すことで、より大きな個別のデータファイルを渡す場合と同じコンパイル済み出力が得られます。

更新時に data/tzdata.zi を複製するだけで済む場合もありますが、tznames/ 配下のタイムゾーン省略名リストを更新するには手作業が必要です。これらは、新しい省略名が考案された場合や、既存の省略名に関連付けられたUTCオフセットが変更された場合に修正する必要があります。これが起こったかどうかを検出するには、data/ 配下に新しいファイルをインストールした後、
	make abbrevs.txt
を実行します。これにより、data/ ファイルに従って現在使用されているすべての省略名を示すファイルが生成されます。これを、tznames/ ファイルが最後に更新された時点のリストである known_abbrevs.txt と比較してください。必要に応じて tznames/ を更新し、同じコミットで known_abbrevs.txt を置き換えてください。通常、既知の省略名の意味が変更された場合は、固定のGMTオフセットではなく、長い形式のゾーン名を参照するように修正するのが適切です。

コアのリグレッションテストスイートは、ゾーンデータと省略名データの単純な検証を行います（特に pg_timezone_names ビューと pg_timezone_abbrevs ビューがエラーをスローしないことを確認します）。提案された更新のクロスチェックとして実行する価値があります。

Windowsの新しいリリース（おそらくService Packを含む）が出た場合は、findtimezone.c のWindowsゾーンからIANAゾーンへのマッピングを更新する必要があるかもしれません。これを行うには2つのアプローチがあります。
1. CLDRプロジェクトの windowsZones.xml ファイルを参照し、そこにリストされているゾーンのうち、我々が持っていないものを追加する。IANAゾーンが複数リストされている場合は、その「territory=001」のマッピングを使用する。
2. 新しいリリースを実行しているWindowsマシン上で src/tools/win32tzlist.pl のスクリプトを実行し、それが検出した新しいタイムゾーンを追加する。（ただし、win32tzlist.pl はどのIANAゾーンにマッピングすべきかを教えてくれないため、これは #1 の完全な代替にはなりません。）
いずれの場合も、古いバージョンで正しく一致させる必要があるため、Windowsから消滅したゾーン名を削除してはいけません。


タイムゾーンコード
================

このディレクトリ内のコードは、現在 tzcode リリース 2020d と同期されています。元の tzcode ライブラリとは多くの見た目上の（あるいはそれ以上の）違いがありますが、アップストリームバージョンの差分は通常、我々のバージョンにも反映させるべきです。それに関するいくつかの注意点を以下に記します。

基本的にはアップストリームのコードをそのまま使用したいところですが、完全に一致させることを妨げるいくつかの考慮事項があります。

* 可読性/保守性のために、我々の規約に合わせてコードを再フォーマットしています。これには pgindent の適用や、アップストリームで多用されている「register」宣言の削除が含まれます。（以前は古いスタイルの関数宣言をC89スタイルに変換することも含まれていましたが、幸いにも彼らがそれを修正してくれました。）

* コードがPostgresの移植性規約に従う必要があります。これには、手作業でハックした #define ではなく、configure の結果に依存することが含まれます（特に private.h を参照）。

* 同様に、古いシステムには存在しない可能性のある <stdint.h> の機能に依存することを避けています。具体的には、int_fast32_t/int_fast64_t ではなく、Postgresの int32 および int64 typedef 定義を使用することを意味します。同様に、INT32_MIN/MAX ではなく PG_INT32_MIN/MAX を使用します。（C99を必要としないすべてのPGバージョンのサポートを終了すれば、<stdint.h> に依存してこれらの一連の差分を削除することは現実的になりますが、その日はまだ来ていません。）

* Postgresは通常、独自の <time.h> 関数を持つシステム上でビルドされるため、それらとの競合を避けなければなりません。このため、typedef time_t を pg_time_t に名前変更し、他の公開されている名前のほとんどについても同様の対応が必要です。

* zic.c の typedef "lineno" は "lineno_t" に名前変更されています。これは、typedef リストに "lineno" があると、他のファイルで同じ名前の変数を使用している場合に pgindent が不適切な動作をするためです。

* tzload() および tzparse() 内部関数を公開し、前者のAPIをわずかに変更しました。これは一部、ファイル自体を開くのではなく、独自の pg_open_tzfile() に依存するようになったためです。

* tzparse() は、TZDEFRULES ゾーンをロードしようとしないように調整されています。

* 不要なコードはかなりの量削除されており、これには非標準のオプションAPIがすべて含まれます。また、localtime.c の末尾に独自の関数をいくつか追加しています。

* zic.c には -P (print_abbrevs) スイッチのサポートを追加しました。これは、前述の現在使用中のゾーン省略名の要約である「abbrevs.txt」を作成するために使用されます。


新しい tzcode リリースを我々のコードと比較する最も便利な方法は、まず tzcode ソースファイルを以下のような sed フィルタに通すことです。

    sed -r \
        -e 's/^([ \t]*)\*\*([ \t])/\1 *\2/' \
        -e 's/^([ \t]*)\*\*$/\1 */' \
        -e 's|^\*/| */|' \
        -e 's/\bregister[ \t]//g' \
        -e 's/\bATTRIBUTE_PURE[ \t]//g' \
        -e 's/int_fast32_t/int32/g' \
        -e 's/int_fast64_t/int64/g' \
        -e 's/intmax_t/int64/g' \
        -e 's/INT32_MIN/PG_INT32_MIN/g' \
        -e 's/INT32_MAX/PG_INT32_MAX/g' \
        -e 's/INTMAX_MIN/PG_INT64_MIN/g' \
        -e 's/INTMAX_MAX/PG_INT64_MAX/g' \
        -e 's/struct[ \t]+tm\b/struct pg_tm/g' \
        -e 's/\btime_t\b/pg_time_t/g' \
        -e 's/lineno/lineno_t/g' \

その後、それらを pgindent に通します。（最初の3つの sed パターンは、ブロックコメントスタイルを pgindent が壊さない形式に変換するためのもので、残りは前述の他の点に対処するためのものです。）その後、ファイルは我々の対応するファイルと直接 diff を取ることができます。また、通常は（同じ方法で処理した後に）以前の tzcode リリースと diff を取り、その差分を我々のファイルに適用しようとするのが役立ちます。これにより、ほとんどの変更を機械的に処理できます。
