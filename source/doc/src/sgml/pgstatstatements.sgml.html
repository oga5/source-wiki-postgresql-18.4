3<sect1 id="pgstatstatements" xreflabel="pg_stat_statements">
4 <title>pg_stat_statements &mdash; track statistics of SQL planning and execution</title>
5
6 <indexterm zone="pgstatstatements">
7  <primary>pg_stat_statements</primary>
8 </indexterm>
9
10 <para>
11  The <filename>pg_stat_statements</filename> module provides a means for
12  tracking planning and execution statistics of all SQL statements executed by
13  a server.
14 </para>
15
16 <para>
17  The module must be loaded by adding <literal>pg_stat_statements</literal> to
18  <xref linkend="guc-shared-preload-libraries"/> in
19  <filename>postgresql.conf</filename>, because it requires additional shared memory.
20  This means that a server restart is needed to add or remove the module.
21  In addition, query identifier calculation must be enabled in order for the
22  module to be active, which is done automatically if <xref linkend="guc-compute-query-id"/>
23  is set to <literal>auto</literal> or <literal>on</literal>, or any third-party
24  module that calculates query identifiers is loaded.
25 </para>
26
27 <para>
28   When <filename>pg_stat_statements</filename> is active, it tracks
29   statistics across all databases of the server.  To access and manipulate
30   these statistics, the module provides views
31   <structname>pg_stat_statements</structname> and
32   <structname>pg_stat_statements_info</structname>,
33   and the utility functions <function>pg_stat_statements_reset</function> and
34   <function>pg_stat_statements</function>.  These are not available globally but
35   can be enabled for a specific database with
36   <command>CREATE EXTENSION pg_stat_statements</command>.
37 </para>
38
39 <sect2 id="pgstatstatements-pg-stat-statements">
40  <title>The <structname>pg_stat_statements</structname> View</title>
41
42  <para>
43   The statistics gathered by the module are made available via a
44   view named <structname>pg_stat_statements</structname>.  This view
45   contains one row for each distinct combination of database ID, user
46   ID, query ID and whether it's a top-level statement or not (up to
47   the maximum number of distinct statements that the module can track).
48   The columns of the view are shown in
49   <xref linkend="pgstatstatements-columns"/>.
50  </para>

582  <para>
583   For security reasons, only superusers and roles with privileges of the
584   <literal>pg_read_all_stats</literal> role are allowed to see the SQL text and
585   <structfield>queryid</structfield> of queries executed by other users.
586   Other users can see the statistics, however, if the view has been installed
587   in their database.
588  </para>
589
590  <para>
591   Plannable queries (that is, <command>SELECT</command>, <command>INSERT</command>,
592   <command>UPDATE</command>, <command>DELETE</command>, and <command>MERGE</command>)
593   and utility commands are combined into a single
594   <structname>pg_stat_statements</structname> entry whenever they have identical query
595   structures according to an internal hash calculation.  Typically, two
596   queries will be considered the same for this purpose if they are
597   semantically equivalent except for the values of literal constants
598   appearing in the query.
599  </para>
600
601  <note>
602   <para>
603    The following details about constant replacement and
604    <structfield>queryid</structfield> only apply when <xref
605    linkend="guc-compute-query-id"/> is enabled.  If you use an external
606    module instead to compute <structfield>queryid</structfield>, you
607    should refer to its documentation for details.
608   </para>
609  </note>
610
611  <para>
612   When a constant's value has been ignored for purposes of matching the query
613   to other queries, the constant is replaced by a parameter symbol, such
614   as <literal>$1</literal>, in the <structname>pg_stat_statements</structname>
615   display.
616   The rest of the query text is that of the first query that had the
617   particular <structfield>queryid</structfield> hash value associated with the
618   <structname>pg_stat_statements</structname> entry.
619  </para>
620
621  <para>
622   Queries on which normalization can be applied may be observed with constant
623   values in <structname>pg_stat_statements</structname>, especially when there
624   is a high rate of entry deallocations. To reduce the likelihood of this
625   happening, consider increasing <varname>pg_stat_statements.max</varname>.
626   The <structname>pg_stat_statements_info</structname> view, discussed below
627   in <xref linkend="pgstatstatements-pg-stat-statements-info"/>,
628   provides statistics about entry deallocations.
629  </para>
630
631  <para>
632   In some cases, queries with visibly different texts might get merged into a
633   single <structname>pg_stat_statements</structname> entry; as explained above,
634   this is expected to happen for semantically equivalent queries.
635   In addition, if the only difference between queries is the number of elements
636   in a list of constants, the list will get squashed down to a single element but shown
637   with a commented-out list indicator:
638
639<screen>
640=# SELECT pg_stat_statements_reset();
641=# SELECT * FROM test WHERE a IN (1, 2, 3, 4, 5, 6, 7);
642=# SELECT * FROM test WHERE a IN (1, 2, 3, 4, 5, 6, 7, 8);
643=# SELECT query, calls FROM pg_stat_statements
644   WHERE query LIKE 'SELECT%';
645-[ RECORD 1 ]------------------------------
646query | SELECT * FROM test WHERE a IN ($1 /*, ... */)
647calls | 2
648</screen>
649
650   In addition to these cases, there is a small chance of hash collisions
651   causing unrelated queries to be merged into one entry.
652   (This cannot happen for queries belonging to different users or databases,
653   however.)
654  </para>
655
656  <para>
657   Since the <structfield>queryid</structfield> hash value is computed on the
658   post-parse-analysis representation of the queries, the opposite is
659   also possible: queries with identical texts might appear as
660   separate entries, if they have different meanings as a result of
661   factors such as different <varname>search_path</varname> settings.
662  </para>
663
664  <para>
665   Consumers of <structname>pg_stat_statements</structname> may wish to use
666   <structfield>queryid</structfield> (perhaps in combination with
667   <structfield>dbid</structfield> and <structfield>userid</structfield>) as a more stable
668   and reliable identifier for each entry than its query text.
669   However, it is important to understand that there are only limited
670   guarantees around the stability of the <structfield>queryid</structfield> hash
671   value.  Since the identifier is derived from the
672   post-parse-analysis tree, its value is a function of, among other
673   things, the internal object identifiers appearing in this representation.
674   This has some counterintuitive implications.  For example,
675   <filename>pg_stat_statements</filename> will consider two apparently-identical
676   queries to be distinct, if they reference for example a function that was
677   dropped and recreated between the executions of the two queries.
678   Conversely, if a table is dropped and recreated between the
679   executions of queries, two apparently-identical queries may be
680   considered the same. However, if the alias for a table is different
681   for otherwise-similar queries, these queries will be considered
682   distinct.
683   The hashing process is also sensitive to differences in
684   machine architecture and other facets of the platform.
685   Furthermore, it is not safe to assume that <structfield>queryid</structfield>
686   will be stable across major versions of <productname>PostgreSQL</productname>.
687  </para>
688
689  <para>
690   Two servers participating in replication based on physical WAL replay can
691   be expected to have identical <structfield>queryid</structfield> values for
692   the same query.  However, logical replication schemes do not promise to
693   keep replicas identical in all relevant details, so
694   <structfield>queryid</structfield> will not be a useful identifier for
695   accumulating costs across a set of logical replicas.
696   If in doubt, direct testing is recommended.
697  </para>
698
699  <para>
700   Generally, it can be assumed that <structfield>queryid</structfield> values
701   are stable between minor version releases of <productname>PostgreSQL</productname>,
702   providing that instances are running on the same machine architecture and
703   the catalog metadata details match.  Compatibility will only be broken
704   between minor versions as a last resort.
705  </para>
706
707  <para>
708   The parameter symbols used to replace constants in
709   representative query texts start from the next number after the
710   highest <literal>$</literal><replaceable>n</replaceable> parameter in the original query
711   text, or <literal>$1</literal> if there was none.  It's worth noting that in
712   some cases there may be hidden parameter symbols that affect this
713   numbering.  For example, <application>PL/pgSQL</application> uses hidden parameter
714   symbols to insert values of function local variables into queries, so that
715   a <application>PL/pgSQL</application> statement like <literal>SELECT i + 1 INTO j</literal>
716   would have representative text like <literal>SELECT i + $2</literal>.
717  </para>
718
719  <para>
720   The representative query texts are kept in an external disk file, and do
721   not consume shared memory.  Therefore, even very lengthy query texts can
722   be stored successfully.  However, if many long query texts are
723   accumulated, the external file might grow unmanageably large.  As a
724   recovery method if that happens, <filename>pg_stat_statements</filename> may
725   choose to discard the query texts, whereupon all existing entries in
726   the <structname>pg_stat_statements</structname> view will show
727   null <structfield>query</structfield> fields, though the statistics associated with
728   each <structfield>queryid</structfield> are preserved.  If this happens, consider
729   reducing <varname>pg_stat_statements.max</varname> to prevent
730   recurrences.
731  </para>
732
733  <para>
734   <structfield>plans</structfield> and <structfield>calls</structfield> aren't
735   always expected to match because planning and execution statistics are
736   updated at their respective end phase, and only for successful operations.
737   For example, if a statement is successfully planned but fails during
738   the execution phase, only its planning statistics will be updated.
739   If planning is skipped because a cached plan is used, only its execution
740   statistics will be updated.
741  </para>
742 </sect2>

805    <term>
806     <function>pg_stat_statements_reset(userid Oid, dbid Oid, queryid
807     bigint, minmax_only boolean) returns timestamp with time zone</function>
808     <indexterm>

812
813    <listitem>
814     <para>
815      <function>pg_stat_statements_reset</function> discards statistics
816      gathered so far by <filename>pg_stat_statements</filename> corresponding
817      to the specified <structfield>userid</structfield>, <structfield>dbid</structfield>
818      and <structfield>queryid</structfield>.  If any of the parameters are not
819      specified, the default value <literal>0</literal>(invalid) is used for
820      each of them and the statistics that match with other parameters will be
821      reset.  If no parameter is specified or all the specified parameters are
822      <literal>0</literal>(invalid), it will discard all statistics.
823      If all statistics in the <filename>pg_stat_statements</filename>
824      view are discarded, it will also reset the statistics in the
825      <structname>pg_stat_statements_info</structname> view.
826      When <structfield>minmax_only</structfield> is <literal>true</literal> only the
827      values of minimum and maximum planning and execution time will be reset (i.e.
828      <structfield>min_plan_time</structfield>, <structfield>max_plan_time</structfield>,
829      <structfield>min_exec_time</structfield> and <structfield>max_exec_time</structfield>
830      fields). The default value for <structfield>minmax_only</structfield> parameter is
831      <literal>false</literal>. Time of last min/max reset performed is shown in
832      <structfield>minmax_stats_since</structfield> field of the
833      <structname>pg_stat_statements</structname> view.
834      This function returns the time of a reset. This time is saved to
835      <structfield>stats_reset</structfield> field of
836      <structname>pg_stat_statements_info</structname> view or to
837      <structfield>minmax_stats_since</structfield> field of the
838      <structname>pg_stat_statements</structname> view if the corresponding reset was
839      actually performed.
840      By default, this function can only be executed by superusers.
841      Access may be granted to others using <command>GRANT</command>.
842     </para>
843    </listitem>
844   </varlistentry>

849     <indexterm>

851      <secondary>function</secondary>
852     </indexterm>
853    </term>
854
855    <listitem>
856     <para>
857      The <structname>pg_stat_statements</structname> view is defined in
858      terms of a function also named <function>pg_stat_statements</function>.
859      It is possible for clients to call
860      the <function>pg_stat_statements</function> function directly, and by
861      specifying <literal>showtext := false</literal> have query text be
862      omitted (that is, the <literal>OUT</literal> argument that corresponds
863      to the view's <structfield>query</structfield> column will return nulls).  This
864      feature is intended to support external tools that might wish to avoid
865      the overhead of repeatedly retrieving query texts of indeterminate
866      length.  Such tools can instead cache the first query text observed
867      for each entry themselves, since that is
868      all <filename>pg_stat_statements</filename> itself does, and then retrieve
869      query texts only as needed.  Since the server stores query texts in a
870      file, this approach may reduce physical I/O for repeated examination
871      of the <structname>pg_stat_statements</structname> data.
872     </para>

879  <title>Configuration Parameters</title>
880
881  <variablelist>

883    <term>
884     <varname>pg_stat_statements.max</varname> (<type>integer</type>)
885     <indexterm>
886      <primary><varname>pg_stat_statements.max</varname> configuration parameter</primary>
887     </indexterm>
888    </term>
889
890    <listitem>
891     <para>
892      <varname>pg_stat_statements.max</varname> is the maximum number of
893      statements tracked by the module (i.e., the maximum number of rows
894      in the <structname>pg_stat_statements</structname> view).  If more distinct
895      statements than that are observed, information about the least-executed
896      statements is discarded.  The number of times such information was
897      discarded can be seen in the
898      <structname>pg_stat_statements_info</structname> view.
899      The default value is 5000.
900      This parameter can only be set at server start.
901     </para>
902    </listitem>
903   </varlistentry>
904
905   <varlistentry>
906    <term>
907     <varname>pg_stat_statements.track</varname> (<type>enum</type>)
908     <indexterm>
909      <primary><varname>pg_stat_statements.track</varname> configuration parameter</primary>
910     </indexterm>
911    </term>
912
913    <listitem>
914     <para>
915      <varname>pg_stat_statements.track</varname> controls which statements
916      are counted by the module.
917      Specify <literal>top</literal> to track top-level statements (those issued
918      directly by clients), <literal>all</literal> to also track nested statements
919      (such as statements invoked within functions), or <literal>none</literal> to
920      disable statement statistics collection.
921      The default value is <literal>top</literal>.
922      Only superusers can change this setting.
923     </para>
924    </listitem>
925   </varlistentry>
926
927   <varlistentry>
928    <term>
929     <varname>pg_stat_statements.track_utility</varname> (<type>boolean</type>)
930     <indexterm>
931      <primary><varname>pg_stat_statements.track_utility</varname> configuration parameter</primary>
932     </indexterm>
933    </term>
934
935    <listitem>
936     <para>
937      <varname>pg_stat_statements.track_utility</varname> controls whether
938      utility commands are tracked by the module.  Utility commands are
939      all those other than <command>SELECT</command>, <command>INSERT</command>,
940      <command>UPDATE</command>, <command>DELETE</command>, and <command>MERGE</command>.
941      The default value is <literal>on</literal>.
942      Only superusers can change this setting.
943     </para>
944    </listitem>
945   </varlistentry>
946
947   <varlistentry>
948    <term>
949     <varname>pg_stat_statements.track_planning</varname> (<type>boolean</type>)
950     <indexterm>
951      <primary><varname>pg_stat_statements.track_planning</varname> configuration parameter</primary>
952     </indexterm>
953    </term>
954
955    <listitem>
956     <para>
957      <varname>pg_stat_statements.track_planning</varname> controls whether
958      planning operations and duration are tracked by the module.
959      Enabling this parameter may incur a noticeable performance penalty,
960      especially when statements with identical query structure are executed
961      by many concurrent connections which compete to update a small number of
962      <structname>pg_stat_statements</structname> entries.
963      The default value is <literal>off</literal>.
964      Only superusers can change this setting.
965     </para>
966    </listitem>
967   </varlistentry>
968
969   <varlistentry>
970    <term>
971     <varname>pg_stat_statements.save</varname> (<type>boolean</type>)
972     <indexterm>
973      <primary><varname>pg_stat_statements.save</varname> configuration parameter</primary>
974     </indexterm>
975    </term>
976
977    <listitem>
978     <para>
979      <varname>pg_stat_statements.save</varname> specifies whether to
980      save statement statistics across server shutdowns.
981      If it is <literal>off</literal> then statistics are not saved at
982      shutdown nor reloaded at server start.
983      The default value is <literal>on</literal>.
984      This parameter can only be set in the <filename>postgresql.conf</filename>
985      file or on the server command line.
986     </para>
987    </listitem>
988   </varlistentry>
989  </variablelist>
990
991  <para>
992   The module requires additional shared memory proportional to
993   <varname>pg_stat_statements.max</varname>.  Note that this
994   memory is consumed whenever the module is loaded, even if
995   <varname>pg_stat_statements.track</varname> is set to <literal>none</literal>.
996  </para>
997
998  <para>
999   These parameters must be set in <filename>postgresql.conf</filename>.
1000   Typical usage might be:
1001
1002<programlisting>
1003# postgresql.conf
1004shared_preload_libraries = 'pg_stat_statements'
1005
1006compute_query_id = on
1007pg_stat_statements.max = 10000
1008pg_stat_statements.track = all
1009</programlisting>
1010  </para>
