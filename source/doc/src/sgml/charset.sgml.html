4 <title>Localization</title>
5
6 <para>
7  This chapter describes the available localization features from the
8  point of view of the administrator.
9  <productname>PostgreSQL</productname> supports two localization
10  facilities:
11
12   <itemizedlist>
13    <listitem>
14     <para>
15      Using the locale features of the operating system to provide
16      locale-specific collation order, number formatting, translated
17      messages, and other aspects.
18      This is covered in <xref linkend="locale"/> and
19      <xref linkend="collation"/>.
20     </para>
21    </listitem>
22
23    <listitem>
24     <para>
25      Providing a number of different character sets to support storing text
26      in all kinds of languages, and providing character set translation
27      between client and server.
28      This is covered in <xref linkend="multibyte"/>.
29     </para>
30    </listitem>
31   </itemizedlist>
32  </para>

35 <sect1 id="locale">
36  <title>Locale Support</title>
37
38  <indexterm zone="locale"><primary>locale</primary></indexterm>
39
40  <para>
41   <firstterm>Locale</firstterm> support refers to an application respecting
42   cultural preferences regarding alphabets, sorting, number
43   formatting, etc.  <productname>PostgreSQL</productname> uses the standard ISO
44   C and <acronym>POSIX</acronym> locale facilities provided by the server operating
45   system.  For additional information refer to the documentation of your
46   system.
47  </para>
48
49  <sect2 id="locale-overview">
50   <title>Overview</title>
51
52   <para>
53    Locale support is automatically initialized when a database
54    cluster is created using <command>initdb</command>.
55    <command>initdb</command> will initialize the database cluster
56    with the locale setting of its execution environment by default,
57    so if your system is already set to use the locale that you want
58    in your database cluster then there is nothing else you need to
59    do.  If you want to use a different locale (or you are not sure
60    which locale your system is set to), you can instruct
61    <command>initdb</command> exactly which locale to use by
62    specifying the <option>--locale</option> option. For example:
63<screen>
64initdb --locale=sv_SE
65</screen>
66   </para>
67
68   <para>
69    This example for Unix systems sets the locale to Swedish
70    (<literal>sv</literal>) as spoken
71    in Sweden (<literal>SE</literal>).  Other possibilities might include
72    <literal>en_US</literal> (U.S. English) and <literal>fr_CA</literal> (French
73    Canadian).  If more than one character set can be used for a
74    locale then the specifications can take the form
75    <replaceable>language_territory.codeset</replaceable>.  For example,
76    <literal>fr_BE.UTF-8</literal> represents the French language (fr) as
77    spoken in Belgium (BE), with a <acronym>UTF-8</acronym> character set
78    encoding.
79   </para>
80
81   <para>
82    What locales are available on your
83    system under what names depends on what was provided by the operating
84    system vendor and what was installed.  On most Unix systems, the command
85    <literal>locale -a</literal> will provide a list of available locales.
86    Windows uses more verbose locale names, such as <literal>German_Germany</literal>
87    or <literal>Swedish_Sweden.1252</literal>, but the principles are the same.
88   </para>
89
90   <para>
91    Occasionally it is useful to mix rules from several locales, e.g.,
92    use English collation rules but Spanish messages.  To support that, a
93    set of locale subcategories exist that control only certain
94    aspects of the localization rules:
95
96    <informaltable>
97     <tgroup cols="2">
98      <colspec colname="col1" colwidth="1*"/>
99      <colspec colname="col2" colwidth="3*"/>
100      <tbody>
101       <row>
102        <entry><envar>LC_COLLATE</envar></entry>
103        <entry>String sort order</entry>
104       </row>
105       <row>
106        <entry><envar>LC_CTYPE</envar></entry>
107        <entry>Character classification (What is a letter? Its upper-case equivalent?)</entry>
108       </row>
109       <row>
110        <entry><envar>LC_MESSAGES</envar></entry>
111        <entry>Language of messages</entry>
112       </row>
113       <row>
114        <entry><envar>LC_MONETARY</envar></entry>
115        <entry>Formatting of currency amounts</entry>
116       </row>
117       <row>
118        <entry><envar>LC_NUMERIC</envar></entry>
119        <entry>Formatting of numbers</entry>
120       </row>
121       <row>
122        <entry><envar>LC_TIME</envar></entry>
123        <entry>Formatting of dates and times</entry>
124       </row>
125      </tbody>
126     </tgroup>
127    </informaltable>
128
129    The category names translate into names of
130    <command>initdb</command> options to override the locale choice
131    for a specific category.  For instance, to set the locale to
132    French Canadian, but use U.S. rules for formatting currency, use
133    <literal>initdb --locale=fr_CA --lc-monetary=en_US</literal>.
134   </para>
135
136   <para>
137    If you want the system to behave as if it had no locale support,
138    use the special locale name <literal>C</literal>, or equivalently
139    <literal>POSIX</literal>.
140   </para>
141
142   <para>
143    Some locale categories must have their values
144    fixed when the database is created.  You can use different settings
145    for different databases, but once a database is created, you cannot
146    change them for that database anymore. <literal>LC_COLLATE</literal>
147    and <literal>LC_CTYPE</literal> are these categories.  They affect
148    the sort order of indexes, so they must be kept fixed, or indexes on
149    text columns would become corrupt.
150    (But you can alleviate this restriction using collations, as discussed
151    in <xref linkend="collation"/>.)
152    The default values for these
153    categories are determined when <command>initdb</command> is run, and
154    those values are used when new databases are created, unless
155    specified otherwise in the <command>CREATE DATABASE</command> command.
156   </para>
157
158   <para>
159    The other locale categories can be changed whenever desired
160    by setting the server configuration parameters
161    that have the same name as the locale categories (see <xref
162    linkend="runtime-config-client-format"/> for details).  The values
163    that are chosen by <command>initdb</command> are actually only written
164    into the configuration file <filename>postgresql.conf</filename> to
165    serve as defaults when the server is started.  If you remove these
166    assignments from <filename>postgresql.conf</filename> then the
167    server will inherit the settings from its execution environment.
168   </para>
169
170   <para>
171    Note that the locale behavior of the server is determined by the
172    environment variables seen by the server, not by the environment
173    of any client.  Therefore, be careful to configure the correct locale settings
174    before starting the server.  A consequence of this is that if
175    client and server are set up in different locales, messages might
176    appear in different languages depending on where they originated.
177   </para>
178
179   <note>
180    <para>
181     When we speak of inheriting the locale from the execution
182     environment, this means the following on most operating systems:
183     For a given locale category, say the collation, the following
184     environment variables are consulted in this order until one is
185     found to be set: <envar>LC_ALL</envar>, <envar>LC_COLLATE</envar>
186     (or the variable corresponding to the respective category),
187     <envar>LANG</envar>.  If none of these environment variables are
188     set then the locale defaults to <literal>C</literal>.
189    </para>
190
191    <para>
192     Some message localization libraries also look at the environment
193     variable <envar>LANGUAGE</envar> which overrides all other locale
194     settings for the purpose of setting the language of messages.  If
195     in doubt, please refer to the documentation of your operating
196     system, in particular the documentation about
197     <application>gettext</application>.
198    </para>
199   </note>
200
201   <para>
202    To enable messages to be translated to the user's preferred language,
203    <acronym>NLS</acronym> must have been selected at build time
204    (<literal>configure --enable-nls</literal>).  All other locale support is
205    built in automatically.
206   </para>
207  </sect2>

209  <sect2 id="locale-behavior">
210   <title>Behavior</title>
211
212   <para>
213    The locale settings influence the following SQL features:
214
215    <itemizedlist>
216     <listitem>
217      <para>
218       Sort order in queries using <literal>ORDER BY</literal> or the standard
219       comparison operators on textual data
220       <indexterm><primary>ORDER BY</primary><secondary>and locales</secondary></indexterm>
221      </para>
222     </listitem>
223
224     <listitem>
225      <para>
226       The <function>upper</function>, <function>lower</function>, and <function>initcap</function>
227       functions
228       <indexterm><primary>upper</primary><secondary>and locales</secondary></indexterm>
229       <indexterm><primary>lower</primary><secondary>and locales</secondary></indexterm>
230      </para>
231     </listitem>
232
233     <listitem>
234      <para>
235       Pattern matching operators (<literal>LIKE</literal>, <literal>SIMILAR TO</literal>,
236       and POSIX-style regular expressions); locales affect both case
237       insensitive matching and the classification of characters by
238       character-class regular expressions
239       <indexterm><primary>LIKE</primary><secondary>and locales</secondary></indexterm>
240       <indexterm><primary>regular expressions</primary><secondary>and locales</secondary></indexterm>
241      </para>
242     </listitem>
243
244     <listitem>
245      <para>
246       The <function>to_char</function> family of functions
247       <indexterm><primary>to_char</primary><secondary>and locales</secondary></indexterm>
248      </para>
249     </listitem>
250
251     <listitem>
252      <para>
253       The ability to use indexes with <literal>LIKE</literal> clauses
254      </para>
255     </listitem>
256    </itemizedlist>
257   </para>
258
259   <para>
260    The drawback of using locales other than <literal>C</literal> or
261    <literal>POSIX</literal> in <productname>PostgreSQL</productname> is its performance
262    impact. It slows character handling and prevents ordinary indexes
263    from being used by <literal>LIKE</literal>. For this reason use locales
264    only if you actually need them.
265   </para>
266
267   <para>
268    As a workaround to allow <productname>PostgreSQL</productname> to use indexes
269    with <literal>LIKE</literal> clauses under a non-C locale, several custom
270    operator classes exist. These allow the creation of an index that
271    performs a strict character-by-character comparison, ignoring
272    locale comparison rules. Refer to <xref linkend="indexes-opclass"/>
273    for more information.  Another approach is to create indexes using
274    the <literal>C</literal> collation, as discussed in
275    <xref linkend="collation"/>.
276   </para>
277  </sect2>

279  <sect2 id="locale-selecting-locales">
280   <title>Selecting Locales</title>
281
282   <para>
283    Locales can be selected in different scopes depending on requirements.
284    The above overview showed how locales are specified using
285    <command>initdb</command> to set the defaults for the entire cluster.  The
286    following list shows where locales can be selected.  Each item provides
287    the defaults for the subsequent items, and each lower item allows
288    overriding the defaults on a finer granularity.
289   </para>
290
291   <orderedlist>
292    <listitem>
293     <para>
294      As explained above, the environment of the operating system provides the
295      defaults for the locales of a newly initialized database cluster.  In
296      many cases, this is enough: if the operating system is configured for
297      the desired language/territory, by default
298      <productname>PostgreSQL</productname> will also behave according
299      to that locale.
300     </para>
301    </listitem>
302
303    <listitem>
304     <para>
305      As shown above, command-line options for <command>initdb</command>
306      specify the locale settings for a newly initialized database cluster.
307      Use this if the operating system does not have the locale configuration
308      you want for your database system.
309     </para>
310    </listitem>
311
312    <listitem>
313     <para>
314      A locale can be selected separately for each database.  The SQL command
315      <command>CREATE DATABASE</command> and its command-line equivalent
316      <command>createdb</command> have options for that.  Use this for example
317      if a database cluster houses databases for multiple tenants with
318      different requirements.
319     </para>
320    </listitem>
321
322    <listitem>
323     <para>
324      Locale settings can be made for individual table columns.  This uses an
325      SQL object called <firstterm>collation</firstterm> and is explained in
326      <xref linkend="collation"/>.  Use this for example to sort data in
327      different languages or customize the sort order of a particular table.
328     </para>
329    </listitem>
330
331    <listitem>
332     <para>
333      Finally, locales can be selected for an individual query.  Again, this
334      uses SQL collation objects.  This could be used to change the sort order
335      based on run-time choices or for ad-hoc experimentation.
336     </para>
337    </listitem>
338   </orderedlist>

341  <sect2 id="locale-providers">
342   <title>Locale Providers</title>
343
344   <para>
345    A locale provider specifies which library defines the locale behavior for
346    collations and character classifications.
347   </para>
348
349   <para>
350    The commands and tools that select the locale settings, as described
351    above, each have an option to select the locale provider. Here is an
352    example to initialize a database cluster using the ICU provider:
353<programlisting>
354initdb --locale-provider=icu --icu-locale=en
355</programlisting>
356    See the description of the respective commands and programs for
357    details.  Note that you can mix locale providers at different
358    granularities, for example use <literal>libc</literal> by default for the
359    cluster but have one database that uses the <literal>icu</literal>
360    provider, and then have collation objects using either provider within
361    those databases.
362   </para>
363
364   <para>
365    Regardless of the locale provider, the operating system is still used to
366    provide some locale-aware behavior, such as messages (see <xref
367    linkend="guc-lc-messages"/>).
368   </para>
369
370   <para>
371    The available locale providers are listed below:
372   </para>
373
374   <variablelist>
375    <varlistentry>
376     <term><literal>builtin</literal></term>
377     <listitem>
378      <para>
379       The <literal>builtin</literal> provider uses built-in operations. Only
380       the <literal>C</literal>, <literal>C.UTF-8</literal>, and
381       <literal>PG_UNICODE_FAST</literal> locales are supported for this
382       provider.
383      </para>
384      <para>
385       The <literal>C</literal> locale behavior is identical to the
386       <literal>C</literal> locale in the libc provider. When using this
387       locale, the behavior may depend on the database encoding.
388      </para>
389      <para>
390       The <literal>C.UTF-8</literal> locale is available only for when the
391       database encoding is <literal>UTF-8</literal>, and the behavior is
392       based on Unicode. The collation uses the code point values only. The
393       regular expression character classes are based on the "POSIX
394       Compatible" semantics, and the case mapping is the "simple" variant.
395      </para>
396      <para>
397       The <literal>PG_UNICODE_FAST</literal> locale is available only when
398       the database encoding is <literal>UTF-8</literal>, and the behavior is
399       based on Unicode. The collation uses the code point values only. The
400       regular expression character classes are based on the "Standard"
401       semantics, and the case mapping is the "full" variant.
402      </para>
403     </listitem>
404    </varlistentry>
405
406    <varlistentry>
407     <term><literal>icu</literal></term>
408     <listitem>
409      <para>
410       The <literal>icu</literal> provider uses the external
411       ICU<indexterm><primary>ICU</primary></indexterm>
412       library. <productname>PostgreSQL</productname> must have been
413       configured with support.
414      </para>
415      <para>
416       ICU provides collation and character classification behavior that is
417       independent of the operating system and database encoding, which is
418       preferable if you expect to transition to other platforms without any
419       change in results. <literal>LC_COLLATE</literal> and
420       <literal>LC_CTYPE</literal> can be set independently of the ICU
421       locale.
422      </para>
423      <note>
424       <para>
425        For the ICU provider, results may depend on the version of the ICU
426        library used, as it is updated to reflect changes in natural language
427        over time.
428       </para>
429      </note>
430     </listitem>
431    </varlistentry>
432
433    <varlistentry>
434     <term><literal>libc</literal></term>
435     <listitem>
436      <para>
437       The <literal>libc</literal> provider uses the operating system's C
438       library. The collation and character classification behavior is
439       controlled by the settings <literal>LC_COLLATE</literal> and
440       <literal>LC_CTYPE</literal>, so they cannot be set independently.
441      </para>
442      <note>
443       <para>
444        The same locale name may have different behavior on different
445        platforms when using the libc provider.
446       </para>
447      </note>
448     </listitem>
449    </varlistentry>
450   </variablelist>

455
456   <sect3 id="icu-locale-names">
457    <title>ICU Locale Names</title>
458
459    <para>
460     The ICU format for the locale name is a <link
461     linkend="icu-language-tag">Language Tag</link>.
462
463<programlisting>
464CREATE COLLATION mycollation1 (provider = icu, locale = 'ja-JP');
465CREATE COLLATION mycollation2 (provider = icu, locale = 'fr');
466</programlisting>
467    </para>
468   </sect3>
469
470   <sect3 id="icu-canonicalization">
471    <title>Locale Canonicalization and Validation</title>
472    <para>
473     When defining a new ICU collation object or database with ICU as the
474     provider, the given locale name is transformed ("canonicalized") into a
475     language tag if not already in that form. For instance,
476
477<screen>
478CREATE COLLATION mycollation3 (provider = icu, locale = 'en-US-u-kn-true');
479NOTICE:  using standard form "en-US-u-kn" for locale "en-US-u-kn-true"
480CREATE COLLATION mycollation4 (provider = icu, locale = 'de_DE.utf8');
481NOTICE:  using standard form "de-DE" for locale "de_DE.utf8"
482</screen>
483
484     If you see this notice, ensure that the <symbol>provider</symbol> and
485     <symbol>locale</symbol> are the expected result. For consistent results
486     when using the ICU provider, specify the canonical <link
487     linkend="icu-language-tag">language tag</link> instead of relying on the
488     transformation.
489    </para>
490
491    <para>
492     A locale with no language name, or the special language name
493     <literal>root</literal>, is transformed to have the language
494     <literal>und</literal> ("undefined").
495    </para>
496
497    <para>
498     ICU can transform most libc locale names, as well as some other formats,
499     into language tags for easier transition to ICU. If a libc locale name is
500     used in ICU, it may not have precisely the same behavior as in libc.
501    </para>
502
503    <para>
504     If there is a problem interpreting the locale name, or if the locale name
505     represents a language or region that ICU does not recognize, you will see
506     the following warning:
507
508<screen>
509CREATE COLLATION nonsense (provider = icu, locale = 'nonsense');
510WARNING:  ICU locale "nonsense" has unknown language "nonsense"
511HINT:  To disable ICU locale validation, set parameter icu_validation_level to DISABLED.
512CREATE COLLATION
513</screen>
514
515     <xref linkend="guc-icu-validation-level"/> controls how the message is
516     reported. Unless set to <literal>ERROR</literal>, the collation will
517     still be created, but the behavior may not be what the user intended.
518    </para>
519   </sect3>

521   <sect3 id="icu-language-tag">
522    <title>Language Tag</title>
523
524    <para>
525     A language tag, defined in BCP 47, is a standardized identifier used to
526     identify languages, regions, and other information about a locale.
527    </para>
528
529    <para>
530     Basic language tags are simply
531     <replaceable>language</replaceable><literal>-</literal><replaceable>region</replaceable>;
532     or even just <replaceable>language</replaceable>. The
533     <replaceable>language</replaceable> is a language code
534     (e.g. <literal>fr</literal> for French), and
535     <replaceable>region</replaceable> is a region code
536     (e.g. <literal>CA</literal> for Canada). Examples:
537     <literal>ja-JP</literal>, <literal>de</literal>, or
538     <literal>fr-CA</literal>.
539    </para>
540
541    <para>
542     Collation settings may be included in the language tag to customize
543     collation behavior. ICU allows extensive customization, such as
544     sensitivity (or insensitivity) to accents, case, and punctuation;
545     treatment of digits within text; and many other options to satisfy a
546     variety of uses.
547    </para>
548
549    <para>
550     To include this additional collation information in a language tag,
551     append <literal>-u</literal>, which indicates there are additional
552     collation settings, followed by one or more
553     <literal>-</literal><replaceable>key</replaceable><literal>-</literal><replaceable>value</replaceable>
554     pairs. The <replaceable>key</replaceable> is the key for a <link
555     linkend="icu-collation-settings">collation setting</link> and
556     <replaceable>value</replaceable> is a valid value for that setting. For
557     boolean settings, the <literal>-</literal><replaceable>key</replaceable>
558     may be specified without a corresponding
559     <literal>-</literal><replaceable>value</replaceable>, which implies a
560     value of <literal>true</literal>.
561    </para>
562
563    <para>
564     For example, the language tag <literal>en-US-u-kn-ks-level2</literal>
565     means the locale with the English language in the US region, with
566     collation settings <literal>kn</literal> set to <literal>true</literal>
567     and <literal>ks</literal> set to <literal>level2</literal>. Those
568     settings mean the collation will be case-insensitive and treat a sequence
569     of digits as a single number:
570
571<screen>
572CREATE COLLATION mycollation5 (provider = icu, deterministic = false, locale = 'en-US-u-kn-ks-level2');
573SELECT 'aB' = 'Ab' COLLATE mycollation5 as result;
574 result
575--------
576 t
577(1 row)
578
579SELECT 'N-45' &lt; 'N-123' COLLATE mycollation5 as result;
580 result
581--------
582 t
583(1 row)
584</screen>
585    </para>
586
587    <para>
588     See <xref linkend="icu-custom-collations"/> for details and additional
589     examples of using language tags with custom collation information for the
590     locale.
591    </para>
592   </sect3>

595  <sect2 id="locale-problems">
596   <title>Problems</title>
597
598   <para>
599    If locale support doesn't work according to the explanation above,
600    check that the locale support in your operating system is
601    correctly configured.  To check what locales are installed on your
602    system, you can use the command <literal>locale -a</literal> if
603    your operating system provides it.
604   </para>
605
606   <para>
607    Check that <productname>PostgreSQL</productname> is actually using the locale
608    that you think it is.  The <envar>LC_COLLATE</envar> and <envar>LC_CTYPE</envar>
609    settings are determined when a database is created, and cannot be
610    changed except by creating a new database.  Other locale
611    settings including <envar>LC_MESSAGES</envar> and <envar>LC_MONETARY</envar>
612    are initially determined by the environment the server is started
613    in, but can be changed on-the-fly.  You can check the active locale
614    settings using the <command>SHOW</command> command.
615   </para>
616
617   <para>
618    The directory <filename>src/test/locale</filename> in the source
619    distribution contains a test suite for
620    <productname>PostgreSQL</productname>'s locale support.
621   </para>
622
623   <para>
624    Client applications that handle server-side errors by parsing the
625    text of the error message will obviously have problems when the
626    server's messages are in a different language.  Authors of such
627    applications are advised to make use of the error code scheme
628    instead.
629   </para>
630
631   <para>
632    Maintaining catalogs of message translations requires the on-going
633    efforts of many volunteers that want to see
634    <productname>PostgreSQL</productname> speak their preferred language well.
635    If messages in your language are currently not available or not fully
636    translated, your assistance would be appreciated.  If you want to
637    help, refer to <xref linkend="nls"/> or write to the developers'
638    mailing list.
639   </para>

644 <sect1 id="collation">
645  <title>Collation Support</title>
646
647  <indexterm zone="collation"><primary>collation</primary></indexterm>
648
649  <para>
650   The collation feature allows specifying the sort order and character
651   classification behavior of data per-column, or even per-operation.
652   This alleviates the restriction that the
653   <symbol>LC_COLLATE</symbol> and <symbol>LC_CTYPE</symbol> settings
654   of a database cannot be changed after its creation.
655  </para>
656
657  <sect2 id="collation-concepts">
658   <title>Concepts</title>
659
660   <para>
661    Conceptually, every expression of a collatable data type has a
662    collation.  (The built-in collatable data types are
663    <type>text</type>, <type>varchar</type>, and <type>char</type>.
664    User-defined base types can also be marked collatable, and of course
665    a <glossterm linkend="glossary-domain">domain</glossterm> over a
666    collatable data type is collatable.)  If the
667    expression is a column reference, the collation of the expression is the
668    defined collation of the column.  If the expression is a constant, the
669    collation is the default collation of the data type of the
670    constant.  The collation of a more complex expression is derived
671    from the collations of its inputs, as described below.
672   </para>
673
674   <para>
675    The collation of an expression can be the <quote>default</quote>
676    collation, which means the locale settings defined for the
677    database.  It is also possible for an expression's collation to be
678    indeterminate.  In such cases, ordering operations and other
679    operations that need to know the collation will fail.
680   </para>
681
682   <para>
683    When the database system has to perform an ordering or a character
684    classification, it uses the collation of the input expression.  This
685    happens, for example, with <literal>ORDER BY</literal> clauses
686    and function or operator calls such as <literal>&lt;</literal>.
687    The collation to apply for an <literal>ORDER BY</literal> clause
688    is simply the collation of the sort key.  The collation to apply for a
689    function or operator call is derived from the arguments, as described
690    below.  In addition to comparison operators, collations are taken into
691    account by functions that convert between lower and upper case
692    letters, such as <function>lower</function>, <function>upper</function>, and
693    <function>initcap</function>; by pattern matching operators; and by
694    <function>to_char</function> and related functions.
695   </para>
696
697   <para>
698    For a function or operator call, the collation that is derived by
699    examining the argument collations is used at run time for performing
700    the specified operation.  If the result of the function or operator
701    call is of a collatable data type, the collation is also used at parse
702    time as the defined collation of the function or operator expression,
703    in case there is a surrounding expression that requires knowledge of
704    its collation.
705   </para>
706
707   <para>
708    The <firstterm>collation derivation</firstterm> of an expression can be
709    implicit or explicit.  This distinction affects how collations are
710    combined when multiple different collations appear in an
711    expression.  An explicit collation derivation occurs when a
712    <literal>COLLATE</literal> clause is used; all other collation
713    derivations are implicit.  When multiple collations need to be
714    combined, for example in a function call, the following rules are
715    used:
716
717    <orderedlist>
718     <listitem>
719      <para>
720       If any input expression has an explicit collation derivation, then
721       all explicitly derived collations among the input expressions must be
722       the same, otherwise an error is raised.  If any explicitly
723       derived collation is present, that is the result of the
724       collation combination.
725      </para>
726     </listitem>
727
728     <listitem>
729      <para>
730       Otherwise, all input expressions must have the same implicit
731       collation derivation or the default collation.  If any non-default
732       collation is present, that is the result of the collation combination.
733       Otherwise, the result is the default collation.
734      </para>
735     </listitem>
736
737     <listitem>
738      <para>
739       If there are conflicting non-default implicit collations among the
740       input expressions, then the combination is deemed to have indeterminate
741       collation.  This is not an error condition unless the particular
742       function being invoked requires knowledge of the collation it should
743       apply.  If it does, an error will be raised at run-time.
744      </para>
745     </listitem>
746    </orderedlist>
747
748    For example, consider this table definition:
749<programlisting>
750CREATE TABLE test1 (
751    a text COLLATE "de_DE",
752    b text COLLATE "es_ES",
753    ...
754);
755</programlisting>
756
757    Then in
758<programlisting>
759SELECT a &lt; 'foo' FROM test1;
760</programlisting>
761    the <literal>&lt;</literal> comparison is performed according to
762    <literal>de_DE</literal> rules, because the expression combines an
763    implicitly derived collation with the default collation.  But in
764<programlisting>
765SELECT a &lt; ('foo' COLLATE "fr_FR") FROM test1;
766</programlisting>
767    the comparison is performed using <literal>fr_FR</literal> rules,
768    because the explicit collation derivation overrides the implicit one.
769    Furthermore, given
770<programlisting>
771SELECT a &lt; b FROM test1;
772</programlisting>
773    the parser cannot determine which collation to apply, since the
774    <structfield>a</structfield> and <structfield>b</structfield> columns have conflicting
775    implicit collations.  Since the <literal>&lt;</literal> operator
776    does need to know which collation to use, this will result in an
777    error.  The error can be resolved by attaching an explicit collation
778    specifier to either input expression, thus:
779<programlisting>
780SELECT a &lt; b COLLATE "de_DE" FROM test1;
781</programlisting>
782    or equivalently
783<programlisting>
784SELECT a COLLATE "de_DE" &lt; b FROM test1;
785</programlisting>
786    On the other hand, the structurally similar case
787<programlisting>
788SELECT a || b FROM test1;
789</programlisting>
790    does not result in an error, because the <literal>||</literal> operator
791    does not care about collations: its result is the same regardless
792    of the collation.
793   </para>
794
795   <para>
796    The collation assigned to a function or operator's combined input
797    expressions is also considered to apply to the function or operator's
798    result, if the function or operator delivers a result of a collatable
799    data type.  So, in
800<programlisting>
801SELECT * FROM test1 ORDER BY a || 'foo';
802</programlisting>
803    the ordering will be done according to <literal>de_DE</literal> rules.
804    But this query:
805<programlisting>
806SELECT * FROM test1 ORDER BY a || b;
807</programlisting>
808    results in an error, because even though the <literal>||</literal> operator
809    doesn't need to know a collation, the <literal>ORDER BY</literal> clause does.
810    As before, the conflict can be resolved with an explicit collation
811    specifier:
812<programlisting>
813SELECT * FROM test1 ORDER BY a || b COLLATE "fr_FR";
814</programlisting>
815   </para>
816  </sect2>

818  <sect2 id="collation-managing">
819   <title>Managing Collations</title>
820
821   <para>
822    A collation is an SQL schema object that maps an SQL name to locales
823    provided by libraries installed in the operating system.  A collation
824    definition has a <firstterm>provider</firstterm> that specifies which
825    library supplies the locale data.  One standard provider name
826    is <literal>libc</literal>, which uses the locales provided by the
827    operating system C library.  These are the locales used by most tools
828    provided by the operating system.  Another provider
829    is <literal>icu</literal>, which uses the external
830    ICU<indexterm><primary>ICU</primary></indexterm> library.  ICU locales can only be
831    used if support for ICU was configured when PostgreSQL was built.
832   </para>
833
834   <para>
835    A collation object provided by <literal>libc</literal> maps to a
836    combination of <symbol>LC_COLLATE</symbol> and <symbol>LC_CTYPE</symbol>
837    settings, as accepted by the <literal>setlocale()</literal> system library call.  (As
838    the name would suggest, the main purpose of a collation is to set
839    <symbol>LC_COLLATE</symbol>, which controls the sort order.  But
840    it is rarely necessary in practice to have an
841    <symbol>LC_CTYPE</symbol> setting that is different from
842    <symbol>LC_COLLATE</symbol>, so it is more convenient to collect
843    these under one concept than to create another infrastructure for
844    setting <symbol>LC_CTYPE</symbol> per expression.)  Also,
845    a <literal>libc</literal> collation
846    is tied to a character set encoding (see <xref linkend="multibyte"/>).
847    The same collation name may exist for different encodings.
848   </para>
849
850   <para>
851    A collation object provided by <literal>icu</literal> maps to a named
852    collator provided by the ICU library.  ICU does not support
853    separate <quote>collate</quote> and <quote>ctype</quote> settings, so
854    they are always the same.  Also, ICU collations are independent of the
855    encoding, so there is always only one ICU collation of a given name in
856    a database.
857   </para>

966  <sect3 id="collation-managing-predefined">
967   <title>Predefined Collations</title>
968
969   <para>
970    If the operating system provides support for using multiple locales
971    within a single program (<function>newlocale</function> and related functions),
972    or if support for ICU is configured,
973    then when a database cluster is initialized, <command>initdb</command>
974    populates the system catalog <literal>pg_collation</literal> with
975    collations based on all the locales it finds in the operating
976    system at the time.
977   </para>
978
979   <para>
980    To inspect the currently available locales, use the query <literal>SELECT
981    * FROM pg_collation</literal>, or the command <command>\dOS+</command>
982    in <application>psql</application>.
983   </para>
984
985  <sect4 id="collation-managing-predefined-libc">
986   <title>libc Collations</title>
987
988   <para>
989    For example, the operating system might
990    provide a locale named <literal>de_DE.utf8</literal>.
991    <command>initdb</command> would then create a collation named
992    <literal>de_DE.utf8</literal> for encoding <literal>UTF8</literal>
993    that has both <symbol>LC_COLLATE</symbol> and
994    <symbol>LC_CTYPE</symbol> set to <literal>de_DE.utf8</literal>.
995    It will also create a collation with the <literal>.utf8</literal>
996    tag stripped off the name.  So you could also use the collation
997    under the name <literal>de_DE</literal>, which is less cumbersome
998    to write and makes the name less encoding-dependent.  Note that,
999    nevertheless, the initial set of collation names is
1000    platform-dependent.
1001   </para>
1002
1003   <para>
1004    The default set of collations provided by <literal>libc</literal> map
1005    directly to the locales installed in the operating system, which can be
1006    listed using the command <literal>locale -a</literal>.  In case
1007    a <literal>libc</literal> collation is needed that has different values
1008    for <symbol>LC_COLLATE</symbol> and <symbol>LC_CTYPE</symbol>, or if new
1009    locales are installed in the operating system after the database system
1010    was initialized, then a new collation may be created using
1011    the <xref linkend="sql-createcollation"/> command.
1012    New operating system locales can also be imported en masse using
1013    the <link linkend="functions-admin-collation"><function>pg_import_system_collations()</function></link> function.
1014   </para>
1015
1016   <para>
1017    Within any particular database, only collations that use that
1018    database's encoding are of interest.  Other entries in
1019    <literal>pg_collation</literal> are ignored.  Thus, a stripped collation
1020    name such as <literal>de_DE</literal> can be considered unique
1021    within a given database even though it would not be unique globally.
1022    Use of the stripped collation names is recommended, since it will
1023    make one fewer thing you need to change if you decide to change to
1024    another database encoding.  Note however that the <literal>default</literal>,
1025    <literal>C</literal>, and <literal>POSIX</literal> collations can be used regardless of
1026    the database encoding.
1027   </para>
1028
1029   <para>
1030    <productname>PostgreSQL</productname> considers distinct collation
1031    objects to be incompatible even when they have identical properties.
1032    Thus for example,
1033<programlisting>
1034SELECT a COLLATE "C" &lt; b COLLATE "POSIX" FROM test1;
1035</programlisting>
1036    will draw an error even though the <literal>C</literal> and <literal>POSIX</literal>
1037    collations have identical behaviors.  Mixing stripped and non-stripped
1038    collation names is therefore not recommended.
1039   </para>

1180   <sect3 id="collation-nondeterministic">
1181    <title>Nondeterministic Collations</title>
1182
1183    <para>
1184     A collation is either <firstterm>deterministic</firstterm> or
1185     <firstterm>nondeterministic</firstterm>.  A deterministic collation uses
1186     deterministic comparisons, which means that it considers strings to be
1187     equal only if they consist of the same byte sequence.  Nondeterministic
1188     comparison may determine strings to be equal even if they consist of
1189     different bytes.  Typical situations include case-insensitive comparison,
1190     accent-insensitive comparison, as well as comparison of strings in
1191     different Unicode normal forms.  It is up to the collation provider to
1192     actually implement such insensitive comparisons; the deterministic flag
1193     only determines whether ties are to be broken using bytewise comparison.
1194     See also <ulink url="https://www.unicode.org/reports/tr10">Unicode Technical
1195     Standard 10</ulink> for more information on the terminology.
1196    </para>
1197
1198    <para>
1199     To create a nondeterministic collation, specify the property
1200     <literal>deterministic = false</literal> to <command>CREATE
1201     COLLATION</command>, for example:
1202<programlisting>
1203CREATE COLLATION ndcoll (provider = icu, locale = 'und', deterministic = false);
1204</programlisting>
1205     This example would use the standard Unicode collation in a
1206     nondeterministic way.  In particular, this would allow strings in
1207     different normal forms to be compared correctly.  More interesting
1208     examples make use of the ICU customization facilities explained above.
1209     For example:
1210<programlisting>
1211CREATE COLLATION case_insensitive (provider = icu, locale = 'und-u-ks-level2', deterministic = false);
1212CREATE COLLATION ignore_accents (provider = icu, locale = 'und-u-ks-level1-kc-true', deterministic = false);
1213</programlisting>
1214    </para>
1215
1216    <para>
1217     All standard and predefined collations are deterministic, all
1218     user-defined collations are deterministic by default.  While
1219     nondeterministic collations give a more <quote>correct</quote> behavior,
1220     especially when considering the full power of Unicode and its many
1221     special cases, they also have some drawbacks.  Foremost, their use leads
1222     to a performance penalty.  Note, in particular, that B-tree cannot use
1223     deduplication with indexes that use a nondeterministic collation.  Also,
1224     certain operations are not possible with nondeterministic collations,
1225     such as some pattern matching operations.  Therefore, they should be used
1226     only in cases where they are specifically wanted.
1227    </para>
1228
1229    <tip>
1230     <para>
1231      To deal with text in different Unicode normalization forms, it is also
1232      an option to use the functions/expressions
1233      <function>normalize</function> and <literal>is normalized</literal> to
1234      preprocess or check the strings, instead of using nondeterministic
1235      collations.  There are different trade-offs for each approach.
1236     </para>
1237    </tip>
1238   </sect3>

1241  <sect2 id="icu-custom-collations">
1242   <title>ICU Custom Collations</title>
1243
1244   <para>
1245    ICU allows extensive control over collation behavior by defining new
1246    collations with collation settings as a part of the language tag. These
1247    settings can modify the collation order to suit a variety of needs. For
1248    instance:

1254SELECT 'z' = 'Z' COLLATE ignore_accent_case; -- true

1257CREATE COLLATION upper_first (provider = icu, locale = 'und-u-kf-upper');

1261CREATE COLLATION num_ignore_punct (provider = icu, deterministic = false, locale = 'und-u-ka-shifted-kn');

1267    linkend="icu-collation-settings"/>, or see <xref

1756  <title>Character Set Support</title>
1757
1758  <indexterm zone="multibyte"><primary>character set</primary></indexterm>
1759
1760  <para>
1761   The character set support in <productname>PostgreSQL</productname>
1762   allows you to store text in a variety of character sets (also called
1763   encodings), including
1764   single-byte character sets such as the ISO 8859 series and
1765   multiple-byte character sets such as <acronym>EUC</acronym> (Extended Unix
1766   Code), UTF-8, and Mule internal code.  All supported character sets
1767   can be used transparently by clients, but a few are not supported
1768   for use within the server (that is, as a server-side encoding).
1769   The default character set is selected while
1770   initializing your <productname>PostgreSQL</productname> database
1771   cluster using <command>initdb</command>.  It can be overridden when you
1772   create a database, so you can have multiple
1773   databases each with a different character set.
1774  </para>
1775
1776  <para>
1777   An important restriction, however, is that each database's character set
1778   must be compatible with the database's <envar>LC_CTYPE</envar> (character
1779   classification) and <envar>LC_COLLATE</envar> (string sort order) locale
1780   settings. For <literal>C</literal> or
1781   <literal>POSIX</literal> locale, any character set is allowed, but for other
1782   libc-provided locales there is only one character set that will work
1783   correctly.
1784   (On Windows, however, UTF-8 encoding can be used with any locale.)
1785   If you have ICU support configured, ICU-provided locales can be used
1786   with most but not all server-side encodings.
1787  </para>

2229    <title>Setting the Character Set</title>
2230
2231    <para>
2232     <command>initdb</command> defines the default character set (encoding)
2233     for a <productname>PostgreSQL</productname> cluster. For example,
2234
2235<screen>
2236initdb -E EUC_JP
2237</screen>
2238
2239     sets the default character set to
2240     <literal>EUC_JP</literal> (Extended Unix Code for Japanese).  You
2241     can use <option>--encoding</option> instead of
2242     <option>-E</option> if you prefer longer option strings.
2243     If no <option>-E</option> or <option>--encoding</option> option is
2244     given, <command>initdb</command> attempts to determine the appropriate
2245     encoding to use based on the specified or default locale.
2246    </para>
2247
2248    <para>
2249     You can specify a non-default encoding at database creation time,
2250     provided that the encoding is compatible with the selected locale:
2251
2252<screen>
2253createdb -E EUC_KR -T template0 --lc-collate=ko_KR.euckr --lc-ctype=ko_KR.euckr korean
2254</screen>
2255
2256     This will create a database named <literal>korean</literal> that
2257     uses the character set <literal>EUC_KR</literal>, and locale <literal>ko_KR</literal>.
2258     Another way to accomplish this is to use this SQL command:
2259
2260<programlisting>
2261CREATE DATABASE korean WITH ENCODING 'EUC_KR' LC_COLLATE='ko_KR.euckr' LC_CTYPE='ko_KR.euckr' TEMPLATE=template0;
2262</programlisting>
2263
2264     Notice that the above commands specify copying the <literal>template0</literal>
2265     database.  When copying any other database, the encoding and locale
2266     settings cannot be changed from those of the source database, because
2267     that might result in corrupt data.  For more information see
2268     <xref linkend="manage-ag-templatedbs"/>.
2269    </para>
2270
2271    <para>
2272     The encoding for a database is stored in the system catalog
2273     <literal>pg_database</literal>.  You can see it by using the
2274     <command>psql</command> <option>-l</option> option or the
2275     <command>\l</command> command.
2276
2277<screen>
2278$ <userinput>psql -l</userinput>

2294     <para>
2295      On most modern operating systems, <productname>PostgreSQL</productname>
2296      can determine which character set is implied by the <envar>LC_CTYPE</envar>
2297      setting, and it will enforce that only the matching database encoding is
2298      used.  On older systems it is your responsibility to ensure that you use
2299      the encoding expected by the locale you have selected.  A mistake in
2300      this area is likely to lead to strange behavior of locale-dependent
2301      operations such as sorting.
2302     </para>

2305      <productname>PostgreSQL</productname> will allow superusers to create
2306      databases with <literal>SQL_ASCII</literal> encoding even when
2307      <envar>LC_CTYPE</envar> is not <literal>C</literal> or <literal>POSIX</literal>.  As noted
2308      above, <literal>SQL_ASCII</literal> does not enforce that the data stored in
2309      the database has any particular encoding, and so this choice poses risks
2310      of locale-dependent misbehavior.  Using this combination of settings is
2311      deprecated and may someday be forbidden altogether.
2312     </para>

2316   <sect2 id="multibyte-automatic-conversion">
2317    <title>Automatic Character Set Conversion Between Server and Client</title>
2318
2319    <para>
2320     <productname>PostgreSQL</productname> supports automatic character
2321     set conversion between server and client for many combinations of
2322     character sets (<xref linkend="multibyte-conversions-supported"/>
2323     shows which ones).
2324    </para>
2325
2326    <para>
2327     To enable automatic character set conversion, you have to
2328     tell <productname>PostgreSQL</productname> the character set
2329     (encoding) you would like to use in the client. There are several
2330     ways to accomplish this:
2331
2332     <itemizedlist>
2333      <listitem>
2334       <para>
2335        Using the <command>\encoding</command> command in
2336        <application>psql</application>.
2337        <command>\encoding</command> allows you to change client
2338        encoding on the fly. For
2339        example, to change the encoding to <literal>SJIS</literal>, type:
2340
2341<programlisting>
2342\encoding SJIS
2343</programlisting>
2344       </para>
2345      </listitem>
2346
2347      <listitem>
2348       <para>
2349        <application>libpq</application> (<xref linkend="libpq-control"/>) has functions to control the client encoding.
2350       </para>
2351      </listitem>
2352
2353      <listitem>
2354       <para>
2355        Using <command>SET client_encoding TO</command>.
2356
2357        Setting the client encoding can be done with this SQL command:
2358
2359<programlisting>
2360SET CLIENT_ENCODING TO '<replaceable>value</replaceable>';
2361</programlisting>
2362
2363        Also you can use the standard SQL syntax <literal>SET NAMES</literal>
2364        for this purpose:
2365
2366<programlisting>
2367SET NAMES '<replaceable>value</replaceable>';
2368</programlisting>
2369
2370        To query the current client encoding:
2371
2372<programlisting>
2373SHOW client_encoding;
2374</programlisting>
2375
2376        To return to the default encoding:
2377
2378<programlisting>
2379RESET client_encoding;
2380</programlisting>
2381       </para>
2382      </listitem>
2383
2384      <listitem>
2385       <para>
2386        Using <envar>PGCLIENTENCODING</envar>. If the environment variable
2387        <envar>PGCLIENTENCODING</envar> is defined in the client's
2388        environment, that client encoding is automatically selected
2389        when a connection to the server is made.  (This can
2390        subsequently be overridden using any of the other methods
2391        mentioned above.)
2392       </para>
2393      </listitem>
2394
2395      <listitem>
2396      <para>
2397       Using the configuration variable <xref
2398       linkend="guc-client-encoding"/>. If the
2399       <varname>client_encoding</varname> variable is set, that client
2400       encoding is automatically selected when a connection to the
2401       server is made.  (This can subsequently be overridden using any
2402       of the other methods mentioned above.)
2403       </para>
2404      </listitem>
2405
2406     </itemizedlist>
2407    </para>
2408
2409    <para>
2410     If the conversion of a particular character is not possible
2411     &mdash; suppose you chose <literal>EUC_JP</literal> for the
2412     server and <literal>LATIN1</literal> for the client, and some
2413     Japanese characters are returned that do not have a representation in
2414     <literal>LATIN1</literal> &mdash; an error is reported.
2415    </para>
2416
2417    <para>
2418     If the client character set is defined as <literal>SQL_ASCII</literal>,
2419     encoding conversion is disabled, regardless of the server's character
2420     set.  (However, if the server's character set is
2421     not <literal>SQL_ASCII</literal>, the server will still check that
2422     incoming data is valid for that encoding; so the net effect is as
2423     though the client character set were the same as the server's.)
2424     Just as for the server, use of <literal>SQL_ASCII</literal> is unwise
2425     unless you are working with all-ASCII data.
2426    </para>
2427   </sect2>
