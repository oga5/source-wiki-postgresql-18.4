18 <sect1 id="install-requirements">
19  <title>Requirements</title>
20
21  <para>
22   In general, a modern Unix-compatible platform should be able to run
23   <productname>PostgreSQL</productname>.
24   The platforms that had received specific testing at the
25   time of release are described in <xref linkend="supported-platforms"/>
26   below.
27  </para>
28
29  <para>
30   The following software packages are required for building
31   <productname>PostgreSQL</productname>:
32
33   <itemizedlist>
34    <listitem>
35     <para>
36      <indexterm>
37       <primary>make</primary>
38      </indexterm>
39
40      <acronym>GNU</acronym> <application>make</application> version 3.81 or newer is required; other
41      <application>make</application> programs or older <acronym>GNU</acronym> <application>make</application> versions will <emphasis>not</emphasis> work.
42      (<acronym>GNU</acronym> <application>make</application> is sometimes installed under
43      the name <filename>gmake</filename>.)  To test for <acronym>GNU</acronym>
44      <application>make</application> enter:
45<screen>
46<userinput>make --version</userinput>
47</screen>
48     </para>
49    </listitem>
50
51    <listitem>
52     <para>
53      <indexterm>
54       <primary>Meson</primary>
55      </indexterm>
56
57      Alternatively, <productname>PostgreSQL</productname> can be built using
58      <ulink url="https://mesonbuild.com/">Meson</ulink>.  This is the only
59      option for building <productname>PostgreSQL</productname> on Windows
60      using <productname>Visual Studio</productname>.  For other platforms,
61      using <application>Meson</application> is currently experimental.  If
62      you choose to use <application>Meson</application>, then you don't need
63      <acronym>GNU</acronym> <application>make</application>, but the other
64      requirements below still apply.
65     </para>
66
67     <para>
68      The minimum required version of <application>Meson</application> is 0.54.
69     </para>
70    </listitem>

73     <para>
74      You need an <acronym>ISO</acronym>/<acronym>ANSI</acronym> C compiler (at least
75      C99-compliant). Recent
76      versions of <productname>GCC</productname> are recommended, but
77      <productname>PostgreSQL</productname> is known to build using a wide variety
78      of compilers from different vendors.
79     </para>
80    </listitem>
81
82    <listitem>
83     <para>
84      <application>tar</application> is required to unpack the source
85      distribution, in addition to either
86      <application>gzip</application> or <application>bzip2</application>.
87     </para>
88    </listitem>
89
90    <listitem>
91     <para>
92      <indexterm>
93       <primary>flex</primary>
94      </indexterm>
95      <indexterm>
96       <primary>lex</primary>
97      </indexterm>
98      <indexterm>
99       <primary>bison</primary>
100      </indexterm>
101      <indexterm>
102       <primary>yacc</primary>
103      </indexterm>
104
105      <application>Flex</application> and <application>Bison</application> are
106      required.  Other <application>lex</application> and
107      <application>yacc</application> programs cannot be used.
108      <application>Bison</application> needs to be at least version 2.3.
109     </para>
110    </listitem>
111
112    <listitem>
113     <para>
114      <indexterm>
115       <primary>perl</primary>
116      </indexterm>
117
118      <application>Perl</application> 5.14 or later is needed during the build
119      process and to run some test suites.  (This requirement is separate from
120      the requirements for building <application>PL/Perl</application>; see
121      below.)
122     </para>
123    </listitem>
124
125    <listitem>
126     <para>
127      <indexterm>
128       <primary>readline</primary>
129      </indexterm>
130      <indexterm>
131       <primary>libedit</primary>
132      </indexterm>
133
134      The <acronym>GNU</acronym> <productname>Readline</productname> library is used by
135      default.  It allows <application>psql</application> (the
136      PostgreSQL command line SQL interpreter) to remember each
137      command you type, and allows you to use arrow keys to recall and
138      edit previous commands.  This is very helpful and is strongly
139      recommended.  If you don't want to use it then you must specify
140      the <option>--without-readline</option> option to
141      <filename>configure</filename>. As an alternative, you can often use the
142      BSD-licensed <filename>libedit</filename> library, originally
143      developed on <productname>NetBSD</productname>. The
144      <filename>libedit</filename> library is
145      GNU <productname>Readline</productname>-compatible and is used if
146      <filename>libreadline</filename> is not found, or if
147      <option>--with-libedit-preferred</option> is used as an
148      option to <filename>configure</filename>. If you are using a package-based
149      Linux distribution, be aware that you need both the
150      <literal>readline</literal> and <literal>readline-devel</literal> packages, if
151      those are separate in your distribution.
152     </para>
153    </listitem>
154
155    <listitem>
156     <para>
157      <indexterm>
158       <primary>zlib</primary>
159      </indexterm>
160
161      The <productname>zlib</productname> compression library is
162      used by default. If you don't want to use it then you must
163      specify the <option>--without-zlib</option> option to
164      <filename>configure</filename>. Using this option disables
165      support for compressed archives in <application>pg_dump</application> and
166      <application>pg_restore</application>.
167     </para>
168    </listitem>
169
170    <listitem>
171     <para>
172      The ICU library is used by default. If you don't want to use it then you must specify the <option>--without-icu</option> option to <filename>configure</filename>. Using this option disables support for ICU collation features (see <xref linkend="collation"/>).
173     </para>
174     <para>
175      ICU support requires the <productname>ICU4C</productname> package to be
176      installed.  The minimum required version of
177      <productname>ICU4C</productname> is currently 4.2.
178     </para>
179
180     <para>
181      By default,
182      <productname>pkg-config</productname><indexterm><primary>pkg-config</primary></indexterm>
183      will be used to find the required compilation options.  This is
184      supported for <productname>ICU4C</productname> version 4.6 and later.
185      For older versions, or if <productname>pkg-config</productname> is not
186      available, the variables <envar>ICU_CFLAGS</envar> and
187      <envar>ICU_LIBS</envar> can be specified to
188      <filename>configure</filename>, like in this example:
189<programlisting>
190./configure ... ICU_CFLAGS='-I/some/where/include' ICU_LIBS='-L/some/where/lib -licui18n -licuuc -licudata'
191</programlisting>
192      (If <productname>ICU4C</productname> is in the default search path
193      for the compiler, then you still need to specify nonempty strings in
194      order to avoid use of <productname>pkg-config</productname>, for
195      example, <literal>ICU_CFLAGS=' '</literal>.)
196     </para>
197    </listitem>
198   </itemizedlist>
199  </para>

207    <listitem>
208     <para>
209      To build the server programming language
210      <application>PL/Perl</application> you need a full
211      <productname>Perl</productname> installation, including the
212      <filename>libperl</filename> library and the header files.

218      recent <productname>Perl</productname> versions, but it was not

220      installed Perl at your site.  <filename>configure</filename> will fail
221      if building <application>PL/Perl</application> is selected but it cannot
222      find a shared <filename>libperl</filename>.  In that case, you will have
223      to rebuild and install <productname>Perl</productname> manually to be
224      able to build <application>PL/Perl</application>.  During the
225      configuration process for <productname>Perl</productname>, request a
226      shared library.
227     </para>

229     <para>

231      <application>PL/Perl</application>, you should ensure that the

234      will show whether this is the case).
235     </para>
236    </listitem>

415  <title>Installation Procedure</title>
416
417  <procedure>
418
419  <step id="configure">
420   <title>Configuration</title>
421
422   <indexterm zone="configure">
423    <primary>configure</primary>
424   </indexterm>
425
426   <para>
427    The first step of the installation procedure is to configure the
428    source tree for your system and choose the options you would like.
429    This is done by running the <filename>configure</filename> script. For a
430    default installation simply enter:
431<screen>
432<userinput>./configure</userinput>
433</screen>
434    This script will run a number of tests to determine values for various
435    system dependent variables and detect any quirks of your
436    operating system, and finally will create several files in the
437    build tree to record what it found.
438   </para>
439
440   <para>
441    You can also run <filename>configure</filename> in a directory outside
442    the source tree, and then build there, if you want to keep the build
443    directory separate from the original source files.  This procedure is
444    called a
445    <indexterm><primary>VPATH</primary></indexterm><firstterm>VPATH</firstterm>
446    build.  Here's how:
447<screen>
448<userinput>mkdir build_dir</userinput>
449<userinput>cd build_dir</userinput>
450<userinput>/path/to/source/tree/configure [options go here]</userinput>
451<userinput>make</userinput>
452</screen>
453   </para>
454
455   <para>
456    The default configuration will build the server and utilities, as
457    well as all client applications and interfaces that require only a
458    C compiler. All files will be installed under
459    <filename>/usr/local/pgsql</filename> by default.
460   </para>
461
462   <para>
463    You can customize the build and installation process by supplying one
464    or more command line options to <filename>configure</filename>.
465    Typically you would customize the install location, or the set of
466    optional features that are built.  <filename>configure</filename>
467    has a large number of options, which are described in
468    <xref linkend="configure-options"/>.
469   </para>
470
471   <para>
472    Also, <filename>configure</filename> responds to certain environment
473    variables, as described in <xref linkend="configure-envvars"/>.
474    These provide additional ways to customize the configuration.
475   </para>
476  </step>
477
478  <step id="build">
479   <title>Build</title>
480
481   <para>
482    To start the build, type either of:
483<screen>
484<userinput>make</userinput>
485<userinput>make all</userinput>
486</screen>
487    (Remember to use <acronym>GNU</acronym> <application>make</application>.)
488    The build will take a few minutes depending on your
489    hardware.
490   </para>
491
492  <para>
493   If you want to build everything that can be built, including the
494   documentation (HTML and man pages), and the additional modules
495   (<filename>contrib</filename>), type instead:
496<screen>
497<userinput>make world</userinput>
498</screen>
499  </para>
500
501  <para>
502   If you want to build everything that can be built, including the
503   additional modules (<filename>contrib</filename>), but without
504   the documentation, type instead:
505<screen>
506<userinput>make world-bin</userinput>
507</screen>
508   </para>
509
510   <para>
511    If you want to invoke the build from another makefile rather than
512    manually, you must unset <varname>MAKELEVEL</varname> or set it to zero,
513    for instance like this:
514<programlisting>
515build-postgresql:
516        $(MAKE) -C postgresql MAKELEVEL=0 all
517</programlisting>
518    Failure to do that can lead to strange error messages, typically about
519    missing header files.
520   </para>
521  </step>
522
523  <step>
524   <title>Regression Tests</title>
525
526   <indexterm>
527    <primary>regression test</primary>
528   </indexterm>
529
530   <para>
531    If you want to test the newly built server before you install it,
532    you can run the regression tests at this point. The regression
533    tests are a test suite to verify that <productname>PostgreSQL</productname>
534    runs on your machine in the way the developers expected it
535    to. Type:
536<screen>
537<userinput>make check</userinput>
538</screen>
539    (This won't work as root; do it as an unprivileged user.)
540    See <xref linkend="regress"/> for
541    detailed information about interpreting the test results. You can
542    repeat this test at any later time by issuing the same command.
543   </para>
544  </step>
545
546  <step id="install">
547   <title>Installing the Files</title>
548
549   <note>
550    <para>
551     If you are upgrading an existing system be sure to read
552     <xref linkend="upgrading"/>,
553     which has instructions about upgrading a
554     cluster.
555    </para>
556   </note>
557
558   <para>
559    To install <productname>PostgreSQL</productname> enter:
560<screen>
561<userinput>make install</userinput>
562</screen>
563    This will install files into the directories that were specified
564    in <xref linkend="configure"/>. Make sure that you have appropriate
565    permissions to write into that area. Normally you need to do this
566    step as root. Alternatively, you can create the target
567    directories in advance and arrange for appropriate permissions to
568    be granted.
569   </para>
570
571   <para>
572    To install the documentation (HTML and man pages), enter:
573<screen>
574<userinput>make install-docs</userinput>
575</screen>
576   </para>
577
578   <para>
579    If you built the world above, type instead:
580<screen>
581<userinput>make install-world</userinput>
582</screen>
583    This also installs the documentation.
584   </para>
585
586   <para>
587    If you built the world without the documentation above, type instead:
588<screen>
589<userinput>make install-world-bin</userinput>
590</screen>
591   </para>
592
593   <para>
594    You can use <literal>make install-strip</literal> instead of
595    <literal>make install</literal> to strip the executable files and
596    libraries as they are installed.  This will save some space.  If
597    you built with debugging support, stripping will effectively
598    remove the debugging support, so it should only be done if
599    debugging is no longer needed.  <literal>install-strip</literal>
600    tries to do a reasonable job saving space, but it does not have
601    perfect knowledge of how to strip every unneeded byte from an
602    executable file, so if you want to save all the disk space you
603    possibly can, you will have to do manual work.
604   </para>
605
606   <para>
607    The standard installation provides all the header files needed for client
608    application development as well as for server-side program
609    development, such as custom functions or data types written in C.
610   </para>
611
612   <formalpara>
613    <title>Client-only installation:</title>
614    <para>
615     If you want to install only the client applications and
616     interface libraries, then you can use these commands:
617<screen>
618<userinput>make -C src/bin install</userinput>
619<userinput>make -C src/include install</userinput>
620<userinput>make -C src/interfaces install</userinput>
621<userinput>make -C doc install</userinput>
622</screen>
623    <filename>src/bin</filename> has a few binaries for server-only use,
624    but they are small.
625    </para>
626   </formalpara>
627  </step>
628  </procedure>
629
630  <formalpara>
631   <title>Uninstallation:</title>
632   <para>
633    To undo the installation use the command <command>make
634    uninstall</command>. However, this will not remove any created directories.
635   </para>
636  </formalpara>
637
638  <formalpara>
639   <title>Cleaning:</title>
640
641   <para>
642    After the installation you can free disk space by removing the built
643    files from the source tree with the command <command>make
644    clean</command>. This will preserve the files made by the <command>configure</command>
645    program, so that you can rebuild everything with <command>make</command>
646    later on. To reset the source tree to the state in which it was
647    distributed, use <command>make distclean</command>. If you are going to
648    build for several platforms within the same source tree you must do
649    this and re-configure for each platform.  (Alternatively, use
650    a separate build tree for each platform, so that the source tree
651    remains unmodified.)
652   </para>
653  </formalpara>
654
655  <para>
656   If you perform a build and then discover that your <command>configure</command>
657   options were wrong, or if you change anything that <command>configure</command>
658   investigates (for example, software upgrades), then it's a good
659   idea to do <command>make distclean</command> before reconfiguring and
660   rebuilding.  Without this, your changes in configuration choices
661   might not propagate everywhere they need to.
662  </para>

680   <sect3 id="configure-options-locations">
681    <title>Installation Locations</title>
682
683     <para>
684      These options control where <literal>make install</literal> will put
685      the files.  The <option>--prefix</option> option is sufficient for
686      most cases.  If you have special needs, you can customize the
687      installation subdirectories with the other options described in this
688      section.  Beware however that changing the relative locations of the
689      different subdirectories may render the installation non-relocatable,
690      meaning you won't be able to move it after installation.
691      (The <literal>man</literal> and <literal>doc</literal> locations are
692      not affected by this restriction.)  For relocatable installs, you
693      might want to use the <literal>--disable-rpath</literal> option
694      described later.
695     </para>

841     <note>
842      <para>
843       Care has been taken to make it possible to install
844       <productname>PostgreSQL</productname> into shared installation locations
845       (such as <filename>/usr/local/include</filename>) without
846       interfering with the namespace of the rest of the system. First,
847       the string <quote><literal>/postgresql</literal></quote> is
848       automatically appended to <varname>datadir</varname>,
849       <varname>sysconfdir</varname>, and <varname>docdir</varname>,
850       unless the fully expanded directory name already contains the
851       string <quote><literal>postgres</literal></quote> or
852       <quote><literal>pgsql</literal></quote>. For example, if you choose
853       <filename>/usr/local</filename> as prefix, the documentation will
854       be installed in <filename>/usr/local/doc/postgresql</filename>,
855       but if the prefix is <filename>/opt/postgres</filename>, then it
856       will be in <filename>/opt/postgres/doc</filename>. The public C
857       header files of the client interfaces are installed into
858       <varname>includedir</varname> and are namespace-clean. The
859       internal header files and the server header files are installed
860       into private directories under <varname>includedir</varname>. See
861       the documentation of each interface for information about how to
862       access its header files. Finally, a private subdirectory will
863       also be created, if appropriate, under <varname>libdir</varname>
864       for dynamically loadable modules.
865      </para>
866     </note>

3275  <sect2 id="install-post-shlibs">
3276   <title>Shared Libraries</title>
3277
3278   <indexterm>
3279    <primary>shared library</primary>
3280   </indexterm>
3281
3282   <para>
3283    On some systems with shared libraries
3284    you need to tell the system how to find the newly installed
3285    shared libraries.  The systems on which this is
3286    <emphasis>not</emphasis> necessary include
3287    <systemitem class="osname">FreeBSD</systemitem>,
3288    <systemitem class="osname">Linux</systemitem>,
3289    <systemitem class="osname">NetBSD</systemitem>, <systemitem
3290    class="osname">OpenBSD</systemitem>, and
3291    <systemitem class="osname">Solaris</systemitem>.
3292   </para>
3293
3294   <para>
3295    The method to set the shared library search path varies between
3296    platforms, but the most widely-used method is to set the
3297    environment variable <envar>LD_LIBRARY_PATH</envar> like so: In Bourne
3298    shells (<command>sh</command>, <command>ksh</command>, <command>bash</command>, <command>zsh</command>):
3299<programlisting>
3300LD_LIBRARY_PATH=/usr/local/pgsql/lib
3301export LD_LIBRARY_PATH
3302</programlisting>
3303    or in <command>csh</command> or <command>tcsh</command>:
3304<programlisting>
3305setenv LD_LIBRARY_PATH /usr/local/pgsql/lib
3306</programlisting>
3307    Replace <literal>/usr/local/pgsql/lib</literal> with whatever you set
3308    <option><literal>--libdir</literal></option> to in <xref linkend="configure"/>.
3309    You should put these commands into a shell start-up file such as
3310    <filename>/etc/profile</filename> or <filename>~/.bash_profile</filename>.  Some
3311    good information about the caveats associated with this method can
3312    be found at <ulink
3313    url="http://xahlee.info/UnixResource_dir/_/ldpath.html"></ulink>.
3314   </para>
3315
3316   <para>
3317    On some systems it might be preferable to set the environment
3318    variable <envar>LD_RUN_PATH</envar> <emphasis>before</emphasis>
3319    building.
3320   </para>
3321
3322   <para>
3323    On <systemitem class="osname">Cygwin</systemitem>, put the library
3324    directory in the <envar>PATH</envar> or move the
3325    <filename>.dll</filename> files into the <filename>bin</filename>
3326    directory.
3327   </para>
3328
3329   <para>
3330    If in doubt, refer to the manual pages of your system (perhaps
3331    <command>ld.so</command> or <command>rld</command>). If you later
3332    get a message like:
3333<screen>
3334psql: error in loading shared libraries
3335libpq.so.2.1: cannot open shared object file: No such file or directory
3336</screen>
3337    then this step was necessary.  Simply take care of it then.
3338   </para>
3339
3340   <para>
3341    <indexterm>
3342     <primary>ldconfig</primary>
3343    </indexterm>
3344    If you are on <systemitem class="osname">Linux</systemitem> and you have root
3345    access, you can run:
3346<programlisting>
3347/sbin/ldconfig /usr/local/pgsql/lib
3348</programlisting>
3349    (or equivalent directory) after installation to enable the
3350    run-time linker to find the shared libraries faster.  Refer to the
3351    manual page of <command>ldconfig</command> for more information.  On
3352    <systemitem class="osname">FreeBSD</systemitem>, <systemitem
3353    class="osname">NetBSD</systemitem>, and <systemitem
3354    class="osname">OpenBSD</systemitem> the command is:
3355<programlisting>
3356/sbin/ldconfig -m /usr/local/pgsql/lib
3357</programlisting>
3358    instead.  Other systems are not known to have an equivalent
3359    command.
3360   </para>

4025    <title>Special Considerations for 64-Bit Windows</title>
4026    <para>
4027     PostgreSQL will only build for the x64 architecture on 64-bit Windows.
4028    </para>
4029    <para>
4030     Mixing 32- and 64-bit versions in the same build tree is not supported.
4031     The build system will automatically detect if it's running in a 32- or
4032     64-bit environment, and build PostgreSQL accordingly. For this reason, it
4033     is important to start the correct command prompt before building.
4034    </para>
4035    <para>
4036     To use a server-side third party library such as <productname>Python</productname> or
4037     <productname>OpenSSL</productname>, this library <emphasis>must</emphasis> also be
4038     64-bit. There is no support for loading a 32-bit library in a 64-bit
4039     server. Several of the third party libraries that PostgreSQL supports may
4040     only be available in 32-bit versions, in which case they cannot be used with
4041     64-bit PostgreSQL.
