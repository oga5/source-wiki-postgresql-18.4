79 <refsect1>
80  <title>Description</title>
81
82  <para>
83   <command>COMMENT</command> stores, replaces, or removes the comment on a
84   database object.
85  </para>
86
87  <para>
88   Only one comment string is stored for each object.  Issuing a new
89   <command>COMMENT</command> command for the same object replaces the
90   existing comment.  Specifying <literal>NULL</literal> or an empty
91   string (<literal>''</literal>) removes the comment.  Comments are
92   automatically dropped when their object is dropped.
93  </para>
94
95  <para>
96   A <literal>SHARE UPDATE EXCLUSIVE</literal> lock is acquired on the
97   object to be commented.
98  </para>
99
100  <para>
101   For most kinds of object, only the object's owner can set the comment.
102   Roles don't have owners, so the rule for <literal>COMMENT ON ROLE</literal> is
103   that you must be superuser to comment on a superuser role, or have the
104   <literal>CREATEROLE</literal> privilege and have been granted
105   <literal>ADMIN OPTION</literal> on the target role.
106   Likewise, access methods don't have owners either; you must be superuser
107   to comment on an access method.
108   Of course, a superuser can comment on anything.
109  </para>
110
111  <para>
112    Comments can be viewed using <application>psql</application>'s
113    <command>\d</command> family of commands.
114    Other user interfaces to retrieve comments can be built atop
115    the same built-in functions that <application>psql</application> uses, namely
116    <function>obj_description</function>, <function>col_description</function>,
117    and <function>shobj_description</function>
118    (see <xref linkend="functions-info-comment-table"/>).
119  </para>
120 </refsect1>

289 <refsect1>
290  <title>Notes</title>
291
292  <para>
293   There is presently no security mechanism for viewing comments: any user
294   connected to a database can see all the comments for objects in
295   that database.  For shared objects such as
296   databases, roles, and tablespaces, comments are stored globally so any
297   user connected to any database in the cluster can see all the comments
298   for shared objects.  Therefore, don't put security-critical
299   information in comments.
300  </para>
301 </refsect1>
