36 <refsect1>
37  <title>Description</title>
38 <para>
39  <application>pgbench</application> is a simple program for running benchmark
40  tests on <productname>PostgreSQL</productname>.  It runs the same sequence of SQL
41  commands over and over, possibly in multiple concurrent database sessions,
42  and then calculates the average transaction rate (transactions per second).
43  By default, <application>pgbench</application> tests a scenario that is
44  loosely based on TPC-B, involving five <command>SELECT</command>,
45  <command>UPDATE</command>, and <command>INSERT</command> commands per transaction.
46  However, it is easy to test other cases by writing your own transaction
47  script files.
48 </para>
49
50 <para>
51  Typical output from <application>pgbench</application> looks like:
52
53<screen>
54transaction type: &lt;builtin: TPC-B (sort of)&gt;
55scaling factor: 10
56query mode: simple
57number of clients: 10
58number of threads: 1
59maximum number of tries: 1
60number of transactions per client: 1000
61number of transactions actually processed: 10000/10000
62number of failed transactions: 0 (0.000%)
63latency average = 11.013 ms
64latency stddev = 7.351 ms
65initial connection time = 45.758 ms
66tps = 896.967014 (without initial connection time)
67</screen>
68
69  The first seven lines report some of the most important parameter
70  settings.
71  The sixth line reports the maximum number of tries for transactions with
72  serialization or deadlock errors (see <xref linkend="failures-and-retries"/>
73  for more information).
74  The eighth line reports the number of transactions completed
75  and intended (the latter being just the product of number of clients
76  and number of transactions per client); these will be equal unless the run
77  failed before completion or some SQL command(s) failed.  (In
78  <option>-T</option> mode, only the actual number of transactions is printed.)
79  The next line reports the number of failed transactions due to
80  serialization or deadlock errors (see <xref linkend="failures-and-retries"/>
81  for more information).
82  The last line reports the number of transactions per second.
83 </para>
84
85  <para>
86   The default TPC-B-like transaction test requires specific tables to be
87   set up beforehand.  <application>pgbench</application> should be invoked with
88   the <option>-i</option> (initialize) option to create and populate these
89   tables.  (When you are testing a custom script, you don't need this
90   step, but will instead need to do whatever setup your test needs.)
91   Initialization looks like:
92
93<programlisting>
94pgbench -i <optional> <replaceable>other-options</replaceable> </optional> <replaceable>dbname</replaceable>
95</programlisting>
96
97   where <replaceable>dbname</replaceable> is the name of the already-created
98   database to test in.  (You may also need <option>-h</option>,
99   <option>-p</option>, and/or <option>-U</option> options to specify how to
100   connect to the database server.)
101  </para>
102
103  <caution>
104   <para>
105    <literal>pgbench -i</literal> creates four tables <structname>pgbench_accounts</structname>,
106    <structname>pgbench_branches</structname>, <structname>pgbench_history</structname>, and
107    <structname>pgbench_tellers</structname>,
108    destroying any existing tables of these names.
109    Be very careful to use another database if you have tables having these
110    names!
111   </para>
112  </caution>
113
114  <para>
115   At the default <quote>scale factor</quote> of 1, the tables initially
116   contain this many rows:
117<screen>
118table                   # of rows
119---------------------------------
120pgbench_branches        1
121pgbench_tellers         10
122pgbench_accounts        100000
123pgbench_history         0
124</screen>
125   You can (and, for most purposes, probably should) increase the number
126   of rows by using the <option>-s</option> (scale factor) option.  The
127   <option>-F</option> (fillfactor) option might also be used at this point.
128  </para>
129
130  <para>
131   Once you have done the necessary setup, you can run your benchmark
132   with a command that doesn't include <option>-i</option>, that is
133
134<programlisting>
135pgbench <optional> <replaceable>options</replaceable> </optional> <replaceable>dbname</replaceable>
136</programlisting>
137
138   In nearly all cases, you'll need some options to make a useful test.
139   The most important options are <option>-c</option> (number of clients),
140   <option>-t</option> (number of transactions), <option>-T</option> (time limit),
141   and <option>-f</option> (specify a custom script file).
142   See below for a full list.
143  </para>
144 </refsect1>

218           </para>
219          </listitem>
220         </varlistentry>

222          <term><literal>g</literal> or <literal>G</literal> (Generate data, client-side or server-side)</term>
223          <listitem>
224           <para>
225            Generate data and load it into the standard tables,
226            replacing any data already present.
227           </para>
228           <para>
229            With <literal>g</literal> (client-side data generation),
230            data is generated in <command>pgbench</command> client and then
231            sent to the server. This uses the client/server bandwidth
232            extensively through a <command>COPY</command>.
233            <command>pgbench</command> uses the <option>FREEZE</option> option

235            or later of <productname>PostgreSQL</productname> to speed up
236            subsequent <command>VACUUM</command>.

238            print one message every 100,000 rows while generating data for all
239            tables.
240           </para>
241           <para>

243            only small queries are sent from the <command>pgbench</command>
244            client and then data is actually generated in the server.
245            No significant bandwidth is required for this variant, but
246            the server will do more work.
247            Using <literal>G</literal> causes logging not to print any progress
248            message while generating data.
249           </para>

252            generation (equivalent to <literal>g</literal>).
253           </para>

415

418    benchmarking arguments:
419
420    <variablelist>

425       <para>

523      <listitem>
524       <para>
525        Transactions that last more than <replaceable>limit</replaceable> milliseconds
526        are counted and reported separately, as <firstterm>late</firstterm>.
527       </para>

530        lag behind schedule by more than <replaceable>limit</replaceable> ms, and thus

534       </para>

540        and not their number, use <literal>--max-tries=0</literal>. By
541        default, the option <option>--max-tries</option> is set to 1 and
542        transactions with serialization/deadlock errors are not retried. See

544        retrying such transactions.
545       </para>
546       </listitem>

642      <listitem>
643       <para>
644        Execute transactions targeting the specified rate instead of running
645        as fast as possible (the default).  The rate is given in transactions
646        per second.  If the targeted rate is above the maximum possible rate,
647        the rate limit won't impact the results.
648       </para>
649       <para>
650        The rate is targeted by starting transactions along a
651        Poisson-distributed schedule time line.  The expected start time
652        schedule moves forward based on when the client first started, not
653        when the previous transaction ended.  That approach means that when
654        transactions go past their original scheduled end time, it is
655        possible for later ones to catch up again.
656       </para>
657       <para>
658        When throttling is active, the transaction latency reported at the
659        end of the run is calculated from the scheduled start times, so it
660        includes the time each transaction had to wait for the previous
661        transaction to finish. The wait time is called the schedule lag time,
662        and its average and maximum are also reported separately. The
663        transaction latency with respect to the actual transaction start time,
664        i.e., the time spent executing the transaction in the database, can be
665        computed by subtracting the schedule lag time from the reported
666        latency.
667       </para>

671        a transaction can lag behind so much that it is already over the
672        latency limit when the previous transaction ends, because the latency
673        is calculated from the scheduled start time. Such transactions are
674        not sent to the server, but are skipped altogether and counted
675        separately.
676       </para>

681        clients and threads. When the average transaction execution time is

1043  <title>Notes</title>
1044
1045 <refsect2 id="transactions-and-scripts" xreflabel="What Is the &quot;Transaction&quot; Actually Performed in pgbench?">
1046  <title>What Is the <quote>Transaction</quote> Actually Performed in <application>pgbench</application>?</title>
1047
1048  <para>
1049   <application>pgbench</application> executes test scripts chosen randomly
1050   from a specified list.
1051   The scripts may include built-in scripts specified with <option>-b</option>
1052   and user-provided scripts specified with <option>-f</option>.
1053   Each script may be given a relative weight specified after an
1054   <literal>@</literal> so as to change its selection probability.
1055   The default weight is <literal>1</literal>.
1056   Scripts with a weight of <literal>0</literal> are ignored.
1057 </para>
1058
1059  <para>
1060   The default built-in transaction script (also invoked with <option>-b tpcb-like</option>)
1061   issues seven commands per transaction over randomly chosen <literal>aid</literal>,
1062   <literal>tid</literal>, <literal>bid</literal> and <literal>delta</literal>.
1063   The scenario is inspired by the TPC-B benchmark, but is not actually TPC-B,
1064   hence the name.
1065  </para>
1066
1067  <orderedlist>
1068   <listitem><para><literal>BEGIN;</literal></para></listitem>
1069   <listitem><para><literal>UPDATE pgbench_accounts SET abalance = abalance + :delta WHERE aid = :aid;</literal></para></listitem>
1070   <listitem><para><literal>SELECT abalance FROM pgbench_accounts WHERE aid = :aid;</literal></para></listitem>
1071   <listitem><para><literal>UPDATE pgbench_tellers SET tbalance = tbalance + :delta WHERE tid = :tid;</literal></para></listitem>
1072   <listitem><para><literal>UPDATE pgbench_branches SET bbalance = bbalance + :delta WHERE bid = :bid;</literal></para></listitem>
1073   <listitem><para><literal>INSERT INTO pgbench_history (tid, bid, aid, delta, mtime) VALUES (:tid, :bid, :aid, :delta, CURRENT_TIMESTAMP);</literal></para></listitem>
1074   <listitem><para><literal>END;</literal></para></listitem>
1075  </orderedlist>
1076
1077  <para>
1078   If you select the <literal>simple-update</literal> built-in (also <option>-N</option>),
1079   steps 4 and 5 aren't included in the transaction.
1080   This will avoid update contention on these tables, but
1081   it makes the test case even less like TPC-B.
1082  </para>
1083
1084  <para>
1085   If you select the <literal>select-only</literal> built-in (also <option>-S</option>),
1086   only the <command>SELECT</command> is issued.
1087  </para>
1088 </refsect2>

1090 <refsect2>
1091  <title>Custom Scripts</title>
1092
1093  <para>
1094   <application>pgbench</application> has support for running custom
1095   benchmark scenarios by replacing the default transaction script
1096   (described above) with a transaction script read from a file
1097   (<option>-f</option> option).  In this case a <quote>transaction</quote>
1098   counts as one execution of a script file.
1099  </para>
1100
1101  <para>
1102   A script file contains one or more SQL commands terminated by
1103   semicolons.  Empty lines and lines beginning with
1104   <literal>--</literal> are ignored.  Script files can also contain
1105   <quote>meta commands</quote>, which are interpreted by <application>pgbench</application>
1106   itself, as described below.
1107  </para>
1108
1109  <note>
1110   <para>
1111    Before <productname>PostgreSQL</productname> 9.6, SQL commands in script files
1112    were terminated by newlines, and so they could not be continued across
1113    lines.  Now a semicolon is <emphasis>required</emphasis> to separate consecutive
1114    SQL commands (though an SQL command does not need one if it is followed
1115    by a meta command).  If you need to create a script file that works with
1116    both old and new versions of <application>pgbench</application>, be sure to write
1117    each SQL command on a single line ending with a semicolon.
1118   </para>
1119   <para>
1120    It is assumed that <application>pgbench</application> scripts do not contain
1121    incomplete blocks of SQL
1122    transactions. If at runtime the client reaches the end of the script without
1123    completing the last transaction block, it will be aborted.
1124   </para>
1125  </note>
1126
1127  <para>
1128   There is a simple variable-substitution facility for script files.
1129   Variable names must consist of letters (including non-Latin letters),
1130   digits, and underscores, with the first character not being a digit.
1131   Variables can be set by the command-line <option>-D</option> option,
1132   explained above, or by the meta commands explained below.
1133   In addition to any variables preset by <option>-D</option> command-line options,
1134   there are a few variables that are preset automatically, listed in
1135   <xref linkend="pgbench-automatic-variables"/>. A value specified for these
1136   variables using <option>-D</option> takes precedence over the automatic presets.
1137   Once set, a variable's
1138   value can be inserted into an SQL command by writing
1139   <literal>:</literal><replaceable>variablename</replaceable>.  When running more than
1140   one client session, each session has its own set of variables.
1141   <application>pgbench</application> supports up to 255 variable uses in one
1142   statement.
1143  </para>
1144
1145   <table id="pgbench-automatic-variables">
1146    <title>pgbench Automatic Variables</title>
1147    <tgroup cols="2">
1148     <colspec colname="col1" colwidth="1*"/>
1149     <colspec colname="col2" colwidth="2*"/>
1150     <thead>
1151      <row>
1152       <entry>Variable</entry>

1154      </row>

1161      </row>

1166      </row>

1181  <para>
1182   Script file meta commands begin with a backslash (<literal>\</literal>) and
1183   normally extend to the end of the line, although they can be continued
1184   to additional lines by writing backslash-return.
1185   Arguments to a meta command are separated by white space.
1186   These meta commands are supported:
1187  </para>
1188
1189  <variablelist>
1190   <varlistentry id="pgbench-metacommand-gset">
1191    <term>
1192     <literal>\gset [<replaceable>prefix</replaceable>]</literal>
1193     <literal>\aset [<replaceable>prefix</replaceable>]</literal>
1194    </term>
1195
1196    <listitem>
1197     <para>
1198      These commands may be used to end SQL queries, taking the place of the
1199      terminating semicolon (<literal>;</literal>).
1200     </para>
1201
1202     <para>
1203      When the <literal>\gset</literal> command is used, the preceding SQL query is
1204      expected to return one row, the columns of which are stored into variables
1205      named after column names, and prefixed with <replaceable>prefix</replaceable>
1206      if provided.
1207     </para>
1208
1209     <para>
1210      When the <literal>\aset</literal> command is used, all combined SQL queries
1211      (separated by <literal>\;</literal>) have their columns stored into variables
1212      named after column names, and prefixed with <replaceable>prefix</replaceable>
1213      if provided. If a query returns no row, no assignment is made and the variable
1214      can be tested for existence to detect this. If a query returns more than one
1215      row, the last value is kept.
1216     </para>
1217
1218     <para>
1219      <literal>\gset</literal> and <literal>\aset</literal> cannot be used in
1220      pipeline mode, since the query results are not yet available by the time
1221      the commands would need them.
1222     </para>
1223
1224     <para>
1225      The following example puts the final account balance from the first query
1226      into variable <replaceable>abalance</replaceable>, and fills variables
1227      <replaceable>p_two</replaceable> and <replaceable>p_three</replaceable>
1228      with integers from the third query.
1229      The result of the second query is discarded.
1230      The result of the two last combined queries are stored in variables
1231      <replaceable>four</replaceable> and <replaceable>five</replaceable>.
1232<programlisting>
1233UPDATE pgbench_accounts
1234  SET abalance = abalance + :delta
1235  WHERE aid = :aid
1236  RETURNING abalance \gset
1237-- compound of two queries
1238SELECT 1 \;
1239SELECT 2 AS two, 3 AS three \gset p_

2105   <para>
2106    The <literal>random</literal> function generates values using a uniform
2107    distribution, that is all the values are drawn within the specified
2108    range with equal probability. The <literal>random_exponential</literal>,
2109    <literal>random_gaussian</literal> and <literal>random_zipfian</literal>
2110    functions require an additional double parameter which determines the precise
2111    shape of the distribution.
2112   </para>
2113
2114   <itemizedlist>
2115    <listitem>
2116     <para>
2117      For an exponential distribution, <replaceable>parameter</replaceable>
2118      controls the distribution by truncating a quickly-decreasing
2119      exponential distribution at <replaceable>parameter</replaceable>, and then
2120      projecting onto integers between the bounds.
2121      To be precise, with
2122<literallayout>
2123f(x) = exp(-parameter * (x - min) / (max - min + 1)) / (1 - exp(-parameter))
2124</literallayout>
2125      Then value <replaceable>i</replaceable> between <replaceable>min</replaceable> and
2126      <replaceable>max</replaceable> inclusive is drawn with probability:
2127      <literal>f(i) - f(i + 1)</literal>.
2128     </para>
2129
2130     <para>
2131      Intuitively, the larger the <replaceable>parameter</replaceable>, the more
2132      frequently values close to <replaceable>min</replaceable> are accessed, and the
2133      less frequently values close to <replaceable>max</replaceable> are accessed.
2134      The closer to 0 <replaceable>parameter</replaceable> is, the flatter (more
2135      uniform) the access distribution.
2136      A crude approximation of the distribution is that the most frequent 1%
2137      values in the range, close to <replaceable>min</replaceable>, are drawn
2138      <replaceable>parameter</replaceable>% of the time.
2139      The <replaceable>parameter</replaceable> value must be strictly positive.
2140     </para>
2141    </listitem>
2142
2143    <listitem>
2144     <para>
2145      For a Gaussian distribution, the interval is mapped onto a standard
2146      normal distribution (the classical bell-shaped Gaussian curve) truncated
2147      at <literal>-parameter</literal> on the left and <literal>+parameter</literal>
2148      on the right.
2149      Values in the middle of the interval are more likely to be drawn.
2150      To be precise, if <literal>PHI(x)</literal> is the cumulative distribution
2151      function of the standard normal distribution, with mean <literal>mu</literal>
2152      defined as <literal>(max + min) / 2.0</literal>, with
2153<literallayout>
2154f(x) = PHI(2.0 * parameter * (x - mu) / (max - min + 1)) /
2155       (2.0 * PHI(parameter) - 1)
2156</literallayout>
2157      then value <replaceable>i</replaceable> between <replaceable>min</replaceable> and
2158      <replaceable>max</replaceable> inclusive is drawn with probability:
2159      <literal>f(i + 0.5) - f(i - 0.5)</literal>.
2160      Intuitively, the larger the <replaceable>parameter</replaceable>, the more
2161      frequently values close to the middle of the interval are drawn, and the
2162      less frequently values close to the <replaceable>min</replaceable> and
2163      <replaceable>max</replaceable> bounds. About 67% of values are drawn from the
2164      middle <literal>1.0 / parameter</literal>, that is a relative
2165      <literal>0.5 / parameter</literal> around the mean, and 95% in the middle
2166      <literal>2.0 / parameter</literal>, that is a relative
2167      <literal>1.0 / parameter</literal> around the mean; for instance, if
2168      <replaceable>parameter</replaceable> is 4.0, 67% of values are drawn from the
2169      middle quarter (1.0 / 4.0) of the interval (i.e., from
2170      <literal>3.0 / 8.0</literal> to <literal>5.0 / 8.0</literal>) and 95% from
2171      the middle half (<literal>2.0 / 4.0</literal>) of the interval (second and third
2172      quartiles). The minimum allowed <replaceable>parameter</replaceable>
2173      value is 2.0.
2174     </para>
2175    </listitem>
2176    <listitem>
2177     <para>
2178      <literal>random_zipfian</literal> generates a bounded Zipfian
2179      distribution.
2180      <replaceable>parameter</replaceable> defines how skewed the distribution
2181      is. The larger the <replaceable>parameter</replaceable>, the more
2182      frequently values closer to the beginning of the interval are drawn.
2183      The distribution is such that, assuming the range starts from 1,
2184      the ratio of the probability of drawing <replaceable>k</replaceable>
2185      versus drawing <replaceable>k+1</replaceable> is
2186      <literal>((<replaceable>k</replaceable>+1)/<replaceable>k</replaceable>)**<replaceable>parameter</replaceable></literal>.
2187      For example, <literal>random_zipfian(1, ..., 2.5)</literal> produces
2188      the value <literal>1</literal> about <literal>(2/1)**2.5 =
2189      5.66</literal> times more frequently than <literal>2</literal>, which
2190      itself is produced <literal>(3/2)**2.5 = 2.76</literal> times more
2191      frequently than <literal>3</literal>, and so on.
2192     </para>
2193     <para>
2194      <application>pgbench</application>'s implementation is based on
2195      "Non-Uniform Random Variate Generation", Luc Devroye, p. 550-551,
2196      Springer 1986.  Due to limitations of that algorithm,
2197      the <replaceable>parameter</replaceable> value is restricted to
2198      the range [1.001, 1000].
2199     </para>
2200    </listitem>
2201   </itemizedlist>

2203   <note>
2204    <para>
2205      When designing a benchmark which selects rows non-uniformly, be aware
2206      that the rows chosen may be correlated with other data such as IDs from
2207      a sequence or the physical row ordering, which may skew performance
2208      measurements.
2209    </para>
2210    <para>
2211      To avoid this, you may wish to use the <function>permute</function>
2212      function, or some other additional step with similar effect, to shuffle
2213      the selected rows and remove such correlations.
2214    </para>
2215   </note>

2217  <para>
2218    Hash functions <literal>hash</literal>, <literal>hash_murmur2</literal> and
2219    <literal>hash_fnv1a</literal> accept an input value and an optional seed parameter.
2220    In case the seed isn't provided the value of <literal>:default_seed</literal>
2221    is used, which is initialized randomly unless set by the command-line
2222    <literal>-D</literal> option.
2223  </para>
2224
2225  <para>
2226    <literal>permute</literal> accepts an input value, a size, and an optional
2227    seed parameter.  It generates a pseudorandom permutation of integers in
2228    the range <literal>[0, size)</literal>, and returns the index of the input
2229    value in the permuted values.  The permutation chosen is parameterized by
2230    the seed, which defaults to <literal>:default_seed</literal>, if not
2231    specified.  Unlike the hash functions, <literal>permute</literal> ensures
2232    that there are no collisions or holes in the output values.  Input values
2233    outside the interval are interpreted modulo the size.  The function raises
2234    an error if the size is not positive.  <function>permute</function> can be
2235    used to scatter the distribution of non-uniform random functions such as
2236    <literal>random_zipfian</literal> or <literal>random_exponential</literal>
2237    so that values drawn more often are not trivially correlated.  For
2238    instance, the following <application>pgbench</application> script
2239    simulates a possible real world workload typical for social media and
2240    blogging platforms where a few accounts generate excessive load:
2241
2242<programlisting>
2243\set size 1000000
2244\set r random_zipfian(1, :size, 1.07)
2245\set k 1 + permute(:r, :size)
2246</programlisting>
2247
2248    In some cases several distinct distributions are needed which don't correlate
2249    with each other and this is when the optional seed parameter comes in handy:
2250
2251<programlisting>
2252\set k1 1 + permute(:r, :size, :default_seed + 123)
2253\set k2 1 + permute(:r, :size, :default_seed + 321)
2254</programlisting>
2255
2256    A similar behavior can also be approximated with <function>hash</function>:
2257
2258<programlisting>
2259\set size 1000000
2260\set r random_zipfian(1, 100 * :size, 1.07)
2261\set k 1 + abs(hash(:r)) % :size
2262</programlisting>
2263
2264    However, since <function>hash</function> generates collisions, some values
2265    will not be reachable and others will be more frequent than expected from
2266    the original distribution.
2267  </para>

2280SELECT abalance FROM pgbench_accounts WHERE aid = :aid;
2281UPDATE pgbench_tellers SET tbalance = tbalance + :delta WHERE tid = :tid;

2283INSERT INTO pgbench_history (tid, bid, aid, delta, mtime) VALUES (:tid, :bid, :aid, :delta, CURRENT_TIMESTAMP);
2284END;
2285</programlisting>

2783 <refsect2 id="failures-and-retries" xreflabel="Failures and Serialization/Deadlock Retries">
2784  <title>Failures and Serialization/Deadlock Retries</title>
2785
2786  <para>
2787   When executing <application>pgbench</application>, there are three main types
2788   of errors:
2789   <itemizedlist>
2790     <listitem>
2791       <para>
2792         Errors of the main program. They are the most serious and always result
2793         in an immediate exit from <application>pgbench</application> with the
2794         corresponding error message. They include:
2795         <itemizedlist>
2796           <listitem>
2797             <para>
2798               errors at the beginning of <application>pgbench</application>
2799               (e.g. an invalid option value);
2800             </para>
2801           </listitem>
2802           <listitem>
2803             <para>
2804               errors in the initialization mode (e.g. the query to create
2805               tables for built-in scripts fails);
2806             </para>
2807           </listitem>
2808           <listitem>
2809             <para>
2810               errors before starting threads (e.g. could not connect to the
2811               database server, syntax error in the meta command, thread
2812               creation failure);
2813             </para>
2814           </listitem>
2815           <listitem>
2816             <para>
2817               internal <application>pgbench</application> errors (which are
2818               supposed to never occur...).
2819             </para>
2820           </listitem>
2821         </itemizedlist></para>
2822     </listitem>
2823     <listitem>
2824       <para>
2825         Errors when the thread manages its clients (e.g. the client could not
2826         start a connection to the database server / the socket for connecting
2827         the client to the database server has become invalid). In such cases
2828         all clients of this thread stop while other threads continue to work.
2829         However, <option>--exit-on-abort</option> is specified, all of the
2830         threads stop immediately in this case.
2831       </para>
2832     </listitem>
2833     <listitem>
2834       <para>
2835         Direct client errors. They lead to immediate exit from
2836         <application>pgbench</application> with the corresponding error message
2837         in the case of an internal <application>pgbench</application>
2838         error (which are supposed to never occur...) or when
2839         <option>--exit-on-abort</option> is specified. Otherwise in the worst
2840         case they only lead to the abortion of the failed client while other
2841         clients continue their run (but some client errors are handled without
2842         an abortion of the client and reported separately, see below). Later in
2843         this section it is assumed that the discussed errors are only the
2844         direct client errors and they are not internal
2845         <application>pgbench</application> errors.
2846       </para>
2847     </listitem>
2848   </itemizedlist>
2849  </para>
2850
2851  <para>
2852   A client's run is aborted in case of a serious error; for example, the
2853   connection with the database server was lost or the end of script was reached
2854   without completing the last transaction. In addition, if execution of an SQL
2855   or meta command fails for reasons other than serialization or deadlock errors,
2856   the client is aborted. Otherwise, if an SQL command fails with serialization or
2857   deadlock errors, the client is not aborted. In such cases, the current
2858   transaction is rolled back, which also includes setting the client variables
2859   as they were before the run of this transaction (it is assumed that one
2860   transaction script contains only one transaction; see
2861   <xref linkend="transactions-and-scripts"/> for more information).
2862   Transactions with serialization or deadlock errors are repeated after
2863   rollbacks until they complete successfully or reach the maximum
2864   number of tries (specified by the <option>--max-tries</option> option) / the maximum
2865   time of retries (specified by the <option>--latency-limit</option> option) / the end
2866   of benchmark (specified by the <option>--time</option> option). If
2867   the last trial run fails, this transaction will be reported as failed but
2868   the client is not aborted and continues to work.
2869  </para>
2870
2871  <note>
2872   <para>
2873    Without specifying the <option>--max-tries</option> option, a transaction will
2874    never be retried after a serialization or deadlock error because its default
2875    value is 1. Use an unlimited number of tries (<literal>--max-tries=0</literal>)
2876    and the <option>--latency-limit</option> option to limit only the maximum time
2877    of tries. You can also use the <option>--time</option> option to limit the
2878    benchmark duration under an unlimited number of tries.
2879   </para>
2880   <para>
2881    Be careful when repeating scripts that contain multiple transactions: the
2882    script is always retried completely, so successful transactions can be
2883    performed several times.
2884   </para>
2885   <para>
2886    Be careful when repeating transactions with shell commands. Unlike the
2887    results of SQL commands, the results of shell commands are not rolled back,
2888    except for the variable value of the <command>\setshell</command> command.
2889   </para>
2890  </note>
2891
2892  <para>
2893   The latency of a successful transaction includes the entire time of
2894   transaction execution with rollbacks and retries. The latency is measured
2895   only for successful transactions and commands but not for failed transactions
2896   or commands.
2897  </para>
2898
2899  <para>
2900   The main report contains the number of failed transactions. If the
2901   <option>--max-tries</option> option is not equal to 1, the main report also
2902   contains statistics related to retries: the total number of retried
2903   transactions and total number of retries. The per-script report inherits all
2904   these fields from the main report. The per-statement report displays retry
2905   statistics only if the <option>--max-tries</option> option is not equal to 1.
2906  </para>
2907
2908  <para>
2909   If you want to group failures by basic types in per-transaction and
2910   aggregation logs, as well as in the main and per-script reports, use the
2911   <option>--failures-detailed</option> option. If you also want to distinguish
2912   all errors and failures (errors without retrying) by type including which
2913   limit for retries was exceeded and how much it was exceeded by for the
2914   serialization/deadlock failures, use the <option>--verbose-errors</option>
2915   option.
2916  </para>

2935 <refsect2>
2936  <title>Good Practices</title>
2937
2938  <para>
2939   It is very easy to use <application>pgbench</application> to produce completely
2940   meaningless numbers.  Here are some guidelines to help you get useful
2941   results.
2942  </para>
2943
2944  <para>
2945   In the first place, <emphasis>never</emphasis> believe any test that runs
2946   for only a few seconds.  Use the <option>-t</option> or <option>-T</option> option
2947   to make the run last at least a few minutes, so as to average out noise.
2948   In some cases you could need hours to get numbers that are reproducible.
2949   It's a good idea to try the test run a few times, to find out if your
2950   numbers are reproducible or not.
2951  </para>
2952
2953  <para>
2954   For the default TPC-B-like test scenario, the initialization scale factor
2955   (<option>-s</option>) should be at least as large as the largest number of
2956   clients you intend to test (<option>-c</option>); else you'll mostly be
2957   measuring update contention.  There are only <option>-s</option> rows in
2958   the <structname>pgbench_branches</structname> table, and every transaction wants to
2959   update one of them, so <option>-c</option> values in excess of <option>-s</option>
2960   will undoubtedly result in lots of transactions blocked waiting for
2961   other transactions.
2962  </para>
2963
2964  <para>
2965   The default test scenario is also quite sensitive to how long it's been
2966   since the tables were initialized: accumulation of dead rows and dead space
2967   in the tables changes the results.  To understand the results you must keep
2968   track of the total number of updates and when vacuuming happens.  If
2969   autovacuum is enabled it can result in unpredictable changes in measured
2970   performance.
2971  </para>
2972
2973  <para>
2974   A limitation of <application>pgbench</application> is that it can itself become
2975   the bottleneck when trying to test a large number of client sessions.
2976   This can be alleviated by running <application>pgbench</application> on a different
2977   machine from the database server, although low network latency will be
2978   essential.  It might even be useful to run several <application>pgbench</application>
2979   instances concurrently, on several client machines, against the same
2980   database server.
2981  </para>
