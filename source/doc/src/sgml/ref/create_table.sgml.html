117 <refsect1 id="sql-createtable-description">
118  <title>Description</title>
119
120  <para>
121   <command>CREATE TABLE</command> will create a new, initially empty table
122   in the current database. The table will be owned by the user issuing the
123   command.
124  </para>
125
126  <para>
127   If a schema name is given (for example, <literal>CREATE TABLE
128   myschema.mytable ...</literal>) then the table is created in the specified
129   schema.  Otherwise it is created in the current schema.  Temporary
130   tables exist in a special schema, so a schema name cannot be given
131   when creating a temporary table.  The name of the table must be
132   distinct from the name of any other relation (table, sequence, index, view,
133   materialized view, or foreign table) in the same schema.
134  </para>
135
136  <para>
137   <command>CREATE TABLE</command> also automatically creates a data
138   type that represents the composite type corresponding
139   to one row of the table.  Therefore, tables cannot have the same
140   name as any existing data type in the same schema.
141  </para>
142
143  <para>
144   The optional constraint clauses specify constraints (tests) that
145   new or updated rows must satisfy for an insert or update operation
146   to succeed.  A constraint is an SQL object that helps define the
147   set of valid values in the table in various ways.
148  </para>
149
150  <para>
151   There are two ways to define constraints: table constraints and
152   column constraints.  A column constraint is defined as part of a
153   column definition.  A table constraint definition is not tied to a
154   particular column, and it can encompass more than one column.
155   Every column constraint can also be written as a table constraint;
156   a column constraint is only a notational convenience for use when the
157   constraint only affects one column.
158  </para>
159
160  <para>
161   To be able to create a table, you must have <literal>USAGE</literal>
162   privilege on all column types or the type in the <literal>OF</literal>
163   clause, respectively.
164  </para>

172   <varlistentry id="sql-createtable-temporary">
173    <term><literal>TEMPORARY</literal> or <literal>TEMP</literal></term>
174    <listitem>
175     <para>
176      If specified, the table is created as a temporary table.
177      Temporary tables are automatically dropped at the end of a
178      session, or optionally at the end of the current transaction
179      (see <literal>ON COMMIT</literal> below).  The default
180      search_path includes the temporary schema first and so identically
181      named existing permanent tables are not chosen for new plans
182      while the temporary table exists, unless they are referenced
183      with schema-qualified names. Any indexes created on a temporary
184      table are automatically temporary as well.
185     </para>
186
187     <para>
188      The <link linkend="autovacuum">autovacuum daemon</link> cannot
189      access and therefore cannot vacuum or analyze temporary tables.
190      For this reason, appropriate vacuum and analyze operations should be
191      performed via session SQL commands.  For example, if a temporary
192      table is going to be used in complex queries, it is wise to run
193      <command>ANALYZE</command> on the temporary table after it is populated.
194     </para>
195
196     <para>
197      Optionally, <literal>GLOBAL</literal> or <literal>LOCAL</literal>
198      can be written before <literal>TEMPORARY</literal> or <literal>TEMP</literal>.
199      This presently makes no difference in <productname>PostgreSQL</productname>
200      and is deprecated; see
201      <xref linkend="sql-createtable-compatibility"/> below.
202     </para>
203    </listitem>
204   </varlistentry>

206   <varlistentry id="sql-createtable-unlogged">
207    <term><literal>UNLOGGED</literal></term>
208    <listitem>
209     <para>
210      If specified, the table is created as an unlogged table.  Data written
211      to unlogged tables is not written to the write-ahead log (see <xref
212      linkend="wal"/>), which makes them considerably faster than ordinary
213      tables.  However, they are not crash-safe: an unlogged table is
214      automatically truncated after a crash or unclean shutdown.  The contents
215      of an unlogged table are also not replicated to standby servers.
216      Any indexes created on an unlogged table are automatically unlogged as
217      well.
218     </para>
219
220     <para>
221      If this is specified, any sequences created together with the unlogged
222      table (for identity or serial columns) are also created as unlogged.
223     </para>
224
225     <para>
226      This form is not supported for partitioned tables.
227     </para>
228    </listitem>

305   <varlistentry id="sql-createtable-parms-storage">
306    <term>
307     <literal>STORAGE { PLAIN | EXTERNAL | EXTENDED | MAIN | DEFAULT }</literal>
308     <indexterm>
309      <primary>TOAST</primary>
310      <secondary>per-column storage settings</secondary>
311     </indexterm>
312    </term>
313    <listitem>
314     <para>
315      This form sets the storage mode for the column. This controls whether this
316      column is held inline or in a secondary <acronym>TOAST</acronym> table,
317      and whether the data should be compressed or not. <literal>PLAIN</literal>
318      must be used for fixed-length values such as <type>integer</type> and is
319      inline, uncompressed. <literal>MAIN</literal> is for inline, compressible
320      data. <literal>EXTERNAL</literal> is for external, uncompressed data, and
321      <literal>EXTENDED</literal> is for external, compressed data.
322      Writing <literal>DEFAULT</literal> sets the storage mode to the default
323      mode for the column's data type.  <literal>EXTENDED</literal> is the
324      default for most data types that support non-<literal>PLAIN</literal>
325      storage.
326      Use of <literal>EXTERNAL</literal> will make substring operations on
327      very large <type>text</type> and <type>bytea</type> values run faster,
328      at the penalty of increased storage space.
329      See <xref linkend="storage-toast"/> for more information.
330     </para>
331    </listitem>

335    <term><literal>COMPRESSION <replaceable class="parameter">compression_method</replaceable></literal></term>
336    <listitem>
337     <para>
338      The <literal>COMPRESSION</literal> clause sets the compression method
339      for the column.  Compression is supported only for variable-width data
340      types, and is used only when the column's storage mode
341      is <literal>main</literal> or <literal>extended</literal>.
342      (See <xref linkend="sql-altertable"/> for information on
343      column storage modes.) Setting this property for a partitioned table
344      has no direct effect, because such tables have no storage of their own,
345      but the configured value will be inherited by newly-created partitions.
346      The supported compression methods are <literal>pglz</literal> and
347      <literal>lz4</literal>.  (<literal>lz4</literal> is available only if
348      <option>--with-lz4</option> was used when building
349      <productname>PostgreSQL</productname>.)  In addition,
350      <replaceable class="parameter">compression_method</replaceable>
351      can be <literal>default</literal> to explicitly specify the default
352      behavior, which is to consult the
353      <xref linkend="guc-default-toast-compression"/> setting at the time of
354      data insertion to determine the method to use.
355     </para>
356    </listitem>

359   <varlistentry id="sql-createtable-parms-inherits">
360    <term><literal>INHERITS ( <replaceable>parent_table</replaceable> [, ... ] )</literal></term>
361    <listitem>
362     <para>
363      The optional <literal>INHERITS</literal> clause specifies a list of
364      tables from which the new table automatically inherits all
365      columns.  Parent tables can be plain tables or foreign tables.
366     </para>
367
368     <para>
369      Use of <literal>INHERITS</literal> creates a persistent relationship
370      between the new child table and its parent table(s).  Schema
371      modifications to the parent(s) normally propagate to children
372      as well, and by default the data of the child table is included in
373      scans of the parent(s).
374     </para>
375
376     <para>
377      If the same column name exists in more than one parent
378      table, an error is reported unless the data types of the columns
379      match in each of the parent tables.  If there is no conflict,
380      then the duplicate columns are merged to form a single column in
381      the new table.  If the column name list of the new table
382      contains a column name that is also inherited, the data type must
383      likewise match the inherited column(s), and the column
384      definitions are merged into one.  If the
385      new table explicitly specifies a default value for the column,
386      this default overrides any defaults from inherited declarations
387      of the column.  Otherwise, any parents that specify default
388      values for the column must all specify the same default, or an
389      error will be reported.
390     </para>
391
392     <para>
393      <literal>CHECK</literal> constraints are merged in essentially the same way as
394      columns: if multiple parent tables and/or the new table definition
395      contain identically-named <literal>CHECK</literal> constraints, these
396      constraints must all have the same check expression, or an error will be
397      reported.  Constraints having the same name and expression will
398      be merged into one copy.  A constraint marked <literal>NO INHERIT</literal> in a
399      parent will not be considered.  Notice that an unnamed <literal>CHECK</literal>
400      constraint in the new table will never be merged, since a unique name
401      will always be chosen for it.
402     </para>
403
404     <para>
405      Column <literal>STORAGE</literal> settings are also copied from parent tables.
406     </para>
407
408     <para>
409      If a column in the parent table is an identity column, that property is
410      not inherited.  A column in the child table can be declared identity
411      column if desired.
412     </para>
413    </listitem>
414   </varlistentry>

416   <varlistentry id="sql-createtable-parms-partition-by">
417    <term><literal>PARTITION BY { RANGE | LIST | HASH } ( { <replaceable class="parameter">column_name</replaceable> | ( <replaceable class="parameter">expression</replaceable> ) } [ <replaceable class="parameter">opclass</replaceable> ] [, ...] ) </literal></term>
418    <listitem>
419     <para>
420      The optional <literal>PARTITION BY</literal> clause specifies a strategy
421      of partitioning the table.  The table thus created is called a
422      <firstterm>partitioned</firstterm> table.  The parenthesized list of
423      columns or expressions forms the <firstterm>partition key</firstterm>
424      for the table.  When using range or hash partitioning, the partition key
425      can include multiple columns or expressions (up to 32, but this limit can
426      be altered when building <productname>PostgreSQL</productname>), but for
427      list partitioning, the partition key must consist of a single column or
428      expression.
429     </para>
430
431     <para>
432      Range and list partitioning require a btree operator class, while hash
433      partitioning requires a hash operator class.  If no operator class is
434      specified explicitly, the default operator class of the appropriate
435      type will be used; if no default operator class exists, an error will
436      be raised.  When hash partitioning is used, the operator class used
437      must implement support function 2 (see <xref linkend="xindex-support"/>
438      for details).
439     </para>
440
441     <para>
442      A partitioned table is divided into sub-tables (called partitions),
443      which are created using separate <literal>CREATE TABLE</literal> commands.
444      The partitioned table is itself empty.  A data row inserted into the
445      table is routed to a partition based on the value of columns or
446      expressions in the partition key.  If no existing partition matches
447      the values in the new row, an error will be reported.
448     </para>
449
450     <para>
451      See <xref linkend="ddl-partitioning"/> for more discussion on table
452      partitioning.
453     </para>
454
455    </listitem>

458   <varlistentry id="sql-createtable-partition">
459    <term><literal>PARTITION OF <replaceable class="parameter">parent_table</replaceable> { FOR VALUES <replaceable class="parameter">partition_bound_spec</replaceable> | DEFAULT }</literal></term>
460    <listitem>
461     <para>
462      Creates the table as a <firstterm>partition</firstterm> of the specified
463      parent table. The table can be created either as a partition for specific
464      values using <literal>FOR VALUES</literal> or as a default partition
465      using <literal>DEFAULT</literal>.  Any indexes, constraints and
466      user-defined row-level triggers that exist in the parent table are cloned
467      on the new partition.
468     </para>
469
470     <para>
471      The <replaceable class="parameter">partition_bound_spec</replaceable>
472      must correspond to the partitioning method and partition key of the
473      parent table, and must not overlap with any existing partition of that
474      parent.  The form with <literal>IN</literal> is used for list partitioning,
475      the form with <literal>FROM</literal> and <literal>TO</literal> is used
476      for range partitioning, and the form with <literal>WITH</literal> is used
477      for hash partitioning.
478     </para>
479
480     <para>
481      <replaceable class="parameter">partition_bound_expr</replaceable> is
482      any variable-free expression (subqueries, window functions, aggregate
483      functions, and set-returning functions are not allowed).  Its data type
484      must match the data type of the corresponding partition key column.
485      The expression is evaluated once at table creation time, so it can
486      even contain volatile expressions such as
487      <literal><function>CURRENT_TIMESTAMP</function></literal>.
488     </para>
489
490     <para>
491      When creating a list partition, <literal>NULL</literal> can be
492      specified to signify that the partition allows the partition key
493      column to be null.  However, there cannot be more than one such
494      list partition for a given parent table.  <literal>NULL</literal>
495      cannot be specified for range partitions.
496     </para>
497
498     <para>
499      When creating a range partition, the lower bound specified with
500      <literal>FROM</literal> is an inclusive bound, whereas the upper
501      bound specified with <literal>TO</literal> is an exclusive bound.
502      That is, the values specified in the <literal>FROM</literal> list
503      are valid values of the corresponding partition key columns for this
504      partition, whereas those in the <literal>TO</literal> list are
505      not.  Note that this statement must be understood according to the
506      rules of row-wise comparison (<xref linkend="row-wise-comparison"/>).
507      For example, given <literal>PARTITION BY RANGE (x,y)</literal>, a partition
508      bound <literal>FROM (1, 2) TO (3, 4)</literal>
509      allows <literal>x=1</literal> with any <literal>y&gt;=2</literal>,
510      <literal>x=2</literal> with any non-null <literal>y</literal>,
511      and <literal>x=3</literal> with any <literal>y&lt;4</literal>.
512     </para>
513
514     <para>
515      The special values <literal>MINVALUE</literal> and <literal>MAXVALUE</literal>
516      may be used when creating a range partition to indicate that there
517      is no lower or upper bound on the column's value. For example, a
518      partition defined using <literal>FROM (MINVALUE) TO (10)</literal> allows
519      any values less than 10, and a partition defined using
520      <literal>FROM (10) TO (MAXVALUE)</literal> allows any values greater than
521      or equal to 10.
522     </para>
523
524     <para>
525      When creating a range partition involving more than one column, it
526      can also make sense to use <literal>MAXVALUE</literal> as part of the lower
527      bound, and <literal>MINVALUE</literal> as part of the upper bound. For
528      example, a partition defined using
529      <literal>FROM (0, MAXVALUE) TO (10, MAXVALUE)</literal> allows any rows
530      where the first partition key column is greater than 0 and less than
531      or equal to 10. Similarly, a partition defined using
532      <literal>FROM ('a', MINVALUE) TO ('b', MINVALUE)</literal> allows any rows
533      where the first partition key column starts with "a".
534     </para>
535
536     <para>
537      Note that if <literal>MINVALUE</literal> or <literal>MAXVALUE</literal> is used for
538      one column of a partitioning bound, the same value must be used for all
539      subsequent columns.  For example, <literal>(10, MINVALUE, 0)</literal> is not
540      a valid bound; you should write <literal>(10, MINVALUE, MINVALUE)</literal>.
541     </para>
542
543     <para>
544      Also note that some element types, such as <literal>timestamp</literal>,
545      have a notion of "infinity", which is just another value that can
546      be stored. This is different from <literal>MINVALUE</literal> and
547      <literal>MAXVALUE</literal>, which are not real values that can be stored,
548      but rather they are ways of saying that the value is unbounded.
549      <literal>MAXVALUE</literal> can be thought of as being greater than any
550      other value, including "infinity" and <literal>MINVALUE</literal> as being
551      less than any other value, including "minus infinity". Thus the range
552      <literal>FROM ('infinity') TO (MAXVALUE)</literal> is not an empty range; it
553      allows precisely one value to be stored &mdash; "infinity".
554     </para>
555
556     <para>
557      If <literal>DEFAULT</literal> is specified, the table will be
558      created as the default partition of the parent table.  This option
559      is not available for hash-partitioned tables.  A partition key value
560      not fitting into any other partition of the given parent will be
561      routed to the default partition.
562     </para>
563
564     <para>
565      When a table has an existing <literal>DEFAULT</literal> partition and
566      a new partition is added to it, the default partition must
567      be scanned to verify that it does not contain any rows which properly
568      belong in the new partition.  If the default partition contains a
569      large number of rows, this may be slow.  The scan will be skipped if
570      the default partition is a foreign table or if it has a constraint which
571      proves that it cannot contain rows which should be placed in the new
572      partition.
573     </para>
574
575     <para>
576      When creating a hash partition, a modulus and remainder must be specified.
577      The modulus must be a positive integer, and the remainder must be a
578      non-negative integer less than the modulus.  Typically, when initially
579      setting up a hash-partitioned table, you should choose a modulus equal to
580      the number of partitions and assign every table the same modulus and a
581      different remainder (see examples, below).   However, it is not required
582      that every partition have the same modulus, only that every modulus which
583      occurs among the partitions of a hash-partitioned table is a factor of the
584      next larger modulus.  This allows the number of partitions to be increased
585      incrementally without needing to move all the data at once.  For example,
586      suppose you have a hash-partitioned table with 8 partitions, each of which
587      has modulus 8, but find it necessary to increase the number of partitions
588      to 16.  You can detach one of the modulus-8 partitions, create two new
589      modulus-16 partitions covering the same portion of the key space (one with
590      a remainder equal to the remainder of the detached partition, and the
591      other with a remainder equal to that value plus 8), and repopulate them
592      with data.  You can then repeat this -- perhaps at a later time -- for
593      each modulus-8 partition until none remain.  While this may still involve
594      a large amount of data movement at each step, it is still better than
595      having to create a whole new table and move all the data at once.
596     </para>
597
598     <para>
599      A partition must have the same column names and types as the partitioned
600      table to which it belongs. Modifications to the column names or types of
601      a partitioned table will automatically propagate to all partitions.
602      <literal>CHECK</literal> constraints will be inherited automatically by
603      every partition, but an individual partition may specify additional
604      <literal>CHECK</literal> constraints; additional constraints with the
605      same name and condition as in the parent will be merged with the parent
606      constraint.  Defaults may be specified separately for each partition.
607      But note that a partition's default value is not applied when inserting
608      a tuple through a partitioned table.
609     </para>
610
611     <para>
612      Rows inserted into a partitioned table will be automatically routed to
613      the correct partition.  If no suitable partition exists, an error will
614      occur.
615     </para>
616
617     <para>
618      Operations such as <command>TRUNCATE</command>
619      which normally affect a table and all of its
620      inheritance children will cascade to all partitions, but may also be
621      performed on an individual partition.
622     </para>
623
624     <para>
625      Note that creating a partition using <literal>PARTITION OF</literal>
626      requires taking an <literal>ACCESS EXCLUSIVE</literal> lock on the
627      parent partitioned table.  Likewise, dropping a partition
628      with <command>DROP TABLE</command> requires taking
629      an <literal>ACCESS EXCLUSIVE</literal> lock on the parent table.
630      It is possible to use <link linkend="sql-altertable"><command>ALTER
631      TABLE ATTACH/DETACH PARTITION</command></link> to perform these
632      operations with a weaker lock, thus reducing interference with
633      concurrent operations on the partitioned table.
634     </para>

996    <listitem>
997     <para>
998      The <literal>UNIQUE</literal> constraint specifies that a
999      group of one or more columns of a table can contain
1000      only unique values. The behavior of a unique table constraint
1001      is the same as that of a unique column constraint, with the
1002      additional capability to span multiple columns.  The constraint
1003      therefore enforces that any two rows must differ in at least one
1004      of these columns.
1005     </para>
1006

1008      If the <literal>WITHOUT OVERLAPS</literal> option is specified for the
1009      last column, then that column is checked for overlaps instead of
1010      equality.  In that case, the other columns of the constraint will allow
1011      duplicates so long as the duplicates don't overlap in the
1012      <literal>WITHOUT OVERLAPS</literal> column.  (This is sometimes called a
1013      temporal key, if the column is a range of dates or timestamps, but
1014      <productname>PostgreSQL</productname> allows ranges over any base type.)
1015      In effect, such a constraint is enforced with an <literal>EXCLUDE</literal>
1016      constraint rather than a <literal>UNIQUE</literal> constraint.  So for
1017      example <literal>UNIQUE (id, valid_at WITHOUT OVERLAPS)</literal> behaves
1018      like <literal>EXCLUDE USING GIST (id WITH =, valid_at WITH
1019      &amp;&amp;)</literal>.  The <literal>WITHOUT OVERLAPS</literal> column
1020      must have a range or multirange type.  Empty ranges/multiranges are
1021      not permitted.  The non-<literal>WITHOUT OVERLAPS</literal> columns of
1022      the constraint can be any type that can be compared for equality in a
1023      GiST index.  By default, only range types are supported, but you can use
1024      other types by adding the <xref linkend="btree-gist"/> extension (which
1025      is the expected way to use this feature).
1026     </para>
1027

1029      For the purpose of a unique constraint, null values are not
1030      considered equal, unless <literal>NULLS NOT DISTINCT</literal> is
1031      specified.
1032     </para>

1035      Each unique constraint should name a set of columns that is
1036      different from the set of columns named by any other unique or
1037      primary key constraint defined for the table.  (Otherwise, redundant
1038      unique constraints will be discarded.)
1039     </para>
1040
1041     <para>
1042      When establishing a unique constraint for a multi-level partition
1043      hierarchy, all the columns in the partition key of the target
1044      partitioned table, as well as those of all its descendant partitioned
1045      tables, must be included in the constraint definition.
1046     </para>
1047
1048     <para>
1049      Adding a unique constraint will automatically create a unique btree
1050      index on the column or group of columns used in the constraint.  But if
1051      the constraint includes a <literal>WITHOUT OVERLAPS</literal> clause, it
1052      will use a GiST index.  The created index has the same name as the
1053      unique constraint.
1054     </para>

1057      The optional <literal>INCLUDE</literal> clause adds to that index
1058      one or more columns that are simply <quote>payload</quote>: uniqueness
1059      is not enforced on them, and the index cannot be searched on the basis
1060      of those columns.  However they can be retrieved by an index-only scan.
1061      Note that although the constraint is not enforced on included columns,

1074     <para>
1075      The <literal>PRIMARY KEY</literal> constraint specifies that a column or
1076      columns of a table can contain only unique (non-duplicate), nonnull
1077      values. Only one primary key can be specified for a table, whether as a
1078      column constraint or a table constraint.
1079     </para>
1080
1081     <para>
1082      The primary key constraint should name a set of columns that is
1083      different from the set of columns named by any unique
1084      constraint defined for the same table.  (Otherwise, the unique
1085      constraint is redundant and will be discarded.)
1086     </para>
1087
1088     <para>
1089      <literal>PRIMARY KEY</literal> enforces the same data constraints as
1090      a combination of <literal>UNIQUE</literal> and <literal>NOT
1091      NULL</literal>.  However,
1092      identifying a set of columns as the primary key also provides metadata
1093      about the design of the schema, since a primary key implies that other
1094      tables can rely on this set of columns as a unique identifier for rows.
1095     </para>
1096
1097     <para>
1098      When placed on a partitioned table, <literal>PRIMARY KEY</literal>
1099      constraints share the restrictions previously described
1100      for <literal>UNIQUE</literal> constraints.
1101     </para>
1102
1103     <para>
1104      Adding a <literal>PRIMARY KEY</literal> constraint will automatically
1105      create a unique btree index on the column or group of columns used in
1106      the constraint, or GiST if <literal>WITHOUT OVERLAPS</literal> was
1107      specified.
1108     </para>
1109
1110     <para>
1111      The optional <literal>INCLUDE</literal> clause adds to that index
1112      one or more columns that are simply <quote>payload</quote>: uniqueness
1113      is not enforced on them, and the index cannot be searched on the basis
1114      of those columns.  However they can be retrieved by an index-only scan.
1115      Note that although the constraint is not enforced on included columns,
1116      it still depends on them.  Consequently, some operations on such columns
1117      (e.g., <literal>DROP COLUMN</literal>) can cause cascaded constraint and
1118      index deletion.
1119     </para>

1123   <varlistentry id="sql-createtable-exclude">
1124    <term><literal>EXCLUDE [ USING <replaceable class="parameter">index_method</replaceable> ] ( <replaceable class="parameter">exclude_element</replaceable> WITH <replaceable class="parameter">operator</replaceable> [, ... ] ) <replaceable class="parameter">index_parameters</replaceable> [ WHERE ( <replaceable class="parameter">predicate</replaceable> ) ]</literal></term>
1125    <listitem>
1126     <para>
1127      The <literal>EXCLUDE</literal> clause defines an exclusion
1128      constraint, which guarantees that if
1129      any two rows are compared on the specified column(s) or
1130      expression(s) using the specified operator(s), not all of these
1131      comparisons will return <literal>TRUE</literal>.  If all of the
1132      specified operators test for equality, this is equivalent to a
1133      <literal>UNIQUE</literal> constraint, although an ordinary unique constraint
1134      will be faster.  However, exclusion constraints can specify
1135      constraints that are more general than simple equality.
1136      For example, you can specify a constraint that
1137      no two rows in the table contain overlapping circles
1138      (see <xref linkend="datatype-geometric"/>) by using the
1139      <literal>&amp;&amp;</literal> operator.
1140      The operator(s) are required to be commutative.
1141     </para>
1142
1143     <para>
1144      Exclusion constraints are implemented using
1145      an index that has the same name as the constraint, so each specified
1146      operator must be associated with an appropriate operator class
1147      (see <xref linkend="indexes-opclass"/>) for the index access
1148      method <replaceable>index_method</replaceable>.
1149      Each <replaceable class="parameter">exclude_element</replaceable>
1150      defines a column of the index, so it can optionally specify a collation,
1151      an operator class, operator class parameters, and/or ordering options;
1152      these are described fully under <xref linkend="sql-createindex"/>.
1153     </para>
1154
1155     <para>
1156      The access method must support <literal>amgettuple</literal> (see <xref
1157      linkend="indexam"/>); at present this means <acronym>GIN</acronym>
1158      cannot be used.  Although it's allowed, there is little point in using
1159      B-tree or hash indexes with an exclusion constraint, because this
1160      does nothing that an ordinary unique constraint doesn't do better.
1161      So in practice the access method will always be <acronym>GiST</acronym> or
1162      <acronym>SP-GiST</acronym>.
1163     </para>

1167      exclusion constraint on a subset of the table; internally this creates a
1168      partial index. Note that parentheses are required around the predicate.
1169     </para>

1171     <para>
1172      When establishing an exclusion constraint for a multi-level partition
1173      hierarchy, all the columns in the partition key of the target
1174      partitioned table, as well as those of all its descendant partitioned
1175      tables, must be included in the constraint definition.  Additionally,
1176      those columns must be compared using the equality operator.  These
1177      restrictions ensure that potentially-conflicting rows will exist in the
1178      same partition.  The constraint may also refer to other columns which
1179      are not a part of any partition key, which can be compared using any
1180      appropriate operator.

1192    [ ON UPDATE <replaceable class="parameter">referential_action</replaceable> ]</literal>
1193    (table constraint)</term>

1195    <listitem>
1196     <para>
1197      These clauses specify a foreign key constraint, which requires
1198      that a group of one or more columns of the new table must only
1199      contain values that match values in the referenced
1200      column(s) of some row of the referenced table.  If the <replaceable
1201      class="parameter">refcolumn</replaceable> list is omitted, the
1202      primary key of the <replaceable class="parameter">reftable</replaceable>
1203      is used.  Otherwise, the <replaceable class="parameter">refcolumn</replaceable>
1204      list must refer to the columns of a non-deferrable unique or primary key
1205      constraint or be the columns of a non-partial unique index.
1206     </para>
1207
1208     <para>
1209      If the last column is marked with <literal>PERIOD</literal>, it is
1210      treated in a special way.  While the non-<literal>PERIOD</literal>
1211      columns are compared for equality (and there must be at least one of
1212      them), the <literal>PERIOD</literal> column is not.  Instead, the
1213      constraint is considered satisfied if the referenced table has matching
1214      records (based on the non-<literal>PERIOD</literal> parts of the key)
1215      whose combined <literal>PERIOD</literal> values completely cover the
1216      referencing record's.  In other words, the reference must have a
1217      referent for its entire duration.  This column must be a range or
1218      multirange type.  In addition, the referenced table must have a primary
1219      key or unique constraint declared with <literal>WITHOUT
1220      OVERLAPS</literal>. Finally, if the foreign key has a PERIOD
1221      <replaceable class="parameter">column_name</replaceable> specification
1222      the corresponding <replaceable class="parameter">refcolumn</replaceable>,
1223      if present, must also be marked <literal>PERIOD</literal>.  If the
1224      <replaceable class="parameter">refcolumn</replaceable> clause is omitted,
1225      and thus the reftable's primary key constraint chosen, the primary key
1226      must have its final column marked <literal>WITHOUT OVERLAPS</literal>.
1227     </para>
1228
1229     <para>
1230      For each pair of referencing and referenced column, if they are of a
1231      collatable data type, then the collations must either be both
1232      deterministic or else both the same.  This ensures that both columns
1233      have a consistent notion of equality.
1234     </para>

1236     <para>
1237      The user
1238      must have <literal>REFERENCES</literal> permission on the referenced
1239      table (either the whole table, or the specific referenced columns).  The
1240      addition of a foreign key constraint requires a
1241      <literal>SHARE ROW EXCLUSIVE</literal> lock on the referenced table.
1242      Note that foreign key constraints cannot be defined between temporary
1243      tables and permanent tables.
1244     </para>
1245
1246     <para>
1247      A value inserted into the referencing column(s) is matched against the
1248      values of the referenced table and referenced columns using the
1249      given match type.  There are three match types: <literal>MATCH
1250      FULL</literal>, <literal>MATCH PARTIAL</literal>, and <literal>MATCH
1251      SIMPLE</literal> (which is the default).  <literal>MATCH
1252      FULL</literal> will not allow one column of a multicolumn foreign key
1253      to be null unless all foreign key columns are null; if they are all
1254      null, the row is not required to have a match in the referenced table.
1255      <literal>MATCH SIMPLE</literal> allows any of the foreign key columns
1256      to be null; if any of them are null, the row is not required to have a
1257      match in the referenced table.
1258      <literal>MATCH PARTIAL</literal> is not yet implemented.
1259      (Of course, <literal>NOT NULL</literal> constraints can be applied to the
1260      referencing column(s) to prevent these cases from arising.)
1261     </para>

1270      in the referenced table is being updated to a new value. If the
1271      row is updated, but the referenced column is not actually
1272      changed, no action is done. Referential actions are executed as part of
1273      the data changing command, even if the constraint is deferred.  There

1575    table's secondary <acronym>TOAST</acronym> table, if any

1591    </term>
1592    <listitem>
1593     <para>
1594      The fillfactor for a table is a percentage between 10 and 100.
1595      100 (complete packing) is the default.  When a smaller fillfactor
1596      is specified, <command>INSERT</command> operations pack table pages only
1597      to the indicated percentage; the remaining space on each page is
1598      reserved for updating rows on that page.  This gives <command>UPDATE</command>
1599      a chance to place the updated copy of a row on the same page as the
1600      original, which is more efficient than placing it on a different
1601      page, and makes <link linkend="storage-hot">heap-only tuple
1602      updates</link> more likely.
1603      For a table whose entries are never updated, complete packing is the
1604      best choice, but in heavily updated tables smaller fillfactors are
1605      appropriate.  This parameter cannot be set for TOAST tables.
1606     </para>

2008    <para>
2009     <productname>PostgreSQL</productname> automatically creates an
2010     index for each unique constraint and primary key constraint to
2011     enforce uniqueness.  Thus, it is not necessary to create an
2012     index explicitly for primary key columns.  (See <xref
2013     linkend="sql-createindex"/> for more information.)
2014    </para>
2015
2016    <para>
2017     Unique constraints and primary keys are not inherited in the
2018     current implementation.  This makes the combination of
2019     inheritance and unique constraints rather dysfunctional.
2020    </para>
2021
2022    <para>
2023     A table cannot have more than 1600 columns.  (In practice, the
2024     effective limit is usually lower because of tuple-length constraints.)

2355 <refsect1 id="sql-createtable-compatibility" xreflabel="Compatibility">
2356  <title>Compatibility</title>
2357
2358  <para>
2359   The <command>CREATE TABLE</command> command conforms to the
2360   <acronym>SQL</acronym> standard, with exceptions listed below.
2361  </para>
2362
2363  <refsect2>
2364   <title>Temporary Tables</title>
2365
2366   <para>
2367    Although the syntax of <literal>CREATE TEMPORARY TABLE</literal>
2368    resembles that of the SQL standard, the effect is not the same.  In the
2369    standard,
2370    temporary tables are defined just once and automatically exist (starting
2371    with empty contents) in every session that needs them.
2372    <productname>PostgreSQL</productname> instead
2373    requires each session to issue its own <literal>CREATE TEMPORARY
2374    TABLE</literal> command for each temporary table to be used.  This allows
2375    different sessions to use the same temporary table name for different
2376    purposes, whereas the standard's approach constrains all instances of a
2377    given temporary table name to have the same table structure.
2378   </para>
2379
2380   <para>
2381    The standard's definition of the behavior of temporary tables is
2382    widely ignored.  <productname>PostgreSQL</productname>'s behavior
2383    on this point is similar to that of several other SQL databases.
2384   </para>
2385
2386   <para>
2387    The SQL standard also distinguishes between global and local temporary
2388    tables, where a local temporary table has a separate set of contents for
2389    each SQL module within each session, though its definition is still shared
2390    across sessions.  Since <productname>PostgreSQL</productname> does not
2391    support SQL modules, this distinction is not relevant in
2392    <productname>PostgreSQL</productname>.
2393   </para>
2394
2395   <para>
2396    For compatibility's sake, <productname>PostgreSQL</productname> will
2397    accept the <literal>GLOBAL</literal> and <literal>LOCAL</literal> keywords
2398    in a temporary table declaration, but they currently have no effect.
2399    Use of these keywords is discouraged, since future versions of
2400    <productname>PostgreSQL</productname> might adopt a more
2401    standard-compliant interpretation of their meaning.
2402   </para>
2403
2404   <para>
2405    The <literal>ON COMMIT</literal> clause for temporary tables
2406    also resembles the SQL standard, but has some differences.
2407    If the <literal>ON COMMIT</literal> clause is omitted, SQL specifies that the
2408    default behavior is <literal>ON COMMIT DELETE ROWS</literal>.  However, the
2409    default behavior in <productname>PostgreSQL</productname> is
2410    <literal>ON COMMIT PRESERVE ROWS</literal>.  The <literal>ON COMMIT
2411    DROP</literal> option does not exist in SQL.
2412   </para>
2413  </refsect2>
2414
2415  <refsect2>
2416   <title>Non-Deferred Uniqueness Constraints</title>
2417
2418   <para>
2419    When a <literal>UNIQUE</literal> or <literal>PRIMARY KEY</literal> constraint is
2420    not deferrable, <productname>PostgreSQL</productname> checks for
2421    uniqueness immediately whenever a row is inserted or modified.
2422    The SQL standard says that uniqueness should be enforced only at
2423    the end of the statement; this makes a difference when, for example,
2424    a single command updates multiple key values.  To obtain
2425    standard-compliant behavior, declare the constraint as
2426    <literal>DEFERRABLE</literal> but not deferred (i.e., <literal>INITIALLY
2427    IMMEDIATE</literal>).  Be aware that this can be significantly slower than
2428    immediate uniqueness checking.
2429   </para>
2430  </refsect2>
2431
2432  <refsect2>
2433   <title>Column Check Constraints</title>
2434
2435   <para>
2436    The SQL standard says that <literal>CHECK</literal> column constraints
2437    can only refer to the column they apply to; only <literal>CHECK</literal>
2438    table constraints can refer to multiple columns.
2439    <productname>PostgreSQL</productname> does not enforce this
2440    restriction; it treats column and table check constraints alike.
2441   </para>
2442  </refsect2>
2443
2444  <refsect2>
2445   <title><literal>EXCLUDE</literal> Constraint</title>
2446
2447   <para>
2448    The <literal>EXCLUDE</literal> constraint type is a
2449    <productname>PostgreSQL</productname> extension.
2450   </para>
2451  </refsect2>
2452
2453  <refsect2>
2454   <title>Foreign Key Constraints</title>
2455
2456   <para>
2457    The ability to specify column lists in the foreign key actions
2458    <literal>SET DEFAULT</literal> and <literal>SET NULL</literal> is a
2459    <productname>PostgreSQL</productname> extension.
2460   </para>
2461
2462   <para>
2463    It is a <productname>PostgreSQL</productname> extension that a
2464    foreign key constraint may reference columns of a unique index instead of
2465    columns of a primary key or unique constraint.
2466   </para>
2467  </refsect2>
2468
2469  <refsect2>
2470   <title><literal>NULL</literal> <quote>Constraint</quote></title>
2471
2472   <para>
2473    The <literal>NULL</literal> <quote>constraint</quote> (actually a
2474    non-constraint) is a <productname>PostgreSQL</productname>
2475    extension to the SQL standard that is included for compatibility with some
2476    other database systems (and for symmetry with the <literal>NOT
2477    NULL</literal> constraint).  Since it is the default for any
2478    column, its presence is simply noise.
2479   </para>
2480  </refsect2>
2481
2482  <refsect2>
2483   <title>Constraint Naming</title>
2484
2485   <para>
2486    The SQL standard says that table and domain constraints must have names
2487    that are unique across the schema containing the table or domain.
2488    <productname>PostgreSQL</productname> is laxer: it only requires
2489    constraint names to be unique across the constraints attached to a
2490    particular table or domain.  However, this extra freedom does not exist
2491    for index-based constraints (<literal>UNIQUE</literal>,
2492    <literal>PRIMARY KEY</literal>, and <literal>EXCLUDE</literal>
2493    constraints), because the associated index is named the same as the
2494    constraint, and index names must be unique across all relations within
2495    the same schema.
2496   </para>
2497  </refsect2>
2498
2499  <refsect2>
2500   <title>Inheritance</title>
2501
2502   <para>
2503    Multiple inheritance via the <literal>INHERITS</literal> clause is
2504    a <productname>PostgreSQL</productname> language extension.
2505    SQL:1999 and later define single inheritance using a
2506    different syntax and different semantics.  SQL:1999-style
2507    inheritance is not yet supported by
2508    <productname>PostgreSQL</productname>.
2509   </para>
2510  </refsect2>
2511
2512  <refsect2>
2513   <title>Zero-Column Tables</title>
2514
2515   <para>
2516    <productname>PostgreSQL</productname> allows a table of no columns
2517    to be created (for example, <literal>CREATE TABLE foo();</literal>).  This
2518    is an extension from the SQL standard, which does not allow zero-column
2519    tables.  Zero-column tables are not in themselves very useful, but
2520    disallowing them creates odd special cases for <command>ALTER TABLE
2521    DROP COLUMN</command>, so it seems cleaner to ignore this spec restriction.
2522   </para>
2523  </refsect2>
2524
2525  <refsect2>
2526   <title>Multiple Identity Columns</title>
2527
2528   <para>
2529    <productname>PostgreSQL</productname> allows a table to have more than one
2530    identity column.  The standard specifies that a table can have at most one
2531    identity column.  This is relaxed mainly to give more flexibility for
2532    doing schema changes or migrations.  Note that
2533    the <command>INSERT</command> command supports only one override clause
2534    that applies to the entire statement, so having multiple identity columns
2535    with different behaviors is not well supported.
2536   </para>
2537  </refsect2>

2543    The options <literal>STORED</literal> and <literal>VIRTUAL</literal> are
2544    not standard but are also used by other SQL implementations.  The SQL
2545    standard does not specify the storage of generated columns.
2546   </para>
