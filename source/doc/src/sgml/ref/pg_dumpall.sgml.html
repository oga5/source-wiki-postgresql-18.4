30 <refsect1 id="app-pg-dumpall-description">
31  <title>Description</title>
32
33  <para>
34   <application>pg_dumpall</application> is a utility for writing out
35   (<quote>dumping</quote>) all <productname>PostgreSQL</productname> databases
36   of a cluster into one script file.  The script file contains
37   <acronym>SQL</acronym> commands that can be used as input to <xref
38   linkend="app-psql"/> to restore the databases.  It does this by
39   calling <xref linkend="app-pgdump"/> for each database in the cluster.
40   <application>pg_dumpall</application> also dumps global objects
41   that are common to all databases, namely database roles, tablespaces,
42   and privilege grants for configuration parameters.
43   (<application>pg_dump</application> does not save these objects.)
44  </para>
45
46  <para>
47   Since <application>pg_dumpall</application> reads tables from all
48   databases you will most likely have to connect as a database
49   superuser in order to produce a complete dump.  Also you will need
50   superuser privileges to execute the saved script in order to be
51   allowed to add roles and create databases.
52  </para>
53
54  <para>
55   The SQL script will be written to the standard output.  Use the
56   <option>-f</option>/<option>--file</option> option or shell operators to
57   redirect it into a file.
58  </para>
59
60  <para>
61  <application>pg_dumpall</application> needs to connect several
62  times to the <productname>PostgreSQL</productname> server (once per
63  database).  If you use password authentication it will ask for
64  a password each time. It is convenient to have a
65  <filename>~/.pgpass</filename> file in such cases. See <xref
66  linkend="libpq-pgpass"/> for more information.
67  </para>
68
69  <warning>
70   <para>
71    Restoring a dump causes the destination to execute arbitrary code of the
72    source superusers' choice.  Partial dumps and partial restores do not limit
73    that.  If the source superusers are not trusted, the dumped SQL statements
74    must be inspected before restoring.  Note that the client running the dump
75    and restore need not trust the source or destination superusers.
76   </para>
77  </warning>

699   <variablelist>
700     <varlistentry>
701      <term><option>-d <replaceable class="parameter">connstr</replaceable></option></term>

706        linkend="libpq-connstring">connection string</link>;  these

708       </para>

872 <refsect1>
873  <title>Notes</title>
874
875  <para>
876   Since <application>pg_dumpall</application> calls
877   <application>pg_dump</application> internally, some diagnostic
878   messages will refer to <application>pg_dump</application>.
879  </para>
880
881  <para>
882   The <option>--clean</option> option can be useful even when your
883   intention is to restore the dump script into a fresh cluster.  Use of
884   <option>--clean</option> authorizes the script to drop and re-create the
885   built-in <literal>postgres</literal> and <literal>template1</literal>
886   databases, ensuring that those databases will retain the same properties
887   (for instance, locale and encoding) that they had in the source cluster.
888   Without the option, those databases will retain their existing
889   database-level properties, as well as any pre-existing contents.
890  </para>
891
892  <para>
893   When <option>--statistics</option> is specified,
894   <command>pg_dumpall</command> will include most optimizer statistics in the
895   resulting dump file. This does not include all statistics, such as
896   those created explicitly with <xref linkend="sql-createstatistics"/>,
897   custom statistics added by an extension, or statistics collected by the
898   cumulative statistics system.  Therefore, it may still be useful to
899   run <command>ANALYZE</command> on each database after restoring from a dump
900   file to ensure optimal performance.  You can also run <command>vacuumdb -a
901   -z</command> to analyze all databases.
902  </para>
903
904  <para>
905   The dump script should not be expected to run completely without errors.
906   In particular, because the script will issue <command>CREATE ROLE</command>
907   for every role existing in the source cluster, it is certain to get a
908   <quote>role already exists</quote> error for the bootstrap superuser,
909   unless the destination cluster was initialized with a different bootstrap
910   superuser name.  This error is harmless and should be ignored.  Use of
911   the <option>--clean</option> option is likely to produce additional
912   harmless error messages about non-existent objects, although you can
913   minimize those by adding <option>--if-exists</option>.
914  </para>
915
916  <para>
917   <application>pg_dumpall</application> requires all needed
918   tablespace directories to exist before the restore;  otherwise,
919   database creation will fail for databases in non-default
920   locations.
921  </para>
922
923  <para>
924   It is generally recommended to use the <option>-X</option>
925   (<option>--no-psqlrc</option>) option when restoring a database from a
926   <application>pg_dumpall</application> script to ensure a clean restore
927   process and prevent potential conflicts with non-default
928   <application>psql</application> configurations. Additionally, because
929   the <application>pg_dumpall</application> script may
930   include <application>psql</application> meta-commands, it may be
931   incompatible with clients other than <application>psql</application>.
932  </para>
933 </refsect1>

948<screen>
949<prompt>$</prompt> <userinput>psql -X -f db.out -d postgres</userinput>
950</screen>

952   script file created by <application>pg_dumpall</application> will
953   contain the appropriate commands to create and connect to the saved
954   databases.  An exception is that if you specified <option>--clean</option>,
955   you must connect to the <literal>postgres</literal> database initially;
956   the script will attempt to drop other databases immediately, and that
957   will fail for the database you are connected to.
