46 <refsect1>
47  <title>Description</title>
48
49  <para>
50   <application>pg_createsubscriber</application> creates a new logical
51   replica from a physical standby server.  All tables in the specified
52   database are included in the <link linkend="logical-replication">logical
53   replication</link> setup.  A pair of
54   publication and subscription objects are created for each database.  It
55   must be run at the target server.
56  </para>
57
58  <para>
59   After a successful run, the state of the target server is analogous to a
60   fresh logical replication setup.  The main difference between the logical
61   replication setup and <application>pg_createsubscriber</application> is how
62   the data synchronization is done. <application>pg_createsubscriber</application>
63   does not copy the initial table data. It does only the synchronization phase,
64   which ensures each table is brought up to a synchronized state.
65  </para>
66
67  <para>
68   <application>pg_createsubscriber</application> targets large database
69   systems because in logical replication setup, most of the time is spent
70   doing the initial data copy.  Furthermore, a side effect of this long time
71   spent synchronizing data is usually a large amount of changes to be applied
72   (that were produced during the initial data copy), which increases even
73   more the time when the logical replica will be available. For smaller
74   databases, it is recommended to set up logical replication with initial data
75   synchronization.  For details, see the <command>CREATE SUBSCRIPTION</command>
76   <link linkend="sql-createsubscription-params-with-copy-data">
77   <literal>copy_data</literal></link> option.
78
79  </para>

349  <title>Notes</title>
350
351  <refsect2>
352   <title>Prerequisites</title>
353
354   <para>
355    There are some prerequisites for
356    <application>pg_createsubscriber</application> to convert the target server
357    into a logical replica.  If these are not met, an error will be reported.
358    The source and target servers must have the same major version as the
359    <application>pg_createsubscriber</application>.  The given target data
360    directory must have the same system identifier as the source data
361    directory.  The given database user for the target data directory must have
362    privileges for creating <link
363    linkend="sql-createsubscription">subscriptions</link> and using <link
364    linkend="pg-replication-origin-advance"><function>pg_replication_origin_advance()</function></link>.
365   </para>
366
367   <para>
368    The target server must be used as a physical standby.  The target server
369    must have <xref linkend="guc-max-active-replication-origins"/> and <xref
370    linkend="guc-max-logical-replication-workers"/> configured to a value
371    greater than or equal to the number of specified databases.  The target
372    server must have <xref linkend="guc-max-worker-processes"/> configured to a
373    value greater than the number of specified databases.  The target server
374    must accept local connections. If you are planning to use the
375    <option>--enable-two-phase</option> switch then you will also need to set
376    the <xref linkend="guc-max-prepared-transactions"/> appropriately.
377   </para>
378
379   <para>
380    The source server must accept connections from the target server.  The
381    source server must not be in recovery. The source server must have <xref
382    linkend="guc-wal-level"/> as <literal>logical</literal>.  The source server
383    must have <xref linkend="guc-max-replication-slots"/> configured to a value
384    greater than or equal to the number of specified databases plus existing
385    replication slots.  The source server must have <xref
386    linkend="guc-max-wal-senders"/> configured to a value greater than or equal
387    to the number of specified databases and existing WAL sender processes.
388   </para>
389  </refsect2>
390
391  <refsect2>
392   <title>Warnings</title>
393
394   <para>
395    If <application>pg_createsubscriber</application> fails after the target
396    server was promoted, then the data directory is likely not in a state that
397    can be recovered.  In such case, creating a new standby server is
398    recommended.
399   </para>
400
401   <para>
402    <application>pg_createsubscriber</application> usually starts the target
403    server with different connection settings during transformation.  Hence,
404    connections to the target server should fail.
405   </para>
406
407   <para>
408    Since DDL commands are not replicated by logical replication, avoid
409    executing DDL commands that change the database schema while running
410    <application>pg_createsubscriber</application>.  If the target server has
411    already been converted to logical replica, the DDL commands might not be
412    replicated, which might cause an error.
413   </para>
414
415   <para>
416    If <application>pg_createsubscriber</application> fails while processing,
417    objects (publications, replication slots) created on the source server are
418    removed.  The removal might fail if the target server cannot connect to
419    the source server.  In such a case, a warning message will inform the
420    objects left.  If the target server is running, it will be stopped.
421   </para>
422
423   <para>
424    If the replication is using <xref linkend="guc-primary-slot-name"/>, it
425    will be removed from the source server after the logical replication
426    setup.
427   </para>
428
429   <para>
430    If the target server is a synchronous replica, transaction commits on the
431    primary might wait for replication while running
432    <application>pg_createsubscriber</application>.
433   </para>
434
435   <para>
436    Unless the <option>--enable-two-phase</option> switch is specified,
437    <application>pg_createsubscriber</application> sets up logical
438    replication with two-phase commit disabled.  This means that any
439    prepared transactions will be replicated at the time
440    of <command>COMMIT PREPARED</command>, without advance preparation.
441    Once setup is complete, you can manually drop and re-create the
442    subscription(s) with
443    the <link linkend="sql-createsubscription-params-with-two-phase"><literal>two_phase</literal></link>
444    option enabled.
445   </para>
446
447   <para>
448    <application>pg_createsubscriber</application> changes the system
449    identifier using <application>pg_resetwal</application>.  It would avoid
450    situations in which the target server might use WAL files from the source
451    server.  If the target server has a standby, replication will break and a
452    fresh standby should be created.
453   </para>
454
455   <para>
456    Replication failures can occur if required WAL files are missing. To prevent
457    this, the source server must set
458    <xref linkend="guc-max-slot-wal-keep-size"/> to <literal>-1</literal> to
459    ensure that required WAL files are not prematurely removed.
460   </para>
461  </refsect2>

463  <refsect2>
464   <title>How It Works</title>
465
466   <para>
467    The basic idea is to have a replication start point from the source server
468    and set up a logical replication to start from this point:
469   </para>
470
471   <procedure>
472    <step>
473     <para>
474      Start the target server with the specified command-line options.  If the
475      target server is already running,
476      <application>pg_createsubscriber</application> will terminate with an
477      error.
478     </para>
479    </step>
480
481    <step>
482     <para>
483      Check if the target server can be converted.  There are also a few
484      checks on the source server.  If any of the prerequisites are not met,
485      <application>pg_createsubscriber</application> will terminate with an
486      error.
487     </para>
488    </step>
489
490    <step>
491     <para>
492      Create a publication and replication slot for each specified database on
493      the source server.  Each publication is created using <link
494      linkend="sql-createpublication-params-for-all-tables"><literal>FOR ALL
495      TABLES</literal></link>.  If the <option>--publication</option> option
496      is not specified, the publication has the following name pattern:
497      <quote><literal>pg_createsubscriber_%u_%x</literal></quote> (parameter:
498      database <parameter>oid</parameter>, random <parameter>int</parameter>).
499      If the <option>--replication-slot</option> option is not specified, the
500      replication slot has the following name pattern:
501      <quote><literal>pg_createsubscriber_%u_%x</literal></quote> (parameters:
502      database <parameter>oid</parameter>, random <parameter>int</parameter>).
503      These replication slots will be used by the subscriptions in a future
504      step.  The last replication slot LSN is used as a stopping point in the
505      <xref linkend="guc-recovery-target-lsn"/> parameter and by the
506      subscriptions as a replication start point.  It guarantees that no
507      transaction will be lost.
508     </para>
509    </step>
510
511    <step>
512     <para>
513      Write recovery parameters into the target data directory and restart the
514      target server.  It specifies an LSN (<xref
515      linkend="guc-recovery-target-lsn"/>) of the write-ahead log location up
516      to which recovery will proceed.  It also specifies
517      <literal>promote</literal> as the action that the server should take
518      once the recovery target is reached.  Additional <link
519      linkend="runtime-config-wal-recovery-target">recovery parameters</link>
520      are added to avoid unexpected behavior during the recovery process such
521      as end of the recovery as soon as a consistent state is reached (WAL
522      should be applied until the replication start location) and multiple
523      recovery targets that can cause a failure.  This step finishes once the
524      server ends standby mode and is accepting read-write transactions.  If
525      <option>--recovery-timeout</option> option is set,
526      <application>pg_createsubscriber</application> terminates if recovery
527      does not end until the given number of seconds.
528     </para>
529    </step>
530
531    <step>
532     <para>
533      Create a subscription for each specified database on the target server.
534      If the <option>--subscription</option> option is not specified, the
535      subscription has the following name pattern:
536      <quote><literal>pg_createsubscriber_%u_%x</literal></quote> (parameters:
537      database <parameter>oid</parameter>, random <parameter>int</parameter>).
538      It does not copy existing data from the source server.  It does not
539      create a replication slot.  Instead, it uses the replication slot that
540      was created in a previous step.  The subscription is created but it is
541      not enabled yet.  The reason is the replication progress must be set to
542      the replication start point before starting the replication.
543     </para>
544    </step>
545
546    <step>
547     <para>
548      Drop publications on the target server that were replicated because they
549      were created before the replication start location.  It has no use on
550      the subscriber.
551     </para>
552    </step>
553
554    <step>
555     <para>
556      Set the replication progress to the replication start point for each
557      subscription.  When the target server starts the recovery process, it
558      catches up to the replication start point.  This is the exact LSN to be
559      used as a initial replication location for each subscription.  The
560      replication origin name is obtained since the subscription was created.
561      The replication origin name and the replication start point are used in
562      <link
563      linkend="pg-replication-origin-advance"><function>pg_replication_origin_advance()</function></link>
564      to set up the initial replication location.
565     </para>
566    </step>
567
568    <step>
569     <para>
570      Enable the subscription for each specified database on the target server.
571      The subscription starts applying transactions from the replication start
572      point.
573     </para>
574    </step>
575
576    <step>
577     <para>
578      If the standby server was using <xref linkend="guc-primary-slot-name"/>,
579      it has no use from now on so drop it.
580     </para>
581    </step>
582
583    <step>
584     <para>
585      If the standby server contains <link
586      linkend="logicaldecoding-replication-slots-synchronization">failover
587      replication slots</link>, they cannot be synchronized anymore, so drop
588      them.
589     </para>
590    </step>
591
592    <step>
593     <para>
594      Update the system identifier on the target server. The
595      <xref linkend="app-pgresetwal"/> is run to modify the system identifier.
596      The target server is stopped as a <command>pg_resetwal</command> requirement.
597     </para>
598    </step>
599   </procedure>
