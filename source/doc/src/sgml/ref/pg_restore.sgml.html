36 <refsect1 id="app-pgrestore-description">
37  <title>Description</title>
38
39  <para>
40   <application>pg_restore</application> is a utility for restoring a
41   <productname>PostgreSQL</productname> database from an archive
42   created by <xref linkend="app-pgdump"/> in one of the non-plain-text
43   formats.  It will issue the commands necessary to reconstruct the
44   database to the state it was in at the time it was saved.  The
45   archive files also allow <application>pg_restore</application> to
46   be selective about what is restored, or even to reorder the items
47   prior to being restored. The archive files are designed to be
48   portable across architectures.
49  </para>
50
51  <para>
52   <application>pg_restore</application> can operate in two modes.
53   If a database name is specified, <application>pg_restore</application>
54   connects to that database and restores archive contents directly into
55   the database.  Otherwise, a script containing the SQL
56   commands necessary to rebuild the database is created and written
57   to a file or standard output.  This script output is equivalent to
58   the plain text output format of <application>pg_dump</application>.
59   Some of the options controlling the output are therefore analogous to
60   <application>pg_dump</application> options.
61  </para>
62
63  <para>
64   Obviously, <application>pg_restore</application> cannot restore information
65   that is not present in the archive file.  For instance, if the
66   archive was made using the <quote>dump data as
67   <command>INSERT</command> commands</quote> option,
68   <application>pg_restore</application> will not be able to load the data
69   using <command>COPY</command> statements.
70  </para>
71
72  <warning>
73   <para>
74    Restoring a dump causes the destination to execute arbitrary code of the
75    source superusers' choice.  Partial dumps and partial restores do not limit
76    that.  If the source superusers are not trusted, the dumped SQL statements
77    must be inspected before restoring.  Non-plain-text dumps can be inspected
78    by using <application>pg_restore</application>'s <option>--file</option>
79    option.  Note that the client running the dump and restore need not trust
80    the source or destination superusers.
81   </para>
82  </warning>
83 </refsect1>

262      <term><option>-j <replaceable class="parameter">number-of-jobs</replaceable></option></term>
263      <term><option>--jobs=<replaceable class="parameter">number-of-jobs</replaceable></option></term>
264      <listitem>
265       <para>
266        Run the most time-consuming steps
267        of <application>pg_restore</application> &mdash; those that load data,
268        create indexes, or create constraints &mdash; concurrently, using up
269        to <replaceable class="parameter">number-of-jobs</replaceable>
270        concurrent sessions.  This option can dramatically reduce the time
271        to restore a large database to a server running on a
272        multiprocessor machine.  This option is ignored when emitting a script
273        rather than connecting directly to a database server.
274       </para>
275
276       <para>
277        Each job is one process or one thread, depending on the
278        operating system, and uses a separate connection to the
279        server.
280       </para>
281
282       <para>
283        The optimal value for this option depends on the hardware
284        setup of the server, of the client, and of the network.
285        Factors include the number of CPU cores and the disk setup.  A
286        good place to start is the number of CPU cores on the server,
287        but values larger than that can also lead to faster restore
288        times in many cases.  Of course, values that are too high will
289        lead to decreased performance because of thrashing.
290       </para>
291
292       <para>
293        Only the custom and directory archive formats are supported
294        with this option.
295        The input must be a regular file or directory (not, for example, a
296        pipe or standard input).  Also, multiple
297        jobs cannot be used together with the
298        option <option>--single-transaction</option>.
299       </para>
300      </listitem>

574       <para>

577        <xref linkend="guc-row-security"/> to off, to ensure

594     <varlistentry>
595      <term><option>--filter=<replaceable class="parameter">filename</replaceable></option></term>
596      <listitem>
597       <para>
598        Specify a filename from which to read patterns for objects excluded
599        or included from restore. The patterns are interpreted according to the
600        same rules as
601        <option>-n</option>/<option>--schema</option> for including objects in schemas,
602        <option>-N</option>/<option>--exclude-schema</option> for excluding objects in schemas,
603        <option>-P</option>/<option>--function</option> for restoring named functions,
604        <option>-I</option>/<option>--index</option> for restoring named indexes,
605        <option>-t</option>/<option>--table</option> for restoring named tables
606        or <option>-T</option>/<option>--trigger</option> for restoring triggers.
607        To read from <literal>STDIN</literal>, use <filename>-</filename> as the
608        filename.  The <option>--filter</option> option can be specified in
609        conjunction with the above listed options for including or excluding
610        objects, and can also be specified more than once for multiple filter
611        files.
612       </para>
613
614       <para>
615        The file lists one database pattern per row, with the following format:
616<synopsis>
617{ include | exclude } { function | index | schema | table | trigger } <replaceable class="parameter">PATTERN</replaceable>
618</synopsis>
619       </para>
620
621       <para>
622        The first keyword specifies whether the objects matched by the pattern
623        are to be included or excluded. The second keyword specifies the type
624        of object to be filtered using the pattern:
625        <itemizedlist>
626         <listitem>
627          <para>
628           <literal>function</literal>: functions, works like the
629           <option>-P</option>/<option>--function</option> option. This keyword
630           can only be used with the <literal>include</literal> keyword.
631          </para>
632         </listitem>
633         <listitem>
634          <para>
635           <literal>index</literal>: indexes, works like the
636           <option>-I</option>/<option>--indexes</option> option. This keyword
637           can only be used with the <literal>include</literal> keyword.
638          </para>
639         </listitem>
640         <listitem>
641          <para>
642           <literal>schema</literal>: schemas, works like the
643           <option>-n</option>/<option>--schema</option> and
644           <option>-N</option>/<option>--exclude-schema</option> options.
645          </para>
646         </listitem>
647         <listitem>
648          <para>
649           <literal>table</literal>: tables, works like the
650           <option>-t</option>/<option>--table</option> option. This keyword
651           can only be used with the <literal>include</literal> keyword.
652          </para>
653         </listitem>
654         <listitem>
655          <para>
656           <literal>trigger</literal>: triggers, works like the

1089 <refsect1 id="app-pgrestore-notes">
1090  <title>Notes</title>
1091
1092  <para>
1093   If your installation has any local additions to the
1094   <literal>template1</literal> database, be careful to load the output of
1095   <application>pg_restore</application> into a truly empty database;
1096   otherwise you are likely to get errors due to duplicate definitions
1097   of the added objects.  To make an empty database without any local
1098   additions, copy from <literal>template0</literal> not <literal>template1</literal>, for example:
1099<programlisting>
1100CREATE DATABASE foo WITH TEMPLATE template0;
1101</programlisting>
1102  </para>
1103
1104  <para>
1105   The limitations of <application>pg_restore</application> are detailed below.
1106
1107   <itemizedlist>
1108    <listitem>
1109     <para>
1110      When restoring data to a pre-existing table and the option
1111      <option>--disable-triggers</option> is used,
1112      <application>pg_restore</application> emits commands
1113      to disable triggers on user tables before inserting the data, then emits commands to
1114      re-enable them after the data has been inserted.  If the restore is stopped in the
1115      middle, the system catalogs might be left in the wrong state.
1116     </para>
1117    </listitem>
1118
1119    <listitem>
1120     <para><application>pg_restore</application> cannot restore large objects
1121      selectively;  for instance, only those for a specific table.  If
1122      an archive contains large objects, then all large objects will be
1123      restored, or none of them if they are excluded via <option>-L</option>,
1124      <option>-t</option>, or other options.
1125     </para>
1126    </listitem>
1127
1128   </itemizedlist>
1129  </para>
1130
1131  <para>
1132   See also the <xref linkend="app-pgdump"/> documentation for details on
1133   limitations of <application>pg_dump</application>.
1134  </para>
1135
1136  <para>
1137   By default, <command>pg_restore</command> will restore optimizer statistics
1138   if included in the dump file.  If not all statistics were restored, it may
1139   be useful to run <command>ANALYZE</command> on each restored table so the
1140   optimizer has useful statistics; see <xref
1141   linkend="vacuum-for-statistics"/> and <xref linkend="autovacuum"/> for more
1142   information.
1143  </para>
1144
1145 </refsect1>

1148 <refsect1 id="app-pgrestore-examples">
1149  <title>Examples</title>
1150
1151  <para>
1152   Assume we have dumped a database called <literal>mydb</literal> into a
1153   custom-format dump file:
1154
1155<screen>
1156<prompt>$</prompt> <userinput>pg_dump -Fc mydb &gt; db.dump</userinput>
1157</screen>
1158  </para>
1159
1160  <para>
1161   To drop the database and recreate it from the dump:
1162
1163<screen>
1164<prompt>$</prompt> <userinput>dropdb mydb</userinput>
1165<prompt>$</prompt> <userinput>pg_restore -C -d postgres db.dump</userinput>
1166</screen>
1167
1168   The database named in the <option>-d</option> switch can be any database existing
1169   in the cluster; <application>pg_restore</application> only uses it to issue the
1170   <command>CREATE DATABASE</command> command for <literal>mydb</literal>.  With
1171   <option>-C</option>, data is always restored into the database name that appears
1172   in the dump file.
1173  </para>
1174
1175  <para>
1176   To restore the dump into a new database called <literal>newdb</literal>:
1177
1178<screen>
1179<prompt>$</prompt> <userinput>createdb -T template0 newdb</userinput>
1180<prompt>$</prompt> <userinput>pg_restore -d newdb db.dump</userinput>
1181</screen>
1182
1183   Notice we don't use <option>-C</option>, and instead connect directly to the
1184   database to be restored into.  Also note that we clone the new database
1185   from <literal>template0</literal> not <literal>template1</literal>, to ensure it is
1186   initially empty.
1187  </para>
1188
1189  <para>
1190   To reorder database items, it is first necessary to dump the table of
1191   contents of the archive:
1192<screen>
1193<prompt>$</prompt> <userinput>pg_restore -l db.dump &gt; db.list</userinput>
1194</screen>
1195   The listing file consists of a header and one line for each item, e.g.:
1196<programlisting>
1197;
1198; Archive created at Mon Sep 14 13:55:39 2009
1199;     dbname: DBDEMOS
1200;     TOC Entries: 81
1201;     Compression: 9
1202;     Dump Version: 1.10-0
1203;     Format: CUSTOM
1204;     Integer: 4 bytes
1205;     Offset: 8 bytes
1206;     Dumped from database version: 8.3.5
1207;     Dumped by pg_dump version: 8.3.8
1208;
1209;
1210; Selected TOC Entries:
1211;
12123; 2615 2200 SCHEMA - public pasha
12131861; 0 0 COMMENT - SCHEMA public pasha
12141862; 0 0 ACL - public pasha
1215317; 1247 17715 TYPE public composite pasha
1216319; 1247 25899 DOMAIN public domain0 pasha
1217</programlisting>
1218   Semicolons start a comment, and the numbers at the start of lines refer to the
1219   internal archive ID assigned to each item.
1220  </para>
1221
1222  <para>
1223   Lines in the file can be commented out, deleted, and reordered. For example:
1224<programlisting>
122510; 145433 TABLE map_resolutions postgres
1226;2; 145344 TABLE species postgres
1227;4; 145359 TABLE nt_header postgres
12286; 145402 TABLE species_records postgres
1229;8; 145416 TABLE ss_old postgres
1230</programlisting>
1231   could be used as input to <application>pg_restore</application> and would only restore
1232   items 10 and 6, in that order:
1233<screen>
1234<prompt>$</prompt> <userinput>pg_restore -L db.list db.dump</userinput>
1235</screen></para>
