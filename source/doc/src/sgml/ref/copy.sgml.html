19  <refpurpose>copy data between a file and a table</refpurpose>

22 <refsynopsisdiv>
23<synopsis>
24COPY <replaceable class="parameter">table_name</replaceable> [ ( <replaceable class="parameter">column_name</replaceable> [, ...] ) ]
25    FROM { '<replaceable class="parameter">filename</replaceable>' | PROGRAM '<replaceable class="parameter">command</replaceable>' | STDIN }
26    [ [ WITH ] ( <replaceable class="parameter">option</replaceable> [, ...] ) ]

30    TO { '<replaceable class="parameter">filename</replaceable>' | PROGRAM '<replaceable class="parameter">command</replaceable>' | STDOUT }
31    [ [ WITH ] ( <replaceable class="parameter">option</replaceable> [, ...] ) ]

44    FORCE_NOT_NULL { ( <replaceable class="parameter">column_name</replaceable> [, ...] ) | * }
45    FORCE_NULL { ( <replaceable class="parameter">column_name</replaceable> [, ...] ) | * }

53 <refsect1>
54  <title>Description</title>
55
56  <para>
57   <command>COPY</command> moves data between
58   <productname>PostgreSQL</productname> tables and standard file-system
59   files. <command>COPY TO</command> copies the contents of a table
60   <emphasis>to</emphasis> a file, while <command>COPY FROM</command> copies
61   data <emphasis>from</emphasis> a file to a table (appending the data to
62   whatever is in the table already).  <command>COPY TO</command>
63   can also copy the results of a <command>SELECT</command> query.
64  </para>
65
66  <para>
67   If a column list is specified, <command>COPY TO</command> copies only
68   the data in the specified columns to the file.  For <command>COPY
69   FROM</command>, each field in the file is inserted, in order, into the
70   specified column.  Table columns not specified in the <command>COPY
71   FROM</command> column list will receive their default values.
72  </para>
73
74  <para>
75   <command>COPY</command> with a file name instructs the
76   <productname>PostgreSQL</productname> server to directly read from
77   or write to a file. The file must be accessible by the
78   <productname>PostgreSQL</productname> user (the user ID the server
79   runs as) and the name must be specified from the viewpoint of the
80   server. When <literal>PROGRAM</literal> is specified, the server
81   executes the given command and reads from the standard output of the
82   program, or writes to the standard input of the program. The command
83   must be specified from the viewpoint of the server, and be executable
84   by the <productname>PostgreSQL</productname> user.  When
85   <literal>STDIN</literal> or <literal>STDOUT</literal> is
86   specified, data is transmitted via the connection between the
87   client and the server.
88  </para>
89
90  <para>
91    Each backend running <command>COPY</command> will report its progress
92    in the <structname>pg_stat_progress_copy</structname> view. See
93    <xref linkend="copy-progress-reporting"/> for details.
94  </para>
95
96  <para>
97    By default, <command>COPY</command> will fail if it encounters an error
98    during processing. For use cases where a best-effort attempt at loading
99    the entire file is desired, the <literal>ON_ERROR</literal> clause can
100    be used to specify some other behavior.
101  </para>

230    <term><literal>FREEZE</literal></term>
231    <listitem>
232     <para>
233      Requests copying the data with rows already frozen, just as they
234      would be after running the <command>VACUUM FREEZE</command> command.
235      This is intended as a performance option for initial data loading.
236      Rows will be frozen only if the table being loaded has been created
237      or truncated in the current subtransaction, there are no cursors

241      This option is only allowed in <command>COPY FROM</command>.

258      a comma in <literal>CSV</literal> format.

274      This option is not allowed when using <literal>binary</literal> format.
275     </para>

293      Specifies the string that represents a default value. Each time the string
294      is found in the input file, the default value of the corresponding column
295      will be used.
296      This option is allowed only in <command>COPY FROM</command>, and only when
297      not using <literal>binary</literal> format.

308      names from the table.  On input, the first line is discarded when this
309      option is set to <literal>true</literal> (or equivalent Boolean value).

311      of the columns in the header line must match the actual column names of
312      the table, in order;  otherwise an error is raised.
313      This option is not allowed when using <literal>binary</literal> format.
314      The <literal>MATCH</literal> option is only valid for <command>COPY
315      FROM</command> commands.

324      Specifies the quoting character to be used when a data value is quoted.
325      The default is double-quote.
326      This must be a single one-byte character.
327      This option is allowed only when using <literal>CSV</literal> format.
328     </para>

330   </varlistentry>

333    <term><literal>ESCAPE</literal></term>
334    <listitem>
335     <para>
336      Specifies the character that should appear before a
337      data character that matches the <literal>QUOTE</literal> value.
338      The default is the same as the <literal>QUOTE</literal> value (so that
339      the quoting character is doubled if it appears in the data).

342     </para>

347    <term><literal>FORCE_QUOTE</literal></term>
348    <listitem>
349     <para>
350      Forces quoting to be
351      used for all non-<literal>NULL</literal> values in each specified column.
352      <literal>NULL</literal> output is never quoted. If <literal>*</literal> is specified,
353      non-<literal>NULL</literal> values will be quoted in all columns.
354      This option is allowed only in <command>COPY TO</command>, and only when
355      using <literal>CSV</literal> format.
356     </para>
357    </listitem>
358   </varlistentry>
359
360   <varlistentry>
361    <term><literal>FORCE_NOT_NULL</literal></term>
362    <listitem>
363     <para>
364      Do not match the specified columns' values against the null string.
365      In the default case where the null string is empty, this means that
366      empty values will be read as zero-length strings rather than nulls,
367      even when they are not quoted.
368      If <literal>*</literal> is specified, the option will be applied to all columns.
369      This option is allowed only in <command>COPY FROM</command>, and only when
370      using <literal>CSV</literal> format.
371     </para>
372    </listitem>
373   </varlistentry>
374
375   <varlistentry>
376    <term><literal>FORCE_NULL</literal></term>
377    <listitem>
378     <para>
379      Match the specified columns' values against the null string, even
380      if it has been quoted, and if a match is found set the value to
381      <literal>NULL</literal>. In the default case where the null string is empty,
382      this converts a quoted empty string into NULL.
383      If <literal>*</literal> is specified, the option will be applied to all columns.
384      This option is allowed only in <command>COPY FROM</command>, and only when
385      using <literal>CSV</literal> format.
386     </para>
387    </listitem>
388   </varlistentry>
389
390   <varlistentry>
391    <term><literal>ON_ERROR</literal></term>
392    <listitem>
393     <para>
394      Specifies how to behave when encountering an error converting a column's
395      input value into its data type.
396      An <replaceable class="parameter">error_action</replaceable> value of
397      <literal>stop</literal> means fail the command, while
398      <literal>ignore</literal> means discard the input row and continue with the next one.
399      The default is <literal>stop</literal>.
400     </para>
401     <para>
402      The <literal>ignore</literal> option is applicable only for <command>COPY FROM</command>
403      when the <literal>FORMAT</literal> is <literal>text</literal> or <literal>csv</literal>.
404     </para>
405     <para>
406      A <literal>NOTICE</literal> message containing the ignored row count is
407      emitted at the end of the <command>COPY FROM</command> if at least one
408      row was discarded. When <literal>LOG_VERBOSITY</literal> option is set to
409      <literal>verbose</literal>, a <literal>NOTICE</literal> message
410      containing the line of the input file and the column name whose input
411      conversion has failed is emitted for each discarded row.
412      When it is set to <literal>silent</literal>, no message is emitted
413      regarding ignored rows.
414     </para>
415    </listitem>
416   </varlistentry>
417
418   <varlistentry>
419    <term><literal>REJECT_LIMIT</literal></term>
420    <listitem>
421     <para>
422      Specifies the maximum number of errors tolerated while converting a
423      column's input value to its data type, when <literal>ON_ERROR</literal> is
424      set to <literal>ignore</literal>.
425      If the input causes more errors than the specified value, the <command>COPY</command>
426      command fails, even with <literal>ON_ERROR</literal> set to <literal>ignore</literal>.
427      This clause must be used with <literal>ON_ERROR</literal>=<literal>ignore</literal>
428      and <replaceable class="parameter">maxerror</replaceable> must be positive <type>bigint</type>.
429      If not specified, <literal>ON_ERROR</literal>=<literal>ignore</literal>
430      allows an unlimited number of errors, meaning <command>COPY</command> will
431      skip all erroneous data.
432     </para>
433    </listitem>
434   </varlistentry>

437    <term><literal>ENCODING</literal></term>
438    <listitem>
439     <para>
440      Specifies that the file is encoded in the <replaceable
441      class="parameter">encoding_name</replaceable>.  If this option is
442      omitted, the current client encoding is used. See the Notes below
443      for more details.
444     </para>
445    </listitem>
446   </varlistentry>

453      command: <literal>default</literal>, <literal>verbose</literal>, or
454      <literal>silent</literal>.
455      If <literal>verbose</literal> is specified, additional messages are
456      emitted during processing.
457      <literal>silent</literal> suppresses both verbose and default messages.
458     </para>
459     <para>

468    <listitem>
469   <para>
470    The optional <literal>WHERE</literal> clause has the general form
471<synopsis>
472WHERE <replaceable class="parameter">condition</replaceable>
473</synopsis>
474    where <replaceable class="parameter">condition</replaceable> is

476    <type>boolean</type>.  Any row that does not satisfy this

500   tag of the form
501<screen>
502COPY <replaceable class="parameter">count</replaceable>
503</screen>

505   of rows copied.
506  </para>

508  <note>
509   <para>
510    <application>psql</application> will print this command tag only if the command
511    was not <literal>COPY ... TO STDOUT</literal>, or the

519 <refsect1>
520  <title>Notes</title>
521
522   <para>
523    <command>COPY TO</command> can be used with plain
524    tables and populated materialized views.
525    For example,
526    <literal>COPY <replaceable class="parameter">table</replaceable>
527    TO</literal> copies the same rows as
528    <literal>SELECT * FROM ONLY <replaceable class="parameter">table</replaceable></literal>.
529    However it doesn't directly support other relation types,
530    such as partitioned tables, inheritance child tables, or views.
531    To copy all rows from such relations, use <literal>COPY (SELECT * FROM
532    <replaceable class="parameter">table</replaceable>) TO</literal>.
533   </para>
534
535   <para>
536    <command>COPY FROM</command> can be used with plain, foreign, or
537    partitioned tables or with views that have
538    <literal>INSTEAD OF INSERT</literal> triggers.
539   </para>
540
541   <para>
542    You must have select privilege on the table
543    whose values are read by <command>COPY TO</command>, and
544    insert privilege on the table into which values
545    are inserted by <command>COPY FROM</command>.  It is sufficient
546    to have column privileges on the column(s) listed in the command.
547   </para>
548
549   <para>
550    If row-level security is enabled for the table, the relevant
551    <command>SELECT</command> policies will apply to <literal>COPY
552    <replaceable class="parameter">table</replaceable> TO</literal> statements.
553    Currently, <command>COPY FROM</command> is not supported for tables
554    with row-level security. Use equivalent <command>INSERT</command>
555    statements instead.
556   </para>
557
558   <para>
559    Files named in a <command>COPY</command> command are read or written
560    directly by the server, not by the client application. Therefore,
561    they must reside on or be accessible to the database server machine,
562    not the client. They must be accessible to and readable or writable
563    by the <productname>PostgreSQL</productname> user (the user ID the
564    server runs as), not the client. Similarly,
565    the command specified with <literal>PROGRAM</literal> is executed directly
566    by the server, not by the client application, must be executable by the
567    <productname>PostgreSQL</productname> user.
568    <command>COPY</command> naming a file or command is only allowed to
569    database superusers or users who are granted one of the roles
570    <literal>pg_read_server_files</literal>,
571    <literal>pg_write_server_files</literal>,
572    or <literal>pg_execute_server_program</literal>, since it allows reading
573    or writing any file or running a program that the server has privileges to
574    access.
575   </para>
576
577   <para>
578    Do not confuse <command>COPY</command> with the
579    <application>psql</application> instruction
580    <command><link linkend="app-psql-meta-commands-copy">\copy</link></command>. <command>\copy</command> invokes
581    <command>COPY FROM STDIN</command> or <command>COPY TO
582    STDOUT</command>, and then fetches/stores the data in a file
583    accessible to the <application>psql</application> client. Thus,
584    file accessibility and access rights depend on the client rather
585    than the server when <command>\copy</command> is used.
586   </para>
587
588   <para>
589    It is recommended that the file name used in <command>COPY</command>
590    always be specified as an absolute path. This is enforced by the
591    server in the case of <command>COPY TO</command>, but for
592    <command>COPY FROM</command> you do have the option of reading from
593    a file specified by a relative path. The path will be interpreted
594    relative to the working directory of the server process (normally
595    the cluster's data directory), not the client's working directory.
596   </para>
597
598   <para>
599    Executing a command with <literal>PROGRAM</literal> might be restricted
600    by the operating system's access control mechanisms, such as SELinux.
601   </para>
602
603   <para>
604    <command>COPY FROM</command> will invoke any triggers and check
605    constraints on the destination table. However, it will not invoke rules.
606   </para>
607
608   <para>
609    For identity columns, the <command>COPY FROM</command> command will always
610    write the column values provided in the input data, like
611    the <command>INSERT</command> option <literal>OVERRIDING SYSTEM
612    VALUE</literal>.
613   </para>
614
615   <para>
616    <command>COPY</command> input and output is affected by
617    <varname>DateStyle</varname>. To ensure portability to other
618    <productname>PostgreSQL</productname> installations that might use
619    non-default <varname>DateStyle</varname> settings,
620    <varname>DateStyle</varname> should be set to <literal>ISO</literal> before
621    using <command>COPY TO</command>.  It is also a good idea to avoid dumping
622    data with <varname>IntervalStyle</varname> set to
623    <literal>sql_standard</literal>, because negative interval values might be
624    misinterpreted by a server that has a different setting for
625    <varname>IntervalStyle</varname>.
626   </para>
627
628   <para>
629    Input data is interpreted according to <literal>ENCODING</literal>
630    option or the current client encoding, and output data is encoded
631    in <literal>ENCODING</literal> or the current client encoding, even
632    if the data does not pass through the client but is read from or
633    written to a file directly by the server.
634   </para>
635
636   <para>
637    The <command>COPY FROM</command> command physically inserts input rows
638    into the table as it progresses.  If the command fails, these rows are
639    left in a deleted state; these rows will not be visible, but still
640    occupy disk space. This might amount to considerable
641    wasted disk space if the failure happened well into a large copy
642    operation. <command>VACUUM</command> should be used to recover the
643    wasted space.
644   </para>
645
646   <para>
647    <literal>FORCE_NULL</literal> and <literal>FORCE_NOT_NULL</literal> can be used
648    simultaneously on the same column. This results in converting quoted
649    null strings to null values and unquoted null strings to empty strings.
650   </para>
651
652 </refsect1>
653
654 <refsect1 id="sql-copy-file-formats" xreflabel="File Formats">
655  <title>File Formats</title>
656
657  <refsect2 id="sql-copy-text-format" xreflabel="Text Format">
658   <title>Text Format</title>
659
660   <para>
661    When the <literal>text</literal> format is used,
662    the data read or written is a text file with one line per table row.
663    Columns in a row are separated by the delimiter character.
664    The column values themselves are strings generated by the
665    output function, or acceptable to the input function, of each
666    attribute's data type.  The specified null string is used in
667    place of columns that are null.
668    <command>COPY FROM</command> will raise an error if any line of the
669    input file contains more or fewer columns than are expected.
670   </para>
671
672   <para>
673    End of data can be represented by a line containing just
674    backslash-period (<literal>\.</literal>).  An end-of-data marker is
675    not necessary when reading from a file, since the end of file
676    serves perfectly well; in that context this provision exists only for
677    backward compatibility.  However, <application>psql</application>
678    uses <literal>\.</literal> to terminate a <literal>COPY FROM
679    STDIN</literal> operation (that is, reading
680    in-line <command>COPY</command> data in an SQL script).  In that
681    context the rule is needed to be able to end the operation before the
682    end of the script.
683   </para>
684
685   <para>
686    Backslash characters (<literal>\</literal>) can be used in the
687    <command>COPY</command> data to quote data characters that might
688    otherwise be taken as row or column delimiters. In particular, the
689    following characters <emphasis>must</emphasis> be preceded by a backslash if
690    they appear as part of a column value: backslash itself,
691    newline, carriage return, and the current delimiter character.
692   </para>
693
694   <para>
695    The specified null string is sent by <command>COPY TO</command> without
696    adding any backslashes; conversely, <command>COPY FROM</command> matches
697    the input against the null string before removing backslashes.  Therefore,
698    a null string such as <literal>\N</literal> cannot be confused with
699    the actual data value <literal>\N</literal> (which would be represented
700    as <literal>\\N</literal>).
701   </para>
702
703   <para>
704    The following special backslash sequences are recognized by
705    <command>COPY FROM</command>:
706
707   <informaltable>
708    <tgroup cols="2">
709     <thead>
710      <row>
711       <entry>Sequence</entry>
712       <entry>Represents</entry>
713      </row>
714     </thead>
715
716     <tbody>
717      <row>
718       <entry><literal>\b</literal></entry>
719       <entry>Backspace (ASCII 8)</entry>
720      </row>
721      <row>
722       <entry><literal>\f</literal></entry>
723       <entry>Form feed (ASCII 12)</entry>
724      </row>
725      <row>
726       <entry><literal>\n</literal></entry>
727       <entry>Newline (ASCII 10)</entry>
728      </row>
729      <row>
730       <entry><literal>\r</literal></entry>
731       <entry>Carriage return (ASCII 13)</entry>
732      </row>
733      <row>
734       <entry><literal>\t</literal></entry>
735       <entry>Tab (ASCII 9)</entry>
736      </row>
737      <row>
738       <entry><literal>\v</literal></entry>
739       <entry>Vertical tab (ASCII 11)</entry>
740      </row>
741      <row>
742       <entry><literal>\</literal><replaceable>digits</replaceable></entry>
743       <entry>Backslash followed by one to three octal digits specifies
744       the byte with that numeric code</entry>
745      </row>
746      <row>
747       <entry><literal>\x</literal><replaceable>digits</replaceable></entry>
748       <entry>Backslash <literal>x</literal> followed by one or two hex digits specifies
749       the byte with that numeric code</entry>
750      </row>
751     </tbody>
752    </tgroup>
753   </informaltable>
754
755    Presently, <command>COPY TO</command> will never emit an octal or
756    hex-digits backslash sequence, but it does use the other sequences
757    listed above for those control characters.
758   </para>
759
760   <para>
761    Any other backslashed character that is not mentioned in the above table
762    will be taken to represent itself.  However, beware of adding backslashes
763    unnecessarily, since that might accidentally produce a string matching the
764    end-of-data marker (<literal>\.</literal>) or the null string (<literal>\N</literal> by
765    default).  These strings will be recognized before any other backslash
766    processing is done.
767   </para>
768
769   <para>
770    It is strongly recommended that applications generating <command>COPY</command> data convert
771    data newlines and carriage returns to the <literal>\n</literal> and
772    <literal>\r</literal> sequences respectively.  At present it is
773    possible to represent a data carriage return by a backslash and carriage
774    return, and to represent a data newline by a backslash and newline.
775    However, these representations might not be accepted in future releases.
776    They are also highly vulnerable to corruption if the <command>COPY</command> file is
777    transferred across different machines (for example, from Unix to Windows
778    or vice versa).
779   </para>
780
781   <para>
782     All backslash sequences are interpreted after encoding conversion.
783     The bytes specified with the octal and hex-digit backslash sequences must
784     form valid characters in the database encoding.
785   </para>
786
787   <para>
788    <command>COPY TO</command> will terminate each row with a Unix-style
789    newline (<quote><literal>\n</literal></quote>).  Servers running on Microsoft Windows instead
790    output carriage return/newline (<quote><literal>\r\n</literal></quote>), but only for
791    <command>COPY</command> to a server file; for consistency across platforms,
792    <command>COPY TO STDOUT</command> always sends <quote><literal>\n</literal></quote>
793    regardless of server platform.
794    <command>COPY FROM</command> can handle lines ending with newlines,
795    carriage returns, or carriage return/newlines.  To reduce the risk of
796    error due to un-backslashed newlines or carriage returns that were
797    meant as data, <command>COPY FROM</command> will complain if the line
798    endings in the input are not all alike.
799   </para>
800  </refsect2>
801
802  <refsect2 id="sql-copy-csv-format" xreflabel="CSV Format">
803   <title>CSV Format</title>
804
805   <para>
806    This format option is used for importing and exporting the Comma-
807    Separated Value (<literal>CSV</literal>) file format used by many other
808    programs, such as spreadsheets. Instead of the escaping rules used by
809    <productname>PostgreSQL</productname>'s standard text format, it
810    produces and recognizes the common <literal>CSV</literal> escaping mechanism.
811   </para>
812
813   <para>
814    The values in each record are separated by the <literal>DELIMITER</literal>
815    character. If the value contains the delimiter character, the
816    <literal>QUOTE</literal> character, the <literal>NULL</literal> string, a carriage
817    return, or line feed character, then the whole value is prefixed and
818    suffixed by the <literal>QUOTE</literal> character, and any occurrence
819    within the value of a <literal>QUOTE</literal> character or the
820    <literal>ESCAPE</literal> character is preceded by the escape character.
821    You can also use <literal>FORCE_QUOTE</literal> to force quotes when outputting
822    non-<literal>NULL</literal> values in specific columns.
823   </para>
824
825   <para>
826    The <literal>CSV</literal> format has no standard way to distinguish a
827    <literal>NULL</literal> value from an empty string.
828    <productname>PostgreSQL</productname>'s <command>COPY</command> handles this by quoting.
829    A <literal>NULL</literal> is output as the <literal>NULL</literal> parameter string
830    and is not quoted, while a non-<literal>NULL</literal> value matching the
831    <literal>NULL</literal> parameter string is quoted.  For example, with the
832    default settings, a <literal>NULL</literal> is written as an unquoted empty
833    string, while an empty string data value is written with double quotes
834    (<literal>""</literal>). Reading values follows similar rules. You can
835    use <literal>FORCE_NOT_NULL</literal> to prevent <literal>NULL</literal> input
836    comparisons for specific columns. You can also use
837    <literal>FORCE_NULL</literal> to convert quoted null string data values to
838    <literal>NULL</literal>.
839   </para>
840
841   <para>
842    Because backslash is not a special character in the <literal>CSV</literal>
843    format, the end-of-data marker used in text mode (<literal>\.</literal>)
844    is not normally treated as special when reading <literal>CSV</literal>
845    data.  An exception is that <application>psql</application> will terminate
846    a <literal>COPY FROM STDIN</literal> operation (that is, reading
847    in-line <command>COPY</command> data in an SQL script) at a line containing
848    only <literal>\.</literal>, whether it is text or <literal>CSV</literal>
849    mode.
850   </para>
851
852   <note>
853    <para>
854     <productname>PostgreSQL</productname> versions before v18 always
855     recognized unquoted <literal>\.</literal> as an end-of-data marker,
856     even when reading from a separate file.  For compatibility with older
857     versions, <command>COPY TO</command> will quote <literal>\.</literal>
858     when it's alone on a line, even though this is no longer necessary.
859    </para>
860   </note>
861
862   <note>
863    <para>
864     In <literal>CSV</literal> format, all characters are significant. A quoted value
865     surrounded by white space, or any characters other than
866     <literal>DELIMITER</literal>, will include those characters. This can cause
867     errors if you import data from a system that pads <literal>CSV</literal>
868     lines with white space out to some fixed width. If such a situation
869     arises you might need to preprocess the <literal>CSV</literal> file to remove
870     the trailing white space, before importing the data into
871     <productname>PostgreSQL</productname>.
872    </para>
873   </note>
874
875   <note>
876    <para>
877     <literal>CSV</literal> format will both recognize and produce <literal>CSV</literal> files with quoted
878     values containing embedded carriage returns and line feeds. Thus
879     the files are not strictly one line per table row like text-format
880     files.
881    </para>
882   </note>
883
884   <note>
885    <para>
886     Many programs produce strange and occasionally perverse <literal>CSV</literal> files,
887     so the file format is more a convention than a standard. Thus you
888     might encounter some files that cannot be imported using this
889     mechanism, and <command>COPY</command> might produce files that other
890     programs cannot process.
891    </para>
892   </note>
893
894  </refsect2>
895
896  <refsect2 id="sql-copy-binary-format" xreflabel="Binary Format">
897   <title>Binary Format</title>
898
899   <para>
900    The <literal>binary</literal> format option causes all data to be
901    stored/read as binary format rather than as text.  It is
902    somewhat faster than the text and <literal>CSV</literal> formats,
903    but a binary-format file is less portable across machine architectures and
904    <productname>PostgreSQL</productname> versions.
905    Also, the binary format is very data type specific; for example
906    it will not work to output binary data from a <type>smallint</type> column
907    and read it into an <type>integer</type> column, even though that would work
908    fine in text format.
909   </para>
910
911   <para>
912    The <literal>binary</literal> file format consists
913    of a file header, zero or more tuples containing the row data, and
914    a file trailer.  Headers and data are in network byte order.
915   </para>
916
917   <note>
918    <para>
919     <productname>PostgreSQL</productname> releases before 7.4 used a
920     different binary file format.
921    </para>
922   </note>
923
924   <refsect3>
925    <title>File Header</title>
926
927    <para>
928     The file header consists of 15 bytes of fixed fields, followed
929     by a variable-length header extension area.  The fixed fields are:
930
931    <variablelist>
932     <varlistentry>
933      <term>Signature</term>
934      <listitem>
935       <para>
93611-byte sequence <literal>PGCOPY\n\377\r\n\0</literal> &mdash; note that the zero byte
937is a required part of the signature.  (The signature is designed to allow
938easy identification of files that have been munged by a non-8-bit-clean
939transfer.  This signature will be changed by end-of-line-translation
940filters, dropped zero bytes, dropped high bits, or parity changes.)
941       </para>
942      </listitem>
943     </varlistentry>
944
945     <varlistentry>
946      <term>Flags field</term>
947      <listitem>
948       <para>
94932-bit integer bit mask to denote important aspects of the file format. Bits
950are numbered from 0 (<acronym>LSB</acronym>) to 31 (<acronym>MSB</acronym>).  Note that
951this field is stored in network byte order (most significant byte first),
952as are all the integer fields used in the file format.  Bits
95316&ndash;31 are reserved to denote critical file format issues; a reader
954should abort if it finds an unexpected bit set in this range. Bits 0&ndash;15
955are reserved to signal backwards-compatible format issues; a reader
956should simply ignore any unexpected bits set in this range. Currently
957only one flag bit is defined, and the rest must be zero:
958        <variablelist>
959         <varlistentry>
960          <term>Bit 16</term>
961          <listitem>
962           <para>
963            If 1, OIDs are included in the data; if 0, not. Oid system columns
964            are not supported in <productname>PostgreSQL</productname>
965            anymore, but the format still contains the indicator.
966           </para>
967          </listitem>
968         </varlistentry>
969        </variablelist></para>
970      </listitem>
971     </varlistentry>
972
973     <varlistentry>
974      <term>Header extension area length</term>
975      <listitem>
976       <para>
97732-bit integer, length in bytes of remainder of header, not including self.
978Currently, this is zero, and the first tuple follows
979immediately.  Future changes to the format might allow additional data
980to be present in the header.  A reader should silently skip over any header
981extension data it does not know what to do with.
982       </para>
983      </listitem>
984     </varlistentry>
985    </variablelist>
986    </para>
987
988    <para>
989The header extension area is envisioned to contain a sequence of
990self-identifying chunks.  The flags field is not intended to tell readers
991what is in the extension area.  Specific design of header extension contents
992is left for a later release.
993    </para>
994
995    <para>
996     This design allows for both backwards-compatible header additions (add
997     header extension chunks, or set low-order flag bits) and
998     non-backwards-compatible changes (set high-order flag bits to signal such
999     changes, and add supporting data to the extension area if needed).
1000    </para>
1001   </refsect3>
1002
1003   <refsect3>
1004    <title>Tuples</title>
1005    <para>
1006Each tuple begins with a 16-bit integer count of the number of fields in the
1007tuple.  (Presently, all tuples in a table will have the same count, but that
1008might not always be true.)  Then, repeated for each field in the tuple, there
1009is a 32-bit length word followed by that many bytes of field data.  (The
1010length word does not include itself, and can be zero.)  As a special case,
1011-1 indicates a NULL field value.  No value bytes follow in the NULL case.
1012    </para>
1013
1014    <para>
1015There is no alignment padding or any other extra data between fields.
1016    </para>
1017
1018    <para>
1019Presently, all data values in a binary-format file are
1020assumed to be in binary format (format code one).  It is anticipated that a
1021future extension might add a header field that allows per-column format codes
1022to be specified.
1023    </para>
1024
1025    <para>
1026To determine the appropriate binary format for the actual tuple data you
1027should consult the <productname>PostgreSQL</productname> source, in
1028particular the <function>*send</function> and <function>*recv</function> functions for
1029each column's data type (typically these functions are found in the
1030<filename>src/backend/utils/adt/</filename> directory of the source
1031distribution).
1032    </para>
1033
1034    <para>
1035If OIDs are included in the file, the OID field immediately follows the
1036field-count word.  It is a normal field except that it's not included in the
1037field-count.  Note that oid system columns are not supported in current
1038versions of <productname>PostgreSQL</productname>.
1039    </para>
1040   </refsect3>
1041
1042   <refsect3>
1043    <title>File Trailer</title>
1044
1045    <para>
1046     The file trailer consists of a 16-bit integer word containing -1.  This
1047     is easily distinguished from a tuple's field-count word.
1048    </para>
1049
1050    <para>
1051     A reader should report an error if a field-count word is neither -1
1052     nor the expected number of columns.  This provides an extra
1053     check against somehow getting out of sync with the data.
1054    </para>
1055   </refsect3>
