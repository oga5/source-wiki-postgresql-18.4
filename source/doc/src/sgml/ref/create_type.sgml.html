65 <refsect1>
66  <title>Description</title>
67
68  <para>
69   <command>CREATE TYPE</command> registers a new data type for use in
70   the current database.  The user who defines a type becomes its
71   owner.
72  </para>
73
74  <para>
75   If a schema name is given then the type is created in the specified
76   schema.  Otherwise it is created in the current schema.  The type
77   name must be distinct from the name of any existing type or domain
78   in the same schema.  (Because tables have associated data types,
79   the type name must also be distinct from the name of any existing
80   table in the same schema.)
81  </para>
82
83  <para>
84   There are five forms of <command>CREATE TYPE</command>, as shown in the
85   syntax synopsis above.  They respectively create a <firstterm>composite
86   type</firstterm>, an <firstterm>enum type</firstterm>, a <firstterm>range type</firstterm>, a
87   <firstterm>base type</firstterm>, or a <firstterm>shell type</firstterm>.  The first four
88   of these are discussed in turn below.  A shell type is simply a placeholder
89   for a type to be defined later; it is created by issuing <command>CREATE
90   TYPE</command> with no parameters except for the type name.  Shell types
91   are needed as forward references when creating range types and base types,
92   as discussed in those sections.
93  </para>
94
95  <refsect2>
96   <title>Composite Types</title>
97
98  <para>
99   The first form of <command>CREATE TYPE</command>
100   creates a composite type.
101   The composite type is specified by a list of attribute names and data types.
102   An attribute's collation can be specified too, if its data type is
103   collatable.  A composite type is essentially the same as the row type
104   of a table, but using <command>CREATE TYPE</command> avoids the need to
105   create an actual table when all that is wanted is to define a type.
106   A stand-alone composite type is useful, for example, as the argument or
107   return type of a function.
108  </para>
109
110  <para>
111   To be able to create a composite type, you must
112   have <literal>USAGE</literal> privilege on all attribute types.
113  </para>
114  </refsect2>
115
116  <refsect2 id="sql-createtype-enum">
117   <title>Enumerated Types</title>
118
119   <para>
120    The second form of <command>CREATE TYPE</command> creates an enumerated
121    (enum) type, as described in <xref linkend="datatype-enum"/>.
122    Enum types take a list of quoted labels, each of which
123    must be less than <symbol>NAMEDATALEN</symbol> bytes long (64 bytes in a
124    standard <productname>PostgreSQL</productname> build).  (It is possible to
125    create an enumerated type with zero labels, but such a type cannot be used
126    to hold values before at least one label is added using <link
127    linkend="sql-altertype"><command>ALTER TYPE</command></link>.)
128   </para>
129  </refsect2>
130
131  <refsect2 id="sql-createtype-range">
132   <title>Range Types</title>
133
134   <para>
135    The third form of <command>CREATE TYPE</command> creates a new
136    range type, as described in <xref linkend="rangetypes"/>.
137   </para>
138
139   <para>
140    The range type's <replaceable class="parameter">subtype</replaceable> can
141    be any type with an associated b-tree operator class (to determine the
142    ordering of values for the range type).  Normally the subtype's default
143    b-tree operator class is used to determine ordering; to use a non-default
144    operator class, specify its name with <replaceable
145    class="parameter">subtype_opclass</replaceable>.  If the subtype is
146    collatable, and you want to use a non-default collation in the range's
147    ordering, specify the desired collation with the <replaceable
148    class="parameter">collation</replaceable> option.
149   </para>
150
151   <para>
152    The optional <replaceable class="parameter">canonical</replaceable>
153    function must take one argument of the range type being defined, and
154    return a value of the same type.  This is used to convert range values
155    to a canonical form, when applicable.  See <xref
156    linkend="rangetypes-defining"/> for more information.  Creating a
157    <replaceable class="parameter">canonical</replaceable> function
158    is a bit tricky, since it must be defined before the range type can be
159    declared.  To do this, you must first create a shell type, which is a
160    placeholder type that has no properties except a name and an
161    owner.  This is done by issuing the command <literal>CREATE TYPE
162    <replaceable>name</replaceable></literal>, with no additional parameters.  Then
163    the function can be declared using the shell type as argument and result,
164    and finally the range type can be declared using the same name.  This
165    automatically replaces the shell type entry with a valid range type.
166   </para>
167
168   <para>
169    The optional <replaceable class="parameter">subtype_diff</replaceable>
170    function must take two values of the
171    <replaceable class="parameter">subtype</replaceable> type as argument,
172    and return a <type>double precision</type> value representing the
173    difference between the two given values.  While this is optional,
174    providing it allows much greater efficiency of GiST indexes on columns of
175    the range type.  See <xref linkend="rangetypes-defining"/> for more
176    information.
177   </para>
178
179   <para>
180    The optional <replaceable class="parameter">multirange_type_name</replaceable>
181    parameter specifies the name of the corresponding multirange type.  If not
182    specified, this name is chosen automatically as follows.
183    If the range type name contains the substring <literal>range</literal>, then
184    the multirange type name is formed by replacement of the <literal>range</literal>
185    substring with <literal>multirange</literal> in the range
186    type name.  Otherwise, the multirange type name is formed by appending a
187    <literal>_multirange</literal> suffix to the range type name.
188   </para>
189  </refsect2>
190
191  <refsect2>
192   <title>Base Types</title>
193
194  <para>
195   The fourth form of <command>CREATE TYPE</command> creates a new base type
196   (scalar type).  To create a new base type, you must be a superuser.
197   (This restriction is made because an erroneous type definition could
198   confuse or even crash the server.)
199  </para>
200
201  <para>
202   The parameters can appear in any order, not only that
203   illustrated above, and most are optional.  You must register
204   two or more functions (using <command>CREATE FUNCTION</command>) before
205   defining the type.  The support functions
206   <replaceable class="parameter">input_function</replaceable> and
207   <replaceable class="parameter">output_function</replaceable>
208   are required, while the functions
209   <replaceable class="parameter">receive_function</replaceable>,
210   <replaceable class="parameter">send_function</replaceable>,
211   <replaceable class="parameter">type_modifier_input_function</replaceable>,
212   <replaceable class="parameter">type_modifier_output_function</replaceable>,
213   <replaceable class="parameter">analyze_function</replaceable>, and
214   <replaceable class="parameter">subscript_function</replaceable>
215   are optional.  Generally these functions have to be coded in C
216   or another low-level language.
217  </para>
218
219  <para>
220   The <replaceable class="parameter">input_function</replaceable>
221   converts the type's external textual representation to the internal
222   representation used by the operators and functions defined for the type.
223   <replaceable class="parameter">output_function</replaceable>
224   performs the reverse transformation.  The input function can be
225   declared as taking one argument of type <type>cstring</type>,
226   or as taking three arguments of types
227   <type>cstring</type>, <type>oid</type>, <type>integer</type>.
228   The first argument is the input text as a C string, the second
229   argument is the type's own OID (except for array types, which instead
230   receive their element type's OID),
231   and the third is the <literal>typmod</literal> of the destination column, if known
232   (-1 will be passed if not).
233   The input function must return a value of the data type itself.
234   Usually, an input function should be declared STRICT; if it is not,
235   it will be called with a NULL first parameter when reading a NULL
236   input value.  The function must still return NULL in this case, unless
237   it raises an error.
238   (This case is mainly meant to support domain input functions, which
239   might need to reject NULL inputs.)
240   The output function must be
241   declared as taking one argument of the new data type.
242   The output function must return type <type>cstring</type>.
243   Output functions are not invoked for NULL values.
244  </para>
245
246  <para>
247   The optional <replaceable class="parameter">receive_function</replaceable>
248   converts the type's external binary representation to the internal
249   representation.  If this function is not supplied, the type cannot
250   participate in binary input.  The binary representation should be
251   chosen to be cheap to convert to internal form, while being reasonably
252   portable.  (For example, the standard integer data types use network
253   byte order as the external binary representation, while the internal
254   representation is in the machine's native byte order.)  The receive
255   function should perform adequate checking to ensure that the value is
256   valid.
257   The receive function can be declared as taking one argument of type
258   <type>internal</type>, or as taking three arguments of types
259   <type>internal</type>, <type>oid</type>, <type>integer</type>.
260   The first argument is a pointer to a <type>StringInfo</type> buffer
261   holding the received byte string; the optional arguments are the
262   same as for the text input function.
263   The receive function must return a value of the data type itself.
264   Usually, a receive function should be declared STRICT; if it is not,
265   it will be called with a NULL first parameter when reading a NULL
266   input value.  The function must still return NULL in this case, unless
267   it raises an error.
268   (This case is mainly meant to support domain receive functions, which
269   might need to reject NULL inputs.)
270   Similarly, the optional
271   <replaceable class="parameter">send_function</replaceable> converts
272   from the internal representation to the external binary representation.
273   If this function is not supplied, the type cannot participate in binary
274   output.  The send function must be
275   declared as taking one argument of the new data type.
276   The send function must return type <type>bytea</type>.
277   Send functions are not invoked for NULL values.
278  </para>
279
280  <para>
281   You should at this point be wondering how the input and output functions
282   can be declared to have results or arguments of the new type, when they
283   have to be created before the new type can be created.  The answer is that
284   the type should first be defined as a <firstterm>shell type</firstterm>, which is a
285   placeholder type that has no properties except a name and an owner.  This
286   is done by issuing the command <literal>CREATE TYPE
287   <replaceable>name</replaceable></literal>, with no additional parameters.  Then the
288   C I/O functions can be defined referencing the shell type.  Finally,
289   <command>CREATE TYPE</command> with a full definition replaces the shell entry
290   with a complete, valid type definition, after which the new type can be
291   used normally.
292  </para>
293
294  <para>
295   The optional
296   <replaceable class="parameter">type_modifier_input_function</replaceable>
297   and <replaceable class="parameter">type_modifier_output_function</replaceable>
298   are needed if the type supports modifiers, that is optional constraints
299   attached to a type declaration, such as <literal>char(5)</literal> or
300   <literal>numeric(30,2)</literal>.  <productname>PostgreSQL</productname> allows
301   user-defined types to take one or more simple constants or identifiers as
302   modifiers.  However, this information must be capable of being packed into a
303   single non-negative integer value for storage in the system catalogs.  The
304   <replaceable class="parameter">type_modifier_input_function</replaceable>
305   is passed the declared modifier(s) in the form of a <type>cstring</type>
306   array.  It must check the values for validity (throwing an error if they
307   are wrong), and if they are correct, return a single non-negative
308   <type>integer</type> value that will be stored as the column <quote>typmod</quote>.
309   Type modifiers will be rejected if the type does not have a
310   <replaceable class="parameter">type_modifier_input_function</replaceable>.
311   The <replaceable class="parameter">type_modifier_output_function</replaceable>
312   converts the internal integer typmod value back to the correct form for
313   user display.  It must return a <type>cstring</type> value that is the exact
314   string to append to the type name; for example <type>numeric</type>'s
315   function might return <literal>(30,2)</literal>.
316   It is allowed to omit the
317   <replaceable class="parameter">type_modifier_output_function</replaceable>,
318   in which case the default display format is just the stored typmod integer
319   value enclosed in parentheses.
320  </para>
321
322  <para>
323   The optional <replaceable class="parameter">analyze_function</replaceable>
324   performs type-specific statistics collection for columns of the data type.
325   By default, <command>ANALYZE</command> will attempt to gather statistics using
326   the type's <quote>equals</quote> and <quote>less-than</quote> operators, if there
327   is a default b-tree operator class for the type.  For non-scalar types
328   this behavior is likely to be unsuitable, so it can be overridden by
329   specifying a custom analysis function.  The analysis function must be
330   declared to take a single argument of type <type>internal</type>, and return
331   a <type>boolean</type> result.  The detailed API for analysis functions appears
332   in <filename>src/include/commands/vacuum.h</filename>.
333  </para>
334
335  <para>
336   The optional <replaceable class="parameter">subscript_function</replaceable>
337   allows the data type to be subscripted in SQL commands.  Specifying this
338   function does not cause the type to be considered a <quote>true</quote>
339   array type; for example, it will not be a candidate for the result type
340   of <literal>ARRAY[]</literal> constructs.  But if subscripting a value
341   of the type is a natural notation for extracting data from it, then
342   a <replaceable class="parameter">subscript_function</replaceable> can
343   be written to define what that means.  The subscript function must be
344   declared to take a single argument of type <type>internal</type>, and
345   return an <type>internal</type> result, which is a pointer to a struct
346   of methods (functions) that implement subscripting.
347   The detailed API for subscript functions appears
348   in <filename>src/include/nodes/subscripting.h</filename>.
349   It may also be useful to read the array implementation
350   in <filename>src/backend/utils/adt/arraysubs.c</filename>,
351   or the simpler code
352   in <filename>contrib/hstore/hstore_subs.c</filename>.
353   Additional information appears in
354   <xref linkend="sql-createtype-array"/> below.
355  </para>
356
357  <para>
358   While the details of the new type's internal representation are only
359   known to the I/O functions and other functions you create to work with
360   the type, there are several properties of the internal representation
361   that must be declared to <productname>PostgreSQL</productname>.
362   Foremost of these is
363   <replaceable class="parameter">internallength</replaceable>.
364   Base data types can be fixed-length, in which case
365   <replaceable class="parameter">internallength</replaceable> is a
366   positive integer, or variable-length, indicated by setting
367   <replaceable class="parameter">internallength</replaceable>
368   to <literal>VARIABLE</literal>.  (Internally, this is represented
369   by setting <literal>typlen</literal> to -1.)  The internal representation of all
370   variable-length types must start with a 4-byte integer giving the total
371   length of this value of the type.  (Note that the length field is often
372   encoded, as described in <xref linkend="storage-toast"/>; it's unwise
373   to access it directly.)
374  </para>
375
376  <para>
377   The optional flag <literal>PASSEDBYVALUE</literal> indicates that
378   values of this data type are passed by value, rather than by
379   reference.  Types passed by value must be fixed-length, and their internal
380   representation cannot be larger than the size of the <type>Datum</type> type
381   (4 bytes on some machines, 8 bytes on others).
382  </para>
383
384  <para>
385   The <replaceable class="parameter">alignment</replaceable> parameter
386   specifies the storage alignment required for the data type.  The
387   allowed values equate to alignment on 1, 2, 4, or 8 byte boundaries.
388   Note that variable-length types must have an alignment of at least
389   4, since they necessarily contain an <type>int4</type> as their first component.
390  </para>
391
392  <para>
393   The <replaceable class="parameter">storage</replaceable> parameter
394   allows selection of storage strategies for variable-length data
395   types.  (Only <literal>plain</literal> is allowed for fixed-length
396   types.)  <literal>plain</literal> specifies that data of the type
397   will always be stored in-line and not compressed.
398   <literal>extended</literal> specifies that the system will first
399   try to compress a long data value, and will move the value out of
400   the main table row if it's still too long.
401   <literal>external</literal> allows the value to be moved out of the
402   main table, but the system will not try to compress it.
403   <literal>main</literal> allows compression, but discourages moving
404   the value out of the main table.  (Data items with this storage
405   strategy might still be moved out of the main table if there is no
406   other way to make a row fit, but they will be kept in the main
407   table preferentially over <literal>extended</literal> and
408   <literal>external</literal> items.)
409  </para>
410
411  <para>
412   All <replaceable class="parameter">storage</replaceable> values other
413   than <literal>plain</literal> imply that the functions of the data type
414   can handle values that have been <firstterm>toasted</firstterm>, as described
415   in <xref linkend="storage-toast"/> and <xref linkend="xtypes-toast"/>.
416   The specific other value given merely determines the default TOAST
417   storage strategy for columns of a toastable data type; users can pick
418   other strategies for individual columns using <literal>ALTER TABLE
419   SET STORAGE</literal>.
420  </para>
421
422  <para>
423   The <replaceable class="parameter">like_type</replaceable> parameter
424   provides an alternative method for specifying the basic representation
425   properties of a data type: copy them from some existing type. The values of
426   <replaceable class="parameter">internallength</replaceable>,
427   <replaceable class="parameter">passedbyvalue</replaceable>,
428   <replaceable class="parameter">alignment</replaceable>, and
429   <replaceable class="parameter">storage</replaceable> are copied from the
430   named type.  (It is possible, though usually undesirable, to override
431   some of these values by specifying them along with the <literal>LIKE</literal>
432   clause.)  Specifying representation this way is especially useful when
433   the low-level implementation of the new type <quote>piggybacks</quote> on an
434   existing type in some fashion.
435  </para>
436
437  <para>
438   The <replaceable class="parameter">category</replaceable> and
439   <replaceable class="parameter">preferred</replaceable> parameters can be
440   used to help control which implicit cast will be applied in ambiguous
441   situations.  Each data type belongs to a category named by a single ASCII
442   character, and each type is either <quote>preferred</quote> or not within its
443   category.  The parser will prefer casting to preferred types (but only from
444   other types within the same category) when this rule is helpful in
445   resolving overloaded functions or operators.  For more details see <xref
446   linkend="typeconv"/>.  For types that have no implicit casts to or from any
447   other types, it is sufficient to leave these settings at the defaults.
448   However, for a group of related types that have implicit casts, it is often
449   helpful to mark them all as belonging to a category and select one or two
450   of the <quote>most general</quote> types as being preferred within the category.
451   The <replaceable class="parameter">category</replaceable> parameter is
452   especially useful when adding a user-defined type to an existing built-in
453   category, such as the numeric or string types.  However, it is also
454   possible to create new entirely-user-defined type categories.  Select any
455   ASCII character other than an upper-case letter to name such a category.
456  </para>
457
458  <para>
459   A default value can be specified, in case a user wants columns of the
460   data type to default to something other than the null value.
461   Specify the default with the <literal>DEFAULT</literal> key word.
462   (Such a default can be overridden by an explicit <literal>DEFAULT</literal>
463   clause attached to a particular column.)
464  </para>
465
466  <para>
467   To indicate that a type is a fixed-length array type,
468   specify the type of the array
469   elements using the <literal>ELEMENT</literal> key word.  For example, to
470   define an array of 4-byte integers (<type>int4</type>), specify
471   <literal>ELEMENT = int4</literal>.  For more details,
472   see <xref linkend="sql-createtype-array"/> below.
473  </para>
474
475  <para>
476   To indicate the delimiter to be used between values in the external
477   representation of arrays of this type, <replaceable
478   class="parameter">delimiter</replaceable> can be
479   set to a specific character.  The default delimiter is the comma
480   (<literal>,</literal>).  Note that the delimiter is associated
481   with the array element type, not the array type itself.
482  </para>
483
484  <para>
485   If the optional Boolean
486   parameter <replaceable class="parameter">collatable</replaceable>
487   is true, column definitions and expressions of the type may carry
488   collation information through use of
489   the <literal>COLLATE</literal> clause.  It is up to the
490   implementations of the functions operating on the type to actually
491   make use of the collation information; this does not happen
492   automatically merely by marking the type collatable.
493  </para>
494  </refsect2>

496  <refsect2 id="sql-createtype-array" xreflabel="Array Types">
497   <title>Array Types</title>
498
499   <para>
500    Whenever a user-defined type is created,
501    <productname>PostgreSQL</productname> automatically creates an
502    associated array type, whose name consists of the element type's
503    name prepended with an underscore, and truncated if necessary to keep
504    it less than <symbol>NAMEDATALEN</symbol> bytes long.  (If the name
505    so generated collides with an existing type name, the process is
506    repeated until a non-colliding name is found.)
507    This implicitly-created array type is variable length and uses the
508    built-in input and output functions <literal>array_in</literal> and
509    <literal>array_out</literal>.  Furthermore, this type is what the system
510    uses for constructs such as <literal>ARRAY[]</literal> over the
511    user-defined type.  The array type tracks any changes in its
512    element type's owner or schema, and is dropped if the element type is.
513   </para>
514
515   <para>
516    You might reasonably ask why there is an <option>ELEMENT</option>
517    option, if the system makes the correct array type automatically.
518    The main case where it's useful to use <option>ELEMENT</option> is when you are
519    making a fixed-length type that happens to be internally an array of a number of
520    identical things, and you want to allow these things to be accessed
521    directly by subscripting, in addition to whatever operations you plan
522    to provide for the type as a whole.  For example, type <type>point</type>
523    is represented as just two floating-point numbers, which can be accessed
524    using <literal>point[0]</literal> and <literal>point[1]</literal>.
525    Note that
526    this facility only works for fixed-length types whose internal form
527    is exactly a sequence of identical fixed-length fields.
528    For historical reasons (i.e., this is clearly wrong but it's far too
529    late to change it), subscripting of fixed-length array types starts from
530    zero, rather than from one as for variable-length arrays.
531   </para>
532
533   <para>
534    Specifying the <option>SUBSCRIPT</option> option allows a data type to
535    be subscripted, even though the system does not otherwise regard it as
536    an array type.  The behavior just described for fixed-length arrays is
537    actually implemented by the <option>SUBSCRIPT</option> handler
538    function <function>raw_array_subscript_handler</function>, which is
539    used automatically if you specify <option>ELEMENT</option> for a
540    fixed-length type without also writing <option>SUBSCRIPT</option>.
541   </para>
542
543   <para>
544    When specifying a custom <option>SUBSCRIPT</option> function, it is
545    not necessary to specify <option>ELEMENT</option> unless
546    the <option>SUBSCRIPT</option> handler function needs to
547    consult <structfield>typelem</structfield> to find out what to return.
548    Be aware that specifying <option>ELEMENT</option> causes the system to
549    assume that the new type contains, or is somehow physically dependent on,
550    the element type; thus for example changing properties of the element
551    type won't be allowed if there are any columns of the dependent type.
552   </para>
553  </refsect2>

852 <refsect1 id="sql-createtype-notes">
853  <title>Notes</title>
854
855  <para>
856   Because there are no restrictions on use of a data type once it's been
857   created, creating a base type or range type is tantamount to granting
858   public execute permission on the functions mentioned in the type definition.
859   This is usually
860   not an issue for the sorts of functions that are useful in a type
861   definition.  But you might want to think twice before designing a type
862   in a way that would require <quote>secret</quote> information to be used
863   while converting it to or from external form.
864  </para>
865
866  <para>
867   Before <productname>PostgreSQL</productname> version 8.3, the name of
868   a generated array type was always exactly the element type's name with one
869   underscore character (<literal>_</literal>) prepended.  (Type names were
870   therefore restricted in length to one fewer character than other names.)
871   While this is still usually the case, the array type name may vary from
872   this in case of maximum-length names or collisions with user type names
873   that begin with underscore.  Writing code that depends on this convention
874   is therefore deprecated.  Instead, use
875   <structname>pg_type</structname>.<structfield>typarray</structfield> to locate the array type
876   associated with a given type.
877  </para>
878
879  <para>
880   It may be advisable to avoid using type and table names that begin with
881   underscore.  While the server will change generated array type names to
882   avoid collisions with user-given names, there is still risk of confusion,
883   particularly with old client software that may assume that type names
884   beginning with underscores always represent arrays.
885  </para>
886
887  <para>
888   Before <productname>PostgreSQL</productname> version 8.2, the shell-type
889   creation syntax
890   <literal>CREATE TYPE <replaceable>name</replaceable></literal> did not exist.
891   The way to create a new base type was to create its input function first.
892   In this approach, <productname>PostgreSQL</productname> will first see
893   the name of the new data type as the return type of the input function.
894   The shell type is implicitly created in this situation, and then it
895   can be referenced in the definitions of the remaining I/O functions.
896   This approach still works, but is deprecated and might be disallowed in
897   some future release.  Also, to avoid accidentally cluttering
898   the catalogs with shell types as a result of simple typos in function
899   definitions, a shell type will only be made this way when the input
900   function is written in C.
901  </para>
902
903  <para>
904   In <productname>PostgreSQL</productname> version 16 and later,
905   it is desirable for base types' input functions to
906   return <quote>soft</quote> errors using the
907   new <function>errsave()</function>/<function>ereturn()</function>
908   mechanism, rather than throwing <function>ereport()</function>
909   exceptions as in previous versions.
910   See <filename>src/backend/utils/fmgr/README</filename> for more
911   information.
912  </para>

916 <refsect1>
917  <title>Examples</title>
918
919  <para>
920   This example creates a composite type and uses it in
921   a function definition:
922<programlisting>
923CREATE TYPE compfoo AS (f1 int, f2 text);
924
925CREATE FUNCTION getfoo() RETURNS SETOF compfoo AS $$
926    SELECT fooid, fooname FROM foo
927$$ LANGUAGE SQL;
928</programlisting>
929  </para>
930
931  <para>
932   This example creates an enumerated type and uses it in
933   a table definition:
934<programlisting>
935CREATE TYPE bug_status AS ENUM ('new', 'open', 'closed');
936
937CREATE TABLE bug (
938    id serial,
939    description text,
940    status bug_status
941);
942</programlisting>
943  </para>
944
945  <para>
946   This example creates a range type:
947<programlisting>
948CREATE TYPE float8_range AS RANGE (subtype = float8, subtype_diff = float8mi);
949</programlisting>
950  </para>
951
952  <para>
953   This example creates the base data type <type>box</type> and then uses the
954   type in a table definition:
955<programlisting>
956CREATE TYPE box;
957
958CREATE FUNCTION my_box_in_function(cstring) RETURNS box AS ... ;
959CREATE FUNCTION my_box_out_function(box) RETURNS cstring AS ... ;
960
961CREATE TYPE box (
962    INTERNALLENGTH = 16,
963    INPUT = my_box_in_function,
964    OUTPUT = my_box_out_function
965);
966
967CREATE TABLE myboxes (
968    id integer,
969    description box
970);
971</programlisting>
972  </para>
973
974  <para>
975   If the internal structure of <type>box</type> were an array of four
976   <type>float4</type> elements, we might instead use:
977<programlisting>
978CREATE TYPE box (
979    INTERNALLENGTH = 16,
980    INPUT = my_box_in_function,
981    OUTPUT = my_box_out_function,
982    ELEMENT = float4
983);
984</programlisting>
985   which would allow a box value's component numbers to be accessed
986   by subscripting.  Otherwise the type behaves the same as before.
987  </para>
988
989  <para>
990   This example creates a large object type and uses it in
991   a table definition:
992<programlisting>
993CREATE TYPE bigobj (
994    INPUT = lo_filein, OUTPUT = lo_fileout,
995    INTERNALLENGTH = VARIABLE
996);
997CREATE TABLE big_objs (
998    id integer,
999    obj bigobj
1000);
1001</programlisting>
1002  </para>
1003
1004  <para>
1005   More examples, including suitable input and output functions, are
1006   in <xref linkend="xtypes"/>.
1007  </para>

1010 <refsect1 id="sql-createtype-compatibility">
1011  <title>Compatibility</title>
1012
1013  <para>
1014   The first form of the <command>CREATE TYPE</command> command, which
1015   creates a composite type, conforms to the <acronym>SQL</acronym> standard.
1016   The other forms are <productname>PostgreSQL</productname>
1017   extensions.  The <command>CREATE TYPE</command> statement in
1018   the <acronym>SQL</acronym> standard also defines other forms that are not
1019   implemented in <productname>PostgreSQL</productname>.
1020  </para>
1021
1022  <para>
1023   The ability to create a composite type with zero attributes is
1024   a <productname>PostgreSQL</productname>-specific deviation from the
1025   standard (analogous to the same case in <command>CREATE TABLE</command>).
1026  </para>
