33 <refsect1>
34  <title>Description</title>
35
36    <para>
37     <application>psql</application> is a terminal-based front-end to
38     <productname>PostgreSQL</productname>. It enables you to type in
39     queries interactively, issue them to
40     <productname>PostgreSQL</productname>, and see the query results.
41     Alternatively, input can be from a file or from command line
42     arguments. In addition, <application>psql</application> provides a
43     number of meta-commands and various shell-like features to
44     facilitate writing scripts and automating a wide variety of tasks.
45    </para>
46 </refsect1>

89    <varlistentry id="app-psql-option-command">
90      <term><option>-c <replaceable class="parameter">command</replaceable></option></term>
91      <term><option>--command=<replaceable class="parameter">command</replaceable></option></term>
92      <listitem>
93      <para>
94       Specifies that <application>psql</application> is to execute the given
95       command string, <replaceable class="parameter">command</replaceable>.
96       This option can be repeated and combined in any order with
97       the <option>-f</option> option.  When either <option>-c</option>
98       or <option>-f</option> is specified, <application>psql</application>
99       does not read commands from standard input; instead it terminates
100       after processing all the <option>-c</option> and <option>-f</option>
101       options in sequence.
102      </para>
103      <para>
104       <replaceable class="parameter">command</replaceable> must be either
105       a command string that is completely parsable by the server (i.e.,
106       it contains no <application>psql</application>-specific features),
107       or a single backslash command. Thus you cannot mix
108       <acronym>SQL</acronym> and <application>psql</application>
109       meta-commands within a <option>-c</option> option. To achieve that,
110       you could use repeated <option>-c</option> options or pipe the string
111       into <application>psql</application>, for example:
112<programlisting>
113psql -c '\x' -c 'SELECT * FROM foo;'
114</programlisting>
115       or
116<programlisting>
117echo '\x \\ SELECT * FROM foo;' | psql
118</programlisting>
119       (<literal>\\</literal> is the separator meta-command.)
120      </para>
121      <para>
122       Each <acronym>SQL</acronym> command string passed
123       to <option>-c</option> is sent to the server as a single request.
124       Because of this, the server executes it as a single transaction even
125       if the string contains multiple <acronym>SQL</acronym> commands,
126       unless there are explicit <command>BEGIN</command>/<command>COMMIT</command>
127       commands included in the string to divide it into multiple
128       transactions.  (See <xref linkend="protocol-flow-multi-statement"/>
129       for more details about how the server handles multi-query strings.)
130      </para>
131      <para>
132       If having several commands executed in one transaction is not desired,
133       use repeated <option>-c</option> commands or feed multiple commands to
134       <application>psql</application>'s standard input,
135       either using <application>echo</application> as illustrated above, or
136       via a shell here-document, for example:
137<programlisting>
138psql &lt;&lt;EOF
139\x
140SELECT * FROM foo;
141EOF
142</programlisting></para>
143      </listitem>

198    <varlistentry id="app-psql-option-file">
199      <term><option>-f <replaceable class="parameter">filename</replaceable></option></term>
200      <term><option>--file=<replaceable class="parameter">filename</replaceable></option></term>
201      <listitem>
202      <para>
203       Read commands from the
204       file <replaceable class="parameter">filename</replaceable>,
205       rather than standard input.
206       This option can be repeated and combined in any order with
207       the <option>-c</option> option.  When either <option>-c</option>
208       or <option>-f</option> is specified, <application>psql</application>
209       does not read commands from standard input; instead it terminates
210       after processing all the <option>-c</option> and <option>-f</option>
211       options in sequence.
212       Except for that, this option is largely equivalent to the
213       meta-command <command>\i</command>.
214      </para>
215
216      <para>

220       interactive input with input from files.  Note however that Readline
221       is not used in this case (much as if <option>-n</option> had been
222       specified).
223      </para>
224
225      <para>
226      Using this option is subtly different from writing <literal>psql
227      &lt; <replaceable
228      class="parameter">filename</replaceable></literal>. In general,
229      both will do what you expect, but using <literal>-f</literal>
230      enables some nice features such as error messages with line
231      numbers. There is also a slight chance that using this option will
232      reduce the start-up overhead. On the other hand, the variant using
233      the shell's input redirection is (in theory) guaranteed to yield
234      exactly the same output you would have received had you entered
235      everything by hand.
236      </para>
237      </listitem>

577     <varlistentry id="app-psql-option-single-transaction">
578      <term><option>-1</option></term>
579      <term><option>--single-transaction</option></term>
580      <listitem>
581       <para>
582        This option can only be used in combination with one or more
583        <option>-c</option> and/or <option>-f</option> options.  It causes
584        <application>psql</application> to issue a <command>BEGIN</command> command
585        before the first such option and a <command>COMMIT</command> command after
586        the last one, thereby wrapping all the commands into a single
587        transaction. If any of the commands fails and the variable
588        <varname>ON_ERROR_STOP</varname> was set, a
589        <command>ROLLBACK</command> command is sent instead. This ensures that
590        either all the commands complete successfully, or no changes are
591        applied.
592       </para>
593
594       <para>
595        If the commands themselves
596        contain <command>BEGIN</command>, <command>COMMIT</command>,
597        or <command>ROLLBACK</command>, this option will not have the desired
598        effects.  Also, if an individual command cannot be executed inside a
599        transaction block, specifying this option will cause the whole
600        transaction to fail.
601       </para>
602      </listitem>

640  <title>Usage</title>
641
642  <refsect2 id="r2-app-psql-connecting">
643    <title>Connecting to a Database</title>
644
645    <para>
646    <application>psql</application> is a regular
647    <productname>PostgreSQL</productname> client application. In order
648    to connect to a database you need to know the name of your target
649    database, the host name and port number of the server, and what
650    database user name you want to connect as. <application>psql</application>
651    can be told about those parameters via command line options, namely
652    <option>-d</option>, <option>-h</option>, <option>-p</option>, and
653    <option>-U</option> respectively. If an argument is found that does
654    not belong to any option it will be interpreted as the database name
655    (or the database user name, if the database name is already given). Not all
656    of these options are required; there are useful defaults. If you omit the host
657    name, <application>psql</application> will connect via a Unix-domain socket
658    to a server on the local host, or via TCP/IP to <literal>localhost</literal> on
659    Windows. The default port number is
660    determined at compile time.
661    Since the database server uses the same default, you will not have
662    to specify the port in most cases. The default database user name is your
663    operating-system user name. Once the database user name is determined, it
664    is used as the default database name.
665    Note that you cannot
666    just connect to any database under any database user name. Your database
667    administrator should have informed you about your access rights.
668    </para>
669
670    <para>
671    When the defaults aren't quite right, you can save yourself
672    some typing by setting the environment variables
673    <envar>PGDATABASE</envar>, <envar>PGHOST</envar>,
674    <envar>PGPORT</envar> and/or <envar>PGUSER</envar> to appropriate
675    values. (For additional environment variables, see <xref
676    linkend="libpq-envars"/>.) It is also convenient to have a
677    <filename>~/.pgpass</filename> file to avoid regularly having to type in
678    passwords. See <xref linkend="libpq-pgpass"/> for more information.
679    </para>
680
681    <para>
682     An alternative way to specify connection parameters is in a
683     <parameter>conninfo</parameter> string or
684     a <acronym>URI</acronym>, which is used instead of a database
685     name. This mechanism give you very wide control over the
686     connection. For example:
687<programlisting>
688$ <userinput>psql "service=myservice sslmode=require"</userinput>
689$ <userinput>psql postgresql://dbmaster:5433/mydb?sslmode=require</userinput>
690</programlisting>
691     This way you can also use <acronym>LDAP</acronym> for connection
692     parameter lookup as described in <xref linkend="libpq-ldap"/>.
693     See <xref linkend="libpq-paramkeywords"/> for more information on all the
694     available connection options.
695    </para>
696
697    <para>
698    If the connection could not be made for any reason (e.g., insufficient
699    privileges, server is not running on the targeted host, etc.),
700    <application>psql</application> will return an error and terminate.
701    </para>
702
703    <para>
704     If both standard input and standard output are a
705     terminal, then <application>psql</application> sets the client
706     encoding to <quote>auto</quote>, which will detect the
707     appropriate client encoding from the locale settings
708     (<envar>LC_CTYPE</envar> environment variable on Unix systems).
709     If this doesn't work out as expected, the client encoding can be
710     overridden using the environment
711     variable <envar>PGCLIENTENCODING</envar>.
712    </para>
713  </refsect2>

715  <refsect2 id="r2-app-psql-4">
716    <title>Entering SQL Commands</title>
717
718    <para>
719    In normal operation, <application>psql</application> provides a
720    prompt with the name of the database to which
721    <application>psql</application> is currently connected, followed by
722    the string <literal>=&gt;</literal>. For example:
723<programlisting>
724$ <userinput>psql testdb</userinput>
725psql (&version;)
726Type "help" for help.
727
728testdb=&gt;
729</programlisting>
730    </para>
731
732    <para>
733    At the prompt, the user can type in <acronym>SQL</acronym> commands.
734    Ordinarily, input lines are sent to the server when a
735    command-terminating semicolon is reached. An end of line does not
736    terminate a command.  Thus commands can be spread over several lines for
737    clarity. If the command was sent and executed without error, the results
738    of the command are displayed on the screen.
739    </para>
740
741    <para>
742    If untrusted users have access to a database that has not adopted a
743    <link linkend="ddl-schemas-patterns">secure schema usage pattern</link>,
744    begin your session by removing publicly-writable schemas
745    from <varname>search_path</varname>.  One can
746    add <literal>options=-csearch_path=</literal> to the connection string or
747    issue <literal>SELECT pg_catalog.set_config('search_path', '',
748    false)</literal> before other SQL commands.  This consideration is not
749    specific to <application>psql</application>; it applies to every interface
750    for executing arbitrary SQL commands.
751    </para>
752
753    <para>
754    Whenever a command is executed, <application>psql</application> also polls
755    for asynchronous notification events generated by
756    <link linkend="sql-listen"><command>LISTEN</command></link> and
757    <link linkend="sql-notify"><command>NOTIFY</command></link>.
758    </para>
759
760    <para>
761    While C-style block comments are passed to the server for
762    processing and removal, SQL-standard comments are removed by
763    <application>psql</application>.
764    </para>
765  </refsect2>

767  <refsect2 id="app-psql-meta-commands">
768    <title>Meta-Commands</title>
769
770    <para>
771    Anything you enter in <application>psql</application> that begins
772    with an unquoted backslash is a <application>psql</application>
773    meta-command that is processed by <application>psql</application>
774    itself. These commands make
775    <application>psql</application> more useful for administration or
776    scripting. Meta-commands are often called slash or backslash commands.
777    </para>
778
779    <para>
780    The format of a <application>psql</application> command is the backslash,
781    followed immediately by a command verb, then any arguments. The arguments
782    are separated from the command verb and each other by any number of
783    whitespace characters.
784    </para>
785
786    <para>
787    To include whitespace in an argument you can quote it with
788    single quotes. To include a single quote in an argument,
789    write two single quotes within single-quoted text.
790    Anything contained in single quotes is
791    furthermore subject to C-like substitutions for
792    <literal>\n</literal> (new line), <literal>\t</literal> (tab),
793    <literal>\b</literal> (backspace), <literal>\r</literal> (carriage return),
794    <literal>\f</literal> (form feed),
795    <literal>\</literal><replaceable>digits</replaceable> (octal), and
796    <literal>\x</literal><replaceable>digits</replaceable> (hexadecimal).
797    A backslash preceding any other character within single-quoted text
798    quotes that single character, whatever it is.
799    </para>
800
801    <para>
802    If an unquoted colon (<literal>:</literal>) followed by a
803    <application>psql</application> variable name appears within an argument, it is
804    replaced by the variable's value, as described in <xref
805    linkend="app-psql-interpolation"/> below.
806    The forms <literal>:'<replaceable>variable_name</replaceable>'</literal> and
807    <literal>:"<replaceable>variable_name</replaceable>"</literal> described there
808    work as well.
809    The <literal>:{?<replaceable>variable_name</replaceable>}</literal> syntax allows
810    testing whether a variable is defined. It is substituted by
811    TRUE or FALSE.
812    Escaping the colon with a backslash protects it from substitution.
813    </para>
814
815    <para>
816    Within an argument, text that is enclosed in backquotes
817    (<literal>`</literal>) is taken as a command line that is passed to the
818    shell.  The output of the command (with any trailing newline removed)
819    replaces the backquoted text.  Within the text enclosed in backquotes,
820    no special quoting or other processing occurs, except that appearances
821    of <literal>:<replaceable>variable_name</replaceable></literal> where
822    <replaceable>variable_name</replaceable> is a <application>psql</application> variable name
823    are replaced by the variable's value.  Also, appearances of
824    <literal>:'<replaceable>variable_name</replaceable>'</literal> are replaced by the
825    variable's value suitably quoted to become a single shell command
826    argument.  (The latter form is almost always preferable, unless you are
827    very sure of what is in the variable.)  Because carriage return and line
828    feed characters cannot be safely quoted on all platforms, the
829    <literal>:'<replaceable>variable_name</replaceable>'</literal> form prints an
830    error message and does not substitute the variable value when such
831    characters appear in the value.
832    </para>
833
834    <para>
835    Some commands take an <acronym>SQL</acronym> identifier (such as a
836    table name) as argument. These arguments follow the syntax rules
837    of <acronym>SQL</acronym>: Unquoted letters are forced to
838    lowercase, while double quotes (<literal>"</literal>) protect letters
839    from case conversion and allow incorporation of whitespace into
840    the identifier.  Within double quotes, paired double quotes reduce
841    to a single double quote in the resulting name.  For example,
842    <literal>FOO"BAR"BAZ</literal> is interpreted as <literal>fooBARbaz</literal>,
843    and <literal>"A weird"" name"</literal> becomes <literal>A weird"
844    name</literal>.
845    </para>
846
847    <para>
848    Parsing for arguments stops at the end of the line, or when another
849    unquoted backslash is found.  An unquoted backslash
850    is taken as the beginning of a new meta-command. The special
851    sequence <literal>\\</literal> (two backslashes) marks the end of
852    arguments and continues parsing <acronym>SQL</acronym> commands, if
853    any. That way <acronym>SQL</acronym> and
854    <application>psql</application> commands can be freely mixed on a
855    line. But in any case, the arguments of a meta-command cannot
856    continue beyond the end of the line.
857    </para>
858
859    <para>
860    Many of the meta-commands act on the <firstterm>current query buffer</firstterm>.
861    This is simply a buffer holding whatever SQL command text has been typed
862    but not yet sent to the server for execution.  This will include previous
863    input lines as well as any text appearing before the meta-command on the
864    same line.
865    </para>
866
867    <para>
868    Many of the meta-commands also allow <literal>x</literal> to be appended
869    as an option. This will cause the results to be displayed in expanded
870    mode, as if <command>\x</command> or <command>\pset expanded</command>
871    had been used.
872    </para>

2708        A conditional block must begin with an <command>\if</command> and end
2709        with an <command>\endif</command>.  In between there may be any number

2713        the commands forming a conditional block.
2714        </para>

2717        their argument(s) and evaluate them as a Boolean expression.  If the
2718        expression yields <literal>true</literal> then processing continues

2724        the same block are not evaluated but are treated as false.  Lines
2725        following an <command>\else</command> are processed only if no earlier
2726        matching <command>\if</command> or <command>\elif</command> succeeded.
2727        </para>

2730        of an <command>\if</command> or <command>\elif</command> command
2731        is subject to variable interpolation and backquote expansion, just
2732        like any other backslash command argument.  After that it is evaluated
2733        like the value of an on/off option variable.  So a valid value
2734        is any unambiguous case-insensitive match for one of:
2735        <literal>true</literal>, <literal>false</literal>, <literal>1</literal>,
2736        <literal>0</literal>, <literal>on</literal>, <literal>off</literal>,
2737        <literal>yes</literal>, <literal>no</literal>.  For example,
2738        <literal>t</literal>, <literal>T</literal>, and <literal>tR</literal>
2739        will all be considered to be <literal>true</literal>.
2740        </para>
2741        <para>
2742        Expressions that do not properly evaluate to true or false will
2743        generate a warning and be treated as false.
2744        </para>

2747        backslash commands, but queries are not sent to the server, and
2748        backslash commands other than conditionals

2750        <command>\else</command>, <command>\endif</command>) are

2752        Variable references in skipped lines are not expanded, and backquote
2753        expansion is not performed either.

2757        the same source file. If EOF is reached on the main input file or an
2758        <command>\include</command>-ed file before all local
2759        <command>\if</command>-blocks have been closed,

4032   <title>Patterns</title>
4033
4034   <indexterm>
4035    <primary>patterns</primary>
4036    <secondary>in psql and pg_dump</secondary>
4037   </indexterm>
4038
4039  <para>
4040   The various <literal>\d</literal> commands accept a <replaceable
4041   class="parameter">pattern</replaceable> parameter to specify the
4042   object name(s) to be displayed.  In the simplest case, a pattern
4043   is just the exact name of the object.  The characters within a
4044   pattern are normally folded to lower case, just as in SQL names;
4045   for example, <literal>\dt FOO</literal> will display the table named
4046   <literal>foo</literal>.  As in SQL names, placing double quotes around
4047   a pattern stops folding to lower case.  Should you need to include
4048   an actual double quote character in a pattern, write it as a pair
4049   of double quotes within a double-quote sequence; again this is in
4050   accord with the rules for SQL quoted identifiers.  For example,
4051   <literal>\dt "FOO""BAR"</literal> will display the table named
4052   <literal>FOO"BAR</literal> (not <literal>foo"bar</literal>).  Unlike the normal
4053   rules for SQL names, you can put double quotes around just part
4054   of a pattern, for instance <literal>\dt FOO"FOO"BAR</literal> will display
4055   the table named <literal>fooFOObar</literal>.
4056  </para>
4057
4058  <para>
4059   Whenever the <replaceable class="parameter">pattern</replaceable> parameter
4060   is omitted completely, the <literal>\d</literal> commands display all objects
4061   that are visible in the current schema search path &mdash; this is
4062   equivalent to using <literal>*</literal> as the pattern.
4063   (An object is said to be <firstterm>visible</firstterm> if its
4064   containing schema is in the search path and no object of the same
4065   kind and name appears earlier in the search path. This is equivalent to the
4066   statement that the object can be referenced by name without explicit
4067   schema qualification.)
4068   To see all objects in the database regardless of visibility,
4069   use <literal>*.*</literal> as the pattern.
4070  </para>
4071
4072  <para>
4073   Within a pattern, <literal>*</literal> matches any sequence of characters
4074   (including no characters) and <literal>?</literal> matches any single character.
4075   (This notation is comparable to Unix shell file name patterns.)
4076   For example, <literal>\dt int*</literal> displays tables whose names
4077   begin with <literal>int</literal>.  But within double quotes, <literal>*</literal>
4078   and <literal>?</literal> lose these special meanings and are just matched
4079   literally.
4080  </para>
4081
4082  <para>
4083   A relation pattern that contains a dot (<literal>.</literal>) is interpreted as a schema
4084   name pattern followed by an object name pattern.  For example,
4085   <literal>\dt foo*.*bar*</literal> displays all tables whose table name
4086   includes <literal>bar</literal> that are in schemas whose schema name
4087   starts with <literal>foo</literal>.  When no dot appears, then the pattern
4088   matches only objects that are visible in the current schema search path.
4089   Again, a dot within double quotes loses its special meaning and is matched
4090   literally.  A relation pattern that contains two dots (<literal>.</literal>)
4091   is interpreted as a database name followed by a schema name pattern followed
4092   by an object name pattern.  The database name portion will not be treated as
4093   a pattern and must match the name of the currently connected database, else
4094   an error will be raised.
4095  </para>
4096
4097  <para>
4098   A schema pattern that contains a dot (<literal>.</literal>) is interpreted
4099   as a database name followed by a schema name pattern.  For example,
4100   <literal>\dn mydb.*foo*</literal> displays all schemas whose schema name
4101   includes <literal>foo</literal>.  The database name portion will not be
4102   treated as a pattern and must match the name of the currently connected
4103   database, else an error will be raised.
4104  </para>
4105
4106  <para>
4107   Advanced users can use regular-expression notations such as character
4108   classes, for example <literal>[0-9]</literal> to match any digit.  All regular
4109   expression special characters work as specified in
4110   <xref linkend="functions-posix-regexp"/>, except for <literal>.</literal> which
4111   is taken as a separator as mentioned above, <literal>*</literal> which is
4112   translated to the regular-expression notation <literal>.*</literal>,
4113   <literal>?</literal> which is translated to <literal>.</literal>, and
4114   <literal>$</literal> which is matched literally.  You can emulate
4115   these pattern characters at need by writing
4116   <literal>?</literal> for <literal>.</literal>,
4117   <literal>(<replaceable class="parameter">R</replaceable>+|)</literal> for
4118   <literal><replaceable class="parameter">R</replaceable>*</literal>, or
4119   <literal>(<replaceable class="parameter">R</replaceable>|)</literal> for
4120   <literal><replaceable class="parameter">R</replaceable>?</literal>.
4121   <literal>$</literal> is not needed as a regular-expression character since
4122   the pattern must match the whole name, unlike the usual
4123   interpretation of regular expressions (in other words, <literal>$</literal>
4124   is automatically appended to your pattern).  Write <literal>*</literal> at the
4125   beginning and/or end if you don't wish the pattern to be anchored.
4126   Note that within double quotes, all regular expression special characters
4127   lose their special meanings and are matched literally.  Also, the regular
4128   expression special characters are matched literally in operator name
4129   patterns (i.e., the argument of <literal>\do</literal>).
4130  </para>
4131  </refsect3>

4816   <refsect3 id="app-psql-interpolation" xreflabel="SQL Interpolation">
4817    <title><acronym>SQL</acronym> Interpolation</title>
4818
4819    <para>
4820    A key feature of <application>psql</application>
4821    variables is that you can substitute (<quote>interpolate</quote>)
4822    them into regular <acronym>SQL</acronym> statements, as well as the
4823    arguments of meta-commands.  Furthermore,
4824    <application>psql</application> provides facilities for
4825    ensuring that variable values used as SQL literals and identifiers are
4826    properly quoted.  The syntax for interpolating a value without
4827    any quoting is to prepend the variable name with a colon
4828    (<literal>:</literal>).  For example,
4829<programlisting>
4830testdb=&gt; <userinput>\set foo 'my_table'</userinput>
4831testdb=&gt; <userinput>SELECT * FROM :foo;</userinput>
4832</programlisting>
4833    would query the table <literal>my_table</literal>. Note that this
4834    may be unsafe: the value of the variable is copied literally, so it can
4835    contain unbalanced quotes, or even backslash commands. You must make sure
4836    that it makes sense where you put it.
4837    </para>
4838
4839    <para>
4840    When a value is to be used as an SQL literal or identifier, it is
4841    safest to arrange for it to be quoted.  To quote the value of
4842    a variable as an SQL literal, write a colon followed by the variable
4843    name in single quotes.  To quote the value as an SQL identifier, write
4844    a colon followed by the variable name in double quotes.
4845    These constructs deal correctly with quotes and other special
4846    characters embedded within the variable value.
4847    The previous example would be more safely written this way:
4848<programlisting>
4849testdb=&gt; <userinput>\set foo 'my_table'</userinput>
4850testdb=&gt; <userinput>SELECT * FROM :"foo";</userinput>
4851</programlisting>
4852    </para>
4853
4854    <para>
4855    Variable interpolation will not be performed within quoted
4856    <acronym>SQL</acronym> literals and identifiers.  Therefore, a
4857    construction such as <literal>':foo'</literal> doesn't work to produce a quoted
4858    literal from a variable's value (and it would be unsafe if it did work,
4859    since it wouldn't correctly handle quotes embedded in the value).
4860    </para>
4861
4862    <para>
4863    One example use of this mechanism is to
4864    copy the contents of a file into a table column.
4865    First load the file into a variable and then interpolate the variable's
4866    value as a quoted string:
4867<programlisting>
4868testdb=&gt; <userinput>\set content `cat my_file.txt`</userinput>
4869testdb=&gt; <userinput>INSERT INTO my_table VALUES (:'content');</userinput>
4870</programlisting>
4871    (Note that this still won't work if <filename>my_file.txt</filename> contains NUL bytes.
4872    <application>psql</application> does not support embedded NUL bytes in variable values.)
4873    </para>
4874
4875    <para>
4876    Since colons can legally appear in SQL commands, an apparent attempt
4877    at interpolation (that is, <literal>:name</literal>,
4878    <literal>:'name'</literal>, or <literal>:"name"</literal>) is not
4879    replaced unless the named variable is currently set. In any case, you
4880    can escape a colon with a backslash to protect it from substitution.
4881    </para>
4882
4883    <para>
4884    The <literal>:{?<replaceable>name</replaceable>}</literal> special syntax returns TRUE
4885    or FALSE depending on whether the variable exists or not, and is thus
4886    always substituted, unless the colon is backslash-escaped.
4887    </para>
4888
4889    <para>
4890    The colon syntax for variables is standard <acronym>SQL</acronym> for
4891    embedded query languages, such as <application>ECPG</application>.
4892    The colon syntaxes for array slices and type casts are
4893    <productname>PostgreSQL</productname> extensions, which can sometimes
4894    conflict with the standard usage.  The colon-quote syntax for escaping a
4895    variable's value as an SQL literal or identifier is a
4896    <application>psql</application> extension.
4897    </para>
