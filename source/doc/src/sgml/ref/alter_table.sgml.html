148 <refsect1>
149  <title>Description</title>
150
151  <para>
152   <command>ALTER TABLE</command> changes the definition of an existing table.
153   There are several subforms described below. Note that the lock level required
154   may differ for each subform. An <literal>ACCESS EXCLUSIVE</literal> lock is
155   acquired unless explicitly noted. When multiple subcommands are given, the
156   lock acquired will be the strictest one required by any subcommand.
157
158  <variablelist>
159   <varlistentry id="sql-altertable-desc-add-column">
160    <term><literal>ADD [ COLUMN ] [ IF NOT EXISTS ]</literal></term>
161    <listitem>
162     <para>
163      This form adds a new column to the table, using the same syntax as
164      <link linkend="sql-createtable"><command>CREATE TABLE</command></link>. If <literal>IF NOT EXISTS</literal>
165      is specified and a column already exists with this name,
166      no error is thrown.
167     </para>
168    </listitem>
169   </varlistentry>
170
171   <varlistentry id="sql-altertable-desc-drop-column">
172    <term><literal>DROP [ COLUMN ] [ IF EXISTS ]</literal></term>
173    <listitem>
174     <para>
175      This form drops a column from a table.  Indexes and
176      table constraints involving the column will be automatically
177      dropped as well.
178      Multivariate statistics referencing the dropped column will also be
179      removed if the removal of the column would cause the statistics to
180      contain data for only a single column.
181      You will need to say <literal>CASCADE</literal> if anything outside the table
182      depends on the column, for example, foreign key references or views.
183      If <literal>IF EXISTS</literal> is specified and the column
184      does not exist, no error is thrown. In this case a notice
185      is issued instead.
186     </para>
187    </listitem>
188   </varlistentry>
189
190   <varlistentry id="sql-altertable-desc-set-data-type">
191    <term><literal>SET DATA TYPE</literal></term>
192    <listitem>
193     <para>
194      This form changes the type of a column of a table. Indexes and
195      simple table constraints involving the column will be automatically
196      converted to use the new column type by reparsing the originally
197      supplied expression.
198      The optional <literal>COLLATE</literal> clause specifies a collation
199      for the new column; if omitted, the collation is the default for the
200      new column type.
201      The optional <literal>USING</literal>
202      clause specifies how to compute the new column value from the old;
203      if omitted, the default conversion is the same as an assignment
204      cast from old data type to new.  A  <literal>USING</literal>
205      clause must be provided if there is no implicit or assignment
206      cast from old to new type.
207     </para>
208
209     <para>
210      When this form is used, the column's statistics are removed,
211      so running <link linkend="sql-analyze"><command>ANALYZE</command></link>
212      on the table afterwards is recommended.
213      For a virtual generated column, <command>ANALYZE</command>
214      is not necessary because such columns never have statistics.
215     </para>
216    </listitem>
217   </varlistentry>
218
219   <varlistentry id="sql-altertable-desc-set-drop-default">
220    <term><literal>SET</literal>/<literal>DROP DEFAULT</literal></term>
221    <listitem>
222     <para>
223      These forms set or remove the default value for a column (where
224      removal is equivalent to setting the default value to NULL).  The new
225      default value will only apply in subsequent <command>INSERT</command>
226      or <command>UPDATE</command> commands; it does not cause rows already
227      in the table to change.
228     </para>
229    </listitem>
230   </varlistentry>
231
232   <varlistentry id="sql-altertable-desc-set-drop-not-null">
233    <term><literal>SET</literal>/<literal>DROP NOT NULL</literal></term>
234    <listitem>
235     <para>
236      These forms change whether a column is marked to allow null
237      values or to reject null values.
238     </para>
239
240     <para>
241      <literal>SET NOT NULL</literal> may only be applied to a column
242      provided none of the records in the table contain a
243      <literal>NULL</literal> value for the column.  Ordinarily this is
244      checked during the <literal>ALTER TABLE</literal> by scanning the
245      entire table, unless <literal>NOT VALID</literal> is specified;
246      however, if a valid <literal>CHECK</literal> constraint exists
247      (and is not dropped in the same command) which proves no
248      <literal>NULL</literal> can exist, then the table scan is skipped.
249      If a column has an invalid not-null constraint,
250      <literal>SET NOT NULL</literal> validates it.
251     </para>
252
253     <para>
254      If this table is a partition, one cannot perform <literal>DROP NOT NULL</literal>
255      on a column if it is marked <literal>NOT NULL</literal> in the parent
256      table.  To drop the <literal>NOT NULL</literal> constraint from all the
257      partitions, perform <literal>DROP NOT NULL</literal> on the parent
258      table.  Even if there is no <literal>NOT NULL</literal> constraint on the
259      parent, such a constraint can still be added to individual partitions,
260      if desired; that is, the children can disallow nulls even if the parent
261      allows them, but not the other way around.  It is also possible to drop
262      the <literal>NOT NULL</literal> constraint from <literal>ONLY</literal>
263      the parent table, which does not remove it from the children.
264     </para>
265    </listitem>
266   </varlistentry>
267
268   <varlistentry id="sql-altertable-desc-set-expression">
269    <term><literal>SET EXPRESSION AS</literal></term>
270    <listitem>
271     <para>
272      This form replaces the expression of a generated column.  Existing data
273      in a stored generated column is rewritten and all the future changes
274      will apply the new generation expression.
275     </para>
276
277     <para>
278      When this form is used on a stored generated column, its statistics
279      are removed, so running
280      <link linkend="sql-analyze"><command>ANALYZE</command></link>
281      on the table afterwards is recommended.
282      For a virtual generated column, <command>ANALYZE</command>
283      is not necessary because such columns never have statistics.
284     </para>
285    </listitem>
286   </varlistentry>
287
288   <varlistentry id="sql-altertable-desc-drop-expression">
289    <term><literal>DROP EXPRESSION [ IF EXISTS ]</literal></term>
290    <listitem>
291     <para>
292      This form turns a stored generated column into a normal base column.
293      Existing data in the columns is retained, but future changes will no
294      longer apply the generation expression.
295     </para>
296
297     <para>
298      This form is currently only supported for stored generated columns (not
299      virtual ones).
300     </para>
301
302     <para>
303      If <literal>DROP EXPRESSION IF EXISTS</literal> is specified and the
304      column is not a generated column, no error is thrown.  In this case a
305      notice is issued instead.
306     </para>
307    </listitem>
308   </varlistentry>

310   <varlistentry id="sql-altertable-desc-generated-identity">
311    <term><literal>ADD GENERATED { ALWAYS | BY DEFAULT } AS IDENTITY</literal></term>
312    <term><literal>SET GENERATED { ALWAYS | BY DEFAULT }</literal></term>
313    <term><literal>DROP IDENTITY [ IF EXISTS ]</literal></term>
314    <listitem>
315     <para>
316      These forms change whether a column is an identity column or change the
317      generation attribute of an existing identity column.
318      See <link linkend="sql-createtable"><command>CREATE TABLE</command></link> for details.
319      Like <literal>SET DEFAULT</literal>, these forms only affect the
320      behavior of subsequent <command>INSERT</command>
321      and <command>UPDATE</command> commands; they do not cause rows
322      already in the table to change.
323     </para>
324
325     <para>
326      If <literal>DROP IDENTITY IF EXISTS</literal> is specified and the
327      column is not an identity column, no error is thrown.  In this case a
328      notice is issued instead.
329     </para>
330    </listitem>

346   <varlistentry id="sql-altertable-desc-set-statistics">
347    <term><literal>SET STATISTICS</literal></term>
348    <listitem>
349     <para>
350      This form
351      sets the per-column statistics-gathering target for subsequent
352      <link linkend="sql-analyze"><command>ANALYZE</command></link> operations.
353      The target can be set in the range 0 to 10000.  Set it
354      to <literal>DEFAULT</literal> to revert to using the system default
355      statistics target (<xref linkend="guc-default-statistics-target"/>).
356      (Setting to a value of -1 is an obsolete way spelling to get the same
357      outcome.)
358      For more information on the use of statistics by the
359      <productname>PostgreSQL</productname> query planner, refer to
360      <xref linkend="planner-stats"/>.
361     </para>
362     <para>
363      <literal>SET STATISTICS</literal> acquires a
364      <literal>SHARE UPDATE EXCLUSIVE</literal> lock.
365     </para>
366    </listitem>
367   </varlistentry>

369   <varlistentry id="sql-altertable-desc-set-attribute-option">
370    <term><literal>SET ( <replaceable class="parameter">attribute_option</replaceable> = <replaceable class="parameter">value</replaceable> [, ... ] )</literal></term>
371    <term><literal>RESET ( <replaceable class="parameter">attribute_option</replaceable> [, ... ] )</literal></term>
372    <listitem>
373     <para>
374      This form sets or resets per-attribute options.  Currently, the only
375      defined per-attribute options are <literal>n_distinct</literal> and
376      <literal>n_distinct_inherited</literal>, which override the
377      number-of-distinct-values estimates made by subsequent
378      <link linkend="sql-analyze"><command>ANALYZE</command></link>
379      operations. <literal>n_distinct</literal> affects the statistics for the
380      table itself, while <literal>n_distinct_inherited</literal> affects the
381      statistics gathered for the table plus its inheritance children, and for
382      the statistics gathered for partitioned tables.  When the value
383      specified is a positive value, the query planner will assume that the
384      column contains exactly the specified number of distinct nonnull values.
385      Fractional values may also be specified by using values below 0 and
386      above or equal to -1.  This instructs the query planner to estimate the
387      number of distinct values by multiplying the absolute value of the
388      specified number by the estimated number of rows in the table.  For
389      example, a value of -1 implies that all values in the column are
390      distinct, while a value of -0.5 implies that each value appears twice on
391      average.  This can be useful when the size of the table changes over
392      time.  For more information on the use of statistics by the
393      <productname>PostgreSQL</productname> query planner, refer to
394      <xref linkend="planner-stats"/>.
395     </para>
396     <para>
397      Changing per-attribute options acquires a
398      <literal>SHARE UPDATE EXCLUSIVE</literal> lock.
399     </para>
400    </listitem>

403   <varlistentry id="sql-altertable-desc-set-storage">
404    <term>
405     <literal>SET STORAGE { PLAIN | EXTERNAL | EXTENDED | MAIN | DEFAULT }</literal>
406     <indexterm>
407      <primary>TOAST</primary>
408      <secondary>per-column storage settings</secondary>
409     </indexterm>
410    </term>
411    <listitem>
412     <para>
413      This form sets the storage mode for a column. This controls whether this
414      column is held inline or in a secondary <acronym>TOAST</acronym> table,
415      and whether the data
416      should be compressed or not. <literal>PLAIN</literal> must be used
417      for fixed-length values such as <type>integer</type> and is
418      inline, uncompressed. <literal>MAIN</literal> is for inline,
419      compressible data. <literal>EXTERNAL</literal> is for external,
420      uncompressed data, and <literal>EXTENDED</literal> is for external,
421      compressed data.
422      Writing <literal>DEFAULT</literal> sets the storage mode to the default
423      mode for the column's data type.  <literal>EXTENDED</literal> is the
424      default for most data types that support non-<literal>PLAIN</literal>
425      storage.
426      Use of <literal>EXTERNAL</literal> will make substring operations on
427      very large <type>text</type> and <type>bytea</type> values run faster,
428      at the penalty of increased storage space.
429      Note that <literal>ALTER TABLE ... SET STORAGE</literal> doesn't itself
430      change anything in the table; it just sets the strategy to be pursued
431      during future table updates.
432      See <xref linkend="storage-toast"/> for more information.
433     </para>
434    </listitem>
435   </varlistentry>

442     <para>
443      This form sets the compression method for a column, determining how

1033    <term><literal>ATTACH PARTITION <replaceable class="parameter">partition_name</replaceable> { FOR VALUES <replaceable class="parameter">partition_bound_spec</replaceable> | DEFAULT }</literal></term>
1034    <listitem>
1035     <para>
1036      This form attaches an existing table (which might itself be partitioned)
1037      as a partition of the target table. The table can be attached
1038      as a partition for specific values using <literal>FOR VALUES</literal>
1039      or as a default partition by using <literal>DEFAULT</literal>.
1040      For each index in the target table, a corresponding
1041      one will be created in the attached table; or, if an equivalent
1042      index already exists, it will be attached to the target table's index,
1043      as if <command>ALTER INDEX ATTACH PARTITION</command> had been executed.
1044      Note that if the existing table is a foreign table, it is currently not
1045      allowed to attach the table as a partition of the target table if there
1046      are <literal>UNIQUE</literal> indexes on the target table.  (See also
1047      <xref linkend="sql-createforeigntable"/>.)  For each user-defined
1048      row-level trigger that exists in the target table, a corresponding one
1049      is created in the attached table.
1050     </para>
1051
1052     <para>
1053      A partition using <literal>FOR VALUES</literal> uses same syntax for
1054      <replaceable class="parameter">partition_bound_spec</replaceable> as
1055      <link linkend="sql-createtable"><command>CREATE TABLE</command></link>.
1056      The partition bound specification
1057      must correspond to the partitioning strategy and partition key of the
1058      target table.  The table to be attached must have all the same columns
1059      as the target table and no more; moreover, the column types must also
1060      match.  Also, it must have all the <literal>NOT NULL</literal> and
1061      <literal>CHECK</literal> constraints of the target table, not marked
1062      <literal>NO INHERIT</literal>.  Currently
1063      <literal>FOREIGN KEY</literal> constraints are not considered.
1064      <literal>UNIQUE</literal> and <literal>PRIMARY KEY</literal> constraints
1065      from the parent table will be created in the partition, if they don't
1066      already exist.
1067     </para>
1068
1069     <para>
1070      If the new partition is a regular table, a full table scan is performed
1071      to check that existing rows in the table do not violate the partition
1072      constraint. It is possible to avoid this scan by adding a valid
1073      <literal>CHECK</literal> constraint to the table that allows only
1074      rows satisfying the desired partition constraint before running this
1075      command. The <literal>CHECK</literal> constraint will be used to
1076      determine that the table need not be scanned to validate the partition
1077      constraint. This does not work, however, if any of the partition keys
1078      is an expression and the partition does not accept
1079      <literal>NULL</literal> values. If attaching a list partition that will
1080      not accept <literal>NULL</literal> values, also add a
1081      <literal>NOT NULL</literal> constraint to the partition key column,
1082      unless it's an expression.
1083     </para>
1084
1085     <para>
1086      If the new partition is a foreign table, nothing is done to verify
1087      that all the rows in the foreign table obey the partition constraint.
1088      (See the discussion in <xref linkend="sql-createforeigntable"/> about
1089      constraints on the foreign table.)
1090     </para>
1091
1092     <para>
1093      When a table has a default partition, defining a new partition changes
1094      the partition constraint for the default partition. The default
1095      partition can't contain any rows that would need to be moved to the new
1096      partition, and will be scanned to verify that none are present. This
1097      scan, like the scan of the new partition, can be avoided if an
1098      appropriate <literal>CHECK</literal> constraint is present. Also like
1099      the scan of the new partition, it is always skipped when the default
1100      partition is a foreign table.
1101     </para>
1102
1103     <para>
1104      Attaching a partition acquires a
1105      <literal>SHARE UPDATE EXCLUSIVE</literal> lock on the parent table,
1106      in addition to the <literal>ACCESS EXCLUSIVE</literal> locks on the table
1107      being attached and on the default partition (if any).
1108     </para>
1109
1110     <para>
1111      Further locks must also be held on all sub-partitions if the table being
1112      attached is itself a partitioned table.  Likewise if the default
1113      partition is itself a partitioned table.  The locking of the
1114      sub-partitions can be avoided by adding a <literal>CHECK</literal>
1115      constraint as described in
1116      <xref linkend="ddl-partitioning-declarative-maintenance"/>.
1117     </para>
1118    </listitem>

1121   <varlistentry id="sql-altertable-detach-partition">
1122    <term><literal>DETACH PARTITION <replaceable class="parameter">partition_name</replaceable> [ CONCURRENTLY | FINALIZE ]</literal></term>
1123
1124    <listitem>
1125     <para>
1126      This form detaches the specified partition of the target table.  The detached
1127      partition continues to exist as a standalone table, but no longer has any
1128      ties to the table from which it was detached.  Any indexes that were
1129      attached to the target table's indexes are detached.  Any triggers that
1130      were created as clones of those in the target table are removed.
1131      <literal>SHARE</literal> lock is obtained on any tables that reference
1132      this partitioned table in foreign key constraints.
1133     </para>
1134     <para>
1135      If <literal>CONCURRENTLY</literal> is specified, it runs using a reduced
1136      lock level to avoid blocking other sessions that might be accessing the
1137      partitioned table.  In this mode, two transactions are used internally.
1138      During the first transaction, a <literal>SHARE UPDATE EXCLUSIVE</literal>
1139      lock is taken on both parent table and partition, and the partition is
1140      marked as undergoing detach; at that point, the transaction is committed
1141      and all other transactions using the partitioned table are waited for.
1142      Once all those transactions have completed, the second transaction
1143      acquires <literal>SHARE UPDATE EXCLUSIVE</literal> on the partitioned
1144      table and <literal>ACCESS EXCLUSIVE</literal> on the partition,
1145      and the detach process completes.  A <literal>CHECK</literal> constraint
1146      that duplicates the partition constraint is added to the partition.
1147      <literal>CONCURRENTLY</literal> cannot be run in a transaction block and
1148      is not allowed if the partitioned table contains a default partition.
1149     </para>
1150     <para>
1151      If <literal>FINALIZE</literal> is specified, a previous
1152      <literal>DETACH CONCURRENTLY</literal> invocation that was canceled or
1153      interrupted is completed.
1154      At most one partition in a partitioned table can be pending detach at
1155      a time.
1156     </para>

1158   </varlistentry>

1160  </variablelist>

1163  <para>
1164   All the forms of <command>ALTER TABLE</command> that act on a single table,
1165   except <literal>RENAME</literal>, <literal>SET SCHEMA</literal>,
1166   <literal>ATTACH PARTITION</literal>, and
1167   <literal>DETACH PARTITION</literal> can be combined into
1168   a list of multiple alterations to be applied together.  For example, it
1169   is possible to add several columns and/or alter the type of several
1170   columns in a single command.  This is particularly useful with large
1171   tables, since only one pass over the table need be made.
1172  </para>

1428 <refsect1 id="sql-altertable-notes" xreflabel="Notes">
1429  <title>Notes</title>
1430
1431   <para>
1432    The key word <literal>COLUMN</literal> is noise and can be omitted.
1433   </para>
1434
1435   <para>
1436    When a column is added with <literal>ADD COLUMN</literal> and a
1437    non-volatile <literal>DEFAULT</literal> is specified, the default value is
1438    evaluated at the time of the statement and the result stored in the
1439    table's metadata, where it will be returned when any existing rows are
1440    accessed.  The value will be only applied when the table is rewritten,
1441    making the <command>ALTER TABLE</command> very fast even on large tables.
1442    If no column constraints are specified, NULL is used as the
1443    <literal>DEFAULT</literal>.  In neither case is a rewrite of the table
1444    required.
1445   </para>
1446
1447   <para>
1448    Adding a column with a volatile <literal>DEFAULT</literal>
1449    (e.g., <function>clock_timestamp()</function>), a stored generated column,
1450    an identity column, or a column with a domain data type that has
1451    constraints will cause the entire table and its indexes to be rewritten.
1452    Adding a virtual generated column never requires a rewrite.
1453   </para>
1454
1455   <para>
1456    Changing the type of an existing column will normally cause the entire table
1457    and its indexes to be rewritten.
1458    As an exception, when changing the type of an existing column,
1459    if the <literal>USING</literal> clause does not change the column
1460    contents and the old type is either binary coercible to the new type
1461    or an unconstrained domain over the new type, a table rewrite is not
1462    needed.  However, indexes will still be rebuilt unless the system
1463    can verify that the new index would be logically equivalent to the
1464    existing one.  For example, if the collation for a column has been
1465    changed, an index rebuild is required because the new sort
1466    order might be different.  However, in the absence of a collation
1467    change, a column can be changed from <type>text</type> to
1468    <type>varchar</type> (or vice versa) without rebuilding the indexes
1469    because these data types sort identically.
1470   </para>
1471
1472   <para>
1473    Table and/or index
1474    rebuilds may take a significant amount of time for a large table,
1475    and will temporarily require as much as double the disk space.
1476   </para>
1477
1478   <para>
1479    Adding a <literal>CHECK</literal> or <literal>NOT NULL</literal>
1480    constraint requires scanning the table to verify that existing rows meet the
1481    constraint, but does not require a table rewrite.  If a <literal>CHECK</literal>
1482    constraint is added as <literal>NOT ENFORCED</literal>, no verification will
1483    be performed.
1484   </para>
1485
1486   <para>
1487    Similarly, when attaching a new partition it may be scanned to verify that
1488    existing rows meet the partition constraint.
1489   </para>
1490
1491   <para>
1492    The main reason for providing the option to specify multiple changes
1493    in a single <command>ALTER TABLE</command> is that multiple table scans or
1494    rewrites can thereby be combined into a single pass over the table.
1495   </para>
1496
1497   <para>
1498    Scanning a large table to verify new foreign-key, check, or not-null constraints
1499    can take a long time, and other updates to the table are locked out
1500    until the <command>ALTER TABLE ADD CONSTRAINT</command> command is
1501    committed.  The main purpose of the <literal>NOT VALID</literal>
1502    constraint option is to reduce the impact of adding a constraint on
1503    concurrent updates.  With <literal>NOT VALID</literal>,
1504    the <command>ADD CONSTRAINT</command> command does not scan the table
1505    and can be committed immediately.  After that, a <literal>VALIDATE
1506    CONSTRAINT</literal> command can be issued to verify that existing rows
1507    satisfy the constraint.  The validation step does not need to lock out
1508    concurrent updates, since it knows that other transactions will be
1509    enforcing the constraint for rows that they insert or update; only
1510    pre-existing rows need to be checked.  Hence, validation acquires only
1511    a <literal>SHARE UPDATE EXCLUSIVE</literal> lock on the table being
1512    altered.  (If the constraint is a foreign key then a <literal>ROW
1513    SHARE</literal> lock is also required on the table referenced by the
1514    constraint.)  In addition to improving concurrency, it can be useful to
1515    use <literal>NOT VALID</literal> and <literal>VALIDATE
1516    CONSTRAINT</literal> in cases where the table is known to contain
1517    pre-existing violations.  Once the constraint is in place, no new
1518    violations can be inserted, and the existing problems can be corrected
1519    at leisure until <literal>VALIDATE CONSTRAINT</literal> finally
1520    succeeds.
1521   </para>
1522
1523   <para>
1524    The <literal>DROP COLUMN</literal> form does not physically remove
1525    the column, but simply makes it invisible to SQL operations.  Subsequent
1526    insert and update operations in the table will store a null value for the
1527    column. Thus, dropping a column is quick but it will not immediately
1528    reduce the on-disk size of your table, as the space occupied
1529    by the dropped column is not reclaimed.  The space will be
1530    reclaimed over time as existing rows are updated.
1531   </para>
1532
1533   <para>
1534    To force immediate reclamation of space occupied by a dropped column,
1535    you can execute one of the forms of <command>ALTER TABLE</command> that
1536    performs a rewrite of the whole table.  This results in reconstructing
1537    each row with the dropped column replaced by a null value.
1538   </para>
1539
1540   <para>
1541    The rewriting forms of <command>ALTER TABLE</command> are not MVCC-safe.
1542    After a table rewrite, the table will appear empty to concurrent
1543    transactions, if they are using a snapshot taken before the rewrite
1544    occurred.  See <xref linkend="mvcc-caveats"/> for more details.
1545   </para>
1546
1547   <para>
1548    The <literal>USING</literal> option of <literal>SET DATA TYPE</literal> can actually
1549    specify any expression involving the old values of the row; that is, it
1550    can refer to other columns as well as the one being converted.  This allows
1551    very general conversions to be done with the <literal>SET DATA TYPE</literal>
1552    syntax.  Because of this flexibility, the <literal>USING</literal>
1553    expression is not applied to the column's default value (if any); the
1554    result might not be a constant expression as required for a default.
1555    This means that when there is no implicit or assignment cast from old to
1556    new type, <literal>SET DATA TYPE</literal> might fail to convert the default even
1557    though a <literal>USING</literal> clause is supplied.  In such cases,
1558    drop the default with <literal>DROP DEFAULT</literal>, perform the <literal>ALTER
1559    TYPE</literal>, and then use <literal>SET DEFAULT</literal> to add a suitable new
1560    default.  Similar considerations apply to indexes and constraints involving
1561    the column.
1562   </para>
1563
1564   <para>
1565    If a table has any descendant tables, it is not permitted to add,
1566    rename, or change the type of a column in the parent table without doing
1567    the same to the descendants.  This ensures that the descendants always
1568    have columns matching the parent.  Similarly, a <literal>CHECK</literal>
1569    constraint cannot be renamed in the parent without also renaming it in
1570    all descendants, so that <literal>CHECK</literal> constraints also match
1571    between the parent and its descendants.  (That restriction does not apply
1572    to index-based constraints, however.)
1573    Also, because selecting from the parent also selects from its descendants,
1574    a constraint on the parent cannot be marked valid unless it is also marked
1575    valid for those descendants.  In all of these cases, <command>ALTER TABLE
1576    ONLY</command> will be rejected.
1577   </para>
1578
1579   <para>
1580    A recursive <literal>DROP COLUMN</literal> operation will remove a
1581    descendant table's column only if the descendant does not inherit
1582    that column from any other parents and never had an independent
1583    definition of the column.  A nonrecursive <literal>DROP
1584    COLUMN</literal> (i.e., <command>ALTER TABLE ONLY ... DROP
1585    COLUMN</command>) never removes any descendant columns, but
1586    instead marks them as independently defined rather than inherited.
1587    A nonrecursive <literal>DROP COLUMN</literal> command will fail for a
1588    partitioned table, because all partitions of a table must have the same
1589    columns as the partitioning root.
1590   </para>
1591
1592   <para>
1593    The actions for identity columns (<literal>ADD
1594    GENERATED</literal>, <literal>SET</literal> etc., <literal>DROP
1595    IDENTITY</literal>), as well as the actions
1596    <literal>CLUSTER</literal>, <literal>OWNER</literal>,
1597    and <literal>TABLESPACE</literal> never recurse to descendant tables;
1598    that is, they always act as though <literal>ONLY</literal> were specified.
1599    Actions affecting trigger states recurse to partitions of partitioned
1600    tables (unless <literal>ONLY</literal> is specified), but never to
1601    traditional-inheritance descendants.
1602    Adding a constraint recurses only for <literal>CHECK</literal> constraints
1603    that are not marked <literal>NO INHERIT</literal>.
1604   </para>
1605
1606   <para>
1607    Changing any part of a system catalog table is not permitted.
1608   </para>
1609
1610   <para>
1611    Refer to <xref linkend="sql-createtable"/> for a further description of valid
1612    parameters. <xref linkend="ddl"/> has further information on
1613    inheritance.
1614   </para>
1615 </refsect1>

1617 <refsect1>
1618  <title>Examples</title>
1619
1620  <para>
1621   To add a column of type <type>varchar</type> to a table:
1622<programlisting>
1623ALTER TABLE distributors ADD COLUMN address varchar(30);
1624</programlisting>
1625   That will cause all existing rows in the table to be filled with null
1626   values for the new column.
1627  </para>
1628
1629  <para>
1630   To add a column with a non-null default:
1631<programlisting>
1632ALTER TABLE measurements
1633  ADD COLUMN mtime timestamp with time zone DEFAULT now();
1634</programlisting>
1635   Existing rows will be filled with the current time as the value of the
1636   new column, and then new rows will receive the time of their insertion.
1637  </para>
1638
1639  <para>
1640   To add a column and fill it with a value different from the default to
1641   be used later:
1642<programlisting>
1643ALTER TABLE transactions
1644  ADD COLUMN status varchar(30) DEFAULT 'old',
1645  ALTER COLUMN status SET default 'current';
1646</programlisting>
1647   Existing rows will be filled with <literal>old</literal>, but then
1648   the default for subsequent commands will be <literal>current</literal>.
1649   The effects are the same as if the two sub-commands had been issued
1650   in separate <command>ALTER TABLE</command> commands.
1651  </para>
1652
1653  <para>
1654   To drop a column from a table:
1655<programlisting>
1656ALTER TABLE distributors DROP COLUMN address RESTRICT;
1657</programlisting>
1658  </para>
1659
1660  <para>
1661   To change the types of two existing columns in one operation:
1662<programlisting>
1663ALTER TABLE distributors
1664    ALTER COLUMN address TYPE varchar(80),
1665    ALTER COLUMN name TYPE varchar(100);
1666</programlisting>
1667  </para>
1668
1669  <para>
1670   To change an integer column containing Unix timestamps to <type>timestamp
1671   with time zone</type> via a <literal>USING</literal> clause:
1672<programlisting>
1673ALTER TABLE foo
1674    ALTER COLUMN foo_timestamp SET DATA TYPE timestamp with time zone
1675    USING
1676        timestamp with time zone 'epoch' + foo_timestamp * interval '1 second';
1677</programlisting>
1678  </para>
1679
1680  <para>
1681   The same, when the column has a default expression that won't automatically
1682   cast to the new data type:
1683<programlisting>
1684ALTER TABLE foo
1685    ALTER COLUMN foo_timestamp DROP DEFAULT,
1686    ALTER COLUMN foo_timestamp TYPE timestamp with time zone
1687    USING
1688        timestamp with time zone 'epoch' + foo_timestamp * interval '1 second',
1689    ALTER COLUMN foo_timestamp SET DEFAULT now();
1690</programlisting>
1691  </para>
1692
1693  <para>
1694   To rename an existing column:
1695<programlisting>
1696ALTER TABLE distributors RENAME COLUMN address TO city;
1697</programlisting>
1698  </para>
1699
1700  <para>
1701   To rename an existing table:
1702<programlisting>
1703ALTER TABLE distributors RENAME TO suppliers;
1704</programlisting>
1705  </para>
1706
1707  <para>
1708   To rename an existing constraint:
1709<programlisting>
1710ALTER TABLE distributors RENAME CONSTRAINT zipchk TO zip_check;
1711</programlisting>
1712  </para>
1713
1714  <para>
1715   To add a not-null constraint to a column:
1716<programlisting>
1717ALTER TABLE distributors ALTER COLUMN street SET NOT NULL;
1718</programlisting>
1719   To remove a not-null constraint from a column:
1720<programlisting>
1721ALTER TABLE distributors ALTER COLUMN street DROP NOT NULL;
1722</programlisting>
1723  </para>
1724
1725  <para>
1726   To add a check constraint to a table and all its children:
1727<programlisting>
1728ALTER TABLE distributors ADD CONSTRAINT zipchk CHECK (char_length(zipcode) = 5);
1729</programlisting>
1730  </para>
1731
1732  <para>
1733   To add a check constraint only to a table and not to its children:
1734<programlisting>
1735ALTER TABLE distributors ADD CONSTRAINT zipchk CHECK (char_length(zipcode) = 5) NO INHERIT;
1736</programlisting>
1737   (The check constraint will not be inherited by future children, either.)
1738  </para>
1739
1740  <para>
1741   To remove a check constraint from a table and all its children:
1742<programlisting>
1743ALTER TABLE distributors DROP CONSTRAINT zipchk;
1744</programlisting>
1745  </para>
1746
1747  <para>
1748   To remove a check constraint from one table only:
1749<programlisting>
1750ALTER TABLE ONLY distributors DROP CONSTRAINT zipchk;
1751</programlisting>
1752   (The check constraint remains in place for any child tables.)
1753  </para>
1754
1755  <para>
1756   To add a foreign key constraint to a table:
1757<programlisting>
1758ALTER TABLE distributors ADD CONSTRAINT distfk FOREIGN KEY (address) REFERENCES addresses (address);
1759</programlisting>
1760  </para>
1761
1762  <para>
1763   To add a foreign key constraint to a table with the least impact on other work:
1764<programlisting>
1765ALTER TABLE distributors ADD CONSTRAINT distfk FOREIGN KEY (address) REFERENCES addresses (address) NOT VALID;
1766ALTER TABLE distributors VALIDATE CONSTRAINT distfk;
1767</programlisting>
1768  </para>
1769
1770  <para>
1771   To add a (multicolumn) unique constraint to a table:
1772<programlisting>
1773ALTER TABLE distributors ADD CONSTRAINT dist_id_zipcode_key UNIQUE (dist_id, zipcode);
1774</programlisting>
1775  </para>
1776
1777  <para>
1778   To add an automatically named primary key constraint to a table, noting
1779   that a table can only ever have one primary key:
1780<programlisting>
1781ALTER TABLE distributors ADD PRIMARY KEY (dist_id);
1782</programlisting>
1783  </para>
1784
1785  <para>
1786   To move a table to a different tablespace:
1787<programlisting>
1788ALTER TABLE distributors SET TABLESPACE fasttablespace;
1789</programlisting>
1790  </para>
1791
1792  <para>
1793   To move a table to a different schema:
1794<programlisting>
1795ALTER TABLE myschema.distributors SET SCHEMA yourschema;
1796</programlisting>
1797  </para>
1798
1799  <para>
1800   To recreate a primary key constraint, without blocking updates while the
1801   index is rebuilt:
1802<programlisting>
1803CREATE UNIQUE INDEX CONCURRENTLY dist_id_temp_idx ON distributors (dist_id);
1804ALTER TABLE distributors DROP CONSTRAINT distributors_pkey,
1805    ADD CONSTRAINT distributors_pkey PRIMARY KEY USING INDEX dist_id_temp_idx;
1806</programlisting></para>
1807
1808  <para>
1809   To attach a partition to a range-partitioned table:
1810<programlisting>
1811ALTER TABLE measurement
1812    ATTACH PARTITION measurement_y2016m07 FOR VALUES FROM ('2016-07-01') TO ('2016-08-01');
1813</programlisting></para>
1814
1815  <para>
1816   To attach a partition to a list-partitioned table:
1817<programlisting>
1818ALTER TABLE cities
1819    ATTACH PARTITION cities_ab FOR VALUES IN ('a', 'b');
1820</programlisting></para>
1821
1822  <para>
1823   To attach a partition to a hash-partitioned table:
1824<programlisting>
1825ALTER TABLE orders
1826    ATTACH PARTITION orders_p4 FOR VALUES WITH (MODULUS 4, REMAINDER 3);
1827</programlisting></para>
1828
1829  <para>
1830   To attach a default partition to a partitioned table:
1831<programlisting>
1832ALTER TABLE cities
1833    ATTACH PARTITION cities_partdef DEFAULT;
1834</programlisting></para>
1835
1836  <para>
1837   To detach a partition from a partitioned table:
1838<programlisting>
1839ALTER TABLE measurement
1840    DETACH PARTITION measurement_y2015m12;
1841</programlisting></para>

1845 <refsect1>
1846  <title>Compatibility</title>
1847
1848  <para>
1849   The forms <literal>ADD [COLUMN]</literal>,
1850   <literal>DROP [COLUMN]</literal>, <literal>DROP IDENTITY</literal>, <literal>RESTART</literal>,
1851   <literal>SET DEFAULT</literal>, <literal>SET DATA TYPE</literal> (without <literal>USING</literal>),
1852   <literal>SET GENERATED</literal>, and <literal>SET <replaceable>sequence_option</replaceable></literal>
1853   conform with the SQL standard.
1854   The form <literal>ADD <replaceable>table_constraint</replaceable></literal>
1855   conforms with the SQL standard when the <literal>USING INDEX</literal> and
1856   <literal>NOT VALID</literal> clauses are omitted and the constraint type is
1857   one of <literal>CHECK</literal>, <literal>UNIQUE</literal>, <literal>PRIMARY KEY</literal>,
1858   or <literal>REFERENCES</literal>.
1859   The other forms are
1860   <productname>PostgreSQL</productname> extensions of the SQL standard.
1861   Also, the ability to specify more than one manipulation in a single
1862   <command>ALTER TABLE</command> command is an extension.
1863  </para>
1864
1865  <para>
1866   <command>ALTER TABLE DROP COLUMN</command> can be used to drop the only
1867   column of a table, leaving a zero-column table.  This is an
1868   extension of SQL, which disallows zero-column tables.
1869  </para>
