3<chapter id="ddl">
4 <title>Data Definition</title>
5
6 <para>
7  This chapter covers how one creates the database structures that
8  will hold one's data.  In a relational database, the raw data is
9  stored in tables, so the majority of this chapter is devoted to
10  explaining how tables are created and modified and what features are
11  available to control what data is stored in the tables.
12  Subsequently, we discuss how tables can be organized into
13  schemas, and how privileges can be assigned to tables.  Finally,
14  we will briefly look at other features that affect the data storage,
15  such as inheritance, table partitioning, views, functions, and
16  triggers.
17 </para>
18
19 <sect1 id="ddl-basics">
20  <title>Table Basics</title>
21
22  <indexterm zone="ddl-basics">
23   <primary>table</primary>
24  </indexterm>
25
26  <indexterm>
27   <primary>row</primary>
28  </indexterm>
29
30  <indexterm>
31   <primary>column</primary>
32  </indexterm>
33
34  <para>
35   A table in a relational database is much like a table on paper: It
36   consists of rows and columns.  The number and order of the columns
37   is fixed, and each column has a name.  The number of rows is
38   variable &mdash; it reflects how much data is stored at a given moment.
39   SQL does not make any guarantees about the order of the rows in a
40   table.  When a table is read, the rows will appear in an unspecified order,
41   unless sorting is explicitly requested.  This is covered in <xref
42   linkend="queries"/>.  Furthermore, SQL does not assign unique
43   identifiers to rows, so it is possible to have several completely
44   identical rows in a table.  This is a consequence of the
45   mathematical model that underlies SQL but is usually not desirable.
46   Later in this chapter we will see how to deal with this issue.
47  </para>
48
49  <para>
50   Each column has a data type.  The data type constrains the set of
51   possible values that can be assigned to a column and assigns
52   semantics to the data stored in the column so that it can be used
53   for computations.  For instance, a column declared to be of a
54   numerical type will not accept arbitrary text strings, and the data
55   stored in such a column can be used for mathematical computations.
56   By contrast, a column declared to be of a character string type
57   will accept almost any kind of data but it does not lend itself to
58   mathematical calculations, although other operations such as string
59   concatenation are available.
60  </para>
61
62  <para>
63   <productname>PostgreSQL</productname> includes a sizable set of
64   built-in data types that fit many applications.  Users can also
65   define their own data types.  Most built-in data types have obvious
66   names and semantics, so we defer a detailed explanation to <xref
67   linkend="datatype"/>.  Some of the frequently used data types are
68   <type>integer</type> for whole numbers, <type>numeric</type> for
69   possibly fractional numbers, <type>text</type> for character
70   strings, <type>date</type> for dates, <type>time</type> for
71   time-of-day values, and <type>timestamp</type> for values
72   containing both date and time.
73  </para>
74
75  <indexterm>
76   <primary>table</primary>
77   <secondary>creating</secondary>
78  </indexterm>
79
80  <para>
81   To create a table, you use the aptly named <xref
82   linkend="sql-createtable"/> command.
83   In this command you specify at least a name for the new table, the
84   names of the columns and the data type of each column.  For
85   example:
86<programlisting>
87CREATE TABLE my_first_table (
88    first_column text,
89    second_column integer
90);
91</programlisting>
92   This creates a table named <literal>my_first_table</literal> with
93   two columns.  The first column is named
94   <literal>first_column</literal> and has a data type of
95   <type>text</type>; the second column has the name
96   <literal>second_column</literal> and the type <type>integer</type>.
97   The table and column names follow the identifier syntax explained
98   in <xref linkend="sql-syntax-identifiers"/>.  The type names are
99   usually also identifiers, but there are some exceptions.  Note that the
100   column list is comma-separated and surrounded by parentheses.
101  </para>
102
103  <para>
104   Of course, the previous example was heavily contrived.  Normally,
105   you would give names to your tables and columns that convey what
106   kind of data they store.  So let's look at a more realistic
107   example:
108<programlisting>
109CREATE TABLE products (
110    product_no integer,
111    name text,
112    price numeric
113);
114</programlisting>
115   (The <type>numeric</type> type can store fractional components, as
116   would be typical of monetary amounts.)
117  </para>
118
119  <tip>
120   <para>
121    When you create many interrelated tables it is wise to choose a
122    consistent naming pattern for the tables and columns.  For
123    instance, there is a choice of using singular or plural nouns for
124    table names, both of which are favored by some theorist or other.
125   </para>
126  </tip>
127
128  <para>
129   There is a limit on how many columns a table can contain.
130   Depending on the column types, it is between 250 and 1600.
131   However, defining a table with anywhere near this many columns is
132   highly unusual and often a questionable design.
133  </para>
134
135  <indexterm>
136   <primary>table</primary>
137   <secondary>removing</secondary>
138  </indexterm>
139
140  <para>
141   If you no longer need a table, you can remove it using the <xref
142   linkend="sql-droptable"/> command.
143   For example:
144<programlisting>
145DROP TABLE my_first_table;
146DROP TABLE products;
147</programlisting>
148   Attempting to drop a table that does not exist is an error.
149   Nevertheless, it is common in SQL script files to unconditionally
150   try to drop each table before creating it, ignoring any error
151   messages, so that the script works whether or not the table exists.
152   (If you like, you can use the <literal>DROP TABLE IF EXISTS</literal> variant
153   to avoid the error messages, but this is not standard SQL.)
154  </para>
155
156  <para>
157   If you need to modify a table that already exists, see <xref
158   linkend="ddl-alter"/> later in this chapter.
159  </para>
160
161  <para>
162   With the tools discussed so far you can create fully functional
163   tables.  The remainder of this chapter is concerned with adding
164   features to the table definition to ensure data integrity,
165   security, or convenience.  If you are eager to fill your tables with
166   data now you can skip ahead to <xref linkend="dml"/> and read the
167   rest of this chapter later.
168  </para>

171 <sect1 id="ddl-default">
172  <title>Default Values</title>
173
174  <indexterm zone="ddl-default">
175   <primary>default value</primary>
176  </indexterm>
177
178  <para>
179   A column can be assigned a default value.  When a new row is
180   created and no values are specified for some of the columns, those
181   columns will be filled with their respective default values.  A
182   data manipulation command can also request explicitly that a column
183   be set to its default value, without having to know what that value is.
184   (Details about data manipulation commands are in <xref linkend="dml"/>.)
185  </para>
186
187  <para>
188   <indexterm><primary>null value</primary><secondary>default value</secondary></indexterm>
189   If no default value is declared explicitly, the default value is the
190   null value.  This usually makes sense because a null value can
191   be considered to represent unknown data.
192  </para>
193
194  <para>
195   In a table definition, default values are listed after the column
196   data type.  For example:
197<programlisting>
198CREATE TABLE products (
199    product_no integer,
200    name text,
201    price numeric <emphasis>DEFAULT 9.99</emphasis>
202);
203</programlisting>
204  </para>
205
206  <para>
207   The default value can be an expression, which will be
208   evaluated whenever the default value is inserted
209   (<emphasis>not</emphasis> when the table is created).  A common example
210   is for a <type>timestamp</type> column to have a default of <literal>CURRENT_TIMESTAMP</literal>,
211   so that it gets set to the time of row insertion.  Another common
212   example is generating a <quote>serial number</quote> for each row.
213   In <productname>PostgreSQL</productname> this is typically done by
214   something like:
215<programlisting>
216CREATE TABLE products (
217    product_no integer <emphasis>DEFAULT nextval('products_product_no_seq')</emphasis>,
218    ...
219);
220</programlisting>
221   where the <literal>nextval()</literal> function supplies successive values
222   from a <firstterm>sequence object</firstterm> (see <xref
223   linkend="functions-sequence"/>). This arrangement is sufficiently common
224   that there's a special shorthand for it:
225<programlisting>
226CREATE TABLE products (
227    product_no <emphasis>SERIAL</emphasis>,
228    ...
229);
230</programlisting>
231   The <literal>SERIAL</literal> shorthand is discussed further in <xref
232   linkend="datatype-serial"/>.
233  </para>
234 </sect1>

236 <sect1 id="ddl-identity-columns">
237  <title>Identity Columns</title>
238
239  <indexterm zone="ddl-identity-columns">
240   <primary>identity column</primary>
241  </indexterm>
242
243  <para>
244   An identity column is a special column that is generated automatically from
245   an implicit sequence.  It can be used to generate key values.
246  </para>
247
248  <para>
249   To create an identity column, use the <literal>GENERATED ...
250   AS IDENTITY</literal> clause in <command>CREATE TABLE</command>, for example:
251<programlisting>
252CREATE TABLE people (
253    id bigint <emphasis>GENERATED ALWAYS AS IDENTITY</emphasis>,
254    ...,
255);
256</programlisting>
257   or alternatively
258<programlisting>
259CREATE TABLE people (
260    id bigint <emphasis>GENERATED BY DEFAULT AS IDENTITY</emphasis>,
261    ...,
262);
263</programlisting>
264   See <xref linkend="sql-createtable"/> for more details.
265  </para>
266
267  <para>
268   If an <command>INSERT</command> command is executed on the table with the
269   identity column and no value is explicitly specified for the identity
270   column, then a value generated by the implicit sequence is inserted.  For
271   example, with the above definitions and assuming additional appropriate
272   columns, writing
273<programlisting>
274INSERT INTO people (name, address) VALUES ('A', 'foo');
275INSERT INTO people (name, address) VALUES ('B', 'bar');
276</programlisting>
277   would generate values for the <literal>id</literal> column starting at 1
278   and result in the following table data:
279<screen>
280 id | name | address
281----+------+---------
282  1 | A    | foo
283  2 | B    | bar
284</screen>
285   Alternatively, the keyword <literal>DEFAULT</literal> can be specified in
286   place of a value to explicitly request the sequence-generated value, like
287<programlisting>
288INSERT INTO people (id, name, address) VALUES (<emphasis>DEFAULT</emphasis>, 'C', 'baz');
289</programlisting>
290   Similarly, the keyword <literal>DEFAULT</literal> can be used in
291   <command>UPDATE</command> commands.
292  </para>
293
294  <para>
295   Thus, in many ways, an identity column behaves like a column with a default
296   value.
297  </para>
298
299  <para>
300   The clauses <literal>ALWAYS</literal> and <literal>BY DEFAULT</literal> in
301   the column definition determine how explicitly user-specified values are
302   handled in <command>INSERT</command> and <command>UPDATE</command>
303   commands.  In an <command>INSERT</command> command, if
304   <literal>ALWAYS</literal> is selected, a user-specified value is only
305   accepted if the <command>INSERT</command> statement specifies
306   <literal>OVERRIDING SYSTEM VALUE</literal>.  If <literal>BY
307   DEFAULT</literal> is selected, then the user-specified value takes
308   precedence.  Thus, using <literal>BY DEFAULT</literal> results in a
309   behavior more similar to default values, where the default value can be
310   overridden by an explicit value, whereas <literal>ALWAYS</literal> provides
311   some more protection against accidentally inserting an explicit value.
312  </para>
313
314  <para>
315   The data type of an identity column must be one of the data types supported
316   by sequences.  (See <xref linkend="sql-createsequence"/>.)  The properties
317   of the associated sequence may be specified when creating an identity
318   column (see <xref linkend="sql-createtable"/>) or changed afterwards (see
319   <xref linkend="sql-altertable"/>).
320  </para>
321
322  <para>
323   An identity column is automatically marked as <literal>NOT NULL</literal>.
324   An identity column, however, does not guarantee uniqueness.  (A sequence
325   normally returns unique values, but a sequence could be reset, or values
326   could be inserted manually into the identity column, as discussed above.)
327   Uniqueness would need to be enforced using a <literal>PRIMARY KEY</literal>
328   or <literal>UNIQUE</literal> constraint.
329  </para>
330
331  <para>
332   In table inheritance hierarchies, identity columns and their properties in
333   a child table are independent of those in its parent tables.  A child table
334   does not inherit identity columns or their properties automatically from
335   the parent. During <command>INSERT</command> or <command>UPDATE</command>,
336   a column is treated as an identity column if that column is an identity
337   column in the table named in the statement, and the corresponding identity
338   properties are applied.
339  </para>
340
341  <para>
342   Partitions inherit identity columns from the partitioned table.  They
343   cannot have their own identity columns.  The properties of a given identity
344   column are consistent across all the partitions in the partition hierarchy.
345  </para>
346 </sect1>

348 <sect1 id="ddl-generated-columns">
349  <title>Generated Columns</title>
350
351  <indexterm zone="ddl-generated-columns">
352   <primary>generated column</primary>
353  </indexterm>
354
355  <para>
356   A generated column is a special column that is always computed from other
357   columns.  Thus, it is for columns what a view is for tables.  There are two
358   kinds of generated columns: stored and virtual.  A stored generated column
359   is computed when it is written (inserted or updated) and occupies storage
360   as if it were a normal column.  A virtual generated column occupies no
361   storage and is computed when it is read.  Thus, a virtual generated column
362   is similar to a view and a stored generated column is similar to a
363   materialized view (except that it is always updated automatically).
364  </para>
365
366  <para>
367   To create a generated column, use the <literal>GENERATED ALWAYS
368   AS</literal> clause in <command>CREATE TABLE</command>, for example:
369<programlisting>
370CREATE TABLE people (
371    ...,
372    height_cm numeric,
373    height_in numeric <emphasis>GENERATED ALWAYS AS (height_cm / 2.54)</emphasis>
374);
375</programlisting>
376   A generated column is by default of the virtual kind.  Use the keywords
377   <literal>VIRTUAL</literal> or <literal>STORED</literal> to make the choice
378   explicit.  See <xref linkend="sql-createtable"/> for more details.
379  </para>
380
381  <para>
382   A generated column cannot be written to directly.  In
383   <command>INSERT</command> or <command>UPDATE</command> commands, a value
384   cannot be specified for a generated column, but the keyword
385   <literal>DEFAULT</literal> may be specified.
386  </para>
387
388  <para>
389   Consider the differences between a column with a default and a generated
390   column.  The column default is evaluated once when the row is first
391   inserted if no other value was provided; a generated column is updated
392   whenever the row changes and cannot be overridden.  A column default may
393   not refer to other columns of the table; a generation expression would
394   normally do so.  A column default can use volatile functions, for example
395   <literal>random()</literal> or functions referring to the current time;
396   this is not allowed for generated columns.
397  </para>
398
399  <para>
400   Several restrictions apply to the definition of generated columns and
401   tables involving generated columns:
402
403   <itemizedlist>
404    <listitem>
405     <para>
406      The generation expression can only use immutable functions and cannot
407      use subqueries or reference anything other than the current row in any
408      way.
409     </para>
410    </listitem>
411    <listitem>
412     <para>
413      A generation expression cannot reference another generated column.
414     </para>
415    </listitem>
416    <listitem>
417     <para>
418      A generation expression cannot reference a system column, except
419      <varname>tableoid</varname>.
420     </para>
421    </listitem>
422    <listitem>
423     <para>
424      A virtual generated column cannot have a user-defined type, and the
425      generation expression of a virtual generated column must not reference
426      user-defined functions or types, that is, it can only use built-in
427      functions or types.  This applies also indirectly, such as for functions
428      or types that underlie operators or casts.  (This restriction does not
429      exist for stored generated columns.)
430     </para>
431    </listitem>
432    <listitem>
433     <para>
434      A generated column cannot have a column default or an identity definition.
435     </para>
436    </listitem>
437    <listitem>
438     <para>
439      A generated column cannot be part of a partition key.
440     </para>
441    </listitem>
442    <listitem>
443     <para>
444      Foreign tables can have generated columns.  See <xref
445      linkend="sql-createforeigntable"/> for details.
446     </para>
447    </listitem>
448    <listitem>
449     <para>For inheritance and partitioning:</para>
450     <itemizedlist>
451      <listitem>
452       <para>
453        If a parent column is a generated column, its child column must also
454        be a generated column of the same kind (stored or virtual); however,
455        the child column can have a different generation expression.
456       </para>
457
458       <para>
459        For stored generated columns, the generation expression that is
460        actually applied during insert or update of a row is the one
461        associated with the table that the row is physically in.  (This is
462        unlike the behavior for column defaults: for those, the default value
463        associated with the table named in the query applies.)  For virtual
464        generated columns, the generation expression of the table named in the
465        query applies when a table is read.
466       </para>
467      </listitem>
468      <listitem>
469       <para>
470        If a parent column is not a generated column, its child column must
471        not be generated either.
472       </para>
473      </listitem>
474      <listitem>
475       <para>
476        For inherited tables, if you write a child column definition without
477        any <literal>GENERATED</literal> clause in <command>CREATE TABLE
478        ... INHERITS</command>, then its <literal>GENERATED</literal> clause
479        will automatically be copied from the parent.  <command>ALTER TABLE
480        ... INHERIT</command> will insist that parent and child columns
481        already match as to generation status, but it will not require their
482        generation expressions to match.
483       </para>
484      </listitem>
485      <listitem>
486       <para>
487        Similarly for partitioned tables, if you write a child column
488        definition without any <literal>GENERATED</literal> clause
489        in <command>CREATE TABLE ... PARTITION OF</command>, then
490        its <literal>GENERATED</literal> clause will automatically be copied
491        from the parent.  <command>ALTER TABLE ... ATTACH PARTITION</command>
492        will insist that parent and child columns already match as to
493        generation status, but it will not require their generation
494        expressions to match.
495       </para>
496      </listitem>
497      <listitem>
498       <para>
499        In case of multiple inheritance, if one parent column is a generated
500        column, then all parent columns must be generated columns.  If they
501        do not all have the same generation expression, then the desired
502        expression for the child must be specified explicitly.
503       </para>
504      </listitem>
505     </itemizedlist>
506    </listitem>
507   </itemizedlist>
508  </para>
509
510  <para>
511   Additional considerations apply to the use of generated columns.
512   <itemizedlist>
513    <listitem>
514     <para>
515      Generated columns maintain access privileges separately from their
516      underlying base columns.  So, it is possible to arrange it so that a
517      particular role can read from a generated column but not from the
518      underlying base columns.
519     </para>
520
521     <para>
522      For virtual generated columns, this is only fully secure if the
523      generation expression uses only leakproof functions (see <xref
524      linkend="sql-createfunction"/>), but this is not enforced by the system.
525     </para>
526    </listitem>
527    <listitem>
528     <para>
529      Privileges of functions used in generation expressions are checked when
530      the expression is actually executed, on write or read respectively, as
531      if the generation expression had been called directly from the query
532      using the generated column.  The user of a generated column must have
533      permissions to call all functions used by the generation expression.
534      Functions in the generation expression are executed with the privileges
535      of the user executing the query or the function owner, depending on
536      whether the functions are defined as <literal>SECURITY INVOKER</literal>
537      or <literal>SECURITY DEFINER</literal>.
538      <!-- matches create_view.sgml -->
539     </para>
540    </listitem>
541    <listitem>
542     <para>
543      Generated columns are, conceptually, updated after
544      <literal>BEFORE</literal> triggers have run.  Therefore, changes made to
545      base columns in a <literal>BEFORE</literal> trigger will be reflected in
546      generated columns.  But conversely, it is not allowed to access
547      generated columns in <literal>BEFORE</literal> triggers.
548     </para>
549    </listitem>
550    <listitem>
551     <para>
552      Generated columns are allowed to be replicated during logical replication
553      according to the <command>CREATE PUBLICATION</command> parameter
554      <link linkend="sql-createpublication-params-with-publish-generated-columns">
555      <literal>publish_generated_columns</literal></link> or by including them
556      in the column list of the <command>CREATE PUBLICATION</command> command.
557      This is currently only supported for stored generated columns.
558      See <xref linkend="logical-replication-gencols"/> for details.
559     </para>
560    </listitem>
561   </itemizedlist>
562  </para>
563 </sect1>

565 <sect1 id="ddl-constraints">
566  <title>Constraints</title>
567
568  <indexterm zone="ddl-constraints">
569   <primary>constraint</primary>
570  </indexterm>
571
572  <para>
573   Data types are a way to limit the kind of data that can be stored
574   in a table.  For many applications, however, the constraint they
575   provide is too coarse.  For example, a column containing a product
576   price should probably only accept positive values.  But there is no
577   standard data type that accepts only positive numbers.  Another issue is
578   that you might want to constrain column data with respect to other
579   columns or rows.  For example, in a table containing product
580   information, there should be only one row for each product number.
581  </para>
582
583  <para>
584   To that end, SQL allows you to define constraints on columns and
585   tables.  Constraints give you as much control over the data in your
586   tables as you wish.  If a user attempts to store data in a column
587   that would violate a constraint, an error is raised.  This applies
588   even if the value came from the default value definition.
589  </para>
590
591  <sect2 id="ddl-constraints-check-constraints">
592   <title>Check Constraints</title>
593
594   <indexterm>
595    <primary>check constraint</primary>
596   </indexterm>
597
598   <indexterm>
599    <primary>constraint</primary>
600    <secondary>check</secondary>
601   </indexterm>
602
603   <para>
604    A check constraint is the most generic constraint type.  It allows
605    you to specify that the value in a certain column must satisfy a
606    Boolean (truth-value) expression.  For instance, to require positive
607    product prices, you could use:
608<programlisting>
609CREATE TABLE products (
610    product_no integer,
611    name text,
612    price numeric <emphasis>CHECK (price &gt; 0)</emphasis>
613);
614</programlisting>
615   </para>
616
617   <para>
618    As you see, the constraint definition comes after the data type,
619    just like default value definitions.  Default values and
620    constraints can be listed in any order.  A check constraint
621    consists of the key word <literal>CHECK</literal> followed by an
622    expression in parentheses.  The check constraint expression should
623    involve the column thus constrained, otherwise the constraint
624    would not make too much sense.
625   </para>
626
627   <indexterm>
628    <primary>constraint</primary>
629    <secondary>name</secondary>
630   </indexterm>
631
632   <para>
633    You can also give the constraint a separate name.  This clarifies
634    error messages and allows you to refer to the constraint when you
635    need to change it.  The syntax is:
636<programlisting>
637CREATE TABLE products (
638    product_no integer,
639    name text,
640    price numeric <emphasis>CONSTRAINT positive_price</emphasis> CHECK (price &gt; 0)
641);
642</programlisting>
643    So, to specify a named constraint, use the key word
644    <literal>CONSTRAINT</literal> followed by an identifier followed
645    by the constraint definition.  (If you don't specify a constraint
646    name in this way, the system chooses a name for you.)
647   </para>
648
649   <para>
650    A check constraint can also refer to several columns.  Say you
651    store a regular price and a discounted price, and you want to
652    ensure that the discounted price is lower than the regular price:
653<programlisting>
654CREATE TABLE products (
655    product_no integer,
656    name text,
657    price numeric CHECK (price &gt; 0),
658    discounted_price numeric CHECK (discounted_price &gt; 0),
659    <emphasis>CHECK (price &gt; discounted_price)</emphasis>
660);
661</programlisting>
662   </para>
663
664   <para>
665    The first two constraints should look familiar.  The third one
666    uses a new syntax.  It is not attached to a particular column,
667    instead it appears as a separate item in the comma-separated
668    column list.  Column definitions and these constraint
669    definitions can be listed in mixed order.
670   </para>
671
672   <para>
673    We say that the first two constraints are column constraints, whereas the
674    third one is a table constraint because it is written separately
675    from any one column definition.  Column constraints can also be
676    written as table constraints, while the reverse is not necessarily
677    possible, since a column constraint is supposed to refer to only the
678    column it is attached to.  (<productname>PostgreSQL</productname> doesn't
679    enforce that rule, but you should follow it if you want your table
680    definitions to work with other database systems.)  The above example could
681    also be written as:
682<programlisting>
683CREATE TABLE products (
684    product_no integer,
685    name text,
686    price numeric,
687    CHECK (price &gt; 0),
688    discounted_price numeric,
689    CHECK (discounted_price &gt; 0),
690    CHECK (price &gt; discounted_price)
691);
692</programlisting>
693    or even:
694<programlisting>
695CREATE TABLE products (
696    product_no integer,
697    name text,
698    price numeric CHECK (price &gt; 0),
699    discounted_price numeric,
700    CHECK (discounted_price &gt; 0 AND price &gt; discounted_price)
701);
702</programlisting>
703    It's a matter of taste.
704   </para>
705
706   <para>
707    Names can be assigned to table constraints in the same way as
708    column constraints:
709<programlisting>
710CREATE TABLE products (
711    product_no integer,
712    name text,
713    price numeric,
714    CHECK (price &gt; 0),
715    discounted_price numeric,
716    CHECK (discounted_price &gt; 0),
717    <emphasis>CONSTRAINT valid_discount</emphasis> CHECK (price &gt; discounted_price)
718);
719</programlisting>
720   </para>
721
722   <indexterm>
723    <primary>null value</primary>
724    <secondary sortas="check constraints">with check constraints</secondary>
725   </indexterm>
726
727   <para>
728    It should be noted that a check constraint is satisfied if the
729    check expression evaluates to true or the null value.  Since most
730    expressions will evaluate to the null value if any operand is null,
731    they will not prevent null values in the constrained columns.  To
732    ensure that a column does not contain null values, the not-null
733    constraint described in the next section can be used.
734   </para>
735
736   <note>
737    <para>
738     <productname>PostgreSQL</productname> does not support
739     <literal>CHECK</literal> constraints that reference table data other than
740     the new or updated row being checked.  While a <literal>CHECK</literal>
741     constraint that violates this rule may appear to work in simple
742     tests, it cannot guarantee that the database will not reach a state
743     in which the constraint condition is false (due to subsequent changes
744     of the other row(s) involved).  This would cause a database dump and
745     restore to fail.  The restore could fail even when the complete
746     database state is consistent with the constraint, due to rows not
747     being loaded in an order that will satisfy the constraint.  If
748     possible, use <literal>UNIQUE</literal>, <literal>EXCLUDE</literal>,
749     or <literal>FOREIGN KEY</literal> constraints to express
750     cross-row and cross-table restrictions.
751    </para>
752
753    <para>
754     If what you desire is a one-time check against other rows at row
755     insertion, rather than a continuously-maintained consistency
756     guarantee, a custom <link linkend="triggers">trigger</link> can be used
757     to implement that.  (This approach avoids the dump/restore problem because
758     <application>pg_dump</application> does not reinstall triggers until after
759     restoring data, so that the check will not be enforced during a
760     dump/restore.)
761    </para>
762   </note>
763
764   <note>
765    <para>
766     <productname>PostgreSQL</productname> assumes that
767     <literal>CHECK</literal> constraints' conditions are immutable, that
768     is, they will always give the same result for the same input row.
769     This assumption is what justifies examining <literal>CHECK</literal>
770     constraints only when rows are inserted or updated, and not at other
771     times.  (The warning above about not referencing other table data is
772     really a special case of this restriction.)
773    </para>
774
775    <para>
776     An example of a common way to break this assumption is to reference a
777     user-defined function in a <literal>CHECK</literal> expression, and
778     then change the behavior of that
779     function.  <productname>PostgreSQL</productname> does not disallow
780     that, but it will not notice if there are rows in the table that now
781     violate the <literal>CHECK</literal> constraint. That would cause a
782     subsequent database dump and restore to fail.
783     The recommended way to handle such a change is to drop the constraint
784     (using <command>ALTER TABLE</command>), adjust the function definition,
785     and re-add the constraint, thereby rechecking it against all table rows.
786    </para>
787   </note>
788  </sect2>

790  <sect2 id="ddl-constraints-not-null">
791   <title>Not-Null Constraints</title>
792
793   <indexterm>
794    <primary>not-null constraint</primary>
795   </indexterm>
796
797   <indexterm>
798    <primary>constraint</primary>
799    <secondary>NOT NULL</secondary>
800   </indexterm>
801
802   <para>
803    A not-null constraint simply specifies that a column must not
804    assume the null value.  A syntax example:
805<programlisting>
806CREATE TABLE products (
807    product_no integer <emphasis>NOT NULL</emphasis>,
808    name text <emphasis>NOT NULL</emphasis>,
809    price numeric
810);
811</programlisting>
812    An explicit constraint name can also be specified, for example:
813<programlisting>
814CREATE TABLE products (
815    product_no integer NOT NULL,
816    name text <emphasis>CONSTRAINT products_name_not_null</emphasis> NOT NULL,
817    price numeric
818);
819</programlisting>
820   </para>
821
822   <para>
823    A not-null constraint is usually written as a column constraint.  The
824    syntax for writing it as a table constraint is
825<programlisting>
826CREATE TABLE products (
827    product_no integer,
828    name text,
829    price numeric,
830    <emphasis>NOT NULL product_no</emphasis>,
831    <emphasis>NOT NULL name</emphasis>
832);
833</programlisting>
834    But this syntax is not standard and mainly intended for use by
835    <application>pg_dump</application>.
836   </para>
837
838   <para>
839    A not-null constraint is functionally equivalent to creating a check
840    constraint <literal>CHECK (<replaceable>column_name</replaceable>
841    IS NOT NULL)</literal>, but in
842    <productname>PostgreSQL</productname> creating an explicit
843    not-null constraint is more efficient.
844   </para>
845
846   <para>
847    Of course, a column can have more than one constraint.  Just write
848    the constraints one after another:
849<programlisting>
850CREATE TABLE products (
851    product_no integer NOT NULL,
852    name text NOT NULL,
853    price numeric NOT NULL CHECK (price &gt; 0)
854);
855</programlisting>
856    The order doesn't matter.  It does not necessarily determine in which
857    order the constraints are checked.
858   </para>
859
860   <para>
861    However, a column can have at most one explicit not-null constraint.
862   </para>
863
864   <para>
865    The <literal>NOT NULL</literal> constraint has an inverse: the
866    <literal>NULL</literal> constraint.  This does not mean that the
867    column must be null, which would surely be useless.  Instead, this
868    simply selects the default behavior that the column might be null.
869    The <literal>NULL</literal> constraint is not present in the SQL
870    standard and should not be used in portable applications.  (It was
871    only added to <productname>PostgreSQL</productname> to be
872    compatible with some other database systems.)  Some users, however,
873    like it because it makes it easy to toggle the constraint in a
874    script file.  For example, you could start with:
875<programlisting>
876CREATE TABLE products (
877    product_no integer NULL,
878    name text NULL,
879    price numeric NULL
880);
881</programlisting>
882    and then insert the <literal>NOT</literal> key word where desired.
883   </para>
884
885   <tip>
886    <para>
887     In most database designs the majority of columns should be marked
888     not null.
889    </para>
890   </tip>
891  </sect2>

893  <sect2 id="ddl-constraints-unique-constraints">
894   <title>Unique Constraints</title>
895
896   <indexterm>
897    <primary>unique constraint</primary>
898   </indexterm>
899
900   <indexterm>
901    <primary>constraint</primary>
902    <secondary>unique</secondary>
903   </indexterm>
904
905   <para>
906    Unique constraints ensure that the data contained in a column, or a
907    group of columns, is unique among all the rows in the
908    table.  The syntax is:
909<programlisting>
910CREATE TABLE products (
911    product_no integer <emphasis>UNIQUE</emphasis>,
912    name text,
913    price numeric
914);
915</programlisting>
916    when written as a column constraint, and:
917<programlisting>
918CREATE TABLE products (
919    product_no integer,
920    name text,
921    price numeric,
922    <emphasis>UNIQUE (product_no)</emphasis>
923);
924</programlisting>
925    when written as a table constraint.
926   </para>
927
928   <para>
929    To define a unique constraint for a group of columns, write it as a
930    table constraint with the column names separated by commas:
931<programlisting>
932CREATE TABLE example (
933    a integer,
934    b integer,
935    c integer,
936    <emphasis>UNIQUE (a, c)</emphasis>
937);
938</programlisting>
939    This specifies that the combination of values in the indicated columns
940    is unique across the whole table, though any one of the columns
941    need not be (and ordinarily isn't) unique.
942   </para>
943
944   <para>
945    You can assign your own name for a unique constraint, in the usual way:
946<programlisting>
947CREATE TABLE products (
948    product_no integer <emphasis>CONSTRAINT must_be_different</emphasis> UNIQUE,
949    name text,
950    price numeric
951);
952</programlisting>
953   </para>
954
955   <para>
956    Adding a unique constraint will automatically create a unique B-tree
957    index on the column or group of columns listed in the constraint.
958    A uniqueness restriction covering only some rows cannot be written as
959    a unique constraint, but it is possible to enforce such a restriction by
960    creating a unique <link linkend="indexes-partial">partial index</link>.
961   </para>
962
963   <indexterm>
964    <primary>null value</primary>
965    <secondary sortas="unique constraints">with unique constraints</secondary>
966   </indexterm>
967
968   <para>
969    In general, a unique constraint is violated if there is more than
970    one row in the table where the values of all of the
971    columns included in the constraint are equal.
972    By default, two null values are not considered equal in this
973    comparison.  That means even in the presence of a
974    unique constraint it is possible to store duplicate
975    rows that contain a null value in at least one of the constrained
976    columns.  This behavior can be changed by adding the clause <literal>NULLS
977    NOT DISTINCT</literal>, like
978<programlisting>
979CREATE TABLE products (
980    product_no integer UNIQUE <emphasis>NULLS NOT DISTINCT</emphasis>,
981    name text,
982    price numeric
983);
984</programlisting>
985    or
986<programlisting>
987CREATE TABLE products (
988    product_no integer,
989    name text,
990    price numeric,
991    UNIQUE <emphasis>NULLS NOT DISTINCT</emphasis> (product_no)
992);
993</programlisting>
994    The default behavior can be specified explicitly using <literal>NULLS
995    DISTINCT</literal>.  The default null treatment in unique constraints is
996    implementation-defined according to the SQL standard, and other
997    implementations have a different behavior.  So be careful when developing
998    applications that are intended to be portable.
999   </para>
1000  </sect2>

1002  <sect2 id="ddl-constraints-primary-keys">
1003   <title>Primary Keys</title>
1004
1005   <indexterm>
1006    <primary>primary key</primary>
1007   </indexterm>
1008
1009   <indexterm>
1010    <primary>constraint</primary>
1011    <secondary>primary key</secondary>
1012   </indexterm>
1013
1014   <para>
1015    A primary key constraint indicates that a column, or group of columns,
1016    can be used as a unique identifier for rows in the table.  This
1017    requires that the values be both unique and not null.  So, the following
1018    two table definitions accept the same data:
1019<programlisting>
1020CREATE TABLE products (
1021    product_no integer UNIQUE NOT NULL,
1022    name text,
1023    price numeric
1024);
1025</programlisting>
1026
1027<programlisting>
1028CREATE TABLE products (
1029    product_no integer <emphasis>PRIMARY KEY</emphasis>,
1030    name text,
1031    price numeric
1032);
1033</programlisting>
1034   </para>
1035
1036   <para>
1037    Primary keys can span more than one column; the syntax
1038    is similar to unique constraints:
1039<programlisting>
1040CREATE TABLE example (
1041    a integer,
1042    b integer,
1043    c integer,
1044    <emphasis>PRIMARY KEY (a, c)</emphasis>
1045);
1046</programlisting>
1047   </para>
1048
1049   <para>
1050    Adding a primary key will automatically create a unique B-tree index
1051    on the column or group of columns listed in the primary key, and will
1052    force the column(s) to be marked <literal>NOT NULL</literal>.
1053   </para>
1054
1055   <para>
1056    A table can have at most one primary key.  (There can be any number
1057    of unique constraints, which combined with not-null constraints are functionally almost the
1058    same thing, but only one can be identified as the primary key.)
1059    Relational database theory
1060    dictates that every table must have a primary key.  This rule is
1061    not enforced by <productname>PostgreSQL</productname>, but it is
1062    usually best to follow it.
1063   </para>
1064
1065   <para>
1066    Primary keys are useful both for
1067    documentation purposes and for client applications.  For example,
1068    a GUI application that allows modifying row values probably needs
1069    to know the primary key of a table to be able to identify rows
1070    uniquely.  There are also various ways in which the database system
1071    makes use of a primary key if one has been declared; for example,
1072    the primary key defines the default target column(s) for foreign keys
1073    referencing its table.
1074   </para>
1075  </sect2>
1076
1077  <sect2 id="ddl-constraints-fk">
1078   <title>Foreign Keys</title>
1079
1080   <indexterm>
1081    <primary>foreign key</primary>
1082   </indexterm>
1083
1084   <indexterm>
1085    <primary>constraint</primary>
1086    <secondary>foreign key</secondary>
1087   </indexterm>
1088
1089   <indexterm>
1090    <primary>referential integrity</primary>
1091   </indexterm>
1092
1093   <para>
1094    A foreign key constraint specifies that the values in a column (or
1095    a group of columns) must match the values appearing in some row
1096    of another table.
1097    We say this maintains the <firstterm>referential
1098    integrity</firstterm> between two related tables.
1099   </para>
1100
1101   <para>
1102    Say you have the product table that we have used several times already:
1103<programlisting>
1104CREATE TABLE products (
1105    product_no integer PRIMARY KEY,
1106    name text,
1107    price numeric
1108);
1109</programlisting>
1110    Let's also assume you have a table storing orders of those
1111    products.  We want to ensure that the orders table only contains
1112    orders of products that actually exist.  So we define a foreign
1113    key constraint in the orders table that references the products
1114    table:
1115<programlisting>
1116CREATE TABLE orders (
1117    order_id integer PRIMARY KEY,
1118    product_no integer <emphasis>REFERENCES products (product_no)</emphasis>,
1119    quantity integer
1120);
1121</programlisting>
1122    Now it is impossible to create orders with non-NULL
1123    <structfield>product_no</structfield> entries that do not appear in the
1124    products table.
1125   </para>
1126
1127   <para>
1128    We say that in this situation the orders table is the
1129    <firstterm>referencing</firstterm> table and the products table is
1130    the <firstterm>referenced</firstterm> table.  Similarly, there are
1131    referencing and referenced columns.
1132   </para>
1133
1134   <para>
1135    You can also shorten the above command to:
1136<programlisting>
1137CREATE TABLE orders (
1138    order_id integer PRIMARY KEY,
1139    product_no integer <emphasis>REFERENCES products</emphasis>,
1140    quantity integer
1141);
1142</programlisting>
1143    because in absence of a column list the primary key of the
1144    referenced table is used as the referenced column(s).
1145   </para>
1146
1147   <para>
1148    You can assign your own name for a foreign key constraint,
1149    in the usual way.
1150   </para>
1151
1152   <para>
1153    A foreign key can also constrain and reference a group of columns.
1154    As usual, it then needs to be written in table constraint form.
1155    Here is a contrived syntax example:
1156<programlisting>
1157CREATE TABLE t1 (
1158  a integer PRIMARY KEY,
1159  b integer,
1160  c integer,
1161  <emphasis>FOREIGN KEY (b, c) REFERENCES other_table (c1, c2)</emphasis>
1162);
1163</programlisting>
1164    Of course, the number and type of the constrained columns need to
1165    match the number and type of the referenced columns.
1166   </para>
1167
1168   <indexterm>
1169    <primary>foreign key</primary>
1170    <secondary>self-referential</secondary>
1171   </indexterm>
1172
1173   <para>
1174    Sometimes it is useful for the <quote>other table</quote> of a
1175    foreign key constraint to be the same table; this is called
1176    a <firstterm>self-referential</firstterm> foreign key.  For
1177    example, if you want rows of a table to represent nodes of a tree
1178    structure, you could write
1179<programlisting>
1180CREATE TABLE tree (
1181    node_id integer PRIMARY KEY,
1182    parent_id integer REFERENCES tree,
1183    name text,
1184    ...
1185);
1186</programlisting>
1187    A top-level node would have NULL <structfield>parent_id</structfield>,
1188    while non-NULL <structfield>parent_id</structfield> entries would be
1189    constrained to reference valid rows of the table.
1190   </para>
1191
1192   <para>
1193    A table can have more than one foreign key constraint.  This is
1194    used to implement many-to-many relationships between tables.  Say
1195    you have tables about products and orders, but now you want to
1196    allow one order to contain possibly many products (which the
1197    structure above did not allow).  You could use this table structure:
1198<programlisting>
1199CREATE TABLE products (
1200    product_no integer PRIMARY KEY,
1201    name text,
1202    price numeric
1203);
1204
1205CREATE TABLE orders (
1206    order_id integer PRIMARY KEY,
1207    shipping_address text,
1208    ...
1209);
1210
1211CREATE TABLE order_items (
1212    product_no integer REFERENCES products,
1213    order_id integer REFERENCES orders,
1214    quantity integer,
1215    PRIMARY KEY (product_no, order_id)
1216);
1217</programlisting>
1218    Notice that the primary key overlaps with the foreign keys in
1219    the last table.
1220   </para>
1221
1222   <indexterm>
1223    <primary>CASCADE</primary>
1224    <secondary>foreign key action</secondary>
1225   </indexterm>
1226
1227   <indexterm>
1228    <primary>RESTRICT</primary>
1229    <secondary>foreign key action</secondary>
1230   </indexterm>
1231
1232   <para>
1233    We know that the foreign keys disallow creation of orders that
1234    do not relate to any products.  But what if a product is removed
1235    after an order is created that references it?  SQL allows you to
1236    handle that as well.  Intuitively, we have a few options:
1237    <itemizedlist spacing="compact">
1238     <listitem><para>Disallow deleting a referenced product</para></listitem>
1239     <listitem><para>Delete the orders as well</para></listitem>
1240     <listitem><para>Something else?</para></listitem>
1241    </itemizedlist>
1242   </para>
1243
1244   <para>
1245    To illustrate this, let's implement the following policy on the
1246    many-to-many relationship example above: when someone wants to
1247    remove a product that is still referenced by an order (via
1248    <literal>order_items</literal>), we disallow it.  If someone
1249    removes an order, the order items are removed as well:
1250<programlisting>
1251CREATE TABLE products (
1252    product_no integer PRIMARY KEY,
1253    name text,
1254    price numeric
1255);
1256
1257CREATE TABLE orders (
1258    order_id integer PRIMARY KEY,
1259    shipping_address text,
1260    ...
1261);
1262
1263CREATE TABLE order_items (
1264    product_no integer REFERENCES products <emphasis>ON DELETE RESTRICT</emphasis>,
1265    order_id integer REFERENCES orders <emphasis>ON DELETE CASCADE</emphasis>,
1266    quantity integer,
1267    PRIMARY KEY (product_no, order_id)
1268);
1269</programlisting>
1270   </para>
1271
1272   <para>
1273    The default <literal>ON DELETE</literal> action is <literal>ON DELETE NO
1274    ACTION</literal>; this does not need to be specified.  This means that the
1275    deletion in the referenced table is allowed to proceed.  But the
1276    foreign-key constraint is still required to be satisfied, so this
1277    operation will usually result in an error.  But checking of foreign-key
1278    constraints can also be deferred to later in the transaction (not covered
1279    in this chapter).  In that case, the <literal>NO ACTION</literal> setting
1280    would allow other commands to <quote>fix</quote> the situation before the
1281    constraint is checked, for example by inserting another suitable row into
1282    the referenced table or by deleting the now-dangling rows from the
1283    referencing table.
1284   </para>
1285
1286   <para>
1287    <literal>RESTRICT</literal> is a stricter setting than <literal>NO
1288    ACTION</literal>.  It prevents deletion of a referenced row.
1289    <literal>RESTRICT</literal> does not allow the check to be deferred until
1290    later in the transaction.
1291   </para>
1292
1293   <para>
1294    <literal>CASCADE</literal> specifies that when a referenced row is deleted,
1295    row(s) referencing it should be automatically deleted as well.
1296   </para>
1297
1298   <para>
1299    There are two other options:
1300    <literal>SET NULL</literal> and <literal>SET DEFAULT</literal>.
1301    These cause the referencing column(s) in the referencing row(s)
1302    to be set to nulls or their default
1303    values, respectively, when the referenced row is deleted.
1304    Note that these do not excuse you from observing any constraints.
1305    For example, if an action specifies <literal>SET DEFAULT</literal>
1306    but the default value would not satisfy the foreign key constraint, the
1307    operation will fail.
1308   </para>
1309
1310   <para>
1311    The appropriate choice of <literal>ON DELETE</literal> action depends on
1312    what kinds of objects the related tables represent.  When the referencing
1313    table represents something that is a component of what is represented by
1314    the referenced table and cannot exist independently, then
1315    <literal>CASCADE</literal> could be appropriate.  If the two tables
1316    represent independent objects, then <literal>RESTRICT</literal> or
1317    <literal>NO ACTION</literal> is more appropriate; an application that
1318    actually wants to delete both objects would then have to be explicit about
1319    this and run two delete commands.  In the above example, order items are
1320    part of an order, and it is convenient if they are deleted automatically
1321    if an order is deleted.  But products and orders are different things, and
1322    so making a deletion of a product automatically cause the deletion of some
1323    order items could be considered problematic.  The actions <literal>SET
1324    NULL</literal> or <literal>SET DEFAULT</literal> can be appropriate if a
1325    foreign-key relationship represents optional information.  For example, if
1326    the products table contained a reference to a product manager, and the
1327    product manager entry gets deleted, then setting the product's product
1328    manager to null or a default might be useful.
1329   </para>
1330
1331   <para>
1332    The actions <literal>SET NULL</literal> and <literal>SET DEFAULT</literal>
1333    can take a column list to specify which columns to set.  Normally, all
1334    columns of the foreign-key constraint are set; setting only a subset is
1335    useful in some special cases.  Consider the following example:
1336<programlisting>
1337CREATE TABLE tenants (
1338    tenant_id integer PRIMARY KEY
1339);
1340
1341CREATE TABLE users (
1342    tenant_id integer REFERENCES tenants ON DELETE CASCADE,
1343    user_id integer NOT NULL,
1344    PRIMARY KEY (tenant_id, user_id)
1345);
1346
1347CREATE TABLE posts (
1348    tenant_id integer REFERENCES tenants ON DELETE CASCADE,
1349    post_id integer NOT NULL,
1350    author_id integer,
1351    PRIMARY KEY (tenant_id, post_id),
1352    FOREIGN KEY (tenant_id, author_id) REFERENCES users ON DELETE SET NULL <emphasis>(author_id)</emphasis>
1353);
1354</programlisting>
1355    Without the specification of the column, the foreign key would also set
1356    the column <literal>tenant_id</literal> to null, but that column is still
1357    required as part of the primary key.
1358   </para>
1359
1360   <para>
1361    Analogous to <literal>ON DELETE</literal> there is also
1362    <literal>ON UPDATE</literal> which is invoked when a referenced
1363    column is changed (updated).  The possible actions are the same,
1364    except that column lists cannot be specified for <literal>SET
1365    NULL</literal> and <literal>SET DEFAULT</literal>.
1366    In this case, <literal>CASCADE</literal> means that the updated values of the
1367    referenced column(s) should be copied into the referencing row(s).
1368    There is also a noticeable difference between <literal>ON UPDATE NO
1369    ACTION</literal> (the default) and <literal>ON UPDATE RESTRICT</literal>.
1370    The former will allow the update to proceed and the foreign-key constraint
1371    will be checked against the state after the update.  The latter will
1372    prevent the update to run even if the state after the update would still
1373    satisfy the constraint.  This prevents updating a referenced row to a
1374    value that is distinct but compares as equal (for example, a character
1375    string with a different case variant, if a character string type with a
1376    case-insensitive collation is used).
1377   </para>
1378
1379   <para>
1380    Normally, a referencing row need not satisfy the foreign key constraint
1381    if any of its referencing columns are null.  If <literal>MATCH FULL</literal>
1382    is added to the foreign key declaration, a referencing row escapes
1383    satisfying the constraint only if all its referencing columns are null
1384    (so a mix of null and non-null values is guaranteed to fail a
1385    <literal>MATCH FULL</literal> constraint).  If you don't want referencing rows
1386    to be able to avoid satisfying the foreign key constraint, declare the
1387    referencing column(s) as <literal>NOT NULL</literal>.
1388   </para>
1389
1390   <para>
1391    A foreign key must reference columns that either are a primary key or
1392    form a unique constraint, or are columns from a non-partial unique index.
1393    This means that the referenced columns always have an index to allow
1394    efficient lookups on whether a referencing row has a match.  Since a
1395    <command>DELETE</command> of a row from the referenced table or an
1396    <command>UPDATE</command> of a referenced column will require a scan of
1397    the referencing table for rows matching the old value, it is often a good
1398    idea to index the referencing columns too.  Because this is not always
1399    needed, and there are many choices available on how to index, the
1400    declaration of a foreign key constraint does not automatically create an
1401    index on the referencing columns.
1402   </para>
1403
1404   <para>
1405    More information about updating and deleting data is in <xref
1406    linkend="dml"/>.  Also see the description of foreign key constraint
1407    syntax in the reference documentation for
1408    <xref linkend="sql-createtable"/>.
1409   </para>
1410  </sect2>

1412  <sect2 id="ddl-constraints-exclusion">
1413   <title>Exclusion Constraints</title>
1414
1415   <indexterm>
1416    <primary>exclusion constraint</primary>
1417   </indexterm>
1418
1419   <indexterm>
1420    <primary>constraint</primary>
1421    <secondary>exclusion</secondary>
1422   </indexterm>
1423
1424   <para>
1425    Exclusion constraints ensure that if any two rows are compared on
1426    the specified columns or expressions using the specified operators,
1427    at least one of these operator comparisons will return false or null.

1429<programlisting>
1430CREATE TABLE circles (
1431    c circle,
1432    EXCLUDE USING gist (c WITH &amp;&amp;)
1433);

1438    See also <link linkend="sql-createtable-exclude"><command>CREATE
1439    TABLE ... CONSTRAINT ... EXCLUDE</command></link> for details.
1440   </para>

1443    Adding an exclusion constraint will automatically create an index
1444    of the type specified in the constraint declaration.
1445   </para>

1595
1596  <para>
1597   When you create a table and you realize that you made a mistake, or
1598   the requirements of the application change, you can drop the
1599   table and create it again.  But this is not a convenient option if
1600   the table is already filled with data, or if the table is
1601   referenced by other database objects (for instance a foreign key
1602   constraint).  Therefore <productname>PostgreSQL</productname>
1603   provides a family of commands to make modifications to existing
1604   tables.  Note that this is conceptually distinct from altering
1605   the data contained in the table: here we are interested in altering
1606   the definition, or structure, of the table.
1607  </para>
1608
1609  <para>
1610   You can:
1611   <itemizedlist spacing="compact">
1612    <listitem>

1614    </listitem>

1617    </listitem>
1618    <listitem>
1619     <para>Add constraints</para>
1620    </listitem>
1621    <listitem>
1622     <para>Remove constraints</para>
1623    </listitem>
1624    <listitem>
1625     <para>Change default values</para>
1626    </listitem>
1627    <listitem>
1628     <para>Change column data types</para>
1629    </listitem>
1630    <listitem>
1631     <para>Rename columns</para>
1632    </listitem>
1633    <listitem>
1634     <para>Rename tables</para>
1635    </listitem>
1636   </itemizedlist>

1639   <xref linkend="sql-altertable"/>
1640   command, whose reference page contains details beyond those given

1644  <sect2 id="ddl-alter-adding-a-column">
1645   <title>Adding a Column</title>
1646
1647   <indexterm>
1648    <primary>column</primary>
1649    <secondary>adding</secondary>
1650   </indexterm>
1651
1652   <para>
1653    To add a column, use a command like:
1654<programlisting>
1655ALTER TABLE products ADD COLUMN description text;
1656</programlisting>
1657    The new column is initially filled with whatever default
1658    value is given (null if you don't specify a <literal>DEFAULT</literal> clause).
1659   </para>
1660
1661   <tip>
1662    <para>
1663     Adding a column with a constant default value does not require each row of
1664     the table to be updated when the <command>ALTER TABLE</command> statement
1665     is executed. Instead, the default value will be returned the next time
1666     the row is accessed, and applied when the table is rewritten, making
1667     the <command>ALTER TABLE</command> very fast even on large tables.
1668    </para>
1669
1670    <para>
1671     If the default value is volatile (e.g., <function>clock_timestamp()</function>)
1672     each row will need to be updated with the value calculated at the time
1673     <command>ALTER TABLE</command> is executed. To avoid a potentially
1674     lengthy update operation, particularly if you intend to fill the column
1675     with mostly nondefault values anyway, it may be preferable to add the
1676     column with no default, insert the correct values using
1677     <command>UPDATE</command>, and then add any desired default as described
1678     below.
1679    </para>
1680   </tip>
1681
1682   <para>
1683    You can also define constraints on the column at the same time,
1684    using the usual syntax:
1685<programlisting>
1686ALTER TABLE products ADD COLUMN description text CHECK (description &lt;&gt; '');
1687</programlisting>
1688    In fact all the options that can be applied to a column description
1689    in <command>CREATE TABLE</command> can be used here.  Keep in mind however
1690    that the default value must satisfy the given constraints, or the
1691    <literal>ADD</literal> will fail.  Alternatively, you can add
1692    constraints later (see below) after you've filled in the new column
1693    correctly.
1694   </para>

1696  </sect2>

1699   <title>Removing a Column</title>
1700
1701   <indexterm>
1702    <primary>column</primary>
1703    <secondary>removing</secondary>
1704   </indexterm>
1705
1706   <para>
1707    To remove a column, use a command like:
1708<programlisting>
1709ALTER TABLE products DROP COLUMN description;
1710</programlisting>
1711    Whatever data was in the column disappears.  Table constraints involving
1712    the column are dropped, too.  However, if the column is referenced by a
1713    foreign key constraint of another table,
1714    <productname>PostgreSQL</productname> will not silently drop that
1715    constraint.  You can authorize dropping everything that depends on
1716    the column by adding <literal>CASCADE</literal>:
1717<programlisting>
1718ALTER TABLE products DROP COLUMN description CASCADE;
1719</programlisting>
1720    See <xref linkend="ddl-depend"/> for a description of the general
1721    mechanism behind this.
1722   </para>

1725  <sect2 id="ddl-alter-adding-a-constraint">
1726   <title>Adding a Constraint</title>
1727
1728   <indexterm>
1729    <primary>constraint</primary>
1730    <secondary>adding</secondary>
1731   </indexterm>
1732
1733   <para>
1734    To add a constraint, the table constraint syntax is used.  For example:
1735<programlisting>
1736ALTER TABLE products ADD CHECK (name &lt;&gt; '');
1737ALTER TABLE products ADD CONSTRAINT some_name UNIQUE (product_no);
1738ALTER TABLE products ADD FOREIGN KEY (product_group_id) REFERENCES product_groups;
1739</programlisting>
1740   </para>
1741
1742   <para>
1743    To add a not-null constraint, which is normally not written as a table
1744    constraint, this special syntax is available:
1745<programlisting>
1746ALTER TABLE products ALTER COLUMN product_no SET NOT NULL;
1747</programlisting>
1748    This command silently does nothing if the column already has a
1749    not-null constraint.
1750   </para>
1751
1752   <para>
1753    The constraint will be checked immediately, so the table data must
1754    satisfy the constraint before it can be added.
1755   </para>

1758  <sect2 id="ddl-alter-removing-a-constraint">
1759   <title>Removing a Constraint</title>
1760
1761   <indexterm>
1762    <primary>constraint</primary>
1763    <secondary>removing</secondary>
1764   </indexterm>
1765
1766   <para>
1767    To remove a constraint you need to know its name.  If you gave it
1768    a name then that's easy.  Otherwise the system assigned a
1769    generated name, which you need to find out.  The
1770    <application>psql</application> command <literal>\d
1771    <replaceable>tablename</replaceable></literal> can be helpful
1772    here; other interfaces might also provide a way to inspect table
1773    details.  Then the command is:
1774<programlisting>
1775ALTER TABLE products DROP CONSTRAINT some_name;
1776</programlisting>
1777   </para>
1778
1779   <para>
1780    As with dropping a column, you need to add <literal>CASCADE</literal> if you
1781    want to drop a constraint that something else depends on.  An example
1782    is that a foreign key constraint depends on a unique or primary key
1783    constraint on the referenced column(s).
1784   </para>
1785
1786   <para>
1787    Simplified syntax is available to drop a not-null constraint:
1788<programlisting>
1789ALTER TABLE products ALTER COLUMN product_no DROP NOT NULL;
1790</programlisting>
1791    This mirrors the <literal>SET NOT NULL</literal> syntax for adding a
1792    not-null constraint.  This command will silently do nothing if the column
1793    does not have a not-null constraint.  (Recall that a column can have at
1794    most one not-null constraint, so it is never ambiguous which constraint
1795    this command acts on.)
1796   </para>

1890 <sect1 id="ddl-priv">
1891  <title>Privileges</title>
1892
1893  <indexterm zone="ddl-priv">
1894   <primary>privilege</primary>
1895  </indexterm>
1896
1897  <indexterm>
1898   <primary>permission</primary>
1899   <see>privilege</see>
1900  </indexterm>
1901
1902  <indexterm zone="ddl-priv">
1903   <primary>owner</primary>
1904  </indexterm>
1905
1906  <indexterm zone="ddl-priv">
1907   <primary>GRANT</primary>
1908  </indexterm>
1909
1910  <indexterm zone="ddl-priv">
1911   <primary>REVOKE</primary>
1912  </indexterm>
1913
1914  <indexterm zone="ddl-priv">
1915   <primary><acronym>ACL</acronym></primary>
1916  </indexterm>
1917
1918  <indexterm zone="ddl-priv-default">
1919   <primary>privilege</primary>
1920   <secondary>default</secondary>
1921  </indexterm>
1922
1923  <para>
1924   When an object is created, it is assigned an owner. The
1925   owner is normally the role that executed the creation statement.
1926   For most kinds of objects, the initial state is that only the owner
1927   (or a superuser) can do anything with the object. To allow
1928   other roles to use it, <firstterm>privileges</firstterm> must be
1929   granted.
1930  </para>
1931
1932  <para>
1933   There are different kinds of privileges: <literal>SELECT</literal>,
1934   <literal>INSERT</literal>, <literal>UPDATE</literal>, <literal>DELETE</literal>,
1935   <literal>TRUNCATE</literal>, <literal>REFERENCES</literal>, <literal>TRIGGER</literal>,
1936   <literal>CREATE</literal>, <literal>CONNECT</literal>, <literal>TEMPORARY</literal>,
1937   <literal>EXECUTE</literal>, <literal>USAGE</literal>, <literal>SET</literal>,
1938   <literal>ALTER SYSTEM</literal>, and <literal>MAINTAIN</literal>.
1939   The privileges applicable to a particular
1940   object vary depending on the object's type (table, function, etc.).
1941   More detail about the meanings of these privileges appears below.
1942   The following sections and chapters will also show you how
1943   these privileges are used.
1944  </para>
1945
1946  <para>
1947   The right to modify or destroy an object is inherent in being the
1948   object's owner, and cannot be granted or revoked in itself.
1949   (However, like all privileges, that right can be inherited by
1950   members of the owning role; see <xref linkend="role-membership"/>.)
1951  </para>
1952
1953  <para>
1954   An object can be assigned to a new owner with an <command>ALTER</command>
1955   command of the appropriate kind for the object, for example
1956<programlisting>
1957ALTER TABLE <replaceable>table_name</replaceable> OWNER TO <replaceable>new_owner</replaceable>;
1958</programlisting>
1959   Superusers can always do this; ordinary roles can only do it if they are
1960   both the current owner of the object (or inherit the privileges of the
1961   owning role) and able to <literal>SET ROLE</literal> to the new owning role.
1962   All object privileges of the old owner are transferred to the new owner
1963   along with the ownership.
1964  </para>
1965
1966  <para>
1967   To assign privileges, the <xref linkend="sql-grant"/> command is
1968   used. For example, if <literal>joe</literal> is an existing role, and
1969   <literal>accounts</literal> is an existing table, the privilege to
1970   update the table can be granted with:
1971<programlisting>
1972GRANT UPDATE ON accounts TO joe;
1973</programlisting>
1974   Writing <literal>ALL</literal> in place of a specific privilege grants all
1975   privileges that are relevant for the object type.
1976  </para>
1977
1978  <para>
1979   The special <quote>role</quote> name <literal>PUBLIC</literal> can
1980   be used to grant a privilege to every role on the system.  Also,
1981   <quote>group</quote> roles can be set up to help manage privileges when
1982   there are many users of a database &mdash; for details see
1983   <xref linkend="user-manag"/>.
1984  </para>
1985
1986  <para>
1987   To revoke a previously-granted privilege, use the fittingly named
1988   <xref linkend="sql-revoke"/> command:
1989<programlisting>
1990REVOKE ALL ON accounts FROM PUBLIC;
1991</programlisting>
1992  </para>
1993
1994  <para>
1995   Ordinarily, only the object's owner (or a superuser) can grant or
1996   revoke privileges on an object.  However, it is possible to grant a
1997   privilege <quote>with grant option</quote>, which gives the recipient
1998   the right to grant it in turn to others.  If the grant option is
1999   subsequently revoked then all who received the privilege from that
2000   recipient (directly or through a chain of grants) will lose the
2001   privilege.  For details see the <xref linkend="sql-grant"/> and
2002   <xref linkend="sql-revoke"/> reference pages.
2003  </para>
2004
2005  <para>
2006   An object's owner can choose to revoke their own ordinary privileges,
2007   for example to make a table read-only for themselves as well as others.
2008   But owners are always treated as holding all grant options, so they
2009   can always re-grant their own privileges.
2010  </para>
2011
2012  <para>
2013   The available privileges are:
2014
2015   <variablelist>
2016    <varlistentry id="ddl-priv-select">
2017     <term><literal>SELECT</literal></term>
2018     <listitem>
2019      <para>

2529   appended if the privilege was granted with grant option.  For example,
2530   <literal>calvin=r*w/hobbes</literal> specifies that the role
2531   <literal>calvin</literal> has the privilege

2615   on a per-user basis, which rows can be returned by normal queries
2616   or inserted, updated, or deleted by data modification commands.
2617   This feature is also known as <firstterm>Row-Level Security</firstterm>.
2618   By default, tables do not have any policies, so that if a user has
2619   access privileges to a table according to the SQL privilege system,
2620   all rows within it are equally available for querying or updating.
2621  </para>
2622
2623  <para>
2624   When row security is enabled on a table (with
2625   <link linkend="sql-altertable">ALTER TABLE ... ENABLE ROW LEVEL
2626   SECURITY</link>), all normal access to the table for selecting rows or
2627   modifying rows must be allowed by a row security policy.  (However, the
2628   table's owner is typically not subject to row security policies.)  If no
2629   policy exists for the table, a default-deny policy is used, meaning that
2630   no rows are visible or can be modified.  Operations that apply to the
2631   whole table, such as <command>TRUNCATE</command> and <literal>REFERENCES</literal>,
2632   are not subject to row security.
2633  </para>
2634
2635  <para>
2636   Row security policies can be specific to commands, or to roles, or to
2637   both.  A policy can be specified to apply to <literal>ALL</literal>
2638   commands, or to <literal>SELECT</literal>, <literal>INSERT</literal>, <literal>UPDATE</literal>,
2639   or <literal>DELETE</literal>.  Multiple roles can be assigned to a given
2640   policy, and normal role membership and inheritance rules apply.
2641  </para>
2642
2643  <para>
2644   To specify which rows are visible or modifiable according to a policy,
2645   an expression is required that returns a Boolean result.  This
2646   expression will be evaluated for each row prior to any conditions or
2647   functions coming from the user's query.  (The only exceptions to this
2648   rule are <literal>leakproof</literal> functions, which are guaranteed to
2649   not leak information; the optimizer may choose to apply such functions
2650   ahead of the row-security check.)  Rows for which the expression does
2651   not return <literal>true</literal> will not be processed.  Separate expressions
2652   may be specified to provide independent control over the rows which are
2653   visible and the rows which are allowed to be modified.  Policy
2654   expressions are run as part of the query and with the privileges of the
2655   user running the query, although security-definer functions can be used
2656   to access data not available to the calling user.
2657  </para>
2658
2659  <para>
2660   Superusers and roles with the <literal>BYPASSRLS</literal> attribute always
2661   bypass the row security system when accessing a table.  Table owners
2662   normally bypass row security as well, though a table owner can choose to
2663   be subject to row security with <link linkend="sql-altertable">ALTER
2664   TABLE ... FORCE ROW LEVEL SECURITY</link>.
2665  </para>
2666
2667  <para>
2668   Enabling and disabling row security, as well as adding policies to a
2669   table, is always the privilege of the table owner only.
2670  </para>
2671
2672  <para>
2673   Policies are created using the <xref linkend="sql-createpolicy"/>
2674   command, altered using the <xref linkend="sql-alterpolicy"/> command,
2675   and dropped using the <xref linkend="sql-droppolicy"/> command.  To
2676   enable and disable row security for a given table, use the
2677   <xref linkend="sql-altertable"/> command.
2678  </para>
2679
2680  <para>
2681   Each policy has a name and multiple policies can be defined for a
2682   table.  As policies are table-specific, each policy for a table must
2683   have a unique name.  Different tables may have policies with the
2684   same name.
2685  </para>
2686
2687  <para>
2688   When multiple policies apply to a given query, they are combined using
2689   either <literal>OR</literal> (for permissive policies, which are the
2690   default) or using <literal>AND</literal> (for restrictive policies).
2691   The <literal>OR</literal> behavior is similar to the rule that a given
2692   role has the privileges
2693   of all roles that they are a member of.  Permissive vs. restrictive
2694   policies are discussed further below.
2695  </para>
2696
2697  <para>
2698   As a simple example, here is how to create a policy on
2699   the <literal>account</literal> relation to allow only members of
2700   the <literal>managers</literal> role to access rows, and only rows of their
2701   accounts:
2702  </para>
2703
2704<programlisting>
2705CREATE TABLE accounts (manager text, company text, contact_email text);
2706
2707ALTER TABLE accounts ENABLE ROW LEVEL SECURITY;
2708
2709CREATE POLICY account_managers ON accounts TO managers
2710    USING (manager = current_user);
2711</programlisting>
2712
2713  <para>
2714   The policy above implicitly provides a <literal>WITH CHECK</literal>
2715   clause identical to its <literal>USING</literal> clause, so that the
2716   constraint applies both to rows selected by a command (so a manager
2717   cannot <command>SELECT</command>, <command>UPDATE</command>,
2718   or <command>DELETE</command> existing rows belonging to a different
2719   manager) and to rows modified by a command (so rows belonging to a
2720   different manager cannot be created via <command>INSERT</command>
2721   or <command>UPDATE</command>).
2722  </para>
2723
2724  <para>
2725   If no role is specified, or the special user name
2726   <literal>PUBLIC</literal> is used, then the policy applies to all
2727   users on the system.  To allow all users to access only their own row in
2728   a <literal>users</literal> table, a simple policy can be used:
2729  </para>
2730
2731<programlisting>
2732CREATE POLICY user_policy ON users
2733    USING (user_name = current_user);
2734</programlisting>
2735
2736  <para>
2737   This works similarly to the previous example.
2738  </para>
2739
2740  <para>
2741   To use a different policy for rows that are being added to the table
2742   compared to those rows that are visible, multiple policies can be
2743   combined.  This pair of policies would allow all users to view all rows
2744   in the <literal>users</literal> table, but only modify their own:
2745  </para>
2746
2747<programlisting>
2748CREATE POLICY user_sel_policy ON users
2749    FOR SELECT
2750    USING (true);
2751CREATE POLICY user_mod_policy ON users
2752    USING (user_name = current_user);
2753</programlisting>
2754
2755  <para>
2756   In a <command>SELECT</command> command, these two policies are combined
2757   using <literal>OR</literal>, with the net effect being that all rows
2758   can be selected.  In other command types, only the second policy applies,
2759   so that the effects are the same as before.
2760  </para>
2761
2762  <para>
2763   Row security can also be disabled with the <command>ALTER TABLE</command>
2764   command.  Disabling row security does not remove any policies that are
2765   defined on the table; they are simply ignored.  Then all rows in the
2766   table are visible and modifiable, subject to the standard SQL privileges
2767   system.
2768  </para>
2769
2770  <para>
2771   Below is a larger example of how this feature can be used in production
2772   environments.  The table <literal>passwd</literal> emulates a Unix password
2773   file:

2933   In some contexts it is important to be sure that row security is
2934   not being applied.  For example, when taking a backup, it could be
2935   disastrous if row security silently caused some rows to be omitted
2936   from the backup.  In such a situation, you can set the
2937   <xref linkend="guc-row-security"/> configuration parameter
2938   to <literal>off</literal>.  This does not in itself bypass row security;
2939   what it does is throw an error if any query's results would get filtered
2940   by a policy.  The reason for the error can then be investigated and
2941   fixed.

2944  <para>
2945   In the examples above, the policy expressions consider only the current
2946   values in the row to be accessed or updated.  This is the simplest and
2947   best-performing case; when possible, it's best to design row security
2948   applications to work this way.  If it is necessary to consult other rows
2949   or other tables to make a policy decision, that can be accomplished using
2950   sub-<command>SELECT</command>s, or functions that contain <command>SELECT</command>s,
2951   in the policy expressions.  Be aware however that such accesses can
2952   create race conditions that could allow information leakage if care is
2953   not taken.  As an example, consider the following table design:
2954  </para>
2955
2956<programlisting>
2957-- definition of privilege groups
2958CREATE TABLE groups (group_id int PRIMARY KEY,
2959                     group_name text NOT NULL);
2960
2961INSERT INTO groups VALUES
2962  (1, 'low'),
2963  (2, 'medium'),
2964  (5, 'high');
2965
2966GRANT ALL ON groups TO alice;  -- alice is the administrator
2967GRANT SELECT ON groups TO public;
2968
2969-- definition of users' privilege levels
2970CREATE TABLE users (user_name text PRIMARY KEY,
2971                    group_id int NOT NULL REFERENCES groups);
2972
2973INSERT INTO users VALUES
2974  ('alice', 5),
2975  ('bob', 2),
2976  ('mallory', 2);
2977
2978GRANT ALL ON users TO alice;
2979GRANT SELECT ON users TO public;
2980
2981-- table holding the information to be protected
2982CREATE TABLE information (info text,
2983                          group_id int NOT NULL REFERENCES groups);
2984
2985INSERT INTO information VALUES
2986  ('barely secret', 1),
2987  ('slightly secret', 2),
2988  ('very secret', 5);

2990ALTER TABLE information ENABLE ROW LEVEL SECURITY;
2991
2992-- a row should be visible to/updatable by users whose security group_id is
2993-- greater than or equal to the row's group_id
2994CREATE POLICY fp_s ON information FOR SELECT
2995  USING (group_id &lt;= (SELECT group_id FROM users WHERE user_name = current_user));
2996CREATE POLICY fp_u ON information FOR UPDATE
2997  USING (group_id &lt;= (SELECT group_id FROM users WHERE user_name = current_user));
2998

3000GRANT ALL ON information TO public;
3001</programlisting>

3003  <para>
3004   Now suppose that <literal>alice</literal> wishes to change the <quote>slightly
3005   secret</quote> information, but decides that <literal>mallory</literal> should not
3006   be trusted with the new content of that row, so she does:
3007  </para>
3008
3009<programlisting>
3010BEGIN;
3011UPDATE users SET group_id = 1 WHERE user_name = 'mallory';
3012UPDATE information SET info = 'secret from mallory' WHERE group_id = 2;
3013COMMIT;
3014</programlisting>
3015
3016  <para>
3017   That looks safe; there is no window wherein <literal>mallory</literal> should be
3018   able to see the <quote>secret from mallory</quote> string.  However, there is
3019   a race condition here.  If <literal>mallory</literal> is concurrently doing,
3020   say,
3021<programlisting>
3022SELECT * FROM information WHERE group_id = 2 FOR UPDATE;
3023</programlisting>
3024   and her transaction is in <literal>READ COMMITTED</literal> mode, it is possible
3025   for her to see <quote>secret from mallory</quote>.  That happens if her
3026   transaction reaches the <structname>information</structname> row just
3027   after <literal>alice</literal>'s does.  It blocks waiting
3028   for <literal>alice</literal>'s transaction to commit, then fetches the updated
3029   row contents thanks to the <literal>FOR UPDATE</literal> clause.  However, it
3030   does <emphasis>not</emphasis> fetch an updated row for the
3031   implicit <command>SELECT</command> from <structname>users</structname>, because that
3032   sub-<command>SELECT</command> did not have <literal>FOR UPDATE</literal>; instead
3033   the <structname>users</structname> row is read with the snapshot taken at the start
3034   of the query.  Therefore, the policy expression tests the old value
3035   of <literal>mallory</literal>'s privilege level and allows her to see the
3036   updated row.
3037  </para>
3038
3039  <para>
3040   There are several ways around this problem.  One simple answer is to use
3041   <literal>SELECT ... FOR SHARE</literal> in sub-<command>SELECT</command>s in row
3042   security policies.  However, that requires granting <literal>UPDATE</literal>
3043   privilege on the referenced table (here <structname>users</structname>) to the
3044   affected users, which might be undesirable.  (But another row security
3045   policy could be applied to prevent them from actually exercising that
3046   privilege; or the sub-<command>SELECT</command> could be embedded into a security
3047   definer function.)  Also, heavy concurrent use of row share locks on the
3048   referenced table could pose a performance problem, especially if updates
3049   of it are frequent.  Another solution, practical if updates of the
3050   referenced table are infrequent, is to take an
3051   <literal>ACCESS EXCLUSIVE</literal> lock on the
3052   referenced table when updating it, so that no concurrent transactions
3053   could be examining old row values.  Or one could just wait for all
3054   concurrent transactions to end after committing an update of the
3055   referenced table and before making changes that rely on the new security
3056   situation.
3057  </para>
3058
3059  <para>
3060   For additional details see <xref linkend="sql-createpolicy"/>
3061   and <xref linkend="sql-altertable"/>.
3062  </para>

3067  <title>Schemas</title>
3068
3069  <indexterm zone="ddl-schemas">
3070   <primary>schema</primary>
3071  </indexterm>
3072
3073  <para>
3074   A <productname>PostgreSQL</productname> database cluster contains
3075   one or more named databases.  Roles and a few other object types are
3076   shared across the entire cluster.  A client connection to the server
3077   can only access data in a single database, the one specified in the
3078   connection request.
3079  </para>
3080
3081  <note>
3082   <para>
3083    Users of a cluster do not necessarily have the privilege to access every
3084    database in the cluster.  Sharing of role names means that there
3085    cannot be different roles named, say, <literal>joe</literal> in two databases
3086    in the same cluster; but the system can be configured to allow
3087    <literal>joe</literal> access to only some of the databases.
3088   </para>
3089  </note>
3090
3091  <para>
3092   A database contains one or more named <firstterm>schemas</firstterm>, which
3093   in turn contain tables.  Schemas also contain other kinds of named
3094   objects, including data types, functions, and operators.  Within one
3095   schema, two objects of the same type cannot have the same name.
3096   Furthermore, tables, sequences, indexes, views, materialized views, and
3097   foreign tables share the same namespace, so that, for example, an index and
3098   a table must have different names if they are in the same schema.  The same
3099   object name can be used in different schemas without conflict; for
3100   example, both <literal>schema1</literal> and <literal>myschema</literal> can
3101   contain tables named <literal>mytable</literal>.  Unlike databases,
3102   schemas are not rigidly separated: a user can access objects in any
3103   of the schemas in the database they are connected to, if they have
3104   privileges to do so.
3105  </para>
3106
3107  <para>
3108   There are several reasons why one might want to use schemas:
3109
3110   <itemizedlist>
3111    <listitem>
3112     <para>
3113      To allow many users to use one database without interfering with
3114      each other.
3115     </para>
3116    </listitem>
3117
3118    <listitem>
3119     <para>
3120      To organize database objects into logical groups to make them
3121      more manageable.
3122     </para>
3123    </listitem>
3124
3125    <listitem>
3126     <para>
3127      Third-party applications can be put into separate schemas so
3128      they do not collide with the names of other objects.
3129     </para>
3130    </listitem>
3131   </itemizedlist>
3132
3133   Schemas are analogous to directories at the operating system level,
3134   except that schemas cannot be nested.
3135  </para>

3215   <para>

3217    (since this is one of the ways to restrict the activities of your
3218    users to well-defined namespaces).  The syntax for that is:
3219<programlisting>
3220CREATE SCHEMA <replaceable>schema_name</replaceable> AUTHORIZATION <replaceable>user_name</replaceable>;
3221</programlisting>
3222    You can even omit the schema name, in which case the schema name
3223    will be the same as the user name.  See <xref
3224    linkend="ddl-schemas-patterns"/> for how this can be useful.
3225   </para>
3226
3227   <para>
3228    Schema names beginning with <literal>pg_</literal> are reserved for
3229    system purposes and cannot be created by users.
3230   </para>
3231  </sect2>

3238    <secondary>public</secondary>
3239   </indexterm>
3240
3241   <para>
3242    In the previous sections we created tables without specifying any
3243    schema names.  By default such tables (and other objects) are
3244    automatically put into a schema named <quote>public</quote>.  Every new
3245    database contains such a schema.  Thus, the following are equivalent:
3246<programlisting>
3247CREATE TABLE products ( ... );

3249    and:
3250<programlisting>
3251CREATE TABLE public.products ( ... );
3252</programlisting>

3256  <sect2 id="ddl-schemas-path">
3257   <title>The Schema Search Path</title>
3258
3259   <indexterm>
3260    <primary>search path</primary>
3261   </indexterm>
3262
3263   <indexterm>
3264    <primary>unqualified name</primary>
3265   </indexterm>
3266
3267   <indexterm>
3268    <primary>name</primary>
3269    <secondary>unqualified</secondary>
3270   </indexterm>
3271
3272   <para>
3273    Qualified names are tedious to write, and it's often best not to
3274    wire a particular schema name into applications anyway.  Therefore
3275    tables are often referred to by <firstterm>unqualified names</firstterm>,
3276    which consist of just the table name.  The system determines which table
3277    is meant by following a <firstterm>search path</firstterm>, which is a list
3278    of schemas to look in.  The first matching table in the search path
3279    is taken to be the one wanted.  If there is no match in the search
3280    path, an error is reported, even if matching table names exist
3281    in other schemas in the database.
3282   </para>
3283
3284  <para>
3285    The ability to create like-named objects in different schemas complicates
3286    writing a query that references precisely the same objects every time.  It
3287    also opens up the potential for users to change the behavior of other
3288    users' queries, maliciously or accidentally.  Due to the prevalence of
3289    unqualified names in queries and their use
3290    in <productname>PostgreSQL</productname> internals, adding a schema
3291    to <varname>search_path</varname> effectively trusts all users having
3292    <literal>CREATE</literal> privilege on that schema.  When you run an
3293    ordinary query, a malicious user able to create objects in a schema of
3294    your search path can take control and execute arbitrary SQL functions as
3295    though you executed them.
3296   </para>
3297
3298   <indexterm>
3299    <primary>schema</primary>
3300    <secondary>current</secondary>
3301   </indexterm>
3302
3303   <para>
3304    The first schema named in the search path is called the current schema.
3305    Aside from being the first schema searched, it is also the schema in
3306    which new tables will be created if the <command>CREATE TABLE</command>
3307    command does not specify a schema name.
3308   </para>
3309
3310   <indexterm>
3311    <primary><varname>search_path</varname> configuration parameter</primary>
3312   </indexterm>
3313
3314   <para>
3315    To show the current search path, use the following command:
3316<programlisting>
3317SHOW search_path;
3318</programlisting>
3319    In the default setup this returns:
3320<screen>
3321 search_path
3322--------------
3323 "$user", public
3324</screen>
3325    The first element specifies that a schema with the same name as
3326    the current user is to be searched.  If no such schema exists,
3327    the entry is ignored.  The second element refers to the
3328    public schema that we have seen already.
3329   </para>
3330
3331   <para>
3332    The first schema in the search path that exists is the default
3333    location for creating new objects.  That is the reason that by
3334    default objects are created in the public schema.  When objects
3335    are referenced in any other context without schema qualification
3336    (table modification, data modification, or query commands) the
3337    search path is traversed until a matching object is found.
3338    Therefore, in the default configuration, any unqualified access
3339    again can only refer to the public schema.
3340   </para>
3341
3342   <para>
3343    To put our new schema in the path, we use:
3344<programlisting>
3345SET search_path TO myschema,public;
3346</programlisting>
3347    (We omit the <literal>$user</literal> here because we have no
3348    immediate need for it.)  And then we can access the table without
3349    schema qualification:
3350<programlisting>
3351DROP TABLE mytable;
3352</programlisting>
3353    Also, since <literal>myschema</literal> is the first element in
3354    the path, new objects would by default be created in it.
3355   </para>
3356
3357   <para>
3358    We could also have written:
3359<programlisting>
3360SET search_path TO myschema;
3361</programlisting>
3362    Then we no longer have access to the public schema without
3363    explicit qualification.  There is nothing special about the public
3364    schema except that it exists by default.  It can be dropped, too.
3365   </para>
3366
3367   <para>
3368    See also <xref linkend="functions-info"/> for other ways to manipulate
3369    the schema search path.
3370   </para>
3371
3372   <para>
3373    The search path works in the same way for data type names, function names,
3374    and operator names as it does for table names.  Data type and function
3375    names can be qualified in exactly the same way as table names.  If you
3376    need to write a qualified operator name in an expression, there is a
3377    special provision: you must write
3378<synopsis>
3379<literal>OPERATOR(</literal><replaceable>schema</replaceable><literal>.</literal><replaceable>operator</replaceable><literal>)</literal>
3380</synopsis>
3381    This is needed to avoid syntactic ambiguity.  An example is:
3382<programlisting>
3383SELECT 3 OPERATOR(pg_catalog.+) 4;
3384</programlisting>
3385    In practice one usually relies on the search path for operators,
3386    so as not to have to write anything so ugly as that.
3387   </para>
3388  </sect2>

3390  <sect2 id="ddl-schemas-priv">
3391   <title>Schemas and Privileges</title>
3392
3393   <indexterm zone="ddl-schemas-priv">
3394    <primary>privilege</primary>
3395    <secondary sortas="schemas">for schemas</secondary>
3396   </indexterm>
3397
3398   <para>
3399    By default, users cannot access any objects in schemas they do not
3400    own.  To allow that, the owner of the schema must grant the
3401    <literal>USAGE</literal> privilege on the schema.  By default, everyone
3402    has that privilege on the schema <literal>public</literal>.  To allow
3403    users to make use of the objects in a schema, additional privileges might
3404    need to be granted, as appropriate for the object.
3405   </para>
3406
3407   <para>
3408    A user can also be allowed to create objects in someone else's schema.  To
3409    allow that, the <literal>CREATE</literal> privilege on the schema needs to
3410    be granted.  In databases upgraded from
3411    <productname>PostgreSQL</productname> 14 or earlier, everyone has that
3412    privilege on the schema <literal>public</literal>.
3413    Some <link linkend="ddl-schemas-patterns">usage patterns</link> call for
3414    revoking that privilege:
3415<programlisting>
3416REVOKE CREATE ON SCHEMA public FROM PUBLIC;
3417</programlisting>
3418    (The first <quote>public</quote> is the schema, the second
3419    <quote>public</quote> means <quote>every user</quote>.  In the
3420    first sense it is an identifier, in the second sense it is a
3421    key word, hence the different capitalization; recall the
3422    guidelines from <xref linkend="sql-syntax-identifiers"/>.)
3423   </para>
3424  </sect2>

3426  <sect2 id="ddl-schemas-catalog">
3427   <title>The System Catalog Schema</title>
3428
3429   <indexterm zone="ddl-schemas-catalog">
3430    <primary>system catalog</primary>
3431    <secondary>schema</secondary>
3432   </indexterm>
3433
3434   <para>
3435    In addition to <literal>public</literal> and user-created schemas, each
3436    database contains a <literal>pg_catalog</literal> schema, which contains
3437    the system tables and all the built-in data types, functions, and
3438    operators.  <literal>pg_catalog</literal> is always effectively part of
3439    the search path.  If it is not named explicitly in the path then
3440    it is implicitly searched <emphasis>before</emphasis> searching the path's
3441    schemas.  This ensures that built-in names will always be
3442    findable.  However, you can explicitly place
3443    <literal>pg_catalog</literal> at the end of your search path if you
3444    prefer to have user-defined names override built-in names.
3445   </para>
3446
3447   <para>
3448    Since system table names begin with <literal>pg_</literal>, it is best to
3449    avoid such names to ensure that you won't suffer a conflict if some
3450    future version defines a system table named the same as your
3451    table.  (With the default search path, an unqualified reference to
3452    your table name would then be resolved as the system table instead.)
3453    System tables will continue to follow the convention of having
3454    names beginning with <literal>pg_</literal>, so that they will not
3455    conflict with unqualified user-table names so long as users avoid
3456    the <literal>pg_</literal> prefix.
3457   </para>
3458  </sect2>

3460  <sect2 id="ddl-schemas-patterns">
3461   <title>Usage Patterns</title>
3462
3463   <para>
3464    Schemas can be used to organize your data in many ways.
3465    A <firstterm>secure schema usage pattern</firstterm> prevents untrusted
3466    users from changing the behavior of other users' queries.  When a database
3467    does not use a secure schema usage pattern, users wishing to securely
3468    query that database would take protective action at the beginning of each
3469    session.  Specifically, they would begin each session by
3470    setting <varname>search_path</varname> to the empty string or otherwise
3471    removing schemas that are writable by non-superusers
3472    from <varname>search_path</varname>.  There are a few usage patterns
3473    easily supported by the default configuration:
3474    <itemizedlist>
3475     <listitem>
3476      <para>
3477       Constrain ordinary users to user-private schemas.
3478       To implement this pattern, first ensure that no schemas have
3479       public <literal>CREATE</literal> privileges.  Then, for every user
3480       needing to create non-temporary objects, create a schema with the
3481       same name as that user, for example
3482       <literal>CREATE SCHEMA alice AUTHORIZATION alice</literal>.
3483       (Recall that the default search path starts
3484       with <literal>$user</literal>, which resolves to the user
3485       name. Therefore, if each user has a separate schema, they access
3486       their own schemas by default.)  This pattern is a secure schema
3487       usage pattern unless an untrusted user is the database owner or
3488       has been granted <literal>ADMIN OPTION</literal> on a relevant role,
3489       in which case no secure schema usage pattern exists.
3490      </para>
3491      <!-- A database owner can attack the database's users via "CREATE SCHEMA
3492           trojan; ALTER DATABASE $mydb SET search_path = trojan, public;". -->
3493
3494      <para>
3495       In <productname>PostgreSQL</productname> 15 and later, the default
3496       configuration supports this usage pattern.  In prior versions, or
3497       when using a database that has been upgraded from a prior version,
3498       you will need to remove the public <literal>CREATE</literal>
3499       privilege from the <literal>public</literal> schema (issue
3500       <literal>REVOKE CREATE ON SCHEMA public FROM PUBLIC</literal>).
3501       Then consider auditing the <literal>public</literal> schema for
3502       objects named like objects in schema <literal>pg_catalog</literal>.
3503      </para>
3504      <!-- "DROP SCHEMA public" is inferior to this REVOKE, because pg_dump
3505           doesn't preserve that DROP. -->
3506     </listitem>
3507
3508     <listitem>
3509      <para>
3510       Remove the public schema from the default search path, by modifying
3511       <link linkend="config-setting-configuration-file"><filename>postgresql.conf</filename></link>
3512       or by issuing <literal>ALTER ROLE ALL SET search_path =
3513       "$user"</literal>.  Then, grant privileges to create in the public
3514       schema.  Only qualified names will choose public schema objects.  While
3515       qualified table references are fine, calls to functions in the public
3516       schema <link linkend="typeconv-func">will be unsafe or
3517       unreliable</link>.  If you create functions or extensions in the public
3518       schema, use the first pattern instead.  Otherwise, like the first
3519       pattern, this is secure unless an untrusted user is the database owner
3520       or has been granted <literal>ADMIN OPTION</literal> on a relevant role.
3521      </para>
3522     </listitem>
3523
3524     <listitem>
3525      <para>
3526       Keep the default search path, and grant privileges to create in the
3527       public schema.  All users access the public schema implicitly.  This
3528       simulates the situation where schemas are not available at all, giving
3529       a smooth transition from the non-schema-aware world.  However, this is
3530       never a secure pattern.  It is acceptable only when the database has a
3531       single user or a few mutually-trusting users.  In databases upgraded
3532       from <productname>PostgreSQL</productname> 14 or earlier, this is the
3533       default.
3534      </para>
3535     </listitem>
3536    </itemizedlist>
3537   </para>
3538
3539   <para>
3540    For any pattern, to install shared applications (tables to be used by
3541    everyone, additional functions provided by third parties, etc.), put them
3542    into separate schemas.  Remember to grant appropriate privileges to allow
3543    the other users to access them.  Users can then refer to these additional
3544    objects by qualifying the names with a schema name, or they can put the
3545    additional schemas into their search path, as they choose.
3546   </para>
3547  </sect2>

3549  <sect2 id="ddl-schemas-portability">
3550   <title>Portability</title>
3551
3552   <para>
3553    In the SQL standard, the notion of objects in the same schema
3554    being owned by different users does not exist.  Moreover, some
3555    implementations do not allow you to create schemas that have a
3556    different name than their owner.  In fact, the concepts of schema
3557    and user are nearly equivalent in a database system that
3558    implements only the basic schema support specified in the
3559    standard.  Therefore, many users consider qualified names to
3560    really consist of
3561    <literal><replaceable>user_name</replaceable>.<replaceable>table_name</replaceable></literal>.
3562    This is how <productname>PostgreSQL</productname> will effectively
3563    behave if you create a per-user schema for every user.
3564   </para>
3565
3566   <para>
3567    Also, there is no concept of a <literal>public</literal> schema in the
3568    SQL standard.  For maximum conformance to the standard, you should
3569    not use the <literal>public</literal> schema.
3570   </para>
3571
3572   <para>
3573    Of course, some SQL database systems might not implement schemas
3574    at all, or provide namespace support by allowing (possibly
3575    limited) cross-database access.  If you need to work with those
3576    systems, then maximum portability would be achieved by not using
3577    schemas at all.
3578   </para>
3579  </sect2>

3582 <sect1 id="ddl-inherit">
3583  <title>Inheritance</title>
3584
3585  <indexterm>
3586   <primary>inheritance</primary>
3587  </indexterm>
3588
3589  <indexterm>
3590   <primary>table</primary>
3591   <secondary>inheritance</secondary>
3592  </indexterm>
3593
3594  <para>
3595   <productname>PostgreSQL</productname> implements table inheritance,
3596   which can be a useful tool for database designers.  (SQL:1999 and
3597   later define a type inheritance feature, which differs in many
3598   respects from the features described here.)
3599  </para>
3600
3601  <para>
3602   Let's start with an example: suppose we are trying to build a data
3603   model for cities.  Each state has many cities, but only one
3604   capital. We want to be able to quickly retrieve the capital city
3605   for any particular state. This can be done by creating two tables,
3606   one for state capitals and one for cities that are not
3607   capitals. However, what happens when we want to ask for data about
3608   a city, regardless of whether it is a capital or not? The
3609   inheritance feature can help to resolve this problem. We define the
3610   <structname>capitals</structname> table so that it inherits from
3611   <structname>cities</structname>:
3612
3613<programlisting>
3614CREATE TABLE cities (
3615    name            text,
3616    population      float,
3617    elevation       int     -- in feet
3618);
3619
3620CREATE TABLE capitals (
3621    state           char(2)
3622) INHERITS (cities);
3623</programlisting>
3624
3625   In this case, the <structname>capitals</structname> table <firstterm>inherits</firstterm>
3626   all the columns of its parent table, <structname>cities</structname>. State
3627   capitals also have an extra column, <structfield>state</structfield>, that shows
3628   their state.
3629  </para>
3630
3631  <para>
3632   In <productname>PostgreSQL</productname>, a table can inherit from
3633   zero or more other tables, and a query can reference either all
3634   rows of a table or all rows of a table plus all of its descendant tables.
3635   The latter behavior is the default.
3636   For example, the following query finds the names of all cities,
3637   including state capitals, that are located at an elevation over
3638   500 feet:
3639
3640<programlisting>
3641SELECT name, elevation
3642    FROM cities
3643    WHERE elevation &gt; 500;
3644</programlisting>
3645
3646   Given the sample data from the <productname>PostgreSQL</productname>
3647   tutorial (see <xref linkend="tutorial-sql-intro"/>), this returns:
3648
3649<programlisting>
3650   name    | elevation
3651-----------+-----------
3652 Las Vegas |      2174
3653 Mariposa  |      1953
3654 Madison   |       845
3655</programlisting>
3656  </para>
3657
3658  <para>
3659   On the other hand, the following query finds all the cities that
3660   are not state capitals and are situated at an elevation over 500 feet:
3661
3662<programlisting>
3663SELECT name, elevation
3664    FROM ONLY cities
3665    WHERE elevation &gt; 500;
3666
3667   name    | elevation
3668-----------+-----------
3669 Las Vegas |      2174
3670 Mariposa  |      1953
3671</programlisting>
3672  </para>
3673
3674  <para>
3675   Here the <literal>ONLY</literal> keyword indicates that the query
3676   should apply only to <structname>cities</structname>, and not any tables
3677   below <structname>cities</structname> in the inheritance hierarchy.  Many
3678   of the commands that we have already discussed &mdash;
3679   <command>SELECT</command>, <command>UPDATE</command> and
3680   <command>DELETE</command> &mdash; support the
3681   <literal>ONLY</literal> keyword.
3682  </para>
3683
3684  <para>
3685   You can also write the table name with a trailing <literal>*</literal>
3686   to explicitly specify that descendant tables are included:
3687
3688<programlisting>
3689SELECT name, elevation
3690    FROM cities*
3691    WHERE elevation &gt; 500;
3692</programlisting>
3693
3694   Writing <literal>*</literal> is not necessary, since this behavior is always
3695   the default.  However, this syntax is still supported for
3696   compatibility with older releases where the default could be changed.
3697  </para>
3698
3699  <para>
3700   In some cases you might wish to know which table a particular row
3701   originated from. There is a system column called
3702   <structfield>tableoid</structfield> in each table which can tell you the
3703   originating table:
3704
3705<programlisting>
3706SELECT c.tableoid, c.name, c.elevation
3707FROM cities c
3708WHERE c.elevation &gt; 500;
3709</programlisting>
3710
3711   which returns:
3712
3713<programlisting>
3714 tableoid |   name    | elevation
3715----------+-----------+-----------
3716   139793 | Las Vegas |      2174
3717   139793 | Mariposa  |      1953
3718   139798 | Madison   |       845
3719</programlisting>
3720
3721   (If you try to reproduce this example, you will probably get
3722   different numeric OIDs.)  By doing a join with
3723   <structname>pg_class</structname> you can see the actual table names:
3724
3725<programlisting>
3726SELECT p.relname, c.name, c.elevation
3727FROM cities c, pg_class p
3728WHERE c.elevation &gt; 500 AND c.tableoid = p.oid;
3729</programlisting>
3730
3731   which returns:
3732
3733<programlisting>
3734 relname  |   name    | elevation
3735----------+-----------+-----------
3736 cities   | Las Vegas |      2174
3737 cities   | Mariposa  |      1953
3738 capitals | Madison   |       845
3739</programlisting>
3740  </para>
3741
3742  <para>
3743   Another way to get the same effect is to use the <type>regclass</type>
3744   alias type, which will print the table OID symbolically:
3745
3746<programlisting>
3747SELECT c.tableoid::regclass, c.name, c.elevation
3748FROM cities c
3749WHERE c.elevation &gt; 500;
3750</programlisting>
3751  </para>
3752
3753  <para>
3754   Inheritance does not automatically propagate data from
3755   <command>INSERT</command> or <command>COPY</command> commands to
3756   other tables in the inheritance hierarchy. In our example, the
3757   following <command>INSERT</command> statement will fail:
3758<programlisting>
3759INSERT INTO cities (name, population, elevation, state)
3760VALUES ('Albany', NULL, NULL, 'NY');
3761</programlisting>
3762   We might hope that the data would somehow be routed to the
3763   <structname>capitals</structname> table, but this does not happen:
3764   <command>INSERT</command> always inserts into exactly the table
3765   specified.  In some cases it is possible to redirect the insertion
3766   using a rule (see <xref linkend="rules"/>).  However that does not
3767   help for the above case because the <structname>cities</structname> table
3768   does not contain the column <structfield>state</structfield>, and so the
3769   command will be rejected before the rule can be applied.
3770  </para>
3771
3772  <para>
3773   All check constraints and not-null constraints on a parent table are
3774   automatically inherited by its children, unless explicitly specified
3775   otherwise with <literal>NO INHERIT</literal> clauses.  Other types of constraints
3776   (unique, primary key, and foreign key constraints) are not inherited.
3777  </para>
3778
3779  <para>
3780   A table can inherit from more than one parent table, in which case it has
3781   the union of the columns defined by the parent tables.  Any columns
3782   declared in the child table's definition are added to these.  If the
3783   same column name appears in multiple parent tables, or in both a parent
3784   table and the child's definition, then these columns are <quote>merged</quote>
3785   so that there is only one such column in the child table.  To be merged,
3786   columns must have the same data types, else an error is raised.
3787   Inheritable check constraints and not-null constraints are merged in a
3788   similar fashion.  Thus, for example, a merged column will be marked
3789   not-null if any one of the column definitions it came from is marked
3790   not-null.  Check constraints are merged if they have the same name,
3791   and the merge will fail if their conditions are different.
3792  </para>
3793
3794  <para>
3795   Table inheritance is typically established when the child table is
3796   created, using the <literal>INHERITS</literal> clause of the
3797   <link linkend="sql-createtable"><command>CREATE TABLE</command></link>
3798   statement.
3799   Alternatively, a table which is already defined in a compatible way can
3800   have a new parent relationship added, using the <literal>INHERIT</literal>
3801   variant of <link linkend="sql-altertable"><command>ALTER TABLE</command></link>.
3802   To do this the new child table must already include columns with
3803   the same names and types as the columns of the parent. It must also include
3804   check constraints with the same names and check expressions as those of the
3805   parent. Similarly an inheritance link can be removed from a child using the
3806   <literal>NO INHERIT</literal> variant of <command>ALTER TABLE</command>.
3807   Dynamically adding and removing inheritance links like this can be useful
3808   when the inheritance relationship is being used for table
3809   partitioning (see <xref linkend="ddl-partitioning"/>).
3810  </para>
3811
3812  <para>
3813   One convenient way to create a compatible table that will later be made
3814   a new child is to use the <literal>LIKE</literal> clause in <command>CREATE
3815   TABLE</command>. This creates a new table with the same columns as
3816   the source table. If there are any <literal>CHECK</literal>
3817   constraints defined on the source table, the <literal>INCLUDING
3818   CONSTRAINTS</literal> option to <literal>LIKE</literal> should be
3819   specified, as the new child must have constraints matching the parent
3820   to be considered compatible.
3821  </para>
3822
3823  <para>
3824   A parent table cannot be dropped while any of its children remain. Neither
3825   can columns or check constraints of child tables be dropped or altered
3826   if they are inherited
3827   from any parent tables. If you wish to remove a table and all of its
3828   descendants, one easy way is to drop the parent table with the
3829   <literal>CASCADE</literal> option (see <xref linkend="ddl-depend"/>).
3830  </para>
3831
3832  <para>
3833   <command>ALTER TABLE</command> will
3834   propagate any changes in column data definitions and check
3835   constraints down the inheritance hierarchy.  Again, dropping
3836   columns that are depended on by other tables is only possible when using
3837   the <literal>CASCADE</literal> option. <command>ALTER
3838   TABLE</command> follows the same rules for duplicate column merging
3839   and rejection that apply during <command>CREATE TABLE</command>.
3840  </para>
3841
3842  <para>
3843   Inherited queries perform access permission checks on the parent table
3844   only.  Thus, for example, granting <literal>UPDATE</literal> permission on
3845   the <structname>cities</structname> table implies permission to update rows in
3846   the <structname>capitals</structname> table as well, when they are
3847   accessed through <structname>cities</structname>.  This preserves the appearance
3848   that the data is (also) in the parent table.  But
3849   the <structname>capitals</structname> table could not be updated directly
3850   without an additional grant.  In a similar way, the parent table's row
3851   security policies (see <xref linkend="ddl-rowsecurity"/>) are applied to
3852   rows coming from child tables during an inherited query.  A child table's
3853   policies, if any, are applied only when it is the table explicitly named
3854   in the query; and in that case, any policies attached to its parent(s) are
3855   ignored.
3856  </para>
3857
3858  <para>
3859   Foreign tables (see <xref linkend="ddl-foreign-data"/>) can also
3860   be part of inheritance hierarchies, either as parent or child
3861   tables, just as regular tables can be.  If a foreign table is part
3862   of an inheritance hierarchy then any operations not supported by
3863   the foreign table are not supported on the whole hierarchy either.
3864  </para>
3865
3866 <sect2 id="ddl-inherit-caveats">
3867  <title>Caveats</title>
3868
3869  <para>
3870   Note that not all SQL commands are able to work on
3871   inheritance hierarchies.  Commands that are used for data querying,
3872   data modification, or schema modification
3873   (e.g., <literal>SELECT</literal>, <literal>UPDATE</literal>, <literal>DELETE</literal>,
3874   most variants of <literal>ALTER TABLE</literal>, but
3875   not <literal>INSERT</literal> or <literal>ALTER TABLE ...
3876   RENAME</literal>) typically default to including child tables and
3877   support the <literal>ONLY</literal> notation to exclude them.
3878   The majority of commands that do database maintenance and tuning
3879   (e.g., <literal>REINDEX</literal>) only work on individual, physical
3880   tables and do not support recursing over inheritance hierarchies.
3881   However, both <literal>VACUUM</literal> and <literal>ANALYZE</literal>
3882   commands default to including child tables and the <literal>ONLY</literal>
3883   notation is supported to allow them to be excluded.  The respective
3884   behavior of each individual command is documented in its reference page
3885   (<xref linkend="sql-commands"/>).
3886  </para>
3887
3888  <para>
3889   A serious limitation of the inheritance feature is that indexes (including
3890   unique constraints) and foreign key constraints only apply to single
3891   tables, not to their inheritance children. This is true on both the
3892   referencing and referenced sides of a foreign key constraint. Thus,
3893   in the terms of the above example:
3894
3895   <itemizedlist>
3896    <listitem>
3897     <para>
3898      If we declared <structname>cities</structname>.<structfield>name</structfield> to be
3899      <literal>UNIQUE</literal> or a <literal>PRIMARY KEY</literal>, this would not stop the
3900      <structname>capitals</structname> table from having rows with names duplicating
3901      rows in <structname>cities</structname>.  And those duplicate rows would by
3902      default show up in queries from <structname>cities</structname>.  In fact, by
3903      default <structname>capitals</structname> would have no unique constraint at all,
3904      and so could contain multiple rows with the same name.
3905      You could add a unique constraint to <structname>capitals</structname>, but this
3906      would not prevent duplication compared to <structname>cities</structname>.
3907     </para>
3908    </listitem>
3909
3910    <listitem>
3911     <para>
3912      Similarly, if we were to specify that
3913      <structname>cities</structname>.<structfield>name</structfield> <literal>REFERENCES</literal> some
3914      other table, this constraint would not automatically propagate to
3915      <structname>capitals</structname>.  In this case you could work around it by
3916      manually adding the same <literal>REFERENCES</literal> constraint to
3917      <structname>capitals</structname>.
3918     </para>
3919    </listitem>
3920
3921    <listitem>
3922     <para>
3923      Specifying that another table's column <literal>REFERENCES
3924      cities(name)</literal> would allow the other table to contain city names, but
3925      not capital names.  There is no good workaround for this case.
3926     </para>
3927    </listitem>
3928   </itemizedlist>
3929
3930   Some functionality not implemented for inheritance hierarchies is
3931   implemented for declarative partitioning.
3932   Considerable care is needed in deciding whether partitioning with legacy
3933   inheritance is useful for your application.
3934  </para>
3935
3936   </sect2>

3940   <title>Table Partitioning</title>
3941
3942   <indexterm>
3943    <primary>partitioning</primary>
3944   </indexterm>
3945
3946   <indexterm>
3947    <primary>table</primary>
3948    <secondary>partitioning</secondary>
3949   </indexterm>

3952    <primary>partitioned table</primary>
3953   </indexterm>
3954
3955   <para>
3956    <productname>PostgreSQL</productname> supports basic table
3957    partitioning. This section describes why and how to implement
3958    partitioning as part of your database design.
3959   </para>
3960
3961   <sect2 id="ddl-partitioning-overview">
3962     <title>Overview</title>
3963
3964    <para>
3965     Partitioning refers to splitting what is logically one large table into
3966     smaller physical pieces.  Partitioning can provide several benefits:
3967    <itemizedlist>
3968     <listitem>
3969      <para>
3970       Query performance can be improved dramatically in certain situations,
3971       particularly when most of the heavily accessed rows of the table are in a
3972       single partition or a small number of partitions.  Partitioning
3973       effectively substitutes for the upper tree levels of indexes,
3974       making it more likely that the heavily-used parts of the indexes
3975       fit in memory.
3976      </para>
3977     </listitem>
3978
3979     <listitem>
3980      <para>
3981       When queries or updates access a large percentage of a single
3982       partition, performance can be improved by using a
3983       sequential scan of that partition instead of using an
3984       index, which would require random-access reads scattered across the
3985       whole table.
3986      </para>
3987     </listitem>
3988
3989     <listitem>
3990      <para>
3991       Bulk loads and deletes can be accomplished by adding or removing
3992       partitions, if the usage pattern is accounted for in the
3993       partitioning design.  Dropping an individual partition
3994       using <command>DROP TABLE</command>, or doing <command>ALTER TABLE
3995       DETACH PARTITION</command>, is far faster than a bulk
3996       operation.  These commands also entirely avoid the
3997       <command>VACUUM</command> overhead caused by a bulk <command>DELETE</command>.
3998      </para>
3999     </listitem>
4000
4001     <listitem>
4002      <para>
4003       Seldom-used data can be migrated to cheaper and slower storage media.
4004      </para>
4005     </listitem>
4006    </itemizedlist>
4007
4008     These benefits will normally be worthwhile only when a table would
4009     otherwise be very large. The exact point at which a table will
4010     benefit from partitioning depends on the application, although a
4011     rule of thumb is that the size of the table should exceed the physical
4012     memory of the database server.
4013    </para>
4014
4015    <para>
4016     <productname>PostgreSQL</productname> offers built-in support for the
4017     following forms of partitioning:
4018
4019     <variablelist>
4020      <varlistentry id="ddl-partitioning-overview-range">
4021       <term>Range Partitioning</term>
4022
4023       <listitem>
4024        <para>
4025         The table is partitioned into <quote>ranges</quote> defined
4026         by a key column or set of columns, with no overlap between
4027         the ranges of values assigned to different partitions.  For
4028         example, one might partition by date ranges, or by ranges of
4029         identifiers for particular business objects.
4030         Each range's bounds are understood as being inclusive at the
4031         lower end and exclusive at the upper end.  For example, if one
4032         partition's range is from <literal>1</literal>
4033         to <literal>10</literal>, and the next one's range is
4034         from <literal>10</literal> to <literal>20</literal>, then
4035         value <literal>10</literal> belongs to the second partition not
4036         the first.
4037        </para>
4038       </listitem>
4039      </varlistentry>
4040
4041      <varlistentry id="ddl-partitioning-overview-list">
4042       <term>List Partitioning</term>
4043
4044       <listitem>
4045        <para>
4046         The table is partitioned by explicitly listing which key value(s)
4047         appear in each partition.
4048        </para>
4049       </listitem>
4050      </varlistentry>
4051
4052      <varlistentry id="ddl-partitioning-overview-hash">
4053       <term>Hash Partitioning</term>
4054
4055       <listitem>
4056        <para>
4057         The table is partitioned by specifying a modulus and a remainder for
4058         each partition. Each partition will hold the rows for which the hash
4059         value of the partition key divided by the specified modulus will
4060         produce the specified remainder.
4061        </para>
4062       </listitem>
4063      </varlistentry>
4064     </variablelist>
4065
4066     If your application needs to use other forms of partitioning not listed
4067     above, alternative methods such as inheritance and
4068     <literal>UNION ALL</literal> views can be used instead.  Such methods
4069     offer flexibility but do not have some of the performance benefits
4070     of built-in declarative partitioning.
4071    </para>
4072   </sect2>
4073
4074  <sect2 id="ddl-partitioning-declarative">
4075   <title>Declarative Partitioning</title>
4076
4077   <para>
4078    <productname>PostgreSQL</productname> allows you to declare
4079    that a table is divided into partitions.  The table that is divided
4080    is referred to as a <firstterm>partitioned table</firstterm>.  The
4081    declaration includes the <firstterm>partitioning method</firstterm>
4082    as described above, plus a list of columns or expressions to be used
4083    as the <firstterm>partition key</firstterm>.
4084   </para>
4085
4086   <para>
4087    The partitioned table itself is a <quote>virtual</quote> table having
4088    no storage of its own.  Instead, the storage belongs
4089    to <firstterm>partitions</firstterm>, which are otherwise-ordinary
4090    tables associated with the partitioned table.
4091    Each partition stores a subset of the data as defined by its
4092    <firstterm>partition bounds</firstterm>.
4093    All rows inserted into a partitioned table will be routed to the
4094    appropriate one of the partitions based on the values of the partition
4095    key column(s).
4096    Updating the partition key of a row will cause it to be moved into a
4097    different partition if it no longer satisfies the partition bounds
4098    of its original partition.
4099   </para>
4100
4101   <para>
4102    Partitions may themselves be defined as partitioned tables, resulting
4103    in <firstterm>sub-partitioning</firstterm>.  Although all partitions
4104    must have the same columns as their partitioned parent, partitions may
4105    have their
4106    own indexes, constraints and default values, distinct from those of other
4107    partitions.  See <xref linkend="sql-createtable"/> for more details on
4108    creating partitioned tables and partitions.
4109   </para>
4110
4111   <para>
4112    It is not possible to turn a regular table into a partitioned table or
4113    vice versa.  However, it is possible to add an existing regular or
4114    partitioned table as a partition of a partitioned table, or remove a
4115    partition from a partitioned table turning it into a standalone table;
4116    this can simplify and speed up many maintenance processes.
4117    See <xref linkend="sql-altertable"/> to learn more about the
4118    <command>ATTACH PARTITION</command> and <command>DETACH PARTITION</command>
4119    sub-commands.
4120   </para>
4121
4122   <para>
4123    Partitions can also be <link linkend="ddl-foreign-data">foreign
4124    tables</link>, although considerable care is needed because it is then
4125    the user's responsibility that the contents of the foreign table
4126    satisfy the partitioning rule.  There are some other restrictions as
4127    well.  See <xref linkend="sql-createforeigntable"/> for more
4128    information.
4129   </para>
4130
4131   <sect3 id="ddl-partitioning-declarative-example">
4132    <title>Example</title>
4133
4134   <para>
4135    Suppose we are constructing a database for a large ice cream company.
4136    The company measures peak temperatures every day as well as ice cream
4137    sales in each region. Conceptually, we want a table like:
4138
4139<programlisting>
4140CREATE TABLE measurement (
4141    city_id         int not null,
4142    logdate         date not null,
4143    peaktemp        int,
4144    unitsales       int
4145);
4146</programlisting>
4147
4148    We know that most queries will access just the last week's, month's or
4149    quarter's data, since the main use of this table will be to prepare
4150    online reports for management.  To reduce the amount of old data that
4151    needs to be stored, we decide to keep only the most recent 3 years
4152    worth of data. At the beginning of each month we will remove the oldest
4153    month's data.  In this situation we can use partitioning to help us meet
4154    all of our different requirements for the measurements table.
4155   </para>
4156
4157   <para>
4158    To use declarative partitioning in this case, use the following steps:
4159
4160    <orderedlist spacing="compact">
4161     <listitem>
4162      <para>
4163       Create the <structname>measurement</structname> table as a partitioned
4164       table by specifying the <literal>PARTITION BY</literal> clause, which
4165       includes the partitioning method (<literal>RANGE</literal> in this
4166       case) and the list of column(s) to use as the partition key.
4167
4168<programlisting>
4169CREATE TABLE measurement (
4170    city_id         int not null,
4171    logdate         date not null,
4172    peaktemp        int,
4173    unitsales       int
4174) PARTITION BY RANGE (logdate);
4175</programlisting>
4176      </para>
4177     </listitem>
4178
4179     <listitem>
4180      <para>
4181       Create partitions.  Each partition's definition must specify bounds
4182       that correspond to the partitioning method and partition key of the
4183       parent.  Note that specifying bounds such that the new partition's
4184       values would overlap with those in one or more existing partitions will
4185       cause an error.
4186      </para>
4187
4188      <para>
4189       Partitions thus created are in every way normal
4190       <productname>PostgreSQL</productname>
4191       tables (or, possibly, foreign tables).  It is possible to specify a
4192       tablespace and storage parameters for each partition separately.
4193      </para>
4194
4195      <para>
4196       For our example, each partition should hold one month's worth of
4197       data, to match the requirement of deleting one month's data at a
4198       time.  So the commands might look like:
4199
4200<programlisting>
4201CREATE TABLE measurement_y2006m02 PARTITION OF measurement
4202    FOR VALUES FROM ('2006-02-01') TO ('2006-03-01');
4203
4204CREATE TABLE measurement_y2006m03 PARTITION OF measurement
4205    FOR VALUES FROM ('2006-03-01') TO ('2006-04-01');
4206
4207...
4208CREATE TABLE measurement_y2007m11 PARTITION OF measurement
4209    FOR VALUES FROM ('2007-11-01') TO ('2007-12-01');
4210
4211CREATE TABLE measurement_y2007m12 PARTITION OF measurement
4212    FOR VALUES FROM ('2007-12-01') TO ('2008-01-01')
4213    TABLESPACE fasttablespace;
4214
4215CREATE TABLE measurement_y2008m01 PARTITION OF measurement
4216    FOR VALUES FROM ('2008-01-01') TO ('2008-02-01')
4217    WITH (parallel_workers = 4)
4218    TABLESPACE fasttablespace;
4219</programlisting>
4220
4221       (Recall that adjacent partitions can share a bound value, since
4222       range upper bounds are treated as exclusive bounds.)
4223      </para>
4224
4225      <para>
4226       If you wish to implement sub-partitioning, again specify the
4227       <literal>PARTITION BY</literal> clause in the commands used to create
4228       individual partitions, for example:
4229
4230<programlisting>
4231CREATE TABLE measurement_y2006m02 PARTITION OF measurement
4232    FOR VALUES FROM ('2006-02-01') TO ('2006-03-01')
4233    PARTITION BY RANGE (peaktemp);
4234</programlisting>
4235
4236       After creating partitions of <structname>measurement_y2006m02</structname>,
4237       any data inserted into <structname>measurement</structname> that is mapped to
4238       <structname>measurement_y2006m02</structname> (or data that is
4239       directly inserted into <structname>measurement_y2006m02</structname>,
4240       which is allowed provided its partition constraint is satisfied)
4241       will be further redirected to one of its
4242       partitions based on the <structfield>peaktemp</structfield> column.  The partition
4243       key specified may overlap with the parent's partition key, although
4244       care should be taken when specifying the bounds of a sub-partition
4245       such that the set of data it accepts constitutes a subset of what
4246       the partition's own bounds allow; the system does not try to check
4247       whether that's really the case.
4248      </para>
4249
4250      <para>
4251       Inserting data into the parent table that does not map
4252       to one of the existing partitions will cause an error; an appropriate
4253       partition must be added manually.
4254      </para>
4255
4256      <para>
4257       It is not necessary to manually create table constraints describing
4258       the partition boundary conditions for partitions.  Such constraints
4259       will be created automatically.
4260      </para>
4261     </listitem>
4262
4263     <listitem>
4264      <para>
4265       Create an index on the key column(s), as well as any other indexes you
4266       might want, on the partitioned table. (The key index is not strictly
4267       necessary, but in most scenarios it is helpful.)
4268       This automatically creates a matching index on each partition, and
4269       any partitions you create or attach later will also have such an
4270       index.
4271       An index or unique constraint declared on a partitioned table
4272       is <quote>virtual</quote> in the same way that the partitioned table
4273       is: the actual data is in child indexes on the individual partition
4274       tables.
4275
4276<programlisting>
4277CREATE INDEX ON measurement (logdate);
4278</programlisting>
4279      </para>
4280     </listitem>
4281
4282      <listitem>
4283       <para>
4284        Ensure that the <xref linkend="guc-enable-partition-pruning"/>
4285        configuration parameter is not disabled in <filename>postgresql.conf</filename>.
4286        If it is, queries will not be optimized as desired.
4287       </para>
4288      </listitem>
4289    </orderedlist>
4290   </para>
4291
4292   <para>
4293    In the above example we would be creating a new partition each month, so
4294    it might be wise to write a script that generates the required DDL
4295    automatically.
4296   </para>
4297   </sect3>

4299   <sect3 id="ddl-partitioning-declarative-maintenance">
4300    <title>Partition Maintenance</title>
4301
4302    <para>
4303      Normally the set of partitions established when initially defining the
4304      table is not intended to remain static.  It is common to want to
4305      remove partitions holding old data and periodically add new partitions for
4306      new data. One of the most important advantages of partitioning is
4307      precisely that it allows this otherwise painful task to be executed
4308      nearly instantaneously by manipulating the partition structure, rather
4309      than physically moving large amounts of data around.
4310    </para>
4311
4312    <para>
4313     The simplest option for removing old data is to drop the partition that
4314     is no longer necessary:
4315<programlisting>
4316DROP TABLE measurement_y2006m02;
4317</programlisting>
4318     This can very quickly delete millions of records because it doesn't have
4319     to individually delete every record.  Note however that the above command
4320     requires taking an <literal>ACCESS EXCLUSIVE</literal> lock on the parent
4321     table.
4322    </para>
4323
4324   <para>
4325     Another option that is often preferable is to remove the partition from
4326     the partitioned table but retain access to it as a table in its own
4327     right.  This has two forms:
4328
4329<programlisting>
4330ALTER TABLE measurement DETACH PARTITION measurement_y2006m02;
4331ALTER TABLE measurement DETACH PARTITION measurement_y2006m02 CONCURRENTLY;
4332</programlisting>
4333
4334     These allow further operations to be performed on the data before
4335     it is dropped. For example, this is often a useful time to back up
4336     the data using <command>COPY</command>, <application>pg_dump</application>, or
4337     similar tools. It might also be a useful time to aggregate data
4338     into smaller formats, perform other data manipulations, or run
4339     reports.  The first form of the command requires an
4340     <literal>ACCESS EXCLUSIVE</literal> lock on the parent table.
4341     Adding the <literal>CONCURRENTLY</literal> qualifier as in the second
4342     form allows the detach operation to require only
4343     <literal>SHARE UPDATE EXCLUSIVE</literal> lock on the parent table, but see
4344     <link linkend="sql-altertable-detach-partition"><literal>ALTER TABLE ... DETACH PARTITION</literal></link>
4345     for details on the restrictions.
4346   </para>
4347
4348   <para>
4349     Similarly we can add a new partition to handle new data. We can create an
4350     empty partition in the partitioned table just as the original partitions
4351     were created above:
4352
4353<programlisting>
4354CREATE TABLE measurement_y2008m02 PARTITION OF measurement
4355    FOR VALUES FROM ('2008-02-01') TO ('2008-03-01')
4356    TABLESPACE fasttablespace;
4357</programlisting>
4358
4359     As an alternative to creating a new partition, it is sometimes more
4360     convenient to create a new table separate from the partition structure
4361     and attach it as a partition later.  This allows new data to be loaded,
4362     checked, and transformed prior to it appearing in the partitioned table.
4363     Moreover, the <literal>ATTACH PARTITION</literal> operation requires
4364     only a <literal>SHARE UPDATE EXCLUSIVE</literal> lock on the
4365     partitioned table rather than the <literal>ACCESS EXCLUSIVE</literal>
4366     lock required by <command>CREATE TABLE ... PARTITION OF</command>,
4367     so it is more friendly to concurrent operations on the partitioned table;
4368     see <link linkend="sql-altertable-attach-partition"><literal>ALTER TABLE ... ATTACH PARTITION</literal></link>
4369     for additional details.  The
4370     <link linkend="sql-createtable-parms-like"><literal>CREATE TABLE ... LIKE</literal></link>
4371     option can be helpful to avoid tediously repeating the parent table's
4372     definition; for example:
4373
4374<programlisting>
4375CREATE TABLE measurement_y2008m02
4376  (LIKE measurement INCLUDING DEFAULTS INCLUDING CONSTRAINTS)
4377  TABLESPACE fasttablespace;
4378
4379ALTER TABLE measurement_y2008m02 ADD CONSTRAINT y2008m02
4380   CHECK ( logdate &gt;= DATE '2008-02-01' AND logdate &lt; DATE '2008-03-01' );
4381
4382\copy measurement_y2008m02 from 'measurement_y2008m02'
4383-- possibly some other data preparation work
4384
4385ALTER TABLE measurement ATTACH PARTITION measurement_y2008m02
4386    FOR VALUES FROM ('2008-02-01') TO ('2008-03-01' );
4387</programlisting>
4388    </para>
4389
4390    <para>
4391     Note that when running the <command>ATTACH PARTITION</command> command,
4392     the table will be scanned to validate the partition constraint while
4393     holding an <literal>ACCESS EXCLUSIVE</literal> lock on that partition.
4394     As shown above, it is recommended to avoid this scan by creating a
4395     <literal>CHECK</literal> constraint matching the expected partition
4396     constraint on the table prior to attaching it.  Once the
4397     <command>ATTACH PARTITION</command> is complete, it is recommended to drop
4398     the now-redundant <literal>CHECK</literal> constraint.
4399     If the table being attached is itself a partitioned table, then each of its
4400     sub-partitions will be recursively locked and scanned until either a
4401     suitable <literal>CHECK</literal> constraint is encountered or the leaf
4402     partitions are reached.
4403    </para>
4404
4405    <para>
4406     Similarly, if the partitioned table has a <literal>DEFAULT</literal>
4407     partition, it is recommended to create a <literal>CHECK</literal>
4408     constraint which excludes the to-be-attached partition's constraint.  If
4409     this is not done, the <literal>DEFAULT</literal> partition will be
4410     scanned to verify that it contains no records which should be located in
4411     the partition being attached.  This operation will be performed whilst
4412     holding an <literal>ACCESS EXCLUSIVE</literal> lock on the <literal>
4413     DEFAULT</literal> partition.  If the <literal>DEFAULT</literal> partition
4414     is itself a partitioned table, then each of its partitions will be
4415     recursively checked in the same way as the table being attached, as
4416     mentioned above.
4417    </para>
4418
4419    <para>
4420     As mentioned earlier, it is possible to create indexes on partitioned
4421     tables so that they are applied automatically to the entire hierarchy.
4422     This can be very convenient as not only will all existing partitions be
4423     indexed, but any future partitions will be as well.  However, one
4424     limitation when creating new indexes on partitioned tables is that it
4425     is not possible to use the <literal>CONCURRENTLY</literal>
4426     qualifier, which could lead to long lock times.  To avoid this, you can
4427     use <command>CREATE INDEX ON ONLY</command> the partitioned table, which
4428     creates the new index marked as invalid, preventing automatic application
4429     to existing partitions.  Instead, indexes can then be created individually
4430     on each partition using <literal>CONCURRENTLY</literal> and
4431     <firstterm>attached</firstterm> to the partitioned index on the parent
4432     using <command>ALTER INDEX ... ATTACH PARTITION</command>.  Once indexes for
4433     all the partitions are attached to the parent index, the parent index will
4434     be marked valid automatically.  Example:
4435<programlisting>
4436CREATE INDEX measurement_usls_idx ON ONLY measurement (unitsales);
4437
4438CREATE INDEX CONCURRENTLY measurement_usls_200602_idx
4439    ON measurement_y2006m02 (unitsales);
4440ALTER INDEX measurement_usls_idx
4441    ATTACH PARTITION measurement_usls_200602_idx;
4442...
4443</programlisting>
4444
4445     This technique can be used with <literal>UNIQUE</literal> and
4446     <literal>PRIMARY KEY</literal> constraints too; the indexes are created
4447     implicitly when the constraint is created.  Example:
4448<programlisting>
4449ALTER TABLE ONLY measurement ADD UNIQUE (city_id, logdate);
4450
4451ALTER TABLE measurement_y2006m02 ADD UNIQUE (city_id, logdate);
4452ALTER INDEX measurement_city_id_logdate_key
4453    ATTACH PARTITION measurement_y2006m02_city_id_logdate_key;
4454...
4455</programlisting>
4456    </para>
4457   </sect3>
4458
4459   <sect3 id="ddl-partitioning-declarative-limitations">
4460    <title>Limitations</title>
4461
4462   <para>
4463    The following limitations apply to partitioned tables:
4464    <itemizedlist>
4465     <listitem>
4466      <para>
4467       To create a unique or primary key constraint on a partitioned table,
4468       the partition keys must not include any expressions or function calls
4469       and the constraint's columns must include all of the partition key
4470       columns.  This limitation exists because the individual indexes making
4471       up the constraint can only directly enforce uniqueness within their own
4472       partitions; therefore, the partition structure itself must guarantee
4473       that there are not duplicates in different partitions.
4474      </para>
4475     </listitem>
4476
4477     <listitem>
4478      <para>
4479       Similarly an exclusion constraint must include all the
4480       partition key columns. Furthermore the constraint must compare those
4481       columns for equality (not e.g. <literal>&amp;&amp;</literal>).
4482       Again, this limitation stems from not being able to enforce
4483       cross-partition restrictions. The constraint may include additional
4484       columns that aren't part of the partition key, and it may compare
4485       those with any operators you like.
4486      </para>
4487     </listitem>
4488
4489     <listitem>
4490      <para>
4491       <literal>BEFORE ROW</literal> triggers on <literal>INSERT</literal>
4492       cannot change which partition is the final destination for a new row.
4493      </para>
4494     </listitem>
4495
4496     <listitem>
4497      <para>
4498       Mixing temporary and permanent relations in the same partition tree is
4499       not allowed.  Hence, if the partitioned table is permanent, so must be
4500       its partitions and likewise if the partitioned table is temporary.  When
4501       using temporary relations, all members of the partition tree have to be
4502       from the same session.
4503      </para>
4504     </listitem>
4505    </itemizedlist>
4506    </para>
4507
4508    <para>
4509     Individual partitions are linked to their partitioned table using
4510     inheritance behind-the-scenes.  However, it is not possible to use
4511     all of the generic features of inheritance with declaratively
4512     partitioned tables or their partitions, as discussed below.  Notably,
4513     a partition cannot have any parents other than the partitioned table
4514     it is a partition of, nor can a table inherit from both a partitioned
4515     table and a regular table.  That means partitioned tables and their
4516     partitions never share an inheritance hierarchy with regular tables.
4517    </para>
4518
4519    <para>
4520     Since a partition hierarchy consisting of the partitioned table and its
4521     partitions is still an inheritance hierarchy,
4522     <structfield>tableoid</structfield> and all the normal rules of
4523     inheritance apply as described in <xref linkend="ddl-inherit"/>, with
4524     a few exceptions:
4525
4526     <itemizedlist>
4527      <listitem>
4528       <para>
4529        Partitions cannot have columns that are not present in the parent.  It
4530        is not possible to specify columns when creating partitions with
4531        <command>CREATE TABLE</command>, nor is it possible to add columns to
4532        partitions after-the-fact using <command>ALTER TABLE</command>.
4533        Tables may be added as a partition with <command>ALTER TABLE
4534        ... ATTACH PARTITION</command> only if their columns exactly match
4535        the parent.
4536       </para>
4537      </listitem>
4538
4539      <listitem>
4540       <para>
4541        Both <literal>CHECK</literal> and <literal>NOT NULL</literal>
4542        constraints of a partitioned table are always inherited by all its
4543        partitions; it is not allowed to create <literal>NO INHERIT</literal>
4544        constraints of those types.
4545        You cannot drop a constraint of those types if the same constraint
4546        is present in the parent table.
4547       </para>
4548      </listitem>
4549
4550      <listitem>
4551       <para>
4552        Using <literal>ONLY</literal> to add or drop a constraint on only
4553        the partitioned table is supported as long as there are no
4554        partitions.  Once partitions exist, using <literal>ONLY</literal>
4555        will result in an error for any constraints other than
4556        <literal>UNIQUE</literal> and <literal>PRIMARY KEY</literal>.
4557        Instead, constraints on the partitions
4558        themselves can be added and (if they are not present in the parent
4559        table) dropped.
4560       </para>
4561      </listitem>
4562
4563      <listitem>
4564       <para>
4565        As a partitioned table does not have any data itself, attempts to use
4566        <command>TRUNCATE</command> <literal>ONLY</literal> on a partitioned
4567        table will always return an error.
4568       </para>
4569      </listitem>
4570     </itemizedlist>
4571    </para>
4572    </sect3>
4573   </sect2>

4575   <sect2 id="ddl-partitioning-using-inheritance">
4576    <title>Partitioning Using Inheritance</title>
4577
4578    <para>
4579     While the built-in declarative partitioning is suitable for most
4580     common use cases, there are some circumstances where a more flexible
4581     approach may be useful.  Partitioning can be implemented using table
4582     inheritance, which allows for several features not supported
4583     by declarative partitioning, such as:
4584
4585     <itemizedlist>
4586      <listitem>
4587       <para>
4588        For declarative partitioning, partitions must have exactly the same set
4589        of columns as the partitioned table, whereas with table inheritance,
4590        child tables may have extra columns not present in the parent.
4591       </para>
4592      </listitem>
4593
4594      <listitem>
4595       <para>
4596        Table inheritance allows for multiple inheritance.
4597       </para>
4598      </listitem>
4599
4600      <listitem>
4601       <para>
4602        Declarative partitioning only supports range, list and hash
4603        partitioning, whereas table inheritance allows data to be divided in a
4604        manner of the user's choosing.  (Note, however, that if constraint
4605        exclusion is unable to prune child tables effectively, query performance
4606        might be poor.)
4607       </para>
4608      </listitem>
4609     </itemizedlist>
4610    </para>
4611
4612    <sect3 id="ddl-partitioning-inheritance-example">
4613     <title>Example</title>
4614
4615     <para>
4616      This example builds a partitioning structure equivalent to the
4617      declarative partitioning example above.  Use
4618      the following steps:
4619
4620      <orderedlist spacing="compact">
4621       <listitem>
4622        <para>
4623         Create the <quote>root</quote> table, from which all of the
4624         <quote>child</quote> tables will inherit.  This table will contain no data.  Do not
4625         define any check constraints on this table, unless you intend them
4626         to be applied equally to all child tables.  There is no point in
4627         defining any indexes or unique constraints on it, either.  For our
4628         example, the root table is the <structname>measurement</structname>
4629         table as originally defined:
4630
4631<programlisting>
4632CREATE TABLE measurement (
4633    city_id         int not null,
4634    logdate         date not null,
4635    peaktemp        int,
4636    unitsales       int
4637);
4638</programlisting>
4639        </para>
4640       </listitem>
4641
4642       <listitem>
4643        <para>
4644         Create several <quote>child</quote> tables that each inherit from
4645         the root table.  Normally, these tables will not add any columns
4646         to the set inherited from the root.  Just as with declarative
4647         partitioning, these tables are in every way normal
4648         <productname>PostgreSQL</productname> tables (or foreign tables).
4649        </para>
4650
4651        <para>
4652<programlisting>
4653CREATE TABLE measurement_y2006m02 () INHERITS (measurement);
4654CREATE TABLE measurement_y2006m03 () INHERITS (measurement);
4655...
4656CREATE TABLE measurement_y2007m11 () INHERITS (measurement);
4657CREATE TABLE measurement_y2007m12 () INHERITS (measurement);
4658CREATE TABLE measurement_y2008m01 () INHERITS (measurement);
4659</programlisting>
4660        </para>
4661       </listitem>
4662
4663       <listitem>
4664        <para>
4665         Add non-overlapping table constraints to the child tables to
4666         define the allowed key values in each.
4667        </para>
4668
4669        <para>
4670         Typical examples would be:
4671<programlisting>
4672CHECK ( x = 1 )
4673CHECK ( county IN ( 'Oxfordshire', 'Buckinghamshire', 'Warwickshire' ))
4674CHECK ( outletID &gt;= 100 AND outletID &lt; 200 )
4675</programlisting>
4676         Ensure that the constraints guarantee that there is no overlap
4677         between the key values permitted in different child tables.  A common
4678         mistake is to set up range constraints like:
4679<programlisting>
4680CHECK ( outletID BETWEEN 100 AND 200 )
4681CHECK ( outletID BETWEEN 200 AND 300 )
4682</programlisting>
4683         This is wrong since it is not clear which child table the key
4684         value 200 belongs in.
4685         Instead, ranges should be defined in this style:
4686
4687<programlisting>
4688CREATE TABLE measurement_y2006m02 (
4689    CHECK ( logdate &gt;= DATE '2006-02-01' AND logdate &lt; DATE '2006-03-01' )
4690) INHERITS (measurement);
4691
4692CREATE TABLE measurement_y2006m03 (
4693    CHECK ( logdate &gt;= DATE '2006-03-01' AND logdate &lt; DATE '2006-04-01' )
4694) INHERITS (measurement);
4695
4696...
4697CREATE TABLE measurement_y2007m11 (
4698    CHECK ( logdate &gt;= DATE '2007-11-01' AND logdate &lt; DATE '2007-12-01' )
4699) INHERITS (measurement);
4700
4701CREATE TABLE measurement_y2007m12 (
4702    CHECK ( logdate &gt;= DATE '2007-12-01' AND logdate &lt; DATE '2008-01-01' )
4703) INHERITS (measurement);
4704
4705CREATE TABLE measurement_y2008m01 (
4706    CHECK ( logdate &gt;= DATE '2008-01-01' AND logdate &lt; DATE '2008-02-01' )
4707) INHERITS (measurement);
4708</programlisting>
4709        </para>
4710       </listitem>
4711
4712       <listitem>
4713        <para>
4714         For each child table, create an index on the key column(s),
4715         as well as any other indexes you might want.
4716<programlisting>
4717CREATE INDEX measurement_y2006m02_logdate ON measurement_y2006m02 (logdate);
4718CREATE INDEX measurement_y2006m03_logdate ON measurement_y2006m03 (logdate);
4719CREATE INDEX measurement_y2007m11_logdate ON measurement_y2007m11 (logdate);
4720CREATE INDEX measurement_y2007m12_logdate ON measurement_y2007m12 (logdate);
4721CREATE INDEX measurement_y2008m01_logdate ON measurement_y2008m01 (logdate);
4722</programlisting>
4723        </para>
4724       </listitem>
4725
4726       <listitem>
4727        <para>
4728         We want our application to be able to say <literal>INSERT INTO
4729         measurement ...</literal> and have the data be redirected into the
4730         appropriate child table.  We can arrange that by attaching
4731         a suitable trigger function to the root table.
4732         If data will be added only to the latest child, we can
4733         use a very simple trigger function:
4734
4735<programlisting>
4736CREATE OR REPLACE FUNCTION measurement_insert_trigger()
4737RETURNS TRIGGER AS $$
4738BEGIN
4739    INSERT INTO measurement_y2008m01 VALUES (NEW.*);
4740    RETURN NULL;
4741END;
4742$$
4743LANGUAGE plpgsql;
4744</programlisting>
4745        </para>
4746
4747        <para>
4748         After creating the function, we create a trigger which
4749         calls the trigger function:
4750
4751<programlisting>
4752CREATE TRIGGER insert_measurement_trigger
4753    BEFORE INSERT ON measurement
4754    FOR EACH ROW EXECUTE FUNCTION measurement_insert_trigger();
4755</programlisting>
4756
4757         We must redefine the trigger function each month so that it always
4758         inserts into the current child table.  The trigger definition does
4759         not need to be updated, however.
4760        </para>
4761
4762        <para>
4763         We might want to insert data and have the server automatically
4764         locate the child table into which the row should be added. We
4765         could do this with a more complex trigger function, for example:
4766
4767<programlisting>
4768CREATE OR REPLACE FUNCTION measurement_insert_trigger()
4769RETURNS TRIGGER AS $$
4770BEGIN
4771    IF ( NEW.logdate &gt;= DATE '2006-02-01' AND
4772         NEW.logdate &lt; DATE '2006-03-01' ) THEN
4773        INSERT INTO measurement_y2006m02 VALUES (NEW.*);
4774    ELSIF ( NEW.logdate &gt;= DATE '2006-03-01' AND
4775            NEW.logdate &lt; DATE '2006-04-01' ) THEN
4776        INSERT INTO measurement_y2006m03 VALUES (NEW.*);
4777    ...
4778    ELSIF ( NEW.logdate &gt;= DATE '2008-01-01' AND
4779            NEW.logdate &lt; DATE '2008-02-01' ) THEN
4780        INSERT INTO measurement_y2008m01 VALUES (NEW.*);
4781    ELSE
4782        RAISE EXCEPTION 'Date out of range.  Fix the measurement_insert_trigger() function!';
4783    END IF;
4784    RETURN NULL;
4785END;
4786$$
4787LANGUAGE plpgsql;
4788</programlisting>
4789
4790         The trigger definition is the same as before.
4791         Note that each <literal>IF</literal> test must exactly match the
4792         <literal>CHECK</literal> constraint for its child table.
4793        </para>
4794
4795        <para>
4796         While this function is more complex than the single-month case,
4797         it doesn't need to be updated as often, since branches can be
4798         added in advance of being needed.
4799        </para>
4800
4801        <note>
4802         <para>
4803          In practice, it might be best to check the newest child first,
4804          if most inserts go into that child.  For simplicity, we have
4805          shown the trigger's tests in the same order as in other parts
4806          of this example.
4807         </para>
4808        </note>
4809
4810        <para>
4811         A different approach to redirecting inserts into the appropriate
4812         child table is to set up rules, instead of a trigger, on the
4813         root table.  For example:
4814
4815<programlisting>
4816CREATE RULE measurement_insert_y2006m02 AS
4817ON INSERT TO measurement WHERE
4818    ( logdate &gt;= DATE '2006-02-01' AND logdate &lt; DATE '2006-03-01' )
4819DO INSTEAD
4820    INSERT INTO measurement_y2006m02 VALUES (NEW.*);
4821...
4822CREATE RULE measurement_insert_y2008m01 AS
4823ON INSERT TO measurement WHERE
4824    ( logdate &gt;= DATE '2008-01-01' AND logdate &lt; DATE '2008-02-01' )
4825DO INSTEAD
4826    INSERT INTO measurement_y2008m01 VALUES (NEW.*);
4827</programlisting>
4828
4829         A rule has significantly more overhead than a trigger, but the
4830         overhead is paid once per query rather than once per row, so this
4831         method might be advantageous for bulk-insert situations.  In most
4832         cases, however, the trigger method will offer better performance.
4833        </para>
4834
4835        <para>
4836         Be aware that <command>COPY</command> ignores rules.  If you want to
4837         use <command>COPY</command> to insert data, you'll need to copy into the
4838         correct child table rather than directly into the root. <command>COPY</command>
4839         does fire triggers, so you can use it normally if you use the trigger
4840         approach.
4841        </para>
4842
4843        <para>
4844         Another disadvantage of the rule approach is that there is no simple
4845         way to force an error if the set of rules doesn't cover the insertion
4846         date; the data will silently go into the root table instead.
4847        </para>
4848       </listitem>
4849
4850       <listitem>
4851        <para>
4852         Ensure that the <xref linkend="guc-constraint-exclusion"/>
4853         configuration parameter is not disabled in
4854         <filename>postgresql.conf</filename>; otherwise
4855         child tables may be accessed unnecessarily.
4856        </para>
4857       </listitem>
4858      </orderedlist>
4859     </para>
4860
4861     <para>
4862      As we can see, a complex table hierarchy could require a
4863      substantial amount of DDL.  In the above example we would be creating
4864      a new child table each month, so it might be wise to write a script that
4865      generates the required DDL automatically.
4866     </para>
4867    </sect3>

4870     <title>Maintenance for Inheritance Partitioning</title>
4871     <para>
4872      To remove old data quickly, simply drop the child table that is no longer
4873      necessary:
4874<programlisting>
4875DROP TABLE measurement_y2006m02;
4876</programlisting>
4877     </para>
4878
4879    <para>
4880     To remove the child table from the inheritance hierarchy table but retain access to
4881     it as a table in its own right:
4882
4883<programlisting>
4884ALTER TABLE measurement_y2006m02 NO INHERIT measurement;
4885</programlisting>
4886    </para>
4887
4888    <para>
4889     To add a new child table to handle new data, create an empty child table
4890     just as the original children were created above:
4891
4892<programlisting>
4893CREATE TABLE measurement_y2008m02 (
4894    CHECK ( logdate &gt;= DATE '2008-02-01' AND logdate &lt; DATE '2008-03-01' )
4895) INHERITS (measurement);
4896</programlisting>
4897
4898     Alternatively, one may want to create and populate the new child table
4899     before adding it to the table hierarchy.  This could allow data to be
4900     loaded, checked, and transformed before being made visible to queries on
4901     the parent table.
4902
4903<programlisting>
4904CREATE TABLE measurement_y2008m02
4905  (LIKE measurement INCLUDING DEFAULTS INCLUDING CONSTRAINTS);
4906ALTER TABLE measurement_y2008m02 ADD CONSTRAINT y2008m02
4907   CHECK ( logdate &gt;= DATE '2008-02-01' AND logdate &lt; DATE '2008-03-01' );
4908\copy measurement_y2008m02 from 'measurement_y2008m02'
4909-- possibly some other data preparation work
4910ALTER TABLE measurement_y2008m02 INHERIT measurement;
4911</programlisting>
4912    </para>

4915   <sect3 id="ddl-partitioning-inheritance-caveats">
4916    <title>Caveats</title>
4917
4918    <para>
4919     The following caveats apply to partitioning implemented using
4920     inheritance:
4921     <itemizedlist>
4922      <listitem>
4923       <para>
4924        There is no automatic way to verify that all of the
4925        <literal>CHECK</literal> constraints are mutually
4926        exclusive.  It is safer to create code that generates
4927        child tables and creates and/or modifies associated objects than
4928        to write each by hand.
4929       </para>
4930      </listitem>
4931
4932      <listitem>
4933       <para>
4934        Indexes and foreign key constraints apply to single tables and not
4935        to their inheritance children, hence they have some
4936        <link linkend="ddl-inherit-caveats">caveats</link> to be aware of.
4937       </para>
4938      </listitem>
4939
4940      <listitem>
4941       <para>
4942        The schemes shown here assume that the values of a row's key column(s)
4943        never change, or at least do not change enough to require it to move to another partition.
4944        An <command>UPDATE</command> that attempts
4945        to do that will fail because of the <literal>CHECK</literal> constraints.
4946        If you need to handle such cases, you can put suitable update triggers
4947        on the child tables, but it makes management of the structure
4948        much more complicated.
4949       </para>
4950      </listitem>
4951
4952      <listitem>
4953       <para>
4954        Manual <command>VACUUM</command> and <command>ANALYZE</command>
4955        commands will automatically process all inheritance child tables.  If
4956        this is undesirable, you can use the <literal>ONLY</literal> keyword.
4957        A command like:
4958<programlisting>
4959ANALYZE ONLY measurement;
4960</programlisting>
4961        will only process the root table.
4962       </para>
4963      </listitem>
4964
4965      <listitem>
4966       <para>
4967        <command>INSERT</command> statements with <literal>ON CONFLICT</literal>
4968        clauses are unlikely to work as expected, as the <literal>ON CONFLICT</literal>
4969        action is only taken in case of unique violations on the specified
4970        target relation, not its child relations.
4971       </para>
4972      </listitem>
4973
4974      <listitem>
4975       <para>
4976        Triggers or rules will be needed to route rows to the desired
4977        child table, unless the application is explicitly aware of the
4978        partitioning scheme.  Triggers may be complicated to write, and will
4979        be much slower than the tuple routing performed internally by
4980        declarative partitioning.
4981       </para>
4982      </listitem>
4983     </itemizedlist>
4984    </para>
4985   </sect3>

4988  <sect2 id="ddl-partition-pruning">
4989   <title>Partition Pruning</title>
4990
4991   <indexterm>
4992    <primary>partition pruning</primary>
4993   </indexterm>
4994
4995   <para>
4996    <firstterm>Partition pruning</firstterm> is a query optimization technique
4997    that improves performance for declaratively partitioned tables.
4998    As an example:
4999
5000<programlisting>
5001SET enable_partition_pruning = on;                 -- the default
5002SELECT count(*) FROM measurement WHERE logdate &gt;= DATE '2008-01-01';
5003</programlisting>
5004
5005    Without partition pruning, the above query would scan each of the
5006    partitions of the <structname>measurement</structname> table. With
5007    partition pruning enabled, the planner will examine the definition
5008    of each partition and prove that the partition need not
5009    be scanned because it could not contain any rows meeting the query's
5010    <literal>WHERE</literal> clause.  When the planner can prove this, it
5011    excludes (<firstterm>prunes</firstterm>) the partition from the query
5012    plan.
5013   </para>
5014
5015   <para>
5016    By using the EXPLAIN command and the <xref
5017    linkend="guc-enable-partition-pruning"/> configuration parameter, it's
5018    possible to show the difference between a plan for which partitions have
5019    been pruned and one for which they have not.  A typical unoptimized
5020    plan for this type of table setup is:
5021<programlisting>
5022SET enable_partition_pruning = off;
5023EXPLAIN SELECT count(*) FROM measurement WHERE logdate &gt;= DATE '2008-01-01';
5024                                    QUERY PLAN
5025-------------------------------------------------------------------&zwsp;----------------
5026 Aggregate  (cost=188.76..188.77 rows=1 width=8)
5027   -&gt;  Append  (cost=0.00..181.05 rows=3085 width=0)
5028         -&gt;  Seq Scan on measurement_y2006m02  (cost=0.00..33.12 rows=617 width=0)
5029               Filter: (logdate &gt;= '2008-01-01'::date)
5030         -&gt;  Seq Scan on measurement_y2006m03  (cost=0.00..33.12 rows=617 width=0)
5031               Filter: (logdate &gt;= '2008-01-01'::date)
5032...
5033         -&gt;  Seq Scan on measurement_y2007m11  (cost=0.00..33.12 rows=617 width=0)
5034               Filter: (logdate &gt;= '2008-01-01'::date)
5035         -&gt;  Seq Scan on measurement_y2007m12  (cost=0.00..33.12 rows=617 width=0)
5036               Filter: (logdate &gt;= '2008-01-01'::date)
5037         -&gt;  Seq Scan on measurement_y2008m01  (cost=0.00..33.12 rows=617 width=0)
5038               Filter: (logdate &gt;= '2008-01-01'::date)
5039</programlisting>
5040
5041    Some or all of the partitions might use index scans instead of
5042    full-table sequential scans, but the point here is that there
5043    is no need to scan the older partitions at all to answer this query.
5044    When we enable partition pruning, we get a significantly
5045    cheaper plan that will deliver the same answer:
5046<programlisting>
5047SET enable_partition_pruning = on;
5048EXPLAIN SELECT count(*) FROM measurement WHERE logdate &gt;= DATE '2008-01-01';
5049                                    QUERY PLAN
5050-------------------------------------------------------------------&zwsp;----------------
5051 Aggregate  (cost=37.75..37.76 rows=1 width=8)
5052   -&gt;  Seq Scan on measurement_y2008m01  (cost=0.00..33.12 rows=617 width=0)
5053         Filter: (logdate &gt;= '2008-01-01'::date)
5054</programlisting>
5055   </para>
5056
5057   <para>
5058    Note that partition pruning is driven only by the constraints defined
5059    implicitly by the partition keys, not by the presence of indexes.
5060    Therefore it isn't necessary to define indexes on the key columns.
5061    Whether an index needs to be created for a given partition depends on
5062    whether you expect that queries that scan the partition will
5063    generally scan a large part of the partition or just a small part.
5064    An index will be helpful in the latter case but not the former.
5065   </para>
5066
5067   <para>
5068    Partition pruning can be performed not only during the planning of a
5069    given query, but also during its execution.  This is useful as it can
5070    allow more partitions to be pruned when clauses contain expressions
5071    whose values are not known at query planning time, for example,
5072    parameters defined in a <command>PREPARE</command> statement, using a
5073    value obtained from a subquery, or using a parameterized value on the
5074    inner side of a nested loop join.  Partition pruning during execution
5075    can be performed at any of the following times:
5076
5077    <itemizedlist>
5078     <listitem>
5079      <para>
5080       During initialization of the query plan.  Partition pruning can be
5081       performed here for parameter values which are known during the
5082       initialization phase of execution.  Partitions which are pruned
5083       during this stage will not show up in the query's
5084       <command>EXPLAIN</command> or <command>EXPLAIN ANALYZE</command>.
5085       It is possible to determine the number of partitions which were
5086       removed during this phase by observing the
5087       <quote>Subplans Removed</quote> property in the
5088       <command>EXPLAIN</command> output.  It's important to note that any
5089       partitions removed by the partition pruning done at this stage are
5090       still locked at the beginning of execution.
5091      </para>
5092     </listitem>
5093
5094     <listitem>
5095      <para>
5096       During actual execution of the query plan.  Partition pruning may
5097       also be performed here to remove partitions using values which are
5098       only known during actual query execution.  This includes values
5099       from subqueries and values from execution-time parameters such as
5100       those from parameterized nested loop joins.  Since the value of
5101       these parameters may change many times during the execution of the
5102       query, partition pruning is performed whenever one of the
5103       execution parameters being used by partition pruning changes.
5104       Determining if partitions were pruned during this phase requires
5105       careful inspection of the <literal>loops</literal> property in
5106       the <command>EXPLAIN ANALYZE</command> output.  Subplans
5107       corresponding to different partitions may have different values
5108       for it depending on how many times each of them was pruned during
5109       execution.  Some may be shown as <literal>(never executed)</literal>
5110       if they were pruned every time.
5111      </para>
5112     </listitem>
5113    </itemizedlist>
5114   </para>
5115
5116   <para>
5117    Partition pruning can be disabled using the
5118    <xref linkend="guc-enable-partition-pruning"/> setting.
5119   </para>
5120  </sect2>

5122  <sect2 id="ddl-partitioning-constraint-exclusion">
5123   <title>Partitioning and Constraint Exclusion</title>
5124
5125   <indexterm>
5126    <primary>constraint exclusion</primary>
5127   </indexterm>
5128
5129   <para>
5130    <firstterm>Constraint exclusion</firstterm> is a query optimization
5131    technique similar to partition pruning.  While it is primarily used
5132    for partitioning implemented using the legacy inheritance method, it can be
5133    used for other purposes, including with declarative partitioning.
5134   </para>
5135
5136   <para>
5137    Constraint exclusion works in a very similar way to partition
5138    pruning, except that it uses each table's <literal>CHECK</literal>
5139    constraints &mdash; which gives it its name &mdash; whereas partition
5140    pruning uses the table's partition bounds, which exist only in the
5141    case of declarative partitioning.  Another difference is that
5142    constraint exclusion is only applied at plan time; there is no attempt
5143    to remove partitions at execution time.
5144   </para>
5145
5146   <para>
5147    The fact that constraint exclusion uses <literal>CHECK</literal>
5148    constraints, which makes it slow compared to partition pruning, can
5149    sometimes be used as an advantage: because constraints can be defined
5150    even on declaratively-partitioned tables, in addition to their internal
5151    partition bounds, constraint exclusion may be able
5152    to elide additional partitions from the query plan.
5153   </para>
5154
5155   <para>
5156    The default (and recommended) setting of
5157    <xref linkend="guc-constraint-exclusion"/> is neither
5158    <literal>on</literal> nor <literal>off</literal>, but an intermediate setting
5159    called <literal>partition</literal>, which causes the technique to be
5160    applied only to queries that are likely to be working on inheritance partitioned
5161    tables.  The <literal>on</literal> setting causes the planner to examine
5162    <literal>CHECK</literal> constraints in all queries, even simple ones that
5163    are unlikely to benefit.
5164   </para>
5165
5166   <para>
5167    The following caveats apply to constraint exclusion:
5168
5169   <itemizedlist>
5170    <listitem>
5171     <para>
5172      Constraint exclusion is only applied during query planning, unlike
5173      partition pruning, which can also be applied during query execution.
5174     </para>
5175    </listitem>
5176
5177    <listitem>
5178     <para>
5179      Constraint exclusion only works when the query's <literal>WHERE</literal>
5180      clause contains constants (or externally supplied parameters).
5181      For example, a comparison against a non-immutable function such as
5182      <function>CURRENT_TIMESTAMP</function> cannot be optimized, since the
5183      planner cannot know which child table the function's value might fall
5184      into at run time.
5185     </para>
5186    </listitem>
5187
5188    <listitem>
5189     <para>
5190      Keep the partitioning constraints simple, else the planner may not be
5191      able to prove that child tables might not need to be visited.  Use simple
5192      equality conditions for list partitioning, or simple
5193      range tests for range partitioning, as illustrated in the preceding
5194      examples.  A good rule of thumb is that partitioning constraints should
5195      contain only comparisons of the partitioning column(s) to constants
5196      using B-tree-indexable operators, because only B-tree-indexable
5197      column(s) are allowed in the partition key.
5198     </para>
5199    </listitem>
5200
5201    <listitem>
5202     <para>
5203      All constraints on all children of the parent table are examined
5204      during constraint exclusion, so large numbers of children are likely
5205      to increase query planning time considerably.  So the legacy
5206      inheritance based partitioning will work well with up to perhaps a
5207      hundred child tables; don't try to use many thousands of children.
5208     </para>
5209    </listitem>
5210
5211   </itemizedlist>
5212   </para>
5213  </sect2>

5215  <sect2 id="ddl-partitioning-declarative-best-practices">
5216   <title>Best Practices for Declarative Partitioning</title>
5217
5218   <para>
5219    The choice of how to partition a table should be made carefully, as the
5220    performance of query planning and execution can be negatively affected by
5221    poor design.
5222   </para>
5223
5224   <para>
5225    One of the most critical design decisions will be the column or columns
5226    by which you partition your data.  Often the best choice will be to
5227    partition by the column or set of columns which most commonly appear in
5228    <literal>WHERE</literal> clauses of queries being executed on the
5229    partitioned table.  <literal>WHERE</literal> clauses that are compatible
5230    with the partition bound constraints can be used to prune unneeded
5231    partitions.  However, you may be forced into making other decisions by
5232    requirements for the <literal>PRIMARY KEY</literal> or a
5233    <literal>UNIQUE</literal> constraint.  Removal of unwanted data is also a
5234    factor to consider when planning your partitioning strategy.  An entire
5235    partition can be detached fairly quickly, so it may be beneficial to
5236    design the partition strategy in such a way that all data to be removed
5237    at once is located in a single partition.
5238   </para>
5239
5240   <para>
5241    Choosing the target number of partitions that the table should be divided
5242    into is also a critical decision to make.  Not having enough partitions
5243    may mean that indexes remain too large and that data locality remains poor
5244    which could result in low cache hit ratios.  However, dividing the table
5245    into too many partitions can also cause issues.  Too many partitions can
5246    mean longer query planning times and higher memory consumption during both
5247    query planning and execution, as further described below.
5248    When choosing how to partition your table,
5249    it's also important to consider what changes may occur in the future.  For
5250    example, if you choose to have one partition per customer and you
5251    currently have a small number of large customers, consider the
5252    implications if in several years you instead find yourself with a large
5253    number of small customers.  In this case, it may be better to choose to
5254    partition by <literal>HASH</literal> and choose a reasonable number of
5255    partitions rather than trying to partition by <literal>LIST</literal> and
5256    hoping that the number of customers does not increase beyond what it is
5257    practical to partition the data by.
5258   </para>
5259
5260   <para>
5261    Sub-partitioning can be useful to further divide partitions that are
5262    expected to become larger than other partitions.
5263    Another option is to use range partitioning with multiple columns in
5264    the partition key.
5265    Either of these can easily lead to excessive numbers of partitions,
5266    so restraint is advisable.
5267   </para>
5268
5269   <para>
5270    It is important to consider the overhead of partitioning during
5271    query planning and execution.  The query planner is generally able to
5272    handle partition hierarchies with up to a few thousand partitions fairly
5273    well, provided that typical queries allow the query planner to prune all
5274    but a small number of partitions.  Planning times become longer and memory
5275    consumption becomes higher when more partitions remain after the planner
5276    performs partition pruning.  Another
5277    reason to be concerned about having a large number of partitions is that
5278    the server's memory consumption may grow significantly over
5279    time, especially if many sessions touch large numbers of partitions.
5280    That's because each partition requires its metadata to be loaded into the
5281    local memory of each session that touches it.
5282   </para>
5283
5284   <para>
5285    With data warehouse type workloads, it can make sense to use a larger
5286    number of partitions than with an <acronym>OLTP</acronym> type workload.
5287    Generally, in data warehouses, query planning time is less of a concern as
5288    the majority of processing time is spent during query execution.  With
5289    either of these two types of workload, it is important to make the right
5290    decisions early, as re-partitioning large quantities of data can be
5291    painfully slow.  Simulations of the intended workload are often beneficial
5292    for optimizing the partitioning strategy.  Never just assume that more
5293    partitions are better than fewer partitions, nor vice-versa.
5294   </para>

5312   <para>
5313    <productname>PostgreSQL</productname> implements portions of the SQL/MED
5314    specification, allowing you to access data that resides outside
5315    PostgreSQL using regular SQL queries.  Such data is referred to as
5316    <firstterm>foreign data</firstterm>.  (Note that this usage is not to be confused
5317    with foreign keys, which are a type of constraint within the database.)
5318   </para>
5319
5320   <para>
5321    Foreign data is accessed with help from a
5322    <firstterm>foreign data wrapper</firstterm>. A foreign data wrapper is a
5323    library that can communicate with an external data source, hiding the

5326    modules; see <xref linkend="contrib"/>.  Other kinds of foreign data

5333    To access foreign data, you need to create a <firstterm>foreign server</firstterm>
5334    object, which defines how to connect to a particular external data source
5335    according to the set of options used by its supporting foreign data
5336    wrapper. Then you need to create one or more <firstterm>foreign
5337    tables</firstterm>, which define the structure of the remote data. A
5338    foreign table can be used in queries just like a normal table, but a
5339    foreign table has no storage in the PostgreSQL server.  Whenever it is
5340    used, <productname>PostgreSQL</productname> asks the foreign data wrapper
5341    to fetch data from the external source, or transmit data to the external
5342    source in the case of update commands.
5343   </para>
5344
5345   <para>
5346    Accessing remote data may require authenticating to the external

5407 <sect1 id="ddl-depend">
5408  <title>Dependency Tracking</title>
5409
5410  <indexterm zone="ddl-depend">
5411   <primary>CASCADE</primary>
5412   <secondary sortas="DROP">with DROP</secondary>
5413  </indexterm>
5414
5415  <indexterm zone="ddl-depend">
5416   <primary>RESTRICT</primary>
5417   <secondary sortas="DROP">with DROP</secondary>
5418  </indexterm>
5419
5420  <para>
5421   When you create complex database structures involving many tables
5422   with foreign key constraints, views, triggers, functions, etc. you
5423   implicitly create a net of dependencies between the objects.
5424   For instance, a table with a foreign key constraint depends on the
5425   table it references.
5426  </para>
5427
5428  <para>
5429   To ensure the integrity of the entire database structure,
5430   <productname>PostgreSQL</productname> makes sure that you cannot
5431   drop objects that other objects still depend on.  For example,
5432   attempting to drop the products table we considered in <xref
5433   linkend="ddl-constraints-fk"/>, with the orders table depending on
5434   it, would result in an error message like this:
5435<screen>
5436DROP TABLE products;
5437
5438ERROR:  cannot drop table products because other objects depend on it
5439DETAIL:  constraint orders_product_no_fkey on table orders depends on table products
5440HINT:  Use DROP ... CASCADE to drop the dependent objects too.
5441</screen>
5442   The error message contains a useful hint: if you do not want to
5443   bother deleting all the dependent objects individually, you can run:
5444<screen>
5445DROP TABLE products CASCADE;
5446</screen>
5447   and all the dependent objects will be removed, as will any objects
5448   that depend on them, recursively.  In this case, it doesn't remove
5449   the orders table, it only removes the foreign key constraint.
5450   It stops there because nothing depends on the foreign key constraint.
5451   (If you want to check what <command>DROP ... CASCADE</command> will do,
5452   run <command>DROP</command> without <literal>CASCADE</literal> and read the
5453   <literal>DETAIL</literal> output.)
5454  </para>
5455
5456  <para>
5457   Almost all <command>DROP</command> commands in <productname>PostgreSQL</productname> support
5458   specifying <literal>CASCADE</literal>.  Of course, the nature of
5459   the possible dependencies varies with the type of the object.  You
5460   can also write <literal>RESTRICT</literal> instead of
5461   <literal>CASCADE</literal> to get the default behavior, which is to
5462   prevent dropping objects that any other objects depend on.
5463  </para>
5464
5465  <note>
5466   <para>
5467    According to the SQL standard, specifying either
5468    <literal>RESTRICT</literal> or <literal>CASCADE</literal> is
5469    required in a <command>DROP</command> command.  No database system actually
5470    enforces that rule, but whether the default behavior
5471    is <literal>RESTRICT</literal> or <literal>CASCADE</literal> varies
5472    across systems.
5473   </para>
5474  </note>
5475
5476  <para>
5477   If a <command>DROP</command> command lists multiple
5478   objects, <literal>CASCADE</literal> is only required when there are
5479   dependencies outside the specified group.  For example, when saying
5480   <literal>DROP TABLE tab1, tab2</literal> the existence of a foreign
5481   key referencing <literal>tab1</literal> from <literal>tab2</literal> would not mean
5482   that <literal>CASCADE</literal> is needed to succeed.
5483  </para>
5484
5485  <para>
5486   For a user-defined function or procedure whose body is defined as a string
5487   literal, <productname>PostgreSQL</productname> tracks
5488   dependencies associated with the function's externally-visible properties,
5489   such as its argument and result types, but <emphasis>not</emphasis> dependencies
5490   that could only be known by examining the function body.  As an example,
5491   consider this situation:
5492
5493<programlisting>
5494CREATE TYPE rainbow AS ENUM ('red', 'orange', 'yellow',
5495                             'green', 'blue', 'purple');
5496
5497CREATE TABLE my_colors (color rainbow, note text);
5498
5499CREATE FUNCTION get_color_note (rainbow) RETURNS text AS
5500  'SELECT note FROM my_colors WHERE color = $1'
5501  LANGUAGE SQL;
5502</programlisting>
5503
5504   (See <xref linkend="xfunc-sql"/> for an explanation of SQL-language
5505   functions.)  <productname>PostgreSQL</productname> will be aware that
5506   the <function>get_color_note</function> function depends on the <type>rainbow</type>
5507   type: dropping the type would force dropping the function, because its
5508   argument type would no longer be defined.  But <productname>PostgreSQL</productname>
5509   will not consider <function>get_color_note</function> to depend on
5510   the <structname>my_colors</structname> table, and so will not drop the function if
5511   the table is dropped.  While there are disadvantages to this approach,
5512   there are also benefits.  The function is still valid in some sense if the
5513   table is missing, though executing it would cause an error; creating a new
5514   table of the same name would allow the function to work again.
5515  </para>
5516
5517  <para>
5518   On the other hand, for an SQL-language function or procedure whose body
5519   is written in SQL-standard style, the body is parsed at function
5520   definition time and all dependencies recognized by the parser are
5521   stored.  Thus, if we write the function above as
5522
5523<programlisting>
5524CREATE FUNCTION get_color_note (rainbow) RETURNS text
5525BEGIN ATOMIC
5526  SELECT note FROM my_colors WHERE color = $1;
5527END;
5528</programlisting>
5529
5530   then the function's dependency on the <structname>my_colors</structname>
5531   table will be known and enforced by <command>DROP</command>.
5532  </para>
