3<sect1 id="xindex">
4 <title>Interfacing Extensions to Indexes</title>
5
6 <indexterm zone="xindex">
7  <primary>index</primary>
8  <secondary>for user-defined data type</secondary>
9 </indexterm>
10
11  <para>
12   The procedures described thus far let you define new types, new
13   functions, and new operators. However, we cannot yet define an
14   index on a column of a new data type.  To do this, we must define an
15   <firstterm>operator class</firstterm> for the new data type.  Later in this
16   section, we will illustrate this concept in an example: a new
17   operator class for the B-tree index method that stores and sorts
18   complex numbers in ascending absolute value order.
19  </para>
20
21  <para>
22   Operator classes can be grouped into <firstterm>operator families</firstterm>
23   to show the relationships between semantically compatible classes.
24   When only a single data type is involved, an operator class is sufficient,
25   so we'll focus on that case first and then return to operator families.
26  </para>
27
28 <sect2 id="xindex-opclass">
29  <title>Index Methods and Operator Classes</title>
30
31  <para>
32   Operator classes are associated with an index access method, such
33   as <link linkend="btree">B-Tree</link>
34   or <link linkend="gin">GIN</link>. Custom index access method may be
35   defined with <xref linkend="sql-create-access-method"/>. See
36   <xref linkend="indexam"/> for details.
37  </para>
38
39  <para>
40   The routines for an index method do not directly know anything
41   about the data types that the index method will operate on.
42   Instead, an <firstterm>operator
43   class</firstterm><indexterm><primary>operator class</primary></indexterm>
44   identifies the set of operations that the index method needs to use
45   to work with a particular data type.  Operator classes are so
46   called because one thing they specify is the set of
47   <literal>WHERE</literal>-clause operators that can be used with an index
48   (i.e., can be converted into an index-scan qualification).  An
49   operator class can also specify some <firstterm>support
50   function</firstterm> that are needed by the internal operations of the
51   index method, but do not directly correspond to any
52   <literal>WHERE</literal>-clause operator that can be used with the index.
53  </para>
54
55  <para>
56   It is possible to define multiple operator classes for the same
57   data type and index method.  By doing this, multiple
58   sets of indexing semantics can be defined for a single data type.
59   For example, a B-tree index requires a sort ordering to be defined
60   for each data type it works on.
61   It might be useful for a complex-number data type
62   to have one B-tree operator class that sorts the data by complex
63   absolute value, another that sorts by real part, and so on.
64   Typically, one of the operator classes will be deemed most commonly
65   useful and will be marked as the default operator class for that
66   data type and index method.
67  </para>
68
69  <para>
70   The same operator class name
71   can be used for several different index methods (for example, both B-tree
72   and hash index methods have operator classes named
73   <literal>int4_ops</literal>), but each such class is an independent
74   entity and must be defined separately.
75  </para>
76 </sect2>
77
78 <sect2 id="xindex-strategies">
79  <title>Index Method Strategies</title>
80
81  <para>
82   The operators associated with an operator class are identified by
83   <quote>strategy numbers</quote>, which serve to identify the semantics of
84   each operator within the context of its operator class.
85   For example, B-trees impose a strict ordering on keys, lesser to greater,
86   and so operators like <quote>less than</quote> and <quote>greater than or equal
87   to</quote> are interesting with respect to a B-tree.
88   Because
89   <productname>PostgreSQL</productname> allows the user to define operators,
90   <productname>PostgreSQL</productname> cannot look at the name of an operator
91   (e.g., <literal>&lt;</literal> or <literal>&gt;=</literal>) and tell what kind of
92   comparison it is.  Instead, the index method defines a set of
93   <quote>strategies</quote>, which can be thought of as generalized operators.
94   Each operator class specifies which actual operator corresponds to each
95   strategy for a particular data type and interpretation of the index
96   semantics.
97  </para>
98

100   The B-tree index method defines five strategies, shown in <xref
101   linkend="xindex-btree-strat-table"/>.
102  </para>

376 <sect2 id="xindex-support">
377  <title>Index Method Support Routines</title>
378
379  <para>
380   Strategies aren't usually enough information for the system to figure
381   out how to use an index.  In practice, the index methods require
382   additional support routines in order to work. For example, the B-tree
383   index method must be able to compare two keys and determine whether one
384   is greater than, equal to, or less than the other.  Similarly, the
385   hash index method must be able to compute hash codes for key values.
386   These operations do not correspond to operators used in qualifications in
387   SQL commands;  they are administrative routines used by
388   the index methods, internally.
389  </para>
390
391  <para>
392   Just as with strategies, the operator class identifies which specific
393   functions should play each of these roles for a given data type and
394   semantic interpretation.  The index method defines the set
395   of functions it needs, and the operator class identifies the correct
396   functions to use by assigning them to the <quote>support function numbers</quote>
397   specified by the index method.
398  </para>
399
400  <para>
401   Additionally, some opclasses allow users to specify parameters which
402   control their behavior.  Each builtin index access method has an optional
403   <function>options</function> support function, which defines a set of
404   opclass-specific parameters.
405  </para>
406
407  <para>
408   B-trees require a comparison support function,
409   and allow four additional support functions to be
410   supplied at the operator class author's option, as shown in <xref
411   linkend="xindex-btree-support-table"/>.
412   The requirements for these support functions are explained further in
413   <xref linkend="btree-support-funcs"/>.
414  </para>

814 <sect2 id="xindex-example">
815  <title>An Example</title>
816
817  <para>
818   Now that we have seen the ideas, here is the promised example of
819   creating a new operator class.
820   (You can find a working copy of this example in
821   <filename>src/tutorial/complex.c</filename> and
822   <filename>src/tutorial/complex.sql</filename> in the source
823   distribution.)
824   The operator class encapsulates
825   operators that sort complex numbers in absolute value order, so we
826   choose the name <literal>complex_abs_ops</literal>.  First, we need
827   a set of operators.  The procedure for defining operators was
828   discussed in <xref linkend="xoper"/>.  For an operator class on
829   B-trees, the operators we require are:
830
831   <itemizedlist spacing="compact">
832    <listitem><simpara>absolute-value less-than (strategy 1)</simpara></listitem>
833    <listitem><simpara>absolute-value less-than-or-equal (strategy 2)</simpara></listitem>
834    <listitem><simpara>absolute-value equal (strategy 3)</simpara></listitem>
835    <listitem><simpara>absolute-value greater-than-or-equal (strategy 4)</simpara></listitem>
836    <listitem><simpara>absolute-value greater-than (strategy 5)</simpara></listitem>
837   </itemizedlist>
838  </para>
839
840  <para>
841   The least error-prone way to define a related set of comparison operators
842   is to write the B-tree comparison support function first, and then write the
843   other functions as one-line wrappers around the support function.  This
844   reduces the odds of getting inconsistent results for corner cases.
845   Following this approach, we first write:
846
847<programlisting><![CDATA[
848#define Mag(c)  ((c)->x*(c)->x + (c)->y*(c)->y)
849
850static int
851complex_abs_cmp_internal(Complex *a, Complex *b)
852{
853    double      amag = Mag(a),
854                bmag = Mag(b);
855
856    if (amag < bmag)
857        return -1;
858    if (amag > bmag)
859        return 1;
860    return 0;
861}
862]]>
863</programlisting>
864
865   Now the less-than function looks like:
866
867<programlisting><![CDATA[
868PG_FUNCTION_INFO_V1(complex_abs_lt);
869
870Datum
871complex_abs_lt(PG_FUNCTION_ARGS)
872{
873    Complex    *a = (Complex *) PG_GETARG_POINTER(0);
874    Complex    *b = (Complex *) PG_GETARG_POINTER(1);
875
876    PG_RETURN_BOOL(complex_abs_cmp_internal(a, b) < 0);
877}
878]]>
879</programlisting>
880
881   The other four functions differ only in how they compare the internal
882   function's result to zero.
883  </para>
884
885  <para>
886   Next we declare the functions and the operators based on the functions
887   to SQL:
888
889<programlisting>
890CREATE FUNCTION complex_abs_lt(complex, complex) RETURNS bool
891    AS '<replaceable>filename</replaceable>', 'complex_abs_lt'
892    LANGUAGE C IMMUTABLE STRICT;
893
894CREATE OPERATOR &lt; (
895   leftarg = complex, rightarg = complex, procedure = complex_abs_lt,
896   commutator = &gt; , negator = &gt;= ,
897   restrict = scalarltsel, join = scalarltjoinsel
898);
899</programlisting>
900   It is important to specify the correct commutator and negator operators,
901   as well as suitable restriction and join selectivity
902   functions, otherwise the optimizer will be unable to make effective
903   use of the index.
904  </para>
905
906  <para>
907   Other things worth noting are happening here:
908
909  <itemizedlist>
910   <listitem>
911    <para>
912     There can only be one operator named, say, <literal>=</literal>
913     and taking type <type>complex</type> for both operands.  In this
914     case we don't have any other operator <literal>=</literal> for
915     <type>complex</type>, but if we were building a practical data
916     type we'd probably want <literal>=</literal> to be the ordinary
917     equality operation for complex numbers (and not the equality of
918     the absolute values).  In that case, we'd need to use some other
919     operator name for <function>complex_abs_eq</function>.
920    </para>
921   </listitem>
922
923   <listitem>
924    <para>
925     Although <productname>PostgreSQL</productname> can cope with
926     functions having the same SQL name as long as they have different
927     argument data types, C can only cope with one global function
928     having a given name.  So we shouldn't name the C function
929     something simple like <filename>abs_eq</filename>.  Usually it's
930     a good practice to include the data type name in the C function
931     name, so as not to conflict with functions for other data types.
932    </para>
933   </listitem>
934
935   <listitem>
936    <para>
937     We could have made the SQL name
938     of the function <filename>abs_eq</filename>, relying on
939     <productname>PostgreSQL</productname> to distinguish it by
940     argument data types from any other SQL function of the same name.
941     To keep the example simple, we make the function have the same
942     names at the C level and SQL level.
943    </para>
944   </listitem>
945  </itemizedlist>
946  </para>
947
948  <para>
949   The next step is the registration of the support routine required
950   by B-trees.  The example C code that implements this is in the same
951   file that contains the operator functions.  This is how we declare
952   the function:
953
954<programlisting>
955CREATE FUNCTION complex_abs_cmp(complex, complex)
956    RETURNS integer
957    AS '<replaceable>filename</replaceable>'
958    LANGUAGE C IMMUTABLE STRICT;
959</programlisting>
960  </para>
961
962  <para>
963   Now that we have the required operators and support routine,
964   we can finally create the operator class:
965
966<programlisting><![CDATA[
967CREATE OPERATOR CLASS complex_abs_ops
968    DEFAULT FOR TYPE complex USING btree AS
969        OPERATOR        1       < ,
970        OPERATOR        2       <= ,
971        OPERATOR        3       = ,
972        OPERATOR        4       >= ,
973        OPERATOR        5       > ,
974        FUNCTION        1       complex_abs_cmp(complex, complex);
975]]>
976</programlisting>
977  </para>
978
979  <para>
980   And we're done!  It should now be possible to create
981   and use B-tree indexes on <type>complex</type> columns.
982  </para>
983
984  <para>
985   We could have written the operator entries more verbosely, as in:
986<programlisting>
987        OPERATOR        1       &lt; (complex, complex) ,
988</programlisting>
989   but there is no need to do so when the operators take the same data type
990   we are defining the operator class for.
991  </para>
992
993  <para>
994   The above example assumes that you want to make this new operator class the
995   default B-tree operator class for the <type>complex</type> data type.
996   If you don't, just leave out the word <literal>DEFAULT</literal>.
997  </para>
998 </sect2>

1000 <sect2 id="xindex-opfamily">
1001  <title>Operator Classes and Operator Families</title>
1002
1003  <para>
1004   So far we have implicitly assumed that an operator class deals with
1005   only one data type.  While there certainly can be only one data type in
1006   a particular index column, it is often useful to index operations that
1007   compare an indexed column to a value of a different data type.  Also,
1008   if there is use for a cross-data-type operator in connection with an
1009   operator class, it is often the case that the other data type has a
1010   related operator class of its own.  It is helpful to make the connections
1011   between related classes explicit, because this can aid the planner in
1012   optimizing SQL queries (particularly for B-tree operator classes, since
1013   the planner contains a great deal of knowledge about how to work with them).
1014  </para>
1015
1016  <para>
1017   To handle these needs, <productname>PostgreSQL</productname>
1018   uses the concept of an <firstterm>operator
1019   family</firstterm><indexterm><primary>operator family</primary></indexterm>.
1020   An operator family contains one or more operator classes, and can also
1021   contain indexable operators and corresponding support functions that
1022   belong to the family as a whole but not to any single class within the
1023   family.  We say that such operators and functions are <quote>loose</quote>
1024   within the family, as opposed to being bound into a specific class.
1025   Typically each operator class contains single-data-type operators
1026   while cross-data-type operators are loose in the family.
1027  </para>
1028
1029  <para>
1030   All the operators and functions in an operator family must have compatible
1031   semantics, where the compatibility requirements are set by the index
1032   method.  You might therefore wonder why bother to single out particular
1033   subsets of the family as operator classes; and indeed for many purposes
1034   the class divisions are irrelevant and the family is the only interesting
1035   grouping.  The reason for defining operator classes is that they specify
1036   how much of the family is needed to support any particular index.
1037   If there is an index using an operator class, then that operator class
1038   cannot be dropped without dropping the index &mdash; but other parts of
1039   the operator family, namely other operator classes and loose operators,
1040   could be dropped.  Thus, an operator class should be specified to contain
1041   the minimum set of operators and functions that are reasonably needed
1042   to work with an index on a specific data type, and then related but
1043   non-essential operators can be added as loose members of the operator
1044   family.
1045  </para>
1046
1047  <para>
1048   As an example, <productname>PostgreSQL</productname> has a built-in
1049   B-tree operator family <literal>integer_ops</literal>, which includes operator
1050   classes <literal>int8_ops</literal>, <literal>int4_ops</literal>, and
1051   <literal>int2_ops</literal> for indexes on <type>bigint</type> (<type>int8</type>),
1052   <type>integer</type> (<type>int4</type>), and <type>smallint</type> (<type>int2</type>)
1053   columns respectively.  The family also contains cross-data-type comparison
1054   operators allowing any two of these types to be compared, so that an index
1055   on one of these types can be searched using a comparison value of another
1056   type.  The family could be duplicated by these definitions:
1057

1161   support function numbers: each number occurs multiple times within the
1162   family.  This is allowed so long as each instance of a
1163   particular number has distinct input data types.  The instances that have
1164   both input types equal to an operator class's input type are the
1165   primary operators and support functions for that operator class,
1166   and in most cases should be declared as part of the operator class rather
1167   than as loose members of the family.
1168  </para>
1169
1170  <para>
1171   In a B-tree operator family, all the operators in the family must sort
1172   compatibly, as is specified in detail in <xref linkend="btree-behavior"/>.
1173   For each
1174   operator in the family there must be a support function having the same
1175   two input data types as the operator.  It is recommended that a family be
1176   complete, i.e., for each combination of data types, all operators are
1177   included.  Each operator class should include just the non-cross-type
1178   operators and support function for its data type.
1179  </para>
1180
1181  <para>
1182   To build a multiple-data-type hash operator family, compatible hash
1183   support functions must be created for each data type supported by the
1184   family.  Here compatibility means that the functions are guaranteed to
1185   return the same hash code for any two values that are considered equal
1186   by the family's equality operators, even when the values are of different
1187   types.  This is usually difficult to accomplish when the types have
1188   different physical representations, but it can be done in some cases.
1189   Furthermore, casting a value from one data type represented in the operator
1190   family to another data type also represented in the operator family via
1191   an implicit or binary coercion cast must not change the computed hash value.
1192   Notice that there is only one support function per data type, not one
1193   per equality operator.  It is recommended that a family be complete, i.e.,
1194   provide an equality operator for each combination of data types.
1195   Each operator class should include just the non-cross-type equality
1196   operator and the support function for its data type.
1197  </para>
1198
1199  <para>
1200   GiST, SP-GiST, and GIN indexes do not have any explicit notion of
1201   cross-data-type operations.  The set of operators supported is just
1202   whatever the primary support functions for a given operator class can
1203   handle.
1204  </para>
1205
1206  <para>
1207   In BRIN, the requirements depends on the framework that provides the
1208   operator classes.  For operator classes based on <literal>minmax</literal>,
1209   the behavior required is the same as for B-tree operator families:
1210   all the operators in the family must sort compatibly, and casts must
1211   not change the associated sort ordering.
1212  </para>
1213
1214  <note>
1215   <para>
1216    Prior to <productname>PostgreSQL</productname> 8.3, there was no concept
1217    of operator families, and so any cross-data-type operators intended to be
1218    used with an index had to be bound directly into the index's operator
1219    class.  While this approach still works, it is deprecated because it
1220    makes an index's dependencies too broad, and because the planner can
1221    handle cross-data-type comparisons more effectively when both data types
1222    have operators in the same operator family.
1223   </para>
1224  </note>
1225 </sect2>

1227 <sect2 id="xindex-opclass-dependencies">
1228  <title>System Dependencies on Operator Classes</title>
1229
1230   <indexterm>
1231    <primary>ordering operator</primary>
1232   </indexterm>
1233
1234  <para>
1235   <productname>PostgreSQL</productname> uses operator classes to infer the
1236   properties of operators in more ways than just whether they can be used
1237   with indexes.  Therefore, you might want to create operator classes
1238   even if you have no intention of indexing any columns of your data type.
1239  </para>
1240
1241  <para>
1242   In particular, there are SQL features such as <literal>ORDER BY</literal> and
1243   <literal>DISTINCT</literal> that require comparison and sorting of values.
1244   To implement these features on a user-defined data type,
1245   <productname>PostgreSQL</productname> looks for the default B-tree operator
1246   class for the data type.  The <quote>equals</quote> member of this operator
1247   class defines the system's notion of equality of values for
1248   <literal>GROUP BY</literal> and <literal>DISTINCT</literal>, and the sort ordering
1249   imposed by the operator class defines the default <literal>ORDER BY</literal>
1250   ordering.
1251  </para>
1252
1253  <para>
1254   If there is no default B-tree operator class for a data type, the system
1255   will look for a default hash operator class.  But since that kind of
1256   operator class only provides equality, it is only able to support grouping
1257   not sorting.
1258  </para>
1259
1260  <para>
1261   When there is no default operator class for a data type, you will get
1262   errors like <quote>could not identify an ordering operator</quote> if you
1263   try to use these SQL features with the data type.
1264  </para>
1265
1266   <note>
1267    <para>
1268     In <productname>PostgreSQL</productname> versions before 7.4,
1269     sorting and grouping operations would implicitly use operators named
1270     <literal>=</literal>, <literal>&lt;</literal>, and <literal>&gt;</literal>.  The new
1271     behavior of relying on default operator classes avoids having to make
1272     any assumption about the behavior of operators with particular names.
1273    </para>
1274   </note>
1275
1276  <para>
1277   Sorting by a non-default B-tree operator class is possible by specifying
1278   the class's less-than operator in a <literal>USING</literal> option,
1279   for example
1280<programlisting>
1281SELECT * FROM mytable ORDER BY somecol USING ~&lt;~;
1282</programlisting>
1283   Alternatively, specifying the class's greater-than operator
1284   in <literal>USING</literal> selects a descending-order sort.
1285  </para>
1286
1287  <para>
1288   Comparison of arrays of a user-defined type also relies on the semantics
1289   defined by the type's default B-tree operator class.  If there is no
1290   default B-tree operator class, but there is a default hash operator class,
1291   then array equality is supported, but not ordering comparisons.
1292  </para>
1293
1294  <para>
1295   Another SQL feature that requires even more data-type-specific knowledge
1296   is the <literal>RANGE</literal> <replaceable>offset</replaceable>
1297   <literal>PRECEDING</literal>/<literal>FOLLOWING</literal> framing option
1298   for window functions (see <xref linkend="syntax-window-functions"/>).
1299   For a query such as
1300<programlisting>
1301SELECT sum(x) OVER (ORDER BY x RANGE BETWEEN 5 PRECEDING AND 10 FOLLOWING)
1302  FROM mytable;
1303</programlisting>
1304   it is not sufficient to know how to order by <literal>x</literal>;
1305   the database must also understand how to <quote>subtract 5</quote> or
1306   <quote>add 10</quote> to the current row's value of <literal>x</literal>
1307   to identify the bounds of the current window frame.  Comparing the
1308   resulting bounds to other rows' values of <literal>x</literal> is
1309   possible using the comparison operators provided by the B-tree operator
1310   class that defines the <literal>ORDER BY</literal> ordering &mdash; but
1311   addition and subtraction operators are not part of the operator class, so
1312   which ones should be used?  Hard-wiring that choice would be undesirable,
1313   because different sort orders (different B-tree operator classes) might
1314   need different behavior.  Therefore, a B-tree operator class can specify
1315   an <firstterm>in_range</firstterm> support function that encapsulates the
1316   addition and subtraction behaviors that make sense for its sort order.
1317   It can even provide more than one in_range support function, in case
1318   there is more than one data type that makes sense to use as the offset
1319   in <literal>RANGE</literal> clauses.
1320   If the B-tree operator class associated with the window's <literal>ORDER
1321   BY</literal> clause does not have a matching in_range support function,
1322   the <literal>RANGE</literal> <replaceable>offset</replaceable>
1323   <literal>PRECEDING</literal>/<literal>FOLLOWING</literal>
1324   option is not supported.
1325  </para>
1326
1327  <para>
1328   Another important point is that an equality operator that
1329   appears in a hash operator family is a candidate for hash joins,
1330   hash aggregation, and related optimizations.  The hash operator family
1331   is essential here since it identifies the hash function(s) to use.
1332  </para>
1333 </sect2>

1335 <sect2 id="xindex-ordering-ops">
1336  <title>Ordering Operators</title>
1337
1338  <para>
1339   Some index access methods (currently, only GiST and SP-GiST) support the concept of
1340   <firstterm>ordering operators</firstterm>.  What we have been discussing so far
1341   are <firstterm>search operators</firstterm>.  A search operator is one for which
1342   the index can be searched to find all rows satisfying
1343   <literal>WHERE</literal>
1344   <replaceable>indexed_column</replaceable>
1345   <replaceable>operator</replaceable>
1346   <replaceable>constant</replaceable>.
1347   Note that nothing is promised about the order in which the matching rows
1348   will be returned.  In contrast, an ordering operator does not restrict the
1349   set of rows that can be returned, but instead determines their order.
1350   An ordering operator is one for which the index can be scanned to return
1351   rows in the order represented by
1352   <literal>ORDER BY</literal>
1353   <replaceable>indexed_column</replaceable>
1354   <replaceable>operator</replaceable>
1355   <replaceable>constant</replaceable>.
1356   The reason for defining ordering operators that way is that it supports
1357   nearest-neighbor searches, if the operator is one that measures distance.
1358   For example, a query like
1359<programlisting><![CDATA[
1360SELECT * FROM places ORDER BY location <-> point '(101,456)' LIMIT 10;
1361]]>
1362</programlisting>
1363   finds the ten places closest to a given target point.  A GiST index
1364   on the location column can do this efficiently because
1365   <literal>&lt;-&gt;</literal> is an ordering operator.
1366  </para>
1367
1368  <para>
1369   While search operators have to return Boolean results, ordering operators
1370   usually return some other type, such as float or numeric for distances.
1371   This type is normally not the same as the data type being indexed.
1372   To avoid hard-wiring assumptions about the behavior of different data
1373   types, the definition of an ordering operator is required to name
1374   a B-tree operator family that specifies the sort ordering of the result
1375   data type.  As was stated in the previous section, B-tree operator families
1376   define <productname>PostgreSQL</productname>'s notion of ordering, so
1377   this is a natural representation.  Since the point <literal>&lt;-&gt;</literal>
1378   operator returns <type>float8</type>, it could be specified in an operator
1379   class creation command like this:
1380<programlisting><![CDATA[
1381OPERATOR 15    <-> (point, point) FOR ORDER BY float_ops
1382]]>
1383</programlisting>
1384   where <literal>float_ops</literal> is the built-in operator family that includes
1385   operations on <type>float8</type>.  This declaration states that the index
1386   is able to return rows in order of increasing values of the
1387   <literal>&lt;-&gt;</literal> operator.
1388  </para>
1389 </sect2>

1391 <sect2 id="xindex-opclass-features">
1392  <title>Special Features of Operator Classes</title>
1393
1394  <para>
1395   There are two special features of operator classes that we have
1396   not discussed yet, mainly because they are not useful
1397   with the most commonly used index methods.
1398  </para>
1399
1400  <para>
1401   Normally, declaring an operator as a member of an operator class
1402   (or family) means that the index method can retrieve exactly the set of rows
1403   that satisfy a <literal>WHERE</literal> condition using the operator.  For example:
1404<programlisting>
1405SELECT * FROM table WHERE integer_column &lt; 4;
1406</programlisting>
1407   can be satisfied exactly by a B-tree index on the integer column.
1408   But there are cases where an index is useful as an inexact guide to
1409   the matching rows.  For example, if a GiST index stores only bounding boxes
1410   for geometric objects, then it cannot exactly satisfy a <literal>WHERE</literal>
1411   condition that tests overlap between nonrectangular objects such as
1412   polygons.  Yet we could use the index to find objects whose bounding
1413   box overlaps the bounding box of the target object, and then do the
1414   exact overlap test only on the objects found by the index.  If this
1415   scenario applies, the index is said to be <quote>lossy</quote> for the
1416   operator.  Lossy index searches are implemented by having the index
1417   method return a <firstterm>recheck</firstterm> flag when a row might or might
1418   not really satisfy the query condition.  The core system will then
1419   test the original query condition on the retrieved row to see whether
1420   it should be returned as a valid match.  This approach works if
1421   the index is guaranteed to return all the required rows, plus perhaps
1422   some additional rows, which can be eliminated by performing the original
1423   operator invocation.  The index methods that support lossy searches
1424   (currently, GiST, SP-GiST and GIN) allow the support functions of individual
1425   operator classes to set the recheck flag, and so this is essentially an
1426   operator-class feature.
1427  </para>
1428
1429  <para>
1430   Consider again the situation where we are storing in the index only
1431   the bounding box of a complex object such as a polygon.  In this
1432   case there's not much value in storing the whole polygon in the index
1433   entry &mdash; we might as well store just a simpler object of type
1434   <type>box</type>.  This situation is expressed by the <literal>STORAGE</literal>
1435   option in <command>CREATE OPERATOR CLASS</command>: we'd write something like:
1436
1437<programlisting>
1438CREATE OPERATOR CLASS polygon_ops
1439    DEFAULT FOR TYPE polygon USING gist AS
1440        ...
1441        STORAGE box;
1442</programlisting>
1443
1444   At present, only the GiST, SP-GiST, GIN and BRIN index methods support a
1445   <literal>STORAGE</literal> type that's different from the column data type.
1446   The GiST <function>compress</function> and <function>decompress</function> support
1447   routines must deal with data-type conversion when <literal>STORAGE</literal>
1448   is used.  SP-GiST likewise requires a <function>compress</function>
1449   support function to convert to the storage type, when that is different;
1450   if an SP-GiST opclass also supports retrieving data, the reverse
1451   conversion must be handled by the <function>consistent</function> function.
1452   In GIN, the <literal>STORAGE</literal> type identifies the type of
1453   the <quote>key</quote> values, which normally is different from the type
1454   of the indexed column &mdash; for example, an operator class for
1455   integer-array columns might have keys that are just integers.  The
1456   GIN <function>extractValue</function> and <function>extractQuery</function> support
1457   routines are responsible for extracting keys from indexed values.
1458   BRIN is similar to GIN: the <literal>STORAGE</literal> type identifies the
1459   type of the stored summary values, and operator classes' support
1460   procedures are responsible for interpreting the summary values
1461   correctly.
1462  </para>
1463 </sect2>
