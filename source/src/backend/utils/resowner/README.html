3Notes About Resource Owners
4===========================
5
6ResourceOwner objects are a concept invented to simplify management of
7query-related resources, such as buffer pins and table locks.  These
8resources need to be tracked in a reliable way to ensure that they will
9be released at query end, even if the query fails due to an error.
10Rather than expecting the entire executor to have bulletproof data
11structures, we localize the tracking of such resources into a single
12module.
13
14The design of the ResourceOwner API is modeled on our MemoryContext API,
15which has proven very flexible and successful in preventing memory leaks.
16In particular we allow ResourceOwners to have child ResourceOwner objects
17so that there can be forests of the things; releasing a parent
18ResourceOwner acts on all its direct and indirect children as well.
19
20(It is tempting to consider unifying ResourceOwners and MemoryContexts
21into a single object type, but their usage patterns are sufficiently
22different that this is probably not really a helpful thing to do.)
23
24We create a ResourceOwner for each transaction or subtransaction as
25well as one for each Portal.  During execution of a Portal, the global
26variable CurrentResourceOwner points to the Portal's ResourceOwner.
27This causes operations such as ReadBuffer and LockAcquire to record
28ownership of the acquired resources in that ResourceOwner object.
29
30When a Portal is closed, any remaining resources (typically only locks)
31become the responsibility of the current transaction.  This is represented
32by making the Portal's ResourceOwner a child of the current transaction's
33ResourceOwner.  resowner.c automatically transfers the resources to the
34parent object when releasing the child.  Similarly, subtransaction
35ResourceOwners are children of their immediate parent.
36
37We need transaction-related ResourceOwners as well as Portal-related ones
38because transactions may initiate operations that require resources (such
39as query parsing) when no associated Portal exists yet.
40

42Usage
43-----
44
45The basic operations on a ResourceOwner are:
46
47* create a ResourceOwner
48
49* associate or deassociate some resource with a ResourceOwner
50
51* release a ResourceOwner's assets (free all owned resources, but not the
52  owner object itself)
53
54* delete a ResourceOwner (including child owner objects); all resources
55  must have been released beforehand
56
57Locks are handled specially because in non-error situations a lock should
58be held until end of transaction, even if it was originally taken by a
59subtransaction or portal.  Therefore, the "release" operation on a child
60ResourceOwner transfers lock ownership to the parent instead of actually
61releasing the lock, if isCommit is true.
62
63Whenever we are inside a transaction, the global variable
64CurrentResourceOwner shows which resource owner should be assigned
65ownership of acquired resources.  Note however that CurrentResourceOwner
66is NULL when not inside any transaction (or when inside a failed
67transaction).  In this case it is not valid to acquire query-lifespan
68resources.
69
70When unpinning a buffer or releasing a lock or cache reference,
71CurrentResourceOwner must point to the same resource owner that was current
72when the buffer, lock, or cache reference was acquired.  It would be possible
73to relax this restriction given additional bookkeeping effort, but at present
74there seems no need.
75
76Adding a new resource type
77--------------------------
78
79ResourceOwner can track ownership of many different kinds of resources.  In
80core PostgreSQL it is used for buffer pins, lmgr locks, and catalog cache
81references, to name a few examples.
82
83To add a new kind of resource, define a ResourceOwnerDesc to describe it.
84For example:
85
86static const ResourceOwnerDesc myresource_desc = {
87	.name = "My fancy resource",
88	.release_phase = RESOURCE_RELEASE_AFTER_LOCKS,
89	.release_priority = RELEASE_PRIO_FIRST,
90	.ReleaseResource = ReleaseMyResource,
91	.DebugPrint = PrintMyResource
92};
93
94ResourceOwnerRemember() and ResourceOwnerForget() functions take a pointer
95to that struct, along with a Datum to represent the resource.  The meaning
96of the Datum depends on the resource type.  Most resource types use it to
97store a pointer to some struct, but it can also be a file descriptor or
98library handle, for example.
99
100The ReleaseResource callback is called when a resource owner is released or
101deleted.  It should release any resources (e.g. close files, free memory)
102associated with the resource.  Because the callback is called during
103transaction abort, it must perform only low-level cleanup with no user
104visible effects.  The callback should not perform operations that could
105fail, like allocate memory.
106
107The optional DebugPrint callback is used in the warning at transaction
108commit, if any resources are leaked.  If not specified, a generic
109implementation that prints the resource name and the resource as a pointer
110is used.
111
112There is another API for other modules to get control during ResourceOwner
113release, so that they can scan their own data structures to find the objects
114that need to be deleted.  See RegisterResourceReleaseCallback function.
115This used to be the only way for extensions to use the resource owner
116mechanism with new kinds of objects; nowadays it is easier to define a custom
117ResourceOwnerDesc struct.
118

120Releasing
121---------
122
123Releasing the resources of a ResourceOwner happens in three phases:
124
1251. "Before-locks" resources
126
1272. Locks
128
1293. "After-locks" resources
130
131Each resource type specifies whether it needs to be released before or after
132locks.  Each resource type also has a priority, which determines the order
133that the resources are released in.  Note that the phases are performed fully
134for the whole tree of resource owners, before moving to the next phase, but
135the priority within each phase only determines the order within that
136ResourceOwner.  Child resource owners are always handled before the parent,
137within each phase.
138
139For example, imagine that you have two ResourceOwners, parent and child,
140as follows:
141
142Parent
143  parent resource BEFORE_LOCKS priority 1
144  parent resource BEFORE_LOCKS priority 2
145  parent resource AFTER_LOCKS priority 10001
146  parent resource AFTER_LOCKS priority 10002
147  Child
148    child resource BEFORE_LOCKS priority 1
149    child resource BEFORE_LOCKS priority 2
150    child resource AFTER_LOCKS priority 10001
151    child resource AFTER_LOCKS priority 10002
152
153These resources would be released in the following order:
154
155child resource BEFORE_LOCKS priority 1
156child resource BEFORE_LOCKS priority 2
157parent resource BEFORE_LOCKS priority 1
158parent resource BEFORE_LOCKS priority 2
159(locks)
160child resource AFTER_LOCKS priority 10001
161child resource AFTER_LOCKS priority 10002
162parent resource AFTER_LOCKS priority 10001
163parent resource AFTER_LOCKS priority 10002
164
165To release all the resources, you need to call ResourceOwnerRelease() three
166times, once for each phase. You may perform additional tasks between the
167phases, but after the first call to ResourceOwnerRelease(), you cannot use
168the ResourceOwner to remember any more resources. You also cannot call
169ResourceOwnerForget on the resource owner to release any previously
170remembered resources "in retail", after you have started the release process.
171
172Normally, you are expected to call ResourceOwnerForget on every resource so
173that at commit, the ResourceOwner is empty (locks are an exception). If there
174are any resources still held at commit, ResourceOwnerRelease will print a
175WARNING on each such resource. At abort, however, we truly rely on the
176ResourceOwner mechanism and it is normal that there are resources to be
177released.
