3GiST Indexing
4=============
5
6This directory contains an implementation of GiST indexing for Postgres.
7
8GiST stands for Generalized Search Tree. It was introduced in the seminal paper
9"Generalized Search Trees for Database Systems", 1995, Joseph M. Hellerstein,
10Jeffrey F. Naughton, Avi Pfeffer:
11
12    http://www.sai.msu.su/~megera/postgres/gist/papers/gist.ps
13    https://dsf.berkeley.edu/papers/sigmod97-gist.pdf
14
15and implemented by J. Hellerstein and P. Aoki in an early version of
16PostgreSQL (more details are available from The GiST Indexing Project
17at Berkeley at http://gist.cs.berkeley.edu/). As a "university"
18project it had a limited number of features and was in rare use.
19
20The current implementation of GiST supports:
21
22  * Variable length keys
23  * Composite keys (multi-key)
24  * Ordered search (nearest-neighbor search)
25  * provides NULL-safe interface to GiST core
26  * Concurrency
27  * Recovery support via WAL logging
28  * Buffering build algorithm
29  * Sorted build method
30
31The support for concurrency implemented in PostgreSQL was developed based on
32the paper "Access Methods for Next-Generation Database Systems" by
33Marcel Kornacker:
34
35    http://www.sai.msu.su/~megera/postgres/gist/papers/concurrency/access-methods-for-next-generation.pdf.gz
36
37Buffering build algorithm for GiST was developed based on the paper "Efficient
38Bulk Operations on Dynamic R-trees" by Lars Arge, Klaus Hinrichs, Jan Vahrenhold
39and Jeffrey Scott Vitter.
40
41    http://citeseerx.ist.psu.edu/viewdoc/download?doi=10.1.1.135.9894&rep=rep1&type=pdf
42
43The original algorithms were modified in several ways:
44
45* They had to be adapted to PostgreSQL conventions. For example, the SEARCH
46  algorithm was considerably changed, because in PostgreSQL the search function
47  should return one tuple (next), not all tuples at once. Also, it should
48  release page locks between calls.
49* Since we added support for variable length keys, it's not possible to
50  guarantee enough free space for all keys on pages after splitting. User
51  defined function picksplit doesn't have information about size of tuples
52  (each tuple may contain several keys as in multicolumn index while picksplit
53  could work with only one key) and pages.
54* We modified original INSERT algorithm for performance reasons. In particular,
55  it is now a single-pass algorithm.
56* Since the papers were theoretical, some details were omitted and we
57  had to find out ourself how to solve some specific problems.
58
59Because of the above reasons, we have revised the interaction of GiST
60core and PostgreSQL WAL system. Moreover, we encountered (and solved)
61a problem of uncompleted insertions when recovering after crash, which
62was not touched in the paper.
63
64Search Algorithm
65----------------
66
67The search code maintains a queue of unvisited items, where an "item" is
68either a heap tuple known to satisfy the search conditions, or an index
69page that is consistent with the search conditions according to inspection
70of its parent page's downlink item.  Initially the root page is searched
71to find unvisited items in it.  Then we pull items from the queue.  A
72heap tuple pointer is just returned immediately; an index page entry
73causes that page to be searched, generating more queue entries.
74
75The queue is kept ordered with heap tuple items at the front, then
76index page entries, with any newly-added index page entry inserted
77before existing index page entries.  This ensures depth-first traversal
78of the index, and in particular causes the first few heap tuples to be
79returned as soon as possible.  That is helpful in case there is a LIMIT
80that requires only a few tuples to be produced.
81
82To implement nearest-neighbor search, the queue entries are augmented
83with distance data: heap tuple entries are labeled with exact distance
84from the search argument, while index-page entries must be labeled with
85the minimum distance that any of their children could have.  Then,
86queue entries are retrieved in smallest-distance-first order, with
87entries having identical distances managed as stated in the previous
88paragraph.
89
90The search algorithm keeps an index page locked only long enough to scan
91its entries and queue those that satisfy the search conditions.  Since
92insertions can occur concurrently with searches, it is possible for an
93index child page to be split between the time we make a queue entry for it
94(while visiting its parent page) and the time we actually reach and scan
95the child page.  To avoid missing the entries that were moved to the right
96sibling, we detect whether a split has occurred by comparing the child
97page's NSN (node sequence number, a special-purpose LSN) to the LSN that
98the parent had when visited.  If it did, the sibling page is immediately
99added to the front of the queue, ensuring that its items will be scanned
100in the same order as if they were still on the original child page.
101
102As is usual in Postgres, the search algorithm only guarantees to find index
103entries that existed before the scan started; index entries added during
104the scan might or might not be visited.  This is okay as long as all
105searches use MVCC snapshot rules to reject heap tuples newer than the time
106of scan start.  In particular, this means that we need not worry about
107cases where a parent page's downlink key is "enlarged" after we look at it.
108Any such enlargement would be to add child items that we aren't interested
109in returning anyway.

112Insert Algorithm
113----------------
114
115INSERT guarantees that the GiST tree remains balanced. User defined key method
116Penalty is used for choosing a subtree to insert; method PickSplit is used for
117the node splitting algorithm; method Union is used for propagating changes
118upward to maintain the tree properties.
119
120To insert a tuple, we first have to find a suitable leaf page to insert to.
121The algorithm walks down the tree, starting from the root, along the path
122of smallest Penalty. At each step:
123
1241. Has this page been split since we looked at the parent? If so, it's
125possible that we should be inserting to the other half instead, so retreat
126back to the parent.
1272. If this is a leaf node, we've found our target node.
1283. Otherwise use Penalty to pick a new target subtree.
1294. Check the key representing the target subtree. If it doesn't already cover
130the key we're inserting, replace it with the Union of the old downlink key
131and the key being inserted. (Actually, we always call Union, and just skip
132the replacement if the Unioned key is the same as the existing key)
1335. Replacing the key in step 4 might cause the page to be split. In that case,
134propagate the change upwards and restart the algorithm from the first parent
135that didn't need to be split.
1366. Walk down to the target subtree, and goto 1.
137
138This differs from the insertion algorithm in the original paper. In the
139original paper, you first walk down the tree until you reach a leaf page, and
140then you adjust the downlink in the parent, and propagate the adjustment up,
141all the way up to the root in the worst case. But we adjust the downlinks to
142cover the new key already when we walk down, so that when we reach the leaf
143page, we don't need to update the parents anymore, except to insert the
144downlinks if we have to split the page. This makes crash recovery simpler:
145after inserting a key to the page, the tree is immediately self-consistent
146without having to update the parents. Even if we split a page and crash before
147inserting the downlink to the parent, the tree is self-consistent because the
148right half of the split is accessible via the rightlink of the left page
149(which replaced the original page).
150
151Note that the algorithm can walk up and down the tree before reaching a leaf
152page, if internal pages need to split while adjusting the downlinks for the
153new key. Eventually, you should reach the bottom, and proceed with the
154insertion of the new tuple.
155
156Once we've found the target page to insert to, we check if there's room
157for the new tuple. If there is, the tuple is inserted, and we're done.
158If it doesn't fit, however, the page needs to be split. Note that it is
159possible that a page needs to be split into more than two pages, if keys have
160different lengths or more than one key is being inserted at a time (which can
161happen when inserting downlinks for a page split that resulted in more than
162two pages at the lower level). After splitting a page, the parent page needs
163to be updated. The downlink for the new page needs to be inserted, and the
164downlink for the old page, which became the left half of the split, needs to
165be updated to only cover those tuples that stayed on the left page. Inserting
166the downlink in the parent can again lead to a page split, recursing up to the
167root page in the worst case.
168
169gistplacetopage is the workhorse function that performs one step of the
170insertion. If the tuple fits, it inserts it to the given page, otherwise
171it splits the page, and constructs the new downlink tuples for the split
172pages. The caller must then call gistplacetopage() on the parent page to
173insert the downlink tuples. The parent page that holds the downlink to
174the child might have migrated as a result of concurrent splits of the
175parent, gistFindCorrectParent() is used to find the parent page.
176
177Splitting the root page works slightly differently. At root split,
178gistplacetopage() allocates the new child pages and replaces the old root
179page with the new root containing downlinks to the new children, all in one
180operation.
181
182
183findPath is a subroutine of findParent, used when the correct parent page
184can't be found by following the rightlinks at the parent level:
185
186findPath( stack item )
187	push stack, [root, 0, 0] // page, LSN, parent
188	while( stack )
189		ptr = top of stack
190		latch( ptr->page, S-mode )
191		if ( ptr->parent->page->lsn < ptr->page->nsn )
192			push stack, [ ptr->page->rightlink, 0, ptr->parent ]
193		end
194		for( each tuple on page )
195			if ( tuple->pagepointer == item->page )
196				return stack
197			else
198				add to stack at the end [tuple->pagepointer,0, ptr]
199			end
200		end
201		unlatch( ptr->page )
202		pop stack
203	end
204
205
206gistFindCorrectParent is used to re-find the parent of a page during
207insertion. It might have migrated to the right since we traversed down the
208tree because of page splits.
209
210findParent( stack item )
211	parent = item->parent
212	if ( parent->page->lsn != parent->lsn )
213		while(true)
214			search parent tuple on parent->page, if found the return
215			rightlink = parent->page->rightlink
216			unlatch( parent->page )
217			if ( rightlink is incorrect )
218				break loop
219			end
220			parent->page = rightlink
221			latch( parent->page, X-mode )
222		end
223		newstack = findPath( item->parent )
224		replace part of stack to new one
225		latch( parent->page, X-mode )
226		return findParent( item )
227	end
228
229pageSplit function decides how to distribute keys to the new pages after
230page split:
231
232pageSplit(page, allkeys)
233	(lkeys, rkeys) = pickSplit( allkeys )
234	if ( page is root )
235		lpage = new page
236	else
237		lpage = page
238	rpage = new page
239	if ( no space left on rpage )
240		newkeys = pageSplit( rpage, rkeys )
241	else
242		push newkeys, union(rkeys)
243	end
244	if ( no space left on lpage )
245		push newkeys, pageSplit( lpage, lkeys )
246	else
247		push newkeys, union(lkeys)
248	end
249	return newkeys

253Concurrency control
254-------------------
255As a rule of thumb, if you need to hold a lock on multiple pages at the
256same time, the locks should be acquired in the following order: child page
257before parent, and left-to-right at the same level. Always acquiring the
258locks in the same order avoids deadlocks.
259
260The search algorithm only looks at and locks one page at a time. Consequently
261there's a race condition between a search and a page split. A page split
262happens in two phases: 1. The page is split 2. The downlink is inserted to the
263parent. If a search looks at the parent page between those steps, before the
264downlink is inserted, it will still find the new right half by following the
265rightlink on the left half. But it must not follow the rightlink if it saw the
266downlink in the parent, or the page will be visited twice!
267
268A split initially marks the left page with the F_FOLLOW_RIGHT flag. If a scan
269sees that flag set, it knows that the right page is missing the downlink, and
270should be visited too. When split inserts the downlink to the parent, it
271clears the F_FOLLOW_RIGHT flag in the child, and sets the NSN field in the
272child page header to match the LSN of the insertion on the parent. If the
273F_FOLLOW_RIGHT flag is not set, a scan compares the NSN on the child and the
274LSN it saw in the parent. If the child's NSN is greater than the LSN seen on
275the parent, the scan looked at the parent page before the downlink was
276inserted, so it should follow the rightlink. Otherwise the scan saw the
277downlink in the parent page, and will/did follow that as usual.
278
279A scan can't normally see a page with the F_FOLLOW_RIGHT flag set, because
280a page split keeps the child pages locked until the downlink has been inserted
281to the parent and the flag cleared again. But if a crash happens in the middle
282of a page split, before the downlinks are inserted into the parent, that will
283leave a page with F_FOLLOW_RIGHT in the tree. Scans handle that just fine,
284but we'll eventually want to fix that for performance reasons. And more
285importantly, dealing with pages with missing downlink pointers in the parent
286would complicate the insertion algorithm. So when an insertion sees a page
287with F_FOLLOW_RIGHT set, it immediately tries to bring the split that
288crashed in the middle to completion by adding the downlink in the parent.

290Buffering build algorithm
291-------------------------
292
293In the buffering index build algorithm, some or all internal nodes have a
294buffer attached to them. When a tuple is inserted at the top, the descend down
295the tree is stopped as soon as a buffer is reached, and the tuple is pushed to
296the buffer. When a buffer gets too full, all the tuples in it are flushed to
297the lower level, where they again hit lower level buffers or leaf pages. This
298makes the insertions happen in more of a breadth-first than depth-first order,
299which greatly reduces the amount of random I/O required.
300
301In the algorithm, levels are numbered so that leaf pages have level zero,
302and internal node levels count up from 1. This numbering ensures that a page's
303level number never changes, even when the root page is split.
304
305Level                    Tree
306
3073                         *
308                      /       \
3092                *                 *
310              /  |  \           /  |  \
3111          *     *     *     *     *     *
312          / \   / \   / \   / \   / \   / \
3130        o   o o   o o   o o   o o   o o   o
314
315* - internal page
316o - leaf page
317
318Internal pages that belong to certain levels have buffers associated with
319them. Leaf pages never have buffers. Which levels have buffers is controlled
320by "level step" parameter: level numbers that are multiples of level_step
321have buffers, while others do not. For example, if level_step = 2, then
322pages on levels 2, 4, 6, ... have buffers. If level_step = 1 then every
323internal page has a buffer.
324
325Level        Tree (level_step = 1)                Tree (level_step = 2)
326
3273                      *                                     *
328                   /       \                             /       \
3292             *(b)              *(b)                *(b)              *(b)
330           /  |  \           /  |  \             /  |  \           /  |  \
3311       *(b)  *(b)  *(b)  *(b)  *(b)  *(b)    *     *     *     *     *     *
332       / \   / \   / \   / \   / \   / \     / \   / \   / \   / \   / \   / \
3330     o   o o   o o   o o   o o   o o   o   o   o o   o o   o o   o o   o o   o
334
335(b) - buffer
336
337Logically, a buffer is just bunch of tuples. Physically, it is divided in
338pages, backed by a temporary file. Each buffer can be in one of two states:
339a) Last page of the buffer is kept in main memory. A node buffer is
340automatically switched to this state when a new index tuple is added to it,
341or a tuple is removed from it.
342b) All pages of the buffer are swapped out to disk. When a buffer becomes too
343full, and we start to flush it, all other buffers are switched to this state.
344
345When an index tuple is inserted, its initial processing can end in one of the
346following points:
3471) Leaf page, if the depth of the index <= level_step, meaning that
348   none of the internal pages have buffers associated with them.
3492) Buffer of topmost level page that has buffers.
350
351New index tuples are processed until one of the buffers in the topmost
352buffered level becomes half-full. When a buffer becomes half-full, it's added
353to the emptying queue, and will be emptied before a new tuple is processed.
354
355Buffer emptying process means that index tuples from the buffer are moved
356into buffers at a lower level, or leaf pages. First, all the other buffers are
357swapped to disk to free up the memory. Then tuples are popped from the buffer
358one by one, and cascaded down the tree to the next buffer or leaf page below
359the buffered node.
360
361Emptying a buffer has the interesting dynamic property that any intermediate
362pages between the buffer being emptied, and the next buffered or leaf level
363below it, become cached. If there are no more buffers below the node, the leaf
364pages where the tuples finally land on get cached too. If there are, the last
365buffer page of each buffer below is kept in memory. This is illustrated in
366the figures below:
367
368   Buffer being emptied to
369     lower-level buffers               Buffer being emptied to leaf pages
370
371               +(fb)                                 +(fb)
372            /     \                                /     \
373        +             +                        +             +
374      /   \         /   \                    /   \         /   \
375    *(ab)   *(ab) *(ab)   *(ab)            x       x     x       x
376
377+    - cached internal page
378x    - cached leaf page
379*    - non-cached internal page
380(fb) - buffer being emptied
381(ab) - buffers being appended to, with last page in memory
382
383In the beginning of the index build, the level-step is chosen so that all those
384pages involved in emptying one buffer fit in cache, so after each of those
385pages have been accessed once and cached, emptying a buffer doesn't involve
386any more I/O. This locality is where the speedup of the buffering algorithm
387comes from.
388
389Emptying one buffer can fill up one or more of the lower-level buffers,
390triggering emptying of them as well. Whenever a buffer becomes too full, it's
391added to the emptying queue, and will be emptied after the current buffer has
392been processed.
393
394To keep the size of each buffer limited even in the worst case, buffer emptying
395is scheduled as soon as a buffer becomes half-full, and emptying it continues
396until 1/2 of the nominal buffer size worth of tuples has been emptied. This
397guarantees that when buffer emptying begins, all the lower-level buffers
398are at most half-full. In the worst case that all the tuples are cascaded down
399to the same lower-level buffer, that buffer therefore has enough space to
400accommodate all the tuples emptied from the upper-level buffer. There is no
401hard size limit in any of the data structures used, though, so this only needs
402to be approximate; small overfilling of some buffers doesn't matter.
403
404If an internal page that has a buffer associated with it is split, the buffer
405needs to be split too. All tuples in the buffer are scanned through and
406relocated to the correct sibling buffers, using the penalty function to decide
407which buffer each tuple should go to.
408
409After all tuples from the heap have been processed, there are still some index
410tuples in the buffers. At this point, final buffer emptying starts. All buffers
411are emptied in top-down order. This is slightly complicated by the fact that
412new buffers can be allocated during the emptying, due to page splits. However,
413the new buffers will always be siblings of buffers that haven't been fully
414emptied yet; tuples never move upwards in the tree. The final emptying loops
415through buffers at a given level until all buffers at that level have been
416emptied, and then moves down to the next level.

418Sorted build method
419-------------------
420
421Sort all input tuples, pack them into GiST leaf pages in the sorted order,
422and create downlinks and internal pages as we go. This method builds the index
423from the bottom up, similar to how the B-tree index is built.
424
425The sorted method is used if the operator classes for all columns have a
426"sortsupport" defined. Otherwise, we fall back on inserting tuples one by one
427with optional buffering.
428
429Sorting GiST build requires good linearization of the sort opclass. That is not
430always the case in multidimensional data. To tackle the anomalies, we buffer
431index tuples and apply a picksplit function that can be multidimensional-aware.

433Bulk delete algorithm (VACUUM)
434------------------------------
435
436VACUUM works in two stages:
437
438In the first stage, we scan the whole index in physical order. To make sure
439that we don't miss any dead tuples because a concurrent page split moved them,
440we check the F_FOLLOW_RIGHT flags and NSN on each page, to detect if the
441page has been concurrently split. If a concurrent page split is detected, and
442one half of the page was moved to a position that we already scanned, we
443"jump backwards" to scan the page again. This is the same mechanism that
444B-tree VACUUM uses, but because we already have NSNs on pages, to detect page
445splits during searches, we don't need a "vacuum cycle ID" concept for that
446like B-tree does.
447
448While we scan all the pages, we also make note of any completely empty leaf
449pages. We will try to unlink them from the tree after the scan. We also record
450the block numbers of all internal pages; they are needed to locate parents of
451the empty pages while unlinking them.
452
453We try to unlink any empty leaf pages from the tree, so that their space can
454be reused. In order to delete an empty page, its downlink must be removed from
455the parent. We scan all the internal pages, whose block numbers we memorized
456in the first stage, and look for downlinks to pages that we have memorized as
457being empty. Whenever we find one, we acquire a lock on the parent and child
458page, re-check that the child page is still empty. Then, we remove the
459downlink and mark the child as deleted, and release the locks.
460
461The insertion algorithm would get confused, if an internal page was completely
462empty. So we never delete the last child of an internal page, even if it's
463empty. Currently, we only support deleting leaf pages.
464
465This page deletion algorithm works on a best-effort basis. It might fail to
466find a downlink, if a concurrent page split moved it after the first stage.
467In that case, we won't be able to remove all empty pages. That's OK, it's
468not expected to happen very often, and hopefully the next VACUUM will clean
469it up.
470
471When we have deleted a page, it's possible that an in-progress search will
472still descend on the page, if it saw the downlink before we removed it. The
473search will see that it is deleted, and ignore it, but as long as that can
474happen, we cannot reuse the page. To "wait out" any in-progress searches, when
475a page is deleted, it's labeled with the current next-transaction counter
476value. The page is not recycled, until that XID is no longer visible to
477anyone. That's much more conservative than necessary, but let's keep it
478simple.
