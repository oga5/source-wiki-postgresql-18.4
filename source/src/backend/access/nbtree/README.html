3Btree Indexing
4==============
5
6This directory contains a correct implementation of Lehman and Yao's
7high-concurrency B-tree management algorithm (P. Lehman and S. Yao,
8Efficient Locking for Concurrent Operations on B-Trees, ACM Transactions
9on Database Systems, Vol 6, No. 4, December 1981, pp 650-670).  We also
10use a simplified version of the deletion logic described in Lanin and
11Shasha (V. Lanin and D. Shasha, A Symmetric Concurrent B-Tree Algorithm,
12Proceedings of 1986 Fall Joint Computer Conference, pp 380-389).
13
14The basic Lehman & Yao Algorithm
15--------------------------------
16
17Compared to a classic B-tree, L&Y adds a right-link pointer to each page,
18to the page's right sibling.  It also adds a "high key" to each page, which
19is an upper bound on the keys that are allowed on that page.  These two
20additions make it possible to detect a concurrent page split, which allows
21the tree to be searched without holding any read locks (except to keep a
22single page from being modified while reading it).
23
24When a search follows a downlink to a child page, it compares the page's
25high key with the search key.  If the search key is greater than the high
26key, the page must've been split concurrently, and you must follow the
27right-link to find the new page containing the key range you're looking
28for.  This might need to be repeated, if the page has been split more than
29once.
30
31Lehman and Yao talk about alternating "separator" keys and downlinks in
32internal pages rather than tuples or records.  We use the term "pivot"
33tuple to refer to tuples which don't point to heap tuples, that are used
34only for tree navigation.  All tuples on non-leaf pages and high keys on
35leaf pages are pivot tuples.  Since pivot tuples are only used to represent
36which part of the key space belongs on each page, they can have attribute
37values copied from non-pivot tuples that were deleted and killed by VACUUM
38some time ago.  A pivot tuple may contain a "separator" key and downlink,
39just a separator key (i.e. the downlink value is implicitly undefined), or
40just a downlink (i.e. all attributes are truncated away).
41
42The requirement that all btree keys be unique is satisfied by treating heap
43TID as a tiebreaker attribute.  Logical duplicates are sorted in heap TID
44order.  This is necessary because Lehman and Yao also require that the key
45range for a subtree S is described by Ki < v <= Ki+1 where Ki and Ki+1 are
46the adjacent keys in the parent page (Ki must be _strictly_ less than v,
47which is assured by having reliably unique keys).  Keys are always unique
48on their level, with the exception of a leaf page's high key, which can be
49fully equal to the last item on the page.
50
51The Postgres implementation of suffix truncation must make sure that the
52Lehman and Yao invariants hold, and represents that absent/truncated
53attributes in pivot tuples have the sentinel value "minus infinity".  The
54later section on suffix truncation will be helpful if it's unclear how the
55Lehman & Yao invariants work with a real world example.
56
57Differences to the Lehman & Yao algorithm
58-----------------------------------------
59
60We have made the following changes in order to incorporate the L&Y algorithm
61into Postgres:
62
63Lehman and Yao don't require read locks, but assume that in-memory
64copies of tree pages are unshared.  Postgres shares in-memory buffers
65among backends.  As a result, we do page-level read locking on btree
66pages in order to guarantee that no record is modified while we are
67examining it.  This reduces concurrency but guarantees correct
68behavior.
69
70We support the notion of an ordered "scan" of an index as well as
71insertions, deletions, and simple lookups.  A scan in the forward
72direction is no problem, we just use the right-sibling pointers that
73L&Y require anyway.  (Thus, once we have descended the tree to the
74correct start point for the scan, the scan looks only at leaf pages
75and never at higher tree levels.)  To support scans in the backward
76direction, we also store a "left sibling" link much like the "right
77sibling".  (This adds an extra step to the L&Y split algorithm: while
78holding the write lock on the page being split, we also lock its former
79right sibling to update that page's left-link.  This is safe since no
80writer of that page can be interested in acquiring a write lock on our
81page.)  A backwards scan has one additional bit of complexity: after
82following the left-link we must account for the possibility that the
83left sibling page got split before we could read it.  So, we have to
84move right until we find a page whose right-link matches the page we
85came from.  (Actually, it's even harder than that; see page deletion
86discussion below.)
87
88Page read locks are held only for as long as a scan is examining a page.
89To minimize lock/unlock traffic, an index scan always searches a leaf page
90to identify all the matching items at once, copying their heap tuple IDs
91into backend-local storage.  The heap tuple IDs are then processed while
92not holding any page lock within the index.  We do continue to hold a pin
93on the leaf page in some circumstances, to protect against concurrent
94deletions (see below).  In this state the scan is effectively stopped
95"between" pages, either before or after the page it has pinned.  This is
96safe in the presence of concurrent insertions and even page splits, because
97items are never moved across pre-existing page boundaries --- so the scan
98cannot miss any items it should have seen, nor accidentally return the same
99item twice.  The scan must remember the page's right-link at the time it
100was scanned, since that is the page to move right to; if we move right to
101the current right-link then we'd re-scan any items moved by a page split.
102We also remember the left-link, and follow it when the scan moves backwards
103(though this requires extra handling to account for concurrent splits of
104the left sibling; see detailed move-left algorithm below).
105
106In most cases we release our lock and pin on a page before attempting
107to acquire pin and lock on the page we are moving to.  In a few places
108it is necessary to lock the next page before releasing the current one.
109This is safe when moving right or up, but not when moving left or down
110(else we'd create the possibility of deadlocks).
111
112Lehman and Yao fail to discuss what must happen when the root page
113becomes full and must be split.  Our implementation is to split the
114root in the same way that any other page would be split, then construct
115a new root page holding pointers to both of the resulting pages (which
116now become siblings on the next level of the tree).  The new root page
117is then installed by altering the root pointer in the meta-data page (see
118below).  This works because the root is not treated specially in any
119other way --- in particular, searches will move right using its link
120pointer if the link is set.  Therefore, searches will find the data
121that's been moved into the right sibling even if they read the meta-data
122page before it got updated.  This is the same reasoning that makes a
123split of a non-root page safe.  The locking considerations are similar too.
124
125When an inserter recurses up the tree, splitting internal pages to insert
126links to pages inserted on the level below, it is possible that it will
127need to access a page above the level that was the root when it began its
128descent (or more accurately, the level that was the root when it read the
129meta-data page).  In this case the stack it made while descending does not
130help for finding the correct page.  When this happens, we find the correct
131place by re-descending the tree until we reach the level one above the
132level we need to insert a link to, and then moving right as necessary.
133(Typically this will take only two fetches, the meta-data page and the new
134root, but in principle there could have been more than one root split
135since we saw the root.  We can identify the correct tree level by means of
136the level numbers stored in each page.  The situation is rare enough that
137we do not need a more efficient solution.)
138
139Lehman and Yao must couple/chain locks as part of moving right when
140relocating a child page's downlink during an ascent of the tree.  This is
141the only point where Lehman and Yao have to simultaneously hold three
142locks (a lock on the child, the original parent, and the original parent's
143right sibling).  We don't need to couple internal page locks for pages on
144the same level, though.  We match a child's block number to a downlink
145from a pivot tuple one level up, whereas Lehman and Yao match on the
146separator key associated with the downlink that was followed during the
147initial descent.  We can release the lock on the original parent page
148before acquiring a lock on its right sibling, since there is never any
149need to deal with the case where the separator key that we must relocate
150becomes the original parent's high key.  Lanin and Shasha don't couple
151locks here either, though they also don't couple locks between levels
152during ascents.  They are willing to "wait and try again" to avoid races.
153Their algorithm is optimistic, which means that "an insertion holds no
154more than one write lock at a time during its ascent".  We more or less
155stick with Lehman and Yao's approach of conservatively coupling parent and
156child locks when ascending the tree, since it's far simpler.
157
158Lehman and Yao assume fixed-size keys, but we must deal with
159variable-size keys.  Therefore there is not a fixed maximum number of
160keys per page; we just stuff in as many as will fit.  When we split a
161page, we try to equalize the number of bytes, not items, assigned to
162pages (though suffix truncation is also considered).  Note we must include
163the incoming item in this calculation, otherwise it is possible to find
164that the incoming item doesn't fit on the split page where it needs to go!
165
166Deleting index tuples during VACUUM
167-----------------------------------
168
169Before deleting a leaf item, we get a full cleanup lock on the target
170page, so that no other backend has a pin on the page when the deletion
171starts.  This is not necessary for correctness in terms of the btree index
172operations themselves; as explained above, index scans logically stop
173"between" pages and so can't lose their place.  The reason we do it is to
174provide an interlock between VACUUM and index scans that are not prepared
175to deal with concurrent TID recycling when visiting the heap.  Since only
176VACUUM can ever mark pointed-to items LP_UNUSED in the heap, and since
177this only ever happens _after_ btbulkdelete returns, having index scans
178hold on to the pin (used when reading from the leaf page) until _after_
179they're done visiting the heap (for TIDs from pinned leaf page) prevents
180concurrent TID recycling.  VACUUM cannot get a conflicting cleanup lock
181until the index scan is totally finished processing its leaf page.
182
183This approach is fairly coarse, so we avoid it whenever possible.  In
184practice most index scans won't hold onto their pin, and so won't block
185VACUUM.  These index scans must deal with TID recycling directly, which is
186more complicated and not always possible.  See later section on making
187concurrent TID recycling safe.
188
189Opportunistic index tuple deletion performs almost the same page-level
190modifications while only holding an exclusive lock.  This is safe because
191there is no question of TID recycling taking place later on -- only VACUUM
192can make TIDs recyclable.  See also simple deletion and bottom-up
193deletion, below.
194
195Because a pin is not always held, and a page can be split even while
196someone does hold a pin on it, it is possible that an indexscan will
197return items that are no longer stored on the page it has a pin on, but
198rather somewhere to the right of that page.  To ensure that VACUUM can't
199prematurely make TIDs recyclable in this scenario, we require btbulkdelete
200to obtain a cleanup lock on every leaf page in the index, even pages that
201don't contain any deletable tuples.  Note that this requirement does not
202say that btbulkdelete must visit the pages in any particular order.
203
204VACUUM's linear scan, concurrent page splits
205--------------------------------------------
206
207VACUUM accesses the index by doing a linear scan to search for deletable
208TIDs, while considering the possibility of deleting empty pages in
209passing.  This is in physical/block order, not logical/keyspace order.
210The tricky part of this is avoiding missing any deletable tuples in the
211presence of concurrent page splits: a page split could easily move some
212tuples from a page not yet passed over by the sequential scan to a
213lower-numbered page already passed over.
214
215To implement this, we provide a "vacuum cycle ID" mechanism that makes it
216possible to determine whether a page has been split since the current
217btbulkdelete cycle started.  If btbulkdelete finds a page that has been
218split since it started, and has a right-link pointing to a lower page
219number, then it temporarily suspends its sequential scan and visits that
220page instead.  It must continue to follow right-links and vacuum dead
221tuples until reaching a page that either hasn't been split since
222btbulkdelete started, or is above the location of the outer sequential
223scan.  Then it can resume the sequential scan.  This ensures that all
224tuples are visited.  It may be that some tuples are visited twice, but
225that has no worse effect than an inaccurate index tuple count (and we
226can't guarantee an accurate count anyway in the face of concurrent
227activity).  Note that this still works if the has-been-recently-split test
228has a small probability of false positives, so long as it never gives a
229false negative.  This makes it possible to implement the test with a small
230counter value stored on each index page.
231
232Deleting entire pages during VACUUM
233-----------------------------------
234
235We consider deleting an entire page from the btree only when it's become
236completely empty of items.  (Merging partly-full pages would allow better
237space reuse, but it seems impractical to move existing data items left or
238right to make this happen --- a scan moving in the opposite direction
239might miss the items if so.)  Also, we *never* delete the rightmost page
240on a tree level (this restriction simplifies the traversal algorithms, as
241explained below).  Page deletion always begins from an empty leaf page.  An
242internal page can only be deleted as part of deleting an entire subtree.
243This is always a "skinny" subtree consisting of a "chain" of internal pages
244plus a single leaf page.  There is one page on each level of the subtree,
245and each level/page covers the same key space.
246
247Deleting a leaf page is a two-stage process.  In the first stage, the page
248is unlinked from its parent, and marked as half-dead.  The parent page must
249be found using the same type of search as used to find the parent during an
250insertion split.  We lock the target and the parent pages, change the
251target's downlink to point to the right sibling, and remove its old
252downlink.  This causes the target page's key space to effectively belong to
253its right sibling.  (Neither the left nor right sibling pages need to
254change their "high key" if any; so there is no problem with possibly not
255having enough space to replace a high key.)  At the same time, we mark the
256target page as half-dead, which causes any subsequent searches to ignore it
257and move right (or left, in a backwards scan).  This leaves the tree in a
258similar state as during a page split: the page has no downlink pointing to
259it, but it's still linked to its siblings.
260
261(Note: Lanin and Shasha prefer to make the key space move left, but their
262argument for doing so hinges on not having left-links, which we have
263anyway.  So we simplify the algorithm by moving the key space right.  This
264is only possible because we don't match on a separator key when ascending
265the tree during a page split, unlike Lehman and Yao/Lanin and Shasha -- it
266doesn't matter if the downlink is re-found in a pivot tuple whose separator
267key does not match the one encountered when inserter initially descended
268the tree.)
269
270To preserve consistency on the parent level, we cannot merge the key space
271of a page into its right sibling unless the right sibling is a child of
272the same parent --- otherwise, the parent's key space assignment changes
273too, meaning we'd have to make bounding-key updates in its parent, and
274perhaps all the way up the tree.  Since we can't possibly do that
275atomically, we forbid this case.  That means that the rightmost child of a
276parent node can't be deleted unless it's the only remaining child, in which
277case we will delete the parent too (see below).
278
279In the second-stage, the half-dead leaf page is unlinked from its siblings.
280We first lock the left sibling (if any) of the target, the target page
281itself, and its right sibling (there must be one) in that order.  Then we
282update the side-links in the siblings, and mark the target page deleted.
283
284When we're about to delete the last remaining child of a parent page, things
285are slightly more complicated.  In the first stage, we leave the immediate
286parent of the leaf page alone, and remove the downlink to the parent page
287instead, from the grandparent.  If it's the last child of the grandparent
288too, we recurse up until we find a parent with more than one child, and
289remove the downlink of that page.  The leaf page is marked as half-dead, and
290the block number of the page whose downlink was removed is stashed in the
291half-dead leaf page.  This leaves us with a chain of internal pages, with
292one downlink each, leading to the half-dead leaf page, and no downlink
293pointing to the topmost page in the chain.
294
295While we recurse up to find the topmost parent in the chain, we keep the
296leaf page locked, but don't need to hold locks on the intermediate pages
297between the leaf and the topmost parent -- insertions into upper tree levels
298happen only as a result of splits of child pages, and that can't happen as
299long as we're keeping the leaf locked.  The internal pages in the chain
300cannot acquire new children afterwards either, because the leaf page is
301marked as half-dead and won't be split.
302
303Removing the downlink to the top of the to-be-deleted subtree/chain
304effectively transfers the key space to the right sibling for all the
305intermediate levels too, in one atomic operation.  A concurrent search might
306still visit the intermediate pages, but it will move right when it reaches
307the half-dead page at the leaf level.  In particular, the search will move to
308the subtree to the right of the half-dead leaf page/to-be-deleted subtree,
309since the half-dead leaf page's right sibling must be a "cousin" page, not a
310"true" sibling page (or a second cousin page when the to-be-deleted chain
311starts at leaf page's grandparent page, and so on).
312
313In the second stage, the topmost page in the chain is unlinked from its
314siblings, and the half-dead leaf page is updated to point to the next page
315down in the chain.  This is repeated until there are no internal pages left
316in the chain.  Finally, the half-dead leaf page itself is unlinked from its
317siblings.
318
319A deleted page cannot be recycled immediately, since there may be other
320processes waiting to reference it (ie, search processes that just left the
321parent, or scans moving right or left from one of the siblings).  These
322processes must be able to observe a deleted page for some time after the
323deletion operation, in order to be able to at least recover from it (they
324recover by moving right, as with concurrent page splits).  Searchers never
325have to worry about concurrent page recycling.
326
327See "Placing deleted pages in the FSM" section below for a description of
328when and how deleted pages become safe for VACUUM to make recyclable.
329
330Page deletion and backwards scans
331---------------------------------
332
333Moving left in a backward scan is complicated because we must consider
334the possibility that the left sibling was just split (meaning we must find
335the rightmost page derived from the left sibling), plus the possibility
336that the page we were just on has now been deleted and hence isn't in the
337sibling chain at all anymore.  So the move-left algorithm becomes:
338
3390. Remember the page we are on as the "original page".
3401. Follow the original page's left-link (we're done if this is zero).
3412. If the current page is live and its right-link matches the "original
342   page", we are done.
3433. Otherwise, move right one or more times looking for a live page whose
344   right-link matches the "original page".  If found, we are done.  (In
345   principle we could scan all the way to the right end of the index, but
346   in practice it seems better to give up after a small number of tries.
347   It's unlikely the original page's sibling split more than a few times
348   while we were in flight to it; if we do not find a matching link in a
349   few tries, then most likely the original page is deleted.)
3504. Return to the "original page".  If it is still live, return to step 1
351   (we guessed wrong about it being deleted, and should restart with its
352   current left-link).  If it is dead, move right until a non-dead page
353   is found (there must be one, since rightmost pages are never deleted),
354   mark that as the new "original page", and return to step 1.
355
356This algorithm is correct because the live page found by step 4 will have
357the same left keyspace boundary as the page we started from.  Therefore,
358when we ultimately exit, it must be on a page whose right keyspace
359boundary matches the left boundary of where we started --- which is what
360we need to be sure we don't miss or re-scan any items.
361
362Page deletion and tree height
363-----------------------------
364
365Because we never delete the rightmost page of any level (and in particular
366never delete the root), it's impossible for the height of the tree to
367decrease.  After massive deletions we might have a scenario in which the
368tree is "skinny", with several single-page levels below the root.
369Operations will still be correct in this case, but we'd waste cycles
370descending through the single-page levels.  To handle this we use an idea
371from Lanin and Shasha: we keep track of the "fast root" level, which is
372the lowest single-page level.  The meta-data page keeps a pointer to this
373level as well as the true root.  All ordinary operations initiate their
374searches at the fast root not the true root.  When we split a page that is
375alone on its level or delete the next-to-last page on a level (both cases
376are easily detected), we have to make sure that the fast root pointer is
377adjusted appropriately.  In the split case, we do this work as part of the
378atomic update for the insertion into the parent level; in the delete case
379as part of the atomic update for the delete (either way, the metapage has
380to be the last page locked in the update to avoid deadlock risks).  This
381avoids race conditions if two such operations are executing concurrently.
382
383Placing deleted pages in the FSM
384--------------------------------
385
386Recycling a page is decoupled from page deletion.  A deleted page can only
387be put in the FSM to be recycled once there is no possible scan or search
388that has a reference to it; until then, it must stay in place with its
389sibling links undisturbed, as a tombstone that allows concurrent searches
390to detect and then recover from concurrent deletions (which are rather
391like concurrent page splits to searchers).  This design is an
392implementation of what Lanin and Shasha call "the drain technique".
393
394We implement the technique by waiting until all active snapshots and
395registered snapshots as of the page deletion are gone; which is overly
396strong, but is simple to implement within Postgres.  When marked fully
397dead, a deleted page is labeled with the next-transaction counter value.
398VACUUM can reclaim the page for re-use when the stored XID is guaranteed
399to be "visible to everyone".  As collateral damage, we wait for snapshots
400taken until the next transaction to allocate an XID commits.  We also wait
401for running XIDs with no snapshots.
402
403Prior to PostgreSQL 14, VACUUM would only place _old_ deleted pages that
404it encounters during its linear scan (pages deleted by a previous VACUUM
405operation) in the FSM.  Newly deleted pages were never placed in the FSM,
406because that was assumed to _always_ be unsafe.  That assumption was
407unnecessarily pessimistic in practice, though -- it often doesn't take
408very long for newly deleted pages to become safe to place in the FSM.
409There is no truly principled way to predict when deleted pages will become
410safe to place in the FSM for recycling -- it might become safe almost
411immediately (long before the current VACUUM completes), or it might not
412even be safe by the time the next VACUUM takes place.  Recycle safety is
413purely a question of maintaining the consistency (or at least the apparent
414consistency) of a physical data structure.  The state within the backend
415running VACUUM is simply not relevant.
416
417PostgreSQL 14 added the ability for VACUUM to consider if it's possible to
418recycle newly deleted pages at the end of the full index scan where the
419page deletion took place.  It is convenient to check if it's safe at that
420point.  This does require that VACUUM keep around a little bookkeeping
421information about newly deleted pages, but that's very cheap.  Using
422in-memory state for this avoids the need to revisit newly deleted pages a
423second time later on -- we can just use safexid values from the local
424bookkeeping state to determine recycle safety in a deferred fashion.
425
426The need for additional FSM indirection after a page deletion operation
427takes place is a natural consequence of the highly permissive rules for
428index scans with Lehman and Yao's design.  In general an index scan
429doesn't have to hold a lock or even a pin on any page when it descends the
430tree (nothing that you'd usually think of as an interlock is held "between
431levels").  At the same time, index scans cannot be allowed to land on a
432truly unrelated page due to concurrent recycling (not to be confused with
433concurrent deletion), because that results in wrong answers to queries.
434Simpler approaches to page deletion that don't need to defer recycling are
435possible, but none seem compatible with Lehman and Yao's design.
436
437Placing an already-deleted page in the FSM to be recycled when needed
438doesn't actually change the state of the page.  The page will be changed
439whenever it is subsequently taken from the FSM for reuse.  The deleted
440page's contents will be overwritten by the split operation (it will become
441the new right sibling page).
442
443Making concurrent TID recycling safe
444------------------------------------
445
446As explained in the earlier section about deleting index tuples during
447VACUUM, we implement a locking protocol that allows individual index scans
448to avoid concurrent TID recycling.  Index scans opt-out (and so drop their
449leaf page pin when visiting the heap) whenever it's safe to do so, though.
450Dropping the pin early is useful because it avoids blocking progress by
451VACUUM.  This is particularly important with index scans used by cursors,
452since idle cursors sometimes stop for relatively long periods of time.  In
453extreme cases, a client application may hold on to an idle cursors for
454hours or even days.  Blocking VACUUM for that long could be disastrous.
455
456Index scans that don't hold on to a buffer pin are protected by holding an
457MVCC snapshot instead.  This more limited interlock prevents wrong answers
458to queries, but it does not prevent concurrent TID recycling itself (only
459holding onto the leaf page pin while accessing the heap ensures that).
460
461Index-only scans can never drop their buffer pin, since they are unable to
462tolerate having a referenced TID become recyclable.  Index-only scans
463typically just visit the visibility map (not the heap proper), and so will
464not reliably notice that any stale TID reference (for a TID that pointed
465to a dead-to-all heap item at first) was concurrently marked LP_UNUSED in
466the heap by VACUUM.  This could easily allow VACUUM to set the whole heap
467page to all-visible in the visibility map immediately afterwards.  An MVCC
468snapshot is only sufficient to avoid problems during plain index scans
469because they must access granular visibility information from the heap
470proper.  A plain index scan will even recognize LP_UNUSED items in the
471heap (items that could be recycled but haven't been just yet) as "not
472visible" -- even when the heap page is generally considered all-visible.
473
474LP_DEAD setting of index tuples by the kill_prior_tuple optimization
475(described in full in simple deletion, below) is also more complicated for
476index scans that drop their leaf page pins.  We must be careful to avoid
477LP_DEAD-marking any new index tuple that looks like a known-dead index
478tuple because it happens to share the same TID, following concurrent TID
479recycling.  It's just about possible that some other session inserted a
480new, unrelated index tuple, on the same leaf page, which has the same
481original TID.  It would be totally wrong to LP_DEAD-set this new,
482unrelated index tuple.
483
484We handle this kill_prior_tuple race condition by having affected index
485scans conservatively assume that any change to the leaf page at all
486implies that it was reached by btbulkdelete in the interim period when no
487buffer pin was held.  This is implemented by not setting any LP_DEAD bits
488on the leaf page at all when the page's LSN has changed.  (That won't work
489with an unlogged index, so for now we don't ever apply the "don't hold
490onto pin" optimization there.)
491
492Fastpath For Index Insertion
493----------------------------
494
495We optimize for a common case of insertion of increasing index key
496values by caching the last page to which this backend inserted the last
497value, if this page was the rightmost leaf page. For the next insert, we
498can then quickly check if the cached page is still the rightmost leaf
499page and also the correct place to hold the current value. We can avoid
500the cost of walking down the tree in such common cases.
501
502The optimization works on the assumption that there can only be one
503non-ignorable leaf rightmost page, and so not even a visible-to-everyone
504style interlock is required.  We cannot fail to detect that our hint was
505invalidated, because there can only be one such page in the B-Tree at
506any time. It's possible that the page will be deleted and recycled
507without a backend's cached page also being detected as invalidated, but
508only when we happen to recycle a block that once again gets recycled as the
509rightmost leaf page.
510
511Simple deletion
512---------------
513
514If a process visits a heap tuple and finds that it's dead and removable
515(ie, dead to all open transactions, not only that process), then we can
516return to the index and mark the corresponding index entry "known dead",
517allowing subsequent index scans to skip visiting the heap tuple.  The
518"known dead" marking works by setting the index item's lp_flags state
519to LP_DEAD.  This is currently only done in plain indexscans, not bitmap
520scans, because only plain scans visit the heap and index "in sync" and so
521there's not a convenient way to do it for bitmap scans.  Note also that
522LP_DEAD bits are often set when checking a unique index for conflicts on
523insert (this is simpler because it takes place when we hold an exclusive
524lock on the leaf page).
525
526Once an index tuple has been marked LP_DEAD it can actually be deleted
527from the index immediately; since index scans only stop "between" pages,
528no scan can lose its place from such a deletion.  We separate the steps
529because we allow LP_DEAD to be set with only a share lock (it's like a
530hint bit for a heap tuple), but physically deleting tuples requires an
531exclusive lock.  We also need to generate a snapshotConflictHorizon for
532each deletion operation's WAL record, which requires additional
533coordinating with the tableam when the deletion actually takes place.
534(snapshotConflictHorizon value may be used to generate a conflict during
535subsequent REDO of the record by a standby.)
536
537Delaying and batching index tuple deletion like this enables a further
538optimization: opportunistic checking of "extra" nearby index tuples
539(tuples that are not LP_DEAD-set) when they happen to be very cheap to
540check in passing (because we already know that the tableam will be
541visiting their table block to generate a snapshotConflictHorizon).  Any
542index tuples that turn out to be safe to delete will also be deleted.
543Simple deletion will behave as if the extra tuples that actually turn
544out to be delete-safe had their LP_DEAD bits set right from the start.
545
546Deduplication can also prevent a page split, but index tuple deletion is
547our preferred approach.  Note that posting list tuples can only have
548their LP_DEAD bit set when every table TID within the posting list is
549known dead.  This isn't much of a problem in practice because LP_DEAD
550bits are just a starting point for deletion.  What really matters is
551that _some_ deletion operation that targets related nearby-in-table TIDs
552takes place at some point before the page finally splits.  That's all
553that's required for the deletion process to perform granular removal of
554groups of dead TIDs from posting list tuples (without the situation ever
555being allowed to get out of hand).
556
557Bottom-Up deletion
558------------------
559
560We attempt to delete whatever duplicates happen to be present on the page
561when the duplicates are suspected to be caused by version churn from
562successive UPDATEs.  This only happens when we receive an executor hint
563indicating that optimizations like heapam's HOT have not worked out for
564the index -- the incoming tuple must be a logically unchanged duplicate
565which is needed for MVCC purposes, suggesting that that might well be the
566dominant source of new index tuples on the leaf page in question.  (Also,
567bottom-up deletion is triggered within unique indexes in cases with
568continual INSERT and DELETE related churn, since that is easy to detect
569without any external hint.)
570
571Simple deletion will already have failed to prevent a page split when a
572bottom-up deletion pass takes place (often because no LP_DEAD bits were
573ever set on the page).  The two mechanisms have closely related
574implementations.  The same WAL records are used for each operation, and
575the same tableam infrastructure is used to determine what TIDs/tuples are
576actually safe to delete.  The implementations only differ in how they pick
577TIDs to consider for deletion, and whether or not the tableam will give up
578before accessing all table blocks (bottom-up deletion lives with the
579uncertainty of its success by keeping the cost of failure low).  Even
580still, the two mechanisms are clearly distinct at the conceptual level.
581
582Bottom-up index deletion is driven entirely by heuristics (whereas simple
583deletion is guaranteed to delete at least those index tuples that are
584already LP_DEAD marked -- there must be at least one).  We have no
585certainty that we'll find even one index tuple to delete.  That's why we
586closely cooperate with the tableam to keep the costs it pays in balance
587with the benefits we receive.  The interface that we use for this is
588described in detail in access/tableam.h.
589
590Bottom-up index deletion can be thought of as a backstop mechanism against
591unnecessary version-driven page splits.  It is based in part on an idea
592from generational garbage collection: the "generational hypothesis".  This
593is the empirical observation that "most objects die young".  Within
594nbtree, new index tuples often quickly appear in the same place, and then
595quickly become garbage.  There can be intense concentrations of garbage in
596relatively few leaf pages with certain workloads (or there could be in
597earlier versions of PostgreSQL without bottom-up index deletion, at
598least).  See doc/src/sgml/btree.sgml for a high-level description of the
599design principles behind bottom-up index deletion in nbtree, including
600details of how it complements VACUUM.
601
602We expect to find a reasonably large number of tuples that are safe to
603delete within each bottom-up pass.  If we don't then we won't need to
604consider the question of bottom-up deletion for the same leaf page for
605quite a while (usually because the page splits, which resolves the
606situation for the time being).  We expect to perform regular bottom-up
607deletion operations against pages that are at constant risk of unnecessary
608page splits caused only by version churn.  When the mechanism works well
609we'll constantly be "on the verge" of having version-churn-driven page
610splits, but never actually have even one.
611
612Our duplicate heuristics work well despite being fairly simple.
613Unnecessary page splits only occur when there are truly pathological
614levels of version churn (in theory a small amount of version churn could
615make a page split occur earlier than strictly necessary, but that's pretty
616harmless).  We don't have to understand the underlying workload; we only
617have to understand the general nature of the pathology that we target.
618Version churn is easy to spot when it is truly pathological.  Affected
619leaf pages are fairly homogeneous.
620
621WAL Considerations
622------------------
623
624The insertion and deletion algorithms in themselves don't guarantee btree
625consistency after a crash.  To provide robustness, we depend on WAL
626replay.  A single WAL entry is effectively an atomic action --- we can
627redo it from the log if it fails to complete.
628
629Ordinary item insertions (that don't force a page split) are of course
630single WAL entries, since they only affect one page.  The same for
631leaf-item deletions (if the deletion brings the leaf page to zero items,
632it is now a candidate to be deleted, but that is a separate action).
633
634An insertion that causes a page split is logged as a single WAL entry for
635the changes occurring on the insertion's level --- including update of the
636right sibling's left-link --- followed by a second WAL entry for the
637insertion on the parent level (which might itself be a page split, requiring
638an additional insertion above that, etc).
639
640For a root split, the follow-on WAL entry is a "new root" entry rather than
641an "insertion" entry, but details are otherwise much the same.
642
643Because splitting involves multiple atomic actions, it's possible that the
644system crashes between splitting a page and inserting the downlink for the
645new half to the parent.  After recovery, the downlink for the new page will
646be missing.  The search algorithm works correctly, as the page will be found
647by following the right-link from its left sibling, although if a lot of
648downlinks in the tree are missing, performance will suffer.  A more serious
649consequence is that if the page without a downlink gets split again, the
650insertion algorithm will fail to find the location in the parent level to
651insert the downlink.
652
653Our approach is to create any missing downlinks on-the-fly, when searching
654the tree for a new insertion.  It could be done during searches, too, but
655it seems best not to put any extra updates in what would otherwise be a
656read-only operation (updating is not possible in hot standby mode anyway).
657It would seem natural to add the missing downlinks in VACUUM, but since
658inserting a downlink might require splitting a page, it might fail if you
659run out of disk space.  That would be bad during VACUUM - the reason for
660running VACUUM in the first place might be that you run out of disk space,
661and now VACUUM won't finish because you're out of disk space.  In contrast,
662an insertion can require enlarging the physical file anyway.  There is one
663minor exception: VACUUM finishes interrupted splits of internal pages when
664deleting their children.  This allows the code for re-finding parent items
665to be used by both page splits and page deletion.
666
667To identify missing downlinks, when a page is split, the left page is
668flagged to indicate that the split is not yet complete (INCOMPLETE_SPLIT).
669When the downlink is inserted to the parent, the flag is cleared atomically
670with the insertion.  The child page is kept locked until the insertion in
671the parent is finished and the flag in the child cleared, but can be
672released immediately after that, before recursing up the tree if the parent
673also needs to be split.  This ensures that incompletely split pages should
674not be seen under normal circumstances; only if insertion to the parent
675has failed for some reason. (It's also possible for a reader to observe
676a page with the incomplete split flag set during recovery; see later
677section on "Scans during Recovery" for details.)
678
679We flag the left page, even though it's the right page that's missing the
680downlink, because it's more convenient to know already when following the
681right-link from the left page to the right page that it will need to have
682its downlink inserted to the parent.
683
684When splitting a non-root page that is alone on its level, the required
685metapage update (of the "fast root" link) is performed and logged as part
686of the insertion into the parent level.  When splitting the root page, the
687metapage update is handled as part of the "new root" action.
688
689Each step in page deletion is logged as a separate WAL entry: marking the
690leaf as half-dead and removing the downlink is one record, and unlinking a
691page is a second record.  If vacuum is interrupted for some reason, or the
692system crashes, the tree is consistent for searches and insertions.  The
693next VACUUM will find the half-dead leaf page and continue the deletion.
694
695Before 9.4, we used to keep track of incomplete splits and page deletions
696during recovery and finish them immediately at end of recovery, instead of
697doing it lazily at the next insertion or vacuum.  However, that made the
698recovery much more complicated, and only fixed the problem when crash
699recovery was performed.  An incomplete split can also occur if an otherwise
700recoverable error, like out-of-memory or out-of-disk-space, happens while
701inserting the downlink to the parent.
702
703Scans during Recovery
704---------------------
705
706nbtree indexes support read queries in Hot Standby mode. Every atomic
707action/WAL record makes isolated changes that leave the tree in a
708consistent state for readers. Readers lock pages according to the same
709rules that readers follow on the primary. (Readers may have to move
710right to recover from a "concurrent" page split or page deletion, just
711like on the primary.)
712
713However, there are a couple of differences in how pages are locked by
714replay/the startup process as compared to the original write operation
715on the primary. The exceptions involve page splits and page deletions.
716The first phase and second phase of a page split are processed
717independently during replay, since they are independent atomic actions.
718We do not attempt to recreate the coupling of parent and child page
719write locks that took place on the primary. This is safe because readers
720never care about the incomplete split flag anyway. Holding on to an
721extra write lock on the primary is only necessary so that a second
722writer cannot observe the incomplete split flag before the first writer
723finishes the split. If we let concurrent writers on the primary observe
724an incomplete split flag on the same page, each writer would attempt to
725complete the unfinished split, corrupting the parent page.  (Similarly,
726replay of page deletion records does not hold a write lock on the target
727leaf page throughout; only the primary needs to block out concurrent
728writers that insert on to the page being deleted.)
729
730WAL replay holds same-level locks in a way that matches the approach
731taken during original execution, though. This prevent readers from
732observing same-level inconsistencies. It's probably possible to be more
733lax about how same-level locks are acquired during recovery (most kinds
734of readers could still move right to recover if we didn't couple
735same-level locks), but we prefer to be conservative here.
736
737During recovery all index scans start with ignore_killed_tuples = false
738and we never set kill_prior_tuple. We do this because the oldest xmin
739on the standby server can be older than the oldest xmin on the primary
740server, which means tuples can be marked LP_DEAD even when they are
741still visible on the standby. We don't WAL log tuple LP_DEAD bits, but
742they can still appear in the standby because of full page writes. So
743we must always ignore them in standby, and that means it's not worth
744setting them either.  (When LP_DEAD-marked tuples are eventually deleted
745on the primary, the deletion is WAL-logged.  Queries that run on a
746standby therefore get much of the benefit of any LP_DEAD setting that
747takes place on the primary.)
748
749Note that we talk about scans that are started during recovery. We go to
750a little trouble to allow a scan to start during recovery and end during
751normal running after recovery has completed. This is a key capability
752because it allows running applications to continue while the standby
753changes state into a normally running server.
754
755The interlocking required to avoid returning incorrect results from
756non-MVCC scans is not required on standby nodes. We still get a full
757cleanup lock when replaying VACUUM records during recovery, but recovery
758does not need to lock every leaf page (only those leaf pages that have
759items to delete) -- that's sufficient to avoid breaking index-only scans
760during recovery (see section above about making TID recycling safe). That
761leaves concern only for plain index scans. (XXX: Not actually clear why
762this is totally unnecessary during recovery.)
763
764MVCC snapshot plain index scans are always safe, for the same reasons that
765they're safe during original execution.  HeapTupleSatisfiesToast() doesn't
766use MVCC semantics, though that's because it doesn't need to - if the main
767heap row is visible then the toast rows will also be visible. So as long
768as we follow a toast pointer from a visible (live) tuple the corresponding
769toast rows will also be visible, so we do not need to recheck MVCC on
770them.

772Other Things That Are Handy to Know
773-----------------------------------
774
775Page zero of every btree is a meta-data page.  This page stores the
776location of the root page --- both the true root and the current effective
777root ("fast" root).  To avoid fetching the metapage for every single index
778search, we cache a copy of the meta-data information in the index's
779relcache entry (rd_amcache).  This is a bit ticklish since using the cache
780implies following a root page pointer that could be stale.  However, a
781backend following a cached pointer can sufficiently verify whether it
782reached the intended page; either by checking the is-root flag when it
783is going to the true root, or by checking that the page has no siblings
784when going to the fast root.  At worst, this could result in descending
785some extra tree levels if we have a cached pointer to a fast root that is
786now above the real fast root.  Such cases shouldn't arise often enough to
787be worth optimizing; and in any case we can expect a relcache flush will
788discard the cached metapage before long, since a VACUUM that's moved the
789fast root pointer can be expected to issue a statistics update for the
790index.
791
792The algorithm assumes we can fit at least three items per page
793(a "high key" and two real data items).  Therefore it's unsafe
794to accept items larger than 1/3rd page size.  Larger items would
795work sometimes, but could cause failures later on depending on
796what else gets put on their page.
797
798"ScanKey" data structures are used in two fundamentally different ways
799in this code, which we describe as "search" scankeys and "insertion"
800scankeys.  A search scankey is the kind passed to btbeginscan() or
801btrescan() from outside the btree code.  The sk_func pointers in a search
802scankey point to comparison functions that return boolean, such as int4lt.
803There might be more than one scankey entry for a given index column, or
804none at all.  (We require the keys to appear in index column order, but
805the order of multiple keys for a given column is unspecified.)  An
806insertion scankey ("BTScanInsert" data structure) uses a similar
807array-of-ScanKey data structure, but the sk_func pointers point to btree
808comparison support functions (ie, 3-way comparators that return int4 values
809interpreted as <0, =0, >0).  In an insertion scankey there is at most one
810entry per index column.  There is also other data about the rules used to
811locate where to begin the scan, such as whether or not the scan is a
812"nextkey" scan.  Insertion scankeys are built within the btree code (eg, by
813_bt_mkscankey()) and are used to locate the starting point of a scan, as
814well as for locating the place to insert a new index tuple.  (Note: in the
815case of an insertion scankey built from a search scankey or built from a
816truncated pivot tuple, there might be fewer keys than index columns,
817indicating that we have no constraints for the remaining index columns.)
818After we have located the starting point of a scan, the original search
819scankey is consulted as each index entry is sequentially scanned to decide
820whether to return the entry and whether the scan can stop (see
821_bt_checkkeys()).

823Notes about suffix truncation
824-----------------------------
825
826We truncate away suffix key attributes that are not needed for a page high
827key during a leaf page split.  The remaining attributes must distinguish
828the last index tuple on the post-split left page as belonging on the left
829page, and the first index tuple on the post-split right page as belonging
830on the right page.  Tuples logically retain truncated key attributes,
831though they implicitly have "negative infinity" as their value, and have no
832storage overhead.  Since the high key is subsequently reused as the
833downlink in the parent page for the new right page, suffix truncation makes
834pivot tuples short.  INCLUDE indexes are guaranteed to have non-key
835attributes truncated at the time of a leaf page split, but may also have
836some key attributes truncated away, based on the usual criteria for key
837attributes.  They are not a special case, since non-key attributes are
838merely payload to B-Tree searches.
839
840The goal of suffix truncation of key attributes is to improve index
841fan-out.  The technique was first described by Bayer and Unterauer (R.Bayer
842and K.Unterauer, Prefix B-Trees, ACM Transactions on Database Systems, Vol
8432, No. 1, March 1977, pp 11-26).  The Postgres implementation is loosely
844based on their paper.  Note that Postgres only implements what the paper
845refers to as simple prefix B-Trees.  Note also that the paper assumes that
846the tree has keys that consist of single strings that maintain the "prefix
847property", much like strings that are stored in a suffix tree (comparisons
848of earlier bytes must always be more significant than comparisons of later
849bytes, and, in general, the strings must compare in a way that doesn't
850break transitive consistency as they're split into pieces).  Suffix
851truncation in Postgres currently only works at the whole-attribute
852granularity, but it would be straightforward to invent opclass
853infrastructure that manufactures a smaller attribute value in the case of
854variable-length types, such as text.  An opclass support function could
855manufacture the shortest possible key value that still correctly separates
856each half of a leaf page split.
857
858There is sophisticated criteria for choosing a leaf page split point.  The
859general idea is to make suffix truncation effective without unduly
860influencing the balance of space for each half of the page split.  The
861choice of leaf split point can be thought of as a choice among points
862*between* items on the page to be split, at least if you pretend that the
863incoming tuple was placed on the page already (you have to pretend because
864there won't actually be enough space for it on the page).  Choosing the
865split point between two index tuples where the first non-equal attribute
866appears as early as possible results in truncating away as many suffix
867attributes as possible.  Evenly balancing space among each half of the
868split is usually the first concern, but even small adjustments in the
869precise split point can allow truncation to be far more effective.
870
871Suffix truncation is primarily valuable because it makes pivot tuples
872smaller, which delays splits of internal pages, but that isn't the only
873reason why it's effective.  Even truncation that doesn't make pivot tuples
874smaller due to alignment still prevents pivot tuples from being more
875restrictive than truly necessary in how they describe which values belong
876on which pages.
877
878While it's not possible to correctly perform suffix truncation during
879internal page splits, it's still useful to be discriminating when splitting
880an internal page.  The split point that implies a downlink be inserted in
881the parent that's the smallest one available within an acceptable range of
882the fillfactor-wise optimal split point is chosen.  This idea also comes
883from the Prefix B-Tree paper.  This process has much in common with what
884happens at the leaf level to make suffix truncation effective.  The overall
885effect is that suffix truncation tends to produce smaller, more
886discriminating pivot tuples, especially early in the lifetime of the index,
887while biasing internal page splits makes the earlier, smaller pivot tuples
888end up in the root page, delaying root page splits.
889
890Logical duplicates are given special consideration.  The logic for
891selecting a split point goes to great lengths to avoid having duplicates
892span more than one page, and almost always manages to pick a split point
893between two user-key-distinct tuples, accepting a completely lopsided split
894if it must.  When a page that's already full of duplicates must be split,
895the fallback strategy assumes that duplicates are mostly inserted in
896ascending heap TID order.  The page is split in a way that leaves the left
897half of the page mostly full, and the right half of the page mostly empty.
898The overall effect is that leaf page splits gracefully adapt to inserts of
899large groups of duplicates, maximizing space utilization.  Note also that
900"trapping" large groups of duplicates on the same leaf page like this makes
901deduplication more efficient.  Deduplication can be performed infrequently,
902without merging together existing posting list tuples too often.
903
904Notes about deduplication
905-------------------------
906
907We deduplicate non-pivot tuples in non-unique indexes to reduce storage
908overhead, and to avoid (or at least delay) page splits.  Note that the
909goals for deduplication in unique indexes are rather different; see later
910section for details.  Deduplication alters the physical representation of
911tuples without changing the logical contents of the index, and without
912adding overhead to read queries.  Non-pivot tuples are merged together
913into a single physical tuple with a posting list (a simple array of heap
914TIDs with the standard item pointer format).  Deduplication is always
915applied lazily, at the point where it would otherwise be necessary to
916perform a page split.  It occurs only when LP_DEAD items have been
917removed, as our last line of defense against splitting a leaf page
918(bottom-up index deletion may be attempted first, as our second last line
919of defense).  We can set the LP_DEAD bit with posting list tuples, though
920only when all TIDs are known dead.
921
922Our lazy approach to deduplication allows the page space accounting used
923during page splits to have absolutely minimal special case logic for
924posting lists.  Posting lists can be thought of as extra payload that
925suffix truncation will reliably truncate away as needed during page
926splits, just like non-key columns from an INCLUDE index tuple.
927Incoming/new tuples can generally be treated as non-overlapping plain
928items (though see section on posting list splits for information about how
929overlapping new/incoming items are really handled).
930
931The representation of posting lists is almost identical to the posting
932lists used by GIN, so it would be straightforward to apply GIN's varbyte
933encoding compression scheme to individual posting lists.  Posting list
934compression would break the assumptions made by posting list splits about
935page space accounting (see later section), so it's not clear how
936compression could be integrated with nbtree.  Besides, posting list
937compression does not offer a compelling trade-off for nbtree, since in
938general nbtree is optimized for consistent performance with many
939concurrent readers and writers.  Compression would also make the deletion
940of a subset of TIDs from a posting list slow and complicated, which would
941be a big problem for workloads that depend heavily on bottom-up index
942deletion.
943
944A major goal of our lazy approach to deduplication is to limit the
945performance impact of deduplication with random updates.  Even concurrent
946append-only inserts of the same key value will tend to have inserts of
947individual index tuples in an order that doesn't quite match heap TID
948order.  Delaying deduplication minimizes page level fragmentation.
949
950Deduplication in unique indexes
951-------------------------------
952
953Very often, the number of distinct values that can ever be placed on
954almost any given leaf page in a unique index is fixed and permanent.  For
955example, a primary key on an identity column will usually only have leaf
956page splits caused by the insertion of new logical rows within the
957rightmost leaf page.  If there is a split of a non-rightmost leaf page,
958then the split must have been triggered by inserts associated with UPDATEs
959of existing logical rows.  Splitting a leaf page purely to store multiple
960versions is a false economy.  In effect, we're permanently degrading the
961index structure just to absorb a temporary burst of duplicates.
962
963Deduplication in unique indexes helps to prevent these pathological page
964splits.  Storing duplicates in a space efficient manner is not the goal,
965since in the long run there won't be any duplicates anyway.  Rather, we're
966buying time for standard garbage collection mechanisms to run before a
967page split is needed.
968
969Unique index leaf pages only get a deduplication pass when an insertion
970(that might have to split the page) observed an existing duplicate on the
971page in passing.  This is based on the assumption that deduplication will
972only work out when _all_ new insertions are duplicates from UPDATEs.  This
973may mean that we miss an opportunity to delay a page split, but that's
974okay because our ultimate goal is to delay leaf page splits _indefinitely_
975(i.e. to prevent them altogether).  There is little point in trying to
976delay a split that is probably inevitable anyway.  This allows us to avoid
977the overhead of attempting to deduplicate with unique indexes that always
978have few or no duplicates.
979
980Note: Avoiding "unnecessary" page splits driven by version churn is also
981the goal of bottom-up index deletion, which was added to PostgreSQL 14.
982Bottom-up index deletion is now the preferred way to deal with this
983problem (with all kinds of indexes, though especially with unique
984indexes).  Still, deduplication can sometimes augment bottom-up index
985deletion.  When deletion cannot free tuples (due to an old snapshot
986holding up cleanup), falling back on deduplication provides additional
987capacity.  Delaying the page split by deduplicating can allow a future
988bottom-up deletion pass of the same page to succeed.
989
990Posting list splits
991-------------------
992
993When the incoming tuple happens to overlap with an existing posting list,
994a posting list split is performed.  Like a page split, a posting list
995split resolves a situation where a new/incoming item "won't fit", while
996inserting the incoming item in passing (i.e. as part of the same atomic
997action).  It's possible (though not particularly likely) that an insert of
998a new item on to an almost-full page will overlap with a posting list,
999resulting in both a posting list split and a page split.  Even then, the
1000atomic action that splits the posting list also inserts the new item
1001(since page splits always insert the new item in passing).  Including the
1002posting list split in the same atomic action as the insert avoids problems
1003caused by concurrent inserts into the same posting list --  the exact
1004details of how we change the posting list depend upon the new item, and
1005vice-versa.  A single atomic action also minimizes the volume of extra
1006WAL required for a posting list split, since we don't have to explicitly
1007WAL-log the original posting list tuple.
1008
1009Despite piggy-backing on the same atomic action that inserts a new tuple,
1010posting list splits can be thought of as a separate, extra action to the
1011insert itself (or to the page split itself).  Posting list splits
1012conceptually "rewrite" an insert that overlaps with an existing posting
1013list into an insert that adds its final new item just to the right of the
1014posting list instead.  The size of the posting list won't change, and so
1015page space accounting code does not need to care about posting list splits
1016at all.  This is an important upside of our design; the page split point
1017choice logic is very subtle even without it needing to deal with posting
1018list splits.
1019
1020Only a few isolated extra steps are required to preserve the illusion that
1021the new item never overlapped with an existing posting list in the first
1022place: the heap TID of the incoming tuple has its TID replaced with the
1023rightmost/max heap TID from the existing/originally overlapping posting
1024list.  Similarly, the original incoming item's TID is relocated to the
1025appropriate offset in the posting list (we usually shift TIDs out of the
1026way to make a hole for it).  Finally, the posting-split-with-page-split
1027case must generate a new high key based on an imaginary version of the
1028original page that has both the final new item and the after-list-split
1029posting tuple (page splits usually just operate against an imaginary
1030version that contains the new item/item that won't fit).
1031
1032This approach avoids inventing an "eager" atomic posting split operation
1033that splits the posting list without simultaneously finishing the insert
1034of the incoming item.  This alternative design might seem cleaner, but it
1035creates subtle problems for page space accounting.  In general, there
1036might not be enough free space on the page to split a posting list such
1037that the incoming/new item no longer overlaps with either posting list
1038half --- the operation could fail before the actual retail insert of the
1039new item even begins.  We'd end up having to handle posting list splits
1040that need a page split anyway.  Besides, supporting variable "split points"
1041while splitting posting lists won't actually improve overall space
1042utilization.

1044Notes About Data Representation
1045-------------------------------
1046
1047The right-sibling link required by L&Y is kept in the page "opaque
1048data" area, as is the left-sibling link, the page level, and some flags.
1049The page level counts upwards from zero at the leaf level, to the tree
1050depth minus 1 at the root.  (Counting up from the leaves ensures that we
1051don't need to renumber any existing pages when splitting the root.)
1052
1053The Postgres disk block data format (an array of items) doesn't fit
1054Lehman and Yao's alternating-keys-and-pointers notion of a disk page,
1055so we have to play some games.  (The alternating-keys-and-pointers
1056notion is important for internal page splits, which conceptually split
1057at the middle of an existing pivot tuple -- the tuple's "separator" key
1058goes on the left side of the split as the left side's new high key,
1059while the tuple's pointer/downlink goes on the right side as the
1060first/minus infinity downlink.)
1061
1062On a page that is not rightmost in its tree level, the "high key" is
1063kept in the page's first item, and real data items start at item 2.
1064The link portion of the "high key" item goes unused.  A page that is
1065rightmost has no "high key" (it's implicitly positive infinity), so
1066data items start with the first item.  Putting the high key at the
1067left, rather than the right, may seem odd, but it avoids moving the
1068high key as we add data items.
1069
1070On a leaf page, the data items are simply links to (TIDs of) tuples
1071in the relation being indexed, with the associated key values.
1072
1073On a non-leaf page, the data items are down-links to child pages with
1074bounding keys.  The key in each data item is a strict lower bound for
1075keys on that child page, so logically the key is to the left of that
1076downlink.  The high key (if present) is the upper bound for the last
1077downlink.  The first data item on each such page has no lower bound
1078--- or lower bound of minus infinity, if you prefer.  The comparison
1079routines must treat it accordingly.  The actual key stored in the
1080item is irrelevant, and need not be stored at all.  This arrangement
1081corresponds to the fact that an L&Y non-leaf page has one more pointer
1082than key.  Suffix truncation's negative infinity attributes behave in
1083the same way.
