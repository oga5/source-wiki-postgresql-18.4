3The Postgres Executor
4=====================
5
6The executor processes a tree of "plan nodes".  The plan tree is essentially
7a demand-pull pipeline of tuple processing operations.  Each node, when
8called, will produce the next tuple in its output sequence, or NULL if no
9more tuples are available.  If the node is not a primitive relation-scanning
10node, it will have child node(s) that it calls in turn to obtain input
11tuples.
12
13Refinements on this basic model include:
14
15* Choice of scan direction (forwards or backwards).  Caution: this is not
16currently well-supported.  It works for primitive scan nodes, but not very
17well for joins, aggregates, etc.
18
19* Rescan command to reset a node and make it generate its output sequence
20over again.
21
22* Parameters that can alter a node's results.  After adjusting a parameter,
23the rescan command must be applied to that node and all nodes above it.
24There is a moderately intelligent scheme to avoid rescanning nodes
25unnecessarily (for example, Sort does not rescan its input if no parameters
26of the input have changed, since it can just reread its stored sorted data).
27
28For a SELECT, it is only necessary to deliver the top-level result tuples
29to the client.  For INSERT/UPDATE/DELETE/MERGE, the actual table modification
30operations happen in a top-level ModifyTable plan node.  If the query
31includes a RETURNING clause, the ModifyTable node delivers the computed
32RETURNING rows as output, otherwise it returns nothing.  Handling INSERT
33is pretty straightforward: the tuples returned from the plan tree below
34ModifyTable are inserted into the correct result relation.  For UPDATE,
35the plan tree returns the new values of the updated columns, plus "junk"
36(hidden) column(s) identifying which table row is to be updated.  The
37ModifyTable node must fetch that row to extract values for the unchanged
38columns, combine the values into a new row, and apply the update.  (For a
39heap table, the row-identity junk column is a CTID, but other things may
40be used for other table types.)  For DELETE, the plan tree need only deliver
41junk row-identity column(s), and the ModifyTable node visits each of those
42rows and marks the row deleted.  MERGE is described below.

47Plan Trees and State Trees
48--------------------------
49
50The plan tree delivered by the planner contains a tree of Plan nodes (struct
51types derived from struct Plan).  During executor startup we build a parallel
52tree of identical structure containing executor state nodes --- generally,
53every plan node type has a corresponding executor state node type.  Each node
54in the state tree has a pointer to its corresponding node in the plan tree,
55plus executor state data as needed to implement that node type.  This
56arrangement allows the plan tree to be completely read-only so far as the
57executor is concerned: all data that is modified during execution is in the
58state tree.  Read-only plan trees make life much simpler for plan caching and
59reuse.
60
61A corresponding executor state node may not be created during executor startup
62if the executor determines that an entire subplan is not required due to
63execution time partition pruning determining that no matching records will be
64found there.  This currently only occurs for Append and MergeAppend nodes.  In
65this case the non-required subplans are ignored and the executor state's
66subnode array will become out of sequence to the plan's subplan list.
67
68Each Plan node may have expression trees associated with it, to represent
69its target list, qualification conditions, etc.  These trees are also
70read-only to the executor, but the executor state for expression evaluation
71does not mirror the Plan expression's tree shape, as explained below.
72Rather, there's just one ExprState node per expression tree, although this
73may have sub-nodes for some complex expression node types.
74
75Altogether there are four classes of nodes used in these trees: Plan nodes,
76their corresponding PlanState nodes, Expr nodes, and ExprState nodes.
77(Actually, there are also List nodes, which are used as "glue" in all
78three tree-based representations.)
79

81Expression Trees and ExprState nodes
82------------------------------------
83
84Expression trees, in contrast to Plan trees, are not mirrored into a
85corresponding tree of state nodes.  Instead each separately executable
86expression tree (e.g. a Plan's qual or targetlist) is represented by one
87ExprState node.  The ExprState node contains the information needed to
88evaluate the expression in a compact, linear form.  That compact form is
89stored as a flat array in ExprState->steps[] (an array of ExprEvalStep,
90not ExprEvalStep *).
91
92The reasons for choosing such a representation include:
93- commonly the amount of work needed to evaluate one Expr-type node is
94  small enough that the overhead of having to perform a tree-walk
95  during evaluation is significant.
96- the flat representation can be evaluated non-recursively within a single
97  function, reducing stack depth and function call overhead.
98- such a representation is usable both for fast interpreted execution,
99  and for compiling into native code.
100
101The Plan-tree representation of an expression is compiled into an
102ExprState node by ExecInitExpr().  As much complexity as possible should
103be handled by ExecInitExpr() (and helpers), instead of execution time
104where both interpreted and compiled versions would need to deal with the
105complexity.  Besides duplicating effort between execution approaches,
106runtime initialization checks also have a small but noticeable cost every
107time the expression is evaluated.  Therefore, we allow ExecInitExpr() to
108precompute information that we do not expect to vary across execution of a
109single query, for example the set of CHECK constraint expressions to be
110applied to a domain type.  This could not be done at plan time without
111greatly increasing the number of events that require plan invalidation.
112(Previously, some information of this kind was rechecked on each
113expression evaluation, but that seems like unnecessary overhead.)

116Expression Initialization
117-------------------------
118
119During ExecInitExpr() and similar routines, Expr trees are converted
120into the flat representation.  Each Expr node might be represented by
121zero, one, or more ExprEvalSteps.
122
123Each ExprEvalStep's work is determined by its opcode (of enum ExprEvalOp)
124and it stores the result of its work into the Datum variable and boolean
125null flag variable pointed to by ExprEvalStep->resvalue/resnull.
126Complex expressions are performed by chaining together several steps.
127For example, "a + b" (one OpExpr, with two Var expressions) would be
128represented as two steps to fetch the Var values, and one step for the
129evaluation of the function underlying the + operator.  The steps for the
130Vars would have their resvalue/resnull pointing directly to the appropriate
131args[].value .isnull elements in the FunctionCallInfoBaseData struct that
132is used by the function evaluation step, thus avoiding extra work to copy
133the result values around.
134
135The last entry in a completed ExprState->steps array is always an
136EEOP_DONE_RETURN or EEOP_DONE_NO_RETURN step; this removes the need to
137test for end-of-array while iterating. The former is used when the
138expression returns a value directly, the latter when side-effects of
139expression initialization are the goal (e.g. for projection or
140aggregate transition value computation).
141
142Also, if the expression contains any variable references (to user
143columns of the ExprContext's INNER, OUTER, or SCAN tuples), the steps
144array begins with EEOP_*_FETCHSOME steps that ensure that the relevant
145tuples have been deconstructed to make the required columns directly
146available (cf. slot_getsomeattrs()).  This allows individual Var-fetching
147steps to be little more than an array lookup.
148
149Most of ExecInitExpr()'s work is done by the recursive function
150ExecInitExprRec() and its subroutines.  ExecInitExprRec() maps one Expr
151node into the steps required for execution, recursing as needed for
152sub-expressions.
153
154Each ExecInitExprRec() call has to specify where that subexpression's
155results are to be stored (via the resv/resnull parameters).  This allows
156the above scenario of evaluating a (sub-)expression directly into
157fcinfo->args[].value/isnull, but also requires some care: target Datum/isnull
158variables may not be shared with another ExecInitExprRec() unless the
159results are only needed by steps executing before further usages of those
160target Datum/isnull variables.  Due to the non-recursiveness of the
161ExprEvalStep representation that's usually easy to guarantee.
162
163ExecInitExprRec() pushes new operations into the ExprState->steps array
164using ExprEvalPushStep().  To keep the steps as a consecutively laid out
165array, ExprEvalPushStep() has to repalloc the entire array when there's
166not enough space.  Because of that it is *not* allowed to point directly
167into any of the steps during expression initialization.  Therefore, the
168resv/resnull for a subexpression usually point to some storage that is
169palloc'd separately from the steps array.  For instance, the
170FunctionCallInfoBaseData for a function call step is separately allocated
171rather than being part of the ExprEvalStep array.  The overall result
172of a complete expression is typically returned into the resvalue/resnull
173fields of the ExprState node itself.
174
175Some steps, e.g. boolean expressions, allow skipping evaluation of
176certain subexpressions.  In the flat representation this amounts to
177jumping to some later step rather than just continuing consecutively
178with the next step.  The target for such a jump is represented by
179the integer index in the ExprState->steps array of the step to execute
180next.  (Compare the EEO_NEXT and EEO_JUMP macros in execExprInterp.c.)
181
182Typically, ExecInitExprRec() has to push a jumping step into the steps
183array, then recursively generate steps for the subexpression that might
184get skipped over, then go back and fix up the jump target index using
185the now-known length of the subexpression's steps.  This is handled by
186adjust_jumps lists in execExpr.c.
187
188The last step in constructing an ExprState is to apply ExecReadyExpr(),
189which readies it for execution using whichever execution method has been
190selected.

193Expression Evaluation
194---------------------
195
196To allow for different methods of expression evaluation, and for
197better branch/jump target prediction, expressions are evaluated by
198calling ExprState->evalfunc (via ExecEvalExpr() and friends).
199
200ExecReadyExpr() can choose the method of interpretation by setting
201evalfunc to an appropriate function.  The default execution function,
202ExecInterpExpr, is implemented in execExprInterp.c; see its header
203comment for details.  Special-case evalfuncs are used for certain
204especially-simple expressions.
205
206Note that a lot of the more complex expression evaluation steps, which are
207less performance-critical than the simpler ones, are implemented as
208separate functions outside the fast-path of expression execution, allowing
209their implementation to be shared between interpreted and compiled
210expression evaluation.  This means that these helper functions are not
211allowed to perform expression step dispatch themselves, as the method of
212dispatch will vary based on the caller.  The helpers therefore cannot call
213for the execution of subexpressions; all subexpression results they need
214must be computed by earlier steps.  And dispatch to the following
215expression step must be performed after returning from the helper.
216
217
218Targetlist Evaluation
219---------------------
220
221ExecBuildProjectionInfo builds an ExprState that has the effect of
222evaluating a targetlist into ExprState->resultslot.  A generic targetlist
223expression is executed by evaluating it as discussed above (storing the
224result into the ExprState's resvalue/resnull fields) and then using an
225EEOP_ASSIGN_TMP step to move the result into the appropriate tts_values[]
226and tts_isnull[] array elements of the result slot.  There are special
227fast-path step types (EEOP_ASSIGN_*_VAR) to handle targetlist entries that
228are simple Vars using only one step instead of two.

231MERGE
232-----
233
234MERGE is a multiple-table, multiple-action command: It specifies a target
235table and a source relation, and can contain multiple WHEN MATCHED and
236WHEN NOT MATCHED clauses, each of which specifies one UPDATE, INSERT,
237DELETE, or DO NOTHING actions.  The target table is modified by MERGE,
238and the source relation supplies additional data for the actions.  Each action
239optionally specifies a qualifying expression that is evaluated for each tuple.
240
241In the planner, transform_MERGE_to_join constructs a join between the target
242table and the source relation, with row-identifying junk columns from the target
243table.  This join is an outer join if the MERGE command contains any WHEN NOT
244MATCHED clauses; the ModifyTable node fetches tuples from the plan tree of that
245join.  If the row-identifying columns in the fetched tuple are NULL, then the
246source relation contains a tuple that is not matched by any tuples in the
247target table, so the qualifying expression for each WHEN NOT MATCHED clause is
248evaluated given that tuple as returned by the plan.  If the expression returns
249true, the action indicated by the clause is executed, and no further clauses
250are evaluated.  On the other hand, if the row-identifying columns are not
251NULL, then the matching tuple from the target table can be fetched; qualifying
252expression of each WHEN MATCHED clause is evaluated given both the fetched
253tuple and the tuple returned by the plan.
254
255If no WHEN NOT MATCHED clauses are present, then the join constructed by
256the planner is an inner join, and the row-identifying junk columns are
257always non NULL.
258
259If WHEN MATCHED ends up processing a row that is concurrently updated or deleted,
260EvalPlanQual (see below) is used to find the latest version of the row, and
261that is re-fetched; if it exists, the search for a matching WHEN MATCHED clause
262to use starts at the top.
263
264MERGE does not allow its own type of triggers, but instead fires UPDATE, DELETE,
265and INSERT triggers: row triggers are fired for each row when an action is
266executed for that row.  Statement triggers are fired always, regardless of
267whether any rows match the corresponding clauses.

270Memory Management
271-----------------
272
273A "per query" memory context is created during CreateExecutorState();
274all storage allocated during an executor invocation is allocated in that
275context or a child context.  This allows easy reclamation of storage
276during executor shutdown --- rather than messing with retail pfree's and
277probable storage leaks, we just destroy the memory context.
278
279In particular, the plan state trees and expression state trees described
280in the previous section are allocated in the per-query memory context.
281
282To avoid intra-query memory leaks, most processing while a query runs
283is done in "per tuple" memory contexts, which are so-called because they
284are typically reset to empty once per tuple.  Per-tuple contexts are usually
285associated with ExprContexts, and commonly each PlanState node has its own
286ExprContext to evaluate its qual and targetlist expressions in.
287

289Query Processing Control Flow
290-----------------------------
291
292This is a sketch of control flow for full query processing:
293
294	CreateQueryDesc
295
296	ExecutorStart
297		CreateExecutorState
298			creates per-query context
299		switch to per-query context to run ExecInitNode
300		AfterTriggerBeginQuery
301		ExecInitNode --- recursively scans plan tree
302			ExecInitNode
303				recurse into subsidiary nodes
304			CreateExprContext
305				creates per-tuple context
306			ExecInitExpr
307
308	ExecutorRun
309		ExecProcNode --- recursively called in per-query context
310			ExecEvalExpr --- called in per-tuple context
311			ResetExprContext --- to free memory
312
313	ExecutorFinish
314		ExecPostprocessPlan --- run any unfinished ModifyTable nodes
315		AfterTriggerEndQuery
316
317	ExecutorEnd
318		ExecEndNode --- recursively releases resources
319		FreeExecutorState
320			frees per-query context and child contexts
321
322	FreeQueryDesc
323
324Per above comments, it's not really critical for ExecEndNode to free any
325memory; it'll all go away in FreeExecutorState anyway.  However, we do need to
326be careful to close relations, drop buffer pins, etc, so we do need to scan
327the plan state tree to find these sorts of resources.
328
329
330The executor can also be used to evaluate simple expressions without any Plan
331tree ("simple" meaning "no aggregates and no sub-selects", though such might
332be hidden inside function calls).  This case has a flow of control like
333
334	CreateExecutorState
335		creates per-query context
336
337	CreateExprContext	-- or use GetPerTupleExprContext(estate)
338		creates per-tuple context
339
340	ExecPrepareExpr
341		temporarily switch to per-query context
342		run the expression through expression_planner
343		ExecInitExpr
344
345	Repeatedly do:
346		ExecEvalExprSwitchContext
347			ExecEvalExpr --- called in per-tuple context
348		ResetExprContext --- to free memory
349
350	FreeExecutorState
351		frees per-query context, as well as ExprContext
352		(a separate FreeExprContext call is not necessary)

355EvalPlanQual (READ COMMITTED Update Checking)
356---------------------------------------------
357
358For simple SELECTs, the executor need only pay attention to tuples that are
359valid according to the snapshot seen by the current transaction (ie, they
360were inserted by a previously committed transaction, and not deleted by any
361previously committed transaction).  However, for UPDATE, DELETE, and MERGE it
362is not cool to modify or delete a tuple that's been modified by an open or
363concurrently-committed transaction.  If we are running in SERIALIZABLE
364isolation level then we just raise an error when this condition is seen to
365occur.  In READ COMMITTED isolation level, we must work a lot harder.
366
367The basic idea in READ COMMITTED mode is to take the modified tuple
368committed by the concurrent transaction (after waiting for it to commit,
369if need be) and re-evaluate the query qualifications to see if it would
370still meet the quals.  If so, we regenerate the updated tuple (if we are
371doing an UPDATE) from the modified tuple, and finally update/delete the
372modified tuple.  SELECT FOR UPDATE/SHARE behaves similarly, except that its
373action is just to lock the modified tuple and return results based on that
374version of the tuple.
375
376To implement this checking, we actually re-run the query from scratch for
377each modified tuple (or set of tuples, for SELECT FOR UPDATE), with the
378relation scan nodes tweaked to return only the current tuples --- either
379the original ones, or the updated (and now locked) versions of the modified
380tuple(s).  If this query returns a tuple, then the modified tuple(s) pass
381the quals (and the query output is the suitably modified update tuple, if
382we're doing UPDATE).  If no tuple is returned, then the modified tuple(s)
383fail the quals, so we ignore the current result tuple and continue the
384original query.
385
386In UPDATE/DELETE/MERGE, only the target relation needs to be handled this way.
387In SELECT FOR UPDATE, there may be multiple relations flagged FOR UPDATE,
388so we obtain lock on the current tuple version in each such relation before
389executing the recheck.
390
391It is also possible that there are relations in the query that are not
392to be locked (they are neither the UPDATE/DELETE/MERGE target nor specified
393to be locked in SELECT FOR UPDATE/SHARE).  When re-running the test query
394we want to use the same rows from these relations that were joined to
395the locked rows.  For ordinary relations this can be implemented relatively
396cheaply by including the row TID in the join outputs and re-fetching that
397TID.  (The re-fetch is expensive, but we're trying to optimize the normal
398case where no re-test is needed.)  We have also to consider non-table
399relations, such as a ValuesScan or FunctionScan.  For these, since there
400is no equivalent of TID, the only practical solution seems to be to include
401the entire row value in the join output row.
402
403We disallow set-returning functions in the targetlist of SELECT FOR UPDATE,
404so as to ensure that at most one tuple can be returned for any particular
405set of scan tuples.  Otherwise we'd get duplicates due to the original
406query returning the same set of scan tuples multiple times.  Likewise,
407SRFs are disallowed in an UPDATE's targetlist.  There, they would have the
408effect of the same row being updated multiple times, which is not very
409useful --- and updates after the first would have no effect anyway.

412Asynchronous Execution
413----------------------
414
415In cases where a node is waiting on an event external to the database system,
416such as a ForeignScan awaiting network I/O, it's desirable for the node to
417indicate that it cannot return any tuple immediately but may be able to do so
418at a later time.  A process which discovers this type of situation can always
419handle it simply by blocking, but this may waste time that could be spent
420executing some other part of the plan tree where progress could be made
421immediately.  This is particularly likely to occur when the plan tree contains
422an Append node.  Asynchronous execution runs multiple parts of an Append node
423concurrently rather than serially to improve performance.
424
425For asynchronous execution, an Append node must first request a tuple from an
426async-capable child node using ExecAsyncRequest.  Next, it must execute the
427asynchronous event loop using ExecAppendAsyncEventWait.  Eventually, when a
428child node to which an asynchronous request has been made produces a tuple,
429the Append node will receive it from the event loop via ExecAsyncResponse.  In
430the current implementation of asynchronous execution, the only node type that
431requests tuples from an async-capable child node is an Append, while the only
432node type that might be async-capable is a ForeignScan.
433
434Typically, the ExecAsyncResponse callback is the only one required for nodes
435that wish to request tuples asynchronously.  On the other hand, async-capable
436nodes generally need to implement three methods:
437
4381. When an asynchronous request is made, the node's ExecAsyncRequest callback
439   will be invoked; it should use ExecAsyncRequestPending to indicate that the
440   request is pending for a callback described below.  Alternatively, it can
441   instead use ExecAsyncRequestDone if a result is available immediately.
442
4432. When the event loop wishes to wait or poll for file descriptor events, the
444   node's ExecAsyncConfigureWait callback will be invoked to configure the
445   file descriptor event for which the node wishes to wait.
446
4473. When the file descriptor becomes ready, the node's ExecAsyncNotify callback
448   will be invoked; like #1, it should use ExecAsyncRequestPending for another
449   callback or ExecAsyncRequestDone to return a result immediately.
