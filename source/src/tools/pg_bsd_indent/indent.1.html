96.Sh DESCRIPTION
97The
98.Nm
99utility is a
100.Em C
101program formatter.
102It reformats the
103.Em C
104program in the
105.Ar input-file
106according to the switches.
107The switches which can be
108specified are described below.
109They may appear before or after the file
110names.
111.Pp
112.Sy NOTE :
113If you only specify an
114.Ar input-file ,
115the formatting is
116done `in-place', that is, the formatted file is written back into
117.Ar input-file
118and a backup copy of
119.Ar input-file
120is written in the current directory.
121If
122.Ar input-file
123is named
124.Sq Pa /blah/blah/file ,
125the backup file is named
126.Sq Pa file.BAK .
127.Pp
128If
129.Ar output-file
130is specified,
131.Nm
132checks to make sure that it is different from
133.Ar input-file .
134.Pp
135The options listed below control the formatting style imposed by
136.Nm .
137.Bl -tag -width Op
138.It Fl bacc , nbacc
139If
140.Fl bacc
141is specified, a blank line is forced around every conditional
142compilation block.
143For example, in front of every #ifdef and after every #endif.
144Other blank lines surrounding such blocks will be swallowed.
145Default:
146.Fl nbacc  .
147.It Fl bad , nbad
148If
149.Fl bad
150is specified, a blank line is forced after every block of
151declarations.
152Default:
153.Fl nbad .
154.It Fl badp , nbadp
155This is vaguely similar to
156.Fl bad
157except that it only applies to the first set of declarations
158in a procedure (just after the first `{') and it causes a blank
159line to be generated even if there are no declarations.
160The default is
161.Fl nbadp.
162.It Fl bap , nbap
163If
164.Fl bap
165is specified, a blank line is forced after every procedure body.
166Default:
167.Fl nbap .
168.It Fl bbb , nbbb
169If
170.Fl bbb
171is specified, a blank line is forced before every block comment.
172Default:
173.Fl nbbb .
174.It Fl \&bc , nbc
175If
176.Fl \&bc
177is specified, then a newline is forced after each comma in a declaration.
178.Fl nbc
179turns off this option.
180Default:
181.Fl \&nbc .
182.It Fl \&bl , \&br
183Specifying
184.Fl \&bl
185lines up compound statements like this:
186.Bd -literal -offset indent
187if (...)
188{
189  code
190}
191.Ed
192.Pp
193Specifying
194.Fl \&br
195(the default) makes them look like this:
196.Bd -literal -offset indent
197if (...) {
198  code
199}
200.Ed
201.It Fl bs , nbs
202Whether a blank should always be inserted after sizeof.
203The default is
204.Fl nbs.
205.It Fl c Ns Ar n
206The column in which comments on code start.
207The default is 33.
208.It Fl cp Ns Ar n
209The column in which comments on #else and #endif start.
210The default is 0, which effectively separates the directives from comments by
211a single space.
212.It Fl cd Ns Ar n
213The column in which comments on declarations start.
214The default
215is for these comments to start in the same column as those on code.
216.It Fl cdb , ncdb
217Enables (disables) the placement of comment delimiters on blank lines.
218With
219this option enabled, comments look like this:
220.Bd -literal -offset indent
221	/*
222	 * this is a comment
223	 */
224.Ed
225.Pp
226Rather than like this:
227.Bd -literal -offset indent
228	/* this is a comment */
229.Ed
230.Pp
231This only affects block comments, not comments to the right of
232code.
233The default is
234.Fl cdb .
235.It Fl ce , nce
236Enables (disables) forcing of `else's to cuddle up to the immediately preceding
237`}'.
238The default is
239.Fl \&ce .
240.It Fl \&ci Ns Ar n
241Sets the continuation indent to be
242.Ar n .
243Continuation
244lines will be indented that far from the beginning of the first line of the
245statement.
246Parenthesized expressions have extra indentation added to
247indicate the nesting, unless
248.Fl \&lp
249is in effect
250or the continuation indent is exactly half of the main indent.
251.Fl \&ci
252defaults to the same value as
253.Fl i .
254.It Fl cli Ns Ar n
255Causes case labels to be indented
256.Ar n
257tab stops to the right of the containing
258.Ic switch
259statement.
260.Fl cli0.5
261causes case labels to be indented half a tab stop.
262The
263default is
264.Fl cli0 .
265.It Fl d Ns Ar n
266Controls the placement of comments which are not to the
267right of code.
268For example,
269.Fl \&d\&1
270means that such comments are placed one indentation level to the
271left of code.
272Specifying the default
273.Fl \&d\&0
274lines up these comments with the code.
275See the section on comment
276indentation below.
277.It Fl \&di Ns Ar n
278Specifies the indentation, in character positions,
279of global variable names and all struct/union member names
280relative to the beginning of their type declaration.
281The default is
282.Fl di16 .
283.It Fl dj , ndj
284.Fl \&dj
285left justifies declarations.
286.Fl ndj
287indents declarations the same as code.
288The default is
289.Fl ndj .
290.It Fl \&ei , nei
291Enables (disables) special
292.Ic else-if
293processing.
294If it is enabled, an
295.Ic if
296following an
297.Ic else
298will have the same indentation as the preceding
299.Ic \&if
300statement.
301The default is
302.Fl ei .
303.It Fl eei , neei
304Enables (disables) extra indentation on continuation lines of
305the expression part of
306.Ic if
307and
308.Ic while
309statements.
310These continuation lines will be indented one extra level.
311The default is
312.Fl neei .
313.It Fl fbs , nfbs
314Enables (disables) splitting the function declaration and opening brace
315across two lines.
316The default is
317.Fl fbs .
318.It Fl fc1 , nfc1
319Enables (disables) the formatting of comments that start in column 1.
320Often, comments whose leading `/' is in column 1 have been carefully
321hand formatted by the programmer.
322In such cases,
323.Fl nfc1
324should be
325used.
326The default is
327.Fl fc1 .
328.It Fl fcb , nfcb
329Enables (disables) the formatting of block comments (ones that begin
330with `/*\\n').
331Often, block comments have been not so carefully hand formatted by the
332programmer, but reformatting that would just change the line breaks is not
333wanted.
334In such cases,
335.Fl nfcb
336should be used.
337Block comments are then handled like box comments.
338The default is
339.Fl fcb .
340.It Fl i Ns Ar n
341The number of columns for one indentation level.
342The default is 8.
343.It Fl \&ip , nip
344Enables (disables) the indentation of parameter declarations from the left
345margin.
346The default is
347.Fl \&ip .
348.It Fl l Ns Ar n
349Maximum length of an output line.
350The default is 78.
351.It Fl lc Ns Ar n
352Maximum length of an output line in a block comment.
353The default is 0, which means to limit block comment lines in accordance with
354.Fl l.
355.It Fl \&ldi Ns Ar n
356Specifies the indentation, in character positions,
357of local variable names
358relative to the beginning of their type declaration.
359The default is for local variable names to be indented
360by the same amount as global ones.
361.It Fl \&lp , nlp
362Lines up code surrounded by parentheses in continuation lines.
363With
364.Fl \&lp ,
365if a line
366has a left paren which is not closed on that line, then continuation lines
367will be lined up to start at the character position just after the left
368paren.
369For example, here is how a piece of continued code looks with
370.Fl nlp
371in effect:
372.Bd -literal -offset indent
373p1 = first_procedure(second_procedure(p2, p3),
374\ \ third_procedure(p4, p5));
375.Ed
376.Pp
377With
378.Fl lp
379in effect (the default) the code looks somewhat clearer:
380.Bd -literal -offset indent
381p1\ =\ first_procedure(second_procedure(p2,\ p3),
382\ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ third_procedure(p4,\ p5));
383.Ed
384.Pp
385Inserting two more newlines we get:
386.Bd -literal -offset indent
387p1\ =\ first_procedure(second_procedure(p2,
388\ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ p3),
389\ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ third_procedure(p4,
390\ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ p5));
391.Ed
392.It Fl \&lpl , nlpl
393With
394.Fl \&lpl ,
395code surrounded by parentheses in continuation lines is lined up even if it
396would extend past the right margin.
397With
398.Fl \&nlpl
399(the default), such a line that would extend past the right margin is moved
400left to keep it within the margin, if that does not require placing it to
401the left of the prevailing indentation level.
402These switches have no effect if
403.Fl nlp
404is selected.
405.It Fl npro
406Causes the profile files,
407.Sq Pa ./.indent.pro
408and
409.Sq Pa ~/.indent.pro ,
410to be ignored.
411.It Fl P Ns Ar file
412Read profile from
413.Ar file .
414.It Fl pcs , npcs
415If true
416.Pq Fl pcs
417all procedure calls will have a space inserted between
418the name and the `('.
419The default is
420.Fl npcs .
421.It Fl psl , npsl
422If true
423.Pq Fl psl
424the names of procedures being defined are placed in
425column 1 \- their types, if any, will be left on the previous lines.
426The
427default is
428.Fl psl .
429.It Fl sac , nsac
430Control whether parenthesized type names in casts are followed by a space or
431not.
432The default is
433.Fl nsac .
434.It Fl \&sc , nsc
435Enables (disables) the placement of asterisks (`*'s) at the left edge of all
436comments.
437The default is
438.Fl sc .
439.It Fl sob , nsob
440If
441.Fl sob
442is specified, indent will swallow optional blank lines.
443You can use this to
444get rid of blank lines after declarations.
445Default:
446.Fl nsob .
447.It Fl \&st
448Causes
449.Nm
450to take its input from stdin and put its output to stdout.
451.It Fl ta
452Automatically add all identifiers ending in "_t" to the list
453of type keywords.
454.It Fl T Ns Ar typename
455Adds
456.Ar typename
457to the list of type keywords.
458Names accumulate:
459.Fl T
460can be specified more than once.
461You need to specify all the typenames that
462appear in your program that are defined by
463.Ic typedef
464\- nothing will be
465harmed if you miss a few, but the program will not be formatted as nicely as
466it should.
467This sounds like a painful thing to have to do, but it is really
468a symptom of a problem in C:
469.Ic typedef
470causes a syntactic change in the
471language and
472.Nm
473cannot find all
474instances of
475.Ic typedef .
476.It Fl tpg , ntpg
477If
478.Fl tpg
479is specified, follow Postgres rules about when to use spaces versus
480tabs for indentation, that is, use a space instead of a tab if the
481tab would move only one column and no tab will follow it.
482Default:
483.Fl ntpg .
484.It Fl ts Ns Ar n
485Assumed distance between tab stops.
486The default is 8.
487.It Fl U Ns Ar file
488Adds type names from
489.Ar file
490to the list of type keywords.
491.It Fl ut , nut
492Enables (disables) the use of tab characters in the output.
493The default is
494.Fl ut .
495.It Fl v , \&nv
496.Fl v
497turns on `verbose' mode;
498.Fl \&nv
499turns it off.
500When in verbose mode,
501.Nm
502reports when it splits one line of input into two or more lines of output,
503and gives some size statistics at completion.
504The default is
505.Fl \&nv .
506.It Fl -version
507Causes
508.Nm
509to print its version number and exit.
510.El
511.Pp
512You may set up your own `profile' of defaults to
513.Nm
514by creating a file called
515.Pa .indent.pro
516in your login directory and/or the current directory and including
517whatever switches you like.
518A `.indent.pro' in the current directory takes
519precedence over the one in your login directory.
520If
521.Nm
522is run and a profile file exists, then it is read to set up the program's
523defaults.
524Switches on the command line, though, always override profile
525switches.
526The switches should be separated by spaces, tabs or newlines.
527.Pp
528.Ss Comments
529.Sq Em Box
530.Em comments .
531The
532.Nm
533utility
534assumes that any comment with a dash or star immediately after the start of
535comment (that is, `/*\-' or `/**') is a comment surrounded by a box of stars.
536Each line of such a comment is left unchanged, except that its indentation
537may be adjusted to account for the change in indentation of the first line
538of the comment.
539.Pp
540.Em Straight text .
541All other comments are treated as straight text.
542The
543.Nm
544utility fits as many words (separated by blanks, tabs, or newlines) on a
545line as possible.
546Blank lines break paragraphs.
547.Ss Comment indentation
548If a comment is on a line with code it is started in the `comment column',
549which is set by the
550.Fl c Ns Ns Ar n
551command line parameter.
552Otherwise, the comment is started at
553.Ar n
554indentation levels less than where code is currently being placed, where
555.Ar n
556is specified by the
557.Fl d Ns Ns Ar n
558command line parameter.
559If the code on a line extends past the comment
560column, the comment starts further to the right, and the right margin may be
561automatically extended in extreme cases.
562.Ss Preprocessor lines
563In general,
564.Nm
565leaves preprocessor lines alone.
566The only
567reformatting that it will do is to straighten up trailing comments.
568It
569leaves embedded comments alone.
570Conditional compilation
571.Pq Ic #ifdef...#endif
572is recognized and
573.Nm
574attempts to correctly
575compensate for the syntactic peculiarities introduced.
576.Ss C syntax
577The
578.Nm
579utility understands a substantial amount about the syntax of C, but it
580has a `forgiving' parser.
581It attempts to cope with the usual sorts of
582incomplete and malformed syntax.
583In particular, the use of macros like:
584.Pp
585.Dl #define forever for(;;)
586.Pp
587is handled properly.
588.Sh ENVIRONMENT
589The
590.Nm
591utility uses the
592.Ev HOME
593environment variable.
