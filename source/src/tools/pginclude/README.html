3This directory contains some scripts and information for managing
4header files and includes in PostgreSQL source code.

7include-what-you-use
8====================
9
10include-what-you-use (IWYU) (<https://include-what-you-use.org/>) is a
11tool for finding missing or superfluous includes in C source files.
12
13With a compilation database (compile_commands.json, produced by
14meson), it can be run like this, over the whole source tree:
15
16    iwyu_tool.py -p build .
17
18(this will likely be very noisy) or for individual files:
19
20    iwyu_tool.py -p build src/bin/psql/startup.c
21
22Various other invocation options are available.
23
24It is recommended to use at least version 0.23.  Earlier versions give
25advice that is incompatible with the compiler warning option
26-Wmissing-variable-declarations.
27
28clangd (the language server) can automatically give IWYU-style advice;
29see <https://clangd.llvm.org/guides/include-cleaner>.
30
31The source code contains some "IWYU pragma" comments to tell IWYU
32about some PostgreSQL include file conventions (such as that a header
33such as "postgres.h" should always be included, even if it doesn't
34contribute any symbols used by the particular source file) and to
35silence a few warnings that are difficult to fix.  See
36<https://github.com/include-what-you-use/include-what-you-use/blob/master/docs/IWYUPragmas.md>
37for documentation about those.
38
39Of course, any include changes suggested by this or other tools should
40be checked and verified carefully.  Note that some includes are only
41used on some platforms or with some compilation options, so blindly
42following the produced advice is not recommended.

45headerscheck
46============
47
48This script can be run to verify that all Postgres include files meet
49the project convention that they will compile "standalone", that is
50with no prerequisite headers other than postgres.h (or postgres_fe.h
51or c.h, as appropriate).
52
53A small number of header files are exempted from this requirement,
54and are skipped by the headerscheck script.
55
56The easy way to run the script is to say "make -s headerscheck" in
57the top-level build directory after completing a build.  You should
58have included "--with-perl --with-python" in your configure options,
59else you're likely to get errors about related headers not being found.
60
61A limitation of the current script is that it doesn't know exactly which
62headers are for frontend or backend; when in doubt it uses postgres.h as
63prerequisite, even if postgres_fe.h or c.h would be more appropriate.
64Also note that the contents of macros are not checked; this is intentional.

67headerscheck --cplusplus
68========================
69
70The headerscheck in --cplusplus mode can be run to verify that all
71Postgres include files meet the project convention that they will
72compile as C++ code.  Although the project's coding language is C,
73some people write extensions in C++, so it's helpful for include files
74to be C++-clean.
75
76A small number of header files are exempted from this requirement,
77and are skipped by the script in the --cplusplus mode.
78
79The easy way to run the script is to say "make -s cpluspluscheck" in
80the top-level build directory after completing a build.  You should
81have included "--with-perl --with-python" in your configure options,
82else you're likely to get errors about related headers not being found.
83
84If you are using a non-g++-compatible C++ compiler, you may need to
85override the script's CXXFLAGS setting by setting a suitable environment
86value.
87
88A limitation of the current script is that it doesn't know exactly which
89headers are for frontend or backend; when in doubt it uses postgres.h as
90prerequisite, even if postgres_fe.h or c.h would be more appropriate.
91Also note that the contents of macros are not checked; this is intentional.
