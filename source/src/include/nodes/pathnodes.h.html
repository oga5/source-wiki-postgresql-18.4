630/*----------
631 * RelOptInfo
632 *		Per-relation information for planning/optimization
633 *
634 * For planning purposes, a "base rel" is either a plain relation (a table)
635 * or the output of a sub-SELECT or function that appears in the range table.
636 * In either case it is uniquely identified by an RT index.  A "joinrel"
637 * is the joining of two or more base rels.  A joinrel is identified by
638 * the set of RT indexes for its component baserels, along with RT indexes
639 * for any outer joins it has computed.  We create RelOptInfo nodes for each
640 * baserel and joinrel, and store them in the PlannerInfo's simple_rel_array
641 * and join_rel_list respectively.
642 *
643 * Note that there is only one joinrel for any given set of component
644 * baserels, no matter what order we assemble them in; so an unordered
645 * set is the right datatype to identify it with.
646 *
647 * We also have "other rels", which are like base rels in that they refer to
648 * single RT indexes; but they are not part of the join tree, and are given
649 * a different RelOptKind to identify them.
650 * Currently the only kind of otherrels are those made for member relations
651 * of an "append relation", that is an inheritance set or UNION ALL subquery.
652 * An append relation has a parent RTE that is a base rel, which represents
653 * the entire append relation.  The member RTEs are otherrels.  The parent
654 * is present in the query join tree but the members are not.  The member
655 * RTEs and otherrels are used to plan the scans of the individual tables or
656 * subqueries of the append set; then the parent baserel is given Append
657 * and/or MergeAppend paths comprising the best paths for the individual
658 * member rels.  (See comments for AppendRelInfo for more information.)
659 *
660 * At one time we also made otherrels to represent join RTEs, for use in
661 * handling join alias Vars.  Currently this is not needed because all join
662 * alias Vars are expanded to non-aliased form during preprocess_expression.
663 *
664 * We also have relations representing joins between child relations of
665 * different partitioned tables. These relations are not added to
666 * join_rel_level lists as they are not joined directly by the dynamic
667 * programming algorithm.
668 *
669 * There is also a RelOptKind for "upper" relations, which are RelOptInfos
670 * that describe post-scan/join processing steps, such as aggregation.
671 * Many of the fields in these RelOptInfos are meaningless, but their Path
672 * fields always hold Paths showing ways to do that processing step.
673 *
674 * Parts of this data structure are specific to various scan and join
675 * mechanisms.  It didn't seem worth creating new node types for them.
676 *
677 *		relids - Set of relation identifiers (RT indexes).  This is a base
678 *				 relation if there is just one, a join relation if more;
679 *				 in the join case, RT indexes of any outer joins formed
680 *				 at or below this join are included along with baserels
681 *		rows - estimated number of tuples in the relation after restriction
682 *			   clauses have been applied (ie, output rows of a plan for it)
683 *		consider_startup - true if there is any value in keeping plain paths for
684 *						   this rel on the basis of having cheap startup cost
685 *		consider_param_startup - the same for parameterized paths
686 *		reltarget - Default Path output tlist for this rel; normally contains
687 *					Var and PlaceHolderVar nodes for the values we need to
688 *					output from this relation.
689 *					List is in no particular order, but all rels of an
690 *					appendrel set must use corresponding orders.
691 *					NOTE: in an appendrel child relation, may contain
692 *					arbitrary expressions pulled up from a subquery!
693 *		pathlist - List of Path nodes, one for each potentially useful
694 *				   method of generating the relation
695 *		ppilist - ParamPathInfo nodes for parameterized Paths, if any
696 *		cheapest_startup_path - the pathlist member with lowest startup cost
697 *			(regardless of ordering) among the unparameterized paths;
698 *			or NULL if there is no unparameterized path
699 *		cheapest_total_path - the pathlist member with lowest total cost
700 *			(regardless of ordering) among the unparameterized paths;
701 *			or if there is no unparameterized path, the path with lowest
702 *			total cost among the paths with minimum parameterization
703 *		cheapest_unique_path - for caching cheapest path to produce unique
704 *			(no duplicates) output from relation; NULL if not yet requested
705 *		cheapest_parameterized_paths - best paths for their parameterizations;
706 *			always includes cheapest_total_path, even if that's unparameterized
707 *		direct_lateral_relids - rels this rel has direct LATERAL references to
708 *		lateral_relids - required outer rels for LATERAL, as a Relids set
709 *			(includes both direct and indirect lateral references)
710 *
711 * If the relation is a base relation it will have these fields set:
712 *
713 *		relid - RTE index (this is redundant with the relids field, but
714 *				is provided for convenience of access)
715 *		rtekind - copy of RTE's rtekind field
716 *		min_attr, max_attr - range of valid AttrNumbers for rel
717 *		attr_needed - array of bitmapsets indicating the highest joinrel
718 *				in which each attribute is needed; if bit 0 is set then
719 *				the attribute is needed as part of final targetlist
720 *		attr_widths - cache space for per-attribute width estimates;
721 *					  zero means not computed yet
722 *		nulling_relids - relids of outer joins that can null this rel
723 *		lateral_vars - lateral cross-references of rel, if any (list of
724 *					   Vars and PlaceHolderVars)
725 *		lateral_referencers - relids of rels that reference this one laterally
726 *				(includes both direct and indirect lateral references)
727 *		indexlist - list of IndexOptInfo nodes for relation's indexes
728 *					(always NIL if it's not a table or partitioned table)
729 *		pages - number of disk pages in relation (zero if not a table)
730 *		tuples - number of tuples in relation (not considering restrictions)
731 *		allvisfrac - fraction of disk pages that are marked all-visible
732 *		eclass_indexes - EquivalenceClasses that mention this rel (filled
733 *						 only after EC merging is complete)
734 *		subroot - PlannerInfo for subquery (NULL if it's not a subquery)
735 *		subplan_params - list of PlannerParamItems to be passed to subquery
736 *
737 *		Note: for a subquery, tuples and subroot are not set immediately
738 *		upon creation of the RelOptInfo object; they are filled in when
739 *		set_subquery_pathlist processes the object.
740 *
741 *		For otherrels that are appendrel members, these fields are filled
742 *		in just as for a baserel, except we don't bother with lateral_vars.
743 *
744 * If the relation is either a foreign table or a join of foreign tables that
745 * all belong to the same foreign server and are assigned to the same user to
746 * check access permissions as (cf checkAsUser), these fields will be set:
747 *
748 *		serverid - OID of foreign server, if foreign table (else InvalidOid)
749 *		userid - OID of user to check access as (InvalidOid means current user)
750 *		useridiscurrent - we've assumed that userid equals current user
751 *		fdwroutine - function hooks for FDW, if foreign table (else NULL)
752 *		fdw_private - private state for FDW, if foreign table (else NULL)
753 *
754 * Two fields are used to cache knowledge acquired during the join search
755 * about whether this rel is provably unique when being joined to given other
756 * relation(s), ie, it can have at most one row matching any given row from
757 * that join relation.  Currently we only attempt such proofs, and thus only
758 * populate these fields, for base rels; but someday they might be used for
759 * join rels too:
760 *
761 *		unique_for_rels - list of UniqueRelInfo, each one being a set of other
762 *					rels for which this one has been proven unique
763 *		non_unique_for_rels - list of Relid sets, each one being a set of
764 *					other rels for which we have tried and failed to prove
765 *					this one unique
766 *
767 * The presence of the following fields depends on the restrictions
768 * and joins that the relation participates in:
769 *
770 *		baserestrictinfo - List of RestrictInfo nodes, containing info about
771 *					each non-join qualification clause in which this relation
772 *					participates (only used for base rels)
773 *		baserestrictcost - Estimated cost of evaluating the baserestrictinfo
774 *					clauses at a single tuple (only used for base rels)
775 *		baserestrict_min_security - Smallest security_level found among
776 *					clauses in baserestrictinfo
777 *		joininfo  - List of RestrictInfo nodes, containing info about each
778 *					join clause in which this relation participates (but
779 *					note this excludes clauses that might be derivable from
780 *					EquivalenceClasses)
781 *		has_eclass_joins - flag that EquivalenceClass joins are possible
782 *
783 * Note: Keeping a restrictinfo list in the RelOptInfo is useful only for
784 * base rels, because for a join rel the set of clauses that are treated as
785 * restrict clauses varies depending on which sub-relations we choose to join.
786 * (For example, in a 3-base-rel join, a clause relating rels 1 and 2 must be
787 * treated as a restrictclause if we join {1} and {2 3} to make {1 2 3}; but
788 * if we join {1 2} and {3} then that clause will be a restrictclause in {1 2}
789 * and should not be processed again at the level of {1 2 3}.)	Therefore,
790 * the restrictinfo list in the join case appears in individual JoinPaths
791 * (field joinrestrictinfo), not in the parent relation.  But it's OK for
792 * the RelOptInfo to store the joininfo list, because that is the same
793 * for a given rel no matter how we form it.
794 *
795 * We store baserestrictcost in the RelOptInfo (for base relations) because
796 * we know we will need it at least once (to price the sequential scan)
797 * and may need it multiple times to price index scans.
798 *
799 * A join relation is considered to be partitioned if it is formed from a
800 * join of two relations that are partitioned, have matching partitioning
801 * schemes, and are joined on an equijoin of the partitioning columns.
802 * Under those conditions we can consider the join relation to be partitioned
803 * by either relation's partitioning keys, though some care is needed if
804 * either relation can be forced to null by outer-joining.  For example, an
805 * outer join like (A LEFT JOIN B ON A.a = B.b) may produce rows with B.b
806 * NULL.  These rows may not fit the partitioning conditions imposed on B.
807 * Hence, strictly speaking, the join is not partitioned by B.b and thus
808 * partition keys of an outer join should include partition key expressions
809 * from the non-nullable side only.  However, if a subsequent join uses
810 * strict comparison operators (and all commonly-used equijoin operators are
811 * strict), the presence of nulls doesn't cause a problem: such rows couldn't
812 * match anything on the other side and thus they don't create a need to do
813 * any cross-partition sub-joins.  Hence we can treat such values as still
814 * partitioning the join output for the purpose of additional partitionwise
815 * joining, so long as a strict join operator is used by the next join.
816 *
817 * If the relation is partitioned, these fields will be set:
818 *
819 *		part_scheme - Partitioning scheme of the relation
820 *		nparts - Number of partitions
821 *		boundinfo - Partition bounds
822 *		partbounds_merged - true if partition bounds are merged ones
823 *		partition_qual - Partition constraint if not the root
824 *		part_rels - RelOptInfos for each partition
825 *		all_partrels - Relids set of all partition relids
826 *		partexprs, nullable_partexprs - Partition key expressions
827 *
828 * The partexprs and nullable_partexprs arrays each contain
829 * part_scheme->partnatts elements.  Each of the elements is a list of
830 * partition key expressions.  For partitioned base relations, there is one
831 * expression in each partexprs element, and nullable_partexprs is empty.
832 * For partitioned join relations, each base relation within the join
833 * contributes one partition key expression per partitioning column;
834 * that expression goes in the partexprs[i] list if the base relation
835 * is not nullable by this join or any lower outer join, or in the
836 * nullable_partexprs[i] list if the base relation is nullable.
837 * Furthermore, FULL JOINs add extra nullable_partexprs expressions
838 * corresponding to COALESCE expressions of the left and right join columns,
839 * to simplify matching join clauses to those lists.
840 *
841 * Not all fields are printed.  (In some cases, there is no print support for
842 * the field type.)
843 *----------
844 */

1324/*
1325 * JoinDomains
1326 *
1327 * A "join domain" defines the scope of applicability of deductions made via
1328 * the EquivalenceClass mechanism.  Roughly speaking, a join domain is a set
1329 * of base+OJ relations that are inner-joined together.  More precisely, it is
1330 * the set of relations at which equalities deduced from an EquivalenceClass
1331 * can be enforced or should be expected to hold.  The topmost JoinDomain
1332 * covers the whole query (so its jd_relids should equal all_query_rels).
1333 * An outer join creates a new JoinDomain that includes all base+OJ relids
1334 * within its nullable side, but (by convention) not the OJ's own relid.
1335 * A FULL join creates two new JoinDomains, one for each side.
1336 *

1338 * to multiple join domains.  However, any of its Vars that appear in
1339 * EquivalenceClasses belonging to higher join domains will have nullingrel
1340 * bits preventing them from being evaluated at the rel's scan level, so that
1341 * we will not be able to derive enforceable-at-the-rel-scan-level clauses
1342 * from such ECs.  We define the join domain relid sets this way so that
1343 * domains can be said to be "higher" or "lower" when one domain relid set
1344 * includes another.
1345 *
1346 * The JoinDomains for a query are computed in deconstruct_jointree.
1347 * We do not copy JoinDomain structs once made, so they can be compared
1348 * for equality by simple pointer equality.
1349 */

1359/*
1360 * EquivalenceClasses
1361 *
1362 * Whenever we identify a mergejoinable equality clause A = B that is
1363 * not an outer-join clause, we create an EquivalenceClass containing
1364 * the expressions A and B to record this knowledge.  If we later find another
1365 * equivalence B = C, we add C to the existing EquivalenceClass; this may
1366 * require merging two existing EquivalenceClasses.  At the end of the qual
1367 * distribution process, we have sets of values that are known all transitively
1368 * equal to each other, where "equal" is according to the rules of the btree
1369 * operator family(s) shown in ec_opfamilies, as well as the collation shown
1370 * by ec_collation.  (We restrict an EC to contain only equalities whose
1371 * operators belong to the same set of opfamilies.  This could probably be
1372 * relaxed, but for now it's not worth the trouble, since nearly all equality
1373 * operators belong to only one btree opclass anyway.  Similarly, we suppose
1374 * that all or none of the input datatypes are collatable, so that a single
1375 * collation value is sufficient.)
1376 *
1377 * Strictly speaking, deductions from an EquivalenceClass hold only within
1378 * a "join domain", that is a set of relations that are innerjoined together
1379 * (see JoinDomain above).  For the most part we don't need to account for
1380 * this explicitly, because equality clauses from different join domains
1381 * will contain Vars that are not equal() because they have different
1382 * nullingrel sets, and thus we will never falsely merge ECs from different
1383 * join domains.  But Var-free (pseudoconstant) expressions lack that safety
1384 * feature.  We handle that by marking "const" EC members with the JoinDomain
1385 * of the clause they came from; two nominally-equal const members will be
1386 * considered different if they came from different JoinDomains.  This ensures
1387 * no false EquivalenceClass merges will occur.
1388 *
1389 * We also use EquivalenceClasses as the base structure for PathKeys, letting
1390 * us represent knowledge about different sort orderings being equivalent.
1391 * Since every PathKey must reference an EquivalenceClass, we will end up
1392 * with single-member EquivalenceClasses whenever a sort key expression has
1393 * not been equivalenced to anything else.  It is also possible that such an
1394 * EquivalenceClass will contain a volatile expression ("ORDER BY random()"),
1395 * which is a case that can't arise otherwise since clauses containing
1396 * volatile functions are never considered mergejoinable.  We mark such
1397 * EquivalenceClasses specially to prevent them from being merged with
1398 * ordinary EquivalenceClasses.  Also, for volatile expressions we have
1399 * to be careful to match the EquivalenceClass to the correct targetlist
1400 * entry: consider SELECT random() AS a, random() AS b ... ORDER BY b,a.
1401 * So we record the SortGroupRef of the originating sort clause.
1402 *
1403 * Derived equality clauses are stored in ec_derives_list. For small queries,
1404 * this list is scanned directly during lookup. For larger queries -- e.g.,
1405 * with many partitions or joins -- a hash table (ec_derives_hash) is built
1406 * when the list grows beyond a threshold, for faster lookup. When present,
1407 * the hash table contains the same RestrictInfos and is maintained alongside
1408 * the list. We retain the list even when the hash is used to simplify
1409 * serialization (e.g., in _outEquivalenceClass()) and support
1410 * EquivalenceClass merging.
1411 *
1412 * In contrast, ec_sources holds equality clauses that appear directly in the
1413 * query. These are typically few and do not require a hash table for lookup.
1414 *
1415 * 'ec_members' is a List of all !em_is_child EquivalenceMembers in the class.
1416 * EquivalenceMembers for any RELOPT_OTHER_MEMBER_REL and RELOPT_OTHER_JOINREL
1417 * relations are stored in the 'ec_childmembers' array in the index
1418 * corresponding to the relid, or first component relid in the case of
1419 * RELOPT_OTHER_JOINRELs.  'ec_childmembers' is NULL if the class has no child
1420 * EquivalenceMembers.
1421 *
1422 * For code wishing to look at EquivalenceMembers, if only parent-level
1423 * members are needed, then a simple foreach loop over ec_members is
1424 * sufficient.  When child members are also required, it is best to use the
1425 * functionality provided by EquivalenceMemberIterator.  This visits all
1426 * parent members and only the relevant child members.  The reason for this
1427 * is that large numbers of child EquivalenceMembers can exist in queries to
1428 * partitioned tables with many partitions.  The functionality provided by
1429 * EquivalenceMemberIterator allows efficient access to EquivalenceMembers
1430 * which belong to specific child relids.  See the header comments for
1431 * EquivalenceMemberIterator below for further details.
1432 *
1433 * NB: if ec_merged isn't NULL, this class has been merged into another, and
1434 * should be ignored in favor of using the pointed-to class.
1435 *
1436 * NB: EquivalenceClasses are never copied after creation.  Therefore,
1437 * copyObject() copies pointers to them as pointers, and equal() compares
1438 * pointers to EquivalenceClasses via pointer equality.  This is implemented
1439 * by putting copy_as_scalar and equal_as_scalar attributes on fields that
1440 * are pointers to EquivalenceClasses.  The same goes for EquivalenceMembers.
1441 */
1442typedef struct EquivalenceClass
1443{
1444	pg_node_attr(custom_read_write, no_copy_equal, no_read, no_query_jumble)
1445
1446	NodeTag		type;
1447
1448	List	   *ec_opfamilies;	/* btree operator family OIDs */
1449	Oid			ec_collation;	/* collation, if datatypes are collatable */
1450	int			ec_childmembers_size;	/* # elements in ec_childmembers */
1451	List	   *ec_members;		/* list of EquivalenceMembers */
1452	List	  **ec_childmembers;	/* array of Lists of child members */
1453	List	   *ec_sources;		/* list of generating RestrictInfos */
1454	List	   *ec_derives_list;	/* list of derived RestrictInfos */
1455	struct derives_hash *ec_derives_hash;	/* optional hash table for fast
1456											 * lookup; contains same
1457											 * RestrictInfos as list */
1458	Relids		ec_relids;		/* all relids appearing in ec_members, except
1459								 * for child members (see below) */
1460	bool		ec_has_const;	/* any pseudoconstants in ec_members? */
1461	bool		ec_has_volatile;	/* the (sole) member is a volatile expr */
1462	bool		ec_broken;		/* failed to generate needed clauses? */

1519/*
1520 * EquivalenceMemberIterator
1521 *
1522 * EquivalenceMemberIterator allows efficient access to sets of
1523 * EquivalenceMembers for callers which require access to child members.
1524 * Because partitioning workloads can result in large numbers of child
1525 * members, the child members are not stored in the EquivalenceClass's
1526 * ec_members List.  Instead, these are stored in the EquivalenceClass's

1528 * EquivalenceMemberIterator aims to provide efficient access to parent
1529 * members and child members belonging to specific child relids.

1532 * EquivalenceMemberIterator and that is via the setup_eclass_member_iterator

1535 * setup_eclass_member_iterator.  The calling function defines which
1536 * EquivalenceClass the iterator should be looking at and which child
1537 * relids to also return members for.  child_relids can be passed as NULL, but
1538 * the caller may as well just perform a foreach loop over ec_members as only
1539 * parent-level members will be returned in that case.

1542 * parent-level EquivalenceMembers are returned first, followed by all child
1543 * members for the specified 'child_relids' for all child members which were
1544 * indexed by any of the specified 'child_relids' in add_child_eq_member().

1854/*
1855 * Each IndexClause references a RestrictInfo node from the query's WHERE
1856 * or JOIN conditions, and shows how that restriction can be applied to
1857 * the particular index.  We support both indexclauses that are directly
1858 * usable by the index machinery, which are typically of the form
1859 * "indexcol OP pseudoconstant", and those from which an indexable qual
1860 * can be derived.  The simplest such transformation is that a clause
1861 * of the form "pseudoconstant OP indexcol" can be commuted to produce an
1862 * indexable qual (the index machinery expects the indexcol to be on the
1863 * left always).  Another example is that we might be able to extract an
1864 * indexable range condition from a LIKE condition, as in "x LIKE 'foo%bar'"
1865 * giving rise to "x >= 'foo' AND x < 'fop'".  Derivation of such lossy
1866 * conditions is done by a planner support function attached to the
1867 * indexclause's top-level function or operator.
1868 *

1874 * indexquals are semantically equivalent to the original clause, or
1875 * represent a weaker condition.

1878 * works on, and indexcols is NIL.  But if the clause is a RowCompareExpr,
1879 * indexcol is the index of the leading column, and indexcols is a list of
1880 * all the affected columns.  (Note that indexcols matches up with the
1881 * columns of the actual indexable RowCompareExpr in indexquals, which
1882 * might be different from the original in rinfo.)
1883 *
1884 * An IndexPath's IndexClause list is required to be ordered by index

1886 * (The order of multiple clauses for the same index column is unspecified.)

2550/*
2551 * Restriction clause info.
2552 *
2553 * We create one of these for each AND sub-clause of a restriction condition
2554 * (WHERE or JOIN/ON clause).  Since the restriction clauses are logically
2555 * ANDed, we can use any one of them or any subset of them to filter out
2556 * tuples, without having to evaluate the rest.  The RestrictInfo node itself
2557 * stores data used by the optimizer while choosing the best query plan.
2558 *
2559 * If a restriction clause references a single base relation, it will appear
2560 * in the baserestrictinfo list of the RelOptInfo for that base rel.
2561 *
2562 * If a restriction clause references more than one base+OJ relation, it will
2563 * appear in the joininfo list of every RelOptInfo that describes a strict
2564 * subset of the relations mentioned in the clause.  The joininfo lists are
2565 * used to drive join tree building by selecting plausible join candidates.
2566 * The clause cannot actually be applied until we have built a join rel
2567 * containing all the relations it references, however.
2568 *
2569 * When we construct a join rel that includes all the relations referenced
2570 * in a multi-relation restriction clause, we place that clause into the
2571 * joinrestrictinfo lists of paths for the join rel, if neither left nor
2572 * right sub-path includes all relations referenced in the clause.  The clause
2573 * will be applied at that join level, and will not propagate any further up
2574 * the join tree.  (Note: the "predicate migration" code was once intended to
2575 * push restriction clauses up and down the plan tree based on evaluation
2576 * costs, but it's dead code and is unlikely to be resurrected in the
2577 * foreseeable future.)
2578 *
2579 * Note that in the presence of more than two rels, a multi-rel restriction
2580 * might reach different heights in the join tree depending on the join
2581 * sequence we use.  So, these clauses cannot be associated directly with
2582 * the join RelOptInfo, but must be kept track of on a per-join-path basis.
2583 *
2584 * RestrictInfos that represent equivalence conditions (i.e., mergejoinable
2585 * equalities that are not outerjoin-delayed) are handled a bit differently.
2586 * Initially we attach them to the EquivalenceClasses that are derived from
2587 * them.  When we construct a scan or join path, we look through all the
2588 * EquivalenceClasses and generate derived RestrictInfos representing the
2589 * minimal set of conditions that need to be checked for this particular scan
2590 * or join to enforce that all members of each EquivalenceClass are in fact
2591 * equal in all rows emitted by the scan or join.
2592 *
2593 * The clause_relids field lists the base plus outer-join RT indexes that
2594 * actually appear in the clause.  required_relids lists the minimum set of
2595 * relids needed to evaluate the clause; while this is often equal to
2596 * clause_relids, it can be more.  We will add relids to required_relids when
2597 * we need to force an outer join ON clause to be evaluated exactly at the
2598 * level of the outer join, which is true except when it is a "degenerate"
2599 * condition that references only Vars from the nullable side of the join.
2600 *
2601 * RestrictInfo nodes contain a flag to indicate whether a qual has been
2602 * pushed down to a lower level than its original syntactic placement in the
2603 * join tree would suggest.  If an outer join prevents us from pushing a qual
2604 * down to its "natural" semantic level (the level associated with just the
2605 * base rels used in the qual) then we mark the qual with a "required_relids"
2606 * value including more than just the base rels it actually uses.  By
2607 * pretending that the qual references all the rels required to form the outer
2608 * join, we prevent it from being evaluated below the outer join's joinrel.
2609 * When we do form the outer join's joinrel, we still need to distinguish
2610 * those quals that are actually in that join's JOIN/ON condition from those
2611 * that appeared elsewhere in the tree and were pushed down to the join rel
2612 * because they used no other rels.  That's what the is_pushed_down flag is
2613 * for; it tells us that a qual is not an OUTER JOIN qual for the set of base
2614 * rels listed in required_relids.  A clause that originally came from WHERE
2615 * or an INNER JOIN condition will *always* have its is_pushed_down flag set.
2616 * It's possible for an OUTER JOIN clause to be marked is_pushed_down too,
2617 * if we decide that it can be pushed down into the nullable side of the join.
2618 * In that case it acts as a plain filter qual for wherever it gets evaluated.
2619 * (In short, is_pushed_down is only false for non-degenerate outer join
2620 * conditions.  Possibly we should rename it to reflect that meaning?  But
2621 * see also the comments for RINFO_IS_PUSHED_DOWN, below.)
2622 *
2623 * There is also an incompatible_relids field, which is a set of outer-join
2624 * relids above which we cannot evaluate the clause (because they might null
2625 * Vars it uses that should not be nulled yet).  In principle this could be
2626 * filled in any RestrictInfo as the set of OJ relids that appear above the
2627 * clause and null Vars that it uses.  In practice we only bother to populate
2628 * it for "clone" clauses, as it's currently only needed to prevent multiple
2629 * clones of the same clause from being accepted for evaluation at the same
2630 * join level.
2631 *
2632 * There is also an outer_relids field, which is NULL except for outer join
2633 * clauses; for those, it is the set of relids on the outer side of the
2634 * clause's outer join.  (These are rels that the clause cannot be applied to
2635 * in parameterized scans, since pushing it into the join's outer side would
2636 * lead to wrong answers.)
2637 *
2638 * To handle security-barrier conditions efficiently, we mark RestrictInfo
2639 * nodes with a security_level field, in which higher values identify clauses
2640 * coming from less-trusted sources.  The exact semantics are that a clause
2641 * cannot be evaluated before another clause with a lower security_level value
2642 * unless the first clause is leakproof.  As with outer-join clauses, this
2643 * creates a reason for clauses to sometimes need to be evaluated higher in
2644 * the join tree than their contents would suggest; and even at a single plan
2645 * node, this rule constrains the order of application of clauses.
2646 *
2647 * In general, the referenced clause might be arbitrarily complex.  The
2648 * kinds of clauses we can handle as indexscan quals, mergejoin clauses,
2649 * or hashjoin clauses are limited (e.g., no volatile functions).  The code
2650 * for each kind of path is responsible for identifying the restrict clauses
2651 * it can use and ignoring the rest.  Clauses not implemented by an indexscan,
2652 * mergejoin, or hashjoin will be placed in the plan qual or joinqual field
2653 * of the finished Plan node, where they will be enforced by general-purpose
2654 * qual-expression-evaluation code.  (But we are still entitled to count
2655 * their selectivity when estimating the result tuple count, if we
2656 * can guess what it is...)
2657 *
2658 * When the referenced clause is an OR clause, we generate a modified copy
2659 * in which additional RestrictInfo nodes are inserted below the top-level
2660 * OR/AND structure.  This is a convenience for OR indexscan processing:
2661 * indexquals taken from either the top level or an OR subclause will have
2662 * associated RestrictInfo nodes.
2663 *
2664 * The can_join flag is set true if the clause looks potentially useful as
2665 * a merge or hash join clause, that is if it is a binary opclause with
2666 * nonoverlapping sets of relids referenced in the left and right sides.
2667 * (Whether the operator is actually merge or hash joinable isn't checked,
2668 * however.)
2669 *
2670 * The pseudoconstant flag is set true if the clause contains no Vars of
2671 * the current query level and no volatile functions.  Such a clause can be
2672 * pulled out and used as a one-time qual in a gating Result node.  We keep
2673 * pseudoconstant clauses in the same lists as other RestrictInfos so that
2674 * the regular clause-pushing machinery can assign them to the correct join
2675 * level, but they need to be treated specially for cost and selectivity
2676 * estimates.  Note that a pseudoconstant clause can never be an indexqual
2677 * or merge or hash join clause, so it's of no interest to large parts of
2678 * the planner.
2679 *
2680 * When we generate multiple versions of a clause so as to have versions
2681 * that will work after commuting some left joins per outer join identity 3,
2682 * we mark the one with the fewest nullingrels bits with has_clone = true,
2683 * and the rest with is_clone = true.  This allows proper filtering of
2684 * these redundant clauses, so that we apply only one version of them.
2685 *
2686 * When join clauses are generated from EquivalenceClasses, there may be
2687 * several equally valid ways to enforce join equivalence, of which we need
2688 * apply only one.  We mark clauses of this kind by setting parent_ec to
2689 * point to the generating EquivalenceClass.  Multiple clauses with the same
2690 * parent_ec in the same join are redundant.
2691 *
2692 * Most fields are ignored for equality, since they may not be set yet, and
2693 * should be derivable from the clause anyway.
2694 *
2695 * parent_ec, left_ec, right_ec are not printed, lest it lead to infinite
2696 * recursion in plan tree dump.
2697 */
