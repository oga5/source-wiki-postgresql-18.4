src/test/modules/libpq_pipeline/README
1Test programs and libraries for libpq
2
3If you have Test::Differences installed, any differences in the trace files
4are displayed in a format that's easier to read than the standard format.
5=====================================
6
7This module was developed to test libpq's "pipeline" mode, but it can
8be used for any libpq test that requires specialized C code.
9
10"make check" will run all the tests in the module against a temporary
11server installation.
12
13You can manually run a specific test by running:
14
15    ./libpq_pipeline <name of test> [ <connection string> ]
16
17This will not start a new server, but rather connect to the server
18specified by the connection string, or your default server if you
19leave that out.  To discover the available test names, run:
20
21    ./libpq_pipeline tests
22
23To add a new test to this module, you need to edit libpq_pipeline.c.
24Add a function to perform the test, and arrange for main() to call it
25when the name of your new test is passed to the program.  Don't forget
26to add the name of your test to the print_test_list() function, else
27the TAP test won't run it.
28
29If the order in which Postgres protocol messages are sent is deterministic
30in your test, you should arrange for the message sequence to be verified
31by the TAP test.  First generate a reference trace file, using a command
32like:
33
34   ./libpq_pipeline -t traces/mynewtest.trace mynewtest
35
36Then add your test's name to the list in the $cmptrace definition in the
37t/001_libpq_pipeline.pl file.  Run "make check" a few times to verify
38that the trace output actually is stable.
