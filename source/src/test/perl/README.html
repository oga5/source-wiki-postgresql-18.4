src/test/perl/README
1Perl-based TAP tests
2====================
3
4src/test/perl/ contains shared infrastructure that's used by Perl-based tests
5across the source tree, particularly tests in src/bin and src/test. It's used
6to drive tests for backup and restore, replication, etc - anything that can't
7really be expressed using pg_regress or the isolation test framework.
8
9The tests are invoked via perl's 'prove' command, wrapped in PostgreSQL
10makefiles to handle instance setup etc. See the $(prove_check) and
11$(prove_installcheck) targets in Makefile.global. By default every test in the
12t/ subdirectory is run. Individual test(s) can be run instead by passing
13something like PROVE_TESTS="t/001_testname.pl t/002_othertestname.pl" to make.
14
15By default, to keep the noise low during runs, we do not set any flags via
16PROVE_FLAGS, but this can be done on the 'make' command line if desired, eg:
17
18make check-world PROVE_FLAGS='--verbose'
19
20When a test fails, the terminal output from 'prove' is usually not sufficient
21to diagnose the problem.  Look into the log files that are left under
22tmp_check/log/ to get more info.  Files named 'regress_log_XXX' are log
23output from the perl test scripts themselves, and should be examined first.
24Other files are postmaster logs, and may be helpful as additional data.
25
26The tests default to a timeout of 180 seconds for many individual operations.
27Slow hosts may avoid load-induced, spurious failures by setting environment
28variable PG_TEST_TIMEOUT_DEFAULT to some number of seconds greater than 180.
29Developers may see faster failures by setting that environment variable to
30some lesser number of seconds.
31
32Data directories will also be left behind for analysis when a test fails;
33they are named according to the test filename.  But if the environment
34variable PG_TEST_NOCLEAN is set, the data directories will be retained
35regardless of test status.  This environment variable also prevents the
36test's temporary directories from being removed.

39Writing tests
40-------------
41
42You should prefer to write tests using pg_regress in src/test/regress, or
43isolation tester specs in src/test/isolation, if possible. If not, check to
44see if your new tests make sense under an existing tree in src/test, like
45src/test/ssl, or should be added to one of the suites for an existing utility.
46
47Note that all tests and test tools should have perltidy run on them before
48patches are submitted, using perltidy --profile=src/tools/pgindent/perltidyrc
49
50Tests are written using Perl's Test::More with some PostgreSQL-specific
51infrastructure from src/test/perl providing node management, support for
52invoking 'psql' to run queries and get results, etc. You should read the
53documentation for Test::More before trying to write tests.
54
55Test scripts in the t/ subdirectory of a suite are executed in alphabetical
56order.
57
58Each test script should begin with:
59
60    use strict;
61    use warnings FATAL => 'all';
62    use PostgreSQL::Test::Cluster;
63    use PostgreSQL::Test::Utils;
64    use Test::More;
65
66then it will generally need to set up one or more nodes, run commands
67against them and evaluate the results. For example:
68
69    my $node = PostgreSQL::Test::Cluster->new('primary');
70    $node->init;
71    $node->start;
72
73    my $ret = $node->safe_psql('postgres', 'SELECT 1');
74    is($ret, '1', 'SELECT 1 returns 1');
75
76    $node->stop('fast');
77
78Each test script should end with:
79
80	done_testing();
81
82Test::Builder::Level controls how far up in the call stack a test will look
83at when reporting a failure.  This should be incremented by any subroutine
84which directly or indirectly calls test routines from Test::More, such as
85ok() or is():
86
87    local $Test::Builder::Level = $Test::Builder::Level + 1;
88
89Read the documentation for more on how to write tests:
90
91    perldoc Test::More
92    perldoc Test::Builder
93
94For available PostgreSQL-specific test methods and some example tests read the
95perldoc for the test modules, e.g.:
96
97    perldoc src/test/perl/PostgreSQL/Test/Cluster.pm

99Portability
100-----------
101
102Avoid using any bleeding-edge Perl features.  We have buildfarm animals
103running Perl versions as old as 5.14, so your tests will be expected
104to pass on that.
105
106Also, do not use any non-core Perl modules except IPC::Run.  Or, if you
107must do so for a particular test, arrange to skip the test when the needed
108module isn't present.  If unsure, you can consult Module::CoreList to find
109out whether a given module is part of the Perl core, and which module
110versions shipped with which Perl releases.
111
112One way to test for compatibility with old Perl versions is to use
113perlbrew; see http://perlbrew.pl .  After installing that, do
114
115    export PERLBREW_CONFIGURE_FLAGS='-de -Duseshrplib'
116    perlbrew --force install 5.14.0
117    perlbrew use 5.14.0
118    perlbrew install-cpanm
119    cpanm install Test::Simple@0.98
120    cpanm install IPC::Run@0.79
121    cpanm install ExtUtils::MakeMaker@6.50  # downgrade
122
123TIP: if Test::Simple's utf8 regression test hangs up, try setting a
124UTF8-compatible locale, e.g. "export LANG=en_US.utf8".
125
126Then re-run Postgres' configure to ensure the correct Perl is used when
127running tests.  To verify that the right Perl was found:
128
129    grep ^PERL= config.log
130
131Due to limitations of cpanm, this recipe doesn't exactly duplicate the
132module list of older buildfarm animals.  The discrepancies should seldom
133matter, but if you want to be sure, bypass cpanm and instead manually
134install the desired versions of Test::Simple and IPC::Run.
