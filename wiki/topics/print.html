print

概念

PostgreSQLのソースコード内において、print という名前の関数は、文脈に応じて全く異なる2つの役割で定義されています。

1. **デバッグ用のノードダンプユーティリティ**: バックエンドの構文木や実行計画などの内部データ構造（Node）をシリアライズし、整形して標準出力にダンプする機能。 2. **ECPGテスト用のエラーハンドラ**: 埋め込みSQL（ECPG）の回帰テストにおいて、SQL実行エラー発生時にエラーメッセージと詳細なSQLエラー情報を標準エラー出力へ出力するカスタムハンドラ。

なぜ重要か

  • **デバッグ効率の向上**: バックエンド開発において、複雑な Node 構造体の状態を人間が読める形式で標準出力にダンプできるため、クエリ解析や最適化処理のデバッグに不可欠です。
  • **テストの堅牢性と検証**: ECPGのプリプロセッサテストにおいて、エラーハンドリングが正しく機能しているかを検証し、エラー発生時の詳細情報を標準エラー出力に確実に記録するために重要です。

  • symbol:src/interfaces/ecpg/test/expected/preproc-whenever.c#print: static void print(const char *msg)。ECPGテスト用のエラーハンドラ。

エンドツーエンドの流れ

これら2つの関数は独立して動作し、統一された単一のエンドツーエンドフローは存在しません。それぞれのフローは以下の通りです。

1. バックエンドのデバッグ用 print のフロー

1. 開発者またはデバッガが print(obj) を呼び出します（具体的な呼び出し元コードは提供された範囲からは未解明です）。 2. nodeToStringWithLocations(obj) により、Node 構造体が文字列形式にシリアライズされます。 3. format_node_dump(s) により、シリアライズされた文字列に改行やインデントが挿入され、整形されます。 4. 整形された文字列が printf で標準出力に書き出され、fflush(stdout) で即座にフラッシュされます。 5. 一時バッファが pfree で解放されます。

2. ECPGテスト用 print のフロー

1. ECPGの回帰テスト実行中にSQLエラーが発生します（具体的な呼び出し元フローは提供されたソースコードからは未解明です）。 2. print(msg) が呼び出されます。 3. fprintf を使用して、引数 msg を含むエラーメッセージを標準エラー出力 (stderr) に出力します。 4. sqlprint() を呼び出し、ECPGライブラリが保持する詳細なSQLエラー情報（sqlca 構造体経由）を出力します。

重要な不変条件

  • **バックエンド print**:
  • 引数 obj は有効な PostgreSQL ノード構造体（シリアライズ可能であること）でなければなりません。
  • 本番環境のコードから呼び出されてはならず、デバッグ専用として維持される必要があります。
  • **ECPGテスト用 print**:
  • stderr が有効であり、ECPGライブラリの sqlprint() が呼び出し可能でなければなりません。

よくある落とし穴

  • **バックエンド print**:
  • 引数 obj に NULL を渡すと、内部の nodeToStringWithLocations の実装に依存しますが、セグメンテーションフォールトを引き起こす可能性があります。
  • 複雑なノード構造の場合、format_node_dump による整形が最適化されず、出力が読みづらくなる場合があります。
  • **ECPGテスト用 print**:
  • 引数 msg が NULL の場合、fprintf の動作に依存してクラッシュする可能性があります。
  • この関数はECPGプリプロセッサのテスト生成物の一部であるため、エラーハンドリングロジックを変更する際は、src/interfaces/ecpg/preproc/ 内のプリプロセッサ変換ロジックとの整合性を壊さないよう注意が必要です。
