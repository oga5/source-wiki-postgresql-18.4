pg_valid_server_encoding

概念

pg_valid_server_encoding は、指定されたエンコーディング名（文字列）が PostgreSQL サーバーでサポートされている有効なエンコーディングであるかを検証し、対応する内部エンコーディング ID を取得するための関数です。

なぜ重要か

PostgreSQL では、システム全体で認識可能なエンコーディングと、サーバーサイド（バックエンド）で実際に使用可能なエンコーディングが区別されます。本関数は、データベースの作成（dbcommands.c）や初期化（initdb.c）、拡張機能のロード（extension.c）などの重要な局面において、指定されたエンコーディングがサーバーサイドで正当に利用可能であるかを厳密に検証し、不正なエンコーディングの適用を未然に防ぐ役割を担っています。

関与するモジュール

  • module:src/include/mb

エンドツーエンドの流れ

1. **呼び出し**: 呼び出し元（initdb や dbcommands.c など）が、検証対象のエンコーディング名（const char *name）を引数に指定して pg_valid_server_encoding を呼び出します。 2. **名前から ID への変換**: 内部的に pg_char_to_encoding(name) を呼び出し、文字列を内部エンコーディング ID に変換します。変換に失敗した（負の値が返された）場合は、即座に -1 を返します。 3. **サーバー適合性の検証**: 取得した ID に対して PG_VALID_BE_ENCODING マクロを適用し、サーバーサイドでの使用が許可されているかを確認します。 4. **結果の返却**: 検証を通過した場合はその内部エンコーディング ID を返し、失敗した場合は -1 を返します。

*注意: 本関数が -1 または有効な ID を返した後に、呼び出し元（各コマンドやツール）の内部で具体的にどのようなエラーハンドリングやフォールバックが行われるかという、システム全体における最終的なエンドツーエンドのフローの詳細は、提供されたコンテキストからは明らかではありません。*

重要な不変条件

  • **検証の整合性**: 本関数が返す値が -1 以外である場合、その値は必ず PG_VALID_BE_ENCODING マクロが真を返す範囲内の有効な内部エンコーディング ID でなければなりません。
  • **エラー時の不変性**: 入力文字列が未知であるか、またはサーバーサイドで許可されていないエンコーディングである場合は、例外なく -1 を返さなければなりません。

よくある落とし穴

  • **サーバーとクライアントの混同**: 対象のエンコーディング名が PostgreSQL 全体として既知（pg_char_to_encoding が有効な値を返す）であっても、サーバーサイドでの使用が許可されていない（PG_VALID_BE_ENCODING が偽）場合は -1 が返されます。単に「名前が正しいか」ではなく「サーバーで使えるか」を判定している点に注意が必要です。
  • **定義の二重管理**: 本関数は src/common/encnames.c で実装されていますが、src/include/mb/pg_wchar.h にも関連する定義が存在します。サーバーサイドのエンコーディング検証ロジックを修正・拡張する場合は、PG_VALID_BE_ENCODING マクロの定義と併せて確認・更新する必要があります。
