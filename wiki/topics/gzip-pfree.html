gzip_pfree

概念

gzip_pfree は、PostgreSQL のメモリ管理システム（pfree）を zlib ライブラリのメモリ解放インターフェースに適合させるためのカスタムメモリ解放ラッパー関数です。zlib が内部でメモリを解放する際に、PostgreSQL のメモリコンテキストを介して適切に解放処理が行われるようにします。

なぜ重要か

zlib ライブラリは独自のメモリ管理インターフェースを要求しますが、PostgreSQL 内で動作させるためには、PostgreSQL のメモリ管理機構（palloc/pfree）と統合する必要があります。gzip_pfree を提供することで、メモリリークを防ぎ、PostgreSQL のメモリコンテキストのライフサイクルに沿った安全なメモリ解放が可能になります。

エンドツーエンドの流れ

1. **初期化と登録**: src/backend/backup/basebackup_gzip.c の初期化処理（例: 123行目付近）において、zlib の z_stream 構造体の zfree フィールドに gzip_pfree のポインタが代入されます。 2. **メモリ解放のトリガー**: zlib ライブラリの内部処理でメモリの解放が必要になった際、登録された zfree（すなわち gzip_pfree）が呼び出されます。 3. **PostgreSQL による解放**: gzip_pfree は受け取ったメモリ領域のポインタ（address）を PostgreSQL の pfree に渡し、実際のメモリ解放を行います。

重要な不変条件

  • **有効なメモリ領域**:

    • gzip_pfree に渡される address は、PostgreSQL の palloc 等で割り当てられた有効なメモリ領域である必要があります（ただし、address が NULL の場合は pfree の仕様に基づき何も行われません）。

  • **インターフェースの一致**:

    • zlib の要求するシグネチャ void (*zfree)(void *opaque, void *address) に適合している必要があります。

※ フロントエンド側（fe_utils）における不変条件の詳細は提供された情報からは不明です。

よくある落とし穴

  • **同名関数の混同**:

    • gzip_pfree は src/backend/backup/basebackup_gzip.c と src/fe_utils/astreamer_gzip.c の両方に別々のコンパイル単位として定義されています。これらは異なるコンテキストで動作するため、混同しないよう注意が必要です。

  • **無効なポインタの引き渡し**:

    • address に無効なポインタが渡された場合の挙動は PostgreSQL の pfree に依存するため、不正なポインタが zlib 経由で渡されないようにする必要があります。
