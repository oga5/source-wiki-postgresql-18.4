Size

概念

PostgreSQLにおける Size は、主にメモリサイズ、データ長、配列の要素数、オフセットなどを表現するための標準的な型定義（typedef size_t Size）です。ただし、論理デコーディングの文脈（src/include/replication/output_plugin.h）においては、メッセージやデータバッファのサイズを表現する struct Size としても定義されています。

なぜ重要か

PostgreSQLのコードベース全体でメモリ割り当て、バッファサイズ計算、データ構造のオフセット管理、ハッシュテーブルのキーサイズ指定などを統一的かつ安全に行うために重要です。プラットフォームのアーキテクチャ（32bit/64bit）に依存する size_t のエイリアスとして機能することで、システム全体のメモリ管理の互換性と移植性を担保します。

エンドツーエンドの流れ

現時点では、Size 型が関与する具体的なエンドツーエンドのデータフロー全体（例えば、メモリ確保から解放、あるいは論理デコーディングメッセージの受信からプラグインへの受け渡しまでの完全なライフサイクル）は完全には明らかになっていません。

ただし、部分的な流れとして以下が確認されています。

  • **論理デコーディング**: LogicalDecodeMessageCB などのコールバック関数において、メッセージのバイト長を示す引数として Size が渡され、プラグインが受信メッセージの長さを判定して処理します。
  • **オブジェクトのフラット化**: expandeddatum.h において、展開済みオブジェクトをフラット化するメソッド EOM_flatten_into_method に割り当てサイズとして Size が渡されます。
  • **ハッシュテーブル制御**: hsearch.h において、ハッシュ関数 HashValueFunc の引数 keysize として Size が渡され、ハッシュテーブルの制御に使用されます。

重要な不変条件

  • **非負性**: Size（size_t のエイリアスの場合）は符号なし整数型であり、値は常に非負（0以上）です。
  • **プラットフォーム依存**: プラットフォームのポインタサイズに依存し、オブジェクトの最大サイズを表現するのに十分な範囲（最大値 SIZE_MAX）を持ちます。
  • **バッファ一致性**: メモリバッファのサイズを示す場合、対象となるバッファの実際のバイト数と一致している必要があります。

よくある落とし穴

  • **オーバーフローと切り詰め**: Size（size_t）の最大値を超えるサイズを計算・代入しようとした場合、整数オーバーフローや切り詰めが発生し、不正なメモリ割り当てやバッファオーバーフローを引き起こすリスクがあります。
  • **負の値の代入**: 符号なし型であるため、負の値を代入すると予期せぬ極めて大きな正の整数として解釈されます。
  • **名前の衝突と混同**: src/include/replication/output_plugin.h では struct Size として定義されている一方、他の多くのヘッダー（c.h, hsearch.h など）では typedef size_t Size として定義されています。環境やコンパイル設定、あるいは標準ライブラリの size_t との混同に注意する必要があります。
