link

概念

PostgreSQLのソースコードにおいて、link という名前は主に2つの異なる文脈で使用されています。

1. **プラットフォーム移植性レイヤー（Windows互換）**: Windows環境において、POSIX準拠のハードリンク作成関数 link() をエミュレートする関数。 2. **タイムゾーンコンパイラ（zic）**: タイムゾーン定義ファイル（.zi）内の Link 行から解析された、タイムゾーン名間のリンク情報を保持するための構造体。

なぜ重要か

  • **移植性の確保**: Windows環境では標準のPOSIX link() 関数が提供されていないため、src/port/win32link.c の link() 関数がWindows API（CreateHardLinkA）をラップしてPOSIX互換のインターフェースを提供し、プラットフォーム間の差異を吸収します。
  • **タイムゾーン定義の効率化**: zic ユーティリティにおける struct link は、同一のタイムゾーン規則を持つ異なる地域名（例: リンク元とリンク先）を効率的に管理・作成するために不可欠なデータ構造です。

エンドツーエンドの流れ

これら2つのエンティティは完全に独立しており、直接連携するエンドツーエンドの流れは存在しません。それぞれの独立した処理フローは以下の通りです。

Windows互換 link() 関数

1. 呼び出し元（具体的な箇所は提供情報からは不明）が link(src, dst) を呼び出します。 2. Windows APIの CreateHardLinkA(dst, src, NULL) を呼び出します。 3. 成功した場合は 0 を返し、失敗した場合は GetLastError() から _dosmaperr() を介して errno を設定し、-1 を返します。

zic の struct link

1. zic.c の infile() 等の構文解析関数がタイムゾーン定義ファイルを読み込みます。 2. Link 行を解析し、struct link のインスタンスを構築して情報を格納します。 3. コンパイルの最終段階で dolink() 関数がこの構造体を参照し、指定されたターゲットに対して実際のリンクを作成します。

重要な不変条件

  • **Windows互換 link() 関数**:
  • ハードリンクは同一ボリューム内でのみ作成可能です（Windows API CreateHardLinkA の制約）。
  • 入力パス（src および dst）はマルチバイト文字（ANSI）として扱われます。
  • **zic の struct link**:
  • l_filename と l_linenum は、エラーメッセージ出力時にソースファイルの場所を特定するために正しく保持されなければなりません。
  • l_target は既存のタイムゾーン名、l_linkname は新しく作成するリンク名を指す必要があります。

よくある落とし穴

  • **Windows互換 link() 関数**:
  • CreateHardLinkA を使用しているため、Unicode（UTF-16）パスを直接扱うことはできません。将来的にUnicode対応が必要な場合は、CreateHardLinkW を使用する実装への変更が必要になります。
  • 同一ボリューム外へのリンク作成など、OSの制約に抵触する場合は失敗します。
  • **zic の struct link**:
  • リンク名が既に存在する場合や、ターゲットとなるゾーンが存在しない場合、dolink() によるリンク作成処理が失敗する可能性があります。
  • contrib/amcheck 等に見られる link への言及は、PostgreSQLのインデックス構造に関するものであり、このタイムゾーン構造体とは無関係です。
