SH_SCOPE

概念

SH_SCOPE は、PostgreSQL の汎用ハッシュテーブル生成テンプレート（主に lib/simplehash.h）を使用する際に、自動生成される関数群のスコープ（可視性やインライン化の属性）を指定するためのマクロです。

通常、#define SH_SCOPE static inline や #define SH_SCOPE extern のように定義した上でテンプレートヘッダーをインクルードすることにより、生成されるハッシュテーブル操作関数のスコープを制御します。

なぜ重要か

C言語におけるマクロテンプレートを用いたジェネリックなデータ構造の実装において、関数の重複定義（ODR違反）を防ぎつつ、コンパイラによる最適化（インライン展開など）を最大限に活かすために重要です。

各モジュール（バックアップ、統計情報、クエリ実行エンジンなど）は、自身の用途（単一ファイル内での局所的な利用、または複数ファイル間での共有）に合わせて、ハッシュテーブル関数のスコープを柔軟に制御できます。

エンドツーエンドの流れ

システム全体で統一された単一の実行フローは存在せず、ハッシュテーブルの初期化や利用の流れは各モジュールの機能に依存します。一般的なコード生成および利用の流れは以下の通りです。

1. **パラメータ定義**: ハッシュテーブルのキー型、値型、ハッシュ関数などのパラメータをマクロで定義します。 2. **スコープ定義**: SH_SCOPE マクロを定義し、生成される関数のスコープ（例: static inline）を指定します。 3. **コード生成**: simplehash.h などのテンプレートヘッダーをインクルードし、マクロを展開してハッシュテーブル操作関数を自動生成します。 4. **利用**: 生成された関数（例: hash_search, hash_insert など）を各モジュール内のロジックから呼び出します。

重要な不変条件

  • **定義の順序**: SH_SCOPE は、ハッシュテーブル生成用のテンプレートヘッダーをインクルードする**前**に必ず定義されていなければなりません。
  • **一貫性**: 同一のハッシュテーブル定義に対して、ヘッダー宣言側（例: load_manifest.h）と実装側（例: load_manifest.c）で SH_SCOPE の指定に不整合があってはなりません。

よくある落とし穴

  • **重複定義エラー**: 複数のソースファイルで同じハッシュテーブル名を定義する際、SH_SCOPE を適切に static inline に設定しないと、リンカが重複シンボルエラー（multiple definition）を検出する原因になります。
  • **未定義動作**: SH_SCOPE の定義を忘れたり、インクルードの後に記述したりすると、デフォルトのスコープが適用されるか、コンパイルエラーを引き起こします。
