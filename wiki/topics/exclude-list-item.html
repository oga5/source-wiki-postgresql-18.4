exclude_list_item

概念

exclude_list_item は、PostgreSQLのファイル走査を伴う各種処理（バックアップ、チェックサム検証、データ同期など）において、処理対象から除外すべきファイルやディレクトリのルールを定義するための構造体です。この構造体は、対象の名前（name）と、前方一致（プレフィックス一致）で判定するか完全一致で判定するかを示すフラグ（match_prefix）を保持します。

なぜ重要か

PostgreSQLのデータディレクトリ内には、一時ファイル、WALファイル、あるいは特定のツールが生成する一時的なデータなど、バックアップやチェックサム検証、pg_rewind による同期処理の対象から除外すべきファイルが存在します。exclude_list_item を用いて除外リストを定義することで、各ツールが不要なファイルを誤って処理したり、エラーを引き起こしたりするのを防ぎ、一貫したファイル走査ルールを適用することができます。

関与するモジュール

  • module:src/backend/backup (ベースバックアップ処理)
  • module:src/bin/pg_checksums (チェックサム検証ツール)
  • module:src/bin/pg_rewind (データ同期ツール)

主要エンティティ

  • **symbol:src/backend/backup/basebackup.c#exclude_list_item**: バックアップ対象外のファイルを定義する構造体。excludeFiles 配列で使用されます。
  • **symbol:src/bin/pg_checksums/pg_checksums.c#exclude_list_item**: チェックサム検証対象外のファイルを定義する構造体。skip[] 配列で使用されます。
  • **symbol:src/bin/pg_rewind/filemap.c#exclude_list_item**: pg_rewind の同期対象外のファイルを定義する構造体。excludeFiles 配列で使用されます。

エンドツーエンドの流れ

各ツールにおけるファイル走査時の具体的なエンドツーエンドの処理フロー（走査関数から除外判定関数が呼び出され、最終的にどのようにスキップされるかの詳細なコールスタックや全体フロー）は、提供された情報からは完全には明らかになっていません。

判明している範囲では、以下の流れで処理されます。 1. 各モジュールにおいて、exclude_list_item の静的配列（excludeFiles や skip[]）が定義されます。 2. ファイルシステムを走査する際、判定ロジック（pg_checksums の skipfile 関数や pg_rewind の check_file_excluded 関数など）がこの配列を走査します。 3. 各要素について、match_prefix が true の場合は前方一致、false の場合は完全一致で対象ファイル名と name を比較し、一致した場合はそのファイルを処理対象から除外します。

重要な不変条件

  • **name の非ヌル性**: name はヌル終端文字列（const char *）であり、静的に定義された文字列リテラルを指すことが想定されています。pg_checksums では name が NULL であってはならないとされています。
  • **静的な定義**: 各モジュールにおける exclude_list_item の配列（excludeFiles や skip[]）は、通常 static const（または静的配列）として定義され、実行中に動的に変更されることはありません。

よくある落とし穴

  • **前方一致（match_prefix）の誤用**: match_prefix を true に設定する際、プレフィックスの指定が不適切（例えば、短すぎる文字列や空文字列の指定）であると、意図しないファイルまで広範囲に除外対象となってしまう危険性があります。
  • **モジュール間の同期漏れ**: pg_rewind の除外リストは basebackup.c の定義と同期させる必要があります。また、pg_checksums も含め、各ツールで同名の構造体が独立して定義されているため、新しいシステムファイルや一時ファイルを除外対象に追加する際は、関連するすべてのモジュールで個別に定義を更新する必要があります。
