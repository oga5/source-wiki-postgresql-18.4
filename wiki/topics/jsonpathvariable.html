JsonPathVariable

概念

JsonPathVariable は、PostgreSQLにおいてSQL/JSON Pathの評価を実行する際、外部から渡される変数（名前、型、値、およびNULLフラグなど）の情報を保持するためのデータ構造です。

なぜ重要か

SQL/JSON Pathの実行エンジンに対して、外部から提供された変数を識別させ、その値を評価エンジンが利用できるようにするために重要です。構造体内に変数名の長さ（namelen）をキャッシュとして保持することで、変数検索時（例：GetJsonPathVar() 等の処理）における文字列長計算のコストを削減し、評価処理の効率化に寄与します。

関与するモジュール

  • module:src/backend/utils/adt
  • module:src/include/utils

主要エンティティ

  • symbol:src/include/utils/jsonpath.h#JsonPathVariable：外部変数のメタデータと値を格納する構造体。
  • symbol:src/backend/utils/adt/jsonpath_exec.c#getJsonPathVariable：JSON Path 実行時に変数を取得・評価するための関数。

エンドツーエンドの流れ

提供されたソースコードの抜粋からは、詳細なエンドツーエンドの実行フロー（変数のパースからバインド、最終評価に至る完全なコールスタック）は完全には明らかになっていません。

現時点で判明している部分的な流れは以下の通りです： 1. **変数の準備**: src/backend/executor/execExpr.c や src/include/nodes/execnodes.h（passing_values 句）などを通じて、外部変数の情報が JsonPathVariable 構造体に格納されます。 2. **変数の参照**: JSON Path の実行時（src/backend/utils/adt/jsonpath_exec.c）に、getJsonPathVariable などの関数を介して、これらの変数が参照・評価されます。

重要な不変条件

  • **名前と長さの同期**: namelen は常に strlen(name) の値を保持するキャッシュとして機能する必要があります。
  • **型の一貫性**: value（Datum型）は、typid で指定された型に従って正しく解釈されなければなりません。
  • **NULLフラグの正確性**: isnull フラグは、value が NULL であるかどうかを正確に反映している必要があります。

よくある落とし穴

  • **キャッシュの更新漏れ**: name を変更した際に namelen を同期して更新し忘れると、変数検索時に誤作動を起こす可能性があります。
  • **型不整合とクラッシュ**: name が NULL の場合や、typid が不適切な場合、評価時に型不整合やクラッシュを招くリスクがあります。
  • **NULL値の誤解釈**: isnull フラグが正しく設定されていない場合、value の内容が誤って解釈される可能性があります。
  • **メモリ管理**: 本構造体は palloc を用いて動的に確保されるケースが多いため、メモリのライフサイクル管理に注意が必要です。
