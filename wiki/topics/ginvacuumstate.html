GinVacuumState

概念

GinVacuumState は、GIN (Generalized Inverted Index) インデックスのバキューム処理（VACUUM および VACUUM ANALYZE）において、処理対象のインデックス情報、コールバック関数、メモリコンテキスト、およびバキュームの進捗状態を保持・管理するための構造体です。

なぜ重要か

GINインデックスのバキューム処理は、不要領域の回収、ポスティングツリーの走査、不要ページの削除など、複雑なステップを伴います。GinVacuumState は、これらの処理フェーズ（TID削除判定、メモリ管理、状態保持など）を横断して一貫したコンテキストを提供し、バキューム処理の各関数間で状態を共有・追跡するために不可欠な役割を担っています。

エンドツーエンドの流れ

1. **初期化**: ginbulkdelete および ginvacuumcleanup が起点となり、GinVacuumState をインスタンス化します。この際、対象リレーション (index)、TID削除判定用のコールバック関数 (callback)、コールバック状態 (callback_state)、一時メモリコンテキスト (tmpCxt) などが初期化されます。 2. **走査と判定**: ginVacuumEntryPage や ginVacuumPostingTree などの関数がインデックスの各ページを走査します。 3. **削除と整理**: ginVacuumItemPointers や ginVacuumPostingTreeLeaf (gindatapage.c) において、callback を用いて特定のTIDを削除すべきか判定し、不要になったページは ginDeletePage によって削除されます。 4. **結果の返却**: 処理結果は result フィールド（IndexBulkDeleteResult）に格納され、バキューム終了時に呼び出し元へ返されます。

※なお、並行処理やロック処理が絡む詳細なエンドツーエンドの実行フローや、具体的なページ削除のトリガー条件の全容は、提供された情報からは完全には明らかになっていません。

重要な不変条件

  • index は有効なGINインデックスのリレーションでなければなりません。
  • tmpCxt はバキューム処理中に生成される一時的なメモリ割り当てを管理するために使用され、処理終了時に適切にリセットまたは解放される必要があります。
  • バキューム処理を実行する関数に渡される前に、構造体が適切に初期化されていなければなりません。

よくある落とし穴

  • **メモリリーク**: tmpCxt が適切に管理（リセットまたは解放）されない場合、バキューム処理中にメモリリークが発生する可能性があります。
  • **不完全な削除判定**: callback が NULL の場合、TID の削除判定が正しく行われない可能性があります。
  • **定義の不整合**: 本構造体は ginvacuum.c と gin_private.h の両方で定義（または宣言）されているため、変更時には両方の整合性に注意する必要があります。
  • **並行処理の影響**: ginScanToDelete などの複雑なロック処理を伴う関数に渡されるため、GinVacuumState を介した状態変更が並行処理に与える影響を考慮する必要があります。
