sqlda_t

概念

sqlda_t は、ECPG（Embedded SQL for C）において動的 SQL を実行する際、列情報（メタデータ）やデータ値を保持・管理するための「SQL Descriptor Area (SQLDA)」構造体に対する、モード非依存の汎用的な型名（typedef struct sqlda_compat sqlda_t;）です。

なぜ重要か

動的 SQL の実行時（パラメータのバインドや結果セットの取得時）に、データベースとアプリケーション間でメタデータやデータをやり取りするための共通インターフェースとして機能します。また、プリプロセッサ定義 _ECPG_INFORMIX_H の有無により、Informix 互換モード（sqlda-compat.h）と PostgreSQL ネイティブモード（sqlda-native.h）の構造体定義を透過的に切り替えるスイッチの役割も担っています。

エンドツーエンドの流れ

1. **メモリ確保と初期化**: アプリケーション側で malloc 等を用いて sqlda_t 構造体のメモリを確保し、memset 等で初期化します。 2. **メタデータの取得 / パラメータのバインド**:

  • PREPARE された SQL 文に対して ECPGdescribe() を呼び出し、列数や列名などのメタデータを sqlda_t にロードします。
  • または、EXECUTE や FETCH 文の USING DESCRIPTOR 句において、入力パラメータのバインドや出力結果の格納先として sqlda_t を指定します。

    • 3. **データの処理**: 構造体内の sqlvar_t 配列（各列の型、名前、データポインタ等を保持）を走査し、必要に応じてデータ変換（例: PGTYPESnumeric_to_asc による数値から文字列への変換）や値の参照を行います。 4. **リソースの解放**: 使用後、確保したメモリを free() 等で適切に解放します。

重要な不変条件

  • **コンパイル時設定**: コンパイル時に _ECPG_INFORMIX_H マクロの定義状態が決定され、適切なヘッダー（sqlda-compat.h または sqlda-native.h）の構造体に解決されている必要があります。
  • **メモリの事前確保**: 構造体を使用する前に、適切なメモリ割り当てと初期化が行われている必要があります。
  • **構造の整合性**: 実行する SQL 文の列数やデータ型と、sqlda_t（および内部の sqlvar_t 配列）が保持する定義が一致している必要があります。

よくある落とし穴

  • **メモリ割り当て失敗**: malloc が失敗して NULL ポインタが返された場合、適切なハンドリングを行わないとセグメンテーション違反が発生します。
  • **モードの不一致**: ビルド時のマクロ設定（Informix 互換かネイティブか）と実行環境のモードが一致していない場合、データ構造の解釈に不整合が生じ、予期せぬ動作を引き起こします。
  • **バッファオーバーフロー**: DESCRIBE で取得した実際の列数と、確保した sqlda_t の容量が一致しない場合、データ破損やバッファオーバーフローのリスクがあります。
  • **自動生成ファイルの直接編集**: src/interfaces/ecpg/test/expected/ 以下のファイルは ECPG プリプロセッサによって生成されるテスト用期待値コードであるため、直接編集してはいけません。
