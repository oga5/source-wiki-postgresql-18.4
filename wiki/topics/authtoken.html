AuthToken

概念

AuthToken は、PostgreSQL の認証設定ファイル（pg_hba.conf や pg_ident.conf）に記述された各トークンを表現するためのデータ構造です。トークン文字列そのものに加え、クォートの有無や、正規表現マッチングのためのコンパイル済み正規表現オブジェクトを保持します。

なぜ重要か

認証設定ファイルにおける柔軟なパターンマッチング（正規表現など）を安全かつ効率的に処理するために重要です。特に、正規表現リソース（regex_t）のライフサイクル（コンパイル、実行、解放）をトークン単位で適切に管理し、メモリリークを防ぐ役割を担っています。

主要エンティティ

  • symbol:src/include/libpq/hba.h#AuthToken：トークン情報を保持する構造体。
  • symbol:src/backend/libpq/hba.c#free_auth_token：トークンに関連付けられた正規表現リソースを解放する関数。

エンドツーエンドの流れ

1. **パースと生成**: 設定ファイルのパース時にトークンが検出されると、palloc0 等を用いて AuthToken 構造体が動的に割り当てられます。 2. **正規表現のコンパイルとマッチング**: 必要に応じて regcomp_auth_token や regexec_auth_token を介して正規表現のコンパイルとマッチングが行われます。 3. **複製**: copy_auth_token により、既存のトークン構造体を複製することが可能です。 4. **リソース解放**: トークンが不要になった際（例: expanded_pg_user_token の解放時など）、free_auth_token が呼び出され、内部の正規表現リソースが解放されます。

*注意: 設定ファイルの読み込みから認証判定、そして最終的なメモリ破棄に至る完全なエンドツーエンドの制御フローの全容は、提供された情報からは未検証です。*

重要な不変条件

  • regex フィールドが非NULLである場合、その正規表現オブジェクトは pg_regcomp 等で適切に初期化されていなければなりません。
  • AuthToken の破棄時には、正規表現リソースのリークを防ぐために必ず free_auth_token を経由して内部リソースを解放する必要があります。

よくある落とし穴

  • **構造体自体のメモリ管理**: free_auth_token は内部の正規表現リソース（regex）の解放（pg_regfree）のみを担当します。AuthToken 構造体自体のメモリ解放は呼び出し元が個別に行う必要があります。
  • **メモリ配置の仮定**: string フィールドのメモリ領域が構造体自身と連続して確保される実装（palloc0(sizeof(AuthToken) + toklen + 1)）が存在するため、構造体のサイズ変更やメンバの個別解放を行う際には注意が必要です。
  • **NULLポインタの扱い**: free_auth_token に NULL ポインタを渡した場合の挙動は、提供されたソース抜粋からは保証されていません。
