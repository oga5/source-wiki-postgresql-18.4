pg_hmac_ctx

概念

pg_hmac_ctx は、RFC 2104 に準拠した HMAC（Keyed-Hashing for Message Authentication）計算の状態を保持するための構造体です。ヘッダーファイル src/include/common/hmac.h において不透明な（opaque）構造体として定義されており、具体的な内部構造はカプセル化されています。

実際の具体的な定義は、ビルド環境（OpenSSL の有無など）に応じて以下のいずれかのソースファイルで提供されます。

  • **OpenSSL 非依存（フォールバック実装）**: src/common/hmac.c
  • **OpenSSL 依存実装**: src/common/hmac_openssl.c

なぜ重要か

PostgreSQL は、フロントエンド（クライアントツールなど）とバックエンド（サーバープロセス）の双方で SCRAM 認証などの安全な暗号化・認証処理を必要とします。pg_hmac_ctx は、環境に依存しない統一された HMAC 抽象化レイヤーを提供します。これにより、OpenSSL が利用可能な環境ではその強力な暗号機能を活用し、利用できない環境では独自のフォールバック実装を透過的に切り替えて使用することが可能になります。

主要エンティティ

  • **symbol:src/include/common/hmac.h#pg_hmac_ctx**

    • 外部に公開される不透明な構造体の前方宣言（typedef struct pg_hmac_ctx pg_hmac_ctx;）。呼び出し側は内部メンバに直接アクセスできません。

  • **symbol:src/common/hmac.c#pg_hmac_ctx**

    • OpenSSL がない環境向けのフォールバック実装。内部ハッシュコンテキスト（pg_cryptohash_ctx）、パディングデータ（k_ipad, k_opad）、エラー状態などを保持します。

  • **symbol:src/common/hmac_openssl.c#pg_hmac_ctx**

    • OpenSSL ライブラリの HMAC_CTX をラップする実装。OpenSSL のコンテキストポインタや、バックエンド環境用の ResourceOwner などを保持します。

エンドツーエンドの流れ

HMAC 計算は以下のライフサイクルに従って実行されます。

1. **作成 (pg_hmac_create)**: コンテキストのメモリ領域を確保します。バックエンド環境では palloc（または TopMemoryContext）、フロントエンド環境では malloc が使用されます。 2. **初期化 (pg_hmac_init)**: 指定されたハッシュアルゴリズム（pg_cryptohash_type）と秘密鍵を設定します。フォールバック実装では、鍵がブロックサイズを超える場合にハッシュ化して短縮し、k_ipad / k_opad を生成して内部ハッシュを初期化します。 3. **更新 (pg_hmac_update)**: メッセージデータを入力ストリームとして逐次処理し、内部のハッシュ状態を更新します。 4. **終了 (pg_hmac_final)**: 最終的な MAC 値を算出してバッファに書き出します。 5. **解放 (pg_hmac_free)**: コンテキストを破棄し、メモリを解放します。この際、機密情報の漏洩を防ぐために explicit_bzero を用いて内部状態がゼロクリアされます。

※なお、SCRAM 認証プロセス（src/backend/libpq/auth-scram.c）における具体的な呼び出しシーケンスの全体像や、エラー発生時の詳細な伝播フローの全容は、提供されたコンテキストからは完全には明らかになっていません。

重要な不変条件

  • **カプセル化の維持**: 呼び出し側は pg_hmac_ctx の内部メンバに直接アクセスしてはならず、必ず hmac.h で定義された API を介して操作しなければなりません。
  • **機密情報の消去**: メモリ解放時（pg_hmac_free）には、必ず explicit_bzero を用いてコンテキスト内の機密情報（鍵情報や中間状態）を完全に消去する必要があります。
  • **環境に応じたメモリ管理**: バックエンド環境では PostgreSQL のメモリ管理機構（palloc や ResourceOwner）に統合され、フロントエンド環境では標準の malloc / free が使用されます。
  • **固定バッファサイズ（フォールバック時）**: フォールバック実装における k_ipad および k_opad の配列サイズは、サポートされる最大ブロックサイズ（PG_SHA512_BLOCK_LENGTH）に固定されています。

よくある落とし穴

  • **直接アクセスの試み**: pg_hmac_ctx は不透明な構造体であるため、呼び出し側でメンバ（error や errreason など）を直接参照しようとするとコンパイルエラーになります。エラー情報の取得には pg_hmac_error() などの専用 API を使用する必要があります。
  • **メモリリーク**: 特にフロントエンド環境では、呼び出し側が明示的に pg_hmac_free() を呼び出してメモリを解放する責任があります。
  • **OpenSSL エラーキューの未処理**: OpenSSL 実装を使用している場合、エラー発生時に OpenSSL のエラーキュー（ERR_get_error()）を適切に監視・クリアしないと、後続の暗号処理に影響を及ぼす可能性があります。
