WaitEvent

概念

WaitEvent および WaitEventSet は、PostgreSQL において複数のイベント（ソケットの I/O 状態変化、ラッチのシグナル伝達、ポストマスタープロセスの終了など）を単一のインターフェースで効率的に監視・待機するための抽象化レイヤーです。OS 固有のイベント通知プリミティブ（epoll, kqueue, poll, WaitForMultipleObjects）をラップすることで、プラットフォーム間の差異を吸収します。

なぜ重要か

PostgreSQL の各バックエンドプロセス（postmaster, syslogger, walreceiver など）は、複数の接続ソケットや内部同期用のラッチを同時に監視する必要があります。WaitEventSet を使用することで、システムコールのオーバーヘッドを削減し、高パフォーマンスな多重待ち（I/O マルチプレクシング）をプラットフォームに依存しない統一された API で実現できます。また、ResourceOwner と連携することで、エラー発生時などのリソースリークを防止します。

主要エンティティ

  • **struct WaitEvent** (src/include/storage/waiteventset.h): 発生したイベントの詳細情報（イベント種別のビットマスク、関連ソケット fd、ユーザー定義データ user_data など）を格納し、呼び出し元へ伝達するための構造体。
  • **struct WaitEventSet** (src/backend/storage/ipc/waiteventset.c / src/include/storage/waiteventset.h): 監視対象のイベント群を管理する不透明な（opaque）構造体。OS 固有の待機用データ（epoll_fd やハンドル配列など）を内包します。
  • **CreateWaitEventSet** (src/backend/storage/ipc/waiteventset.c): 指定された最大イベント数 nevents に応じて WaitEventSet を初期化し、OS 固有の通知メカニズムを準備する関数。
  • **FreeWaitEventSet** (src/backend/storage/ipc/waiteventset.c): WaitEventSet が保持する OS リソース（ファイル記述子など）をクリーンアップし、メモリを解放する関数。

エンドツーエンドの流れ

1. **初期化**: 呼び出し元（postmaster や latch.c など）が CreateWaitEventSet を呼び出し、WaitEventSet を生成します。この際、ResourceOwner を指定すると、リソースのライフサイクル管理に登録されます。 2. **イベント登録**: AddWaitEventToSet（※詳細な実装は本コンテキストの範囲外）を呼び出し、監視したいソケットやラッチ、イベントマスク（WL_SOCKET_READABLE など）をセットに追加します。 3. **待機**: WaitEventSetWait（※詳細な実装は本コンテキストの範囲外）を呼び出して、イベントが発生するかタイムアウトに達するまでプロセスをブロックします。内部的には epoll_wait や kevent などの OS システムコールが実行されます。 4. **イベント処理**: イベントを検知すると、発生したイベント情報が WaitEvent 構造体の配列として返されます。呼び出し元はこの配列を走査し、events ビットマスクや user_data に基づいて適切な処理を実行します。 5. **解放**: 利用が終わった段階で FreeWaitEventSet を呼び出し、OS 固有のファイル記述子のクローズやメモリの解放を行います。

> **注意**: AddWaitEventToSet や WaitEventSetWait の具体的な内部アルゴリズム、およびシグナルハンドラとの競合を回避する詳細なフローは、提供されたモジュール情報からは完全には明らかになっていません。

重要な不変条件

  • **イベント数の上限**: WaitEventSet 内の登録イベント数 nevents は、常に確保された最大スペース nevents_space 以下でなければなりません。
  • **ラッチの整合性**: WL_LATCH_SET が指定されている場合、latch ポインタと latch_pos が正しく設定されている必要があります。
  • **Windows での配置順序**: Windows 環境（WAIT_USE_WIN32）では、シグナル待機を保証するために pgwin32_signal_event が必ずハンドル配列の先頭（インデックス 0）に配置されていなければなりません。
  • **有効なポインタ**: FreeWaitEventSet を呼び出す際、引数 set は有効なポインタでなければなりません（内部で NULL チェックのガードが明示されていないため）。

よくある落とし穴

  • **動的削除の未サポート**: 現在の設計では、一度 WaitEventSet に登録したイベントを動的に削除する機能は提供されていません。
  • **タイムアウトのオーバーヘッド**: 待機処理におけるタイムアウトの指定はシステム的なオーバーヘッドを伴うため、可能な限り避けることが推奨されます。
  • **プラットフォーム依存の挙動**: コンパイルフラグ（WAIT_USE_EPOLL、WAIT_USE_KQUEUE、WAIT_USE_WIN32 など）によって内部実装が大きく異なります。特に Windows 環境では reset フラグの処理など、Unix 系とは異なる挙動に注意が必要です。
  • **型安全ではない user_data**: WaitEvent の user_data フィールドは void * 型であるため、キャスト時の整合性を開発者が保証する必要があります。
