select_loop

概念

select_loop は、PostgreSQL のクライアントツール（pg_dump、pg_upgrade、および共通ユーティリティ fe_utils）において、複数のファイル記述子（FD）に対する I/O 待機（select() システムコール）を、シグナル割り込みやキャンセル要求を考慮して安全に実行するためのラッパー関数です。

なぜ重要か

非同期または並行処理を行うクライアントツールにおいて、OS のシグナル割り込み（EINTR / WSAEINTR）やユーザーからのキャンセル要求（CancelRequested）が発生した場合でも、I/O 監視ループが不意に終了したりハングしたりするのを防ぎ、堅牢な並行処理の制御を維持するために重要です。特に Windows 環境と非 Windows 環境におけるエラーコードやタイムアウト処理の差異を吸収する役割を持ちます。

エンドツーエンドの流れ

1. 各ツール（pg_upgrade や fe_utils を利用するツール）の並行処理管理部（例: wait_on_slots）から select_loop が呼び出されます。 2. select_loop は、監視対象のファイル記述子集合（fd_set）と最大ファイル記述子番号（maxFd）を受け取ります。 3. ループ内で select() システムコールを実行します。

  • **fe_utils の実装**: ループ開始時および実行中に CancelRequested が真であるかをチェックし、真であれば -1 を返して即座に中断します。また、Windows 環境では 1 秒のタイムアウトを設定して定期的にキャンセル要求を監視します。
  • **pg_upgrade の実装**: input および output の fd_set を内部で保存・復元しながら select() を呼び出します。

    • 4. select() が EINTR（または Windows の WSAEINTR）で失敗した場合は、ループを継続して再試行します。 5. 準備が整ったファイル記述子がある場合、または致命的なエラーが発生した場合は、ループを抜けて結果を呼び出し元に返します。

※ pg_dump における具体的なエンドツーエンドの流れの詳細は、提供されたコンテキストからは不明です。

重要な不変条件

  • **破壊的更新の回避**: select() システムコールは引数として渡された fd_set を破壊的に更新するため、select_loop の内部（pg_upgrade の場合は save_input/save_output による復元）または呼び出し元（fe_utils の場合）で、呼び出しごとに fd_set を再構築・復元しなければなりません。
  • **キャンセル即時応答**: fe_utils の select_loop においては、CancelRequested が設定されている場合、即座に -1 を返して処理を中断しなければなりません。
  • **maxFd の境界**: pg_upgrade の実装では、maxFd が 0 の場合は即座に 0 を返さなければなりません。

よくある落とし穴

  • **fd_set の再利用**: select_loop に渡す fd_set は select() によって書き換えられるため、呼び出し元で再利用する際に再構築を怠ると、意図しないファイル記述子の監視漏れや誤検知が発生します。
  • **Windows での FD_SETSIZE 制限**: Windows 環境では FD_SETSIZE の制限（デフォルトで 1024）があるため、これを超えるファイル記述子を扱うとエラーになる可能性があります。
  • **プラットフォーム間の差異**: Windows と非 Windows で割り込みエラーコード（EINTR と WSAEINTR）やタイムアウトの扱いが異なるため、プラットフォーム依存のコード修正を行う際は注意が必要です。
