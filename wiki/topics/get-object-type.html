get_object_type

概念

get_object_type は、PostgreSQLのソースコードにおいて、オブジェクトのタイプ（種類）を特定または変換するために使用される関数名です。この名前を持つ関数は、以下の2つの異なる文脈で定義されています。

1. **クライアントツール側 (pg_dump)**: フィルタ設定ファイル内の文字列キーワード（table, schema, function など）を、プログラム内部で利用可能な FilterObjectType 列挙型へ変換します。 2. **バックエンド側 (catalog)**: src/backend/catalog/objectaddress.c に定義されていますが、クライアントツール側とはシグネチャや役割が異なる別関数です。

なぜ重要か

  • **pg_dumpにおける役割**: ユーザーが指定したフィルタ設定ファイルを解析し、文字列からプログラムが理解できる列挙型へ安全にマッピングするために重要です。
  • **バックエンドにおける役割**: 現時点では詳細な役割は不明ですが、システムカタログ内のオブジェクトアドレスに関連する処理を担っていると推測されます。

関与するモジュール

  • module:src/backend/catalog
  • module:src/bin/pg_dump

主要エンティティ

  • symbol:src/bin/pg_dump/filter.c#get_object_type: フィルタ設定ファイルのキーワードを FilterObjectType に変換する静的関数。
  • symbol:src/backend/catalog/objectaddress.c#get_object_type: バックエンド側でオブジェクトタイプを扱う関数（詳細な仕様は本コンテキストからは不明）。

エンドツーエンドの流れ

pg_dump におけるフィルタ解析フロー

1. src/bin/pg_dump/filter.c 内の filter_read_item がフィルタ設定ファイルを読み込みます。 2. 読み込んだキーワード文字列（keyword）とその長さ（size）を引数として、get_object_type を呼び出します。 3. get_object_type は is_keyword_str を用いて、定義済みのオブジェクト型キーワードと順次比較します。 4. 一致するものが見つかった場合、対応する FilterObjectType を objtype に設定して true を返します。一致しない場合は false を返します。

バックエンド側におけるフロー

  • src/backend/catalog/objectaddress.c における get_object_type の具体的なエンドツーエンドの流れは、現時点では不明です。

重要な不変条件

pg_dump 側

  • 入力される keyword は有効な文字列でなければなりません。
  • 出力先ポインタ objtype は書き込み可能なメモリ領域を指している必要があります。
  • 未知のキーワードが渡された場合、objtype の値は更新されず、関数は false を返します。

バックエンド側

  • 現時点では不変条件は不明です。

よくある落とし穴

  • **同名関数の混同**: pg_dump 側の get_object_type は静的（static）関数であり、バックエンド側の同名関数とは直接の関係はありません。シグネチャも異なるため、混同しないよう注意が必要です。
  • **キーワード追加時の更新漏れ**: pg_dump で新しいオブジェクト型をフィルタリング対象として追加する場合、src/bin/pg_dump/filter.c 内の get_object_type の if-else チェーンに新しいキーワードと列挙値のペアを手動で追加する必要があります。
