datetkn

概念

datetkn は、PostgreSQLのバックエンドおよびECPG（埋め込みSQL Cプリプロセッサ）のクライアントライブラリにおいて、日付・時刻文字列の解析（パース）時に使用されるトークン情報を保持するための共通のデータ構造（struct）です。入力文字列中のキーワード（例: NOW, EPOCH）やフィールド型（例: YEAR, MONTH）を識別し、対応する内部表現の型コードや値へとマッピングするために用いられます。

なぜ重要か

日付・時刻のパース処理は、データベースにおいて頻繁かつ高速に行われる必要があります。datetkn 構造体は、文字列トークン、フィールド型コード、および値をコンパクトに保持し、バイナリサーチ（二分探索）による高速なトークン照合を可能にします。また、バックエンド（datetime.h）とECPG（dt.h）の双方で同様の構造体を定義・同期させることで、一貫した日付・時刻解析ロジックを提供しています。

エンドツーエンドの流れ

1. **入力とトークン化**: 解析関数（例: ParseDateTime）が入力文字列をトークンに分割します。 2. **トークン照合**: 分割されたトークン文字列を、datetkn 型の静的配列（datetktbl や deltatktbl など）に対して datebsearch 関数を用いてバイナリサーチ（二分探索）を行います。 3. **値の変換**: 一致する datetkn が見つかった場合、その type（DTK_* マクロに対応）と value を用いて、内部表現（pg_tm や fsec_t など）への変換処理が行われます。

※なお、ECPGライブラリ内における具体的な呼び出し元や、詳細なエンドツーエンドの制御フローの全容は、提供されたソース情報からは完全には特定されていません。

重要な不変条件

  • **NUL終端**: token フィールド（サイズ: TOKMAXLEN + 1）は、常に NUL 終端されている必要があります。
  • **テーブルのソート順**: datetktbl などの静的配列において、datebsearch によるバイナリサーチを正しく機能させるため、要素は常にアルファベット順にソートされていなければなりません。
  • **値の依存性**: value フィールドの意味は、type フィールドの値（型コード）に依存します。

よくある落とし穴

  • **バッファ制限**: トークンの長さが TOKMAXLEN を超える場合、切り詰めや解析エラー、あるいはバッファオーバーランのリスクが生じます。
  • **ソート順の欠如**: 新しいキーワードを追加する際に datetktbl のアルファベット順を維持し忘れると、datebsearch が対象トークンを見つけられなくなります。
  • **同期の不整合**: ECPG側の dt.h に定義されている datetkn はバックエンドの datetime.h と同期しているため、一方を変更する際にもう一方との整合性を損なわないよう注意が必要です。
  • **互換性の影響**: type フィールドのコード値（DTK_*）を変更すると、INTERVAL 型の typmod 等に影響を及ぼし、initdb が必要になる可能性があります。
