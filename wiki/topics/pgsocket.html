pgsocket

概念

pgsocket は、PostgreSQLにおいてプラットフォーム（Unix系およびWindows）間のソケット記述子の違いを抽象化し、移植性を確保するために導入された int 型の typedef 定義です。

なぜ重要か

PostgreSQLはマルチプラットフォームで動作するため、OS依存のソケット型（特にWindowsとUnix系システムの違い）を意識せずにコアコードがネットワーク通信を行えるようにする必要があります。pgsocket を使用することで、ソケット関連の関数シグネチャやデータ構造を統一的に扱うことが可能になり、コードの移植性と保守性が向上します。

関与するモジュール

  • module:src/include
  • module:src/include/replication

主要エンティティ

  • symbol:src/include/port.h#pgsocket: プラットフォーム抽象化の基盤となる pgsocket の型定義。
  • symbol:src/include/replication/walreceiver.h#pgsocket: レプリケーション（WALレシーバーなど）の通信コンテキストで利用される pgsocket の型定義。
  • PGINVALID_SOCKET: 無効なソケット状態を表すために併用される定数。
  • ClientSocket: 接続されたクライアントソケットを保持する構造体。
  • WalReceiverConn: WALレシーバーの接続状態を保持する構造体。
  • walrcv_receive_fn: 待機対象のソケットを特定するために pgsocket を引数として受け取る関数ポインタ。

エンドツーエンドの流れ

ソケットの作成から破棄に至るエンドツーエンドのライフサイクル全体の詳細な流れは、提供された情報からは完全には明らかになっていません。 ただし、部分的な通信の流れとして以下が確認されています： 1. **接続待受と受け入れ**: postmaster.c で待受ソケットの配列が管理され、pqcomm.c の AcceptConnection などで接続が受け入れられます。 2. **認証と通信**: auth.c でIdentサーバーとの通信、または各レプリケーションモジュール（tablesync.c, worker.c, libpqwalreceiver.c）でWAL受信や論理レプリケーションの通信に pgsocket が使用されます。 3. **ソケットの待機**: walrcv_receive_fn などの関数ポインタを介して、待機すべき pgsocket が特定・処理されます。

重要な不変条件

  • **プラットフォーム抽象化の維持**: 内部的には int 型として表現されますが、ソケット関連の関数シグネチャでは int を直接使用せず、必ず pgsocket 型を使用する必要があります。
  • **無効値の表現**: 無効なソケット状態を表す際には、一貫して PGINVALID_SOCKET が使用されます。

よくある落とし穴

  • **Windows環境での互換性**: Windows環境では標準の socket 関数が返す値と pgsocket の互換性を保つため、port/win32_port.h などのレイヤーによるラッピングに注意する必要があります。
  • **型変換の誤り**: int 型の範囲を超える記述子を扱う環境において、単純な int キャストによる適合性の問題が発生する可能性があります。
  • **無効値チェックの欠落**: PGINVALID_SOCKET による適切なエラーチェックや初期化を怠ると、無効なソケットに対する操作が発生し、通信エラーや予期しない動作を引き起こす原因になります。
