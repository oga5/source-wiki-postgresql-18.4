MYTYPE

概念

MYTYPE は、ECPG（Embedded SQL for C）のプリプロセッサ（preproc）および回帰テストにおいて、SQLのホスト変数として利用される構造体（struct mytype）の型エイリアス（typedef struct mytype MYTYPE;）です。データベースから取得した行データを保持し、関数スコープをまたいでデータバインドを行うために使用されます。

なぜ重要か

ECPGプリプロセッサがC言語の構造体をホスト変数として正しく認識し、SQLの実行結果（FETCH 等）をC言語の構造体メンバへ正しくマッピング・バインドできるかを検証するためのテスト用データ構造として重要です。特に、関数スコープ外（out of scope）での変数バインドや、動的メモリ確保（malloc）を伴うユースケースのテストにおいて中心的な役割を果たします。

  • **symbol:src/interfaces/ecpg/test/expected/preproc-outofscope.c#MYTYPE**: プリプロセッサによって生成されたCコード内での実際の利用箇所。get_var1 関数の引数型、malloc によるメモリ確保、および ECPGget_var を介したデータバインド処理で参照されます。

エンドツーエンドの流れ

1. **定義**: struct.h にて typedef struct mytype MYTYPE; として定義されます。構造体は id (int), t (char[64]), d1 (double), d2 (double), c (char[30]) などのメンバで構成されます。 2. **メモリ確保**: preproc-outofscope.c の main 関数内で、malloc を用いて MYTYPE のインスタンスがヒープ上に確保されます。 3. **変数登録**: 確保されたポインタは get_var1 関数に渡され、ECPGの変数管理システムに登録されます。 4. **データバインドと実行**: SQL操作（FETCH など）の実行時、ECPGランタイムは ECPGget_var(0) などのAPIを介してこの構造体のアドレスを特定し、取得したデータを各メンバに格納します。

※なお、ECPGプリプロセッサが .pgc ファイルからこのCコードを生成する詳細な変換ルールや、ランタイム内部の完全なエンドツーエンドのバインドフローの全容は、提供されたシンボル情報からは完全には明らかになっていません。

重要な不変条件

  • **メモリの事前確保と登録**: MYTYPE 構造体へのアクセスやデータ格納を行う前に、メモリが適切に確保され、get_var1 などを通じてECPGランタイムに登録されている必要があります。
  • **インデックスの整合性**: ECPGget_var を介したアクセスは、適切なインデックス（例: 0）で初期化・管理されている必要があります。
  • **レイアウトの一致**: 構造体のメモリレイアウト（メンバの型やサイズ）は、ECPGプリプロセッサが生成する ECPGdo マクロ内の型定義およびサイズ指定（sizeof(struct mytype)）と完全に一致している必要があります。

よくある落とし穴

  • **手動変更による不整合**: MYTYPE はECPGプリプロセッサによって処理・生成されるコードの一部であるため、手動で構造体の定義を変更すると、プリプロセッサが生成するバインド用のメタデータ（sizeof や型指定）と不整合を起こし、不正なメモリアクセスやバインドエラーの原因になります。
  • **メモリ確保の失敗**: malloc によるメモリ確保が失敗した場合、構造体へのアクセスは未定義動作を引き起こします。
  • **NULLインジケータの欠如**: MYTYPE の各フィールドに対応するNULL状態を管理するためには、対となる MYNULLTYPE 構造体を適切に併用する必要があります。これを怠ると、データベース上のNULL値を正しく扱えません。
