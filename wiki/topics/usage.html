usage

概念

usage は、PostgreSQL の各種コマンドラインツール、ユーティリティ、およびテストモジュールにおいて、プログラムの利用方法、サポートされているコマンドラインオプション、接続設定、およびヘルプメッセージを標準出力（またはページャ）に表示するための共通の関数パターンです。

なぜ重要か

ユーザーがコマンドライン引数を誤って指定した場合や、--help や -? オプションを指定した際に、適切な使用方法やオプション一覧を提示することで、ツールの正しい操作を支援します。また、ビルド時のメタデータ（バグ報告先 URL やパッケージ名など）を埋め込んで出力する役割も担っています。

主要エンティティ

各モジュールで定義されている代表的な usage 関数シンボルは以下の通りです。

  • symbol:contrib/vacuumlo/vacuumlo.c#usage: vacuumlo の利用方法と接続オプションを表示する。
  • symbol:src/bin/pg_archivecleanup/pg_archivecleanup.c#usage: pg_archivecleanup のオプションと archive_cleanup_command の設定例を表示する。
  • symbol:src/bin/pg_basebackup/pg_receivewal.c#usage: pg_receivewal のオプション（ディレクトリ、LSN、圧縮設定など）を表示する。
  • symbol:src/bin/pg_basebackup/pg_recvlogical.c#usage: pg_recvlogical のアクション（スロット作成・削除・開始など）とオプションを表示する。
  • symbol:src/bin/pg_checksums/pg_checksums.c#usage: pg_checksums のオプションと PGDATA 環境変数に関する情報を表示する。
  • symbol:src/bin/pg_controldata/pg_controldata.c#usage: pg_controldata のオプションと PGDATA のデフォルト動作を表示する。
  • symbol:src/bin/pg_dump/pg_restore.c#usage: pg_restore の復元制御オプションや接続オプションを表示する。
  • symbol:src/bin/pg_rewind/pg_rewind.c#usage: pg_rewind のソース指定や同期方法などのオプションを表示する。
  • symbol:src/bin/pg_upgrade/option.c#usage: pg_upgrade のオプション、実行前の必須手順、および実行例を表示する。
  • symbol:src/bin/psql/help.c#usage: psql のオプションを、ページャ（pager）の指定に応じてバッファ経由で出力する。
  • symbol:src/test/modules/oauth_validator/oauth_hook_client.c#usage: テスト用クライアント oauth_hook_client のテスト用フラグ一覧を表示する。

エンドツーエンドの流れ

各コマンドラインツールにおける一般的な実行フローは以下の通りですが、詳細なエンドツーエンドの呼び出しフローや、すべてのモジュールにおける共通の呼び出しライフサイクルは完全には統一されておらず、一部のモジュールでは明示的な呼び出し元が提供されたソースコードの範囲内では未確定です。

一般的な想定フロー： 1. ユーザーがコマンドラインからツールを実行する（例: 不正な引数の指定、または -? / --help の指定）。 2. main 関数（または引数解析ルーチン）がこれを検知する。 3. 各モジュール固有の usage 関数が呼び出される。 4. usage 関数内で printf や fputs（psql の場合は PageOutput 経由）を用いてヘルプメッセージを出力する。 5. 呼び出し元、あるいは usage 関数自体によってプログラムが終了（exit）する。

重要な不変条件

  • **プログラム名の参照**: usage 関数は、実行ファイル名（progname または argv[0]）を動的に埋め込んで出力する必要があります。
  • **ビルドメタデータの整合性**: PACKAGE_BUGREPORT、PACKAGE_NAME、PACKAGE_URL などのマクロがビルド時に定義されていることを前提とし、これらを用いてバグ報告先やプロジェクト情報を出力します。
  • **国際化（i18n）**: ユーザー向けのメッセージは、_() マクロ（gettext）を介してローカライズ可能である必要があります。

よくある落とし穴

  • **オプション追加時の同期漏れ**: 新しいコマンドラインオプションを追加・変更した際に、対応する usage 関数内のヘルプメッセージの更新を忘れると、ユーザーに古い情報が表示されてしまいます。
  • **ドキュメントとの不一致**: psql など一部のツールでは、usage の出力内容と公式ドキュメント（SGMLファイルなど）の内容を手動で同期させる必要があり、更新漏れが発生しやすいです。
  • **引数の未定義動作**: oauth_hook_client.c のように argv[0] を直接参照する場合、argv が適切に初期化されていないと未定義動作（セグメンテーションフォールトなど）を引き起こす可能性があります。
