CreateReplicationSlot

概念

CreateReplicationSlot は、PostgreSQLにおいてレプリケーションスロット（物理または論理）を作成するための処理です。この処理は、クライアント側（pg_basebackup などのツール）がサーバーに対して作成要求コマンドを構築・送信するフェーズと、サーバー側（walsender プロセス）がその要求を受け取って実際にスロットの作成、初期化、永続化を行うフェーズの双方に、同名の関数として実装されています。

なぜ重要か

レプリケーションスロットは、スタンバイサーバーや論理デコーディングクライアントが必要とするWAL（Write-Ahead Log）がプライマリサーバー上で削除されないように保護するために不可欠です。クライアントツール（pg_basebackup, pg_receivewal, pg_recvlogical）から安全かつ一貫した方法でスロット作成を要求し、サーバー側で物理・論理の要件（スナップショットのエクスポートやWALの予約など）を満たしながらスロットを初期化・永続化する仕組みを提供するため、システムのデータ整合性と継続的なレプリケーションの基盤となります。

関与するモジュール

  • module:src/backend/replication: サーバー側でのレプリケーション制御および walsender によるスロット作成処理を担当します。
  • module:src/bin/pg_basebackup: クライアント側ツールにおける接続管理とコマンド発行を担当します。

主要エンティティ

  • symbol:src/backend/replication/walsender.c#CreateReplicationSlot: サーバー側でクライアントからの CREATE_REPLICATION_SLOT コマンドを処理し、物理/論理スロットの作成、WAL予約、スナップショット処理、結果送信を行う静的関数。
  • symbol:src/bin/pg_basebackup/streamutil.c#CreateReplicationSlot: クライアント側でサーバー接続（PGconn）を介して適切なオプション付きの CREATE_REPLICATION_SLOT コマンドを構築・実行する関数。

エンドツーエンドの流れ

提供された情報から、クライアントからサーバーへの大まかな流れは以下の通りです。

1. **クライアント側での要求生成**: pg_basebackup などのツールが streamutil.c の CreateReplicationSlot を呼び出します。サーバーのバージョンや指定された引数（物理/論理、一時的か、WAL予約、2フェーズコミット、フェイルオーバーなど）に応じて CREATE_REPLICATION_SLOT コマンド文字列を動的に構築し、PQexec で送信します。 2. **サーバー側でのコマンド受信**: walsender.c のコマンドディスパッチャ（exec_replication_command 等）がコマンドを受信し、サーバー側の CreateReplicationSlot を呼び出します。 3. **サーバー側でのスロット作成と初期化**:

  • **物理スロットの場合**: ReplicationSlotCreate で作成し、必要に応じて ReplicationSlotReserveWal でWALを予約してディスクに保存します。
  • **論理スロットの場合**: ReplicationSlotCreate でスロットを作成し、デコーディングコンテキストの初期化（CreateInitDecodingContext）、開始地点の特定（DecodingContextFindStartpoint）、スナップショット操作（エクスポート等）を行い、永続スロットの場合は ReplicationSlotPersist で保存します。

    • 4. **結果の返却**: サーバーは作成されたスロット名、整合性ポイント（LSN）、スナップショット名などの結果をクライアントへ送信します。クライアント側はこれを受信して成功判定を行います。

※詳細な通信プロトコルや、コマンドディスパッチャから CreateReplicationSlot に至る厳密な内部パスなどの完全なエンドツーエンドのフローは、提供されたモジュール情報からは一部未詳です。

重要な不変条件

  • **排他性**: サーバー側で処理を実行する際、MyReplicationSlot が NULL でなければなりません（すでにスロットを保持している状態で重ねて作成することはできません）。
  • **物理スロットの制約**: 物理スロット作成時（is_physical が true）は、出力プラグイン（plugin）は NULL でなければならず、2フェーズコミット（two_phase）やフェイルオーバー（failover）オプションを同時に指定することはできません。
  • **論理スロットの制約**: 論理スロット作成時（is_physical が false）は、出力プラグイン（plugin）が NULL であってはなりません。また、論理レプリケーションの要件（CheckLogicalDecodingRequirements()）を満たす必要があります。
  • **スナップショット操作の制約**: サーバー側で CRS_USE_SNAPSHOT を使用する場合、トランザクションブロック内であること、REPEATABLE READ 隔離レベルであること、読み取り専用トランザクションであること、クエリ実行前であること、サブトランザクションでないことが必須となります。

よくある落とし穴

  • **サーバーバージョンによる構文差異**: PostgreSQL 15以降では括弧付きの新しいオプション構文が導入されており、クライアント側（streamutil.c）でバージョン判定を誤ると、古いサーバーに対して未対応のオプションを送信してエラーになるか、構文エラーを引き起こす可能性があります。
  • **同名関数の混同**: CreateReplicationSlot という名前の関数は、サーバー側（walsender.c）とクライアント側（streamutil.c）の両方に存在するため、コードリーディングやデバッグ時にどちらを指しているか混同しやすいです。
  • **タイムアウトの懸念**: 論理スロットの初期化（DecodingContextFindStartpoint）は処理に時間がかかる場合があります。サーバー側ではタイムアウトを防ぐために last_reply_timestamp = 0 とする対策が行われていますが、ネットワークやディスクの遅延による影響に注意が必要です。
  • **トランザクション状態の不整合**: スナップショットを使用する際、サーバー側の厳しいトランザクション制約（隔離レベルやサブトランザクションの禁止など）に違反すると、ereport(ERROR) が発生してスロット作成が失敗します。
