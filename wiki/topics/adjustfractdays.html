AdjustFractDays

概念

AdjustFractDays は、浮動小数点数で表現された日数（端数）を、指定されたスケール（倍率）に基づいて整数日数とそれ以下の端数に分割し、日付・時刻構造体の日フィールドおよび秒・マイクロ秒フィールドへ反映・正規化するための内部ユーティリティ関数です。

PostgreSQL のコードベース内では、サーバーバックエンド（datetime.c）と、クライアント側の ECPG ライブラリ（interval.c）の2箇所に、それぞれ異なるシグネチャとデータ構造を持つ同名の static 関数として実装されています。

なぜ重要か

日付や時間間隔（インターバル）の計算において、月や週などの単位から生じる浮動小数点の端数（例: 0.5ヶ月、0.25週など）を、精度を保ちながら正確に日数（tm_mday）およびそれ以下の時間単位（秒やマイクロ秒）へ変換・伝播させるために重要です。これにより、日付計算における端数の不整合を防ぎます。

  • バックエンド用の静的関数。struct pg_itm_in を用いて日付・時刻の内部表現を更新し、オーバーフローチェックを行います。
  • **symbol:src/interfaces/ecpg/pgtypeslib/interval.c#AdjustFractDays**
  • ECPG用の静的関数。標準の struct tm および fsec_t（小数秒）を直接更新します。

エンドツーエンドの流れ

具体的なエンドツーエンドの呼び出しフロー全体（どのSQLクエリやAPI呼び出しからどのような経路をたどってこれらの関数に到達するか）の詳細は、提供されたコンテキストからは完全には明らかになっていません。

各モジュール内における局所的な処理の流れは以下の通りです。

1. バックエンド側 (datetime.c)

1. 日付計算処理などの呼び出し元から、端数 frac、スケール scale、および pg_itm_in 構造体が渡されます。 2. frac に scale を乗算して整数部分（extra_days）を抽出します。 3. pg_add_s32_overflow を使用して、itm_in->tm_mday に extra_days を安全に加算します。 4. 残りの端数（frac - extra_days）を AdjustFractMicroseconds に渡し、USECS_PER_DAY をスケールとしてマイクロ秒単位の調整を行います。

2. ECPG側 (interval.c)

1. インターバル処理の各所（162, 166, 208, 519行目など）から呼び出されます。 2. frac に scale を乗算して総日数を算出し、整数部分（extra_days）を tm->tm_mday に加算します。 3. 残りの端数を算出し、AdjustFractSeconds を呼び出して fsec および tm の秒単位の調整を行います。

重要な不変条件

  • **早期リターン**: いずれの実装においても、入力 frac が 0 の場合は何も処理を行わずに即座にリターンします。
  • **バックエンド版の前提条件**: datetime.c の実装では、入力 frac の絶対値が 1 未満であることを前提として設計されています。
  • **ポインタの有効性**: 渡される構造体ポインタ（itm_in、tm、fsec）は、有効なメモリ領域を指している必要があります。

よくある落とし穴

  • **オーバーフロー処理の差異**:
  • バックエンド版は pg_add_s32_overflow を用いて 32 ビット整数のオーバーフローを検知し、失敗時に false を返します。
  • 一方、ECPG版は void を返し、オーバーフローチェックを行わずに tm->tm_mday に直接加算するため、月や年を跨ぐ繰り越しやオーバーフローの挙動は struct tm の仕様や呼び出し側に依存します。
  • **前提条件の不一致**: バックエンド版を呼び出す際は、frac の絶対値が 1 未満である制約が呼び出し側で守られていることを確認する必要があります。
  • **データ構造の差異**: ECPG版はバックエンドのロジックを移植したものですが、バックエンドの struct pg_itm_in（または struct pg_tm）ではなく標準の struct tm を使用しているため、構造体の差異に注意が必要です。
