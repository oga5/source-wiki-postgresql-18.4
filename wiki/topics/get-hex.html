get_hex

概念

get_hex は、PostgreSQLの複数の独立したモジュール（バックエンド、ECPG、libpq）において、16進数文字（ASCIIの '0'〜'9', 'a'〜'f', 'A'〜'F'）を対応する数値（0〜15）に変換するために定義されている、パフォーマンス重視のインラインヘルパー関数群です。これらは各モジュール内で static inline として個別に実装されています。

なぜ重要か

16進数データのデコード処理（hex_decode など）は、データベースのデータ変換やクライアント・サーバ間の通信において頻繁に発生します。各モジュールで static inline 関数として get_hex を定義し、静的なルックアップテーブル（hexlookup）を参照することで、関数呼び出しのオーバーヘッドを排除し、高速かつ安全な文字変換を実現しています。

主要エンティティ

  • 入力文字 *cp を hexlookup テーブルで変換し、成否を bool で返し、結果を out に格納します。
  • **symbol:src/interfaces/ecpg/ecpglib/data.c#get_hex**:

  • 入力文字 c を hexlookup テーブルで変換し、変換後の値（または失敗時は -1）を直接返します。

エンドツーエンドの流れ

各モジュールにおける具体的なエンドツーエンドのデータフロー全体は、提供されたコンテキストからは完全には明らかになっていません。

ただし、部分的な流れとして、各モジュール内の hex_decode などのデコード処理において、入力文字列から1文字ずつ読み込み、get_hex を呼び出して数値に変換し、最終的なバイナリデータを再構築する流れが推測されます。

重要な不変条件

  • **ASCII範囲の制限**: 入力文字が ASCII 範囲内（127未満）である必要があります。範囲外の文字が渡された場合、ルックアップテーブルの範囲外アクセスを防ぐため、変換を行わずに即座に失敗（false または -1）を返します。
  • **ルックアップテーブルの整合性**: 各ファイルで定義されている hexlookup テーブルは、有効な16進数文字に対しては 0〜15 の値を、無効な文字に対しては -1 を保持している必要があります。

よくある落とし穴

  • **シグネチャと挙動の不一致**: get_hex はモジュールごとにシグネチャ（引数や戻り値の型）が異なります。例えば、バックエンドの実装（encode.c）は成否を bool で返して結果をポインタ経由で書き込みますが、ECPGの実装（data.c）は変換後の値を直接 char として返します。コードを移植・参照する際は、どのモジュールの get_hex であるかを混同しないよう注意が必要です。
  • **libpqの実装詳細の不足**: src/interfaces/libpq/fe-exec.c における get_hex の具体的なシグネチャや振る舞い、不変条件は現時点では提供されておらず不明です。
