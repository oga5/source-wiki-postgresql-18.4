SplitGUCList

概念

SplitGUCList は、PostgreSQLにおいて区切り文字（カンマなど）で区切られたGUC（Grand Unified Configuration）リスト形式の文字列（識別子やファイル名のリスト）を解析し、個別の要素に分割するための関数です。

バックエンド側（src/backend/utils/adt/varlena.c）とクライアントツール側（src/bin/pg_dump/dumputils.c）の双方に同名の関数が存在し、それぞれ異なるデータ構造（List 構造体、または char ** の動的配列）として結果を返しますが、解析ロジックは同様のルールに従います。

なぜ重要か

PostgreSQLの設定パラメータや各種リスト形式の文字列は、カンマなどの区切り文字や引用符（ダブルクォーテーション "）を含む複雑な構文を持つことがあります。SplitGUCList は、引用符のネスト（"" を単一の " に置換する処理など）や空白のスキップ、構文エラーの検出を正しく処理しながら、安全に文字列をトークン化するために重要です。

また、バックエンドと pg_dump などのユーティリティ間で同様のパースロジックを共有・維持するために、それぞれのコンテキストに応じた実装が用意されています。

エンドツーエンドの流れ

現時点では、システム全体における具体的なエンドツーエンドの呼び出しフロー（どの設定読み込みフェーズやダンプ処理フェーズでどのように本関数がトリガーされ、結果がどう伝播するか）の全体像は完全には明らかになっていません。

ただし、各関数単体における処理の流れは以下の通りです：

1. **入力の受け取り**: 解析対象の文字列 rawstring と区切り文字 separator を受け取ります。 2. **メモリ確保**:

  • **バックエンド版**: 内部で lappend を用いて List を構築します。
  • **クライアント版**: rawstring の長さに応じて pg_malloc でポインタ配列を確保します。

    • 3. **パース処理**:

  • 先頭の空白をスキップします。
  • 引用符付き（" で囲まれた範囲。内部の "" は " に置換）または引用符なし（区切り文字や空白まで）の識別子を抽出します。
  • 抽出した識別子を終端（\0）し、リストまたは配列に追加します。

    • 4. **終了判定**:

  • 構文エラー（引用符の閉じ忘れ、空の識別子、不正な文字など）を検出した場合は直ちに false を返します。
  • 正常に終端に達した場合は true を返します。

重要な不変条件

  • **破壊的変更**: 入力引数 rawstring は関数内部で直接書き換えられ、ヌル文字（\0）の挿入や引用符の置換（collapse）が行われます。呼び出し元は、元の文字列を保持する必要がある場合、事前にコピーを作成しなければなりません。
  • **メモリ管理（クライアント版）**: rawstring は pg_malloc で確保された領域である必要があり、出力される namelist 配列は呼び出し元で解放する必要があります。
  • **空の識別子の禁止**: 引用符なしで区切り文字が連続するなどの理由で識別子が空になる場合は、構文エラーとして false が返されます。

よくある落とし穴

  • **定数文字列の渡し**: rawstring を直接書き換える（破壊的読み込みを行う）ため、読み取り専用の定数文字列（const char *）を直接渡すとセグメンテーションフォールトなどの未定義動作を引き起こします。
  • **重複コードの保守**: src/backend/utils/adt/varlena.c と src/bin/pg_dump/dumputils.c に同名の関数が同様のロジックで独立して実装されています。一方のパース規則やバグ修正を行う際は、もう一方の実装との整合性を確認し、両方を更新する必要があります。
