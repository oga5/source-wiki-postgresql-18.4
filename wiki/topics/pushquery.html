pushquery

概念

pushquery は、PostgreSQLの拡張モジュール（contrib）において、クエリの解析（パース）中に生成されたノードを解析状態の連結リストに追加するための内部関数です。同名の関数が複数のモジュールに存在し、それぞれのクエリ表現（例えば、ltree の ltxtquery や intarray のクエリ）の構築に利用されます。

なぜ重要か

クエリの構文解析において、パースされた演算子やオペランドを一時的なツリー構造や連結リスト（NODE）として正しくスタック・構築していくための基礎的な操作を提供します。これにより、パース処理のモジュール性とメモリ管理（PostgreSQLのメモリコンテキストを利用した palloc）が簡素化されます。

関与するモジュール

  • module:contrib/intarray
  • module:contrib/ltree

主要エンティティ

  • symbol:contrib/ltree/ltxtquery_io.c#pushquery: ltxtquery の解析状態（QPRS_STATE）に新しい NODE を追加する関数。
  • symbol:contrib/intarray/_int_bool.c#pushquery: intarray のクエリ解析で用いられる同名関数（シグネチャや実装は ltree のものとは異なります）。

エンドツーエンドの流れ

contrib/ltree における流れは以下の通りです： 1. クエリの解析ロジック（makepol など）が、パースされたトークン（演算子やオペランド）を処理します。 2. 解析中に新しいノードを生成する際、pushquery が呼び出されます。 3. pushquery は palloc を用いて NODE 構造体をメモリ上に確保し、引数で渡されたノード情報（型、値、距離、長さ、フラグなど）を格納します。 4. 確保されたノードは、解析状態を保持する QPRS_STATE の str が指す連結リストの先頭に挿入され、ノード数（state->num）がインクリメントされます。

※ contrib/intarray における具体的なエンドツーエンドの流れ、および両モジュールを跨ぐような共通のフローは現時点では不明です。

重要な不変条件

contrib/ltree/ltxtquery_io.c#pushquery において、以下の不変条件が強制されます：

  • **距離の制限**: 引数 distance は 0xffff 以下でなければなりません。
  • **長さの制限**: 引数 lenval は 0xff 以下でなければなりません。

    • これらを満たさない場合、ERRCODE_INVALID_PARAMETER_VALUE エラーが発生し、処理が中断されます。

※ contrib/intarray 側の pushquery における不変条件は現時点では不明です。

よくある落とし穴

  • **同名関数の混同**: contrib/ltree と contrib/intarray の pushquery は、名前は同じですがシグネチャや対象とするデータ構造が異なる別個の実装です。ソースコードを追う際や移植する際には、これらを混同しないよう注意が必要です。
  • **サイズ制限超過**: ltree の実装では、distance や lenval が内部バイナリ形式の仕様に基づく上限（それぞれ 0xffff、0xff）を超えるとエラーを投げるため、入力データのバリデーションやパース時の境界値に注意する必要があります。
