_include_path

概念

_include_path は、PostgreSQLのECPG（埋め込みSQL）プリプロセッサにおいて、ヘッダーファイルを検索するためのディレクトリパスを保持・管理するための連結リスト構造体です。

なぜ重要か

ECPGプリプロセッサがソースファイルを処理する際、依存するヘッダーファイルを正しく解決する必要があります。_include_path は、コマンドライン引数などで指定された複数のインクルードパスを順序付きのリストとして保持し、プリプロセッサが適切なディレクトリからファイルを探索できるようにするために重要です。

関与するモジュール

  • module:src/interfaces/ecpg/preproc: プリプロセッサの本体であり、_include_path 構造体の定義やリストの管理を行います。
  • module:src/port: 関連エンティティとして get_include_path が挙げられていますが、プリプロセッサとの具体的な連携の詳細は提供された情報からは不明です。

主要エンティティ

  • symbol:src/interfaces/ecpg/preproc/type.h#_include_path: インクルードパスを保持する連結リスト構造体。
  • symbol:src/port/path.c#get_include_path: 関連するパス取得関数（詳細な仕様や役割は未定義）。

エンドツーエンドの流れ

1. **初期化と追加**: ECPGプリプロセッサの起動時（src/interfaces/ecpg/preproc/ecpg.c）に、コマンドライン引数等で指定されたインクルードパスが解析されます。 2. **メモリ確保**: mm_alloc を使用して _include_path 構造体のインスタンスが動的に確保され、パス文字列が格納されます。 3. **リスト構築**: 確保された構造体は、グローバルなリストの起点（include_paths）を介して連結リストとして構築されます。 4. **探索**: プリプロセッサがファイルを検索する際、このリストを順次走査して対象のディレクトリを決定します。

*注意: src/port/path.c#get_include_path がこの流れの中でどのように呼び出されるか、あるいはECPGプリプロセッサとどのように連携するかといった、システム全体にわたる詳細なエンドツーエンドのフローは現時点では不明です。*

重要な不変条件

  • **リストの終端**: リストの最後の要素の next ポインタは、リストの終端を示すために必ず NULL で終了している必要があります。
  • **有効なパス**: path メンバは、有効なディレクトリパスを示す文字列へのポインタである必要があります。
  • **初期状態**: インクルードパスが指定されていない場合、リストの起点（include_paths）は NULL に初期化され、検索対象のパスが存在しない状態を許容します。

よくある落とし穴

  • **メモリ不足時の動作**: _include_path のメモリ確保（mm_alloc）が失敗した場合、プリプロセッサの動作が停止する可能性があります。
  • **メモリ管理と走査**: リストの走査や解放を行う際、next ポインタを正しく辿る再帰的または反復的な処理を行わないと、メモリリークや不正アクセスを引き起こす可能性があります。

  • src/interfaces/ecpg/preproc/ecpg.c: インクルードパスのリスト管理（初期化、追加、メモリ確保）の実装。
