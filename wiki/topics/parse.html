parse

概念

PostgreSQLのソースツリーにおいて、parse という識別子は、異なる目的を持つ複数のモジュールで構文解析（パース）処理を行う関数として定義されています。具体的には、正規表現エンジンにおける正規表現の構文解析と、ソースコード整形ツール（pg_bsd_indent）におけるC言語コードの構文解析という、2つの独立した文脈で用いられています。

なぜ重要か

それぞれの文脈において、parse 関数は処理の核となる役割を担っています。

  • **正規表現エンジン (src/backend/regex)**: 与えられた正規表現文字列を再帰的に解析して抽象構文木（subre ツリー）を構築し、NFA（非決定性有限オートマトン）の状態遷移と論理的に整合させるための中心的な役割を果たします。
  • **コード整形ツール (src/tools/pg_bsd_indent)**: 字句解析されたトークンをスタックベースの構文コンテキストに適用し、インデントレベルの決定や構文の還元（畳み込み）を行うことで、正しいコード整形を実現するための基盤となります。

関与するモジュール

  • module:src/backend/regex
  • module:src/tools/pg_bsd_indent

主要エンティティ

  • **symbol:src/backend/regex/regcomp.c#parse**: 正規表現の構文を再帰的に解析し、subre 構造体を構築する静的関数。
  • **symbol:src/tools/pg_bsd_indent/parse.c#parse**: トークンを受け取り、グローバルな解析状態（parser_state）を更新する関数。

エンドツーエンドの流れ

各モジュールにおける詳細なエンドツーエンドの全体フロー（呼び出し元から最終的な出力にいたる完全なパス）は、提供された情報からは完全には明らかになっていません。現時点で判明している部分的な流れは以下の通りです。

正規表現の解析 (regcomp.c)

1. parse が呼び出され、do-while ループ内で parsebranch() を呼び出して正規表現の各枝（| で区切られた要素）を解析します。 2. 各枝を subre ツリーの兄弟ノードとして連結し、フラグを更新します。 3. 解析終了後、最適化処理（単一の枝の簡略化など）を行い、構築された subre 構造体へのポインタを返します。

インデント整形の解析 (parse.c)

1. 字句解析器からトークンコード tk を受け取って parse(tk) が呼び出されます（indent.c がメインループとして呼び出すことが示唆されています）。 2. else を伴わない if 文の還元などの前処理を行います。 3. switch(tk) により、トークン種別（decl, ifstmt, lbrace など）に応じてスタック操作とインデントレベルの調整を行います。 4. reduce() を呼び出してスタック上のパターンを評価し、文を畳み込みます。

重要な不変条件

  • **正規表現の解析 (regcomp.c)**:
  • 引数 stopper は ')' または EOS（文字列の終端）である必要があります。
  • NOERRN() マクロにより、エラー発生時は即座に NULL を返してエラー状態を伝播させます。
  • 構築される subre ツリーは、指定された NFA の開始状態（init）および終了状態（final）と論理的に整合していなければなりません。
  • **インデント整形の解析 (parse.c)**:
  • スタックポインタ ps.tos は、常にスタック配列 ps.p_stack の有効な範囲内に収まっていなければなりません。
  • reduce() の呼び出しにより、スタックは常に安定した（還元可能なパターンが処理された）状態に保たれる必要があります。

よくある落とし穴

  • **正規表現の解析 (regcomp.c)**:
  • lastbranch の連結処理や branches->flags の更新ロジックは、正規表現の結合性と最適化に直結しているため、不用意に変更すると解析結果や最適化の挙動が壊れる危険性があります。
  • **インデント整形の解析 (parse.c)**:
  • ps.tos がスタックの最大サイズを超えると、errx(1, "Parser stack overflow") によりプログラムが強制終了します。
  • reduce() は内部で無限ループとして実装されているため、新しい構文ルールを追加する際にスタックが正しく収束しないと、無限ループに陥るリスクがあります。
  • ps.p_stack はインデント計算の直接的な根拠となるため、indent_globs.h の定義との整合性を常に維持する必要があります。
