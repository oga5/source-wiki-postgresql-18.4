TupleDesc

概念

TupleDesc（タプル記述子 / Tuple Descriptor）は、PostgreSQLにおいてリレーションやクエリ結果の行構造（列数、各列の型、制約など）を保持するメタデータ構造体（TupleDescData へのポインタ型）です。実行エンジンやアクセスメソッドが、メモリ上のタプルを解釈・変形（Deformation）するための基盤として機能します。

なぜ重要か

実行エンジンがタプルから値を取得したり、タプルのバイト列を解析したりする際、そのタプルがどのような属性構成（列数、型情報など）を持つかを識別・検証するために不可欠です。また、タプル解釈を高速化するための内部構造体 CompactAttribute を保持し、タプルアクセスの最適化に寄与します。

主要エンティティ

  • **symbol:src/backend/access/common/tupdesc.c#CreateTupleDesc**: 指定された属性定義（Form_pg_attribute）の配列から新しい TupleDesc を作成・初期化する関数。
  • **symbol:src/backend/access/common/tupdesc.c#FreeTupleDesc**: TupleDesc およびその内部の制約情報（TupleConstr）に関連するメモリを解放する関数。

エンドツーエンドの流れ

システム全体における完全なエンドツーエンドのライフサイクルは未だ完全には明らかになっていませんが、提供されたモジュールから判明している部分的な流れは以下の通りです。

1. **生成**: CreateTupleDesc が呼び出され（ブートストラップ処理やシステムカタログ初期化時など）、指定された属性数（natts）と属性定義（attrs）に基づいて TupleDesc がメモリ上に確保・初期化されます。この際、内部的に CompactAttribute が生成・同期されます。 2. **利用**: 実行エンジンがタプルから値を取得する際、TupleDesc（例えば execExpr.h 内の fetch 構造体の known_desc）を参照して属性情報（型や列数など）を特定し、タプル変形（Deformation）やアクセス最適化を行います。 3. **解放**: 参照カウント（tdrefcount）が 0 以下になると、FreeTupleDesc が呼び出され、内部の制約情報（デフォルト値、欠損値、チェック制約など）とともにメモリが解放されます。

重要な不変条件

  • **参照カウントによる管理**: TupleDesc のライフサイクルは参照カウント（tdrefcount）によって管理されます。FreeTupleDesc を呼び出す際、tdrefcount が 0 以下であることが前提条件（Assert で検証）となります。ただし、tdrefcount が -1 の場合はキャッシュ管理対象外（スタック上や一時的なメモリコンテキスト）であることを示し、Pin/Release 操作は無効となります。
  • **属性定義の同期**: カタログ由来の属性定義（FormData_pg_attribute）を変更した後は、必ず populate_compact_attribute() を呼び出して compact_attrs（CompactAttribute）と同期させ、整合性を維持しなければなりません。
  • **属性配列の整合性**: CreateTupleDesc の呼び出し時、入力される attrs 配列には少なくとも指定された属性数（natts）個の有効な属性定義が含まれている必要があります。

よくある落とし穴

  • **最適化情報の同期漏れ**: populate_compact_attribute() の呼び出しを怠ると、CompactAttribute と実際の属性定義との間に不整合が生じ、タプル変形時に予期しない動作を引き起こす可能性があります。
  • **参照カウントの管理不備**: 参照カウントの増減（PinTupleDesc / ReleaseTupleDesc）を誤ると、メモリリークが発生するか、もしくは使用中の TupleDesc が早期に解放されてしまう危険性があります。
  • **制約情報のコピー**: CreateTupleDesc は属性定義のコピーのみを行い、制約（TupleConstr）のコピーは行いません。制約が必要な場合は CreateTupleDescCopyConstr などの別の関数を使用する必要があります。
  • **未定義状態の参照**: execExpr.h の fetch 構造体において、known_desc が NULL の場合や fixed フラグが偽の場合は、タプル構造が特定できないか信頼できないため、直接的な TupleDesc の利用には注意が必要です。
