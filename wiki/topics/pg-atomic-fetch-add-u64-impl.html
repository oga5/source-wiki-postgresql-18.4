pg_atomic_fetch_add_u64_impl

概念

pg_atomic_fetch_add_u64_impl は、64ビット符号なし整数（pg_atomic_uint64）に対してアトミックな加算操作を行い、加算前の古い値を返却するためのプラットフォーム固有またはフォールバック用の低レベル実装関数です。

なぜ重要か

PostgreSQL はマルチプロセス環境で動作するため、共有メモリ上のカウンタやリソース管理において、競合（データレース）を起こさずに安全かつ高速に値を更新する必要があります。この関数は、ハードウェアやコンパイラが提供する最適なアトミック命令（またはスピンロックによるシミュレーション）を抽象化し、プラットフォームに依存しない一貫したアトミック操作を提供するために不可欠です。

主要エンティティ

本関数は、動作環境やコンパイラに応じて複数のファイルで個別に定義されています。

  • **symbol:src/backend/port/atomics.c#pg_atomic_fetch_add_u64_impl**

    • ハードウェアによるネイティブな64ビットアトミック操作がサポートされていない環境向けのフォールバック実装。スピンロックを用いてアトミック性をシミュレートします。

  • **symbol:src/include/port/atomics/arch-x86.h#pg_atomic_fetch_add_u64_impl**

    • x86_64 アーキテクチャ向けの実装。インラインアセンブラで lock xaddq 命令を直接実行します。

  • **symbol:src/include/port/atomics/arch-ppc.h#pg_atomic_fetch_add_u64_impl**

    • PowerPC アーキテクチャ向けの実装。ldarx/stdcx. 命令のペアによる LL/SC（Load-Link/Store-Conditional）ループを使用します。

  • **symbol:src/include/port/atomics/generic-msvc.h#pg_atomic_fetch_add_u64_impl**

    • MSVC（Windows 64bit）環境向けの実装。コンパイラ組み込み関数 _InterlockedExchangeAdd64 を使用します。

  • **symbol:src/include/port/atomics/generic-gcc.h#pg_atomic_fetch_add_u64_impl**

    • GCC 環境向けの汎用実装（詳細な内部フローは提供されたコンテキストからは未定義）。

  • **symbol:src/include/port/atomics/generic.h#pg_atomic_fetch_add_u64_impl**

    • プラットフォーム非依存のフォールバックや共通定義を処理するヘッダ。

エンドツーエンドの流れ

上位モジュールがアトミックな加算を要求すると、コンパイル時に適切な実装が選択されます。

1. **呼び出し**: 上位モジュールが pg_atomic_fetch_add_u64 などの抽象化 API を呼び出します。 2. **プラットフォーム固有の処理**:

  • **x86_64**: lock xaddq 命令により、マルチコア間でのキャッシュコヒーレンシを保証しつつ、メモリ上の値を直接加算し、加算前の値をレジスタ経由で返します。
  • **PowerPC**: sync 命令でメモリバリアを張り、ldarx で値をロードして予約をセットし、加算後に stdcx. で書き込みます。書き込みが競合して失敗した場合はループして再試行します。
  • **MSVC**: Windows の _InterlockedExchangeAdd64 を呼び出してアトミックに加算します。
  • **フォールバック (ネイティブサポートなし)**: SpinLockAcquire で ptr->sema をロックし、値を加算して退避しておいた古い値を返した後、SpinLockRelease でロックを解除します。

※ 上位の抽象化 API（pg_atomic_fetch_add_u64）からこれらの _impl 関数がどのようにマクロやインライン関数で条件分岐・マッピングされるかの完全な定義は、提供されたコンテキストからは明らかではありません。

重要な不変条件

  • **アライメントの保証**: ネイティブなハードウェア命令（x86 や PowerPC など）が正しく動作するためには、対象となる pg_atomic_uint64 ポインタ（ptr）がメモリ上で適切にアライメントされている必要があります。
  • **スピンロックの初期化**: ネイティブサポートがなく atomics.c のシミュレーション実装にフォールバックする場合、ptr 構造体内のスピンロック（sema）が事前に適切に初期化されている必要があります。
  • **メモリ順序（メモリバリア）**: 各プラットフォームの実装（PowerPC の sync/lwsync や MSVC の組み込み関数など）は、適切なメモリバリアを内包または適用し、操作前後でのメモリ可視性の整合性を保証します。

よくある落とし穴

  • **シミュレーションによる性能低下**: ネイティブな64ビットアトミック命令がサポートされない環境では、スピンロックによるシミュレーション（atomics.c）にフォールバックするため、ロック競合やデッドロックのリスクが生じ、パフォーマンスが大幅に低下します。
  • **32ビット環境での制限**: arch-x86.h や generic-msvc.h の実装は64ビット環境を前提としており、32ビット環境で呼び出された場合はコンパイルエラーや未定義動作を引き起こす可能性があります。
  • **直接呼び出し**: 開発者はこれらの _impl 関数を直接呼び出すべきではありません。移植性を維持するため、常に pg_atomic_fetch_add_u64 などの共通抽象化インターフェースを経由する必要があります。
