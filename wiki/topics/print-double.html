print_double

概念

print_double は、PostgreSQLのECPG（Embedded SQL in C）テストスイートにおいて、プラットフォーム間で異なる浮動小数点数（double）の指数表記を正規化し、標準出力（stdout）へ書き出すためのヘルパー関数です。

なぜ重要か

Windows環境（WIN32）では、sprintf や printf による浮動小数点数の指数表記が3桁（例: e+001）になる場合があります。一方、非Windows環境では通常2桁（例: e+01）で出力されます。この表記の差異があると、テスト結果と期待値（expectedファイル）との比較（diff）で不一致が発生し、回帰テストが失敗してしまいます。print_double はこの表記揺れを吸収し、テスト結果の安定性と一貫性を担保するために極めて重要です。

エンドツーエンドの流れ

テスト実行時における詳細なエンドツーエンドの呼び出しフロー全体（どのテストケースからどのような順序で呼び出されるかなど）は、提供されたコンテキストからは完全には明らかになっていません。

しかし、本関数単体における処理の流れは以下の通りです：

1. **入力**: double 型の値 x を引数として print_double(x) が呼び出されます。 2. **非Windows環境（WIN32 未定義）**:

  • 標準の printf("%g", x) を直接実行して終了します。

    • 3. **Windows環境（WIN32 定義時）**:

  • sprintf を使用して、x をローカルの文字配列バッファ convert（128バイト）に文字列として書き出します。
  • 文字列の末尾付近を走査し、指数部が3桁（例: e+0xx）であるかを確認します。
  • 3桁の指数部を検出した場合、余分な 0 を1つ削除して2桁（例: e+xx）に詰め、文字列を再構成します。
  • 最終的に、加工後の文字列を printf で標準出力に出力します。

重要な不変条件

  • **スコープの限定**: static 関数として定義されており、インクルードした各ソースファイル内で個別にコンパイルされます。
  • **出力の正規化**: Windows環境下において、指数部が3桁（例: e+001）である場合に、これを必ず2桁（例: e+01）に変換して出力します。
  • **非破壊的なフォールバック**: 指数表記を含まない値や、文字列長が極端に短い場合（例: vallen が 6 未満）、または指数部が既に2桁以下の場合は、変換処理をスキップしてそのまま出力します。

よくある落とし穴

  • **バッファオーバーフローの潜在的リスク**: 内部で128バイトの固定長バッファ（char convert[128]）を使用しており、sprintf による変換結果がこれを超えるような極端な数値表現の場合、バッファオーバーフローを引き起こす可能性があります（ただし、通常の double 出力では十分に収まる想定となっています）。
  • **プラットフォーム依存のオーバーヘッド**: Windows環境でのみ文字列の走査と再構成が発生するため、パフォーマンス特性が異なります。
  • **コードの重複と同期**: printf_hack.h で定義されているほか、複数のテストファイル（compat_informix-dec_test.c、pgtypeslib-num_test.c、pgtypeslib-num_test2.c）に同等のロジックが埋め込まれているため、修正時には影響範囲に注意する必要があります。
