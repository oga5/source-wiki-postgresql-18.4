unlimit_core_size

概念

unlimit_core_size は、プロセスが生成するコアダンプファイルのサイズ制限（ソフトリミット）を、システムが許可する最大値（ハードリミット）まで引き上げるための関数です。これにより、プログラムが異常終了した際にデバッグに必要なコアダンプファイルを確実に取得できるようにします。

なぜ重要か

テストの実行時（pg_regress）やサーバープロセスの管理時（pg_ctl）において、プロセスがクラッシュした際に十分なサイズのコアダンプファイルが生成されないと、事後解析やデバッグが困難になります。本関数を実行して制限を緩和することで、デバッグ情報の欠落を防ぐことができます。

エンドツーエンドの流れ

pg_regress.c における処理の流れは以下の通りです。

1. **初期化**: メイン処理の初期化フェーズから unlimit_core_size が呼び出されます。 2. **現在の制限取得**: getrlimit(RLIMIT_CORE, &lim) を呼び出し、現在のコアダンプ制限を取得します。 3. **判定と更新**:

  • ハードリミット (lim.rlim_max) が 0 の場合、コアダンプが禁止されていると判断し、警告メッセージを出力して処理を終了します。
  • ハードリミットが無限大 (RLIM_INFINITY) であるか、現在のソフトリミット (lim.rlim_cur) がハードリミットより小さい場合、ソフトリミットの値をハードリミットの値まで引き上げます。

    • 4. **制限の適用**: setrlimit(RLIMIT_CORE, &lim) を呼び出し、新しい制限をシステムに適用します。

なお、pg_ctl.c における具体的な呼び出しタイミングや、両モジュール間を跨ぐエンドツーエンドの相互作用の詳細は、提供された情報からは不明です。

重要な不変条件

  • **プラットフォームのサポート**: sys/resource.h が提供する getrlimit、setrlimit、および RLIMIT_CORE がシステムで利用可能である必要があります。
  • **ハードリミットの尊重**: ハードリミットが 0 に設定されている場合は、制限の引き上げを行わずに警告を出力して終了します。
  • その他の不変条件については、現時点では不明です。

よくある落とし穴

  • **ハードリミットが0の環境**: システムやユーザーの設定でハードリミット自体が 0 に制限されている場合、本関数を実行してもコアダンプ制限は解除されず、警告が出力されるのみとなります。
  • **エラーハンドリングの欠如**: setrlimit が権限不足などの理由で失敗した場合、その戻り値のチェックやエラーハンドリングは行われません。
  • **移植性の問題**: プラットフォームによって RLIMIT_CORE の挙動や定義が異なる場合があるため、非UNIX系環境などへの移植時には注意が必要です。
