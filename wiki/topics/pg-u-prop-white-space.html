pg_u_prop_white_space

概念

pg_u_prop_white_space は、指定された Unicode コードポイント（pg_wchar）が Unicode 規格における「空白文字（White Space）」プロパティを保持しているかどうかを判定する機能です。ASCII 範囲の文字に対する高速な判定パスと、それ以外の Unicode 範囲に対する探索パスを組み合わせることで、効率的な判定を実現しています。

なぜ重要か

PostgreSQL が Unicode 文字列を処理する際、様々な言語や規格で定義された空白文字を正確に識別する必要があります。本機能は、ASCII 範囲（0x80 未満）の文字に対しては定数時間（$O(1)$）で判定を行い、それ以上の広域な Unicode 範囲に対しても二分探索を用いて高速に判定を行うことで、文字プロパティ判定のパフォーマンスと正確性を両立させています。

  • **symbol:src/include/common/unicode_category_table.h#PG_U_PROP_WHITE_SPACE**

    • ASCII 範囲の文字プロパティを判定するためのビットマスク定数です。また、同ヘッダーには判定に使用される静的テーブル unicode_white_space などが定義されています。

エンドツーエンドの流れ

1. **呼び出し**: 呼び出し元（category_test.c や unicode_category.c 内部など）から、判定対象の Unicode コードポイント code を引数として pg_u_prop_white_space(code) が呼び出されます。 2. **ASCII 範囲の最適化判定**: 入力された code が 0x80 未満の場合、静的テーブル unicode_opt_ascii を参照し、PG_U_PROP_WHITE_SPACE ビットマスクを用いて即座に結果（true または false）を返します。 3. **広域範囲の二分探索**: code が 0x80 以上の場合、unicode_category_table.h で定義されている空白文字の範囲テーブル unicode_white_space に対し、内部ユーティリティ関数 range_search を用いて二分探索を実行します。 4. **結果の返却**: 探索の結果、該当する範囲内にコードポイントが存在すれば true、存在しなければ false を返します。

※なお、SQL エンジンやパーサなどの上位レイヤーから本関数が呼び出される具体的なエンドツーエンドの全体フローは、提供されたモジュール情報からは未確定です。

重要な不変条件

  • **一貫性の維持**: 0x7F と 0x80 の境界で判定アルゴリズムが切り替わりますが、どちらの経路をたどっても Unicode 規格に準拠した一貫性のある判定結果を返す必要があります。
  • **入力の前提**: 入力される code は、有効な Unicode コードポイント（pg_wchar）である必要があります。
  • **規格の固定性**: 判定基準となるカテゴリ値や範囲テーブルは、Unicode 規格の安定性ポリシーに基づき固定されています。

よくある落とし穴

  • **テーブルの手動変更**:

    • 本関数が参照する unicode_white_space や unicode_opt_ascii などのテーブルは unicode_category_table.h に定義されています。これらは通常、Unicode 規格のデータからスクリプト等を用いて自動生成されるため、手動で直接書き換えると一貫性が失われる原因になります。

  • **パフォーマンス特性の差異**:

    • ASCII 範囲（0x80 未満）は定数時間で判定されますが、それ以外の範囲は二分探索（range_search）を伴うため、大量の非 ASCII 文字を処理する際には相対的に処理コストが変化する点に留意する必要があります。
