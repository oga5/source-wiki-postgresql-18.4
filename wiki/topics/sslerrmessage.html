SSLerrmessage

概念

SSLerrmessage は、OpenSSL のエラーコード（unsigned long）を人間が読める形式のエラーメッセージ文字列に変換するための関数です。PostgreSQL のソースコード内では、バックエンド、フロントエンド（libpq）、および共通暗号化モジュール（cryptohash, hmac）のそれぞれにおいて、同名の static 関数として個別に実装されています。

なぜ重要か

OpenSSL のエラーキューから取得されるエラーコード（ERR_get_error() などによる数値）は、そのままでは開発者や管理者が原因を特定するのが困難です。SSLerrmessage は、OpenSSL の提供する ERR_reason_error_string やシステムエラー（strerror / strerror_r）の情報を統合し、適切なフォールバック処理を行いながら、一貫した可読性の高いエラーメッセージを生成するために重要です。これにより、SSL/TLS 接続や暗号化処理の失敗時のデバッグやログ出力が容易になります。

主要エンティティ

  • **symbol:src/backend/libpq/be-secure-openssl.c#SSLerrmessage**: サーバー側（バックエンド）のSSL通信エラー用。静的バッファ errbuf を使用して const char * を返します。
  • **symbol:src/interfaces/libpq/fe-secure-openssl.c#SSLerrmessage**: クライアント側（libpq）のSSL通信エラー用。malloc で動的にメモリを確保して char * を返します。
  • **symbol:src/common/cryptohash_openssl.c#SSLerrmessage**: 共通暗号ハッシュ処理用。ecode が 0 の場合は NULL を返します。
  • **symbol:src/common/hmac_openssl.c#SSLerrmessage**: 共通HMAC処理用。ecode が 0 の場合は NULL を返します。

エンドツーエンドの流れ

OpenSSL の API 呼び出し（接続確立、証明書の読み込み、ハッシュ計算など）が失敗した際、以下の流れでエラーメッセージが生成されます。

1. 各モジュールで OpenSSL のエラーが発生すると、ERR_get_error() 等によりエラーコード（ecode）が取得されます。 2. 取得された ecode を引数として、それぞれのモジュール内に定義された SSLerrmessage(ecode) が呼び出されます。 3. **変換処理**:

  • ecode が 0 の場合、通信系（be-secure-openssl.c, fe-secure-openssl.c）では "no SSL error reported" を返し、共通系（cryptohash_openssl.c, hmac_openssl.c）では NULL を返します。
  • 0 以外の場合、OpenSSL の ERR_reason_error_string(ecode) を呼び出して標準的なエラー文字列の取得を試みます。
  • 通信系では、OpenSSL 3.0.0 以降のシステムエラー（ERR_SYSTEM_ERROR）の判定や、特定の TLS アラート（fe-secure-openssl.c における SSL_AD_NO_APPLICATION_PROTOCOL など）に応じた詳細なフォールバック処理が行われます。

    • 4. 生成されたエラーメッセージ文字列は、ログ出力、エラー報告、または呼び出し元でのフォールバック処理に利用されます。

※なお、各モジュール間を跨ぐグローバルな呼び出し関係や、エラー発生から最終的なクライアント/サーバー間でのエラー伝播に至る完全なエンドツーエンドのフローの詳細は、提供された情報からは未解明です。

重要な不変条件

  • **メモリ管理の不変条件**:
  • src/interfaces/libpq/fe-secure-openssl.c の実装では、戻り値は malloc で確保されたメモリを指すため、呼び出し元が責任を持って解放しなければなりません（メモリ確保失敗時は静的文字列 ssl_nomem が返されます）。
  • src/backend/libpq/be-secure-openssl.c の実装では、36バイトの静的バッファ errbuf を使用するため、呼び出し元は戻り値の文字列を即座に使用（またはコピー）する必要があります。
  • **エラーコード 0 の扱い**:
  • 通信系（be-secure-openssl.c, fe-secure-openssl.c）では、ecode が 0 の場合に "no SSL error reported" を返します。
  • 共通系（cryptohash_openssl.c, hmac_openssl.c）では、ecode が 0 の場合に NULL を返します。

よくある落とし穴

  • **静的バッファの競合と生存期間**: be-secure-openssl.c の SSLerrmessage は静的バッファ errbuf を使用するため、複数回連続して呼び出すと以前のメッセージが上書きされます。また、マルチスレッド環境での同時呼び出しには注意が必要ですが、現在の PostgreSQL バックエンドのコンテキストでは単一スレッドでの利用を想定しています。
  • **メモリリーク**: fe-secure-openssl.c の SSLerrmessage が返す文字列は動的に確保されているため、呼び出し元で解放を怠るとメモリリークが発生します。
  • **モジュール間の依存関係**: src/common/cryptohash_openssl.c や src/common/hmac_openssl.c で定義されている SSLerrmessage は static 関数ですが、ドキュメントの記述によれば src/backend/libpq/be-secure-openssl.c から参照されているとされており、ビルド構成やリンク時の依存関係に注意が必要です。
