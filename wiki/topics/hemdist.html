hemdist

概念

hemdist は、PostgreSQL の複数の拡張モジュール（contrib）およびコアのデータ型における GiST インデックス実装において、2つのシグネチャ（ビットベクトル）間のハミング距離（またはそれに準ずる差異）を計算するために定義されている関数群です。

GiST インデックスのペナルティ計算（ノード挿入時のコスト評価）や、ページ分割（PickSplit）時のコスト評価の指標として使用されます。

なぜ重要か

GiST インデックスは、データをシグネチャ（ビットベクトル）に圧縮して管理します。新しいデータをインデックスに挿入する際、どのノードに挿入するのが最も適切か（ペナルティが最小か）、またページが溢れたときにどのように分割するのが最適かを決定する必要があります。hemdist は、2つのシグネチャ間の類似度（ハミング距離）を定量化することで、インデックスの検索効率と圧縮効率を最適化するための重要なヒューリスティック計算を提供します。

主要エンティティ

各モジュールで個別に定義されている hemdist 関数が主要なエンティティです。

  • **symbol:contrib/intarray/_intbig_gist.c#hemdist**

    • GISTTYPE オブジェクト間のハミング距離を計算します。引数として siglen（ビットベクトルの長さ）を受け取ります。

  • **symbol:contrib/ltree/_ltree_gist.c#hemdist**

    • ltree_gist 構造体間のハミング距離を計算する静的（static）関数です。引数として siglen を受け取ります。

  • **symbol:src/backend/utils/adt/tsquery_gist.c#hemdist**

    • TSQuerySign（シグネチャ）間のハミング距離を計算する静的（static）関数です。引数に siglen は含まず、2つの TSQuerySign の XOR を取って sizebitvec でカウントします。

  • **その他のエンティティ**

    • contrib/hstore/hstore_gist.c、contrib/pg_trgm/trgm_gist.c、src/backend/utils/adt/tsgistidx.c にも同名の hemdist シンボルが存在します。

エンドツーエンドの流れ

GiST インデックスへの挿入やページ分割の処理において、以下のような流れで呼び出されますが、システム全体にわたる完全なエンドツーエンドの制御フローの詳細は現時点では完全には判明していません。

判明している部分的な流れ

1. **ペナルティ計算 / 分割評価**: GiST インデックスの操作（g_intbig_penalty や g_intbig_picksplit など）の過程で、2つのインデックスエントリーの類似度を評価するために hemdist が呼び出されます。 2. **特殊状態の判定**: intarray や ltree の実装では、シグネチャが「すべて真（ISALLTRUE / LTG_ISALLTRUE）」である特殊な状態を最初に判定します。

  • 両方が ISALLTRUE の場合は距離 0 を返します。
  • 片方のみが ISALLTRUE の場合は、シグネチャの全ビット数からもう一方のセットされているビット数（sizebitvec）を引いた値を返します。

    • 3. **ハミング距離の計算**:

  • 通常状態の場合、hemdistsign を呼び出してビットベクトル間のハミング距離を計算します。
  • tsquery_gist.c の実装では、2つの TSQuerySign の排他的論理和（XOR）を計算し、その結果に対して sizebitvec を呼び出してセットされているビット数をカウントします。

重要な不変条件

  • **シグネチャ長の整合性**:

    • intarray や ltree の実装において、引数として渡される siglen は、対象となるビットベクトルの実際の長さと厳密に整合していなければなりません。不整合がある場合、ビット演算や sizebitvec の処理で予期せぬ動作を引き起こす可能性があります。

  • **特殊状態（ALLTRUE）の優先処理**:

    • intarray や ltree では、全ビットが立っている特殊なノードに対する距離計算が、通常のハミング距離計算よりも優先して処理され、インデックスの圧縮効率を維持する不変のルールとなっています。

  • **TSQuerySign の前提**:

    • tsquery_gist.c において、TSQuerySign はビットマップとして表現されている必要があり、sizebitvec がそのビット数を正確にカウントできることが前提となっています。

よくある落とし穴

  • **同名関数の混同**:

    • hemdist は複数のモジュール（intarray, ltree, tsquery_gist など）で個別に定義されています。これらは名前は同じですが、引数のシグネチャ（siglen の有無など）や対象とするデータ構造（GISTTYPE, ltree_gist, TSQuerySign）が異なります。モジュールをまたいでコードを移植・参照する際には、これらを混同しないよう注意が必要です。

  • **インデックス性能への影響**:

    • hemdist は GiST インデックスの構築や検索の性能に直結するヒューリスティックな計算です。この関数の挙動や計算ロジックを変更すると、インデックスの再構築コストや検索効率に重大な影響を与える可能性があります。
