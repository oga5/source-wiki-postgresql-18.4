formTextDatum

概念

formTextDatum は、指定されたバイト列（char *）と長さ（datalen）から、PostgreSQL の可変長文字列型である text 型（varlena 構造）の Datum を構築するための内部ヘルパー関数です。

なぜ重要か

SP-GiST（Space-Partitioned GiST）インデックスのテキスト処理や、そのテストモジュールにおいて、インデックスの構築、検索、および再構成値（reconstructedValue）の生成時に、text 型の Datum を一貫した形式で動的に生成するために重要です。この関数は、データ長に応じてショートヘッダー形式（VARHDRSZ_SHORT）とロングヘッダー形式（VARHDRSZ）を自動的に選択するため、呼び出し側が varlena のヘッダーサイズや圧縮形式を意識せずに安全に text 型データを構築できます。

エンドツーエンドの流れ

1. **呼び出し**: SP-GiST のインデックス処理やテストモジュールの関数が、部分文字列や再構成された値から新しい text 型の Datum を作成するために formTextDatum を呼び出します。 2. **メモリ確保**: palloc を使用して、datalen + VARHDRSZ 分のメモリ領域を確保します。 3. **ヘッダー設定とコピー**:

  • datalen + VARHDRSZ_SHORT が VARATT_SHORT_MAX 以下の場合は、ショートヘッダー形式（SET_VARSIZE_SHORT）を適用し、データをコピーします。
  • それ以外の場合は、標準のロングヘッダー形式（SET_VARSIZE）を適用し、データをコピーします。

    • 4. **返却**: 構築されたメモリ領域を PointerGetDatum で Datum に変換して呼び出し元に返します。

※なお、SP-GiST全体のインデックス走査や構築における詳細なエンドツーエンドの制御フロー全体は、提供されたコンテキストからは完全には明らかになっていません。

重要な不変条件

  • **メモリコンテキスト**: palloc を使用してメモリを割り当てるため、呼び出し元は適切なメモリコンテキストで実行されていなければなりません。
  • **空データの許容**: datalen が 0 の場合、データ部を持たない空の text 型データが生成されます。

よくある落とし穴

  • **メモリ不足（OOM）**: palloc によるメモリ確保が失敗した場合、PostgreSQL の標準的なエラーハンドリングにより処理がアボートします。
  • **NULLポインタと長さの不整合**: data が NULL であるにもかかわらず datalen が 0 より大きい場合、memcpy の実行時に未定義の動作（セグメンテーションフォールトなど）が発生する可能性があります。
  • **重複定義と保守**: 同名の関数が spgtextproc.c と spgist_name_ops.c の両方に定義されており、共通のロジックを持っています。一方を変更する際は、もう一方のモジュールへの影響や整合性を考慮する必要があります。
