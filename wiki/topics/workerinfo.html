WorkerInfo

概念

WorkerInfo は、PostgreSQLのコードベースにおいて、異なる2つのコンテキストで定義・使用されているデータ構造です。

1. **自動バキューム（バックエンド）**: 自動バキュームワーカープロセスの状態を追跡・管理するためのハンドル（struct WorkerInfoData * の typedef）として機能します。 2. **並列ダンプ（pg_dump）**: pg_dump の並列処理において、リーダープロセスと各ワーカープロセス間の連携や状態を管理するための構造体（struct）として機能します。

なぜ重要か

  • **自動バキュームにおける重要性**: ワーカープロセスの起動から終了にいたるライフサイクル（共有メモリ上の管理リストへの登録、終了時のリソース解放、および他のワーカーへのコスト制限再計算の通知など）を安全に制御するために不可欠です。
  • **pg_dump における重要性**: 並列ダンプの実行時に、リーダープロセスが各ワーカーのデータベース接続や通信スロットの状態を正確に追跡し、並列処理を協調動作させるために必要となります。

主要エンティティ

  • **symbol:src/backend/postmaster/autovacuum.c#WorkerInfo**: 自動バキュームワーカーの状態を指すポインタ型定義。
  • **symbol:src/backend/postmaster/autovacuum.c#FreeWorkerInfo**: ワーカー終了時に共有メモリ上の WorkerInfo を解放し、再調整をトリガーする関数。
  • **symbol:src/bin/pg_dump/parallel.c#WorkerInfo**: pg_dump のリーダーとワーカー間の連携情報を保持する構造体。

エンドツーエンドの流れ

自動バキュームワーカーのライフサイクル

1. **初期化と登録**: ワーカーの起動時に av_startingWorker を介して初期化され、実行中キュー（av_runningWorkers）などの操作に WorkerInfo が使用されます。ワーカープロセス内では、静的変数 MyWorkerInfo を通じて自身の状態が保持されます。 2. **プロセスの終了と解放**: ワーカープロセスが終了する際、on_shmem_exit コールバックを介して FreeWorkerInfo が呼び出されます。 3. **共有メモリの更新**: FreeWorkerInfo は AutovacuumLock を排他取得し、MyWorkerInfo を管理リストから削除して初期化した後、フリーリスト（av_freeWorkers）の先頭に返却します。 4. **再バランスのトリガー**: 最後に AutoVacuumShmem->av_signal[AutoVacRebalance] を true に設定し、残存する他のワーカーに対してコスト制限の再計算を促します。

pg_dump における並列処理フロー

  • pg_dump における WorkerInfo を用いた具体的なインスタンス化、参照箇所、および詳細なエンドツーエンドの制御フローは、提供された情報からは明らかになっていません。

重要な不変条件

  • **自動バキューム側**:
  • WorkerInfo は、必ず共有メモリ領域 AutoVacuumShmem 内で管理される WorkerInfoData インスタンスを指していなければなりません。
  • FreeWorkerInfo の実行時、MyWorkerInfo が有効なワーカー情報を指している必要があります。
  • 共有メモリ上のリスト（フリーリストや実行中キュー）を操作する際は、必ず AutovacuumLock を排他モードで取得している必要があります。
  • **pg_dump 側**:
  • WorkerInfo 構造体の AH メンバはリーダー側のデータベース接続（ArchiveHandle）を指し、slot メンバは当該ワーカーに割り当てられた ParallelSlot を指している必要があります。

よくある落とし穴

  • **名前の混同**: バックエンドの自動バキューム用 WorkerInfo と、クライアントツールである pg_dump の並列処理用 WorkerInfo は、名前は同一ですが全く異なるデータ構造であり、直接的な関連性はありません。開発・保守時にはこれらを混同しないよう注意が必要です。
  • **排他制御の欠如（自動バキューム）**: AutovacuumLock を取得せずに WorkerInfo が関与するリスト操作を行うと、共有メモリのデータ構造に不整合を招く恐れがあります。
  • **デッドロックの危険性（自動バキューム）**: FreeWorkerInfo はプロセス終了処理（on_shmem_exit）の中で確実に実行される必要があります。デッドロックを避けるため、この関数内では AutovacuumLock 以外の新たなロック取得を行わないように設計されています。
  • **NULLアクセス（pg_dump）**: WorkerInfo 内の AH または slot が NULL の状態でアクセスされた場合、並列処理の制御ロジックがクラッシュする原因となります。
