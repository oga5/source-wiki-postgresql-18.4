TopoSort

概念

TopoSort は、依存関係を持つ要素群を整合性のある順序に並び替えるトポロジカルソート（Topological Sort）アルゴリズムの実装です。PostgreSQLにおいては、ダンプデータのオブジェクト復元順序の決定（pg_dump）や、デッドロック検出（storage/lmgr）などの異なる文脈で同名の関数が定義・利用されています。

なぜ重要か

  • **pg_dumpにおける重要性**: データベースオブジェクトを復元する際、テーブルやビュー、関数などの間に依存関係が存在します。これらを正しい順序（依存される側を先に、依存する側を後に）で出力・復元しなければ、整合性エラーが発生して復元に失敗します。TopoSort はこの依存関係を解決した順序を生成するために不可欠です。
  • **storage/lmgrにおける重要性**: デッドロック検出（deadlock.c）においても同名関数が定義されていますが、その詳細な役割や重要性は提供された情報からは不明です。

エンドツーエンドの流れ

pg_dump における流れ

1. **初期化**: 入力された DumpableObject の配列 objs を走査し、各オブジェクトの dumpId に対する「先行制約数（beforeConstraints）」を計算します。 2. **キューへの投入**: 先行制約数が 0 のオブジェクト（依存関係の起点となるもの）を優先度付きキュー（binaryheap）に投入します。 3. **ソート処理**:

  • キューからオブジェクトを1つずつ取り出し、結果配列 ordering の末尾から順に配置します。
  • 取り出したオブジェクトが依存していた他のオブジェクトの先行制約数を減算します。
  • 減算の結果、先行制約数が 0 になったオブジェクトを新たにキューに追加します。

    • 4. **終了判定**: 全てのオブジェクトを処理できれば true を返します。循環参照などにより未処理のオブジェクトが残った状態でキューが空になった場合は false を返します。

※ src/backend/storage/lmgr/deadlock.c における TopoSort のエンドツーエンドの流れは、提供された情報からは不明です。

重要な不変条件

pg_dump における不変条件

  • objs 内の各オブジェクトは dumpId と dependencies（依存先IDリスト）を保持している必要があります。
  • maxDumpId は getMaxDumpId() によって取得可能である必要があります。
  • 循環依存が存在しない場合、すべてのオブジェクトが正しくソートされ、戻り値は true となります。

storage/lmgr における不変条件

  • 提供された情報からは不明です。

よくある落とし穴

  • **循環依存の存在**: 依存関係にループ（循環参照）が含まれている場合、TopoSort はすべてのオブジェクトをソートできず false を返します。この際、ソートできなかったオブジェクトは ordering の先頭側に残ります。循環依存の解消自体は TopoSort の外側（findDependencyLoops など）で行われるため、呼び出し側での適切なリカバリ処理が重要となります。
  • **不正な dumpId**: dumpId が範囲外の場合、pg_fatal が呼び出され、プログラムは異常終了します。
