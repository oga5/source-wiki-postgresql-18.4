report_manifest_error

概念

report_manifest_error は、PostgreSQL のバックアップマニフェスト（backup_manifest）ファイルを解析（パース）する際に、致命的なエラーが発生したことを報告し、プログラムを即座に異常終了させるためのエラーハンドラ（コールバック関数）です。

なぜ重要か

バックアップマニフェストの解析中に不正な構文や致命的な問題が検出された場合、それ以上の処理（バックアップの結合や検証など）を安全に継続することはできません。report_manifest_error は、エラーメッセージを適切にログ出力した上で、プロセスを確実に exit(1) で終了させることで、不整合な状態での処理継続を防ぐ重要な役割を担っています。

主要エンティティ

  • symbol:src/bin/pg_combinebackup/load_manifest.c#report_manifest_error: pg_combinebackup ツールにおいて、マニフェスト読み込み時のエラーハンドラとして機能する static 関数。
  • symbol:src/bin/pg_verifybackup/pg_verifybackup.c#report_manifest_error: pg_verifybackup ツールにおいて、マニフェスト検証時のエラーハンドラとして機能する static 関数。
  • JsonManifestParseContext: マニフェスト解析の状態を管理する構造体。この構造体の error_cb フィールドに上記のエラーハンドラ関数が登録されます。

エンドツーエンドの流れ

1. **コンテキストの初期化と登録**: pg_combinebackup または pg_verifybackup の実行時、マニフェスト解析コンテキスト（JsonManifestParseContext）が作成され、その error_cb フィールドにそれぞれの report_manifest_error 関数が登録されます。 2. **パースエラーの発生**: マニフェストファイルの解析中に構文エラーなどの致命的な問題が検出されると、登録された error_cb（すなわち report_manifest_error）が呼び出されます。 3. **ログ出力と終了**:

  • va_start を用いて可変長引数リストを初期化します。
  • pg_log_generic_v を呼び出し、PG_LOG_ERROR レベルでエラーメッセージをログ（標準エラー出力など）に出力します。
  • va_end で引数リストをクリーンアップします。
  • exit(1) を実行し、プロセスを即座に異常終了させます。

なお、マニフェスト解析の開始から終了に至る全体的な制御フローや、パース処理の具体的な内部アルゴリズムの全容（エンドツーエンドの完全な流れ）は、提供された情報からは完全には特定されていません。

重要な不変条件

  • **非復帰性 (No-return)**: 本関数が呼び出された場合、呼び出し元へ制御が戻ることはなく、必ず exit(1) によりプロセスが終了します。
  • **シグネチャの一致**: pg_combinebackup と pg_verifybackup の両モジュールにおいて、JsonManifestParseContext のエラーコールバックとして登録できるように、同一の関数シグネチャ（static void (JsonManifestParseContext *, const char *, ...)）を維持する必要があります。

よくある落とし穴

  • **リソースの解放漏れ**: 本関数は呼び出されると即座に exit(1) でプロセスを終了させるため、呼び出し元で確保されたメモリやファイル記述子などのリソースは明示的に解放されません（OSによるプロセス終了時の自動回収に依存します）。
  • **定義の重複と属性の不一致**: 本関数は pg_combinebackup と pg_verifybackup の両方にほぼ同一のロジックで個別に定義されています。片方の挙動やシグネチャを変更する際は、もう片方にも同様の修正が必要になるため、同期漏れに注意する必要があります。また、pg_combinebackup 側では pg_noreturn 属性が宣言されている一方、pg_verifybackup 側では static void として定義されているという差異が存在します。
