write_eventlog

概念

write_eventlog は、Windows環境においてオペレーティングシステムのイベントログサービスにメッセージを記録するための関数です。PostgreSQLのコードベース内では、ユーティリティツールである pg_ctl と、バックエンドのエラーハンドリングモジュールの2箇所に同名の関数が存在し、それぞれの文脈でWindowsイベントログへの書き込みを抽象化しています。

なぜ重要か

Windows環境におけるPostgreSQLの運用において、サービスの起動・停止状態やエラー情報をOS標準のイベントログに統合して記録するために重要です。これにより、システム管理者はWindowsのイベントビューアー等を通じてPostgreSQLの挙動や異常を監視できるようになります。

関与するモジュール

  • module:src/backend/utils/error
  • module:src/bin/pg_ctl

主要エンティティ

  • **symbol:src/bin/pg_ctl/pg_ctl.c#write_eventlog**: pg_ctl ユーティリティ内で使用される静的関数。Windows APIを直接呼び出してイベントログを記録します。
  • **symbol:src/backend/utils/error/elog.c#write_eventlog**: バックエンドのログ出力機構の一部として定義されている同名関数（シグネチャや実装の詳細は pg_ctl のものとは異なります）。

エンドツーエンドの流れ

pg_ctl における流れ

1. **呼び出し**: pg_ctl.c 内の各所（サーバー起動待機時のメッセージ出力やエラー報告など）から write_eventlog(level, line) が呼び出されます。 2. **フィルタリング**: silent_mode が有効かつ指定された level が EVENTLOG_INFORMATION_TYPE である場合、処理は何も行わずに終了します。 3. **初期化**: 静的変数 evtHandle が未初期化（INVALID_HANDLE_VALUE）の場合、Windows API の RegisterEventSource を呼び出してイベントソースを登録します。登録に失敗した場合は処理を中断します。 4. **書き込み**: Windows API の ReportEvent を呼び出し、指定された重大度（level）とメッセージ（line）をイベントログに書き込みます。

src/backend/utils/error/elog.c における流れ

*注意: src/backend/utils/error/elog.c に定義されている write_eventlog については、現時点で詳細なエンドツーエンドのフローやシグネチャは未解明です。*

重要な不変条件

  • **プラットフォーム依存性**: 本関数はWindows環境での実行を前提としています。
  • **ハンドルの永続性**: pg_ctl 内の evtHandle は静的変数として保持され、プロセスの生存期間中に初回呼び出し時のみ初期化されます。
  • **イベントソースのフォールバック**: event_source グローバル変数が設定されていない場合は、デフォルトの DEFAULT_EVENT_SOURCE が使用されます。
  • **実装の独立性**: src/bin/pg_ctl/pg_ctl.c と src/backend/utils/error/elog.c の write_eventlog は、名前は同一ですがシグネチャが異なり、完全に独立した別個の実装です。

よくある落とし穴

  • **サイレントな失敗**: RegisterEventSource によるイベントソースの登録に失敗した場合、evtHandle は INVALID_HANDLE_VALUE のままとなり、以降の呼び出しでもエラーを報告することなくログ出力がスキップされます。
  • **ログの欠損**: silent_mode が有効な場合、情報レベル（EVENTLOG_INFORMATION_TYPE）のログが抑制されるため、意図したログが出力されない原因が設定によるものか、あるいは登録失敗によるものかの区別がつきにくくなります。
  • **同名関数の混同**: バックエンド側（elog.c）とツール側（pg_ctl.c）で同名の関数が存在するため、ソースコードを横断的に検索・修正する際には、それぞれのシグネチャや動作コンテキストが異なる点に注意する必要があります。
