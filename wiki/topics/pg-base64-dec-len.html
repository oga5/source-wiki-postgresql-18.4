pg_base64_dec_len

概念

pg_base64_dec_len は、Base64エンコードされた文字列の長さから、デコード後に必要となるバイナリデータの最大バッファサイズ（バイト単位）を算出するための関数です。PostgreSQLのソースコード内では、異なるモジュールに同名の関数が複数定義されています。

なぜ重要か

Base64デコード処理を行う際、デコード後のデータを格納するためのメモリバッファを安全かつ効率的に確保する必要があります。この関数は、入力長に基づいて必要な最大バッファサイズを高速に計算（srclen * 3 / 4 相当）し、メモリ割り当ての基準値を提供するために重要です。

関与するモジュール

  • module:contrib/pgcrypto
  • module:src/backend/utils/adt

主要エンティティ

  • symbol:contrib/pgcrypto/pgp-armor.c#pg_base64_dec_len : contrib/pgcrypto 内で定義されている静的（static）関数。
  • symbol:src/backend/utils/adt/encode.c#pg_base64_dec_len : src/backend/utils/adt に存在する同名の関数（実装は異なります）。

エンドツーエンドの流れ

contrib/pgcrypto における具体的な流れは以下の通りです：

1. pgp_armor_decode などのデコード関連処理が、Base64エンコードされた文字列の長さを取得します。 2. pg_base64_dec_len を呼び出し、デコード後のバイナリデータの最大長を算出します。 3. 算出された最大長に基づいてデコード用のバッファを確保し、実際のデコード処理を実行します。

※ src/backend/utils/adt/encode.c 側の具体的なエンドツーエンドの流れや、両モジュール間での統一的な呼び出しフローは現時点では不明です。

重要な不変条件

  • **最大長の算出**: contrib/pgcrypto の実装では、Base64の4文字が3バイトのバイナリに変換される特性に基づき、(srclen * 3) >> 2 というビット演算を用いて最大長を算出します。
  • **非負の入力**: 入力 srclen が 0 の場合、戻り値は必ず 0 になります。

よくある落とし穴

  • **同名関数の混同**: contrib/pgcrypto/pgp-armor.c の pg_base64_dec_len は static 関数であり、src/backend/utils/adt/encode.c にある同名関数とは実装が異なります。これらは独立して存在しているため、混同しないよう注意が必要です。
  • **概算値とパディング**: 算出される値はあくまで「最大長」の概算であり、実際のデコード結果の長さ（パディング文字 = による調整など）とは厳密に一致しない場合があります。
  • **オーバーフローの可能性**: 非常に大きな srclen が入力された場合、srclen * 3 の計算において符号なし整数のオーバーフローが発生するリスクがあります。
