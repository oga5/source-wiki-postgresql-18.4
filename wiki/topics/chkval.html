CHKVAL

概念

CHKVAL は、PostgreSQLの複数のクエリ評価モジュール（intarray、ltree、全文検索の tsvector/tsgistidx）において、評価対象となるデータセットやコンテキスト情報を条件判定関数（checkcondition 系関数）へ引き渡すために定義されている、ローカルなコンテキスト構造体です。

各モジュールで同名の構造体として独立して定義されていますが、「クエリ評価エンジンと条件判定関数の間で、評価に必要な状態やポインタを void * を介して受け渡す」という共通の設計パターン（カプセル化）を担っています。

なぜ重要か

PostgreSQLのクエリ評価エンジン（TS_execute や ltree_execute など）は、汎用的なクエリツリーの走査ロジックを提供します。一方で、評価対象となるデータ構造（整数配列、ltree ノード、tsvector の語彙配列など）はモジュールごとに大きく異なります。

CHKVAL は、これらの多様なデータ構造を評価エンジンが解釈可能な形式に抽象化し、条件判定関数へ安全に引き渡すための「共通のコンテナ（インターフェース）」として機能します。これにより、評価ロジックの共通化と各データ型に特化したマッチング処理の分離が実現されています。

主要エンティティ

  • **symbol:contrib/intarray/_int_bool.c#CHKVAL**
  • 整数配列の範囲（開始ポインタ arrb、末尾ポインタ arre）を保持します。
  • **symbol:contrib/ltree/ltxtquery_op.c#CHKVAL**
  • 照合対象の ltree ノード（node）と、クエリ内のオペランド文字列（operand）を保持します。
  • **symbol:src/backend/utils/adt/tsgistidx.c#CHKVAL**
  • 全文検索の GiST インデックス照合において、検索対象配列の範囲（開始ポインタ arrb、終了ポインタ arre）を保持します。
  • **symbol:src/backend/utils/adt/tsvector_op.c#CHKVAL**
  • 全文検索クエリ（tsquery）の評価において、tsvector 内の語彙配列（arrb、arre）、語彙文字列領域（values）、および評価用オペランド（operand）を保持します。

エンドツーエンドの流れ

各モジュールにおける評価フローは以下の通りです（モジュール間を跨ぐ統一的な共通シーケンスは存在せず、各モジュール内で独立して完結しています）。

1. **初期化**: クエリ評価の開始時（例: ltxtq_exec や execute などの呼び出し時）に、評価対象データへのポインタを用いて CHKVAL 構造体のインスタンスがローカル変数として初期化されます。 2. **評価エンジンの呼び出し**: 評価エンジン（例: TS_execute や ltree_execute）に対して、初期化した CHKVAL のポインタが void * 型（checkval）として渡されます。 3. **条件判定の実行**: 評価エンジンがクエリツリーを走査する際、登録された条件判定関数（例: checkcondition_arr や checkcondition_str）を呼び出し、引数として void *（CHKVAL）を引き渡します。 4. **キャストとマッチング**: 条件判定関数内において、void * はそれぞれのモジュール固有の CHKVAL * にキャストされ、構造体内のメンバ（配列ポインタやオペランドなど）を参照して、実際のバイナリサーチや文字列比較などのマッチング処理が行われます。

重要な不変条件

  • **ポインタの有効性**: CHKVAL 内に保持されるポインタ（arrb, arre, node, operand, values など）は、評価処理の実行中、常に有効なメモリ領域を指していなければなりません。
  • **範囲の整合性（配列の場合）**: intarray、tsgistidx、tsvector_op において、開始ポインタ arrb は終了ポインタ arre 以下（arrb <= arre）でなければならず、対象の配列はソート済みである必要があります（二分探索による判定の正当性を保証するため）。
  • **キャストの型安全性**: void * を介して渡されるため、呼び出し側が渡す CHKVAL の型と、条件判定関数側でキャストする CHKVAL の型が完全に一致していなければなりません。

よくある落とし穴

  • **名前衝突と誤認**: CHKVAL という名前の構造体は、複数のファイル（_int_bool.c, ltxtquery_op.c, tsgistidx.c, tsvector_op.c）で**個別に定義**されています。これらはグローバルに共有された単一の構造体ではなく、各ファイル内で閉じた定義です。開発やデバッグの際、異なるモジュールの CHKVAL 定義を混同しないよう注意が必要です。
  • **境界外アクセス**: 配列が空の場合や、arrb と arre の関係性が崩れている場合、二分探索や走査ロジックにおいて境界外参照（セグメンテーション違反）を引き起こすリスクがあります。
  • **メモリレイアウトへの依存**: ltxtquery_op.c などの判定関数（checkcondition_str）は、CHKVAL の特定のメンバ順序やメモリレイアウトに強く依存してキャストを行っています。構造体のメンバを変更する場合は、関連するキャスト処理もすべて同期して修正しなければなりません。
