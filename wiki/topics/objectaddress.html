ObjectAddress

概念

ObjectAddress は、PostgreSQL 内に存在するあらゆるデータベースオブジェクト（テーブル、関数、型、列など）を、システムカタログテーブルの物理的な構造に依存することなく、一意に識別するための抽象化されたデータ構造です。

システムカタログ上のオブジェクトを、カタログの OID である classId、オブジェクト自体の OID である objectId、およびオブジェクト内のサブ要素（テーブルの列など）を指す objectSubId の3つの識別子の組み合わせによって特定します。

なぜ重要か

PostgreSQL の内部システム（DDL コマンド、イベントトリガー、権限管理、セキュリティラベルなど）において、オブジェクトの物理的なカタログ構造を直接意識することなく、統一されたインターフェースでオブジェクトを特定・操作できるようにするために極めて重要です。

これにより、以下のような横断的処理を汎用的に実装できます。

  • オブジェクトの所有権確認（check_object_ownership）
  • オブジェクトの人間が読める形式への変換（getObjectIdentity, getObjectDescription）
  • セキュリティプロバイダー（contrib/sepgsql など）によるアクセス制御やセキュリティラベルの管理
  • ユーティリティコマンド実行時のオブジェクト追跡（デパースユーティリティ）

エンドツーエンドの流れ

1. **生成**: DDL コマンドやユーティリティコマンドの実行時、パーサー出力を基に get_object_address が呼び出され、対象オブジェクトの ObjectAddress が生成されます。 2. **追跡・記録**: ユーティリティコマンドの実行過程において、CollectedCommand 構造体の simple.address や simple.secondaryObject に ObjectAddress が格納され、イベントトリガーや監査ログなどの後続処理に引き渡されます。 3. **検証と適用**:

  • セキュリティラベルの変更時（sepgsql_object_relabel など）には、ObjectAddress がセキュリティプロバイダーに渡され、対象オブジェクトに対してラベル付けが可能かどうかの検証（check_object_relabel_type）が行われます。
  • メタデータ取得関数（getObjectIdentity など）に渡されることで、オブジェクトの識別情報が人間が読める文字列に変換されます。

※ 各 DDL コマンドやイベントトリガーにおける具体的な伝播経路など、詳細なエンドツーエンドの実行フローの全容は、提供されたコンテキストからは完全には明らかになっていません。

重要な不変条件

  • **objectSubId の値**: テーブルの列（Column）など、親オブジェクト内のサブ要素を指す場合にのみ非ゼロの値を使用します。それ以外の場合は常に 0 でなければなりません。
  • **同時実行制御**: カタログ操作を行う際は、適切な LOCKMODE を指定して同時実行制御を行う必要があります。
  • **有効なオブジェクト参照**: seclabel.h の検証関数などで const ObjectAddress * として参照される際は、対象オブジェクトの識別情報が有効（実在するオブジェクトを指していること）でなければなりません。
  • **型チェックの徹底**: deparse_utility.h の CollectedCommand 内で共用体の一部として使用されているため、アクセス時には必ず type フィールドによる型チェックを行う必要があります。

よくある落とし穴

  • **存在しないオブジェクトの指定**: 存在しない OID や無効な classId を指定して ObjectAddress を作成した場合、それを利用するカタログ検索関数やセキュリティプロバイダー側の検証ロジックでエラーが発生します。
  • **不適切な objectSubId の使用**: 親オブジェクトが存在しない、あるいはサブ要素を持たないオブジェクトに対して不適切な objectSubId を指定すると、後続の処理で予期せぬエラーを引き起こす原因となります。
  • **構造体自体の検証機能の欠如**: ObjectAddress 自体は単なるデータ保持用の構造体であり、自己検証ロジックを持ちません。そのため、無効なデータが格納された場合の挙動は、それを利用する側の関数に依存します。
