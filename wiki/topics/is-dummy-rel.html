is_dummy_rel

概念

is_dummy_rel は、PostgreSQLのクエリ最適化（プランナ）フェーズにおいて、指定されたリレーション（RelOptInfo）がクエリの制約などによって「確実に空の結果セットを返す（ダミーである）」と証明されているかどうかを判定する仕組みです。

なぜ重要か

クエリの最適化中に特定のリレーションが空（ダミー）であると早期に判定できれば、不要な結合（join）処理やスキャンをスキップするなどの最適化判断が可能になります。これにより、プランナは無駄な実行パスの探索を排除し、効率的な実行プランを生成できます。

関与するモジュール

  • module:src/backend/optimizer/path
  • module:src/include/nodes

主要エンティティ

  • **symbol:src/backend/optimizer/path/joinrels.c#is_dummy_rel**: リレーションがダミーであるかを実際に判定するヘルパー関数。
  • **symbol:src/include/nodes/pathnodes.h#IS_DUMMY_REL**: is_dummy_rel 関数をラップし、外部から簡潔に呼び出すために提供されているマクロ。

エンドツーエンドの流れ

1. プランナが結合の正当性や最適化の判断を行う過程（joinrels.c 内の複数箇所）で、対象リレーションに対して IS_DUMMY_REL マクロが呼び出されます。 2. is_dummy_rel 関数は、対象リレーション（RelOptInfo）のパスリスト（pathlist）を検査します。

  • pathlist が空（NIL）の場合は、直ちに false を返します。
  • パスリストが存在する場合、先頭のパス（linitial）を取得します。

    • 3. 取得したパスが ProjectionPath または ProjectSetPath でラップされている場合、それらの subpath を再帰的に辿り、元のパスを特定します。 4. 最終的に到達したパスに対して IS_DUMMY_APPEND マクロを適用し、そのパスが子を持たない空の Append パスであるかを判定して結果を返します。

※なお、この判定がクエリ全体の最適化プロセスの中でどのようにトリガーされ、最終的な実行プランにどう統合されるかという詳細なエンドツーエンドの全体フローは、提供された情報からは完全には明らかになっていません。

重要な不変条件

  • **パスリストの存在**: rel->pathlist が空（NIL）の場合、関数は常に false を返します。
  • **ダミーリレーションの構造**: ダミーリレーションは、パスリストの先頭に子を持たない Append パスを持つことが期待されます。
  • **ラッパーパスの透過性**: 計画の後半段階で ProjectionPath や ProjectSetPath が追加されている場合でも、それらを再帰的に透過して内部の Append パスまで到達し、正しく判定を行う必要があります。

よくある落とし穴

  • **パスのラップによる判定漏れ**: 計画の進行に伴い、パスが ProjectionPath や ProjectSetPath などの他のパスノードでラップされることがあります。単に先頭のパスの型を直接チェックするだけではダミー判定を見落とす可能性があるため、必ず subpath を再帰的に辿るループ処理を経由して判定を行う必要があります。
