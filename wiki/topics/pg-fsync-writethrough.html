pg_fsync_writethrough

概念

pg_fsync_writethrough は、プラットフォーム固有の「ライトスルー（Write-through）」同期操作をファイル記述子に対して実行し、データの物理的なディスクへの書き込みを保証するための関数です。主に macOS などの F_FULLFSYNC マクロがサポートされている環境において、OSのキャッシュ層をバイパスして物理デバイスへの同期を強制するために使用されます。

なぜ重要か

データベースの信頼性を担保するためには、WAL（Write-Ahead Log）などの重要なデータがディスクに物理的に書き込まれたことを保証する必要があります。通常の fsync ではOSのキャッシュ層に留まる可能性があるプラットフォームにおいて、pg_fsync_writethrough は F_FULLFSYNC を用いて物理的なライトスルー同期を強制し、予期せぬ停電やシステムクラッシュ時におけるデータの永続性を高めます。また、pg_test_fsync ツールを通じて、この同期手法が現在のプラットフォームで利用可能かつ有効であるかを検証するためにも重要です。

エンドツーエンドの流れ

1. 呼び出し元（src/backend/access/transam/xlog.c、src/backend/storage/file/fd.c、または src/bin/pg_test_fsync/pg_test_fsync.c）が、同期対象のファイル記述子 fd を指定して pg_fsync_writethrough(fd) を呼び出します。 2. プラットフォームで F_FULLFSYNC マクロが定義されている場合、システムコール fcntl(fd, F_FULLFSYNC, 0) が実行されます。 3. fcntl が成功した場合は 0 を返し、失敗した場合は -1 を返します。 4. F_FULLFSYNC が定義されていないプラットフォームの場合、errno に ENOSYS を設定した上で、即座に -1 を返します。

※なお、バックエンド全体における具体的な呼び出しトリガーや、トランザクションコミット時などの詳細なエンドツーエンドの制御フローの全貌は、提供された情報からは完全には特定されていません。

重要な不変条件

  • **プラットフォーム依存性**: F_FULLFSYNC マクロが定義されている環境（主に macOS 等）でのみ、実際の fcntl による物理同期操作が実行されます。
  • **非サポート環境での挙動**: F_FULLFSYNC が定義されていない環境では、関数は常に errno を ENOSYS に設定し、-1 を返さなければなりません。

よくある落とし穴

  • **非サポートプラットフォームでのエラー**: macOS 以外の多くのプラットフォーム（Linuxなど）では F_FULLFSYNC が定義されていないため、この関数を呼び出すと常に ENOSYS エラー（-1）が返されます。プラットフォームごとのサポート状況を考慮せずにこの関数に依存すると、同期処理が失敗する原因となります。
  • **パフォーマンスへの影響**: F_FULLFSYNC によるライトスルー同期は、通常の fsync よりも物理ディスクへの書き込みを厳格に強制するため、I/O パフォーマンスに大きなオーバーヘッドを与える可能性があります。
