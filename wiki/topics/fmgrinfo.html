FmgrInfo

概念

FmgrInfo は、PostgreSQLの関数マネージャ（fmgr）において、SQL関数をC言語で実行するために必要なメタデータや実行時情報を保持するキャッシュ構造体です。また、式評価エンジン（execExpr.h）内では、関数実行ステップ（ExprEvalStep）のコンテキストの一部として、関数ルックアップデータや直接的な関数ポインタ（fn_addr）を保持・参照するために利用されます。

なぜ重要か

関数呼び出しのオーバーヘッドを最小化し、実行時のディスパッチを効率化するために重要です。システムカタログからの関数情報のルックアップ結果をキャッシュし、直接的な関数ポインタ（fn_addr）を保持することで、間接参照を避けて高速な関数実行を可能にします。また、fn_strict によるNULL値の自動ハンドリングや、fn_extra を用いた実行時状態の保持など、関数の実行制御と拡張性を支える基盤となっています。

エンドツーエンドの流れ

提供された情報に基づく、関数実行の基本的な流れは以下の通りです。

1. **初期化**: fmgr_info() などの関数を介して、システムカタログから対象関数の情報を取得し、FmgrInfo 構造体にキャッシュします。 2. **式評価への組み込み**: 式評価エンジンにおいて、ExprEvalStep の func メンバ（FmgrInfo *finfo を含む）として関数実行ステップが構築されます。 3. **実行制御と呼び出し**:

  • fn_strict が true の場合、引数に NULL が含まれると関数本体を呼び出さずに NULL を返します。
  • それ以外の場合は、キャッシュされた関数ポインタ fn_addr を直接参照して、実際のC言語実装関数を呼び出します。

なお、fmgr_info の具体的な内部処理や、式評価エンジンにおけるステップ構築から実行に至る完全なエンドツーエンドの制御フローの全容は、提供された情報からは完全には明らかになっていません。

重要な不変条件

  • **引数の範囲**: fn_nargs は 0 から FUNC_MAX_ARGS の範囲内でなければなりません。
  • **メモリコンテキストの整合性**: fn_extra に格納されるデータの生存期間は、fn_mcxt で指定されたメモリコンテキストに依存しなければなりません。
  • **関数ポインタのキャッシュ**: fn_addr は、間接参照を避けるためにキャッシュされた有効な関数ポインタでなければなりません。
  • **引数の一致**: nargs が実際の引数の数と一致していなければなりません。

よくある落とし穴

  • **キャッシュの無効化漏れ**: FmgrInfo の内容はキャッシュされるため、CREATE OR REPLACE FUNCTION などで関数定義が変更された場合、キャッシュの無効化や再初期化を行わないと古い情報で実行される危険性があります。
  • **メモリ管理の不備**: fn_mcxt が不適切な場合、fn_extra へのアクセス時にメモリ破壊や無効なポインタ参照が発生する可能性があります。
  • **未初期化ポインタの呼び出し**: finfo や fn_addr が正しく初期化・ロードされていない状態で関数を呼び出すと、セグメンテーションフォールトが発生します。
  • **引数の不整合**: nargs が実際の引数数と一致しない場合、関数実行時にスタックや引数処理の不整合を招く恐れがあります。
