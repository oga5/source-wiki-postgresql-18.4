LockClauseStrength

概念

LockClauseStrength は、SQLの SELECT FOR 句（FOR UPDATE や FOR SHARE など）で指定される行レベルのロック強度を表現するための列挙型（enum）です。

なぜ重要か

クエリ解析、書き換え、実行計画（プランニング）の各フェーズにおいて、複数のロック指定が競合した際の優先順位を決定したり、外部データラッパー（FDW）がリモートサーバーに対してどのような行ロックを要求すべきかを判定したりするために不可欠なデータ構造だからです。

関与するモジュール

  • module:src/include/foreign
  • module:src/include/nodes

主要エンティティ

  • symbol:src/include/nodes/lockoptions.h#LockClauseStrength : 列挙型の本体定義。以下の値を含みます。
  • LCS_NONE (ロック指定なし)
  • LCS_FORKEYSHARE (FOR KEY SHARE)
  • LCS_FORSHARE (FOR SHARE)
  • LCS_FORNOKEYUPDATE (FOR NO KEY UPDATE)
  • LCS_FORUPDATE (FOR UPDATE)
  • symbol:src/include/foreign/fdwapi.h#LockClauseStrength : FDW API において、リモートの行ロック要件（RowMarkType）を決定する GetForeignRowMarkType などのインターフェースで参照される型定義。

エンドツーエンドの流れ

1. **解析（Parser）**: src/backend/parser/analyze.c がSQL文中のロック句を解析し、LockClauseStrength の値を ParseState や Query 構造体に格納します。 2. **書き換え（Rewriter）**: src/backend/rewrite/rewriteHandler.c がルール適用時にロックポリシーを処理します。 3. **計画（Planner）**: src/backend/optimizer/plan/planner.c（applyLockingClause や select_rowmark_type など）が、競合する複数のロック指定の強さを比較し、最終的なロック要件を決定して PlanRowMark に格納します。 4. **外部データ連携（FDW）**: 外部テーブルのスキャン時、GetForeignRowMarkType などのFDW関数にこの強度が渡され、リモート側で必要な行ロックが決定されます。実行時には ForeignScanState 等にこの強度が保持されます。

重要な不変条件

  • **優先順位の順序**: 列挙値の定義順序はロックの強さと一致しており、値が大きいほど強いロックを意味します（LCS_NONE < LCS_FORKEYSHARE < LCS_FORSHARE < LCS_FORNOKEYUPDATE < LCS_FORUPDATE）。プランナーはこの大小関係を利用して優先順位を判定します。
  • **デフォルト値**: ロック句が存在しない場合は LCS_NONE が設定されます。

よくある落とし穴

  • **内部ロックモードとの混同**: SQLレベルの指定である LockClauseStrength は、ストレージエンジン内部でタプルをロックする際に使用される LockTupleMode とは役割が異なるため、混同しないよう注意が必要です。
  • **未定義値のキャスト**: 範囲外の整数値が LockClauseStrength にキャストされて渡された場合、planner.c や ruleutils.c 内の elog(ERROR, "unrecognized LockClauseStrength %d", ...) によりシステムエラーが発生します。
  • **拡張時の注意**: 新しいロック強度を追加する場合、列挙型の順序が優先順位判定ロジックと整合しているか確認し、LCS_asString やエラーハンドリング箇所（elog）をすべて更新する必要があります。
