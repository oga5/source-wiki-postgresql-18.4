BlockRefTableEntry

概念

BlockRefTableEntry は、特定のリレーションフォークにおける変更済みブロックの状態を保持・管理するためのデータ構造です。BlockRefTable（ハッシュテーブル）の要素として機能し、内部的には変更されたブロックの密度に応じて、メモリ効率を最適化するためにオフセット配列からビットマップへと動的に表現形式を切り替える特徴を持っています。

なぜ重要か

このデータ構造は、増分バックアップ（src/backend/backup/basebackup_incremental.c など）において、どのブロックが変更されたかを追跡・管理するために不可欠です。リレーションの切り詰めや再拡張を追跡するための limit_block を保持し、効率的なブロック参照情報のシリアライズや取得を可能にします。

関与するモジュール

  • module:src/common
  • module:src/include/common

主要エンティティ

  • **symbol:src/common/blkreftable.c#BlockRefTableEntry**: 構造体の実体定義。key（リレーション識別子）、limit_block、チャンク数（nchunks）、および実際のブロック変更状態を格納する chunk_data などのメンバを持ちます。
  • **symbol:src/include/common/blkreftable.h#BlockRefTableEntry**: 不透明な構造体（Opaque Struct）としての宣言。外部モジュールに対して詳細なメンバを隠蔽します。
  • **symbol:src/common/blkreftable.c#CreateBlockRefTableEntry**: BlockRefTableEntry 構造体のメモリを確保し、指定された RelFileLocator と ForkNumber で初期化する関数。

エンドツーエンドの流れ

1. **初期化**: CreateBlockRefTableEntry が呼び出され、palloc0 により BlockRefTableEntry のメモリが確保されます。引数の RelFileLocator と ForkNumber が key にコピーされ、limit_block は InvalidBlockNumber に初期化されます。 2. **ブロック変更の記録**: BlockRefTableMarkBlockModified 等の関数を通じて、特定のブロックの変更状態が chunk_data に記録されます。 3. **情報の取得と書き出し**: BlockRefTableEntryGetBlocks を介して変更ブロックリストを取得したり、BlockRefTableWriteEntry を使用して BlockRefTableWriter 経由で外部ストレージへシリアライズ（書き出し）されます。

*注意: どのタイミングでハッシュテーブルに登録され、どのように解放されるかなど、詳細なエンドツーエンドの全体フローの全容は、提供された情報からは完全には明らかになっていません。*

重要な不変条件

  • **表現形式の一方向性**: 変更密度に応じて内部表現がオフセット配列からビットマップへ動的に切り替わりますが、一度ビットマップへ変換されると、オフセット配列には戻りません。
  • **チャンク管理単位**: chunk は BLOCKS_PER_CHUNK (64KB) 単位で管理されます。
  • **ソート要件**: BlockRefTableWriter を使用して書き出す際、エントリは tablespace -> database -> relfilenumber -> forknumber の順序でソートされている必要があります。
  • **初期値**: CreateBlockRefTableEntry で生成された直後の limit_block は常に InvalidBlockNumber です。

よくある落とし穴

  • **不透明な構造体による制限**: src/include/common/blkreftable.h では typedef struct BlockRefTableEntry BlockRefTableEntry; として宣言されているため、ヘッダー側から直接メンバにアクセスすることはできません。操作は必ず BlockRefTableEntryGetBlocks などの専用インターフェース関数を経由する必要があります。
  • **ソート順の不整合**: シリアライズ時に供給されるエントリの順序が正しくソート（tablespace, database, relfilenumber, forknumberの順）されていない場合、書き込み処理において整合性が保たれなくなる可能性があります。
  • **メモリ不足時のエラー**: CreateBlockRefTableEntry は内部で palloc0 を使用するため、メモリ不足時には PostgreSQL のメモリ管理システムによってエラーが発生します。
