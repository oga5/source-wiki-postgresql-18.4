manifest_data

概念

manifest_data は、PostgreSQL のバックアップマニフェスト（backup_manifest）の内容をメモリ上で保持するためのルート構造体です。バックアップの統合（pg_combinebackup）や検証（pg_verifybackup）において、構成情報、ファイル情報、および WAL 範囲を管理・検索するために使用されます。

なぜ重要か

バックアップマニフェスト全体の解析結果を構造化して保持することで、以下の処理を効率的に行うための基盤となるため重要です。

  • simplehash を使用したハッシュテーブルによる、ファイルパスをキーとした高速なファイル情報の検索。
  • バックアップに含まれる WAL の連続性の検証。
  • system_identifier によるバックアップ元データベースクラスタの一意な識別。

関与するモジュール

  • module:src/bin/pg_combinebackup

主要エンティティ

  • **symbol:src/bin/pg_combinebackup/load_manifest.h#manifest_data**

    • バックアップ統合処理用の構造体。システム識別子、ファイル情報のハッシュテーブル、および WAL 範囲の双方向リスト（先頭と末尾）を保持します。

  • **symbol:src/bin/pg_verifybackup/pg_verifybackup.h#manifest_data**

    • バックアップ検証処理用の構造体。バージョン情報、システム識別子、ファイル情報のハッシュテーブル、および WAL 範囲のリスト（先頭と末尾）を保持します。

エンドツーエンドの流れ

各ツールにおいて、以下のようにマニフェストデータの構築と利用が行われます。

pg_combinebackup における流れ

1. load_backup_manifest() が指定されたディレクトリ内のマニフェストファイルを読み込み、JSON 解析コンテキストを初期化します。 2. 解析されたファイル情報を files ハッシュテーブルに格納します。 3. WAL 範囲情報をリストとして連結し、first_wal_range と last_wal_range に格納します。 4. 構築された manifest_data 構造体を返却し、バックアップ統合処理で利用します。

pg_verifybackup における流れ

1. メインロジック（pg_verifybackup.c 等）がマニフェストファイルを解析し、manifest_data 構造体を構築します。 2. 検証の実行中、ファイルパスをキーとして files ハッシュテーブルを検索し、対応するファイル情報の整合性を確認します。 3. first_wal_range から last_wal_range までの WAL 範囲リストを走査し、WAL の連続性を検証します。

※ 両ツール間を跨ぐ直接的なデータの受け渡しなど、統合的なエンドツーエンドの全体フローは定義されていません。

重要な不変条件

  • **ファイル検索の高速性**: files フィールドは lib/simplehash.h を用いて生成されたハッシュテーブル（manifest_files_hash）であり、ファイルパスをキーとして個別のファイル情報（manifest_file）を高速に検索できなければなりません。
  • **一意なクラスタ識別**: system_identifier は、バックアップ元のデータベースクラスタを一意に識別する値として保持されます。
  • **WAL 範囲の管理**: first_wal_range および last_wal_range は、WAL のタイムラインと LSN 範囲を管理するリスト構造の先頭と末尾を正しく指し示している必要があります。

よくある落とし穴

  • **重複定義と変更の影響**: manifest_data 構造体は pg_combinebackup と pg_verifybackup の両方で個別に定義されています。構造体のフィールドや仕様を変更する場合は、両方のツールへの影響を考慮する必要があります。
  • **ハッシュテーブルの仕様変更**: manifest_files_hash は simplehash.h のテンプレート生成マクロによって定義されているため、ハッシュテーブルの仕様を変更する際は pg_verifybackup.h や load_manifest.c 内の SH_ マクロ定義を確認する必要があります。
  • **メモリ管理とリーク**: pg_combinebackup では複数のバックアップを扱うため、manifest_data が配列として管理されることが想定されています。メモリのライフサイクル管理を誤るとリークの原因になります。また、メモリ確保失敗時には pg_malloc0 等によりプロセスが終了する可能性があります。
