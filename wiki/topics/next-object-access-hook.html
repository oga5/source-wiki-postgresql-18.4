next_object_access_hook

概念

next_object_access_hook は、PostgreSQLにおいてデータベースオブジェクト（テーブル、カラム、スキーマなど）へのアクセスや操作を監視・制御するための object_access_hook を、複数の拡張モジュール間で協調して連鎖（チェーン）呼び出しするための関数ポインタ（またはその退避用変数）です。

なぜ重要か

PostgreSQLのコアシステムが提供するオブジェクトアクセス用のフックポイント（object_access_hook）は、グローバルに1つしか存在しません。複数の拡張モジュール（セキュリティ制御を行う sepgsql や、フックの動作を検証するテストモジュールなど）が同時にオブジェクトアクセスをフックしたい場合、自身のフックを登録する前に既存のフックを next_object_access_hook に退避させ、自処理の実行後にそれを呼び出すことで、競合を避けてすべてのフックを順次実行（連鎖）させることができます。

エンドツーエンドの流れ

1. **モジュールの初期化**: モジュール（sepgsql や test_oat_hooks）がロードされる際、初期化関数内で現在の object_access_hook の値を next_object_access_hook に保存します。 2. **フックの登録**: object_access_hook に、各モジュール独自のカスタムフック関数を設定します。 3. **オブジェクト操作の検知**: SQL実行などによりオブジェクトの作成、変更、削除、アクセスが発生すると、PostgreSQLコアから現在登録されている object_access_hook が呼び出されます。 4. **連鎖呼び出し**: カスタムフック内での処理（セキュリティチェックやログ記録など）が完了した後、退避しておいた next_object_access_hook が NULL でなければ、それを呼び出して次のフックに制御を移します。

*注意: 具体的な引数の伝播や、各モジュール間での呼び出し順序の制御など、詳細なエンドツーエンドの実行フローは提供されたソースコードのコンテキストからは完全には特定されていません。*

重要な不変条件

  • **フックチェーンの維持**:

    • 新しいフックを登録する際は、必ず事前に既存の object_access_hook を next_object_access_hook に退避させなければなりません。これを怠ると、先にロードされていた他のモジュールのフックが呼び出されなくなります。

  • **NULLチェックの必須性**:

    • next_object_access_hook を呼び出す前には、退避されたフックが存在する（NULL ではない）ことを確認する必要があります。

*注意: 複数モジュールがアンロードされる際の復元順序に関する不変条件など、その他の詳細な制約は現時点では未検証です。*

よくある落とし穴

  • **ロード順序への依存**:

    • どのモジュールが先にロードされるかによって next_object_access_hook に格納されるフックが変化するため、特定の実行順序に依存する実装を行うと、予期せぬ動作やセキュリティチェックのバイパスを引き起こす可能性があります。

  • **アンロード時のチェーン破損**:

    • モジュールが動的にアンロードされる場合、退避したフックポインタを正しく復元しないと、解放済みのメモリ領域を指す不正なポインタがフックチェーン内に残り、クラッシュの原因になります。
