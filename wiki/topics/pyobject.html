PyObject

概念

PyObject は、Python C API においてすべての Python オブジェクトを表現するための不透明な構造体（基底型）です。PostgreSQL の PL/Python 拡張およびその関連モジュール（hstore_plpython や jsonb_plpython）において、PostgreSQL 側のデータ表現（Datum など）と Python インタプリタ側のデータ構造（辞書、リスト、文字列、数値など）の間で相互変換を行う際の、共通のデータコンテナとして機能します。

なぜ重要か

PL/Python 拡張機能において、PostgreSQL のデータ型と Python のデータ型をシームレスに連携させるために不可欠なデータ表現であるためです。例えば、hstore 型を Python の dict 型に変換したり、jsonb 型の階層構造を Python のオブジェクトツリー（辞書やリストなど）へ再帰的にマッピングしたりする際に、すべての Python 側データを統一的に扱うための基底型として機能します。

エンドツーエンドの流れ

PL/Python におけるデータ変換の全体的なエンドツーエンドの流れは、提供された情報からは完全には明らかになっていません。ただし、部分的な流れとして以下の処理が確認されています。

1. **PostgreSQL から Python への変換**:

  • PLy_input_convert や PLyDatumToOb 構造体に含まれる変換関数、または hstore_to_plpython や PLyObject_FromJsonbContainer などの関数を介して、PostgreSQL のデータ（Datum、hstore、JsonbContainer など）から PyObject（辞書、リスト、文字列など）が生成されます。
  • 例えば、hstore の各要素は Python の文字列オブジェクトに変換され、PyDict_SetItem を用いて Python の辞書（PyObject）に格納されます。

    • 2. **Python から PostgreSQL への変換**:

  • PLy_output_convert や PLyObToDatum 構造体に含まれる変換関数、または plpython_to_hstore などの関数を介して、PyObject から PostgreSQL の Datum や対応するデータ型へ変換されます。
  • 例えば、plpython_to_hstore では PyMapping_Items を介して Python 辞書のキーと値のペアを抽出します。

重要な不変条件

  • **参照カウント（Reference Counting）の管理**: Python C API のメモリ管理規則に従い、不要になった PyObject に対しては Py_DECREF または Py_XDECREF を呼び出して参照カウントを適切に減らす必要があります。
  • **実行環境の前提**: Python インタプリタが初期化されており、plpython 拡張がロードされている環境下でのみ有効です。
  • **ドメイン制約チェック時の例外**: PLy_output_convert に渡される際、Py_None はドメイン制約チェックのために変換関数へ渡されることが許可されています。

よくある落とし穴

  • **メモリリークとセグメンテーション違反**: Py_DECREF の呼び忘れによるメモリリーク、または過剰な解放（ダブルフリーなど）によるセグメンテーション違反。特に PostgreSQL 側のメモリ管理（palloc 等）と Python 側のメモリ管理（PyObject）が混在する箇所では、エラー発生時（PG_TRY/PG_FINALLY や PG_CATCH ブロック内）のクリーンアップ（Py_XDECREF の呼び出しなど）を徹底する必要があります。
  • **NULL ポインタの逆参照**: メモリ不足などにより Python オブジェクトの生成に失敗した場合、Python C API は NULL を返します。この戻り値を検証せずに操作するとクラッシュの原因となります。
  • **型不一致による例外**: PyMapping_Check などの型確認を行わずに非マッピングオブジェクトに対してマッピング操作を行うと、Python 側の例外が発生します。
