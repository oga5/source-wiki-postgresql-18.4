EncodeDateTime

概念

EncodeDateTime は、PostgreSQLのソースコードにおいて、内部的な日付・時刻表現（struct tm など）を指定されたスタイル（ISO、SQL、German、Postgresなど）に基づいて文字列形式に変換（エンコード）するための関数です。この関数は、クライアント側のECPGライブラリと、サーバーのバックエンド側の両方に同名のシンボルとして存在します。

なぜ重要か

データベースシステムにおいて、日付や時刻のデータをユーザーが読みやすい、あるいは標準規格に準拠した文字列形式で出力することは不可欠です。EncodeDateTime は、タイムゾーン、小数秒、紀元前（BC）表記、欧州形式（EuroDates）などの多様なフォーマット要件を吸収し、一貫した文字列変換処理を提供します。また、ECPG（埋め込みSQL）とバックエンドのそれぞれで同様のエンコード処理を担うことで、システム全体での日付時刻表現の整合性を支えています。

エンドツーエンドの流れ

現時点では、バックエンド側（src/backend/utils/adt/datetime.c）の EncodeDateTime の具体的な呼び出し元や、ECPG側における上位の呼び出しフローなど、システム全体におけるエンドツーエンドの流れは完全には判明していません。

ECPG側（dt_common.c）における局所的な処理の流れは以下の通りです： 1. 入力された style（USE_ISO_DATES, USE_SQL_DATES など）に基づき、switch 文でフォーマット処理を分岐します。 2. 各スタイルに応じた順序で、sprintf を使用して出力先バッファ str へ日付・時刻を書き込みます。 3. print_tz が有効な場合、タイムゾーン名（tzn）またはオフセット（tz）を計算して追記します。 4. USE_POSTGRES_DATES スタイルの場合は、date2j を用いて曜日を算出し、文字列に含めます。

重要な不変条件

ECPG側の EncodeDateTime において、以下の不変条件が確認されています。

  • **タイムゾーン出力の抑制**: tm->tm_isdst < 0 の場合、print_tz は強制的に false となります。
  • **紀元前表記の付与**: tm->tm_year <= 0 の場合、歴史的な「0年」を考慮し、BC（紀元前）表記が付与されます。
  • **小数秒のトリミング**: fsec != 0 の場合、小数秒が追加され、末尾の不要なゼロは TrimTrailingZeros によって削除されます。

※バックエンド側の EncodeDateTime における不変条件は、提供されたコンテキストからは不明です。

よくある落とし穴

  • **バッファオーバーフロー**: ECPG側の EncodeDateTime では、出力先バッファ str のサイズ管理は呼び出し側の責任となっています。十分なバッファサイズが確保されていない場合、sprintf によるメモリ破壊が発生する危険性があります。
  • **非ASCIIタイムゾーン名**: タイムゾーン名 tzn に非ASCII文字が含まれる場合、%.*s による切り詰め処理がエンコーディングを考慮しないため、不正な文字列が生成される可能性があります（ただし、IANAデータベースのTZ略称はすべてASCIIであると想定されています）。
  • **シグネチャの不一致**: バックエンド側の EncodeDateTime とECPG側の同名関数は、引数構成（例：EuroDates の有無など）が異なります。両者を混同して呼び出したり、移植したりしないよう注意が必要です。
