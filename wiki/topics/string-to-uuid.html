string_to_uuid

概念

string_to_uuid は、PostgreSQLにおいて文字列形式のUUIDを解析・変換し、内部的なUUID表現に格納するための静的（static）関数です。

PostgreSQLのソースコード内には、役割や依存ライブラリが異なる2つの同名関数が存在します。 1. **コア実装 (src/backend/utils/adt/uuid.c)**: PostgreSQLコアのUUIDデータ型（pg_uuid_t）への変換を行う関数。 2. **拡張モジュール実装 (contrib/uuid-ossp/uuid-ossp.c)**: OSSP UUIDライブラリの内部表現（uuid_t）への変換を行う関数。

なぜ重要か

UUIDはデータベースにおいて一意識別子として広く利用されます。外部から入力された様々な形式のUUID文字列（標準的なハイフン区切り形式や、中括弧 {} で囲まれた形式など）を、システム内部で効率的に扱えるバイナリ形式（16バイト構造体）に安全かつ正確に変換・検証する役割を担っているため重要です。

また、コアの組み込みUUID型と、外部ライブラリに依存する拡張モジュール（uuid-ossp）のそれぞれで、適切な構造体に適合させるための変換処理を提供しています。

エンドツーエンドの流れ

SQLクエリのパースからこれら関数が呼び出されるまでの完全なエンドツーエンドの呼び出しフローは、提供されたコンテキストからは明らかになっていません。ただし、各モジュール内での部分的な流れは以下の通りです。

  • **コア側 (src/backend/utils/adt/uuid.c)**:

    • 1. uuid_in などのSQL型変換関数（呼び出し元）から呼び出されます。 2. 入力文字列の先頭に { があるかを確認し、16バイト分（UUID_LEN）のループで2文字ずつ16進数として読み取ります。 3. 特定のバイト位置でハイフン（-）をスキップします。 4. { で開始した場合は末尾に } があることを確認し、余分な文字がないか検証します。 5. 構文エラーが発生した場合は ereturn マクロを介してエラーを報告します。

  • **拡張モジュール側 (contrib/uuid-ossp/uuid-ossp.c)**:

    • 1. モジュール内の関数から呼び出されます。 2. OSSP UUIDライブラリの uuid_import を呼び出し、入力文字列を uuid_t オブジェクトにインポートします。 3. 戻り値が UUID_RC_OK でない場合、pguuid_complain を呼び出してエラーを報告します。

重要な不変条件

  • **コア側 (src/backend/utils/adt/uuid.c)**:
  • 入力文字列 source はヌル終端文字列でなければなりません。
  • 変換結果は正確に16バイトのデータとして pg_uuid_t に格納されます。
  • 文字列が { で開始された場合、末尾は必ず } で閉じられていなければなりません。
  • ハイフンは特定のバイト位置でのみスキップが許容されます。
  • **拡張モジュール側 (contrib/uuid-ossp/uuid-ossp.c)**:
  • OSSP UUIDライブラリの uuid_import が成功することを前提とします。失敗した場合は必ず pguuid_complain によるエラー処理（例外送出など）が行われます。

よくある落とし穴

  • **同名関数の混同**:

    • src/backend/utils/adt/uuid.c と contrib/uuid-ossp/uuid-ossp.c にある string_to_uuid は、名前は同じですがシグネチャも内部実装も依存するデータ構造（pg_uuid_t vs uuid_t）も異なります。これらは互いに独立した static 関数です。

  • **フォーマット違反によるエラー**:

    • コア側の実装では、16進数以外の文字が含まれている場合、中括弧 {} が閉じられていない場合、または文字列の末尾に余分な文字が残っている場合に構文エラー（ERRCODE_INVALID_TEXT_REPRESENTATION）となります。

  • **ビルド依存性**:

    • contrib/uuid-ossp 側の関数は、ビルド環境で OSSP UUID ライブラリが有効化されている場合にのみ機能します。
