palloc_extended

概念

palloc_extended は、PostgreSQL のフロントエンドおよびバックエンドの両環境において、フラグ指定による柔軟なメモリ確保を可能にする共通の API インターフェースです。バックエンドのメモリ管理インターフェースとの互換性を維持し、共通コードの移植性を高める役割を持っています。

なぜ重要か

フロントエンドとバックエンドで共有されるコード（例: xlogreader.c など）において、同一のメモリ確保インターフェースを提供することで、コードの共通化を容易にします。また、呼び出し側はフラグ（例: MCXT_ALLOC_ZERO によるゼロクリア、MCXT_ALLOC_NO_OOM による OOM 時のプロセス終了回避など）を指定することで、メモリ確保の挙動を詳細に制御できます。

エンドツーエンドの流れ

フロントエンド環境における処理の流れは以下の通りです： 1. 呼び出し元が palloc_extended(size, flags) を呼び出します。 2. 内部的に pg_malloc_internal(size, flags) が呼び出され、サイズ検証が行われます。 3. malloc を実行します（サイズが 0 の場合は移植性のために malloc(1) として処理されます）。 4. メモリ確保に失敗し、かつ MCXT_ALLOC_NO_OOM フラグが指定されていない場合は、exit(EXIT_FAILURE) によりプロセスが終了します。 5. MCXT_ALLOC_ZERO フラグが指定されている場合、確保されたメモリ領域がゼロクリアされます。 6. 確保されたメモリへのポインタを返します。

※ バックエンド環境（src/backend/utils/mmgr/mcxt.c）におけるメモリコンテキストと連携した詳細なエンドツーエンドの流れは、提供された情報からは不明です。

重要な不変条件

  • **フロントエンド環境（FRONTEND マクロ定義下）**:
  • 確保要求サイズは SIZE_MAX / 2 以下でなければなりません。これを超える要求はエラーとなります。
  • MCXT_ALLOC_NO_OOM フラグが指定されていない限り、メモリ確保失敗時は必ずプロセスが終了（exit）します。
  • **バックエンド環境**:
  • バックエンド側における固有の不変条件は、提供された情報からは不明です。

よくある落とし穴

  • **暗黙のプロセス終了**: フロントエンド環境において MCXT_ALLOC_NO_OOM フラグを指定せずにメモリ確保に失敗した場合、呼び出し元に制御が戻らずにプロセスが終了します。呼び出し側でエラーハンドリングを行いたい場合は、明示的にフラグを指定する必要があります。
  • **サイズ制限**: 物理メモリの空き容量に関わらず、SIZE_MAX / 2 を超えるサイズを指定した場合はエラーとなります。
  • **バックエンド実装との差異**: フロントエンド版は標準の malloc をベースに動作しますが、バックエンド版は PostgreSQL 独自のメモリコンテキストシステム上で動作します。バックエンド側における具体的なライフサイクルや挙動の差異については、提供された情報からは未検証です。
