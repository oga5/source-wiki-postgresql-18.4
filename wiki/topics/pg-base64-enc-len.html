pg_base64_enc_len

概念

pg_base64_enc_len は、Base64 エンコード後のデータに必要なバッファサイズ（バイト数）を算出するための関数です。PostgreSQL のソースコード内には、異なるモジュールに同名の static 関数が2つ存在します。

なぜ重要か

Base64 エンコード処理（3バイトを4文字に変換）に加え、76文字ごとに改行コードを挿入するルール（MIMEやPGP Armor仕様）を考慮した正確なバッファサイズを事前に計算するために重要です。これにより、安全なメモリ割り当てを行い、バッファオーバーフローを防止します。

関与するモジュール

  • module:contrib/pgcrypto
  • module:src/backend/utils/adt

主要エンティティ

  • **symbol:contrib/pgcrypto/pgp-armor.c#pg_base64_enc_len**
  • シグネチャ: static unsigned pg_base64_enc_len(unsigned srclen)
  • 役割: PGP Armor エンコード処理（pgp_armor_encode など）において、必要なバッファサイズを算出します。
  • **symbol:src/backend/utils/adt/encode.c#pg_base64_enc_len**
  • シグネチャ: static uint64 pg_base64_enc_len(const char *src, size_t srclen)
  • 役割: 一般的な Base64 エンコード処理（pg_base64_encode など）において、必要なバッファサイズを算出します。なお、src 引数は現在の実装では使用されていません。

エンドツーエンドの流れ

1. **呼び出し**:

  • contrib/pgcrypto/pgp-armor.c 内のエンコード処理、または src/backend/utils/adt/encode.c 内の pg_base64_encode などのエンコーディング処理フローから、対象データのバイト長（srclen）を引数として呼び出されます。
  • ※ contrib/pgcrypto/pgp-armor.c から src/backend/utils/adt/encode.c の同名関数への参照も報告されていますが、両者が static 定義であるため、具体的なリンクや呼び出しの競合などの詳細なエンドツーエンドの相互作用は完全には明らかになっていません。

    • 2. **サイズ計算**:

  • 基本の Base64 変換サイズとして (srclen + 2) / 3 * 4 を計算します。
  • 76文字ごとに改行を挿入するルールに基づき、追加の改行バイト数 srclen / (76 * 3 / 4) を加算します。

    • 3. **バッファ確保**: 算出されたバイト数が呼び出し元に返され、メモリ割り当てに使用されます。

重要な不変条件

  • **変換比率**: 3バイトの入力データが4バイトの Base64 文字列に変換されること。
  • **改行挿入**: 76文字ごとに改行コードが挿入されること。
  • **入力値**: 入力 srclen は非負の整数であること。

よくある落とし穴

  • **オーバーフローの危険性**: srclen が極めて大きい場合、計算結果が unsigned または uint64 の範囲を超えてオーバーフローする可能性があります。本関数自体はオーバーフローチェックを行わないため、呼び出し側で MaxAllocSize 等を用いた検証を行う必要があります。
  • **重複実装の保守**: 同名の関数が2つのモジュールに独立して定義されているため、将来的に改行ルールなどの仕様変更が発生した場合は、両方の実装の整合性を保つ必要があります。
