PG_U_PROP_JOIN_CONTROL

概念

PG_U_PROP_JOIN_CONTROL は、Unicode 規格における「結合制御文字（Join Control）」プロパティを判定・管理するための識別子およびマクロ定義です。PostgreSQL 内部において、指定された Unicode コードポイントが結合制御文字（ゼロ幅接合子 ZWJ やゼロ幅非接合子 ZWNJ など）に該当するかどうかを効率的に判定するために使用されます。

なぜ重要か

結合制御文字は、テキストのレンダリングや文字の結合挙動を制御するために重要な役割を果たします。PostgreSQL では、Unicode 文字のプロパティ判定を高速化するために、ASCII 範囲（0x80 未満）に対するビットマスクを用いた定数時間判定と、それ以上の範囲に対する二分探索（範囲検索）を組み合わせた最適化を行っており、PG_U_PROP_JOIN_CONTROL はその最適化処理の基盤となっています。

  • **symbol:src/include/common/unicode_category_table.h#PG_U_PROP_JOIN_CONTROL**

エンドツーエンドの流れ

1. **判定の開始**: 呼び出し元（category_test.c や unicode_category.c 内部など）から pg_u_prop_join_control(code) が呼び出されます。 2. **ASCII 最適化パス**:

  • 入力された code が 0x80 未満の場合、静的テーブル unicode_opt_ascii を直接参照します。
  • PG_U_PROP_JOIN_CONTROL ビットマスクを用いて、該当するプロパティが有効であるかを定数時間で判定します。

    • 3. **範囲検索パス**:

  • 入力された code が 0x80 以上の場合、unicode_category_table.h で定義されている静的テーブル unicode_join_control に対して range_search を実行します。
  • コードポイントが定義された範囲内に存在すれば true、存在しなければ false を返します。

※ なお、この判定処理が PostgreSQL の SQL レイヤやパーサ、あるいは特定のエンコーディング変換処理などの全体的なエンドツーエンドフローの中でどのように統合され、最終的にどのように利用されるかという全体像の詳細は、提供されたコンテキストからは明らかではありません。

重要な不変条件

  • **入力型**: 判定対象のコードポイントは常に pg_wchar 型でなければなりません。
  • **Unicode 規格の安定性**: Unicode 規格の安定性ポリシーに基づき、一度定義されたカテゴリおよびプロパティ値は固定されているものとして扱われます。
  • **ASCII 境界**: 0x80 未満のコードポイントに対しては、必ず unicode_opt_ascii テーブルを用いた高速パスが適用されます。

※ その他のシステム全体にわたる不変条件は、現時点では完全には判明していません。
