huge_page_size

概念

huge_page_size は、PostgreSQLが共有メモリを確保する際に使用する Huge Page（ラージページ）のサイズを指定するためのGUC（Grand Unified Configuration）パラメータ、およびそれに関連するシステム制御機構です。

なぜ重要か

Huge Pageを使用することで、TLB（Translation Lookaside Buffer）のキャッシュミスを減らし、大規模な共有メモリ（shared_buffersなど）へのアクセス効率を向上させることができます。huge_page_size パラメータを適切に設定・取得することで、システムデフォルト以外の特定のHuge Pageサイズを明示的に選択し、メモリ割り当てを最適化することが可能になります。

エンドツーエンドの流れ

1. **設定の読み込み**: GUC変数 huge_page_size（guc_tables.c）を介して、ユーザーが希望するHuge Pageサイズが設定されます。 2. **サイズとフラグの取得**: 共有メモリの初期化時（ipci.c や shmem.c などから呼び出される際）に GetHugePageSize が実行されます。

  • **Linux等（sysv_shmem.c）**:
  • MAP_HUGETLB が定義されていない場合は、サイズとフラグに 0 を設定して終了します。
  • /proc/meminfo からシステムデフォルトのHuge Pageサイズ（default_hugepagesize）の取得を試みます。
  • 優先順位（明示的な huge_page_size 設定 ＞ システムデフォルト ＞ フォールバック値 2MB）に従って、最終的な hugepagesize を決定します。
  • MAP_HUGETLB フラグを設定し、必要に応じてページサイズに応じたシフト値をフラグに加算します。
  • **Windows（win32_shmem.c）**:
  • 常にサイズとフラグに 0 を設定して終了します（スタブ実装）。

    • 3. **メモリ確保**: 取得されたサイズとフラグを用いて、実際の共有メモリ確保（mmap 等）が行われます（※ただし、GetHugePageSize 呼び出し以降の具体的なメモリ確保の全体フローやエラーハンドリングの詳細は、提供されたコンテキストからは未確定です）。

重要な不変条件

  • **プラットフォーム依存性**: MAP_HUGETLB が定義されていないプラットフォームや、Windows環境（win32_shmem.c）では、GetHugePageSize は常にサイズおよびフラグとして 0 を返します。
  • **優先順位の不変規則**: sysv_shmem.c において、明示的に設定された huge_page_size は、システムデフォルト（/proc/meminfo から取得）よりも常に優先されます。

よくある落とし穴

  • **Windowsでの未サポート**: Windows環境における GetHugePageSize は現状スタブ実装であり、常に 0 を返すため、Windows上でラージページを有効化するには GetLargePageMinimum などのWindows APIを用いたコードの書き換えが必要です。
  • **アライメントエラー**: 実際のシステム設定がフォールバック値（2MB）より大きい場合、mmap や munmap がアライメントエラーで失敗する可能性があります。
  • **Linux以外の非Windows環境**: Linux以外のプラットフォームでは /proc/meminfo が存在しないため、システムデフォルトの取得ロジックがなく、2MBのフォールバック値に依存します。
