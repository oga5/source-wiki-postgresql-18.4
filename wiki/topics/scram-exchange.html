scram_exchange

概念

scram_exchange は、PostgreSQLにおけるSASL SCRAM-SHA-256認証プロトコルを処理するための、サーバー側（backend）およびクライアント側（libpq）の双方に存在する主要なステートマシン関数です。この関数は、認証プロセスにおけるメッセージの送受信、解析、検証、および状態遷移を制御します。

なぜ重要か

PostgreSQLのセキュアな接続確立において、SCRAM-SHA-256は標準的な認証方式です。scram_exchange は、サーバーとクライアントの間でやり取りされるチャレンジ＆レスポンスのフローを厳密に管理し、タイミング攻撃の防止（モック認証）やチャネルバインディングの適用など、セキュリティ上の重要な要件を担保する役割を担っています。

  • クライアント側（libpq）でサーバーからのメッセージを処理し、証明書や署名の検証を行う。

エンドツーエンドの流れ

サーバーとクライアントの scram_exchange は、SASL認証フレームワークを介して以下のように協調して動作します。ただし、これらを統括する上位のSASLフレームワーク全体の完全なエンドツーエンドフローの詳細は、提供されたコンテキストからは完全には明らかになっていません。

1. **初期化フェーズ**:

  • クライアント側は FE_SCRAM_INIT 状態から開始し、クライアントの nonce を含む最初のメッセージを生成してサーバーへ送信します。
  • サーバー側は SCRAM_AUTH_INIT 状態において、クライアントの最初のメッセージを解析し、サーバーの nonce と salt を含むメッセージを生成して SCRAM_AUTH_SALT_SENT へ移行します。

    • 2. **チャレンジ・レスポンス検証フェーズ**:

  • クライアント側は FE_SCRAM_NONCE_SENT 状態に移行し、サーバーからの salt/nonce を解析してクライアント証明（Proof）を生成し、サーバーへ送信して FE_SCRAM_PROOF_SENT へ移行します。
  • サーバー側は SCRAM_AUTH_SALT_SENT 状態において、クライアントの最終メッセージを解析し、nonce と Proof を検証します（verify_client_proof）。

    • 3. **完了フェーズ**:

  • サーバー側は検証成功時に SCRAM_AUTH_FINISHED 状態へ移行し、セッション鍵を MyProcPort に格納して PG_SASL_EXCHANGE_SUCCESS を返します。
  • クライアント側は FE_SCRAM_PROOF_SENT 状態において、サーバーからの署名を検証し（verify_server_signature）、一致すれば SASL_COMPLETE を返します。

重要な不変条件

  • **サーバー側**:
  • opaq は有効な scram_state 構造体へのポインタでなければなりません。
  • input が NULL の場合は、認証の初期段階（SCRAM_AUTH_INIT）でなければなりません。
  • 認証成功時、MyProcPort に ClientKey および ServerKey が保存されます。
  • **クライアント側**:
  • FE_SCRAM_INIT 以外の状態では、inputlen が 0 でないこと、および input の文字列長と一致することが必須です。
  • input 文字列は、解析過程（read_attr_value 等）で破壊的に変更される可能性があります。

よくある落とし穴

  • **入力データの破壊**: クライアント側の scram_exchange は入力メッセージ（input）を破壊的に処理するため、呼び出し元は再利用や不変性を前提としたバッファの扱いに注意する必要があります。
  • **タイミング攻撃**: サーバー側では、ユーザーが存在しない場合でもモック認証として計算を継続し、意図的に失敗（doomed フラグ）させることで、ユーザーの存在有無が処理時間から漏洩するのを防いでいます。このロジックをバイパスしないように注意が必要です。
  • **プロトコル違反の厳格なチェック**: メッセージの長さ不一致や空メッセージの受信など、プロトコル違反が発生した場合は即座にエラー（PG_SASL_EXCHANGE_FAILURE または SASL_FAILED）を返し、詳細なエラーを記録する必要があります。
