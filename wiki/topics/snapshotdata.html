SnapshotData

概念

SnapshotData は、PostgreSQLにおいて特定のトランザクションから見てどのタプル（行）が可視（参照可能）であるかを判定するための基準情報を保持する構造体です。MVCC（多版同時実行制御）やリカバリ時の状態管理において、可視性判定の抽象化インターフェースとして機能します。

なぜ重要か

PostgreSQLのMVCCにおいて、同時実行される複数のトランザクション間で一貫したデータの可視性を保証するために不可欠なデータ構造です。また、前回のスナップショット生成以降にトランザクションの完了がない場合に再計算を回避する最適化（snapXactCompletionCount による管理）など、パフォーマンス面でも重要な役割を担っています。

主要エンティティ

  • symbol:src/backend/storage/ipc/procarray.c#GetSnapshotData: 現在のトランザクション状態やリカバリ状態に基づいて SnapshotData を構築する関数。

エンドツーエンドの流れ

1. **構築**: GetSnapshotData() が呼び出され、現在のシステム状態（トランザクションやリカバリの状態）を反映した SnapshotData が構築されます。 2. **判定**: HeapTupleSatisfiesMVCC などの可視性判定関数が、構築された SnapshotData を参照してタプルの可視性を判定します。 3. **管理**: スナップショットのライフサイクルは、参照カウント（active_count, regd_count）や pairingheap_node を通じて管理されます。

※ なお、呼び出し元（heapam.c やレプリケーション関連モジュールなど）との詳細な連携や、システム全体における網羅的なエンドツーエンドの制御フローの全貌は、提供されたコンテキストからは完全には明らかになっていません。

重要な不変条件

  • **xminの境界**: xmin 未満のすべてのトランザクションID（XID）は可視です。
  • **xmaxの境界**: xmax 以上のすべての XID は不可視です。
  • **xipの範囲**: xip[] に含まれるすべての ID は、xmin <= id < xmax の範囲を満たします。
  • **subxipの範囲**: subxip[] に含まれるすべての ID は、xmin 以上です。
  • **非MVCC型での例外**: snapshot_type が SNAPSHOT_MVCC 以外の場合、xmin や xmax などの一部のフィールドは特殊な目的で再利用されるか、ゼロ値となります。

よくある落とし穴

  • **型依存のフィールド解釈**: フィールドの解釈は snapshot_type に依存するため、スナップショットの型を誤認してフィールドにアクセスすると、不正な可視性判定を引き起こします。
  • **サブトランザクションのオーバーフロー**: サブトランザクション数が上限を超えると suboverflowed フラグが true になります。この状態を考慮したハンドリングが必要です。
  • **直接的な参照カウント操作**: メモリ管理は snapmgr.h 経由で行う必要があり、active_count や regd_count を直接操作することは禁止されています。
  • **リカバリ時の特殊挙動**: リカバリ中に取得されたスナップショットでは、xip が空になる場合があります。
