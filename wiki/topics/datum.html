Datum

概念

Datum は、PostgreSQL において任意のデータ型（数値、文字列、ポインタなど）を単一のポインタサイズ（通常は uintptr_t）の整数値として統一的に保持・表現するための、極めて基本的な汎用データ型です。値そのものを直接格納する「値渡し（pass-by-value）」と、メモリ上のデータ構造を指すポインタを格納する「参照渡し（pass-by-reference）」の2つの形態を抽象化します。

なお、contrib/sepgsql/label.c のように、一部のモジュールでは同名のローカルな構造体として定義・利用される例外もあります。

なぜ重要か

PostgreSQL の実行エンジン、関数マネージャ（fmgr）、インデックスアクセスメソッド（B-Tree, GIN, SP-GiST, BRIN など）、および各種拡張モジュール（PL/Python, PL/Perl など）は、データ型に依存しない共通のインターフェースを介してデータをやり取りする必要があります。Datum はこの「共通通貨」として機能することで、システムの高い拡張性と柔軟性を支えています。

エンドツーエンドの流れ

Datum を介した具体的なエンドツーエンドのデータ処理フロー（例：SQLクエリからインデックススキャン、PL/Pythonでの型変換を経て結果を返すまでの一連のライフサイクル）の全体像は、提供されたモジュール情報からは完全には特定されていません。

ただし、部分的な流れとして以下が確認できます。 1. **式評価と定数保持**: execExpr.h の constval 構造体などに Datum として定数データが格納され、式評価ステップ（EEOP_CONST）で読み出されます。 2. **アクセスメソッドへの伝達**: tableam.h や amapi.h を介して、テーブルやインデックスのキー・属性値が Datum 配列として渡されます。 3. **外部言語へのマーシャリング**: plpy_typeio.h や hstore_plpython.c において、PostgreSQL の Datum が PLy_input_convert などを経て Python の PyObject へ変換されます。

重要な不変条件

  • **サイズの一致**: Datum は、プラットフォームのポインタサイズ（uintptr_t）と同一のビット幅（64bit環境では8バイト、32bit環境では4バイト）を持つ必要があります。
  • **型情報の非保持**: Datum 自体は型情報を保持しません。そのため、値を取り出す、あるいは解釈する際には、対応する Oid（型識別子）や byValue フラグなどのメタデータと常に整合していなければなりません。
  • **NULLの分離**: Datum 自体は SQL の NULL を表現できません。isnull などの bool 型フラグを別途用意して管理する必要があります（例：execExpr.h の constval 構造体）。
  • **メモリの有効性**: 参照渡し（ポインタ格納）の場合、Datum が参照される期間中、そのポインタが指すメモリ領域（通常は特定の MemoryContext 内）が有効であり続けなければなりません。

よくある落とし穴

  • **型安全性の欠如と誤キャスト**: Datum は単なる整数型（uintptr_t）であるため、コンパイラによる型チェックが働きません。DatumGetPointer や Int32GetDatum などの適切なマクロを使用せずに直接キャストを行うと、メモリ破損やセグメンテーションフォールトを引き起こします。
  • **セキュリティラベル定義との混混同**: contrib/sepgsql/label.c 内では、コアの Datum 型とは異なるローカルな struct として Datum が定義されている箇所があり、コードの解読や修正時に混同しないよう注意が必要です。
  • **ダングリングポインタ**: 参照渡しの Datum を扱う際、参照先のメモリが pfree などで解放された後にアクセスすると、未定義動作を招きます。
