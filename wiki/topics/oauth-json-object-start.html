oauth_json_object_start

概念

oauth_json_object_start は、PostgreSQLのクライアントライブラリ（libpq）およびそのOAuth関連モジュールにおいて、JSONデータの解析中にオブジェクトの開始（{）を検出し、ネストレベルの管理や構造のバリデーションを行うためのコールバック関数です。

なぜ重要か

OAuth認証プロセスなどでやり取りされるJSONデータの整合性を検証し、不正なデータ構造や過度に深いネスト（DoS攻撃などにつながる恐れがあるもの）を早期に検知して処理を安全に中断させるために重要です。この関数は、common/jsonapi.h が提供するJSONパーサーのセマンティックアクションコールバックとして機能します。

エンドツーエンドの流れ

OAuth認証シーケンス全体における詳細なエンドツーエンドのフローは、提供された情報からは完全には明らかになっていません。

現時点で判明している部分的な処理の流れは以下の通りです： 1. JSONパーサーがJSON入力ストリームを解析し、オブジェクトの開始（{）を検出します。 2. パーサーのコールバック（sem.object_start への代入を通じて間接的に登録されたもの）を介して、oauth_json_object_start が呼び出されます。 3. 関数内部で、現在のコンテキスト（json_ctx）のターゲットフィールドの状態を検証し、ネストレベル（ctx->nested）をインクリメントします。 4. ネストレベルが制限を超えている場合や構造違反がある場合は、oauth_json_set_error を呼び出してエラーを記録し、JSON_SEM_ACTION_FAILED を返します。 5. エラーがなければ JSON_SUCCESS を返し、パーサーは解析を続行します。

重要な不変条件

  • **コンテキストの有効性**: 引数 state は、有効な json_ctx 構造体を指すポインタでなければなりません。
  • **最大ネスト制限**: 解析中のJSONオブジェクトのネスト深さ（ctx->nested）は、常に MAX_SASL_NESTING_LEVEL を超えてはなりません。
  • **型の一貫性**: ctx->target_field が設定されている場合、オブジェクトの開始は許可されません（文字列などのプリミティブ値が期待されているため）。

よくある落とし穴

  • **ネスト制限による解析失敗**: MAX_SASL_NESTING_LEVEL を超える深い階層を持つJSON構造を処理しようとすると、正当なデータであっても JSON_SEM_ACTION_FAILED となり解析が失敗します。
  • **コンテキスト状態の不整合**: ctx->target_field の状態管理が不適切な場合、予期しないタイミングで構造違反エラーが記録される可能性があります。
