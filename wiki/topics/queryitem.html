QueryItem

概念

QueryItem は、PostgreSQLの全文検索（Text Search）機能において、検索クエリを表す tsquery の内部ノード（演算子またはオペランド/検索語）を表現するためのデータ構造です。

  • src/include/tsearch/ts_type.h では、ノードのタイプを識別する QueryItemType、演算子を表す QueryOperator、オペランドを表す QueryOperand を格納する共用体（union）として定義されています。
  • src/backend/utils/adt/tsrank.c においても、項目タイプ（type）を保持し、適合度計算などで参照される同名の構造体（struct）として定義されています。

なぜ重要か

QueryItem は、全文検索クエリ（tsquery）の構文木やフラットな配列表現における各要素の核となるデータ構造です。クエリの解析、評価、インデックス（GIN/GiST）の走査、および適合度（ランク）計算など、全文検索に関わるほぼすべての処理がこのデータ構造を介して行われます。この構造体の設計は、ディスク上およびメモリ上の tsquery のバイナリ表現と直接結びついています。

関与するモジュール

  • module:src/backend/utils/adt （特に tsrank.c, tsginidx.c, tsquery.c など）
  • module:src/include/tsearch （特に ts_type.h）

主要エンティティ

  • **symbol:src/include/tsearch/ts_type.h#QueryItem**

    • tsquery 構造体内部の各ノードを表現するための共用体。type（型判別用）、qoperator（演算子データ）、qoperand（オペランドデータ）をメンバに持ちます。

  • **symbol:src/backend/utils/adt/tsrank.c#QueryItem**

    • 適合度計算（calc_rank 等）の文脈などで利用される、項目タイプ（type）を持つ構造体。

エンドツーエンドの流れ

詳細なエンドツーエンドの生成・評価フローの全容は提供されたコンテキストからは完全には明らかではありませんが、大まかな流れは以下の通りです：

1. **解析と構築**: ユーザー入力から ts_parse.c や tsquery.c を通じて tsquery が解析され、その内部表現として QueryItem の配列が構築されます。 2. **評価と走査**: クエリの実行時、tsquery 内の QueryItem 配列が走査されます。type フィールドを参照して、ノードが演算子（QueryOperator）かオペランド（QueryOperand）かを判別し、演算子の評価やオペランドの照合、適合度計算（tsrank.c の calc_rank など）へと分岐します。

重要な不変条件

  • **アライメント**: tsquery は4バイトアライメントを前提としています。
  • **型判別の整合性**: QueryItem の type フィールド（または QueryItemType）は、共用体内のどのメンバ（qoperator または qoperand）が有効であるかを正確に識別しなければなりません。
  • **未知の型に対するエラー**: 実行エンジンが QueryItem を処理する際、未知の type 値を検出した場合は elog(ERROR) により処理を中断する必要があります（tsquery.c:917）。

よくある落とし穴

  • **バイナリ互換性とインデックスへの影響**: QueryItem は tsquery のバイナリ表現の核であるため、この構造を変更すると、既存のインデックス（GIN/GiST）や保存されたデータとの互換性が失われます。
  • **メモリレイアウトとサイズ計算**: tsquery のメモリレイアウトを直接操作する場合、アライメントや varlena ヘッダーの整合性に注意する必要があります。構造体のサイズ計算には、直接 sizeof を使うのではなく、ts_type.h で提供される専用マクロを使用する必要があります。
  • **不正なタイプ値**: 不正な type 値が設定されている場合、tsquery.c 内の findoprnd などの処理関数でエラーが発生します。
  • **スタックオーバーフロー**: 複雑なクエリ構造において再帰的な走査を行う際、スタックオーバーフローが発生する可能性があります。
