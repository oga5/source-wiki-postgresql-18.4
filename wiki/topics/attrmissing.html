AttrMissing

概念

AttrMissing は、ALTER TABLE ADD COLUMN などのテーブル定義変更によって、物理的なタプル（行データ）上に存在しない属性（カラム）が発生した場合に、その属性の読み取り時に供給すべきデフォルト値を保持するためのデータ構造です。

なぜ重要か

PostgreSQLにおいて、テーブルに新しいカラムがデフォルト値付きで追加された際、既存のすべての物理タプルを即座に書き換える（書き換えコストを発生させる）代わりに、物理タプル上に存在しない属性に対してこの構造体を介してデフォルト値を動的に供給することで、スキーマ変更の高速化を実現するために重要です。

関与するモジュール

  • module:src/backend/catalog
  • module:src/include/access

主要エンティティ

  • symbol:src/include/access/tupdesc_details.h#AttrMissing: 欠落した属性のデフォルト値情報を保持する構造体。
  • symbol:src/backend/catalog/heap.c#SetAttrMissing: 欠落属性情報を設定するための関数。

エンドツーエンドの流れ

テーブル定義変更（ALTER TABLE など）が行われた際、新しく追加されたカラムのデフォルト値情報が AttrMissing 構造体に格納されます。その後、タプルスキャンやデータ変換処理において、物理タプル上に該当する属性が存在しないと判断された場合に、この構造体が参照されてデフォルト値が供給されます。

ただし、SetAttrMissing の具体的な呼び出しタイミングや、タプルスキャン時にどのように AttrMissing が解決されるかの詳細なエンドツーエンドの制御フローは、提供された情報からは完全には明らかになっていません。

重要な不変条件

  • am_present が false の場合、属性は欠落しており、かつデフォルト値も存在しない（NULLとして扱われる）ことを示します。
  • am_present が true の場合、am_value に有効なデフォルト値（Datum）が格納されている必要があります。

よくある落とし穴

  • am_value に格納される Datum がポインタ型である場合、そのメモリ管理上の制約（生存期間など）に違反すると、データアクセス時に予期しない問題（メモリ破損や不正アクセスなど）が発生する可能性があります。
  • AttrMissing は tupdesc.h と密接に関連していますが、循環参照を避けるために tupdesc_details.h に分離されています。インクルード時の依存関係の連鎖に注意する必要があります。
