GetAggInitVal

概念

GetAggInitVal は、集約関数（Aggregate Function）の初期値（initval）を、テキスト形式の Datum から指定された遷移データ型（transtype）の内部表現（Datum）へと変換して取得するための共通ユーティリティ関数です。

なぜ重要か

PostgreSQLのシステムカタログ等において、集約関数の初期値はテキスト形式で保持されています。集約処理やウィンドウ関数処理を実行、あるいは最適化（プラン作成）する段階で、このテキスト表現を実際の遷移データ型（transtype）の内部表現（Datum）に変換する必要があります。GetAggInitVal は、この変換処理を一貫した方法で安全に行うために重要です。

エンドツーエンドの流れ

集約処理やウィンドウ関数の初期化、あるいはプラン作成の準備段階において、以下の流れで初期値の変換が行われます。

1. **呼び出し**: 実行エンジン（nodeAgg.c、nodeWindowAgg.c）またはオプティマイザ（prepagg.c）から、テキスト形式の初期値（textInitVal）と遷移型（transtype）を引数として GetAggInitVal が呼び出されます。 2. **型情報の取得**: getTypeInputInfo を呼び出し、指定された transtype に対応する入力関数（typinput）とパラメータ（typioparam）をシステムカタログ等から取得します。 3. **文字列への変換**: TextDatumGetCString を使用して、テキスト形式の textInitVal を一時的な C 文字列（strInitVal）に変換します。 4. **内部表現への変換**: OidInputFunctionCall を呼び出し、取得した入力関数を用いて C 文字列を transtype の Datum 形式に変換します。 5. **メモリ解放**: 変換に使用した一時的な C 文字列を pfree で解放します。 6. **返却**: 変換された Datum を呼び出し元に返します。

なお、これら呼び出し元モジュール間を跨ぐ全体的な制御フローや、どのタイミングでどのモジュールが優先して呼び出されるかといった詳細なエンドツーエンドの相互作用の全貌は、提供された情報からは完全には特定されていません。

重要な不変条件

  • **入力関数の存在**: 引数として渡される transtype に対応する有効な入力関数（Input Function）がシステムカタログから取得可能でなければなりません。
  • **入力データの妥当性**: textInitVal は TextDatumGetCString で処理可能な有効なテキストデータ（text 型の Datum）でなければなりません。
  • **メモリ管理**: 変換処理の過程で一時的に確保された C 文字列（strInitVal）は、関数を抜ける前に必ず pfree によって解放されなければなりません。

よくある落とし穴

  • **無効なデータ型（Oid）の指定**: 存在しない、または入力関数が定義されていない無効な transtype を指定した場合、getTypeInputInfo 内でエラーが発生します。
  • **不正な初期値フォーマット**: textInitVal のテキスト表現が、指定された transtype の入力関数が受け付けるフォーマットと一致しない場合、OidInputFunctionCall 内でパースエラー（実行時エラー）が発生します。
  • **複数箇所での定義・参照**: 本関数は nodeAgg.c、nodeWindowAgg.c、prepagg.c の複数箇所に定義または参照が存在するため、共通ロジックを修正する際には、実行エンジンとオプティマイザの双方に影響が及ぶ点に注意する必要があります。
