RelationInfo

概念

RelationInfo というキーワードは、PostgreSQLのコードベースにおいて、バックエンド（クエリ最適化）とフロントエンド（検証ツール）の2つの異なる文脈で異なる役割を持っています。

1. **バックエンド（オプティマイザ）**: get_relation_info 関数を指します。プランナがクエリ最適化を行うために、対象リレーションの物理的特性（属性、インデックス、統計情報など）を収集し、RelOptInfo 構造体を初期化する役割を担います。 2. **フロントエンド（pg_amcheck）**: RelationInfo 構造体を指します。検証対象となる個々のリレーション（ヒープテーブルまたはB-tree索引）のメタデータを保持し、並列検証プロセスにおける各リレーションの識別とSQL構築を支援します。

なぜ重要か

  • **クエリ最適化の基盤**: バックエンドにおいて、プランナが最適な実行計画を立てるためには、正確なリレーション情報（サイズ、インデックスの有効性、並列ワーカー数など）を迅速かつ正確に取得する必要があります。
  • **並列検証の効率化**: pg_amcheck において、複数のリレーションを並列に検証する際、各検証タスクのコンテキスト（対象リレーション、検証用SQLなど）を管理するために RelationInfo 構造体が不可欠です。

エンドツーエンドの流れ

バックエンドとフロントエンドで処理の流れが独立しています。

バックエンド（プランナ）における流れ

1. 呼び出し元（build_simple_rel など）が、事前に対象リレーションのロックを取得した状態で get_relation_info を呼び出します。 2. table_open を用いて NoLock でリレーションを開き、テーブルAMの存在確認やリカバリ中のアクセス制限をチェックします。 3. RelOptInfo の属性範囲やテーブルスペース情報を設定し、継承親でない場合は estimate_rel_size でサイズを推定します。 4. RelationGetIndexList でインデックス一覧を取得し、有効なインデックスに対して IndexOptInfo を生成して詳細情報を格納します。

フロントエンド（pg_amcheck）における流れ

1. pg_amcheck のリスト走査処理や並列スロットのコンテキスト初期化において、pg_malloc0 を介して RelationInfo 構造体がメモリ確保・初期化されます。 2. prepare_heap_command または prepare_btree_command に RelationInfo が渡され、検証用のSQLコマンドが動的に構築されて sql フィールドに格納されます。 3. 並列実行スロットのハンドラ（verify_heap_slot_handler や verify_btree_slot_handler）内で、この構造体がコンテキストとして参照され、検証が実行されます。 4. 実行後、動的に確保された sql フィールドは pg_free によって解放されます。

*注意: これら2つのコンテキスト（バックエンドのプランナ処理とフロントエンドのpg_amcheckツール）の間の直接的なエンドツーエンドの連携フロー（例: pg_amcheckがバックエンドのget_relation_infoを直接呼び出す詳細な内部シーケンスなど）は、提供された情報からは不明です。*

重要な不変条件

  • **バックエンド (get_relation_info)**:
  • 対象リレーションは、呼び出し前に既にロックが取得されている必要があります（table_open は NoLock で実行されます）。
  • rel->max_attr >= rel->min_attr である必要があります。
  • 属性の attnullability は ATTNULLABLE_UNKNOWN であってはなりません。
  • **フロントエンド (RelationInfo 構造体)**:
  • is_heap が true の場合はヒープテーブル、false の場合はB-tree索引として扱われます。
  • datinfo は他の RelationInfo インスタンスと共有される可能性があります。

よくある落とし穴

  • **バックエンド**:
  • get_relation_info はロック管理を呼び出し元に依存しているため、事前に適切なロックを取得せずに呼び出すと競合や一貫性の欠如を招きます。
  • 継承親テーブルの場合、サイズ推定はスキップされる（後続の set_append_rel_size で処理される）ため、この関数単体でサイズ情報が完結しない点に注意が必要です。
  • **フロントエンド**:
  • RelationInfo の sql フィールドは動的にメモリ確保されるため、適切なタイミングで pg_free による解放を行わないとメモリリークが発生します。
