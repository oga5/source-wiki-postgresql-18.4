ReplicationSlot

概念

ReplicationSlot（レプリケーションスロット）は、PostgreSQLにおいてプライマリ（送信側）サーバーが、スタンバイや論理レプリケーションクライアント（受信側）のために必要なWAL（Write-Ahead Log）や古いシステムカタログのタプルを保持し続けるための状態管理機構です。

メモリ内では ReplicationSlot 構造体として表現され、物理レプリケーションと論理レプリケーションの双方をサポートします。

なぜ重要か

レプリケーションスロットがない場合、受信側の処理が遅延すると、送信側で不要と判断されたWALファイルが削除されたり、バキュームによって古いタプルが回収されたりして、レプリケーションが継続不可能になるリスクがあります。ReplicationSlot は、受信側が必要とする最小限の進捗（LSNやxmin）を送信側に登録・維持させることで、データの早期消失を防ぎ、レプリケーションの整合性と継続性を保証します。

関与するモジュール

  • **module:src/backend/replication**: サーバー側におけるスロットの作成、管理、およびWAL送信（walsender）などのコアロジック。
  • **module:src/bin/pg_basebackup**: クライアントツール（pg_basebackup, pg_receivewal, pg_recvlogical）におけるスロットの作成・利用。
  • **module:src/include/replication**: ReplicationSlot 構造体などのデータ定義。

主要エンティティ

  • **struct ReplicationSlot** (src/include/replication/slot.h): 共有メモリ上のスロット状態を保持する構造体。
  • **ReplicationSlotCreate** (src/backend/replication/slot.c): サーバー側で新しいスロットを割り当て、ディスク上に永続化し、共有メモリ上でアクティブ化する関数。
  • **CreateReplicationSlot (サーバー側)** (src/backend/replication/walsender.c): クライアントからの CREATE_REPLICATION_SLOT コマンドを処理し、物理または論理スロットを作成・初期化する。
  • **CreateReplicationSlot (クライアント側)** (src/bin/pg_basebackup/streamutil.c): libpq を介してサーバーに対して CREATE_REPLICATION_SLOT コマンドを発行するヘルパー関数。
  • **クライアント側変数 replication_slot**: pg_basebackup.c, pg_receivewal.c, pg_recvlogical.c で定義され、コマンドライン引数などから指定された対象スロット名を保持する。

エンドツーエンドの流れ

1. **クライアントからの要求**: pg_basebackup などのクライアントツールが、streamutil.c の CreateReplicationSlot を呼び出し、接続先サーバーに対して CREATE_REPLICATION_SLOT コマンドを送信します。 2. **サーバー側でのコマンド受信と解析**: walsender.c の CreateReplicationSlot がコマンドを受信し、物理スロットか論理スロットかに応じてオプションを解析します。 3. **スロットの作成と初期化**:

  • サーバーは ReplicationSlotCreate を呼び出し、共有メモリ上に ReplicationSlot を確保し、ディスク上に永続化用のディレクトリを作成します。
  • 物理スロットの場合、必要に応じて WAL を予約します（ReplicationSlotReserveWal）。
  • 論理スロットの場合、デコーディングコンテキスト（LogicalDecodingContext）を初期化し、開始地点（LSN）を特定してスナップショットのエクスポートなどを行います。

    • 4. **結果の返却**: 作成されたスロットの情報（LSNやスナップショット名など）がクライアントに返却されます。

*※なお、作成されたスロットを用いた継続的なWALストリーミングやデコーディングの具体的な詳細フロー（エンドツーエンドの全体像）については、提供されたモジュール情報からは完全には網羅されていません。*

重要な不変条件

  • **単一所有権**: スロットを作成または使用する際、現在のバックエンドプロセス（MyReplicationSlot）は既に別のスロットを所有していてはなりません（MyReplicationSlot が NULL であること）。
  • **排他制御**:
  • スロットの追加・削除には ReplicationSlotControlLock（LWLock）の取得が必要です。
  • スロットの作成・割り当て時には ReplicationSlotAllocationLock を排他モードで取得します。
  • スロット内の個別フィールドの更新には、スロット固有の mutex（スピンロック）の取得が必須です。
  • **物理と論理のパラメータ制約**:
  • 物理スロット作成時は、出力プラグイン（plugin）が NULL でなければなりません。
  • 論理スロット作成時は、plugin が NULL であってはなりません。また、two_phase や failover などの論理専用オプションは物理スロットと同時に指定できません。

よくある落とし穴

  • **名前の衝突**: 同名のスロットが既に存在する場合、ERRCODE_DUPLICATE_OBJECT エラーが発生します。ただし、クライアント側で slot_exists_ok が true に設定されている場合は、エラーを無視して成功とみなすことができます。
  • **リソース制限**: max_replication_slots 設定値の上限に達している場合、新しいスロットの作成は ERRCODE_CONFIGURATION_LIMIT_EXCEEDED で失敗します。
  • **トランザクション制約**: 論理スロット作成時にスナップショット操作（CRS_USE_SNAPSHOT）を行う場合、トランザクションブロック内であること、REPEATABLE READ 隔離レベルであること、読み取り専用であることなど、厳しいトランザクション状態の制約を満たさないとエラーになります。
  • **バージョン互換性**: クライアント（streamutil.c）がサーバーにコマンドを送る際、PostgreSQL 15以降で導入された括弧付きの新しいオプション構文とそれ以前の構文の差異を考慮する必要があります。
