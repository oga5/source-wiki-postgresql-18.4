ResourceOwner

概念

ResourceOwner は、クエリやトランザクションの生存期間中に確保されたシステムリソース（バッファピン、ロック、キャッシュ参照、AIO ハンドル、OpenSSL ハンドルなど）を追跡し、適切なタイミングで確実に解放するためのメモリ管理・クリーンアップ機構です。リソースオーナーは親子関係による階層構造を持つことができます。

なぜ重要か

トランザクションのコミット、アボート、またはエラー発生時などに、確保されたリソースがリークすることなく、定義されたフェーズや優先度に従って確実に解放されるようにするために極めて重要です。これにより、システムの安定性とリソース管理の正確性が保証されます。

エンドツーエンドの流れ

リソースの登録から解放、そして削除に至る完全なライフサイクル全体の詳細な流れ（サブトランザクションでの伝播や具体的なエラーハンドリング時の挙動など）は、提供された情報からは完全には記述されておらず、一部未検証です。

現時点で判明している部分的な流れは以下の通りです：

1. **生成**: トランザクション開始時（xact.c）、ポータル作成時（portalmem.c）、またはPL/pgSQLプロシージャ実行時などに ResourceOwnerCreate が呼び出され、必要に応じて親オーナーとリンクされた ResourceOwner が生成されます。 2. **登録と追跡**: リソース確保時に ResourceOwnerRemember を介してリソースを登録し、内部の固定長配列やハッシュテーブルで追跡します。 3. **解放**: ResourceOwnerRelease が呼び出されると、定義されたフェーズ（BEFORE_LOCKS, LOCKS, AFTER_LOCKS）と優先度（RELEASE_PRIO_*）に従い、登録されたコールバックを実行してリソースを解放します。 4. **削除**: リソースが空になった後、ResourceOwnerDelete を呼び出します。これにより、子オーナーが再帰的に削除され、親とのリンクが解除された後、メモリが解放されます。

重要な不変条件

  • **削除時のカレント制限**: 削除対象の owner は、現在のリソースオーナー（CurrentResourceOwner）と同一であってはなりません（ResourceOwnerDelete の前提条件）。
  • **削除時のリソース空条件**: ResourceOwnerDelete を呼び出す際、owner が保持するリソース（配列、ハッシュテーブル、ロック）はすべて解放済みでなければなりません（narr == 0, nhash == 0, nlocks が 0 または MAX_RESOWNER_LOCKS + 1 であること）。
  • **コールバックの安全性**: ReleaseResource コールバックは、コミットまたはアボート後に実行されるため、失敗が許されない非クリティカルなクリーンアップ処理である必要があります。
  • **メモリコンテキスト**: 生成された ResourceOwner は TopMemoryContext に割り当てられます。

よくある落とし穴

  • **リソースリーク**: 追跡対象のリソースが適切に登録・忘却（ResourceOwnerForget）されない場合、リソースリークが発生します。
  • **未解放状態での削除**: リソースが適切に解放されていない状態で ResourceOwnerDelete を呼び出すと、アサーション失敗による異常終了や、リソースリーク、不正なメモリ参照を引き起こす可能性があります。
  • **スタックオーバーフロー**: 階層構造の削除は再帰的に行われるため、極端に深い階層を持つオーナーを削除する際はスタックオーバーフローに注意が必要です。
  • **優先度設定の誤り**: 拡張機能で新しいリソース型を追加する場合、既存の RELEASE_PRIO_* に基づく適切な優先度を設定しなければ、解放順序の不整合を招く恐れがあります。
