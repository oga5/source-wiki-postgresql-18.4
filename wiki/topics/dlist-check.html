dlist_check

概念

dlist_check は、PostgreSQLの二重連結リスト（dlist）のポインタ整合性を検証し、リスト構造の破損を検知するためのデバッグ用関数です。

なぜ重要か

二重連結リストの操作中にポインタの不整合やメモリ破損が発生した場合、それを早期に検知してプロセスの異常動作を防ぐために重要です。特にデバッグビルド（ILIST_DEBUG 有効時）において、リスト操作マクロの実行前後にこの検証を挟むことで、バグの早期発見に貢献します。

関与するモジュール

  • module:src/backend/lib
  • module:src/include/lib

主要エンティティ

  • symbol:src/backend/lib/ilist.c#dlist_check: リストの整合性検証アルゴリズムを実装している関数。
  • symbol:src/include/lib/ilist.h#dlist_check: dlist_check の宣言、およびデバッグ用マクロ（dlist_insert_after など）からの呼び出しを定義しているヘッダー。

エンドツーエンドの流れ

1. src/include/lib/ilist.h 内で定義されたリスト操作マクロ（dlist_insert_after など）が呼び出されます。 2. ILIST_DEBUG が有効な場合、これらのマクロの内部から dlist_check が呼び出されます。 3. dlist_check は、渡された dlist_head ポインタが NULL でないことを確認します。 4. リストがゼロ初期化されている（空の）場合は、検証をスキップして正常終了します。 5. **順方向走査**: head->head.next から開始し、head->head に戻るまで各ノードを辿り、隣接ポインタが NULL でないこと、および相互参照関係が正しいことを検証します。 6. **逆方向走査**: head->head.prev から開始し、同様に head->head に戻るまで逆方向に検証します。 7. 破損が検知された場合、elog(ERROR, ...) を呼び出してプロセスを中断します。

※なお、具体的なマクロ呼び出しの全容や、ILIST_DEBUG 以外のコンテキストにおける詳細なエンドツーエンドの統合フローは、提供された情報からは完全には判明していません。

重要な不変条件

  • **非NULLヘッド**: 入力される head ポインタは NULL であってはなりません。
  • **空リストの許容**: リストが空（head->head.next および prev が NULL）の場合、初期化済みとみなされます。
  • **ポインタの対称性（相互参照関係）**: 任意のノード cur に対して、cur->prev->next == cur かつ cur->next->prev == cur が常に成立しなければなりません。

よくある落とし穴

  • **パフォーマンスへの影響**: dlist_check はリスト全体を走査するため、計算量が $O(n)$ となります。本番環境などパフォーマンスが重視される環境で有効化すると、著しい性能低下を招く可能性があります（そのため、通常は ILIST_DEBUG 有効時のみ動作するように設計されています）。
  • **無限ループの危険性**: リストの循環構造が破損し、終端（head->head）に到達できないような不正なポインタのループが存在する場合、検証処理自体が無限ループに陥る可能性があります。
