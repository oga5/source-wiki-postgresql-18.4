dsm_segment

概念

dsm_segment は、PostgreSQL において実行時に動的に確保・解放可能な共有メモリ領域である**動的共有メモリ（Dynamic Shared Memory: DSM）セグメント**の状態を管理するための構造体です。

本構造体は、ヘッダーファイル（src/include/storage/dsm.h）においては内部メンバが非公開の不透明な（opaque）構造体として定義されており、実装ファイル（src/backend/storage/ipc/dsm.c）において具体的なメンバが定義されています。各バックエンドプロセスが現在接続（アタッチ）しているセグメントのローカルな状態（マッピング情報、リソース所有権、デタッチ時のコールバックなど）を追跡する役割を担います。

なぜ重要か

並列クエリ、バックグラウンドワーカー、パラレルインデックス作成（B-Tree, BRIN, GIN）、パラレル VACUUM などの並列処理において、プロセス間でデータを共有するためのメモリ領域を動的に確保・管理するために不可欠です。

また、トランザクション終了時やプロセス異常終了時におけるメモリリークを防ぐため、ResourceOwner と連携した自動クリーンアップやデタッチ処理のライフサイクル管理を担う中心的なデータ構造となっています。

主要エンティティ

  • **symbol:src/include/storage/dsm.h#dsm_segment**

    • 外部モジュール向けに公開される不透明な構造体定義（typedef struct dsm_segment dsm_segment;）です。

  • **symbol:src/backend/storage/ipc/dsm.c#dsm_segment**

    • 構造体の具体的な定義です。以下の主要なメンバを含みます。

  • node (dlist_node): バックエンドローカルなセグメントリスト（dsm_segment_list）に登録するためのリンク。
  • resowner (ResourceOwner): リソース所有者。トランザクション終了時の自動デタッチに使用されます。
  • handle (dsm_handle): セグメントを一意に識別する識別子。
  • control_slot: 制御セグメント内のスロット番号。
  • mapped_address: マッピング済みのアドレス（未マッピング時は NULL）。
  • mapped_size: マッピングサイズ。
  • on_detach (slist_head): デタッチ時に実行されるコールバックのリスト。
  • impl_private: OS 依存の低レベル実装（dsm_impl.c）が使用するプライベートデータ。

エンドツーエンドの流れ

1. **初期化**: DSM を利用する前提として、dsm_shmem_init() により共有メモリ制御領域の初期化が完了している必要があります。 2. **作成またはアタッチ**:

  • 新規作成時は dsm_create()、既存セグメントへの接続時は dsm_attach() を呼び出すことで dsm_segment が生成・初期化されます。
  • 生成されたセグメントは、バックエンドローカルなリストに登録され、ResourceOwner に関連付けられます。

    • 3. **利用**:

  • dsm_segment_address() を介して、マッピングされた共有メモリ領域のアドレス（mapped_address）を取得し、並列ワーカー間などでデータ共有を行います。
  • dsm_pin_mapping() を使用することで、リソースオーナーの寿命を超えてセッション終了までマッピングを維持（ピン留め）することも可能です。

    • 4. **デタッチと解放**:

  • dsm_detach() が呼び出される（またはトランザクション終了時に ResourceOwner により自動トリガーされる）と、on_detach リストに登録されたコールバックが実行されます。
  • その後、参照カウントが減少され、必要に応じてメモリ領域が解放されます。プロセス終了時には dsm_detach_all() を通じて自動的にクリーンアップされます。

*※なお、OS依存の低レベル実装（dsm_impl.c）との具体的なやり取りや、制御セグメント（dsm_control_header / dsm_control_item）との詳細な同期アルゴリズムの完全なエンドツーエンドのフローは、提供された情報からは一部未解明です。*

重要な不変条件

  • **マッピングの有効性**: mapped_address が NULL でない場合、そのアドレスは mapped_size バイトの有効な共有メモリ領域を指していなければなりません。
  • **リソース所有権の紐付け**: resowner は、トランザクション終了時に自動的にデタッチ処理をトリガーするために正しく設定されている必要があります。
  • **初期化前提**: DSM を利用する前に、dsm_shmem_init() による共有メモリ制御領域の初期化が完了していなければなりません。

よくある落とし穴

  • **直接アクセスの禁止**: dsm_segment は dsm.h において不透明な構造体として定義されているため、外部モジュールから直接メンバ（mapped_address など）にアクセスしてはなりません。必ず dsm_segment_address() などの提供された API を介する必要があります。
  • **NULLポインタアクセス**: mapped_address が NULL の状態でアクセスを試みると、セグメントが正しくマッピングされていないことを意味し、エラーやクラッシュの原因となります。
  • **メモリリーク**: 異常終了時やクリーンアップ時に ResourceOwner による自動解放が適切に行われない場合、セグメントがリークする可能性があります（ただし、ポストマスター起動時のクリーンアップで復旧します）。
  • **低レベル領域の操作**: impl_private は OS 依存の低レベル実装が使用する領域であるため、直接操作してはなりません。

次に読む場所

  • **src/include/storage/dsm.h**: 外部公開されている API（dsm_create, dsm_attach, dsm_detach, dsm_segment_address など）の宣言。
  • **src/backend/storage/ipc/dsm.c**: dsm_segment の具体的な構造体定義およびライフサイクル管理の実装。
