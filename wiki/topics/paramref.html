ParamRef

概念

ParamRef は、SQL文中に明示的に記述されたパラメータ参照（例: $1, $2）を、パースツリー（構文木）内で表現するためのノード構造体です。

なぜ重要か

SQLの解析初期段階（生のパースツリー）において、パラメータの位置やインデックス番号を保持し、後続の意味解析フェーズで適切な型情報や実行可能な Param ノードへと安全に変換・解決するための基礎となるためです。

エンドツーエンドの流れ

1. **構文解析**: パーサーがSQL文をスキャンし、$n 形式のトークンを検出すると、ParamRef ノードを生成します。 2. **意味解析**: 解析器（parse_analyze）の過程で、src/backend/parser/parse_expr.c の transformParamRef 関数が呼び出されます。 3. **パラメータ解決**: ParseState に設定された ParseParamRefHook（sql_fn_param_ref や paramlist_param_ref、fixed_paramref_hook、variable_paramref_hook など）を介して、パラメータの型や値の解決ロジックが実行されます。 4. **ノード変換**: 最終的に、実行可能な Param ノードやその他の式ノードに変換され、クエリツリーに組み込まれます。

重要な不変条件

  • **パラメータ番号**: number フィールドは、1から始まるパラメータのインデックスを保持します。
  • **位置情報**: location フィールドはソーステキスト内のバイトオフセットを指し、不明な場合は -1 となります。
  • **生存期間**: ParamRef は「生のパースツリー」にのみ存在します。解析後のクエリツリーでは Param ノード等に置き換わるため、プランナーやエグゼキューターが直接 ParamRef を扱うことはありません。

よくある落とし穴

  • **ノードの混同**: 構文木上の ParamRef と、実行可能なクエリ木上の Param ノードは別物です。プランナーやエグゼキューターのフェーズで ParamRef を直接操作しようとしないよう注意が必要です。
  • **解決エラー**: パラメータ番号が範囲外である場合、必要なコンテキスト（ParamListInfo 等）が不足している場合、または ParseParamRefHook が未設定の状態でパラメータが参照された場合は、解析エラーが発生します。

次に読む場所

  • src/include/nodes/parsenodes.h: ParamRef 構造体の詳細な定義。
  • src/include/parser/parse_node.h: ParseState や ParseParamRefHook などの解析コンテキスト定義。
  • src/backend/parser/parse_expr.c: transformParamRef による変換処理の実装。
  • src/backend/parser/parse_param.c: fixed_paramref_hook や variable_paramref_hook などのパラメータ解決フックの実装。
