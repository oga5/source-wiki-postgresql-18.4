XLogRecPtr

概念

XLogRecPtr は、PostgreSQLのWAL（Write-Ahead Log）ストリーム内における特定の物理的・論理的なレコード位置を指し示すためのデータ型です。一般に **LSN（Log Sequence Number）** と呼ばれ、src/include/access/xlogdefs.h において uint64（64ビットの符号なし整数値）として定義されています。WALストリーム上の論理的なオフセットを表現するために使用されます。

なぜ重要か

XLogRecPtr は、トランザクションの永続性と一貫性を保証するためのコア機能において、進捗や位置を特定・追跡するための共通の識別子として機能します。WALの書き込み、読み込み、リカバリ、レプリケーション、および論理デコーディングなど、多くのモジュール間で位置情報を正確に共有・比較するために極めて重要です。また、データページが最後に更新された時点を識別するページヘッダ内のLSNとしても使用されます。

エンドツーエンドの流れ

WALの生成からレプリケーション、デコーディングに至る完全なエンドツーエンドの全体フローは、提供されたソースコードの抜粋範囲からはすべてが明らかになっておらず、未確定な部分があります。現時点で判明している部分的な流れは以下の通りです。

1. **WALの生成と位置特定**: トランザクションの変更に伴いWALレコードが生成され、一意の XLogRecPtr が割り当てられます。 2. **WALの受信と永続化（スタンバイ側）**: walreceiver プロセスがプライマリからWALデータを受信し、LogstreamResult 構造体の Write（書き込み位置）および Flush（フラッシュ位置）を更新しながらディスクに書き込み・同期（fsync）を行います。 3. **WALの読み込みと先読み**: リカバリプロセスや解析ツールは XLogReaderState を用いてWALを順次読み込みます。この際、xlogprefetcher.c の LsnReadQueue を介して、必要なデータブロックの先読みI/Oが制御されます。 4. **論理デコーディングと出力**: ReorderBuffer が XLogRecPtr（コミットLSNなど）を基準にトランザクションの変更を整列し、出力プラグインのコールバック（LogicalDecodeCommitCB など）へ位置情報を引き渡します。

重要な不変条件

  • **単調増加**: WALストリームの進行に伴い、生成される XLogRecPtr の値は単調増加します。
  • **書き込みとフラッシュの順序関係**: walreceiver.c の LogstreamResult において、ディスクにフラッシュされた位置は書き込まれた位置を超えてはなりません（Flush <= Write）。
  • **無効値の定義**: 値 0 は InvalidXLogRecPtr として定義され、未初期化または無効なWAL位置を示す不変の基準値として扱われます。

よくある落とし穴

  • **無効値（InvalidXLogRecPtr）の誤用**: 値が 0 である InvalidXLogRecPtr を有効なWAL位置として処理しようとすると、WALリーダーの読み取り失敗や予期せぬ動作を引き起こす原因になります。
  • **WALの切り詰め（ローテーション）**: 参照している XLogRecPtr が指す古いWALセグメントが、チェックポイントなどの処理によって既に削除（切り詰め）されている場合、その位置へのアクセスはエラーとなります。
  • **フォーマット出力時の注意**: printf などの関数で XLogRecPtr を出力する際は、直接整数値として出力するのではなく、必ず LSN_FORMAT_ARGS マクロを使用して "%X/%X" 形式でフォーマットする必要があります。
