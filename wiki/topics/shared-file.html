Shared File

概念

Shared File（共有ファイル）は、PostgreSQLにおいて複数のバックエンドプロセス間で共有される一時ファイル群を管理するための仕組みです。プロセス間での協調的なアクセスを可能にし、不要になった一時ファイルの自動的なライフサイクル管理（クリーンアップ）を提供します。

なぜ重要か

並列クエリ（並列ハッシュジョインや並列ソートなど）を実行する際、複数のバックエンドプロセスが協調して一時データを読み書きする必要があります。Shared File は、これらのプロセス間で一時ファイルを安全に共有し、すべてのプロセスが処理を終えた（またはデタッチした）段階で、関連する一時ファイルを自動的かつ確実に物理削除するために不可欠な役割を担っています。

主要エンティティ

  • **struct:src/include/storage/sharedfileset.h#SharedFileSet**: 共有ファイルセットを管理する中心的な構造体。内部にローカルな一時ファイル管理用の FileSet、スピンロック（slock_t mutex）、および参照カウンタ（int refcnt）を保持します。
  • **symbol:src/backend/storage/file/sharedfileset.c#SharedFileSetInit**: 共有ファイルセットの初期化を行う関数。参照カウンタの初期化や、DSM（動的共有メモリ）デタッチ時の自動クリーンアップ処理の登録を行います。

エンドツーエンドの流れ

提供された情報に基づく、共有ファイルセットのライフサイクルおよび処理の流れは以下の通りです。

1. **初期化**: src/backend/executor/nodeHashjoin.c や src/backend/utils/sort/tuplesort.c などの呼び出し元が SharedFileSetInit を呼び出します。これにより、SharedFileSet 内の mutex がスピンロックとして初期化され、参照カウンタ（refcnt）が 1 に設定されます。また、DSMセグメント（dsm_segment）が指定されている場合は、デタッチ時のコールバックとして SharedFileSetOnDetach が登録されます。 2. **アタッチ**: 他のバックエンドプロセスがアタッチする際、スピンロック（mutex）による保護のもとで refcnt がインクリメントされます。 3. **利用**: src/backend/storage/file/buffile.c や src/backend/utils/sort/logtape.c などを通じて、共有一時ファイルへのアクセスが実行されます。 4. **デタッチとクリーンアップ**: プロセス終了時やエラー発生時などに SharedFileSetOnDetach が呼び出され、refcnt がデクリメントされます。refcnt が 0 に達した時点で FileSetDeleteAll が呼び出され、関連するすべての一時ファイルが物理的に削除されます。

*注意: SharedFileSetAttach や SharedFileSetOnDetach、FileSetDeleteAll の具体的なシグネチャや、これらを含む並列処理全体の詳細なエンドツーエンドフローは、提供された情報からは完全には明らかになっていません。*

重要な不変条件

  • **参照カウンタの保護**: SharedFileSet 内の refcnt に対する操作は、必ず mutex（スピンロック）によって保護されていなければなりません。
  • **共有メモリへの配置**: 複数プロセスからアクセスされるため、SharedFileSet 構造体は必ず共有メモリ（DSM）セグメント上に配置される必要があります。
  • **初期参照カウント**: SharedFileSetInit による初期化直後の refcnt は必ず 1 となります。
  • **自動削除のトリガー**: 参照カウント refcnt が 0 に達したとき、保持しているすべての一時ファイルが物理的に削除されます。

よくある落とし穴

  • **自動クリーンアップの欠落**:

    • SharedFileSetInit の呼び出し時に dsm_segment（seg）に NULL を指定した場合、DSMデタッチ時の自動クリーンアップコールバックが登録されません。この場合、呼び出し元が手動でクリーンアップを管理する必要があり、管理を怠ると一時ファイルが残留する原因になります。

  • **エラーハンドリング時の再入**:

    • SharedFileSetOnDetach はエラー発生時のクリーンアップパスからも呼び出されるため、このコールバック内でさらにエラーを発生させないように堅牢に実装する必要があります。

  • **参照カウントの不整合**:

    • スピンロックの競合やデタッチ処理の欠落が発生すると、参照カウント（refcnt）に不整合が生じ、ファイルが早期に削除されたり、逆に削除されずに残ったりする可能性があります。また、既に破棄された SharedFileSet に対してアタッチを試みた場合はエラーが発生します。
