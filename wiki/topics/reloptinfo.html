RelOptInfo

概念

RelOptInfo は、PostgreSQL のクエリプランナ（最適化エンジン）において、ベースリレーション（テーブルなど）、結合リレーション、あるいはサブクエリや外部テーブルなどのリレーションに関する統計情報、物理的特性、コスト見積もり、および生成されたアクセスパス（Path）の候補を保持するための中心的なデータ構造です。

なぜ重要か

RelOptInfo は、プランナが最適な実行計画を決定するための基盤として機能します。

  • **情報のキャッシュと抽象化**: システムカタログ（pg_class, pg_index, pg_statistic など）から取得した物理属性や統計情報をキャッシュし、コスト計算やパス生成の入力として再利用できるようにします。
  • **結合探索のサポート**: 結合探索（動的計画法やGEQO）の過程で、結合リレーション用の RelOptInfo が動的に構築され、最終的に最も低コストなパス（cheapest_total_path など）を選択するために使用されます。
  • **拡張性の提供**: 外部データラッパー（FDW）やパーティションテーブルなどの特殊なリレーションに対しても、fdw_private などのフィールドを介して固有の最適化情報を保持・伝達する役割を担います。

エンドツーエンドの流れ

提供された情報から推論される大まかな処理の流れは以下の通りですが、プランニング全体の厳密な遷移やコールバックの呼び出し順序などの完全なエンドツーエンドのフローは、現時点ではすべて明らかになっていません。

1. **初期化と情報収集**: プランナがクエリ最適化を開始すると、get_relation_info（src/backend/optimizer/util/plancat.c）が呼び出され、システムカタログから物理属性（ページ数、タプル数、列幅）、インデックス情報、統計情報、制約情報を収集して RelOptInfo を構築・初期化します。 2. **パス生成とコスト見積もり**: 収集されたデータを基に、スキャンや結合のアルゴリズムを表す Path 構造体が生成され、RelOptInfo の pathlist に格納されます。 3. **外部テーブルの最適化（FDW）**: 対象が外部テーブルの場合、FDW は estimate_size や estimate_costs などの関数を通じて RelOptInfo を参照・操作し、fdw_private に独自の最適化情報を格納しながら、外部ソースに対する効率的なアクセス計画を構築します。 4. **最適パスの選択**: 最終的に cheapest_total_path などのフィールドから最適なパスが選択され、実行計画が作成されます。

重要な不変条件

  • **コピーの制限**: RelOptInfo と Path の間には複雑な相互リンクが存在するため、copyObject() などの汎用関数によるコピーはサポートされていません。
  • **リレーション識別**: relids フィールドは Bitmapset であり、対象となるリレーションの集合を表現します。ベースリレーションの場合は relid や indexlist などの固有情報が設定されますが、結合リレーションではこれらは設定されません。
  • **パーティション数の未設定状態**: nparts が -1 の場合は、パーティション数が未設定であることを示します。
  • **FDW専用フィールドの限定利用**: fdw_private や fdwroutine は、対象リレーションが外部テーブルでない場合は使用されません。

よくある落とし穴

  • **型キャストの危険性**: FDW 開発において fdw_private フィールドに独自構造体のデータを格納・参照する際、適切な型キャストが行われないと、メモリ破壊や不正なデータ参照を引き起こす可能性があります。
  • **統計情報の欠如**: 外部テーブルなどの統計情報が欠如している場合、RelOptInfo に基づくコスト見積もりが不正確になり、非効率なプランが選択される原因となります。
  • **構造体変更の影響**: RelOptInfo はプランナの再帰的な処理やコスト見積もりロジックで頻繁に参照されるため、構造体を変更する際はプランナ全体への影響を慎重に評価する必要があります。
  • **パーティションの肥大化**: パーティション分割されたリレーションにおいて、part_rels 配列は境界情報と対応しますが、パーティション数が極めて多い場合にメモリや処理負荷が非常に肥大化する可能性があります。
