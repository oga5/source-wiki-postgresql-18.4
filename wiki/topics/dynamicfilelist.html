DynamicFileList

概念

DynamicFileList は、PostgreSQL プロセスにおいてロード済みの動的ライブラリ（共有オブジェクト）を追跡・管理するための単方向連結リスト構造体です。

なぜ重要か

動的ライブラリの重複ロードを防止し、ロード済みモジュールの走査を可能にするために重要です。また、ロードされたライブラリの ABI 互換性を検証するためのマジックブロック（Pg_magic_struct）や、プラットフォームごとのファイル一意性識別情報（デバイス番号や i-node）を保持する役割を担っています。

エンドツーエンドの流れ

1. **ライブラリのロード要求**: internal_load_library 等が呼び出され、動的ライブラリのロードが試みられます。 2. **重複チェック**: 既存の file_list を走査し、同一の device および inode（Windows 以外の場合）を持つエントリが既に存在するか確認することで、重複ロードを防止します。 3. **メモリ確保と登録**: 重複がない場合、malloc を使用して offsetof(DynamicFileList, filename) + strlen(libname) + 1 のサイズでメモリを確保し、新規エントリを作成して file_list / file_tail に追加します。 4. **走査**: get_next_loaded_module 等を通じて、ロード済みのライブラリリストが走査されます。

※ 外部モジュール（src/backend/commands/extension.c など）から internal_load_library が呼び出される際の上流の完全な制御フローや、エラーハンドリング時の詳細なライフサイクルは、提供された情報からは完全には特定されていません。

重要な不変条件

  • filename は FLEXIBLE_ARRAY_MEMBER を使用した可変長配列であり、必ず構造体の末尾に配置されます。
  • Windows（WIN32）環境以外では、ファイルの一意性を判定するために device と inode が使用されます。
  • 一度ロードされたライブラリは、現在の設計上アンロードされません。

よくある落とし穴

  • **構造体のコピー禁止**: filename が可変長配列であるため、構造体全体の単純なコピーやスタックへの割り当てを行うと、メモリ破壊を引き起こす原因になります。
  • **プラットフォーム間の差異**: Windows 環境では inode フィールドが存在しないため、プラットフォーム固有の条件分岐（#ifndef WIN32）を意識する必要があります。
  • **ABI 不整合**: magic フィールドが指す情報が PG_MODULE_MAGIC と一致しない場合、ロードが拒否されます。
