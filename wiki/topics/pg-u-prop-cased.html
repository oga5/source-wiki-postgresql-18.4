pg_u_prop_cased

概念

pg_u_prop_cased は、指定された Unicode コードポイント（pg_wchar）が「Cased（大文字・小文字の区別を持つ）」プロパティを保持しているかどうかを判定する機能です。

なぜ重要か

Unicode 文字列のケースマッピングや大文字・小文字の処理において、対象の文字がケース属性を持っているかを正確かつ効率的に判定するために重要です。ASCII 範囲（0x80 未満）に対する高速なテーブル参照と、それ以上のコードポイントに対する Unicode カテゴリ判定を組み合わせることで、パフォーマンスと正確性を両立させています。

関与するモジュール

  • module:src/common
  • module:src/include/common

主要エンティティ

  • **symbol:src/common/unicode_category.c#pg_u_prop_cased**: Unicode コードポイントが Cased プロパティを持つかを判定する関数。
  • **symbol:src/include/common/unicode_category_table.h#PG_U_PROP_CASED**: ASCII 範囲の高速判定用テーブル（unicode_opt_ascii）などで使用される、Cased プロパティを示すビットマスク定数。

エンドツーエンドの流れ

1. **入力の受け取り**: 呼び出し元（src/common/unicode_case.c やテストコードなど）から pg_wchar code を受け取ります。 2. **ASCII 最適化パス**:

  • code < 0x80 の場合、静的テーブル unicode_opt_ascii を直接参照します。
  • PG_U_PROP_CASED ビットが立っているかを定数時間で判定し、結果を返します。

    • 3. **一般 Unicode パス**:

  • code >= 0x80 の場合、unicode_category(code) を呼び出してカテゴリを取得し、PG_U_LT_MASK（Letter, Titlecase）と一致するか判定します。
  • さらに、pg_u_prop_lowercase(code) または pg_u_prop_uppercase(code) を呼び出し、いずれかが真であれば true を返します。

※ SQL レベルの関数や上位のテキスト処理機能からこの判定処理に至るまでの全体的なエンドツーエンドの呼び出しフローは、提供された情報からは未詳です。

重要な不変条件

  • **入力型とエンコーディング**: 入力コードポイントは、UTF-8 エンコーディングを前提とした pg_wchar 型である必要があります。
  • **ASCII テーブルの整合性**: 0x80 未満のコードポイントに対する unicode_opt_ascii テーブルの PG_U_PROP_CASED ビット定義は、実際の Unicode 規格のプロパティと一致していなければなりません。

※ システム全体におけるその他の不変条件は、提供された情報からは未詳です。

よくある落とし穴

  • **処理コストの非対称性**: 0x80 未満の文字は定数時間で高速に判定されますが、0x80 以上の文字は内部で二分探索（range_search）を伴う unicode_category() や複数のプロパティ判定関数を呼び出すため、処理コストが異なります。
  • **テーブル生成時の整合性**: 判定ロジックやカテゴリ定義を変更する場合、unicode_category_table.h を生成するスクリプトおよび関連するプロパティ判定関数（pg_u_prop_lowercase、pg_u_prop_uppercase）との整合性を慎重に維持する必要があります。
