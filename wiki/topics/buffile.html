BufFile

概念

BufFile は、OSのファイルサイズ制限や単一ファイル記述子の制約を回避するために、複数の物理ファイル（セグメント）から構成される論理的に単一の巨大なバッファ付きファイルを表現するデータ構造およびAPIです。一時的なデータ保持（ソート、ハッシュ結合、GiSTインデックス構築など）のために、効率的なバッファリング付きI/Oインターフェースを提供します。

なぜ重要か

PostgreSQLでは、クエリ実行中にメモリ（work_mem など）に収まらない大量のデータを一時的にディスクに退避させる必要があります。BufFile は、以下の理由から重要です： 1. **OS制限の回避**: 物理ファイルを自動的に分割（デフォルトで1GBごと）することで、OSのファイルサイズ制限を回避します。 2. **I/O効率の向上**: PGAlignedBlock を用いた内部バッファリングにより、物理的なI/O回数を削減します。 3. **リソース管理の統合**: ResourceOwner や FileSet と連携し、トランザクション終了時の自動クリーンアップや並列クエリ間での一時ファイル共有を安全に行うことができます。

エンドツーエンドの流れ

提供されたソースコード情報からは、BufFile の作成（BufFileCreateTemp など）から読み書き、そしてクローズに至る完全なエンドツーエンドの制御フローの全容は完全には記述されていません。 ただし、提供された情報から以下の部分的な流れが判明しています：

1. **初期化**: BufFileCreateTemp などの関数により、palloc を介して BufFile 構造体がメモリ上に確保・初期化されます。 2. **書き込みとセグメント分割**: BufFileWrite を介してデータが書き込まれます。物理ファイルサイズが MAX_PHYSICAL_FILESIZE（1GB）に達すると、自動的に新しい物理ファイルセグメントが作成・オープンされ、論理的なストリームが継続されます。 3. **読み込み**: BufFileRead を介して、内部バッファ（PGAlignedBlock）を利用しながら効率的にデータが読み込まれます。 4. **クローズと解放**: BufFileClose が呼び出されると、まず BufFileFlush により未書き込みデータが物理ファイルにフラッシュされ、その後、各物理ファイルが FileClose によりクローズされ、最後に BufFile 構造体とファイル配列が pfree により解放されます。

重要な不変条件

  • **セグメントサイズの一貫性**: 最後の物理ファイルを除くすべての物理ファイルセグメントは、正確に MAX_PHYSICAL_FILESIZE（1GB）の長さを持つ必要があります。
  • **メモリコンテキストへの依存**: BufFile 構造体は palloc によって割り当てられるため、その寿命は割り当てられたメモリコンテキストに依存します。
  • **リソース管理**: 一時ファイルのリソース管理を適切に行うため、resowner（ResourceOwner）が適切に設定されている必要があります。

よくある落とし穴

  • **リソースリーク**: BufFileClose を明示的に呼び出さない場合、仮想ファイル記述子がリークする可能性があります。トランザクション終了時やエラー発生時に確実にクローズされるよう、ResourceOwner による管理が必要です。
  • **並列実行時の注意**: 並列クエリ環境で一時ファイルを共有する場合は、直接操作するのではなく、必ず FileSet を介した API を使用する必要があります。
  • **ファイル結合時の「穴」**: BufFileAppend は物理セグメントのリストを連結するだけでデータコピーを行わないため、ファイル間に「穴」が生じる可能性がある点に注意が必要です。
  • **型変換**: BufFileSeek などのシーク操作では、論理的なオフセットを int と off_t で管理しているため、巨大なファイルを操作する際の型変換に留意する必要があります。
