wait_on_slots

概念

wait_on_slots は、PostgreSQL のクライアントツールおよびユーティリティにおいて、複数のデータベース接続（スロット）を非同期に監視し、I/O 準備が整った接続を特定して処理を進めるための内部待機ループ関数です。

この関数は、役割や対象とするデータ構造が異なる2つのモジュールでそれぞれ同名の static 関数として実装されています。 1. **pg_upgrade のタスク管理用** (src/bin/pg_upgrade/task.c) 2. **フロントエンド共通ユーティリティの並列スロット管理用** (src/fe_utils/parallel_slot.c)

なぜ重要か

複数のデータベース接続を並行して処理する際、ブロッキング I/O による遅延を避け、効率的に処理を進行させる必要があります。wait_on_slots は、select()（または内部の select_loop）を用いた I/O 多重化により、準備が整ったソケットからのみデータを読み書き・処理することで、シングルスレッド環境下でも効率的な非同期・並行処理を実現するために重要な役割を果たしています。

関与するモジュール

  • module:src/bin/pg_upgrade
  • module:src/fe_utils

主要エンティティ

  • **symbol:src/bin/pg_upgrade/task.c#wait_on_slots**

    • UpgradeTaskSlot 配列を監視し、準備が整ったスロットの ready フラグを true に更新する静的関数。

  • **symbol:src/fe_utils/parallel_slot.c#wait_on_slots**

    • ParallelSlotArray を監視し、PQconsumeInput や PQgetResult を用いて結果を消費・処理する静的関数。

エンドツーエンドの流れ

2つの実装で処理の流れが異なります。

1. src/bin/pg_upgrade/task.c の流れ

1. **初期化**: 全スロットの ready フラグを false に初期化します。 2. **ソケット登録**: FREE 状態以外の各スロットから PQsocket() でソケットを取得し、select_mode に応じて input または output の fd_set に追加します。 3. **待機**: select_loop() を呼び出し、いずれかのソケットが準備完了になるまで待機します。 4. **フラグ更新**: 待機終了後、準備完了したスロットの ready フラグを true に設定します。 *※ 呼び出し元の並列実行ループの全体像や、ready になった後の具体的なタスク処理の全貌（エンドツーエンドの完全な流れ）は、提供されたコンテキストからは特定されていません。*

2. src/fe_utils/parallel_slot.c の流れ

1. **呼び出し**: ParallelSlotsGetIdle などの呼び出し元から、空きスロットを確保する際やビジー状態の監視時に呼び出されます。 2. **ソケット登録と待機**: 全スロットのソケットを fd_set に登録し、select_loop を呼び出して入力を待機します。この際、SetCancelConn / ResetCancelConn を用いてキャンセル処理が統合されます。 3. **入力消費**: select() が入力を検知したスロットに対し、PQconsumeInput を実行します。 4. **結果取得ループ**: PQisBusy が偽になるまで PQgetResult をループし、結果を消費します。 5. **結果処理**: processQueryResult を呼び出して各結果を処理します。 6. **状態遷移**: 結果が NULL になった場合、そのスロットをアイドル状態（inUse = false）に設定し、ハンドラをクリアします。 *※ 呼び出し元に制御が戻った後の、アプリケーション全体における最終的なデータフローの全貌は、提供されたコンテキストからは特定されていません。*

重要な不変条件

src/bin/pg_upgrade/task.c における不変条件

  • slots 配列は常に有効なメモリ領域を指していなければなりません。
  • process_slot() が呼び出されるたびに、前回の wait_on_slots() で ready とマークされたスロットの処理が完了している必要があります。
  • FREE 状態のスロットは、処理すべきデータベースが存在しないものとして無視されます。

src/fe_utils/parallel_slot.c における不変条件

  • sa->slots[i].connection が NULL でない（接続済みである）必要があります。
  • 呼び出し時に、少なくとも1つの有効な接続が存在しなければなりません。
  • シングルスレッド環境での実行を前提としています。

よくある落とし穴

  • **同名関数の混同**: src/bin/pg_upgrade/task.c と src/fe_utils/parallel_slot.c に同名の static 関数が存在するため、コードリーディングやデバッグの際に対象の実装を取り違えないよう注意が必要です。
  • **ファイルディスクリプタ数の制限**: src/fe_utils/parallel_slot.c では select() を使用しているため、ファイルディスクリプタ数が FD_SETSIZE を超えないようにする必要があります（特に Windows 環境で注意が必要です）。
  • **エラーハンドリングの差異**:
  • pg_upgrade 側では、PQsocket() や select_loop() が失敗すると pg_fatal によりプロセス全体が即座に異常終了します。
  • fe_utils 側では、エラー発生時に false を返して呼び出し元にエラーを伝播します。
  • **ハンドラの実装依存**: fe_utils 側では processQueryResult の戻り値が全体の成否を決定するため、登録するハンドラの実装に不備があると、予期せず処理全体が失敗（false を返却）する原因になります。
