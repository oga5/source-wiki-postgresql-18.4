IndexPath

概念

IndexPath は、PostgreSQLのクエリオプティマイザにおいて、インデックススキャン（Index Scan）またはインデックスオンリースキャン（Index-Only Scan）を実行するためのアクセスパスを表現するデータ構造です。

なぜ重要か

クエリオプティマイザが最適な実行プランを選択する際、テーブルへのアクセス方法としてインデックスの利用は極めて重要です。IndexPath は、特定のインデックス、スキャン方向、インデックス条件（indexclauses）などの情報と、それに伴う推定コストを保持することで、プランナが他のアクセスパス（シーケンシャルスキャンなど）とコストを比較検討できるようにするために不可欠な役割を果たします。

関与するモジュール

  • module:src/backend/optimizer/util
  • module:src/include/nodes

主要エンティティ

  • symbol:src/include/nodes/pathnodes.h#IndexPath: インデックススキャンパスの情報を保持する構造体。
  • symbol:src/backend/optimizer/util/pathnode.c#create_index_path: IndexPath ノードを生成し、初期化およびコスト計算を行う関数。

エンドツーエンドの流れ

プラン作成全体における完全なエンドツーエンドの流れは提供された情報からは明らかではありませんが、IndexPath の生成に関しては以下の流れが確認できます。

1. **呼び出し**: src/backend/optimizer/path/indxpath.c や src/backend/optimizer/plan/planner.c などの呼び出し元から、インデックス情報（IndexOptInfo）やインデックス条件（indexclauses）を伴って create_index_path が呼び出されます。 2. **メモリ確保と初期化**: makeNode(IndexPath) によりメモリが確保され、indexonly フラグに応じて pathtype が T_IndexOnlyScan または T_IndexScan に設定されます。 3. **パラメータ化情報の取得**: get_baserel_parampathinfo を呼び出してパラメータ化情報を設定します。 4. **コスト計算**: cost_index を呼び出して、当該インデックスパスの実行コストを算出し、設定します。 5. **返却**: 初期化とコスト計算が完了した IndexPath 構造体へのポインタが呼び出し元に返されます。

重要な不変条件

  • **並列実行の初期値**: create_index_path 内で生成される IndexPath において、pathnode->path.parallel_aware は常に false に初期化され、parallel_workers は 0 に固定されます。
  • **並列安全性の継承**: pathnode->path.parallel_safe は、対象リレーションの並列考慮フラグ（index->rel->consider_parallel）に基づいて設定されます。
  • **有効なインデックス情報**: 引数として渡される index（IndexOptInfo）は、有効なポインタである必要があります。

よくある落とし穴

  • **NULLポインタの危険性**: create_index_path の引数 index が NULL の場合、内部で index->rel へのアクセスが発生するため、クラッシュを引き起こします。
  • **並列ワーカー数の固定**: create_index_path の内部では parallel_workers が 0 に固定されているため、並列実行を考慮した拡張を行う際には、この初期化処理の挙動に注意する必要があります。
  • **コスト計算の依存**: コスト計算ロジック自体は create_index_path 内に直接記述されておらず、cost_index に依存しているため、コスト計算の挙動を変更する場合は cost_index 側を参照・修正する必要があります。
