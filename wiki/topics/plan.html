Plan

概念

Plan は、PostgreSQLにおけるクエリ実行計画ツリーを構成するすべてのノードの基底となる抽象構造体（およびその型定義）です。プランナによって生成され、エグゼキュータによって消費される実行手順の「設計図」として機能します。C言語における構造体の継承を模倣しており、すべてのプランノード構造体の先頭メンバとして配置されます。

なぜ重要か

Plan は、クエリ最適化エンジン（プランナ）と実行エンジン（エグゼキュータ）を繋ぐ中核的なデータ構造です。また、外部データラッパー（FDW）などの拡張モジュールにおいても、外部テーブルスキャン用の計画ノード（ForeignScan など）の構成要素やサブプランのルートノードとして広く利用されます。共通の基底構造体としてコスト情報、推定サイズ、並列実行フラグ、ツリー構造（左右の子ノード）などを一元管理することで、一貫した計画ツリーの構築と実行を可能にしています。

エンドツーエンドの流れ

1. **計画作成（プランナ）**: プランナ（src/backend/optimizer/）がクエリを解析し、Plan 構造体（およびそれを継承した各種計画ノード）を構築して実行計画ツリーを生成します。 2. **FDWとの連携**: FDWの GetForeignPlan コールバック関数において、サブプラン（外部プラン）のルートノードが Plan ポインタ（outer_plan 引数）として渡されます。 3. **実行（エグゼキュータ）**: エグゼキュータ（src/backend/executor/）が Plan 構造体を参照し、ツリーを再帰的に辿りながら各ノードの実行関数を呼び出します。各ノードは子ノード（lefttree, righttree）から供給されるタプルに対し、qual によるフィルタリングと targetlist に基づく射影を適用してタプルを出力します。

重要な不変条件

  • **直接インスタンス化の禁止**: 本構造体は抽象的な基底であり、直接インスタンス化してはなりません（pg_node_attr(abstract, ...) として定義されています）。
  • **一意なノードID**: plan_node_id は、最終的なプランツリー全体で一意である必要があります。
  • **ツリー構造の方向**: lefttree は「外部（outer）」プラン、righttree は「内部（inner）」プランを指します。
  • **NULLポインタの扱い**: lefttree や righttree が NULL である場合、その方向からの入力が存在しないことを示します。

よくある落とし穴

  • **NULLポインタ参照**: Plan ポインタが NULL である場合、それを参照する関数やマクロ（例: outerPlan() など）を呼び出すとセグメンテーション違反が発生する可能性があります。
  • **直接アクセスの回避**: 構造体メンバへ直接アクセスするのではなく、outerPlan() や innerPlan() などの推奨マクロを使用する必要があります。
  • **パラメータ管理の不備**: extParam や allParam の設定が不適切な場合、パラメータ変更に伴う再スキャン（Rescan）が正しく機能しない可能性があります。
  • **FDW拡張時の注意**: FDW固有の拡張を行う際は、Plan 自体を直接変更するのではなく、ForeignScan などの派生構造体を通じて情報を付加する必要があります。
