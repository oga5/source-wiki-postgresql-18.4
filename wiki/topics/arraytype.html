ArrayType

概念

ArrayType は、PostgreSQLにおける配列データの表現や管理に関連する、複数のレイヤーにまたがる概念です。 具体的には、コアシステム（バックエンド）における標準的な「フラット」な可変長配列のバイナリ表現（struct ArrayType）と、埋め込みSQL（ECPG）クライアントライブラリにおいてホスト変数が配列であるかを識別するための列挙型（enum ARRAY_TYPE）という、2つの異なる文脈で定義・使用されています。

なぜ重要か

  • **バックエンドでの効率的なデータ表現**: struct ArrayType は、ディスク上およびメモリ上での配列データの標準的なレイアウトを規定します。これにより、次元数、要素型、NULLビットマップ、および実際の要素データをコンパクトな単一の可変長（varlena）ブロックとして管理できます。
  • **クライアント側（ECPG）での型識別**: enum ARRAY_TYPE は、ECPGランタイムがホスト変数の配列特性（標準配列かベクトルかなど）を動的に解決し、SQL実行時の適切なデータマッピングや境界チェックを行うために不可欠です。

エンドツーエンドの流れ

バックエンドにおける配列データの処理から、ECPGクライアントライブラリを介したホスト変数へのマッピングに至る、完全なエンドツーエンドの具体的な連携フローは、提供された情報からは未だ明らかになっていません。

現時点で判明している部分的な流れは以下の通りです：

  • **バックエンド側**: PG_GETARG_ARRAYTYPE_P マクロ等を通じて detoast された ArrayType 構造体を取得し、ARR_DATA_PTR マクロ等を用いて要素データやNULLビットマップにアクセスします。
  • **ECPG側**: SQL実行時や型情報キャッシュ（ECPGtype_information_cache）の構築時に、struct variable の isarray メンバ（enum ARRAY_TYPE 型）を参照して、配列の区切り文字判定や境界チェックなどの制御ロジックを動的に切り替えます。

重要な不変条件

  • **メモリ配置とアライメント**: struct ArrayType の配列データは、必ず MAXALIGN 境界で開始される必要があります。
  • **NULLビットマップの有無**: dataoffset が 0 の場合はNULLビットマップが存在せず、0 以外の値の場合はタプルと同様の形式（LSBファースト）でNULLビットマップが格納されます。
  • **ECPGの初期状態**: ECPG_ARRAY_NOT_SET は、初期化状態や型情報が未確定であることを示すデフォルト値として維持されます。

よくある落とし穴

  • **直接的なヘッダー操作**: struct ArrayType の vl_len_ フィールド（varlenaヘッダー）を直接操作することは避ける必要があります。
  • **メモリ破壊のリスク**: dataoffset の設定値が不適切な場合、NULLビットマップの読み取り時にメモリ破壊が発生する可能性があります。また、要素型（elemtype）が正しく設定されていないと、後続のデータアクセスで型不整合が発生します。
  • **TOASTの扱い**: 配列要素を個別に TOAST することは避けるべきであり、配列全体は detoast された状態で格納される必要があります。
