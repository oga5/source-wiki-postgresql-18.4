List

概念

List は、PostgreSQL全体で広く使用される汎用的な連結リストおよび拡張可能配列（expansible array）のデータ構造です。src/include/nodes/pg_list.h において struct List として定義されており、ノードタグ（NodeTag）、要素数（length）、最大容量（max_length）、および要素の配列（ListCell *elements）から構成されます。クエリツリー、プランツリー、演算子、関数、オプション、属性情報など、多様なノードやオブジェクトの集合を保持・管理するための標準的なコンテナとして機能します。

なぜ重要か

PostgreSQLのほぼすべての主要サブシステム（パーサ、プランナ、エグゼキュータ、インデックスアクセスメソッド、FDW、キャッシュ管理など）において、動的な要素の集合をやり取りするための標準的なインターフェースとして機能しているためです。

  • **エグゼキュータ (executor.h)**: 範囲テーブルやパーミッション情報の管理、フック関数での権限チェック対象の走査に使用されます。
  • **インデックスAM (amapi.h)**: 演算子や関数のリストを調整・検証するために使用されます。
  • **FDW (fdwapi.h)**: 外部データラッパー固有の計画情報（fdw_private）やオプションの保持に使用されます。
  • **プランナ (paths.h)**: リレーションやパスの管理に使用されます。
  • **計画キャッシュ (plancache.h)**: 書き換え後のクエリツリーリストの管理に使用されます。

エンドツーエンドの流れ

提供されたソースコードの抜粋からは、システム全体を横断する具体的なエンドツーエンドの制御フロー（例：クエリ入力から実行完了までのリストの完全な変遷）の詳細は完全には明らかになっていません。

ただし、部分的な流れとして、クエリの解析・書き換えプロセスで生成されたクエリツリーリスト（plancache.h の PostRewriteHook など）が、プランナ（paths.h）を経て、エグゼキュータ（executor.h）に渡され、各段階で List 構造体が情報の受け渡しや走査に利用されることが示されています。また、FDW（fdwapi.h）やインデックスAM（amapi.h）などの拡張インターフェースに対しても、必要な情報が List の形で引き渡されます。

重要な不変条件

  • **空リストの表現**: 空のリストは常に NIL（NULL ポインタ）として表現されます。length == 0 である List 構造体インスタンス自体は存在しません。
  • **メモリ配置の安定性**: リストが空でない限り、ヘッダと要素配列のメモリ位置は再配置されないことが保証されています。
  • **メモリ管理**: PostgreSQLのメモリコンテキスト（MemoryContext）に依存して管理され、メモリ不足時にはリスト操作関数がエラーを発生させます。

よくある落とし穴

  • **ループ中の要素削除**: foreach ループでリストを走査している最中に、通常の削除関数を使用して要素を削除すると、イテレータの整合性が崩れます。走査中の削除には必ず foreach_delete_current マクロを使用する必要があります。
  • **型安全性**: リスト要素（ListCell）からノードを取り出す際、単純な lfirst ではなく、型安全性を確保するために lfirst_node(type, lc) を使用することが推奨されます。
  • **空リスト（NIL）のハンドリング**: List * が NIL（NULL）である可能性を考慮せずにアクセスすると、セグメンテーション違反を引き起こす可能性があります。
