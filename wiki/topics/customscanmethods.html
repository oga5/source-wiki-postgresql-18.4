CustomScanMethods

概念

CustomScanMethods は、PostgreSQLの拡張機能が独自のカスタムスキャン（CustomScan）を定義し、実行時にその実行状態（CustomScanState）を生成するためのインターフェースを提供する構造体です。これにより、コアシステムを変更することなく、独自のデータスキャンロジックをプランナーおよびエグゼキュータに統合できます。

なぜ重要か

PostgreSQLのオプティマイザやエグゼキュータは、標準のテーブルスキャンやインデックススキャン以外の特殊なデータアクセス方法（外部データソースへのアクセスや、ハードウェアアクセラレーションを利用したスキャンなど）を直接サポートしていません。CustomScanMethods を使用することで、拡張モジュールは独自のカスタムスキャン計画ノードから実行状態ノードを生成するコールバック（CreateCustomScanState）を定義し、コアシステムに登録・統合することが可能になります。

関与するモジュール

  • module:src/backend/nodes
  • module:src/include/nodes

主要エンティティ

  • **struct CustomScanMethods** (src/include/nodes/extensible.h): カスタムスキャンの名前（CustomName）と、実行状態を生成する関数ポインタ（CreateCustomScanState）を保持する構造体。
  • **GetCustomScanMethods** (src/backend/nodes/extensible.c): 指定されたカスタムスキャン名をキーとして、登録済みの CustomScanMethods を検索・取得する関数。
  • **custom_scan_methods** (src/backend/nodes/extensible.c): 登録された CustomScanMethods を管理するための内部ハッシュテーブル。

エンドツーエンドの流れ

1. **登録**: 拡張モジュールは、CustomScanMethods 構造体を定義し、コアシステムに登録します（登録処理の具体的な関数は提供された情報からは未詳）。これにより、内部ハッシュテーブル custom_scan_methods にメソッドが追加されます。 2. **計画・構築**: プランナーがカスタムスキャンを計画に組み込みます。 3. **検索**: コアシステム（copyfuncs.c や outfuncs.c など）は、必要に応じて GetCustomScanMethods を呼び出し、名前（CustomName）をキーにして登録済みの CustomScanMethods を取得します。 4. **実行状態の生成**: エグゼキュータの実行開始時、取得した CustomScanMethods の CreateCustomScanState コールバックが呼び出され、CustomScan プランノードから CustomScanState 実行状態ノード（Node *）が生成されます。

*注意: 登録処理の具体的な関数（RegisterCustomScanMethods など）の内部実装や、プランナー・エグゼキュータ側での呼び出しの完全なライフサイクルは、提供されたソースコード断片からは完全には明らかになっていません。*

重要な不変条件

  • **一意の名前**: CustomScanMethods の CustomName は一意である必要があります。
  • **名前の長さ制限**: CustomName は EXTNODENAME_MAX_LEN（64バイト）以内の文字列である必要があります。
  • **ハッシュテーブルの初期化**: GetCustomScanMethods を呼び出す際、内部ハッシュテーブル custom_scan_methods が初期化されている必要があります。

よくある落とし穴

  • **名前の不一致**: 拡張機能がメソッドを登録した際の名前と、GetCustomScanMethods で検索する際の名前が一致していない場合、メソッドの取得に失敗します。
  • **エラーハンドリング**: GetCustomScanMethods を呼び出す際、missing_ok を false に設定して存在しない名前を検索すると、システムエラーが発生します。存在しない可能性がある場合は missing_ok を true に設定し、返り値の NULL チェックを行う必要があります。
