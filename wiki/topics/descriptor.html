descriptor

概念

descriptor は、ECPG（Embedded SQL for C）においてSQL記述子（Descriptor）の情報を保持し、ランタイムライブラリ内で管理するためのデータ構造です。SQLの実行結果やメタデータ、およびホスト変数とのマッピング情報を保持する役割を担います。

なぜ重要か

ECPGランタイムが動的SQLを実行する際、クエリ結果（libpq の PGresult）や個々の項目（descriptor_item）を追跡・管理するために不可欠です。これにより、C言語のホストプログラムとPostgreSQLデータベース間で、動的なデータの受け渡しを安全かつ正確に行うことができます。

関与するモジュール

  • module:src/interfaces/ecpg/ecpglib
  • module:src/interfaces/ecpg/preproc

主要エンティティ

  • **symbol:src/interfaces/ecpg/ecpglib/ecpglib_extern.h#descriptor**

    • ECPGランタイムで記述子を表現する struct descriptor 構造体。以下のフィールドを持ちます。

  • name: 記述子を識別する名前。
  • result: libpq の PGresult へのポインタ。
  • next: 記述子リストの次の要素へのポインタ。
  • count: 記述子内の項目数。
  • items: descriptor_item 構造体リストへのポインタ。
  • **symbol:src/interfaces/ecpg/preproc/type.h#descriptor**

    • プリプロセッサ側で記述子を扱うための関連定義。

エンドツーエンドの流れ

プリプロセッサ（preproc）がソースコードを解析してから、ランタイム（ecpglib）が struct descriptor を生成・操作するにいたる、具体的なエンドツーエンドの制御フローや関数呼び出しの詳細は、提供された情報からは完全には明らかになっていません。

一般的には、descriptor.c などのモジュールにおいて、記述子の検索、追加、削除といったリスト操作が実行され、クエリ結果のメタデータ管理に利用されます。

重要な不変条件

  • **単方向リスト構造**: next ポインタを介して、複数の記述子が連結リストを形成します。
  • **メモリ管理**: 記述子およびそのメンバー（name や items）は動的に割り当てられ、解放時には ecpg_free を使用して ecpglib のメモリ管理ポリシーを遵守する必要があります。
  • その他の詳細な状態遷移やスレッド間の不変条件については、提供された情報からは未詳です。

よくある落とし穴

  • **用語の混同**: PostgreSQLのコア機能や contrib/ 配下で頻出する「タプル記述子（tuple descriptor）」は、本ECPGの struct descriptor とは全く異なる概念です。コードリーディングやデバッグの際は混同しないよう注意してください。
  • **メモリリーク**: 記述子名（name）や項目（items）の割り当て失敗時、または記述子破棄時に適切な解放処理（ecpg_free）が行われないと、メモリリークの原因となります。
