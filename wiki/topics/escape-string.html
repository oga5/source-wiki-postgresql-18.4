escape_string

概念

escape_string は、PostgreSQLのソースコード内の複数のモジュール（クライアントツールおよびテストモジュール）において、文字列をSQLリテラルとして安全にエスケープするために定義・使用されている関数です。ただし、モジュールごとに異なるシグネチャや実装が存在します。

なぜ重要か

SQLクエリ内で文字列を安全に扱うためには、文字列リテラルの適切なエスケープが不可欠です。テストモジュール（test_escape）やクライアントツール（psql）において、それぞれの文脈に応じたエスケープ処理を行うためにこの名前の関数が定義されています。

主要エンティティ

  • symbol:src/test/modules/test_escape/test_escape.c#escape_string: テスト用のエスケープ関数。PQExpBuffer に対して、与えられた文字列をシングルクォートで囲み、PQescapeString を用いてエスケープした結果を追加します。
  • symbol:src/bin/psql/tab-complete.in.c#escape_string: psql のタブ補完処理に関連して定義されている同名の関数（詳細な実装は本コンテキストでは未定義）。

エンドツーエンドの流れ

test_escape モジュールにおける具体的な流れは以下の通りです： 1. pe_test_escape_func 構造体から escape_string が呼び出されます。 2. 対象のバッファ（PQExpBuffer target）の末尾に開始クォート ' が追加されます。 3. enlargePQExpBuffer により、エスケープ後の文字列を格納するのに十分な領域（unescaped_len * 2 + 1 バイト）が確保されます。 4. PQescapeString を用いて、未エスケープの文字列がバッファの末尾にエスケープされながら書き込まれます。 5. バッファの長さ（target->len）が更新され、最後に終了クォート ' が追加されます。

なお、src/bin/psql/tab-complete.in.c における escape_string のエンドツーエンドのフローや、システム全体における統一的なエンドツーエンドのフローは現時点では不明です。

重要な不変条件

  • **test_escape における不変条件**:
  • 入力される target は、初期化済みの有効な PQExpBuffer でなければなりません。
  • unescaped は、少なくとも unescaped_len バイトの有効なメモリ領域を指している必要があります。
  • 関数は常に true を返します。
  • psql 側の実装における不変条件は現時点では不明です。

よくある落とし穴

  • **実装の混同**: src/test/modules/test_escape/test_escape.c の escape_string はテスト専用の実装であり、引数として受け取る conn や escape_err を内部で利用していません。実運用コード（psql など）に存在する同名かつ異なるシグネチャの関数と混同しないよう注意が必要です。
  • **メモリ不足**: 非常に大きな unescaped_len が渡された場合、enlargePQExpBuffer 内部でのバッファ拡張に伴い、メモリ不足でプロセスが終了する可能性があります。
