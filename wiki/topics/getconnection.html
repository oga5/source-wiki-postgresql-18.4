GetConnection

概念

GetConnection は、PostgreSQLのソースコード内の異なるモジュールにおいて、データベース接続（PGconn）を取得または確立するために定義されている同名の関数群です。これらは共通の親クラスやインターフェースを持つものではなく、それぞれのコンテキスト（外部データラッパ、ベースバックアップ、ダンプツール）において独立して実装されています。

なぜ重要か

それぞれのモジュールにおいて、接続のライフサイクル管理、キャッシュ、セキュリティ設定、または接続オブジェクトへのアクセスをカプセル化するために重要です。

  • **postgres_fdw**: リモートサーバーへの接続オーバーヘッドを削減するためのキャッシュ管理、トランザクション同期、エラー回復を担います。
  • **pg_basebackup**: レプリケーション専用接続の確立、対話的パスワード要求、およびセキュリティ向上のための環境初期化（search_path のクリア）をカプセル化します。
  • **pg_dump**: アーカイブ処理中に保持されている接続オブジェクトへの安全なアクセス（ゲッター）を提供します。

関与するモジュール

  • module:contrib/postgres_fdw
  • module:src/bin/pg_basebackup
  • module:src/bin/pg_dump

主要エンティティ

  • symbol:contrib/postgres_fdw/connection.c#GetConnection: リモートPostgreSQLサーバーへの接続を取得・キャッシュ管理する関数。
  • symbol:src/bin/pg_basebackup/streamutil.c#GetConnection: レプリケーション専用接続を確立し初期設定を行う関数。
  • symbol:src/bin/pg_dump/pg_backup_db.c#GetConnection: アーカイブハンドルから保持している接続を取得するゲッター関数。

エンドツーエンドの流れ

これらは独立したモジュールで定義された関数であるため、**システム全体を横断する単一のエンドツーエンドの流れは存在しません**。それぞれの関数における処理の流れは以下の通りです。

postgres_fdw における流れ

1. **キャッシュ検索**: UserMapping の OID をキーに ConnectionHash を検索します。 2. **接続確立**: キャッシュにない、または無効化されている場合は make_new_connection で新規接続を確立します。 3. **トランザクション同期**: begin_remote_xact を呼び出し、ローカルのトランザクション深度に合わせてリモート側でトランザクションまたはセーブポイントを開始します。 4. **エラー回復**: 接続失敗時は PG_CATCH ブロックで接続を破棄し、再接続を試行します。

pg_basebackup における流れ

1. **接続情報の構築**: コマンドライン引数や接続文字列からパラメータを生成し、レプリケーションモードを有効化します。 2. **接続試行**: PQconnectdbParams を使用して接続します。パスワードが必要な場合は simple_prompt で対話的に取得し再試行します。 3. **環境初期化**: PostgreSQL 10以降の場合、セキュリティ向上のため search_path をクリアします。

pg_dump における流れ

1. **キャスト**: 引数 Archive *AHX を内部構造体 ArchiveHandle * にキャストします。 2. **取得**: 構造体内の connection メンバを直接参照して返します。

重要な不変条件

モジュール間で共通する不変条件はありませんが、それぞれの関数において以下の不変条件が課されています。

  • **postgres_fdw**:
  • 接続は UserMapping 単位でキャッシュされ、ユーザーごとに認証情報が分離されます。
  • 接続が「汚染」されている（pgfdw_reject_incomplete_xact_state_change で判定）場合、その接続は再利用されません。
  • **pg_basebackup**:
  • dbname と connection_string が同時に非NULLであってはなりません（Assert による検証）。
  • **pg_dump**:
  • 入力される AHX は、有効な ArchiveHandle 構造体へのポインタである必要があります。

よくある落とし穴

  • **postgres_fdw**: リモート側でのトランザクション開始失敗時、再接続後も失敗が続く場合は例外が再スローされるため、エラーハンドリングの挙動に注意が必要です。また、無効化された接続はトランザクション深度が 0 になるまで破棄されません。
  • **pg_basebackup**: PostgreSQL 10未満のサーバーと接続する場合、search_path のクリア処理がスキップされるため、バージョン互換性の分岐ロジックに注意する必要があります。
  • **pg_dump**: 本関数は単なるゲッターであり、接続の確立や再接続は行いません。また、AHX が NULL の場合は構造体のメンバアクセスによりセグメンテーション違反が発生する可能性があります。
  • **呼び出し元の混同**: 提供された情報において、pg_dump の GetConnection の呼び出し元として postgres_fdw の記述が一部見られますが、これらはシグネチャや目的が異なるため、モジュールをまたいだ誤用や混同に注意してください。
