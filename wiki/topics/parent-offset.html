parent_offset

概念

parent_offset は、配列ベースの二分木（バイナリヒープ）において、指定されたノードのインデックス i から、その親ノードのインデックスを算出するためのユーティリティ関数です。

二分ヒープの構造上、インデックス i の左の子は 2*i + 1、右の子は 2*i + 2 に配置されるため、その逆変換として (i - 1) / 2 を計算します。

なぜ重要か

バイナリヒープの操作（特に要素をヒープ内の適切な位置まで浮上させる sift_up 処理やヒープの再構築）において、親ノードとの比較や入れ替えを高速に行うために不可欠です。本関数は static inline として定義されており、コンパイル時にインライン展開されることで、頻繁に呼び出されるヒープ操作のオーバーヘッドを最小限に抑える役割を果たしています。

エンドツーエンドの流れ

1. **ヒープ操作のトリガー**: src/common/binaryheap.c 内での sift_up 処理や、ヒープ構築時のループ処理が実行されます。 2. **親インデックスの計算**: 対象ノードのインデックス i を引数として parent_offset(i) が呼び出され、(i - 1) / 2 の算術演算により親ノードのインデックスが返されます。 3. **ソート処理等での利用**: src/backend/utils/sort/logtape.c などの上位モジュールにおけるソートやテープ操作の文脈で、このヒープ操作が利用されます。

*注意: logtape.c における具体的な呼び出しコンテキストや、システム全体における詳細なエンドツーエンドのデータフローの全容は、提供された情報からは完全には明らかになっていません。*

重要な不変条件

  • **入力値の範囲**: 入力 i は 0 以上の整数であることを前提とします。
  • **ルートノードの計算結果**: ルートノード（インデックス 0）に対して呼び出した場合、整数除算の特性により (0 - 1) / 2 の計算結果として 0 が返されます。

よくある落とし穴

  • **ルートノードの境界条件**: i = 0 の場合に 0（自身と同じインデックス）が返されるため、呼び出し側で「ルートノードに達したか（i > 0）」を適切に判定して処理を終了させないと、無限ループや誤った自己比較が発生する原因になります。
  • **型変換に伴うリスク**: src/backend/utils/sort/logtape.c では uint64 型の引数で呼び出されている箇所があります。parent_offset のシグネチャは int を受け取るように定義されているため、型変換に伴う値の切り捨てやオーバーフローの可能性について、呼び出し側の文脈で慎重に確認する必要があります（この不変条件が常に安全に維持されているかは未検証です）。
