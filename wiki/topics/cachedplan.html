CachedPlan

概念

CachedPlan は、PostgreSQL において実行可能なクエリ計画（PlannedStmt のリストなど）を保持し、それを再利用可能にすることで、クエリの解析（パース）や計画作成（プランニング）のコストを削減するためのデータ構造です。

なぜ重要か

クエリを実行するたびに発生する解析・計画作成コストは、データベース全体のパフォーマンスに影響を与えます。CachedPlan を用いて実行計画をキャッシュし、適切に再利用することで、このオーバーヘッドを大幅に削減できます。また、スキーマ変更などのイベントが発生した際には、キャッシュされた計画を無効化（is_valid = false）して再計画を促す仕組みを提供し、データの整合性と最新スキーマへの追従を両立させています。

エンドツーエンドの流れ

提供されたソースコード情報からは、完全なエンドツーエンドの流れ（特に GetCachedPlan の詳細な内部動作や、実行エンジンへの受け渡し、CompleteCachedPlan との連携など）は完全には明らかになっておらず、一部未検証です。

現在判明している部分的な流れは以下の通りです：

1. **初期化（第1段階）**: CreateCachedPlan を呼び出し、生パースツリー（RawStmt）やクエリ文字列、コマンドタグを基に、専用のメモリコンテキスト（AllocSetContextCreate で作成された長寿命なコンテキスト）内で CachedPlanSource 構造体を初期化・確保します。 2. **計画の生成と保持**: （詳細なフローは未検証ですが）実行計画が作成され、CachedPlan 構造体の stmt_list に PlannedStmt のリストとして格納されます。 3. **再利用と無効化の監視**: 生成された CachedPlan は参照カウント（refcount）により管理され、再利用されます。スキーマ変更などのイベントが発生すると is_valid が false に設定され、次回利用時に再計画（Replan）がトリガーされます。また、saved_xmin を使用してトランザクションの xmin 変化を検知し、再計画を判定します。 4. **解放**: 参照カウント（refcount）が 0 になった時点で、context が示すメモリコンテキストが解放されます。

重要な不変条件

  • **マジックナンバーの整合性**: CachedPlan 構造体の magic フィールドは、常に CACHEDPLAN_MAGIC と一致しなければなりません。
  • **有効性フラグの遵守**: is_valid が false の場合、その計画は再利用できず、必ず再計画（Replan）を行う必要があります。
  • **メモリ管理のライフサイクル**: refcount が 0 になった時点で、その CachedPlan に紐づくメモリコンテキスト（context）は解放されます。
  • **入力制限**: CreateCachedPlan の呼び出し時、query_string は NULL であってはなりません（デバッグビルドでは Assert により停止します）。

よくある落とし穴

  • **参照カウントの管理不備**: refcount の操作（通常は ResourceOwner を介して行われる）に不備があると、メモリリークや不正なメモリ解放を引き起こす原因になります。
  • **無効な計画の実行**: is_valid が false であるにもかかわらず、再計画を行わずに計画を実行しようとした場合、未定義の動作やエラーが発生する可能性があります。
  • **oneshot 計画の監視対象外**: is_oneshot が真に設定されているクエリは、キャッシュ無効化の監視対象外となり、呼び出し元のメモリコンテキストに依存するため、スキーマ変更などの影響を考慮する際に注意が必要です。
  • **初期化の未完了**: CreateCachedPlan は計画キャッシュ作成の「第1段階」に過ぎず、キャッシュを完全に機能させるには CompleteCachedPlan の呼び出しが必要です。
