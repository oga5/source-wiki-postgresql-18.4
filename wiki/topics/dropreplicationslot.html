DropReplicationSlot

概念

DropReplicationSlot は、PostgreSQLにおいて不要になったレプリケーションスロットを削除（破棄）するための処理です。この処理には、クライアント（フロントエンド）側から削除コマンドを送信する実装と、サーバー（バックエンド）側でそのコマンドを受け取って実際にスロットを削除する実装の2つの側面が存在します。

なぜ重要か

レプリケーションスロットは、WAL（Write-Ahead Log）の保持や論理デコードの状態を管理するためにサーバー上でリソースを消費し続けます。不要になったスロットを適切に削除することは、ディスク容量の枯渇を防ぎ、サーバーリソースを解放するために不可欠です。pg_receivewal や pg_recvlogical などのフロントエンドツールが処理をクリーンアップする際などに、この削除処理が実行されます。

関与するモジュール

  • **module:src/backend/replication**: バックエンド側におけるレプリケーションスロットの削除処理およびコマンドの受信を担当します。
  • **module:src/bin/pg_basebackup**: フロントエンドツール（pg_receivewal や pg_recvlogical など）からサーバーに対してスロット削除を要求する処理を担当します。

主要エンティティ

  • **symbol:src/bin/pg_basebackup/streamutil.c#DropReplicationSlot**: フロントエンド側の関数。libpq を介してサーバーへ DROP_REPLICATION_SLOT コマンドを送信します。
  • **symbol:src/backend/replication/walsender.c#DropReplicationSlot**: バックエンド側の関数。サーバー内部でスロット削除要求を処理します（詳細な内部仕様は現時点で未提供）。

エンドツーエンドの流れ

1. **削除のトリガー**: フロントエンドツール（pg_receivewal や pg_recvlogical など）が、不要になったスロットを削除するために streamutil.c の DropReplicationSlot を呼び出します。 2. **コマンドの構築と送信**: フロントエンド側は DROP_REPLICATION_SLOT "slot_name" コマンド文字列を構築し、PQexec を使用してサーバーへ送信します。 3. **サーバー側の処理**: サーバー（バックエンド）側でコマンドが受信され、walsender.c などのバックエンド側処理によって実際のスロット削除が実行されます。 4. **結果の検証**: フロントエンド側は、サーバーからの応答が PGRES_COMMAND_OK であり、かつ返されたデータが 0 行 0 フィールドであることを確認して、成功（true）または失敗（false）を呼び出し元に返します。

*注: バックエンド側（walsender.c 内）の具体的な処理フローや、フロントエンドとバックエンドがどのように完全に同期して動作するかという詳細なエンドツーエンドのフロー全体は、提供された情報からは完全には明らかになっていません。*

重要な不変条件

  • **フロントエンド側の不変条件**:
  • 入力される slot_name は NULL であってはなりません（Assert により検証されます）。
  • 接続 conn はレプリケーションプロトコルが利用可能な有効な状態（PGconn）でなければなりません。
  • 正常に削除された場合、サーバーからの応答は PGRES_COMMAND_OK であり、かつ返されるデータ（タプルやフィールド）は 0 件でなければなりません。

*注: バックエンド側における不変条件（例: アクティブに接続されているスロットは削除できない等）は、提供された情報からは不明です。*

よくある落とし穴

  • **同名関数の混同**: フロントエンド用の DropReplicationSlot（streamutil.c）と、バックエンド用の DropReplicationSlot（walsender.c）は名前が同一ですが、動作するプロセスやコンテキストが全く異なるため、混同しないよう注意が必要です。
  • **NULLポインタの渡し**: フロントエンド関数に slot_name として NULL を渡すと、アサーション違反によりプロセスが異常終了します。
  • **エラーハンドリング**: サーバーから予期せぬデータ（タプルやフィールド）が返された場合や、PGRES_COMMAND_OK 以外のステータスが返された場合は失敗（false）となるため、呼び出し元で適切に成否を判定する必要があります。
