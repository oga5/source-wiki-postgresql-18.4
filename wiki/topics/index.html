Index

概念

PostgreSQL において Index という名前の型定義は、定義されるヘッダーファイル（文脈）によって異なる2つの役割を持っています。

1. **汎用的な符号なし整数型 (src/include/c.h)** PostgreSQL 全体で広く使用される unsigned int のエイリアスです。インデックス（索引）や配列の添字、あるいは特定の識別子を表現するために使用されます。 2. **外部テーブル更新計画の識別子 (src/include/foreign/fdwapi.h)** クエリ計画作成時に、外部テーブル（Foreign Table）に対する更新操作（INSERT, UPDATE, DELETE）の計画を識別するためのインデックス値を保持するために使用されます。

なぜ重要か

  • **コードの標準化と移植性**: c.h で定義された Index は、データベースの内部構造（インデックスやタプル）を扱う際の標準的な型として定着しており、システム全体で一貫した型定義を提供します。
  • **外部データラッパ (FDW) の制御**: fdwapi.h で定義された Index は、クエリ計画フェーズにおいて ModifyTable ノードがどの外部テーブルを操作対象とするかを一意に特定し、FDW が対象の外部テーブルに関する情報を取得するためのキーとして機能します。

関与するモジュール

主要エンティティ

  • **symbol:src/include/c.h#Index**

    • typedef unsigned int Index; として定義される汎用型。contrib/amcheck（verify_common.c, verify_gin.c, verify_nbtree.c など）をはじめ、PostgreSQL の全ソースファイルから利用可能です。

  • **symbol:src/include/foreign/fdwapi.h#Index**

    • FDW API の文脈で定義される型（Oid 型の別名として機能）。外部テーブルの更新計画を生成する関数（PlanForeignModify_function など）の引数として利用されます。

エンドツーエンドの流れ

1. **汎用型としての利用 (c.h)** 特定の単一のエンドツーエンドフローは存在しません。システム全体の様々なデータ構造やアルゴリズムで利用されており、例えば contrib/amcheck モジュールでは、エラーメッセージの出力やデータ構造の記述において、対象のインデックスやタプルを特定するためにこの型が参照されます。 2. **FDW 計画フェーズでの利用 (fdwapi.h)** クエリ計画フェーズにおいて、外部テーブルに対する更新操作が計画される際、PlanForeignModify_function などの関数に resultRelation 引数として Index 値が渡されます。この値をもとに、FDW は対象リレーションの情報を取得します。

*注意: 具体的な関数呼び出しの直接的な遷移や、詳細なエンドツーエンドの制御フローは、提供された情報からは完全には明らかになっていません。*

重要な不変条件

  • **c.h の Index**:
  • unsigned int 型のサイズおよび表現範囲に依存します。
  • **fdwapi.h の Index**:
  • PlannerInfo 構造体内の simple_rel_array におけるインデックスとして有効な値である必要があります。

よくある落とし穴

  • **名前の重複と混同**:

    • contrib モジュールなどのインデックス（B-Tree などの索引）関連コードで多用される Index（unsigned int）と、FDW API の文脈における Index（リレーションの識別子）は、名前が同一ですが意味が異なります。開発時にはどちらの文脈の Index を指しているかを意識する必要があります。

  • **オーバーフロー**:

    • c.h の Index は unsigned int であるため、最大値を超える値を代入した場合にオーバーフローが発生します。プラットフォームごとのビット幅（通常 32 ビット）に依存するため、移植性に注意が必要です。

  • **無効な識別子**:

    • FDW API において無効な Index 値が渡された場合、PlannerInfo からリレーション情報を取得する際にエラーが発生する可能性があります。
