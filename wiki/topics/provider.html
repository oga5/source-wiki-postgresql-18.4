provider

概念

PostgreSQL のソースコードにおいて、provider という用語は複数の異なる文脈で使用されます。

1. **OAuth 2.0 認証プロバイダー**: src/interfaces/libpq-oauth モジュールにおいて、OAuth 2.0 認可サーバーのメタデータやエンドポイント情報を保持するデータ構造（struct provider）として定義されています。 2. **JIT（Just-In-Time）プロバイダー**: src/backend/jit モジュール（jit.c）にも provider というシンボルが存在しますが、その具体的な概念や役割は提供された情報からは不明です。

なぜ重要か

  • **OAuth 2.0 連携の制御**: libpq-oauth において、認可サーバーから取得した Discovery ドキュメントの情報を保持し、Device Authorization や Token Request などの認証フローで正しい通信先 URL を特定するために不可欠です。
  • **JIT コンパイルの管理**: src/backend/jit/jit.c における provider は JIT 処理の実行プロバイダーに関連すると推測されますが、詳細な重要性は現時点で不明です。

関与するモジュール

  • module:src/backend/jit
  • module:src/interfaces/libpq-oauth

主要エンティティ

  • **symbol:src/interfaces/libpq-oauth/oauth-curl.c#provider**: OAuth 2.0 認可サーバーのメタデータ（issuer, token_endpoint, device_authorization_endpoint, grant_types_supported）を保持する struct。
  • **symbol:src/backend/jit/jit.c#provider**: JIT モジュール内のシンボル（詳細な型や役割は未定義）。

エンドツーエンドの流れ

OAuth 2.0 認証フロー

1. **Discovery 情報の取得**: 認証フローの初期段階で、認可サーバーから Discovery ドキュメント（JSON）を取得します。 2. **メタデータの格納**: パースされた JSON から、struct provider の各フィールドにエンドポイント URL やサポートされるグラントタイプ（grant_types_supported）を格納します。 3. **リクエストの送信**: 以降の認可リクエストやトークンリクエストの送信時に、本構造体に格納された URL を参照して通信先を特定します。

JIT プロバイダーのフロー

  • src/backend/jit/jit.c における provider のエンドツーエンドの制御フローや初期化の流れは、提供された情報からは不明です。

重要な不変条件

  • **OAuth 2.0 struct provider**:
  • issuer、token_endpoint、device_authorization_endpoint の各文字列フィールドは、メモリ確保された有効な文字列へのポインタである必要があります。
  • grant_types_supported は libcurl の curl_slist 構造体として管理され、不要になった際には curl_slist_free_all を使用して適切に解放されなければなりません。
  • **JIT プロバイダー**:
  • 不変条件は現時点では不明です。

よくある落とし穴

  • **同名シンボルの混同**: provider という名称は、libpq-oauth の OAuth プロバイダー、jit.c の JIT プロバイダーのほか、セキュリティプロバイダー（sepgsql）や照合順序プロバイダー（collationcmds.c）など、PostgreSQL 内の全く異なる文脈で独立して使用されています。これらを混同しないよう注意が必要です。
  • **不完全なメタデータによる未初期化**: 認可サーバーから返される JSON に必要なエンドポイントが含まれていない場合、struct provider のフィールドが未初期化または NULL になる可能性があります。

  • **src/backend/jit/jit.c**: JIT プロバイダーシンボルの定義と役割。
