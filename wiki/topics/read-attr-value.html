read_attr_value

概念

read_attr_value は、PostgreSQLのSCRAM（Salted Challenge Response Authentication Mechanism）認証において、SCRAMメッセージ文字列から指定された属性キーに対応する値（key=value 形式の value 部分）を抽出するためのユーティリティ関数です。メッセージはカンマ区切りの属性ペアで構成されており、この関数を用いて必要な情報を順次パースします。

なぜ重要か

SCRAM認証プロトコルでは、クライアントとサーバーの間でユーザー名、ナンス（nonce）、チャンネルバインディングなどの様々な属性情報を正確にやり取りする必要があります。read_attr_value は、これらのメッセージから必要な属性値を安全かつ効率的に抽出するために不可欠な役割を担っています。サーバー側とクライアント（libpq）側の双方で同様のパース処理が必要となるため、それぞれのコンテキストに応じた同名の関数が定義されています。

関与するモジュール

  • module:src/backend/libpq (サーバー側の接続・認証処理)
  • module:src/interfaces/libpq (クライアント側の接続・認証処理)

主要エンティティ

  • **symbol:src/backend/libpq/auth-scram.c#read_attr_value**

    • サーバー側の実装です。エラーが発生した場合は、ereport(ERROR, ...) を呼び出して処理を中断します。

  • **symbol:src/interfaces/libpq/fe-auth-scram.c#read_attr_value**

    • クライアント（libpq）側の実装です。エラー発生時は NULL を返し、引数で渡された PQExpBuffer にエラーメッセージを記録します。

エンドツーエンドの流れ

SCRAM認証全体におけるメッセージ送受信の完全なエンドツーエンドの流れは、提供された情報からは特定されていません。ただし、各モジュール内における read_attr_value を用いた局所的な解析の流れは以下の通りです。

1. **サーバー側 (auth-scram.c)**: クライアントから受信したSCRAMメッセージを解析する際、以下の属性を順次抽出するために呼び出されます。

  • channel_binding_type (p 属性)
  • state->client_username (n 属性)
  • state->client_nonce (r 属性)
  • channel_binding (c 属性)
  • state->client_final_nonce (r 属性)

2. **クライアント側 (fe-auth-scram.c)**: サーバーから受信したメッセージ（read_server_first_message など）を解析する際に呼び出されます。

**解析アルゴリズムの共通ステップ**: 1. input ポインタが指す現在位置の文字が、期待する属性キー attr と一致するか検証します。 2. 次の文字が = であることを検証します。 3. = の直後から、カンマ（,）または文字列終端（\0）までを値の範囲として特定します。 4. カンマが見つかった場合は \0 に置換して文字列を分割し、input ポインタを次の属性の開始位置（, の次の文字）へ進めます。 5. 抽出した値の先頭アドレスを返します。

重要な不変条件

  • **入力文字列の破壊的変更**: 本関数は、抽出した値の末尾（カンマの位置）を \0 で置換して文字列を分割します。そのため、input に渡される文字列は必ず書き込み可能（mutable）でなければなりません。
  • **パースの順序依存性**: 解析が進むと input ポインタが次の属性の開始位置に更新されるため、メッセージ内の属性はプロトコルで定義された順序通りにパースされる必要があります。

よくある落とし穴

  • **サーバー版とクライアント版の混同**:

    • 同名でほぼ同様の処理を行いますが、関数シグネチャが異なります。クライアント版（fe-auth-scram.c）はエラーハンドリングのために PQExpBuffer errorMessage を引数に取り、失敗時に NULL を返しますが、サーバー版（auth-scram.c）はエラー時に直接 ereport を発生させます。

  • **元の文字列の破壊**:

    • 関数内部で \0 を挿入して文字列を書き換えるため、呼び出し側で元のメッセージ文字列をそのまま保持し続けたい場合は、事前にコピーを取るなどの対策が必要です。
