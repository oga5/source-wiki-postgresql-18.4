Py_ssize_t

概念

Py_ssize_t は、Python C API においてコンテナのサイズ、インデックス、または文字列の長さを表現するために使用される標準的な符号付き整数型です。プラットフォームのポインタサイズ（64ビット環境では通常 ssize_t）に応じて解決されます。PostgreSQL の PL/Python 拡張モジュール群において、PostgreSQL 側のデータ構造と Python オブジェクト（PyObject）との間のデータ変換やサイズ管理に広く利用されています。

なぜ重要か

PL/Python 拡張モジュール（hstore_plpython、jsonb_plpython、ltree_plpython など）が PostgreSQL のデータ（hstore、jsonb、ltree など）を Python オブジェクトに変換する際、要素数や文字列長を正確に Python 側に伝える必要があります。Py_ssize_t は、Python C API（PyList_New、PyList_SetItem、PLyUnicode_FromStringAndSize など）の引数や戻り値の型として規定されているため、型安全なデータ連携とメモリ確保を行うために不可欠です。

  • **symbol:contrib/jsonb_plpython/jsonb_plpython.c#Py_ssize_t**

    • リストや辞書の要素数カウント、および PLyUnicode_FromStringAndSize_t などの関数ポインタ定義の引数として使用されます。

  • **symbol:contrib/ltree_plpython/ltree_plpython.c#Py_ssize_t**

    • PyList_New や PLyUnicode_FromStringAndSize などの Python API 呼び出し時に、リストの要素数やデータサイズを保持・伝達するために使用されます。

エンドツーエンドの流れ

PostgreSQL 側のデータ（hstore、jsonb、ltree）から Python オブジェクトへの変換処理における具体的なエンドツーエンドの流れは、提供されたソースコードの抜粋からは完全には特定されていません。

ただし、部分的な流れとして以下が確認されています。 1. 各拡張モジュールの変換関数（例: hstore_to_plpython）において、PostgreSQL 側のデータから要素数やサイズを取得し、Py_ssize_t 型の変数（pcount など）に格納します。 2. Py_ssize_t 型のループカウンタ（i など）を用いてループ処理を行い、各要素を処理します。 3. PyList_New や PLyUnicode_FromStringAndSize などの Python C API、あるいは PLyUnicode_FromStringAndSize_t などの関数ポインタを介して、サイズ情報を Python 側に渡して Python オブジェクトを生成・構築します。

重要な不変条件

  • **符号付き整数としての特性**: Python C API の規約に従い、負の値はエラー状態の表現や、特殊なインデックス指定（末尾からのオフセットなど）として解釈される場合があります。
  • **コンテナの許容範囲**: Py_ssize_t で表現されるサイズやインデックスは、対象となる Python コンテナの許容範囲内（オーバーフローしない範囲）でなければなりません。
  • **プラットフォーム依存**: プラットフォームのポインタサイズに依存する型であり、PostgreSQL 固有の型（int32 や int64）とは区別して扱う必要があります。

よくある落とし穴

  • **オーバーフロー**: 非常に大きなデータセットを扱う際、プラットフォームの Py_ssize_t の最大値を超えるインデックスやサイズを指定すると、Python C API 側で OverflowError やメモリ割り当て失敗が発生するリスクがあります。
  • **負の値の誤用**: インデックスとして負の値を渡す際、呼び出し先の Python C API 関数が負の値をサポートしていない場合、予期せぬ動作やクラッシュを引き起こす可能性があります。
  • **PostgreSQL 固有型との混同**: PostgreSQL 側の整数型（int32 や int64）と Py_ssize_t を混同して直接演算や代入を行うと、型サイズや符号の不一致によるバグの原因となります。
