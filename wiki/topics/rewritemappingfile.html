RewriteMappingFile

概念

RewriteMappingFile は、PostgreSQLにおけるテーブル再構築（ヒープの書き換え）の際に、論理デコーディング用のマッピング情報を保持・管理するためのデータ構造（struct）です。テーブル再構築に伴って変化する古いタプルID（TID）と新しいTIDの対応関係を追跡し、ディスク上のファイルへ永続化するための状態（トランザクションID、ファイル記述子、書き込みオフセット、メモリ上のマッピングリストなど）を保持します。

なぜ重要か

テーブルの再構築処理（VACUUM FULL や CLUSTER など）が行われると、物理的なタプル位置（TID）が変更されます。論理レプリケーション（論理デコーディング）が有効な環境において、再構築中のトランザクションを正しくデコードするためには、新旧のTIDの対応関係を正確に把握する必要があります。RewriteMappingFile はこのマッピング情報を pg_logical/mappings 配下のファイルに永続化することで、クラッシュリカバリ時やデコーディング処理時におけるデータの整合性を担保する重要な役割を果たします。

関与するモジュール

  • module:src/backend/access/heap：ヒープの書き換え処理（rewriteheap.c）を担当します。
  • module:src/backend/replication/logical：論理デコーディングおよびリオーダーバッファ処理（reorderbuffer.c）を担当します。

主要エンティティ

  • symbol:src/backend/access/heap/rewriteheap.c#RewriteMappingFile：構造体の定義本体。トランザクションID（xid）、仮想ファイル記述子（vfd）、書き込みオフセット（off）、メモリ上のリスト（mappings）、ファイルパス（path）を管理します。
  • symbol:src/backend/replication/logical/reorderbuffer.c#RewriteMappingFile：論理デコーディング処理において、このマッピング情報を参照・利用します。

エンドツーエンドの流れ

1. **マッピングの追跡と生成**: テーブル再構築処理中、古いヒープから新しいヒープへタプルが移動する際に、新旧TIDの対応関係が生成されます。 2. **メモリ上での管理とディスクへの書き込み**: 生成されたマッピングは RewriteMappingFile 内の mappings リスト（dclist_head）で管理され、仮想ファイル記述子 vfd を通じて pg_logical/mappings 配下のファイルへ逐次書き込まれます。 3. **論理デコーディングでの参照**: reorderbuffer.c などの論理デコーディング処理が、これらのマッピングファイルを参照して、再構築を跨ぐトランザクションの整合性を維持しながらデコードを行います。 4. **クリーンアップ**: チェックポイント時に、不要になったマッピングファイルの同期や削除が行われます。

*注意: 詳細な関数呼び出しシーケンスや、チェックポイント時の具体的な削除トリガー条件など、完全なエンドツーエンドの制御フローの全貌は提供されたコンテキストからは未詳です。*

重要な不変条件

  • **論理レプリケーションの依存性**: 本構造体およびマッピングファイルの生成・利用は、論理レプリケーションが有効な場合にのみ実行されます。
  • **パスの長さ制限**: ファイルパスを格納する path メンバは、MAXPGPATH によるサイズ制限を厳格に遵守する必要があります。
  • **VFD管理**: ファイル記述子 vfd の管理は、PostgreSQLの仮想ファイル記述子（VFD）サブシステム（storage/fd.h）の規則に従う必要があります。

よくある落とし穴

  • **メモリ不足（OOM）**: 大量の未解決タプルが存在する場合、メモリ上のハッシュテーブルやマッピングリスト（mappings）が肥大化し、メモリ不足を引き起こす可能性があります。
  • **ファイルシステムの制限**: マッピングファイルのパスが MAXPGPATH を超過した場合や、ディスク容量不足、書き込み権限の不備によってファイルの作成・書き込みが失敗するリスクがあります。
  • **モジュール間結合**: 本構造体は rewriteheap.c と reorderbuffer.c の両方から直接参照されているため、構造体の定義やレイアウトを変更する際は、双方のモジュールにおける論理レプリケーションの整合性に影響を与えないよう注意が必要です。
