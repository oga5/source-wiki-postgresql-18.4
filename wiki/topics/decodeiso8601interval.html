DecodeISO8601Interval

概念

DecodeISO8601Interval は、ISO 8601 形式の時間間隔（Interval）を表す文字列（例: P1Y2M3DT4H5M6S など）を解析し、システム内部で扱える時間間隔表現（tm 構造体および小数秒 fsec_t）に変換するための関数です。PostgreSQL のコアバックエンドと、クライアント側の埋め込み SQL プリプロセッサ（ECPG）のライブラリの双方に同様のロジックが存在します。

なぜ重要か

ISO 8601 は時間間隔を表現するための標準規格であり、データベース内外でのデータ交換において広く利用されています。この関数は、入力された標準フォーマットの文字列を正確にパースし、データベースエンジンやクライアントアプリケーションが理解できる内部データ構造（interval）へと橋渡しする重要な役割を担っています。

エンドツーエンドの流れ

ECPG 側（interval.c）における解析の流れは以下の通りです：

1. **初期化**: 解析前に ClearPgTm を呼び出し、出力用の tm 構造体と小数秒 fsec を初期化します。 2. **プレフィックス検証**: 入力文字列 str が 'P' で始まっているかを確認します。 3. **走査と数値抽出**: 'T' を境界として日付部と時刻部を走査し、ParseISO8601Number を用いて数値を抽出します。 4. **フィールド加算**: 抽出した数値と単位（Y, M, W, D, H, M, S）に応じて、tm 構造体の各フィールドを加算します。 5. **小数秒の調整**: 小数秒が含まれる場合は、AdjustFractDays または AdjustFractSeconds を用いて tm と fsec に適切に配分します。 6. **代替形式の処理**: YYYYMMDD や HHMMSS などの代替形式にも対応し、特定の桁数条件を満たす場合に解析を行います。

**未判明な事項**: バックエンド側（datetime.c）における具体的な呼び出し元や、システム全体を通じたエンドツーエンドの完全な制御フローは、現時点のコンテキスト情報からは判明していません。

重要な不変条件

  • **入力形式**: 解析対象の文字列 str は必ず 'P' で始まる必要があります。
  • **初期化状態**: 解析処理の開始時に、tm および fsec は ClearPgTm によって初期化されていなければなりません。
  • **コンパイル制限**: -ffast-math コンパイルオプションは使用できません。
  • **出力設定**: 正常に解析が完了した場合、dtype には DTK_DELTA が設定され、関数は 0 を返します。

**未判明な事項**: バックエンド側における不変条件の差異や詳細については、現時点では未検証です。

よくある落とし穴

  • **構造体の違い**: ECPG 側の実装では、バックエンドで使用される struct pg_tm ではなく、標準の struct tm が使用されているため、移植やコードの参照時には注意が必要です。
  • **前提スタイル**: IntervalStyle は INTSTYLE_POSTGRES_VERBOSE が前提となっています。
  • **エラーハンドリング**: 文字列が 'P' で始まらない場合や、長さが 2 未満、あるいは不正なフォーマットが検出された場合は DTERR_BAD_FORMAT などのエラーコードが返されます。呼び出し側でこれらのエラーを適切に処理する必要があります。
