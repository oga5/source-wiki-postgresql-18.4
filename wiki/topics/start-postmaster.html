start_postmaster

概念

start_postmaster は、PostgreSQL のユーティリティ（特に pg_upgrade や pg_ctl）において、Postmaster（PostgreSQL データベースサーバー）プロセスを起動し、必要に応じてその後の接続確認などを行うための関数、または処理の単位です。

なぜ重要か

データベースのアップグレード（pg_upgrade）や制御（pg_ctl）のプロセスにおいて、サーバーインスタンスを安全かつ適切な設定で起動・検証することは、管理操作を自動化し、データの整合性を保つために不可欠です。特に pg_upgrade においては、新旧のクラスターを一時的に起動してメタデータの検証や移行を行うために、この処理が重要な役割を果たします。

エンドツーエンドの流れ

pg_upgrade における start_postmaster の処理フローは以下の通りです。

1. **初期化とフック登録** 初回呼び出し時に、終了時処理として stop_postmaster_atexit を atexit フックに登録します。 2. **コマンド構築** 対象クラスター（ClusterInfo）のバージョンや種類、環境（Windows/非Windows）に応じて、pg_ctl start の起動コマンド文字列を構築します。

  • 非Windows環境では、TCP/IP 接続を無効化し、Unix ソケット権限を 0700 に制限する設定を追加します。
  • 新クラスター（new_cluster）の起動時には、パフォーマンス向上のため synchronous_commit=off、fsync=off、full_page_writes=off を強制的に付与します。

    • 3. **プロセスの起動** exec_prog を使用して pg_ctl start を実行します。 4. **状態の記録** 起動成功時、グローバル構造体 os_info.running_cluster に対象クラスターを記録します。 5. **接続確認** get_db_conn を呼び出し、template1 データベースへの接続を試行してサーバーが実際に稼働しているか確認します。 6. **エラーハンドリング** 起動または接続確認に失敗した場合、report_and_exit_on_error フラグが true であれば pg_fatal でプロセスを終了し、false であれば false を返して呼び出し元に制御を戻します。

※ src/bin/pg_ctl/pg_ctl.c 側の start_postmaster におけるエンドツーエンドの流れは、提供されたコンテキストからは不明です。

重要な不変条件

  • **一時的な最適化設定の適用 (pg_upgrade)**

    • 新クラスターを起動する際、fsync=off などのパラメータが強制的に付与されます。これは、クラッシュ時にはクラスターを再作成すればよいという前提（不変条件）に基づいています。

  • **後処理の保証 (pg_upgrade)**

    • 初回呼び出し時に atexit フック（stop_postmaster_atexit）を登録することで、プログラム終了時に起動した Postmaster が確実に停止されるようにします。

  • ※ pg_ctl 側における不変条件は現時点では不明です。

よくある落とし穴

  • **接続確認と pg_ctl の終了コードの乖離**

    • pg_upgrade において、pg_ctl -w が失敗を返した場合でも、その後の template1 への直接接続確認が成功すれば、処理が継続される場合があります。この挙動を誤解すると、起動成否の判定ロジックで混乱が生じる可能性があります。

  • **クラッシュ時のデータ消失リスク**

    • 新クラスター起動時に fsync=off 等が強制されるため、この起動中にシステムがクラッシュした場合、データが破損しクラスターの再作成が必要になります。

  • ※ pg_ctl 側の実装における落とし穴は、提供されたコンテキストからは不明です。
