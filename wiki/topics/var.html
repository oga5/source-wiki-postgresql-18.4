Var

概念

Var は、PostgreSQLのクエリ処理（解析、計画、実行）において、テーブルの列やサブプランの出力を参照するための変数ノードを表現するデータ構造です。また、実行エンジンにおいては、プランツリー内の変数ノードを保持し、特に行全体（whole-row）の参照を評価するための実行時コンテキストの管理にも関与します。

なぜ重要か

クエリ内の列参照やサブクエリの出力を正しく追跡・評価するために不可欠です。

  • **柔軟な参照**: 物理テーブルの列だけでなく、サブプランの出力や、行全体（whole-row）の参照を統一的に表現できます。
  • **コンテキストの追跡**: 外部結合によるNULL化の追跡（varnullingrels）や、相関サブクエリにおける外側クエリレベルの参照（varlevelsup）をサポートします。
  • **実行時最適化**: 実行時には、不要な列のフィルタリング（JunkFilter）や、NULL値の実行時チェック（slow モード）を効率的に行うためのコンテキストを提供します。

関与するモジュール

  • module:src/include/nodes
  • module:src/include/executor

主要エンティティ

  • **symbol:src/include/nodes/primnodes.h#Var (struct)**: クエリ処理の各段階で変数（列やサブプラン出力）を表現する基本ノード。
  • **symbol:src/include/executor/execExpr.h#Var (struct wholerow)**: 実行エンジンにおいて、行全体（whole-row）の参照を評価・管理するための構造体。元の Var ノードへのポインタや、結果タプルのディスクリプタ（tupdesc）、不要な列を除去するための junkFilter を保持します。

エンドツーエンドの流れ

提供されたソース情報からは、詳細なエンドツーエンドの完全な流れは完全には特定されていませんが、以下の部分的な流れが判明しています。

1. **解析・計画段階**: クエリ内の列参照やサブクエリ出力が Var ノード（nodes/primnodes.h）として表現されます。 2. **実行段階の準備**: 行全体（whole-row）を参照する場合、executor/execExpr.h の wholerow 構造体が用意され、初回実行時（first フラグが真のとき）に必要なリソース（tupdesc や junkFilter）のセットアップが行われます。 3. **評価とフィルタリング**: ExecEvalExpr などの関数によって Var ノードが再帰的に走査され、指定された varno と varattno に基づいて値が抽出されます。行全体評価においては、slow フラグが有効な場合に実行時NULLチェックが行われ、junkFilter を使用して不要な列（resjunk列）が除去されます。

重要な不変条件

  • **特別な属性番号**: varattno が 0 の場合、そのテーブルの全列（whole-row Var）を指します。
  • **特別なリレーション番号**: varno が負の値（INNER_VAR, OUTER_VAR 等）の場合、物理テーブルではなく実行時のサブプラン出力を指します。
  • **ハッシュ・等価性判定の対象外**: vartype, vartypmod, varcollid はクエリのハッシュ化（Jumble）対象外であり、varnosyn と varattnosyn は意味論的な等価性判定において無視されます。
  • **実行時の整合性**: wholerow 構造体の var は有効なプランツリーの Var ノードを指している必要があり、tupdesc は評価対象となるタプルの構造と一致している必要があります。

よくある落とし穴

  • **位置情報の欠落**: location が -1 の場合、トークン位置が不明であることを示します。
  • **照合順序の未定義**: varcollid が InvalidOid の場合、照合順序（Collation）が定義されていないことを示します。
  • **メタデータの変更**: pg_node_attr マクロが付与されたフィールドは、クエリのハッシュ化や等価性判定において特別な扱いを受けるため、フィールドの追加・変更時には注意が必要です。
  • **実行時エラーとパフォーマンス**: wholerow 評価において var が NULL の場合や tupdesc が不整合な場合、実行時に予期せぬ動作を引き起こす可能性があります。また、slow モードでのNULLチェックはパフォーマンスに影響を与える可能性があります。
