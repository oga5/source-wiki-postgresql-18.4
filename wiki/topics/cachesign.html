CACHESIGN

概念

CACHESIGN は、PostgreSQL の GiST インデックス操作において、インデックスキーのシグネチャ（ビットベクトル）をメモリ上にキャッシュし、繰り返し発生するビット演算や距離計算のコストを削減するために使用されるデータ構造です。

構造体は以下の形状で定義されています。

typedef struct
{
	bool		allistrue;
	BITVECP		sign;
} CACHESIGN;
  • allistrue: すべてのビットがセットされている状態（全包含）を示すフラグ。
  • sign: シグネチャを保持するビットベクトル（BITVECP）へのポインタ。

なぜ重要か

GiST インデックスの検索やページ分割（picksplit）の処理では、多数のエントリ間でシグネチャの比較やハミング距離の計算が頻繁に行われます。これらの処理のたびにシグネチャを再計算したり、複雑なビット演算を愚直に実行すると、CPU オーバーヘッドが非常に大きくなります。

CACHESIGN を用いてシグネチャ情報を事前にキャッシュしておくことで、計算コストを大幅に削減し、インデックス操作を高速化できます。

関与するモジュール

  • module:contrib/pg_trgm (3連文字（trigram）インデックスの GiST サポート)
  • module:src/backend/utils/adt (テキスト検索（tsvector）の GiST サポート)

主要エンティティ

  • symbol:contrib/pg_trgm/trgm_gist.c#CACHESIGN: pg_trgm における GiST 検索や距離計算のキャッシュ用構造体。
  • symbol:src/backend/utils/adt/tsgistidx.c#CACHESIGN: tsvector の GiST ページ分割（picksplit）におけるハミング距離計算のキャッシュ用構造体。

エンドツーエンドの流れ

各モジュールにおいて、CACHESIGN は以下のような流れで利用されます。

1. **キャッシュの構築 (fillcache)**: インデックスキー（または SignTSVector）からシグネチャ情報を抽出し、CACHESIGN 構造体にコピーしてキャッシュを構築します。 2. **距離計算・比較 (hemdistcache)**: 構築された CACHESIGN キャッシュのペアを用いて、ハミング距離を計算します。 3. **上位処理での利用**:

  • **contrib/pg_trgm**: GiST 検索時に gtrgm_consistent_cache などの上位キャッシュ構造体と連携し、ノードごとのシグネチャを保持して検索時の一貫性判定や距離計算のオーバーヘッドを抑制します。
  • **src/backend/utils/adt**: gtsvector_picksplit 関数によるページ分割処理において、エントリ間のオーバーラップを最小化するためのハミング距離計算に利用されます。

*注意: キャッシュ配列全体の割り当てから解放に至るメモリコンテキストの厳密なライフサイクルなど、詳細なエンドツーエンドのフローの全容は完全には判明していません。*

重要な不変条件

  • sign メンバは、常に有効なビットベクトル（BITVECP）を指していなければなりません。
  • allistrue が true の場合、sign の内容はすべてのビットがセットされているもの（全包含）として扱われます。
  • キャッシュ操作関数（fillcache や hemdistcache）に渡されるシグネチャ長（siglen）は、実際の sign のサイズと厳密に一致していなければなりません。

よくある落とし穴

  • **定義の重複と非共通化**:

    • CACHESIGN 構造体は contrib/pg_trgm/trgm_gist.c と src/backend/utils/adt/tsgistidx.c の両方で全く同じ形状で独立して定義されています。片方を修正する際は、もう一方の実装にも影響がないか確認する必要があります。

  • **シグネチャ長（siglen）の不整合**:

    • siglen が実際のビットベクトルサイズと一致しない場合、hemdistcache での距離計算結果が不正になったり、メモリ破壊を引き起こす可能性があります。

  • **メモリ管理の依存性**:

    • sign フィールドが指すメモリのライフサイクルは呼び出し元に依存するため、キャッシュ配列の生存期間と palloc によるメモリ割り当てのライフサイクル管理に注意が必要です。

  • **ロジックへの影響**:

    • allistrue フラグの扱いを変更すると、検索時の適合性判定ロジック（gtsvector_consistent など）に影響を及ぼす可能性があります。
