SubscriptionRelState

概念

SubscriptionRelState は、PostgreSQL の論理レプリケーションにおいて、特定のサブスクリプションと個々のテーブル（リレーション）の紐付け状態（同期フェーズや進捗を示す LSN など）をメモリ上で管理するための構造体、およびそれに関連する状態管理の仕組みです。

なぜ重要か

論理レプリケーションでは、サブスクリプション全体だけでなく、個々のテーブル単位で同期の進捗（LSN）や現在の同期フェーズ（state）を追跡する必要があります。SubscriptionRelState 構造体や GetSubscriptionRelState 関数は、メモリ上での状態保持や、システムカタログ（pg_subscription_rel）からの状態取得を仲介し、正確なテーブル同期プロセスの制御を可能にするために重要です。

関与するモジュール

  • module:src/backend/catalog
  • module:src/include/catalog

主要エンティティ

  • **SubscriptionRelState** (symbol:src/include/catalog/pg_subscription_rel.h#SubscriptionRelState)

    • メモリ上でサブスクリプションとリレーションの紐付け状態（リレーション OID、LSN、同期フェーズ）を保持する構造体。

  • **GetSubscriptionRelState** (symbol:src/backend/catalog/pg_subscription.c#GetSubscriptionRelState)

    • 特定のサブスクリプションとテーブルの組み合わせにおける、現在の同期状態（srsubstate）と最終更新 LSN を pg_subscription_rel カタログ（および SUBSCRIPTIONRELMAP システムキャッシュ）から取得する関数。

エンドツーエンドの流れ

1. **状態の更新と保持**: テーブル同期プロセス（tablesync.c）において、同期フェーズの遷移や進捗の更新時に SubscriptionRelState のインスタンスが生成・更新され、メモリ上での状態保持やプロセス間通信に利用されます。 2. **状態の参照**: GetSubscriptionRelState が呼び出されると、以下の流れでカタログから状態が取得されます。

  • SubscriptionRelRelationId に対して AccessShareLock を取得し、競合を回避します。
  • SUBSCRIPTIONRELMAP システムキャッシュを relid と subid で検索します。
  • タプルが存在しない場合、LSN に InvalidXLogRecPtr を設定し、SUBREL_STATE_UNKNOWN を返します。
  • タプルが存在する場合、srsubstate から状態を取得し、srsublsn 属性から LSN を取得して返します。
  • キャッシュタプルを解放し、リレーションをクローズします。

※なお、状態遷移の全体的なエンドツーエンドのフロー（どのタイミングでどの状態に遷移するかなど）の全容は、提供された情報からは完全には明らかになっていません。

重要な不変条件

  • **LSN の追跡**: SubscriptionRelState の lsn フィールドは、同期の進捗を追跡する XLogRecPtr 型（access/xlogdefs.h で定義）です。
  • **状態値の範囲**: state フィールドは、永続化される状態（i, d, f, s, r）またはプロセス間通信専用の非永続状態（w, c）のいずれかを取ります。
  • **存在しないマッピングへの対応**: GetSubscriptionRelState において、指定された subid と relid のマッピングがカタログに存在しない場合は、エラーを発生させずに SUBREL_STATE_UNKNOWN を返し、LSN に InvalidXLogRecPtr を設定します。

よくある落とし穴

  • **永続化形式との混同**: SubscriptionRelState はメモリ内管理用の構造体であり、システムカタログ pg_subscription_rel の物理的な永続化形式である FormData_pg_subscription_rel と混同しないよう注意が必要です。
  • **LSN への直接アクセス**: lsn フィールドは CATALOG_VARLEN マクロの影響を受ける可能性があるため、直接アクセスするのではなく、提供されている API 経由で操作することが推奨されます。
  • **未定義の状態値**: state に未定義の文字が設定された場合の挙動や、状態遷移の完全なバリデーションルールは、提供された情報からは未検証です。
