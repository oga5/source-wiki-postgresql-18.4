create_logical_replication_slot

概念

create_logical_replication_slot は、PostgreSQLにおいて論理レプリケーションスロットを作成・初期化し、指定された出力プラグインの検証を行うための処理（およびそれを担う関数）です。バックエンドのレプリケーション管理機能に実装されており、SQLインターフェースや外部ツールから呼び出されます。

なぜ重要か

論理レプリケーションを開始するためには、WAL（Write-Ahead Log）のデコーディング開始位置や、使用する出力プラグインを定義した「論理レプリケーションスロット」が不可欠です。この処理は、スロットを安全に作成し、プラグインの検証やデコーディング開始位置（LSN）の特定を確実に行うための基盤を提供します。

関与するモジュール

  • module:src/backend/replication: バックエンド側でのスロット作成の実体（slotfuncs.c）を保持します。
  • module:src/bin/pg_basebackup: サブスクライバ作成ツール（pg_createsubscriber.c）からこの処理を呼び出します。

主要エンティティ

  • symbol:src/backend/replication/slotfuncs.c#create_logical_replication_slot: 論理レプリケーションスロットの作成、初期化、プラグイン検証を実際に行うバックエンドの静的（static）関数。
  • symbol:src/bin/pg_basebackup/pg_createsubscriber.c#create_logical_replication_slot: 上記バックエンド関数を呼び出す、または関連する処理を行うクライアント側のエンティティ（詳細な内部実装は提供されたコンテキストからは未詳）。

エンドツーエンドの流れ

クライアント（pg_createsubscriber）からバックエンド（slotfuncs.c）への通信や、SQLインターフェース（pg_create_logical_replication_slot）を経由した呼び出しの完全なエンドツーエンドのフローは、提供されたコンテキストからは完全には明らかになっていません。

バックエンド内部（slotfuncs.c）における処理の流れは以下の通りです： 1. **スロット作成**: ReplicationSlotCreate を呼び出し、共有メモリ上にスロットを確保します。エラー時の自動クリーンアップを容易にするため、初期状態は RS_EPHEMERAL（一時的）として作成されます。 2. **デコーディングコンテキスト作成**: CreateInitDecodingContext を呼び出し、出力プラグインの検証、および restart_lsn と xmin の更新を行います。 3. **開始位置検索**: find_startpoint が真の場合、DecodingContextFindStartpoint を呼び出してデコーディング開始位置を特定します。 4. **クリーンアップ**: FreeDecodingContext を呼び出してコンテキストを解放します。

※スロットの永続化（RS_PERSISTENT への昇格）は、本関数を呼び出した上位の関数で行われます。

重要な不変条件

  • **呼び出し時の状態**: 関数呼び出し時に MyReplicationSlot が NULL である必要があります（Assertにより検証されます）。
  • **権限**: 呼び出し元で適切な権限チェックが完了している必要があります。
  • **初期状態の安全性**: 作成直後のスロットは RS_EPHEMERAL（一時的）として作成され、処理が正常に完了して上位の呼び出し元に制御が戻るまでは永続化（RS_PERSISTENT への昇格）されません。

よくある落とし穴

  • **名前の競合**: 既存のスロット名と競合する場合、ReplicationSlotCreate 内でエラーが発生します。
  • **プラグインの検証失敗**: 指定された論理デコーディングプラグインが読み込めない、または初期化できない場合、CreateInitDecodingContext でエラーが発生します。
  • **トランザクションのロールバック**: 本関数の実行中にトランザクションが失敗またはロールバックされた場合、作成中のスロットは自動的に破棄されます。
