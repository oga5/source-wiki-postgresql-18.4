ITEM

概念

ITEM は、PostgreSQLの拡張モジュールである contrib/intarray および contrib/ltree において、ポーランド記法を用いたブール検索クエリを構成する個々の要素（演算子またはオペランド）を表現するための基本単位となる構造体です。

なぜ重要か

この構造体は、クエリの論理演算（AND, OR, NOT）や値の照合を評価するロジックの基盤として機能します。また、contrib/intarray と contrib/ltree の両モジュール間で同名の構造体が定義されており、メモリレイアウトの互換性を前提とした設計がなされているため、クエリ評価処理の共通性や保守において重要な役割を担っています。

関与するモジュール

  • module:contrib/intarray
  • module:contrib/ltree

主要エンティティ

  • **symbol:contrib/intarray/_int.h#ITEM**
  • type (int16), left (int16), val (int32) フィールドを持つ構造体。QUERYTYPE 構造体の一部として使用されます。
  • **symbol:contrib/ltree/ltree.h#ITEM**
  • type (int16), left (int16), val (int32) に加え、flag (uint8), length (uint8), distance (uint16) フィールドを持つ構造体。ltxtquery 構造体の一部として使用されます。

エンドツーエンドの流れ

1. **クエリの構築**: クエリの構文解析結果に基づき、ITEM の各フィールド（type, left, val など）が設定されます。 2. **メモリ配置**: ITEM は、QUERYTYPE や ltxtquery などの上位クエリ構造体内の可変長配列の一部として、メモリ上に連続して配置されます。 3. **クエリの評価**: _int_bool.c 内の execute 関数や checkcondition_arr、checkcondition_bit、checkcondition_gin などの評価ロジックが、ポーランド記法形式の ITEM 配列を走査します。 4. **条件判定**: 各 ITEM の type を参照して演算子かオペランドかを判別し、left（左側要素へのオフセットやインデックス）や val を用いて論理演算や値の照合を行います。

※構文解析から評価に至る具体的なエンドツーエンドの完全な制御フローの詳細は、提供されたコンテキストからは明らかになっていません。

重要な不変条件

  • **メモリレイアウトの互換性**: contrib/intarray と contrib/ltree の間での ITEM 構造体の互換性が前提となっています。
  • **サイズ計算の正確性**: ITEM は可変長配列（FLEXIBLE_ARRAY_MEMBER など）を伴う構造体の一部として扱われるため、メモリ確保時には COMPUTESIZE マクロ等を用いて適切なサイズを計算する必要があります。
  • **要素数の制限**: QUERYTYPEMAXITEMS マクロにより、メモリ割り当て可能な ITEM の最大数が制限されています。

よくある落とし穴

  • **メモリ確保時のサイズ計算ミス**: QUERYTYPE などのメモリ確保時に COMPUTESIZE マクロを使用しないと、バッファオーバーフローや不正なメモリ参照が発生する危険性があります。
  • **最大要素数の超過**: QUERYTYPEMAXITEMS を超える要素数を保持しようとした場合、メモリ割り当てやインデックス計算で境界違反が発生する可能性があります。
  • **モジュール間の変更不整合**: contrib/ltree と contrib/intarray の両方で ITEM が定義されているため、一方に変更を加える際にもう一方への影響を考慮しないと、互換性が崩れる恐れがあります。
  • **不正な type 値**: type フィールドに不正な値が設定された場合、評価ロジックが予期せぬ動作をする可能性があります。
