TableAmRoutine

概念

TableAmRoutine は、PostgreSQLのテーブルアクセス手法（Table AM: Table Access Method）を抽象化するための関数ポインタ群を保持する構造体です（src/include/access/tableam.h）。これにより、ストレージエンジンに依存しない汎用的なテーブル操作APIを提供し、heap 以外の物理ストレージ実装の追加を可能にします。

なぜ重要か

PostgreSQLにおいて、テーブルへのアクセス方法を抽象化し、異なるストレージエンジン（AM）をプラグイン可能にするためのコアインターフェースであるためです。table_ で始まるラッパー関数を経由して、Relation->rd_tableam に格納された本構造体の関数ポインタを呼び出すことで、具体的なストレージ実装（例：heapam.c）へ処理を委譲（ディスパッチ）する役割を担います。

主要エンティティ

  • **symbol:src/include/access/tableam.h#TableAmRoutine**: テーブルアクセス手法を抽象化する関数ポインタ群を保持する構造体。
  • **symbol:src/backend/access/table/tableamapi.c#GetTableAmRoutine**: 指定されたテーブルアクセス手法（TAM）のハンドラ関数を呼び出し、返された TableAmRoutine 構造体の取得と、必須コールバックの実装有無の検証を行う関数。
  • **symbol:src/backend/utils/cache/relcache.c#InitTableAmRoutine**: リレーションキャッシュ構築時に GetTableAmRoutine を呼び出して TableAmRoutine を初期化する関数（※詳細な実装は提供されたコンテキストからは未詳）。

エンドツーエンドの流れ

1. **初期化**: リレーションキャッシュの構築時（src/backend/utils/cache/relcache.c の InitTableAmRoutine など）に、リレーションのAMハンドラ（relation->rd_amhandler）を引数として GetTableAmRoutine が呼び出されます。 2. **ハンドラ実行と検証**: GetTableAmRoutine は OidFunctionCall0 を介してAMハンドラ関数を実行し、TableAmRoutine 構造体を取得します。その後、構造体の型チェック（IsA マクロ）および必須コールバック関数ポインタが NULL でないかの検証（Assert）を行います。 3. **ディスパッチと実行**: 呼び出し側は table_ プレフィックスのついたラッパー関数を経由して、Relation->rd_tableam（TableAmRoutine へのポインタ）の各関数ポインタ（scan_begin、scan_getnextslot、index_fetch_tuple など）を呼び出し、具体的なストレージ操作を実行します。

*注: 特定の関数呼び出しの連鎖や、システム全体における動的なエンドツーエンドの完全な制御フローは、提供されたコンテキストからはすべてが明らかになっていません。*

重要な不変条件

  • **型チェックの適合**: TableAmRoutine 構造体の type フィールドには、必ず T_TableAmRoutine が設定されていなければなりません（IsA マクロによる型チェックを通過する必要があります）。
  • **必須コールバックの存在**: TableAmRoutine 内のすべての必須関数ポインタ（スキャン、挿入、更新、削除、バキューム、統計情報取得など）は NULL であってはなりません（GetTableAmRoutine 内の Assert で検証されます）。
  • **関連機能の同時実装**: scan_set_tidrange を実装する場合、scan_getnextslot_tidrange も併せて実装する必要があります。
  • **ロックの前提**: 各操作は、呼び出し側が適切なリレーションロックを保持していることを前提とします。

よくある落とし穴

  • **直接呼び出しの禁止**: TableAmRoutine の構造体メンバーを直接呼び出すことは推奨されず、必ず table_ プレフィックスのついたラッパー関数を経由する必要があります。
  • **必須コールバックの未実装**: 新しいテーブルアクセス手法（AM）を追加する際、必須コールバックのいずれかが NULL のままであると、デバッグビルド環境において GetTableAmRoutine 内の Assert によりアサーション失敗が発生します。また、ハンドラ関数が NULL を返したり、型が正しくない場合は elog(ERROR) により処理が中断されます。
  • **TID範囲外の処理**: scan_set_tidrange は、TID がテーブルの範囲外である場合や、無効なオフセットを指定された場合でも適切に処理する必要があります。
