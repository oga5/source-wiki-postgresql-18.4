InternalGrant

概念

InternalGrant は、SQLの GRANT および REVOKE コマンドの解析結果を保持し、パーサーから実行エンジン（システムカタログ更新処理）へ情報を伝達するための中間データ構造（struct）です。権限付与コマンドの実行状態を保持し、DDLの追跡、監査ログ、レプリケーションなどの目的でコマンド構造を管理するために使用されます。

なぜ重要か

InternalGrant は、通常のパースツリー（GrantStmt）から名前解決（OID変換）を経て得られた情報を保持し、システムカタログ（pg_class等）のACLエントリを更新する実行エンジンへパラメータを伝播する役割を担います。また、CollectedCommand 構造体の共用体メンバ（grant.istmt）として保持されることで、階層的なコマンド管理やDDL追跡を可能にします。

エンドツーエンドの流れ

1. **変換**: パーサーが生成した構文木（GrantStmt）から、名前解決（OID変換）を経て InternalGrant 形式へ変換されます（src/backend/catalog/aclchk.c 内の ExecGrantStmt_oids 等が関与）。 2. **正規化**: all_privs が true の場合、privileges フィールドがオブジェクト型に基づき適切なビットマスクに正規化されます。 3. **実行と伝播**: 構築された InternalGrant は、ExecGrant_Relation、ExecGrant_Language_check、ExecGrant_Largeobject、ExecGrant_Parameter などの権限実行関数に引数として渡され、システムカタログの権限情報が更新されます。 4. **追跡**: 階層的なコマンド管理が必要な場合、CollectedCommand の一部として保持されます。

*注意: DDL追跡や監査ログ、レプリケーションにおける具体的なエンドツーエンドの統合フローの全容は、提供された情報からは完全には明らかになっていません。*

重要な不変条件

  • **オブジェクト型と列権限の整合性**: col_privs（列レベルの権限）は、objtype が OBJECT_TABLE である場合にのみ有効です。
  • **権限の正規化**: all_privs が true の場合、privileges フィールドはオブジェクト型に応じた適切なビットマスクに正規化される必要があります。
  • **コマンド種別の整合性**: CollectedCommand 内で使用される場合、type フィールドが適切なコマンド種別であることを前提とします。
  • **ポインタの有効性**: 権限チェックや実行処理の過程で、有効な InternalGrant ポインタが保持されている必要があります。

よくある落とし穴

  • **NULLポインタ**: InternalGrant ポインタが NULL の状態で実行関数に渡された場合、セグメンテーション違反が発生する可能性があります。
  • **変換エラー**: 変換元の GrantStmt が不正な場合、InternalGrant への変換処理でエラーが発生します。
  • **副作用**: all_privs フラグの操作時に privileges フィールドが書き換えられる副作用があるため、構造体の取り扱いには注意が必要です。
  • **モジュール間の結合**: 本構造体は utils/aclchk_internal.h と deparse_utility.h で共有されているため、定義を変更する際は src/backend/catalog/aclchk.c 内の関連する実行関数への影響を考慮する必要があります。
