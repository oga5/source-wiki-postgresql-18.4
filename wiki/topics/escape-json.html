escape_json

概念

escape_json は、PostgreSQLの複数の独立したモジュール（フロントエンドツール、テストモジュール、バックエンドなど）において、与えられた文字列をJSON形式の文字列リテラルとして安全にエスケープするために定義されている関数群です。

これらの関数は、入力されたヌル終端文字列を走査し、JSONの仕様で特別な意味を持つ文字（ダブルクォーテーション、バックスラッシュ、制御文字など）を適切なエスケープシーケンスに変換した上で、StringInfo バッファに追加する役割を持ちます。

なぜ重要か

JSON形式のデータを構築する際、制御文字やダブルクォーテーションなどの特殊文字が適切にエスケープされていないと、生成されたJSONが不正な形式になり、パーサーによる解析失敗やセキュリティ上の問題を引き起こす可能性があります。

escape_json は、以下の用途において、出力されるJSONデータの整合性と仕様準拠性を保証するために重要です。

  • **バックアップマニフェストの生成**: pg_combinebackup において、正しいJSONマニフェストを出力する。
  • **パーサーのテスト**: test_json_parser において、インクリメンタルパーサーのテスト用データを正しく構築する。

エンドツーエンドの流れ

システム全体を横断する統一された単一のエンドツーエンドの流れは存在せず、各モジュールが独立して escape_json を実装・利用しています。

各関数内における処理の流れは以下の通りです： 1. 対象の StringInfo バッファに、開始のダブルクォーテーション（"）を追加します。 2. 入力文字列（str）を1文字ずつ走査します。 3. 特殊文字を検出した場合、対応するエスケープシーケンスに変換してバッファに追加します。

  • \b, \f, \n, \r, \t はそれぞれ対応するシーケンスへ変換。
  • " は \" に、\ は \\ に変換。
  • ASCII値が32未満（スペース未満）の制御文字は、\u00xx 形式のUnicodeエスケープに変換。

    • 4. 通常の文字はそのままバッファに追加します。 5. 最後に、終了のダブルクォーテーション（"）を追加します。

※ 各モジュールにおいて、このエスケープ処理を呼び出す具体的な上位の業務フローや、バックエンド側（src/backend/utils/adt/json.c）における具体的なエンドツーエンドの流れは、提供された情報からは未だ明らかになっていません。

重要な不変条件

  • **二重引用符による囲み**: 出力されるエスケープ済みの文字列は、必ず開始と終了がダブルクォーテーション（"）で囲まれます。
  • **入力の前提条件**: 入力される str は有効なヌル終端文字列であり、出力先の buf は初期化済みの StringInfo バッファでなければなりません。
  • **空文字列の処理**: 入力文字列が空（""）の場合、バッファには ""（ダブルクォーテーション2つのみ）が追加されます。

よくある落とし穴

  • **重複実装とスコープ**: escape_json はそれぞれのファイル内で static 関数として個別に定義されています。これらは非常に類似した振る舞いを持ちますが、共通ライブラリとして一元管理されているわけではありません。そのため、一方の実装に変更を加えても、他方のモジュールの挙動には反映されません。
  • **メモリ不足時の挙動**: StringInfo バッファへの書き込み中にメモリ不足が発生した場合、StringInfo の内部マクロや関数（appendStringInfoCharMacro など）がエラーを発生させる可能性があります。
