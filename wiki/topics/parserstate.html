ParserState

概念

ParserState（または parser_state）は、PostgreSQLのコードベースにおいて、テキストやソースコードの解析（パース）処理中に、現在の解析位置や構文状態を保持・管理するために用いられるデータ構造です。

本トピックが対象とする範囲には、以下の2つの異なる文脈における状態管理構造体が含まれます。 1. **カスタムパーサーの解析状態管理**: PostgreSQLのテスト用カスタムパーサー（test_parser）において、解析対象のテキスト、長さ、および現在の走査位置を追跡する構造体。 2. **コード整形ツールの構文状態管理**: ソースコード整形ツール（pg_bsd_indent）において、インデントレベル、括弧のネスト、宣言の有無、コメントの配置などの構文解析状態を保持する構造体。

なぜ重要か

解析処理は本質的に状態を持つ（stateful）処理であり、トークンの切り出しや構文要素の判別を行うためには、現在どこまで解析が進んでいるか、あるいはどのような構文的文脈（宣言内、括弧内など）にあるかという情報を保持し続ける必要があります。

  • test_parser においては、ステートレスな関数呼び出し間で解析位置（pos）を維持し、一貫したトークン抽出を行うために不可欠です。
  • pg_bsd_indent においては、複雑なC言語ソースコードのインデントや整形ルールを適用するために、ネストレベルや直前のトークンなどの詳細な構文コンテキストを追跡する役割を担います。

エンドツーエンドの流れ

それぞれの文脈における処理の流れは以下の通りです。

test_parser における流れ

1. **初期化**: testprs_start 関数が呼び出され、palloc0 を用いて ParserState のメモリを確保し、解析対象テキストと長さを設定します。 2. **解析と更新**: testprs_getlexeme 関数が呼び出されるたびに、ParserState の pos を起点としてテキストを走査し、トークンを抽出して pos を進めます。 3. **終了処理**: 解析が完了すると、testprs_end 関数が呼び出され、pfree によって ParserState のメモリが解放されます。

pg_bsd_indent における流れ

1. **状態の入力**: 構文解析器（lexi.c や parse.c）がトークンを解析し、その結果や行の属性をグローバルな parser_state インスタンス（ps）の各フィールドに書き込みます。 2. **状態の参照と出力**: dump_line や pr_comment などの関数が ps の状態を参照し、適切なインデントや整形ルールを適用してコードを出力します。 3. **状態の保存・復元**: indent.c などにおいて、必要に応じて transient_state へのコピーや、state_stack を用いた状態の退避・比較が行われます。

※なお、これら2つのモジュール間を跨ぐような統合されたエンドツーエンドのフローは存在しません。

重要な不変条件

  • **test_parser の ParserState**:
  • buffer は解析対象のテキストを保持する有効なメモリ領域を指す必要があります。
  • pos は常に 0 以上 len 以下の範囲内でなければなりません。
  • **pg_bsd_indent の parser_state**:
  • グローバルインスタンス ps は、プログラム全体で共有される単一の状態として扱われます。
  • p_stack や il などの内部スタック構造は、スタックポインタ tos によって厳密に管理される必要があります。

よくある落とし穴

  • **test_parser**:
  • buffer が空（len が 0）の場合、解析ループは直ちに終了します。
  • メモリ管理は PostgreSQL の palloc / pfree に依存しているため、ライフサイクル外での参照や解放漏れに注意する必要があります。
  • **pg_bsd_indent**:
  • 古い BSD のコードベースに由来するため、現代的なカプセル化が行われておらず、グローバル変数として直接アクセスされます。フィールドを変更する際は、indent.c を含む全参照箇所への影響を考慮する必要があります。
  • p_stack (256), il (64), cstk (32), paren_indents (20) などの固定長配列のサイズを超えるネストが発生した場合の挙動は定義されておらず、スタックオーバーフローの危険性があります。また、procname は 100 文字に制限されています。
