stop_postmaster

概念

stop_postmaster は、PostgreSQLのユーティリティやテストスイートにおいて、実行中のPostgreSQLサーバー（Postmaster）プロセスを安全または強制的に停止するための関数です。主にアップグレード処理（pg_upgrade）や回帰テスト（pg_regress）のコンテキストで、サーバーのライフサイクル管理の一環として使用されます。

なぜ重要か

PostgreSQLのアップグレードやテスト実行時には、古いバージョンと新しいバージョンのクラスター、あるいはテスト用のテンポラリインスタンスを起動・停止する必要があります。stop_postmaster は、これらのプロセスを適切に終了させてデータディレクトリの一貫性を保ち、ポートやシステムリソースを解放するために重要です。特に異常終了時（atexit フックなど）に確実にサーバーを停止させることで、孤立したPostmasterプロセスが残るのを防ぎます。

エンドツーエンドの流れ

pg_upgrade における stop_postmaster の処理フローは以下の通りです。

1. **呼び出しのトリガー** アップグレード処理の各フェーズ（チェック、実行完了時、または atexit による異常終了時）で呼び出されます。 2. **対象クラスターの特定** グローバル構造体 os_info.running_cluster を参照し、現在実行中のクラスター（old_cluster または new_cluster）を特定します。 3. **停止モードの決定**

  • in_atexit 引数が true の場合：-m fast モード（高速停止）を選択し、ログ出力を抑制して pg_ctl を実行します。
  • in_atexit 引数が false の場合：-m smart モード（スマート停止）を選択し、ログを出力しつつ pg_ctl を実行します。

    • 4. **コマンド実行** exec_prog を介して pg_ctl stop コマンドを実行します。 5. **状態のリセット** コマンド実行後、os_info.running_cluster を NULL にリセットします。

※ src/test/regress/pg_regress.c における stop_postmaster の具体的なエンドツーエンドの流れや、両モジュールを跨ぐ統合的なエンドツーエンドフローの詳細は現時点では不明です。

重要な不変条件

  • **実行中クラスターの追跡**

    • pg_upgrade において、サーバーを停止するためには os_info.running_cluster が old_cluster または new_cluster を正しく指している必要があります。

  • **状態のクリア**

    • 停止処理が試行された後、コマンドの成否に関わらず os_info.running_cluster は必ず NULL にリセットされます。

よくある落とし穴

  • **状態の早期リセット**

    • pg_ctl による停止コマンドの実行成否に関わらず os_info.running_cluster が NULL にリセットされるため、停止に失敗した場合でも、システム上は「実行中のクラスターはない」とみなされる可能性があります。

  • **NULLチェックによるスキップ**

    • os_info.running_cluster が NULL の状態で呼び出された場合、関数は何もせずに即座にリターンします。意図せずサーバーが起動したままこの状態になると、プロセスが孤立する原因になります。

  • **pg_regress側の挙動**

    • pg_regress.c における stop_postmaster の不変条件や落とし穴については、提供されたコンテキストに情報がないため不明です。
