YYLTYPE

概念

YYLTYPE は、PostgreSQLの各種パーサー（構文解析器）において、ソースコード内のトークンの位置情報を追跡・保持するために使用されるデータ型です。

この型はモジュールごとに異なる定義を持つ場合があります。例えば、ECPGプリプロセッサ（src/interfaces/ecpg/preproc）においては const char *（定数文字列へのポインタ）として定義されており、現在のトークンの位置情報（ファイル名や行番号等に相当する文字列）を保持するために使用されます。

なぜ重要か

パーサーが構文解析を行う際、エラー報告や構文の文脈理解のために「どの位置のトークンを処理しているか」を正確に把握する必要があります。YYLTYPE は、字句解析器（lexer）と構文解析器（parser）の間でトークンの位置情報を共有するための標準的なインターフェースを提供し、正確なエラー箇所の特定やデバッグを可能にするため重要です。

エンドツーエンドの流れ

ECPGプリプロセッサにおける位置情報の処理フローは以下の通りですが、システム全体における完全なエンドツーエンドの制御フローはまだ十分に判明していません。

1. **位置情報の保持**: 字句解析器がトークンを読み進める際、YYLTYPE 型の変数（lookahead_yylloc や cur_yylloc など）に現在の解析位置（ファイル名や行番号等に相当する文字列へのポインタ）を格納します。 2. **位置情報の参照**: 構文解析器（src/interfaces/ecpg/preproc/parser.c など）は、エラー報告や構文解析の文脈処理において、これらの変数から位置情報を参照します。

なお、バックエンドパーサー（src/include/parser）やPL/pgSQL（src/pl/plpgsql/src）における具体的なエンドツーエンドの流れは、現時点では未定義です。

重要な不変条件

  • **ECPGにおける不変条件**:
  • YYLTYPE は const char * 型であるため、指し示す先の文字列は変更不可（読み取り専用）として扱われなければなりません。
  • パーサーの実行中、このポインタは現在の解析位置を指す有効なメモリ領域を指している必要があります。
  • **その他のモジュールにおける不変条件**:
  • 現時点では詳細な不変条件は不明です。

よくある落とし穴

  • **モジュール間での型定義の不一致**:

    • YYLTYPE はPostgreSQLのコンポーネントごとに異なる型として定義されている場合があります。例えば、ECPGプリプロセッサでは const char * ですが、バックエンドのパーサーなど他のコンポーネントでは int など別の型として定義されている可能性があります。モジュールをまたいでコードを移植したり参照したりする際は、対象モジュールにおける YYLTYPE の定義を確認する必要があります。

  • **ポインタの安全性（ECPG）**:

    • ECPGにおいてはポインタ型であるため、初期化されていない場合や無効なメモリ領域を指している場合にデリファレンスを行うと、未定義動作（セグメンテーションフォールトなど）を引き起こす危険性があります。
