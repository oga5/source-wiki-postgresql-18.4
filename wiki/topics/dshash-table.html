dshash_table

概念

dshash_table は、動的共有メモリ（DSA: Dynamic Shared Area）領域上に構築される、プロセス間で共有可能な並行オープンハッシュテーブルを表現する構造体です。

ヘッダーファイル（dshash.h）においては内部メンバが非公開の不透明な構造体（opaque type）として定義されており、ソースファイル（dshash.c）においてバックエンドローカルな状態を保持する具体的な構造体が定義されています。

なぜ重要か

  • **プロセス間共有と並行性**: 複数のプロセスから安全に並行アクセス可能なハッシュテーブルを提供します。内部的にパーティション分割された LWLock を使用することで、アクセス競合を最小化します。
  • **動的スケーラビリティ**: テーブルの負荷状況に応じて、動的にバケットの再配置（リサイズ）を行うことができます。
  • **カプセル化**: 不透明な構造体として定義されているため、利用側は内部の実装詳細（共有メモリポインタの解決やロック管理など）を意識せずに操作できます。

主要エンティティ

  • **symbol:src/include/lib/dshash.h#dshash_table**

  • **symbol:src/backend/lib/dshash.c#dshash_table**

    • バックエンドローカルな状態を保持する具体的な構造体。DSA領域へのポインタ（area）、パラメータ（params）、共有メモリ上の制御ブロック（control）、ローカルにキャッシュされたバケットポインタ（buckets）などを保持します。

エンドツーエンドの流れ

ハッシュテーブルの作成・アタッチから破棄に至る完全なライフサイクルの全容は、提供されたコンテキストからは未検証です。しかし、テーブル操作時の一連の流れは以下のように定義されています。

1. **操作の受付**: dshash_find や dshash_find_or_insert などのAPIに dshash_table のポインタが渡されます。 2. **バケットポインタの検証**: ensure_valid_bucket_pointers を呼び出し、他プロセスによってハッシュテーブルがリサイズされていないかを確認します。リサイズが検知された場合は、ローカルの buckets ポインタを最新の状態に更新します。 3. **パーティションロックの取得**: 操作対象のキーに対応するパーティションの LWLock を取得します。 4. **バケット内操作**: find_in_bucket や insert_into_bucket などの内部関数を通じて、共有メモリ上の要素（dshash_table_item）の検索・挿入・削除を行います。

重要な不変条件

  • **直接ポインタの禁止**: 共有メモリ上のデータにアクセスする際、プロセス固有の直接的な絶対メモリポインタを保持し続けてはならず、必ず dsa_pointer を介してアクセスする必要があります。
  • **ロックの取得順序**: デッドロックを防止するため、複数のパーティションロックを同時に取得する必要がある場合は、必ずパーティション番号の昇順で取得しなければなりません。
  • **関数ポインタの再指定**: ハッシュ関数や比較関数などの関数ポインタはプロセス間で共有できないため、各プロセスがテーブルにアタッチする際に再指定する必要があります。

よくある落とし穴

  • **頻繁なリサイズによるボトルネック**: テーブルのリサイズ（resize）時には、すべてのパーティションのロックを同時に取得する必要があります。そのため、頻繁にリサイズが発生する設計になっていると、システム全体のパフォーマンスが著しく低下します。
  • **シーケンシャルスキャン中のブロッキング**: シーケンシャルスキャン（dshash_seq_status を使用）の実行中は、対象パーティションのロックが保持され続けます。長時間のスキャン処理は、他プロセスによる並行操作を長時間ブロックする原因になります。
  • **不透明型への直接アクセス**: dshash_table の内部メンバは dshash.c 内に隠蔽されているため、外部から直接アクセスすることはできません。必ず提供されている API を介して操作する必要があります。
