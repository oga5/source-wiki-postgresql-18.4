chr

概念

PostgreSQLにおいて、chr は文脈に応じて異なる2つの役割を持ちます。

1. **正規表現エンジンにおける文字型の抽象化**: src/include/regex/regcustom.h において、正規表現エンジンが内部で処理対象とする文字型（pg_wchar のエイリアス）として定義されています。 2. **SQL/組み込み関数としての文字変換**: src/backend/utils/adt/oracle_compat.c に関連する、指定されたコードポイントに対応する文字を返すOracle互換の関数です。

なぜ重要か

  • **移植性と一貫性の確保**: 正規表現エンジン内部で環境依存の文字型を pg_wchar に統一（chr として抽象化）することで、プラットフォームに依存しない正規表現処理を可能にします。
  • **SQL互換性**: Oracle互換関数としての chr は、データベースユーザーが数値コードから文字を生成するための標準的な手段を提供します。

エンドツーエンドの流れ

正規表現エンジンにおける chr 型の利用フローは以下の通りです。

1. 正規表現のパースやマッチング処理（例: src/backend/regex/regc_color.c 内の pg_reg_getcolor や subcolor）において、文字データが chr 型として扱われます。 2. regcustom.h で定義された iscalnum や iscdigit などのマクロを介して、PostgreSQLのワイド文字処理関数（pg_wc_isalnum など）が呼び出され、文字の判定が行われます。

※ SQL関数としての chr のエンドツーエンドの呼び出しフローや、oracle_compat.c における具体的な実装の詳細は、提供されたコンテキストからは未詳です。

重要な不変条件

  • **ビット幅と範囲の整合性**: chr 型（pg_wchar）は32ビット幅を想定しており、CHRBITS は 32 に定義されています。
  • **最大値・最小値の制約**: CHR_MAX（0x7ffffffe）は、CHR_MAX - CHR_MIN + 1 が int 型に収まり、かつ CHR_MAX + 1 が chr 型の変数に収まる必要があります。

よくある落とし穴

  • **マクロの多重評価**: CHR_IS_IN_RANGE(c) マクロは引数 c を複数回評価する可能性があるため、副作用のある式（例: インクリメントを伴う式）を渡してはいけません。
  • **型変更時の影響**: chr 型の定義を変更すると、正規表現エンジンのメモリレイアウトやカラーマップの計算ロジック（MAX_SIMPLE_CHR など）に広範な影響を及ぼすため、整合性の維持に注意が必要です。
  • **名前の混同**: contrib/unaccent/generate_unaccent_rules.py に現れる chr() は Python の組み込み関数であり、PostgreSQL 内部の chr 定義や C 言語の関数とは無関係です。
