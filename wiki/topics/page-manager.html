Page Manager

概念

Page Manager（主に FreePageManager として実装）は、共有メモリ領域内においてページ単位（4096バイト / FPM_PAGE_SIZE）のメモリ割り当ておよび解放を管理する仕組みです。OS標準のメモリ管理が困難な共有メモリ環境下において、断片化を抑制しつつ効率的なメモリ管理を行うための基盤を提供します。

なぜ重要か

動的共有メモリ（DSM）や動的共有メモリ割り当て器（DSA）の管理において、アドレス空間が固定されない共有メモリ内でメモリを効率的にアロケートするために重要です。内部で相対ポインタ（relptr）を使用することで、アドレス空間の変動に対応しながらページ単位の管理を可能にします。

関与するモジュール

  • module:src/backend/utils/mmgr
  • module:src/include/utils

主要エンティティ

  • **symbol:src/include/utils/freepage.h#FreePageManager** (構造体)

    • ページ単位のメモリ管理を行うメインの構造体。共有メモリ内で使用されるため、内部では相対ポインタを用いて管理されます。

  • **symbol:src/backend/utils/mmgr/freepage.c#FreePageManagerGet** (関数)

    • 指定されたページ数（npages）のメモリ領域を FreePageManager から確保し、その開始ページ番号を返すAPIです。

エンドツーエンドの流れ

1. **初期化**: FreePageManagerInitialize を呼び出し、管理対象のベースアドレスを設定します。 2. **確保**: FreePageManagerGet を呼び出してメモリを確保します。

  • 内部関数 FreePageManagerGetInternal により、フリーリストまたは B-tree から適切な空き領域を探索・確保します。
  • 確保処理に伴う B-tree の構造変化を整理するため、FreePageBtreeCleanup を実行します。
  • クリーンアップ結果に基づき fpm->contiguous_pages を更新し、必要に応じて FreePageManagerUpdateLargest で最大連続ページ数を再計算します。
  • 成功時は fpm->free_pages を減算し、開始ページ番号を first_page に格納して true を返します。

    • 3. **解放**: FreePageManagerPut を介して領域をフリーリストへ戻します。その際、B-tree を用いて隣接する空き領域と統合（Coalesce）し、断片化を防止します。

※ 呼び出し元である dsm.c や dsa.c との具体的な連携や、ロック制御（lwlock.h）を交えた詳細な並行処理フローの全貌は、提供された情報からは完全には明らかになっていません。

重要な不変条件

  • **アライメント**: 管理対象領域は FPM_PAGE_SIZE（4096バイト）の倍数でアライメントされていなければなりません。
  • **デバッグ時の整合性検証**: FPM_EXTRA_ASSERTS が有効な場合、FreePageManagerGet の実行時に以下の不変条件が検証されます。
  • 確保成功時、fpm->free_pages が要求された npages 以上であること。
  • 確保後の fpm->free_pages が、管理下の全空きページの合計と一致すること。
  • fpm->contiguous_pages が、管理下の最大連続空きページ数と一致すること。

よくある落とし穴

  • **メモリ枯渇**: 管理対象領域が枯渇した場合、FreePageManagerGet は false を返します。呼び出し側で適切にハンドリングする必要があります。
  • **再帰的確保の懸念**: 確保処理に伴い B-tree のノードが再利用される際、内部で再帰的なメモリ確保が発生する可能性があります。
  • **アドレスの不固定性**: 共有メモリ内でアドレス空間が固定されないため、絶対ポインタを直接保持してはなりません。常に fpm_segment_base マクロ等を使用し、現在のセグメントベースからのオフセット（相対ポインタ）としてアドレスを計算する必要があります。
