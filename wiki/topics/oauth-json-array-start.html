oauth_json_array_start

概念

oauth_json_array_start は、PostgreSQLのクライアントライブラリ（libpq）およびOAuth認証関連モジュールにおいて、JSON解析中に配列の開始（[）を検出し、解析状態（json_ctx）のネストレベルを更新・検証するためのコールバック関数です。

なぜ重要か

OAuth認証フローにおいて、サーバーから返されるJSONレスポンスを安全かつ正確にパースするために重要です。この関数は、不正なJSON構造（トップレベルが配列である、文字列を期待するフィールドに配列が指定されている、ネストが深すぎるなど）を早期に検出し、解析エラーとして処理することで、不正な状態遷移を防ぐ役割を担っています。

エンドツーエンドの流れ

JSON解析の全体的なエンドツーエンドの流れ（どのようにJSONパーサーが起動され、どのような順序で他のコールバックが呼ばれるかなど）の詳細は、提供された情報からは完全には明らかになっていません。

現時点で判明している流れは以下の通りです： 1. JSONパーサーのセマンティクスアクション（sem.array_start）に oauth_json_array_start が登録されます（fe-auth-oauth.c および oauth-curl.c 内）。 2. パーサーが配列の開始（[）を検出すると、本関数が呼び出されます。 3. 本関数内で、現在の解析状態（json_ctx）の検証（ルートレベルチェック、ターゲットフィールドチェック、ネスト深さチェック）が行われます。 4. 検証に失敗した場合は oauth_json_set_error を呼び出してエラーを記録し、JSON_SEM_ACTION_FAILED を返します。成功した場合は JSON_SUCCESS を返します。

重要な不変条件

  • **有効な状態オブジェクト**: 引数 state は、有効な json_ctx 構造体を指していなければなりません。
  • **非ルートレベルでの開始**: 配列の開始は、ルートレベル（ctx->nested == 0）では許可されません。
  • **ターゲットフィールドとの排他**: ctx->target_field が設定されている（文字列値を期待している）場合、配列の開始は許可されません。
  • **ネスト制限**: インクリメント後のネストレベル（ctx->nested）は、MAX_SASL_NESTING_LEVEL を超えてはなりません。

よくある落とし穴

  • **トップレベル配列の拒否**: 解析対象のJSONの最外周が配列（例: [...]）である場合、ctx->nested が 0 の状態で呼び出されるため、解析エラー（JSON_SEM_ACTION_FAILED）となります。
  • **フィールド値としての配列制限**: 特定のターゲットフィールド（ctx->target_field）が設定されているコンテキストで配列が開始されると、文字列型ではないとみなされてエラーになります。
  • **ネストの深さ制限**: MAX_SASL_NESTING_LEVEL を超える複雑な階層を持つJSONはパースできません。
