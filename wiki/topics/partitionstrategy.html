PartitionStrategy

概念

PartitionStrategy は、PostgreSQLにおけるテーブルパーティショニングの分割手法（RANGE、LIST、HASHなど）を定義・識別するための概念です。テーブルがどのルールに基づいてデータを分割するかを決定する基盤となります。

なぜ重要か

テーブルが採用しているパーティショニング戦略を判別することは、クエリのパース、プランニング、および実行時におけるパーティションのルーティングにおいて不可欠です。get_partition_strategy のようなカプセル化された関数を通じて、キャッシュされたメタデータから戦略を迅速に取得することで、システム全体で一貫したパーティション制御を可能にします。

関与するモジュール

  • module:src/include/nodes
  • module:src/include/utils

主要エンティティ

  • **symbol:src/include/nodes/parsenodes.h#PartitionStrategy**: パーティショニング戦略を定義するシンボル。
  • **symbol:src/include/utils/partcache.h#get_partition_strategy**: 指定された PartitionKey から、そのテーブルが採用しているパーティション戦略（RANGE, LIST, HASHなど）を表す整数値を返すインライン関数。

エンドツーエンドの流れ

提供された情報からは、パーティションテーブルの作成からクエリ実行に至る完全なエンドツーエンドの流れは明らかになっていません。

現時点で判明している部分的な流れは以下の通りです： 1. パーティションテーブルのメタデータがロードされ、PartitionKey 構造体（PartitionKeyData）が構築・キャッシュされます。 2. 解析処理（例: src/backend/parser/parse_utilcmd.c）などの呼び出し元が、対象テーブルのパーティション戦略を必要とする際、get_partition_strategy を呼び出します。 3. get_partition_strategy は、引数として受け取った PartitionKey の strategy フィールドに直接アクセスし、その戦略値を呼び出し元に返します。

重要な不変条件

  • **有効なパーティションキー**: get_partition_strategy に渡される PartitionKey 引数は、有効な構造体を指している必要があります。
  • **リレーションオープン時の不変性**: パーティションキーはリレーションが開かれている間は不変であり、get_partition_strategy はキャッシュされた値を直接参照します。

よくある落とし穴

  • **NULLポインタのハンドリング**: get_partition_strategy 内部では引数 key の NULL チェックを行いません。そのため、呼び出し側で PartitionKey が構築済みであることを保証せずに NULL を渡した場合、セグメンテーション違反（Segmentation Fault）が発生します。
