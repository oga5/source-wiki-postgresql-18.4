DSM_IMPL_SYSV

概念

DSM_IMPL_SYSV は、PostgreSQLにおける動的共有メモリ（Dynamic Shared Memory: DSM）の実装方式の一つです。System V 共有メモリシステムコール（shmget, shmat, shmdt, shmctl）を利用して、共有メモリセグメントの作成、アタッチ、デタッチ、および破棄を行う低レベルな機構を提供します。

なぜ重要か

PostgreSQLのマルチプロセスアーキテクチャにおいて、プロセス間で動的にメモリを共有するためのポータブルな手段を提供するためです。オペレーティングシステム標準の System V 共有メモリ API を用いることで、プラットフォーム固有の共有メモリ管理を抽象化し、一貫したインターフェースで動的共有メモリを操作可能にします。

関与するモジュール

  • module:src/backend/storage/ipc
  • module:src/include/storage

主要エンティティ

  • **symbol:src/include/storage/dsm_impl.h#DSM_IMPL_SYSV**

    • System V 共有メモリを用いた動的共有メモリ実装を識別・有効化するための定義です。

  • **symbol:src/backend/storage/ipc/dsm_impl.c#dsm_impl_sysv**

    • System V 共有メモリセグメントに対する具体的な操作（作成、アタッチ、デタッチ、破棄）をカプセル化した静的関数です。

エンドツーエンドの流れ

1. **呼び出し**: src/backend/storage/ipc/dsm_impl.c 内の dsm_impl_op 関数から、指定された操作（dsm_op）とともに dsm_impl_sysv が呼び出されます。 2. **キー生成**: 渡された dsm_handle を key_t にキャストして、System V 共有メモリキーを生成します。 3. **識別子の取得**: *impl_private が空（NULL）の場合、shmget を呼び出して新しい識別子（ident）を取得し、キャッシュします。 4. **操作の実行**: dsm_op に応じて以下のシステムコールを呼び出します。

  • **DSM_OP_CREATE**: shmget でセグメントを新規作成し、shmat でプロセスのアドレス空間にマッピングします。
  • **DSM_OP_ATTACH**: shmctl(IPC_STAT) で既存セグメントのサイズを確認し、shmat でマッピングします。
  • **DSM_OP_DETACH**: shmdt でメモリをデタッチします。
  • **DSM_OP_DESTROY**: shmdt でデタッチした上で、shmctl(IPC_RMID) を用いてセグメントを破棄します。

    • 5. **結果の返却**: 処理が成功した場合は true を返し、失敗した場合は ereport でエラーを報告して false を返します。

*注意: dsm_impl_op より上位の呼び出し元や、PostgreSQL全体の動的共有メモリ管理ライフサイクルにおける完全なエンドツーエンドのフローは、提供されたコンテキストからは未詳です。*

重要な不変条件

  • **キーの制限**: dsm_handle から変換されたキーが IPC_PRIVATE と一致することは許容されません。一致した場合はエラーとして扱われます。
  • **識別子のキャッシュ管理**: impl_private が NULL の場合のみ、shmget を呼び出して新しい識別子を取得し、メモリを割り当てます。
  • **既存セグメントへのアクセス**: 既存セグメントにアクセスする際、shmget のサイズ指定は 0 でなければなりません。
  • **一貫したキー変換**: dsm_handle から key_t への変換時にビットの切り捨てが発生する可能性がありますが、同一のハンドルに対しては常に一貫した変換結果が得られなければなりません。

よくある落とし穴

  • **作成失敗時のリソースリーク**: DSM_OP_CREATE 中に shmget に成功したものの shmat によるマッピングに失敗した場合、作成したセグメントを shmctl(IPC_RMID) で明示的に破棄（クリーンアップ）しなければ、システム上に不要な共有メモリセグメントが残存してしまいます。
  • **キーの衝突**: dsm_handle から key_t への変換におけるビット切り捨てにより、異なるハンドルが同一の key_t にマッピングされる潜在的なリスクがあります（ただし、同一ハンドルに対する変換の一貫性は保証されます）。
