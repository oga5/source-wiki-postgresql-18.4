oauth_json_object_field_start

概念

oauth_json_object_field_start は、PostgreSQLのクライアントライブラリ（libpq）等において、OAuth認証に関連するJSONレスポンス（サーバーからのエラー応答など）を解析する際に使用されるコールバック関数です。JSONパーサーがオブジェクト内の新しいフィールドを検出したときに呼び出され、特定の関心のあるフィールド（status、scope、openid-configuration など）を識別して、その後の値の格納先を決定する役割を持ちます。

なぜ重要か

OAuth認証フローにおいて、サーバーから返されるエラー応答や設定情報を正しく処理することは、認証の成否やエラーハンドリングにおいて極めて重要です。この関数は、JSON解析中に必要なフィールド（エラーのステータス、スコープ、OpenID設定のURIなど）を正確に識別し、解析コンテキスト（json_ctx）の状態を適切に遷移させることで、後続の処理が正しいデータを取得できるようにします。

エンドツーエンドの流れ

提供された情報からは、詳細なエンドツーエンドの呼び出しフロー全体（JSONパーサーの初期化から解析完了、および oauth-curl.c 側の具体的な動作）は完全には明らかになっていません。

現時点で判明している fe-auth-oauth.c 側における部分的な流れは以下の通りです： 1. libpqがOAuth関連の処理（エラー応答の受信など）においてJSONパーサーを起動します。 2. JSONパーサーがオブジェクトのフィールドを検出すると、コールバックとして oauth_json_object_field_start が呼び出されます。 3. 本関数内で、現在のネストレベルがトップレベル（ctx->nested == 1）であることを確認します。 4. フィールド名（name）が既知の定数（ERROR_STATUS_FIELD、ERROR_SCOPE_FIELD、ERROR_OPENID_CONFIGURATION_FIELD）のいずれかと一致するか比較します。 5. 一致した場合、json_ctx 構造体内の target_field および target_field_name を更新し、後続の値解析処理で適切な変数に値が格納されるように準備します。 6. 関数は JSON_SUCCESS を返してパーサーに制御を戻します。

重要な不変条件

  • **ネストレベルの制限**: フィールドの識別処理は、ctx->nested が 1（トップレベルのオブジェクト）である場合にのみ実行されます。ネストされたオブジェクト内の同名フィールドは無視されます。
  • **戻り値の不変性**: 処理の成否やフィールドの一致成否に関わらず、この関数は常に JSON_SUCCESS を返します。
  • **状態のバインド**: 識別されたフィールドに対応する ctx->target_field のポインタ（ctx->status、ctx->scope、ctx->discovery_uri など）が正しく設定される必要があります。

よくある落とし穴

  • **ネストされたフィールドの無視**: 対象のフィールドがJSONのトップレベルではなく、ネストされたオブジェクト内にある場合、ctx->nested == 1 の条件に合致しないため、無視されます。
  • **未定義フィールドの挙動**: 定義されていないフィールド名が渡された場合、target_field は更新されず、エラーも発生せずに静かに無視されます。
  • **複数モジュールでの重複**: fe-auth-oauth.c と oauth-curl.c の両方に同名のシンボルが存在するため、それぞれのモジュールにおける役割の違いや、ビルド・リンク時の依存関係に注意する必要があります（ただし、oauth-curl.c 側の詳細な仕様は現時点では不明です）。
