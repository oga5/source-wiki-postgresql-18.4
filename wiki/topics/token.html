token

概念

token は、PostgreSQLのソースツリーにおいて複数の異なる文脈で使用される名前です。

主として、libpq-oauth モジュールにおける **OAuth 2.0 認可サーバーから返されるトークンレスポンス（またはエラーレスポンス）を保持するためのデータ構造**（struct token）を指します。一方で、コード整形ツール（pg_bsd_indent）やテストモジュール（oauth_validator）など、認証フローとは無関係な文脈でも同名のシンボルが存在します。

なぜ重要か

libpq-oauth における struct token は、OAuth 2.0 認証フローの成否を判定し、取得したアクセストークン情報を後続の処理へ受け渡すための中心的なデータ保持の役割を担っています。認可サーバーからのJSONレスポンス（成功時のアクセストークンやトークンタイプ、または失敗時のエラー詳細）をパースして格納するため、認証連携の成否に直接関わります。

エンドツーエンドの流れ

libpq-oauth における大まかな流れは以下の通りです：

1. 認可サーバーから返された JSON レスポンスを oauth_parse などの関数が解析します。 2. 解析結果が struct token 構造体にマッピングされます。

  • **成功時**：access_token および token_type に値が割り当てられます。
  • **失敗時**：err フィールド（token_error 構造体）にエラー詳細が設定されます。

    • 3. 認証フローのコンテキスト（async_ctx など）において、このトークン情報を基に後続の認証判断が行われます。

なお、oauth_validator や pg_bsd_indent における token シンボルが関与するエンドツーエンドの流れは、提供された情報からは未解明です。

重要な不変条件

  • libpq-oauth の struct token において、成功時には access_token および token_type が適切に割り当てられ、OAuth プロトコル上のエラー発生時には err フィールドが設定されます。
  • JSON パース失敗時や必須フィールドが欠落している場合、構造体内の各フィールドは未初期化または空（NULL 等）になります。
  • メモリ割り当てに失敗した場合、各文字列ポインタは NULL になる可能性があります。

他のモジュールにおける token シンボルの不変条件は未解明です。

よくある落とし穴

  • **メモリリーク**: struct token 内の access_token などの文字列ポインタは、不要になった際に free_token 関数を用いて適切に解放する必要があります。これを怠るとメモリリークの原因になります。
  • **名前の混同**: libpq-oauth の token 構造体は、pg_bsd_indent や contrib/ 配下など、他のモジュールに存在する同名の token シンボル（文字列解析用など）とは全く無関係です。コードの保守や修正の際には、これらを混同しないよう注意が必要です。
