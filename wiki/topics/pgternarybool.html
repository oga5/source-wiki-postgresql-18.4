PGTernaryBool

概念

PGTernaryBool は、通常の真偽値（True/False）に「不明（Unknown）」の状態を加えた3値論理を表現するための列挙型（enum）です。値が未確定である初期状態や、接続設定のデフォルト値、認証状態などを管理するために使用されます。

なぜ重要か

データベース接続や認証処理において、状態が「真」か「偽」かまだ決定していない、あるいは取得できていない「未確定（Unknown）」の状態を明示的に区別することが重要です。PGTernaryBool を使用することで、未初期化状態やデフォルト状態を安全に表現し、誤った真偽判定を防ぐことができます。

エンドツーエンドの流れ

具体的なエンドツーエンドの制御フローや、値がどのように遷移するかについての詳細な情報は、提供されたコンテキストからは不明です。

現時点で判明している利用箇所は以下の通りです：

  • **libpq-oauth 側**: src/interfaces/libpq-oauth/oauth-curl.c 内の init_successful 変数（認証の初期化成功フラグなど）の型として使用されます。
  • **libpq 側**: src/interfaces/libpq/libpq-int.h 内の PGconn 構造体のメンバーである default_transaction_read_only および in_hot_standby の型として使用されます。

重要な不変条件

  • **有効な値の制限**: 保持する値は、必ず定義された3つの列挙子（PG_BOOL_UNKNOWN = 0, PG_BOOL_YES, PG_BOOL_NO）のいずれかでなければなりません。
  • **初期状態の表現**: PG_BOOL_UNKNOWN (0) が初期値または未確定状態を示すために使用されます。

よくある落とし穴

  • **暗黙的なキャストと論理演算**: 整数値としてキャストされた場合、0は PG_BOOL_UNKNOWN を意味します。そのため、通常のC言語の真偽値判定（0が偽、非0が真）をそのまま適用すると意図しない挙動になるため、明示的な比較が必要です。
  • **範囲外の値の代入**: 整数キャスト時に 0, 1, 2 以外の値が代入されると、予期せぬ動作を引き起こす可能性があります。
  • **公開APIとの境界**: 本型は libpq 内部でのみ使用されるプライベートな定義であり、公開 API である libpq-fe.h には含まれていません。
  • **複数モジュール間の共有**: 本型は libpq-int.h と oauth-utils.h の双方で定義・共有されているため、型定義を変更する際は双方のモジュールへの影響を確認する必要があります。
