ExplainState

概念

ExplainState は、PostgreSQLの EXPLAIN コマンド実行時において、実行計画（クエリプラン）の走査、出力フォーマット（TEXT, JSON, YAML, XML等）の設定、出力バッファ、および走査状態を一元的に保持・管理するためのメイン構造体です。

なぜ重要か

EXPLAIN コマンドの実行中、クエリプランの走査から結果の整形・出力に至るまでの状態を管理し、一貫したフォーマットで出力するために不可欠です。また、外部データラッパー（FDW）や拡張モジュール（auto_explain、pg_overexplain など）が、独自のオプションやノード固有の情報（ファイルパスやスキャン情報など）を EXPLAIN 出力に追加するための共通インターフェース（フックやコールバック）としても機能します。

エンドツーエンドの流れ

1. **初期化**: EXPLAIN コマンド実行時に NewExplainState() により ExplainState インスタンスがデフォルト値で生成されます。 2. **オプション解析**: ParseExplainOptionList() 等により、ANALYZE や VERBOSE などのオプションフラグが構造体内に設定されます。 3. **プラン走査と出力生成**: ExplainOnePlan や ExplainPrintPlan が実行計画ツリーを再帰的に走査します。この際、ExplainState のフォーマット設定（format）やインデントレベル（indent）を参照・更新しながら、出力バッファ（str）に結果を蓄積します。 4. **拡張機能・FDWの介入**:

  • 走査中に explain_per_node_hook 等のフックが呼び出され、拡張モジュールが情報を追加します。
  • 外部テーブルの走査時には、FdwRoutine の ExplainForeignScan コールバックを介して、FDW（例: file_fdw の fileExplainForeignScan）が ExplainState に固有の情報を書き出します。

    • 5. **出力**: 最終的に、蓄積されたバッファ（str）の内容が出力先（標準出力やログなど）へ書き出されます。

※詳細なメモリ確保・解放の厳密なタイミングや、エラーハンドリング時のクリーンアップ処理の全貌は、提供された情報からは完全には特定されていません。

重要な不変条件

  • **一貫したフォーマット維持**: 実行計画の走査中、保持するフォーマット設定（format）は一貫して維持される必要があります。
  • **メモリ管理**: str フィールド（StringInfo）は動的に管理されており、メモリリークを防ぐため適切なライフサイクル管理が必要です。
  • **並列クエリの整合性**: 並列クエリ実行時には、ワーカーごとの状態を示す workers_state が適切に設定されている必要があります。

※その他の詳細な不変条件やライフサイクル規則については、提供された情報からは完全には特定されていません。

よくある落とし穴

  • **領域外アクセス**: 拡張機能で extension_state 配列を利用する際、割り当てられたサイズ（extension_state_allocated）を超えたアクセスを行わないよう注意が必要です。
  • **前方宣言による制限**: fdwapi.h などのヘッダーでは ExplainState が前方宣言のみで提供されているため、構造体の詳細なメンバに直接アクセスするには explain_state.h をインクルードする必要があります。
  • **互換性の影響**: 本構造体は多くの拡張機能（contrib 配下のモジュールなど）で頻繁に利用されるため、構造体のフィールド変更は後方互換性に大きな影響を与える可能性があります。
