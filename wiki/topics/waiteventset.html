WaitEventSet

概念

WaitEventSet は、複数のイベント（ソケット、ラッチ、ポストマスターの終了など）を単一のインターフェースで効率的に待機するための抽象化レイヤーを提供する構造体です。OS 固有のイベント通知プリミティブ（epoll, kqueue, poll, WaitForMultipleObjects）をラップし、プラットフォーム間の差異を吸収します。

ヘッダーファイル（waiteventset.h）では不透明な（opaque）構造体として宣言され、実装ファイル（waiteventset.c）で具体的なメンバが定義されています。

なぜ重要か

PostgreSQL の各バックエンドプロセス（postmaster, syslogger, walreceiver など）や拡張モジュール（postgres_fdw など）が、複数の I/O イベントやシグナル、プロセス間通信（ラッチ）を効率的に多重待ちできるようにするために重要です。これにより、OS 固有のシステムコールのオーバーヘッドを削減し、プラットフォーム非依存の統一された API を提供します。

エンドツーエンドの流れ

1. **初期化**: 呼び出し元が CreateWaitEventSet を呼び出してセットを確保し、ResourceOwner に紐付けてリソースのライフサイクルを管理します。 2. **登録**: AddWaitEventToSet を通じて、監視対象のイベント（ラッチやソケットなど）をセットに追加します。この際、OS 固有のシステムコール（epoll_ctl など）でカーネルに登録されます。 3. **待機**: WaitEventSetWait を呼び出すと、イベント発生またはタイムアウトまでプロセスがブロック（停止）します。内部的には OS 固有の待機プリミティブ（epoll_wait, kevent, poll, WaitForMultipleObjects など）が呼び出されます。 4. **復帰と処理**: イベントが発生すると、発生したイベント情報が WaitEvent 構造体の配列として返され、呼び出し元がこれを走査して処理を行います。 5. **解放**: FreeWaitEventSet を呼び出し、リソースをクリーンアップします。

重要な不変条件

  • nevents（登録されたイベント数）は常に nevents_space（確保されたスペース）以下でなければなりません。
  • WL_LATCH_SET が指定された場合、latch ポインタと latch_pos が正しく設定されていなければなりません。
  • 待機対象のイベントには、WL_LATCH_SET や WL_SOCKET_READABLE などの有効なビットマスクを指定する必要があります。
  • コンパイル時に選択されたプラットフォーム固有のメンバ（epoll_fd 等）が、対応する OS プリミティブと整合している必要があります。

よくある落とし穴

  • **動的削除の未サポート**: 現在の設計では、既存の WaitEventSet からイベントを動的に削除する機能は提供されていません。
  • **プラットフォーム依存性**: コンパイル時のフラグ（WAIT_USE_EPOLL 等）によって内部実装が大きく異なります。特に Windows 環境（WaitForMultipleObjects を使用）は Unix 系環境と実装が大きく異なるため、デバッグ時は対象環境のプリミティブを確認する必要があります。
  • **リソース管理**: メモリリークを防ぐため、ResourceOwner を用いた適切なスコープ管理と FreeWaitEventSet による解放を徹底する必要があります。
  • **タイムアウトのオーバーヘッド**: タイムアウト指定はオーバーヘッドを伴うため、可能な限り避けることが推奨されます。
