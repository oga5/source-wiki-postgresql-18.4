JsonbIterator

概念

JsonbIterator は、PostgreSQLにおいてバイナリ形式のJSONデータ（jsonb）を再帰的に走査するためのメモリ内イテレータ構造体です。JsonbContainer 内に格納された要素（オブジェクトや配列など）を順次探索し、ネストされた構造に対しても親子関係を保持しながら走査を行う仕組みを提供します。

なぜ重要か

jsonb は、ディスク上およびメモリ内での圧縮効率を優先したバイナリ表現（JsonbContainer）を採用しています。このため、データ構造に直接ランダムアクセスしたりパースしたりする処理は複雑になります。

JsonbIterator を使用することで、複雑なバイナリ表現を抽象化し、各種組み込み関数、インデックス（GIN）、外部言語プロシージャ（PL/Perl, PL/Python）などの呼び出し元に対して、統一的かつ安全な要素走査インターフェースを提供できます。

関与するモジュール

  • module:src/backend/utils/adt
  • module:src/include/utils

主要エンティティ

  • **symbol:src/include/utils/jsonb.h#JsonbIterator**: 走査状態（対象コンテナ、要素数、現在のインデックス、オフセット、親イテレータへのポインタなど）を保持する構造体。
  • **symbol:src/backend/utils/adt/jsonb_util.c#JsonbIteratorInit**: 指定された JsonbContainer から走査を開始するための JsonbIterator を初期化する関数。

エンドツーエンドの流れ

1. **初期化**: 呼び出し元（jsonb.c、jsonb_gin.c、PL/Perl、PL/Pythonなど）が、走査対象の JsonbContainer を指定して JsonbIteratorInit を呼び出します。 2. **内部イテレータの生成**: JsonbIteratorInit は内部関数 iteratorFromContainer を呼び出し、親イテレータ（parent）を NULL としたルートレベルの JsonbIterator を生成・初期化して返却します。 3. **要素の走査**: 呼び出し元はイテレータを進め（通常は JsonbIteratorNext を使用）、curIndex を 0 から nElems - 1 まで進めながら要素を順次取得します。 4. **ネストされたコンテナの処理**: 走査中にネストされたコンテナに遭遇した場合、内部で新しい JsonbIterator が palloc され、元のイテレータが parent ポインタとして関連付けられます。これにより、再帰的なツリー構造の走査が実現します。

*注意: JsonbIteratorNext の具体的な内部アルゴリズムや、走査完了時のクリーンアップ処理の全体像は、提供されたソース抜粋からは完全には特定されていません。*

重要な不変条件

  • **アライメント**: Jsonb 構造体は4バイト境界でアライメントされている必要があります。
  • **インデックス範囲**: curIndex は常に 0 から nElems - 1 の範囲内で推移します。
  • **親子関係の追跡**: ネストされたコンテナを走査する際、新しく生成されたイテレータの parent フィールドは、呼び出し元の親イテレータを正しく指し示さなければなりません。

*注意: JEntry のフラグ（JENTRY_HAS_OFF）に基づくオフセット計算の正確な境界条件や、可変長データ領域へのアクセス時におけるパディング管理の厳密なルールは、提供された情報からは完全には特定されていません。*

よくある落とし穴

  • **メモリリーク**: ネストされたコンテナを処理する際、内部で新しいイテレータが palloc されます。呼び出し元は、生成されるイテレータの親子関係を適切に管理し、メモリリークを防ぐ必要があります。
  • **スタックオーバーフロー**: 非常に深くネストされた複雑な JSON 構造を走査する際、再帰的な処理によってスタックオーバーフローを引き起こすリスクがあります。
  • **メモリ破壊**: 可変長データ領域へのアクセス時、パディングを考慮したアライメント管理や JEntry のオフセット計算を誤ると、メモリ破壊を招く可能性があります。
