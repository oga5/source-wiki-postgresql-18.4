ParallelSlot

概念

ParallelSlot は、PostgreSQLのフロントエンドツールおよびユーティリティ（pg_dump、pg_restore、pg_amcheck など）において、並列処理を管理するために用いられる構造体です。

この名前を持つ構造体には、役割の異なる2つの定義が存在します。

1. **フロントエンド共通の接続管理 (src/include/fe_utils/parallel_slot.h)**: 並列処理用のデータベース接続（PGconn）を保持し、クエリ実行状態（inUse）や結果処理用のコールバック（handler）を管理するコンテナ。 2. **pg_dump/pg_restore 内部のワーカー管理 (src/bin/pg_dump/parallel.c)**: 並列処理を実行する各ワーカープロセスまたはスレッドの状態（workerStatus）、通信用パイプ、および識別情報（pid や hThread）を保持する構造体。

なぜ重要か

フロントエンドツールがデータベースに対して並列にクエリを実行したり、ダンプ/リストア処理を複数のワーカーに分散して高速化したりする際に、各並列実行コンテキストの状態、通信、および接続を安全かつ効率的に追跡・制御するために不可欠なデータ構造だからです。

関与するモジュール

  • module:src/bin/pg_dump
  • module:src/include/fe_utils

主要エンティティ

  • symbol:src/bin/pg_dump/parallel.c#ParallelSlot : pg_dump 内でワーカープロセス/スレッドの状態や通信パイプを管理する具体的な構造体定義。
  • symbol:src/bin/pg_dump/parallel.h#ParallelSlot : parallel.c 外に対して不透明ポインタ（opaque struct）として開示される宣言。
  • symbol:src/include/fe_utils/parallel_slot.h#ParallelSlot : フロントエンド共通で PGconn 接続と結果ハンドラを管理する構造体定義。

エンドツーエンドの流れ

1. **fe_utils 接続管理の流れ**:

  • ParallelSlotArray 内で複数の ParallelSlot が管理されます。
  • 各スロットは PGconn を保持し、クエリ発行時に inUse が設定され、結果処理用の handler と handler_context が登録されます。
  • クエリ結果が返されると、登録された handler コールバックが実行されます。

    • 2. **pg_dump ワーカー管理の流れ**:

  • リーダープロセスは DispatchJobForTocEntry を通じてジョブを割り当て、パイプ経由でコマンドを送信します。
  • ワーカーは RunWorker ループ内でパイプを監視し、リーダーからの指示に従って ArchiveHandle を使用した処理を実行します。
  • 処理完了後、ワーカーはパイプ経由で結果を返送し、リーダーが ListenToWorkers でこれを回収します。

※なお、これら2つの ParallelSlot 構造体がシステム全体でどのように相互作用し、統合された単一の制御フローを形成しているかという詳細なエンドツーエンドの連携フローは、提供された情報からは完全には明らかになっていません。

重要な不変条件

  • **pg_dump 側**:
  • workerStatus が WRKR_WORKING の場合のみ、callback および callback_data が有効です。
  • parallel.h で定義される ParallelSlot は不透明ポインタであり、その内部メンバへの直接アクセスは parallel.c 内に限定されなければなりません。
  • **fe_utils 側**:
  • connection に対してコマンドやクエリを発行する前に、必要に応じて handler および handler_context を登録する必要があります。未設定の場合、これらは NULL である必要があります。

よくある落とし穴

  • **プラットフォーム依存の制限**:
  • Windows環境では fork が利用できないため、スレッドとソケットを用いたパイプエミュレーション（pgpipe）に依存します。また、MAXIMUM_WAIT_OBJECTS の制限により、並列数は最大 64 に制限され、これに伴い ParallelSlot の配列サイズも制限されます。
  • Windows環境では FD_SETSIZE の制限が並列接続数に影響を与えるため、スロット数設計時に注意が必要です。
  • **メモリ・リソース管理**:
  • fe_utils の handler 内での PQclear の実行有無など、呼び出し元とハンドラ間での結果解放の責務分担に注意する必要があります。
  • シグナルハンドラ内での ParallelSlot へのアクセスは制限されており、安全な関数のみを使用する必要があります。
