HTAB

概念

HTAB は、PostgreSQL の動的ハッシュテーブル（dynahash）を管理するための制御構造体です。

外部モジュールに対しては、src/include/utils/hsearch.h において不透明な（opaque）構造体 typedef struct HTAB HTAB; として宣言されており、カプセル化されています。一方、src/backend/utils/hash/dynahash.c 内部で具体的な構造体メンバが定義されており、ローカルメモリおよび共有メモリ上のハッシュテーブルの状態（セグメント、ハッシュ関数、比較関数、メモリコンテキスト、各種制御フラグなど）を保持します。

なぜ重要か

HTAB は、メモリ内での効率的なデータ検索、挿入、削除を行うための基盤を提供します。

  • **競合の低減**: keysize や ssize などの固定値を構造体内に保持することで、共有メモリ上のヘッダー（HASHHDR）へのアクセス頻度を抑え、プロセス間の競合を最小化します。
  • **高い汎用性**: 共有メモリとローカルメモリの両方に対応しており、contrib/dblink や contrib/pg_stat_statements などの多くの拡張モジュールで幅広く利用されています。

主要エンティティ

  • **symbol:src/include/utils/hsearch.h#HTAB**: 外部に公開される不透明な構造体定義。
  • **symbol:src/backend/utils/hash/dynahash.c#HTAB**: 内部状態（HASHHDR へのポインタ、セグメントディレクトリ、関数ポインタ、メモリコンテキストなど）を保持する具体的な構造体定義。

エンドツーエンドの流れ

提供されたソースコードの定義から、一般的な利用の流れは以下の通りです。

1. **初期化**: HASHCTL 構造体で設定を指定し、hash_create() を呼び出して HTAB を生成・初期化します。 2. **操作**: hash_search() 等の関数に HTAB のポインタを渡すことで、指定されたキーに対する検索、挿入、削除を行います。この際、HTAB 内に登録された関数ポインタ（hash, match, keycopy, alloc）が動的に呼び出されます。 3. **スキャン**: 全要素を走査する場合、hash_seq_init() および hash_seq_search() を用いてシーケンシャルスキャンを行います。 4. **破棄**: hash_destroy() を呼び出すことで、HTAB に関連するメモリリソースを解放します。

※なお、バケット分割のトリガー条件や、共有メモリ上での排他制御の厳密な同期手順など、詳細な内部フローのすべては提供されたコンテキストからは明らかになっていません。

重要な不変条件

  • **カプセル化の維持**: 呼び出し側は HTAB の内部メンバに直接アクセスせず、必ず提供される API（hsearch.h）を介して操作しなければなりません。
  • **セグメントサイズ**: 構造体内の ssize（セグメントサイズ）は必ず2のべき乗でなければなりません。
  • **共有メモリの設定**: 共有メモリ上で使用する場合は、作成時に HASH_SHARED_MEM フラグを指定する必要があります。
  • **挿入禁止状態**: frozen フラグが true の場合、テーブルへの新規挿入は許可されません。
  • **ユーザーデータの配置**: ユーザーデータは HASHELEMENT の直後に配置され、ハッシュキーはユーザーデータの先頭に存在する必要があります。

よくある落とし穴

  • **動的拡張の制限**:
  • isfixed が true に設定されている場合、テーブルの拡張は行われません。
  • 共有メモリ上のテーブルは、hash_create 時に指定した初期サイズを超えて動的にディレクトリを拡張することができません。
  • **スキャン中の操作**: シーケンシャルスキャン中に要素を削除することは可能ですが、スキャン中に他の要素を追加・削除すると未定義の挙動を引き起こす可能性があります。
  • **初期化フラグの欠如**: 共有メモリ使用時のフラグ指定ミスや、必須である HASH_ELEM フラグの欠如は、予期せぬ動作を引き起こす原因となります。
