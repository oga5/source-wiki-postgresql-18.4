NODE

概念

NODE は、PostgreSQLの複数のモジュールにおいて、クエリの構文解析や最適化、クエリツリーの操作を行うために定義されている内部的な構造体（struct）です。

名前は同一ですが、モジュールごとに異なる定義と役割を持っています。

  • **contrib/intarray および contrib/ltree**: クエリ文字列の構文解析中にトークンを一時的に保持し、演算子の優先順位に基づいたポーランド記法（後置記法）への変換を行うための単方向連結リストの要素として使用されます。
  • **src/backend/utils/adt (tsquery_cleanup.c)**: 全文検索クエリ（tsquery）をツリー構造として表現し、不要なノード（NOT 演算子やストップワード）の除去や最適化を再帰的に行うために使用されます。

なぜ重要か

クエリの解析や最適化といった処理において、フラットな入力データ（文字列や配列）を一時的に操作しやすい構造（連結リストや二分木）に変換して処理するために不可欠なデータ構造だからです。

特に、contrib/intarray や contrib/ltree では、ユーザー入力をポーランド記法（ITEM 配列）に変換する際の中間表現として機能します。また、tsquery_cleanup.c では、ストップワードの除去やフレーズ演算子の距離再計算といった複雑なクエリ最適化ロジックを再帰的に適用するための基盤となります。

エンドツーエンドの流れ

モジュールごとに処理の流れが異なります。

1. contrib/intarray および contrib/ltree の場合

1. クエリ文字列の解析プロセス（ltxtq_in や makepol など）において、トークンが順次読み込まれます。 2. palloc を介して NODE 構造体が動的に割り当てられ、トークン情報（タイプや値など）が格納されます。 3. NODE は next ポインタを介して単方向連結リスト（スタック）を形成します。 4. 構築されたリストは、演算子の優先順位に従って走査され、最終的なバイナリ形式である ITEM 配列（ポーランド記法）へと変換されます。 5. 変換後、これらの一時的な NODE リストは破棄されます。

2. src/backend/utils/adt (tsquery_cleanup.c) の場合

1. maketree() 関数により、フラットな QueryItem 配列から NODE を用いた二分木ツリー構造が構築されます。 2. 構築されたツリーに対して、clean_NOT_intree や clean_stopword_intree などの関数が再帰的に適用され、不要なノードの除去やフレーズ演算子の距離再計算が行われます。 3. 処理完了後、plaintree() 関数によって、最適化されたツリー構造から再びフラットな QueryItem 配列へと書き戻されます。

重要な不変条件

  • **連結リストの終端**: contrib/intarray および contrib/ltree の NODE において、リストの終端を示す next ポインタは必ず NULL でなければなりません。
  • **ツリーの構造**: tsquery_cleanup.c の NODE において、各ノードは QueryItem（valnode）を保持するか、あるいは演算子として左右の子ノード（left, right）を保持します。
  • **メモリ管理**: これらすべての NODE 構造体は、永続的なクエリ保存には使用されず、解析やクリーンアップ処理中の一時的なメモリ（palloc による割り当て）としてのみ存在します。

よくある落とし穴

  • **名前の重複と混同**: NODE という名前の構造体は、contrib/intarray、contrib/ltree、および tsquery_cleanup.c でそれぞれ独立して定義されています。これらは互いに直接の関連性を持たない別個の定義であるため、開発や修正の際に混同しないよう注意が必要です。特に contrib/intarray と contrib/ltree は定義が類似しているため、一方を変更する際は他方への影響も考慮する必要があります。
  • **スタックオーバーフロー**: tsquery_cleanup.c などの再帰的なツリー走査や、クエリ解析時の再帰処理において、クエリの階層が極端に深い場合にスタックオーバーフローを引き起こす危険性があります。これを防ぐために check_stack_depth() によるスタック保護を適切に適用する必要があります。
  • **複雑なポインタ操作と距離伝播**: tsquery_cleanup.c におけるストップワード除去時のフレーズ演算子の距離（ladd, radd）の伝播ロジックは、NODE のポインタ操作に強く依存しており非常に複雑です。不適切な操作を行うと、クエリの整合性が失われる可能性があります。
