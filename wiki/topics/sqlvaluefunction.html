SQLValueFunction

概念

SQLValueFunction は、SQL標準で定義されている値関数（CURRENT_USER、CURRENT_DATE など）を表現および評価するためのデータ構造です。構文解析やプランニング段階のノード表現から、実行時の式評価ステップにわたる複数のモジュールで利用されます。

なぜ重要か

SQL標準の値関数は、クエリ実行時のコンテキスト（現在のユーザーや日付など）に応じて動的に評価される必要があります。SQLValueFunction は、これらを式評価エンジン（ExprEvalStep）に統合し、一貫した方法で効率的に評価できるようにします。また、最適化フェーズ（costsize.c や clauses.c など）において、これらのノードが「安定（stable）」であるという前提が利用されるため、クエリプランの生成においても重要な役割を果たします。

エンドツーエンドの流れ

1. **初期化と構築**: クエリ実行の準備段階において、SQLValueFunction ノードから ExprEvalStep が構築され、op->d.sqlvaluefunction.svf に対象の構造体へのポインタが格納されます。 2. **評価の識別**: 式評価エンジン（src/backend/executor/execExprInterp.c 内の評価ループなど）は、ExprEvalStep の opcode を参照して SQLValueFunction 評価ステップを識別します。 3. **実行と出力**: op->d.sqlvaluefunction.svf から関数情報を取得し、対応する内部評価ロジックを呼び出します。評価結果は ExprEvalStep の resvalue および resnull に格納され、後続のステップへ渡されます。

*注意: 呼び出し元・呼び出し先の詳細な関数コールグラフや、具体的な内部評価ロジックの全容など、完全なエンドツーエンドのフローは提供された情報からは未詳です。*

重要な不変条件

  • **安定性（Stability）**: SQLValueFunction は式評価のコンテキストにおいて「安定（stable）」であると見なされます。この不変条件はプランナの最適化フェーズで依存されています。
  • **評価の特定**: 実行エンジンは、この構造体を通じて適切な SQL 値関数を一意に特定し、評価を実行しなければなりません。

よくある落とし穴

  • **型情報の不整合**: 構造体内の type や typmod が不正な場合、src/backend/nodes/nodeFuncs.c などのノード走査処理においてアサーションエラーが発生する可能性があります。
  • **未サポートの関数**: 評価対象の関数が実行エンジン側でサポートされていない場合、実行時エラーを引き起こします。
  • **最適化前提の破壊**: 最適化フェーズ（costsize.c や clauses.c）において本ノードが「安定」である前提が利用されているため、この挙動や前提を変更すると、予期しないプランニングの不具合を招く恐れがあります。
