lo_read

概念

lo_read は、PostgreSQLにおいてラージオブジェクト（Large Object）からデータを読み出すためのAPIです。この名前を持つ関数は、クライアント側（libpq ライブラリ）とサーバー側（バックエンド）の双方に存在し、それぞれ異なる役割を担いながら連携して動作します。

なぜ重要か

ラージオブジェクトは通常のテーブル行の制限を超える大容量データを扱うために使用されます。lo_read は、クライアントアプリケーションがデータベースに格納されたラージオブジェクトから、指定したバッファサイズごとにデータを効率的に読み出すための主要なインターフェースを提供します。

関与するモジュール

  • module:src/backend/libpq
  • module:src/interfaces/libpq

主要エンティティ

  • **symbol:src/backend/libpq/be-fsstubs.c#lo_read**

    • サーバー側で実行される関数。記述子の検証や読み込み権限のチェックを行い、低レベルAPIである inv_read を呼び出して物理的な読み込みを行います。

  • **symbol:src/interfaces/libpq/fe-lobj.c#lo_read**

    • クライアント側で実行される関数。サーバー側の関数OIDを解決し、PQnfn を介してサーバー側の読み込み処理をリモートで呼び出します。

エンドツーエンドの流れ

1. **クライアント側の呼び出し**: クライアントアプリケーション（pg_dump や lo_import などのストリーミング処理）が src/interfaces/libpq/fe-lobj.c の lo_read を呼び出します。 2. **初期化と検証**: クライアント側で lo_initialize が実行され、サーバー側の関数OIDが解決されます。また、読み込み長さ len が INT_MAX を超えていないか検証されます。 3. **サーバーへの要求送信**: 引数（fd、len）が PQArgBlock に設定され、PQnfn を介してサーバー側の関数が呼び出されます。 4. **サーバー側の処理**: サーバー側の src/backend/libpq/be-fsstubs.c の lo_read が呼び出されます。 5. **記述子と権限の検証**: サーバー側で fd が有効な範囲内（0 以上 cookies_size 未満）であり、cookies[fd] が NULL でないこと、および読み込み権限（IFS_RDLOCK）が設定されていることを検証します。 6. **物理的読み込み**: 検証を通過すると、低レベルAPIである inv_read が呼び出され、実際のデータが読み込まれます。 7. **結果の返却**: 読み込まれたデータとバイト数がクライアントに返され、クライアント側のバッファに格納されます。

*注意: クライアントからサーバーへの通信プロトコルの詳細なパケット構造や、inv_read 内部の具体的な物理ストレージへのアクセス不変条件などの完全なエンドツーエンドのフローは、提供された情報からは完全には特定されていません。*

重要な不変条件

  • **クライアント側**:
  • lo_initialize(conn) が事前に成功している必要があります。
  • 読み込み要求サイズ len は INT_MAX を超えてはなりません。
  • ラージオブジェクトの操作は、現在のトランザクション内で実行される必要があります。
  • **サーバー側**:
  • 記述子 fd は有効な範囲（0 以上 cookies_size 未満）であり、cookies[fd] が NULL であってはなりません。
  • 対象のラージオブジェクトは、読み込み権限（IFS_RDLOCK フラグ）を保持してオープンされている必要があります。

よくある落とし穴

  • **同名関数の混同**: クライアント側の lo_read とサーバー側の lo_read はシグネチャや役割が異なります。開発やデバッグの際には、どちらのコンテキストを扱っているかを明確に区別する必要があります。
  • **データサイズ制限**: クライアント側の lo_read の引数 len は size_t 型ですが、内部的に int32 にキャストされてサーバーに送信されるため、INT_MAX を超える一度の読み込みはエラー（-1）になります。非常に大きなデータを一度に転送することはできません。
  • **トランザクションの管理**: ラージオブジェクトの操作はトランザクション内で行う必要があります。失敗時にはトランザクション全体がアボート状態になる可能性があります。
