SH_KEY_TYPE

概念

SH_KEY_TYPE は、PostgreSQL のヘッダーファイル lib/simplehash.h を用いて、型安全かつ高速なインラインハッシュテーブルを生成する際に定義する必須のマクロ（パラメータ）の一つです。 simplehash.h は、C言語でジェネリックなハッシュテーブルを実現するためのマクロテンプレートであり、利用者はハッシュテーブルの定義を生成する前に、キーのデータ型を指定するために SH_KEY_TYPE を定義します。

なぜ重要か

PostgreSQL では、動的なメモリ割り当てや関数ポインタ経由の呼び出しによるオーバーヘッドを削減するため、オープンアドレス法のハッシュテーブルテンプレートである simplehash.h を多用しています。 SH_KEY_TYPE を定義することで、コンパイル時に特定のキー型（例: Oid, BlockNumber, 構造体, 文字列ポインタなど）に特化したハッシュルックアップ、挿入、削除関数がインライン展開可能な形で自動生成されます。これにより、実行時のパフォーマンスが大幅に向上します。

関与するモジュール

提供されたソースコードに基づくと、以下のモジュールが SH_KEY_TYPE を使用して独自のハッシュテーブルを定義しています。

  • **バックエンドコア・オプティマイザ・エグゼキュータ**:
  • module:src/backend/catalog (カタログ・ネームスペース管理)
  • module:src/backend/executor (式評価、グループ化、Memoize ノード)
  • module:src/backend/nodes (TID ビットマップ)
  • module:src/backend/optimizer/path (等価クラスの管理)
  • module:src/include/nodes (実行ノード定義)
  • **ユーティリティ・統計情報**:
  • module:src/backend/utils/activity (pgstat 統計情報、共有メモリ統計情報)
  • module:src/backend/utils/adt (ロケールキャッシュ)
  • **バックアップ・共通処理**:
  • module:src/backend/backup (増分バックアップ処理)
  • module:src/common (ブロック参照テーブル blkreftable.c)
  • **クライアントツール / バイナリ**:
  • module:src/bin/pg_combinebackup (マニフェストロード)
  • module:src/bin/pg_dump (ダンプ対象オブジェクトの管理)
  • module:src/bin/pg_rewind (ファイルマップ管理)
  • module:src/bin/pg_verifybackup (バックアップ検証)

主要エンティティ

各ファイルにおいて、SH_KEY_TYPE は以下のような具体的なキー型とともに定義され、simplehash.h をインクルードすることでハッシュテーブルが生成されます。

  • **symbol:src/backend/backup/basebackup_incremental.c#SH_KEY_TYPE**: 増分バックアップにおけるファイルやブロック情報の管理用キー。
  • **symbol:src/backend/catalog/namespace.c#SH_KEY_TYPE**: スキーマ検索パスや一時名前空間のキャッシュ用キー。
  • **symbol:src/backend/executor/execGrouping.c#SH_KEY_TYPE** / **symbol:src/backend/executor/nodeMemoize.c#SH_KEY_TYPE**: グループ化キーや、Memoize ノードのキャッシュエントリキー。
  • **symbol:src/backend/nodes/tidbitmap.c#SH_KEY_TYPE**: BlockNumber をキーとする TID ビットマップのページエントリ。
  • **symbol:src/backend/utils/activity/pgstat_shmem.c#SH_KEY_TYPE**: 共有メモリ上の統計情報エントリを識別するキー。
  • **symbol:src/bin/pg_rewind/filemap.c#SH_KEY_TYPE**: 比較対象ファイルのパスや識別子をキーとするファイルマップ。
  • **symbol:src/common/blkreftable.c#SH_KEY_TYPE**: ブロック参照テーブルにおけるブロック番号等のキー。

エンドツーエンドの流れ

SH_KEY_TYPE を用いたハッシュテーブルの生成と利用の一般的な流れは以下の通りです。

1. **マクロの定義**: ハッシュテーブルを定義したいソースファイル（またはヘッダーファイル）内で、以下のマクロ群を定義します。

  • SH_PREFIX: 生成される関数や構造体の接頭辞（例: pagetable）
  • SH_KEY_TYPE: キーの型（例: BlockNumber）
  • SH_ELEMENT_TYPE: ハッシュテーブルに格納する要素（構造体）の型
  • SH_DECLARE または SH_DEFINE: 宣言または定義の有効化
  • その他、SH_EQUAL（キー比較関数）や SH_HASH_KEY（ハッシュ値計算関数）など

    • 2. **テンプレートのインクルード**: #include "lib/simplehash.h" を実行します。これにより、指定した SH_KEY_TYPE に適合する構造体やインライン関数（例: pagetable_insert, pagetable_lookup）がコンパイル時に生成されます。 3. **ハッシュテーブルの操作**: 生成された関数を用いて、指定されたキー型（SH_KEY_TYPE）のデータを検索・挿入します。

*注意: 各モジュールにおける具体的なハッシュ関数のアルゴリズムや、衝突解決の挙動など、個別のエンドツーエンドのデータフローの詳細は、それぞれのモジュールの実装に依存しており、統一的な単一のフローとしては定義されていません。*

重要な不変条件

  • **キーの一貫性**: SH_KEY_TYPE で指定された型は、SH_ELEMENT_TYPE で定義される構造体の中に、同一の型を持つメンバ（通常は key という名前）として存在しなければなりません。
  • **ハッシュ関数と等価関数の整合性**: SH_HASH_KEY が返すハッシュ値と SH_EQUAL による比較結果は一貫している必要があります（等しいキーは必ず同じハッシュ値を持つ）。
  • **マクロの未定義化（Undefine）**: simplehash.h をインクルードした後、これらの SH_ から始まるマクロは、同一ファイル内で別のハッシュテーブルを定義する際の衝突を防ぐために、自動的または明示的にアンデフ（#undef）されます。

よくある落とし穴

  • **マクロの再定義エラー**: 同一の C ファイル内で複数の異なる simplehash テーブルを定義する場合、各インクルードの前に SH_KEY_TYPE や SH_PREFIX などのマクロを適切に再定義（および以前の定義をクリア）しないと、コンパイルエラーが発生します。
  • **ポインタ型をキーにする場合の寿命**: SH_KEY_TYPE に char* などのポインタ型を指定した場合、ハッシュテーブルに格納されるのはポインタそのものです。ポインタが指す実体（文字列など）のメモリが解放されたり書き換えられたりすると、ハッシュテーブルの検索が破損します。
  • **ヘッダーでの多重定義**: SH_DEFINE をヘッダーファイル内で不用意に行うと、そのヘッダーをインクルードする複数のソースファイルでシンボルの重複定義エラーが発生します。宣言（SH_DECLARE）と定義（SH_DEFINE）を適切に分離する必要があります。
