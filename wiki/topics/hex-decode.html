hex_decode

概念

hex_decode は、PostgreSQLにおいて16進数形式の文字列（hex string）をバイナリデータにデコードするために使用される関数です。この処理は、データベースバックエンドのデータエンコーディング処理（src/backend/utils/adt）と、ECPG（Embedded SQL for C）クライアントライブラリ（src/interfaces/ecpg/ecpglib）の双方に同名の関数として存在し、それぞれの文脈に応じたデコード処理を提供しています。

なぜ重要か

データベースシステムにおいて、バイナリデータ（bytea など）をテキスト形式（16進数）で表現・転送し、それを元のバイナリ形式に復元するデコード処理は不可欠です。

  • **バックエンド側（src/backend/utils/adt/encode.c）**: SQL関数や内部処理から呼び出され、標準的なエラーハンドリングを伴う安全なデコードを提供します。
  • **クライアント側（src/interfaces/ecpg/ecpglib/data.c）**: ECPGアプリケーションがデータベースから取得した bytea 形式のデータをC言語の変数に変換する際に、空白文字のスキップなどの柔軟なパース処理を担います。

関与するモジュール

  • module:src/interfaces/ecpg/ecpglib

主要エンティティ

  • **symbol:src/backend/utils/adt/encode.c#hex_decode**:
  • シグネチャ: uint64 hex_decode(const char *src, size_t len, char *dst)
  • 役割: 16進数文字列 src をデコードし、dst に格納します。内部的に hex_decode_safe をエラーコンテキスト NULL で呼び出します。
  • **symbol:src/interfaces/ecpg/ecpglib/data.c#hex_decode**:
  • シグネチャ: static unsigned hex_decode(const char *src, unsigned len, char *dst)
  • 役割: ECPGライブラリ内部で bytea 形式の16進数文字列をデコードします。空白文字を無視する特徴があります。

エンドツーエンドの流れ

システム全体における完全なエンドツーエンドのデータフロー（例えば、SQLクエリの受付からクライアントへの返却、およびECPGでのパースに至る一連の全ステップ）は、提供された情報からは完全には明らかになっていません。

現時点で判明している局所的な流れは以下の通りです：

1. バックエンド側でのデコード

  • 内部処理から hex_decode が呼び出されます。
  • hex_decode_safe(src, len, dst, NULL) が呼び出され、実際のデコード処理が委譲されます。
  • デコードされた実際のバイト数が uint64 として返されます。

2. ECPGクライアント側でのデコード

  • src/interfaces/ecpg/ecpglib/data.c:532 において、データ変換および variable->len の計算のために hex_decode が呼び出されます。
  • 入力文字列 src を走査し、空白文字（' ', '\n', '\t', '\r'）をスキップしながら、get_hex を用いて2文字ずつ（1バイトずつ）デコードし dst に書き込みます。
  • 成功するとデコードされたバイト数を返し、不完全なペアがある場合は -1 を返します。

重要な不変条件

  • **バッファサイズの確保**:
  • バックエンド側・ECPG側のいずれにおいても、出力先バッファ dst はデコード後のデータを受け取るために十分なメモリ領域（ECPG側では少なくとも len / 2 バイト以上）が確保されている必要があります。
  • **エラーハンドリングの不変条件**:
  • バックエンド側の hex_decode はエラーコンテキスト（escontext）に NULL を渡して hex_decode_safe を呼び出すため、不正な16進数文字が含まれている場合はトランザクションの中断を伴う標準的なエラーが発生します。
  • ECPG側の hex_decode は、16進数のペアが完結しない（奇数長など）不完全なデータに遭遇した場合、例外を投げるのではなく -1 を返します。

よくある落とし穴

  • **同名関数の混同**:
  • src/backend/utils/adt/encode.c と src/interfaces/ecpg/ecpglib/data.c に同名の hex_decode が定義されていますが、これらはシグネチャ、戻り値の型（uint64 と unsigned）、およびエラー時の挙動（エラー発生によるトランザクション中断 vs -1 返却）が異なります。
  • **空白文字の扱いの違い**:
  • ECPG側の hex_decode は空白文字（スペース、改行、タブ、復帰）を自動的に無視してスキップしますが、バックエンド側の hex_decode（および hex_decode_safe）が同様の空白スキップを行うかどうかは、提供された情報からは保証されていません。
  • **不完全なペアによるエラー**:
  • ECPG側では、入力文字列の長さが奇数などでペアが完結しない場合、デコードに失敗して -1 が返されます。
