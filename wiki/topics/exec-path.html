exec_path

概念

exec_path（および関連する get_exec_path 関数）は、PostgreSQLのバイナリ（pg_createsubscriber や pg_ctl など）が、自身と同一のディレクトリに存在する他の関連実行ファイル（pg_ctl や pg_resetwal など）の絶対パスを特定・保持するための仕組みです。

なぜ重要か

PostgreSQLの各種ツールは、実行環境の環境変数 PATH に依存することなく、自身と同じインストール先ディレクトリにある他のバイナリを安全かつ確実に呼び出す必要があります。これにより、異なるバージョンの PostgreSQL が混在する環境であっても、意図しないバージョンのツールが実行されるのを防ぎ、システムの整合性を保つことができます。

主要エンティティ

  • **symbol:src/bin/pg_basebackup/pg_createsubscriber.c#get_exec_path**: 呼び出し元プログラムの実行パス（argv[0]）を基準に、同一ディレクトリ内から指定されたプログラム（pg_ctl や pg_resetwal）の絶対パスを検索・取得する関数。
  • **symbol:src/bin/pg_ctl/pg_ctl.c#exec_path**: pg_ctl 内で定義または使用されている変数またはシンボル（詳細なコンテキストは未提供）。

エンドツーエンドの流れ

1. **パスの検索開始**: pg_createsubscriber の初期化処理において、自身の実行パス（argv[0]）と検索対象のプログラム名（"pg_ctl" および "pg_resetwal"）を指定して get_exec_path を呼び出します。 2. **同一ディレクトリの探索**: get_exec_path 内部で find_other_exec を呼び出し、自身と同一ディレクトリ内に対象プログラムが存在するか、およびバージョンが一致しているかを検証します。 3. **パスの返却と保持**: 検索に成功すると、動的に割り当てられた絶対パスの文字列（char *）が返され、pg_ctl_path や pg_resetwal_path に格納されて以降のプロセス起動に利用されます。 4. **エラーハンドリング**: 検索に失敗した場合（ファイル未発見またはバージョン不一致）、find_my_exec で自身のパスを特定した上で、pg_fatal を呼び出してプロセスを異常終了させます。

※ src/bin/pg_ctl/pg_ctl.c における exec_path の具体的なエンドツーエンドの流れや、それがどのように利用されるかについては、提供されたコンテキストからは不明です。

重要な不変条件

  • **配置の同一性**: 検索対象のプログラム（progname）は、呼び出し元プログラム（pg_createsubscriber）と必ず**同一のディレクトリ**に存在しなければなりません。
  • **パス長の制限**: 取得される絶対パスの長さは、MAXPGPATH バッファサイズに収まる必要があります。
  • **バージョンの整合性**: find_other_exec による検証において、対象バイナリのバージョンが呼び出し元と一致している必要があります。

※ src/bin/pg_ctl/pg_ctl.c における exec_path に関連する不変条件については、提供されたコンテキストからは不明です。

よくある落とし穴

  • **ディレクトリ構成の不一致**: インストール環境のカスタマイズなどにより、pg_createsubscriber と pg_ctl や pg_resetwal が異なるディレクトリに配置されている場合、get_exec_path はプログラムを発見できず、pg_fatal により即座に異常終了します。
  • **メモリリーク**: get_exec_path が成功した際に返される char * は呼び出し元で適切に解放（free）する必要があります。
  • **バージョン不一致による起動失敗**: 同一ディレクトリ内に同名のバイナリが存在しても、バージョンが一致しない場合は find_other_exec がエラーを返し、実行が拒否されます。
