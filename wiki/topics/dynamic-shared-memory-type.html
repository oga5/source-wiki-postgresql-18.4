dynamic_shared_memory_type

概念

dynamic_shared_memory_type は、PostgreSQL が動的共有メモリ（Dynamic Shared Memory: DSM）を確保・管理する際に使用する、OS依存の実装方式（POSIX、System V、mmap、Windows等）を決定するための設定パラメータおよび変数です。

なぜ重要か

PostgreSQL はマルチプロセスアーキテクチャを採用しており、パラレルクエリなどの機能でプロセス間通信を行うために動的共有メモリを必要とします。OSの環境や制限に応じて最適なDSM実装を選択・制御し、安定したメモリ確保を行うためにこの設定が重要となります。

関与するモジュール

  • module:src/backend/storage/ipc
  • module:src/bin/initdb

主要エンティティ

  • symbol:src/backend/storage/ipc/dsm_impl.c#dynamic_shared_memory_type: バックエンド側で、どのDSM実装を使用してセグメントの作成やアタッチを行うかを判定・制御する変数。
  • symbol:src/bin/initdb/initdb.c#dynamic_shared_memory_type: データベースクラスタの初期化（initdb）時に、プラットフォームに適したデフォルトのDSMタイプを決定・設定するための変数。

エンドツーエンドの流れ

1. **初期化フェーズ**: src/bin/initdb/initdb.c の実行時、システム環境がサポートしているDSMタイプが検証され、最適なデフォルト値が dynamic_shared_memory_type として postgresql.conf に書き込まれます。 2. **実行フェーズ**: PostgreSQL サーバーの起動時および運用時、src/backend/storage/ipc/dsm_impl.c は設定された dynamic_shared_memory_type の値に基づき、対応するOS固有のAPI（POSIX shm_open、System V shmget、または mmap など）を呼び出して動的共有メモリセグメントを操作します。

※プラットフォームごとの具体的な自動検出アルゴリズムや、フォールバック処理の厳密な制御フローは、提供されたコンテキストからは現時点では不明です。

重要な不変条件

  • dynamic_shared_memory_type に指定する値は、ビルド時および実行時のOS環境がサポートしている実装タイプ（POSIX、System V、mmap、Windowsなど）のいずれかでなければなりません。

※サポートされる具体的な値のリストや、無効な値が指定された場合のバリデーションロジックの詳細は、現時点では不明です。

よくある落とし穴

  • **OSカーネル制限による失敗**: 設定されたタイプ（例: posix）に対して、OS側のリソース制限（/dev/shm のサイズ制限や System V の SHMMAX / SHMALL など）が不足している場合、動的共有メモリの確保に失敗し、パラレルクエリなどの機能が動作しなくなる、あるいはサーバーが起動できなくなる可能性があります。
  • **コンテナ環境での不整合**: initdb 実行環境と実際のコンテナ実行環境で共有メモリの権限やマウント状況が異なる場合、自動検出されたデフォルト値が実行時にエラーを引き起こすことがあります。

次に読む場所

  • src/backend/storage/ipc/dsm_impl.c: 各DSM実装（POSIX、System V、mmap等）の具体的な切り替えロジックとシステムコール呼び出しの実装。
  • src/bin/initdb/initdb.c: 初期化時にデフォルトのDSMタイプを決定するロジック。
