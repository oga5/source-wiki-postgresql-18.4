decimal

概念

decimal は、ECPG（Embedded SQL for C）において、固定サイズのメモリ領域で高精度数値（PostgreSQLの NUMERIC 型など）を保持・操作するためのC言語の構造体（struct）です。動的メモリ確保を行わないスタック配置に適した設計となっており、数値の属性（符号、スケール、重みなど）と、固定長配列 digits に格納された実際の桁データを管理します。

なぜ重要か

ECPG環境において、データベースからフェッチされた高精度な数値データをC言語の変数と安全かつ効率的に交換するために重要です。動的メモリ確保を伴う numeric 構造体とは異なり、decimal はスタック上に配置可能な固定サイズ（DECSIZE）の構造体であるため、メモリ管理のオーバーヘッドを削減しつつ、高精度な数値演算やSQLDA（SQL Descriptor Area）を介したデータ転送を可能にします。

関与するモジュール

提供された情報に基づき、以下のモジュールが関与しています。

  • module:src/interfaces/ecpg/include : pgtypes_numeric.h ヘッダーファイルを含み、標準的な decimal 構造体の定義を提供します。
  • module:src/interfaces/ecpg/test/expected : ECPGプリプロセッサのテスト期待値ファイル（preproc-outofscope.c や sql-sqlda.c）を含み、生成されたコード内での decimal 構造体の定義と利用例を示します。

エンドツーエンドの流れ

データベースから取得された数値データがどのように decimal 構造体に格納され、処理されるかの完全なエンドツーエンドの流れは、提供されたソースコードの範囲内では完全には明らかになっていません。

ただし、部分的な流れとして以下が確認できます： 1. **変数の準備**: ECPGプリプロセッサによって生成されたCコード（例: preproc-outofscope.c）において、データベースからフェッチされた数値データの格納先として decimal 構造体の変数が用意されます。 2. **データの解析と表示**: 動的SQL実行時やデータダンプ処理（例: sql-sqlda.c 内の dump_sqlda 関数など）において、SQLDA（sqlda_t や sqlvar_t）を経由して取得された数値データの解析や表示に decimal 構造体が参照されます。 3. **数値演算**: 実際の数値操作は、PGTYPESnumeric_* 系の関数（src/interfaces/ecpg/pgtypeslib/numeric.c に実装）を介して行われます。

重要な不変条件

  • **固定サイズ制限**: 桁データを保持する digits 配列のサイズは DECSIZE 定数によって固定されています。
  • **有効桁数**: ndigits は digits 配列内の有効な桁数を示し、0になる可能性があります。
  • **符号の有効性**: sign メンバは、NUMERIC_POS（正）、NUMERIC_NEG（負）、または NUMERIC_NAN（非数）のいずれかの定義された定数でなければなりません。

よくある落とし穴

  • **精度超過による切り捨て**: DECSIZE を超える桁数の数値を格納しようとした場合、データが切り捨てられるか、構造体の制限により表現不能となります。より高い精度や動的なサイズ変更が必要な場合は、動的メモリ確保を行う numeric 構造体を使用する必要があります。
  • **不正な符号値**: sign メンバに NUMERIC_POS、NUMERIC_NEG、NUMERIC_NAN 以外の不正な値が直接設定された場合、数値の正負判定や演算結果の動作は未定義となります。構造体のメンバを直接操作せず、提供されている API（PGTYPESnumeric_* 系の関数）を介してアクセスすることが推奨されます。
  • **定義の不一致**: テスト期待値ファイル（preproc-outofscope.c や sql-sqlda.c）に埋め込まれた decimal 定義と、標準ヘッダー（pgtypes_numeric.h）の定義との整合性、および _ECPG_INFORMIX_H などのマクロによる定義の切り替え（sqlda-compat.h や sqlda-native.h との関連）に注意する必要があります。
