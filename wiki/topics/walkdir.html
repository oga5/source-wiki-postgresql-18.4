walkdir

概念

walkdir は、指定されたディレクトリツリーを再帰的に走査し、各ファイルおよびディレクトリに対して指定されたアクション（fsync など）を実行するための関数です。PostgreSQLのソースコード内では、フロントエンド用（src/common/file_utils.c）とバックエンド用（src/backend/storage/file/fd.c）で異なる実装が存在します。

なぜ重要か

データベースのクラッシュリカバリやデータの永続性を保証するためには、データディレクトリ内のファイルだけでなく、ディレクトリ自体のメタデータも確実にディスクに同期（fsync）する必要があります。walkdir は、ディレクトリツリー全体を走査して漏れなく同期処理などのアクションを適用するための共通基盤を提供します。

エンドツーエンドの流れ

フロントエンド版（src/common/file_utils.c）における処理の流れは以下の通りです。

1. **除外チェック**: 指定された exclude_dir と現在の path が一致する場合、処理を即座に終了します。 2. **ディレクトリのオープン**: opendir を使用して対象ディレクトリを開きます。 3. **エントリの走査**: readdir でエントリを順次読み込み、. および .. はスキップします。 4. **型判定とアクション実行**: get_dirent_type を用いてエントリの型を判定します。

  • PGFILETYPE_REG (通常ファイル) の場合: 指定された action 関数を実行します。
  • PGFILETYPE_DIR (ディレクトリ) の場合: 再帰的に walkdir を呼び出します。
  • その他の型は無視します。

    • 5. **ディレクトリのクローズ**: closedir でディレクトリを閉じます。 6. **自己同期**: 最後に、ディレクトリ自身に対して action を実行します（ディレクトリメタデータの永続化を保証するため）。

なお、バックエンド版（src/backend/storage/file/fd.c）の walkdir を含めた全体的なエンドツーエンドの統合フローや、呼び出し元（pre_sync_fname や datadir_fsync_fname など）との詳細な連携フローの全貌は、現時点の提供情報からは未解明です。

重要な不変条件

  • **ディレクトリのオープン可能性**: path で指定されたディレクトリは正常にオープン可能でなければなりません。
  • **アクション関数の戻り値**: 渡される action 関数は、成功時に 0 を返すことが期待されます。
  • **ディレクトリメタデータの永続化**: 個別のファイル同期だけでは不十分なため、走査の最後に必ずディレクトリ自身に対しても action を実行し、メタデータの永続化を保証します。

よくある落とし穴

  • **フロントエンドとバックエンドの実装差異**: src/common/file_utils.c の walkdir はフロントエンド用であり、src/backend/storage/file/fd.c にあるバックエンド版の walkdir とは実装が異なります。これらを混同しないよう注意が必要です。
  • **エラーハンドリング**: ディレクトリのオープン失敗や読み込み中のエラーが発生した場合、関数は pg_log_error を出力して終了しますが、呼び出し元へのエラー伝播やリカバリの挙動には注意が必要です。
  • **バッファサイズ制限**: 内部で subpath の生成に MAXPGPATH * 2 のバッファサイズが使用されているため、極端に長いパスを扱う際には境界条件に留意する必要があります。
