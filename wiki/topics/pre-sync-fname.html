pre_sync_fname

概念

pre_sync_fname は、指定されたファイル（またはディレクトリ）に対して、OSのキャッシュからディスクへデータをフラッシュするための「ヒント」を送信する関数です。具体的には、sync_file_range や posix_fadvise を用いて、カーネルに対して非同期の書き込みバックグラウンド処理を促します。

なぜ重要か

本関数は、実際の同期処理（fsync など）を行う前に、OSキャッシュのフラッシュを先行して開始（プレシンク）させるために重要です。これにより、一括での同期処理時のI/O負荷を分散し、システム全体のパフォーマンス低下を防ぐことができます。バックエンドの pg_flush_data() と同等の役割をフロントエンド向けに提供します。

関与するモジュール

  • module:src/backend/storage/file
  • module:src/common

主要エンティティ

  • **symbol:src/common/file_utils.c#pre_sync_fname**: フロントエンドや共通ユーティリティ向けに実装されている pre_sync_fname 関数。
  • **symbol:src/backend/storage/file/fd.c#pre_sync_fname**: バックエンドのファイルストレージ管理において定義または利用される pre_sync_fname 関連処理（fd.c 内の walkdir などから呼び出されます）。

エンドツーエンドの流れ

1. **呼び出し**: src/common/file_utils.c 内の walkdir、src/backend/storage/file/fd.c 内の walkdir、または src/bin/pg_upgrade/relfilenumber.c などの呼び出し元から、対象のファイルパス fname とディレクトリ判定フラグ isdir を指定して pre_sync_fname が呼び出されます。 2. **ファイルのオープン**: PG_FLUSH_DATA_WORKS マクロが定義されている環境において、対象ファイルを読み取り専用（O_RDONLY）でオープンします。 3. **フラッシュヒントの送信**: プラットフォームのサポート状況に応じて、以下のいずれかのシステムコールを実行してOSにフラッシュのヒントを送ります。

  • HAVE_SYNC_FILE_RANGE が定義されている場合: sync_file_range
  • USE_POSIX_FADVISE かつ POSIX_FADV_DONTNEED が定義されている場合: posix_fadvise

    • 4. **クローズと終了**: 実行結果に関わらず、オープンしたファイル記述子をクローズします。成功時、またはファイルオープン失敗時に EACCES やディレクトリに対する EISDIR が発生した場合は 0 を返し、その他のエラー時は -1 を返します。

※なお、本関数が呼び出された後に、ディスクへの最終的な同期（fsync 等）に至る全体的な制御フローの全容や、バックエンド側における詳細な呼び出しコンテキストは、提供された情報からは完全には明らかになっていません。

重要な不変条件

  • **非ブロッキングなヒントとしての動作**: 本関数はあくまでOSへの「ヒント」の送信を目的としており、内部でのフラッシュ処理自体の失敗（システムコールのエラーなど）は致命的なエラーとして扱われず、処理を中断させません。
  • **マクロ定義への依存**: PG_FLUSH_DATA_WORKS が定義されていない環境では、本関数は何も行わずに 0 を返します。

よくある落とし穴

  • **完全な同期の保証はない**: pre_sync_fname はデータのフラッシュを「促す」だけであり、ディスクへの永続化（耐久性）を保証するものではありません。データの整合性を保証するためには、別途 fsync などの適切な同期処理を行う必要があります。
  • **エラーハンドリングの非対称性**: ファイルのオープン失敗時に EACCES や EISDIR は無視されて 0 が返されますが、それ以外のエラー（リソース不足など）では -1 が返されるため、呼び出し側での戻り値の扱いに注意が必要です。
