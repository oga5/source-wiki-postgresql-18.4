GucSource

概念

GucSource は、PostgreSQLのGUC（Grand Unified Configuration）システムにおいて、設定パラメータがどのソース（デフォルト値、設定ファイル、環境変数、SQLセッションなど）から提供されたかを識別するための列挙型（enum）です。

なぜ重要か

GUCシステムにおいて、設定値の優先順位（どのソースからの設定が既存の設定を上書きできるか）を決定し、管理するために不可欠です。例えば、設定ファイル（PGC_S_FILE）による設定よりも、セッション中の SET コマンド（PGC_S_SESSION）による設定を優先するといった制御は、この GucSource の値に基づいて行われます。また、check_* フック関数などの設定変更プロセスにおいて、変更元を特定する基準としても機能します。

関与するモジュール

  • module:src/backend/utils/misc
  • module:src/include/utils

主要エンティティ

  • symbol:src/include/utils/guc.h#GucSource: 設定パラメータのソースを示す列挙型。
  • symbol:src/backend/utils/misc/guc.c#set_guc_source: ソースを設定するための関数（詳細な振る舞いは現時点で未検証）。

エンドツーエンドの流れ

設定パラメータが変更される際、その変更を要求したソース（例: postgresql.conf の読み込み時は PGC_S_FILE、クライアントからの SET コマンド実行時は PGC_S_SESSION）に対応する GucSource の値が指定されます。 この値は、設定変更を試みる関数や check_* フック関数に渡され、現在の設定値のソースと比較されます。新しいソースの優先度が現在のソースと同等以上である場合にのみ、設定値が上書きされます。

※ただし、set_guc_source を含めた具体的な関数呼び出しの連鎖や、システム全体における詳細なエンドツーエンドの制御フローは、提供された情報からは完全には明らかになっていません。

重要な不変条件

  • **優先順位の直結**: GucSource 列挙型の定義順序は、設定の優先順位や適用可否を決定するロジックに直結しています。
  • **エラー報告の境界**: PGC_S_INTERACTIVE は、エラー報告における境界線として機能します。

よくある落とし穴

  • **列挙値の順序変更**: GucSource の列挙値の順序を変更すると、GUCシステム全体の優先順位判定ロジックに影響を及ぼすため、極めて慎重な検討が必要です。
  • **コンテキストの誤解**: フック関数（check_*）内で GucSource を使用する際は、設定が適用されるコンテキスト（GucContext など）を正しく理解して処理を行う必要があります。

次に読む場所

  • src/include/utils/guc.h: GucSource 列挙型の定義（PGC_S_* 定数群）および関連する GucContext などのデータ構造。
  • src/backend/utils/misc/guc.c: set_guc_source を含む、GUCシステムの具体的な実装。
