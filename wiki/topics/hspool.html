HSpool

概念

HSpool（Hash Spool）は、PostgreSQL のハッシュインデックス構築（ビルド）時に、インデックスエントリを一時的に保持・ソートするためのスプール（一時格納領域）を管理する構造体です。

ヘッダーファイル src/include/access/hash.h においては、構造体の内部定義を隠蔽した不透明な型（opaque struct）として typedef struct HSpool HSpool; と宣言されており、具体的な構造体のメンバー定義は src/backend/access/hash/hashsort.c にカプセル化されています。

なぜ重要か

ハッシュインデックスの構築において、インデックスエントリをバケット番号およびハッシュ値に基づいて事前にソートし、インデックスページへ順次挿入することで、ディスクアクセスの局所性を最適化し、構築パフォーマンスを大幅に向上させるために重要です。ソートを行わずにランダムな順序で挿入すると、ディスク I/O のオーバーヘッドが極めて大きくなります。

エンドツーエンドの流れ

ハッシュインデックス構築時における HSpool を介したデータフローは以下の通りです。

1. **初期化**: hashbuild（src/backend/access/hash/hash.c）から _h_spoolinit が呼び出され、HSpool 構造体が初期化されます。この際、内部の Tuplesortstate やバケットマスク情報（high_mask, low_mask, max_buckets）が設定されます。 2. **データの蓄積**: インデックス対象のタプルデータ（ItemPointer、Datum 値、isnull 配列）が _h_spool 関数を介して sortstate に投入され、一時的に蓄積されます。 3. **ソートと挿入**: _h_indexbuild が呼び出され、内部で tuplesort_performsort を用いてバケット番号順にタプルがソートされます。その後、ソートされたタプルを順次取り出し、_hash_doinsert を用いて実際のハッシュインデックスへ書き込みます。 4. **破棄**: 構築完了後、_h_spooldestroy を呼び出して HSpool および関連するリソース（sortstate など）を解放します。

*注意: hashbuild や _hash_doinsert の詳細な内部処理、およびエラー発生時のクリーンアップ処理などのエンドツーエンドの完全なフローは、提供されたコンテキストからは明らかになっていません。*

重要な不変条件

  • **マスク情報の同期**: HSpool 内の high_mask、low_mask、max_buckets は、対象ハッシュインデックスのメタデータ管理と完全に同期していなければなりません。
  • **リソース管理のライフサイクル**: _h_spoolinit で初期化された HSpool は、処理終了時に必ず _h_spooldestroy によって確実に破棄されなければなりません。
  • **カプセル化の維持**: HSpool は不透明な構造体として定義されているため、hashsort.c 以外の外部モジュールから構造体の内部フィールド（sortstate など）に直接アクセスしてはならず、必ず _h_ 接頭辞を持つ専用の関数群を介して操作しなければなりません。

よくある落とし穴

  • **メモリ消費量の増大**: 大規模なインデックス作成時、maintenance_work_mem の制限により、ソート処理（Tuplesortstate）に伴うメモリ消費量が増大し、リソースが逼迫する可能性があります。
  • **メタデータの不整合**: バケットマスク（high_mask, low_mask）の計算がインデックスのメタデータと不整合を起こすと、挿入先バケットを正しく特定できなくなる可能性があります。
  • **直接アクセスの試み**: hash.h をインクルードする外部ファイルから HSpool のメンバに直接アクセスしようとすると、不透明な構造体であるためコンパイルエラーになります。必ず提供されている API 関数を使用する必要があります。
