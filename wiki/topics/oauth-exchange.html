oauth_exchange

概念

oauth_exchange は、PostgreSQL における SASL OAUTHBEARER 認証メカニズムにおいて、クライアントとサーバー間でのプロトコル交換（ハンドシェイク）を処理するための仕組みです。サーバー側（src/backend/libpq/auth-oauth.c）では、クライアントから送信されたトークンの検証や状態遷移の管理を行います。クライアント側（src/interfaces/libpq/fe-auth-oauth.c）にも同名のシンボルが存在し、プロトコル交換の対となる処理を担っています。

なぜ重要か

RFC 7628 に準拠した OAUTHBEARER 認証を安全かつ正確に実行するために重要です。クライアントから送られてくる認証トークンの抽出、検証、およびエラーハンドリングを適切に行い、メモリ上の機密情報（トークンなど）を安全に消去（explicit_bzero）することで、セキュアな認証プロセスを保証します。

エンドツーエンドの流れ

サーバー側における処理の流れは以下の通りです：

1. **初期ハンドシェイク**: クライアントからの入力 input が NULL の場合、空のチャレンジを返して継続します。 2. **状態チェック**: OAUTH_STATE_INIT または OAUTH_STATE_ERROR 状態であることを確認します。 3. **メッセージ解析**: クライアントからのメッセージを解析し、チャネルバインディングフラグ（'n' または 'y'）と認証属性を確認します。 4. **トークン抽出**: parse_kvpairs_for_auth を使用して auth キーの値（トークン）を抽出します。 5. **検証**: validate 関数を呼び出してトークンを検証します。

  • **成功時**: 状態を OAUTH_STATE_FINISHED に更新し、PG_SASL_EXCHANGE_SUCCESS を返します。
  • **失敗時**: generate_error_response でエラー情報を生成し、状態を OAUTH_STATE_ERROR に遷移させて PG_SASL_EXCHANGE_CONTINUE を返します。

    • 6. **クリーンアップ**: トークンを含むメモリ領域を explicit_bzero で消去します。

※ クライアント側（fe-auth-oauth.c）の具体的な内部処理、およびクライアントとサーバー間を跨ぐ完全なエンドツーエンドの相互作用の詳細は、提供されたコンテキストからはまだ判明していません。

重要な不変条件

  • **入力と状態の整合性**: input が NULL の場合、セッション状態（ctx->state）は必ず OAUTH_STATE_INIT でなければなりません。
  • **入力長の整合性**: inputlen は input の実際の文字列長と一致していなければなりません。
  • **状態遷移**: 認証成功時には、セッション状態が必ず OAUTH_STATE_FINISHED に更新されます。
  • **機密情報の消去**: 処理終了時には、トークンを含むメモリ領域（input_copy）が explicit_bzero によって確実に消去されなければなりません。

よくある落とし穴

  • **プロトコル違反**: メッセージが空、長さの不一致、不正なチャネルバインディングフラグ、または必須属性の欠如がある場合、ERRCODE_PROTOCOL_VIOLATION エラーが発生します。
  • **未サポート機能**: 認可ID（authzid）が含まれている場合、ERRCODE_FEATURE_NOT_SUPPORTED が報告され処理が失敗します。
  • **検証失敗時の挙動**: トークンが無効な場合、即座に完全に切断するのではなく、エラー応答を生成して認証を継続（再試行）させる状態（OAUTH_STATE_ERROR）に遷移します。
