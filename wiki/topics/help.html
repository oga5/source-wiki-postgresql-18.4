help

概念

PostgreSQL のソースツリーにおいて、help は各コマンドラインツール、ユーティリティ、およびサーバーバイナリ（postgres）がサポートするコマンドラインオプション、使用方法（Usage）、接続パラメータ、およびバグ報告先などのヘルプ情報を標準出力に表示するための関数群です。

なぜ重要か

各バイナリ（サーバー、クライアントユーティリティ、プリプロセッサなど）が提供するコマンドラインインターフェース（CLI）の仕様をユーザーに提示し、正しいオプション指定を促すための共通のユーザーインターフェース（UI）パターンとして機能します。新しいオプションが追加された際には、この関数を更新することでドキュメントと実装の乖離を防ぐ役割を持ちます。

主要エンティティ

詳細なコンテキストが提供されている主要な help 関数シンボルは以下の通りです。

  • **symbol:contrib/oid2name/oid2name.c#help**: oid2name コマンドの利用方法と接続パラメータを表示します。
  • **symbol:src/backend/main/main.c#help**: postgres サーバーバイナリの標準・開発者向け・シングルユーザーモード用オプションを表示します。
  • **symbol:src/bin/pg_config/pg_config.c#help**: pg_config のオプション一覧を表示します。
  • **symbol:src/bin/pg_dump/pg_dumpall.c#help**: pg_dumpall の出力制御および接続オプションを表示します。
  • **symbol:src/bin/pg_walsummary/pg_walsummary.c#help**: pg_walsummary のオプション一覧を表示します。
  • **symbol:src/bin/scripts/clusterdb.c#help**: clusterdb コマンドのオプションを表示します。
  • **symbol:src/bin/scripts/createdb.c#help**: createdb コマンドのデータベース作成・接続オプションを表示します。
  • **symbol:src/bin/scripts/createuser.c#help**: createuser コマンドのロール属性・接続オプションを表示します。
  • **symbol:src/bin/scripts/dropdb.c#help**: dropdb コマンドの削除・接続オプションを表示します。
  • **symbol:src/bin/scripts/dropuser.c#help**: dropuser コマンドの削除・接続オプションを表示します。
  • **symbol:src/bin/scripts/pg_isready.c#help**: pg_isready コマンドの接続確認オプションを表示します。
  • **symbol:src/bin/scripts/reindexdb.c#help**: reindexdb コマンドの再構築・接続オプションを表示します。
  • **symbol:src/bin/scripts/vacuumdb.c#help**: vacuumdb コマンドのバキューム・接続オプションを表示します。
  • **symbol:src/interfaces/ecpg/preproc/ecpg.c#help**: 埋め込み SQL プリプロセッサ ecpg のオプションを表示します。

エンドツーエンドの流れ

各コマンドラインツールまたはサーバーの起動時（main 関数）において、コマンドライン引数の解析が行われます。

1. ユーザーが --help または -? オプションを指定してバイナリを実行します。 2. 引数解析処理がこれを検知し、該当するモジュールの help 関数を呼び出します。

  • 多くの help 関数は progname（実行ファイル名）を引数として受け取り、出力メッセージに埋め込みます。

    • 3. help 関数は、標準ライブラリの printf を用いて、国際化マクロ（_()）を適用しながらヘルプメッセージを標準出力（stdout）に書き出します。 4. 呼び出し元（main）または help 実行後に、プログラムは通常終了します。

※ help 関数自体の実行後に直接 exit を呼ぶか、呼び出し側に制御を戻して終了処理を行うかなどの詳細なエンドツーエンドの制御フローは、個々のバイナリの実装に依存するため一律の共通フローとしては未検証です。

重要な不変条件

  • **引数の有効性**: progname を引数として受け取る help 関数（例: main.c#help, oid2name.c#help など）では、progname が有効な文字列ポインタ（非 NULL）であることが前提となります。
  • **ビルドマクロの存在**: PACKAGE_BUGREPORT、PACKAGE_NAME、PACKAGE_URL などのプロジェクト共通マクロがビルド環境で定義されている必要があります。
  • **同期の維持**: 各ツールのコマンドラインオプションの定義（getopt のスイッチや関連データ構造）を追加・変更した際は、対応する help 関数の出力テキストも必ず同期して更新されなければなりません。

よくある落とし穴

  • **翻訳（国際化）の漏れ**: メッセージの追加や変更時に _() マクロ（gettext）によるラップを忘れると、多言語環境での翻訳が適用されなくなります。
  • **条件付きコンパイルによる不一致**: src/backend/main/main.c の USE_SSL や src/interfaces/ecpg/preproc/ecpg.c の YYDEBUG のように、ビルド時のマクロ定義によって表示されるオプションが変化する場合があります。ビルド構成が異なると、実際のオプションとヘルプ表示に乖離が生じる可能性があります。
  • **エラーハンドリングの欠如**: 標準出力（stdout）がクローズされている、またはパイプラインが切れている状態で printf が実行された場合、書き込みエラーが発生しますが、多くの help 関数内では明示的なエラーハンドリングやエラーコードの返却は行われません。

次に読む場所

  • **src/backend/main/main.c**: サーバーのオプション定義とヘルプ表示。
  • **src/bin/pg_config/pg_config.c**: オプション定義と info_items テーブルの同期例。
  • **src/bin/scripts/**: 各クライアントスクリプト（createdb.c, createuser.c など）における標準的な CLI ヘルプ実装パターン。
  • **src/interfaces/ecpg/preproc/ecpg.c**: プリプロセッサにおける条件付きヘルプ表示の例。
