JsonLexContext

概念

JsonLexContext は、PostgreSQLにおけるJSONデータの字句解析（Lexical Analysis）プロセスにおいて、現在の解析状態、入力バッファ、進捗、およびエラー情報を一元管理するための構造体です。

なぜ重要か

JSONのパース処理は、バックエンド（サーバー側）とフロントエンド（クライアント側、例えばlibpqなど）の両方で共通して必要とされます。JsonLexContext は、入力文字列のポインタや長さ、現在のトークン位置、ネストの深さ（lex_level）、エラーメッセージなどを保持し、インクリメンタル（増分）解析やエスケープ処理の有無に応じた柔軟なメモリ管理を可能にします。これにより、一貫したJSON解析インターフェースが提供され、メモリリークを防ぎつつ効率的なパースが実現されます。

主要エンティティ

  • **symbol:src/include/common/jsonapi.h#JsonLexContext**: 解析状態、入力バッファ、トークン情報、エラーメッセージなどを保持するメイン構造体。
  • **symbol:src/common/jsonapi.c#freeJsonLexContext**: JsonLexContext に関連付けられたリソース（文字列バッファやインクリメンタル解析用スタックなど）を安全に解放し、メモリリークを防ぐための関数。

エンドツーエンドの流れ

1. **初期化と設定**: 呼び出し元（src/backend/utils/adt/json.c、jsonb.c、jsonfuncs.c など）が JsonLexContext をインスタンス化し、入力文字列（input）、長さ（input_length）、および各種フラグ（incremental や need_escapes など）を設定します。 2. **字句解析の実行**: jsonapi.c 内の解析ロジック（pg_parse_json など）が JsonLexContext を操作し、入力ストリームから順次 JSON トークンを切り出し、token_type や lex_level を更新しながら、JsonSemAction コールバック関数と連携して解析を進めます。 3. **リソースの解放**: 解析処理の終了時、またはエラー発生時に、呼び出し元が freeJsonLexContext を呼び出します。これにより、コンテキスト内の文字列バッファ（strval、errormsg）や、インクリメンタル解析用の状態（inc_state、pstack）が適切に解放されます。

重要な不変条件

  • **行番号の開始**: line_number は必ず 1 から開始されます。
  • **エスケープ処理とバッファ**: need_escapes が true の場合のみ、デコードされた文字列を格納する strval が使用されます。
  • **安全な解放**: freeJsonLexContext に NULL またはメモリ不足を示す特殊な静的ポインタ failed_oom が渡された場合、関数は何もせず安全に復帰しなければなりません。
  • **構造体の処理フラグ**: JSONLEX_FREE_STRUCT フラグが設定されている場合は構造体自体も解放されますが、設定されていない場合は構造体の中身がゼロクリアされます。

よくある落とし穴

  • **メモリ消費の増大**: need_escapes を true に設定すると、文字列のデコード処理が行われるためメモリ消費量が増加します。不要な場合は false に設定することが推奨されます。
  • **トークンの所有権管理**: 長期実行プロセスにおいて、コールバックに渡される文字列トークンの解放責任を誤るとメモリリークの原因となります。setJsonLexContextOwnsTokens の設定により、パーサー側に所有権を適切に委譲する必要があります。
  • **フロントエンドとバックエンドの差異**: 本構造体および関連関数はバックエンドとフロントエンドの両方で利用されます。メモリ管理には FREE マクロが使用され、環境（palloc / malloc）に応じて抽象化されているため、変更時には両環境への影響を考慮しなければなりません。
