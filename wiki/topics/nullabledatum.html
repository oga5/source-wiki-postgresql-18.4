NullableDatum

概念

NullableDatum は、PostgreSQLの汎用データ型である Datum と、その値が NULL であるかどうかを示す bool 型のフラグ（isnull）をペアとして保持する構造体です。

なぜ重要か

実行エンジン内でのデータ受け渡しにおいて、値と NULL 状態を分離せずに単一のメモリブロックとして扱うことで、キャッシュ効率（空間効率と局所性）を高める役割を果たします。バックエンドの実行エンジン全体で頻繁に使用されるため、そのメモリレイアウトはシステム全体のパフォーマンスに影響を与えます。

エンドツーエンドの流れ

詳細なエンドツーエンドの制御フローは提供された情報からは完全には明らかになっていません。 ただし、以下のコンポーネント間でデータが受け渡されることが分かっています： 1. **インデックス属性の処理**: src/backend/catalog/index.c の AppendAttributeTuples 等で属性処理に使用されます。 2. **式評価**: src/backend/executor/execExpr.c において、式評価時の引数や結果の保持に使用されます。 3. **関数呼び出し**: src/backend/executor/execExprInterp.c において、fcinfo->args を介した関数呼び出し引数の保持に使用されます。

重要な不変条件

  • value フィールドは Datum 型であり、sizeof(void *) と同じサイズであると仮定されます。
  • isnull が true の場合、value の内容は不定であるか、または無視される必要があります。

よくある落とし穴

  • **メモリ割り当て時のサイズ計算**: palloc 等を用いて NullableDatum の配列を確保する際、サイズ計算には必ず sizeof(NullableDatum) を使用する必要があります。
  • **アライメントとパディング**: プラットフォームごとのアライメント規則に従うため、構造体内にパディング領域が生じます。フィールドの追加や変更を行う場合は、メモリレイアウトへの影響に注意する必要があります。
