ColumnRef

概念

ColumnRef は、SQL文中に現れる列参照（例: column、table.column、あるいは * のようなワイルドカード）を表現するためのパースツリー（構文木）ノードです。

なぜ重要か

SQLの構文解析（パース）段階で抽出された生の列名や修飾子、ソーステキスト上の位置情報を一時的に保持し、その後の意味解析フェーズで実際のデータベースオブジェクト（テーブルの列など）や Var ノード等へ正しく解決・変換するための起点となるため重要です。また、エラー発生時にユーザーへ正確なエラー位置を示す役割も担っています。

関与するモジュール

  • module:src/include/nodes
  • module:src/include/parser

主要エンティティ

  • **symbol:src/include/nodes/parsenodes.h#ColumnRef**

    • 列参照を表現する構造体定義です。ノードタグ（type）、フィールド名のリスト（fields）、およびソースコード上の位置（location）を保持します。

  • **symbol:src/include/parser/parse_node.h#ColumnRef**

    • 解析フェーズにおける ColumnRef の解決に関連する定義です。標準の解決ロジックの前に呼び出されるカスタムフック PreParseColumnRefHook などが定義されています。

エンドツーエンドの流れ

1. **生成**: パーサー（gram.y）がSQLを解析する際、または COPY コマンド処理（copy.c）の実行時に ColumnRef ノードが生成されます。 2. **走査・解決**: 意味解析器（analyze.c）が ParseState の持つ範囲テーブル（Range Table）や名前空間（Namespace）と照合しながら fields リストを走査し、参照先の列を特定します。

  • PreParseColumnRefHook が設定されている場合は、標準の解決ロジックの前にカスタムの解決処理が実行されます。
  • A_Star（*）が含まれる場合は、ワイルドカード展開の対象として処理されます。

    • 3. **変換**: 解析が完了すると、クエリツリー内の適切な参照（Var ノードなど）へと変換されます。

*注: 具体的な関数呼び出しチェーンの全容など、詳細なエンドツーエンドの流れは提供されたソース範囲内では完全には網羅されていません。*

重要な不変条件

  • **fields の構成**: fields リストには、列名を表す String ノード、または A_Star（*）のみが含まれている必要があります。
  • **location の保持**: location フィールドには、SQL文字列内の有効なバイトオフセット（不明な場合は -1）が格納されていなければなりません。
  • **実行時の非存在**: 本構造体はパーサー・解析器の一時的な出力であるため、クエリの実行時には存在せず、必ず Var ノード等に変換されていなければなりません。

よくある落とし穴

  • **位置情報の欠落**: location が適切に設定されていない（または -1 のまま放置される）場合、エラーメッセージの発生箇所（カーソル位置）を特定できなくなります。
  • **不正なフィールド**: fields が空であるか、想定外のノードタイプを含んでいる場合、意味解析フェーズで予期せぬエラーが発生する可能性があります。
  • **名前解決の失敗**: 参照された列名が現在の名前空間や範囲テーブルに存在しない場合、解析エラーが発生します。
