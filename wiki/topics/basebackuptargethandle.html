BaseBackupTargetHandle

概念

BaseBackupTargetHandle は、PostgreSQLのベースバックアップ処理において、バックアップの出力先（ターゲット）を抽象化するためのハンドル構造体です。選択されたターゲットの検証済み情報と、それに関連する詳細な引数を保持します。

なぜ重要か

バックアップデータの書き込み先（ターゲット）固有の検証処理や、バックアップデータを転送する bbsink パイプラインの生成処理をカプセル化するためです。これにより、バックアップのメインフロー（basebackup.c）は特定のターゲット実装に直接依存することなく、抽象化されたインターフェースを通じて安全にバックアップを制御できます。

関与するモジュール

  • module:src/include/backup

主要エンティティ

  • **symbol:src/include/backup/basebackup_target.h#BaseBackupTargetHandle**: 外部モジュール向けに公開される不透明な（opaque）構造体としての宣言。
  • **symbol:src/backend/backup/basebackup_target.c#BaseBackupTargetHandle**: 構造体の具体的な定義。ターゲットの定義を指す BaseBackupTargetType *type と、検証済みの詳細情報を指す void *detail_arg をメンバに持ちます。

エンドツーエンドの流れ

1. **ハンドルの生成と検証**: バックアップ処理の開始時に BaseBackupGetTargetHandle() が呼び出されます。指定されたターゲット名に対応する BaseBackupTargetType が検索され、ターゲット固有の検証関数が実行されます。検証が成功すると、本構造体がインスタンス化されます。 2. **シンクの構築**: 生成されたハンドルは BaseBackupGetSink() に渡されます。ハンドルが保持する type メンバを通じてターゲット固有の関数（get_sink など）が呼び出され、バックアップデータを処理する bbsink パイプラインが構築されます。

*注意: バックアップ処理全体の詳細なライフサイクルや、エラー発生時のクリーンアップ処理の全容など、上記以外のエンドツーエンドフローの詳細は提供されたコンテキストからは不明です。*

重要な不変条件

  • **カプセル化の維持**: 本構造体はヘッダーファイル（basebackup_target.h）において不透明（opaque）として定義されており、外部モジュールから直接メンバにアクセスすることはできません。必ず提供される API を通じて操作する必要があります。
  • **検証済み状態の保証**: ハンドルは BaseBackupGetTargetHandle() によって検証が成功した状態でのみ取得されます。
  • **メモリコンテキスト**: 本構造体は TopMemoryContext 上でメモリ確保されることが想定されています。

よくある落とし穴

  • **定義の不整合**: 本構造体はヘッダーファイルとCソースファイルの両方に定義が存在するため、構造体の内部定義を変更する際は双方の整合性を保つ必要があります。
  • **メモリ管理**: 拡張モジュールが新しいターゲットを追加して detail_arg に独自のデータ構造を格納する場合、そのライフサイクルは TopMemoryContext に依存するため、メモリリークや不正アクセスが発生しないよう注意が必要です。
  • **検証失敗時のハンドリング**: ターゲットの検索や検証（check_detail）に失敗した場合、有効なハンドルは返されず、バックアップ処理は開始されません。
