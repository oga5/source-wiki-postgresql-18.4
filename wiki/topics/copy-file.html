copy_file

概念

PostgreSQL 18.4 において、copy_file は異なるコンテキスト（サーバーのバックエンド処理と、クライアント側のバックアップ結合ツール）で定義されている、ファイル複製のための関数です。

  • **バックエンド側（src/backend/storage/file/copydir.c）**: OSキャッシュへの負荷を抑え、データ整合性を考慮しながらバッファリングコピーを行います。
  • **クライアントツール側（src/bin/pg_combinebackup/copy_file.c）**: バックアップ再構築プロセスにおいて、指定された戦略（クローン、コピー、リンクなど）に基づいてファイルを転送する抽象化レイヤーを提供します。

なぜ重要か

  • **バックエンド側**: アーカイブ処理（例: basic_archive）などで、OSキャッシュを圧迫せずに効率的かつ安全にファイルを複製し、システム全体のI/Oパフォーマンスを維持するために重要です。
  • **クライアントツール側 (pg_combinebackup)**: 異なるコピー戦略（CLONE, COPY, LINK など）やチェックサム計算、ドライラン（シミュレーション）を統一的に扱う抽象化を提供し、バックアップ再構築の柔軟性と信頼性を確保するために重要です。

エンドツーエンドの流れ

それぞれのコンテキストにおける処理の流れは以下の通りです。なお、呼び出し元から本関数に至る全体的なエンドツーエンドのフロー（特に pg_combinebackup 内での具体的な呼び出し元）は、提供された情報からは未確定です。

バックエンド側 (src/backend/storage/file/copydir.c)

1. **初期化**: COPY_BUF_SIZE（8 * BLCKSZ）のバッファを確保します。 2. **ファイルオープン**: OpenTransientFile を使用し、ソースを読み取り専用、コピー先を書き込み専用（新規作成）で開きます。 3. **データコピー**:

  • ループ内で read と write を繰り返し、データを転送します。
  • CHECK_FOR_INTERRUPTS() を呼び出し、シグナルによる中断を監視します。
  • FLUSH_DISTANCE（プラットフォームにより1MBまたは32MB）ごとに pg_flush_data を呼び出し、OSキャッシュをフラッシュします。
  • 転送中、pgstat_report_wait_start/end を使用して待機イベントを記録します。

    • 4. **終了処理**: 最後に残りのデータをフラッシュし、ファイルを閉じてバッファを解放します。 ※ 呼び出し元として contrib/basic_archive/basic_archive.c が確認されています。

クライアントツール側 (src/bin/pg_combinebackup/copy_file.c)

1. **ドライラン確認**: dry_run モードであれば、ソースファイルを開いて閉じ、存在を確認します。 2. **戦略選択**: copy_method に基づき、実行する関数ポインタ（strategy_implementation）とログ用名称を決定します。 3. **ログ出力**: 実行前（またはドライラン時）に、選択された戦略とチェックサムの有無をログ出力します。 4. **実行**: 選択された strategy_implementation（copy_file_clone, copy_file_blocks, copy_file_by_range, copy_file_copyfile, copy_file_link のいずれか）を呼び出し、実際の I/O を実行します。

重要な不変条件

  • **バックエンド側**:
  • コピー元ファイル（fromfile）は読み取り可能でなければなりません。
  • コピー先ファイル（tofile）は新規作成可能でなければなりません（既存ファイルがある場合は失敗します）。
  • palloc を使用するため、PostgreSQL のメモリコンテキストが有効である必要があります。
  • **クライアントツール側**:
  • dry_run が true の場合であっても、ソースファイルが読み取り可能（open 可能）でなければなりません。
  • Windows 環境において COPY_METHOD_COPY が指定された場合、自動的に COPY_METHOD_COPYFILE へ変換されます。

よくある落とし穴

  • **バックエンド側**:
  • 本関数は fsync を直接実行しません。そのため、データの永続性を保証するには、呼び出し側でコピー完了後に fsync_fname 等を用いて明示的に同期を行う必要があります。
  • FLUSH_DISTANCE はプラットフォーム（特にmacOS）の特性を考慮したチューニング値であるため、変更時には注意が必要です。
  • **クライアントツール側**:
  • OSが選択されたコピー戦略（例: クローンやハードリンク）をサポートしていない場合、呼び出された内部の実装関数内で pg_fatal が発生し、プロセスが終了する可能性があります。
  • dry_run モードは単なるログ出力のシミュレーションではなく、実際にファイルシステムへのアクセス権限を検証（open の実行）するため、ファイルが存在しない場合はエラーとなります。
