numeric

概念

numeric は、PostgreSQLの NUMERIC 型データをC言語プログラム（特にECPG: Embedded SQL for C）内で表現・管理するための動的メモリ管理型構造体です。固定長の decimal 型とは異なり、動的にサイズが変化する高精度数値を保持することができます。

なぜ重要か

ECPGプリプロセッサによるSQL操作（FETCH など）において、データベースから取得した NUMERIC 型の値をC言語の構造体にマッピングし、データバインドや高精度な数値演算を安全に行うために不可欠なデータ構造です。

エンドツーエンドの流れ

提供された情報からは、完全なエンドツーエンドのフローは未検証です。判明している部分的な流れは以下の通りです。

1. **入力**: データベースからフェッチされた数値データ、または PGTYPESnumeric_from_asc 等の関数を介して文字列や他の数値型から変換されたデータが numeric 構造体に格納されます。 2. **仲介**: ECPGプリプロセッサが生成したコード（ECPGget_var 等のランタイム関数）を通じて、SQLの NUMERIC 型とC言語の構造体間でマッピングが行われます。 3. **出力**: PGTYPESnumeric_to_asc 等の関数を介して文字列や他の数値型へ変換されるか、SQL操作を通じてデータベースへ書き込まれます。

※ バックエンド（src/backend/utils/adt/numeric.c）とECPG側の連携や、具体的な計算処理の全体フローは現時点では不明です。

重要な不変条件

  • **メモリ管理**: numeric 構造体は動的メモリ確保を前提としており、ライフサイクル管理のために PGTYPESnumeric_new() で確保し、PGTYPESnumeric_free() で解放する必要があります。
  • **ポインタの一貫性**: buf メンバは、digits 配列を保持するために動的に確保されたメモリ領域の先頭を指す必要があります。
  • **有効な符号**: sign メンバは NUMERIC_POS、NUMERIC_NEG、または NUMERIC_NAN のいずれかでなければなりません。
  • **桁数**: ndigits は 0 を取り得ます（0の場合は数値がゼロまたは未定義として扱われる可能性があります）。

よくある落とし穴

  • **直接操作によるメモリ破壊**: 構造体のメンバ（特に buf や digits）を直接変更すると、メモリリークや不正なメモリ参照を引き起こす可能性があります。必ず提供される公開API（PGTYPESnumeric_*）を介して操作する必要があります。
  • **初期化不足**: buf が適切に初期化または割り当てられていない場合、数値演算や表示時に予期せぬエラーやクラッシュが発生するリスクがあります。
