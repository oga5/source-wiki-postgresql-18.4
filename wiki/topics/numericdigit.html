NumericDigit

概念

NumericDigit は、PostgreSQLにおける高精度数値型（NUMERIC 型）の内部表現において、数値を構成する個々の「桁（digit）」を保持するための基本データ型です。この型は、PostgreSQLのコアバックエンドと、埋め込みSQLプリプロセッサであるECPG（Embedded SQL for C）の両方で定義されていますが、それぞれのコンテキストに応じて異なる基本型（signed char / int16 または unsigned char）として実装されています。

なぜ重要か

高精度な数値演算をサポートする NUMERIC 型の内部構造体（NumericData、NumericVar、およびECPGの numeric や decimal）において、メモリ効率を維持しつつ、各桁のデータを正確に保持・操作するための基礎となるため重要です。バックエンドでの算術演算や、クライアント・サーバー間での数値データの受け渡しにおいて、この型を介してデータが処理されます。

エンドツーエンドの流れ

バックエンドとECPGクライアント間における NumericDigit データの直接的な受け渡しや、詳細なエンドツーエンドのデータフローは、提供されたソースコードの文脈からは完全には特定されていません。

一般的に想定される流れは以下の通りです： 1. **バックエンド側**: データベース内の NUMERIC 型データは、NumericData 構造体の n_data 配列（NumericDigit 型）に格納されます。算術演算（add_var や sub_var など）の実行時には、メモリ上の NumericVar に展開されて操作されます。 2. **ECPGクライアント側**: SQLDA（SQL Descriptor Area）などを介して数値データを取得・操作する際、ECPG側の numeric または decimal 構造体の配列要素（NumericDigit 型）としてデータが格納・処理されます。

重要な不変条件

  • **バックエンド側**:
  • 各 NumericDigit は、基数 NBASE（現在は 10000）の範囲内（0 から NBASE - 1）の値を保持する必要があります。これにより、1つの NumericDigit に10進数4桁が格納される設計上の不変条件が維持されます。
  • **ECPG側**:
  • unsigned char 型として定義されているため、非負の整数値（0〜255）のみを保持します。
  • 構造体（特に固定長配列を持つ decimal）の最大サイズ（DECSIZE など）を超えたアクセスを行ってはなりません。

よくある落とし穴

  • **定義の不一致**: バックエンド側（signed char または int16）とECPG側（unsigned char）で型定義が異なります。これらを混同してキャストや代入を行うと、符号の有無やサイズの違いにより、予期せぬ数値の破壊やコンパイル警告が発生する可能性があります。
  • **バッファオーバーラン**: NumericVar や decimal などの構造体でメモリを確保・操作する際、NumericDigit 配列の境界チェックを誤ると、メモリ破壊や未定義動作を引き起こすリスクがあります。
