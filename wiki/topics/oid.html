Oid

概念

Oid（Object Identifier）は、PostgreSQLデータベースシステムにおいて、テーブル、インデックス、関数、データ型、演算子族などの各種データベースオブジェクトを一意に識別するための基本データ型（通常は unsigned int）です。

なぜ重要か

Oid は、PostgreSQLのほぼすべてのサブシステムを横断する共通の「キー」として機能するため極めて重要です。システムカタログの検索、キャッシュ（syscacheやrelcache）のルックアップ、式評価エンジンでの型判定、インデックスアクセスメソッドのメタデータ管理、DDL追跡、およびキャッシュ無効化イベントの通知など、オブジェクトの特定が必要なあらゆる場面で利用されます。

関与するモジュール

提供されたコンテキストに基づき、以下のモジュールが Oid を使用してオブジェクトの識別や操作を行っています。

  • module:src/bin/pg_dump（アーカイブ処理対象のオブジェクト特定）
  • module:src/include（基本型定義）
  • module:src/include/access（インデックスアクセスメソッドのメタデータ管理）
  • module:src/include/catalog（システムカタログ内のオブジェクト識別、名前解決、オブジェクトアクセスフック）
  • module:src/include/commands（EXPLAIN出力時のインデックス名取得フック）
  • module:src/include/executor（式評価ステップにおける変数の型情報保持）
  • module:src/include/foreign（外部リレーションのサイズ取得等のFDW API）
  • module:src/include/nodes（式ノードの評価結果や関数識別子の表現）
  • module:src/include/replication（WALレシーバ関連）
  • module:src/include/tcop（DDL追跡やイベントトリガー用のCollectedCommand構造体）
  • module:src/include/utils（キャッシュ無効化イベントの対象リレーション特定）

主要エンティティ

  • **symbol:src/include/postgres_ext.h#Oid**: バックエンドとフロントエンド（libpq等）で共有される基本型定義（typedef unsigned int Oid;）。
  • **symbol:src/bin/pg_dump/pg_backup_archiver.h#Oid**: アーカイブ処理（StartLOPtrType 等）でオブジェクトを特定するための定義。
  • **symbol:src/include/access/amapi.h#Oid**: インデックスアクセスメソッド（amtranslate_strategy_function 等）で演算子族などを特定するための定義。
  • **symbol:src/include/catalog/namespace.h#Oid**: 名前解決（RangeVarGetRelid 等）でオブジェクトIDを取得するための定義。
  • **symbol:src/include/catalog/objectaccess.h#Oid**: オブジェクトアクセスフック（object_access_hook_type）で操作対象を特定するための定義。
  • **symbol:src/include/commands/explain.h#Oid**: EXPLAINフック（explain_get_index_name_hook_type）でインデックスを特定するための定義。
  • **symbol:src/include/executor/execExpr.h#Oid**: 式評価ステップ（ExprEvalStep 内の var.vartype）で変数の型を特定するための定義。
  • **symbol:src/include/foreign/fdwapi.h#Oid**: FDW API（GetForeignRelSize_function 等）で外部テーブルを特定するための定義。
  • **symbol:src/include/nodes/nodeFuncs.h#Oid**: 式ノード（FuncExpr, OpExpr 等）で評価結果や関数を特定するための定義。
  • **symbol:src/include/tcop/deparse_utility.h#Oid**: DDL追跡（CollectedCommand 内の alterTable.objectId / classId）で対象オブジェクトを特定するための定義。
  • **symbol:src/include/utils/inval.h#Oid**: キャッシュ無効化（RelcacheCallbackFunction 等）で影響を受けるリレーションを特定するための定義。

エンドツーエンドの流れ

具体的なエンドツーエンドのフロー（例：クエリ実行からキャッシュ無効化、あるいはダンプ出力までの完全なライフサイクル）は、提供されたモジュール情報だけでは完全には明らかになっていません。

ただし、部分的な流れとして以下が確認されています。

1. **名前解決とオブジェクト特定**: RangeVar などの入力から RangeVarGetRelid などの名前解決関数を介して対象オブジェクトの Oid が取得されます。 2. **式評価と実行**: プランナーが生成した式ツリーから Oid（vartype など）が設定され、実行時に適切な型変換や関数呼び出しのディスパッチに利用されます。 3. **キャッシュ無効化とフック**: オブジェクトの変更時には Oid を用いて object_access_hook や RelcacheCallbackFunction が呼び出され、キャッシュの無効化や監査ログの生成が行われます。

重要な不変条件

  • **一意性**: Oid は、システムカタログ内の各オブジェクトに対して一意な値として割り当てられます（データベースクラスタ全体、またはデータベース単位）。
  • **無効値の予約**: InvalidOid（通常 0）は、オブジェクトが存在しない、または無効であることを示す予約値として定義されています。
  • **型の一貫性**: Oid はシステム全体で一貫して unsigned int（32ビット符号なし整数）として扱われ、フロントエンドとバックエンドの間で共通のインターフェースを提供します。

よくある落とし穴

  • **InvalidOid のハンドリング漏れ**: 名前解決やオブジェクト検索の結果として InvalidOid が返された場合、呼び出し元で適切にエラーハンドリングを行わないと、システムカタログの検索失敗や予期しない動作を引き起こす可能性があります。
  • **Oid の再利用と生存期間**: システムカタログの更新やオブジェクトの削除・再作成に伴い、古い Oid が無効になったり、別のオブジェクトに再利用されたりする可能性があるため、永続的な参照として保持する際は注意が必要です。
  • **共用体アクセスの前提条件**: deparse_utility.h の CollectedCommand のように、Oid が共用体（alterTable 構造体など）の一部として定義されている場合、親構造体の type フィールドを確認せずにアクセスすると、不正なメモリ参照を引き起こす危険性があります。
