DIR

概念

DIR は、Windows環境（MSVC）においてPOSIX準拠のディレクトリ走査（opendir, readdir, closedir）を行うための状態保持用構造体です。WindowsのネイティブAPIである FindFirstFile および FindNextFile による検索ハンドルと、現在のディレクトリ情報を管理します。

なぜ重要か

PostgreSQLのバックエンドコード（例: xlog.c や slru.c など）は、プラットフォームに依存しない共通のコードベースとしてPOSIX標準のディレクトリ走査APIに依存しています。しかし、Windows環境はこれらをネイティブでサポートしていません。

DIR 構造体とその関連関数を提供することで、プラットフォーム間の差異を吸収し、バックエンドの共通コードをWindows上でもそのまま動作させることが可能になります。

主要エンティティ

  • **symbol:src/include/port/win32_msvc/dirent.h#DIR**

    • ヘッダーファイルで定義される不透明な型定義（opaque structure）です。 ``c typedef struct DIR DIR; ``

  • **symbol:src/port/dirent.c#DIR**

    • 実際の構造体の定義です。以下のメンバを持ち、Windows APIの検索ハンドルと走査状態を保持します。 ``c struct DIR { char *dirname; struct dirent ret; HANDLE handle; }; ``

エンドツーエンドの流れ

1. **初期化 (opendir)**

  • 呼び出し元（xlog.c や slru.c など）がディレクトリパスを指定して opendir を呼び出します。
  • 内部でパスに * を付与して検索文字列を作成し、Windows APIの FindFirstFile を実行して検索ハンドル handle を取得します。

    • 2. **走査 (readdir)**

  • 呼び出し元が readdir を繰り返し呼び出します。
  • 内部で FindNextFile を呼び出し、取得したファイル属性やファイル名を struct dirent 型の ret メンバにマッピングして、そのポインタを返します。

    • 3. **終了 (closedir)**

  • 呼び出し元が closedir を呼び出します。
  • 内部で FindClose を呼び出して handle を解放し、割り当てられたメモリをクリーンアップします。

*注意: 各バックエンド呼び出し元における具体的なエラーハンドリングや、すべての呼び出し元における詳細なエンドツーエンドの統合フローの全容は、提供されたコンテキストからは完全には明らかになっていません。*

重要な不変条件

  • **Windows環境専用**: 本構造体および関連APIは、Windows環境（MSVC）でのみ使用されます。
  • **有効な検索ハンドル**: opendir で正常に初期化された後、handle は FindFirstFile によって取得された有効なWindows検索ハンドル（INVALID_HANDLE_VALUE 以外）でなければなりません。
  • **固定値のフィールド**: Windows APIの制約により、ret（struct dirent）内の d_ino（iノード番号）および d_reclen は常に 0 に固定されます。

よくある落とし穴

  • **不透明型の不整合**: DIR 構造体の実体は src/port/dirent.c で定義され、ヘッダー src/include/port/win32_msvc/dirent.h では不透明な型として宣言されています。構造体のメンバを変更する場合は、必ず双方の整合性を保つ必要があります。
  • **POSIXの完全な模倣ではない**: 本実装はPostgreSQLのビルドに必要な最小限の機能セットのみを提供しています。例えば、d_ino はサポートされておらず常に 0 が返されるため、iノード番号に依存するコードはWindows環境で正しく動作しません。
  • **パス長制限**: Windowsの MAX_PATH 制限に依存します。
