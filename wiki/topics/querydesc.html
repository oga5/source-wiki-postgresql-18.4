QueryDesc

概念

QueryDesc（クエリ記述子）は、PostgreSQLのクエリ実行エンジンにおいて、実行対象の計画（PlannedStmt）と、その実行に必要な環境情報（スナップショット、パラメータ、出力先など）を一元管理し、実行ライフサイクル全体を通じて保持するためのパッケージ構造体です。

なぜ重要か

クエリ実行の各フェーズ（Start -> Run -> Finish -> End）において、プランツリー、実行時パラメータ、および実行結果の出力先（DestReceiver）などの実行状態を一貫して維持・伝達するために不可欠です。また、実行エンジンのフック（ExecutorStart_hook など）や外部プラグイン（auto_explain や pg_stat_statements など）が実行コンテキストにアクセスする際の共通インターフェースとしても機能します。

  • **symbol:src/include/executor/executor.h#QueryDesc (typedef)**: 実行ライフサイクル関数等で参照される QueryDesc の型定義。
  • **symbol:src/backend/tcop/pquery.c#CreateQueryDesc (function)**: QueryDesc 構造体を palloc により割り当て、初期化する関数。
  • **symbol:src/backend/tcop/pquery.c#FreeQueryDesc (function)**: QueryDesc が保持するリソース（スナップショットなど）を解放し、メモリをクリーンアップする関数。

エンドツーエンドの流れ

1. **生成と初期化**: CreateQueryDesc により QueryDesc 構造体が確保され、計画（PlannedStmt）、ソーステキスト、出力先（DestReceiver）、パラメータ、スナップショットなどが格納されます。この際、スナップショットは RegisterSnapshot により登録されます。 2. **実行準備**: ExecutorStart が QueryDesc を受け取り、クエリ全体の実行状態を表す EState や実行ノードツリーである PlanState を構築して QueryDesc 内に設定します。 3. **実行**: ExecutorRun が QueryDesc を参照し、プランツリーを走査して結果を dest（DestReceiver）へ渡します。 4. **解放**: クエリ実行終了後、FreeQueryDesc を呼び出すことで、登録されたスナップショットの解除（UnregisterSnapshot）および QueryDesc 自体のメモリ解放（pfree）が行われます。

重要な不変条件

  • **スナップショットの参照管理**: CreateQueryDesc で登録された snapshot および crosscheck_snapshot は、FreeQueryDesc において UnregisterSnapshot により参照カウントが減らされなければなりません。
  • **解放時の実行状態**: FreeQueryDesc を呼び出す際、qdesc->estate が NULL であること（実行中のクエリではないこと）が Assert により保証されている必要があります。したがって、必ず ExecutorEnd 等で実行状態が終了した後に呼び出す必要があります。
  • **ユーティリティコマンドの制限**: ユーティリティコマンドの場合、実行エンジン（ExecutorRun）に渡してはならないという制約があります。

よくある落とし穴

  • **不適切な初期化状態での実行**: QueryDesc が適切に初期化されていない状態で実行関数に渡された場合、実行エンジンは未定義の動作を引き起こす可能性があります。
  • **実行中の解放**: 実行中のクエリに対して FreeQueryDesc を呼び出すことはできません（qdesc->estate が NULL でない場合は Assert により異常終了します）。
  • **ユーティリティコマンドの誤用**: ユーティリティコマンドを誤って ExecutorRun に渡した場合の動作は未定義です。
  • **プラグイン用領域の競合**: totaltime フィールドはコアシステムでは使用されずプラグイン用に予約されていますが、複数のプラグインが競合して利用する際には注意が必要です。
