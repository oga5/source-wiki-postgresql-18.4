RangeVarGetRelidExtended

定義

  • 種別: function
  • 定義箇所: src/backend/catalog/namespace.c:440

目的

リレーション名（RangeVar）からその OID を解決し、必要に応じてロックを取得する。DDL 操作による名前解決結果の変更を考慮し、整合性を保つための再試行ロジックを備えた主要なエントリーポイント。

シグネチャまたは形状

Oid RangeVarGetRelidExtended(const RangeVar *relation, LOCKMODE lockmode, uint32 flags, RangeVarGetRelidCallback callback, void *callback_arg)

入力と出力

  • **入力**:
  • relation: 解決対象のリレーション名（スキーマ名、カタログ名を含む）。
  • lockmode: 取得するロックモード（NoLock の場合はロックを取得しない）。
  • flags: 動作フラグ（RVR_MISSING_OK, RVR_NOWAIT, RVR_SKIP_LOCKED 等）。
  • callback: 解決後に呼び出されるオプションのコールバック関数。
  • callback_arg: コールバックに渡す引数。
  • **出力**:
  • 解決されたリレーションの Oid。見つからない場合は InvalidOid。

前提条件と不変条件

  • relation->catalogname が指定されている場合、現在のデータベース名と一致しなければならない。
  • RVR_NOWAIT と RVR_SKIP_LOCKED フラグを同時に指定することはできない（Assert で検証）。
  • lockmode が NoLock でない場合、DDL による無効化メッセージを監視し、必要に応じて再試行を行う。

アルゴリズムまたは振る舞い

1. **初期化**: SharedInvalidMessageCounter を記録し、無効化メッセージの発生を追跡する。 2. **名前解決**:

  • RELPERSISTENCE_TEMP の場合: 一時名前空間を優先して検索。
  • スキーマ名が指定されている場合: 指定されたスキーマ内を検索。
  • それ以外: search_path に基づき検索。

    • 3. **コールバック**: 解決した OID に対して callback を実行。 4. **ロック取得**: lockmode が NoLock でない場合、LockRelationOid を実行。ロック取得時に無効化メッセージを処理する。 5. **整合性チェック**: ロック取得後に SharedInvalidMessageCounter を確認し、値が変化していれば名前解決から再試行する。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/commands/cluster.c, indexcmds.c, lockcmds.c, matview.c, policy.c 等のコマンド実行モジュール。
  • **呼び出し先**: get_database_name, LookupExplicitNamespace, get_relname_relid, RelnameGetRelid, AcceptInvalidationMessages, LockRelationOid, UnlockRelationOid, ConditionalLockRelationOid。

関連データ構造

  • RangeVar: 解決対象のリレーション情報を保持する構造体。
  • LOCKMODE: ロックの強度を定義する列挙型。

失敗モードと境界条件

  • **データベース不一致**: catalogname が現在の DB と異なる場合、ERRCODE_FEATURE_NOT_SUPPORTED でエラー。
  • **スキーマ不一致**: 一時テーブルでスキーマ名が指定された場合、ERRCODE_INVALID_TABLE_DEFINITION でエラー。
  • **ロック競合**: RVR_NOWAIT 指定時にロックが取得できない場合、ERRCODE_LOCK_NOT_AVAILABLE でエラー（RVR_SKIP_LOCKED の場合は DEBUG1 メッセージのみ）。
  • **未検出**: missing_ok が指定されていない場合、リレーションが見つからなければエラーが発生する可能性がある。

保守担当者向けメモ

  • 本関数は DDL 実行中の名前解決の揺らぎを吸収する設計である。コールバック内でロックを取得するような実装は避けること（デッドロックのリスク）。
  • lockmode == NoLock の場合、無効化メッセージのチェックを行わないため、呼び出し側で適切なロックを保持している必要がある。
