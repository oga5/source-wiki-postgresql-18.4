LogicalRepRelMapEntry

定義

  • 種別: struct
  • 定義箇所: src/include/replication/logicalrelation.h:19

目的

論理レプリケーションにおいて、リモート（パブリッシャー）のテーブル定義とローカル（サブスクライバー）のテーブル定義を紐付け、データ適用に必要なマッピング情報と同期状態を保持するために使用されます。

シグネチャまたは形状

typedef struct LogicalRepRelMapEntry
{
    LogicalRepRelation remoterel;
    bool        localrelvalid;
    Oid         localreloid;
    Relation    localrel;
    AttrMap    *attrmap;
    bool        updatable;
    Oid         localindexoid;
    char        state;
    XLogRecPtr  statelsn;
} LogicalRepRelMapEntry;

入力と出力

本構造体は関数ではなくデータ構造ですが、logicalrep_rel_open() 等の関数を通じて、リモートIDに対応するローカルリレーション情報や属性マップが格納・更新されます。

前提条件と不変条件

  • localrelvalid が false の場合、次に logicalrep_rel_open() が呼び出された際に、派生情報（属性マップやインデックス情報）の再検証が必要です。
  • localrel が開かれている間は、保持しているロックにより情報が有効であるとみなされます。
  • localrel がクローズされている場合、localrel フィールドは NULL となります。

アルゴリズムまたは振る舞い

  • リモートのスキーマとローカルのスキーマが異なる場合（カラムの順序や有無）、attrmap を用いて属性の変換を行います。
  • updatable フラグは、当該テーブルに対して UPDATE や DELETE 操作が可能かどうかを判定するために使用されます。
  • localindexoid は、更新・削除操作のキーとして使用するインデックス（Replica Identity）を特定します。

呼び出し元と呼び出し先

  • 呼び出し元: src/backend/replication/logical/relation.c 内の各関数（logicalrep_relmap_update, logicalrep_rel_open, logicalrep_rel_mark_updatable 等）で参照・操作されます。
  • 呼び出し先: 構造体であるため直接の呼び出し先はありませんが、保持する Relation 型のフィールドを通じてリレーションキャッシュ（relcache）と相互作用します。

関連データ構造

  • LogicalRepRelation: リモート側のリレーション定義。
  • AttrMap: ローカル属性とリモート属性の対応付け。
  • Relation: ローカルのテーブル定義（relcacheエントリ）。

失敗モードと境界条件

  • localrelvalid が偽の状態で再検証に失敗した場合、レプリケーションの適用処理が継続できなくなる可能性があります。
  • localindexoid が InvalidOid の場合、更新や削除操作に必要なインデックスが見つからないことを示します。

保守担当者向けメモ

  • localrelvalid フラグの管理は本構造体の整合性を維持する上で極めて重要です。
  • パーティションテーブル等で利用する際は、単一の LogicalRepRelMapEntry がすべてのパーティションをカバーできない点に注意してください（src/backend/replication/logical/relation.c:44 参照）。
