RelMapFile

定義

  • 種別: struct
  • 定義箇所: src/backend/utils/cache/relmapper.c:89

目的

PostgreSQLの「マッピングされたカタログ」において、OIDと物理ファイル番号（RelFileNumber）の対応関係を永続化するためのデータ構造です。マップファイル（pg_filenode.map）のディスク上のレイアウトを定義します。

シグネチャまたは形状

typedef struct RelMapFile
{
    int32       magic;          /* always RELMAPPER_FILEMAGIC */
    int32       num_mappings;   /* number of valid RelMapping entries */
    RelMapping  mappings[MAX_MAPPINGS];
    pg_crc32c   crc;            /* CRC of all above */
} RelMapFile;

入力と出力

  • **入力**: 構造体内の各フィールド（magic、num_mappings、mappings）は、マップファイルの読み込み時にディスクから初期化されます。
  • **出力**: crc フィールドは、構造体内の他の全フィールドの整合性を検証するために使用されます。

前提条件と不変条件

  • magic フィールドは常に RELMAPPER_FILEMAGIC である必要があります。
  • mappings 配列内の有効なエントリ数は num_mappings によって管理されます。
  • crc フィールドは、構造体全体の内容と一致している必要があります。

アルゴリズムまたは振る舞い

  • 本構造体は、マップファイルの読み書きおよびメモリ上のキャッシュ状態（shared_map, local_map 等）の保持に使用されます。
  • トランザクション処理中、未コミットの変更を追跡するために、同じ構造体が active_shared_updates や pending_local_updates といった変数として利用されます。

呼び出し元と呼び出し先

  • **呼び出し元**: relmapper.c 内の各関数（apply_map_update, merge_map_updates 等）が、この構造体を引数として操作します。
  • **呼び出し先**: 特定の関数呼び出し関係はソース抜粋からは不明です。

関連データ構造

  • RelMapping: mappings 配列の要素として使用される、個別のOIDとファイル番号の対応を保持する構造体。

失敗モードと境界条件

  • magic 番号の不一致や crc チェックの失敗は、マップファイルの破損を意味し、PostgreSQL起動時の致命的なエラー（FATAL）を引き起こします。
  • MAX_MAPPINGS を超えるマッピングは保持できません。

保守担当者向けメモ

  • 本構造体は、アトミックな更新を保証するためにテンポラリファイル経由でディスクに書き出されます。
  • トランザクションのコミット時に、この構造体の内容がWALに記録され、リカバリ時に relmap_redo を通じて復元されます。
