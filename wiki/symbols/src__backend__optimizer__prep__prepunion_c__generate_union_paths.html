generate_union_paths

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/prep/prepunion.c:675

目的

UNION または UNION ALL 演算の実行パスを生成し、集合演算ツリーのノードを物理的な実行プラン（Append や MergeAppend）へ変換する。

シグネチャまたは形状

static RelOptInfo *generate_union_paths(SetOperationStmt *op, PlannerInfo *root, List *refnames_tlist, List **pTargetList)

入力と出力

  • **入力**:
  • op: 処理対象の SetOperationStmt。
  • root: 現在のクエリの PlannerInfo。
  • refnames_tlist: 参照名を含むターゲットリスト。
  • **出力**:
  • 戻り値: 集合演算の結果を表す RelOptInfo。
  • pTargetList: 構築されたターゲットリストが格納される。

前提条件と不変条件

  • op は UNION または UNION ALL である必要がある。
  • UNION（ALL なし）の場合、型変換やソート順序の整合性が求められる。
  • 集合演算の子ノードに対して plan_union_children が実行済みであること。

アルゴリズムまたは振る舞い

1. **子ノードの統合**: plan_union_children を呼び出し、マージ可能な UNION ノードを特定する。 2. **ターゲットリスト生成**: generate_append_tlist を用いて、Append ノードに必要なターゲットリストを構築する。 3. **ソートの検討**: UNION（ALL なし）の場合、generate_setop_grouplist でグループ化条件を特定し、ソート可能であれば make_pathkeys_for_sortclauses でパスキーを生成する。 4. **子パスの構築**: build_setop_child_paths を呼び出し、各子ノードに対して適切なパスを生成する。 5. **パスの収集**: 各子ノードの最安パス（cheapest_total_path）を収集し、create_append_path で Append パスを作成する。 6. **並列処理の検討**: consider_parallel が有効な場合、各子ノードの並列パスを収集し、並列 Append の可能性を評価する。 7. **結果の構築**: fetch_upper_rel を使用して UPPERREL_SETOP リレーションを作成し、コストと行数を見積もる。

呼び出し元と呼び出し先

  • **呼び出し元**: recurse_set_operations（src/backend/optimizer/prep/prepunion.c:275）から呼び出される。
  • **呼び出し先**: plan_union_children, generate_append_tlist, generate_setop_grouplist, build_setop_child_paths, create_append_path, fetch_upper_rel 等。

関連データ構造

  • SetOperationStmt: 集合演算の論理構造。
  • RelOptInfo: 集合演算の結果を保持するリレーション情報。
  • Path: 生成される実行パス。
  • PlannerInfo: クエリ全体のプランニング状態。

失敗モードと境界条件

  • 子ノードでソート済みパスが見つからない場合、try_sorted フラグが解除され、ソートを伴わないパス生成にフォールバックする。
  • 並列パスが有効でない場合、consider_parallel が無効化される。
  • 型の不一致によりターゲットリストに変換処理が追加された場合、ソート順序の維持が困難になることがある。

保守担当者向けメモ

  • 本関数は UNION 系のパス生成の中核であり、Append ノードのコスト見積もりに直結する。
  • trivial_tlist の扱いは最適化の鍵であり、型変換が不要な場合に効率的なパス生成を可能にする。
  • 並列 Append のワーカー数計算式はヒューリスティックな推定値であるため、変更時は add_paths_to_append_rel との整合性に注意すること。
