llvm_build_inline_plan

定義

  • 種別: function
  • 定義箇所: src/backend/jit/llvm/llvmjit_inline.cpp:182

目的

現在の LLVM モジュール内で未定義の外部関数参照を解析し、コストベースのヒューリスティックに基づいて、インライン化すべき関数とモジュールの対応関係（ImportMapTy）を構築する。

シグネチャまたは形状

static std::unique_ptr<ImportMapTy> llvm_build_inline_plan(LLVMContextRef lc, llvm::Module *mod)

入力と出力

  • **入力**:
  • lc: LLVM コンテキスト (LLVMContextRef)。
  • mod: インライン化対象の LLVM モジュール (llvm::Module*)。
  • **出力**:
  • インポートすべき関数名とモジュールパスのマップを保持する std::unique_ptr<ImportMapTy>。

前提条件と不変条件

  • defaultSearchPath に $libdir/postgres が追加可能であること（追加できない場合は nullptr を返す）。
  • インライン化対象の関数定義は、LLVM のビットコードインデックスから取得可能であること。
  • funcDef は外部リンケージ（hasExternalLinkage）を持つ必要がある。

アルゴリズムまたは振る舞い

1. **初期化**: モジュール内の全関数を走査し、宣言のみ（isDeclaration）かつ組み込み関数（isIntrinsic）でないものを worklist に追加する。 2. **探索**: worklist が空になるまで以下の処理を繰り返す。

  • シンボル名からモジュール名と関数名を分離し、GUID を計算する。
  • summaries_for_guid を用いてインデックスから関数定義を検索する。
  • fs->instCount() がコスト制限を超えていないか確認する。
  • load_module_cached で対象モジュールをロードし、function_inlinable を呼び出して依存関係を含めたインライン化の可否を判定する。

    • 3. **計画確定**: インライン化可能と判定された場合、importVars に対象関数を追加し、globalsToInline マップに記録する。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/jit/llvm/llvmjit_inline.cpp 内のインライン化処理フローから呼び出される。
  • **呼び出し先**:
  • add_module_to_inline_search_path
  • llvm_split_symbol_name
  • summaries_for_guid
  • load_module_cached
  • function_inlinable

関連データ構造

  • ImportMapTy: インポートする関数とモジュールのマッピング。
  • FunctionInlineStates: 各関数のインライン化状態（コスト制限、処理済みフラグ等）を管理。
  • InlineWorkList: 処理待ちの関数リスト。
  • InlineSearchPath: モジュール検索パス。

失敗モードと境界条件

  • **検索パスの欠如**: defaultSearchPath が空の場合、nullptr を返す。
  • **メタデータ取得失敗**: defMod->materializeMetadata() または funcDef->materialize() が失敗した場合、FATAL エラーで終了する。
  • **ハッシュ衝突**: 関数名の GUID 衝突により定義が見つからない場合、そのエントリはスキップされる。
  • **コスト超過**: instCount が inlineState.costLimit を超える場合、インライン化は見送られる。

保守担当者向けメモ

  • LLVM_VERSION_MAJOR >= 21 の条件分岐により、GUID 計算方法が LLVM のバージョンに依存している点に注意。
  • inline_cost_decay_factor (0.5) が再帰的なコスト計算に影響を与えるため、インライン化の深さに応じて許容値が厳しくなる。
  • INLINE_DEBUG マクロを有効にすると、判定プロセスが elog で詳細に出力される。
