SplitPageLayout

定義

  • **種別**: struct
  • **定義箇所**: src/include/access/gist_private.h:191

目的

GiSTインデックスのページ分割処理において、分割後の新しいページレイアウト情報を保持し、インデックスの再構築や更新を制御するために使用されます。

シグネチャまたは形状

typedef struct SplitPageLayout
{
    gistxlogPage block;
    IndexTupleData *list;
    int            lenlist;
    IndexTuple     itup;
    Page           page;
    Buffer         buffer;
    struct SplitPageLayout *next;
} SplitPageLayout;

入力と出力

本構造体は関数への引数や戻り値として渡され、分割後のページ状態（page）、書き込み対象のバッファ（buffer）、およびページ内のタプルリスト（list）を伝達します。

前提条件と不変条件

  • next ポインタにより、複数のページ分割結果を連結したリスト構造を形成可能です。
  • buffer は、すべての処理が完了した後に書き込まれる対象として管理されます。

アルゴリズムまたは振る舞い

GiSTのページ分割処理（gistSplit）において、分割によって生成された各ページの情報を格納します。gistxlogPage を介してWAL記録に必要な情報を保持し、最終的に buffer を通じてディスクへの書き込みを確定させる一連のフローをサポートします。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/access/gist/gist.c、src/backend/access/gist/gistbuild.c、src/backend/access/gist/gistxlog.c 内の各関数が本構造体をインスタンス化し、分割処理の管理に使用しています。
  • **呼び出し先**: 本構造体自体はデータ保持用であり、直接的な関数呼び出し先は定義されていません。

関連データ構造

  • gistxlogPage: 分割されたページのWAL記録用データ。
  • IndexTupleData: ページ内に格納されるインデックスエントリ。
  • Buffer: 共有バッファ管理におけるページバッファ。

失敗モードと境界条件

  • メモリ割り当て失敗時（palloc0 等）の挙動については、PostgreSQLの標準的なメモリ管理エラーに依存します。
  • 連結リストの終端は next が NULL であることで判定されます。

保守担当者向けメモ

  • 本構造体は gistSplit 関数の結果を保持する役割を担います。
  • gist_private.h 内の他の構造体（GISTInsertStack 等）と連携して動作するため、変更時はGiSTの挿入・分割ロジック全体への影響を考慮してください。
