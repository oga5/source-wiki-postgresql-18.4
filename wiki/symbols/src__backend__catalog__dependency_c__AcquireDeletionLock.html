AcquireDeletionLock

定義

  • 種別: function
  • 定義箇所: src/backend/catalog/dependency.c:1495

目的

データベースオブジェクトの削除処理に先立ち、対象オブジェクトに対して適切なロックを取得する。

シグネチャまたは形状

void AcquireDeletionLock(const ObjectAddress *object, int flags)

入力と出力

  • object: ロック対象のオブジェクトアドレス（ObjectAddress 構造体へのポインタ）。
  • flags: 削除処理のオプションフラグ（例: PERFORM_DELETION_CONCURRENTLY）。
  • 戻り値: なし。

前提条件と不変条件

  • object が指す構造体は有効であること。
  • 呼び出し元は、対象オブジェクトの削除権限や整合性チェックを事前に考慮している必要がある。

アルゴリズムまたは振る舞い

1. object->classId が RelationRelationId（リレーション）の場合:

  • flags に PERFORM_DELETION_CONCURRENTLY が含まれる場合、ShareUpdateExclusiveLock を取得する。
  • それ以外の場合、AccessExclusiveLock を取得する。

    • 2. object->classId が AuthMemRelationId の場合、AccessExclusiveLock を共有オブジェクトに対して取得する。 3. 上記以外の場合、対象オブジェクトに対して AccessExclusiveLock を取得する。

呼び出し元と呼び出し先

  • 呼び出し元: src/backend/catalog/dependency.c 内の削除関連関数、および src/backend/catalog/pg_shdepend.c。
  • 呼び出し先: LockRelationOid, LockSharedObject, LockDatabaseObject（いずれもロックマネージャの関数）。

関連データ構造

  • ObjectAddress: 削除対象を識別するための構造体。
  • RelationRelationId, AuthMemRelationId: ロック対象を判別するためのカタログID。

失敗モードと境界条件

  • ロック取得時にデッドロックが発生する可能性がある（ロックマネージャの挙動に依存）。
  • 無効な classId や objectId が渡された場合の挙動は、ロックマネージャ側のエラー処理に依存する。

保守担当者向けメモ

  • DROP INDEX CONCURRENTLY のような特殊なケースでは、index_drop() が後続でロックを昇格させる前提で、一時的に弱いロック（ShareUpdateExclusiveLock）を取得する設計となっている。
  • 新しいオブジェクトタイプを追加する際は、適切なロック関数（LockRelationOid 等）が選択されるよう条件分岐の確認が必要。
