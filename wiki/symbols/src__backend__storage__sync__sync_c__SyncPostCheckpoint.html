SyncPostCheckpoint

定義

  • 種別: function
  • 定義箇所: src/backend/storage/sync/sync.c:201

目的

チェックポイント完了後に、削除待ちとなっているファイル（pendingUnlinks）を物理的に削除し、リストを整理する。

シグネチャまたは形状

void SyncPostCheckpoint(void)

入力と出力

  • 入力: なし（グローバル変数 pendingUnlinks および checkpoint_cycle_ctr を参照）
  • 出力: なし（副作用としてファイルシステム上のファイル削除およびメモリ解放を行う）

前提条件と不変条件

  • pendingUnlinks リストには、削除対象の PendingUnlinkEntry が格納されていること。
  • 新しい要求はリストの末尾に追加されるため、cycle_ctr が現在の checkpoint_cycle_ctr と一致するエントリは、今回のチェックポイント以降に追加されたものとみなされる。

アルゴリズムまたは振る舞い

1. pendingUnlinks リストを先頭から走査する。 2. canceled フラグが立っているエントリはスキップする。 3. cycle_ctr が現在の checkpoint_cycle_ctr に達した場合、それ以降のエントリは今回のチェックポイント後に登録されたものと判断し、走査を終了する。 4. syncsw テーブルから該当ハンドラの sync_unlinkfiletag を呼び出し、ファイルを削除する。 5. 削除失敗時、ENOENT（ファイルが存在しない）以外のエラーであれば警告を出力する。 6. 処理中、一定数（UNLINKS_PER_ABSORB）の削除ごとに AbsorbSyncRequests() を呼び出し、fsync 要求の取り込みを継続する。 7. 処理済みのエントリをリストから削除し、メモリを解放する。

呼び出し元と呼び出し先

  • 呼び出し元: src/backend/access/transam/xlog.c
  • 呼び出し先:
  • syncsw[...].sync_unlinkfiletag (ハンドラ関数)
  • AbsorbSyncRequests()

関連データ構造

  • pendingUnlinks: 削除待ちファイルリスト。
  • PendingUnlinkEntry: 削除対象ファイルの情報（タグ、サイクルカウンタ、キャンセル状態）を保持する構造体。
  • syncsw: 各種ストレージハンドラの関数テーブル。

失敗モードと境界条件

  • **競合**: ファイルが既に削除されている場合（ENOENT）、エラーを無視して続行する。
  • **サイクルカウンタのラップアラウンド**: 連続するチェックポイントの失敗によりカウンタが一周した場合、削除が1チェックポイント分遅延する可能性があるが、許容範囲内として処理される。
  • **メモリ不足**: リスト操作時に pfree を行うが、リストの断片化やメモリ確保の失敗については本関数内での明示的なハンドリングは記述されていない。

保守担当者向けメモ

  • 本関数はチェックポイント処理の最終段階で呼び出される。
  • AbsorbSyncRequests() を呼び出すことで、削除処理中に fsync 要求の処理が長時間停止することを防いでいる。
  • リストの走査終了条件（cycle_ctr == checkpoint_cycle_ctr）は、チェックポイントの境界を正しく管理するための重要な不変条件である。
