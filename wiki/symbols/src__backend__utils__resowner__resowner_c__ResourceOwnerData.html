ResourceOwnerData

定義

  • 種別: struct
  • 定義箇所: src/backend/utils/resowner/resowner.c:112

目的

トランザクションやクエリの生存期間中に確保されたリソース（バッファピン、ロック、メモリコンテキスト等）を追跡し、階層的な管理と確実な解放を行うための内部状態保持構造体。

シグネチャまたは形状

struct ResourceOwnerData {
    ResourceOwner parent;
    ResourceOwner firstchild;
    ResourceOwner nextchild;
    const char *name;
    bool releasing;
    bool sorted;
    uint8 nlocks;
    uint8 narr;
    uint32 nhash;
    ResourceElem arr[RESOWNER_ARRAY_SIZE];
    ResourceElem *hash;
    uint32 capacity;
    uint32 grow_at;
    LOCALLOCK *locks[MAX_RESOWNER_LOCKS];
    dlist_head aio_handles;
};

入力と出力

  • **入力**: リソース登録時（ResourceOwnerRemember）に ResourceElem が追加され、nlocks, narr, nhash が更新される。
  • **出力**: 解放時（ResourceOwnerRelease）に、優先順位に従ってソートされたリソースが順次解放される。

前提条件と不変条件

  • ResourceOwnerRemember() を呼び出す前に、必ず ResourceOwnerEnlarge() で十分な容量を確保する必要がある。
  • 解放処理（ResourceOwnerRelease）が開始された後は、新規リソースの登録や個別の解放は禁止される。
  • リソースの解放は BEFORE_LOCKS -> LOCKS -> AFTER_LOCKS の順序で行われる。

アルゴリズムまたは振る舞い

  • **リソース管理**: 高速な固定長配列（arr）と、オーバーフロー時に使用されるハッシュテーブル（hash）の二段構えでリソースを保持する。
  • **階層処理**: parent, firstchild, nextchild を用いてツリー構造を形成し、再帰的にリソースを解放する。
  • **ソート**: 解放フェーズに入ると、sorted フラグが立てられ、優先順位に基づいてリソースがソートされる。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/storage/aio/aio.c 内の pgaio_io_resowner_register, pgaio_io_acquire, pgaio_io_acquire_nb 等から参照されている。
  • **呼び出し先**: 本構造体自体はデータ保持用であり、具体的な解放処理は ResourceOwnerRelease 等の関数が本構造体のメンバを操作することで実行される。

関連データ構造

  • ResourceElem: 個々のリソース情報を保持する要素。
  • LOCALLOCK: ローカルロックの追跡用。
  • dlist_head: AIO ハンドル管理用リスト。

失敗モードと境界条件

  • **ハッシュテーブルの拡張**: nhash が grow_at に達すると、ハッシュテーブルの再構築（リハッシュ）が発生する。
  • **メモリ管理**: TopMemoryContext に割り当てられるため、明示的に ResourceOwnerDelete() を呼ばない限りメモリリークの原因となる。

保守担当者向けメモ

  • releasing フラグと sorted フラグは、ResourceOwnerReleaseAllOfKind() による一時的な状態変更と、通常の解放処理を区別するために個別に管理されている。
  • AIO ハンドルはクリティカルセクションでの登録が必要なため、通常の ResourceElem 機構ではなく dlist_head を介して管理される。
