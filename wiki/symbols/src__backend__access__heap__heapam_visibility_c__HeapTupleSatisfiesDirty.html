HeapTupleSatisfiesDirty

定義

  • 種別: function
  • 定義箇所: src/backend/access/heap/heapam_visibility.c:742

目的

タプルが「ダーティ（更新中または未確定）」な状態にあるかどうかを判定し、現在のトランザクションから見て可視であるかを判断します。主に、他のトランザクションによる更新の競合や、投機的挿入（speculative insertion）の追跡に使用されます。

シグネチャまたは形状

bool HeapTupleSatisfiesDirty(HeapTuple htup, Snapshot snapshot, Buffer buffer)

入力と出力

  • **入力**:
  • htup: 判定対象のタプル。
  • snapshot: 可視性判定結果を格納する構造体（xmin, xmax, speculativeToken が更新される）。
  • buffer: タプルが格納されているバッファ。
  • **出力**:
  • bool: タプルが可視であれば true、不可視であれば false を返します。

前提条件と不変条件

  • 対象のバッファに対して、少なくとも共有ロック（Shared Lock）を保持している必要があります。
  • htup->t_self が有効であり、t_tableOid が設定されている必要があります。

アルゴリズムまたは振る舞い

1. **初期化**: snapshot の xmin, xmax を InvalidTransactionId に、speculativeToken を 0 にリセットします。 2. **Xmin 判定**:

  • 未コミットの場合、HEAP_MOVED_OFF や HEAP_MOVED_IN フラグを確認し、古いバイナリアップグレード時の特殊処理を行います。
  • 現在のトランザクションによる挿入であれば、xmax の状態に応じて可視性を判定します。
  • 他のトランザクションによる挿入で進行中であれば、xmin を snapshot->xmin に設定し、投機的挿入であれば speculativeToken も取得して true を返します。
  • コミット済みであればヒントビットを更新します。

    • 3. **Xmax 判定**:

  • 挿入トランザクションがコミット済みの場合、t_infomask を確認します。
  • HEAP_XMAX_INVALID であれば可視（true）。
  • HEAP_XMAX_COMMITTED であれば、ロックのみか更新かを確認し、更新済みなら不可視（false）。
  • マルチトランザクションや進行中の更新トランザクションがある場合、それに応じて snapshot->xmax を設定し、可視性を決定します。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/access/heap/heapam.c:6165
  • src/backend/access/heap/heapam_handler.c:484
  • src/backend/access/heap/heapam_visibility.c:1789
  • **呼び出し先**:
  • SetHintBits (ヒントビット更新用)
  • TransactionIdIsCurrentTransactionId, TransactionIdIsInProgress, TransactionIdDidCommit (トランザクション状態確認用)
  • HeapTupleHeaderGetRawXmin, HeapTupleHeaderGetRawXmax 等のアクセサ

関連データ構造

  • HeapTuple: 判定対象のタプル構造体。
  • Snapshot: 可視性判定の結果や状態を保持する構造体。
  • Buffer: タプルが属するバッファ。

失敗モードと境界条件

  • TransactionId が無効な場合や、トランザクションがアボート/クラッシュした場合は、ヒントビットを HEAP_XMIN_INVALID または HEAP_XMAX_INVALID に設定し、適切に false または true を返します。
  • 投機的挿入のトークンが 0 の場合はアサーションエラーとなります。

保守担当者向けメモ

  • 本関数は、HeapTupleSatisfiesVisibility のような汎用的なディスパッチャではなく、特定のダーティ状態を判定するための特殊な関数です。
  • TransactionIdIsInProgress を TransactionIdDidCommit より先にチェックする順序は、コミット直後のレースコンディションを回避するために厳守してください。
