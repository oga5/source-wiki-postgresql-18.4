cached_function_compile

定義

  • 種別: function
  • 定義箇所: src/backend/utils/cache/funccache.c:479

目的

関数実行データのキャッシュ管理におけるメインエントリポイント。関数定義（pg_proc）と実行コンテキストに基づき、コンパイル済み関数を検索、検証、または新規生成して提供する。

シグネチャまたは形状

CachedFunction *cached_function_compile(
    FunctionCallInfo fcinfo,
    CachedFunction *function,
    CachedFunctionCompileCallback ccallback,
    CachedFunctionDeleteCallback dcallback,
    Size cacheEntrySize,
    bool includeResultType,
    bool forValidator
)

入力と出力

  • **入力**:
  • fcinfo: 関数呼び出し情報。
  • function: 既存のキャッシュエントリ（またはNULL）。
  • ccallback: 言語固有のコンパイル処理を行うコールバック関数。
  • dcallback: キャッシュ削除時に呼び出されるコールバック関数。
  • cacheEntrySize: キャッシュエントリのサイズ。
  • includeResultType, forValidator: キャッシュキー生成用のフラグ。
  • **出力**: コンパイル済み CachedFunction 構造体へのポインタ。

前提条件と不変条件

  • cacheEntrySize は sizeof(CachedFunction) 以上であること。
  • キャッシュエントリは TopMemoryContext に割り当てられ、バックエンドの生存期間中保持される。
  • pg_proc の xmin および ctid を用いて、キャッシュの有効性を検証する。

アルゴリズムまたは振る舞い

1. **検索**: fcinfo から CachedFunctionHashKey を生成し、ハッシュテーブルを検索する。 2. **検証**: キャッシュが存在する場合、pg_proc の xmin/ctid と比較し、定義が最新か確認する。古い場合は delete_function を呼び出し、必要に応じて再検索する。 3. **コンパイル**: キャッシュミスまたは無効な場合、ccallback を実行して関数をコンパイルする。コンパイル失敗時は PG_TRY/PG_CATCH ブロックによりメモリリークを防ぐ。 4. **挿入**: コンパイル済みエントリをハッシュテーブルに登録し、関数を返す。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/executor/functions.c
  • src/pl/plpgsql/src/pl_comp.c
  • **呼び出し先**:
  • compute_function_hashkey
  • cfunc_hashtable_lookup
  • cfunc_hashtable_insert
  • delete_function
  • SearchSysCache1 / ReleaseSysCache
  • ccallback (引数として渡された関数)

関連データ構造

  • CachedFunction: コンパイル済み関数の基底構造体。
  • CachedFunctionHashKey: キャッシュ検索用のキー構造体。
  • FunctionCallInfo: 関数呼び出しのメタデータ。

失敗モードと境界条件

  • pg_proc の検索に失敗した場合、elog(ERROR) を発生させる。
  • コンパイル中にエラーが発生した場合、PG_RE_THROW により例外を再送出し、新規確保したメモリを解放する。
  • use_count が 0 でないキャッシュエントリは、実行中のクエリが存在する可能性があるため、即座に解放されない場合がある。

保守担当者向けメモ

  • 本関数は TopMemoryContext を使用するため、メモリリークには注意が必要。
  • ccallback 内での内部的なメモリリーク防止は、コールバック側の責務である。
  • 多相型関数の場合、実行時に確定した引数型がキャッシュキーに反映される。
