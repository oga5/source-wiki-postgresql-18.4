process_target_wal_block_change

定義

  • 種別: function
  • 定義箇所: src/bin/pg_rewind/filemap.c:352

目的

WAL 解析中にターゲットデータディレクトリで発生したブロック変更を追跡し、ソースからコピーすべきページを特定して記録する。

シグネチャまたは形状

void process_target_wal_block_change(ForkNumber forknum, RelFileLocator rlocator, BlockNumber blkno)

入力と出力

  • **入力**:
  • forknum: 変更されたフォーク番号。
  • rlocator: 変更されたリレーションの識別子。
  • blkno: 変更されたブロック番号。
  • **出力**: なし（副作用として file_entry_t 内のビットマップを更新）。

前提条件と不変条件

  • 対象ファイルが filehash に存在し、かつ isrelfile であること。
  • ターゲットファイルが FILE_TYPE_REGULAR であること。
  • 変更対象のブロックが、ソースとターゲット双方のファイルサイズ内に収まっていること。

アルゴリズムまたは振る舞い

1. 指定された rlocator と blkno からファイルパスを生成し、lookup_filehash_entry で対応する file_entry_t を取得する。 2. ファイルがハッシュテーブルに存在しない場合、またはターゲットに存在しない場合は無視する。 3. ファイルがターゲットに存在する場合、target_type が通常ファイルであることを確認する。 4. ソースにもファイルが存在する場合、ブロックオフセットがソースおよびターゲットのファイルサイズを超えていないか検証する。 5. 条件を満たす場合、datapagemap_add を呼び出し、当該ファイルの target_pages_to_overwrite ビットマップにブロック番号を追加する。

呼び出し元と呼び出し先

  • **呼び出し元**: src/bin/pg_rewind/parsexlog.c:481 (WAL 解析処理)
  • **呼び出し先**:
  • datasegpath (パス生成)
  • lookup_filehash_entry (ハッシュテーブル検索)
  • datapagemap_add (ビットマップ更新)
  • pg_fatal (エラーハンドリング)

関連データ構造

  • file_entry_t: ファイルのメタデータおよび同期アクションを保持。
  • datapagemap: ページレベルの変更追跡用ビットマップ。
  • RelFileLocator: リレーションの物理的な位置を特定する構造体。

失敗モードと境界条件

  • **非通常ファイル**: target_type が FILE_TYPE_REGULAR でない場合、pg_fatal により異常終了する。
  • **ファイル不在**: ソースまたはターゲットのいずれか一方（あるいは両方）にファイルが存在しない場合、変更は無視される。
  • **範囲外**: ブロックオフセットがファイルサイズを超える場合、ビットマップへの追加は行われない。

保守担当者向けメモ

  • 本関数はメインフォークの変更追跡を前提としています。他のフォークは常に全体コピーされるため、本関数での処理対象外です。
  • datasegpath で生成されたパスは pfree で解放されるため、メモリリークに注意してください。
