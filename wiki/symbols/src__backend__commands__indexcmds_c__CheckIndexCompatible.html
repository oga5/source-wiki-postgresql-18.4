CheckIndexCompatible

定義

  • 種別: function
  • 定義箇所: src/backend/commands/indexcmds.c:177

目的

既存のインデックスが、指定された新しいインデックス定義と互換性があるかを判定します。互換性がある場合、インデックスの再構築を回避できる可能性があります。

シグネチャまたは形状

bool CheckIndexCompatible(Oid oldId,
                          const char *accessMethodName,
                          const List *attributeList,
                          const List *exclusionOpNames,
                          bool isWithoutOverlaps)

入力と出力

  • **入力**:
  • oldId: 既存インデックスのOID。
  • accessMethodName: アクセスメソッド名。
  • attributeList: インデックス対象属性のリスト。
  • exclusionOpNames: 除外演算子名のリスト。
  • isWithoutOverlaps: 重複なし制約の有無。
  • **出力**:
  • bool: 互換性がある場合は true、それ以外は false を返します。

前提条件と不変条件

  • 呼び出し元は、対象のインデックス（oldId）が属するリレーションに対して適切なロックを保持している必要があります。
  • attributeList の長さは 1 以上 INDEX_MAX_KEYS 以下である必要があります。
  • 述語（indpred）や式（indexprs）を含むインデックスは、互換性がないものとして扱われます。

アルゴリズムまたは振る舞い

1. 指定されたアクセスメソッドのルーチンを取得し、属性リストから新しいインデックスの演算子クラス、照合順序、型情報を計算します。 2. 既存インデックスの pg_index カタログ情報を取得し、述語や式が存在しないこと、および indisvalid が真であることを確認します。 3. 演算子クラスおよび照合順序が一致するかを比較します。 4. 多相型（Polymorphic type）を使用している場合、列の型が一致するかを検証します。 5. 演算子クラスのオプション（opclass options）を比較します。 6. 除外演算子が指定されている場合、その演算子および多相型の入力型が一致するかを検証します。 7. すべての条件を満たす場合に true を返します。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/commands/indexcmds.c
  • src/backend/commands/tablecmds.c (ATPostAlterTypeParse からの呼び出し)
  • **呼び出し先**:
  • IndexGetRelation, SearchSysCache1, GetIndexAmRoutine, makeIndexInfo, ComputeIndexAttrs, SysCacheGetAttrNotNull, index_open, get_opclass_input_type, TupleDescAttr, get_attoptions, CompareOpclassOptions, RelationGetExclusionInfo, op_input_types, index_close 等。

関連データ構造

  • IndexInfo: インデックスの定義情報を保持する構造体。
  • Form_pg_index: pg_index システムカタログの行データ。
  • IndexAmRoutine: インデックスアクセスメソッドのインターフェース。

失敗モードと境界条件

  • アクセスメソッドが見つからない場合、エラーが発生します。
  • 既存インデックスのキャッシュルックアップに失敗した場合、エラーが発生します。
  • 述語や式が含まれるインデックス、または無効なインデックスに対しては、即座に false を返します。
  • 演算子クラス、照合順序、オプション、または除外演算子の不一致は、互換性なし（false）と判定されます。

保守担当者向けメモ

  • 本関数は REINDEX や ALTER TABLE におけるインデックス再構築の最適化判断に使用されます。
  • 多相型（IsPolymorphicType）の判定ロジックは、型変更時の互換性維持において重要です。
  • 比較ロジックの追加や変更を行う際は、pg_index の属性定義との整合性に注意してください。
