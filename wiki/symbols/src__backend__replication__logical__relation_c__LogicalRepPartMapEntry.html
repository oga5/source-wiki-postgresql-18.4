LogicalRepPartMapEntry

定義

  • 種別: struct
  • 定義箇所: src/backend/replication/logical/relation.c:51

目的

論理レプリケーションにおいて、パーティションテーブルの各リーフパーティションに対するマッピング情報を保持するためのキャッシュエントリ構造体です。

シグネチャまたは形状

typedef struct LogicalRepPartMapEntry
{
    Oid            partoid;        /* LogicalRepPartMap's key */
    LogicalRepRelMapEntry relmapentry;
} LogicalRepPartMapEntry;

入力と出力

本構造体はデータ構造として定義されており、関数としての入出力は持ちません。ハッシュテーブル LogicalRepPartMap のエントリとして使用されます。

前提条件と不変条件

  • partoid は、対応するリーフパーティションの Oid を保持します。
  • relmapentry には、そのパーティションに関連付けられた論理レプリケーション用のマッピング情報（属性マップやローカルリレーションのポインタ等）が格納されます。

アルゴリズムまたは振る舞い

  • 本構造体は、パーティションテーブルの各リーフパーティションごとに個別のマッピングを管理するために使用されます。
  • hash_search を用いて LogicalRepPartMap から検索・挿入が行われます。
  • memset により初期化されることが確認されています。

呼び出し元と呼び出し先

  • 呼び出し元: src/backend/replication/logical/relation.c 内のパーティション管理関数群（hash_seq_search を使用した走査や、hash_search を使用したエントリの取得・生成処理）。
  • 呼び出し先: 本構造体自体はデータ保持用であり、直接関数を呼び出すことはありません。

関連データ構造

  • LogicalRepRelMapEntry: 本構造体の一部として埋め込まれており、リレーションの属性マッピングや状態を保持します。
  • LogicalRepPartMap: 本構造体を格納するハッシュテーブル。

失敗モードと境界条件

  • ハッシュテーブルのメモリ不足時や、無効な Oid が指定された場合の挙動については、本構造体の定義からは直接判断できません。

保守担当者向けメモ

  • パーティションテーブルの処理において、親テーブルのマップを共有せず、リーフパーティションごとに本構造体を作成して管理する設計になっています。
  • LogicalRepRelMapEntry の変更が本構造体内の relmapentry に与える影響を考慮してください。
