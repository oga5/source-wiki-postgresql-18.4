ResolveRecoveryConflictWithSnapshotFullXid

定義

  • **種別**: 関数
  • **定義箇所**: src/backend/storage/ipc/standby.c:511

目的

ホットスタンバイ環境において、プライマリから受信したWALレコードが示すスナップショットの競合を解決します。FullTransactionId（64bit）を引数に取り、必要に応じて32bitのトランザクションIDへ変換した上で競合解決処理を委譲します。

シグネチャまたは形状

void ResolveRecoveryConflictWithSnapshotFullXid(
    FullTransactionId snapshotConflictHorizon,
    bool isCatalogRel,
    RelFileLocator locator
)

入力と出力

  • **入力**:
  • snapshotConflictHorizon: 競合の基準となる64bitトランザクションID。
  • isCatalogRel: 対象がシステムカタログリレーションであるかを示すフラグ。
  • locator: 競合が発生したリレーションの識別子。
  • **出力**: なし（副作用としてクエリのキャンセルや待機が発生する可能性があります）。

前提条件と不変条件

  • 渡された snapshotConflictHorizon が現在の nextXid に対して極端に古い（ラップアラウンド済み）場合、競合は発生しないものとみなされ、処理はスキップされます。

アルゴリズムまたは振る舞い

1. 現在の nextXid を取得し、snapshotConflictHorizon との差分を計算します。 2. 差分が MaxTransactionId / 2 未満である場合、まだ競合の可能性があると判断します。 3. FullTransactionId を32bitの TransactionId に切り詰め、ResolveRecoveryConflictWithSnapshot を呼び出して実際の競合解決（クエリの待機またはキャンセル）を実行します。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/access/gist/gistxlog.c
  • src/backend/access/nbtree/nbtxlog.c
  • **呼び出し先**:
  • ResolveRecoveryConflictWithSnapshot（競合解決の実行）
  • ReadNextFullTransactionId（現在のXID取得）
  • U64FromFullTransactionId / XidFromFullTransactionId（型変換）

関連データ構造

  • FullTransactionId: 64bitトランザクションID。
  • RelFileLocator: リレーションの物理的な位置情報。

失敗モードと境界条件

  • **ラップアラウンド**: 渡されたXIDが非常に古い場合、計算上の差分が大きくなり、競合解決処理がスキップされます。これは、そのXIDが既にどのスナップショットからも参照されていないことを前提としています。

保守担当者向けメモ

  • 本関数は ResolveRecoveryConflictWithSnapshot のラッパーとして機能します。64bit XIDを扱う必要があるWALレコード（GiSTやB-treeのインデックス操作など）から呼び出されます。
  • 競合解決のロジックを変更する場合は、内部で呼び出している ResolveRecoveryConflictWithSnapshot の仕様を確認してください。
