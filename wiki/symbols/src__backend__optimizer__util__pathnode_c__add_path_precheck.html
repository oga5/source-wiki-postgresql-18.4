add_path_precheck

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/util/pathnode.c:690

目的

新しいパスを RelOptInfo のパスリストに追加する前に、既存のパスによってそのパスが明らかに支配（劣後）されているかを判定し、不要なパス生成を早期に回避する。

シグネチャまたは形状

bool add_path_precheck(RelOptInfo *parent_rel, int disabled_nodes,
                       Cost startup_cost, Cost total_cost,
                       List *pathkeys, Relids required_outer)

入力と出力

  • **入力**:
  • parent_rel: 対象となるリレーション情報。
  • disabled_nodes: 無効化されたノード数。
  • startup_cost, total_cost: 新規パスのコスト。
  • pathkeys: 新規パスのソート順序。
  • required_outer: パラメータ化されたパスの外部参照リレーション。
  • **出力**:
  • bool: 新規パスが検討に値する場合は true、既存のパスに支配されており破棄すべき場合は false を返す。

前提条件と不変条件

  • parent_rel->pathlist は、disabled_nodes の昇順、次いで total_cost の昇順でソートされている必要がある。
  • パラメータ化されたパス（required_outer が非空）の場合、pathkeys は NIL として扱われる。

アルゴリズムまたは振る舞い

1. parent_rel->pathlist を走査し、コストとソート順序に基づき既存パスと比較する。 2. 既存パスの disabled_nodes が新規パスより多い場合、または total_cost が新規パスのコストを十分に下回る場合は走査を終了する。 3. 以下の条件をすべて満たす既存パスが見つかった場合、新規パスは支配されていると判断し false を返す：

  • コスト（起動コストおよび総コスト）が新規パスと同等かそれ以上である。
  • pathkeys が新規パスと同等か、より優れている。
  • required_outer が一致している。

    • 4. 走査完了まで支配関係が見つからなければ true を返す。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/optimizer/path/joinpath.c (複数箇所)
  • src/backend/optimizer/util/pathnode.c (例: add_partial_path 等の内部)
  • **呼び出し先**:
  • compare_pathkeys: パスキーの比較に使用。
  • bms_equal: required_outer の比較に使用。

関連データ構造

  • RelOptInfo: パスリストを保持するリレーション情報。
  • Path: 比較対象となるパス構造体。
  • List: pathkeys および pathlist の管理に使用。

失敗モードと境界条件

  • STD_FUZZ_FACTOR を用いた浮動小数点数の曖昧比較を行うため、極めて微小なコスト差は無視される。
  • required_outer が異なるパス同士は、この関数では直接的な支配関係の判定対象外となる。

保守担当者向けメモ

  • 本関数は add_path の最適化版であり、コスト計算が重い処理の前に呼び出すことで計算コストを削減する。
  • 比較ロジックは compare_path_costs_fuzzily と整合性を保つ必要がある。
