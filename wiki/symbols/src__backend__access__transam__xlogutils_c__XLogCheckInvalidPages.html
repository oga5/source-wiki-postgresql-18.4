XLogCheckInvalidPages

定義

  • 種別: function
  • 定義箇所: src/backend/access/transam/xlogutils.c:233

目的

WALリプレイの終了時またはリカバリの節目において、存在しないページへの参照が残存していないかを検証し、整合性を保証する。

シグネチャまたは形状

void XLogCheckInvalidPages(void)

入力と出力

  • 入力: なし（グローバル変数 invalid_page_tab を参照）
  • 出力: なし（副作用としてログ出力または PANIC によるプロセス停止）

前提条件と不変条件

  • 本関数はリカバリプロセス中のみ呼び出されることを前提とする。
  • invalid_page_tab が初期化されている場合、そのハッシュテーブル内の全エントリを走査する。
  • 処理終了後、invalid_page_tab は破棄され、ポインタは NULL にリセットされる。

アルゴリズムまたは振る舞い

1. invalid_page_tab が NULL の場合は即座に終了する。 2. ハッシュテーブル invalid_page_tab を走査し、各エントリに対して report_invalid_page を呼び出して WARNING レベルのログを出力する。 3. 走査の結果、一つでも無効なページが検出された場合、ignore_invalid_pages フラグの状態を確認する。

  • ignore_invalid_pages が真であれば WARNING を出力する。
  • 偽であれば PANIC を発生させ、リカバリを中断する。

    • 4. 最後に hash_destroy を用いてハッシュテーブルを解放し、invalid_page_tab を NULL に設定する。

呼び出し元と呼び出し先

  • 呼び出し元: src/backend/access/transam/xlogrecovery.c:2256
  • 呼び出し先: report_invalid_page（内部関数）、hash_seq_init, hash_seq_search, hash_destroy（utils/hsearch.h）

関連データ構造

  • invalid_page_tab: 無効なページ情報を保持するハッシュテーブル。
  • xl_invalid_page: ハッシュテーブルに格納されるエントリ構造体。

失敗モードと境界条件

  • invalid_page_tab が空の場合: 何もせず正常終了する。
  • 無効なページが検出された場合: 原則として PANIC を発生させ、データベースの整合性エラーとしてリカバリを停止させる。

保守担当者向けメモ

  • 本関数はリカバリの最終段階で呼び出されることを想定している。通常のバックエンドプロセスから呼び出すと、意図しない PANIC を引き起こす可能性があるため注意が必要。
  • ignore_invalid_pages はデバッグや特定のリカバリシナリオでのみ使用される設定である。
