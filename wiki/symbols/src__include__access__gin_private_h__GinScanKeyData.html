GinScanKeyData

定義

  • 種別: struct
  • 定義箇所: src/include/access/gin_private.h:269

目的

GIN インデックススキャンにおける単一のクエリ条件式を保持し、インデックス内のタプルがクエリ条件を満たすかを判定するための状態を管理します。

シグネチャまたは形状

typedef struct GinScanKeyData {
    uint32      nentries;
    uint32      nuserentries;
    GinScanEntry *scanEntry;
    GinScanEntry *requiredEntries;
    int         nrequired;
    GinScanEntry *additionalEntries;
    int         nadditional;
    GinTernaryValue *entryRes;
    bool        (*boolConsistentFn) (GinScanKey key);
    GinTernaryValue (*triConsistentFn) (GinScanKey key);
    FmgrInfo   *consistentFmgrInfo;
    FmgrInfo   *triConsistentFmgrInfo;
    Oid         collation;
    Datum       query;
    Datum      *queryValues;
    GinNullCategory *queryCategories;
    Pointer    *extra_data;
    StrategyNumber strategy;
    int32       searchMode;
    OffsetNumber attnum;
    bool        excludeOnly;
    ItemPointerData curItem;
    bool        curItemMatches;
    bool        recheckCurItem;
    bool        isFinished;
} GinScanKeyData;

入力と出力

  • **入力**: ginbeginscan 等で初期化され、extractQueryFn によって分解されたクエリ情報や、consistentFn 呼び出しに必要な関数ポインタおよびデータを受け取ります。
  • **出力**: consistentFn を通じて、現在のタプル（curItem）がクエリ条件に一致するか（curItemMatches）および再チェックが必要か（recheckCurItem）を判定結果として提供します。

前提条件と不変条件

  • nentries は常に 0 より大きい値である必要があります。
  • queryValues、queryCategories、extra_data の各配列は nuserentries 個の要素を持ちます。
  • excludeOnly が真の場合、そのキー単独では一致タプルを列挙できません。

アルゴリズムまたは振る舞い

1. **クエリ評価**: consistentFn（boolConsistentFn または triConsistentFn）を呼び出し、entryRes に格納された各エントリの照合結果に基づいて、インデックス項目がクエリを満たすかを判定します。 2. **スキャン制御**: curItem で現在テスト中の TID を保持します。isFinished が真になると、すべての入力エントリのストリームが終了したことを示し、以降の TID に対して成功することはありません。 3. **最適化**: excludeOnly キーは、他の非 excludeOnly キーが存在する場合にのみフィルタリングに使用されます。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/access/gin/ginscan.c 内の関数（ginbeginscan 等）がこの構造体を割り当て、初期化します。
  • **呼び出し先**: consistentFn 経由で、インデックス定義時に指定された一貫性関数が呼び出されます。

関連データ構造

  • GinScanEntry: scanEntry 配列が保持する各検索条件のデータ構造。
  • GinTernaryValue: entryRes で使用される三値論理値。
  • FmgrInfo: consistentFn を実行するための関数マネージャ情報。

失敗モードと境界条件

  • excludeOnly キーのみが存在する場合、GIN_CAT_EMPTY_QUERY エントリが自動的に追加されない限り、正しい結果が得られない可能性があります。
  • isFinished が正しく更新されない場合、スキャンが終了せず無限ループや誤った結果を返す可能性があります。

保守担当者向けメモ

  • GinScanKeyData は GIN スキャンの核心であり、consistentFn の呼び出し規約と entryRes の更新ロジックを理解することが重要です。
  • excludeOnly の変換ロジックは、インデックススキャンの効率と正確性に直結するため、変更時には注意が必要です。
