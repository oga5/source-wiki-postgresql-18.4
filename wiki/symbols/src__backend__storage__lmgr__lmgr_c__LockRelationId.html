LockRelationId

定義

  • 種別: function
  • 定義箇所: src/backend/storage/lmgr/lmgr.c:184

目的

指定されたリレーション識別子（LockRelId）に対して、指定されたロックモードでロックを取得します。ロック取得後、リレーションキャッシュの整合性を保つためのインバリデーション処理を必要に応じて実行します。

シグネチャまたは形状

void LockRelationId(LockRelId *relid, LOCKMODE lockmode)

入力と出力

  • **入力**:
  • relid: ロック対象のリレーションを特定する LockRelId 構造体へのポインタ。
  • lockmode: 取得するロックのモード（LOCKMODE）。
  • **出力**: なし（成功時はロックを取得し、失敗時はエラーをスローして中断します）。

前提条件と不変条件

  • relid が指す構造体は有効である必要があります。
  • ロック取得後、LOCKACQUIRE_ALREADY_CLEAR 以外の結果が返された場合、AcceptInvalidationMessages() を呼び出してキャッシュの整合性を維持する必要があります。

アルゴリズムまたは振る舞い

1. SET_LOCKTAG_RELATION マクロを使用して、relid から LOCKTAG を構築します。 2. LockAcquireExtended を呼び出し、共有メモリ上で指定されたロックモードの取得を試みます。 3. ロック取得の結果が LOCKACQUIRE_ALREADY_CLEAR でない場合、AcceptInvalidationMessages() を呼び出して、ロック取得中に発生した可能性のあるスキーマ変更等のインバリデーションメッセージを処理します。 4. MarkLockClear(locallock) を呼び出し、当該ロックがクリーンな状態であることを記録します。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/commands/dbcommands.c (複数箇所)
  • **呼び出し先**:
  • LockAcquireExtended
  • AcceptInvalidationMessages
  • MarkLockClear

関連データ構造

  • LockRelId: データベースIDとリレーションIDを保持する構造体。
  • LOCKTAG: ロック対象を一意に識別するためのタグ。
  • LOCALLOCK: プロセスローカルなロック管理構造体。

失敗モードと境界条件

  • LockAcquireExtended が内部でエラーを検出した場合、トランザクションは中断されます。
  • デッドロックが発生した場合、PostgreSQLのデッドロック検出器によりエラーがスローされます。

保守担当者向けメモ

  • 本関数は LockRelationOid と同様のインバリデーションロジックを共有しています。修正を行う際は、src/backend/storage/lmgr/lmgr.c 内の関連するコメントおよび LockRelationOid の実装を参照してください。
