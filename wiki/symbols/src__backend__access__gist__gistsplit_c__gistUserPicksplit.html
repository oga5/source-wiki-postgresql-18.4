gistUserPicksplit

定義

  • 種別: function
  • 定義箇所: src/backend/access/gist/gistsplit.c:414

目的

GiSTインデックスのページ分割において、ユーザー定義の picksplit メソッドを呼び出し、その結果を評価・調整する。特にマルチカラムインデックスにおいて、分割の品質を向上させるための再帰的な最適化を制御する。

シグネチャまたは形状

static bool gistUserPicksplit(Relation r, GistEntryVector *entryvec, int attno, GistSplitVector *v, IndexTuple *itup, int len, GISTSTATE *giststate)

入力と出力

  • **入力**:
  • r: 対象のリレーション。
  • entryvec: 分割対象のインデックスエントリベクトル。
  • attno: 現在処理中の列インデックス。
  • v: 分割状態を保持する構造体。
  • itup: インデックスタプルの配列。
  • len: タプル数。
  • giststate: GiSTの実行状態。
  • **出力**:
  • bool: 分割が不十分であり、後続の列で再分割（再帰）が必要な場合に true を返す。

前提条件と不変条件

  • entryvec 内に NULL キーは存在しない。
  • ユーザー定義の picksplit が片側に全タプルを寄せるような不適切な分割を行った場合、genericPickSplit によるフォールバックが行われる。
  • 戻り値が true の場合、呼び出し元は後続の列を使用して再分割を試みる必要がある。

アルゴリズムまたは振る舞い

1. **初期分割**: ユーザー定義の picksplit 関数を呼び出し、初期分割を実行する。 2. **フォールバック**: 分割結果が空（片側のみ）の場合、genericPickSplit を実行して補正する。 3. **二次分割の処理**: supportSecondarySplit を呼び出し、必要に応じて二次分割を処理する。 4. **最適化（Don't-care タプルの特定）**:

  • 現在の列でペナルティなしで移動可能なタプル（don't-care）を findDontCares で特定する。
  • don't-care タプルが存在する場合、それらを分割から除外し、gistunionsubkey でユニオンキーを再計算する。
  • don't-care タプルが1つのみの場合は placeOne でペナルティの低い側へ割り当てる。
  • 複数ある場合は true を返し、後続列での再分割を促す。

呼び出し元と呼び出し先

  • **呼び出し元**: gistSplitByKey（ソースコード内の文脈より）
  • **呼び出し先**:
  • FunctionCall2Coll (ユーザー定義 picksplit 実行)
  • genericPickSplit
  • supportSecondarySplit
  • findDontCares
  • removeDontCares
  • gistunionsubkey
  • placeOne

関連データ構造

  • GistEntryVector: 分割対象のエントリ群。
  • GistSplitVector: 分割結果を保持する構造体。
  • GIST_SPLITVEC: ユーザー定義メソッドとやり取りするための分割ベクトル。
  • GISTSTATE: GiSTインデックスの操作状態。

失敗モードと境界条件

  • **分割失敗**: ユーザー定義の picksplit が片側に全タプルを配置した場合、DEBUG1 レベルで警告を発行し、genericPickSplit で強制的に分割する。
  • **縮退した分割**: 左側と右側のユニオンキーが等しい場合、分割は縮退していると判断し、後続列での再分割を要求する。
  • **Don't-care タプルによる空分割**: don't-care タプルを除去した結果、どちらかの側が空になる場合、その分割を破棄して後続列での再分割を優先する。

保守担当者向けメモ

  • 本関数はマルチカラム最適化の要であり、attno をインクリメントしながら再帰的に呼び出される。
  • gistunionsubkey によるユニオンキーの再計算は、後続の gistUserPicksplit 呼び出し時に正しい値が渡されることを保証するために不可欠である。
  • NumDontCare == 1 の特殊ケースでは、ペナルティ比較のみで配置を決定するが、この際ユニオンキーは一時的に不整合となる可能性がある。これは最上位の呼び出し元で修正される設計となっている。
