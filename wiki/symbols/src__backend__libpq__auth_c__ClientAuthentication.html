ClientAuthentication

定義

  • 種別: function
  • 定義箇所: src/backend/libpq/auth.c:378

目的

PostgreSQLのクライアント認証プロセスのメインエントリーポイントとして、接続要求（Port）に基づき、設定された認証方式に従ってユーザーの身元確認と認証フローを制御する。

シグネチャまたは形状

void ClientAuthentication(Port *port)

入力と出力

  • **入力**: Port *port（接続情報、HBA設定、認証状態を含む構造体）
  • **出力**: なし（認証成功時は制御を戻し、失敗時は FATAL エラーにより接続を終了する）

前提条件と不変条件

  • hba_getauthmethod(port) が呼び出され、port->hba に認証方式が設定されていること。
  • 認証処理の開始前に CHECK_FOR_INTERRUPTS() が実行されること。
  • port->hba->clientcert が clientCertOff でない場合、ルート証明書ストアがロード済みであり、かつ port->peer_cert_valid が真であること。

アルゴリズムまたは振る舞い

1. **事前検証**: HBA設定に基づき、クライアント証明書の検証が必要な場合は証明書の有効性を確認する。 2. **方式分岐**: port->hba->auth_method に基づき switch 文で認証ハンドラを分岐する。

  • uaReject: 明示的な拒否設定により FATAL エラーを発生させる。
  • uaImplicitReject: マッチするHBAエントリがない場合に FATAL エラーを発生させる。
  • uaGSS: GSSAPI認証を実行する。port->gss が未初期化の場合はメモリを割り当て、暗号化状態に応じて pg_GSS_checkauth 等を呼び出す。

    • 3. **エラー処理**: 認証失敗時は、セキュリティを考慮しつつ、接続情報（ホスト名、ユーザー名、暗号化状態）を含む詳細なログを記録して接続を終了する。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/utils/init/postinit.c:253
  • **呼び出し先**:
  • hba_getauthmethod
  • secure_loaded_verify_locations
  • pg_getnameinfo_all
  • pg_GSS_checkauth（uaGSS の場合）
  • MemoryContextAllocZero
  • ereport（エラー発生時）

関連データ構造

  • Port: 接続状態、HBA設定、認証情報を保持する主要構造体。
  • pg_gssinfo: GSSAPI認証情報を保持する構造体。

失敗モードと境界条件

  • **認証失敗**: uaReject または uaImplicitReject に該当する場合、または認証プロトコルが検証に失敗した場合、FATAL エラーにより接続が切断される。
  • **証明書エラー**: clientCert が要求されているにもかかわらず、ルート証明書ストアが未ロード、またはクライアント証明書が無効な場合は FATAL エラーとなる。
  • **割り込み**: CHECK_FOR_INTERRUPTS() により、処理中に割り込みが発生した場合は中断される。

保守担当者向けメモ

  • 新しい認証方式を追加する場合は、switch 文に新しい case を追加し、対応するハンドラを実装すること。
  • 認証失敗時のエラーメッセージは、攻撃者に情報を与えすぎないよう設計されているため、変更時はセキュリティへの影響を考慮すること。
  • 認証フック ClientAuthentication_hook が存在する場合、認証後に外部プラグインが制御を受け取る可能性がある。
