add_other_rels_to_query

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/plan/initsplan.c:207

目的

クエリ内の継承関係を持つリレーション（inh フラグが有効なもの）を特定し、それらの子リレーションをプランナの管理対象に追加します。

シグネチャまたは形状

void add_other_rels_to_query(PlannerInfo *root)

入力と出力

  • **入力**: PlannerInfo *root（クエリのプランニング情報を保持する構造体）
  • **出力**: なし（副作用として root->simple_rel_array 等が更新されます）

前提条件と不変条件

  • root->simple_rel_array が初期化されている必要があります。
  • RELOPT_BASEREL としてマークされているリレーションのみが処理対象となります。

アルゴリズムまたは振る舞い

1. root->simple_rel_array を走査し、各リレーション（RelOptInfo）と対応するレンジテーブルエントリ（RangeTblEntry）を取得します。 2. rel == NULL のスロットや、既に RELOPT_BASEREL 以外としてマークされているリレーションはスキップします。 3. rte->inh が真（継承関係あり）の場合、expand_inherited_rtentry を呼び出し、子リレーションを生成・追加します。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/optimizer/path/allpaths.c:1034
  • src/backend/optimizer/plan/planmain.c:276
  • **呼び出し先**:
  • expand_inherited_rtentry（継承テーブルの展開用）

関連データ構造

  • PlannerInfo: プランナのグローバル状態を保持。
  • RelOptInfo: リレーションの最適化情報を保持。
  • RangeTblEntry: レンジテーブルエントリ。

失敗モードと境界条件

  • rel が NULL の場合、空のスロットとして安全にスキップされます。
  • 継承関係を持たない（inh が偽の）テーブルについては、何も行われません。

保守担当者向けメモ

  • 本関数はプランニングの初期段階で、継承テーブルの展開を確実に行うために呼び出されます。
  • expand_inherited_rtentry への依存が強いため、継承処理のロジック変更時は本関数の挙動も再確認してください。
