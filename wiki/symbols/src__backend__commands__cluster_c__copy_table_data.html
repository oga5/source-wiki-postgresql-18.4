copy_table_data

定義

  • 種別: function
  • 定義箇所: src/backend/commands/cluster.c:830

目的

CLUSTER または VACUUM FULL 操作において、古いテーブル（OldHeap）から新しいテーブル（NewHeap）へデータを物理的にコピーし、再構築を行う。

シグネチャまたは形状

static void copy_table_data(Relation NewHeap, Relation OldHeap, Relation OldIndex, 
                            bool verbose, bool *pSwapToastByContent, 
                            TransactionId *pFreezeXid, MultiXactId *pCutoffMulti)

入力と出力

  • **入力**:
  • NewHeap, OldHeap: 操作対象のテーブルリレーション。
  • OldIndex: クラスタリングに使用するインデックス（指定がない場合は NULL）。
  • verbose: 処理状況をログ出力するかどうかのフラグ。
  • **出力**:
  • pSwapToastByContent: TOAST テーブルをコンテンツベースでスワップすべきかを示すフラグ。
  • pFreezeXid, pCutoffMulti: 更新後のテーブルの凍結トランザクションIDおよびマルチトランザクションID。

前提条件と不変条件

  • NewHeap と OldHeap のタプル記述子（TupleDesc）の属性数（natts）が一致していること。
  • OldHeap に TOAST テーブルが存在する場合、競合を防ぐために AccessExclusiveLock が取得されていること。

アルゴリズムまたは振る舞い

1. **TOAST ロック**: OldHeap に TOAST テーブルがある場合、オートバキュームによる競合を防ぐためロックを取得。 2. **TOAST スワップ判定**: 両方のテーブルが TOAST テーブルを持つ場合、pSwapToastByContent を true に設定し、NewHeap->rd_toastoid を OldHeap の TOAST OID に一時的に設定する。 3. **バキューム閾値計算**: vacuum_get_cutoffs を使用して、タプルの凍結および削除に必要な閾値を算出。既存の relfrozenxid および relminmxid を下回らないよう調整。 4. **スキャン方式決定**: OldIndex が B-tree インデックスであれば plan_cluster_use_sort を呼び出し、インデックススキャンかシーケンシャルスキャン＋ソートかを決定。 5. **データコピー**: table_relation_copy_for_cluster を呼び出し、AM（アクセスメソッド）固有のロジックでデータを物理的にコピー。 6. **後処理**: rd_toastoid を InvalidOid にリセットし、処理結果をログ出力する。

呼び出し元と呼び出し先

  • **呼び出し元**: rebuild_relation（src/backend/commands/cluster.c 内）。
  • **呼び出し先**: get_namespace_name, LockRelationOid, vacuum_get_cutoffs, plan_cluster_use_sort, table_relation_copy_for_cluster, RelationGetNumberOfBlocks, pg_rusage_show 等。

関連データ構造

  • Relation: テーブルおよびインデックスのメタデータ構造。
  • VacuumParams, VacuumCutoffs: バキューム処理のパラメータと閾値管理用構造体。

失敗モードと境界条件

  • OldIndex が NULL の場合、インデックス順序の最適化は行われず、通常のシーケンシャルスキャンによるコピーとなる。
  • TOAST テーブルの有無により、スワップ戦略（コンテンツベースかリンクベースか）が分岐する。

保守担当者向けメモ

  • 本関数は AM（アクセスメソッド）抽象化レイヤーを介してデータをコピーするため、特定のストレージエンジンに依存しない設計となっている。
  • rd_toastoid の一時的な書き換えは、finish_heap_swap までの整合性を維持するための重要なハックである。変更時は toast_save_datum への影響を考慮すること。
