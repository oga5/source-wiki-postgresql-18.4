_DestReceiver

定義

  • 種別: struct
  • 定義箇所: src/include/tcop/dest.h:115

目的

クエリ実行結果（タプル）の出力先を抽象化し、実行エンジンが宛先の詳細を知ることなくタプルを処理できるようにするためのインターフェースを提供します。

シグネチャまたは形状

struct _DestReceiver
{
    bool (*receiveSlot) (TupleTableSlot *slot, DestReceiver *self);
    void (*rStartup) (DestReceiver *self, int operation, TupleDesc typeinfo);
    void (*rShutdown) (DestReceiver *self);
    void (*rDestroy) (DestReceiver *self);
    CommandDest mydest;
};

入力と出力

  • **入力**: 各関数ポインタは DestReceiver *self を引数として受け取り、receiveSlot は処理対象の TupleTableSlot を、rStartup は操作種別と TupleDesc を受け取ります。
  • **出力**: receiveSlot は処理の継続可否を示す bool 値を返します。その他の関数は void です。

前提条件と不変条件

  • receiveSlot が false を返した場合、実行エンジンは処理を直ちに終了する必要があります。
  • rDestroy は、動的に割り当てられたレシーバーに対してのみメモリ解放を行います。静的に割り当てられたレシーバー（None_Receiver 等）の場合、この関数は何もしません。

アルゴリズムまたは振る舞い

1. **初期化**: rStartup により、クエリ実行単位の準備が行われます。 2. **タプル処理**: タプルが生成されるたびに receiveSlot が呼び出されます。 3. **終了**: 実行完了後に rShutdown が呼び出され、必要に応じて rDestroy でオブジェクト自体が破棄されます。

呼び出し元と呼び出し先

  • **呼び出し元**: 実行エンジン（tcop モジュール内のクエリ実行フロー）。
  • **呼び出し先**: 具体的な実装（src/backend/tcop/dest.c や各レシーバー実装ファイル）に依存します。
  • **注記**: 本ソース excerpt の範囲外での具体的な呼び出し元・呼び出し先の詳細は提供されていません。

関連データ構造

  • TupleTableSlot: receiveSlot で渡されるタプル保持構造体。
  • CommandDest: レシーバーの種別を識別する列挙型。
  • DestReceiver: _DestReceiver の typedef 名。

失敗モードと境界条件

  • receiveSlot の戻り値が false の場合、実行エンジンは早期終了を強制されます。
  • メモリ管理の観点から、静的割り当てと動的割り当てのレシーバーで rDestroy の挙動が異なるため、適切なライフサイクル管理が求められます。

保守担当者向けメモ

  • 本構造体は src/include/tcop/dest.h で定義され、src/backend/tcop/dest.c で実装が管理されています。
  • 構造体の末尾にはプライベートフィールドが追加される可能性があるため、拡張時は注意してください。
