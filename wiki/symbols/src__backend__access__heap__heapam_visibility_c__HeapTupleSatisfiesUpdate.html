HeapTupleSatisfiesUpdate

定義

  • 種別: function
  • 定義箇所: src/backend/access/heap/heapam_visibility.c:457

目的

更新操作（UPDATE/DELETE）の対象となるタプルが、現在のトランザクションおよびコマンドから見て「更新可能」な状態にあるかを判定する。

シグネチャまたは形状

TM_Result HeapTupleSatisfiesUpdate(HeapTuple htup, CommandId curcid, Buffer buffer)

入力と出力

  • **入力**:
  • htup: 判定対象のヒープタプル。
  • curcid: 現在のコマンドID。
  • buffer: タプルを含むバッファ（ヒントビット更新時に使用）。
  • **出力**:
  • TM_Result 型の判定結果（TM_Ok, TM_Invisible, TM_Updated, TM_Deleted, TM_BeingModified, TM_SelfModified 等）。

前提条件と不変条件

  • 対象バッファに対して少なくとも共有ロック（Shared Lock）を保持していること。
  • htup->t_self が有効なアイテムポインタであること。
  • htup->t_tableOid が InvalidOid でないこと。

アルゴリズムまたは振る舞い

1. **Xminの確認**: タプルの挿入トランザクションがコミット済みかを確認する。未コミットの場合は TM_Invisible を返す。 2. **古いバイナリアップグレード対応**: HEAP_MOVED_OFF または HEAP_MOVED_IN フラグを確認し、トランザクション状態に応じて可視性を判定する。 3. **自己更新の判定**: 自身のトランザクションが挿入したタプルの場合、コマンドID（curcid）を比較し、スキャン開始後に挿入されたものは TM_Invisible とする。 4. **Xmaxの確認**:

  • HEAP_XMAX_INVALID の場合は TM_Ok。
  • HEAP_XMAX_COMMITTED の場合、t_ctid を比較して他者による更新（TM_Updated）か削除（TM_Deleted）かを判定する。
  • ロックのみ（HEAP_XMAX_IS_LOCKED_ONLY）の場合は、ロック状態に応じて TM_Ok または TM_BeingModified を返す。

    • 5. **ヒントビットの更新**: 判定過程でトランザクションの確定状態が判明した場合、SetHintBits を呼び出してタプルヘッダーを更新する。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/access/heap/heapam.c 内の heap_update や heap_delete 等の更新系関数、および contrib/pgrowlocks/pgrowlocks.c。
  • **呼び出し先**: SetHintBits, TransactionIdIsCurrentTransactionId, TransactionIdIsInProgress, TransactionIdDidCommit, MultiXactIdIsRunning 等のトランザクション管理関数。

関連データ構造

  • HeapTuple: 判定対象のタプル構造体。
  • TM_Result: 判定結果を示す列挙型。
  • Buffer: バッファ管理構造体。

失敗モードと境界条件

  • **TM_Invisible**: タプルが現在のスナップショットから見えない、または未コミットの挿入によるもの。
  • **TM_BeingModified**: 他のトランザクションによってロックまたは更新中。
  • **TM_Updated / TM_Deleted**: 他のトランザクションによって既に更新または削除済み。
  • **レースコンディション**: コミット直後のトランザクション状態判定において、TransactionIdIsInProgress を TransactionIdDidCommit より先にチェックする順序を厳守することで回避している。

保守担当者向けメモ

  • 本関数は heap_update 等の低レベルな更新処理の根幹である。
  • t_infomask のフラグ（HEAP_XMAX_IS_MULTI 等）の解釈が複雑であるため、変更時は MultiXact 関連のロジックに注意すること。
  • ヒントビット更新によりバッファがダーティになるため、呼び出し側で適切なロック管理が必要である。
