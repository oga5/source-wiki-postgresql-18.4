table_slot_callbacks

定義

  • src/backend/access/table/tableam.c:58

目的

リレーションの種類に基づき、タプルを格納するための適切な TupleTableSlotOps（スロット操作コールバック）を決定して返します。

シグネチャまたは形状

const TupleTableSlotOps *table_slot_callbacks(Relation relation)

入力と出力

  • **入力**: Relation relation（対象となるリレーション）
  • **出力**: const TupleTableSlotOps *（スロット操作を定義する関数テーブルへのポインタ）

前提条件と不変条件

  • relation は有効なポインタである必要があります。
  • rd_tableam が設定されている場合、その実装が slot_callbacks を提供していることが前提となります。

アルゴリズムまたは振る舞い

1. relation->rd_tableam が存在する場合、そのアクセスメソッド固有の slot_callbacks を呼び出し、結果を返します。 2. rd_tableam が存在しない場合、relkind を評価します：

  • RELKIND_FOREIGN_TABLE の場合: 互換性維持のため TTSOpsHeapTuple を返します。
  • RELKIND_VIEW または RELKIND_PARTITIONED_TABLE の場合: TTSOpsVirtual を返します。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/commands/tablecmds.c
  • src/backend/executor/execUtils.c
  • src/backend/executor/nodeBitmapHeapscan.c
  • src/backend/executor/nodeIndexonlyscan.c
  • src/backend/executor/nodeIndexscan.c
  • **呼び出し先**:
  • relation->rd_tableam->slot_callbacks（動的ディスパッチ）

関連データ構造

  • Relation: テーブルのメタデータ構造体。
  • TupleTableSlotOps: スロット操作を抽象化する関数ポインタ群。
  • TTSOpsHeapTuple, TTSOpsVirtual: 具体的なスロット操作実装。

失敗モードと境界条件

  • rd_tableam が NULL であり、かつ relkind が上記条件（FOREIGN, VIEW, PARTITIONED）のいずれにも該当しない場合、コードパスは定義されていません（アサーション失敗の可能性があります）。

保守担当者向けメモ

  • 本関数は、特定のテーブルアクセスメソッドに依存しない汎用的なスロット選択ロジックを中央集権化しています。
  • FDW（外部テーブル）に対する TTSOpsHeapTuple の使用は歴史的な経緯によるものです。将来的な変更の際は、FDW側の互換性に注意してください。
