FreeSpaceMapPrepareTruncateRel

定義

  • **種別**: function
  • **定義箇所**: src/backend/storage/freespace/freespace.c:284

目的

リレーションの切り詰め（Truncate）に先立ち、対応する Free Space Map (FSM) フォークのサイズを調整する準備を行う。不要となったヒープブロックに対応する FSM 領域をクリアし、切り詰め後の FSM ブロック数を算出する。

シグネチャまたは形状

BlockNumber FreeSpaceMapPrepareTruncateRel(Relation rel, BlockNumber nblocks)

入力と出力

  • **入力**:
  • rel: 対象のリレーション情報。
  • nblocks: 切り詰め後のリレーションの目標ブロック数。
  • **出力**:
  • 切り詰め後の FSM フォークのブロック数（BlockNumber）。切り詰めが不要な場合は InvalidBlockNumber を返す。

前提条件と不変条件

  • 対象リレーションに FSM フォークが存在しない場合、何もせず InvalidBlockNumber を返す。
  • FSM の更新はクリティカルセクション内で行われる。
  • FSM は「ヒント」として機能するため、この操作は物理的な整合性を維持しつつ、不要な領域を効率的に削除することを目的とする。

アルゴリズムまたは振る舞い

1. smgrexists を使用し、FSM フォークの存在を確認する。 2. fsm_get_location を呼び出し、切り詰め対象となる最初のヒープブロックに対応する FSM アドレスとスロット位置を特定する。 3. first_removed_slot > 0 の場合:

  • 該当する FSM ページを fsm_readbuf で読み込み、排他ロックを取得する。
  • fsm_truncate_avail を呼び出し、指定スロット以降をクリアする。
  • MarkBufferDirty でバッファをダーティにし、必要に応じて log_newpage_buffer で WAL ログを出力する。
  • バッファを解放し、新しい FSM ブロック数を計算して返す。

    • 4. first_removed_slot == 0 の場合:

  • 物理的な FSM ブロック数を計算し、既存の FSM サイズと比較して切り詰めが必要か判定する。

関連データ構造

  • Relation: リレーション情報。
  • FSMAddress: FSM 内の論理アドレス。
  • Buffer: バッファマネージャの管理単位。

失敗モードと境界条件

  • FSM フォークが存在しない場合、InvalidBlockNumber を返す。
  • FSM が既に目標サイズ以下である場合、InvalidBlockNumber を返す。
  • fsm_readbuf が無効なバッファを返した場合、処理を中断し InvalidBlockNumber を返す。

保守担当者向けメモ

  • 本関数は WAL リプレイ時にも呼び出される可能性がある。
  • log_newpage_buffer の呼び出しはオプションであり、フルページイメージの保存に関する README の指針に従っている。
  • fsm_truncate_avail による変更は非クリティカルと見なされるが、効率化のために明示的にダーティマークが付与される。
