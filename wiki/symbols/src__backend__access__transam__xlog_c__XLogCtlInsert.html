XLogCtlInsert

定義

  • 種別: struct
  • 定義箇所: src/backend/access/transam/xlog.c:397

目的

WAL（Write-Ahead Log）挿入プロセスにおける共有状態を管理し、複数のバックエンドによる WAL 挿入の同期と、挿入に関連するグローバルな制御パラメータを保持する。

シグネチャまたは形状

typedef struct XLogCtlInsert
{
    slock_t     insertpos_lck;
    uint64      CurrBytePos;
    uint64      PrevBytePos;
    char        pad[PG_CACHE_LINE_SIZE];
    XLogRecPtr  RedoRecPtr;
    bool        fullPageWrites;
    int         runningBackups;
    XLogRecPtr  lastBackupStart;
    WALInsertLockPadded *WALInsertLocks;
} XLogCtlInsert;

入力と出力

本構造体は関数ではなくデータ構造であるため、直接的な入力・出力はない。各フィールドは共有メモリ上で管理され、WAL 挿入時に各バックエンドによって読み書きされる。

前提条件と不変条件

  • CurrBytePos および PrevBytePos の更新には insertpos_lck（スピンロック）が必要。
  • fullPageWrites や RedoRecPtr などのフィールドを読み取るには、いずれかの挿入ロック（WALInsertLock）の保持が必要。
  • これらのフィールドを更新（変更）するには、すべての挿入ロックを保持する必要がある。
  • CurrBytePos と PrevBytePos は、キャッシュラインの競合を避けるため、他のフィールドとは別のキャッシュラインに配置される。

アルゴリズムまたは振る舞い

  • **WAL 予約**: CurrBytePos を用いて、次に挿入される WAL レコードの領域を予約する。
  • **リンク管理**: PrevBytePos は直前に予約されたレコードの開始位置を保持し、新しいレコードの prev-link として使用される。
  • **設定の伝播**: fullPageWrites はプロセスローカルな値ではなく、本構造体の共有値が使用される。これにより、SIGHUP 等による設定変更が全バックエンドに即座に反映される。

呼び出し元と呼び出し先

  • **呼び出し元**: XLogCtl 構造体の一部として、src/backend/access/transam/xlog.c 内の WAL 管理関数群（XLogInsertRecord 等）から参照される。
  • **呼び出し先**: 本構造体自体はデータ保持用であり、直接関数を呼び出すことはない。

関連データ構造

  • XLogCtlData: 本構造体を含む、WAL 管理の全状態を保持する親構造体。
  • WALInsertLockPadded: WAL 挿入の競合を制御するためのロック構造体。

失敗モードと境界条件

  • **競合**: 複数のバックエンドが同時に WAL を挿入しようとする際、insertpos_lck や WALInsertLocks による同期が不十分な場合、データ破損や LSN の不整合が発生する可能性がある。
  • **キャッシュライン**: pad フィールドによるパディングが不十分な環境では、CurrBytePos への頻繁なアクセスが他のフィールドの読み取り性能を低下させる可能性がある。

保守担当者向けメモ

  • CurrBytePos と PrevBytePos は XLogRecPtr ではなく「使用可能なバイト位置」として格納されている点に注意すること（変換には XLogBytePosToRecPtr() を使用）。
  • fullPageWrites の更新は、全挿入ロックを必要とするため非常に高コストである。頻繁な変更は避ける設計となっている。
