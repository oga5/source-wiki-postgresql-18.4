HSpool

定義

  • 種別: struct
  • 定義箇所: src/backend/access/hash/hashsort.c:39

目的

ハッシュインデックス構築時、インデックスエントリをバケット番号およびハッシュ値に基づいてソートし、インデックスページへの挿入効率（ディスクアクセス局所性）を最適化するためのコンテキスト構造体です。

シグネチャまたは形状

struct HSpool
{
    Tuplesortstate *sortstate;
    Relation    index;
    uint32      high_mask;
    uint32      low_mask;
    uint32      max_buckets;
};

入力と出力

本構造体は _h_spoolinit によって初期化され、以下の操作で利用されます。

  • **入力**: _h_spool を介してタプルデータを受け取り、sortstate に蓄積します。
  • **出力**: _h_indexbuild を通じてソート済みのタプルをインデックスへ書き出します。

前提条件と不変条件

  • high_mask、low_mask、max_buckets は、インデックスのメタデータ管理と同期している必要があります。
  • sortstate は tuplesort.c の機能を利用して管理されます。

アルゴリズムまたは振る舞い

1. **蓄積**: _h_spool によりタプルをソート領域へ投入します。 2. **ソート**: _h_indexbuild 内で tuplesort_performsort を呼び出し、バケット番号順に並び替えます。 3. **挿入**: ソートされたタプルを順次取得し、_hash_doinsert を用いてインデックスへ書き込みます。

呼び出し元と呼び出し先

  • **呼び出し元**: hashbuild（src/backend/access/hash/hash.c）がインデックス構築の初期化時に利用します。
  • **呼び出し先**: tuplesort.c（ソート処理）、_hash_doinsert（インデックス挿入）。

関連データ構造

  • Tuplesortstate: ソート処理の内部状態を保持します。
  • Relation: インデックスのリレーション情報を保持します。

失敗モードと境界条件

  • maintenance_work_mem の制限により、大規模なインデックス作成時にはメモリ消費量が増大します。
  • バケットマスクの計算がインデックスのメタデータと不整合を起こすと、挿入先バケットが正しく特定できなくなる可能性があります。

保守担当者向けメモ

  • 本構造体によるソートはパフォーマンス最適化を目的としており、ソート順が崩れてもインデックスの論理的な整合性は維持されます。
