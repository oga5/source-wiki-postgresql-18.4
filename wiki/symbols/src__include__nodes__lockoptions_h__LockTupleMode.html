LockTupleMode

定義

  • 種別: enum
  • 定義箇所: src/include/nodes/lockoptions.h:49

目的

PostgreSQLの内部処理において、タプル（行）に対して適用するロックの強度を表現するために使用されます。

シグネチャまたは形状

typedef enum LockTupleMode
{
    LockTupleKeyShare,
    LockTupleShare,
    LockTupleNoKeyExclusive,
    LockTupleExclusive,
} LockTupleMode;

入力と出力

本シンボルは列挙型であり、関数への引数として渡されるか、構造体のメンバとして保持されます。具体的な入出力値は各列挙子（LockTupleKeyShare 等）です。

前提条件と不変条件

  • 列挙値の順序は、ロックの強度（優先順位）を反映しています。
  • LockClauseStrength（SQLレベルの指定）と密接に関連しており、低レイヤーのタプルロック処理を規定します。

アルゴリズムまたは振る舞い

  • LockTupleKeyShare: SELECT FOR KEY SHARE に対応。
  • LockTupleShare: SELECT FOR SHARE に対応。
  • LockTupleNoKeyExclusive: SELECT FOR NO KEY UPDATE およびキー列を更新しない UPDATE に対応。
  • LockTupleExclusive: SELECT FOR UPDATE、キー列を更新する UPDATE、および DELETE に対応。

呼び出し元と呼び出し先

  • **呼び出し元（参照箇所）**: src/backend/access/heap/heapam.c 内の多数の関数（heap_acquire_tuplock、get_mxact_status_for_lock 等）で、タプルロックの強度指定として参照されています。
  • **呼び出し先**: 本シンボル自体は列挙型定義であり、関数を呼び出すことはありません。

関連データ構造

  • LockClauseStrength: SQLレベルのロック強度定義。
  • LockWaitPolicy: ロック競合時の待機ポリシー。
  • RowMarkClause (src/include/nodes/parsenodes.h): 本列挙型を利用してロック要件を保持します。
  • PlanRowMark (src/include/nodes/plannodes.h): 実行計画におけるロック要件保持に使用されます。

失敗モードと境界条件

  • 列挙値の範囲外の値が渡された場合の挙動は、利用側の実装に依存します。
  • 優先順位判定ロジック（applyLockingClause 等）において、意図しない順序で比較が行われるとロック競合の解決が正しく行われない可能性があります。

保守担当者向けメモ

  • 新しいロックモードを追加する際は、既存の列挙順序が applyLockingClause 等での優先順位判定ロジックと整合しているか確認してください。
  • 本列挙型は低レイヤーのタプルロック処理を直接制御するため、変更の影響範囲は広範囲に及びます。
