PageHeaderData

定義

  • **種別**: struct
  • **定義箇所**: src/include/storage/bufpage.h:159

目的

PostgreSQLの「スロット付きページ（Slotted Page）」構造の先頭に配置されるメタデータ構造体です。ページ内の空き領域管理、整合性検証、およびタプルへのアクセスに必要な情報を保持します。

シグネチャまたは形状

typedef struct PageHeaderData
{
    PageXLogRecPtr pd_lsn;
    uint16      pd_checksum;
    uint16      pd_flags;
    LocationIndex pd_lower;
    LocationIndex pd_upper;
    LocationIndex pd_special;
    uint16      pd_pagesize_version;
    TransactionId pd_prune_xid;
    ItemIdData  pd_linp[FLEXIBLE_ARRAY_MEMBER];
} PageHeaderData;

入力と出力

  • **入力**: ページ初期化時（PageInit）に各フィールドが設定されます。データ追加時（PageAddItemExtended）には pd_lower と pd_upper が更新されます。
  • **出力**: ページ内のタプル位置を特定するためのラインポインタ配列（pd_linp）や、空き領域の範囲（pd_lower, pd_upper）を提供します。

前提条件と不変条件

  • **配置**: ページ先頭に固定配置されます。
  • **成長方向**: ラインポインタはページ先頭から後方へ、タプルデータはページ末尾から前方へ配置されます。
  • **不変条件**: pd_lower はラインポインタ配列の末尾を指し、pd_upper はタプル格納領域の開始位置を指します。常に pd_lower <= pd_upper が維持されます。

アルゴリズムまたは振る舞い

  • **空き領域管理**: pd_lower は次に割り当てるラインポインタの位置を、pd_upper は次に割り当てるタプルの開始位置を示します。
  • **検証**: pd_checksum はディスク書き込み直前に計算され、読み込み時に PageIsVerified 関数によって整合性が確認されます。
  • **ヒントビット**: pd_flags に含まれる一部のビット（PD_HAS_FREE_LINES 等）はWALに記録されないヒント情報として機能します。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/storage/page/bufpage.c 内のページ操作関数群（PageAddItemExtended 等）、src/backend/access/nbtree/nbtsort.c、および src/bin/pg_upgrade/file.c 等から参照されます。
  • **呼び出し先**: 構造体自体はデータ保持用であり、直接関数を呼び出すことはありません。

関連データ構造

  • ItemIdData: pd_linp 配列の各要素として使用されるラインポインタ。
  • PageXLogRecPtr: ページの最終更新を示すLSN。
  • LocationIndex: ページ内のオフセット値を表す型。

失敗モードと境界条件

  • **破損**: pd_checksum の不一致はページ破損を示します。
  • **領域不足**: pd_lower と pd_upper が交差する場合、ページ内に新しいアイテムを格納する空き領域が存在しません。

保守担当者向けメモ

  • pd_linp は FLEXIBLE_ARRAY_MEMBER を使用しているため、構造体のサイズは可変です。SizeOfPageHeaderData マクロを使用してヘッダーサイズを計算してください。
  • pd_lsn はページ組織化の有無に関わらず、ブロックの変更履歴を追跡するために使用されます。
