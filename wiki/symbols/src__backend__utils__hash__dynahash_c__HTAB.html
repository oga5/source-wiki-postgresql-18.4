HTAB

定義

  • 種別: struct
  • 定義箇所: src/backend/utils/hash/dynahash.c:219

目的

PostgreSQLの動的ハッシュテーブルを管理するための制御構造体。ローカルメモリおよび共有メモリ上のハッシュテーブルの状態を保持し、検索、挿入、削除操作の基盤となります。

シグネチャまたは形状

struct HTAB {
    HASHHDR    *hctl;           /* 共有制御情報へのポインタ */
    HASHSEGMENT *dir;           /* セグメント開始位置のディレクトリ */
    HashValueFunc hash;         /* ハッシュ関数 */
    HashCompareFunc match;      /* キー比較関数 */
    HashCopyFunc keycopy;       /* キーコピー関数 */
    HashAllocFunc alloc;        /* メモリ割り当て関数 */
    MemoryContext hcxt;         /* デフォルトアロケータ使用時のメモリコンテキスト */
    char       *tabname;        /* エラーメッセージ用テーブル名 */
    bool        isshared;       /* 共有メモリ上のテーブルか */
    bool        isfixed;        /* 拡張不可フラグ */
    bool        frozen;         /* 挿入禁止フラグ */
    Size        keysize;        /* キーのバイト長 */
    long        ssize;          /* セグメントサイズ（2のべき乗） */
    int         sshift;         /* セグメントシフト量 */
};

入力と出力

本構造体は関数ではなくデータ構造であるため、直接的な入力・出力はありません。各フィールドは hash_create() 等の関数によって初期化され、hash_search() 等の操作関数によって参照・更新されます。

前提条件と不変条件

  • ssize は必ず2のべき乗である必要があります。
  • isshared が true の場合、テーブルは共有メモリ上に配置されます。
  • frozen が true の場合、テーブルへの新規挿入は許可されません。

アルゴリズムまたは振る舞い

  • **競合低減**: keysize や ssize などの固定値を構造体内に保持することで、共有メモリ上のヘッダーへのアクセス頻度を抑え、競合を最小化します。
  • **動的拡張**: isfixed が false の場合、負荷に応じてバケットの分割とディレクトリの拡張を行います。
  • **メモリ管理**: alloc 関数ポインタと hcxt を介して、動的なメモリ割り当てを抽象化します。

呼び出し元と呼び出し先

  • **呼び出し元**: 以下のモジュール等で利用されています（直接的な参照の証拠）。
  • contrib/dblink/dblink.c
  • contrib/pg_stat_statements/pg_stat_statements.c
  • contrib/pg_trgm/trgm_regexp.c
  • contrib/postgres_fdw/connection.c
  • contrib/postgres_fdw/shippable.c
  • contrib/tablefunc/tablefunc.c
  • **呼び出し先**: 構造体内の関数ポインタ（hash, match, keycopy, alloc）を通じて、実行時に動的に決定されます。

関連データ構造

  • HASHHDR: 共有メモリ上に配置されるハッシュテーブルのヘッダー情報。
  • HASHSEGMENT: ハッシュテーブルのディレクトリを構成するセグメント。
  • HASHBUCKET: ハッシュテーブル内の個々のバケット。

失敗モードと境界条件

  • isfixed が true の場合、テーブルの拡張は行われません。
  • 共有メモリ上のテーブルは、hash_create 時に指定した初期サイズを超えて動的にディレクトリを拡張することができません。
  • シーケンシャルスキャン中に要素を削除することは可能ですが、スキャン中の他の要素の追加・削除は未定義の挙動を引き起こす可能性があります。

保守担当者向けメモ

  • 本構造体は各バックエンドが保持するローカルな制御構造体です。共有メモリ上の永続的な状態については HASHHDR を参照してください。
  • 共有メモリ上のテーブルでは frozen 状態の管理が重要です。
  • 構造体内のフィールド追加・変更時は、hash_create での初期化ロジックと整合性を取る必要があります。
