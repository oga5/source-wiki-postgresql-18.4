domain_check_internal

定義

  • 種別: function
  • 定義箇所: src/backend/utils/adt/domains.c:370

目的

ドメイン型に対する値の制約検証を、キャッシュを利用して効率的に実行する。

シグネチャまたは形状

static bool domain_check_internal(Datum value, bool isnull, Oid domainType,
                                  void **extra, MemoryContext mcxt,
                                  Node *escontext)

入力と出力

  • **入力**:
  • value: 検証対象の Datum 値。
  • isnull: 値が NULL かどうかを示すフラグ。
  • domainType: 対象となるドメイン型の Oid。
  • extra: キャッシュされた DomainIOData 構造体へのポインタ（void **）。
  • mcxt: メモリコンテキスト。
  • escontext: エラー報告用コンテキスト（Node *）。
  • **出力**:
  • bool: 制約違反がない場合は true、違反がある場合は false を返す。

前提条件と不変条件

  • extra が指す領域には、以前の呼び出しで生成された DomainIOData が格納されているか、NULL である必要がある。
  • domainType が変更された場合、キャッシュは自動的に再構築される。
  • escontext を介したエラー報告がサポートされている。

アルゴリズムまたは振る舞い

1. **キャッシュの確認**: extra に格納された DomainIOData を確認する。NULL であるか、domain_type が現在の domainType と一致しない場合は、domain_state_setup を呼び出してキャッシュを初期化・更新する。 2. **制約検証**: domain_check_input を呼び出し、値がドメインの制約（NOT NULL や CHECK 制約）を満たしているか検証する。 3. **結果判定**: SOFT_ERROR_OCCURRED(escontext) を使用して、検証中にエラーが発生しなかったかを判定し、その結果を返す。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/utils/adt/domains.c 内の domain_check および domain_check_safe から呼び出される。
  • **呼び出し先**:
  • domain_state_setup: キャッシュの初期化に使用。
  • domain_check_input: 実際の制約評価ロジックを実行。

関連データ構造

  • DomainIOData: ドメインの制約情報や実行コンテキストを保持するキャッシュ構造体。

失敗モードと境界条件

  • domain_state_setup が失敗した場合、例外が発生する可能性がある。
  • escontext が提供されている場合、制約違反は例外を投げずに escontext への記録として処理される。

保守担当者向けメモ

  • 本関数は、同一のドメイン型に対する連続した呼び出しにおいて、カタログ検索を回避するために extra 引数によるキャッシュを利用している。
  • mcxt が NULL の場合は CurrentMemoryContext が使用される。メモリ管理の整合性を保つため、呼び出し元でのコンテキスト管理に注意すること。
