append_pathkeys

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/path/pathkeys.c:106

目的

既存のパスキーリスト（target）に対し、別のパスキーリスト（source）から冗長でない要素を順次追加し、正規化された新しいパスキーリストを構築します。

シグネチャまたは形状

List *append_pathkeys(List *target, List *source)

入力と出力

  • **入力**:
  • target: 追加先の PathKey リスト（NIL であってはならない）。
  • source: 追加元の PathKey リスト。
  • **出力**:
  • target に source の要素を適切に結合した新しい List。

前提条件と不変条件

  • target は NIL であってはなりません（Assert(target != NIL)）。
  • source 内の各 PathKey は、pathkey_is_redundant によって target に対して冗長性が判定されます。

アルゴリズムまたは振る舞い

1. target リストが空でないことを検証します。 2. source リストの各要素（PathKey）を順に走査します。 3. 各 PathKey について pathkey_is_redundant を呼び出し、既存の target リストに対して冗長でない場合のみ、lappend を使用して target リストの末尾に追加します。 4. 最終的に構築されたリストを返します。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/optimizer/plan/planner.c: GROUP BY やウィンドウ関数のパスキー構築時に呼び出されます。
  • **呼び出し先**:
  • pathkey_is_redundant (判定)
  • lappend (リスト操作)

関連データ構造

  • PathKey: 順序を表現するデータ構造。
  • List: PostgreSQL の汎用リスト構造。

失敗モードと境界条件

  • target が NIL の場合、アサーション違反により異常終了します。
  • source が NIL の場合、target を変更せずにそのまま返します。

保守担当者向けメモ

  • 本関数は、既存の順序リストを拡張する際に「冗長なソートキーを排除する」というプランナの最適化方針を強制します。
  • pathkey_is_redundant のロジック変更が、本関数の結果に直接影響を与える点に注意してください。
