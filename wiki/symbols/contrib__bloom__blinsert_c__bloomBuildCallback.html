bloomBuildCallback

定義

  • 種別: function
  • 定義箇所: contrib/bloom/blinsert.c:74

目的

Bloom 索引構築時、ヒープスキャンから渡された各タプルを処理し、Bloom フィルタページへ追加するコールバック関数です。

シグネチャまたは形状

static void bloomBuildCallback(Relation index, ItemPointer tid, Datum *values, bool *isnull, bool tupleIsAlive, void *state);

入力と出力

  • **入力**:
  • index: 構築対象のインデックスリレーション。
  • tid: タプルの物理位置。
  • values, isnull: タプルの属性値と NULL 判定配列。
  • tupleIsAlive: タプルの生存状態。
  • state: BloomBuildState 構造体へのポインタ。
  • **出力**: なし（副作用として state 内のキャッシュページおよびインデックス状態を更新）。

前提条件と不変条件

  • state は BloomBuildState 型としてキャスト可能であること。
  • buildstate->tmpCtx が初期化されていること。
  • ページが満杯の場合、flushCachedPage を経て新しいページが初期化されること。

アルゴリズムまたは振る舞い

1. buildstate->tmpCtx にメモリコンテキストを切り替え、BloomFormTuple でタプルを生成します。 2. BloomPageAddItem を呼び出し、現在のキャッシュページへの追加を試みます。 3. 追加に成功した場合、buildstate->count をインクリメントします。 4. 追加に失敗した場合（ページ満杯）:

  • flushCachedPage で現在のページをディスクへ書き出します。
  • CHECK_FOR_INTERRUPTS() を実行し、割り込みを確認します。
  • initCachedPage で新しいページを初期化し、再度 BloomPageAddItem を実行します。

    • 5. 最後に buildstate->indtuples を更新し、メモリコンテキストを復元・リセットします。

呼び出し元と呼び出し先

  • **呼び出し元**: blbuild（contrib/bloom/blinsert.c 内）から table_index_build_scan を介して呼び出されます。
  • **呼び出し先**:
  • BloomFormTuple
  • BloomPageAddItem
  • flushCachedPage
  • initCachedPage
  • MemoryContextSwitchTo / MemoryContextReset

関連データ構造

  • BloomBuildState: 構築中のインデックス状態、メモリコンテキスト、キャッシュページを保持。
  • BloomTuple: インデックスに格納される形式のタプル。

失敗モードと境界条件

  • 空のページにタプルを追加できない場合、elog(ERROR) を発生させます。
  • メモリ不足時は MemoryContext の管理下でエラーが発生する可能性があります。

保守担当者向けメモ

  • 本関数は table_index_build_scan からのコールバックとして動作するため、シグネチャの変更には注意が必要です。
  • GenericXLog を使用したページ書き出し処理が含まれるため、WAL 整合性に影響を与えるロジックです。
