WAIT_ORDER

定義

  • **種別**: struct
  • **定義箇所**: src/backend/storage/lmgr/deadlock.c:57

目的

デッドロック解消のために提案される、ロック待ち行列の新しい順序を保持するためのデータ構造です。

シグネチャまたは形状

typedef struct
{
	LOCK	   *lock;			/* 対象となるロック */
	PGPROC	  **procs;			/* 新しい順序での PGPROC ポインタ配列 */
	int			nProcs;			/* 配列内の要素数 */
} WAIT_ORDER;

入力と出力

本構造体は関数への引数や戻り値として直接渡されるのではなく、waitOrders 配列の要素としてメモリ上に確保され、デッドロック検出アルゴリズム内で状態を保持するために使用されます。

前提条件と不変条件

  • lock は有効な LOCK 構造体を指している必要があります。
  • procs は nProcs 個の PGPROC ポインタを保持する十分な領域を確保している必要があります。

アルゴリズムまたは振る舞い

デッドロック検出プロセスにおいて、循環参照を解消するためにロック待ち行列の順序を再構成する際、その「提案された順序」を記録します。DeadLockCheck が解決可能と判断した場合、この構造体に基づいて実際のロック待ち行列が書き換えられます。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/storage/lmgr/deadlock.c 内のデッドロック検出ロジック（DeadLockCheck 等）により、waitOrders 配列の要素として初期化・参照されます。
  • **呼び出し先**: 本構造体自体はデータ保持用であり、関数を呼び出すことはありません。

関連データ構造

  • LOCK: ロック待ち行列を管理する構造体。
  • PGPROC: ロックを待機しているプロセスを表す構造体。
  • waitOrders: WAIT_ORDER 構造体の配列（deadlock.c 内で静的に確保）。

失敗モードと境界条件

  • MaxBackends に基づいてメモリが確保されるため、メモリ不足時には palloc が失敗する可能性があります。
  • nProcs が実際の procs 配列のサイズと一致しない場合、メモリ破壊や不正なアクセスを引き起こす可能性があります。

保守担当者向けメモ

  • 本構造体はデッドロック解消の試行中にのみ使用される一時的なデータです。
  • waitOrders 配列は InitDeadLockChecking で初期化されるため、この構造体を利用する際は初期化が完了していることを前提としてください。
