目的

ネイティブな int128 型と、PostgreSQL 独自の INT128 構造体、およびその内部ビット表現（上位/下位 64 ビット）を同一メモリ領域で保持し、相互変換や比較検証を行うための型です。

シグネチャまたは形状

typedef union {
    int128      i128;
    INT128      I128;
    union {
#ifdef WORDS_BIGENDIAN
        int64   hi;
        uint64  lo;
#else
        uint64  lo;
        int64   hi;
#endif
    }           hl;
} test128;

入力と出力

  • 入力: 構造体内の各メンバ（i128, I128, hl）への代入。
  • 出力: 構造体内の各メンバからの値の読み出し。

前提条件と不変条件

  • ネイティブ型 int128 と INT128 構造体、および hl 構造体が同一のメモリレイアウトを持つという前提に基づいています。
  • エンディアン（WORDS_BIGENDIAN）に応じて、hi（上位）と lo（下位）の配置が適切に切り替わります。

アルゴリズムまたは振る舞い

  • 複数の異なる型表現を共用体としてラップすることで、同一のビットパターンを異なる型として解釈・比較することを可能にします。

呼び出し元と呼び出し先

  • 呼び出し元: src/tools/testint128.c 内の main 関数にて、テスト用の変数 t1, t2 としてインスタンス化されています。
  • 呼び出し先: 情報なし。

関連データ構造

  • int128: コンパイラがネイティブサポートする 128 ビット整数型。
  • INT128: PostgreSQL 独自の実装による 128 ビット整数構造体。

失敗モードと境界条件

  • プラットフォームのエンディアン設定が正しくない場合、hi と lo の値が逆転し、検証ロジックが誤った結果を生成します。
  • コンパイラが int128 をサポートしていない環境では、本型の定義自体が成立しません。

保守担当者向けメモ

  • 本型はテスト専用のユーティリティです。本番コードでの使用は想定されていません。
  • メモリレイアウトの互換性に依存しているため、コンパイラやアーキテクチャの変更時には、ビット配置の整合性が維持されているか注意が必要です。
