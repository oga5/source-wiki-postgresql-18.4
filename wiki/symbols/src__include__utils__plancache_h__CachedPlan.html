CachedPlan

定義

  • 種別: struct
  • 定義箇所: src/include/utils/plancache.h:159

目的

実行可能なクエリ計画を保持し、再利用を可能にするためのデータ構造。解析・計画作成コストを削減するために使用される。

シグネチャまたは形状

typedef struct CachedPlan
{
	int			magic;
	List	   *stmt_list;
	bool		is_oneshot;
	bool		is_saved;
	bool		is_valid;
	Oid			planRoleId;
	bool		dependsOnRole;
	TransactionId saved_xmin;
	int			generation;
	int			refcount;
	MemoryContext context;
} CachedPlan;

入力と出力

  • **入力**: 構造体生成時に stmt_list（PlannedStmt のリスト）が設定される。
  • **出力**: 参照カウント (refcount) を通じて管理され、ReleaseCachedPlan によりメモリが解放される。

前提条件と不変条件

  • magic フィールドは常に CACHEDPLAN_MAGIC と一致する必要がある。
  • refcount が 0 になった時点で、context が示すメモリコンテキストが解放される。
  • is_valid が false の場合、その計画は再利用できず、再計画（Replan）が必要となる。

アルゴリズムまたは振る舞い

  • **ライフサイクル管理**: 参照カウント方式を採用。
  • **無効化**: スキーマ変更等のイベント発生時、is_valid フラグが false に設定される。
  • **oneshot 計画**: is_oneshot が真の場合、キャッシュ無効化の対象外となり、呼び出し元のメモリコンテキストに依存する。
  • **再計画判定**: saved_xmin を使用し、トランザクションの xmin が変化した際に再計画をトリガーする。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/commands/prepare.c、src/backend/executor/functions.c、src/backend/executor/spi.c 等で参照されている。
  • **呼び出し先**: 構造体内部で List（nodes/pg_list.h）や MemoryContext を保持する。具体的な関数呼び出し関係はソース抜粋からは特定不可。

関連データ構造

  • CachedPlanSource: 計画のソース（テキストや解析木）を管理する上位構造体。
  • PlannedStmt: stmt_list に格納される実行計画の単位。
  • ResourceOwner: 計画の生存期間を管理するために利用される。

失敗モードと境界条件

  • refcount の管理不備はメモリリークを引き起こす。
  • is_valid が false の状態で計画を実行しようとした場合、未定義の動作またはエラーが発生する可能性がある。

保守担当者向けメモ

  • 構造体の整合性チェックには CACHEDPLAN_MAGIC を使用すること。
  • is_oneshot クエリはキャッシュ無効化の監視対象外である点に注意。
  • 参照カウントの操作は ResourceOwner を介して適切に行うこと。
