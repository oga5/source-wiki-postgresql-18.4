SlruSharedData

定義

  • 種別: struct
  • 定義箇所: src/include/access/slru.h:61

目的

共有メモリ上で SLRU（Simple LRU）バッファプールの状態を保持し、メタデータ（CLOG, SUBTRANS 等）の効率的なキャッシュと並行アクセス制御を行うための構造体です。

シグネチャまたは形状

typedef struct SlruSharedData {
    int            num_slots;
    char         **page_buffer;
    SlruPageStatus *page_status;
    bool          *page_dirty;
    int64         *page_number;
    int           *page_lru_count;
    LWLockPadded  *buffer_locks;
    LWLockPadded  *bank_locks;
    int           *bank_cur_lru_count;
    XLogRecPtr    *group_lsn;
    int            lsn_groups_per_page;
    pg_atomic_uint64 latest_page_number;
    int            slru_stats_idx;
} SlruSharedData;

入力と出力

本構造体自体はデータ保持用であり、関数ではありません。各フィールドは SLRU のバッファ管理状態（ページ内容、状態、ロック、LRU カウンタ、WAL 同期情報）を格納します。

前提条件と不変条件

  • page_number および page_lru_count は、page_status が EMPTY の場合は未定義です。
  • bank_cur_lru_count はバンク単位で管理され、ページアクセスごとにインクリメントされます。
  • ページが「最も最近使用された（MRU）」状態であることは、page_lru_count[slotno] に現在の bank_cur_lru_count[bankno] を代入することで示されます。
  • ページ年齢が INT_MAX を超えない限り、カウンタのラップアラウンドが発生しても LRU 計算は正しく動作します。

アルゴリズムまたは振る舞い

  • **バッファ管理**: 固定サイズの num_slots を管理し、バンク単位で LRU 置換を行います。
  • **ロック階層**: bank_locks はメモリ上のバッファスロット検索を保護し、buffer_locks は各スロットの I/O を保護します。デッドロック回避のため、I/O 中はバンクのロックを解放します。
  • **最新ページ保護**: latest_page_number を参照し、ログの末尾ページが置換対象から除外されるように制御します。

呼び出し元と呼び出し先

  • 呼び出し元: src/backend/access/transam/slru.c 内の初期化および管理関数（SimpleLruInit 等）が本構造体のメモリ確保や初期化を行います。
  • 呼び出し先: 本構造体はデータ保持用であり、直接的な関数呼び出し先はありません。

関連データ構造

  • SlruCtlData: SLRU インスタンスを制御する非共有構造体。
  • SlruPageStatus: ページの状態（EMPTY, READ_IN_PROGRESS, VALID, WRITE_IN_PROGRESS）。
  • LWLockPadded: 並行性制御のためのパディング付き軽量ロック。

失敗モードと境界条件

  • bank_cur_lru_count のラップアラウンド: ページ年齢が INT_MAX を超えると LRU 計算が破綻する可能性があります。
  • メモリ不足: 共有メモリ確保時に num_slots が適切に設定されていない場合、バッファ管理が機能しません。

保守担当者向けメモ

  • 本構造体は共有メモリ上に配置されるため、フィールドの追加や変更は共有メモリレイアウトの整合性に影響を与えます。
  • group_lsn は pg_xact 等で WAL フラッシュを強制するために使用されますが、不要な場合は NULL に設定可能です。
