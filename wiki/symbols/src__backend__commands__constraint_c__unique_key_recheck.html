unique_key_recheck

定義

  • 種別: function
  • 定義箇所: src/backend/commands/constraint.c:38

目的

遅延制約（Deferrable Constraints）の検証を行うトリガー関数。トランザクションのコミット時や SET CONSTRAINTS コマンド実行時に、挿入または更新された行が一意性制約や排他制約に違反していないかを事後的に再確認する。

シグネチャまたは形状

Datum unique_key_recheck(PG_FUNCTION_ARGS)

入力と出力

  • **入力**: PG_FUNCTION_ARGS（トリガー実行コンテキスト TriggerData を含む）。
  • **出力**: Datum 型（常に PointerGetDatum(NULL) を返す）。

前提条件と不変条件

  • AFTER ROW トリガーとして呼び出される必要がある。
  • INSERT または UPDATE イベントに対してのみ有効。
  • 呼び出し元がトリガーマネージャであることを前提とする。

アルゴリズムまたは振る舞い

1. **プロトコル検証**: CALLED_AS_TRIGGER および TRIGGER_FIRED_AFTER / TRIGGER_FIRED_FOR_ROW を確認し、違反時はエラーを投げる。 2. **行の取得**: TriggerData から対象行の TID を取得。 3. **生存確認**: table_index_fetch_tuple を使用し、対象行が現在も有効か確認する。無効な場合はチェックをスキップして終了する。 4. **準備**: 対象インデックスを開き（RowExclusiveLock）、BuildIndexInfo で情報を取得。式や排他制約がある場合は EState を作成する。 5. **値抽出**: FormIndexDatum でインデックスキー値を抽出する。 6. **制約チェック**:

  • **一意性制約**: index_insert を UNIQUE_CHECK_EXISTING モードで呼び出し、既存エントリとの衝突を確認する。
  • **排他制約**: check_exclusion_constraint を呼び出し、競合をチェックする。

    • 7. **後処理**: EState の解放、スロットの破棄、インデックスのクローズを行う。

呼び出し元と呼び出し先

  • **呼び出し元**: トリガーマネージャ（src/backend/catalog/index.c 等でトリガーとして登録）。
  • **呼び出し先**: table_index_fetch_tuple, index_open, BuildIndexInfo, FormIndexDatum, index_insert, check_exclusion_constraint, index_insert_cleanup 等。

関連データ構造

  • TriggerData: トリガーの実行情報。
  • IndexInfo: インデックスのメタデータ。
  • EState: 式評価のための実行状態。
  • TupleTableSlot: 行データを保持するスロット。

失敗モードと境界条件

  • **プロトコル違反**: トリガー以外からの呼び出し、または AFTER ROW 以外のイベントで呼び出された場合にエラーとなる。
  • **行の削除**: 挿入後に削除された行についてはチェックをスキップする。
  • **HOT更新**: HOT更新によりトリガーが起動しない場合があるため、インデックスAM側のチェックと本関数の役割分担が重要となる。

保守担当者向けメモ

  • 歴史的経緯により、関数名は unique_key_recheck だが、排他制約のチェックも包含している。
  • index_insert 呼び出し時に UNIQUE_CHECK_EXISTING を使用している点は、実際の挿入ではなく既存データの検証であることを示している。
