pg_logical_slot_get_binary_changes

定義

  • 種別: function
  • 定義箇所: src/backend/replication/logical/logicalfuncs.c:348

目的

SQLインターフェースを通じて、指定された論理レプリケーションスロットからWALの変更内容をバイナリ形式で取得し、スロットの読み取り位置を更新します。

シグネチャまたは形状

Datum pg_logical_slot_get_binary_changes(PG_FUNCTION_ARGS);

入力と出力

  • **入力**: PG_FUNCTION_ARGS（SQL関数として呼び出される際の引数群。スロット名、LSN位置、オプション等を含む）。
  • **出力**: Datum（取得した変更内容を含む結果セットを Tuplestore として返却）。

前提条件と不変条件

  • 本関数は pg_logical_slot_get_changes_guts を呼び出すためのラッパーです。
  • 内部的に get（読み取り位置の更新を行う）および binary（バイナリ形式での出力）のフラグを有効にして処理を実行します。

アルゴリズムまたは振る舞い

1. pg_logical_slot_get_changes_guts(fcinfo, true, true) を呼び出します。 2. 第2引数 true により、読み取ったWALレコードの確認（confirm）を行い、スロットの読み取り位置を進めます。 3. 第3引数 true により、出力プラグインに対してバイナリ形式でのデータ生成を指示します。 4. 処理結果として得られた Tuplestore を呼び出し元へ返却します。

呼び出し元と呼び出し先

  • **呼び出し元**: SQLインターフェース（ユーザーによるSQLクエリ実行時）。
  • **呼び出し先**: pg_logical_slot_get_changes_guts（変更抽出ロジックの核となる関数）。

関連データ構造

  • DecodingOutputState: 出力プラグインから受け取ったデータを保持し、Tuplestore へ変換するために使用されます。

失敗モードと境界条件

  • 存在しないスロットを指定した場合や、権限不足、あるいは論理デコーディングのコンテキスト初期化に失敗した場合はエラーが発生します。
  • pg_logical_slot_get_changes_guts 内の PG_TRY/PG_CATCH ブロックにより、エラー発生時にはリソースリークを防ぐためのクリーンアップが行われます。

保守担当者向けメモ

  • 本関数は単なるラッパーであり、実質的なロジックは pg_logical_slot_get_changes_guts に集約されています。
  • 動作の詳細は src/backend/replication/logical/logicalfuncs.c の pg_logical_slot_get_changes_guts 実装を参照してください。
