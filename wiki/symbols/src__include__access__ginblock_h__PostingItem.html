PostingItem

定義

  • 種別: struct
  • 定義箇所: src/include/access/ginblock.h:183

目的

GIN（Generalized Inverted Index）の非リーフページにおいて、子ページへのポインタと対応するキー（TID）を保持するためのデータ構造です。

シグネチャまたは形状

typedef struct
{
	BlockIdData child_blkno;
	ItemPointerData key;
} PostingItem;

入力と出力

  • 入力: GINインデックスの非リーフページ内のデータ領域。
  • 出力: child_blkno（子ブロック番号）および key（インデックスキーとしての ItemPointerData）。

前提条件と不変条件

  • BlockNumber ではなく BlockIdData を使用することで、パディングによるメモリの浪費を回避しています。
  • 非リーフページ内での配置は、GinDataPageGetPostingItem マクロ等を通じてオフセット計算が行われることを前提としています。

アルゴリズムまたは振る舞い

  • GINインデックスの非リーフページにおいて、特定のキーに対応する子ページを検索するためのリンクとして機能します。
  • src/backend/access/gin/gindatapage.c 内の関数（GinDataPageAddPostingItem 等）により、ページ内への追加やメモリ移動（memmove）が行われます。

呼び出し元と呼び出し先

  • 呼び出し元（参照箇所）:
  • contrib/amcheck/verify_gin.c: インデックスの整合性検証時に使用。
  • src/backend/access/gin/gindatapage.c: 非リーフページ内のアイテム検索および追加処理で使用。
  • 呼び出し先:
  • 本構造体自体はデータ保持用であり、関数を呼び出すことはありません。

関連データ構造

  • BlockIdData: child_blkno の型。
  • ItemPointerData: key の型。
  • GinDataPageGetPostingItem: 本構造体へアクセスするためのマクロ。

失敗モードと境界条件

  • ページ内のアイテム数（maxoff）と sizeof(PostingItem) を用いた計算において、メモリレイアウトの整合性が崩れると誤ったポインタを参照するリスクがあります。
  • memmove を用いたページ内操作時に、オフセット計算が不適切であるとデータ破損を招く可能性があります。

保守担当者向けメモ

  • BlockIdData を使用しているため、直接 BlockNumber を代入しようとせず、適切な変換関数やマクロを介して操作してください。
  • 構造体のサイズ変更は、ページ内のアイテム配置計算（sizeof(PostingItem) を使用する箇所）に直接影響するため、慎重に行ってください。
