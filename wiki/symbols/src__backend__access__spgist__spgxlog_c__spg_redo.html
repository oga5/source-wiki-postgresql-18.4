spg_redo

定義

  • **種別**: 関数
  • **定義箇所**: src/backend/access/spgist/spgxlog.c:934

目的

SP-GiST インデックスの WAL（Write-Ahead Log）リプレイにおけるメインエントリポイントとして、記録された操作をインデックスページ上で再現する。

シグネチャまたは形状

void spg_redo(XLogReaderState *record)

入力と出力

  • **入力**: XLogReaderState *record（リプレイ対象の WAL レコード）
  • **出力**: なし（副作用としてインデックスページの状態を更新）

前提条件と不変条件

  • opCtx が初期化済みであること。
  • リプレイ処理の前後でメモリコンテキストの切り替えとリセットが行われる。

アルゴリズムまたは振る舞い

1. XLogRecGetInfo を使用して WAL レコードの操作タイプを抽出する。 2. opCtx にメモリコンテキストを切り替える。 3. 操作タイプに基づき、以下のいずれかのリプレイ関数へ処理を委譲する:

  • spgRedoAddLeaf, spgRedoMoveLeafs, spgRedoAddNode, spgRedoSplitTuple, spgRedoPickSplit, spgRedoVacuumLeaf, spgRedoVacuumRoot, spgRedoVacuumRedirect

    • 4. 未知の操作タイプが指定された場合は PANIC を発生させる。 5. 処理完了後、元のメモリコンテキストに戻し、opCtx をリセットする。

呼び出し元と呼び出し先

  • **呼び出し元**: src/include/access/rmgrlist.h において、SP-GiST のリソースマネージャ（RM_SPGIST_ID）のコールバックとして登録されている。
  • **呼び出し先**: 各種 spgRedo* 関数（上記「アルゴリズム」参照）。

関連データ構造

  • XLogReaderState: WAL レコードの読み取り状態を保持する構造体。
  • opCtx: リプレイ処理中に使用される一時的なメモリコンテキスト。

失敗モードと境界条件

  • 未知の操作タイプ（info）が渡された場合、elog(PANIC, ...) によりリカバリが停止する。

保守担当者向けメモ

  • 本関数は SP-GiST の WAL リプレイのディスパッチャである。新しい WAL レコードタイプを追加した際は、必ず switch 文にケースを追加すること。
  • リプレイ関数内では、スタンバイサーバーでの整合性を保つため、親ノードよりも子ページの更新を優先する順序が推奨される。
