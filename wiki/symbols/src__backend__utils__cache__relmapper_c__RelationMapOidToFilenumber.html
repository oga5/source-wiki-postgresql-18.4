RelationMapOidToFilenumber

定義

  • src/backend/utils/cache/relmapper.c:164

目的

指定されたリレーションOIDに対応する物理ファイル番号（RelFileNumber）を、メモリ上のマッピング情報から検索して返します。

シグネチャまたは形状

RelFileNumber RelationMapOidToFilenumber(Oid relationId, bool shared)

入力と出力

  • **入力**:
  • relationId: 検索対象のリレーションOID。
  • shared: 共有カタログを検索する場合は true、ローカルカタログの場合は false。
  • **出力**:
  • 該当する RelFileNumber。見つからない場合は InvalidRelFileNumber を返します。

前提条件と不変条件

  • 検索は、トランザクション内で保留中の更新（active_shared_updates または active_local_updates）を優先し、その後にメインのマップ（shared_map または local_map）を検索します。
  • マップファイルの内容は、RelationMapInitialize 等によって事前にメモリへロードされている必要があります。

アルゴリズムまたは振る舞い

1. shared 引数に基づき、検索対象の更新用マップとメインマップを決定します。 2. まず、トランザクション内で保留中の更新マップを走査し、relationId が一致するエントリがあればその mapfilenumber を即座に返します。 3. 保留中の更新で見つからない場合、メインのマップを走査し、一致するエントリがあればその mapfilenumber を返します。 4. いずれのマップにも存在しない場合、InvalidRelFileNumber を返します。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/commands/cluster.c
  • src/backend/utils/adt/dbsize.c
  • src/backend/utils/cache/relcache.c
  • **呼び出し先**:
  • 本関数は直接的な関数呼び出しを行わず、メモリ上の静的変数（active_shared_updates, shared_map, active_local_updates, local_map）を参照します。

関連データ構造

  • RelMapFile: マッピング情報を保持する構造体。
  • RelFileNumber: 戻り値の型。

失敗モードと境界条件

  • 指定されたOIDがどのマップにも存在しない場合、InvalidRelFileNumber が返されます。これは呼び出し側で適切に処理される必要があります。

保守担当者向けメモ

  • 本関数は、pg_class を介して物理ファイル位置を特定できない特殊なカタログ（マッピングされたカタログ）の解決に使用されます。
  • 検索順序（保留中更新 → メインマップ）は、トランザクションの整合性を保つために重要です。
