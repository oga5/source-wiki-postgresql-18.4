PLy_procedure_valid

定義

  • 種別: function
  • 定義箇所: src/pl/plpython/plpy_procedure.c:414

目的

キャッシュされたPL/Pythonプロシージャ（PLyProcedure）が、システムカタログ上の最新の関数定義（pg_proc）と一致しているかを検証する。

シグネチャまたは形状

static bool PLy_procedure_valid(PLyProcedure *proc, HeapTuple procTup)

入力と出力

  • **入力**:
  • proc: 検証対象のキャッシュ済みプロシージャ構造体。
  • procTup: システムカタログから取得した最新の pg_proc タプル。
  • **出力**:
  • bool: キャッシュが有効であれば true、無効または proc が NULL であれば false を返す。

前提条件と不変条件

  • procTup は有効な HeapTuple である必要がある。
  • キャッシュの有効性は、proc 内に保持されたトランザクションID（fn_xmin）およびタプル識別子（fn_tid）と、procTup の現在の値が一致することに依存する。

アルゴリズムまたは振る舞い

1. proc が NULL の場合は直ちに false を返す。 2. proc->fn_xmin と procTup の t_data から取得した xmin を比較する。 3. proc->fn_tid と procTup->t_self を ItemPointerEquals で比較する。 4. 両者が一致する場合のみ true を返し、それ以外は false を返す。

呼び出し元と呼び出し先

  • **呼び出し元**: src/pl/plpython/plpy_procedure.c 内の PLy_procedure_get（101行目付近）から呼び出される。
  • **呼び出し先**: ItemPointerEquals（PostgreSQLコアのユーティリティ関数）および HeapTupleHeaderGetRawXmin マクロ。

関連データ構造

  • PLyProcedure: キャッシュされた関数情報を保持する構造体。
  • HeapTuple: システムカタログ pg_proc の行データ。

失敗モードと境界条件

  • proc が NULL の場合、検証は失敗する。
  • 関数定義が ALTER FUNCTION 等で更新された場合、xmin または tid が変化するため、本関数は false を返し、キャッシュの再構築を促す。

保守担当者向けメモ

  • 本関数はキャッシュの整合性チェックの要である。関数定義の変更を検知する唯一の手段であるため、pg_proc の構造変更時には本ロジックが正しく追従しているか確認すること。
