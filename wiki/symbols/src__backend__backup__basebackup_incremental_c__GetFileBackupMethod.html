GetFileBackupMethod

定義

  • 種別: function
  • 定義箇所: src/backend/backup/basebackup_incremental.c:662

目的

特定のファイルに対して、フルバックアップを行うか、変更ブロックのみを抽出する増分バックアップを行うかを判定する。

シグネチャまたは形状

FileBackupMethod GetFileBackupMethod(IncrementalBackupInfo *ib, const char *path,
                                     Oid dboid, Oid spcoid,
                                     RelFileNumber relfilenumber, ForkNumber forknum,
                                     unsigned segno, size_t size,
                                     unsigned *num_blocks_required,
                                     BlockNumber *relative_block_numbers,
                                     unsigned *truncation_block_length)

入力と出力

  • **入力**:
  • ib: 増分バックアップのメタデータ構造体。
  • path, dboid, spcoid, relfilenumber, forknum, segno, size: 対象ファイルの識別情報およびサイズ。
  • **出力**:
  • 戻り値: BACK_UP_FILE_FULLY または BACK_UP_FILE_INCREMENTALLY。
  • num_blocks_required: 増分バックアップ時に必要なブロック数。
  • relative_block_numbers: 変更されたブロックの相対番号配列。
  • truncation_block_length: 切り詰め後のブロック長。

前提条件と不変条件

  • PrepareForIncrementalBackup が呼び出された後に実行されること（ib->buf.data == NULL で検証）。
  • spcoid および relfilenumber は有効な値であること。

アルゴリズムまたは振る舞い

1. **フルバックアップの強制条件**:

  • ファイルサイズが BLCKSZ の倍数でない、または RELSEG_SIZE を超える場合。
  • forknum が FSM_FORKNUM（空き領域マップ）である場合。
  • 前回のバックアップマニフェストに存在しないファイルの場合。
  • データベース全体が新規作成されたと判断される場合。

    • 2. **増分判定**:

  • BlockRefTable を参照し、変更がない場合は増分バックアップを選択（ただしサイズが0の場合はフルバックアップ）。
  • 変更ブロック数がファイル全体の90%を超える場合、効率化のためフルバックアップに切り替える。

    • 3. **ブロック情報の整理**:

  • 変更ブロック番号をソートし、セグメント内の相対位置に変換する。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/backup/basebackup.c
  • **呼び出し先**: backup_file_lookup, GetIncrementalFilePath, BlockRefTableGetEntry, BlockRefTableEntryGetBlocks, qsort

関連データ構造

  • IncrementalBackupInfo: 増分バックアップのメタデータ管理。
  • BlockRefTable: 変更ブロックの追跡用テーブル。
  • FileBackupMethod: 判定結果（列挙型）。

失敗モードと境界条件

  • ブロック番号の計算時にオーバーフローが発生した場合、内部エラーとして ereport(ERROR, ...) を発行する。
  • 90%ルールにより、変更量が多い場合は増分バックアップの利点が失われるためフルバックアップへフォールバックする。

保守担当者向けメモ

  • 90%の閾値はヒューリスティックであり、将来的に設定可能にすることが検討されている。
  • BlockRefTable のメモリ消費量に注意が必要。
