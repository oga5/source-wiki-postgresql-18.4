parser_state

定義

  • **種別**: struct
  • **定義箇所**: src/tools/pg_bsd_indent/indent_globs.h:234

目的

ソースコードの構文解析中における現在の状態（インデントレベル、括弧のネスト、宣言の有無、コメントの配置など）を保持・管理するための主要なデータ構造です。

入力と出力

  • **入力**: 構文解析器（lexi.c や parse.c）によって、トークン解析結果や行の属性が各フィールドに書き込まれます。
  • **出力**: dump_line や pr_comment などの関数が、現在の状態を参照して整形後のコードやコメントを出力します。

前提条件と不変条件

  • ps（グローバルインスタンス）は、プログラム全体で共有される単一の状態として扱われます。
  • p_stack や il などのスタック構造は、tos（スタックポインタ）によって管理されます。

アルゴリズムまたは振る舞い

  • **状態追跡**: 解析中のコード行の属性（in_decl, in_stmt 等）を更新し、後続の行のインデントや整形ルールを決定します。
  • **バッファ管理**: dump_line が行を書き出す際、bl_line や com_lines などの状態を更新します。
  • **再帰的解析の補助**: search_brace フラグにより、if や while 後の文の開始まで情報をバッファリングする制御を行います。

呼び出し元と呼び出し先

  • **呼び出し元**: indent.c（transient_state としてコピー利用）、lexi.c（lexi 関数の引数として渡される）、parse.c。
  • **呼び出し先**: 本構造体自体はデータ保持用であり、関数を呼び出すことはありません。

関連データ構造

  • ps: indent_globs.h で定義されるグローバルな parser_state インスタンス。
  • state_stack[5], match_state[5]: 状態の保存や比較に使用される parser_state の配列。

失敗モードと境界条件

  • **スタックオーバーフロー**: p_stack (256), il (64), cstk (32), paren_indents (20) の各配列サイズを超えるネストが発生した場合の挙動は定義されていません。
  • **バッファ制限**: procname は 100 文字に制限されています。

保守担当者向けメモ

  • 本構造体はグローバル変数として扱われるため、変更時は indent.c を含む全参照箇所への影響を考慮してください。
  • CHECK_SIZE_* マクロによるメモリ再配置が発生する場合、ポインタの整合性に注意が必要です。
  • 古い BSD コードベースであるため、現代的なカプセル化は行われていません。各フィールドの役割は indent_globs.h のコメントを参照してください。
