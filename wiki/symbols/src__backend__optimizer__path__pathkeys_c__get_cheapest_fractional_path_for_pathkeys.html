get_cheapest_fractional_path_for_pathkeys

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/path/pathkeys.c:665

目的

指定されたパスキー（pathkeys）を満たし、かつ指定された割合（fraction）のタプルを取得する際のコストが最小となる Path を、候補リストの中から選択する。

シグネチャまたは形状

Path *get_cheapest_fractional_path_for_pathkeys(List *paths,
                                                List *pathkeys,
                                                Relids required_outer,
                                                double fraction)

入力と出力

  • **入力**:
  • paths: 検討対象となる Path ノードのリスト。
  • pathkeys: 満たすべきソート順序を表すパスキーリスト。
  • required_outer: パスが依存する外部リレーションの集合。
  • fraction: コスト計算に使用する取得割合（0.0〜1.0）。
  • **出力**:
  • 条件を満たす中で最もコストが低い Path へのポインタ。該当するものがない場合は NULL を返す。

前提条件と不変条件

  • paths リスト内の各 Path は、正規化された pathkeys を保持している必要がある。
  • compare_fractional_path_costs を用いてコストの優劣を判定する。

アルゴリズムまたは振る舞い

1. paths リストを順次走査する。 2. 現在の matched_path が既に存在する場合、compare_fractional_path_costs を呼び出し、現在のパスの方がコストが高ければスキップする。 3. pathkeys_contained_in を用いて、対象パスが要求された pathkeys を提供可能か判定する。 4. bms_is_subset を用いて、対象パスの外部依存関係（PATH_REQ_OUTER）が required_outer のサブセットであるかを確認する。 5. 上記の条件をすべて満たす場合、そのパスを matched_path として更新し、リストの最後まで繰り返す。

関連データ構造

  • Path: クエリ実行計画のノード。
  • List: パスやパスキーを保持するリスト構造。
  • Relids: リレーションIDの集合（ビットマップ）。

失敗モードと境界条件

  • paths が空の場合、matched_path は NULL のまま返される。
  • 条件を満たすパスが存在しない場合、NULL が返される。

保守担当者向けメモ

  • コスト比較はパスキーの比較よりも計算コストが低いため、先にコスト比較を行うことで最適化を図っている。この前提が現在も有効かどうかは継続的な確認が必要である（ソース内の XXX コメント参照）。
