table_index_fetch_tuple_check

定義

  • 種別: function
  • 定義箇所: src/backend/access/table/tableam.c:208

目的

指定されたリレーションおよびタプルID（TID）に対し、スナップショットに基づいたタプルの存在確認を行います。

シグネチャまたは形状

bool table_index_fetch_tuple_check(Relation rel, ItemPointer tid, Snapshot snapshot, bool *all_dead)

入力と出力

  • **入力**:
  • rel: 対象のリレーション。
  • tid: 検索対象のタプルID。
  • snapshot: 可視性判定に使用するスナップショット。
  • all_dead: 呼び出し元から渡されるポインタ。タプルが完全にデッドであるかどうかの判定結果が格納される。
  • **出力**:
  • bool: タプルが見つかり、かつスナップショットに対して可視である場合に true を返す。

前提条件と不変条件

  • rel は有効なリレーションである必要があります。
  • 内部で一時的な TupleTableSlot を作成して使用するため、呼び出し元でスロットを管理する必要はありません。

アルゴリズムまたは振る舞い

1. table_slot_create を呼び出し、リレーションに対応する一時的なスロットを確保します。 2. table_index_fetch_begin を呼び出し、インデックスフェッチ用のスキャン状態を初期化します。 3. table_index_fetch_tuple を呼び出し、指定された tid のタプルをスロットに読み込み、可視性を判定します。 4. table_index_fetch_end を呼び出し、スキャン状態を解放します。 5. ExecDropSingleTupleTableSlot を呼び出し、手順1で確保したスロットを破棄します。 6. 判定結果（found）を返します。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/access/nbtree/nbtinsert.c (インデックス挿入時の重複チェック等で使用)
  • **呼び出し先**:
  • table_slot_create
  • table_index_fetch_begin
  • table_index_fetch_tuple
  • table_index_fetch_end
  • ExecDropSingleTupleTableSlot

関連データ構造

  • Relation: テーブルのメタデータ構造。
  • ItemPointer: タプルを特定するための物理的な位置情報。
  • Snapshot: トランザクションの可視性制御用構造体。
  • TupleTableSlot: タプルを一時的に保持するための構造体。

失敗モードと境界条件

  • 内部でメモリ確保（table_slot_create）を行うため、メモリ不足時にはエラーが発生する可能性があります。
  • table_index_fetch_tuple が内部で参照するストレージ層の制約により、無効な tid が渡された場合の挙動は各アクセスメソッドの実装に依存します。

保守担当者向けメモ

  • 本関数は、インデックスフェッチのライフサイクル（開始・取得・終了）を単一の関数にカプセル化したラッパーです。
  • 頻繁に呼び出されるパスで使用される場合、スロットの作成・破棄コストが無視できない可能性があるため、パフォーマンス要件に応じて最適化を検討してください。
