index_checkable

定義

  • 種別: function
  • 定義箇所: contrib/amcheck/verify_common.c:158

目的

指定されたリレーションが検証可能なインデックスであるかを判定し、検証の前提条件を満たしているかを確認します。

シグネチャまたは形状

bool index_checkable(Relation rel, Oid am_id)

入力と出力

  • **入力**:
  • rel: 検証対象のリレーション（Relation 構造体）。
  • am_id: 期待されるアクセス手法（AM）の OID。
  • **出力**:
  • 成功時: true（amcheck_index_mainfork_expected の戻り値に依存）。
  • 失敗時: 内部で ereport(ERROR, ...) を呼び出し、処理を中断します。

前提条件と不変条件

  • 対象リレーションが有効なインデックスであること。
  • 対象が他セッションの一時テーブルではないこと。
  • 対象インデックスが indisvalid フラグを保持していること。

アルゴリズムまたは振る舞い

1. **型とAMの検証**: rel がインデックスであり、かつ指定された am_id と一致するかを確認します。不一致の場合はシステムカタログから情報を取得し、エラーを報告します。 2. **一時テーブルのチェック**: RELATION_IS_OTHER_TEMP(rel) を使用し、他セッションの一時テーブルであればエラーを報告します。 3. **有効性のチェック**: rel->rd_index->indisvalid を確認し、無効なインデックスであればエラーを報告します。 4. **メインフォークの確認**: 最後に amcheck_index_mainfork_expected(rel) を呼び出し、その結果を返します。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • contrib/amcheck/verify_common.c 内の検証ロジック（amcheck_lock_relation_and_check 等）。
  • **呼び出し先**:
  • amcheck_index_mainfork_expected
  • SearchSysCache1（エラー報告時）
  • ereport

関連データ構造

  • Relation: 検証対象のインデックス情報。
  • Form_pg_am: アクセス手法の名称取得に使用。

失敗モードと境界条件

  • **ERRCODE_FEATURE_NOT_SUPPORTED**: 以下の条件で発生します。
  • 指定された am_id とインデックスの実際の AM が異なる場合。
  • 他セッションの一時テーブルである場合。
  • インデックスが有効（indisvalid）でない場合。

保守担当者向けメモ

  • 本関数は検証のゲートキーパーとして機能します。新しい検証ロジックを追加する際は、必ず本関数を介して対象の妥当性を確認してください。
  • amcheck_index_mainfork_expected の戻り値が最終的な判定結果となります。
