add_partial_path_precheck

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/util/pathnode.c:923

目的

並列実行可能なパス（partial path）を候補リストに追加する前に、そのパスが既存のパスと比較して明らかに劣っているか、あるいは採用する価値があるかを事前に判定し、不要な計算を回避する。

シグネチャまたは形状

bool add_partial_path_precheck(RelOptInfo *parent_rel, int disabled_nodes, Cost total_cost, List *pathkeys)

入力と出力

  • **入力**:
  • parent_rel: 対象となるリレーション情報。
  • disabled_nodes: 無効化されたプランノードのビットマスク。
  • total_cost: 評価対象パスの総コスト。
  • pathkeys: 評価対象パスのソート順序。
  • **出力**:
  • bool: パスを候補として検討すべき場合は true、即座に破棄すべき場合は false を返す。

前提条件と不変条件

  • parent_rel->partial_pathlist には既に検討済みの部分パスが格納されている。
  • 部分パスの評価において、起動コスト（startup cost）は考慮されない（並列実行時は完了まで実行されるため）。

アルゴリズムまたは振る舞い

1. **部分パスリストとの比較**: parent_rel->partial_pathlist を走査し、既存のパスと pathkeys を比較する。

  • STD_FUZZ_FACTOR を用いた曖昧なコスト比較を行い、明らかに劣る場合は false、明らかに優れている場合は true を返す。

    • 2. **非並列プランとの比較**: 部分パスリストでの判定が確定しない場合、add_path_precheck を呼び出し、非並列プランとの比較を行う。

  • この際、起動コストと総コストの両方に total_cost を渡す。

    • 3. 上記のいずれのチェックも通過した場合に true を返す。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/optimizer/path/joinpath.c (複数箇所)
  • **呼び出し先**:
  • compare_pathkeys
  • add_path_precheck

関連データ構造

  • RelOptInfo: パスリストを保持するリレーション情報。
  • Path: パスのコストやソート順序を保持する構造体。
  • PathKeysComparison: パスキーの比較結果（PATHKEYS_DIFFERENT, PATHKEYS_BETTER1 等）。

失敗モードと境界条件

  • 既存のパスとコスト差が STD_FUZZ_FACTOR 以内の場合、即座の破棄は行われない。
  • add_path_precheck が false を返した場合、この関数も false を返す。

保守担当者向けメモ

  • add_path() と異なり、この関数は常に pathkeys を比較する。これは partial_pathlist が通常短いため、早期に決定を下すことで効率化を図る設計である。
