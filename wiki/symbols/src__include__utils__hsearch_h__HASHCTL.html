HASHCTL

定義

  • 種別: struct
  • 定義箇所: src/include/utils/hsearch.h:65

目的

PostgreSQL の動的ハッシュテーブル（dynahash）を作成する際、その構成パラメータや動作を制御するための設定構造体です。

シグネチャまたは形状

typedef struct HASHCTL
{
    long        num_partitions;
    long        ssize;
    long        dsize;
    long        max_dsize;
    Size        keysize;
    Size        entrysize;
    HashValueFunc hash;
    HashCompareFunc match;
    HashCopyFunc keycopy;
    HashAllocFunc alloc;
    MemoryContext hcxt;
    HASHHDR    *hctl;
} HASHCTL;

入力と出力

  • **入力**: hash_create() 関数に引数として渡されます。各フィールドは、対応するフラグ（HASH_PARTITION, HASH_ELEM, HASH_FUNCTION 等）が hash_create の第3引数で指定された場合にのみ参照されます。
  • **出力**: なし（本構造体は設定の保持のみに使用されます）。

前提条件と不変条件

  • HASH_ELEM フラグは現在必須であり、keysize（キーのバイト長）と entrysize（ユーザー要素の合計バイト長）の指定が求められます。
  • num_partitions を指定する場合、その値は2のべき乗である必要があります。
  • 共有メモリを使用する場合は HASH_SHARED_MEM フラグが必須であり、その際 hctl フィールドに共有メモリ上のヘッダー位置を指定する必要があります。

アルゴリズムまたは振る舞い

hash_create() 呼び出し時に、この構造体の各フィールドがハッシュテーブルの初期化ロジックに読み込まれます。 1. **メモリ管理**: hcxt が指定されればそのメモリコンテキストを使用し、alloc が指定されればカスタムアロケータを使用します。 2. **ハッシュ計算**: hash（ハッシュ関数）、match（比較関数）、keycopy（キーコピー関数）が指定されれば、デフォルトの動作を上書きします。 3. **構造制御**: num_partitions, ssize, dsize 等により、ハッシュテーブルの物理的なレイアウトや拡張性を決定します。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • contrib/dblink/dblink.c
  • contrib/pg_stat_statements/pg_stat_statements.c
  • contrib/pg_trgm/trgm_regexp.c
  • contrib/postgres_fdw/connection.c および shippable.c
  • contrib/tablefunc/tablefunc.c
  • src/backend/access/common/heaptuple.c
  • src/backend/access/gist/gistbuild.c および gistbuildbuffers.c
  • src/backend/access/hash/hashpage.c
  • src/backend/access/heap/rewriteheap.c
  • **呼び出し先**: 本構造体自体はデータ保持用であり、直接関数を呼び出すことはありません。

関連データ構造

  • HTAB: hash_create() の結果として生成されるハッシュテーブル管理構造体。
  • HASHHDR: 共有メモリ上のハッシュテーブルヘッダー。
  • MemoryContext: メモリ割り当てに使用されるコンテキスト。

失敗モードと境界条件

  • 必須フラグ（HASH_ELEM）が指定されていない場合、hash_create() は正常に初期化できません。
  • num_partitions が2のべき乗でない場合、ハッシュテーブルの動作は保証されません。
  • 共有メモリ設定時に hctl が適切に初期化されていない場合、メモリ破壊やセグメンテーション違反を引き起こす可能性があります。

保守担当者向けメモ

  • 本構造体は dynahash の公開インターフェースの一部です。フィールドを追加・変更する場合は、hash_create() 内のフラグ判定ロジックとの整合性に注意してください。
  • ユーザーデータは HASHELEMENT の直後に配置され、ハッシュキーはユーザーデータの先頭にあることが期待されるため、keysize と entrysize の設定には注意が必要です。
