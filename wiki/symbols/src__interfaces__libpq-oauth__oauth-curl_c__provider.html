provider

定義

  • 種別: struct
  • 定義箇所: src/interfaces/libpq-oauth/oauth-curl.c:114

目的

OAuth 2.0 認可サーバーに関するメタデータおよびエンドポイント情報を保持するためのデータ構造です。

シグネチャまたは形状

struct provider
{
    char       *issuer;
    char       *token_endpoint;
    char       *device_authorization_endpoint;
    struct curl_slist *grant_types_supported;
};

入力と出力

  • **入力**: 認可サーバーから取得した Discovery ドキュメント（JSON）の内容をパースして各フィールドに格納します。
  • **出力**: 認証フローの各ステップ（Device Authorization や Token Request）において、通信先 URL を特定するために参照されます。

前提条件と不変条件

  • 各文字列フィールド（issuer, token_endpoint, device_authorization_endpoint）は、メモリ確保された文字列へのポインタである必要があります。
  • grant_types_supported は libcurl の curl_slist 構造体として管理され、不要になった際には適切に解放される必要があります。

アルゴリズムまたは振る舞い

本構造体は、OAuth 2.0 の Discovery プロセスを通じて得られた情報を保持するコンテナとして機能します。認証フローの初期段階でサーバーから取得したメタデータを格納し、以降の認可リクエストおよびトークンリクエスト時に、対応するエンドポイント URL を提供します。

呼び出し元と呼び出し先

  • **呼び出し元**: ソースコード内の直接的な呼び出し元は、提供された情報からは特定できません。
  • **呼び出し先**: 本構造体自体はデータ保持用であり、関数呼び出しは行いません。
  • **注意**: ツールが提示した contrib/ や src/backend/ 内の provider という名称の参照は、本構造体とは異なる文脈（セキュリティプロバイダーや照合順序プロバイダー）を指しており、本シンボルとは無関係です。

関連データ構造

  • async_ctx: 認証フローの全体状態を管理するコンテキストであり、本構造体を保持する可能性があります。
  • curl_slist: libcurl のリスト構造体であり、サポートされている認可グラントタイプを保持するために使用されます。

失敗モードと境界条件

  • メモリ不足により各フィールドの割り当てが失敗する可能性があります。
  • 認可サーバーから返される JSON に必要なエンドポイントが含まれていない場合、本構造体のフィールドが未初期化または NULL になる可能性があります。

保守担当者向けメモ

  • 本構造体は libpq-oauth サブシステム専用です。他の PostgreSQL コンポーネント（contrib/sepgsql や collationcmds.c 等）で見られる同名のシンボルと混同しないよう注意してください。
  • grant_types_supported の解放には curl_slist_free_all を使用する必要があります。
