generate_recursion_path

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/prep/prepunion.c:356

目的

再帰的 UNION クエリのための実行パス（RecursiveUnion）を生成し、対応する RelOptInfo を構築する。

シグネチャまたは形状

static RelOptInfo *generate_recursion_path(SetOperationStmt *setOp, PlannerInfo *root, List *refnames_tlist, List **pTargetList)

入力と出力

  • **入力**:
  • setOp: 処理対象の SetOperationStmt（UNION である必要がある）。
  • root: 現在のクエリの PlannerInfo。
  • refnames_tlist: 参照名を含むターゲットリスト。
  • **出力**:
  • 戻り値: 生成された RelOptInfo。
  • pTargetList: 構築されたターゲットリストが格納される。

前提条件と不変条件

  • setOp->op は SETOP_UNION でなければならない（それ以外はエラー）。
  • root->wt_param_id が 0 以上であること（作業テーブルの ID が割り当て済みであること）。
  • UNION（ALL なし）の場合、すべてのカラム型がハッシュ可能であること。

アルゴリズムまたは振る舞い

1. **再帰的プランニング**: recurse_set_operations を用いて、左辺（非再帰項）と右辺（再帰項）を個別にプランニングする。 2. **パスの取得**: 各辺の cheapest_total_path を取得し、root->non_recursive_path に左辺のパスを一時的に設定して右辺のプランニングを補助する。 3. **ターゲットリスト生成**: generate_append_tlist を呼び出し、両辺の出力を統合するターゲットリストを作成する。 4. **リレーション構築**: fetch_upper_rel で UPPERREL_SETOP を取得し、ターゲットリストを適用する。 5. **グループ化**: UNION ALL でない場合は generate_setop_grouplist でグループ化条件を特定し、ハッシュ可能か検証する。 6. **パス生成**: create_recursiveunion_path で RecursiveUnion パスを作成し、add_path でリレーションに追加後、postprocess_setop_rel を実行する。

呼び出し元と呼び出し先

  • **呼び出し元**: recurse_set_operations（src/backend/optimizer/prep/prepunion.c:146 付近）。
  • **呼び出し先**: recurse_set_operations, build_setop_child_paths, generate_append_tlist, fetch_upper_rel, create_pathtarget, generate_setop_grouplist, grouping_is_hashable, create_recursiveunion_path, add_path, postprocess_setop_rel。

関連データ構造

  • SetOperationStmt: 集合演算の定義。
  • RelOptInfo: プランニング中のリレーション情報。
  • Path: 実行パスの基底構造。
  • PlannerInfo: クエリ全体のプランニング状態。

失敗モードと境界条件

  • SETOP_UNION 以外の演算子が渡された場合、elog(ERROR) を発生させる。
  • UNION（ALL なし）においてカラム型がハッシュ可能でない場合、ereport(ERROR) で ERRCODE_FEATURE_NOT_SUPPORTED を送出する。

保守担当者向けメモ

  • 本関数は再帰的 UNION に特化しており、通常の UNION とは異なり、左右の入力を Append に統合せず、RecursiveUnion ノードとして処理する点に注意すること。
  • dNumGroups の見積もりは現在、最悪ケース（入力サイズの合計）として計算されている。
