PendingFsyncEntry

定義

  • 種別: struct
  • 定義箇所: src/backend/storage/sync/sync.c:56

目的

チェックポイント処理において、ファイル同期（fsync）要求をハッシュテーブル内で管理するためのエントリ構造体です。重複する要求の集約や、チェックポイントの境界を跨いだ処理順序の制御に使用されます。

シグネチャまたは形状

typedef struct
{
	FileTag		tag;			/* 同期対象の識別子（ハンドラとファイル） */
	CycleCtr	cycle_ctr;		/* 最古の要求の sync_cycle_ctr */
	bool		canceled;		/* 最近キャンセルされた場合に true */
} PendingFsyncEntry;

入力と出力

  • 本構造体自体はデータ保持用であり、関数ではありません。
  • ハッシュテーブル pendingOps のエントリとして使用され、hash_search や hash_seq_search を通じて読み書きされます。

前提条件と不変条件

  • tag は同期対象のファイルを一意に識別する必要があります。
  • cycle_ctr は、チェックポイントの実行サイクルを管理するために使用され、要求の有効性を判定する基準となります。

アルゴリズムまたは振る舞い

  • pendingOps ハッシュテーブル内で、同一ファイルに対する複数の同期要求を PendingFsyncEntry として集約します。
  • canceled フラグにより、同期実行前に削除されたファイル等の要求を適切にスキップします。
  • ProcessSyncRequests 等の関数内で、ハッシュテーブルを走査する際の型として利用されます。

呼び出し元と呼び出し先

  • 呼び出し元: src/backend/storage/sync/sync.c 内の ProcessSyncRequests、RememberSyncRequest 等の関数が、ハッシュテーブル操作の型として本構造体を参照しています。
  • 呼び出し先: 構造体自体はデータ保持用であり、直接的な関数呼び出しは行いません。

関連データ構造

  • FileTag: 同期対象を特定するための識別子。
  • CycleCtr: チェックポイントのサイクル管理用カウンタ。
  • pendingOps: 本構造体を格納するハッシュテーブル。

失敗モードと境界条件

  • メモリ不足: pendingOps への挿入時にメモリ確保が失敗する可能性があります。
  • 競合: ファイルが削除された後に fsync が試行される場合、canceled フラグや errno の確認により競合を解決します。

保守担当者向けメモ

  • pendingOpsCxt メモリコンテキスト上で確保されるため、チェックポイント処理中のメモリ確保には注意が必要です。
  • cycle_ctr の値は、チェックポイントの無限ループを防ぐための重要な指標であるため、更新ロジックを変更する際は注意してください。
