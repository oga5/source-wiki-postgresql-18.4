ginScanToDelete

定義

  • 種別: function
  • 定義箇所: src/backend/access/gin/ginvacuum.c:245

目的

GINインデックスのポストティングツリーを再帰的に走査し、空になったページを特定して物理的に削除（再利用可能化）します。

シグネチャまたは形状

static bool ginScanToDelete(GinVacuumState *gvs, BlockNumber blkno, bool isRoot, DataPageDeleteStack *parent, OffsetNumber myoff)

入力と出力

  • **入力**:
  • gvs: バキューム処理の状態管理構造体。
  • blkno: 現在走査中のブロック番号。
  • isRoot: 現在のページがルートかどうかを示すフラグ。
  • parent: 親ノードの削除スタック情報。
  • myoff: 親ノード内での自身のオフセット番号。
  • **出力**:
  • bool: ページが削除された場合に true を返します。

前提条件と不変条件

  • GinPageIsData(page) が真であること（データページであること）。
  • ルートページおよび最右端のブランチは削除対象外です。
  • ポストティングツリーの削除時、親から子へのロック順序が厳密に管理されています。

アルゴリズムまたは振る舞い

1. **スタック管理**: 非ルートページの場合、DataPageDeleteStack を用いて親ノードとの関係を保持します。 2. **再帰的走査**: 非リーフページの場合、子ノードに対して再帰的に ginScanToDelete を呼び出します。 3. **空判定**: リーフページまたは子ノードを持たないページが空であるかを確認します。 4. **削除実行**: ページが空かつ最右端でない場合、ginDeletePage を呼び出して物理的な削除を行います。 5. **バッファ管理**: 削除されなかったページは、後続の処理のために leftBuffer として保持し、削除された場合はバッファを解放します。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • ginVacuumPostingTree (間接的)
  • ginScanToDelete (自身を再帰的に呼び出し)
  • **呼び出し先**:
  • ginScanToDelete (再帰)
  • ginDeletePage
  • ReadBufferExtended, LockBuffer, ReleaseBuffer 等のバッファ管理関数

関連データ構造

  • GinVacuumState: バキューム処理の状態。
  • DataPageDeleteStack: 削除対象のページ階層を追跡するためのスタック構造。

失敗モードと境界条件

  • **ルートページ**: isRoot が真の場合、削除は行われません。
  • **最右端ページ**: GinPageRightMost が真の場合、削除は行われません。
  • **ロック**: 複雑なロック処理を伴うため、デッドロック回避のための順序制御が不可欠です。

保守担当者向けメモ

  • 本関数はポストティングツリーの構造を深く操作するため、変更時には親ノードのダウンリンク更新ロジックに細心の注意を払ってください。
  • leftBuffer の管理は、隣接ページとの整合性を保つために重要です。
