ltq_regex

定義

  • 種別: function
  • 定義箇所: contrib/ltree/lquery_op.c:277

目的

ltree 型のデータが lquery 型のパターンに適合するかを判定し、PostgreSQL の ~ 演算子のバックエンドロジックとして機能する。

シグネチャまたは形状

Datum ltq_regex(PG_FUNCTION_ARGS)

入力と出力

  • **入力**: PG_FUNCTION_ARGS を介して ltree 型（第1引数）と lquery 型（第2引数）を受け取る。
  • **出力**: bool 型の判定結果を Datum として返す。

前提条件と不変条件

  • 入力された ltree および lquery は有効なメモリ領域にある必要がある。
  • PG_GETARG_LTREE_P および PG_GETARG_LQUERY_P マクロにより、必要に応じてデータがデトースト（detoast）される。

アルゴリズムまたは振る舞い

1. PG_GETARG_LTREE_P および PG_GETARG_LQUERY_P を使用して引数を取得する。 2. checkCond 関数を呼び出し、lquery のレベル構造と ltree のパス構造を照合する。 3. PG_FREE_IF_COPY を使用して、必要に応じてコピーされた引数メモリを解放する。 4. PG_RETURN_BOOL を介してマッチング結果を返す。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • contrib/ltree/_ltree_op.c（array_iterator 経由）
  • contrib/ltree/lquery_op.c（DirectFunctionCall2 経由）
  • contrib/ltree/ltree_gist.c（DirectFunctionCall2 経由）
  • **呼び出し先**:
  • checkCond（マッチングの主要ロジック）

関連データ構造

  • ltree: 階層パスを保持する構造体。
  • lquery: 検索パターンを保持する構造体。

失敗モードと境界条件

  • checkCond 内部で再帰的な走査が行われるため、極端に深い階層構造を持つデータに対してはスタックオーバーフローのリスクがある（check_stack_depth() による保護が想定される）。
  • メモリ管理は PostgreSQL の PG_FREE_IF_COPY に依存する。

保守担当者向けメモ

  • 本関数は ltree ~ lquery 演算子のエントリポイントである。
  • 比較ロジックの詳細は checkCond に依存しているため、マッチングアルゴリズムの変更は checkCond を参照すること。
  • PG_FUNCTION_INFO_V1(ltq_regex) により、PostgreSQL の関数マネージャに登録されている。
