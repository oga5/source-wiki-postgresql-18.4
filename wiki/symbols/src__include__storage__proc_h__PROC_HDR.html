PROC_HDR

定義

  • 種別: struct
  • 定義箇所: src/include/storage/proc.h:383

目的

PostgreSQLの全バックエンドプロセスを管理するためのグローバルなヘッダー構造体です。共有メモリ上に配置され、各プロセスの状態（PGPROC）の集約、空きリストの管理、およびプロセス間での高速な状態共有を提供します。

シグネチャまたは形状

typedef struct PROC_HDR {
    PGPROC         *allProcs;
    TransactionId  *xids;
    XidCacheStatus *subxidStates;
    uint8          *statusFlags;
    uint32          allProcCount;
    dlist_head      freeProcs;
    dlist_head      autovacFreeProcs;
    dlist_head      bgworkerFreeProcs;
    dlist_head      walsenderFreeProcs;
    pg_atomic_uint32 procArrayGroupFirst;
    pg_atomic_uint32 clogGroupFirst;
    ProcNumber      walwriterProc;
    ProcNumber      checkpointerProc;
    int             spins_per_delay;
    int             startupBufferPinWaitBufId;
} PROC_HDR;

入力と出力

  • **入力**: 共有メモリ初期化時に ShmemInitStruct を通じて割り当てられます。
  • **出力**: システム内の各バックエンドプロセスが、自身の状態更新や他プロセスのトランザクション状態参照のために本構造体を参照します。

前提条件と不変条件

  • allProcs 配列は、準備済みトランザクション（prepared transactions）用のダミーを除いた全 PGPROC を保持します。
  • xids, subxidStates, statusFlags は、procarray 内の各 PGPROC の状態をミラーリングしており、整合性が保たれる必要があります。
  • walwriterProc や checkpointerProc などの補助プロセスは、システム内で同時に1つのみ存在することが前提です。

アルゴリズムまたは振る舞い

  • **プロセス管理**: freeProcs 等のリストを用いて、新しいバックエンドプロセスに PGPROC スロットを割り当てます。
  • **状態共有**: 各 PGPROC の主要フィールドをミラー配列（xids 等）に保持することで、ProcArray スキャン等の処理においてキャッシュ効率を高めた高速な参照を可能にします。
  • **同期**: procArrayGroupFirst および clogGroupFirst を用いて、グループ単位でのトランザクション状態更新や待機制御を行います。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/storage/lmgr/proc.c にて初期化および管理されます。clog.c, xlog.c, checkpointer.c, procarray.c 等の各サブシステムからグローバル変数 ProcGlobal を介して参照されます。
  • **呼び出し先**: 本構造体自体はデータ構造であり、関数を直接呼び出すことはありません。

関連データ構造

  • PGPROC: 各バックエンドの個別状態を保持する構造体。
  • dlist_head: プロセスリスト管理に使用される双方向リスト。
  • ProcNumber: プロセス識別番号。
  • TransactionId, XidCacheStatus: トランザクション状態の管理に使用。

失敗モードと境界条件

  • 共有メモリの割り当て失敗時、システム起動時に致命的なエラーが発生します。
  • allProcCount を超えるプロセスが要求された場合、スロット不足によりプロセス生成が失敗します。
  • ミラー配列と PGPROC 本体の状態に不整合が生じた場合、デッドロック検出やトランザクション可視性判定に誤りが生じる可能性があります。

保守担当者向けメモ

  • PGPROC のフィールドにはロックなしでアクセス可能なものと、ProcArrayLock 等の保持が必須なものが混在しています。変更時は ProcArray 周りの設計ドキュメントを併読してください。
  • pgxactoff を用いたインデックスアクセスはキャッシュ効率を最適化するための設計です。複雑な並行制御が絡むため、慎重な変更が求められます。
