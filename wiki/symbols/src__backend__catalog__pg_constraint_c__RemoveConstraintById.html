RemoveConstraintById

定義

  • 種別: function
  • 定義箇所: src/backend/catalog/pg_constraint.c:911

目的

システムカタログ pg_constraint から指定された OID を持つ制約エントリを削除し、関連するメタデータ（リレーションのチェック制約数など）を整合させる。

シグネチャまたは形状

void RemoveConstraintById(Oid conId)

入力と出力

  • 入力: conId (削除対象の制約の OID)
  • 出力: なし（副作用としてカタログおよびリレーションキャッシュの更新を行う）

前提条件と不変条件

  • pg_constraint カタログに対する RowExclusiveLock が内部で取得される。
  • 対象の制約がリレーションに属する場合、そのリレーションに対して AccessExclusiveLock が取得される。
  • 制約が CHECK 制約である場合、対象リレーションの pg_class.relchecks カウントが正しく維持されている必要がある。

アルゴリズムまたは振る舞い

1. pg_constraint を開き、対象の制約をシステムキャッシュから検索する。 2. 制約がリレーションに属する場合 (conrelid が有効):

  • 対象リレーションを AccessExclusiveLock で開く。
  • 制約種別が CONSTRAINT_CHECK の場合、pg_class の該当行を更新し、relchecks をデクリメントする。

    • 3. 制約がドメインに属する場合 (contypid が有効):

  • 現時点では特別な処理は行わない。

    • 4. CatalogTupleDelete を使用して pg_constraint から対象行を削除する。 5. システムキャッシュを解放し、開いたリレーションを閉じる。

呼び出し元と呼び出し先

  • 呼び出し元:
  • src/backend/catalog/dependency.c:1396
  • 呼び出し先:
  • table_open, table_close
  • SearchSysCache1, SearchSysCacheCopy1, ReleaseSysCache
  • CatalogTupleUpdate, CatalogTupleDelete
  • elog (エラーハンドリング)

関連データ構造

  • pg_constraint: 制約情報を保持するシステムカタログ。
  • pg_class: リレーションのメタデータ（relchecks を含む）。
  • Form_pg_constraint: pg_constraint 行の構造体表現。

失敗モードと境界条件

  • 指定された conId が pg_constraint に存在しない場合、elog(ERROR) によりトランザクションがアボートされる。
  • CHECK 制約削除時に relchecks が 0 である場合、整合性エラーとして elog(ERROR) が発生する。
  • 未知の制約タイプ（リレーションにもドメインにも属さない）が渡された場合、elog(ERROR) が発生する。

保守担当者向けメモ

  • relchecks の更新はバックエンド間でのリレーションキャッシュ再構築を強制する重要な処理であるため、変更時は注意が必要。
