delay_execution_planner

定義

  • 種別: function
  • 定義箇所: src/test/modules/delay_execution/delay_execution.c:41

目的

PostgreSQLのプランニング処理直後に意図的な遅延を挿入し、並行性テストにおけるセッション間の同期を制御する。

シグネチャまたは形状

static PlannedStmt *
delay_execution_planner(Query *parse, const char *query_string,
                        int cursorOptions, ParamListInfo boundParams)

入力と出力

  • **入力**:
  • parse: 解析済みのクエリツリー。
  • query_string: 元のクエリ文字列。
  • cursorOptions: カーソルオプションのビットマスク。
  • boundParams: パラメータ情報。
  • **出力**:
  • PlannedStmt *: プランニングの結果生成された実行計画。

前提条件と不変条件

  • planner_hook に本関数が登録されていること。
  • post_planning_lock_id が0以外の場合にのみ遅延処理が実行される。
  • アドバイザリロックの取得・解放後に AcceptInvalidationMessages() を呼び出し、キャッシュの整合性を維持すること。

アルゴリズムまたは振る舞い

1. prev_planner_hook が設定されていればそれを呼び出し、そうでなければ standard_planner を呼び出してプランニングを実行する。 2. post_planning_lock_id が0でない場合、pg_advisory_lock_int8 でロックを取得し、直後に pg_advisory_unlock_int8 で解放する。これにより、他セッションが同IDでロックを保持している間、実行を待機させる。 3. AcceptInvalidationMessages() を呼び出し、待機中に発生したキャッシュ無効化メッセージを処理する。 4. 生成された PlannedStmt を呼び出し元へ返す。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • planner_hook を介してPostgreSQLのプランナから呼び出される。
  • **呼び出し先**:
  • prev_planner_hook または standard_planner
  • pg_advisory_lock_int8 (DirectFunctionCall1経由)
  • pg_advisory_unlock_int8 (DirectFunctionCall1経由)
  • AcceptInvalidationMessages()

関連データ構造

  • PlannedStmt: プランニング結果を保持する構造体。
  • post_planning_lock_id: 遅延に使用するアドバイザリロックIDを保持するGUC変数。
  • prev_planner_hook: 以前のプランナフックを保持する変数。

失敗モードと境界条件

  • post_planning_lock_id が0の場合、遅延処理はスキップされる。
  • アドバイザリロック関数はキャッシュ無効化メッセージを処理しないため、AcceptInvalidationMessages() の呼び出しを省略するとキャッシュ不整合のリスクがある。

保守担当者向けメモ

  • 本関数はテスト専用であり、本番環境での使用は想定されていない。
  • 他の拡張モジュールとの共存を考慮し、prev_planner_hook を介したフックの連鎖を維持すること。
