SupportRequestIndexCondition

定義

  • **種別**: struct
  • **定義箇所**: src/include/nodes/supportnodes.h:223

目的

関数呼び出しをインデックス検索可能な条件式（Index Condition）に変換可能かどうかを判断し、変換に必要なメタデータを提供するためのリクエスト構造体です。

シグネチャまたは形状

typedef struct SupportRequestIndexCondition
{
    NodeTag     type;
    /* 入力フィールド */
    struct PlannerInfo *root;
    Oid         funcid;
    Node       *node;
    int         indexarg;
    struct IndexOptInfo *index;
    int         indexcol;
    Oid         opfamily;
    Oid         indexcollation;
    /* 出力フィールド */
    bool        lossy;
} SupportRequestIndexCondition;

入力と出力

  • **入力**:
  • root: プランナーのインフラストラクチャ情報。
  • funcid: 問い合わせ対象の関数OID。
  • node: 関数を呼び出しているパースノード。
  • indexarg: インデックス列と一致する関数引数の位置。
  • index: 対象インデックスの情報。
  • indexcol: 対象インデックス列のインデックス（0始まり）。
  • opfamily: インデックス列の演算子族。
  • indexcollation: インデックス列の照合順序。
  • **出力**:
  • lossy: インデックス条件が関数呼び出しと完全に等価であれば false、そうでなければ true を設定します。

前提条件と不変条件

  • サポート関数がリクエストを処理できない場合は、エラーを発生させず NULL を返す必要があります。
  • 構造体自体は一時的なものであり、サポート関数内で永続的なメモリ確保は行いません。

アルゴリズムまたは振る舞い

1. プランナーが特定の関数に対してインデックス利用の最適化を試みる際、本構造体を生成します。 2. 登録されたサポート関数が本構造体を受け取り、インデックス条件への変換が可能か判定します。 3. 変換可能な場合、lossy フラグを適切に設定し、情報を返します。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/optimizer/path/indxpath.c
  • **呼び出し先**:
  • src/backend/utils/adt/like_support.c
  • src/backend/utils/adt/network.c

    • (上記ファイル内のサポート関数が本構造体を処理します)

関連データ構造

  • PlannerInfo: プランナーの全体的な状態を保持。
  • IndexOptInfo: インデックスの最適化情報を保持。
  • Node: 汎用的なノード構造。

失敗モードと境界条件

  • サポート関数が変換をサポートしていない場合、呼び出し元は NULL を受け取り、インデックス条件への変換を断念します。

保守担当者向けメモ

  • 本構造体はプランナーだけでなく実行時にも利用される可能性があるため、プランナー専用のロジックに限定しないよう注意してください。
  • lossy フィールドは、インデックスによるフィルタリングが完全一致か、あるいは追加の再チェックが必要かを示す重要なフラグです。
