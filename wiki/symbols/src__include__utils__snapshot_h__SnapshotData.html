SnapshotData

定義

  • 種別: struct
  • 定義箇所: src/include/utils/snapshot.h:138

目的

特定のトランザクションから見て、どのタプル（行）が可視であるかを判定するための基準となる情報を保持する。MVCC（多版同時実行制御）やリカバリ時の状態管理において、可視性判定の抽象化インターフェースとして機能する。

シグネチャまたは形状

typedef struct SnapshotData {
    SnapshotType snapshot_type;
    TransactionId xmin;
    TransactionId xmax;
    TransactionId *xip;
    uint32 xcnt;
    TransactionId *subxip;
    int32 subxcnt;
    bool suboverflowed;
    bool takenDuringRecovery;
    bool copied;
    CommandId curcid;
    uint32 speculativeToken;
    struct GlobalVisState *vistest;
    uint32 active_count;
    uint32 regd_count;
    pairingheap_node ph_node;
    uint64 snapXactCompletionCount;
} SnapshotData;

入力と出力

  • **入力**: GetSnapshotData() 等により、現在のトランザクション状態やリカバリ状態に基づいて構築される。
  • **出力**: HeapTupleSatisfiesMVCC 等の可視性判定関数が本構造体を参照し、タプルの可視性を判定する。SNAPSHOT_DIRTY 型等の場合は、構造体が進行中のトランザクションIDを返すための出力引数として利用される。

前提条件と不変条件

  • xmin 未満のすべての XID は可視である。
  • xmax 以上のすべての XID は不可視である。
  • xip[] に含まれるすべての ID は xmin <= id < xmax を満たす。
  • subxip[] に含まれるすべての ID は xmin 以上である。
  • snapshot_type が SNAPSHOT_MVCC 以外の場合、一部のフィールド（xmin, xmax 等）は特殊な目的で再利用されるか、ゼロ値となる。

アルゴリズムまたは振る舞い

  • MVCC スナップショットは、xmin と xmax、および進行中のトランザクションリスト（xip, subxip）を用いてタプルの可視性を判定する。
  • snapXactCompletionCount を保持し、前回のスナップショット生成以降にトランザクションの完了がない場合に再計算を回避する最適化を行う。
  • 参照カウント（active_count, regd_count）によりライフサイクルを管理する。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/access/heap/heapam.c, src/backend/executor/execReplication.c, src/backend/replication/logical/reorderbuffer.c 等でインスタンス化やコピーが行われる。
  • **呼び出し先**: lib/pairingheap.h の関数が ph_node を通じて登録済みスナップショットの管理に利用される。

関連データ構造

  • SnapshotType: スナップショットの動作を決定する列挙型。
  • GlobalVisState: SNAPSHOT_NON_VACUUMABLE 等でバキューム可否判定に使用される。
  • pairingheap_node: スナップショットマネージャーによる管理用。

失敗モードと境界条件

  • subxip 配列がオーバーフローした場合、suboverflowed フラグが true となる。
  • リカバリ中に取得されたスナップショットでは xip が空になる場合がある。
  • フィールドの解釈は snapshot_type に依存するため、型を誤認したアクセスは不正な可視性判定を招く。

保守担当者向けメモ

  • 歴史的経緯によりフィールドが多いため、将来的に NodeTag を用いたサブクラス化が検討されている。
  • xmin や xmax は MVCC 以外でも再利用されるため、変更時は HeapTupleSatisfiesDirty 等への影響を考慮すること。
  • メモリ管理は snapmgr.h 経由で行い、active_count や regd_count を直接操作しないこと。
