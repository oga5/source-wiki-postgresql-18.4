hash_xlog_split_allocate_page

定義

  • 種別: function
  • 定義箇所: src/backend/access/hash/hash_xlog.c:310

目的

ハッシュインデックスのバケット分割時に発生する、新しいバケットの割り当ておよびメタページの更新をWALリプレイとして再現する。

シグネチャまたは形状

static void hash_xlog_split_allocate_page(XLogReaderState *record)

入力と出力

  • **入力**: XLogReaderState *record（リプレイ対象のWALレコード）
  • **出力**: なし（副作用としてバッファプール上のページ状態を更新）

前提条件と不変条件

  • WALリプレイ中であるため、他のプロセスによる同時アクセスは発生しない。
  • 冪等性を保持する（既に適用済みの変更に対して再実行しても安全であること）。
  • oldbuf（旧バケット）、newbuf（新バケット）、metabuf（メタページ）の順で処理を行う。

アルゴリズムまたは振る舞い

1. **旧バケットの更新**: XLogReadBufferForRedoExtended で旧バケットを読み込み、xl_hash_split_allocate_page の情報に基づきフラグとリンク先（hasho_prevblkno）を更新する。 2. **新バケットの初期化**: RBM_ZERO_AND_CLEANUP_LOCK を指定して新バケットを確保し、_hash_initbuf で初期化する。 3. **メタページの更新**: metabuf を読み込み、hashm_maxbucket を更新する。

  • XLH_SPLIT_META_UPDATE_MASKS フラグがある場合、hashm_lowmask と hashm_highmask を更新する。
  • XLH_SPLIT_META_UPDATE_SPLITPOINT フラグがある場合、hashm_spares 配列と hashm_ovflpoint を更新する。

    • 4. **終了処理**: 各バッファをダーティとしてマークし、LSNを記録した後、ロックを解放する。

呼び出し元と呼び出し先

  • **呼び出し元**: hash_redo（src/backend/access/hash/hash_xlog.c 内のディスパッチャ）から呼び出される。
  • **呼び出し先**: XLogReadBufferForRedoExtended, XLogReadBufferForRedo, _hash_initbuf, BufferGetPage, HashPageGetOpaque, HashPageGetMeta, PageSetLSN, MarkBufferDirty, UnlockReleaseBuffer 等。

関連データ構造

  • xl_hash_split_allocate_page: WALレコードのデータ構造。
  • HashPageOpaque: バケットページのメタデータ。
  • HashMetaPage: ハッシュインデックスのメタページ構造体。

失敗モードと境界条件

  • XLogReadBufferForRedo 等のバッファ読み込み失敗時は、PostgreSQLの標準的なエラーハンドリング（パニック等）に依存する。
  • ページイメージが復元された場合でも、special space は再適用する必要があるため、明示的に更新処理を行う。

保守担当者向けメモ

  • 本関数は、通常のインデックス操作と異なり、ロックの取得順序や保持期間がリプレイ用に最適化されている。
  • メタページの更新ロジックは、WALレコード内のフラグ（XLH_SPLIT_META_UPDATE_*）に基づいて条件分岐するため、構造変更時にはフラグの整合性に注意すること。
