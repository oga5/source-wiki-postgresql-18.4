PLy_procedure_munge_source

定義

  • 種別: function
  • 定義箇所: src/pl/plpython/plpy_procedure.c:428

目的

SQLで定義された関数本体のソースコードを、Pythonの def 文として実行可能な形式に変換（munge）します。

シグネチャまたは形状

static char *PLy_procedure_munge_source(const char *name, const char *src)

入力と出力

  • **入力**:
  • name: 生成するPython関数の名前。
  • src: SQL関数として定義されたPythonソースコードの文字列。
  • **出力**:
  • def <name>():\n\t... という形式で整形された、動的に割り当てられた文字列（palloc 済み）。

前提条件と不変条件

  • 入力 src はヌル終端文字列である必要があります。
  • 出力バッファは (strlen(src) * 2) + strlen(name) + 16 のサイズで確保され、変換後の文字列を格納するのに十分な容量を持つことが期待されます。

アルゴリズムまたは振る舞い

1. 指定された関数名を用いて def <name>():\n\t というヘッダーを生成します。 2. 入力ソース src を1文字ずつ走査します。 3. 改行コード（\n または \r\n）を検知するたびに、Pythonのインデントを維持するために \n\t に置換します。 4. 最後に2つの改行を追加して文字列を終端します。 5. 処理後のポインタが確保したバッファサイズを超えていないかを確認し、超過している場合は FATAL エラーを発生させます。

呼び出し元と呼び出し先

  • **呼び出し元**: src/pl/plpython/plpy_procedure.c 内の関数（PLy_procedure_compile 等から呼び出されることが確認されています）。
  • **呼び出し先**: PostgreSQLのメモリ管理関数 palloc およびエラー報告関数 elog。

関連データ構造

  • PLyProcedure: 本関数で生成されたソースコードを保持・コンパイルする対象となる構造体。

失敗モードと境界条件

  • **バッファオーバーラン**: 変換後の文字列が確保したメモリ領域を超過した場合、elog(FATAL, ...) によりプロセスが終了します。
  • **メモリ不足**: palloc が失敗した場合、PostgreSQLの標準的なメモリ管理機構によりエラーが送出されます。

保守担当者向けメモ

  • 本関数は単純な文字列置換によるインデント調整を行っています。複雑なPythonコードや特殊なエスケープシーケンスを含むソースコードを扱う場合、意図しない構文エラーが発生する可能性があるため、デバッグ時には生成された文字列を確認してください。
