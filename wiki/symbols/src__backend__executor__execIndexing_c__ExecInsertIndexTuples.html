ExecInsertIndexTuples

定義

  • 種別: function
  • 定義箇所: src/backend/executor/execIndexing.c:309

目的

テーブルへのタプル挿入に伴い、関連するすべてのインデックスを更新し、一意性制約および排他制約の整合性を強制します。

シグネチャまたは形状

List *ExecInsertIndexTuples(ResultRelInfo *resultRelInfo,
                            TupleTableSlot *slot,
                            EState *estate,
                            bool update,
                            bool noDupErr,
                            bool *specConflict,
                            List *arbiterIndexes,
                            bool onlySummarizing)

入力と出力

  • **入力**:
  • resultRelInfo: 対象リレーションのインデックス情報を含む構造体。
  • slot: 挿入対象のタプル。
  • estate: 実行状態。
  • update: 更新操作かどうかを示すフラグ。
  • noDupErr: 一意性違反時にエラーとせず、競合を報告するかどうか。
  • arbiterIndexes: 競合チェック対象のインデックスリスト。
  • onlySummarizing: サマリーインデックスのみを更新するかどうか。
  • **出力**:
  • List *: 遅延制約チェックが必要なインデックスの OID リスト。

前提条件と不変条件

  • slot->tts_tid が有効な ItemPointer であること。
  • slot->tts_tableOid が resultRelInfo のリレーション OID と一致すること。

アルゴリズムまたは振る舞い

1. **初期化**: EState の ExprContext を設定し、対象タプルをスキャン対象としてセットします。 2. **インデックス走査**: resultRelInfo に保持された全インデックスを順次処理します。

  • ii_ReadyForInserts が偽のインデックスはスキップします。
  • 部分インデックスの場合、ii_Predicate を評価し、不適合ならスキップします。

    • 3. **データ生成**: FormIndexDatum を呼び出し、インデックスキー値を生成します。 4. **インデックス挿入**: index_insert を呼び出し、インデックス AM による挿入と一意性チェックを実行します。

  • noDupErr が真の場合、UNIQUE_CHECK_PARTIAL を使用して遅延チェックを行います。

    • 5. **制約チェック**: 排他制約 (ii_ExclusionOps) が存在する場合、挿入後に競合チェックを行います。投機的挿入時は CEOUC_LIVELOCK_PREVENTING_WAIT を使用してデッドロックを回避します。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/executor/nodeModifyTable.c、src/backend/commands/copyfrom.c、src/backend/executor/execReplication.c などから呼び出されます。
  • **呼び出し先**: FormIndexDatum, index_insert, ExecQual, ExecPrepareQual 等。

関連データ構造

  • ResultRelInfo: インデックス情報 (ri_IndexRelationInfo) を保持。
  • IndexInfo: インデックスの属性や制約情報を保持。
  • TupleTableSlot: 挿入対象のタプルを保持。

失敗モードと境界条件

  • 一意性制約違反時: noDupErr が偽であればエラーを発生させます。
  • 排他制約違反時: 競合検出時に待機モードに応じた処理（エラー、待機、またはリチェックキューへの追加）が行われます。

保守担当者向けメモ

  • 本関数は INSERT ... ON CONFLICT の投機的挿入をサポートしており、競合時の挙動は noDupErr および arbiterIndexes によって制御されます。
  • index_unchanged_by_update を用いて、UPDATE時にキーが変更されていないインデックスの更新をスキップする最適化が行われています。
