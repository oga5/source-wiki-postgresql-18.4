TIDBitmap

定義

  • 種別: struct
  • 定義箇所: src/include/nodes/tidbitmap.h:41

目的

PostgreSQL における TID（Tuple Identifier）の集合をメモリ効率よく管理するための不透明なデータ構造です。インデックススキャンやビットマップヒープスキャンにおいて、対象タプルの位置情報を保持するために使用されます。

シグネチャまたは形状

typedef struct TIDBitmap TIDBitmap;

構造体の内部メンバは隠蔽されており、直接アクセスは推奨されません。

入力と出力

本構造体は API を通じて操作されます。

  • **入力**: tbm_add_tuples 等の関数を介して ItemPointer（ブロック番号とオフセット番号）を受け取ります。
  • **出力**: tbm_iterate を通じて TBMIterateResult（ブロック番号、損失フラグ、再チェック要否）を返します。

前提条件と不変条件

  • **メモリ制限**: 生成時に指定されたメモリ制限内で動作します。
  • **損失付き格納（Lossy Storage）**: メモリ制限を超過した場合、個別のオフセット情報を破棄し、ページ単位の管理へ自動的に切り替わります。
  • **再検証**: 損失付きモードで格納されたページについては、イテレータから返される recheck フラグが真となるため、ヒープ上のタプルを再検証する必要があります。

アルゴリズムまたは振る舞い

1. **初期化**: tbm_create により生成。 2. **構築**: tbm_add_tuples や tbm_add_page で TID を追加。 3. **走査**: tbm_begin_iterate でイテレータを初期化し、tbm_iterate で結果を取得。 4. **終了**: tbm_end_iterate でリソースを解放。

関連データ構造

  • TBMIterateResult: イテレータが返す結果構造体。
  • TBMIterator: プライベートおよび共有イテレータをラップする構造体。
  • ItemPointer: TID を構成するブロック番号とオフセット番号の型。

失敗モードと境界条件

  • **メモリ不足**: メモリ制限に達すると損失付きモードへ移行し、詳細なオフセット情報が失われます。
  • **共有メモリ**: パラレルクエリで使用する場合、dsa_area を介したライフサイクル管理が不適切な場合にエラーが発生する可能性があります。

保守担当者向けメモ

  • 内部構造は tidbitmap.c で定義されているため、変更時は同ファイルを併せて参照してください。
  • イテレータから返される recheck フラグの扱いは、ヒープスキャン時の正確性に直結するため注意が必要です。
  • パラレルクエリ対応時は、共有イテレータのライフサイクル管理に留意してください。
