heap_fetch_toast_slice

定義

  • 種別: function
  • 定義箇所: src/backend/access/heap/heaptoast.c:625

目的

TOASTテーブルから、指定された valueid に対応するデータの特定範囲（スライス）を読み出し、メモリ上の result バッファにコピーする。

シグネチャまたは形状

void heap_fetch_toast_slice(Relation toastrel, Oid valueid, int32 attrsize,
                            int32 sliceoffset, int32 slicelength,
                            struct varlena *result)

入力と出力

  • **入力**:
  • toastrel: 対象のTOASTリレーション。
  • valueid: 取得対象データのOID。
  • attrsize: データの総サイズ。
  • sliceoffset: 読み出し開始位置。
  • slicelength: 読み出す長さ。
  • result: データを格納する varlena 構造体（呼び出し元で十分な領域が確保されている必要がある）。
  • **出力**: なし（result にデータが書き込まれる）。

前提条件と不変条件

  • toastrel は有効なTOASTリレーションであること。
  • endchunk <= totalchunks であること。
  • 取得される各チャンクは、TOASTテーブルのインデックス順（valueid, chunkidx）に従って順次処理される。
  • 取得したチャンクデータは VARATT_IS_EXTENDED でない（インライン化されている）必要がある。

アルゴリズムまたは振る舞い

1. toast_open_indexes を呼び出し、TOASTテーブルのインデックスを取得する。 2. sliceoffset と slicelength から、読み出すべきチャンクの範囲（startchunk から endchunk）を計算する。 3. systable_beginscan_ordered を用いて、指定された valueid とチャンク範囲を条件とするインデックススキャンを開始する。 4. systable_getnext_ordered でチャンクを順次取得し、以下の検証を行う:

  • チャンク番号が期待値と一致するか。
  • チャンクサイズが期待値と一致するか。

    • 5. 取得したチャンクデータを memcpy で result の適切なオフセット位置へコピーする。 6. 全てのチャンクが取得できたことを確認し、systable_endscan_ordered および toast_close_indexes でリソースを解放する。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/access/heap/heapam_handler.c (テーブルアクセスメソッドのインターフェースとして参照)
  • **呼び出し先**:
  • toast_open_indexes / toast_close_indexes
  • systable_beginscan_ordered / systable_getnext_ordered / systable_endscan_ordered
  • get_toast_snapshot
  • fastgetattr
  • elog / ereport (エラーハンドリング)

関連データ構造

  • Relation: TOASTリレーションのメタデータ。
  • ScanKeyData: インデックススキャンの条件設定に使用。
  • HeapTuple: TOASTテーブルから取得されるタプル。
  • varlena: 可変長データの格納構造体。

失敗モードと境界条件

  • **データ破損**: チャンク番号の不一致、範囲外のチャンク、または期待されるサイズと実際のチャンクサイズが異なる場合に ERRCODE_DATA_CORRUPTED を発生させる。
  • **欠損**: 期待される全てのチャンクが取得できなかった場合、エラーを発生させる。
  • **不正なチャンク**: チャンクデータが VARATT_IS_EXTENDED である場合、エラーを発生させる。

保守担当者向けメモ

  • 本関数はTOASTテーブルの物理的なチャンク構造に強く依存している。TOASTのチャンクサイズ（TOAST_MAX_CHUNK_SIZE）や格納形式に変更がある場合は注意が必要。
  • result バッファのメモリ管理は呼び出し元に委ねられているため、バッファオーバーランを防ぐために slicelength の妥当性検証が呼び出し元で適切に行われていることを前提としている。
