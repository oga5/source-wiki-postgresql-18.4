PLy_exec_trigger

定義

  • 種別: function
  • 定義箇所: src/pl/plpython/plpy_exec.c:318

目的

PostgreSQL のトリガー呼び出しを Python 関数へ橋渡しし、トリガーデータの構築、実行、および戻り値（OK, SKIP, MODIFY）の処理を行うメインエントリーポイント。

シグネチャまたは形状

HeapTuple PLy_exec_trigger(FunctionCallInfo fcinfo, PLyProcedure *proc)

入力と出力

  • **入力**:
  • fcinfo: PostgreSQL の関数呼び出し情報。TriggerData を保持。
  • proc: 実行対象の PL/Python プロシージャ情報。
  • **出力**:
  • HeapTuple: トリガー処理の結果として変更されたタプル、あるいは NULL（SKIP の場合）。

前提条件と不変条件

  • fcinfo がトリガーとして呼び出されていること（Assert(CALLED_AS_TRIGGER(fcinfo))）。
  • リレーションのタプル記述子（tupdesc）が変更された場合、実行のたびに PLy_input/output_setup_func および setup_tuple が再実行され、型変換情報が更新される。
  • 再帰呼び出し時には PLy_global_args_push が呼び出され、終了時に PLy_global_args_pop が保証されること。

アルゴリズムまたは振る舞い

1. **型変換の準備**: rel_descr を取得し、必要に応じて入出力変換関数をセットアップする。 2. **スタック管理**: 再帰呼び出しに備え、現在の引数状態を PLy_global_args_push で退避する。 3. **実行ブロック (PG_TRY)**:

  • SPI_register_trigger_data で SPI にトリガーデータを登録。
  • PLy_trigger_build_args で Python 用の引数を構築。
  • PLy_procedure_call で Python コードを実行。
  • SPI_finish で SPI 接続を終了。
  • 戻り値の解析: None または "OK" は正常終了、"SKIP" は NULL を返却、"MODIFY" は PLy_modify_tuple を呼び出してタプルを更新。

    • 4. **クリーンアップ (PG_FINALLY)**: PLy_global_args_pop を実行し、Python オブジェクトの参照カウントを解放する。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/pl/plpython/plpy_main.c (直接の呼び出し)
  • **呼び出し先**:
  • PLy_output_setup_func, PLy_input_setup_func, PLy_output_setup_tuple, PLy_input_setup_tuple
  • PLy_global_args_push, PLy_global_args_pop
  • SPI_register_trigger_data, SPI_finish
  • PLy_trigger_build_args, PLy_procedure_call, PLy_modify_tuple

関連データ構造

  • FunctionCallInfo: PostgreSQL の関数呼び出し情報。
  • PLyProcedure: PL/Python プロシージャの定義とキャッシュ。
  • TriggerData: トリガー実行時のコンテキスト情報。
  • HeapTuple: PostgreSQL のタプル構造。

失敗モードと境界条件

  • **型不一致**: Python からの戻り値が期待される文字列（"OK", "SKIP", "MODIFY"）や None 以外の場合、ERRCODE_DATA_EXCEPTION でエラーを報告。
  • **SPI 失敗**: SPI_finish が失敗した場合、elog(ERROR) を発生させる。
  • **MODIFY の誤用**: DELETE トリガーで "MODIFY" が返された場合、警告を発行して無視する。

保守担当者向けメモ

  • PG_TRY ブロック内での Py_XDECREF 漏れはメモリリークを引き起こすため、PG_FINALLY での解放処理を厳守すること。
  • トリガーのタプル記述子が変更される可能性があるため、型変換のセットアップはキャッシュの有無を確認しつつ毎回実行される設計となっている。
