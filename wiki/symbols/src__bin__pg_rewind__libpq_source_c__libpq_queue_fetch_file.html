libpq_queue_fetch_file

定義

  • **種別**: 関数
  • **定義箇所**: src/bin/pg_rewind/libpq_source.c:324

目的

リモートサーバーからファイル全体を取得するためのリクエストをキューに追加し、ローカルのターゲットファイルを即座に切り詰める（truncate）ことで、同期準備を行う。

シグネチャまたは形状

static void libpq_queue_fetch_file(rewind_source *source, const char *path, size_t len)

入力と出力

  • **入力**:
  • source: rewind_source 構造体へのポインタ。
  • path: 取得対象ファイルのパス。
  • len: 取得対象ファイルのサイズ。
  • **出力**: なし（副作用としてファイル操作とキューイングを行う）。

前提条件と不変条件

  • open_target_file(path, true) が呼び出され、ローカルのターゲットファイルが書き込み用に開かれ、切り詰められる。
  • ファイルサイズが MAX_CHUNK_SIZE より小さい場合でも、少なくとも MAX_CHUNK_SIZE 分の範囲をリクエストする。これにより、ソース側でファイルが拡大していた場合に備える。

アルゴリズムまたは振る舞い

1. open_target_file を呼び出し、ローカルのターゲットファイルを切り詰める。 2. Max(len, MAX_CHUNK_SIZE) を計算し、ファイル全体（または最小チャンクサイズ分）を読み込むための範囲リクエストを libpq_queue_fetch_range に渡す。 3. この処理は、設定ファイルなどの非WALログファイルに対して有効である。

呼び出し元と呼び出し先

  • **呼び出し元**: ソースコード内の参照から、rewind_source インターフェースの関数ポインタとして設定されていることが確認できる（src/bin/pg_rewind/libpq_source.c:91）。具体的な実行時の呼び出し元はソースコードの文脈からは特定できない。
  • **呼び出し先**:
  • open_target_file
  • libpq_queue_fetch_range

関連データ構造

  • rewind_source: 通信レイヤーのインターフェースを保持する構造体。

失敗モードと境界条件

  • **競合状態**: ファイルコピー中にソース側でファイルが変更された場合、古いバージョンと新しいバージョンが混在した「引き裂かれた（torn）」ファイルがコピーされる可能性がある。これは pg_basebackup と同様の既知の挙動である。
  • **ファイルサイズ**: MAX_CHUNK_SIZE を超えるファイルについては、指定された len までが取得対象となる。

保守担当者向けメモ

  • 本関数は rewind_source インターフェースの一部として実装されている。
  • ファイルの切り詰めをリクエスト受信時ではなく、キューイング時に行う設計となっている。これは、どのリクエストがファイル全体を対象としているかを追跡する複雑さを避けるためである。
