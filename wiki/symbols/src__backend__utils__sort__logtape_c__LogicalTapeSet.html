LogicalTapeSet

定義

  • **種別**: struct
  • **定義箇所**: src/backend/utils/sort/logtape.c:187

目的

単一の物理ファイル（BufFile）上で複数の論理テープを管理し、外部ソートにおけるディスク使用量を最小化するためのコンテナ構造体です。読み込み済みブロックの再利用や、並列ソート時のワーカー間での領域管理を担います。

シグネチャまたは形状

struct LogicalTapeSet {
    BufFile    *pfile;
    SharedFileSet *fileset;
    int         worker;
    int64       nBlocksAllocated;
    int64       nBlocksWritten;
    int64       nHoleBlocks;
    bool        forgetFreeSpace;
    int64      *freeBlocks;
    int64       nFreeBlocks;
    Size        freeBlocksLen;
    bool        enable_prealloc;
};

入力と出力

  • **入力**: LogicalTapeSetCreate 等により初期化され、LogicalTapeWrite や LogicalTapeRead の操作対象として渡されます。
  • **出力**: 物理ファイルへの書き込み、またはフリーリスト（freeBlocks）によるブロックの再利用状態の更新。

前提条件と不変条件

  • nBlocksAllocated >= nBlocksWritten が常に成立します。
  • freeBlocks は最小ヒープとして管理され、再利用可能なブロック番号を保持します。
  • forgetFreeSpace が真の場合、解放されたブロックはフリーリストに記録されず、再利用されません。

アルゴリズムまたは振る舞い

1. **ブロック割り当て**: ltsGetBlock を通じてブロックを取得します。フリーリストに空きがある場合はそこから取得し、ない場合は物理ファイルを拡張します。 2. **ブロック解放**: ltsReleaseBlock により、読み込み済みのブロックをフリーリストへ戻し、将来の書き込みで再利用可能にします。 3. **ファイル管理**: nBlocksWritten は物理ファイルの現在のサイズを追跡し、nHoleBlocks は並列ソート時のファイル連結に伴う未使用領域を管理します。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/executor/nodeAgg.c（hashagg_spill_init 等）から参照されています。
  • **呼び出し先**: ltsCreateTape, ltsWriteBlock, ltsReadBlock, ltsGetBlock, ltsGetFreeBlock, ltsGetPreallocBlock, ltsReleaseBlock などの内部関数が本構造体を操作します。

関連データ構造

  • BufFile: 物理的なI/Oを抽象化する基盤。
  • LogicalTape: 本セット内で管理される個別の論理テープ状態。
  • TapeBlockTrailer: 各ブロックの末尾に付与されるメタデータ。

失敗モードと境界条件

  • **メモリ不足**: freeBlocks 配列の拡張時に発生する可能性があります。
  • **ディスクフル**: 物理ファイルの拡張時に BufFile 経由で発生します。
  • **論理エラー**: LogicalTapeFreeze 呼び出し後にブロックを解放しようとした場合、再利用ロジックが停止するため注意が必要です。

保守担当者向けメモ

  • freeBlocks は動的配列であり、freeBlocksLen で現在の容量を管理しています。
  • 並列ソート環境下では worker ID を用いて共有ファイルセット内の領域を識別します。
  • ブロックの再利用は「一度読み込んだデータは二度と必要としない」というソートの特性に依存しています。この前提を崩す操作（LogicalTapeFreeze 等）を行う際は、再利用ロジックへの影響を考慮してください。
