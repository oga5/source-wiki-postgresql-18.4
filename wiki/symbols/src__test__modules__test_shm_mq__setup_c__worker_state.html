worker_state

定義

  • 種別: struct
  • 定義箇所: src/test/modules/test_shm_mq/setup.c:25

目的

テスト環境で起動された複数のバックグラウンドワーカーのハンドルを管理し、それらの生存期間と状態を追跡するためのコンテナ構造体です。

シグネチャまたは形状

typedef struct
{
	int			nworkers;
	BackgroundWorkerHandle *handle[FLEXIBLE_ARRAY_MEMBER];
} worker_state;

入力と出力

  • 入力: nworkers（ワーカー数）に基づき、可変長配列 handle を含むメモリ領域を確保して使用します。
  • 出力: なし（構造体としてのデータ保持が主目的）。

前提条件と不変条件

  • handle 配列は FLEXIBLE_ARRAY_MEMBER を使用しているため、インスタンス化の際は offsetof(worker_state, handle) を用いた適切なメモリ確保が必要です。
  • ワーカーの生存期間を維持するため、通常 TopTransactionContext に割り当てられます。

アルゴリズムまたは振る舞い

  • setup_background_workers 関数内で初期化され、登録された各バックグラウンドワーカーのハンドルを格納します。
  • 格納されたハンドルは、ワーカーの起動待機（wait_for_workers_to_become_ready）や、ワーカーの異常終了監視（check_worker_status）に使用されます。

呼び出し元と呼び出し先

  • 呼び出し元: setup_background_workers、wait_for_workers_to_become_ready、check_worker_status、および cleanup_background_workers（on_dsm_detach フック経由）から参照されます。
  • 呼び出し先: 本構造体自体はデータ保持用であり、関数を呼び出すことはありません。

関連データ構造

  • BackgroundWorkerHandle: PostgreSQLのバックグラウンドワーカーを制御するためのハンドル。
  • test_shm_mq_header: 共有メモリ上の制御構造体。

失敗モードと境界条件

  • メモリ確保不足: nworkers に対して十分なメモリを確保せずに構造体を初期化した場合、handle 配列へのアクセスでバッファオーバーランが発生します。
  • ライフサイクル不一致: TopTransactionContext 以外で確保された場合、トランザクション終了時にハンドルが無効化され、クリーンアップ処理が失敗する可能性があります。

保守担当者向けメモ

  • 本構造体は test_shm_mq モジュールのテスト環境構築において、ワーカーのライフサイクル管理の要となります。
  • FLEXIBLE_ARRAY_MEMBER を使用しているため、構造体のサイズ計算には必ず offsetof マクロを使用してください。
