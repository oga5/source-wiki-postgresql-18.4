ProcArrayStruct

定義

  • 種別: struct
  • 定義箇所: src/backend/storage/ipc/procarray.c:71

目的

PostgreSQLの共有メモリ上で、全アクティブなバックエンドプロセス（PGPROC）の配列および、ホットスタンバイ時に必要なトランザクション追跡情報（KnownAssignedXids）を一元管理するためのメイン構造体です。

シグネチャまたは形状

typedef struct ProcArrayStruct {
    int         numProcs;
    int         maxProcs;
    int         maxKnownAssignedXids;
    int         numKnownAssignedXids;
    int         tailKnownAssignedXids;
    int         headKnownAssignedXids;
    TransactionId lastOverflowedXid;
    TransactionId replication_slot_xmin;
    TransactionId replication_slot_catalog_xmin;
    int         pgprocnos[FLEXIBLE_ARRAY_MEMBER];
} ProcArrayStruct;

入力と出力

本構造体は共有メモリ上に配置され、ProcArrayLock を介してアクセスされます。

  • **入力**: トランザクションの開始・終了、WALからのリカバリ情報、レプリケーションスロットの更新により状態が変更されます。
  • **出力**: GetSnapshotData 等の関数を通じて、MVCC判定に必要な実行中トランザクションのリストや境界値を提供します。

前提条件と不変条件

  • pgprocnos は PROCARRAY_MAXPROCS エントリを保持する可変長配列として扱われます。
  • lastOverflowedXid の更新には ProcArrayLock の排他ロックが必要であり、読み取りには共有ロックが必要です。
  • 共有メモリ上の状態変更時は、マルチプロセッサ環境での整合性を保証するためメモリバリア（pg_read_barrier / pg_write_barrier）が適切に使用される必要があります。

アルゴリズムまたは振る舞い

  • **プロセス管理**: numProcs により有効なプロセス数を追跡します。
  • **KnownAssignedXids**: ホットスタンバイ時に、プライマリから受信したトランザクション情報を循環バッファとして管理します。head と tail のインデックスを用いて、オーバーフローを防ぎつつ追跡を行います。
  • **レプリケーション管理**: レプリケーションスロットごとの xmin および catalog_xmin を保持し、不要な行の削除を抑制します。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/storage/ipc/procarray.c 内の各関数（ProcArrayAdd, ProcArrayRemove, GetSnapshotData 等）が、静的変数 procArray を通じて本構造体を操作します。
  • **呼び出し先**: 本構造体自体はデータ保持用であり、直接関数を呼び出すことはありません。

関連データ構造

  • PGPROC: pgprocnos 配列が指し示す各バックエンドのプロセス状態。
  • ProcArrayLock: 本構造体へのアクセスを制御する LWLock。
  • GlobalVisState: 可視性判定のための状態構造体。

失敗モードと境界条件

  • maxProcs を超えるプロセス登録はできません。
  • KnownAssignedXids のバッファが溢れる場合、lastOverflowedXid を用いて追跡情報を退避させる必要があります。

保守担当者向けメモ

  • ProcArrayLock は競合が激しいため、パフォーマンス最適化（グループ化処理など）が随時行われています。変更を加える際はロックの粒度に注意してください。
  • ホットスタンバイ時のロジックは通常のバックエンド管理と異なるため、standby.c および WAL 処理との整合性を確認してください。
