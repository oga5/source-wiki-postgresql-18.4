amcheck_lock_relation_and_check

定義

  • 種別: function
  • 定義箇所: contrib/amcheck/verify_common.c:59

目的

インデックスの整合性検証を行う際、対象リレーションの安全なロック取得、実行権限の昇格、GUC（設定パラメータ）の保護、および検証後のクリーンアップを一元管理するメインワークホース関数です。

シグネチャまたは形状

void amcheck_lock_relation_and_check(Oid indrelid,
                                     Oid am_id,
                                     IndexDoCheckCallback check,
                                     LOCKMODE lockmode,
                                     void *state)

入力と出力

  • indrelid: 検証対象インデックスの OID。
  • am_id: 検証対象のアクセス手法（AM）の OID。
  • check: 実際の検証ロジックを実行するコールバック関数。
  • lockmode: ロックの強さ（例: ShareLock）。
  • state: コールバック関数に渡される状態データ。
  • 出力: なし（検証失敗時はエラーをスロー）。

前提条件と不変条件

  • デッドロックを回避するため、インデックスよりも先に親テーブルのロックを取得する必要があります。
  • 検証関数はテーブル所有者の権限で実行される必要があります。
  • ホットスタンバイモードにおいて、parentcheck が有効な場合はエラーを発生させます。

アルゴリズムまたは振る舞い

1. **ロック取得**: IndexGetRelation で親テーブルを特定し、デッドロックを防ぐためにテーブル→インデックスの順でロックを取得します。 2. **コンテキスト設定**: SetUserIdAndSecContext によりテーブル所有者の権限へ昇格し、NewGUCNestLevel で設定変更を追跡可能にします。 3. **検証実行**: index_checkable で対象が検証可能か判定し、問題なければコールバック関数 check を呼び出します。 4. **後処理**: AtEOXact_GUC で GUC の変更をロールバックし、ユーザーIDとセキュリティコンテキストを復元します。 5. **解放**: index_close および table_close でロックを解放します。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • contrib/amcheck/verify_gin.c
  • contrib/amcheck/verify_nbtree.c
  • **呼び出し先**:
  • IndexGetRelation, table_open, table_close, index_open, index_close
  • GetUserIdAndSecContext, SetUserIdAndSecContext
  • NewGUCNestLevel, AtEOXact_GUC
  • index_checkable
  • 引数として渡された check コールバック関数

関連データ構造

  • Relation: PostgreSQL のリレーション構造体。
  • IndexDoCheckCallback: 検証処理を定義する関数ポインタ型（verify_common.h で定義）。

失敗モードと境界条件

  • indrelid がインデックスではない場合、または親テーブルの特定に失敗した場合はエラーとなります。
  • インデックスのドロップや再作成との競合が発生し、親テーブルの特定が不整合となった場合は ERRCODE_UNDEFINED_TABLE でエラーを報告します。

保守担当者向けメモ

  • 本関数はスーパーユーザー権限での実行を前提としています。
  • ロックは検証終了後に即時解放されますが、これは呼び出されるルーチンが共有キャッシュの無効化をトリガーしないため安全です。
