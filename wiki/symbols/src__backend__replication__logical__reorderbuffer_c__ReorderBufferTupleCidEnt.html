ReorderBufferTupleCidEnt

定義

  • **種別**: struct
  • **定義箇所**: src/backend/replication/logical/reorderbuffer.c:142

目的

論理デコーディングにおいて、特定のタプルに対するコマンドID（CID）の範囲を追跡し、カタログ変更の整合性を維持するために使用されるハッシュテーブルのエントリ構造体です。

シグネチャまたは形状

typedef struct ReorderBufferTupleCidEnt
{
    ReorderBufferTupleCidKey key;
    CommandId    cmin;
    CommandId    cmax;
    CommandId    combocid; /* デバッグ用 */
} ReorderBufferTupleCidEnt;

入力と出力

  • **入力**: ReorderBufferTupleCidKey をキーとしてハッシュテーブルに格納されます。
  • **出力**: 保持している cmin および cmax を通じて、トランザクション内の特定のタプルが可視であったコマンド範囲を特定します。

前提条件と不変条件

  • ReorderBufferTupleCidKey をキーとするハッシュテーブルの一部として管理されます。
  • combocid フィールドはデバッグ目的でのみ使用されます。

アルゴリズムまたは振る舞い

  • 論理デコーディング中に発生するカタログ変更を追跡する際、タプルごとの CID 範囲を記録・更新するために利用されます。
  • ハッシュテーブルのサイズ設定（hash_ctl.entrysize）において、この構造体のサイズが直接参照されています。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/replication/logical/reorderbuffer.c 内のハッシュテーブル操作関数（hash_seq_search 等）から参照されています。
  • **呼び出し先**: 具体的な関数呼び出し関係はソースコードの断片からは特定できません。

関連データ構造

  • ReorderBufferTupleCidKey: 本構造体のキーとして使用されます。
  • CommandId: PostgreSQL のトランザクション内でのコマンド順序を識別する型です。

失敗モードと境界条件

  • ハッシュテーブルへの挿入や検索時に、キーが一致しない場合やメモリ不足が発生した場合の挙動は、ハッシュテーブルの実装（dynahash）に依存します。

保守担当者向けメモ

  • 本構造体は reorderbuffer.c 内のハッシュテーブル管理において中心的な役割を果たします。変更を加える際は、キー構造体 ReorderBufferTupleCidKey との整合性に注意してください。
