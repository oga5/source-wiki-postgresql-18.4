RecoveryLockEntry

定義

  • 種別: struct
  • 定義箇所: src/backend/storage/ipc/standby.c:53

目的

ホットスタンバイ環境において、プライマリサーバーで発生した排他ロック（AccessExclusiveLock）をスタンバイ側で追跡・再現するためのデータ構造です。

シグネチャまたは形状

typedef struct RecoveryLockEntry
{
    xl_standby_lock key;            /* ハッシュキー: xid, dbOid, relOid */
    struct RecoveryLockEntry *next; /* チェーン用リンク */
} RecoveryLockEntry;

入力と出力

  • **入力**: xl_standby_lock 型のキー（トランザクションID、データベースOID、リレーションOIDの組み合わせ）。
  • **出力**: なし（構造体としてメモリ上に保持される）。

前提条件と不変条件

  • RecoveryLockHash ハッシュテーブルの要素として使用されます。
  • 各エントリは、プライマリから受信した XLOG_STANDBY_LOCK レコードに対応します。

アルゴリズムまたは振る舞い

  • RecoveryLockHash 内でハッシュキーとして xl_standby_lock を使用し、ロック状態を管理します。
  • next ポインタにより、ハッシュ衝突時や同一キーに関連する複数のロック情報をチェーン構造で保持します。

呼び出し元と呼び出し先

  • **呼び出し元**: standby_redo() 内の XLOG_STANDBY_LOCK 処理ロジックにおいて、ロックの登録や検索時に使用されます。
  • **呼び出し先**: 特定の関数呼び出しはソース内に明示されていませんが、ハッシュテーブル操作関数（hash_search 等）によって管理されます。

関連データ構造

  • xl_standby_lock: ロックを識別するためのキー構造体。
  • RecoveryLockHash: 本構造体を格納するハッシュテーブル。
  • RecoveryLockXidEntry: トランザクション単位でロックを管理するための関連構造体。

失敗モードと境界条件

  • ハッシュテーブルのメモリ不足時に hash_search が失敗する可能性があります。
  • 構造体サイズは sizeof(RecoveryLockEntry) としてハッシュテーブル初期化時に固定されます。

保守担当者向けメモ

  • 本構造体はスタンバイ側でのロック模倣の基盤です。next ポインタによるチェーン構造を操作する際は、メモリリークやポインタの不整合に注意してください。
  • 呼び出し元・呼び出し先の詳細なコールグラフは、現在のソース抜粋からは限定的であるため、必要に応じて standby_redo 周辺のコードパスを追跡してください。
