pgssEntry

定義

  • 種別: struct
  • 定義箇所: contrib/pg_stat_statements/pg_stat_statements.c:231

目的

PostgreSQLのクエリ統計を保持するハッシュテーブルの各要素（エントリ）を定義する構造体です。個々のSQL文に関するリソース使用量とメタデータを管理します。

シグネチャまたは形状

typedef struct pgssEntry
{
	pgssHashKey key;			/* ハッシュキー（構造体の先頭である必要がある） */
	Counters	counters;		/* クエリの統計情報 */
	Size		query_offset;	/* 外部ファイル内のクエリテキストのオフセット */
	int			query_len;		/* クエリ文字列の有効長、または -1 */
	int			encoding;		/* クエリテキストのエンコーディング */
	TimestampTz stats_since;	/* エントリ割り当て時のタイムスタンプ */
	TimestampTz minmax_stats_since; /* 最小/最大値が最後にリセットされたタイムスタンプ */
	slock_t		mutex;			/* カウンタのみを保護するスピンロック */
} pgssEntry;

入力と出力

  • 入力: pgss_store 等の関数を通じて、クエリID（key）および実行統計（counters）が更新されます。
  • 出力: 統計情報の永続化時やビューの参照時に、構造体内の各フィールドが読み取られます。

前提条件と不変条件

  • key フィールドは構造体の先頭に配置される必要があります。
  • mutex は Counters の更新時にのみ使用されます。ハッシュテーブル自体の構造変更には、別途 pgssSharedState が管理する LWLock が必要です。

アルゴリズムまたは振る舞い

  • クエリ実行時に pgss_store がハッシュテーブルを検索し、存在しない場合は entry_alloc によって新規作成されます。
  • クエリ文字列自体は共有メモリの節約のため外部ファイルに保存され、本構造体には query_offset として位置情報のみが保持されます。
  • mutex を用いて、複数のバックエンドプロセスによる同時更新から Counters を保護します。

呼び出し元と呼び出し先

  • 呼び出し元: entry_alloc（新規作成）、pgss_store（検索・更新）、および統計のダンプ/読み込み処理（fread/fwrite）から参照されます。
  • 呼び出し先: 直接的な関数呼び出しはありませんが、Counters 構造体や pgssHashKey を内包しています。

関連データ構造

  • pgssHashKey: ハッシュテーブルの検索キー。
  • Counters: 実行時間、バッファ使用量、WAL使用量などの統計データ。
  • pgssSharedState: 本構造体を格納するハッシュテーブルを含む共有メモリ状態。

失敗モードと境界条件

  • query_len が -1 の場合、クエリテキストが正しく取得できていないか、無効であることを示します。
  • 共有メモリ上のハッシュテーブルが満杯の場合、entry_alloc による新規作成が失敗する可能性があります。

保守担当者向けメモ

  • mutex（スピンロック）とハッシュテーブルの LWLock を併用する際は、デッドロックを避けるため必ず LWLock を先に取得する階層構造を守ってください。
  • クエリテキストのガベージコレクション（GC）実行時、query_offset の整合性を維持するためにファイル操作とメモリ上のオフセット管理を厳密に同期させる必要があります。
