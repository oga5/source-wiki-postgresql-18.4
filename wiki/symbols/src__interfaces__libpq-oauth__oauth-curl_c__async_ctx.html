async_ctx

定義

  • **種別**: struct
  • **定義箇所**: src/interfaces/libpq-oauth/oauth-curl.c:228

目的

OAuth 2.0 Device Authorization Grant フローにおける非同期処理の状態を保持し、複数回の呼び出しにわたって認証プロセスを継続させるためのコンテキスト構造体です。

シグネチャまたは形状

struct async_ctx {
    enum OAuthStep step;
    int timerfd;
    pgsocket mux;
    CURLM *curlm;
    CURL *curl;
    struct curl_slist *headers;
    PQExpBufferData work_data;
    const char *errctx;
    PQExpBufferData errbuf;
    char curl_err[CURL_ERROR_SIZE];
    struct provider provider;
    struct device_authz authz;
    int running;
    bool user_prompted;
    bool used_basic_auth;
    bool debugging;
    int dbg_num_calls;
};

入力と出力

本構造体は関数引数として渡され、認証フローの進捗に応じて内部フィールド（step、provider、authz 等）が更新されます。出力として、エラー発生時には errbuf および curl_err に詳細な診断情報が格納されます。

前提条件と不変条件

  • work_data を使用する際は、事前に内容をクリアする必要があります。
  • errctx は静的割り当てを指す必要があり、解放してはなりません。
  • 認証フローの各ステップ（OAuthStep）において、必要なデータ（provider や authz）が適切にキャッシュされている必要があります。

アルゴリズムまたは振る舞い

  • libcurl のマルチハンドル（curlm）とイージーハンドル（curl）を管理し、非同期 I/O 多重化（mux）を介して認証フローを駆動します。
  • 認証フローの各段階で発生したエラーを errctx（操作の文脈）、errbuf（論理エラー）、curl_err（libcurl の詳細エラー）の3要素に分類して保持し、最終的に統合されたエラーメッセージを生成します。
  • running フラグにより、非同期処理が現在進行中であるかを追跡します。

呼び出し元と呼び出し先

  • **呼び出し元**: pg_fe_run_oauth_flow 等の認証フロー制御関数。
  • **呼び出し先**: free_async_ctx（破棄時）、check_content_type、parse_oauth_json 等のユーティリティ関数。
  • *注: 呼び出し元・呼び出し先の詳細なコールグラフは、提供されたソース抜粋の範囲外です。*

関連データ構造

  • OAuthStep: 認証フローの現在地を示す列挙型。
  • provider: 認可サーバーのメタデータ。
  • device_authz: デバイス認可リクエストの結果。
  • PQExpBufferData: 文字列バッファ管理用。

失敗モードと境界条件

  • ネットワークエラーやプロトコルエラーが発生した場合、errbuf にエラー内容が記録されます。
  • curl_err は libcurl からの未翻訳のエラー情報を含みます。
  • 認証フローの再試行が必要な場合、timerfd を用いて待機時間を管理します。

保守担当者向けメモ

  • 本構造体は libpq の接続状態を跨いで生存するため、メモリリークを防ぐために free_async_ctx による適切な解放が必須です。
  • デバッグ時は debugging フラグを有効にすることで、dbg_num_calls 等を用いた詳細なトレースが可能です。
