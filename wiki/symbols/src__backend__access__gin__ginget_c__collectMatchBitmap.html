collectMatchBitmap

定義

  • 種別: function
  • 定義箇所: src/backend/access/gin/ginget.c:120

目的

GINインデックスの特定の検索エントリに対し、合致するTID（タプル識別子）をビットマップ形式で収集する。

シグネチャまたは形状

static bool collectMatchBitmap(GinBtreeData *btree, GinBtreeStack *stack, GinScanEntry scanEntry, Snapshot snapshot)

入力と出力

  • **入力**:
  • btree: GIN B-treeの走査状態。
  • stack: 現在のインデックスページ位置を保持するスタック。
  • scanEntry: 検索対象のエントリ情報。
  • snapshot: 述語ロック用のスナップショット。
  • **出力**:
  • bool: 処理が正常に完了した場合は true。インデックス構造の変更により再走査が必要な場合は false。

前提条件と不変条件

  • stack->buffer は呼び出し時点でロックされている必要がある。
  • scanEntry->matchBitmap は本関数内で tbm_create により初期化される。

アルゴリズムまたは振る舞い

1. tbm_create でビットマップを初期化。 2. PredicateLockPage で対象のリーフページをロック。 3. moveRightIfItNeeded を用いて、対象の属性番号に合致するインデックスエントリを走査。 4. 部分一致検索（isPartialMatch）の場合、comparePartialFn を用いて一致判定を行い、条件に応じて走査を終了または継続。 5. GIN_SEARCH_MODE_ALL モードの場合、NULLアイテムに到達した時点で走査を終了。 6. エントリが PostingTree を指す場合:

  • デッドロック回避のため、一時的にバッファのロックを解除。
  • scanPostingTree を呼び出し、TIDをビットマップに収集。
  • 再度バッファをロックし、インデックス構造の変更がないか確認。変更がある場合は false を返して再走査を促す。

    • 7. エントリが PostingList の場合、後続の処理（ソース抜粋外）へ移行。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/access/gin/ginget.c 内の走査ロジックから呼び出される。
  • **呼び出し先**: tbm_create, PredicateLockPage, moveRightIfItNeeded, BufferGetPage, PageGetItem, gintuple_get_attrnum, gintuple_get_key, FunctionCall4Coll, GinIsPostingTree, GinGetPostingTree, datumCopy, LockBuffer, scanPostingTree, ereport 等。

関連データ構造

  • GinBtreeData: GINインデックスの走査状態を保持。
  • GinBtreeStack: インデックス走査中のページ位置スタック。
  • GinScanEntry: 検索キーごとの状態管理構造体。
  • TIDBitmap: 収集されたTIDを保持するビットマップ構造。

失敗モードと境界条件

  • **再走査**: Posting Treeの走査中にインデックス構造が変更された場合、false を返して呼び出し元に再試行を要求する。
  • **エラー**: インデックス内でタプルを再発見できない場合、ereport(ERROR, ...) により処理を中断する。
  • **部分一致**: 検索キーがNULLの場合、または部分一致条件で不一致と判定された場合は即座に終了する。

保守担当者向けメモ

  • Posting Tree走査中のロック解除はデッドロック回避に不可欠である。
  • idatum のコピーと解放（datumCopy/pfree）は、メモリ管理の観点から attbyval フラグに依存している点に注意すること。
