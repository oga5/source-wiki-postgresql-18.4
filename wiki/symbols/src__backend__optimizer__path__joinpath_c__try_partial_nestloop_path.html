try_partial_nestloop_path

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/path/joinpath.c:949

目的

並列クエリ実行を考慮したネステッドループ結合（Nestloop Join）のパスを生成し、コスト評価を経て有効であれば結合リレーションのパスリストに追加する。

シグネチャまたは形状

static void try_partial_nestloop_path(PlannerInfo *root,
                                      RelOptInfo *joinrel,
                                      Path *outer_path,
                                      Path *inner_path,
                                      List *pathkeys,
                                      JoinType jointype,
                                      JoinPathExtraData *extra)

入力と出力

  • **入力**:
  • root: プランナのグローバル情報。
  • joinrel: 結合対象のリレーション情報。
  • outer_path, inner_path: 結合する外側および内側のパス。
  • pathkeys: 結合結果のソート順序。
  • jointype: 結合の種類（INNER, LEFT 等）。
  • extra: 結合に必要なメタデータ（restrictlist 等）。
  • **出力**: なし（成功時に joinrel->partial_pathlist を更新）。

前提条件と不変条件

  • joinrel->lateral_relids が空であること。
  • outer_path が外部パラメータに依存していないこと（PATH_REQ_OUTER が空）。
  • 内側パスがパラメータ化されている場合、そのパラメータが外側パスのリレーションによって完全に充足されること。
  • パラメータ化された内側パスを子リレーションへ再パラメータ化できない場合は、パス生成を中止する。

アルゴリズムまたは振る舞い

1. **パラメータ検証**: 内側パスのパラメータ依存関係が外側パスで解決可能かを確認する。 2. **コスト見積もり**: initial_cost_nestloop を呼び出し、結合コストを算出する。 3. **事前チェック**: add_partial_path_precheck を使用し、コストやパスキーに基づき、このパスが採用に値するかを判定する。 4. **パス生成と登録**: 上記を通過した場合、create_nestloop_path でパスを作成し、add_partial_path を介して joinrel に登録する。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/optimizer/path/joinpath.c 内の結合パス生成ロジック（add_paths_to_joinrel 等から間接的に呼び出される）。
  • **呼び出し先**:
  • initial_cost_nestloop (コスト計算)
  • add_partial_path_precheck (採用可否判定)
  • create_nestloop_path (パスノード生成)
  • add_partial_path (パスの登録)

関連データ構造

  • RelOptInfo: 結合対象リレーションの情報を保持。
  • Path: 結合パスの基本構造。
  • JoinCostWorkspace: 結合コスト計算用の一時領域。
  • JoinPathExtraData: 結合処理に必要な追加情報。

失敗モードと境界条件

  • パラメータ化の充足条件を満たさない場合、パスは生成されず早期リターンする。
  • add_partial_path_precheck により、コストが既存のパスと比較して劣ると判断された場合、パスは追加されない。

保守担当者向けメモ

  • 本関数は並列実行可能なパスのみを扱う。通常の（非並列）ネステッドループについては try_nestloop_path を参照すること。
  • パラメータ化されたパスの再パラメータ化（path_is_reparameterizable_by_child）の判定ロジックは、プランナの結合最適化において重要な制約となる。
