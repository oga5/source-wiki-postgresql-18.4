timeout_params

定義

  • **種別**: struct
  • **定義箇所**: src/backend/utils/misc/timeout.c:26

目的

PostgreSQLのバックエンドプロセスにおいて、個別のタイムアウトイベントの構成、状態管理、および期限切れ時のコールバック実行を定義するデータ構造です。

シグネチャまたは形状

typedef struct timeout_params
{
    TimeoutId    index;
    volatile bool active;
    volatile bool indicator;
    timeout_handler_proc timeout_handler;
    TimestampTz  start_time;
    TimestampTz  fin_time;
    int          interval_in_ms;
} timeout_params;

入力と出力

本構造体は関数ではなくデータ型です。各フィールドは以下の役割を持ちます。

  • index: タイムアウト理由を識別するID。
  • active: active_timeouts[] 配列に含まれているかを示すフラグ。
  • indicator: タイムアウトが発生したことを示すフラグ。
  • timeout_handler: 期限切れ時に実行される関数ポインタ。
  • start_time, fin_time: タイムアウトの開始時刻と期限時刻。
  • interval_in_ms: 繰り返し実行の間隔（0の場合は単発）。

前提条件と不変条件

  • active および indicator フィールドは、シグナルハンドラから変更される可能性があるため volatile 修飾されています。
  • active_timeouts 配列内で管理される際、期限順にソートされることが期待されます。

アルゴリズムまたは振る舞い

  • SIGALRM シグナル発生時、handle_sig_alarm が active_timeouts を走査し、期限切れの timeout_params を特定します。
  • 繰り返し実行が必要なタスク（interval_in_ms > 0）は、次回の期限を計算して再スケジュールされます。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/utils/misc/timeout.c 内の all_timeouts 配列および active_timeouts 配列の要素として参照されます。
  • **呼び出し先**: timeout_handler フィールドに登録された関数が、期限切れ時に実行されます。

関連データ構造

  • TimeoutId: タイムアウト理由を識別する列挙型。
  • all_timeouts: 全てのタイムアウト設定を保持する静的配列。
  • active_timeouts: 現在有効なタイムアウトを期限順に保持する配列。

失敗モードと境界条件

  • シグナルハンドラ内で実行されるため、ハンドラ内での elog(FATAL) や longjmp は状態の不整合を招くリスクがあります。
  • active_timeouts のソート順序が崩れると、タイムアウトの発生タイミングが正しく管理されなくなります。

保守担当者向けメモ

  • 本構造体のフィールド変更時は、シグナル安全性（sig_atomic_t 等の考慮）を維持してください。
  • タイムアウトが期待通りに発生しない場合は、indicator フラグの状態と active_timeouts のソート順序を確認してください。
