ConflictType

定義

  • 種別: enum
  • 定義箇所: src/include/replication/conflict.h:31

目的

論理レプリケーションの適用時に発生する競合の種類を識別し、ログ出力や統計情報の記録に使用します。

シグネチャまたは形状

typedef enum {
    CT_INSERT_EXISTS,
    CT_UPDATE_ORIGIN_DIFFERS,
    CT_UPDATE_EXISTS,
    CT_UPDATE_MISSING,
    CT_DELETE_ORIGIN_DIFFERS,
    CT_DELETE_MISSING,
    CT_MULTIPLE_UNIQUE_CONFLICTS
} ConflictType;

入力と出力

  • 本シンボルは列挙型であり、入力や出力を持つ関数ではありません。
  • 統計情報収集関数 pgstat_report_subscription_conflict や、エラー報告関数 errcode_apply_conflict 等の引数として渡されます。

前提条件と不変条件

  • 競合の発生は、論理レプリケーションワーカーがリモートの変更をローカルに適用する過程で検出されます。
  • CT_INSERT_EXISTS、CT_UPDATE_EXISTS、CT_MULTIPLE_UNIQUE_CONFLICTS は一意制約違反に関連します。
  • CT_UPDATE_ORIGIN_DIFFERS および CT_DELETE_ORIGIN_DIFFERS は、レプリケーション元の不一致に関連します。

アルゴリズムまたは振る舞い

  • 適用プロセスがローカルテーブル操作中に制約違反やデータ不整合を検知した際、該当する ConflictType を特定します。
  • 特定された型は、ReportApplyConflict() を通じてエラーログの生成および pgstat による統計情報の更新に使用されます。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/replication/logical/worker.c（競合検出時）、src/backend/executor/execReplication.c 等。
  • **呼び出し先**: 本シンボル自体は型定義であるため呼び出し先を持ちませんが、pgstat_report_subscription_conflict や errcode_apply_conflict 等の関数に引数として渡されます。

関連データ構造

  • ConflictTupleInfo: 競合を引き起こしたローカル行のメタデータ（トランザクション情報等）を保持する構造体。

失敗モードと境界条件

  • 複雑なルール（除外制約違反など）を伴う競合は、本列挙型では網羅されておらず、将来的な拡張対象とされています。

保守担当者向けメモ

  • ConflictType に新しい値を追加する場合、pgstat 関連の統計情報収集コードへの影響を必ず確認してください。
  • track_commit_timestamp が無効な場合、競合情報に含まれるタイムスタンプが取得できない可能性があるため、呼び出し側で考慮が必要です。
