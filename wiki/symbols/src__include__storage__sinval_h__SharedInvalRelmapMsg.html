SharedInvalRelmapMsg

定義

  • 種別: struct
  • 定義箇所: src/include/storage/sinval.h:99

目的

リレーションマップ（Relmap）の変更を他のバックエンドに通知し、共有キャッシュの整合性を維持するためのメッセージ構造体です。

シグネチャまたは形状

typedef struct
{
    int8    id;         /* メッセージ型識別子（先頭必須） */
    Oid     dbId;       /* データベースID（共有カタログの場合は0） */
} SharedInvalRelmapMsg;

入力と出力

  • **入力**: id（メッセージ種別）、dbId（対象データベースのOID）。
  • **出力**: なし（本構造体は共有メモリ上の無効化キューに書き込まれるデータ形式です）。

前提条件と不変条件

  • id フィールドは構造体の先頭に配置されなければなりません。
  • dbId が 0 の場合、共有カタログに関連するリレーションマップの無効化を意味します。

アルゴリズムまたは振る舞い

本構造体は、リレーションマップが更新された際に共有メモリ上の無効化キューへ投入されます。受信側のバックエンドは、このメッセージを読み取ることで、自身のローカルなリレーションマップキャッシュを無効化または再ロードします。他のキャッシュ無効化メッセージと異なり、リレーションマップの無効化はトランザクションのコミットを待たずに即時送信されます。

呼び出し元と呼び出し先

  • **呼び出し元**: ソースコード上での直接的な呼び出し元は特定されていませんが、リレーションマップを更新するストレージ管理層から利用されます。
  • **呼び出し先**: 本構造体自体はデータ保持用であり、関数呼び出しを含みません。

関連データ構造

  • SharedInvalidationMessage: 本構造体を包含する共用体。
  • RelFileLocator: リレーションの物理的な位置を特定するために使用されます。

失敗モードと境界条件

  • キューが溢れた場合、PROCSIG_CATCHUP_INTERRUPT を介した強制的なキャッシュ同期が発生します。
  • 共有メモリ上のキュー操作における競合は、sinval モジュールの排他制御メカニズムによって管理されます。

保守担当者向けメモ

  • 本メッセージは即時送信されるため、トランザクションのロールバックが発生しても無効化が取り消されない点に注意してください。
  • id フィールドの型定義や順序を変更する場合、SharedInvalidationMessage 共用体との整合性を維持する必要があります。
