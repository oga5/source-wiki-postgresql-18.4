BTPageOpaqueData

定義

  • 種別: struct
  • 定義箇所: src/include/access/nbtree.h:63

目的

PostgreSQL の B-tree インデックスにおける各ページの末尾に配置される特殊領域（Opaque Data）を定義します。ページ間のリンク、ツリー階層、および同時実行制御のための状態を保持します。

シグネチャまたは形状

typedef struct BTPageOpaqueData
{
    BlockNumber btpo_prev;      /* 左兄弟ページ、または最左端の場合は P_NONE */
    BlockNumber btpo_next;      /* 右兄弟ページ、または最右端の場合は P_NONE */
    uint32      btpo_level;     /* ツリーレベル（リーフページは 0） */
    uint16      btpo_flags;     /* フラグビット（BTP_LEAF, BTP_ROOT 等） */
    BTCycleId   btpo_cycleid;   /* 直近のページ分割時のバキュームサイクル ID */
} BTPageOpaqueData;

入力と出力

本構造体はデータ保持を目的としており、関数としての入力・出力はありません。ページ初期化時や読み込み時に、バッファ内の特殊領域としてメモリ上に展開されます。

前提条件と不変条件

  • ページ末尾に配置される必要があり、サイズは MAXALIGN(sizeof(BTPageOpaqueData)) に調整されます。
  • btpo_level はリーフページで 0 となり、ルートに向かって増加します。
  • btpo_prev および btpo_next は、インデックスの物理的な兄弟関係を維持するために使用されます。

アルゴリズムまたは振る舞い

  • **ページ管理**: Lehman & Yao のアルゴリズムに基づき、ページ分割時に btpo_next を更新して兄弟ページをリンクします。
  • **同時実行制御**: btpo_cycleid は、バキューム処理中にページ分割が発生した際、スキャンが正しく継続できるように追跡するために使用されます。

呼び出し元と呼び出し先

  • **呼び出し元（参照・操作）**:
  • src/backend/access/nbtree/nbtpage.c: ページ初期化および検証。
  • src/backend/access/nbtree/nbtdedup.c: 重複排除処理。
  • src/backend/access/nbtree/nbtsplitloc.c: ページ分割位置の決定。
  • contrib/pageinspect/btreefuncs.c: ページ構造の検査。
  • contrib/amcheck/verify_nbtree.c: インデックスの整合性チェック。
  • **呼び出し先**:
  • 本構造体はデータ構造定義であり、直接的な関数呼び出しは行いません。

関連データ構造

  • BTMetaPageData: インデックスのメタページ構造。
  • BTScanOpaqueData: スキャン中の状態保持用構造体。

失敗モードと境界条件

  • **サイズ不整合**: PageGetSpecialSize が MAXALIGN(sizeof(BTPageOpaqueData)) と一致しない場合、ページ破損とみなされます。
  • **境界値**: 最左端ページでは btpo_prev が P_NONE、最右端ページでは btpo_next が P_NONE となります。

保守担当者向けメモ

  • BTREE_VERSION の変更に伴い、本構造体のフィールド追加や変更が必要になる場合があります。
  • 16ビットの btpo_cycleid は、バキュームサイクルを追跡するための制限付き領域である点に留意してください。
