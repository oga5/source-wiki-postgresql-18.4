pg_logical_slot_get_changes_guts

定義

  • 種別: function
  • 定義箇所: src/backend/replication/logical/logicalfuncs.c:98

目的

論理デコーディングの変更抽出処理（get/peek および text/binary）を統合管理し、SQLインターフェース経由でWALの変更内容を Tuplestore に出力する。

シグネチャまたは形状

static Datum pg_logical_slot_get_changes_guts(FunctionCallInfo fcinfo, bool confirm, bool binary)

入力と出力

  • **入力**:
  • fcinfo: SQL関数呼び出し情報。
  • confirm: true の場合、読み取り位置を更新する。
  • binary: true の場合、バイナリ形式で出力する。
  • **出力**:
  • Datum: Tuplestore に格納された結果セット。

前提条件と不変条件

  • 呼び出し前に CheckSlotPermissions および CheckLogicalDecodingRequirements による検証が行われる。
  • options 配列は1次元かつ偶数要素数（キーと値のペア）である必要がある。
  • 出力プラグインがバイナリ出力をサポートしていない場合、binary=false の要求と矛盾するとエラーとなる。

アルゴリズムまたは振る舞い

1. **初期化**: DecodingOutputState を確保し、SQL関数の戻り値用メモリコンテキストへ切り替える。 2. **オプション解析**: 引数の配列をデコードし、DefElem リストを構築する。 3. **コンテキスト作成**: CreateDecodingContext を呼び出し、WAL読み取りおよび書き込み用コールバックを登録する。 4. **同期**: WaitForStandbyConfirmation により、指定されたLSNまでスタンバイサーバーの確認を待機する。 5. **デコーディングループ**:

  • XLogBeginRead で開始位置を設定。
  • XLogReadRecord でWALを順次読み込み、LogicalDecodingProcessRecord で処理する。
  • upto_lsn または upto_nchanges の制限に達するまで繰り返す。

    • 6. **終了処理**: PG_TRY ブロック内で実行され、エラー発生時もリソースの整合性を維持する。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • pg_logical_slot_get_changes
  • pg_logical_slot_peek_changes
  • pg_logical_slot_get_binary_changes
  • pg_logical_slot_peek_binary_changes
  • **呼び出し先**:
  • CreateDecodingContext, XLogReadRecord, LogicalDecodingProcessRecord, WaitForStandbyConfirmation, ReplicationSlotAcquire, InvalidateSystemCaches 等。

関連データ構造

  • DecodingOutputState: 出力プラグインからのデータを一時保持し、Tuplestore へ変換するための状態構造体。
  • LogicalDecodingContext: デコーディングの実行コンテキスト。

失敗モードと境界条件

  • **エラー**:
  • スロット名やオプション配列がNULLの場合。
  • オプション配列が1次元でない、または要素数が奇数の場合。
  • 出力プラグインの形式と要求された形式（テキスト/バイナリ）が不一致の場合。
  • WALレコードの読み取りに失敗した場合。
  • **境界条件**:
  • upto_lsn または upto_nchanges が指定されている場合、その条件を満たした時点でループを終了する。

保守担当者向けメモ

  • PG_TRY/PG_CATCH ブロックは、論理デコーディング中のシステムキャッシュクリアとリソースリーク防止のために不可欠である。
  • CurrentResourceOwner の管理は、トランザクション管理に伴う副作用を抑えるための重要な回避策である。
