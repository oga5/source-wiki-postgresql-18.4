find_my_exec

定義

  • 種別: function
  • 定義箇所: src/common/exec.c:159

目的

現在のプロセス自身の実行ファイルの絶対パスを特定し、検証を行う。

シグネチャまたは形状

int find_my_exec(const char *argv0, char *retpath);

入力と出力

  • argv0: プロセス起動時の argv[0]。
  • retpath: 解決された絶対パスを格納するバッファ（MAXPGPATH以上のサイズが必要）。
  • 戻り値: 成功時は 0、失敗時は -1 を返す。

前提条件と不変条件

  • retpath は MAXPGPATH バイト以上の書き込み可能なメモリ領域であること。
  • 実行ファイルの検証には validate_exec が使用され、ファイルが存在し、かつ実行可能である必要がある。

アルゴリズムまたは振る舞い

1. **直接パスの確認**: argv0 にディレクトリ区切り文字が含まれる場合、それを直接 validate_exec で検証し、成功すれば normalize_exec_path を呼び出して絶対パス化して終了する。 2. **Windows環境の特例**: WIN32 定義時は、区切り文字がない場合でもカレントディレクトリを先に確認する。 3. **PATH環境変数の走査**: 上記で特定できない場合、環境変数 PATH を順次走査する。

  • 各パス要素に対して argv0 を結合し、canonicalize_path で正規化する。
  • validate_exec で検証し、成功すれば normalize_exec_path を呼び出して終了する。

    • 4. **エラー処理**: どの候補でも実行ファイルが見つからない場合、log_error を通じてエラーを報告し -1 を返す。

呼び出し元と呼び出し先

  • **呼び出し元**: postmaster.c, miscinit.c, initdb.c, pg_basebackup, pg_config.c, pg_ctl.c, pg_dumpall.c, pg_rewind.c, pg_upgrade.c, pg_verifybackup.c など、PostgreSQLの主要なバイナリ群。
  • **呼び出し先**: validate_exec, normalize_exec_path, first_dir_separator, first_path_var_separator, join_path_components, canonicalize_path, log_error。

関連データ構造

  • MAXPGPATH: パス文字列の最大長を規定する定数。

失敗モードと境界条件

  • argv0 が無効なパスである場合、validate_exec の結果に応じてエラーログが出力される。
  • PATH 環境変数が未設定または空の場合、検索は行われずエラーとなる。
  • validate_exec が -2（ファイルは存在するが読み取り不可など）を返した場合、その候補はスキップされ、次の PATH 要素の探索が継続される。

保守担当者向けメモ

  • Windows環境における .exe 拡張子の補完や、OS固有のパス解決ロジックが validate_exec および normalize_exec_path に依存している点に注意すること。
  • log_error マクロを使用しているため、呼び出し元のコンテキスト（フロントエンドかバックエンドか）に応じて適切なエラー出力が行われる。
