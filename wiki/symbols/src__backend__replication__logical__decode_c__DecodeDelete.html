DecodeDelete

定義

  • 種別: function
  • 定義箇所: src/backend/replication/logical/decode.c:1019

目的

ヒープの削除操作を示す WAL レコードを解析し、論理デコーディング用の変更情報として再構築して reorderbuffer にキューイングする。

シグネチャまたは形状

static void DecodeDelete(LogicalDecodingContext *ctx, XLogRecordBuffer *buf)

入力と出力

  • **入力**:
  • ctx: 論理デコーディングのコンテキスト。
  • buf: 解析対象の WAL レコードおよび LSN 情報を含むバッファ。
  • **出力**: なし（副作用として ctx->reorder に変更が追加される）。

前提条件と不変条件

  • buf->record は有効な XLogReaderState を指していること。
  • データベース OID が ctx->slot->data.database と一致する場合のみ処理を継続する。
  • FilterByOrigin によるフィルタリング条件に合致しない場合のみ処理を継続する。

アルゴリズムまたは振る舞い

1. WAL レコードから xl_heap_delete 構造体を取得する。 2. ターゲットのデータベース OID を確認し、不一致なら早期リターンする。 3. FilterByOrigin を呼び出し、レプリケーションオリジンによるフィルタリングを行う。 4. ReorderBufferAllocChange を用いて変更オブジェクトを確保する。 5. レコードのフラグ (XLH_DELETE_IS_SUPER) に応じて、アクションを REORDER_BUFFER_CHANGE_INTERNAL_SPEC_ABORT または REORDER_BUFFER_CHANGE_DELETE に設定する。 6. レコードに古いタプル情報 (XLH_DELETE_CONTAINS_OLD) が含まれる場合、DecodeXLogTuple を使用してタプルデータをデコードし、change->data.tp.oldtuple に格納する。 7. change->data.tp.clear_toast_afterwards を true に設定する。 8. ReorderBufferQueueChange を呼び出し、トランザクション ID と LSN を紐付けて変更をキューに投入する。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/replication/logical/decode.c:511（LogicalDecodingProcessRecord 内のディスパッチ処理から呼び出される）。
  • **呼び出し先**:
  • XLogRecGetData, XLogRecGetBlockTag, XLogRecGetOrigin, XLogRecGetDataLen, XLogRecGetXid
  • FilterByOrigin
  • ReorderBufferAllocChange, ReorderBufferAllocTupleBuf, ReorderBufferQueueChange
  • DecodeXLogTuple

関連データ構造

  • LogicalDecodingContext: デコーディングの全体状態。
  • XLogRecordBuffer: WAL レコードのラッパー。
  • xl_heap_delete: ヒープ削除レコードの物理構造。
  • ReorderBufferChange: 論理変更の抽象表現。

失敗モードと境界条件

  • データベース OID が一致しない場合、何も行わず終了する。
  • FilterByOrigin が真を返す場合、何も行わず終了する。
  • XLH_DELETE_CONTAINS_OLD フラグが立っている場合、XLogRecGetDataLen が SizeOfHeapDelete + SizeOfHeapHeader より大きいことを保証するアサーションが存在する。

保守担当者向けメモ

  • 本関数はヒープ削除の物理レコードを論理的な変更に変換する役割を持つ。
  • XLH_DELETE_IS_SUPER フラグは、論理デコーディングにおいて特殊なアボート処理（SPEC_ABORT）をトリガーするために使用される。
  • TOAST データのクリアフラグ (clear_toast_afterwards) は常に true に設定される。
