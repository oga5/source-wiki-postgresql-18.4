pending_label

定義

  • 種別: struct
  • 定義箇所: contrib/sepgsql/label.c:66

目的

トランザクションおよびサブトランザクション内でのセキュリティラベルの変更を追跡・管理するためのデータ構造です。

シグネチャまたは形状

typedef struct
{
	SubTransactionId subid;
	char	   *label;
} pending_label;

入力と出力

  • **入力**: subid（サブトランザクションID）、label（セキュリティコンテキスト文字列）。
  • **出力**: なし（構造体定義）。

前提条件と不変条件

  • client_label_pending リストに格納され、トランザクションのコミットまたはロールバック時に処理されることを前提としています。
  • label はメモリ管理された文字列へのポインタです。

アルゴリズムまたは振る舞い

  • sepgsql_set_client_label() 等の関数において、ラベル変更を一時的に保持するためにインスタンス化されます。
  • トランザクションの階層構造（サブトランザクション）とラベル変更を紐付け、ロールバック時に適切なラベル状態へ復元するための情報を保持します。

呼び出し元と呼び出し先

  • **呼び出し元**: contrib/sepgsql/label.c 内のラベル管理関数（sepgsql_set_client_label 等）が、palloc0 を用いて本構造体を生成し、client_label_pending リストに追加します。
  • **呼び出し先**: 本構造体自体はデータ保持用であり、直接関数を呼び出すことはありません。

関連データ構造

  • client_label_pending: 本構造体を要素として保持するリスト。
  • SubTransactionId: トランザクションの階層を識別するための型。

失敗モードと境界条件

  • メモリ不足時: palloc0 によるメモリ確保が失敗した場合、PostgreSQL のエラーハンドリング機構によりトランザクションがアボートされます。
  • リスト操作: llast() や lfirst() を使用してリストを走査する際、リストが空である場合にアクセスすると境界エラーが発生する可能性があります。

保守担当者向けメモ

  • 本構造体は CurTransactionContext 上で管理されるため、メモリリークを防ぐためにトランザクション終了時の適切な解放処理が必須です。
  • sepgsql_xact_callback 内で TopMemoryContext へのコピーが行われる点に注意してください。
