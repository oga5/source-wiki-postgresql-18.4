pg_buffercache_pages

定義

  • **種別**: function
  • **定義箇所**: contrib/pg_buffercache/pg_buffercache_pages.c:109

目的

PostgreSQL の共有バッファキャッシュの状態を SQL インターフェース経由で可視化するための Set-Returning Function (SRF) です。全バッファのメタデータを取得し、テーブル形式で返します。

シグネチャまたは形状

Datum pg_buffercache_pages(PG_FUNCTION_ARGS)

入力と出力

  • **入力**: PG_FUNCTION_ARGS (SQL 経由の呼び出し引数)
  • **出力**: Datum (SRF としてのタプルセット)
  • **返却される列**: bufferid, relfilenode, reltablespace, reldatabase, relforknumber, relblocknumber, isdirty, usage_count, pinning_backends (オプション)

前提条件と不変条件

  • 初回呼び出し時に NBuffers 分のメモリを確保し、全バッファをスキャンします。
  • バッファヘッダーのロック (LockBufHdr) を個別に取得して情報を抽出しますが、システム全体の一貫したスナップショットは保証されません。
  • expected_tupledesc が期待する列数と一致しない場合、エラーを発生させます。

アルゴリズムまたは振る舞い

1. **初回呼び出し**:

  • FuncCallContext を初期化し、BufferCachePagesContext を作成します。
  • NBuffers 分の BufferCachePagesRec 配列を確保します。
  • 全バッファをループで走査し、各バッファヘッダーをロックしてメタデータ（ダーティフラグ、使用回数、参照カウント等）を配列にコピーします。

    • 2. **各呼び出し**:

  • funcctx->call_cntr に基づき、キャッシュされたレコードからタプルを構築します。
  • バッファが未使用または無効な場合、bufferid 以外の列を NULL として返します。

    • 3. **終了**:

  • 全バッファの処理完了後、SRF_RETURN_DONE を返します。

呼び出し元と呼び出し先

  • **呼び出し元**: SQL インターフェース（PG_FUNCTION_INFO_V1 マクロにより登録）。
  • **呼び出し先**: GetBufferDescriptor, LockBufHdr, UnlockBufHdr, BufTagGetRelNumber, BufTagGetForkNum, BUF_STATE_GET_USAGECOUNT, BUF_STATE_GET_REFCOUNT, heap_form_tuple 等。

関連データ構造

  • BufferCachePagesContext: SRF の呼び出し間で状態を保持する構造体。
  • BufferCachePagesRec: 個々のバッファのメタデータを格納する構造体。
  • BufferDesc: バッファ管理の基本構造体。

失敗モードと境界条件

  • **メモリ消費**: NBuffers が大きい場合、MemoryContextAllocHuge による大量のメモリ確保が発生します。
  • **ロック競合**: 全バッファを順次ロックするため、高負荷時にはバッファ管理の競合を引き起こす可能性があります。
  • **型不整合**: 呼び出し側の期待するタプル定義が NUM_BUFFERCACHE_PAGES_MIN_ELEM 未満または NUM_BUFFERCACHE_PAGES_ELEM を超える場合、エラーとなります。

保守担当者向けメモ

  • 本関数はシステム全体のスナップショットを保証しません。調査時点の各バッファの状態を個別に取得するものである点に留意してください。
  • バージョン 1.0 からのアップグレードをサポートするため、pinning_backends 列の有無を動的に判定するロジックが含まれています。
