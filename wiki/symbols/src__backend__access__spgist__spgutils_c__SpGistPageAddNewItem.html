SpGistPageAddNewItem

定義

  • 種別: function
  • 定義箇所: src/backend/access/spgist/spgutils.c:1202

目的

SP-GiSTインデックスページに新しいアイテムを挿入します。ページ内にプレースホルダー（削除済みタプルの跡地）が存在する場合、可能な限りその領域を再利用して断片化を抑制します。

シグネチャまたは形状

OffsetNumber SpGistPageAddNewItem(SpGistState *state, Page page, Item item, Size size, OffsetNumber *startOffset, bool errorOK)

入力と出力

  • **入力**:
  • state: SP-GiSTの実行時状態。
  • page: 挿入対象のページ。
  • item: 挿入するタプルデータ。
  • size: アイテムのサイズ。
  • startOffset: プレースホルダー検索を開始するオフセット（ヒント）。
  • errorOK: 挿入失敗時にエラーを投げるか否かのフラグ。
  • **出力**:
  • 成功時: 挿入された位置の OffsetNumber を返します。
  • 失敗時: errorOK が真であれば InvalidOffsetNumber を返し、偽であれば ERROR を発生させます。

前提条件と不変条件

  • ページは SpGistPageOpaque 構造体で管理されている必要があります。
  • プレースホルダー再利用時、PageGetExactFreeSpace(page) + SGDTSIZE >= MAXALIGN(size) が満たされている必要があります。
  • プレースホルダーを削除した後に PageAddItem が失敗した場合は、データ整合性維持のため PANIC を発生させます。

アルゴリズムまたは振る舞い

1. ページ内にプレースホルダーが存在し、かつ十分な空き領域がある場合、プレースホルダーの検索を試みます。 2. startOffset が指定されていればその位置から、なければ先頭から SPGIST_PLACEHOLDER 状態のタプルを探します。 3. プレースホルダーが見つかった場合、PageIndexTupleDelete で削除し、その位置に PageAddItem で新しいアイテムを挿入します。成功すれば nPlaceholder を減算し、startOffset を更新します。 4. プレースホルダーが見つからない、あるいは再利用できない場合は、ページ末尾への通常の挿入（PageAddItem）を行います。

呼び出し元と呼び出し先

  • **呼び出し元**: 主に src/backend/access/spgist/spgdoinsert.c 内の各関数から、インデックス挿入処理の一部として呼び出されます。
  • **呼び出し先**: PageGetExactFreeSpace, PageGetMaxOffsetNumber, PageGetItem, PageGetItemId, PageIndexTupleDelete, PageAddItem などの標準的なページ操作関数を呼び出します。

関連データ構造

  • SpGistPageOpaque: ページヘッダーのSP-GiST固有情報（nPlaceholder を保持）。
  • SpGistDeadTuple: プレースホルダー判定に使用されるタプル構造。

失敗モードと境界条件

  • **プレースホルダー再利用失敗**: プレースホルダー削除後に PageAddItem が失敗した場合、回復不能として PANIC を発生させます。
  • **領域不足**: プレースホルダーがない、またはサイズが大きすぎて挿入できない場合、PageAddItem が InvalidOffsetNumber を返します。このとき errorOK が偽であれば ERROR となります。

保守担当者向けメモ

  • startOffset は挿入効率を上げるためのヒントであり、検索が失敗した場合は自動的にリセットされます。
  • プレースホルダーの再利用は、インデックスの肥大化を防ぐ重要な処理です。タプル構造の変更時には SGDTSIZE やアライメントの計算が正しく維持されているか確認してください。
