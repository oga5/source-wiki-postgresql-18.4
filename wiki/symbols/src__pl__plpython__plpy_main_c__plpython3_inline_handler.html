plpython3_inline_handler

定義

  • 種別: function
  • 定義箇所: src/pl/plpython/plpy_main.c:264

目的

PostgreSQL の DO 文（匿名コードブロック）として提供された Python コードを実行するためのハンドラです。

シグネチャまたは形状

Datum plpython3_inline_handler(PG_FUNCTION_ARGS)

入力と出力

  • **入力**: PG_FUNCTION_ARGS を介して InlineCodeBlock 構造体を受け取ります。これには実行するソースコードと言語 OID が含まれます。
  • **出力**: PG_RETURN_VOID() を返し、戻り値はありません。

前提条件と不変条件

  • PLy_initialize() が呼び出され、Python インタプリタが初期化されている必要があります。
  • 実行コンテキストのスタック操作（PLy_push_execution_context / PLy_pop_execution_context）は、エラー発生時を含め必ず対になるように管理される必要があります。

アルゴリズムまたは振る舞い

1. PLy_initialize で Python 環境を準備し、SPI_connect_ext で SPI 接続を確立します。 2. 匿名ブロック実行用の PLyProcedure 構造体を初期化し、メモリコンテキストを割り当てます。 3. PLy_push_execution_context で実行コンテキストをスタックに積みます。 4. PG_TRY ブロック内で以下を実行します。

  • plpython_inline_error_callback をエラーコンテキストスタックに登録します。
  • PLy_procedure_compile でソースコードをコンパイルします。
  • PLy_exec_function を呼び出し、コードを実行します。

    • 5. PG_CATCH ブロックでは、コンテキストのポップ、手続きの削除、Python エラーのクリアを行い、例外を再スローします。 6. 正常終了時は、コンテキストをポップし、手続きを削除して終了します。

呼び出し元と呼び出し先

  • **呼び出し元**: PostgreSQL の関数呼び出し機構（PG_FUNCTION_INFO_V1 マクロにより登録）。
  • **呼び出し先**: PLy_initialize, SPI_connect_ext, PLy_push_execution_context, PLy_procedure_compile, PLy_exec_function, PLy_pop_execution_context, PLy_procedure_delete 等。

関連データ構造

  • InlineCodeBlock: 実行対象のコードブロック情報。
  • PLyProcedure: 実行する手続きの定義とキャッシュ。
  • PLyExecutionContext: 実行中のコンテキスト管理用スタック構造体。

失敗モードと境界条件

  • Python コードの実行中にエラーが発生した場合、PG_CATCH ブロックによりコンテキストが正しく解放され、例外が再スローされます。
  • SPI_finish() が plpy_exec.c 側で呼ばれる設計となっており、ハンドラ側との責務分担に注意が必要です。

保守担当者向けメモ

  • plpython_inline_error_callback は現在 exec_ctx を使用していませんが、将来的な一貫性のために plpython3_call_handler と同様の形式で渡されています。
  • proc.result.typoid は現在 VOIDOID に固定されていますが、将来的に PLy_output_setup_func を使用する可能性がある点に留意してください。
