RelationAddBlocks

定義

  • 種別: function
  • 定義箇所: src/backend/access/heap/hio.c:237

目的

リレーションを物理的に拡張し、新しいページを割り当てて初期化する。既存の空き領域が不足している場合に呼び出され、必要に応じて FSM（Free Space Map）への登録や BulkInsertState を用いた効率的なバッファ確保を行う。

シグネチャまたは形状

static Buffer RelationAddBlocks(Relation relation, BulkInsertState bistate, int num_pages, bool use_fsm, bool *did_unlock)

入力と出力

  • **入力**:
  • relation: 拡張対象のリレーション。
  • bistate: 大量挿入時の状態管理構造体（NULL 可）。
  • num_pages: 要求するページ数。
  • use_fsm: FSM を使用するかどうかのフラグ。
  • did_unlock: 呼び出し元でバッファロックの再確認が必要かを示す出力用フラグ。
  • **出力**:
  • 戻り値: 確保された最初のページの Buffer。

前提条件と不変条件

  • MAX_BUFFERS_TO_EXTEND_BY（64）を超えるページ数の拡張は不可。
  • 拡張されたページは初期化（PageInit）され、ダーティとしてマークされる。
  • ページが既に空でない場合は elog(ERROR) を発生させる。

アルゴリズムまたは振る舞い

1. **拡張サイズの決定**: num_pages と現在の待機者数、bistate の履歴に基づき、拡張するページ数を計算する。 2. **リレーションの拡張**: ExtendBufferedRelBy を呼び出し、バッファを確保する。最初のページはロックされた状態で返される。 3. **初期化**: 確保したページに対して PageInit を実行し、ダーティマークを付与する。 4. **FSM への登録**: use_fsm が有効な場合、確保したページのうち使用しない分を FSM に登録する。 5. **ロック管理**: FSM 登録を行う場合、I/O を避けるためバッファロックを解除し、did_unlock を true に設定する。 6. **ピンの解放**: 最初のページ（戻り値）以外のピンを解放する。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/access/heap/hio.c 内の RelationGetBufferForTuple から呼び出される。
  • **呼び出し先**: ExtendBufferedRelBy, BufferGetPage, PageInit, MarkBufferDirty, LockBuffer, ReleaseBuffer, RecordPageWithFreeSpace, FreeSpaceMapVacuumRange 等。

関連データ構造

  • Relation: リレーション情報。
  • BulkInsertState: 大量挿入時のバッファ状態管理。
  • Buffer: バッファマネージャの管理単位。

失敗モードと境界条件

  • ページが空でない場合、elog(ERROR) により処理が中断される。
  • MAX_BUFFERS_TO_EXTEND_BY を超える要求は、Min 関数により制限される。

保守担当者向けメモ

  • 本関数はバッファロックを保持したままページ初期化を行うため、ロックの解放タイミング（特に FSM 登録時）には注意が必要。
  • did_unlock が true の場合、呼び出し元でバッファの状態が変化している可能性があるため、追加の検証が必要となる。
