ginVacuumPostingTreeLeaves

定義

  • 種別: function
  • 定義箇所: src/backend/access/gin/ginvacuum.c:344

目的

GINインデックスのポストティングツリーにおいて、リーフページを走査し、各ページに対してバキューム処理を実行する。また、処理の結果として空になったリーフページが存在するかを判定する。

シグネチャまたは形状

static bool ginVacuumPostingTreeLeaves(GinVacuumState *gvs, BlockNumber blkno)

入力と出力

  • **入力**:
  • gvs: バキューム処理の状態を保持する GinVacuumState 構造体へのポインタ。
  • blkno: ポストティングツリーのルートまたは走査開始位置となるブロック番号。
  • **出力**:
  • bool: 走査したリーフページの中に空（GinDataLeafPageIsEmpty）のものが含まれていた場合は true、それ以外は false を返す。

前提条件と不変条件

  • blkno は有効なポストティングツリーのページを指している必要がある。
  • 処理中、対象ページは GIN_EXCLUSIVE ロック下で操作される。
  • GinPageIsData(page) が真であること（データページであること）が前提。

アルゴリズムまたは振る舞い

1. **リーフの探索**: 指定された blkno から開始し、ツリーを降下して最も左側のリーフページを特定する。降下中は GIN_SHARE ロックを使用し、リーフ到達時に GIN_EXCLUSIVE ロックへ昇格する。 2. **リーフのバキューム**: rightlink を辿りながら、すべてのリーフページに対して ginVacuumPostingTreeLeaf を呼び出し、不要なTIDの削除を行う。 3. **メモリ管理**: 各リーフの処理は gvs->tmpCxt メモリコンテキストで行われ、反復ごとにリセットされる。 4. **空ページ判定**: 各リーフの処理後、ページが空になったかをチェックし、一つでも空のページがあれば hasVoidPage を true に設定する。 5. **終了**: rightlink が InvalidBlockNumber に達した時点で走査を終了する。

呼び出し元と呼び出し先

  • **呼び出し元**: ginVacuumPostingTree（src/backend/access/gin/ginvacuum.c 内の推論される呼び出し元）。
  • **呼び出し先**:
  • ReadBufferExtended, LockBuffer, BufferGetPage, UnlockReleaseBuffer（バッファ管理）
  • ginVacuumPostingTreeLeaf（各リーフのバキューム実行）
  • GinDataLeafPageIsEmpty（空判定）
  • MemoryContextSwitchTo, MemoryContextReset（メモリ管理）

関連データ構造

  • GinVacuumState: バキュームのコンテキスト情報（インデックスリレーション、メモリコンテキスト、戦略等）。
  • PostingItem: ポストティングツリー内のノードを指す構造体。

失敗モードと境界条件

  • ツリーが空、またはルートのみで構成されている場合でも正しく動作する。
  • ページが空になった場合、物理的な削除は本関数ではなく、呼び出し元で後続の処理（ginScanToDelete 等）が行われることを想定している。

保守担当者向けメモ

  • 本関数はポストティングツリーのリーフを横断的に処理する。ロックの取得・解放順序はデッドロックを避けるため厳密に管理されている。
  • ginVacuumPostingTreeLeaf の呼び出し前後でメモリコンテキストの切り替えを行っているため、メモリリークには注意が必要である。
