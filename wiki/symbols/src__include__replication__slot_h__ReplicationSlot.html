ReplicationSlot

定義

  • 種別: struct
  • 定義箇所: src/include/replication/slot.h:155

目的

レプリケーションスロットのメモリ内表現を保持し、プライマリサーバーがスタンバイや論理レプリケーションクライアントのために必要なWALや古いタプルを保持し続けるための状態管理を行う。

シグネチャまたは形状

typedef struct ReplicationSlot {
    slock_t mutex;
    bool in_use;
    pid_t active_pid;
    bool just_dirtied;
    bool dirty;
    TransactionId effective_xmin;
    TransactionId effective_catalog_xmin;
    ReplicationSlotPersistentData data;
    LWLock io_in_progress_lock;
    ConditionVariable active_cv;
    TransactionId candidate_catalog_xmin;
    XLogRecPtr candidate_xmin_lsn;
    XLogRecPtr candidate_restart_valid;
    XLogRecPtr candidate_restart_lsn;
    XLogRecPtr last_saved_confirmed_flush;
    TimestampTz inactive_since;
    XLogRecPtr last_saved_restart_lsn;
} ReplicationSlot;

入力と出力

  • **入力**: 共有メモリ上のスロット制御構造（ReplicationSlotCtlData）の一部として初期化され、WALの進捗やクライアントからの確認応答に基づいて各フィールドが更新される。
  • **出力**: data フィールド（ReplicationSlotPersistentData）がチェックポイント時にディスクへ永続化される。

前提条件と不変条件

  • スロットの追加・削除には ReplicationSlotControlLock（LWLock）が必要。
  • スロット内の個別のフィールド更新には、スロット固有の mutex（スピンロック）の取得が必須。
  • 論理デコーディングにおいて、effective_xmin はクラッシュリカバリ後も整合性を保つため、ディスクに書き込まれた最新の値を反映しなければならない。

アルゴリズムまたは振る舞い

  • **状態管理**: in_use フラグでスロットの有効性を管理し、active_pid で現在ストリーミング中のプロセスを追跡する。
  • **論理デコーディング**: candidate_* フィールドを使用して、クライアントからのフラッシュ確認に基づき catalog_xmin や restart_lsn を安全に進める。
  • **永続化**: just_dirtied および dirty フラグにより変更を検知し、チェックポイント時にディスクへフラッシュする。
  • **排他制御**: io_in_progress_lock により、I/O操作中のスロットに対する競合を防ぐ。

呼び出し元と呼び出し先

  • **呼び出し元**: 以下のソースファイルで ReplicationSlot 型への参照が確認されている。
  • src/backend/replication/logical/logical.c
  • src/backend/replication/logical/origin.c
  • src/backend/replication/logical/reorderbuffer.c
  • src/backend/replication/logical/slotsync.c
  • **呼び出し先**: 構造体自体はデータ保持が主目的であり、特定の関数呼び出しは含まない。操作には src/backend/replication/slot.c で定義された関数群が使用される。

関連データ構造

  • ReplicationSlotPersistentData: ディスクに書き込まれる永続属性。
  • ReplicationSlotCtlData: 全スロットを保持する共有メモリ上の配列。
  • ConditionVariable: active_pid の変化を通知するために使用。

失敗モードと境界条件

  • effective_xmin の不適切な管理は、論理デコーディングにおける誤ったデータ削除や整合性欠如を招く。
  • 適切なロックを取得せずにフィールドを更新した場合、共有メモリ上のデータ破損が発生する可能性がある。

保守担当者向けメモ

  • ReplicationSlot 内の data フィールドのみが永続化対象である点に注意すること。
  • 論理デコーディングの整合性維持のため、effective_xmin の更新ロジックには細心の注意を払うこと。
  • スロットの同期（スタンバイ側）とプライマリ側の操作で、inactive_since 等のフィールドの解釈が異なる場合がある。
