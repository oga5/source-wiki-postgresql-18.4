RelationPreTruncate

定義

  • src/backend/catalog/storage.c:449

目的

リレーションの切り詰め（Truncate）操作に先立ち、対象リレーションが保持する同期待ち状態（PendingRelSync）に対して、切り詰めが実行されることを記録する。

シグネチャまたは形状

void RelationPreTruncate(Relation rel)

入力と出力

  • **入力**: rel (対象リレーションの Relation 構造体)
  • **出力**: なし（副作用として pendingSyncHash 内のフラグを更新）

前提条件と不変条件

  • pendingSyncHash が初期化されていること。初期化されていない場合、本関数は何も行わず即座に復帰する。
  • rel から取得されるストレージロケータが、現在管理中の同期待ちハッシュテーブル内に存在していること。

アルゴリズムまたは振る舞い

1. グローバル変数 pendingSyncHash が存在するか確認する。存在しない場合は終了する。 2. RelationGetSmgr(rel) を介してリレーションのストレージロケータを取得する。 3. pendingSyncHash に対して HASH_FIND を実行し、該当する PendingRelSync エントリを検索する。 4. エントリが見つかった場合、その is_truncated フィールドを true に設定する。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/catalog/storage.c 内の切り詰め処理に関連する箇所（詳細はソースコード上の直接的な呼び出し箇所を参照）。
  • **呼び出し先**: hash_search（ハッシュテーブル検索関数）。

関連データ構造

  • PendingRelSync: 同期待ち状態を保持する構造体。is_truncated フラグを含む。
  • pendingSyncHash: 現在のトランザクションにおける同期待ちリレーションを管理するハッシュテーブル。

失敗モードと境界条件

  • pendingSyncHash が NULL の場合、処理は安全にスキップされる。
  • 指定されたリレーションが pendingSyncHash に登録されていない場合、hash_search は NULL を返し、何も更新されない。

保守担当者向けメモ

  • 本関数は、リレーションの物理的な切り詰めがコミット時に適切に処理されるよう、同期待ち状態をマークするために使用される。
  • is_truncated フラグの更新は、後続の同期処理（smgrDoPendingSyncs 等）における挙動に影響を与える可能性があるため、変更時は整合性に注意すること。
