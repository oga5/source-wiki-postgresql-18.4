create_join_plan

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/plan/createplan.c:1080

目的

JoinPath ノードを、実行エンジンが処理可能な Plan ノードへ変換する。

シグネチャまたは形状

Plan *create_join_plan(PlannerInfo *root, JoinPath *best_path)

入力と出力

  • **入力**:
  • root: プランナのグローバル情報へのポインタ。
  • best_path: 変換対象となる JoinPath 構造体。
  • **出力**:
  • 変換された Plan 構造体へのポインタ。

前提条件と不変条件

  • best_path->path.pathtype は T_MergeJoin、T_HashJoin、または T_NestLoop のいずれかである必要がある。

アルゴリズムまたは振る舞い

1. best_path->path.pathtype に基づき、以下のいずれかの関数を呼び出して Plan ノードを生成する。

  • T_MergeJoin: create_mergejoin_plan
  • T_HashJoin: create_hashjoin_plan
  • T_NestLoop: create_nestloop_plan

    • 2. 該当するノードタイプでない場合は elog(ERROR) を発生させる。 3. get_gating_quals を使用して、結合条件に含まれる擬似定数（pseudoconstant）句を抽出する。 4. 擬似定数句が存在する場合、create_gating_plan を呼び出し、ゲート用 Result ノードをプランツリーに挿入する。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/optimizer/plan/createplan.c 内の create_plan_recurse（推測されるフローに基づく）。
  • **呼び出し先**:
  • create_mergejoin_plan
  • create_hashjoin_plan
  • create_nestloop_plan
  • get_gating_quals
  • create_gating_plan

関連データ構造

  • JoinPath: 結合のパス情報。
  • Plan: 実行計画の基底構造体。
  • PlannerInfo: プランナのコンテキスト情報。

失敗モードと境界条件

  • 未知の pathtype が渡された場合、エラーが発生する。
  • 擬似定数句が存在する場合、create_gating_plan によってプランツリーの構造が変更される。

保守担当者向けメモ

  • ソースコード内に #ifdef NOT_USED で囲まれた古いロジック（get_loc_restrictinfo 等）が存在するが、現在は実行されない。
  • 結合ノード固有の最適化や制約の追加が必要な場合は、switch 文の各ケースまたはゲート処理の後にロジックを追加する。
