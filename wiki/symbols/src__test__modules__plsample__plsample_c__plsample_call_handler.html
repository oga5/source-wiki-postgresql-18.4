plsample_call_handler

定義

  • 種別: function
  • 定義箇所: src/test/modules/plsample/plsample.c:38

目的

PostgreSQL の手続き型言語（PL）ハンドラのエントリポイントとして、呼び出し種別（関数、トリガー、イベントトリガー）を判別し、適切なサブハンドラへ処理をディスパッチする。

シグネチャまたは形状

Datum plsample_call_handler(PG_FUNCTION_ARGS)

入力と出力

  • **入力**: PG_FUNCTION_ARGS（FunctionCallInfo 構造体へのポインタを含むマクロ）
  • **出力**: Datum 型の戻り値。各サブハンドラの実行結果を返す。

前提条件と不変条件

  • fcinfo->context を通じて呼び出しコンテキストが提供されている必要がある。
  • エラー発生時でもリソース解放が可能なよう、PG_TRY/PG_FINALLY ブロック内で実行される。

アルゴリズムまたは振る舞い

1. PG_TRY ブロックを開始する。 2. CALLED_AS_TRIGGER(fcinfo) を判定し、真であれば plsample_trigger_handler を呼び出す。 3. CALLED_AS_EVENT_TRIGGER(fcinfo) を判定し、真であればイベントトリガー用の処理を行う（現在の実装では本体なし）。 4. 上記いずれでもない場合、通常の関数として plsample_func_handler を呼び出す。 5. PG_FINALLY ブロックでリソースのクリーンアップを行い、PG_END_TRY で終了する。

呼び出し元と呼び出し先

  • **呼び出し元**: PostgreSQL の関数マネージャ（fmgr）。PG_FUNCTION_INFO_V1(plsample_call_handler) マクロにより登録される。
  • **呼び出し先**:
  • plsample_trigger_handler（トリガー呼び出し時）
  • plsample_func_handler（通常関数呼び出し時）

関連データ構造

  • FunctionCallInfo (fcinfo): 関数呼び出しの引数やコンテキスト情報。
  • TriggerData: fcinfo->context に格納されるトリガー実行時の情報。
  • EventTriggerData: fcinfo->context に格納されるイベントトリガー実行時の情報。

失敗モードと境界条件

  • イベントトリガーとして呼び出された場合、現在の実装では処理が空である。
  • PG_TRY ブロック内で発生したエラーは PostgreSQL のエラーハンドリング機構に委ねられる。

保守担当者向けメモ

  • 本関数はディスパッチャであるため、新しい言語機能を追加する際は、まずここで呼び出し種別を正しく判定し、適切なハンドラへ転送するように拡張すること。
