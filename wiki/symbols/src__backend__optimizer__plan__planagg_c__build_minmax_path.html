build_minmax_path

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/plan/planagg.c:316

目的

集約関数（MIN/MAX）をインデックススキャンを利用した LIMIT 1 サブクエリへ置換するためのパスを構築します。

シグネチャまたは形状

static bool build_minmax_path(PlannerInfo *root, MinMaxAggInfo *mminfo,
                              Oid eqop, Oid sortop, bool reverse_sort, bool nulls_first)

入力と出力

  • **入力**:
  • root: 現在のクエリレベルの PlannerInfo。
  • mminfo: 最適化対象の集約関数に関するメタデータ。
  • eqop, sortop: ソートおよび等価比較に使用する演算子。
  • reverse_sort, nulls_first: ソート順序の指定。
  • **出力**:
  • bool: パスの構築に成功した場合は true、失敗した場合は false。成功時、mminfo 内の subroot, path, pathcost が更新されます。

前提条件と不変条件

  • can_minmax_aggs による事前検証が完了していること。
  • subroot の構築において、join_info_list, eq_classes, placeholder_list が空であることを前提とします。
  • mminfo->target は行型（rowtype）ではないことが保証されている必要があります。

アルゴリズムまたは振る舞い

1. **サブクエリ環境の構築**: 現在の PlannerInfo をクローンし、query_level をインクリメントしてサブクエリ用の subroot を作成します。 2. **クエリの書き換え**: targetList を対象集約式のみに絞り、HAVING や DISTINCT を削除します。 3. **制約の追加**: IS NOT NULL 制約を jointree->quals に追加し、指定されたソート順序で ORDER BY 句を構成します。 4. **LIMIT 1 の適用**: limitCount を 1 に設定します。 5. **パス探索**: query_planner を呼び出し、最適なインデックススキャンパスを探索します。 6. **パスの確定**: get_cheapest_fractional_path_for_pathkeys を用いて、コストが最小となるパスを選択し、apply_projection_to_path で射影を適用します。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/optimizer/plan/planagg.c 内の preprocess_minmax_aggregates（直接の呼び出し）。
  • **呼び出し先**: copyObject, makeTargetEntry, makeNode, IncrementVarSublevelsUp, query_planner, SS_identify_outer_params, SS_charge_for_initplans, get_cheapest_fractional_path_for_pathkeys, apply_projection_to_path 等。

関連データ構造

  • PlannerInfo: クエリのプランニング状態を保持。
  • MinMaxAggInfo: 最適化対象の集約関数に関するメタデータ。
  • Path: 生成された実行パス。

失敗モードと境界条件

  • get_cheapest_fractional_path_for_pathkeys が適切なパスを見つけられない場合、false を返して終了します。
  • クエリが単一テーブルを参照していない場合や、複雑な結合が含まれる場合は本関数による最適化は適用されません。

保守担当者向けメモ

  • 本関数は PlannerInfo のクローンを作成するため、メモリ管理に注意が必要です。
  • subquery_planner を経由しないため、SS_identify_outer_params 等のクリーンアップ処理を手動で呼び出す必要があります。
