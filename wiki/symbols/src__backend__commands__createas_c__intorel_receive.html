intorel_receive

定義

  • 種別: function
  • 定義箇所: src/backend/commands/createas.c:582

目的

CREATE TABLE AS (CTAS) またはマテリアライズドビュー作成時、クエリ実行エンジンから渡されるタプルをターゲットテーブルへ挿入する。

シグネチャまたは形状

static bool intorel_receive(TupleTableSlot *slot, DestReceiver *self)

入力と出力

  • **入力**:
  • slot: 挿入対象のタプルを保持する TupleTableSlot。
  • self: DR_intorel 構造体としてキャストされる DestReceiver。
  • **出力**:
  • 常に true を返す。

前提条件と不変条件

  • self は DR_intorel 型の構造体であること。
  • ターゲットとなるリレーションは新規作成されたものであり、インデックスは存在しない前提で動作する。

アルゴリズムまたは振る舞い

1. myState->into->skipData を確認し、WITH NO DATA オプションが指定されている場合は挿入処理をスキップする。 2. オプションが指定されていない場合、table_tuple_insert を呼び出し、myState->rel に対してタプルを挿入する。

  • 入力 slot の型とターゲットリレーションの型が一致しない場合でも table_tuple_insert が許容する範囲で処理を行う。

    • 3. 処理完了後、true を返して呼び出し元へ制御を戻す。

呼び出し元と呼び出し先

  • **呼び出し元**: クエリ実行エンジン（ExecutorRun 等の過程で DestReceiver を介して呼び出される）。
  • **呼び出し先**: table_tuple_insert。

関連データ構造

  • DR_intorel: ターゲットリレーション (rel)、挿入オプション (ti_options)、バルク挿入状態 (bistate) を保持する構造体。
  • TupleTableSlot: 挿入されるタプルデータ。

失敗モードと境界条件

  • WITH NO DATA 指定時はタプル挿入を行わず正常終了する。
  • ターゲットリレーションの型と slot の型が異なる場合、効率は低下するが挿入は実行される。

保守担当者向けメモ

  • 本関数は DestReceiver インターフェースの一部として実装されている。
  • table_tuple_insert を使用しているため、テーブルアクセスメソッド (AM) の抽象化レイヤーに依存している。
  • 現在の設計では、挿入対象のテーブルにインデックスが存在しないことを前提とした最適化が行われている。
