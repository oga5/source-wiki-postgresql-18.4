check_index_is_clusterable

定義

  • 種別: function
  • 定義箇所: src/backend/commands/cluster.c:493

目的

指定されたインデックスが、テーブルのクラスタリング（物理的な再構成）に使用可能かどうかを検証します。

シグネチャまたは形状

void check_index_is_clusterable(Relation OldHeap, Oid indexOid, LOCKMODE lockmode)

入力と出力

  • **入力**:
  • OldHeap: クラスタリング対象のテーブルリレーション。
  • indexOid: 検証対象のインデックスのOID。
  • lockmode: インデックスを開く際に使用するロックモード。
  • **出力**: なし。検証に失敗した場合は ereport(ERROR, ...) により処理を中断します。

前提条件と不変条件

  • indexOid で指定されたインデックスは、OldHeap に属している必要があります。
  • インデックスのアクセスメソッドがクラスタリングをサポートしている必要があります。
  • インデックスは完全なインデックス（部分インデックスではない）である必要があります。
  • インデックスは有効（indisvalid が真）である必要があります。

アルゴリズムまたは振る舞い

1. index_open を使用して指定されたインデックスを開きます。 2. インデックスが対象テーブルのインデックスであるかを確認し、不一致の場合はエラーを発生させます。 3. インデックスのアクセスメソッド（rd_indam->amclusterable）を確認し、クラスタリング非対応の場合はエラーを発生させます。 4. インデックスが部分インデックス（indpred が存在）である場合、クラスタリングはサポートされていないためエラーを発生させます。 5. インデックスが有効でない（indisvalid が偽）場合、クラスタリングは安全ではないためエラーを発生させます。 6. 最後に index_close を呼び出し、リレーションキャッシュの参照カウントを解放します（ロックは保持したまま）。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/commands/cluster.c 内のクラスタリング処理フロー。
  • src/backend/commands/tablecmds.c 内の関連処理。
  • **呼び出し先**:
  • index_open
  • RelationGetRelid
  • RelationGetRelationName
  • heap_attisnull
  • index_close
  • ereport

関連データ構造

  • Relation: テーブルおよびインデックスのリレーション記述子。
  • Form_pg_index: インデックスのメタデータ（rd_index 経由で参照）。
  • IndexAmRoutine: インデックスアクセスメソッドの機能定義（rd_indam 経由で参照）。

失敗モードと境界条件

  • 指定されたOIDがインデックスではない、または対象テーブルと無関係な場合にエラーとなります。
  • クラスタリング非対応のアクセスメソッド（例: 一部の特殊なインデックス）を指定した場合にエラーとなります。
  • 部分インデックスや、CREATE INDEX CONCURRENTLY の失敗等により無効化されたインデックスを指定した場合にエラーとなります。

保守担当者向けメモ

  • 本関数は、クラスタリング実行前の安全装置として機能します。
  • 将来的に部分インデックスのクラスタリングを許可する場合、全行をコピーするためのシーケンシャルスキャン処理を追加する必要がありますが、現状ではコストと複雑性の観点から禁止されています。
