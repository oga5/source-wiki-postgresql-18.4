AlteredTableInfo

定義

  • 種別: struct
  • 定義箇所: src/backend/commands/tablecmds.c:168

目的

ALTER TABLE コマンドの実行中、対象となるテーブルごとの変更状態や、フェーズを跨いで引き継ぐべき情報を保持するために使用される構造体です。

シグネチャまたは形状

typedef struct AlteredTableInfo {
    Oid         relid;
    char        relkind;
    TupleDesc   oldDesc;
    Relation    rel;
    List       *subcmds[AT_NUM_PASSES];
    List       *constraints;
    List       *newvals;
    List       *afterStmts;
    bool        verify_new_notnull;
    int         rewrite;
    bool        chgAccessMethod;
    Oid         newAccessMethod;
    Oid         newTableSpace;
    bool        chgPersistence;
    char        newrelpersistence;
    Expr       *partition_constraint;
    bool        validate_default;
    List       *changedConstraintOids;
    List       *changedConstraintDefs;
    List       *changedIndexOids;
    List       *changedIndexDefs;
    char       *replicaIdentityIndex;
    char       *clusterOnIndex;
    List       *changedStatisticsOids;
    List       *changedStatisticsDefs;
} AlteredTableInfo;

入力と出力

本構造体は関数引数として渡され、ALTER TABLE の各フェーズ（Prep, Rewrite Catalogs, Rewrite Tables）で状態の読み書きが行われます。直接的な入力/出力関数はありません。

前提条件と不変条件

  • rel フィールドはフェーズ2（ATRewriteCatalogs）実行中にのみ一時的に設定され、それ以外は NULL であることが期待されます。
  • subcmds は AT_NUM_PASSES 分のリストを持ち、コマンドの実行順序を制御します。

アルゴリズムまたは振る舞い

ALTER TABLE の処理フローにおいて以下の役割を担います： 1. **Phase 1 (Prep)**: コマンドを解析し、本構造体に情報を格納してキュー（wqueue）に積む。 2. **Phase 2 (Rewrite Catalogs)**: システムカタログの更新を行い、必要に応じて rel フィールドを使用してテーブルを開く。 3. **Phase 3 (Rewrite Tables)**: テーブルデータの書き換えや制約の検証を行う。

呼び出し元と呼び出し先

  • **呼び出し元**: ATGetQueueEntry 等により、ALTER TABLE のメイン処理フローで生成・取得されます。
  • **呼び出し先**: ATExecCmd、ATRewriteTable、ATParseTransformCmd、ATExecAddColumn 等の ATExec* 系関数群が本構造体を引数として受け取り、操作を行います。

関連データ構造

  • AlterTableCmd: subcmds リストに含まれる個別のコマンド定義。
  • NewConstraint: constraints リストに含まれる制約情報。
  • NewColumnValue: newvals リストに含まれる列のデフォルト値情報。

失敗モードと境界条件

  • rewrite フィールドにより、強制的なテーブル書き換えが必要な理由が管理されます。
  • newAccessMethod や newTableSpace が 0 の場合は「変更なし」を意味します。

保守担当者向けメモ

  • 本構造体は ALTER TABLE のフェーズ分けを支える中心的なデータ構造です。
  • 複雑な制約（特に外部キー）の追加は Phase 3 での検証が必要となるため、constraints リストの管理には注意が必要です。
  • AT_PASS_* 列挙型による実行順序の制御と、本構造体の subcmds 配列の整合性に留意してください。
