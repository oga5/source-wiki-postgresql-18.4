ginvacuumcleanup

定義

  • 種別: function
  • 定義箇所: src/backend/access/gin/ginvacuum.c:686

目的

GINインデックスのクリーンアップ処理を実行し、保留中の挿入の処理、不要なページの再利用判定、統計情報の更新、およびFSM（Free Space Map）のバキュームを行います。

シグネチャまたは形状

IndexBulkDeleteResult *ginvacuumcleanup(IndexVacuumInfo *info, IndexBulkDeleteResult *stats)

入力と出力

  • **入力**:
  • info: バキューム対象のインデックス情報を含む IndexVacuumInfo 構造体。
  • stats: 統計情報を保持する IndexBulkDeleteResult 構造体（NULLの場合、内部で初期化される）。
  • **出力**:
  • 更新された IndexBulkDeleteResult 構造体へのポインタ。

前提条件と不変条件

  • info->analyze_only が真の場合、AutoVacuumワーカプロセスであれば保留中の挿入を処理し、それ以外は何も行わずに終了します。
  • stats がNULLの場合、ginInsertCleanup を呼び出して保留中の挿入を処理します。
  • インデックスのロックが必要な場合（!RELATION_IS_LOCAL）、拡張時に ExclusiveLock を取得します。

アルゴリズムまたは振る舞い

1. **初期化と保留処理**: analyze_only モードまたは stats がNULLの場合に ginInsertCleanup を実行します。 2. **統計情報の準備**: インデックスのタプル数や推定カウントを stats に設定します。 3. **全ページ走査**: インデックスの全ページを GIN_ROOT_BLKNO から順に走査します。

  • vacuum_delay_point を呼び出し、バキュームの遅延を制御します。
  • GinPageIsRecyclable で再利用可能と判定されたページを RecordFreeIndexPage でFSMに返却します。
  • 再利用不可の場合、ページ種別（データページかエントリページか）に応じて idxStat を更新します。

    • 4. **メタデータ更新**: ginUpdateStats を呼び出し、集計したページ数やエントリ数をメタページに反映します。 5. **FSMバキューム**: IndexFreeSpaceMapVacuum を呼び出し、FSMを整理します。 6. **終了処理**: 最終的なページ数を取得し、結果を返します。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/access/gin/ginutil.c: amroutine->amvacuumcleanup として登録。
  • src/backend/commands/analyze.c: ANALYZE処理の一部として参照。
  • **呼び出し先**:
  • ginInsertCleanup, initGinState, LockRelationForExtension, RelationGetNumberOfBlocks, ReadBufferExtended, LockBuffer, GinPageIsRecyclable, RecordFreeIndexPage, GinPageIsData, GinPageIsList, GinPageIsLeaf, PageGetMaxOffsetNumber, UnlockReleaseBuffer, ginUpdateStats, IndexFreeSpaceMapVacuum。

関連データ構造

  • IndexVacuumInfo: バキュームの制御パラメータ。
  • IndexBulkDeleteResult: バキュームの結果統計。
  • GinState: GINインデックスの状態保持用。
  • GinStatsData: インデックスの統計集計用。

失敗モードと境界条件

  • stats がNULLの場合、メモリ割り当て（palloc0）が失敗する可能性があります。
  • インデックスが空の場合、走査範囲はルートのみとなります。
  • GinPageIsRecyclable の判定により、ルートページは再利用対象から除外されます。

保守担当者向けメモ

  • ginvacuumcleanup は VACUUM ANALYZE 等のクリーンアップフェーズの主エントリポイントです。
  • XXX コメントにある通り、インデックスのタプル数はヒープのタプル数で代用しており、部分インデックス等の場合には正確な値ではない点に注意してください。
  • needLock フラグによるロック制御は、ローカルインデックスと共有インデックスの差異を吸収するために使用されています。
