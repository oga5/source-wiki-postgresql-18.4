ExecMakeTableFunctionResult

定義

  • 種別: function
  • 定義箇所: src/backend/executor/execSRF.c:100

目的

集合を返す関数（SRF）を実行し、その結果を Tuplestore にマテリアライズ（蓄積）する。呼び出し元が反復的に結果を取得できるように、関数呼び出しのプロトコル（ValuePerCall）を抽象化する。

シグネチャまたは形状

Tuplestorestate *ExecMakeTableFunctionResult(SetExprState *setexpr,
                                            ExprContext *econtext,
                                            MemoryContext argContext,
                                            TupleDesc expectedDesc,
                                            bool randomAccess)

入力と出力

  • **入力**:
  • setexpr: SRF の実行状態を保持する構造体。
  • econtext: 式評価用のコンテキスト。
  • argContext: 引数評価および関数呼び出しの生存期間を管理するメモリコンテキスト。
  • expectedDesc: 期待される結果の行記述子。
  • randomAccess: Tuplestore でのランダムアクセスを許可するかどうか。
  • **出力**:
  • 結果を格納した Tuplestorestate へのポインタ。

前提条件と不変条件

  • argContext は、関数呼び出しのループ全体で生存する必要がある（econtext の per-tuple コンテキストとは異なる）。
  • argContext は、関数呼び出しのたびにリセットされる。
  • 関数が ValuePerCall モードの場合、ReturnSetInfo プロトコルに従い、isDone が ExprEndResult になるまでループ処理を行う。

アルゴリズムまたは振る舞い

1. **初期化**: argContext をリセットし、メモリコンテキストを切り替える。ReturnSetInfo 構造体を初期化し、関数呼び出しのメタデータを準備する。 2. **引数評価**: ExecEvalFuncArgs を呼び出し、関数に渡す引数を評価する。関数が strict で引数に NULL が含まれる場合は処理をスキップする。 3. **実行ループ**:

  • econtext->ecxt_per_tuple_memory をリセットし、メモリリークを防ぐ。
  • FunctionCallInvoke を使用して関数を呼び出す。
  • rsinfo.returnMode が SFRM_ValuePerCall の場合、結果を Tuplestore に蓄積する。
  • rsinfo.isDone が ExprEndResult に達するまで繰り返す。

    • 4. **終了**: 蓄積された Tuplestorestate を呼び出し元に返す。

呼び出し元と呼び出し先

  • **呼び出し元**: nodeFunctionscan.c（FROM 句での SRF 実行）、functions.c など。
  • **呼び出し先**: ExecEvalFuncArgs, FunctionCallInvoke, tuplestore_begin_heap, ExecEvalExpr（elidedFuncState の場合）。

関連データ構造

  • SetExprState: SRF の実行状態。
  • ReturnSetInfo: SRF と実行エンジン間の通信用構造体。
  • Tuplestorestate: 結果を一時的に保持するストレージ。
  • ExprContext: 式評価のコンテキスト。

失敗モードと境界条件

  • 関数が strict で引数が NULL の場合、結果は返されない。
  • randomAccess が指定された場合、Tuplestore はランダムアクセス可能なモードで初期化される。
  • elidedFuncState が存在する場合、通常の SRF ではなく汎用的な式として評価される。

保守担当者向けメモ

  • argContext と econtext->ecxt_per_tuple_memory の役割分担を維持すること。前者は関数呼び出しの生存期間を管理し、後者は各呼び出しのメモリリークをクリーンアップする。
  • ReturnSetInfo の isDone フラグの管理がプロトコル遵守の鍵となる。
