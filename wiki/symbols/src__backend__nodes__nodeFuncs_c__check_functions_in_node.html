check_functions_in_node

定義

  • 種別: function
  • 定義箇所: src/backend/nodes/nodeFuncs.c:1905

目的

ノードツリーに含まれる関数呼び出しや演算子に関連する OID を抽出し、指定されたコールバック関数を用いてそれらの関数が特定の条件（可変性やユーザー定義など）を満たすかを判定する。

シグネチャまたは形状

bool check_functions_in_node(Node *node, check_function_callback checker, void *context)

入力と出力

  • **入力**:
  • node: 走査対象のノードツリー。
  • checker: 各ノードから抽出された関数 OID を引数に取り、bool を返すコールバック関数。
  • context: コールバック関数に渡される任意のデータポインタ。
  • **出力**:
  • bool: コールバック関数が true を返した場合は直ちに true を返し、ツリー全体を走査しても条件に合致しない場合は false を返す。

前提条件と不変条件

  • node は有効なノード構造体である必要がある。
  • checker は関数 OID を受け取り、条件判定を行う関数であること。
  • OpExpr や ScalarArrayOpExpr のような演算子ノードに対しては、内部で set_opfuncid 等を呼び出し、関数 OID が解決済みであることを保証する。

アルゴリズムまたは振る舞い

1. nodeTag(node) に基づき、ノードの種類を判定する。 2. Aggref, WindowFunc, FuncExpr の場合は、直接保持している関数 OID を checker に渡す。 3. OpExpr, DistinctExpr, NullIfExpr の場合は、set_opfuncid を呼び出して OID を解決してから checker を実行する。 4. ScalarArrayOpExpr の場合は、set_sa_opfuncid を呼び出して OID を解決してから checker を実行する。 5. CoerceViaIO の場合は、結果型の入力関数と引数型の出力関数の両方を checker で検証する。 6. RowCompareExpr の場合は、保持する演算子リストを走査し、各演算子に対応する関数 OID を get_opcode で取得して checker を実行する。 7. 上記以外のノード型は無視される。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/catalog/heap.c
  • src/backend/commands/publicationcmds.c
  • src/backend/optimizer/util/clauses.c
  • **呼び出し先**:
  • checker (コールバック関数)
  • set_opfuncid
  • set_sa_opfuncid
  • getTypeInputInfo / getTypeOutputInfo
  • get_opcode
  • exprType

関連データ構造

  • Node: 走査対象の基底構造体。
  • Aggref, WindowFunc, FuncExpr, OpExpr, ScalarArrayOpExpr, CoerceViaIO, RowCompareExpr: 処理対象となる具体的なノード型。

失敗モードと境界条件

  • node が NULL の場合、あるいはサポートされていないノード型が渡された場合は、何もせず false を返す。
  • checker が true を返した時点で走査を中断する。
  • 演算子の関数 ID が解決できない場合、set_opfuncid 等の内部実装に依存する。

保守担当者向けメモ

  • 新しいノード型が関数呼び出しを含むようになった場合、この関数の switch 文にケースを追加する必要がある。
  • この関数は再帰的な走査を行わない（単一ノードのチェックのみ）。ツリー全体を走査する場合は、expression_tree_walker を併用する設計が一般的である。
