sepgsql_utility_command

定義

  • 種別: function
  • 定義箇所: contrib/sepgsql/hooks.c:315

目的

PostgreSQL の ProcessUtility_hook として機能し、ユーティリティコマンド実行時のセキュリティコンテキストの管理および特定のコマンドに対するアクセス制御を行います。

シグネチャまたは形状

static void sepgsql_utility_command(
    PlannedStmt *pstmt,
    const char *queryString,
    bool readOnlyTree,
    ProcessUtilityContext context,
    ParamListInfo params,
    QueryEnvironment *queryEnv,
    DestReceiver *dest,
    QueryCompletion *qc
);

入力と出力

  • **入力**: 実行対象の PlannedStmt およびユーティリティコマンドの実行に必要なコンテキスト情報。
  • **出力**: なし（副作用として、LOAD コマンドの拒否や、後続のフック/標準処理への委譲を行う）。

前提条件と不変条件

  • sepgsql_context_info グローバル変数を一時的に保存・復元し、コマンド実行中のみ現在の nodeTag を保持します。
  • PG_TRY / PG_FINALLY ブロックにより、エラー発生時でも必ず sepgsql_context_info が元の状態に復元されることが保証されています。

アルゴリズムまたは振る舞い

1. 現在の sepgsql_context_info を保存します。 2. pstmt->utilityStmt のノードタイプを確認します。

  • T_CreatedbStmt: テンプレートデータベース名を sepgsql_context_info に保存します。
  • T_LoadStmt: SELinux が強制モード（enforcing mode）の場合、LOAD コマンドをエラーとして拒否します。

    • 3. next_ProcessUtility_hook が存在する場合はそれを呼び出し、存在しない場合は standard_ProcessUtility を呼び出して処理を継続します。 4. 処理終了後、保存しておいた sepgsql_context_info を復元します。

呼び出し元と呼び出し先

  • **呼び出し元**: PostgreSQL コアの ProcessUtility フック機構（contrib/sepgsql/hooks.c:482 にてフックとして登録）。
  • **呼び出し先**:
  • next_ProcessUtility_hook（連鎖している場合）
  • standard_ProcessUtility（連鎖がない場合）

関連データ構造

  • sepgsql_context_info_t: コマンド実行中のメタデータ（コマンドタイプやテンプレート名）を保持する構造体。
  • PlannedStmt: 実行計画を保持するコア構造体。

失敗モードと境界条件

  • sepgsql_getenforce() が真の状態で LOAD コマンドが実行された場合、ERRCODE_INSUFFICIENT_PRIVILEGE でエラーを発生させます。
  • 他のユーティリティコマンドについては、詳細な解析が必要なため、本関数内では明示的な権限チェックを行わず、後続の処理に委ねています。

保守担当者向けメモ

  • LOAD コマンドを禁止している理由は、バイナリモジュールによるフックの不正な上書きを防ぐためです。
  • フックの連鎖（next_ProcessUtility_hook）は、他のセキュリティ拡張機能との共存を可能にするために必須です。
