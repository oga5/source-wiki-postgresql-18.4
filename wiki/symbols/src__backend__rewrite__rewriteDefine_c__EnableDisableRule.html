EnableDisableRule

定義

  • 種別: function
  • 定義箇所: src/backend/rewrite/rewriteDefine.c:690

目的

指定されたリレーションに関連付けられた書き換えルール（Rewrite Rule）の有効/無効状態を更新し、システムカタログへの反映とリレーションキャッシュの無効化を行う。

シグネチャまたは形状

void EnableDisableRule(Relation rel, const char *rulename, char fires_when)

入力と出力

  • rel: 対象リレーションのオープン済みリレーション記述子。
  • rulename: 更新対象のルール名。
  • fires_when: 設定する新しい状態（ev_enabled カラムの値）。
  • 出力: なし（副作用としてシステムカタログおよびキャッシュを更新）。

前提条件と不変条件

  • rel は適切なロック（AccessExclusiveLock 等）で保護されている必要がある。
  • 呼び出し元は、対象リレーションの所有者権限を持っている必要がある。
  • ルール名は対象リレーションに対して一意であること。

アルゴリズムまたは振る舞い

1. pg_rewrite システムカタログを RowExclusiveLock で開く。 2. RULERELNAME システムキャッシュを使用して、指定されたルールを検索する。存在しない場合はエラーを報告する。 3. object_ownercheck を呼び出し、現在のユーザーがリレーションの所有者であることを確認する。 4. ルールの現在の状態（ev_enabled）と fires_when を比較する。 5. 状態が異なる場合、CatalogTupleUpdate を使用して pg_rewrite を更新し、changed フラグを立てる。 6. InvokeObjectPostAlterHook を呼び出し、変更を通知する。 7. 変更があった場合、CacheInvalidateRelcache(rel) を呼び出し、全バックエンドのリレーションキャッシュを無効化する。

呼び出し元と呼び出し先

  • src/backend/commands/tablecmds.c:17227
  • 呼び出し先:

関連データ構造

  • Relation: リレーション記述子。
  • Form_pg_rewrite: pg_rewrite カタログのタプル構造。
  • HeapTuple: システムカタログのタプル表現。

失敗モードと境界条件

  • 指定されたルール名がリレーション内に存在しない場合、ERRCODE_UNDEFINED_OBJECT でエラー終了する。
  • 呼び出し元がリレーションの所有者でない場合、aclcheck_error によりエラー終了する。

保守担当者向けメモ

  • 本関数はカタログを直接更新するため、呼び出し側で適切なロックが確保されていることを前提としている。
  • CacheInvalidateRelcache の呼び出しは必須であり、これを省略すると他のバックエンドが古いルール定義を参照し続けるため注意が必要。
