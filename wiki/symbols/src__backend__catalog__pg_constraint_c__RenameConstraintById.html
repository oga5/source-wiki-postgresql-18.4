RenameConstraintById

定義

  • 種別: function
  • 定義箇所: src/backend/catalog/pg_constraint.c:1002

目的

システムカタログ pg_constraint 内の既存の制約名を新しい名前に変更する。

シグネチャまたは形状

void RenameConstraintById(Oid conId, const char *newname)

入力と出力

  • conId (Oid): 名前を変更する制約のOID。
  • newname (const char *): 新しい制約名。
  • 出力: なし（成功時はカタログを更新し、失敗時はエラーをスローする）。

前提条件と不変条件

  • 呼び出し元は、対象の制約に対して適切なロック（RowExclusiveLock 等）を保持している必要がある。
  • 指定された newname が、対象のテーブルまたはドメインにおいて既に存在する場合はエラーとなる。

アルゴリズムまたは振る舞い

1. pg_constraint テーブルを RowExclusiveLock でオープンする。 2. CONSTROID システムキャッシュを使用して、対象制約のタプルをコピーとして取得する。見つからない場合はエラーを発生させる。 3. 制約がテーブルに属している場合、ConstraintNameIsUsed を用いて newname がそのテーブル内で重複していないか確認する。重複時は ERRCODE_DUPLICATE_OBJECT でエラーとする。 4. 制約がドメインに属している場合も同様に重複チェックを行う。 5. namestrcpy を使用してタプル内の conname を更新する。 6. CatalogTupleUpdate を呼び出し、カタログを更新する。 7. InvokeObjectPostAlterHook を呼び出し、変更を通知する。 8. 取得したタプルを解放し、テーブルをクローズする。

呼び出し元と呼び出し先

  • 呼び出し元: src/backend/commands/tablecmds.c 内の ALTER TABLE 関連処理から呼び出される。
  • 呼び出し先: table_open, SearchSysCacheCopy1, ConstraintNameIsUsed, CatalogTupleUpdate, InvokeObjectPostAlterHook, heap_freetuple, table_close 等。

関連データ構造

  • pg_constraint: 制約情報を保持するシステムカタログ。
  • Form_pg_constraint: pg_constraint 行の構造体。

失敗モードと境界条件

  • SearchSysCacheCopy1 で制約が見つからない場合、elog(ERROR) により処理が中断される。
  • ConstraintNameIsUsed により名前の重複が検出された場合、ereport(ERROR) により処理が中断される。

保守担当者向けメモ

  • 本関数は pg_constraint の行を直接更新するため、カタログの整合性を維持するために適切なロック管理が必須である。
  • NOT NULL 制約など、特殊な制約の更新が必要な場合は、本関数が適切に処理できるか確認すること。
