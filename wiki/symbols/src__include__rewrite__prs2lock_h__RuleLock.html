RuleLock

定義

  • 種別: struct
  • 定義箇所: src/include/rewrite/prs2lock.h:40

目的

リレーションに適用される書き換えルール（RewriteRule）の集合を保持・管理するためのコンテナ構造体です。名称に「Lock」とありますが、マルチスレッドの排他制御を行う同期プリミティブではなく、ルールシステムにおけるデータ保持の仕組みを指します。

シグネチャまたは形状

typedef struct RuleLock
{
	int			numLocks;
	RewriteRule **rules;
} RuleLock;

入力と出力

  • **入力**: リレーションキャッシュ（RelCache）構築時、システムカタログ pg_rewrite から読み出されたルール情報が本構造体にマッピングされます。
  • **出力**: クエリ書き換え処理（rewriteHandler）において、リレーションに関連付けられたルール群として参照されます。

前提条件と不変条件

  • rules は RewriteRule へのポインタ配列を指します。
  • リレーションごとに生成され、リレーションキャッシュの一部として保持されます。

アルゴリズムまたは振る舞い

  • リレーションが読み込まれる際、カタログ情報から動的に構築されます。
  • クエリ書き換え時、この構造体を介して適用すべきルールが特定され、クエリツリーの変換が行われます。
  • equalRuleLocks 関数（src/backend/utils/cache/relcache.c）により、二つの RuleLock が等価であるかの比較が可能です。

呼び出し元と呼び出し先

  • **呼び出し元（参照元）**:
  • src/backend/rewrite/rewriteHandler.c: クエリ書き換えロジック内で参照。
  • src/backend/utils/adt/tid.c: 参照あり。
  • src/backend/utils/cache/relcache.c: リレーションキャッシュの構築および管理（MemoryContextAlloc によるメモリ確保など）で使用。
  • **呼び出し先**:
  • RewriteRule: 本構造体は RewriteRule のポインタ配列を保持します。

関連データ構造

  • RewriteRule: 個別の書き換えルールを定義する構造体。
  • Relation: リレーションキャッシュにおいて rd_rules フィールドとして本構造体を保持します。

失敗モードと境界条件

  • ルール数が多いテーブルでは、本構造体が保持するポインタ配列のサイズが大きくなるため、メモリ消費量に注意が必要です。
  • 本構造体自体は排他制御を行わないため、リレーションキャッシュの整合性管理は上位のキャッシュ機構に依存します。

保守担当者向けメモ

  • 名称の「Lock」は歴史的経緯によるものであり、排他制御用ではない点に注意してください。
  • src/backend/utils/cache/relcache.c におけるリレーションキャッシュの構築プロセスが、本構造体のライフサイクルを理解する上で最も重要です。
