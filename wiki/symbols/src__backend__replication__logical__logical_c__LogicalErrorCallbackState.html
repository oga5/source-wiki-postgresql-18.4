LogicalErrorCallbackState

定義

  • 種別: struct
  • 定義箇所: src/backend/replication/logical/logical.c:50

目的

論理デコーディングのプラグインコールバック実行時にエラーが発生した際、エラーコンテキスト（どのプラグインのどの処理で失敗したか）を詳細に報告するための状態保持構造体です。

シグネチャまたは形状

typedef struct LogicalErrorCallbackState
{
	LogicalDecodingContext *ctx;
	const char *callback_name;
	XLogRecPtr	report_location;
} LogicalErrorCallbackState;

入力と出力

  • **入力**: LogicalDecodingContext へのポインタ、実行中のコールバック名、およびエラー報告対象の WAL 位置（XLogRecPtr）。
  • **出力**: なし（本構造体はエラー発生時の error_context_stack に登録される情報源として機能します）。

前提条件と不変条件

  • ctx は有効な LogicalDecodingContext を指している必要があります。
  • callback_name は静的な文字列定数であることが期待されます。

アルゴリズムまたは振る舞い

論理デコーディングの各コールバックラッパー（commit_cb_wrapper 等）内で、エラー発生時に詳細な情報を付与するために使用されます。具体的には、コールバック実行前にこの構造体をスタックにプッシュし、実行後にポップすることで、PostgreSQL のエラーハンドリング機構が失敗時に「どのプラグインのどのコールバックで、どの LSN の処理中にエラーが起きたか」をログに出力できるようにします。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/replication/logical/logical.c 内の各コールバックラッパー関数（commit_cb_wrapper, change_cb_wrapper 等）。
  • **呼び出し先**: 直接的な関数呼び出しは行いません。エラー発生時に PostgreSQL のエラー報告システムによって参照されます。

関連データ構造

  • LogicalDecodingContext: デコーディング全体の状態を保持する構造体。

失敗モードと境界条件

  • ctx が NULL の状態でエラーが発生した場合、エラー報告時に不正なメモリ参照が発生する可能性があります。
  • report_location が無効な LSN（InvalidXLogRecPtr）である場合、エラーログには位置情報が含まれません。

保守担当者向けメモ

  • 本構造体は error_context_stack に登録されるため、コールバックの実行が終了した後は必ずスタックから除去される必要があります。
  • 呼び出し元として logical.c 内の複数のラッパー関数が本構造体をスタック上に確保して利用しています。新しいコールバックラッパーを追加する際は、既存のパターンに従って本構造体を利用してください。
