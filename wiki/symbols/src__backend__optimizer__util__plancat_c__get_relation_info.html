get_relation_info

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/util/plancat.c:116

目的

プランナがクエリ最適化を行うために必要なリレーションの物理的特性（属性、インデックス、統計情報等）を収集し、RelOptInfo 構造体を初期化する。

シグネチャまたは形状

void get_relation_info(PlannerInfo *root, Oid relationObjectId, bool inhparent, RelOptInfo *rel)

入力と出力

  • **入力**:
  • root: プランナのグローバル情報。
  • relationObjectId: 対象リレーションの OID。
  • inhparent: 継承ツリーの親テーブルであるかを示すフラグ。
  • rel: 初期化対象の RelOptInfo 構造体。
  • **出力**: なし（rel 構造体の各フィールドを直接更新する）。

前提条件と不変条件

  • 対象リレーションは、呼び出し前に既にロックが取得されている必要がある（table_open は NoLock で実行される）。
  • rel->max_attr >= rel->min_attr であること。
  • 属性の attnullability は ATTNULLABLE_UNKNOWN であってはならない。

アルゴリズムまたは振る舞い

1. table_open でリレーションを開き、テーブルAMの存在確認およびリカバリ中のアクセス制限チェックを行う。 2. RelOptInfo の属性範囲（min_attr, max_attr）およびテーブルスペース情報を設定する。 3. notnullattnums を構築する（継承親またはパーティションテーブルの場合）。 4. 継承親でない場合、estimate_rel_size を呼び出してページ数やタプル数を推定する。 5. RelationGetParallelWorkers で並列ワーカー数を取得する。 6. インデックス情報を収集する：

  • RelationGetIndexList でインデックス一覧を取得。
  • 各インデックスに対し、indisvalid（有効性）および indcheckxmin（トランザクション可視性）を検証。
  • 有効なインデックスに対して IndexOptInfo を生成し、詳細情報を格納する。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/optimizer/util/relnode.c (build_simple_rel)
  • **呼び出し先**:
  • table_open, RelationGetNumberOfAttributes, RelationGetForm, RelationIsPermanent, RecoveryInProgress, RelationGetParallelWorkers, RelationGetIndexList, index_open, index_close, estimate_rel_size, makeNode, palloc, bms_add_member, ereport 等。

関連データ構造

  • RelOptInfo: プランナがリレーションを表現するために使用する主要構造体。
  • IndexOptInfo: インデックスの特性を保持する構造体。
  • PlannerInfo: クエリ全体のプランニング情報を保持。

失敗モードと境界条件

  • **エラー**:
  • テーブルAMが存在しないリレーション（かつ特定の relkind 以外）を開こうとした場合。
  • リカバリ中に一時テーブルや未ログ（unlogged）テーブルにアクセスしようとした場合。
  • **境界条件**:
  • 継承親テーブルの場合、サイズ推定はスキップされる（set_append_rel_size で後続処理されるため）。
  • 無効なインデックスや、特定のトランザクション条件下で利用不可なインデックスは無視される。

保守担当者向けメモ

  • 本関数はカタログアクセスの窓口であり、ロック管理は呼び出し元に依存する。
  • get_relation_info_hook が存在する場合、プラグインによる介入が可能である点に留意すること。
  • インデックスの述語や属性情報の書き換え（varno の調整）については、プランナの他のユーティリティ関数との連携が必要となる。
