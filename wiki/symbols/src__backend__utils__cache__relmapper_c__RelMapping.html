RelMapping

定義

  • 種別: struct
  • 定義箇所: src/backend/utils/cache/relmapper.c:83

目的

PostgreSQLの「マッピングされたカタログ」において、特定のカタログのOIDと、それに対応する物理ファイル番号（RelFileNumber）のペアを保持するために使用されます。

シグネチャまたは形状

typedef struct RelMapping
{
    Oid            mapoid;          /* カタログのOID */
    RelFileNumber  mapfilenumber;   /* 対応するリレーションファイル番号 */
} RelMapping;

入力と出力

本構造体はデータ保持用であり、関数ではありません。入力としてOIDとファイル番号のペアを格納し、出力としてそれらの値を保持します。

前提条件と不変条件

  • mapoid はシステムカタログを識別する有効なOIDである必要があります。
  • mapfilenumber は物理的なファイル位置を特定するための有効な番号である必要があります。

アルゴリズムまたは振る舞い

本構造体自体はロジックを持ちませんが、RelMapFile 構造体内の mappings 配列の要素として利用されます。システムはこれを用いて、通常の pg_class 参照では解決できないカタログの物理ファイル位置を特定します。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/utils/cache/relmapper.c 内のマップファイル管理ロジックが、この構造体の配列を操作します。
  • **呼び出し先**: 情報なし。

関連データ構造

  • RelMapFile: RelMapping の配列（mappings）を保持する上位の構造体。
  • MAX_MAPPINGS: RelMapFile 内の mappings 配列のサイズを定義する定数。

失敗モードと境界条件

  • mappings 配列のサイズ（MAX_MAPPINGS）を超えるマッピングは保持できません。
  • マップファイルが破損している場合、この構造体を通じて読み込まれるデータも無効となり、システム起動時に致命的なエラー（FATAL）が発生します。

保守担当者向けメモ

  • 本構造体は pg_filenode.map ファイルのメモリ上での表現の一部です。変更を加える際は、マップファイルのシリアライズ/デシリアライズ処理およびCRCチェックとの整合性に注意してください。
