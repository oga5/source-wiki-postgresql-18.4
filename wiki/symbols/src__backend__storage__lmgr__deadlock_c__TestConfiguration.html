TestConfiguration

定義

  • **種別**: function
  • **定義箇所**: src/backend/storage/lmgr/deadlock.c:377

目的

現在のロック制約セットに基づき、デッドロックの有無を判定し、解決可能な「ソフトエッジ（解消可能な競合）」が存在するかを評価します。

シグネチャまたは形状

static int TestConfiguration(PGPROC *startProc)

入力と出力

  • **入力**: startProc (デッドロック検出の起点となるプロセス)
  • **出力**:
  • 0: デッドロックなし。
  • >0: 検出されたソフトエッジの数。
  • -1: ハードデッドロック（解決不能）の検出、または制約の不整合、メモリ不足。

前提条件と不変条件

  • ExpandConstraints が現在の制約セットに対して整合性を保っていること。
  • nPossibleConstraints および maxPossibleConstraints が作業領域として十分な容量を確保していること。

アルゴリズムまたは振る舞い

1. **メモリ確認**: 作業領域が不足している場合は -1 を返す。 2. **制約展開**: ExpandConstraints を呼び出し、現在の制約セットを待ち順序に展開する。不整合があれば -1 を返す。 3. **サイクル探索**: 以下の順序で FindLockCycle を実行し、ロック待ちグラフの循環を探索する。

  • 現在の制約に含まれる各プロセスの waiter および blocker。
  • 起点となる startProc。

    • 4. **判定**:

  • サイクルが見つかり、かつソフトエッジが 0 の場合は「ハードデッドロック」とみなし -1 を返す。
  • ソフトエッジが見つかった場合はその数を保持し、最終的に返却する。

呼び出し元と呼び出し先

  • **呼び出し元**: DeadLockCheck (同ファイル内)
  • **呼び出し先**: ExpandConstraints, FindLockCycle

関連データ構造

  • PGPROC: ロック待ち状態を保持するプロセス構造体。
  • EDGE: 待ちグラフ上のエッジ情報。
  • curConstraints: 現在適用されている制約のリスト。

失敗モードと境界条件

  • **ハードデッドロック**: 解決不可能な循環参照が検出された場合、-1 を返して処理を中断する。
  • **メモリ不足**: nPossibleConstraints + MaxBackends > maxPossibleConstraints の場合、処理を継続できず -1 を返す。

保守担当者向けメモ

  • 本関数は DeadLockCheck から呼び出され、ロック待ち行列の再構成の可否を判断する重要なステップです。
  • FindLockCycle の戻り値が nSoftEdges == 0 の場合にハードデッドロックと判定するロジックは、デッドロック解消アルゴリズムの根幹です。
