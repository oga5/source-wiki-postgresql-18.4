check_index_page

定義

  • 種別: function
  • 定義箇所: contrib/amcheck/verify_gin.c:663

目的

GINインデックスのページヘッダーおよびページ構造の物理的な整合性を検証する。

シグネチャまたは形状

static void check_index_page(Relation rel, Buffer buffer, BlockNumber blockNo)

入力と出力

  • **入力**:
  • rel: 検証対象のインデックスリレーション。
  • buffer: 検証対象のページを含むバッファ。
  • blockNo: 検証対象のブロック番号。
  • **出力**: なし。検証失敗時は ereport(ERROR, ...) により例外を発生させる。

前提条件と不変条件

  • ReadBuffer により、ページヘッダーが有効（PageHeaderIsValid）または全ゼロであることが保証されている必要がある。
  • ページが全ゼロ（PageIsNew）ではないこと。
  • ページの特殊領域（Special Area）のサイズが GinPageOpaqueData のサイズと一致していること。

アルゴリズムまたは振る舞い

1. **ゼロページチェック**: ページが PageIsNew の場合、破損とみなしてエラーを報告する。 2. **特殊領域チェック**: PageGetSpecialSize を確認し、GINインデックスの不透明データ構造とサイズが一致しない場合はエラーを報告する。 3. **削除済みページチェック**: ページが削除済み（GinPageIsDeleted）の場合、以下の不変条件を検証する。

  • 内部ページ（非リーフ）であってはならない。
  • タプル（MaxOffsetNumber）を含んでいてはならない。

    • 4. **通常ページチェック**: 削除済みでない場合、ページ内のタプル数が MaxIndexTuplesPerPage を超えていないかを確認する。

呼び出し元と呼び出し先

  • **呼び出し元**: contrib/amcheck/verify_gin.c 内のインデックス走査プロセス（例: gin_check_parent_keys_consistency 等のループ内）。
  • **呼び出し先**: BufferGetPage, PageIsNew, RelationGetRelationName, BufferGetBlockNumber, PageGetSpecialSize, GinPageIsDeleted, GinPageIsLeaf, PageGetMaxOffsetNumber 等のアクセサ関数。

関連データ構造

  • Relation: インデックスリレーション。
  • Buffer: ページバッファ。
  • GinPageOpaqueData: GINインデックスのページヘッダー末尾に配置される不透明データ。

失敗モードと境界条件

  • **ERRCODE_INDEX_CORRUPTED**: 以下のいずれかの場合に発生する。
  • ページが全ゼロである。
  • 特殊領域のサイズが不正である。
  • 削除済みページが内部ページである、またはタプルを保持している。
  • ページ内のタプル数が許容上限を超えている。

保守担当者向けメモ

  • 本関数は物理的なページ構造の健全性を確認する低レベルな検証を行う。論理的なツリー構造の検証は、呼び出し元の関数群（gin_check_parent_keys_consistency 等）が担当する。
  • 破損検知時には REINDEX を推奨するエラーヒントを付与すること。
