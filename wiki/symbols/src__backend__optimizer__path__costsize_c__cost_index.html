cost_index

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/path/costsize.c:559

目的

インデックススキャンパス（IndexPath）の実行コスト（startup_cost, total_cost）および行数推定値を算出し、対象パス構造体に設定します。

シグネチャまたは形状

void cost_index(IndexPath *path, PlannerInfo *root, double loop_count, bool partial_path)

入力と出力

  • **入力**:
  • path: コスト計算対象の IndexPath 構造体。
  • root: プランナのグローバル情報。
  • loop_count: スキャンが繰り返される回数。
  • partial_path: 並列クエリの一部であるかを示すフラグ。
  • **出力**:
  • path 構造体の path.rows, path.startup_cost, path.total_cost, indextotalcost, indexselectivity フィールドを更新。

前提条件と不変条件

  • baserel は RelOptInfo であり、index は IndexOptInfo であること。
  • baserel->relid > 0 かつ baserel->rtekind == RTE_RELATION であること。

アルゴリズムまたは振る舞い

1. **行数とクエリ条件の特定**: path->path.param_info の有無に応じて行数を設定し、インデックスで処理できない条件（qpquals）を抽出します。 2. **インデックスアクセス手法のコスト推定**: index->amcostestimate を呼び出し、インデックス自体の読み込みコスト、選択率、相関などを取得します。 3. **I/Oコストの算出**:

  • Mackert & Lohmanのモデルに基づき、インデックスの相関や loop_count を考慮してヒープページへのアクセス数を推定します。
  • インデックスのみのスキャン（Index-Only Scan）の場合、可視性マップ（Visibility Map）を考慮してヒープフェッチ数を削減します。
  • 完全に相関がある場合とない場合でコストを補間し、最終的なI/Oコストを決定します。

関連データ構造

  • IndexPath: コスト計算の対象となる構造体。
  • PlannerInfo: プランナのコンテキスト。
  • IndexOptInfo: インデックスの統計情報。

失敗モードと境界条件

  • loop_count > 1 の場合、繰り返し実行時のキャッシュ効果を考慮した計算が行われます。
  • インデックスのみのスキャン（T_IndexOnlyScan）時は、ヒープアクセスが抑制されます。

保守担当者向けメモ

  • 本関数はMackert & Lohmanのモデルに基づいた複雑なI/Oコスト計算を含みます。
  • indexCorrelation を用いたコストの線形補間については、その妥当性について議論の余地がある旨のコメントがソース内に存在します。
