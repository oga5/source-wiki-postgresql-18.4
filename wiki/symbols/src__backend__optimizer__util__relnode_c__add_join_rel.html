add_join_rel

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/util/relnode.c:626

目的

新しく作成された結合リレーション（RelOptInfo）を、プランナの管理リストおよび補助的なハッシュテーブルに登録する。

シグネチャまたは形状

static void add_join_rel(PlannerInfo *root, RelOptInfo *joinrel)

入力と出力

  • root: 現在のクエリプランニング状態を保持する PlannerInfo 構造体へのポインタ。
  • joinrel: 登録対象となる結合リレーションの RelOptInfo 構造体。
  • 出力: なし（副作用として root 内のデータ構造を更新）。

前提条件と不変条件

  • joinrel は既に初期化済みであること。
  • GEQO（遺伝的クエリ最適化）を使用する場合、リストの末尾に追加される必要がある。
  • ハッシュテーブルが存在する場合、登録しようとする joinrel の relids が既にハッシュテーブル内に存在していてはならない（Assert(!found) による検証）。

アルゴリズムまたは振る舞い

1. root->join_rel_list の末尾に joinrel を追加する。 2. root->join_rel_hash が存在する場合：

  • joinrel->relids をキーとして hash_search を実行し、エントリを確保する。
  • 確保した JoinHashEntry の join_rel フィールドに joinrel を格納する。

呼び出し元と呼び出し先

  • 呼び出し元: src/backend/optimizer/util/relnode.c 内の build_join_rel 関数（848行目、997行目付近）。
  • 呼び出し先: lappend（リスト操作）、hash_search（ハッシュテーブル操作）。

関連データ構造

  • PlannerInfo: プランナのグローバル状態。
  • RelOptInfo: 結合リレーションの定義。
  • JoinHashEntry: ハッシュテーブル内のエントリ構造体。

失敗モードと境界条件

  • root->join_rel_hash が既に存在し、かつ同じ relids を持つリレーションが既に登録されている場合、Assert により停止する。

保守担当者向けメモ

  • 本関数は join_rel_list と join_rel_hash の同期を維持する役割を持つ。ハッシュテーブルは動的に作成されるため、呼び出し側で root->join_rel_hash の有無を確認する必要はない（本関数内で判定済み）。
