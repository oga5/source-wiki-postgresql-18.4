XLogRecoveryCtlData

定義

  • 種別: struct
  • 定義箇所: src/backend/access/transam/xlogrecovery.c:311

目的

リカバリ処理（クラッシュリカバリ、アーカイブリカバリ、スタンバイモード）における共有メモリ上の状態管理を行う。リカバリの進行状況、スタンバイの制御フラグ、および同期用プリミティブを保持する。

シグネチャまたは形状

typedef struct XLogRecoveryCtlData {
    bool                SharedHotStandbyActive;
    bool                SharedPromoteIsTriggered;
    Latch               recoveryWakeupLatch;
    XLogRecPtr          lastReplayedReadRecPtr;
    XLogRecPtr          lastReplayedEndRecPtr;
    TimeLineID          lastReplayedTLI;
    XLogRecPtr          replayEndRecPtr;
    TimeLineID          replayEndTLI;
    TimestampTz         recoveryLastXTime;
    TimestampTz         currentChunkStartTime;
    RecoveryPauseState  recoveryPauseState;
    ConditionVariable   recoveryNotPausedCV;
    slock_t             info_lck;
} XLogRecoveryCtlData;

入力と出力

本構造体は関数ではなくデータ構造であるため、直接的な入力・出力はない。ただし、info_lck を介して複数のプロセス（スタートアッププロセス、WAL受信プロセス等）間で状態が読み書きされる。

前提条件と不変条件

  • info_lck は、構造体内の共有変数（SharedHotStandbyActive 等）へのアクセスを保護するために使用される。
  • recoveryWakeupLatch は、WALの到着待ちやプロモーション要求待ちの際にスタートアッププロセスを起床させるために使用される。
  • recoveryConflict 発生時の待機には、本構造体の recoveryWakeupLatch ではなく、スタートアッププロセスの procLatch を使用する。

アルゴリズムまたは振る舞い

  • **リカバリ状態の追跡**: lastReplayed* フィールドにより、最後に正常に適用されたWALレコードの位置とタイムラインを記録する。
  • **Redo実行中の状態**: replayEndRecPtr は、現在Redo関数内で処理中のレコードの終了位置を指す。それ以外の場合は lastReplayedEndRecPtr と一致する。
  • **リカバリ一時停止**: recoveryPauseState を通じてリカバリの停止状態を管理し、recoveryNotPausedCV を用いて待機中のプロセスを制御する。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/access/transam/xlogrecovery.c 内の XLogRecoveryShmemInit（初期化時）およびリカバリ実行ループ内の各関数。
  • **呼び出し先**: 構造体自体はデータ保持用であり、関数を呼び出すことはない。ただし、volatile 修飾子を伴ってリカバリ状態の監視に使用される。

関連データ構造

  • Latch: プロセス間通知用。
  • XLogRecPtr: WAL上の位置を示すポインタ。
  • RecoveryPauseState: リカバリの一時停止状態を定義する列挙型。
  • ConditionVariable: リカバリ停止状態の同期用。

失敗モードと境界条件

  • 共有メモリ上の構造体であるため、info_lck による排他制御が正しく行われない場合、競合状態が発生する可能性がある。
  • recoveryWakeupLatch と procLatch の用途を混同して使用すると、デッドロックやシグナルハンドリングの不整合を招く恐れがある。

保守担当者向けメモ

  • 本構造体はスタートアッププロセスがリカバリを制御するための中心的な共有メモリ領域である。
  • 構造体内のフィールドを追加・変更する場合は、XLogRecoveryShmemSize および XLogRecoveryShmemInit の更新が必要となる。
  • 呼び出し元・呼び出し先の詳細な依存関係については、xlogrecovery.c 内の XLogRecoveryCtl ポインタ経由のアクセス箇所を参照すること。
