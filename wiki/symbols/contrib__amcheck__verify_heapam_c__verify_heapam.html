verify_heapam

定義

  • 種別: function
  • 定義箇所: contrib/amcheck/verify_heapam.c:249

目的

PostgreSQL のヒープリレーション（テーブル）の物理的な整合性を検証し、破損を検出して SQL レベルで報告する。

シグネチャまたは形状

Datum verify_heapam(PG_FUNCTION_ARGS)

入力と出力

  • **入力**: PG_FUNCTION_ARGS を介して以下の引数を受け取る。

    • 1. relid (OID): 検証対象のリレーション。 2. on_error_stop (bool): 破損検出時に検証を停止するか。 3. check_toast (bool): TOAST テーブルを検証対象に含めるか。 4. skip (text): スキップオプション（all-visible, all-frozen, none）。 5. first_block (int64, optional): 開始ブロック番号。 6. last_block (int64, optional): 終了ブロック番号。

  • **出力**: Datum (SRF: Set Returning Function)。検出された破損情報を tuplestore に格納して返却する。

前提条件と不変条件

  • 対象リレーションはヒープアクセスメソッドを使用している必要がある。
  • 読み取り専用の検証ツールであるが、破損データへのアクセスによりサーバーがクラッシュするリスクがある。
  • RecoveryInProgress() 中のアンログドテーブルは検証不可のためスキップされる。

アルゴリズムまたは振る舞い

1. **初期化**: 引数を解析し、HeapCheckContext を初期化。tuplestore を準備する。 2. **検証準備**: 対象リレーションを AccessShareLock でオープンし、リレーションの種類と AM を確認。 3. **範囲設定**: 指定されたブロック範囲（first_block から last_block）の妥当性を検証。 4. **TOAST 準備**: check_toast が有効な場合、関連する TOAST テーブルとインデックスをオープンする。 5. **走査と検証**: read_stream API を用いてページを順次読み込み、タプルヘッダー、可視性、属性、および TOAST データの整合性をチェックする。

呼び出し元と呼び出し先

  • **呼び出し元**: SQL インターフェース経由で呼び出される（pg_amcheck 等のツールから利用）。
  • **呼び出し先**: relation_open, table_open, toast_open_indexes, GetAccessStrategy, GetTransactionSnapshot 等のバックエンド関数。

関連データ構造

  • HeapCheckContext: 検証中の状態（リレーション、TOAST 情報、キャッシュされた XID 等）を保持する構造体。
  • tuplestore: 破損報告を蓄積するための構造。

失敗モードと境界条件

  • **エラー**: 引数が NULL の場合、サポート外の relkind/AM の場合、または無効なブロック範囲を指定した場合はエラーを報告する。
  • **スキップ**: 空のリレーションや、リカバリ中のアンログドテーブルは検証せずに終了する。

保守担当者向けメモ

  • 本関数は検証のメインエントリーポイントであり、HeapCheckContext の初期化と検証パラメータのバリデーションを担当する。
  • 破損報告は tuplestore を介して行われるため、大量の破損が検出された場合のメモリ消費に注意すること。
