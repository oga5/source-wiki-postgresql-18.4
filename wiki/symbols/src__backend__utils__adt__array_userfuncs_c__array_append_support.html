array_append_support

定義

  • 種別: function
  • 定義箇所: src/backend/utils/adt/array_userfuncs.c:192

目的

array_append 関数に対するプランナサポート関数として、インプレース（in-place）更新が可能な場合に最適化を適用する。

シグネチャまたは形状

Datum array_append_support(PG_FUNCTION_ARGS)

入力と出力

  • **入力**: PG_FUNCTION_ARGS を介して SupportRequestModifyInPlace 構造体を含む Node を受け取る。
  • **出力**: 最適化が適用可能な場合は該当する Param ノードを、それ以外の場合は NULL を Datum として返す。

前提条件と不変条件

  • 入力された rawreq が SupportRequestModifyInPlace 型である必要がある。
  • 配列の引数が、割り当て対象の配列と一致する場合のみ最適化が有効となる。

アルゴリズムまたは振る舞い

1. PG_GETARG_POINTER(0) で取得したノードが SupportRequestModifyInPlace かどうかを判定する。 2. 該当する場合、リクエスト内の引数リストから最初の引数（Param）を取り出す。 3. その引数が PARAM_EXTERN であり、かつリクエスト内の paramid と一致する場合、その Param ノードを最適化対象として返す。 4. 条件を満たさない場合は NULL を返す。

呼び出し元と呼び出し先

  • **呼び出し元**: PostgreSQL のプランナ（直接的な呼び出し元情報はソースコード上では明示されていない）。
  • **呼び出し先**: IsA マクロおよび linitial マクロを使用。

関連データ構造

  • SupportRequestModifyInPlace: プランナがインプレース更新を要求する際に使用するノード構造体。
  • Param: クエリパラメータを表すノード。

失敗モードと境界条件

  • 入力ノードが SupportRequestModifyInPlace ではない場合、最適化は行われず NULL が返される。
  • 引数が Param ではない、あるいは paramid が一致しない場合も同様に NULL が返される。

保守担当者向けメモ

  • 本関数はプランナの最適化パスの一部であり、array_append の実行効率を向上させるためのヒントを提供する。
  • SupportRequestModifyInPlace の構造に変更があった場合は、本関数の判定ロジックも追従する必要がある。
