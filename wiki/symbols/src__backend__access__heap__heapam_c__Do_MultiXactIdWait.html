Do_MultiXactIdWait

定義

  • 種別: function
  • 定義箇所: src/backend/access/heap/heapam.c:7703

目的

MultiXactId（複数トランザクションによるロック）に含まれる各メンバーのトランザクション状態を評価し、競合が発生している場合に待機（または待機失敗の報告）を行う。

シグネチャまたは形状

static bool Do_MultiXactIdWait(
    MultiXactId multi, 
    MultiXactStatus status,
    uint16 infomask, 
    bool nowait,
    Relation rel, 
    ItemPointer ctid, 
    XLTW_Oper oper,
    int *remaining, 
    bool logLockFailure
)

入力と出力

  • **入力**:
  • multi: 待機対象の MultiXactId。
  • status: 自身のロック要求の MultiXactStatus。
  • infomask: タプルの infomask（ロック状態の判定に使用）。
  • nowait: true の場合、待機せずに即座に失敗を返す。
  • rel, ctid, oper: 待機時に使用するリレーション情報およびロック待機操作のコンテキスト。
  • remaining: 待機が必要な残りのメンバー数を格納するポインタ（NULL可）。
  • logLockFailure: 待機失敗時にログを出力するかどうか。
  • **出力**:
  • bool: 待機が成功したか、あるいは競合がなかった場合は true。nowait が true で競合により待機が必要な場合は false。

前提条件と不変条件

  • HEAP_LOCKED_UPGRADED(infomask) が真の場合、pg_upgrade 以前のタプルとみなされ、待機処理はスキップされる。
  • メンバーの取得には GetMultiXactIdMembers を使用する。

アルゴリズムまたは振る舞い

1. GetMultiXactIdMembers を呼び出し、MultiXactId に含まれるメンバーリストを取得する。 2. 各メンバーについて以下の処理を行う：

  • 自トランザクションがメンバーである場合は、remaining をカウントしてスキップする。
  • DoLockModesConflict を使用し、自身のロック要求とメンバーのステータスが競合するか判定する。
  • 競合しない場合、メンバーが実行中であれば remaining をカウントしてスキップする。
  • 競合する場合：
  • nowait が true なら ConditionalXactLockTableWait を呼び出し、結果を返す。
  • nowait が false なら XactLockTableWait を呼び出し、ロックが解放されるまで待機する。

    • 3. 最後に取得したメンバーリストを pfree で解放する。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/access/heap/heapam.c 内の複数箇所（heap_delete 等のロック関連処理から呼び出される）。
  • **呼び出し先**:
  • GetMultiXactIdMembers
  • TransactionIdIsCurrentTransactionId
  • DoLockModesConflict
  • LOCKMODE_from_mxstatus
  • TransactionIdIsInProgress
  • ConditionalXactLockTableWait
  • XactLockTableWait

関連データ構造

  • MultiXactId, MultiXactStatus, MultiXactMember
  • Relation, ItemPointer
  • XLTW_Oper

失敗モードと境界条件

  • nowait が true の場合、競合するメンバーが存在すると false を返し、待機を行わない。
  • remaining が指定されている場合、競合の有無に関わらず、処理対象のメンバー数が計算され格納される。

保守担当者向けメモ

  • 本関数は MultiXactId のメンバーを反復処理するため、メンバー数が多い場合にパフォーマンスへの影響を考慮する必要がある。
  • 待機時のエラーコンテキスト設定は XactLockTableWait 側に委譲されており、本関数内では明示的なコールバック設定を行わない設計となっている。
