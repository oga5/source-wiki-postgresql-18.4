PortalData

定義

  • 種別: struct
  • 定義箇所: src/include/utils/portal.h:115

目的

PostgreSQLにおけるクエリ実行のライフサイクルと状態を管理するための中心的な構造体。SQLカーソルやプロトコルレベルのクエリ実行において、実行戦略、リソース、実行結果、およびカーソル位置を保持する。

シグネチャまたは形状

typedef struct PortalData {
    const char *name;
    const char *prepStmtName;
    MemoryContext portalContext;
    ResourceOwner resowner;
    void (*cleanup) (Portal portal);
    SubTransactionId createSubid;
    SubTransactionId activeSubid;
    int createLevel;
    const char *sourceText;
    CommandTag commandTag;
    QueryCompletion qc;
    List *stmts;
    CachedPlan *cplan;
    ParamListInfo portalParams;
    QueryEnvironment *queryEnv;
    PortalStrategy strategy;
    int cursorOptions;
    PortalStatus status;
    bool portalPinned;
    bool autoHeld;
    QueryDesc *queryDesc;
    TupleDesc tupDesc;
    int16 *formats;
    Snapshot portalSnapshot;
    Tuplestorestate *holdStore;
    MemoryContext holdContext;
    Snapshot holdSnapshot;
    bool atStart;
    bool atEnd;
    uint64 portalPos;
    TimestampTz creation_time;
    bool visible;
} PortalData;

入力と出力

  • **入力**: CreatePortal 等で初期化され、PortalDefineQuery を通じてクエリ（PlannedStmts）やパラメータ（ParamListInfo）が割り当てられる。
  • **出力**: 実行結果は Tuplestorestate（holdStore）にバッファリングされ、クライアントへ提供される。

前提条件と不変条件

  • portalSnapshot は、クエリ実行中のデータ一貫性を保証するために必要であり、TOASTデータのデトーストに不可欠。
  • atStart が真の場合、portalPos は必ず 0 である。
  • portalPinned が真のポータルは、明示的に解放されるまで破棄できない。

アルゴリズムまたは振る舞い

1. **初期化**: CreatePortal によりメモリコンテキストとリソースオーナーが割り当てられる。 2. **実行**: PortalStatus が READY から ACTIVE に遷移し、queryDesc を用いて Executor がクエリを実行する。 3. **カーソル移動**: portalPos を更新し、atStart / atEnd フラグを管理する。 4. **保持**: トランザクションを跨ぐ場合、holdStore に結果を格納し、holdSnapshot を用いて一貫性を維持する。 5. **終了**: PortalDrop によりリソースが解放される。

呼び出し元と呼び出し先

  • **呼び出し元**: 提供されたソース情報からは、具体的な呼び出し元関数は特定できない。
  • **呼び出し先**: Executor（queryDesc を介した実行）、メモリ管理関数、リソース管理関数。

関連データ構造

  • PortalStrategy: 実行戦略（PORTAL_ONE_SELECT 等）。
  • PortalStatus: ライフサイクル状態（NEW, READY, ACTIVE, DONE 等）。
  • QueryDesc: 実行記述子。
  • Tuplestorestate: 結果保持用ストレージ。
  • ResourceOwner: リソース管理用。

失敗モードと境界条件

  • PORTAL_ONE_SELECT 以外の戦略では、部分実行やサスペンドはサポートされない。
  • holdStore を使用する場合、TOASTデータのデトーストが不十分だと、スナップショット解放後にデータが消失するリスクがある。

保守担当者向けメモ

  • ポータルは「1つのソースSQLクエリ」に対して1つ作成される。書き換えルールにより複数のプランツリーが生成される場合も同様。
  • portalPos の操作は、ポータル移動ルーチンのみが行うこと。
  • holdSnapshot は、holdStore 内のタプルがTOAST参照を含む可能性がある場合に保持する必要がある。
