LogicalRepTyp

定義

  • 種別: struct
  • 定義箇所: src/include/replication/logicalproto.h:119

目的

論理レプリケーションにおいて、パブリッシャー側から送信されるデータ型の情報を保持し、サブスクライバー側で型定義を識別・管理するために使用されます。

シグネチャまたは形状

typedef struct LogicalRepTyp
{
	Oid			remoteid;		/* リモート側の型の一意なID */
	char	   *nspname;		/* リモート側の型のスキーマ名 */
	char	   *typname;		/* リモート側の型の名前 */
} LogicalRepTyp;

入力と出力

本構造体は、論理レプリケーションのワイヤプロトコルを通じて受信した型情報を格納するコンテナとして機能します。関数 logicalrep_read_typ を通じて、ストリームから読み取られたデータが本構造体に設定されます。

前提条件と不変条件

  • remoteid はリモートデータベース内での型の一意性を保証する Oid です。
  • nspname および typname は、受信したプロトコルデータから動的に割り当てられた文字列を指すことが想定されます。

アルゴリズムまたは振る舞い

論理レプリケーションのデコードプロセスにおいて、リモート側の型定義をローカル環境で再現または照合する際に利用されます。サブスクライバー側のワーカープロセスが、プロトコル経由で送られてきた型情報を本構造体にデシリアライズし、必要に応じて型変換や検証を行います。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/replication/logical/worker.c において、受信した型情報の保持用としてインスタンス化されます。
  • **呼び出し先**: src/backend/replication/logical/proto.c 内の logicalrep_read_typ 関数が、本構造体へのデータ読み込みを担当します。

関連データ構造

  • LogicalRepRelation: リレーション定義を保持する構造体であり、型情報と組み合わせてテーブル構造の再現に使用されます。

失敗モードと境界条件

  • nspname や typname がメモリ管理の不整合により解放済み領域を指すリスクがあるため、ライフサイクル管理には注意が必要です。
  • 受信した remoteid がローカル環境で存在しない型を指している場合、後続のデータ適用処理でエラーが発生する可能性があります。

保守担当者向けメモ

  • 本構造体はワイヤプロトコルの一部であるため、フィールドの追加や変更を行う際は、プロトコルバージョン（LOGICALREP_PROTO_VERSION_NUM）の更新および logicalrep_read_typ / logicalrep_write_typ 関数の同期的な修正が必須です。
