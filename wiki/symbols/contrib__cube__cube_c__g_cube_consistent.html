g_cube_consistent

定義

  • 種別: function
  • 定義箇所: contrib/cube/cube.c:394

目的

PostgreSQL の GiST 索引において、検索条件（クエリ）がインデックス内の特定のノードと一致するかを判定するサポート関数です。

シグネチャまたは形状

Datum g_cube_consistent(PG_FUNCTION_ARGS)

入力と出力

  • **入力**:
  • entry (GISTENTRY *): 評価対象のインデックスエントリ。
  • query (NDBOX *): 検索条件となる多次元ボックス。
  • strategy (StrategyNumber): 実行する演算子の戦略番号。
  • recheck (bool *): 検索結果の再評価が必要かを示すフラグ。
  • **出力**:
  • Datum (bool): 検索条件に一致する場合は true、一致しない場合は false を返します。
  • *recheck: 常に false が設定されます（本関数による判定は完全であるため）。

前提条件と不変条件

  • recheck は常に false に設定されます。これは、この関数が提供する判定ロジックが完全（exact）であることを示します。
  • query は PG_FREE_IF_COPY により、必要に応じてメモリ解放が行われます。

アルゴリズムまたは振る舞い

1. GIST_LEAF(entry) を用いて、現在のノードがリーフノードか内部ノードかを判定します。 2. リーフノードの場合は g_cube_leaf_consistent を呼び出し、インデックスキーとクエリを比較します。 3. 内部ノードの場合は g_cube_internal_consistent を呼び出し、インデックスキーとクエリを比較します。 4. 判定結果を PG_RETURN_BOOL で返します。

呼び出し元と呼び出し先

  • **呼び出し元**: PostgreSQL の GiST 索引インターフェースから呼び出されます（直接的な呼び出し元コードはソース内に明示されていません）。
  • **呼び出し先**:
  • g_cube_leaf_consistent
  • g_cube_internal_consistent

関連データ構造

  • NDBOX: 多次元ボックスを表現するデータ構造。
  • GISTENTRY: GiST 索引のノード情報を保持する構造体。

失敗モードと境界条件

  • entry が無効な場合や、query の次元数とインデックスの次元数が一致しない場合の挙動については、本ソース excerpt 内には記述がありません。

保守担当者向けメモ

  • 本関数は GiST 索引の検索ロジックの分岐点です。新しい空間演算子を追加する際は、g_cube_leaf_consistent および g_cube_internal_consistent 側の実装を確認してください。
