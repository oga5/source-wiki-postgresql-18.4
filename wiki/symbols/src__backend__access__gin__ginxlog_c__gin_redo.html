gin_redo

定義

  • 種別: function
  • 定義箇所: src/backend/access/gin/ginxlog.c:725

目的

GIN（Generalized Inverted Index）インデックスのWAL（Write-Ahead Log）リプレイ処理におけるメインエントリポイントとして機能し、WALレコードの種別に応じて適切な再実行関数へ処理を振り分けます。

シグネチャまたは形状

void gin_redo(XLogReaderState *record)

入力と出力

  • **入力**: XLogReaderState *record（リプレイ対象のWALレコード状態）
  • **出力**: なし（副作用としてインデックスページの状態を更新）

前提条件と不変条件

  • opCtx がリプレイ処理用の一時的なメモリコンテキストとして利用可能であること。
  • GINインデックスのリプレイ処理は冪等（idempotent）であること。

アルゴリズムまたは振る舞い

1. XLogRecGetInfo を使用してWALレコードの種別（info）を抽出します。 2. opCtx にメモリコンテキストを切り替えます。 3. info コードに基づき、以下のいずれかの関数を呼び出します：

  • ginRedoCreatePTree
  • ginRedoInsert
  • ginRedoSplit
  • ginRedoVacuumPage
  • ginRedoVacuumDataLeafPage
  • ginRedoDeletePage
  • ginRedoUpdateMetapage
  • ginRedoInsertListPage
  • ginRedoDeleteListPages

    • 4. 未知のオペコードが指定された場合は PANIC で停止します。 5. 処理終了後、メモリコンテキストを元に戻し、opCtx をリセットします。

呼び出し元と呼び出し先

  • **呼び出し元**: PostgreSQLのリカバリマネージャ（src/include/access/rmgrlist.h で RM_GIN_ID に登録され、WALリプレイ時に呼び出される）。
  • **呼び出し先**: 上記「アルゴリズムまたは振る舞い」に記載された各 ginRedo* 関数。

関連データ構造

  • XLogReaderState: WALレコードを読み取るための状態構造体。
  • opCtx: リプレイ処理中に使用されるメモリコンテキスト。

失敗モードと境界条件

  • 未知のWALオペコードが渡された場合、elog(PANIC, ...) によりシステムが停止します。

保守担当者向けメモ

  • GINインデックスは現在競合処理を必要としませんが、将来的にB-treeのような最適化（VACUUM外でのタプル削除など）を導入する場合は、本関数内で対応が必要となります。
  • 新しいWALレコードタイプを追加した場合は、switch 文へのケース追加と、gin_mask 関数でのマスク対象定義を忘れないようにしてください。
