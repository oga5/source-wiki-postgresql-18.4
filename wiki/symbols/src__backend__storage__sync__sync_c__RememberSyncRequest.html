RememberSyncRequest

定義

  • 種別: function
  • 定義箇所: src/backend/storage/sync/sync.c:486

目的

チェックポインタプロセスにおいて、ファイル同期（fsync）または削除（unlink）の要求を管理キュー（ハッシュテーブルまたはリスト）へ登録、あるいは既存の要求をキャンセルするために使用されます。

シグネチャまたは形状

void RememberSyncRequest(const FileTag *ftag, SyncRequestType type)

入力と出力

  • ftag: 操作対象のファイルを識別する FileTag 構造体へのポインタ。
  • type: 要求の種類（SYNC_REQUEST, SYNC_FORGET_REQUEST, SYNC_FILTER_REQUEST, SYNC_UNLINK_REQUEST）。
  • 戻り値: なし。

前提条件と不変条件

  • pendingOps ハッシュテーブルが初期化されていること（Assert(pendingOps)）。
  • メモリ確保は pendingOpsCxt コンテキスト内で行われる。
  • SYNC_REQUEST の場合、既存のエントリが存在しても cycle_ctr は更新されず、最も古い要求のサイクルが維持される。

アルゴリズムまたは振る舞い

要求の種類に応じて以下の処理を実行します。 1. **SYNC_FORGET_REQUEST**: 指定された ftag に一致する fsync 要求をハッシュテーブルから検索し、canceled フラグを立てる。 2. **SYNC_FILTER_REQUEST**: ハッシュテーブル内の fsync 要求および pendingUnlinks リスト内の削除要求を走査し、sync_filetagmatches が真となるすべてのエントリの canceled フラグを立てる。 3. **SYNC_UNLINK_REQUEST**: 新しい PendingUnlinkEntry を作成し、pendingUnlinks リストに追加する。 4. **SYNC_REQUEST**: 指定された ftag でハッシュテーブルを検索（HASH_ENTER）。新規または以前にキャンセルされたエントリであれば、現在の sync_cycle_ctr を設定し、canceled を偽にする。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/postmaster/checkpointer.c および src/backend/storage/sync/sync.c 内の要求転送処理。
  • **呼び出し先**: hash_search, hash_seq_init, hash_seq_search, palloc, MemoryContextSwitchTo, lappend。また、syncsw テーブル内の sync_filetagmatches 関数ポインタを介して各ハンドラの比較ロジックを呼び出す。

関連データ構造

  • pendingOps: fsync 要求を管理するハッシュテーブル。
  • pendingUnlinks: 削除待ちのファイルリスト。
  • PendingFsyncEntry, PendingUnlinkEntry: 要求を保持する構造体。
  • syncsw: 各種ハンドラの同期関数を抽象化したテーブル。

失敗モードと境界条件

  • メモリ不足時には palloc によりエラーが発生する可能性がある。
  • SYNC_FILTER_REQUEST 実行時、対象のハンドラが存在しない場合や sync_filetagmatches が適切に定義されていない場合は意図したフィルタリングが行われない可能性がある。

保守担当者向けメモ

  • 本関数はチェックポインタプロセス内でのみ実行されることを想定している。
  • cycle_ctr の管理は、チェックポイント処理の無限ループを防ぐための重要な不変条件であるため、変更時には注意が必要。
