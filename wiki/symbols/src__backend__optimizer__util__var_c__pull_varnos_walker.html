pull_varnos_walker

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/util/var.c:160

目的

式ツリーを走査し、指定されたクエリレベル（sublevels_up）に属するすべてのリレーションID（varno）を収集してビットマップセット（varnos）に格納する。

シグネチャまたは形状

static bool pull_varnos_walker(Node *node, pull_varnos_context *context);

入力と出力

  • **入力**:
  • node: 走査対象のノード。
  • context: 走査状態を保持する pull_varnos_context 構造体。sublevels_up（現在の階層）、varnos（結果のビットマップ）、root（プランナのルート情報）を含む。
  • **出力**:
  • 戻り値: 走査を継続する場合は false、停止する場合は true。
  • 副作用: context->varnos に見つかったリレーションIDが追加される。

前提条件と不変条件

  • context->sublevels_up は、現在走査中のノードが何階層上のクエリに属しているかを示す。
  • PlaceHolderVar の処理において、context->root が NULL の場合は phrels を使用し、NULL でない場合は ph_eval_at を優先して使用する。

アルゴリズムまたは振る舞い

1. **Var ノード**: varlevelsup が context->sublevels_up と一致する場合、varno および varnullingrels を context->varnos に追加する。 2. **CurrentOfExpr ノード**: sublevels_up == 0 の場合、cvarno を追加する。 3. **PlaceHolderVar ノード**:

  • phlevelsup が一致する場合、PlaceHolderInfo の有無に応じて ph_eval_at または phrels を使用して varnos を更新する。
  • 翻訳された PlaceHolderVar の場合は、ph_eval_at を適切に変換して適用する。
  • phnullingrels を常に結果に含める。

    • 4. **Query ノード**: sublevels_up をインクリメントし、query_tree_walker を用いてサブクエリ内を再帰的に走査する。 5. **その他**: expression_tree_walker を用いてツリーを再帰的に走査する。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • pull_varnos (src/backend/optimizer/util/var.c:73)
  • pull_varnos_of_level (src/backend/optimizer/util/var.c:127, 153)
  • 自身による再帰呼び出し (query_tree_walker および expression_tree_walker 経由)
  • **呼び出し先**:
  • query_tree_walker
  • expression_tree_walker
  • bms_add_member, bms_add_members, bms_equal, bms_difference, bms_join (ビットマップ操作関数)

関連データ構造

  • pull_varnos_context: 走査のコンテキストを保持する構造体。
  • Var, PlaceHolderVar, CurrentOfExpr, Query: 走査対象となるノード型。
  • PlaceHolderInfo: PlaceHolderVar の評価位置情報を保持する構造体。

失敗モードと境界条件

  • node が NULL の場合は直ちに false を返す。
  • PlaceHolderInfo が見つからない場合、保守的なフォールバックとして phrels を使用する。
  • サブクエリの走査において sublevels_up の管理が不適切な場合、誤った階層の変数を収集する可能性がある。

保守担当者向けメモ

  • PlaceHolderVar の処理は複雑であり、特に appendrel 翻訳時の ph_eval_at の再計算ロジックは、プランナの構造変更に影響を受けやすいため注意が必要。
  • sublevels_up のインクリメント・デクリメントは Query ノードの走査前後で確実に行うこと。
