RenameRole

定義

  • **種別**: function
  • **定義箇所**: src/backend/commands/user.c:1333

目的

データベース内の既存ロール名を新しい名称に変更する。

シグネチャまたは形状

ObjectAddress RenameRole(const char *oldname, const char *newname)

入力と出力

  • **入力**:
  • oldname: 変更対象の既存ロール名。
  • newname: 新しいロール名。
  • **出力**:
  • 変更されたロールの ObjectAddress を返す。

前提条件と不変条件

  • oldname で指定されたロールが存在しなければならない。
  • newname は既存のロール名と重複してはならない。
  • セッションユーザー自身および現在のユーザー（GetOuterUserId）の名前変更は禁止されている。
  • pg_ で始まる予約済み名称への変更、または予約済み名称の変更は禁止されている。
  • 実行ユーザーは、対象ロールがスーパーユーザーである場合はスーパーユーザーである必要があり、そうでない場合は CREATEROLE 権限かつ対象ロールに対する ADMIN 権限が必要である。

アルゴリズムまたは振る舞い

1. pg_authid システムカタログを RowExclusiveLock で開く。 2. AUTHNAME システムキャッシュを使用して oldname のタプルを検索する。 3. セッションユーザーや現在のユーザーの変更、予約語のチェック、権限チェックを行う。 4. newname が既に存在しないか確認する。 5. pg_authid の rolname 属性を更新する。 6. MD5形式のパスワードが設定されている場合、ユーザー名がソルトとして使用されているため、パスワードをクリアし通知を発行する。 7. CatalogTupleUpdate を実行し、InvokeObjectPostAlterHook を呼び出す。 8. ObjectAddressSet でアドレスを構築し、キャッシュを解放して終了する。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/commands/alter.c:385 (推論: ALTER ROLE ... RENAME TO コマンド実行時)。
  • **呼び出し先**: ソースコード内で table_open, SearchSysCache1, GetSessionUserId, IsReservedName, heap_modify_tuple, CatalogTupleUpdate, InvokeObjectPostAlterHook 等のPostgreSQL内部関数を呼び出す。

関連データ構造

  • pg_authid: ロール情報を保持するシステムカタログ。
  • ObjectAddress: 操作対象オブジェクトの識別子。

失敗モードと境界条件

  • **ERRCODE_UNDEFINED_OBJECT**: oldname が存在しない場合。
  • **ERRCODE_FEATURE_NOT_SUPPORTED**: セッションユーザーや現在のユーザーを変更しようとした場合。
  • **ERRCODE_RESERVED_NAME**: pg_ で始まる名称を使用しようとした場合。
  • **ERRCODE_DUPLICATE_OBJECT**: newname が既に存在する場合。
  • **ERRCODE_INSUFFICIENT_PRIVILEGE**: 権限チェックに失敗した場合。

保守担当者向けメモ

  • MD5 パスワードのクリア処理は、ロール名変更に伴う認証情報の不整合を防ぐための必須ロジックである。
  • pg_authid のロックは table_close 後もコミットまで保持される点に注意すること。
  • ENFORCE_REGRESSION_TEST_NAME_RESTRICTIONS が定義されている場合、テスト用ロール名の命名規則が強制される。
