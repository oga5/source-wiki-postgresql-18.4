canonicalize_ec_expression

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/path/equivclass.c:544

目的

等価クラス（EquivalenceClass）に含める式を、要求された型および照合順序（collation）に適合するように正規化する。

シグネチャまたは形状

Expr *canonicalize_ec_expression(Expr *expr, Oid req_type, Oid req_collation)

入力と出力

  • **入力**:
  • expr: 正規化対象の式。
  • req_type: 要求されるデータ型（Oid）。
  • req_collation: 要求される照合順序（Oid）。
  • **出力**:
  • 要求された型と照合順序を持つ Expr。必要に応じて RelabelType ノードでラップされる。

前提条件と不変条件

  • eval_const_expressions が既に適用済みであることを前提とする。
  • 多相型（Polymorphic type）または RECORDOID が req_type として指定された場合、入力式の型が優先される。

アルゴリズムまたは振る舞い

1. req_type が多相型または RECORDOID の場合、req_type を入力式の型で上書きする。 2. 入力式の型および照合順序が要求と一致しているか確認する。 3. 一致しない場合、applyRelabelType を使用して型変換を行う。

  • 型が異なる場合は typmod を -1 に設定する。
  • 型が同じで照合順序のみ異なる場合は、元の typmod を維持する。

    • 4. applyRelabelType を用いることで、定数フラット性（const-flatness）を維持する。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/optimizer/path/equivclass.c (複数箇所)
  • src/backend/optimizer/path/pathkeys.c:1178
  • **呼び出し先**:
  • exprType (マクロ)
  • exprCollation (マクロ)
  • exprTypmod (マクロ)
  • applyRelabelType

関連データ構造

  • Expr
  • Node

失敗モードと境界条件

  • 特筆すべき失敗モードは定義されていない。型変換が不要な場合は、入力された式がそのまま返される。

保守担当者向けメモ

  • 本関数は RelabelType ノードを生成する際、定数式の最適化結果を破壊しないよう applyRelabelType を経由している。修正を加える際は、この定数フラット性の維持が重要である。
