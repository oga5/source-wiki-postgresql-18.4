DefineIndex

定義

  • 種別: function
  • 定義箇所: src/backend/commands/indexcmds.c:541

目的

インデックス作成（CREATE INDEX）のメインエントリポイントとして、インデックス定義の検証、カタログ登録、および構築プロセスの制御を行う。

シグネチャまたは形状

ObjectAddress DefineIndex(Oid tableId, IndexStmt *stmt, Oid indexRelationId, Oid parentIndexId, Oid parentConstraintId, int total_parts, bool is_alter_table, bool check_rights, bool check_not_in_use, bool skip_build, bool quiet);

入力と出力

  • **入力**: 対象テーブルの tableId、インデックス定義 stmt、および構築オプション（concurrent 等）。
  • **出力**: 作成されたインデックスの ObjectAddress を返す。

前提条件と不変条件

  • 呼び出し元は、対象テーブルに対して適切なロックを保持している必要がある（デッドロック回避のため、ヒープをロックしてからインデックスをロックする順序が厳守される）。
  • 一時テーブルに対しては、CONCURRENTLY 指定があっても非並行構築が強制される。
  • インデックス列数は INDEX_MAX_KEYS を超えてはならない。

アルゴリズムまたは振る舞い

1. **初期化**: NewGUCNestLevel で設定を保存し、RestrictSearchPath を実行。 2. **並行性の判定**: 一時テーブル以外で stmt->concurrent が指定されている場合に並行構築モードを有効化。 3. **進捗報告**: pgstat_progress_start_command を呼び出し、構築状況を追跡。 4. **検証**: インデックス対象の列数を確認し、サポートされていないリレーション種別（RELKIND）の場合はエラーを投げる。 5. **ロック取得**: 並行構築時は ShareUpdateExclusiveLock、通常時は ShareLock を取得。 6. **権限昇格**: テーブル所有者のユーザーIDとセキュリティコンテキストに切り替え、SECURITY_RESTRICTED_OPERATION を適用。 7. **構築フェーズ**: カタログ登録、アクセスメソッドの検証、および必要に応じて並行構築の各フェーズ（カタログ登録、構築、検証、スワップ）を実行。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/catalog/index.c、src/backend/commands/tablecmds.c 等から呼び出される。
  • **呼び出し先**: index_create（カタログ登録）、table_open（テーブルオープン）、pgstat_progress_*（進捗報告）、SetUserIdAndSecContext（権限管理）等。

関連データ構造

  • IndexStmt: インデックス定義を保持するノード。
  • Relation: 対象テーブルのメタデータ。
  • ObjectAddress: 作成されたインデックスの識別子。
  • IndexInfo: インデックスの構築に必要な情報を格納する構造体。

失敗モードと境界条件

  • 列数が 0 の場合、または INDEX_MAX_KEYS を超える場合は ERROR を発生させる。
  • サポート外のリレーション種別に対しては ERRCODE_WRONG_OBJECT_TYPE でエラーを発生させる。
  • 権限不足やロック競合が発生した場合、トランザクションのロールバックを伴うエラーとなる。

保守担当者向けメモ

  • REINDEX CONCURRENTLY 等の並行処理では、複数のトランザクションをまたぐため MemoryContext の切り替えとトランザクション管理に注意が必要。
  • パーティションテーブルに対する再帰的なインデックス作成処理が含まれる。
  • 既存の CheckIndexCompatible や WaitForOlderSnapshots との連携により、インデックスの整合性を維持している。
