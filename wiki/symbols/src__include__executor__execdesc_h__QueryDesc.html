QueryDesc

定義

  • 種別: struct
  • 定義箇所: src/include/executor/execdesc.h:33

目的

PostgreSQLのクエリ実行エンジンにおいて、実行対象の計画（PlannedStmt）と実行に必要な環境情報（スナップショット、パラメータ、出力先など）を一元管理し、実行ライフサイクル全体を通じて保持するためのパッケージ構造体です。

シグネチャまたは形状

typedef struct QueryDesc {
    /* CreateQueryDescにより提供されるフィールド */
    CmdType         operation;
    PlannedStmt    *plannedstmt;
    const char     *sourceText;
    Snapshot        snapshot;
    Snapshot        crosscheck_snapshot;
    DestReceiver   *dest;
    ParamListInfo   params;
    QueryEnvironment *queryEnv;
    int             instrument_options;

    /* ExecutorStartにより設定されるフィールド */
    TupleDesc       tupDesc;
    EState         *estate;
    PlanState      *planstate;

    /* ExecutePlanにより設定されるフィールド */
    bool            already_executed;

    /* プラグイン用予約領域 */
    struct Instrumentation *totaltime;
} QueryDesc;

入力と出力

  • **入力**: CreateQueryDesc 関数により、プランナの出力（PlannedStmt）および実行環境情報が入力され初期化されます。
  • **出力**: 実行エンジン（ExecutorRun）が保持する dest（DestReceiver）を通じて、クエリ結果がクライアントやテーブル等へ出力されます。

前提条件と不変条件

  • ユーティリティコマンドの場合、実行エンジン（ExecutorRun）に渡してはならないという制約があります。
  • totaltime フィールドはコアシステムでは使用されず、常に NULL です（プラグインによる拡張用）。

アルゴリズムまたは振る舞い

1. **生成**: CreateQueryDesc でパッケージ化されます。 2. **初期化**: ExecutorStart が QueryDesc を受け取り、estate（クエリ全体の実行状態）および planstate（実行ノードツリー）を構築して格納します。 3. **実行**: ExecutorRun が QueryDesc を参照し、プランツリーを走査して結果を dest へ渡します。 4. **解放**: FreeQueryDesc により、保持するメモリやリソースがクリーンアップされます。

呼び出し元と呼び出し先

  • **呼び出し元（直接的な言及）**:
  • contrib/auto_explain/auto_explain.c（explain_ExecutorStart 等）
  • contrib/pg_stat_statements/pg_stat_statements.c（pgss_ExecutorStart 等）
  • **呼び出し先**:
  • 構造体内部で PlannedStmt, EState, PlanState, DestReceiver 等の型を参照しています。具体的な関数呼び出しの依存関係はソース抜粋からは特定できません。

関連データ構造

  • PlannedStmt: プランナの出力。
  • EState: クエリ実行全体の共有状態。
  • PlanState: 実行計画ノードのツリー構造。
  • DestReceiver: 結果の出力先インターフェース。

失敗モードと境界条件

  • 実行前に already_executed フラグが true である場合、再実行に関する制御が行われます。
  • ユーティリティコマンドを誤って ExecutorRun に渡した場合の動作は未定義です。

保守担当者向けメモ

  • QueryDesc が保持する estate と planstate の関係を理解することが、実行フローを追うための最短ルートです。
  • 実行エンジン以外のコンポーネント（プラグイン等）が totaltime を利用する際は、コアシステムとの競合に注意してください。
