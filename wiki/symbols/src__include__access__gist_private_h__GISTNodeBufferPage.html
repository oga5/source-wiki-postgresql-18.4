GISTNodeBufferPage

定義

  • **種別**: struct
  • **定義箇所**: src/include/access/gist_private.h:46

目的

GiSTインデックス構築時のバッファリング処理において、メモリ上に一時的なページデータを保持するための構造体です。

シグネチャまたは形状

typedef struct {
    BlockNumber prev;
    uint32      freespace;
    char        tupledata[FLEXIBLE_ARRAY_MEMBER];
} GISTNodeBufferPage;

入力と出力

  • **入力**: IndexTuple をページ内に配置する際、gistPlaceItupToPage を通じてデータが書き込まれます。
  • **出力**: gistGetItupFromPage を通じて、ページ内の IndexTuple を取得可能です。

前提条件と不変条件

  • tupledata は FLEXIBLE_ARRAY_MEMBER として定義されており、可変長のデータを格納します。
  • ページ内のデータオフセットは BUFFER_PAGE_DATA_OFFSET マクロ（MAXALIGN(offsetof(GISTNodeBufferPage, tupledata))）によりアライメントが保証されます。

アルゴリズムまたは振る舞い

  • インデックス構築時に gistAllocateNewPageBuffer を介してメモリ上に割り当てられます。
  • MemoryContextAllocZero を使用してゼロ初期化された状態で確保されます。
  • 大量挿入時のランダムI/Oを削減するため、一時ファイルへ書き出す前段階のバッファとして機能します。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/access/gist/gistbuildbuffers.c 内の関数群（gistAllocateNewPageBuffer 等）から参照・操作されます。
  • **呼び出し先**: 本構造体自体はデータ保持用であり、直接的な関数呼び出しは行いません。

関連データ構造

  • GISTBuildBuffers: 本構造体を管理する上位の構築用構造体。
  • IndexTuple: ページ内に格納されるインデックスエントリ。

失敗モードと境界条件

  • BLCKSZ（ブロックサイズ）から BUFFER_PAGE_DATA_OFFSET を差し引いた値が、格納可能な最大データサイズとなります。これを超える書き込みは境界外アクセスのリスクを伴います。

保守担当者向けメモ

  • 本構造体はインデックス構築時の最適化専用であり、通常のインデックススキャンや挿入パスでは使用されません。
  • tupledata へのアクセスには、必ず BUFFER_PAGE_DATA_OFFSET を使用してアライメントを考慮してください。
