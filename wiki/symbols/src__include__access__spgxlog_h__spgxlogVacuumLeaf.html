spgxlogVacuumLeaf

定義

  • 種別: struct
  • 定義箇所: src/include/access/spgxlog.h:201

目的

SP-GiST インデックスのバキューム処理（リーフページ内のタプル整理）を WAL に記録するための構造体です。

シグネチャまたは形状

typedef struct spgxlogVacuumLeaf
{
    uint16      nDead;
    uint16      nPlaceholder;
    uint16      nMove;
    uint16      nChain;
    spgxlogState stateSrc;
    OffsetNumber offsets[FLEXIBLE_ARRAY_MEMBER];
} spgxlogVacuumLeaf;

入力と出力

  • **入力**: バキューム対象のリーフページにおける、削除（DEAD）、プレースホルダ化、移動、再チェーンが必要なタプルの数および各オフセット番号。
  • **出力**: WAL レコードとしてシリアライズされたデータ。offsets 配列には、DEAD、PLACEHOLDER、移動元、移動先、更新対象の nextOffset リンク、挿入する nextOffset リンクの順でオフセット番号が格納されます。

前提条件と不変条件

  • offsets 配列は可変長であり、構造体直後に続くアライメントされていないデータとして扱われます。
  • 構造体内の各カウント（nDead, nPlaceholder, nMove, nChain）は、offsets 配列内の対応する要素数と一致している必要があります。

アルゴリズムまたは振る舞い

1. バキューム操作時に、対象となるタプルのオフセット情報を構造体に詰め込みます。 2. XLogInsert を介して WAL に書き出されます。 3. リカバリ時（spg_redo）、この構造体を読み取り、対象ページ上のタプルを DEAD 状態への変更、プレースホルダへの変換、またはリンクの再構築を行います。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/access/spgist/spgvacuum.c にて、バキューム処理中にこの構造体がインスタンス化されます。
  • **呼び出し先**: src/backend/access/spgist/spgxlog.c の spg_redo 関数が、この構造体を使用して物理ページの更新を行います。
  • **その他**: src/backend/access/rmgrdesc/spgdesc.c にて、ログのデコード（説明表示）のために参照されています。

関連データ構造

  • spgxlogState: リカバリに必要なインデックス状態を保持します。
  • OffsetNumber: storage/off.h で定義される、ページ内のアイテムオフセット。
  • SizeOfSpgxlogVacuumLeaf: offsetof(spgxlogVacuumLeaf, offsets) として定義される、固定部分のサイズ計算用マクロ。

失敗モードと境界条件

  • 可変長配列 offsets を扱う際、ポインタ演算を誤るとメモリ破壊を引き起こす可能性があります。
  • 構造体内のデータはアライメントされていないため、直接的なキャストではなく memcpy 等による安全なアクセスが求められます。

保守担当者向けメモ

  • FLEXIBLE_ARRAY_MEMBER を使用しているため、サイズ計算には必ず SizeOfSpgxlogVacuumLeaf マクロを使用してください。
  • offsets 配列の順序は、構造体のコメントに記載された順序と厳密に一致させる必要があります。リカバリロジックを変更する際は、この順序の整合性に注意してください。
