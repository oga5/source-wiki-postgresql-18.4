spgVacPendingItem

定義

  • 種別: struct
  • 定義箇所: src/backend/access/spgist/spgvacuum.c:33

目的

SP-GiST インデックスの VACUUM 処理中に、後続の処理で再訪（クリーンアップ）が必要な REDIRECT 先の TID を保持するためのリスト要素です。

シグネチャまたは形状

typedef struct spgVacPendingItem
{
    ItemPointerData tid;            /* 訪問対象のリダイレクト先 */
    bool            done;           /* 処理済みフラグ */
    struct spgVacPendingItem *next; /* リストのリンク */
} spgVacPendingItem;

入力と出力

  • 入力: ItemPointerData 型の TID。
  • 出力: なし（構造体としてメモリ上に保持される）。

前提条件と不変条件

  • next ポインタは、リストの終端では NULL である必要があります。
  • done フラグは、spgprocesspending 等の処理ルーチンによって、対象の TID が適切に処理されたか否かを追跡するために使用されます。

アルゴリズムまたは振る舞い

  • spgvacuumscan 等の走査中に発見された REDIRECT 先の TID を格納します。
  • spgAddPendingTID を通じてリストに追加され、spgprocesspending によってリスト内の各項目が順次処理されます。
  • 処理が完了した項目は done フラグにより管理され、最終的にリストから解放されます。

呼び出し元と呼び出し先

  • 呼び出し元: spgAddPendingTID（リストへの追加）、spgprocesspending（リストの走査と処理）。
  • 呼び出し先: なし（データ構造体のため）。

関連データ構造

  • spgBulkDeleteState: この構造体をリストのヘッド（pendingList）として保持します。
  • ItemPointerData: インデックス内の物理的な位置を指すために使用されます。

失敗モードと境界条件

  • メモリ不足: palloc によるリスト要素の確保時に発生する可能性があります。
  • リストの重複: 同一の TID が複数回リストに追加される可能性があり、処理側で done フラグや重複チェックを行う必要があります。

保守担当者向けメモ

  • 本構造体は spgvacuum.c 内の動的なリスト管理に使用されます。リストのメモリ解放漏れがないよう、spgClearPendingList 等のクリーンアップ関数との整合性に注意してください。
