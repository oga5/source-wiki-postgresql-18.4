TupleTableSlot

定義

  • 種別: struct
  • 定義箇所: src/include/executor/tuptable.h:114

目的

PostgreSQL 実行エンジンにおいて、プランノード間で受け渡されるタプルを保持・抽象化するためのコンテナです。物理的なタプル（ヒープ、バッファ等）と仮想タプルを統一的なインターフェースで扱います。

シグネチャまたは形状

typedef struct TupleTableSlot
{
    NodeTag             type;
    uint16              tts_flags;
    AttrNumber          tts_nvalid;
    const TupleTableSlotOps *const tts_ops;
    TupleDesc           tts_tupleDescriptor;
    Datum              *tts_values;
    bool               *tts_isnull;
    MemoryContext       tts_mcxt;
    ItemPointerData     tts_tid;
    Oid                 tts_tableOid;
} TupleTableSlot;

入力と出力

  • **入力**: TupleDesc（型定義）、各種物理タプル（ExecStore... 系関数経由）。
  • **出力**: tts_values および tts_isnull 配列を介した属性値の提供。

前提条件と不変条件

  • tts_ops はスロットの挙動を定義する関数ポインタテーブルであり、初期化時に設定され変更不可。
  • tts_nvalid は現在有効な属性数を示し、slot_getattr 呼び出し時に必要に応じて更新される。
  • スロット内のデータは tts_mcxt で指定されたメモリコンテキストの生存期間に依存する。

アルゴリズムまたは振る舞い

  • **遅延評価**: タプルからの属性値抽出は、slot_getattr が呼ばれるまで行われない。
  • **抽象化**: tts_ops を通じて、物理的なストレージ形式（ヒープ、バッファ、仮想タプル等）を意識せずにデータアクセスが可能。
  • **マテリアライズ**: ExecMaterializeSlot を呼び出すことで、外部リソースへの依存を断ち切り、メモリ上に実体化できる。
  • **クリア**: ExecClearTuple により、スロット内のデータを解放し、再利用可能な状態にする。

呼び出し元と呼び出し先

  • **呼び出し元**: contrib/amcheck/verify_nbtree.c、contrib/file_fdw/file_fdw.c、contrib/postgres_fdw/postgres_fdw.c 等の各スキャンノードや外部データラッパー。
  • **呼び出し先**: tts_ops が指す関数ポインタ（getsomeattrs 等）。具体的な呼び出し先の実装は TupleTableSlotOps に依存する。

関連データ構造

  • TupleTableSlotOps: スロットの挙動を定義する仮想関数テーブル。
  • TupleDesc: タプルの属性定義。
  • MemoryContext: スロットのメモリ管理領域。

失敗モードと境界条件

  • tts_nvalid が属性数を超えてアクセスされた場合、未定義の動作を引き起こす可能性がある。
  • 仮想タプルを使用する場合、参照元のメモリ領域が解放される前にスロットへアクセスすると不正なメモリ参照が発生する。

保守担当者向けメモ

  • tts_ops の実装を追加することで、コアコードを変更せずに新しいタプル形式をサポート可能。
  • 仮想タプルは物理コピーを避ける最適化であるため、生存期間管理には細心の注意を払うこと。
