ApplyWalRecord

定義

  • 種別: function
  • 定義箇所: src/backend/access/transam/xlogrecovery.c:1927

目的

WALレコードを解析し、対応するリソースマネージャのRedoルーチンを実行してデータベースの状態を復元する。リカバリプロセスにおけるWAL適用ループの主要な処理単位である。

シグネチャまたは形状

static void ApplyWalRecord(XLogReaderState *xlogreader, XLogRecord *record, TimeLineID *replayTLI)

入力と出力

  • **入力**:
  • xlogreader: WALレコードを読み込むための状態管理構造体。
  • record: 適用対象のWALレコード。
  • replayTLI: 現在のタイムラインID（ポインタ経由で更新される）。
  • **出力**: なし（副作用としてデータベース状態の更新および共有メモリのリカバリ状態更新を行う）。

前提条件と不変条件

  • xlogreader は有効なWALレコードを保持していること。
  • XLogRecoveryCtl 共有メモリ構造体が初期化されていること。
  • rm_redo 実行中、エラー発生時に備えて error_context_stack が適切に設定されること。

アルゴリズムまたは振る舞い

1. **エラーハンドリング設定**: rm_redo_error_callback をエラーコンテキストスタックに登録。 2. **XID更新**: AdvanceNextFullTransactionIdPastXid を呼び出し、次回のトランザクションIDをレコードのXIDより先に進める。 3. **タイムライン切り替え判定**: レコードが RM_XLOG_ID であり、かつチェックポイントやリカバリ終了を示す場合、タイムラインの切り替えを検証・実行する。 4. **共有メモリ更新**: XLogRecoveryCtl の replayEndRecPtr 等を更新し、リカバリの進捗を記録。 5. **スタンバイ処理**: ホットスタンバイモードであれば、XIDを既知の割り当て済みトランザクションとして記録。 6. **Redo実行**: xlogrecovery_redo（リカバリ専用処理）および各リソースマネージャの rm_redo を実行。 7. **整合性検証**: XLR_CHECK_CONSISTENCY フラグがある場合、バックアップページの整合性を検証。 8. **事後処理**: lastReplayed 関連のポインタを更新し、必要に応じてWAL送信プロセスの起床、リカバリ整合性のチェック、およびタイムライン切り替え時の不要ファイル削除を行う。

呼び出し元と呼び出し先

  • **呼び出し元**: PerformWalRecovery (src/backend/access/transam/xlogrecovery.c)
  • **呼び出し先**:
  • GetRmgr(record->xl_rmid).rm_redo()
  • xlogrecovery_redo()
  • checkTimeLineSwitch()
  • verifyBackupPageConsistency()
  • WalSndWakeup()
  • WalRcvForceReply()
  • CheckRecoveryConsistency()
  • RemoveNonParentXlogFiles()

関連データ構造

  • XLogReaderState: WAL読み込み状態。
  • XLogRecord: WALレコード本体。
  • XLogRecoveryCtlData: リカバリの共有メモリ状態。

失敗モードと境界条件

  • rm_redo 実行中にエラーが発生した場合、ereport によりリカバリが中断される。
  • タイムライン切り替え時に checkTimeLineSwitch の検証に失敗した場合、エラーが発生する。
  • 整合性チェック（verifyBackupPageConsistency）で不一致が検出された場合、パニックまたはエラーとなる。

保守担当者向けメモ

  • 本関数はリカバリの心臓部であり、各リソースマネージャの rm_redo を呼び出すため、ここでの変更は全リソースマネージャの動作に影響する。
  • タイムライン切り替え時の RemoveNonParentXlogFiles は、将来のWALセグメントを削除する破壊的な処理を含むため、慎重に扱うこと。
  • XLogRecoveryCtl へのアクセスには info_lck スピンロックが必須である。
