btgettuple

定義

  • 種別: function
  • 定義箇所: src/backend/access/nbtree/nbtree.c:225

目的

B-tree インデックススキャンにおいて、指定された方向（dir）に従って次のタプルを取得します。

シグネチャまたは形状

bool btgettuple(IndexScanDesc scan, ScanDirection dir)

入力と出力

  • **入力**:
  • scan: インデックススキャンの状態を保持する IndexScanDesc 構造体。
  • dir: スキャン方向（前方または後方）。
  • **出力**:
  • bool: タプルが見つかった場合は true、それ以外は false を返します。

前提条件と不変条件

  • scan->heapRelation が NULL でないこと（Assert による検証）。
  • B-tree インデックスは非損失（lossy）であるため、scan->xs_recheck は常に false に設定されます。

アルゴリズムまたは振る舞い

1. スキャンが未初期化の場合、_bt_first() を呼び出して最初のアイテムを取得します。 2. スキャンが既に初期化されている場合:

  • scan->kill_prior_tuple が真であれば、直前に取得したタプルを削除対象として so->killedItems に記録します。
  • _bt_next() を呼び出して次のアイテムへ進みます。

    • 3. so->numArrayKeys が存在する場合、現在のプリミティブスキャンが終了しても、_bt_start_prim_scan() を用いて次のプリミティブスキャンを継続的に試行します。 4. タプルが見つかった時点でループを終了し、結果を返します。

呼び出し元と呼び出し先

  • **呼び出し元**: インデックスアクセスメソッドのルーチン（amgettuple）として登録されており、PostgreSQL の実行エンジンから呼び出されます。
  • **呼び出し先**:
  • _bt_first(): 初回スキャン時のエントリポイント。
  • _bt_next(): 継続スキャン時のエントリポイント。
  • _bt_start_prim_scan(): ScalarArrayOp 等の複数プリミティブスキャンが必要な場合の制御。

関連データ構造

  • IndexScanDesc: スキャン全体のメタデータ。
  • BTScanOpaque (scan->opaque): B-tree 特有のスキャン状態（現在の位置 currPos、削除対象タプル killedItems 等）。

失敗モードと境界条件

  • MaxTIDsPerBTreePage を超える削除対象タプルが記録される場合、超過分は無視されます。
  • スキャン方向の反転により、同一アイテムが複数回削除対象として記録される可能性がありますが、本関数では最適化を行わずそのまま処理します。

保守担当者向けメモ

  • 本関数はスキャンライフサイクルのメインループです。ページ境界を跨ぐ処理や並列スキャンの詳細は nbtsearch.c の _bt_next 等に委譲されています。
  • 削除対象タプルの追跡は、インデックスページを離れる直前に一括処理される前提で実装されています。
