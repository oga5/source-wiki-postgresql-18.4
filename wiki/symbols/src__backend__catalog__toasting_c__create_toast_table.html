create_toast_table

定義

  • 種別: function
  • 定義箇所: src/backend/catalog/toasting.c:126

目的

指定されたリレーションに対して、TOASTテーブルおよびそのインデックスを構築し、システムカタログへの登録と依存関係の設定を行う内部作業用関数です。

シグネチャまたは形状

static bool create_toast_table(Relation rel, Oid toastOid, Oid toastIndexOid,
                               Datum reloptions, LOCKMODE lockmode, bool check,
                               Oid OIDOldToast)

入力と出力

  • **入力**:
  • rel: 対象となるメインリレーション。
  • toastOid, toastIndexOid: 新規作成するテーブル/インデックスのOID（指定がない場合は InvalidOid）。
  • reloptions: TOASTテーブルに適用するオプション。
  • lockmode: ロックモード。
  • check: ロックモードの妥当性チェックを行うかどうかのフラグ。
  • OIDOldToast: バイナリアップグレード時に使用する既存のTOASTテーブルOID。
  • **出力**:
  • bool: TOASTテーブルが正常に作成された場合は true、既に存在する場合や作成不要と判断された場合は false を返します。

前提条件と不変条件

  • 対象リレーションが既にTOASTテーブルを持っている場合、処理は直ちに終了します。
  • check が true の場合、lockmode は AccessExclusiveLock である必要があります。
  • TOASTテーブルの各カラム（chunk_id, chunk_seq, chunk_data）は、再帰的なTOAST発生を防ぐため、強制的に TYPSTORAGE_PLAIN に設定されます。
  • TOASTテーブルの各カラムは圧縮が無効化されます。

アルゴリズムまたは振る舞い

1. **存在確認**: rel->rd_rel->reltoastrelid を確認し、既にTOASTテーブルがあれば終了。 2. **生成判定**: needs_toast_table を呼び出し、TOASTテーブルが必要か判定（IsBinaryUpgrade 時は例外処理あり）。 3. **スキーマ定義**: chunk_id (OID), chunk_seq (INT4), chunk_data (BYTEA) を持つタプル記述子を作成。 4. **名前解決**: pg_toast_ プレフィックスを付与したテーブル名およびインデックス名を生成。 5. **テーブル作成**: heap_create_with_catalog を呼び出し、RELKIND_TOASTVALUE としてテーブルを作成。 6. **可視化**: CommandCounterIncrement() を実行し、作成したテーブルを後続処理から参照可能にする。 7. **インデックス構築**: chunk_id と chunk_seq をキーとするユニークインデックスを定義（スライスアクセス効率化のため）。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • NewRelationCreateToastTable
  • AlterTableCreateToastTable
  • CheckAndCreateToastTable
  • **呼び出し先**:
  • needs_toast_table
  • heap_create_with_catalog
  • table_open
  • makeNode (IndexInfoの作成)

関連データ構造

  • Relation: メインリレーションおよび作成されるTOASTリレーション。
  • TupleDesc: TOASTテーブルのスキーマ定義。
  • IndexInfo: TOASTテーブル用インデックスの定義情報。

失敗モードと境界条件

  • check が true かつ lockmode が AccessExclusiveLock 以外の場合、elog(ERROR) が発生します。
  • バイナリアップグレードモードにおいて、binary_upgrade_next_toast_pg_class_oid が無効な場合は作成をスキップします。

保守担当者向けメモ

  • TOASTテーブルのインデックスは、単一の chunk_id でも動作可能ですが、スライスアクセス（部分読み込み）の効率化と重複チェックのため、chunk_id と chunk_seq の複合インデックスが必須となっています。
  • chunk_data は bytea 型であり、それ自体がTOAST可能であるため、再帰的なTOASTを防ぐための TYPSTORAGE_PLAIN 設定は極めて重要です。
