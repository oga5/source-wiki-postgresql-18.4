get_index_paths

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/path/indxpath.c:715

目的

指定されたインデックスと適合条件（IndexClauseSet）に基づき、利用可能な IndexPath を生成し、リレーションのパスリストへの登録およびビットマップスキャン用パスの収集を行う。

シグネチャまたは形状

static void get_index_paths(PlannerInfo *root, RelOptInfo *rel, IndexOptInfo *index, IndexClauseSet *clauses, List **bitindexpaths)

入力と出力

  • **入力**:
  • root: プランナのグローバル情報。
  • rel: 対象のリレーション情報。
  • index: 使用するインデックスのメタデータ。
  • clauses: インデックスに適合する条件セット。
  • **出力**:
  • bitindexpaths: ビットマップスキャンとして利用可能なパスが追加されるリスト（ポインタ経由）。

前提条件と不変条件

  • index は rel に属している必要がある。
  • clauses は match_restriction_clauses_to_index 等で事前に収集された有効な条件セットであること。

アルゴリズムまたは振る舞い

1. **パス生成**: build_index_paths を呼び出し、インデックス条件から IndexPath リストを生成する。この際、インデックスがネイティブサポートしない ScalarArrayOpExpr の除外フラグを制御する。 2. **パスの登録と選別**: 生成された各パスを走査する。

  • amhasgettuple が真であれば、add_path を呼び出し、通常のインデックススキャンパスとして登録する。
  • amhasgetbitmap が真かつ、パスがソート目的のみでない（または選択率が 1.0 未満）場合、bitindexpaths に追加する。

    • 3. **非ネイティブ対応**: インデックスがネイティブサポートしない ScalarArrayOpExpr が存在する場合、ST_BITMAPSCAN モードで再度 build_index_paths を実行し、その結果を bitindexpaths に追加する。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/optimizer/path/indxpath.c 内の create_index_paths 等から呼び出される（ソース内の直接的な記述に基づく）。
  • **呼び出し先**:
  • build_index_paths
  • add_path
  • lappend / list_concat

関連データ構造

  • PlannerInfo: プランナ全体のコンテキスト。
  • RelOptInfo: リレーションの最適化情報。
  • IndexOptInfo: インデックスのプロパティ。
  • IndexClauseSet: インデックス列に適合する条件の集合。
  • IndexPath: 生成されるインデックスパスノード。

失敗モードと境界条件

  • インデックスが amhasgettuple も amhasgetbitmap もサポートしない場合、パスは生成されず、いずれのリストにも追加されない。
  • 部分インデックス（Partial Index）の場合、index->predOK に基づいて適合性が判定される。

保守担当者向けメモ

  • 本関数は、通常のインデックススキャンとビットマップスキャンの両方のパス生成を統括する。
  • ScalarArrayOpExpr の扱いにおいて、インデックスのネイティブサポート有無で処理が分岐するため、AM（アクセスメソッド）の機能拡張時にはこの分岐ロジックに注意すること。
