BtreeCheckState

定義

  • 種別: struct
  • 定義箇所: contrib/amcheck/verify_nbtree.c:70

目的

B-Tree インデックスの検証プロセス全体を通じて、検証対象のインデックス情報、現在の走査状態、および整合性チェックに必要な補助データ（Bloom フィルタ等）を保持するために使用されます。

シグネチャまたは形状

typedef struct BtreeCheckState {
    Relation rel;
    Relation heaprel;
    bool heapkeyspace;
    bool readonly;
    bool heapallindexed;
    bool rootdescend;
    bool checkunique;
    MemoryContext targetcontext;
    BufferAccessStrategy checkstrategy;
    IndexInfo *indexinfo;
    Snapshot snapshot;
    Page target;
    BlockNumber targetblock;
    XLogRecPtr targetlsn;
    IndexTuple lowkey;
    BlockNumber prevrightlink;
    bool previncompletesplit;
    bloom_filter *filter;
    int64 heaptuplespresent;
} BtreeCheckState;

入力と出力

本構造体は関数引数として渡され、検証の進捗に応じて内部の「可変状態（Mutable state）」フィールドが更新されます。直接的な入力・出力関数ではなく、検証アルゴリズムのコンテキストとして機能します。

前提条件と不変条件

  • readonly が設定されている場合のみ、lowkey（左兄弟のハイキー）が保持されます。
  • heapallindexed または checkunique が有効な場合、snapshot が適切に初期化されている必要があります。
  • 検証開始時に rel（インデックス）および heaprel（ヒープ）が正しく設定されている必要があります。

アルゴリズムまたは振る舞い

1. **初期化**: 検証開始時に、インデックスの属性や検証モード（readonly, heapallindexed 等）を固定状態として保持します。 2. **ページ走査**: target, targetblock, targetlsn を更新しながら、インデックスの各ページを順次検証します。 3. **リンク追跡**: prevrightlink および previncompletesplit を用いて、下位レベルのページ分割状態とダウンリンクの整合性を追跡します。 4. **ヒープ検証**: filter（Bloom フィルタ）を使用して、インデックスとヒープ間のタプル存在確認を行います。

呼び出し元と呼び出し先

  • **呼び出し元**: bt_index_check および bt_index_parent_check が本構造体を初期化し、検証の起点となります。
  • **呼び出し先**: 以下の関数が BtreeCheckState *state を引数として受け取り、状態を参照または更新します。
  • bt_check_level_from_leftmost
  • bt_target_page_check
  • bt_child_check
  • bt_downlink_missing_check
  • bt_entry_unique_check
  • bt_recheck_sibling_links
  • heap_entry_is_visible

関連データ構造

  • Relation: 検証対象のインデックスおよびヒープ。
  • IndexInfo: 一意性チェックに使用されるインデックス情報。
  • bloom_filter: heapallindexed モードでのヒープ整合性検証に使用。
  • IndexTuple: lowkey として保持されるページ間の境界情報。

失敗モードと境界条件

  • インデックスが破損している場合、PageGetItemIdCareful 等の防御的関数を通じて検証が中断またはエラー報告されます。
  • readonly モードでない場合、親子の整合性チェックの一部がスキップされる可能性があります。
  • メモリ不足時、targetcontext の管理下でエラーが発生する可能性があります。

保守担当者向けメモ

  • 本構造体は contrib/amcheck/verify_nbtree.c 内の検証ロジック全体で共有されます。フィールドを追加する際は、それが「不変（初期化時のみ設定）」か「可変（ページごとに更新）」かを明確に区別してください。
  • readonly モードの有無によって、一部のフィールド（lowkey 等）の有効性が変わる点に注意してください。
