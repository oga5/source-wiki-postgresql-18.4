heap_xlog_delete

定義

  • 種別: function
  • 定義箇所: src/backend/access/heap/heapam_xlog.c:340

目的

ヒープページに対する削除操作のWAL（Write-Ahead Log）リプレイを実行し、ページ上のタプル状態を更新して整合性を維持する。

シグネチャまたは形状

static void heap_xlog_delete(XLogReaderState *record)

入力と出力

  • **入力**: XLogReaderState *record（リプレイ対象のWALレコード）。
  • **出力**: なし（副作用としてバッファおよび可視性マップを更新）。

前提条件と不変条件

  • record は xl_heap_delete 構造体を含む有効なWALレコードであること。
  • 対象ページが XLogReadBufferForRedo によって取得可能であること。
  • ページ上の ItemId が正常（ItemIdIsNormal）であること。

アルゴリズムまたは振る舞い

1. **可視性マップの更新**: XLH_DELETE_ALL_VISIBLE_CLEARED フラグが設定されている場合、対象ブロックの可視性マップビットをクリアする。 2. **バッファの読み込み**: XLogReadBufferForRedo を呼び出し、ページが最新でない（BLK_NEEDS_REDO）場合にのみ以下の更新を行う。

  • **タプルヘッダーの更新**: infomask および infomask2 を修正し、HOT更新フラグをクリアする。
  • **Xmax/Xminの設定**: XLH_DELETE_IS_SUPER フラグに基づき、Xmaxを設定するか、Xminを無効化する。
  • **Cmaxの設定**: FirstCommandId を設定する。
  • **メタデータ更新**: ページをプルーニング候補としてマークし、必要に応じて PageClearAllVisible を実行する。
  • **ctidの更新**: パーティション移動フラグに応じて t_ctid を設定する。

    • 3. **コミット**: ページのLSNを更新し、バッファをダーティとしてマークする。 4. **終了処理**: バッファが取得されている場合はロックを解放する。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/access/heap/heapam_xlog.c 内の heap_redo（推測されるディスパッチャ）。
  • **呼び出し先**: XLogRecGetData, XLogRecGetBlockTag, visibilitymap_pin, visibilitymap_clear, XLogReadBufferForRedo, PageGetItemId, fix_infomask_from_infobits, PageSetPrunable, PageSetLSN, MarkBufferDirty, UnlockReleaseBuffer 等。

関連データ構造

  • xl_heap_delete: 削除操作のWALレコード構造体。
  • XLogReaderState: WALレコード読み取り状態。
  • Buffer: ヒープページを保持するバッファ。
  • HeapTupleHeader: 更新対象のタプルヘッダー。

失敗モードと境界条件

  • PageGetMaxOffsetNumber(page) < xlrec->offnum または !ItemIdIsNormal(lp) の場合、PANIC を発生させてリプレイを停止する。

保守担当者向けメモ

  • 本関数は物理的なページ操作に特化している。論理的な整合性はWAL生成側（heapam.c）で保証されている前提である。
  • XLH_DELETE_ALL_VISIBLE_CLEARED フラグの処理は、ヒープページが既に最新であっても実行される必要があるため、バッファ取得の条件分岐の外側に配置されている。
