ginxlogInsert

定義

  • 種別: struct
  • 定義箇所: src/include/access/ginxlog.h:37

目的

GINインデックスの挿入操作におけるWAL（Write-Ahead Log）レコードの共通ヘッダーとして機能します。

シグネチャまたは形状

typedef struct {
    uint16 flags; /* GIN_INSERT_ISLEAF および/または GIN_INSERT_ISDATA */
} ginxlogInsert;

入力と出力

  • **入力**: WALレコード生成時に flags フィールドが設定され、後続データとして挿入内容が配置されます。
  • **出力**: WAL再生（REDO）時に、この構造体を通じて挿入操作の種別（リーフかデータか）が識別されます。

前提条件と不変条件

  • 本構造体の直後に可変長データが配置されるレイアウトを前提としています。
  • 構造体内のフィールド順序とサイズは、バイナリ互換性を維持するため厳密に管理されています。
  • 本構造体自体は16ビットアライメントですが、後続する構造体（ginxlogInsertEntry 等）は、本構造体に付随する場合にのみ16ビットアライメントとなります。

アルゴリズムまたは振る舞い

1. **記録**: GINインデックス更新時、flags を設定し、XLogRegisterData を通じてWALに書き出されます。 2. **再生**: gin_redo 関数がWALレコードを読み取り、本構造体をキャストして取得します。 3. **分岐**: flags の値に基づき、リーフページかデータページか、あるいは非リーフページの場合は左右の子ブロック番号（BlockIdData[2]）を読み取り、適切なREDO処理へ分岐します。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/access/gin/ginbtree.c においてWALレコードの登録に使用されています。
  • **呼び出し先**: src/backend/access/gin/ginxlog.c のREDO処理および src/backend/access/rmgrdesc/gindesc.c のWAL記述子生成処理で参照されています。

関連データ構造

  • ginxlogInsertEntry
  • ginxlogRecompressDataLeaf
  • BlockIdData

失敗モードと境界条件

  • 構造体にフィールドを追加する際、アライメントによるパディングが発生するとWALのバイナリ互換性が破壊されるリスクがあります。
  • 16ビットアライメントの制約を逸脱するフィールド追加は避ける必要があります。

保守担当者向けメモ

  • 本構造体は可変長データを伴うため、memcpy 等での直接的な読み書きが行われます。変更時はバイナリレイアウトの整合性に細心の注意を払ってください。
  • トラブルシューティング時には src/backend/access/rmgrdesc/gindesc.c の gin_desc 関数を介して pg_waldump で内容を確認可能です。
