make_expanded_record_from_typeid

定義

  • 種別: function
  • 定義箇所: src/backend/utils/adt/expandedrecord.c:68

目的

指定された型ID（Oid）に基づき、メモリ上で操作可能な空の「展開されたレコード（Expanded Record）」オブジェクトを生成します。

シグネチャまたは形状

ExpandedRecordHeader *make_expanded_record_from_typeid(Oid type_id, int32 typmod, MemoryContext parentcontext)

入力と出力

  • **入力**:
  • type_id: 対象となる複合型またはレコード型の Oid。
  • typmod: 型修飾子（RECORDOID の場合に利用）。
  • parentcontext: オブジェクトのメモリコンテキストを作成するための親コンテキスト。
  • **出力**:
  • 初期化された ExpandedRecordHeader 構造体へのポインタ。

前提条件と不変条件

  • type_id が RECORDOID でない場合、typcache を通じて複合型またはドメイン型として解決可能である必要があります。
  • 解決された型が複合型でない場合、エラー（ERRCODE_WRONG_OBJECT_TYPE）が発生します。
  • 生成されたレコードは、初期状態では論理的に空（フィールド値未設定）です。

アルゴリズムまたは振る舞い

1. **型解決**: type_id に応じて typcache から TupleDesc を取得します。ドメイン型の場合はその基底型を辿ります。 2. **メモリ確保**: parentcontext の下に専用のメモリコンテキストを作成し、ヘッダーおよびフィールド格納用の dvalues/dnulls 配列を一度に確保します。 3. **初期化**: EOH_init_header を呼び出し、ER_methods を設定して展開オブジェクトとして登録します。 4. **メタデータ設定**: 型情報、typmod、および tupdesc_id をヘッダーに格納します。 5. **参照カウント管理**: TupleDesc が参照カウントを持つ場合、メモリコンテキストのリセットコールバック（ER_mc_callback）を登録し、自身で参照カウントをインクリメントします。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/pl/plpgsql/src/pl_exec.c（PL/pgSQLの実行エンジンにおいて、レコード変数の初期化や更新時に利用）
  • **呼び出し先**:
  • lookup_type_cache, lookup_rowtype_tupdesc, assign_record_type_identifier
  • AllocSetContextCreate, MemoryContextAlloc, EOH_init_header
  • MemoryContextRegisterResetCallback, ReleaseTupleDesc

関連データ構造

  • ExpandedRecordHeader: 展開レコードのメイン構造体。
  • TupleDesc: 複合型の構造定義。
  • TypeCacheEntry: 型情報のキャッシュ。

失敗モードと境界条件

  • 指定された type_id が複合型でない場合、ereport によりエラー終了します。
  • メモリ不足時には palloc 関連関数によりエラーが発生します。

保守担当者向けメモ

  • 本関数は ER_FLAG_DVALUES_VALID や ER_FLAG_FVALUE_VALID を設定しないため、呼び出し側は必要に応じてフィールド値を設定する必要があります。
  • TupleDesc の参照カウント管理はメモリコンテキストのライフサイクルに依存しているため、コールバックの登録ロジックを変更する際は注意が必要です。
