dsm_segment

定義

  • **種別**: struct
  • **定義箇所**: src/backend/storage/ipc/dsm.c:66

目的

バックエンドローカルな動的共有メモリ（DSM）セグメントの管理を行う構造体です。各バックエンドが現在接続しているセグメントの状態を追跡し、リソース所有権やマッピング情報を保持します。

シグネチャまたは形状

struct dsm_segment
{
    dlist_node    node;             /* dsm_segment_list 内のリンク */
    ResourceOwner resowner;         /* リソース所有者 */
    dsm_handle    handle;           /* セグメント識別子 */
    uint32        control_slot;     /* 制御セグメント内のスロット番号 */
    void         *impl_private;     /* 実装依存のプライベートデータ */
    void         *mapped_address;   /* マッピング済みアドレス（未マッピング時は NULL） */
    Size          mapped_size;      /* マッピングサイズ */
    slist_head    on_detach;        /* デタッチ時コールバックリスト */
};

入力と出力

本構造体は関数ではなくデータ構造であるため、直接的な入出力はありません。ただし、dsm_create や dsm_attach によって初期化され、dsm_detach によって参照されるライフサイクル管理の単位となります。

前提条件と不変条件

  • mapped_address が NULL でない場合、そのアドレスは mapped_size 分の有効な共有メモリ領域を指している必要があります。
  • resowner は、トランザクション終了時に自動的にデタッチ処理をトリガーするために使用されます。

アルゴリズムまたは振る舞い

  • **リソース追跡**: dlist_node を用いて、各バックエンドが保持するセグメントリスト（dsm_segment_list）に登録されます。
  • **デタッチ処理**: dsm_detach が呼び出されると、on_detach リストに登録されたコールバックが実行され、その後リソースが解放されます。
  • **参照管理**: 制御セグメント内の control_slot を通じて、全バックエンド間で共有される参照カウントと同期します。

  • **呼び出し先**:
  • 本構造体自体はデータ構造であり、特定の関数を呼び出すことはありません。操作は dsm.c 内の API（dsm_detach 等）を介して行われます。

関連データ構造

  • dsm_control_header: 制御セグメントの全体レイアウトを定義。
  • dsm_control_item: 制御セグメント内の個別のセグメント状態を管理。
  • ResourceOwner: リソースの生存期間を管理。
  • dlist_node / slist_head: lib/ilist.h で定義されるリスト管理構造。

失敗モードと境界条件

  • mapped_address が NULL の状態でアクセスを試みた場合、セグメントが正しくマッピングされていないことを示します。
  • 異常終了時に resowner によるクリーンアップが適切に行われない場合、セグメントがリークする可能性があります（ポストマスター起動時のクリーンアップで復旧）。

保守担当者向けメモ

  • impl_private は OS 依存の低レベル実装（dsm_impl.c）が使用する領域であり、直接操作すべきではありません。
  • dsm_detach はエラーを投げない設計であるため、本構造体を用いたクリーンアップ処理は堅牢である必要があります。
