pull_varnos_context

定義

  • 種別: struct
  • 定義箇所: src/backend/optimizer/util/var.c:33

目的

式ツリーの走査中に、特定のクエリレベルに関連する varno（リレーション番号）を収集するための状態を保持するコンテキスト構造体です。

シグネチャまたは形状

typedef struct
{
	Relids		varnos;         /* 収集されたリレーション番号のセット */
	PlannerInfo *root;          /* プランナのグローバル情報 */
	int			sublevels_up;   /* 現在走査中のサブクエリの深さ */
} pull_varnos_context;

入力と出力

本構造体は関数への引数として渡され、走査結果を蓄積するために使用されます。

  • **入力**: root（プランナ情報）および sublevels_up（初期値）を設定して初期化します。
  • **出力**: 走査完了後、varnos フィールドに収集された Relids が格納されます。

前提条件と不変条件

  • sublevels_up は、走査対象の式ツリーにおける現在のネストレベルを正しく反映している必要があります。
  • varnos は、bms_add_member 等のビットマップセット操作関数で適切に管理されることを前提としています。

アルゴリズムまたは振る舞い

expression_tree_walker を利用した走査処理において、以下の役割を果たします。 1. pull_varnos_walker がツリー内の Var ノードを検出した際、その varlevelsup がコンテキストの sublevels_up と一致するかを確認します。 2. 一致する場合、その varno を varnos ビットマップセットに追加します。 3. サブクエリや SubLink に遭遇した際、sublevels_up を適切に調整して再帰的に走査を継続します。

呼び出し元と呼び出し先

  • **呼び出し元**: pull_varnos および pull_varnos_of_level からインスタンス化され、利用されます。
  • **呼び出し先**: pull_varnos_walker がこの構造体を引数として受け取り、走査を実行します。

関連データ構造

  • Relids: varno を保持するためのビットマップセット型。
  • PlannerInfo: プランナの全体的な状態を保持する構造体。
  • Var: 式ツリー内の変数ノード。

失敗モードと境界条件

  • root が NULL の場合、プランナ情報に依存する処理で参照エラーが発生する可能性があります。
  • sublevels_up の管理が不適切な場合、サブクエリ内の変数を誤って現在のレベルの変数として収集する可能性があります。

保守担当者向けメモ

  • 本構造体は pull_varnos_walker と密結合しています。走査ロジックを変更する場合は、このコンテキストが保持するフィールドが十分か確認してください。
  • PlaceHolderVar は Var と同様に扱われるため、必要に応じて PlaceHolderVar の varno 抽出ロジックとの整合性を維持してください。
