BlockRefTableEntry

定義

  • 種別: struct
  • 定義箇所: src/common/blkreftable.c:110

目的

特定のリレーションフォークにおける変更済みブロックの状態を保持・管理するためのデータ構造です。

シグネチャまたは形状

struct BlockRefTableEntry
{
    BlockRefTableKey key;
    BlockNumber limit_block;
    char        status;
    uint32      nchunks;
    uint16     *chunk_size;
    uint16     *chunk_usage;
    BlockRefTableChunk *chunk_data;
};

入力と出力

本構造体は関数ではなくデータ構造であるため、直接的な入力・出力はありません。ただし、BlockRefTableMarkBlockModified 等の関数を通じて、key（リレーション識別子）に基づいた変更状態の更新や、シリアライズ処理におけるバイナリ出力の対象となります。

前提条件と不変条件

  • chunk は BLOCKS_PER_CHUNK (64KB) 単位で管理されます。
  • 変更密度に応じて、内部表現はオフセット配列からビットマップへ動的に切り替わります。一度ビットマップへ変換されると、オフセット配列には戻りません。
  • シリアライズ時には、tablespace -> database -> relfilenumber -> forknumber の順序でソートされている必要があります。

アルゴリズムまたは振る舞い

  • BlockRefTableMarkBlockModified が呼び出されると、対応するエントリが取得され、chunk データが更新されます。
  • nchunks は管理対象のチャンク数を保持し、chunk_data は実際のブロック変更状態を格納します。
  • limit_block はリレーションの切り詰めや再拡張を追跡するために使用されます。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/backup/basebackup_incremental.c および src/common/blkreftable.c 内の各関数（BlockRefTableWriteEntry 等）から参照されています。
  • **呼び出し先**: 本構造体自体はデータ保持用であり、直接関数を呼び出すことはありません。

関連データ構造

  • BlockRefTableKey: エントリを特定するためのキー。
  • BlockRefTableChunk: 実際のブロック変更状態を保持するデータ構造。
  • BlockRefTable: 本エントリを格納するハッシュテーブル。

失敗モードと境界条件

  • MAX_ENTRIES_PER_CHUNK に達した際のビットマップへの切り替え処理が境界条件となります。
  • メモリ割り当て失敗時の挙動については、ソースコードの文脈からは明示されていません。

保守担当者向けメモ

  • 本構造体は simplehash を使用したハッシュテーブルの要素として定義されています。
  • src/include/common/blkreftable.h にも定義が存在するため、変更時は両方の整合性に注意してください。
