BlockInfoRecord

定義

  • 種別: struct
  • 定義箇所: contrib/pg_prewarm/autoprewarm.c:55

目的

共有バッファの状態を永続化および復元するために、個々のデータブロックを識別するメタデータを保持する。

シグネチャまたは形状

typedef struct BlockInfoRecord
{
    Oid         database;
    Oid         tablespace;
    RelFileNumber filenumber;
    ForkNumber  forknum;
    BlockNumber blocknum;
} BlockInfoRecord;

入力と出力

  • 本構造体は、autoprewarm.blocks ファイルから読み込まれた情報、または共有バッファから抽出された情報を格納する。
  • プリウォーム処理時には、この構造体に基づいて対象ブロックの読み込みが行われる。

前提条件と不変条件

  • データベースの再起動時、この構造体の配列が動的共有メモリ（DSM）上に展開される。
  • プリウォーム実行前に、qsort を用いてデータベースやファイル番号順にソートされることが一般的である。

アルゴリズムまたは振る舞い

  • 共有バッファのダンプ時、各ブロックの識別子として本構造体が生成される。
  • 復元時、リーダーワーカーがファイルから読み込んだ BlockInfoRecord の配列を DSM に配置し、各データベース担当ワーカーがこれを参照して read_stream API を介したブロック読み込みを実行する。

呼び出し元と呼び出し先

  • 呼び出し元: autoprewarm.c 内の各関数（apw_load_buffers 等）が、DSM 領域の確保やソート、ブロック情報の読み取りのために本構造体を使用している。
  • 呼び出し先: 直接的な関数呼び出しは行わないデータ構造である。

関連データ構造

  • AutoPrewarmSharedState: 共有メモリ上で本構造体の配列を管理するための制御構造体。

失敗モードと境界条件

  • 共有メモリの容量制限により、格納可能な BlockInfoRecord の数には上限がある。
  • データベースの削除やテーブルのドロップにより、記録された BlockInfoRecord が指すブロックが物理的に存在しなくなった場合、プリウォーム処理時にスキップされる必要がある。

保守担当者向けメモ

  • 本構造体は autoprewarm.blocks ファイルの形式と密接に関連している。ファイル形式を変更する場合は、本構造体の定義と読み書きロジックの両方を更新すること。
