get_row_security_policies

定義

  • 種別: function
  • 定義箇所: src/backend/rewrite/rowsecurity.c:97

目的

クエリのリライトフェーズにおいて、指定されたリレーション（テーブル）に対する行レベルセキュリティ（RLS）ポリシーを収集し、クエリツリーに適用すべき制約（securityQuals および WithCheckOptions）を生成する。

シグネチャまたは形状

void get_row_security_policies(Query *root, RangeTblEntry *rte, int rt_index,
                               List **securityQuals, List **withCheckOptions,
                               bool *hasRowSecurity, bool *hasSubLinks)

入力と出力

  • **入力**:
  • root: 現在のクエリツリー。
  • rte: 対象となるリレーションの範囲テーブルエントリ。
  • rt_index: リレーションのインデックス。
  • **出力**:
  • securityQuals: 読み取り制限用のクエリリスト。
  • withCheckOptions: 書き込み制限用の制約リスト。
  • hasRowSecurity: RLSが有効であるか、または環境変化による再プランが必要な場合に true。
  • hasSubLinks: 生成された制約にサブリンクが含まれる場合に true。

前提条件と不変条件

  • rte->rtekind は RTE_RELATION である必要がある（Assert で検証）。
  • 対象が通常のテーブルまたはパーティションテーブルでない場合、処理は即座に終了する。
  • RLSが有効な場合、Permissive ポリシーは OR で、Restrictive ポリシーは AND で結合される。
  • ポリシーが存在しない場合、デフォルト拒否（always-false）の制約が追加される。

アルゴリズムまたは振る舞い

1. **初期化**: 出力引数をデフォルト値（NIL, false）に設定する。 2. **RLS状態判定**: check_enable_rls を呼び出し、対象テーブルのRLS状態を確認する。RLS_NONE_ENV の場合は hasRowSecurity を true にして終了する。 3. **ポリシー取得**: get_policies_for_relation を使用して、コマンドタイプ（SELECT, UPDATE等）に応じたポリシーリストを取得する。 4. **制約の注入**:

  • SELECT で UPDATE 権限が必要な場合、先に UPDATE 用の USING ポリシーを適用する。
  • SELECT, UPDATE, DELETE に対して add_security_quals を呼び出し、読み取り制限を適用する。
  • UPDATE, DELETE, MERGE で SELECT 権限が必要な場合、追加で SELECT 用の USING ポリシーを適用する。
  • INSERT, UPDATE に対して add_with_check_options を呼び出し、書き込み時の整合性を検証する。

関連データ構造

  • Query, RangeTblEntry: クエリツリー構造。
  • RTEPermissionInfo: 権限情報。
  • RowSecurityPolicy (間接的): get_policies_for_relation を通じて参照。

失敗モードと境界条件

  • RLS_NONE: RLSが設定されていない場合、何も行わずに戻る。
  • RLS_NONE_ENV: 環境依存でRLSが有効になる可能性がある場合、再プランフラグを立てて戻る。
  • Assert 違反: rt_index が root->resultRelation と一致しない場合（INSERT/UPDATE時）に発生する可能性がある。

保守担当者向けメモ

  • MERGE コマンド処理時は、UPDATE/DELETE/INSERTの各ポリシーが状況に応じて適用されるため、ロジックの追跡には注意が必要。
  • 拡張モジュールによるポリシー追加は get_policies_for_relation 内のフック経由で行われる。
