ExecMaterializeSlot

定義

  • 種別: static inline function
  • 定義箇所: src/include/executor/tuptable.h:475

目的

タプルテーブルスロット内のタプルをメモリ上に実体化（マテリアライズ）し、外部リソース（バッファや他のメモリ領域）への依存を断ち切るために使用されます。

シグネチャまたは形状

static inline void ExecMaterializeSlot(TupleTableSlot *slot)

入力と出力

  • **入力**: TupleTableSlot *slot - 操作対象となるタプルテーブルスロット。
  • **出力**: なし（副作用としてスロットの状態が更新される）。

前提条件と不変条件

  • slot は有効な TupleTableSlot を指している必要があります。
  • slot->tts_ops が適切に初期化されており、materialize 関数ポインタが実装されている必要があります。

アルゴリズムまたは振る舞い

1. スロットの tts_ops（TupleTableSlotOps）構造体を参照します。 2. tts_ops->materialize 関数ポインタを呼び出し、スロット固有の実装に従ってタプルのコピーやメモリ確保を実行します。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/commands/copyfrom.c
  • src/backend/commands/trigger.c
  • src/backend/executor/execMain.c
  • src/backend/executor/execReplication.c
  • src/backend/executor/execTuples.c
  • src/backend/executor/functions.c
  • src/backend/executor/nodeModifyTable.c
  • **呼び出し先**:
  • slot->tts_ops->materialize（スロットの型に応じた実装関数）。

関連データ構造

  • TupleTableSlot: タプルを保持する基本構造体。
  • TupleTableSlotOps: スロットの挙動を定義する仮想関数テーブル。

失敗モードと境界条件

  • スロットが既にクリアされている場合や、マテリアライズできない状態のスロットに対して呼び出した場合の挙動は、各 tts_ops の実装に依存します。
  • メモリ不足時には、materialize 実装内でエラーが発生する可能性があります。

保守担当者向けメモ

  • 本関数は単なるラッパーであり、実際の処理は tts_ops に委譲されます。スロットの型（TTSOpsVirtual, TTSOpsHeapTuple 等）ごとのマテリアライズ挙動を理解するには、各 TupleTableSlotOps の実装を確認してください。
