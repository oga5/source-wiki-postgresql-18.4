has_subclass

定義

  • 種別: function
  • 定義箇所: src/backend/catalog/pg_inherits.c:354

目的

指定されたリレーション（テーブル）が子テーブルを持っている可能性があるかを判定する。主にクエリ最適化や継承展開の際、継承関係の探索をスキップするための効率化ハックとして利用される。

シグネチャまたは形状

bool has_subclass(Oid relationId)

入力と出力

  • **入力**: relationId (Oid) - 検査対象のリレーションのOID。
  • **出力**: bool - pg_class の relhassubclass フラグの値。

前提条件と不変条件

  • 指定された relationId がシステムカタログ pg_class に存在すること。
  • 呼び出し元は、対象のリレーションに対して適切なロックを保持していることが推奨される。

アルゴリズムまたは振る舞い

1. RELOID システムキャッシュを使用して、指定された relationId に対応する pg_class のタプルを検索する。 2. タプルが見つからない場合はエラーを発生させる。 3. 取得したタプルの relhassubclass フィールド（Form_pg_class 構造体）を参照する。 4. システムキャッシュを解放し、取得したフラグ値を返す。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/catalog/pg_inherits.c (複数箇所)
  • src/backend/optimizer/plan/planner.c
  • src/backend/optimizer/util/inherit.c (過去の参照履歴あり)
  • **呼び出し先**:
  • SearchSysCache1
  • GETSTRUCT
  • ReleaseSysCache
  • elog

関連データ構造

  • Form_pg_class: pg_class カタログの行構造体。relhassubclass フィールドを含む。
  • HeapTuple: システムキャッシュから取得されるタプルデータ。

失敗モードと境界条件

  • **キャッシュルックアップ失敗**: 指定された relationId が存在しない場合、elog(ERROR, ...) により処理が中断される。
  • **精度の限界**: 本関数は「かつて子が存在した」場合でも真を返す可能性があるため、継承関係の正確な判定には使用できない。正確な判定が必要な場合は、他の継承探索関数を使用する必要がある。

保守担当者向けメモ

  • 本関数はあくまで効率化のためのヒントとして使用すること。
  • relhassubclass フラグは、一度でも子テーブルが作成されると真になり、その後子テーブルが削除されても偽に戻らない場合があるため、真であっても必ずしも現在の子テーブルの存在を保証しない。
