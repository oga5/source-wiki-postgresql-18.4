SQLFunctionCache

定義

  • 種別: struct
  • 定義箇所: src/backend/executor/functions.c:142

目的

SQL関数（SQL言語で記述された関数）の実行状態を保持し、関数内のクエリ実行、プランキャッシュの再利用、および結果セットの管理を行うための構造体です。

シグネチャまたは形状

typedef struct SQLFunctionCache {
    SQLFunctionHashEntry *func;
    bool        active;
    bool        lazyEvalOK;
    bool        shutdown_reg;
    bool        lazyEval;
    bool        randomAccess;
    bool        ownSubcontext;
    ParamListInfo paramLI;
    Tuplestorestate *tstore;
    MemoryContext tscontext;
    JunkFilter *junkFilter;
    int         jf_generation;
    CachedPlan *cplan;
    ResourceOwner cowner;
    int         next_query_index;
    execution_state *eslist;
    execution_state *esarray;
    int         esarray_len;
    int         error_query_index;
    MemoryContext fcontext;
    MemoryContext jfcontext;
    MemoryContext subcontext;
    MemoryContextCallback mcb;
} SQLFunctionCache;

入力と出力

  • **入力**: FmgrInfo の fn_extra フィールドを介して保持され、関数呼び出しごとの引数（ParamListInfo）や実行コンテキストを管理します。
  • **出力**: 実行結果は Tuplestorestate に蓄積されるか、単一のタプルとして返されます。

前提条件と不変条件

  • FmgrInfo の fn_extra に格納される補助データとして機能します。
  • fcontext は構造体自体と関連するすべての補助データを保持するメモリコンテキストです。
  • active フラグは、現在そのキャッシュエントリが実行中であることを示します。

アルゴリズムまたは振る舞い

1. **初期化**: init_sql_fcache により生成され、関数定義に基づくキャッシュエントリとリンクされます。 2. **実行制御**: 関数内の各クエリを順次処理します。next_query_index で次に実行すべきクエリを追跡し、execution_state を通じて各クエリの実行状態を管理します。 3. **遅延評価**: lazyEval が有効な場合、セットを返す関数（SRF）において全行を一度に計算せず、呼び出しごとに1行ずつ結果を生成します。 4. **終了処理**: すべてのクエリ完了後、ShutdownSQLFunction 等によりリソースが解放されます。

呼び出し元と呼び出し先

  • **呼び出し元**: fmgr_sql がこの構造体を使用してSQL関数の実行を制御します。
  • **呼び出し先**: postquel_start（クエリ開始）、ExecutorRun（クエリ実行）、postquel_end（クエリ終了）を呼び出します。
  • ※具体的な呼び出し元・呼び出し先の網羅的なリストは提供されていません。

関連データ構造

  • SQLFunctionHashEntry: 関数定義に基づく長寿命なキャッシュ。
  • execution_state: 関数内の個々のクエリの実行状態を保持するリスト。
  • Tuplestorestate: 関数がセットを返す場合のデータ蓄積先。
  • JunkFilter: 戻り値から不要な列を除去するために使用。

失敗モードと境界条件

  • active フラグが立っている状態でエラーが発生した場合、再入や不整合を防ぐための処理が必要です。
  • error_query_index はエラー発生時のトレースバックを支援するために使用されます。
  • lazyEval が安全でない場合（lazyEvalOK が false）、全結果を一度に計算する必要があります。

保守担当者向けメモ

  • 本構造体は FmgrInfo の fn_extra に紐付くため、メモリ管理（fcontext 等）のライフサイクルに注意してください。
  • lazyEval モードの切り替えは、セットを返す関数のパフォーマンスに直結する重要なロジックです。
  • エラー発生時のコンテキスト保持のため、sql_exec_error_callback との連携が不可欠です。
