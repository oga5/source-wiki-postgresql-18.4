RelfilenumberMapEntry

定義

  • 種別: struct
  • 定義箇所: src/backend/utils/cache/relfilenumbermap.c:41

目的

(tablespace, relfilenumber) のペアから relid (pg_class.oid) へのマッピングを保持し、RelfilenumberMapHash ハッシュテーブル内でキャッシュエントリとして機能します。

シグネチャまたは形状

typedef struct {
    RelfilenumberMapKey key; /* 検索キー - 構造体の先頭である必要がある */
    Oid         relid;       /* pg_class.oid */
} RelfilenumberMapEntry;

入力と出力

  • 入力: RelfilenumberMapKey を含む構造体としてハッシュテーブルに格納される。
  • 出力: キャッシュヒット時に relid フィールドが参照され、対応する OID が返される。

前提条件と不変条件

  • key フィールドは構造体の先頭に配置されなければならない（ハッシュテーブルの検索キーとして機能するため）。
  • relid には有効な OID、または検索失敗を示す InvalidOid が格納される。

アルゴリズムまたは振る舞い

  • RelfilenumberMapHash ハッシュテーブルの各エントリとして使用される。
  • キャッシュミス発生時にカタログ検索結果を格納し、次回以降の高速な逆引きを可能にする。
  • pg_class の変更に伴う無効化イベントが発生した際、該当するエントリがハッシュテーブルから削除される。

呼び出し元と呼び出し先

  • 呼び出し元: src/backend/utils/cache/relfilenumbermap.c 内のハッシュテーブル操作関数（hash_seq_search 等）およびエントリ生成処理。
  • 呼び出し先: なし（データ構造定義のため）。

関連データ構造

  • RelfilenumberMapKey: 本構造体の検索キーとして使用される。
  • RelfilenumberMapHash: 本構造体を要素として管理するハッシュテーブル。

失敗モードと境界条件

  • 検索対象のファイル番号が存在しない場合、relid に InvalidOid を保持することで「負のキャッシュ」として機能する。
  • 一時テーブルはバックエンド固有であるため、本構造体によるキャッシュの対象外となる。

保守担当者向けメモ

  • 本構造体のサイズは ctl.entrysize を通じてハッシュテーブルの初期化時に使用されるため、メンバの追加・変更時には注意が必要。
