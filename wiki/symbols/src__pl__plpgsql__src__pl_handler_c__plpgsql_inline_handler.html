plpgsql_inline_handler

定義

  • 種別: function
  • 定義箇所: src/pl/plpgsql/src/pl_handler.c:315

目的

PostgreSQL における DO 文（匿名コードブロック）の実行を仲介するハンドラ関数です。

シグネチャまたは形状

Datum plpgsql_inline_handler(PG_FUNCTION_ARGS)

入力と出力

  • **入力**: PG_FUNCTION_ARGS を介して InlineCodeBlock 構造体（ソーステキストおよびアトミック性フラグを含む）を受け取る。
  • **出力**: Datum 型の実行結果を返す。

前提条件と不変条件

  • SPI_connect_ext により SPI マネージャへの接続が確立されていること。
  • 実行に使用する EState および ResourceOwner は、トランザクションレベルのリソースとは独立して管理され、DO ブロック内の COMMIT/ROLLBACK を跨いで生存する必要がある。
  • 実行終了時（成功・失敗問わず）に、割り当てた EState、ResourceOwner、およびコンパイル済み関数メモリが解放されること。

アルゴリズムまたは振る舞い

1. SPI_connect_ext を呼び出し、DO ブロックの実行コンテキストを準備する。 2. plpgsql_compile_inline を使用してソーステキストを PLpgSQL_function にコンパイルする。 3. 実行に必要な EState と ResourceOwner を作成する。 4. PG_TRY ブロック内で plpgsql_exec_function を呼び出し、コードブロックを実行する。 5. PG_CATCH ブロックにて、エラー発生時に plpgsql_subxact_cb を呼び出してサブトランザクション関連のクリーンアップを行い、リソースを解放した後にエラーを再スローする。 6. 正常終了時、または PG_CATCH 後のクリーンアップ処理として、EState の破棄、ResourceOwner の解放、関数メモリの解放、および SPI_finish を実行する。

呼び出し元と呼び出し先

  • **呼び出し元**: PostgreSQL の関数マネージャ（直接的な呼び出し元情報はソースコード上では明示されていないが、DO 文実行時に起動される）。
  • **呼び出し先**: SPI_connect_ext, plpgsql_compile_inline, CreateExecutorState, ResourceOwnerCreate, plpgsql_exec_function, plpgsql_subxact_cb, FreeExecutorState, ReleaseAllPlanCacheRefsInOwner, ResourceOwnerDelete, plpgsql_free_function_memory, SPI_finish。

関連データ構造

  • InlineCodeBlock: 実行対象のソースコードとアトミック性フラグを保持。
  • PLpgSQL_function: コンパイルされた関数表現。
  • EState: 実行状態を保持する構造体。
  • ResourceOwner: リソース管理用オブジェクト。

失敗モードと境界条件

  • SPI_finish が失敗した場合、elog(ERROR) によりエラーが報告される。
  • DO ブロック内でエラーが発生した場合、PG_CATCH ブロックを経由してリソースのクリーンアップが行われ、エラーが伝播される。
  • func->cfunc.use_count が 0 にならない場合は Assert により停止する。

保守担当者向けメモ

  • 本関数は PG_TRY/PG_FINALLY 的な構造（PG_CATCH を含む）を用いてリソースリークを防止している。特に ResourceOwner はトランザクションに依存しないため、DO ブロック内のトランザクション制御の影響を受けないよう注意が必要である。
  • plpgsql_subxact_cb の呼び出しは、DO ブロック内のサブトランザクション異常終了時に simple_econtext_stack を適切にクリーンアップするために必須である。
