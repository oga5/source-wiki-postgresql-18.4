standard_planner

定義

  • **種別**: function
  • **定義箇所**: src/backend/optimizer/plan/planner.c:320

目的

PostgreSQLにおけるクエリ最適化の標準的な実装を提供し、解析済みの Query ツリーから実行可能な PlannedStmt を生成する。

シグネチャまたは形状

PlannedStmt *standard_planner(Query *parse, const char *query_string, int cursorOptions, ParamListInfo boundParams)

入力と出力

  • **入力**:
  • parse: 最適化対象の Query ツリー。
  • query_string: クエリ文字列（デバッグやログ用）。
  • cursorOptions: カーソル動作を制御するビットマスク（CURSOR_OPT_PARALLEL_OK, CURSOR_OPT_FAST_PLAN, CURSOR_OPT_SCROLL 等）。
  • boundParams: 実行時にバインドされるパラメータ情報。
  • **出力**:
  • 実行可能な PlannedStmt 構造体へのポインタ。

前提条件と不変条件

  • 入力された Query ツリーは、パーサーによって解析済みであること。
  • standard_planner は入力された Query ツリーを直接書き換える（破壊的変更を行う）可能性がある。
  • 並列クエリの可否は、cursorOptions、GUC設定、およびクエリ内の関数属性に基づいて決定される。

アルゴリズムまたは振る舞い

1. **初期化**: PlannerGlobal 構造体を生成し、サブクエリ間で共有される状態を初期化する。 2. **並列性の判定**: クエリが並列実行可能か（parallelModeOK）を判定し、必要に応じて parallelModeNeeded を設定する。 3. **コスト評価の準備**: cursorOptions に基づき、最適化の対象となるタプル割合（tuple_fraction）を決定する。 4. **サブクエリのプランニング**: subquery_planner を呼び出し、クエリツリーの再帰的な最適化を行う。 5. **パス選択**: fetch_upper_rel で最終的なリレーションを取得し、get_cheapest_fractional_path で最適なパスを選択する。 6. **プラン生成**: create_plan を用いてパスをプランツリーに変換する。 7. **後処理**: スクロール可能なカーソルの場合は ExecSupportsBackwardScan を確認し、必要に応じて materialize_finished_plan を適用する。また、デバッグ設定に応じて Gather ノードの追加を試みる。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/optimizer/plan/planner.c (内部呼び出し)
  • contrib/pg_stat_statements/pg_stat_statements.c
  • src/test/modules/delay_execution/delay_execution.c
  • **呼び出し先**:
  • subquery_planner(): プランニングの主要な再帰処理。
  • fetch_upper_rel(): 上層リレーションの取得。
  • get_cheapest_fractional_path(): 最適なパスの選択。
  • create_plan(): パスからプランノードへの変換。
  • ExecSupportsBackwardScan() / materialize_finished_plan(): カーソル制御。

関連データ構造

  • Query: 入力となるクエリツリー。
  • PlannedStmt: 出力となる実行計画。
  • PlannerGlobal: プランニング全体で共有されるグローバル状態。
  • PlannerInfo: 特定のクエリレベルの最適化情報。
  • Path / Plan: 最適化過程および最終的な実行計画ノード。

失敗モードと境界条件

  • debug_parallel_query が有効な場合、プランの並列安全性に依存して Gather ノードが追加されるが、initPlan の配置制限により一部のケースでスキップされる。
  • cursor_tuple_fraction が 1.0 以上の場合は 0.0（全タプル）として扱われ、0.0 以下の場合は非常に小さな値（1e-10）として扱われる。

保守担当者向けメモ

  • 本関数は planner_hook が設定されている場合にスキップされる可能性があるため、フックの挙動と併せて確認すること。
  • Query ツリーを破壊的に変更するため、呼び出し側で元のツリーを保持する必要がある場合は注意が必要。
  • 並列実行の判定ロジックは、将来的な機能拡張（並列挿入など）により変更される可能性がある。
