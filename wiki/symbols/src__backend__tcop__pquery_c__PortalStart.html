PortalStart

定義

  • 種別: function
  • 定義箇所: src/backend/tcop/pquery.c:433

目的

ポータルの実行準備を行い、クエリの実行戦略（PortalStrategy）を確定させて実行可能な状態（PORTAL_READY）へ移行させる。

シグネチャまたは形状

void PortalStart(Portal portal, ParamListInfo params, int eflags, Snapshot snapshot)

入力と出力

  • **入力**:
  • portal: 実行対象のポータル構造体。
  • params: クエリ実行に使用するパラメータリスト。
  • eflags: 実行エンジンに渡すフラグ（EXEC_FLAG_*）。
  • snapshot: 実行に使用するトランザクションスナップショット。
  • **出力**: なし（副作用としてポータル内部状態を更新）。

前提条件と不変条件

  • PortalIsValid(portal) が真であること。
  • portal->status が PORTAL_DEFINED であること。
  • 実行終了時に portal->status は PORTAL_READY に更新される。

アルゴリズムまたは振る舞い

1. **環境設定**: ActivePortal、CurrentResourceOwner、PortalContext をポータル固有のものに切り替える。エラー発生時に備え PG_TRY ブロックで囲む。 2. **戦略決定**: ChoosePortalStrategy を呼び出し、ポータルの実行戦略を決定する。 3. **戦略別処理**:

  • PORTAL_ONE_SELECT: スナップショットをプッシュし、CreateQueryDesc でクエリ記述子を作成。ExecutorStart を呼び出して実行準備を行い、カーソル位置を初期化する。
  • PORTAL_ONE_RETURNING / PORTAL_ONE_MOD_WITH: 実行結果のタプル記述子（tupDesc）を生成し、カーソル位置を初期化する。
  • PORTAL_UTIL_SELECT: ユーティリティコマンド用のタプル記述子を生成し、カーソル位置を初期化する。
  • PORTAL_MULTI_QUERY: 特段の準備は行わない。

    • 4. **エラーハンドリング**: PG_CATCH ブロックにて、実行中にエラーが発生した場合は MarkPortalFailed を呼び出し、ポータルを失敗状態にマークする。 5. **終了処理**: コンテキストを復元し、ステータスを PORTAL_READY に更新する。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/commands/portalcmds.c
  • src/backend/commands/prepare.c
  • src/backend/executor/spi.c
  • src/backend/tcop/postgres.c
  • **呼び出し先**:
  • ChoosePortalStrategy
  • CreateQueryDesc
  • ExecutorStart
  • PushActiveSnapshot / PopActiveSnapshot
  • MarkPortalFailed

関連データ構造

  • Portal: 実行状態を保持するコンテナ。
  • PlannedStmt: 実行計画。
  • QueryDesc: クエリ実行に必要な情報セット。
  • PortalStrategy: 実行モード（PORTAL_ONE_SELECT 等）。

失敗モードと境界条件

  • PORTAL_DEFINED 以外のステータスのポータルが渡された場合、アサーションエラーとなる。
  • 実行中にエラーが発生した場合、PG_CATCH によりポータルは PORTAL_FAILED 状態となり、例外が再スローされる。

保守担当者向けメモ

  • 本関数はポータルの実行準備のみを行い、実際のデータ処理は PortalRun が担当する。
  • PORTAL_ONE_SELECT 以外の戦略では ExecutorStart が即座には呼ばれないため、実行時の遅延初期化に注意すること。
  • PortalContext の切り替えによりメモリ管理が行われているため、コンテキストの復元漏れは致命的なメモリリークや不正なポインタ参照を招く。
