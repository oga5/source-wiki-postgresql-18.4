SyncOps

定義

  • 種別: struct
  • 定義箇所: src/backend/storage/sync/sync.c:84

目的

PostgreSQLのストレージサブシステムにおいて、異なるファイルタイプ（MD, CLOG等）に対する同期および削除操作を抽象化するための関数テーブルを定義する。

シグネチャまたは形状

typedef struct SyncOps
{
    int (*sync_syncfiletag) (const FileTag *ftag, char *path);
    int (*sync_unlinkfiletag) (const FileTag *ftag, char *path);
    bool (*sync_filetagmatches) (const FileTag *ftag, const FileTag *candidate);
} SyncOps;

入力と出力

  • sync_syncfiletag: FileTag とパスを受け取り、同期処理を実行して成功時に0を返す。
  • sync_unlinkfiletag: FileTag とパスを受け取り、削除処理を実行して成功時に0を返す。
  • sync_filetagmatches: 2つの FileTag を比較し、一致する場合に true を返す。

前提条件と不変条件

  • 各関数ポインタは、特定のストレージハンドラ（MDなど）の仕様に従って実装されている必要がある。
  • FileTag は、同期または削除対象のファイルを一意に識別できる必要がある。

アルゴリズムまたは振る舞い

本構造体は、syncsw 配列の要素として保持され、ProcessSyncRequests 等の関数から呼び出される。各ハンドラは、この構造体を通じて、ファイルシステムの種類に依存しない汎用的な同期・削除インターフェースを提供する。

呼び出し元と呼び出し先

  • 呼び出し元: src/backend/storage/sync/sync.c 内の syncsw 配列定義（95行目）にて初期化され、同期要求処理時に参照される。
  • 呼び出し先: 各ハンドラ（MD等）で定義された具体的な関数実装。

関連データ構造

  • FileTag: 同期対象ファイルを識別するためのタグ構造体。
  • syncsw: SyncOps 型の静的配列。

失敗モードと境界条件

  • 各関数ポインタが NULL の場合、呼び出し時にクラッシュする可能性がある（実装上の注意）。
  • sync_syncfiletag および sync_unlinkfiletag は、ファイル操作失敗時に適切な errno を設定し、呼び出し元へエラーを伝播させる必要がある。

保守担当者向けメモ

  • 新しいストレージタイプを追加する場合は、この構造体に準拠した新しい SyncOps インスタンスを syncsw 配列に追加すること。
  • 呼び出し元情報の詳細は、sync.c 内の syncsw を参照する各関数の実装を確認すること。
