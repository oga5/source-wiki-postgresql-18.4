assign_query_collations

定義

  • 種別: function
  • 定義箇所: src/backend/parser/parse_collate.c:100

目的

クエリ内の全式に対して照合順序（Collation）の割り当てと解決を開始するエントリーポイントです。

シグネチャまたは形状

void assign_query_collations(ParseState *pstate, Query *query)

入力と出力

  • **入力**:
  • pstate: 解析状態を保持する ParseState 構造体へのポインタ。
  • query: 照合順序を割り当てる対象の Query 構造体。
  • **出力**: なし（戻り値は void）。副作用として、クエリ内の各ノードに照合順序が書き込まれます。

前提条件と不変条件

  • Query 内の RTE（Range Table Entries）およびサブクエリは、本関数が呼び出される前に既に処理済みである必要があります。これは、それらを参照する Var ノードが正しい照合順序を持って生成されることを保証するためです。

アルゴリズムまたは振る舞い

1. query_tree_walker を使用して、クエリ内のすべての式を再帰的に走査します。 2. 走査の際、assign_query_collations_walker をコールバック関数として使用します。 3. QTW_IGNORE_RANGE_TABLE および QTW_IGNORE_CTE_SUBQUERIES フラグを指定し、既に処理済みの範囲テーブルとCTEサブクエリを走査対象から除外します。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/parser/analyze.c 内の複数の箇所（例: analyze.c:619, 1044, 1515 等）から、クエリ解析の最終段階で呼び出されます。
  • **呼び出し先**: query_tree_walker を介して assign_query_collations_walker を呼び出します。

関連データ構造

  • Query: 処理対象となるクエリツリー。
  • ParseState: 解析状態を保持する構造体。

失敗モードと境界条件

  • 照合順序の競合が発生した場合、内部の走査関数（assign_query_collations_walker 等）がエラーを投げる可能性があります。
  • 範囲テーブルやCTEサブクエリが未処理の状態で呼び出された場合、照合順序の解決が不完全になる可能性があります。

保守担当者向けメモ

  • 本関数はクエリ解析の「後処理」フェーズに位置します。構文解析中に照合順序を決定するコストを避けるための設計です。
  • 修正を行う際は、assign_query_collations_walker がどのようにノードを訪問し、照合順序を伝播させているかを確認してください。
