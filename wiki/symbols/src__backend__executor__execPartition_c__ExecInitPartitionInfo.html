ExecInitPartitionInfo

定義

  • 種別: function
  • 定義箇所: src/backend/executor/execPartition.c:501

目的

パーティションテーブルへのデータ操作（INSERT/UPDATE/MERGE）において、特定のパーティションに対する ResultRelInfo をオンデマンドで初期化し、制約チェックやプロジェクション情報を構築します。

シグネチャまたは形状

static ResultRelInfo *ExecInitPartitionInfo(
    ModifyTableState *mtstate, 
    EState *estate,
    PartitionTupleRouting *proute,
    PartitionDispatch dispatch,
    ResultRelInfo *rootResultRelInfo,
    int partidx
)

入力と出力

  • **入力**:
  • mtstate: 実行中の ModifyTable ノードの状態。
  • estate: クエリ実行状態。
  • proute: パーティションルーティング情報。
  • dispatch: 対象パーティションを含む階層のディスパッチ情報。
  • rootResultRelInfo: ルートテーブルの ResultRelInfo。
  • partidx: 対象パーティションのインデックス。
  • **出力**: 初期化された ResultRelInfo 構造体へのポインタ。

前提条件と不変条件

  • proute->memcxt が有効なメモリコンテキストであること。
  • mtstate が保持する resultRelInfo 配列の先頭要素が、属性マッピングの基準として利用可能であること。
  • パーティションが INSERT 操作に対して有効なターゲットであること（CheckValidResultRel による検証）。

アルゴリズムまたは振る舞い

1. **メモリ管理**: proute->memcxt に切り替えてメモリを確保します。 2. **リレーションのオープン**: 指定されたパーティションを RowExclusiveLock で開きます。 3. **構造体初期化**: ResultRelInfo を作成し、InitResultRelInfo で初期化します。 4. **インデックス準備**: パーティションにインデックスが存在する場合、ExecOpenIndices を呼び出して挿入時の競合チェックに備えます。 5. **制約・プロジェクション構築**:

  • withCheckOptionLists が存在する場合、属性番号を変換して ri_WithCheckOptions および ri_WithCheckOptionExprs を構築します。
  • returningLists が存在する場合、属性変換を行い、ExecBuildProjectionInfo を用いて ri_projectReturning を初期化します。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/executor/execPartition.c 内のルーティング処理（ExecFindPartition 等）から、未初期化のパーティションに遭遇した際に呼び出されます。
  • **呼び出し先**: table_open, makeNode, InitResultRelInfo, CheckValidResultRel, ExecOpenIndices, build_attrmap_by_name, map_variable_attnos, ExecInitQual, ExecBuildProjectionInfo 等。

関連データ構造

  • ResultRelInfo: パーティションの実行時情報を保持。
  • ModifyTableState: ModifyTable ノードの実行状態。
  • PartitionTupleRouting: ルーティング環境全体を管理。
  • PartitionDispatch: パーティション階層の探索情報。

失敗モードと境界条件

  • パーティションが INSERT 不可な状態である場合、CheckValidResultRel によりエラーが発生します。
  • 属性マッピング（part_attmap）の構築に失敗した場合、または型変換が不整合な場合は予期せぬ動作を引き起こす可能性があります。

保守担当者向けメモ

  • 本関数は INSERT だけでなく、UPDATE や MERGE のタプルルーティング時にも再利用されます。
  • withCheckOptionLists や returningLists の変換ロジックは、ルートテーブルとパーティション間の属性番号の差異を吸収するために重要です。変更時は nodeModifyTable.c との整合性に注意してください。
