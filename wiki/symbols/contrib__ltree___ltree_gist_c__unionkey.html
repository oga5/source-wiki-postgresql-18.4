unionkey

定義

  • 種別: function
  • 定義箇所: contrib/ltree/_ltree_gist.c:141

目的

GiSTインデックス構築時、複数のシグネチャ（ビットベクトル）を論理和（OR）演算によって統合し、親ノードのシグネチャを更新する。

シグネチャまたは形状

int32 unionkey(BITVECP sbase, ltree_gist *add, int siglen)

入力と出力

  • **入力**:
  • sbase: 更新対象となるベースのビットベクトル（BITVECP）。
  • add: 追加する ltree_gist 構造体。
  • siglen: ビットベクトルの長さ（バイト単位）。
  • **出力**:
  • 1: add が全ビットセット状態（LTG_ISALLTRUE）である場合。
  • 0: 通常のビット演算が完了した場合。

前提条件と不変条件

  • sbase は少なくとも siglen バイトのメモリ領域を確保している必要がある。
  • add は LTG_SIGN マクロによってビットベクトル部分を抽出可能な形式であること。

アルゴリズムまたは振る舞い

1. add が全ビットセット状態であるかを確認する。該当する場合は即座に 1 を返す。 2. そうでない場合、ALOOPBYTE(siglen) マクロを使用して sbase の各バイトに対し、add のシグネチャ（sadd）とのビット単位の論理和（|=）を実行する。 3. 演算終了後、0 を返す。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • contrib/ltree/_ltree_gist.c:167
  • 他の拡張モジュール（hstore, intarray, pg_trgm, tsgistidx）からも同名の関数が呼び出されているが、これらは各モジュール固有の型を引数に取るため、本関数とはシグネチャが異なる可能性がある。
  • **呼び出し先**:
  • LTG_SIGN (マクロ)
  • LTG_ISALLTRUE (マクロ)
  • ALOOPBYTE (マクロ)

関連データ構造

  • ltree_gist: ltree 型のGiSTインデックス用構造体。
  • BITVECP: ビットベクトルを指すポインタ型。

失敗モードと境界条件

  • siglen が実際のビットベクトルのサイズと一致しない場合、メモリ破壊や不正なアクセスが発生する可能性がある。

保守担当者向けメモ

  • 本関数は ltree 専用の実装であるが、PostgreSQLの他のGiSTインデックスモジュール（hstore 等）にも同名の関数が存在する。修正時は他のモジュールとの整合性に注意すること。
