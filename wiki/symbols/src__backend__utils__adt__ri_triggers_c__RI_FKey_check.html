RI_FKey_check

定義

  • 種別: function
  • 定義箇所: src/backend/utils/adt/ri_triggers.c:249

目的

外部キー制約の整合性を検証するためのトリガープロシージャ。挿入または更新操作時に、参照先の主キー（または一意キー）が存在するかを確認し、制約違反がある場合はエラーを報告する。

シグネチャまたは形状

static Datum RI_FKey_check(TriggerData *trigdata)

入力と出力

  • **入力**: TriggerData *trigdata（トリガー実行時のコンテキスト情報）。
  • **出力**: Datum 型。整合性が確認された場合は PointerGetDatum(NULL) を返し、制約違反時はエラーをスローする。

前提条件と不変条件

  • 呼び出し元は、対象タプルに対するバッファピンを保持している必要がある（ロックは不要）。
  • ri_FetchConstraintInfo により、制約メタデータが取得可能であること。
  • 参照先テーブル（PKテーブル）は RowShareLock モードでオープンされる。

アルゴリズムまたは振る舞い

1. **メタデータ取得**: ri_FetchConstraintInfo を呼び出し、制約情報を取得する。 2. **生存確認**: table_tuple_satisfies_snapshot を使用し、対象タプルが SnapshotSelf に対して有効か確認する。無効な場合は即座に終了する。 3. **NULLチェック**: ri_NullCheck を実行し、外部キー列のNULL状態に応じた制約ルール（MATCH FULL/SIMPLE）を適用する。 4. **クエリ準備**: SPI_connect を経由し、ri_FetchPreparedPlan でキャッシュされたSPIプランを検索する。 5. **クエリ生成**: キャッシュがない場合、SELECT 1 FROM ... FOR KEY SHARE クエリを動的に構築する。期間型（Temporal FK）が含まれる場合は、範囲の包含関係を検証する HAVING 句付きのクエリを生成する。

関連データ構造

  • TriggerData: トリガー実行時のデータ構造。
  • RI_ConstraintInfo: 外部キー制約のメタデータ。
  • RI_QueryKey: SPIプランをキャッシュするためのキー。
  • SPIPlanPtr: 実行されるSQLクエリのプラン。

失敗モードと境界条件

  • **MATCH FULL違反**: 外部キー列の一部のみがNULLの場合、ERRCODE_FOREIGN_KEY_VIOLATION を発生させる。
  • **タプル無効**: SnapshotSelf で無効と判定された場合、チェックをスキップする。
  • **メモリ管理**: キャッシュされたSPIプランは永続化されるため、大量の制約変更時には注意が必要。

保守担当者向けメモ

  • 本関数は FOR KEY SHARE を使用して参照整合性を保証する。
  • 期間型（Temporal FK）のサポートが含まれており、hasperiod フラグに応じてクエリ構築ロジックが分岐する。
