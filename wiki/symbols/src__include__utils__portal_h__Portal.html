Portal

定義

  • 種別: typedef (構造体 PortalData へのポインタ)
  • 定義箇所: src/include/utils/portal.h:113

目的

実行中または実行可能なクエリの状態を抽象化し、SQLカーソルおよびプロトコルレベルのクエリ実行を管理する。実行エンジン（Executor）とクライアント間の橋渡しを行う。

シグネチャまたは形状

typedef struct PortalData *Portal;

入力と出力

  • 入力: CreatePortal により生成され、PortalDefineQuery を通じてクエリが割り当てられる。
  • 出力: 実行結果は Tuplestore にバッファリングされ、クライアントへ提供される。

前提条件と不変条件

  • 1つのソースSQLクエリに対して1つのポータルが作成される。
  • PortalIsValid(p) マクロにより有効性を判定可能。
  • PORTAL_ONE_SELECT 以外の戦略では、部分的な実行やサスペンドはサポートされない。

アルゴリズムまたは振る舞い

1. **初期化**: CreatePortal で生成され、PortalDefineQuery でクエリが割り当てられる。 2. **実行**: PortalStatus が READY から ACTIVE に遷移し、QueryDesc を介して Executor が呼び出される。 3. **状態管理**: PortalStrategy（実行戦略）と PortalStatus（ライフサイクル）を保持・追跡する。 4. **終了**: 実行完了後に PortalStatus が DONE に遷移し、PortalDrop によりメモリコンテキストやリソースオーナーが解放される。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/access/common/printtup.c (結果出力用)
  • src/backend/access/transam/xact.c (トランザクション管理内)
  • src/backend/commands/portalcmds.c (ポータル操作コマンド)
  • src/backend/commands/functioncmds.c (関数実行コンテキスト)
  • **呼び出し先**:
  • 提供されたソース excerpt からは、具体的な呼び出し先関数は特定できない。

関連データ構造

  • PortalData: ポータルの全状態を保持する実体。
  • PortalStrategy: クエリ実行戦略（PORTAL_ONE_SELECT 等）。
  • PortalStatus: ライフサイクル状態（NEW, READY, ACTIVE, DONE 等）。
  • QueryDesc: 実行記述子。
  • ResourceOwner: リソース管理用。

失敗モードと境界条件

  • PortalDrop を呼び出さない場合、メモリリークやリソースの解放漏れが発生する。
  • PORTAL_ONE_SELECT 以外の戦略でサスペンドを試みた場合の挙動は未定義（サポート外）。

保守担当者向けメモ

  • ポータルは書き換えルールによって複数のプランツリーが生成される場合でも、ユーザーからは単一のクエリとして扱われる。
  • portalSnapshot はクエリ実行中の一貫性保証（特にTOASTデータのデトースト）に不可欠である。
