CreateLockFile

定義

  • **種別**: function
  • **定義箇所**: src/backend/utils/init/miscinit.c:1209

目的

指定されたファイルパスを使用してロックファイルを作成し、PostgreSQLの二重起動を防止します。既存のロックファイルが存在する場合は、その内容（PID）を検証し、プロセスが生存しているかを確認します。

シグネチャまたは形状

static void CreateLockFile(const char *filename, bool amPostmaster,
                           const char *socketDir,
                           bool isDDLock, const char *refName)

入力と出力

  • **入力**:
  • filename: 作成するロックファイルのパス。
  • amPostmaster: 呼び出し元がPostmasterであるかを示すフラグ。
  • socketDir: ソケットディレクトリのパス。
  • isDDLock: データディレクトリのロックであるかを示すフラグ。
  • refName: エラーメッセージで使用する参照名（データディレクトリパスやソケットファイル名）。
  • **出力**: なし（成功時はファイルを生成し、失敗時は ereport(FATAL, ...) によりプロセスを終了します）。

前提条件と不変条件

  • ロックファイルは O_EXCL フラグ付きで作成され、アトミックな排他制御を保証します。
  • ファイルのパーミッションは pg_file_create_mode（通常0600/0640）で保護されます。
  • 自身のPID、親プロセスのPID、および環境変数 PG_GRANDPARENT_PID で指定された祖父プロセスのPIDは、ロックファイル内のPIDと一致しても「古いロック」とみなされます。

アルゴリズムまたは振る舞い

1. 自身のPIDおよび関連する親・祖父プロセスのPIDを取得します。 2. 最大100回のリトライループで open(..., O_CREAT | O_EXCL) を試行します。 3. ファイル作成に失敗した場合、既存ファイルの内容（PID）を読み取ります。 4. 読み取ったPIDが生存しているか kill(other_pid, 0) で確認します。

  • プロセスが生存している場合、FATAL エラーで終了します。
  • プロセスが存在しない場合、または権限エラー（EPERM）の場合は、ロックが古いと判断して処理を継続します。

    • 5. データディレクトリロック（isDDLock）の場合、共有メモリセグメントの生存確認へ進みます（ソース抜粋の範囲外）。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/utils/init/miscinit.c 内の CreateDataDirLockFile および CreateSocketLockFile から呼び出されます。
  • **呼び出し先**:
  • open, read, close, getpid, getppid, getenv, atoi, kill, ereport, pgstat_report_wait_start/end などのシステムコールおよびPostgreSQL内部関数。

関連データ構造

  • MAXPGPATH: ファイルパスの最大長。
  • pg_file_create_mode: ファイル作成時のパーミッション設定。

失敗モードと境界条件

  • **EEXIST/EACCES**: ファイル作成失敗時に既存のロックを調査します。
  • **空のロックファイル**: ファイルが空の場合は FATAL エラーとなります。
  • **PIDの不一致**: 読み取ったPIDが生存している場合、二重起動とみなして FATAL エラーとなります。
  • **リトライ上限**: 100回試行しても作成できない場合は FATAL エラーとなります。

保守担当者向けメモ

  • 本関数は proc_exit に依存してロックファイルを削除する設計であるため、異常終了時にはロックファイルが残存する可能性があります。
  • Windows環境では getppid() が存在しないため、親PIDのチェックはスキップされます。
  • kill(other_pid, 0) の結果が EPERM の場合、異なるユーザーIDによるプロセスであると判断し、安全のためロックを無視します。
