log_split_page

定義

  • 種別: function
  • 定義箇所: src/backend/access/hash/hashpage.c:1473

目的

ハッシュインデックスのバケット分割操作に関連するページ状態をWAL（Write-Ahead Log）に記録する。

シグネチャまたは形状

static void log_split_page(Relation rel, Buffer buf)

入力と出力

  • rel: 操作対象のインデックスリレーション。
  • buf: 分割操作が行われた対象ページを含むバッファ。
  • 出力: なし（副作用としてWALログの書き込みとページLSNの更新を行う）。

前提条件と不変条件

  • RelationNeedsWAL(rel) が真の場合のみWAL記録が実行される。
  • 呼び出し側は、対象のバッファに対して適切なロックを保持している必要がある。

アルゴリズムまたは振る舞い

1. RelationNeedsWAL を確認し、WALが必要なリレーションであれば以下の処理を行う。 2. XLogBeginInsert を呼び出し、WAL挿入シーケンスを開始する。 3. XLogRegisterBuffer を使用し、対象バッファを REGBUF_FORCE_IMAGE および REGBUF_STANDARD フラグ付きで登録する。 4. XLogInsert を呼び出し、RM_HASH_ID リソースマネージャおよび XLOG_HASH_SPLIT_PAGE レコードタイプでログを書き込む。 5. 書き込まれたWALレコードのLSN（Log Sequence Number）を取得し、PageSetLSN を用いて対象ページのLSNを更新する。

呼び出し元と呼び出し先

  • 呼び出し元: src/backend/access/hash/hashpage.c 内のバケット分割処理（1189行目、1240行目）。
  • 呼び出し先: PostgreSQLのWAL管理サブシステム（XLogBeginInsert, XLogRegisterBuffer, XLogInsert 等）。

関連データ構造

  • Relation: インデックスのメタデータ管理に使用。
  • Buffer: ページデータへのアクセスに使用。
  • XLogRecPtr: WALレコードのポインタ。

失敗モードと境界条件

  • WAL書き込み時にストレージ領域不足やI/Oエラーが発生した場合、PostgreSQLの標準的なエラーハンドリングに従い例外が発生する。
  • rel がWALを必要としない場合（例: 一時テーブル等）、本関数は何も行わず正常終了する。

保守担当者向けメモ

  • 本関数は XLOG_HASH_SPLIT_PAGE レコードを生成する。リカバリ時にこのレコードを正しく再適用できるよう、hash_xlog.c 側の再実行ロジックと整合性を保つこと。
  • REGBUF_FORCE_IMAGE を使用しているため、ページ全体がWALに書き込まれる。パフォーマンスへの影響を考慮すること。
