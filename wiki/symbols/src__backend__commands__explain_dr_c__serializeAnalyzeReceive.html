serializeAnalyzeReceive

定義

  • 種別: function
  • 定義箇所: src/backend/commands/explain_dr.c:104

目的

EXPLAIN (SERIALIZE) 実行時に、クエリ結果のタプルをクライアントへ送信することなく、シリアライズ処理（データ変換やTOAST解除）にかかるコストを計測・収集する。

シグネチャまたは形状

static bool serializeAnalyzeReceive(TupleTableSlot *slot, DestReceiver *self);

入力と出力

  • **入力**:
  • slot: 処理対象のタプルを含む TupleTableSlot。
  • self: SerializeDestReceiver 型としてキャストされる DestReceiver インスタンス。
  • **出力**: 常に true を返す。副作用として self 内の metrics（バイト数、時間、バッファ使用量）が更新される。

前提条件と不変条件

  • self は SerializeDestReceiver 構造体へのポインタであること。
  • slot は有効なタプル記述子（tts_tupleDescriptor）を持つこと。
  • ネットワーク送信は行わないため、pq_endmessage_reuse() は呼び出さないこと。

アルゴリズムまたは振る舞い

1. **計測開始**: es->timing および es->buffers が有効な場合、開始時点の時刻とバッファ使用量を記録。 2. **準備**: 必要に応じて serialize_prepare_info を呼び出し、属性情報をキャッシュ。slot_getallattrs でタプルをデコード。 3. **シリアライズ**: tmpcontext に切り替え、pq_beginmessage_reuse を使用して DataRow メッセージ形式でデータを構築。

  • テキスト形式の場合は OutputFunctionCall、バイナリ形式の場合は SendFunctionCall を使用。

    • 4. **集計**: 構築したメッセージの長さ（buf->len）を metrics.bytesSent に加算。 5. **後処理**: メモリコンテキストを復元・リセットし、計測が有効であれば終了時刻とバッファ使用量の差分を metrics に累積。

呼び出し元と呼び出し先

  • **呼び出し元**: 明示的な呼び出し元情報は提供されていないが、DestReceiver の receiveSlot フィールドに本関数が登録される（src/backend/commands/explain_dr.c:280）。
  • **呼び出し先**: serialize_prepare_info, slot_getallattrs, MemoryContextSwitchTo, pq_beginmessage_reuse, pq_sendint16, OutputFunctionCall, SendFunctionCall, MemoryContextReset, INSTR_TIME_... マクロ群, BufferUsageAccumDiff。

関連データ構造

  • SerializeDestReceiver: 計測状態とバッファを保持する構造体。
  • TupleTableSlot: 処理対象のタプルデータ。
  • DestReceiver: 汎用的なタプル受信インターフェース。

失敗モードと境界条件

  • slot が空の場合や、型変換関数が失敗した場合は例外が発生する可能性がある。
  • メモリコンテキストの切り替え失敗や、tmpcontext の管理不備によるメモリリークのリスクがある（MemoryContextReset により抑制）。

保守担当者向けメモ

  • 本関数は printtup() のロジックを模倣しているため、データ構築ロジックを変更する際は printtup.c との整合性に注意すること。
  • ネットワーク負荷を除外するため、pq_endmessage_reuse() を呼び出さない設計を維持すること。
