index_create

定義

  • 種別: function
  • 定義箇所: src/backend/catalog/index.c:725

目的

PostgreSQLにおいて、新しいインデックスリレーションをカタログに登録し、物理的なインデックス構造を作成するためのメインエントリポイントです。

シグネチャまたは形状

Oid index_create(Relation heapRelation, const char *indexRelationName, Oid indexRelationId, Oid parentIndexRelid, Oid parentConstraintId, RelFileNumber relFileNumber, IndexInfo *indexInfo, const List *indexColNames, Oid accessMethodId, Oid tableSpaceId, const Oid *collationIds, const Oid *opclassIds, const Datum *opclassOptions, const int16 *coloptions, const NullableDatum *stattargets, Datum reloptions, bits16 flags, bits16 constr_flags, bool allow_system_table_mods, bool is_internal, Oid *constraintId)

入力と出力

  • **入力**: 対象テーブル（heapRelation）、インデックス名、各種OID、インデックス属性情報（IndexInfo）、演算子クラス情報、フラグ等。
  • **出力**: 作成されたインデックスの Oid を返します。IF NOT EXISTS 指定時に既に存在する場合は InvalidOid を返します。

前提条件と不変条件

  • indexInfo->ii_NumIndexAttrs は 1 以上である必要があります。
  • システムカタログに対するユーザー定義インデックスは、allow_system_table_mods が真でない限り禁止されます。
  • 共有リレーションのインデックス作成は initdb 中のみ許可されます。
  • パーティション化されたインデックスは、INDEX_CREATE_SKIP_BUILD フラグが設定されている必要があります。

アルゴリズムまたは振る舞い

1. **初期検証**: インデックス名が重複していないか、システムカタログへの不正な操作ではないかを確認します。 2. **互換性チェック**: 特定の演算子クラス（text_pattern_ops 等）と非決定論的照合順序（nondeterministic collation）の組み合わせを拒否します。 3. **制約チェック**: 並行作成（CONCURRENTLY）がシステムカタログや排他制約に対して行われていないか検証します。 4. **カタログ操作**: pg_class をロックし、インデックスのメタデータ定義を準備します。 5. **物理作成**: relFileNumber が無効な場合、ストレージ層を通じて物理ファイルを確保します。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/commands/indexcmds.c（CREATE INDEX コマンド処理）、src/backend/catalog/toasting.c（TOASTテーブルのインデックス作成）など。
  • **呼び出し先**: table_open, get_collation_isdeterministic, SearchSysCache1, get_relname_relid 等のカタログおよびストレージ管理関数。

関連データ構造

  • Relation: インデックス対象のテーブル情報。
  • IndexInfo: インデックスの属性や演算子情報を含むメタデータ構造。
  • pg_class: インデックスのカタログ定義を保持するシステムテーブル。

失敗モードと境界条件

  • **重複エラー**: 同名のインデックスが既に存在する場合（IF NOT EXISTS がない場合）。
  • **機能制限エラー**: システムカタログへの並行インデックス作成や、非対応の照合順序使用時。
  • **キャッシュエラー**: 演算子クラスのキャッシュルックアップ失敗時。

保守担当者向けメモ

  • 本関数はインデックス作成の初期段階を担います。物理的なデータ構築（index_build）やカタログへの最終的な依存関係登録は、この後の処理で行われます。
  • CONCURRENTLY フラグに関連する状態遷移は複雑であるため、indisready や indisvalid の更新タイミングについては、呼び出し元の indexcmds.c との整合性に注意してください。
