has_legal_joinclause

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/path/joinrels.c:1234

目的

指定されたリレーション（rel）が、クエリ内の他のいずれかのリレーションと、正当な結合条件（joinclause）を介して結合可能であるかを判定する。

シグネチャまたは形状

static bool has_legal_joinclause(PlannerInfo *root, RelOptInfo *rel)

入力と出力

  • **入力**:
  • root: クエリ全体のプランナ情報。
  • rel: 結合可能性を検証する対象のリレーション。
  • **出力**:
  • bool: 少なくとも1つの他のリレーションと正当な結合条件で結合可能な場合は true、それ以外は false。

前提条件と不変条件

  • root->initial_rels が初期化されていること。
  • rel は RelOptInfo として有効であること。

アルゴリズムまたは振る舞い

1. root->initial_rels 内の各リレーション（rel2）を順次走査する。 2. rel と rel2 が既に重複している（bms_overlap）場合はスキップする。 3. have_relevant_joinclause を呼び出し、両者の間に結合条件が存在するかを確認する。 4. 結合条件が存在する場合、join_is_legal を呼び出し、その結合がクエリの制約（外部結合の順序等）に照らして正当であるかを検証する。 5. 正当な結合と判定された場合、直ちに true を返し、走査を終了する。 6. すべての候補を走査しても正当な結合が見つからない場合は false を返す。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/optimizer/path/joinrels.c (複数箇所)
  • **呼び出し先**:
  • have_relevant_joinclause
  • join_is_legal
  • bms_overlap, bms_union, bms_free (ビットマップ操作)

関連データ構造

  • PlannerInfo: クエリ全体のプランナ状態。
  • RelOptInfo: 結合対象となるリレーションのメタデータ。
  • SpecialJoinInfo: 外部結合等の制約情報（join_is_legal 内部で使用）。

失敗モードと境界条件

  • initial_rels が空の場合、常に false を返す。
  • 結合条件が存在しても join_is_legal が false を返す場合、そのペアは無視される。

保守担当者向けメモ

  • 本関数は、結合順序の制約チェックにおいて「コストのかかるテスト」として位置づけられている。
  • join_is_legal の結果に依存しているため、外部結合や LATERAL 参照の制約が変更された場合は本関数の挙動も影響を受ける。
