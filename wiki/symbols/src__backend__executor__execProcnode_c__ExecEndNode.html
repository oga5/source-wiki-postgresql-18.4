ExecEndNode

定義

  • 種別: function
  • 定義箇所: src/backend/executor/execProcnode.c:561

目的

クエリ実行ツリー内の各ノード（PlanState）に対して、実行完了後のリソース解放および終了処理を再帰的に実行する。

シグネチャまたは形状

void ExecEndNode(PlanState *node)

入力と出力

  • **入力**: PlanState *node（終了対象となる実行ノードのポインタ）
  • **出力**: なし（void）

前提条件と不変条件

  • node が NULL の場合、本関数は何もせず即座に復帰する。
  • スタック深さのチェック（check_stack_depth()）が実行され、スタックオーバーフローを防止する。
  • node->chgParam が存在する場合は解放され、NULL にリセットされる。

アルゴリズムまたは振る舞い

1. node が NULL であれば終了。 2. check_stack_depth() を呼び出し、スタックの安全性を確認する。 3. node->chgParam が非 NULL であれば、bms_free() でメモリを解放し、ポインタを NULL にする。 4. nodeTag(node) を用いてノードタイプを判定し、対応する ExecEnd<Node> 関数（例: ExecEndSeqScan, ExecEndHashJoin 等）を呼び出す。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/executor/execMain.c（ExecutorEnd 等から）
  • src/backend/executor/nodeAgg.c
  • src/backend/executor/nodeAppend.c
  • **呼び出し先**: 各ノードタイプ専用の終了関数（ExecEndResult, ExecEndSeqScan, ExecEndHashJoin など、nodeTag に応じた関数）。

関連データ構造

  • PlanState: 実行ツリーの各ノードを表現する構造体。
  • chgParam: ノードの状態変更を追跡するビットマップセット。

失敗モードと境界条件

  • node が不正なポインタである場合、nodeTag(node) の評価時にクラッシュする可能性がある。
  • 再帰的な呼び出し構造を持つため、プランツリーが極端に深い場合はスタックオーバーフローのリスクがある（check_stack_depth() により緩和されている）。

保守担当者向けメモ

  • 本関数は巨大な switch 文で構成されている。新しいプランノードタイプを追加した際は、必ず本関数の switch 文に該当する case を追加し、適切な終了関数を呼び出す必要がある。
