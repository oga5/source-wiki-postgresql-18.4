pg_logical_slot_peek_binary_changes

定義

  • 種別: function
  • 定義箇所: src/backend/replication/logical/logicalfuncs.c:357

目的

論理レプリケーションスロットからWALの変更内容をバイナリ形式で読み取ります。この関数は「peek（覗き見）」動作を行うため、読み取った変更によってスロットの消費位置（confirmed_flush_lsn）は更新されません。

シグネチャまたは形状

Datum pg_logical_slot_peek_binary_changes(PG_FUNCTION_ARGS)

入力と出力

  • **入力**: PG_FUNCTION_ARGS（SQLインターフェース経由で、スロット名、開始LSN、アップトゥLSN、および出力プラグインオプションを受け取ります）。
  • **出力**: Datum（Tuplestore に格納されたバイナリ形式の変更レコードセットを返します）。

前提条件と不変条件

  • 論理デコーディングが有効なデータベース環境で実行される必要があります。
  • 指定されたレプリケーションスロットが存在し、アクティブである必要があります。

アルゴリズムまたは振る舞い

1. 内部関数 pg_logical_slot_get_changes_guts を呼び出します。 2. 第2引数に false を渡し、スロットの読み取り位置を更新しない（peek）動作を指定します。 3. 第3引数に true を渡し、出力形式をバイナリとして指定します。 4. pg_logical_slot_get_changes_guts が実行するデコーディングループにより、WALレコードが処理され、結果が Tuplestore に蓄積されて返却されます。

呼び出し元と呼び出し先

  • **呼び出し元**: SQLインターフェース（ユーザーによる関数呼び出し）。
  • **呼び出し先**: pg_logical_slot_get_changes_guts（本関数は実質的なラッパーです）。

関連データ構造

  • DecodingOutputState: 出力プラグインから生成されたバイナリデータを一時保持し、Tuplestore へ変換するために使用されます。

失敗モードと境界条件

  • 指定したスロットが存在しない場合や、権限がない場合はエラーが発生します。
  • WALの読み取り中にエラーが発生した場合、PG_TRY/PG_CATCH ブロックによりリソースリークを防ぐためのクリーンアップが行われます。

保守担当者向けメモ

  • 本関数は pg_logical_slot_get_changes_guts の単なるラッパーです。ロジックの変更が必要な場合は、logicalfuncs.c 内の pg_logical_slot_get_changes_guts を参照してください。
  • 読み取り位置を更新しないため、繰り返し呼び出しても同じ変更内容が取得されます。
