SetLatch

定義

  • 種別: function
  • 定義箇所: src/backend/storage/ipc/latch.c:289

目的

ラッチをセットし、そのラッチを待機しているプロセス（自身または他プロセス）を起床させる。シグナルハンドラ内からの呼び出しにも対応する。

シグネチャまたは形状

void SetLatch(Latch *latch)

入力と出力

  • **入力**: latch (操作対象の Latch 構造体へのポインタ)
  • **出力**: なし

前提条件と不変条件

  • pg_memory_barrier() により、本関数呼び出し前のメモリ変更が確実にフラッシュされる。
  • latch->is_set が既に true の場合、関数は即座に終了する。
  • latch->maybe_sleeping が false の場合、通知処理を行わずに終了する。

アルゴリズムまたは振る舞い

1. メモリバリアを適用する。 2. is_set フラグを true に設定する。 3. 再度メモリバリアを適用する。 4. maybe_sleeping フラグを確認し、待機者がいない場合は終了する。 5. OS 依存の通知処理を実行する:

  • **非 Windows**: owner_pid を取得し、自身であれば WakeupMyProc() を、他プロセスであれば WakeupOtherProc(owner_pid) を呼び出す。
  • **Windows**: latch->event に格納されたハンドルに対して SetEvent() を実行する。

呼び出し元と呼び出し先

  • **呼び出し元**: parallel.c, xlog.c, xlogrecovery.c, async.c, autovacuum.c, checkpointer.c, interrupt.c 等の各モジュールから、プロセス間通知や待機解除のために呼び出される。
  • **呼び出し先**: WakeupMyProc(), WakeupOtherProc() (非 Windows)、または SetEvent() (Windows)。

関連データ構造

  • Latch: ラッチの状態（is_set, maybe_sleeping, owner_pid または event）を保持する構造体。

失敗モードと境界条件

  • **競合**: owner_pid の取得はアトミックではない可能性があるが、実用上は pid_t が 32bit 整数に収まるため許容される。
  • **誤通知**: 稀に誤ったプロセスにシグナルが送られる可能性があるが、PostgreSQL のプロセスは不要な SIGUSR1 を無視する設計のため問題とならない。
  • **Windows**: SetEvent() のエラーはシグナルハンドラ等のクリティカルパスを考慮し、意図的に無視される。

保守担当者向けメモ

  • WaitLatch/ResetLatch との組み合わせにおいて、待機側はループの最後で待機するコーディングパターンに従う必要がある。これにより、ループ開始前に発生した SetLatch を見逃す競合を防ぐ。
  • シグナルハンドラから呼び出される可能性があるため、内部で errno を破壊しないよう注意が必要である。
