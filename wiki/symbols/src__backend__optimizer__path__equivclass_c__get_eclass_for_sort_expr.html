get_eclass_for_sort_expr

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/path/equivclass.c:735

目的

ソートやグループ化に使用される式に対応する EquivalenceClass (EC) を取得、または必要に応じて新規作成する。

シグネチャまたは形状

EquivalenceClass *get_eclass_for_sort_expr(PlannerInfo *root, Expr *expr, List *opfamilies, Oid opcintype, Oid collation, Index sortref, Relids rel, bool create_it)

入力と出力

  • **入力**:
  • root: プランナのグローバル情報。
  • expr: 評価対象の式。
  • opfamilies, opcintype, collation: 式の等価性を判定するための演算子族、型、照合順序。
  • sortref: ソート参照番号（揮発性関数の判定に使用）。
  • rel: 関連するリレーションの集合。
  • create_it: 一致するECがない場合に新規作成するかどうかのフラグ。
  • **出力**: 一致する（または新規作成された）EquivalenceClass へのポインタ。create_it が false で一致がない場合は NULL。

前提条件と不変条件

  • expr は canonicalize_ec_expression によって正規化される。
  • 揮発性関数を含むECは、sortref が一致しない限りマッチングから除外される。
  • 定数メンバーは、同一の JoinDomain に属する場合のみマッチング対象となる。

アルゴリズムまたは振る舞い

1. root->eq_classes を走査し、照合順序、演算子族、および式の内容が一致する既存の EC を検索する。 2. 一致が見つかった場合、その EC を返す。 3. 一致がなく create_it が true の場合、新しい EquivalenceClass を root->planner_cxt 上に割り当てて初期化する。 4. 新規 EC に式をメンバーとして追加し、root->eq_classes に登録する。 5. すでに EC のマージ処理が完了している場合（root->ec_merging_done）、新しい EC を関連するリレーションの eclass_indexes に追加する。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/optimizer/path/pathkeys.c 内の複数箇所（make_pathkeys_for_sortclauses 等）から呼び出されることが確認されている。
  • **呼び出し先**: canonicalize_ec_expression, setup_eclass_member_iterator, eclass_member_iterator_next, makeNode, add_eq_member, pull_varnos, copyObject, contain_volatile_functions 等を呼び出す。

関連データ構造

  • EquivalenceClass: 等価な式の集合を管理する構造体。
  • EquivalenceMember: EC 内の個別の式を保持する構造体。
  • PlannerInfo: プランナの全体状態を保持する構造体。

失敗モードと境界条件

  • 揮発性関数を含むECで sortref が 0 の場合はエラー (elog) となる。
  • 揮発性関数、集合を返す関数、集約関数、ウィンドウ関数が式に含まれる場合、定数フラグ (ec_has_const) が適切にクリアされる。

保守担当者向けメモ

  • 本関数は root->eq_classes を直接操作するため、メモリコンテキストの切り替え (MemoryContextSwitchTo) が必須である。
  • ec_merging_done が真の場合のインデックス更新処理は、プランナの後半フェーズで EC が追加された際の整合性を保つために重要である。
