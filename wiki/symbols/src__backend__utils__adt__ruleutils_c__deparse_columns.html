deparse_columns

定義

  • 種別: struct
  • 定義箇所: src/backend/utils/adt/ruleutils.c:234

目的

クエリの逆コンパイル（deparse）処理において、各レンジテーブルエントリ（RTE）に関連付けられた列名の管理および名前解決の状態を保持するために使用されます。

シグネチャまたは形状

typedef struct {
    int          num_cols;
    char       **colnames;
    int          num_new_cols;
    char       **new_colnames;
    bool        *is_new_col;
    bool         printaliases;
    List        *parentUsing;
    int          leftrti;
    int          rightrti;
    int         *leftattnos;
    int         *rightattnos;
    List        *usingNames;
    HTAB        *names_hash;
} deparse_columns;

入力と出力

本構造体は関数引数として渡され、逆コンパイルの過程で各フィールドが更新されます。

  • **入力**: set_relation_column_names や set_join_column_names 等の関数により、カタログ情報やクエリツリーから名前が抽出・設定されます。
  • **出力**: 最終的に colnames や new_colnames が参照され、SQL 文字列生成時の列エイリアスとして出力されます。

前提条件と不変条件

  • colnames は、クエリ解析時に存在した列のエイリアスを保持し、削除された列は NULL となります。
  • new_colnames は、現在のカタログ定義に基づいて再解析した場合の列名を含みます。
  • is_new_col は new_colnames と並列の配列であり、元の解析以降に追加された列を識別します。
  • names_hash は、列名割り当て関数実行中のみ有効であり、それ以外は NULL です。

アルゴリズムまたは振る舞い

  • **列名管理**: colnames は varattno で直接インデックス可能です。
  • **JOIN RTE の特殊処理**: JOIN RTE の場合、leftattnos と rightattnos を用いて子 RTE との対応関係を保持します。JOIN USING の場合は usingNames にマージされた列名が格納されます。
  • **ハッシュ最適化**: 大規模なリレーションに対しては、names_hash を使用して列名の重複チェックを効率化します。

呼び出し元と呼び出し先

  • **呼び出し元**: rtable_columns リスト（deparse_context 内）の一部として管理されます。
  • **呼び出し先**: expand_colnames_array_to, build_colinfo_names_hash, add_to_names_hash, destroy_colinfo_names_hash 等の内部関数によって操作されます。

関連データ構造

  • deparse_context: 本構造体をリストとして保持する上位のコンテキスト構造体。
  • HTAB: names_hash フィールドで使用されるハッシュテーブル。
  • List: parentUsing および usingNames で使用される PostgreSQL のリスト構造。

失敗モードと境界条件

  • **JOIN RTE**: 名前が重複する場合、usingNames 等を用いて一意性を確保するロジックが働きます。
  • **削除された列**: colnames 内の NULL エントリは、列が削除されていることを示しますが、クエリ内で使用されない限り問題にはなりません。

保守担当者向けメモ

  • 本構造体は ruleutils.c 内の複雑な名前解決ロジックの核です。変更を加える際は、set_relation_column_names および set_join_column_names への影響を十分に確認してください。
  • names_hash のライフサイクル管理（作成と破棄）が適切に行われているか注意してください。
