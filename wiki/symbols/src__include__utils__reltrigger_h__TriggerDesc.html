TriggerDesc

定義

  • 種別: struct
  • 定義箇所: src/include/utils/reltrigger.h:47

目的

リレーション（テーブル）に関連付けられた全トリガーのメタデータを保持し、実行エンジンが特定のイベントに対するトリガーの有無を高速に判定するための構造体です。

シグネチャまたは形状

typedef struct TriggerDesc
{
    Trigger    *triggers;
    int         numtriggers;
    bool        trig_insert_before_row;
    /* ... (各イベント・タイミングごとのboolフラグ群) ... */
    bool        trig_delete_old_table;
} TriggerDesc;

入力と出力

  • **入力**: リレーションオープン時に pg_trigger システムカタログから読み込まれたトリガー情報。
  • **出力**: 実行エンジンが DML 操作時に参照するトリガーの存在フラグおよびトリガー定義配列。

前提条件と不変条件

  • relcache によって構築され、RelationData 構造体に格納されます。
  • トリガーの追加・削除時には、配列内の情報と各ブールフラグの整合性を維持する必要があります。

アルゴリズムまたは振る舞い

  • 実行エンジンは、DML 操作時に TriggerDesc 内のブールフラグ（例: trig_insert_before_row）をチェックします。
  • フラグが false の場合、トリガー配列を走査することなく処理をスキップし、実行オーバーヘッドを最小化します。

呼び出し元と呼び出し先

  • **呼び出し元（参照・操作）**:
  • src/backend/commands/trigger.c: SetTriggerFlags（フラグ更新）、CopyTriggerDesc（構造体のコピー）、およびトリガーの構築処理。
  • **呼び出し先**:
  • 本構造体自体はデータ保持用であり、直接関数を呼び出すことはありません。

関連データ構造

  • Trigger: 個々のトリガー属性（OID、関数OID、タイプ、引数など）を保持する構造体。
  • RelationData (src/include/utils/rel.h): 本構造体をメンバとして保持します。

失敗モードと境界条件

  • 行レベルの TRUNCATE トリガーは存在しないため、TriggerDesc には対応するフラグが含まれていません。
  • トリガーが定義されていない場合、すべてのブールフラグは false となります。

保守担当者向けメモ

  • 本構造体は rel.h からインクルードされるため、依存関係を最小限に保つ必要があります。
  • 多数のブールフラグはパフォーマンス最適化のためのヒントです。新しいトリガータイプを追加する際は、対応するフラグの追加と、SetTriggerFlags 等でのフラグ更新ロジックの修正を忘れないでください。
