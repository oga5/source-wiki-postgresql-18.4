PLy_input_setup_func

定義

  • 種別: function
  • 定義箇所: src/pl/plpython/plpy_typeio.c:417

目的

SQLデータ型（Datum）をPythonオブジェクトへ変換するための変換構造体（PLyDatumToOb）を初期化し、型に応じた適切な変換関数ポインタを割り当てる。

シグネチャまたは形状

void PLy_input_setup_func(PLyDatumToOb *arg, MemoryContext arg_mcxt, Oid typeOid, int32 typmod, PLyProcedure *proc)

入力と出力

  • **入力**:
  • arg: 初期化対象の PLyDatumToOb 構造体へのポインタ。
  • arg_mcxt: 変換情報の割り当てに使用するメモリコンテキスト。
  • typeOid: 変換対象のSQL型OID。
  • typmod: 型修飾子。
  • proc: 呼び出し元のPL/Pythonプロシージャ情報。
  • **出力**: なし（arg 構造体のフィールドを直接更新する）。

前提条件と不変条件

  • 再帰的な型走査を行うため、スタックオーバーフローを防ぐための check_stack_depth() が呼び出しの先頭で実行される。
  • RECORD 型は特殊ケースとして扱われ、typcache を経由せずハードコードされた属性値が設定される。
  • ドメイン型は再帰的にベース型まで掘り下げられる。

アルゴリズムまたは振る舞い

1. **型情報の取得**: lookup_type_cache を使用して型情報を取得し、arg に格納する。 2. **変換メソッドの決定**:

  • **ドメイン型**: ベース型に対して再帰的に PLy_input_setup_func を呼び出す。
  • **配列型**: PLyList_FromArray を設定し、要素型に対して再帰的に初期化を行う。
  • **変換関数（Transform）**: get_transform_fromsql が定義されていれば PLyObject_FromTransform を設定する。
  • **複合型**: PLyDict_FromComposite を設定し、フィールド情報を保持する構造を準備する。
  • **スカラ型**: 型OIDに基づき、PLyBool_FromBool や PLyLong_FromInt32 などの専用関数を割り当てる。デフォルトの場合は PLyUnicode_FromScalar を使用する。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/pl/plpython/plpy_cursorobject.c
  • src/pl/plpython/plpy_exec.c
  • src/pl/plpython/plpy_procedure.c
  • src/pl/plpython/plpy_spi.c
  • src/pl/plpython/plpy_typeio.c（自身を再帰呼び出し）
  • **呼び出し先**:
  • lookup_type_cache, check_stack_depth, MemoryContextAllocZero, IsTrueArrayType, get_transform_fromsql, fmgr_info_cxt, getTypeOutputInfo 等のPostgreSQL内部関数。

関連データ構造

  • PLyDatumToOb: 変換ロジックと型情報を保持する構造体。
  • PLyProcedure: プロシージャの実行コンテキスト。
  • TypeCacheEntry: SQL型のキャッシュ情報。

失敗モードと境界条件

  • 再帰的な構造（配列やドメインのネスト）が深すぎる場合、スタックオーバーフローが発生する可能性がある。
  • RECORD 型は実行時に型が決定されるため、初期化時点では tupdescid が INVALID_TUPLEDESC_IDENTIFIER に設定される。

保守担当者向けメモ

  • 変換関数は再帰的であるため、メモリ管理には注意が必要。arg_mcxt は呼び出し元で適切に管理されていることを前提とする。
  • 新しいスカラ型を追加する場合は、switch 文にケースを追加すること。
  • ドメイン型は中間レベルをスキップしてベース型に直接到達する設計であるため、ドメインに対する変換関数（Transform）はサポートされていない。
