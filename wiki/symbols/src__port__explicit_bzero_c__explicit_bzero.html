explicit_bzero

定義

  • 種別: function
  • 定義箇所: src/port/explicit_bzero.c:21

目的

メモリ領域を安全にゼロクリアします。コンパイラの最適化（Dead Store Elimination）によって、機密情報の消去処理が「不要な書き込み」とみなされ削除されることを防ぎます。

シグネチャまたは形状

void explicit_bzero(void *buf, size_t len);

入力と出力

  • buf: ゼロクリア対象のメモリ領域へのポインタ。
  • len: クリアするバイト数。
  • 戻り値: なし。

前提条件と不変条件

  • buf は少なくとも len バイトの書き込み可能なメモリ領域を指している必要があります。
  • 本関数は、パスワードや暗号鍵などの機密情報をメモリから消去する用途を想定しています。

アルゴリズムまたは振る舞い

1. プラットフォームに応じて最適なゼロクリア手法を選択します。 2. 提供されたソースコードの抜粋では、memset_s(buf, len, 0, len) を呼び出します。 3. 環境により memset_s が利用できない場合は、volatile な関数ポインタを介して最適化を回避するフォールバック実装（bzero2）が使用されます。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/libpq/auth-oauth.c
  • src/backend/libpq/be-secure-common.c
  • src/common/cryptohash.c
  • src/common/cryptohash_openssl.c
  • src/common/hmac.c
  • src/common/hmac_openssl.c
  • **呼び出し先**:
  • memset_s（プラットフォーム依存）
  • または内部関数 bzero2（フォールバック時）

関連データ構造

  • 特になし。

失敗モードと境界条件

  • buf が NULL の場合や、len が確保された領域を超えている場合の動作は、標準ライブラリの memset_s の仕様に依存します。

保守担当者向けメモ

  • 本関数は OpenSSH の実装を参考にしています。
  • コンパイラの最適化を回避する手法として、フォールバック環境では volatile 修飾された関数ポインタ bzero_p を使用しています。
  • プラットフォーム固有のセキュリティ関数（memset_s や SecureZeroMemory）が利用可能な場合は、それらが優先的に使用される設計です。
