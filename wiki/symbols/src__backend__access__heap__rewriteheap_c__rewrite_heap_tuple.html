rewrite_heap_tuple

定義

  • **種別**: function
  • **定義箇所**: src/backend/access/heap/rewriteheap.c:340

目的

テーブル再構築（VACUUM FULL や CLUSTER 等）において、古いヒープから新しいヒープへタプルをコピーし、更新チェーン（ctid）の整合性を維持しながら書き込む。

シグネチャまたは形状

void rewrite_heap_tuple(RewriteState state, HeapTuple old_tuple, HeapTuple new_tuple)

入力と出力

  • **入力**:
  • state: 再構築状態を管理する RewriteState 構造体。
  • old_tuple: コピー元のタプル。
  • new_tuple: 書き込み対象の新しいタプル（コピー先）。
  • **出力**: なし（副作用として新しいヒープへの書き込みおよび内部ハッシュテーブルの更新を行う）。

前提条件と不変条件

  • 呼び出し元は対象テーブルに対して AccessExclusiveLock を保持している必要がある。
  • state->rs_cxt メモリコンテキストが適切に設定されていること。
  • new_tuple の可視性情報は old_tuple から引き継がれるが、HOT（Heap Only Tuple）関連のステータスビットは意図的にクリアされる。

アルゴリズムまたは振る舞い

1. **初期化**: old_tuple の可視性情報を new_tuple にコピーし、heap_freeze_tuple を呼び出してタプルの凍結処理を行う。 2. **更新チェーンの解決**:

  • タプルが更新チェーンの一部である場合、rs_old_new_tid_map を検索する。
  • 既に新しい位置が判明していれば ctid を更新して書き込む。
  • 未判明であれば rs_unresolved_tups に一時保存し、処理を中断する。

    • 3. **書き込みと連鎖処理**:

  • raw_heap_insert を用いて新しいヒープへ書き込む。
  • logical_rewrite_heap_tuple を呼び出し、論理デコーディング用のマッピングを記録する。
  • 挿入したタプルが更新チェーンの「B」側である場合、対応する「A」側が rs_unresolved_tups に存在すれば、それを取り出して ctid を修正し、再帰的に書き込みを行う（ループ処理）。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/access/heap/heapam_handler.c:2402
  • **呼び出し先**:
  • heap_freeze_tuple
  • hash_search (rs_old_new_tid_map, rs_unresolved_tups 操作)
  • raw_heap_insert
  • logical_rewrite_heap_tuple
  • heap_copytuple / heap_freetuple

関連データ構造

  • RewriteStateData: 再構築中の状態管理。
  • OldToNewMapping: 更新チェーンの TID マッピング。
  • UnresolvedTup: 未解決の更新チェーンタプル。

失敗モードと境界条件

  • **メモリ不足**: 更新チェーンが非常に長く、未解決タプルが大量に滞留した場合、メモリ不足に陥る可能性がある。
  • **論理デコーディング**: 論理レプリケーションが有効な場合、マッピングファイルが pg_logical/mappings に生成されるため、ディスク容量や権限の影響を受ける。

保守担当者向けメモ

  • t_infomask2 の HOT ステータスビットは意図的にクリアされている。将来的にこれらを保持する必要があるか検討が必要。
  • 更新チェーンの解決にはインメモリハッシュテーブルを使用しているため、巨大なテーブルの再構築時はメモリ消費量に注意すること。
