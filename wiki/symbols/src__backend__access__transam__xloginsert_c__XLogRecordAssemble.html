XLogRecordAssemble

定義

  • 種別: function
  • 定義箇所: src/backend/access/transam/xloginsert.c:547

目的

登録済みのバッファ情報およびデータチャンクを走査し、WAL レコードのヘッダー、ブロックヘッダー、およびデータ本体を連結した XLogRecData チェーンを構築する。

シグネチャまたは形状

XLogRecData *XLogRecordAssemble(RmgrId rmid, uint8 info, XLogRecPtr RedoRecPtr, bool doPageWrites, XLogRecPtr *fpw_lsn, int *num_fpi, bool *topxid_included)

入力と出力

  • **入力**:
  • rmid: リソースマネージャ ID。
  • info: レコードのフラグ情報。
  • RedoRecPtr: 現在の Redo 挿入位置。
  • doPageWrites: フルページ書き込み（FPI）を許可するかどうか。
  • **出力**:
  • 戻り値: 組み立てられた XLogRecData チェーンの先頭ポインタ。
  • fpw_lsn: 最初のフルページ書き込みが必要な LSN。
  • num_fpi: 構築されたフルページイメージの総数。
  • topxid_included: トップトランザクション ID がレコードに含まれているか。

前提条件と不変条件

  • XLogBeginInsert() によって作業領域が初期化されていること。
  • XLogRegisterBuffer() 等でバッファやデータが登録済みであること。
  • 本関数は同一レコードに対して複数回呼び出される可能性があるため、rdata チェーンの操作は再入可能である必要がある。

アルゴリズムまたは振る舞い

1. **ヘッダー初期化**: hdr_scratch を使用して固定サイズの XLogRecord ヘッダーを配置する。 2. **整合性チェック**: wal_consistency_checking[rmid] が有効な場合、info に XLR_CHECK_CONSISTENCY を付与する。 3. **ブロック走査**: max_registered_block_id までループし、登録された各バッファを処理する。

  • REGBUF_FORCE_IMAGE 等のフラグに基づき、フルページ書き込み（FPI）が必要か判定する。
  • FPI が必要な場合、ページ内の「ホール（pd_lower から pd_upper）」を特定し、wal_compression 設定に応じて圧縮を行う。
  • 構築したブロックヘッダーおよびイメージデータを XLogRecData チェーンに連結する。

    • 4. **チェーン構築**: 各データチャンクを XLogRecData 構造体で繋ぎ、最終的なレコード構造を形成する。

呼び出し元と呼び出し先

  • **呼び出し元**: XLogInsert (src/backend/access/transam/xloginsert.c:520)
  • **呼び出し先**:
  • XLogCompressBackupBlock (圧縮時)
  • PageGetLSN (ページ LSN 取得時)
  • その他、registered_buffers 配列へのアクセス。

関連データ構造

  • XLogRecData: WAL レコードのデータ断片を保持するリンクドリスト構造体。
  • registered_buffer: 登録されたブロックのメタデータと状態を保持する構造体。
  • XLogRecord: WAL レコードの固定ヘッダー。

失敗モードと境界条件

  • wal_compression が有効な場合、圧縮アルゴリズムの成否に応じて is_compressed フラグが制御される。
  • ページヘッダーが標準的でない場合、ホール除去は行われない。
  • fpw_lsn は InvalidXLogRecPtr で初期化され、必要に応じて更新される。

保守担当者向けメモ

  • 本関数は XLogInsert の中心的な組み立てロジックであり、hdr_scratch を介したメモリ操作を行うため、バッファオーバーランに注意すること。
  • XLogRecData チェーンの連結処理は、複数回呼び出しに耐えうるよう設計されている必要がある。
