GinNewBuffer

定義

  • 種別: function
  • 定義箇所: src/backend/access/gin/ginutil.c:304

目的

GIN インデックス用の新しいページバッファを確保します。既存の空き領域（FSM）を再利用するか、必要に応じてインデックスファイルを拡張します。

シグネチャまたは形状

Buffer GinNewBuffer(Relation index)

入力と出力

  • **入力**: index (対象となる GIN インデックスの Relation 構造体)
  • **出力**: 確保およびロックされた Buffer。

前提条件と不変条件

  • 呼び出し元は、インデックスに対する適切なロックを保持している必要があります。
  • 返されるバッファは、呼び出し元が使用可能な状態（ロック済み）で提供されます。

アルゴリズムまたは振る舞い

1. **FSM 検索**: GetFreeIndexPage を使用して、FSM (Free Space Map) から再利用可能なブロック番号を取得します。 2. **再利用の試行**:

  • ReadBuffer でページを読み込みます。
  • ConditionalLockBuffer でロックを試みます。
  • GinPageIsRecyclable でページが再利用可能か判定し、条件を満たせばそのバッファを返します。
  • 再利用不可またはロック失敗時は、バッファを解放して次のブロックを探索します。

    • 3. **ファイル拡張**: FSM から有効なページが得られない場合、ExtendBufferedRel を呼び出してインデックスファイルを物理的に拡張し、新しいバッファを確保して返します。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/access/gin/ginbtree.c
  • src/backend/access/gin/gindatapage.c
  • src/backend/access/gin/ginfast.c
  • src/backend/access/gin/gininsert.c
  • **呼び出し先**:
  • GetFreeIndexPage
  • ReadBuffer
  • ConditionalLockBuffer
  • GinPageIsRecyclable
  • BufferGetPage
  • LockBuffer
  • ReleaseBuffer
  • ExtendBufferedRel

関連データ構造

  • Relation: インデックスのメタデータ管理に使用。
  • Buffer: PostgreSQL のバッファ管理単位。
  • BlockNumber: ページ識別子。

失敗モードと境界条件

  • FSM が空の場合、自動的にファイル拡張が行われます。
  • 他のプロセスがページを再利用している可能性があるため、ConditionalLockBuffer を用いて競合を回避しています。

保守担当者向けメモ

  • 本関数は GIN インデックスのページ割り当ての基点となります。
  • GinPageIsRecyclable の判定ロジックが変更された場合、再利用効率に直接影響します。
