RelFileLocator

定義

  • 種別: struct
  • 定義箇所: src/include/storage/relfilelocator.h:58

目的

PostgreSQLにおいて、テーブルやインデックスなどのリレーションが物理的に格納されている場所を特定するための識別子を提供します。ストレージサブシステムがファイルシステム上のパスへアクセスする際の「住所」として機能します。

シグネチャまたは形状

typedef struct RelFileLocator
{
	Oid			spcOid;			/* tablespace */
	Oid			dbOid;			/* database */
	RelFileNumber relNumber;	/* relation */
} RelFileLocator;

入力と出力

  • **入力**: 構造体の各フィールド（spcOid, dbOid, relNumber）は、カタログやリレーションキャッシュから取得された解決済みのOIDおよびリレーション番号です。
  • **出力**: 本構造体は、relpath() 等の関数へ渡されることで、最終的にファイルシステム上の物理パス文字列を生成するための基盤データとなります。

前提条件と不変条件

  • **正規化**: pg_class に存在する省略形（デフォルトテーブルスペース等）は許容されず、必ず解決済みのOIDを格納する必要があります。
  • **パディングの禁止**: ハッシュテーブルのキーとして利用されるため、構造体のメンバ追加や変更を行う際は、パディングが発生しないよう厳密に管理する必要があります。

アルゴリズムまたは振る舞い

本構造体自体はデータ保持を目的としており、実行時の制御フローは持ちません。ストレージマネージャ（smgr）や磁気ディスクマネージャ（md.c）が物理ファイルI/Oを行う際、この識別子を基にパスを特定します。

関連データ構造

  • RelFileLocatorBackend: RelFileLocator に backend プロセス番号を加えた構造体。一時リレーションの識別に用いられます。

失敗モードと境界条件

  • RelFileLocatorEquals マクロによる比較において、パディングが含まれると誤判定の原因となります。
  • RelFileLocatorBackend を使用する場合、backend フィールドが INVALID_PROC_NUMBER であれば共有リレーション、それ以外であればバックエンドローカルな一時リレーションとして扱われます。

保守担当者向けメモ

  • 構造体メンバの変更はハッシュ計算に影響を与えるため、慎重に行ってください。
  • 本構造体は物理パス生成の起点となるため、ストレージ層の変更時には必ず整合性を確認してください。
