record_send

定義

  • 種別: function
  • 定義箇所: src/backend/utils/adt/rowtypes.c:686

目的

複合型（Composite Type）または匿名レコードを、PostgreSQLのバイナリ転送形式（bytea）にシリアライズする。

シグネチャまたは形状

Datum record_send(PG_FUNCTION_ARGS)

入力と出力

  • **入力**: PG_FUNCTION_ARGS を介して HeapTupleHeader を受け取る。
  • **出力**: bytea 型のバイナリデータ（PG_RETURN_BYTEA_P）。

前提条件と不変条件

  • check_stack_depth() により、再帰的なレコード構造に対するスタックオーバーフロー保護が行われる。
  • 削除された列（attisdropped）は出力から除外される。
  • fn_extra を使用して、レコード構造に対応するI/O関数情報をキャッシュする。型やtypmodが変更されない限り、このキャッシュは再利用される。

アルゴリズムまたは振る舞い

1. lookup_rowtype_tupdesc を用いてタプルの型定義を取得する。 2. heap_deform_tuple を使用して、タプルをフィールド値（Datum）とNULLフラグの配列に分解する。 3. pq_begintypsend でバイナリバッファを開始し、削除済み列を除いた列数を書き込む。 4. 各フィールドを順次処理する:

  • 列がNULLの場合、データ長として -1 を書き込む。
  • 非NULLの場合、各型のバイナリ出力関数（getTypeBinaryOutputInfo で取得）を呼び出し、その結果をバッファに書き込む。

    • 5. pq_endtypsend で構築したバッファを bytea として返す。

呼び出し元と呼び出し先

  • **呼び出し元**: ソースコード内の明示的な呼び出し元は特定されていないが、PostgreSQLのバイナリ出力プロトコルの一部として機能する。
  • **呼び出し先**:
  • lookup_rowtype_tupdesc
  • heap_deform_tuple
  • pq_begintypsend / pq_sendint32 / pq_sendbytes / pq_endtypsend
  • getTypeBinaryOutputInfo
  • SendFunctionCall

関連データ構造

  • RecordIOData: フィールドごとのI/O関数情報や型情報を保持するキャッシュ構造体。
  • ColumnIOData: 個別の列に対する型情報および関数ポインタ。
  • TupleDesc: タプルの構造定義。
  • HeapTupleHeader: 処理対象のタプルデータ。

失敗モードと境界条件

  • レコード内にレコードが含まれる場合、再帰的な処理が行われるためスタック深度に依存する。
  • fn_extra のキャッシュは、呼び出し間でレコードの型定義が変更されないことを前提としている。

保守担当者向けメモ

  • attisdropped の判定により、物理的な列位置と論理的な出力順序が一致しない場合がある点に注意すること。
  • fn_extra のメモリ管理は fcinfo->flinfo->fn_mcxt に依存しており、関数呼び出しのライフサイクルに紐付いている。
