heap_update

定義

  • 種別: function
  • 定義箇所: src/backend/access/heap/heapam.c:3241

目的

リレーション内の既存タプルを新しいタプルで置き換えます。HOT（Heap-Only Tuple）更新の判定、行レベルロックの獲得、およびMVCC整合性の維持を担います。

シグネチャまたは形状

TM_Result heap_update(Relation relation, ItemPointer otid, HeapTuple newtup,
                      CommandId cid, Snapshot crosscheck, bool wait,
                      TM_FailureData *tmfd, LockTupleMode *lockmode,
                      TU_UpdateIndexes *update_indexes)

入力と出力

  • **入力**:
  • relation: 対象のリレーション。
  • otid: 更新対象タプルのTID（ItemPointer）。
  • newtup: 更新後の新しいタプルデータ。
  • cid: 現在のコマンドID。
  • crosscheck: 可視性確認用のスナップショット。
  • wait: 競合時に待機するかどうかのフラグ。
  • **出力**:
  • TM_Result: 更新結果（成功、削除済み、更新済み等）。
  • tmfd: 更新失敗時の詳細情報（TIDやXMAX）。
  • lockmode: 獲得した行ロックのモード。
  • update_indexes: インデックス更新が必要かどうかのフラグ。

前提条件と不変条件

  • ItemPointerIsValid(otid) が真であること。
  • 並列操作（Parallel Mode）中ではないこと。
  • AssertHasSnapshotForToast(relation) により、TOAST処理に必要なスナップショットが保持されていること。
  • 更新対象のタプルが LP_NORMAL 状態であること（システムカタログ等の例外を除く）。

アルゴリズムまたは振る舞い

1. **属性解析**: RelationGetIndexAttrBitmap を使用し、更新対象列がインデックスキーやレプリカアイデンティティに含まれるかを判定。 2. **バッファ確保**: 対象ブロックを読み込み、必要に応じて可視性マップ（VM）をピン留めし、排他ロックを獲得。 3. **可視性判定**: HeapTupleSatisfiesUpdate を呼び出し、対象タプルが更新可能か確認。 4. **競合処理**: 他のトランザクションが更新中の場合、wait が真であれば heap_acquire_tuplock を経由して待機。 5. **更新実行**:

  • インデックスキーに変更がない場合はHOT更新を試行。
  • 変更がある場合は通常更新として処理。
  • 必要に応じてTOAST処理を実行。

    • 6. **終了処理**: クリティカルセクション内でページ更新とWAL記録を行い、バッファロックを解除。

呼び出し元と呼び出し先

  • **呼び出し元**: 実行エンジン（ExecUpdate 等）から呼び出されることが想定されますが、ソースコード上では heapam.c 内の heap_inplace_lock 等からの参照や、heapam_handler.c を介したテーブルAMインターフェースとしての利用が確認されています。
  • **呼び出し先**: HeapTupleSatisfiesUpdate, RelationGetIndexAttrBitmap, HeapDetermineColumnsInfo, LockBuffer, ReadBuffer, MultiXactIdSetOldestMember 等。

関連データ構造

  • Relation: 対象テーブルのメタデータ。
  • HeapTuple: タプルデータ。
  • TM_Result: 操作結果の列挙型。
  • Bitmapset: 更新対象属性の管理に使用。
  • Buffer: ページバッファ。

失敗モードと境界条件

  • **TM_Deleted**: 対象タプルが既に削除されている場合。
  • **TM_Invisible**: 対象タプルが現在のスナップショットから不可視である場合。
  • **TM_BeingModified**: 他のトランザクションが更新中で、wait が偽の場合。
  • **エラー**: 並列操作中の呼び出しは ERRCODE_INVALID_TRANSACTION_STATE でエラー終了します。

保守担当者向けメモ

  • 本関数は heapam.c 内の非常に複雑な関数の一つであり、HOT更新と通常更新の分岐条件は HeapDetermineColumnsInfo の結果に依存します。
  • 競合時の待機処理（l2 ラベルへのループバック）は、デッドロックを避けるため慎重に実装されています。変更を加える際は、既存のロック獲得順序を厳守してください。
