plan_union_children

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/prep/prepunion.c:1268

目的

UNION または UNION ALL 演算において、同一の演算特性を持つ子ノードを再帰的にフラット化し、個別のサブクエリとしてプランニングする。

シグネチャまたは形状

static List *plan_union_children(PlannerInfo *root,
                                 SetOperationStmt *top_union,
                                 List *refnames_tlist,
                                 List **tlist_list,
                                 List **istrivial_tlist)

入力と出力

  • **入力**:
  • root: プランナのグローバル情報。
  • top_union: 処理対象の SetOperationStmt。
  • refnames_tlist: 参照名を含むターゲットリスト。
  • tlist_list: 各子ノードのターゲットリストを格納するリストへのポインタ。
  • istrivial_tlist: 各子ノードのターゲットリストが自明（変換不要）かを示すフラグリストへのポインタ。
  • **出力**:
  • 処理された各子ノードのプラン結果（List）。

前提条件と不変条件

  • top_union の colTypes および colCollations が、フラット化対象の各子ノードと一致している必要がある。
  • tlist_list および istrivial_tlist は、関数呼び出し時に空のリストとして初期化される。

アルゴリズムまたは振る舞い

1. pending_rels リストを初期化し、top_union を追加する。 2. pending_rels が空になるまでループ処理を行う。 3. setOp が SetOperationStmt であり、かつ top_union と演算の種類（op）、all フラグ、列型、照合順序が一致する場合、その子ノード（larg, rarg）を pending_rels に追加して再帰的にフラット化する。 4. 一致しない場合、recurse_set_operations を呼び出して子ノードを個別にプランニングする。

  • top_union が UNION ALL でない場合、ソート順序を維持するために top_union を親演算子として渡す。

    • 5. 得られた結果を result リストに追加し、ターゲットリスト情報を出力用ポインタに格納する。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/optimizer/prep/prepunion.c 内の generate_union_paths（行706付近）。
  • **呼び出し先**: recurse_set_operations。

関連データ構造

  • SetOperationStmt: 集合演算の論理構造。
  • PlannerInfo: プランニング中のクエリ全体のメタデータ。
  • List: PostgreSQL の汎用リスト構造。

失敗モードと境界条件

  • 集合演算のネストが深い場合、再帰呼び出しによりスタックオーバーフローが発生する可能性がある（ファイル全体の文脈より）。
  • top_union と子ノード間で型や照合順序が不一致の場合、フラット化は行われず個別にプランニングされる。

保守担当者向けメモ

  • 本関数は UNION のフラット化を行うことで、不要な中間ノードの生成を抑制する最適化を担う。
  • top_union->all が真の場合、ソート順序の制約が緩和されるため、recurse_set_operations への引数に注意すること。
