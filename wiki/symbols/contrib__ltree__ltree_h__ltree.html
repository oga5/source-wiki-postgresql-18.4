ltree

定義

  • 種別: struct
  • 定義箇所: contrib/ltree/ltree.h:42

目的

ドット区切りのラベルパスを格納するための主要なデータ構造です。階層構造を持つパスをメモリ上に保持し、PostgreSQL の varlena 型として扱えるように設計されています。

シグネチャまたは形状

typedef struct {
    int32    vl_len_;
    uint16   numlevel;
    char     data[FLEXIBLE_ARRAY_MEMBER];
} ltree;

入力と出力

  • **入力**: ltree_in 関数等を通じて文字列から変換されたラベルの集合。
  • **出力**: varlena 形式のデータとしてメモリ上に配置され、SQL インターフェースを通じてクエリやインデックス操作に利用される。

前提条件と不変条件

  • vl_len_ は varlena ヘッダーであり、直接操作してはならない。
  • data フィールドは ltree_level 構造体の配列であり、MAXALIGN に基づくアライメントが必須である。
  • sizeof を直接使用するとアライメント違反が発生するため、サイズ計算には LTREE_HDRSIZE 等の専用マクロを使用する必要がある。

アルゴリズムまたは振る舞い

  • 階層的なラベルパスをシリアライズされた形式で保持する。
  • numlevel フィールドにより、格納されているラベルの総数を管理する。
  • 可変長データとして、PostgreSQL のメモリ管理システムと統合されている。

呼び出し元と呼び出し先

  • **呼び出し元**: contrib/ltree/_ltree_gist.c（GiST インデックス操作）、contrib/ltree/_ltree_op.c（演算子実装）等から参照されている。
  • **呼び出し先**: 特定の関数呼び出しはソース抜粋に含まれていない。

関連データ構造

  • ltree_level: data 配列内に格納される個々のラベル要素。
  • lquery: ltree を検索するためのパターン構造体。
  • ltree_gist: GiST インデックス構築時に使用される格納形式。

失敗モードと境界条件

  • MAXALIGN を考慮しないメモリ操作は、アライメント違反を引き起こす。
  • varlena ヘッダーの不適切な操作は、データ破損やセグメンテーション違反の原因となる。

保守担当者向けメモ

  • FLEXIBLE_ARRAY_MEMBER を使用しているため、構造体の拡張やサイズ計算を行う際は必ず LTREE_HDRSIZE マクロを介すること。
  • ディスクフォーマットとの互換性が重要であるため、構造体のフィールド順序や型定義を変更する際は注意が必要である。
