ginVacuumPostingTree

定義

  • 種別: function
  • 定義箇所: src/backend/access/gin/ginvacuum.c:407

目的

GINインデックスのポストティングツリーをバキュームし、不要なリーフページの削除および空になったページの再利用処理を行う。

シグネチャまたは形状

static void ginVacuumPostingTree(GinVacuumState *gvs, BlockNumber rootBlkno)

入力と出力

  • **入力**:
  • gvs: バキューム処理の状態を保持する GinVacuumState へのポインタ。
  • rootBlkno: 対象となるポストティングツリーのルートブロック番号。
  • **出力**: なし（副作用としてインデックスページを更新）。

前提条件と不変条件

  • ginVacuumPostingTreeLeaves が呼び出され、ツリー内のリーフページが検査済みであること。
  • 空のページが存在する場合、ツリー全体の再スキャンと構造整理が必要となる。

アルゴリズムまたは振る舞い

1. ginVacuumPostingTreeLeaves を呼び出し、空のページが存在するか判定する。 2. 空ページが存在する場合：

  • ルートページを ReadBufferExtended で読み込み、LockBufferForCleanup で排他ロックを取得する（同時挿入の防止）。
  • DataPageDeleteStack を初期化し、ginScanToDelete を呼び出して空ページの削除とツリー構造の再帰的な整理を実行する。
  • 削除処理で使用したスタックメモリを解放する。
  • バッファのロックを解除し、解放する。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/access/gin/ginvacuum.c:669
  • **呼び出し先**:

関連データ構造

  • GinVacuumState: バキュームの実行状態を管理。
  • DataPageDeleteStack: 削除対象のページ階層を追跡するためのスタック構造。

失敗モードと境界条件

  • ginVacuumPostingTreeLeaves が空ページを検出しなかった場合、本関数は何も行わず終了する。
  • ポストティングツリーのルートロック取得時に競合が発生する可能性がある（LockBufferForCleanup による待機）。

保守担当者向けメモ

  • 本関数はポストティングツリーの物理的な再構成を伴うため、ロック順序の厳守が必要。
  • ginScanToDelete 内でのロック処理と併せて、デッドロックが発生しないよう設計されている。
