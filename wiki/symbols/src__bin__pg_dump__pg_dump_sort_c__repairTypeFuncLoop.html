repairTypeFuncLoop

定義

  • 種別: function
  • 定義箇所: src/bin/pg_dump/pg_dump_sort.c:939

目的

型（Type）と関数（Function）の間に発生する循環依存関係を解消し、ダンプ順序の整合性を確保する。

シグネチャまたは形状

static void repairTypeFuncLoop(DumpableObject *typeobj, DumpableObject *funcobj)

入力と出力

  • typeobj: 循環に関与する型オブジェクト。
  • funcobj: 循環に関与する関数オブジェクト。
  • 出力: なし（副作用として依存関係グラフを更新）。

前提条件と不変条件

  • typeobj は TypeInfo 構造体としてキャスト可能であること。
  • 循環依存の検出後、トポロジカルソートを再試行する文脈で呼び出されることを想定している。

アルゴリズムまたは振る舞い

1. funcobj から typeobj への直接的な依存関係を削除する（removeObjectDependency）。 2. typeobj がシェル型（shellType）を持つ場合、funcobj からそのシェル型への依存関係を新たに追加する（addObjectDependency）。 3. funcobj がダンプ対象としてマークされている場合、対応するシェル型もダンプ対象としてマークし、定義（DUMP_COMPONENT_DEFINITION）を含めるようフラグを更新する。

呼び出し元と呼び出し先

  • 呼び出し元: src/bin/pg_dump/pg_dump_sort.c 内の循環依存解決ロジック（1184行目および1191行目）。
  • 呼び出し先: removeObjectDependency, addObjectDependency。

関連データ構造

  • DumpableObject: ダンプ対象の汎用オブジェクト構造体。
  • TypeInfo: 型情報を保持する構造体（shellType メンバを含む）。

失敗モードと境界条件

  • typeInfo->shellType が NULL の場合、シェル型への依存関係追加処理はスキップされる。

保守担当者向けメモ

  • 本関数は、PostgreSQLのオブジェクト依存関係における「ハック」的な解決策の一つである。循環参照の解消ロジックを変更する際は、TopoSort の結果に与える影響を十分に検証すること。
