PsqlScanStateData

定義

  • **種別**: struct
  • **定義箇所**: src/include/fe_utils/psqlscan_int.h:84

目的

PostgreSQLのクライアントツール（psql等）における字句解析器の全状態を保持し、再入可能な解析を実現するための構造体です。複数のFlex定義ファイル間で状態を共有し、ネストされた解析や変数展開を管理します。

シグネチャまたは形状

typedef struct PsqlScanStateData {
    yyscan_t    scanner;
    PQExpBuffer output_buf;
    StackElem  *buffer_stack;
    YY_BUFFER_STATE scanbufhandle;
    char       *scanbuf;
    const char *scanline;
    int         encoding;
    bool        safe_encoding;
    bool        std_strings;
    const char *curline;
    const char *refline;
    int         cur_line_no;
    const char *cur_line_ptr;
    int         start_state;
    int         state_before_str_stop;
    int         paren_depth;
    int         xcdepth;
    char       *dolqstart;
    int         identifier_count;
    char        identifiers[4];
    int         begin_depth;
    const PsqlScanCallbacks *callbacks;
    void       *cb_passthrough;
} PsqlScanStateData;

入力と出力

  • **入力**: scanbuf および scanline を通じて入力文字列を受け取り、buffer_stack を介して変数展開用のバッファを管理します。
  • **出力**: output_buf に解析結果を蓄積します。

前提条件と不変条件

  • start_state は yylex() の開始および終了状態を保持し、psql_scan_reset が呼ばれるまで状態を維持します。
  • scanbufhandle、scanbuf、scanline は常に最外層のバッファを参照し、スタックされた変数展開用バッファには影響されません。
  • マルチバイト文字の整合性を保つため、emit() 関数経由で 0xFF 置換処理を行う必要があります。

アルゴリズムまたは振る舞い

  • **再入可能性**: yyscan_t を保持することで、複数の解析操作をネスト可能です。
  • **バッファ管理**: psqlscan_push_new_buffer によりスタックへバッファを追加し、解析対象を切り替えます。
  • **状態追跡**: 括弧の深さ (paren_depth)、コメントの深さ (xcdepth)、BEGIN...END ブロックの深さ (begin_depth) を管理し、構文解析の文脈を維持します。

呼び出し元と呼び出し先

  • **呼び出し元**: src/include/fe_utils/psqlscan.h にて PsqlScanState 型として定義され、psqlの字句解析処理全般で使用されます。
  • **呼び出し先**: 構造体内の callbacks フィールドを通じて、外部から提供されたコールバック関数を呼び出します。

関連データ構造

  • StackElem: 変数展開時に使用されるバッファスタックの要素。
  • yyscan_t: Flexの再入可能なスキャナ状態を指すポインタ。
  • PsqlScanCallbacks: 字句解析器が利用するコールバック関数の集合。

失敗モードと境界条件

  • psqlscan_emit を介さずに直接バッファを操作した場合、マルチバイト文字が破壊されるリスクがあります。
  • 異なるFlexバージョンで生成されたコードを混在させると、yyscan_t の互換性問題が発生する可能性があります。

保守担当者向けメモ

  • 本構造体は psqlscan.l および psqlscanslash.l で共有されます。変更時は両方の解析器への影響を考慮してください。
  • 0xFF 置換ロジックは複雑であるため、マルチバイト文字の取り扱いには細心の注意を払ってください。
