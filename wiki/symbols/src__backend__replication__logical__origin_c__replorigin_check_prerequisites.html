replorigin_check_prerequisites

定義

  • 種別: function
  • 定義箇所: src/backend/replication/logical/origin.c:189

目的

論理レプリケーション起点の操作を行う前に、システム構成および現在の実行状態が適切であるかを検証する。

シグネチャまたは形状

static void replorigin_check_prerequisites(bool check_origins, bool recoveryOK)

入力と出力

  • check_origins (bool): true の場合、max_active_replication_origins が設定されているかを確認する。
  • recoveryOK (bool): true の場合、リカバリ中であっても操作を許可する。
  • 出力: なし。条件を満たさない場合は ereport(ERROR, ...) により処理を中断する。

前提条件と不変条件

  • max_active_replication_origins が 0 の場合、check_origins が true であればエラーとなる。
  • リカバリ中（RecoveryInProgress() が真）かつ recoveryOK が false の場合、エラーとなる。

アルゴリズムまたは振る舞い

1. check_origins が真かつ max_active_replication_origins == 0 の場合、論理レプリケーション起点が利用不可である旨のエラーを発生させる。 2. recoveryOK が偽かつシステムがリカバリ中である場合、リカバリ中の起点操作は許可されない旨のエラーを発生させる。

呼び出し元と呼び出し先

  • 呼び出し元: src/backend/replication/logical/origin.c 内の各種起点操作関数（replorigin_create, replorigin_drop, replorigin_advance 等）から、操作の直前に呼び出される。
  • 呼び出し先: RecoveryInProgress()（PostgreSQL内部関数）。

関連データ構造

  • max_active_replication_origins (GUC変数)

失敗モードと境界条件

  • max_active_replication_origins が 0 に設定されている環境で起点操作を試みた場合、ERRCODE_OBJECT_NOT_IN_PREREQUISITE_STATE で失敗する。
  • スタンバイノード等でリカバリ中に起点操作を試みた場合、ERRCODE_READ_ONLY_SQL_TRANSACTION で失敗する。

保守担当者向けメモ

  • 本関数は、論理レプリケーション起点の管理APIにおけるガード節として機能する。
  • 新たな起点操作関数を追加する際は、必ず本関数による事前チェックを呼び出すこと。
