ReorderBufferToastEnt

定義

  • **種別**: struct
  • **定義箇所**: src/backend/replication/logical/reorderbuffer.c:177

目的

論理デコーディング中に、WALから分割されて読み込まれたTOASTチャンクを一時的に保持し、再結合（デトースト）して完全なタプルを復元するための管理構造体です。

シグネチャまたは形状

typedef struct ReorderBufferToastEnt
{
    Oid         chunk_id;        /* toast_table.chunk_id */
    int32       last_chunk_seq;  /* 最後に確認したチャンクのシーケンス番号 */
    Size        num_chunks;      /* 既に確認済みのチャンク数 */
    Size        size;            /* 確認済みチャンクの合計サイズ */
    dlist_head  chunks;          /* チャンクの連結リスト */
    struct varlena *reconstructed; /* 再構築されたvarlenaデータ */
} ReorderBufferToastEnt;

入力と出力

  • **入力**: WALレコードから抽出された個別のTOASTチャンクデータ。
  • **出力**: reconstructed フィールドに格納される、再結合済みの完全な varlena 構造体。

前提条件と不変条件

  • chunk_id は、対象となるTOASTテーブル内での一意な識別子として機能します。
  • chunks リスト内の各チャンクは、last_chunk_seq を通じて順序が管理されます。
  • reconstructed は、すべてのチャンクが揃った後に生成され、メインタプルから参照されます。

アルゴリズムまたは振る舞い

1. WALからTOASTチャンクが到着するたびに、対応する ReorderBufferToastEnt がハッシュテーブルから検索または作成されます。 2. チャンクは chunks リストに追加され、num_chunks と size が更新されます。 3. すべてのチャンクが揃った時点で、それらを連結し、reconstructed フィールドに完全なデータを保持します。 4. 最終的に、この構造体はメモリ上のタプル再構築プロセスの一部として利用されます。

呼び出し元と呼び出し先

  • **呼び出し元**: reorderbuffer.c 内のTOAST再構築ロジック（hash_seq_search を含むハッシュテーブル操作関数群）。
  • **呼び出し先**: 不明（ソースコードの文脈上、本構造体はデータ保持用であり、特定の関数を呼び出すインターフェースではありません）。

関連データ構造

  • ReorderBuffer: 本構造体を管理するハッシュテーブルを保持するメイン構造体。
  • varlena: PostgreSQLの可変長データ型。

失敗モードと境界条件

  • **メモリ不足**: 大規模なTOASTデータが蓄積される場合、logical_decoding_work_mem の制限に抵触する可能性があります。
  • **チャンク欠損**: WALの読み取り順序や欠損により、すべてのチャンクが揃わない場合、再構築が完了しません。

保守担当者向けメモ

  • 本構造体はハッシュテーブルのエントリとして使用されます（sizeof(ReorderBufferToastEnt) がハッシュテーブルの entrysize として指定されています）。
  • dlist_head を使用しているため、チャンクの追加・削除はリスト操作のコストに依存します。
  • デバッグ時は、ReorderBufferToastEnt を格納するハッシュテーブルの走査（hash_seq_search）箇所を確認してください。
