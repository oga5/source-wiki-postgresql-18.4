pg_relation_size

定義

  • 種別: function
  • 定義箇所: src/backend/utils/adt/dbsize.c:363

目的

指定されたリレーション（テーブルやインデックス等）の物理的なディスクサイズをバイト単位で取得します。

シグネチャまたは形状

Datum pg_relation_size(PG_FUNCTION_ARGS)

入力と出力

  • **入力**:
  • relOid (OID): 対象リレーションのOID。
  • forkName (text): 対象とするフォーク名（例: 'main', 'fsm', 'vm'）。
  • **出力**:
  • int64: リレーションのバイトサイズ。対象リレーションが存在しない場合は NULL を返します。

前提条件と不変条件

  • 対象リレーションに対して AccessShareLock を取得して処理を行います。
  • リレーションのオープン時に存在しない場合（既に削除済みの場合など）は、エラーを発生させず NULL を返します。

アルゴリズムまたは振る舞い

1. try_relation_open を使用して、指定されたOIDのリレーションを AccessShareLock モードで開きます。 2. リレーションが取得できない場合は PG_RETURN_NULL() を返します。 3. forkname_to_number を用いて、入力されたフォーク名を内部的な数値へ変換します。 4. calculate_relation_size を呼び出し、リレーションのロケータとフォーク番号に基づき物理サイズを算出します。 5. relation_close を呼び出してロックを解放します。 6. 算出されたサイズを int64 として返却します。

呼び出し元と呼び出し先

  • **呼び出し先**:
  • try_relation_open
  • forkname_to_number
  • text_to_cstring
  • calculate_relation_size
  • relation_close
  • **呼び出し元**:
  • contrib/postgres_fdw/deparse.c（SQL生成時の参照として利用）
  • その他、SQLインターフェース経由でユーザーから直接呼び出されます。

関連データ構造

  • Relation: リレーションのメタデータ構造体。
  • RelFileLocator: リレーションの物理的な位置を特定するための構造体。

失敗モードと境界条件

  • 指定されたOIDがシステムカタログ上に存在しない場合、try_relation_open が NULL を返し、本関数は安全に NULL を返却します。
  • 権限チェックについては、本関数が呼び出される前に object_aclcheck 等のメカニズムによりアクセス権限が検証されることが想定されています。

保守担当者向けメモ

  • 本関数は、クエリ実行中にリレーションが削除される可能性を考慮し、エラーではなく NULL を返す設計となっています。
  • calculate_relation_size への依存があるため、リレーションのフォーク構造に変更があった場合は、本関数の呼び出し引数やロジックの整合性を確認してください。
