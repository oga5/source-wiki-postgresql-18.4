create_append_plan

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/plan/createplan.c:1215

目的

AppendPath ノードを、実行エンジンが処理可能な Append プランノードへ変換します。複数の子パスを統合し、必要に応じてソートや非同期実行の準備を行います。

シグネチャまたは形状

Plan *create_append_plan(PlannerInfo *root, AppendPath *best_path, int flags)

入力と出力

  • **入力**:
  • root: プランナのグローバル情報。
  • best_path: 変換対象の AppendPath。
  • flags: プラン生成時の制約フラグ。
  • **出力**:
  • 変換された Plan ノード（Append または Result）。

前提条件と不変条件

  • best_path->subpaths が空の場合、定数 FALSE を持つ Result ノードが生成されます。
  • 子プランはすべて同一のターゲットリスト（tlist）を返す必要があります。
  • ソートが必要な場合、prepare_sort_from_pathkeys を用いて子プランの tlist が期待されるソートキーと一致することが保証されます。

アルゴリズムまたは振る舞い

1. **空パスの処理**: 子パスが存在しない場合、FALSE を返す Result ノードを生成して終了します。 2. **プラン初期化**: Append ノードを生成し、ターゲットリストを設定します。 3. **ソート準備**: pathkeys が指定されている場合、prepare_sort_from_pathkeys を呼び出し、ソート列情報を計算して Append ノードの tlist を調整します。 4. **子プランの再帰的生成**: create_plan_recurse を用いて各子パスを変換します。

  • ソートが必要な場合、子プランが順序を満たしていなければ Sort ノードを挿入します。
  • 非同期実行が有効な場合、mark_async_capable_plan を試行します。

    • 5. **パーティションプルーニング**: enable_partition_pruning が有効な場合、実行時のパーティション除外に必要な情報を収集します。

呼び出し元と呼び出し先

  • **呼び出し元**: create_plan_recurse（src/backend/optimizer/plan/createplan.c:422）から呼び出されます。
  • **呼び出し先**: build_path_tlist, make_result, makeNode(Append), prepare_sort_from_pathkeys, create_plan_recurse, make_sort, label_sort_with_costsize, mark_async_capable_plan, extract_actual_clauses 等。

関連データ構造

  • AppendPath: 入力となるパス構造。
  • Append: 出力されるプラン構造。
  • PlannerInfo: プランナのコンテキスト情報。

失敗モードと境界条件

  • 子プランのターゲットリストが Append ノードの期待と一致しない場合、elog(ERROR) により処理が中断されます。
  • 子パスが空の場合、Result ノードによるダミー実行計画が生成されます。

保守担当者向けメモ

  • Append ノード自体は子プランが1つの場合でも削除されません（setrefs.c での処理に委ねられます）。
  • ソート情報の整合性チェックは厳格に行われており、memcmp を用いてソートキーの定義が一致することを確認しています。
