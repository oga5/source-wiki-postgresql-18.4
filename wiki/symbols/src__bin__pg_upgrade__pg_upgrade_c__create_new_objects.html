create_new_objects

定義

  • 種別: function
  • 定義箇所: src/bin/pg_upgrade/pg_upgrade.c:571

目的

新クラスタにおいて pg_restore を実行し、データベーススキーマを再構築する。

シグネチャまたは形状

static void create_new_objects(void)

入力と出力

  • **入力**: グローバル変数 old_cluster (旧クラスタ情報) および new_cluster (新クラスタ情報) を参照。
  • **出力**: なし（副作用として新クラスタへのスキーマ復元を実行）。

前提条件と不変条件

  • old_cluster および new_cluster が初期化済みであること。
  • template1 データベースの変更がディスクに書き込まれている必要がある（関数内で CHECKPOINT を実行して保証）。

アルゴリズムまたは振る舞い

1. **template1 の準備**: template1 データベースに対して CHECKPOINT を実行し、状態を確定させる。 2. **template1 の復元**: template1 は他のデータベースと並行処理できないため、単独で pg_restore を実行（--clean --create オプションを使用）。 3. **他データベースの復元**: template1 以外の全データベースに対し、pg_restore を並行実行。

  • postgres データベースには --clean --create を適用。
  • その他には --create を適用。
  • 並行実行数 (user_opts.jobs) に応じて --transaction-size を調整。

    • 4. **終了処理**: 全子プロセスの終了を待ち、check_ok() で完了を通知。 5. **事後処理**:

  • PostgreSQL 9.2 以前からのアップグレードの場合、set_frozenxids(true) を呼び出す。
  • get_db_rel_and_slot_infos(&new_cluster) を呼び出し、新クラスタの情報を更新する。

呼び出し元と呼び出し先

  • **呼び出し元**: main 関数 (src/bin/pg_upgrade/pg_upgrade.c:170)。
  • **呼び出し先**:
  • connectToServer, executeQueryOrDie, PQclear, PQfinish (libpq)
  • prep_status_progress, pg_log, end_progress_output, check_ok
  • exec_prog, parallel_exec_prog
  • reap_child
  • set_frozenxids
  • get_db_rel_and_slot_infos

関連データ構造

  • old_cluster, new_cluster: クラスタ情報を保持するグローバル構造体。
  • DbInfo: データベース情報を保持する構造体。
  • log_opts: ログ設定を保持するグローバル構造体。
  • user_opts: ユーザー指定オプション（並列数など）を保持するグローバル構造体。

失敗モードと境界条件

  • pg_restore がエラーを返した場合、--exit-on-error により処理が中断される。
  • template1 の復元中に接続失敗が発生する可能性があるため、非並列で処理される。
  • 並列実行時、--transaction-size が小さすぎるとロック競合が発生し、大きすぎるとメモリ消費が増大する（Max(txn_size, 10) で下限を保護）。

保守担当者向けメモ

  • template1 の復元は、データベースのドロップと再作成を伴うため、接続の排他性が重要である。
  • RESTORE_TRANSACTION_SIZE 定数は、スキーマ復元のパフォーマンスに直結するため、調整時は並列数とのバランスを考慮すること。
  • Windows 環境では exec_prog 経由のプロセス管理に依存しているため、reap_child の呼び出し漏れがないよう注意が必要。
