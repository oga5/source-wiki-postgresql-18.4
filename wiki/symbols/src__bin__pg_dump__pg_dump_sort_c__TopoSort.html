TopoSort

定義

  • 種別: function
  • 定義箇所: src/bin/pg_dump/pg_dump_sort.c:609

目的

データベースオブジェクトの依存関係に基づき、復元時に整合性が保たれるようなトポロジカルソート順序を生成する。

シグネチャまたは形状

static bool TopoSort(DumpableObject **objs, int numObjs, DumpableObject **ordering, int *nOrdering)

入力と出力

  • **入力**:
  • objs: ソート対象の DumpableObject ポインタ配列。
  • numObjs: 配列内のオブジェクト数。
  • **出力**:
  • ordering: ソート結果が格納される配列。
  • nOrdering: 実際にソートされた（またはソート不能で残った）オブジェクト数。
  • **戻り値**: ソートが完全に成功した場合は true、循環依存等により一部ソートできなかった場合は false。

前提条件と不変条件

  • objs 内の各オブジェクトは dumpId と dependencies（依存先IDリスト）を保持している必要がある。
  • maxDumpId は getMaxDumpId() によって取得可能であること。
  • 内部で使用する binaryheap は int_cmp 関数により優先度を管理する。

アルゴリズムまたは振る舞い

1. 各オブジェクトの依存関係を走査し、各IDに対する「先行制約数（beforeConstraints）」を計算する。 2. 先行制約数が 0 のオブジェクトを優先度付きキュー（binaryheap）に投入する。 3. キューからオブジェクトを取り出し、結果配列の末尾から順に配置する。 4. 取り出したオブジェクトが依存していた他のオブジェクトの先行制約数を減算し、0 になったものをキューに追加する。 5. 全てのオブジェクトを処理できた場合は成功。循環参照等でキューが空になっても未処理のオブジェクトが残る場合は失敗とみなす。

呼び出し元と呼び出し先

  • **呼び出し元**: src/bin/pg_dump/pg_dump_sort.c 内のループ処理から呼び出される。
  • **呼び出し先**: binaryheap 関連関数（allocate, build, add, remove_first, empty, free）、pg_malloc0, pg_malloc, pg_fatal, getMaxDumpId, int_cmp。
  • *注記*: src/backend/storage/lmgr/deadlock.c にも同名の関数が存在するが、本ページは pg_dump モジュールの実装を対象としている。

関連データ構造

  • DumpableObject: ダンプ対象のデータベースオブジェクト構造体。
  • binaryheap: 優先度付きキューとして使用されるデータ構造。
  • beforeConstraints: 各 dumpId が何個のオブジェクトに先行されるかを保持する配列。
  • idMap: dumpId から objs 配列内のインデックスを引くためのマップ。

失敗モードと境界条件

  • numObjs <= 0: 即座に true を返す。
  • invalid dumpId: dumpId が範囲外の場合、pg_fatal を呼び出し異常終了する。
  • 循環依存: 依存関係にループが含まれる場合、全てのオブジェクトをソートできず false を返す。この際、ソートできなかったオブジェクトが ordering の先頭側に残る。

保守担当者向けメモ

  • 本関数は O(N log N) で動作する。
  • 循環依存の解消は本関数の外側（findDependencyLoops 等）で行われるため、本関数が false を返した後のリカバリ処理が重要となる。
  • ソート順序の安定性はテスト結果に影響するため、変更時は DOTypeNameCompare 等のタイブレーク処理との整合性に注意すること。
