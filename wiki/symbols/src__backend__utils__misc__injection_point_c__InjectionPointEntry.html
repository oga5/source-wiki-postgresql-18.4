InjectionPointEntry

定義

  • **種別**: struct
  • **定義箇所**: src/backend/utils/misc/injection_point.c:41

目的

共有メモリ上で管理されるインジェクションポイント（実行コードへのフック）の定義を保持します。テストやデバッグ目的で、外部ライブラリの関数を動的に呼び出すためのメタデータを格納します。

シグネチャまたは形状

typedef struct InjectionPointEntry
{
    pg_atomic_uint64 generation;
    char        name[INJ_NAME_MAXLEN];
    char        library[INJ_LIB_MAXLEN];
    char        function[INJ_FUNC_MAXLEN];
    char        private_data[INJ_PRIVATE_MAXLEN];
} InjectionPointEntry;

入力と出力

  • **入力**: 登録時に name, library, function, private_data が設定されます。
  • **出力**: 実行時に library と function が動的ロードされ、private_data がコールバックへ渡されます。

前提条件と不変条件

  • 本構造体は共有メモリ上に配置されます。
  • generation カウンタにより、ロックフリーな読み取りが可能です。
  • 偶数の generation はスロットが未使用、奇数は使用中であることを示します。

アルゴリズムまたは振る舞い

  • **読み取り**:

    • 1. generation を読み取る。 2. フィールドを読み取る。 3. 再度 generation を読み取り、値が一致すれば読み取ったデータは有効とみなす。不一致の場合はリサイクルされたと判断し破棄する。

  • **書き込み**:

    • 1. InjectionPointLock を取得する。 2. 全フィールドを書き込む。 3. メモリバリアを挟んで generation を更新する。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/utils/misc/injection_point.c 内の InjectionPointAttach, InjectionPointRun, InjectionPointCacheRefresh 等が本構造体を参照・操作します。
  • **呼び出し先**: 本構造体自体はデータ構造であり、直接関数を呼び出しません。

関連データ構造

  • InjectionPointsCtl: 共有メモリ上の全エントリを管理する構造体。
  • InjectionPointCacheEntry: バックエンドローカルなキャッシュ用構造体。

失敗モードと境界条件

  • generation の不一致による読み取り失敗（並行更新によるリサイクル）。
  • INJ_NAME_MAXLEN 等の固定長バッファを超える名前やパスの指定。

保守担当者向けメモ

  • 本番環境での使用は想定されていません。
  • ロックフリー読み取りのプロトコル（generation の前後チェック）を厳守してください。
  • ビルド時に --enable-injection-points が指定されている場合のみ有効です。
