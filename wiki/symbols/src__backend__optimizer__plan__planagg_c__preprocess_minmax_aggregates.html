preprocess_minmax_aggregates

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/plan/planagg.c:72

目的

クエリ内の MIN/MAX 集約関数を、インデックススキャンを利用した効率的な LIMIT 1 サブクエリへと最適化可能か判定し、実行計画パスを生成する。

シグネチャまたは形状

void preprocess_minmax_aggregates(PlannerInfo *root)

入力と出力

  • **入力**: PlannerInfo *root（現在のクエリのプランナ情報）
  • **出力**: なし（副作用として root->minmax_aggs の設定および UPPERREL_GROUP_AGG へのパス追加を行う）

前提条件と不変条件

  • root->minmax_aggs は呼び出し時点で NIL であること。
  • クエリに集約関数が含まれていること。
  • GROUP BY 句、ウィンドウ関数、FOR UPDATE、集合演算（UNION 等）、CTE が含まれていないこと。
  • クエリが単一のテーブル（または継承された単一のテーブル）を参照していること。

アルゴリズムまたは振る舞い

1. **検証**: クエリ構造が最適化対象（単一テーブル、集約のみ）かを確認し、不適合な場合は早期リターンする。 2. **解析**: can_minmax_aggs を呼び出し、すべての集約関数がインデックス利用可能な MIN/MAX であるか検証する。 3. **パス構築**: 各集約関数に対し、build_minmax_path を用いてインデックススキャンによるパス生成を試みる。 4. **パラメータ生成**: 最適化が確定した場合、各集約用に SS_make_initplan_output_param で出力用パラメータを作成する。 5. **パス登録**: create_minmaxagg_path で生成したパスを UPPERREL_GROUP_AGG に追加する。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/optimizer/plan/planner.c:1860 (grouping_planner 内)
  • **呼び出し先**:
  • can_minmax_aggs
  • get_equality_op_for_ordering_op
  • build_minmax_path
  • SS_make_initplan_output_param
  • fetch_upper_rel
  • create_minmaxagg_path
  • create_pathtarget

関連データ構造

  • PlannerInfo: クエリ全体のプランナ状態。
  • MinMaxAggInfo: 最適化対象の集約関数に関するメタデータ。
  • UPPERREL_GROUP_AGG: 集約処理用のリレーションパス。

失敗モードと境界条件

  • クエリが複数のテーブルを参照している場合や、複雑な句（GROUP BY 等）がある場合は最適化をスキップする。
  • can_minmax_aggs が false を返す場合、または各集約関数に対してインデックス利用可能なパスが構築できない場合は、何も行わず終了する。

保守担当者向けメモ

  • 本関数は grouping_planner の前処理フェーズで実行される。
  • MinMaxAggPath は並列実行（parallel-safe）には対応していない。
  • build_minmax_path 内で PlannerInfo のクローンが行われるため、メモリ消費に注意が必要。
