EPlan

定義

  • 種別: struct
  • 定義箇所: contrib/spi/refint.c:23

目的

トリガー関数内で使用される SPI (Server Programming Interface) の実行計画をキャッシュし、セッション内での再利用を可能にすることでクエリ実行のオーバーヘッドを削減する。

シグネチャまたは形状

typedef struct
{
	char	   *ident;
	int			nplans;
	SPIPlanPtr *splan;
} EPlan;

入力と出力

本構造体はデータ保持用であり、関数ではないため直接的な入出力はない。内部的には ident（識別子）、nplans（計画数）、splan（SPIPlanPtr の配列）を保持する。

前提条件と不変条件

  • splan は SPI_prepare によって生成された実行計画を指す必要がある。
  • キャッシュされた計画は TopMemoryContext に保持される必要がある（SPI_keepplan を使用）。

アルゴリズムまたは振る舞い

1. find_plan 関数により、特定の ident（トリガー名とリレーションIDの組み合わせ）に対応する EPlan が検索される。 2. 該当する計画が存在しない場合、新規に EPlan 構造体が割り当てられ、SPI_prepare で生成された計画が格納される。 3. 既存の計画は SPI_execp を通じて実行され、整合性チェックに利用される。

呼び出し元と呼び出し先

  • **呼び出し元**: find_plan 関数が本構造体の検索およびメモリ割り当て（palloc/repalloc）を行う。また、check_primary_key および check_foreign_key 内で実行計画の管理用として参照される。
  • **呼び出し先**: 本構造体自体はデータ構造であり、関数を呼び出さない。

関連データ構造

  • SPIPlanPtr: SPI 実行計画へのポインタ。EPlan 内の splan 配列に格納される。
  • FPlans / PPlans: refint.c 内で定義される EPlan 型の静的変数。それぞれ外部キー用、主キー用のキャッシュとして機能する。

失敗モードと境界条件

  • メモリ不足時に palloc または repalloc が失敗する可能性がある。
  • ident の不一致や、キャッシュされた計画が現在のコンテキストで無効化された場合、整合性チェックが失敗する可能性がある。

保守担当者向けメモ

  • 本構造体は古いスタイルのトリガー実装用である。現代の PostgreSQL では標準の FOREIGN KEY 制約の使用が推奨される。
  • EPlan の配列を動的に拡張する際は repalloc を使用しているため、メモリ管理に注意すること。
