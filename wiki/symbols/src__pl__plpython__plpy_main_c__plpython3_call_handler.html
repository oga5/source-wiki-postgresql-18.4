plpython3_call_handler

定義

  • 種別: function
  • 定義箇所: src/pl/plpython/plpy_main.c:193

目的

PostgreSQL の SQL 関数実行時における PL/Python 言語ハンドラのメインエントリポイントです。Python インタプリタの初期化、実行コンテキストの管理、および関数またはトリガーの実行を制御します。

シグネチャまたは形状

Datum plpython3_call_handler(PG_FUNCTION_ARGS)

入力と出力

  • **入力**: PG_FUNCTION_ARGS (PostgreSQL の関数呼び出し規約に基づく引数セット)
  • **出力**: Datum (Python 関数の実行結果、またはトリガー実行時の HeapTuple を変換した Datum)

前提条件と不変条件

  • PLy_initialize() が呼び出され、Python インタプリタが利用可能であること。
  • PLy_push_execution_context() によりスタックへ積まれた実行コンテキストは、正常終了時および例外発生時の両方で必ず PLy_pop_execution_context() により解放されること。
  • error_context_stack へのコールバック登録は、PG_TRY ブロックのスコープ内で管理されること。

アルゴリズムまたは振る舞い

1. **初期化**: PLy_initialize を呼び出し、Python 環境を準備します。 2. **SPI 接続**: CallContext を確認し、アトミックでない呼び出しの場合は SPI_OPT_NONATOMIC を指定して SPI_connect_ext を実行します。 3. **コンテキスト管理**: PLy_push_execution_context で実行コンテキストをスタックに積みます。 4. **実行**: PG_TRY ブロック内で以下を行います。

  • plpython_error_callback をエラーコンテキストスタックに登録します。
  • CALLED_AS_TRIGGER を判定し、トリガーであれば PLy_procedure_get と PLy_exec_trigger を、関数であれば PLy_procedure_get と PLy_exec_function を呼び出します。

    • 5. **後処理**: PG_CATCH ブロックで例外発生時にコンテキストをポップし、PyErr_Clear で Python エラーをクリアして再スローします。正常終了時も同様にコンテキストをポップします。

呼び出し元と呼び出し先

  • **呼び出し元**: PostgreSQL の関数呼び出し機構（PG_FUNCTION_INFO_V1 マクロにより登録）。
  • **呼び出し先**:
  • PLy_initialize
  • SPI_connect_ext
  • PLy_push_execution_context / PLy_pop_execution_context
  • PLy_procedure_get
  • PLy_exec_trigger / PLy_exec_function
  • plpython_error_callback

関連データ構造

  • PLyExecutionContext: 実行中の手続きやメモリコンテキストを保持するスタック構造体。
  • ErrorContextCallback: PostgreSQL のエラー報告用コールバック構造体。
  • PLyProcedure: ロードされた Python 手続きのキャッシュ構造体。
  • TriggerData: トリガー実行時に渡されるコンテキスト情報。

失敗モードと境界条件

  • **メモリ管理**: PLy_pop_execution_context が漏れるとメモリリークが発生します。
  • **例外処理**: Python 側で例外が発生した場合、PG_CATCH ブロックにより PostgreSQL のエラーとして再スローされます。
  • **設計上の懸念**: SPI_finish() が本関数内ではなく plpy_exec.c 側で呼び出される設計となっており、責務の分離が不完全です。

保守担当者向けメモ

  • PG_TRY ブロックの開始前にエラーを投げる可能性のある処理を配置しないでください。
  • plerrcontext は PG_TRY 構造によって自動的にポップされるため、明示的な解除は不要です。
