ExecuteQuery

定義

  • 種別: function
  • 定義箇所: src/backend/commands/prepare.c:149

目的

SQLの EXECUTE 文を実行し、キャッシュされたプリペアドステートメントを検索・実行する。

シグネチャまたは形状

void ExecuteQuery(ParseState *pstate,
                  ExecuteStmt *stmt, IntoClause *intoClause,
                  ParamListInfo params,
                  DestReceiver *dest, QueryCompletion *qc)

入力と出力

  • **入力**:
  • pstate: 解析状態。
  • stmt: 実行対象の EXECUTE 文構造体。
  • intoClause: CREATE TABLE ... AS EXECUTE の場合に指定される句。
  • params: 実行時に提供されるパラメータリスト。
  • dest: クエリ結果の送信先。
  • qc: クエリ完了状態を格納する構造体。
  • **出力**: なし（副作用としてクエリを実行し、結果を dest に送信する）。

前提条件と不変条件

  • 対象のプリペアドステートメントは、固定結果（fixed-result）を持つ必要がある。
  • GetCachedPlan と PortalDefineQuery の間では、エラーを発生させる可能性のある処理を行ってはならない（計画の参照カウントリークを防ぐため）。

アルゴリズムまたは振る舞い

1. FetchPreparedStatement を使用してハッシュテーブルから対象のステートメントを検索する。 2. パラメータが存在する場合、CreateExecutorState で実行状態を作成し、EvaluateParams でパラメータを評価する。 3. CreateNewPortal で内部実行用のポータルを作成し、visible を false に設定する。 4. GetCachedPlan で実行可能な計画を取得し、PortalDefineQuery でポータルに定義する。 5. intoClause が存在する場合、対象が SELECT 文であることを検証し、GetIntoRelEFlags で実行フラグを設定する。 6. PortalStart で実行を開始し、PortalRun でクエリを実行する。 7. PortalDrop でポータルを破棄し、必要に応じて実行状態を解放する。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/commands/createas.c
  • src/backend/tcop/utility.c
  • **呼び出し先**:
  • FetchPreparedStatement, CreateExecutorState, EvaluateParams, CreateNewPortal, MemoryContextStrdup, GetCachedPlan, PortalDefineQuery, GetIntoRelEFlags, PortalStart, PortalRun, PortalDrop, FreeExecutorState

関連データ構造

  • PreparedStatement: プリペアドステートメントの管理構造体。
  • CachedPlan: キャッシュされたクエリ計画。
  • Portal: クエリ実行のコンテキスト。
  • ExecuteStmt: EXECUTE 文の構文解析結果。

失敗モードと境界条件

  • 存在しないステートメントへのアクセス: FetchPreparedStatement がエラーを投げる。
  • 可変結果を持つキャッシュ計画の実行: elog(ERROR) で停止する。
  • CREATE TABLE ... AS EXECUTE において SELECT 以外が指定された場合: ERRCODE_WRONG_OBJECT_TYPE でエラーを報告する。

保守担当者向けメモ

  • prepared_queries はセッションローカルであり、他セッションと共有されない。
  • intoClause を伴う実行時の eflags や count の制御は、将来的な拡張（INSERT ... RETURNING 等）を考慮して慎重に扱う必要がある。
