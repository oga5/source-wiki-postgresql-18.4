index_check_primary_key

定義

  • src/backend/catalog/index.c:201

目的

テーブルに対する主キー制約の追加が妥当であるかを検証する。既存の主キーとの重複チェック、インデックス属性の制約確認、および主キーとして不適切なインデックス構成の排除を行う。

シグネチャまたは形状

void index_check_primary_key(Relation heapRel,
                             const IndexInfo *indexInfo,
                             bool is_alter_table,
                             const IndexStmt *stmt)

入力と出力

  • heapRel: 対象テーブルのリレーション。
  • indexInfo: 検証対象のインデックス情報。
  • is_alter_table: ALTER TABLE 操作中であるかを示すフラグ。
  • stmt: インデックス定義文（エラーメッセージ生成に使用）。
  • 戻り値: なし。制約違反時は ereport(ERROR, ...) によりトランザクションを中断する。

前提条件と不変条件

  • heapRel は有効なリレーションであること。
  • indexInfo は BuildIndexInfo 等で構築された有効な構造体であること。

アルゴリズムまたは振る舞い

1. **重複チェック**: is_alter_table が真、または対象がパーティションである場合、relationHasPrimaryKey を呼び出し、既に主キーが存在すればエラーを投げる。 2. **インデックス属性の検証**:

  • ii_NullsNotDistinct が設定されている場合、主キーには使用できないためエラーを投げる。
  • インデックスを構成する各属性（ii_IndexAttrNumbers）を走査する。
  • 属性番号が 0（式インデックス）の場合、主キーには使用できないためエラーを投げる。
  • システム属性（負の番号）はスキップする。
  • SearchSysCache2 を使用して pg_attribute を検索し、attnotnull フラグを確認する。NOT NULL でない属性が含まれている場合はエラーを投げる。

    • 3. **リソース解放**: 検索したシステムキャッシュを ReleaseSysCache で解放する。

関連データ構造

  • Relation: テーブルのメタデータ構造。
  • IndexInfo: インデックスの属性や特性を保持する構造体。
  • IndexStmt: SQLの CREATE INDEX 文の解析結果。
  • Form_pg_attribute: pg_attribute カタログの行構造。

失敗モードと境界条件

  • **主キー重複**: 既に主キーが存在するテーブルに主キーを追加しようとした場合。
  • **NULLS NOT DISTINCT**: NULLS NOT DISTINCT が有効なインデックスを主キーに指定した場合。
  • **式インデックス**: 主キーに式（attnum == 0）が含まれる場合。
  • **NOT NULL 制約欠如**: 主キー列が NOT NULL として定義されていない場合。
  • **キャッシュ検索失敗**: pg_attribute の検索に失敗した場合（elog(ERROR)）。

保守担当者向けメモ

  • 本関数は制約の論理的な妥当性をチェックするものであり、物理的なインデックス構築の可否とは別に、カタログ上の定義整合性を保証する役割を持つ。
  • attnum < 0（システム属性）の扱いは、主キーとして使用できないという前提に基づきスキップされている。
