SetRelationRuleStatus

定義

  • 種別: function
  • 定義箇所: src/backend/rewrite/rewriteSupport.c:52

目的

指定されたテーブル（リレーション）の pg_class.relhasrules フラグを更新し、全バックエンドに対してリレーションキャッシュ（relcache）の再構築を強制することで、ルール定義の変更をシステム全体に反映させます。

シグネチャまたは形状

void SetRelationRuleStatus(Oid relationId, bool relHasRules)

入力と出力

  • **入力**:
  • relationId (Oid): 対象テーブルのOID。
  • relHasRules (bool): 設定する relhasrules の値。
  • **出力**: なし（副作用としてカタログ更新とキャッシュ無効化を実行）。

前提条件と不変条件

  • pg_class テーブルに対する RowExclusiveLock が取得可能であること。
  • 指定された relationId がシステムカタログ pg_class に存在すること（存在しない場合はエラーが発生します）。

アルゴリズムまたは振る舞い

1. table_open を使用して pg_class を RowExclusiveLock で開きます。 2. SearchSysCacheCopy1 を用いて、対象テーブルのタプルをシステムキャッシュからコピーします。 3. classForm->relhasrules の値が引数 relHasRules と異なる場合のみ、CatalogTupleUpdate を呼び出してカタログを更新します。 4. 値の変更有無に関わらず、CacheInvalidateRelcacheByTuple を呼び出し、全バックエンドの relcache を無効化します。 5. 最後にタプルを解放し、テーブルをクローズします。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/rewrite/rewriteDefine.c:485（ルールの作成・削除プロセスから呼び出されます）。
  • **呼び出し先**: table_open, SearchSysCacheCopy1, GETSTRUCT, CatalogTupleUpdate, CacheInvalidateRelcacheByTuple, heap_freetuple, table_close。

関連データ構造

  • pg_class: 更新対象となるシステムカタログ。
  • Form_pg_class: pg_class タプルの構造体。

失敗モードと境界条件

  • SearchSysCacheCopy1 が対象タプルを見つけられない場合、elog(ERROR) により処理が中断されます。

保守担当者向けメモ

  • relhasrules の値が変更されない場合でも CacheInvalidateRelcacheByTuple が実行されるのは、ルール定義の変更を確実に他バックエンドへ伝播させるための意図的な設計です。この挙動を変更する際は、キャッシュ整合性への影響を十分に考慮してください。
