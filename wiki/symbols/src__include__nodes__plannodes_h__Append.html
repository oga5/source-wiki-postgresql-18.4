Append

定義

  • 種別: struct
  • 定義箇所: src/include/nodes/plannodes.h:341

目的

複数の子プランノードを順次実行し、その結果を結合して出力するための実行計画ノードです。主にパーティションテーブルの結合や UNION ALL クエリなどで使用されます。

シグネチャまたは形状

typedef struct Append
{
    Plan        plan;
    Bitmapset  *apprelids;
    List       *appendplans;
    int         nasyncplans;
    int         first_partial_plan;
    int         part_prune_index;
} Append;

入力と出力

  • **入力**: appendplans リストに格納された複数の子プランノード。
  • **出力**: 各子プランから生成されたタプルの集合。

前提条件と不変条件

  • plan メンバが構造体の先頭に配置されており、Plan 構造体としてキャスト可能です。
  • appendplans 内のプランは、first_partial_plan インデックスを境界として、前半が非並列（non-partial）プラン、後半が並列（partial）プランとして構成されます。

アルゴリズムまたは振る舞い

1. エグゼキュータは appendplans に含まれる各子プランを順次（または並列に）実行します。 2. 各子プランから返されるタプルを、そのまま親ノードへ透過的に渡します。 3. part_prune_index が -1 でない場合、実行時にパーティションプルーニングが行われ、不要な子プランの実行がスキップされます。

呼び出し元と呼び出し先

  • **呼び出し元**: プランナ（src/backend/optimizer/）がクエリ最適化時にこのノードを生成します。
  • **呼び出し先**: エグゼキュータ（src/backend/executor/）がこのノードを走査し、各子プランの実行関数を呼び出します。
  • **補足**: ツールによる解析では、contrib/pg_overexplain/pg_overexplain.c において、デバッグや説明目的でこの構造体のメンバ（apprelids）にアクセスしている事例が確認されています。

関連データ構造

  • Plan: 基底構造体。コストやターゲットリストを保持。
  • Bitmapset: apprelids を管理するために使用。
  • List: appendplans を保持するために使用。

失敗モードと境界条件

  • part_prune_index が -1 の場合、実行時プルーニングは無効化されます。
  • appendplans が空の場合、このノードはタプルを返さずに終了します。

保守担当者向けメモ

  • lefttree/righttree ではなく appendplans リストを走査する点に注意してください。
  • first_partial_plan の値は、並列クエリ実行時のプラン選択において重要な意味を持ちます。
  • 実行時の状態管理については、src/include/nodes/execnodes.h で定義される AppendState を参照してください。
