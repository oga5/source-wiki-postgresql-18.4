JoinHashEntry

定義

  • 種別: struct
  • 定義箇所: src/backend/optimizer/util/relnode.c:38

目的

結合リレーション（joinrel）をハッシュテーブル内で管理するためのエントリ構造体です。プランナが結合リレーションを高速に検索・再利用するために使用されます。

シグネチャまたは形状

typedef struct JoinHashEntry
{
	Relids		join_relids;	/* hash key --- MUST BE FIRST */
	RelOptInfo *join_rel;
} JoinHashEntry;

入力と出力

  • 入力: ハッシュテーブル検索時のキーとして join_relids を使用します。
  • 出力: 検索成功時に対応する RelOptInfo 構造体へのポインタを保持します。

前提条件と不変条件

  • join_relids は構造体の先頭に配置される必要があります（ハッシュテーブルのキーとして機能するため）。

アルゴリズムまたは振る舞い

  • hash_search 関数を通じて、join_relids をキーとして RelOptInfo を取得または登録するために使用されます。
  • 結合リレーションの数が増加した際に、リスト検索からハッシュ検索へ移行する際のデータ格納単位となります。

呼び出し元と呼び出し先

  • 呼び出し元（参照箇所）:
  • src/backend/optimizer/util/relnode.c: build_join_rel_hash（ハッシュテーブル作成時）
  • src/backend/optimizer/util/relnode.c: find_join_rel（検索時）
  • src/backend/optimizer/util/relnode.c: add_join_rel（登録時）
  • 呼び出し先: 情報なし。

関連データ構造

  • RelOptInfo: 結合リレーションの情報を保持する主要な構造体。
  • Relids: 結合対象のリレーションを識別するビットマップセット。

失敗モードと境界条件

  • ハッシュテーブルが初期化されていない場合、find_join_rel 等の関数で root->join_rel_hash が NULL である可能性を考慮する必要があります。

保守担当者向けメモ

  • 本構造体は hash_search のエントリサイズとして sizeof(JoinHashEntry) が指定されています。構造体を変更する場合は、ハッシュテーブルの定義と整合性が取れているか確認してください。
  • join_relids が構造体の先頭にあるという不変条件は、ハッシュテーブルのキー抽出ロジックに直結するため、順序の変更は厳禁です。
