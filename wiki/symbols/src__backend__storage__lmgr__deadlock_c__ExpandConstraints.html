ExpandConstraints

定義

  • 種別: function
  • 定義箇所: src/backend/storage/lmgr/deadlock.c:789

目的

デッドロック解消のために提案された制約リスト（EDGE）を基に、ロック待ち行列の順序（WAIT_ORDER）を再構成し、整合性が取れるかを検証する。

シグネチャまたは形状

static bool ExpandConstraints(EDGE *constraints, int nConstraints)

入力と出力

  • **入力**:
  • constraints: 適用対象となる制約エッジの配列。
  • nConstraints: 制約の数。
  • **出力**:
  • bool: 順序の再構成および検証に成功した場合は true、矛盾が見つかり失敗した場合は false を返す。

前提条件と不変条件

  • nWaitOrderProcs は MaxBackends を超えないこと（Assert により検証）。
  • constraints 配列内の各要素は、ロック待ち行列の順序に対する制約を表す。

アルゴリズムまたは振る舞い

1. 制約リストを末尾から逆順に走査する（最後に追加された制約が矛盾を引き起こす可能性が最も高いため）。 2. 各制約に含まれる LOCK に対して、まだリスト化されていない場合は新しい waitOrders エントリを割り当てる。 3. dclist_count を用いてロック待ちプロセスの数を取得し、作業領域を確保する。 4. TopoSort を呼び出し、当該ロックに対するトポロジカルソートを実行して順序を決定する。

呼び出し元と呼び出し先

  • **呼び出し元**: DeadLockCheck (src/backend/storage/lmgr/deadlock.c:395)
  • **呼び出し先**: TopoSort

関連データ構造

  • EDGE: 制約を保持する構造体。
  • LOCK: ロック状態を管理する構造体。
  • waitOrders: 順序付けられたロックとプロセスのリストを保持するグローバルな作業領域。

失敗モードと境界条件

  • TopoSort が矛盾を検出した場合、関数は false を返して終了する。
  • nWaitOrderProcs が MaxBackends を超える場合、アサーション違反が発生する。

保守担当者向けメモ

  • 本関数は DeadLockCheck の一部として、ロック待ち行列の再構成を試みる重要なステップである。
  • TopoSort の結果に依存するため、トポロジカルソートのロジック変更時には本関数の整合性も再確認が必要である。
