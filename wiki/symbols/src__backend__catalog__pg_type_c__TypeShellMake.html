TypeShellMake

定義

  • **種別**: function
  • **定義箇所**: src/backend/catalog/pg_type.c:56

目的

システムカタログ pg_type に「シェル型（未定義のプレースホルダー）」を登録します。これにより、循環参照を伴う型定義において、型名のみを先行してカタログに記録し、後続の完全な定義を待機させることが可能になります。

シグネチャまたは形状

ObjectAddress TypeShellMake(const char *typeName, Oid typeNamespace, Oid ownerId)

入力と出力

  • **入力**:
  • typeName: 作成する型の名前。
  • typeNamespace: 型が属する名前空間のOID。
  • ownerId: 型の所有者のOID。
  • **出力**:
  • ObjectAddress: 作成されたシェル型のカタログ上の位置（classId と objectId）。

前提条件と不変条件

  • typeName は有効なポインタである必要があります。
  • TYPTYPE_PSEUDO として登録されるため、通常の型としては使用できません。
  • typisdefined は false に設定され、未定義状態であることが明示されます。
  • typinput および typoutput には F_SHELL_IN / F_SHELL_OUT が割り当てられます。

アルゴリズムまたは振る舞い

1. pg_type リレーションを RowExclusiveLock でオープンします。 2. pg_type の各属性を初期化し、シェル型として必要なダミー値（typtype = TYPTYPE_PSEUDO 等）をセットします。 3. IsBinaryUpgrade モードの場合は binary_upgrade_next_pg_type_oid を使用し、それ以外の場合は GetNewOidWithIndex で新しいOIDを取得します。 4. CatalogTupleInsert を使用して、初期化されたタプルを pg_type に挿入します。 5. IsBootstrapProcessingMode でない場合、GenerateTypeDependencies を呼び出し、拡張機能との依存関係などを記録します。 6. InvokeObjectPostCreateHook を呼び出し、作成後のフック処理を実行します。 7. リレーションをクローズし、作成した型の ObjectAddress を返します。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/commands/functioncmds.c
  • src/backend/commands/typecmds.c
  • **呼び出し先**:
  • table_open, table_close
  • heap_form_tuple, heap_freetuple
  • CatalogTupleInsert
  • GenerateTypeDependencies
  • InvokeObjectPostCreateHook
  • GetNewOidWithIndex

関連データ構造

  • pg_type: 型情報を保持するシステムカタログ。
  • ObjectAddress: 作成されたオブジェクトを特定するための構造体。
  • TupleDesc: pg_type のタプル記述子。

失敗モードと境界条件

  • IsBinaryUpgrade モードにおいて binary_upgrade_next_pg_type_oid が有効なOIDでない場合、エラーを報告します。
  • カタログへの挿入失敗や、メモリ不足等のシステムエラーが発生した場合は、PostgreSQLの標準的なエラー処理機構により中断されます。

保守担当者向けメモ

  • 本関数は循環参照解決のための重要なステップです。typisdefined が false であるため、この型を直接使用しようとするとエラーになる設計となっています。
  • GenerateTypeDependencies の呼び出し時に make extension dependency を true に設定している点に注意してください。
