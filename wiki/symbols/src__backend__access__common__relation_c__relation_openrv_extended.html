relation_openrv_extended

定義

  • 種別: function
  • 定義箇所: src/backend/access/common/relation.c:171

目的

RangeVar（スキーマ修飾名など）で指定されたリレーションを特定し、ロックを取得した上でリレーション記述子（Relation）を開く。missing_ok フラグにより、リレーションが存在しない場合の挙動を制御する。

シグネチャまたは形状

Relation relation_openrv_extended(const RangeVar *relation, LOCKMODE lockmode, bool missing_ok)

入力と出力

  • **入力**:
  • relation: 開く対象のリレーションを指定する RangeVar 構造体。
  • lockmode: 取得するロックモード（NoLock を含む）。
  • missing_ok: true の場合、リレーションが見つからなくてもエラーにせず NULL を返す。
  • **出力**:
  • 成功時: 開かれたリレーションの Relation 記述子。
  • 失敗時（missing_ok=true かつリレーション不在時）: NULL。

前提条件と不変条件

  • lockmode が NoLock 以外の場合、リレーションキャッシュの整合性を保つために AcceptInvalidationMessages() が呼び出される。
  • RangeVarGetRelid を介して名前解決が行われる。

アルゴリズムまたは振る舞い

1. lockmode が NoLock でない場合、AcceptInvalidationMessages() を呼び出し、キャッシュ無効化メッセージを処理する。 2. RangeVarGetRelid を呼び出し、指定された RangeVar からリレーションの OID を取得する。 3. OID が無効（存在しない）かつ missing_ok が true の場合、NULL を返して終了する。 4. 上記以外の場合、relation_open(relOid, NoLock) を呼び出し、既にロック取得済みの状態でリレーション記述子を取得して返す。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/access/table/table.c
  • src/backend/catalog/objectaddress.c
  • **呼び出し先**:
  • AcceptInvalidationMessages
  • RangeVarGetRelid
  • relation_open

関連データ構造

  • RangeVar: リレーションの名前とスキーマ情報を保持する構造体。
  • Relation: リレーションのメタデータと状態を保持する記述子。
  • LOCKMODE: ロックの種類を指定する列挙型。

失敗モードと境界条件

  • missing_ok が false の場合、リレーションが見つからないと RangeVarGetRelid 内でエラーが発生する。
  • lockmode が NoLock でない場合、ロック取得の競合が発生する可能性がある。

保守担当者向けメモ

  • 本関数は relation_openrv の拡張版であり、missing_ok フラグによる柔軟なエラーハンドリングを提供する。
  • relation_open を呼び出す際は NoLock を渡しているため、ロック取得は本関数内の RangeVarGetRelid で完結している必要がある。
