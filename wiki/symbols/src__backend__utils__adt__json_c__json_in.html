json_in

定義

  • 種別: function
  • 定義箇所: src/backend/utils/adt/json.c:106

目的

入力されたC文字列（cstring）を検証し、PostgreSQLの json 型として内部表現（text）に変換します。

シグネチャまたは形状

Datum json_in(PG_FUNCTION_ARGS)

入力と出力

  • **入力**: PG_GETARG_CSTRING(0) を介して取得される、JSON形式の文字列。
  • **出力**: text 型としてシリアライズされたJSONデータ。検証に失敗した場合は NULL を返します。

前提条件と不変条件

  • 入力文字列は、pg_parse_json_or_errsave による構文解析に合格する必要があります。
  • 内部表現は text 型と同一であり、戻り値は PG_RETURN_TEXT_P を通じて返されます。

アルゴリズムまたは振る舞い

1. 入力されたC文字列を cstring_to_text で text 型に変換します。 2. makeJsonLexContext を使用して字句解析コンテキストを初期化します。 3. pg_parse_json_or_errsave を呼び出し、JSONの構文妥当性を検証します。 4. 検証が成功した場合、変換済みの text データを返します。失敗した場合は PG_RETURN_NULL() を実行します。

呼び出し元と呼び出し先

  • **呼び出し元**: SQLインターフェースを介して、json 型へのキャストや入力時に呼び出されます。
  • **呼び出し先**:
  • cstring_to_text
  • makeJsonLexContext
  • pg_parse_json_or_errsave

関連データ構造

  • JsonLexContext: JSONの字句解析および構文解析の状態を保持する構造体。

失敗モードと境界条件

  • 入力文字列が不正なJSON構文である場合、pg_parse_json_or_errsave がエラーを検出し、関数は NULL を返します。

保守担当者向けメモ

  • 本関数は json 型の入力におけるゲートキーパーです。検証ロジックを変更する場合は、pg_parse_json_or_errsave の仕様と整合性を保つ必要があります。
  • 内部表現が text 型と同一であるため、メモリ管理はPostgreSQLの text 型のライフサイクルに従います。
