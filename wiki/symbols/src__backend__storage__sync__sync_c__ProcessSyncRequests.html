ProcessSyncRequests

定義

  • 種別: function
  • 定義箇所: src/backend/storage/sync/sync.c:285

目的

蓄積されたファイル同期（fsync）要求をハッシュテーブルから読み出し、物理ディスクへの書き込みを確実に実行する。チェックポイント処理の一部として機能する。

シグネチャまたは形状

void ProcessSyncRequests(void)

入力と出力

  • **入力**: グローバル変数 pendingOps（fsync 要求を保持するハッシュテーブル）および sync_cycle_ctr（サイクル管理用カウンタ）。
  • **出力**: なし（副作用としてディスク同期を実行し、完了した要求を pendingOps から削除する）。

前提条件と不変条件

  • pendingOps が初期化されていること（未初期化時は ERROR を送出）。
  • チェックポイント処理中にのみ呼び出されること。
  • 以前の呼び出しが失敗していた場合、sync_cycle_ctr を用いて古いエントリの整合性を強制的に修正する。

アルゴリズムまたは振る舞い

1. **要求の吸収**: AbsorbSyncRequests() を呼び出し、最新の fsync 要求をハッシュテーブルに取り込む。 2. **サイクル管理**: sync_cycle_ctr をインクリメントし、今回処理すべき要求と次回以降に回すべき要求を区別する。 3. **ハッシュ走査**: pendingOps を走査し、cycle_ctr が現在のサイクルと一致しない（＝古い）エントリのみを処理対象とする。 4. **同期実行**: syncsw テーブル経由で各ハンドラの sync_syncfiletag を呼び出す。

  • enableFsync が無効な場合はスキップ。
  • 実行中に AbsorbSyncRequests() を定期的に呼び出し、キューのオーバーフローを防止する。
  • ファイルが存在しない場合（ENOENT 等）は、AbsorbSyncRequests() を経由してキャンセル要求が届いていないか確認し、リトライまたはエラー処理を行う。

    • 5. **クリーンアップ**: 処理完了したエントリを pendingOps から削除する。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/access/transam/xlog.c（チェックポイント処理時）。
  • **呼び出し先**: AbsorbSyncRequests(), syncsw（各ハンドラの同期関数）, hash_seq_search(), hash_search(), elog/ereport。

関連データ構造

  • pendingOps: fsync 要求を管理するハッシュテーブル。
  • PendingFsyncEntry: ハッシュテーブル内の各エントリ。
  • syncsw: 同期処理を抽象化したハンドラテーブル。
  • sync_cycle_ctr: 処理の境界を管理するカウンタ。

失敗モードと境界条件

  • **同期失敗**: ファイルアクセスエラーが発生した場合、data_sync_elevel に基づきエラーを報告する。ただし、ファイルが削除済み（FILE_POSSIBLY_DELETED）の場合はリトライを試みる。
  • **チェックポイントの無限ループ防止**: サイクルカウンタにより、処理中に新しく追加された要求を次回の実行へ遅延させる。
  • **メモリ不足**: pendingOpsCxt を使用して、クリティカルセクション内でのメモリ確保を許容する。

保守担当者向けメモ

  • sync_in_progress フラグは、前回の実行が中断された場合に古いエントリの cycle_ctr を更新するために使用される。
  • FSYNCS_PER_ABSORB ごとに要求の吸収を行うことで、長時間の同期処理中もキューの肥大化を抑制している。
