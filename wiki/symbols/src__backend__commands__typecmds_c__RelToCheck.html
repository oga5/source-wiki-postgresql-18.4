RelToCheck

定義

  • 種別: struct
  • 定義箇所: src/backend/commands/typecmds.c:79

目的

ドメイン制約の検証時に影響を受けるリレーション（テーブル）と、その中の特定の属性（列）を追跡するために使用される構造体です。

シグネチャまたは形状

typedef struct
{
	Relation	rel;			/* 開かれ、ロックされたリレーション */
	int			natts;			/* 注目する属性の数 */
	int		   *atts;			/* 属性番号の配列 */
} RelToCheck;

入力と出力

  • 本構造体は、palloc を用いて動的に割り当てられます。
  • atts 配列は、RelationGetNumberOfAttributes(rel) に基づく十分な長さで確保される必要があります。

前提条件と不変条件

  • rel フィールドは、既にオープンかつロックされているリレーションを指している必要があります。
  • atts 配列には、検証対象となる属性番号が格納されます。

アルゴリズムまたは振る舞い

  • ドメイン制約の変更や検証プロセスにおいて、影響を受けるリレーションのリストを構築する際に利用されます。
  • 複数の属性が同一リレーションに属する場合、それらを一つの RelToCheck エントリに集約して管理することが可能です。

呼び出し元と呼び出し先

  • 呼び出し元: src/backend/commands/typecmds.c 内のドメイン制約処理に関連する関数群（lfirst を介してリストから取得されるケースが確認されています）。
  • 呼び出し先: 本構造体自体はデータ保持用であり、直接的な関数呼び出しは行いません。

関連データ構造

  • Relation: access/rel.h で定義されるリレーション記述子。

失敗モードと境界条件

  • メモリ不足: palloc による割り当て失敗の可能性があります。
  • 属性番号の不整合: natts と atts 配列の長さがリレーションの定義と一致しない場合、後続の処理で不正なメモリ参照が発生する可能性があります。

保守担当者向けメモ

  • atts 配列の動的確保と解放のライフサイクルが、リスト管理と整合していることを確認してください。
  • 属性の重複排除や集約ロジックに変更を加える際は、src/backend/commands/typecmds.c 内のリスト構築箇所を精査してください。
