load_external_function

定義

  • 種別: function
  • 定義箇所: src/backend/utils/fmgr/dfmgr.c:94

目的

指定された共有ライブラリから特定の関数シンボルを検索し、必要に応じてライブラリをプロセス空間へロードします。

シグネチャまたは形状

void *load_external_function(const char *filename, const char *funcname, bool signalNotFound, void **filehandle);

入力と出力

  • **入力**:
  • filename: ライブラリのファイル名（$libdir/ プレフィックスを含む）。
  • funcname: 検索対象の関数名。
  • signalNotFound: 関数が見つからない場合にエラーを報告するかどうかを指定するフラグ。
  • filehandle: ライブラリのハンドルを格納するためのポインタ（不要な場合は NULL）。
  • **出力**:
  • 成功時: 検索した関数のアドレス（void *）。
  • 失敗時: signalNotFound が true の場合はエラー終了、false の場合は NULL を返却。

前提条件と不変条件

  • $libdir/ で始まるパスのうち、単純な名前（サブディレクトリを含まないもの）は、ライブラリ検索パスを利用するためにプレフィックスが除去されます。
  • 共有ライブラリは、ロード時に PG_MODULE_MAGIC による ABI 互換性チェックを通過する必要があります。

アルゴリズムまたは振る舞い

1. **パス正規化**: filename が $libdir/ で始まり、かつネストされたパスでない場合、プレフィックスを除去します。 2. **パス展開**: expand_dynamic_library_name を呼び出し、ライブラリの絶対パスを取得します。 3. **ライブラリロード**: internal_load_library を呼び出し、ライブラリが未ロードであれば dlopen を実行してロードします。 4. **ハンドル保存**: filehandle が指定されていれば、ロードされたライブラリのハンドルを格納します。 5. **シンボル検索**: dlsym を使用してライブラリ内から funcname を検索します。 6. **エラーハンドリング**: 関数が見つからず signalNotFound が true の場合、ERRCODE_UNDEFINED_FUNCTION でエラーを報告します。 7. **クリーンアップ**: 展開されたパス文字列を解放し、関数ポインタを返します。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • contrib/hstore_plperl/hstore_plperl.c
  • contrib/hstore_plpython/hstore_plpython.c
  • **呼び出し先**:
  • expand_dynamic_library_name
  • internal_load_library
  • dlsym (POSIX)
  • ereport / pfree

関連データ構造

  • DynamicFileList: ロード済みライブラリを管理する連結リスト。

失敗モードと境界条件

  • **関数未検出**: signalNotFound が true の場合、ereport によりトランザクションが中断されます。
  • **ライブラリロード失敗**: internal_load_library 内で ABI 不整合やファイル不在が発生した場合、エラーが発生します。

保守担当者向けメモ

  • 本関数は一度ロードされたライブラリをアンロードする機能を持たないため、メモリ管理はプロセス終了まで継続されます。
  • Windows 環境では dlsym は OS 固有の API に抽象化されています。
