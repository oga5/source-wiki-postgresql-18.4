Tuplesortstate

定義

  • **種別**: struct
  • **定義箇所**: src/backend/utils/sort/tuplesort.c:185

目的

PostgreSQLにおける汎用的なタプルソート操作の実行状態を保持するメイン構造体です。メモリ内ソートから外部マージソートへの移行、並列ソートの調整、および一時ファイル（論理テープ）の管理を統括します。

シグネチャまたは形状

Tuplesortstate は、ソートの進行状況（TupSortStatus）、メモリ使用量制限（work_mem 関連）、ソート対象タプルの配列（memtuples）、および外部ソート用の論理テープセット（LogicalTapeSet）など、ソートに必要な全コンテキストを保持します。

入力と出力

  • **入力**: tuplesort_begin_common を通じて初期化され、tuplesort_puttuple_common によりタプルが投入されます。
  • **出力**: tuplesort_gettuple_common を通じて、メモリまたは一時ファイルからソート済みのタプルが順次返却されます。

前提条件と不変条件

  • work_mem に基づくメモリ制限を厳密に管理します。
  • TSS_INITIAL（初期状態）、TSS_SORTEDINMEM（メモリ内ソート完了）、TSS_BUILDRUNS（ラン構築中）、TSS_FINALMERGE（マージ中）、TSS_SORTEDONTAPE（テープ上ソート完了）の各状態遷移に従います。
  • スラブアロケータ使用時は、USEMEM/LACKMEM によるメモリ追跡は無効化されます。

アルゴリズムまたは振る舞い

1. **メモリ内ソート**: memtuples 配列にタプルを蓄積し、クイックソート等で処理します。 2. **外部ソートへの移行**: メモリ制限を超過すると、dumptuples を呼び出して一時ファイルへ書き出し、TSS_BUILDRUNS 状態へ移行します。 3. **マージ**: mergeruns により k-way マージアルゴリズムを実行し、最終的なソート済み順序を生成します。 4. **並列処理**: Sharedsort を介してリーダーとワーカー間でテープセットを共有し、並列マージを調整します。

呼び出し元と呼び出し先

  • **呼び出し元**: brin.c, gininsert.c, gistbuild.c, hashsort.c 等のインデックス構築処理から参照されています。
  • **呼び出し先**: logtape.c（テープ管理）、tuplesortvariants.c（比較・読み書き）、lib/sort_template.h（ソートロジック）を利用します。

関連データ構造

  • TupSortStatus: ソートの進行状況を示す列挙型。
  • SortTuple: ソート対象のタプルを保持する構造体。
  • LogicalTapeSet: 一時ファイル上の論理テープ管理オブジェクト。
  • Sharedsort: 並列ソート用の共有メモリ状態。
  • SlabSlot: マージフェーズで使用されるスラブアロケータの管理単位。

失敗モードと境界条件

  • **メモリ不足**: work_mem 制限により、外部ソートへの移行が必須となります。
  • **境界条件**: bounded ソート（LIMIT 句等）が指定されている場合、ヒープソートを用いてメモリ消費を最小限に抑えます。
  • **EOF**: eof_reached フラグにより、カーソル操作時の終端到達を管理します。

保守担当者向けメモ

  • work_mem の管理は LACKMEM / USEMEM マクロに依存しているため、メモリ追跡ロジックの変更には注意が必要です。
  • 並列ソートのデバッグ時は、LEADER(state) や WORKER(state) マクロを使用してプロセスの役割を特定してください。
  • trace_sort GUC を有効にすると、ソートの進行状況やリソース使用量がログに出力されます。
