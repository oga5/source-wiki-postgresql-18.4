fill_val

定義

  • 種別: static inline function
  • 定義箇所: src/backend/access/common/heaptuple.c:274

目的

タプル構築時に、個々の属性値をメモリ上の指定位置へ書き込み、必要に応じてアライメント調整、圧縮、およびNULLビットマップの更新を行う。

シグネチャまたは形状

static inline void fill_val(CompactAttribute *att, bits8 **bit, int *bitmask, char **dataP, uint16 *infomask, Datum datum, bool isnull)

入力と出力

  • **入力**:
  • att: 属性のメタデータ（型、長さ、アライメント情報）。
  • bit: NULLビットマップの現在位置（NULL値を含まない場合は NULL）。
  • bitmask: 現在のビット位置を追跡するマスク。
  • dataP: 書き込み先のメモリポインタ（更新される）。
  • infomask: タプルの状態フラグ（HEAP_HASNULL, HEAP_HASVARWIDTH, HEAP_HASEXTERNAL が設定される）。
  • datum: 書き込む属性値。
  • isnull: 値がNULLかどうかを示すフラグ。
  • **出力**: なし（dataP が指すメモリへの書き込みと、infomask 等のポインタ経由の更新）。

前提条件と不変条件

  • dataP は書き込み可能なメモリ領域を指していること。
  • att のアライメント情報（attalignby）に基づき、CPU境界に合わせた書き込みが行われる。
  • attlen == -2（cstring）の場合、attalignby は sizeof(char) であること。

アルゴリズムまたは振る舞い

1. **NULL処理**: bit が有効な場合、NULLビットマップを更新する。isnull が真であれば infomask に HEAP_HASNULL を立てて即座に終了する。 2. **データ書き込み**: 属性の型に応じて処理を分岐する。

  • **Pass-by-value**: att_nominal_alignby でアライメント調整後、値をコピー。
  • **Varlena (attlen == -1)**: 外部データ（TOAST）の展開、ショート形式への変換、または通常の4バイトヘッダー形式でのコピーを行う。必要に応じて infomask にフラグを付与。
  • **Cstring (attlen == -2)**: アライメントなしでコピー。
  • **固定長Pass-by-reference**: att_nominal_alignby でアライメント調整後、値をコピー。

    • 3. **ポインタ更新**: 書き込んだデータ長分だけ dataP を進める。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/access/common/heaptuple.c 内のタプル構築処理（heap_fill_tuple 等）。
  • **呼び出し先**: att_nominal_alignby（マクロ）、store_att_byval、EOH_flatten_into、memcpy 等のメモリ操作関数。

関連データ構造

  • CompactAttribute: 属性の物理レイアウト情報。
  • Datum: PostgreSQLの汎用データ表現。
  • ExpandedObjectHeader: 展開されたTOASTオブジェクトのヘッダー。

失敗モードと境界条件

  • attlen が不正な場合、アサーションエラーが発生する可能性がある。
  • メモリ領域が不足している場合、memcpy 等によりメモリ破壊が発生する（呼び出し元で十分なバッファを確保していることが前提）。

保守担当者向けメモ

  • att_nominal_alignby をポインタ値に対して直接使用している点はハックであるため、変更時は注意が必要。
  • Varlenaの圧縮・変換ロジックは src/backend/access/heap/heaptoast.c の処理と密接に関連している。
