populate_joinrel_with_paths

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/path/joinrels.c:884

目的

結合対象となる2つのリレーション（rel1, rel2）から結合リレーション（joinrel）を構築し、利用可能なパスを生成・登録する。また、結合条件の評価に基づき、リレーションが空（dummy）であると判断される場合の最適化を行う。

シグネチャまたは形状

static void populate_joinrel_with_paths(PlannerInfo *root, RelOptInfo *rel1,
                                        RelOptInfo *rel2, RelOptInfo *joinrel,
                                        SpecialJoinInfo *sjinfo, List *restrictlist)

入力と出力

  • **入力**:
  • root: クエリ全体のプランナ情報。
  • rel1, rel2: 結合対象の入力リレーション。
  • joinrel: 構築対象の結合リレーション。
  • sjinfo: 結合の制約情報（SpecialJoinInfo）。
  • restrictlist: 結合条件のリスト。
  • **出力**: なし（joinrel 構造体内部のパスリストを更新する副作用を持つ）。

前提条件と不変条件

  • sjinfo->jointype に基づいて処理が分岐する。
  • 結合条件が定数的に偽（constant-false）である場合、該当リレーションまたは結合リレーション自体を dummy としてマークする。
  • FULL JOIN の場合、マージ結合またはハッシュ結合が不可能な条件であればエラーを発生させる。

アルゴリズムまたは振る舞い

1. sjinfo->jointype に応じたスイッチ処理を実行。 2. 各結合タイプ（INNER, LEFT, FULL, SEMI, ANTI）ごとに、入力リレーションの dummy 判定および restriction_is_constant_false による最適化を行う。 3. add_paths_to_joinrel を呼び出し、rel1 を外側、rel2 を内側とするパス（およびその逆）を生成・追加する。 4. SEMI JOIN の場合、必要に応じて create_unique_path を用いた一意化パスの検討を行う。 5. 最後に try_partitionwise_join を呼び出し、パーティション単位の結合最適化を試みる。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/optimizer/path/joinrels.c 内の結合探索プロセス（join_search_one_level 等から間接的に呼び出される）。
  • **呼び出し先**:
  • is_dummy_rel, mark_dummy_rel
  • restriction_is_constant_false
  • add_paths_to_joinrel
  • create_unique_path
  • try_partitionwise_join
  • ereport (エラー発生時)

関連データ構造

  • RelOptInfo: 結合対象および結果のリレーション情報。
  • SpecialJoinInfo: 結合タイプや制約条件を保持。
  • PlannerInfo: プランナ全体のコンテキスト。

失敗モードと境界条件

  • FULL JOIN において、マージ結合もハッシュ結合も適用できない結合条件が指定された場合、ERRCODE_FEATURE_NOT_SUPPORTED でエラー終了する。
  • 未知の jointype が渡された場合、elog によりエラー終了する。

保守担当者向けメモ

  • 本関数は結合パス生成の最終的な振り分けを行うため、新しい結合アルゴリズムや最適化ルールを追加する際は、各 jointype のケース文を確認すること。
  • FULL JOIN のエラー処理は、プランナがこれ以上探索を継続できない制約に基づいている。
