SortSupportData

定義

  • 種別: struct
  • 定義箇所: src/include/utils/sortsupport.h:60

目的

PostgreSQL のソート処理において、データ型固有の比較ロジックや省略キー（Abbreviated Key）変換ルーチンを保持し、低オーバーヘッドな比較メカニズムを提供するための状態管理構造体です。

シグネチャまたは形状

typedef struct SortSupportData {
    MemoryContext ssup_cxt;
    Oid           ssup_collation;
    bool          ssup_reverse;
    bool          ssup_nulls_first;
    AttrNumber    ssup_attno;
    void         *ssup_extra;
    int         (*comparator) (Datum x, Datum y, SortSupport ssup);
    bool          abbreviate;
    Datum       (*abbrev_converter) (Datum original, SortSupport ssup);
    bool        (*abbrev_abort) (int memtupcount, SortSupport ssup);
    int         (*abbrev_full_comparator) (Datum x, Datum y, SortSupport ssup);
} SortSupportData;

入力と出力

  • **入力**: BTSORTSUPPORT_PROC プロシージャを介して初期化されます。ssup_cxt や ssup_collation は初期化時に設定され、以降変更不可です。
  • **出力**: comparator 関数ポインタは、比較結果（<0, 0, >0）を返します。省略キー比較が不確定（0）を返した場合、abbrev_full_comparator が最終的な判定を行います。

前提条件と不変条件

  • ssup_extra に格納するデータは、必ず ssup_cxt コンテキスト内に割り当てる必要があります。
  • comparator は NULL を返さず、入力される x, y も NULL ではないことが保証されます。
  • 省略キーを使用する場合、abbrev_converter と abbrev_full_comparator の設定が必須です。

アルゴリズムまたは振る舞い

1. **初期化**: PrepareSortSupport 系関数により、対象の型に応じた比較関数が登録されます。 2. **比較**: ApplySortComparator を通じて比較が実行されます。省略キーが有効な場合、まず高速な近似比較を行い、結果が不確定な場合にのみ abbrev_full_comparator を呼び出します。 3. **最適化**: abbrev_abort を用いて、省略キーの重複が多い場合に変換コストを回避する戦略をとることが可能です。

呼び出し元と呼び出し先

  • **呼び出し元**: tuplesort.c（ソートエンジン）、nodeIndexscan.c、nodeMergejoin.c、nodeSetOp.c 等の実行ノードや、analyze.c、nbtsort.c 等のユーティリティから参照されます。
  • **呼び出し先**: BTSORTSUPPORT_PROC で定義された型固有の関数、および ApplySortAbbrevFullComparator 等のインライン関数。

関連データ構造

  • SortSupport: SortSupportData へのポインタ型。
  • MemoryContext: ssup_cxt で使用されるメモリ管理コンテキスト。

失敗モードと境界条件

  • 省略キーの比較結果が 0 の場合、それは等価を意味せず「判定不能」を意味するため、必ず abbrev_full_comparator によるフォールバックが必要です。
  • abbrev_converter が設定されていない場合、省略キー最適化は無効となります。

保守担当者向けメモ

  • ssup_reverse および ssup_nulls_first は初期化後も変更可能です。
  • comparator は、コアコードの判断により、省略キー用から完全比較用へ動的に切り替えられる場合があります。
  • CPU キャッシュ効率を最大化するため、省略キーのエンコーディング設計には注意が必要です。
