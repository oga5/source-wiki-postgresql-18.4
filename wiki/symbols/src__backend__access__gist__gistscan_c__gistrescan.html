gistrescan

定義

  • 種別: function
  • 定義箇所: src/backend/access/gist/gistscan.c:126

目的

GiSTインデックススキャンの状態をリセットし、新しい検索条件（ScanKey）や順序付け（OrderBy）を適用してスキャンを再開する。

シグネチャまたは形状

void gistrescan(IndexScanDesc scan, ScanKey key, int nkeys, ScanKey orderbys, int norderbys)

入力と出力

  • **入力**:
  • scan: スキャン状態を保持する IndexScanDesc 構造体。
  • key: 新しい検索条件（ScanKey）。NULL の場合は更新しない。
  • nkeys: 検索条件の数（無視される）。
  • orderbys: 新しい順序付け条件（ScanKey）。NULL の場合は更新しない。
  • norderbys: 順序付け条件の数（無視される）。
  • **出力**: なし（副作用として scan 内部の状態が更新される）。

前提条件と不変条件

  • scan->opaque は GISTScanOpaque 型としてキャスト可能であること。
  • インデックススキャンの初回実行時、so->queueCxt は so->giststate->scanCxt と一致していること。

アルゴリズムまたは振る舞い

1. **検索キューの管理**:

  • 初回呼び出し時は既存の scanCxt を使用。2回目以降は専用の queueCxt を作成またはリセットし、メモリ効率を最適化する。

    • 2. **インデックス専用スキャンの初期化**:

  • xs_want_itup が真かつ未初期化の場合、タプル記述子と専用メモリコンテキストを作成する。

    • 3. **ヒープの再構築**:

  • pairingheap_allocate を用いて、距離計算関数 pairingheap_GISTSearchItem_cmp を指定した新しい検索キューを生成する。

    • 4. **スキャンキーの更新**:

  • key が提供された場合、fmgr_info_copy を使用して Consistent サポート関数をバインドする。既存の fn_extra は再スキャン後も保持される。

    • 5. **順序付けキーの更新**:

  • orderbys が提供された場合、Distance サポート関数をバインドする準備を行う。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/access/gist/gist.c 内の amrescan インターフェース経由で呼び出される。
  • **呼び出し先**: AllocSetContextCreate, MemoryContextReset, MemoryContextSwitchTo, pairingheap_allocate, fmgr_info_copy, CreateTemplateTupleDesc 等のメモリ管理およびサポート関数。

関連データ構造

  • GISTScanOpaque: スキャン中の検索キューやメモリコンテキストを保持する内部構造体。
  • IndexScanDesc: PostgreSQLの標準的なインデックススキャン記述子。
  • pairingheap: 検索順序を制御するための優先度付きキュー。

失敗モードと境界条件

  • key が NULL の場合、検索条件の更新はスキップされる。
  • sk_flags に SK_ISNULL が含まれ、かつ SK_SEARCHNULL 等が指定されていない場合、so->qual_ok が false に設定され、検索結果が空になることが示唆される。
  • 複数回の再スキャンが発生する場合、古い queueCxt はリセットされ、メモリの断片化を抑制する。

保守担当者向けメモ

  • nkeys および norderbys 引数は無視されるため、実際のデータ数は scan 構造体から取得すること。
  • fn_extra の保持処理は、Consistent 関数がキャッシュを利用している場合に重要となる。
  • メモリコンテキストの切り替え（scanCxt と queueCxt）は、スキャンを繰り返す際のオーバーヘッドを最小化するための設計である。
