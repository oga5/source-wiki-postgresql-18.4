BlockRefTableSerializedEntry

定義

  • 種別: struct
  • 定義箇所: src/common/blkreftable.c:155

目的

ブロック参照テーブルをディスクへシリアライズする際、各リレーションフォークのメタデータを保持するための転送用構造体です。

シグネチャまたは形状

typedef struct BlockRefTableSerializedEntry
{
    RelFileLocator rlocator;  /* リレーションの識別子 */
    ForkNumber    forknum;    /* フォーク番号 */
    BlockNumber   limit_block;/* リレーションの切り詰め境界 */
    uint32        nchunks;    /* 保持するチャンク数 */
} BlockRefTableSerializedEntry;

入力と出力

  • 本構造体は、メモリ上の BlockRefTable から情報を抽出して生成されます。
  • シリアライズ処理において、ファイルへの書き込み順序を決定するためのソート対象として使用されます。

前提条件と不変条件

  • シリアライズ時は tablespace -> database -> relfilenumber -> forknumber の順序でソートされる必要があります。
  • nchunks は、当該リレーションフォークに関連付けられた変更済みブロック群を管理するチャンクの総数を示します。

アルゴリズムまたは振る舞い

1. BlockRefTable 内の全エントリを本構造体の配列として抽出します。 2. qsort を用いて、定義された順序に従いソートを行います。 3. ソートされた順序で、マジックナンバー、本構造体（メタデータ）、および実際のチャンクデータの順にバイナリ出力されます。

呼び出し元と呼び出し先

  • 呼び出し元: src/common/blkreftable.c 内のシリアライズ処理（WriteBlockRefTable 等）から参照されます。
  • 呼び出し先: 構造体自体はデータ保持用であり、関数呼び出しは行いません。

関連データ構造

  • BlockRefTable: メモリ上のメイン構造体。
  • RelFileLocator: リレーションを特定するための識別子。
  • BlockRefTableChunk: 実際のブロック変更状態を保持する構造体。

失敗モードと境界条件

  • メモリ不足時: palloc_array による配列確保時に失敗する可能性があります。
  • 整合性チェック: 読み込み時に memcmp を用いてゼロ初期化されたエントリとの比較が行われる場合があります。

保守担当者向けメモ

  • 本構造体はディスク上のバイナリ形式と密接に関連しているため、フィールドの追加や変更を行う際は、シリアライズ/デシリアライズ処理（WriteBlockRefTable および BlockRefTableReader）との整合性を維持してください。
