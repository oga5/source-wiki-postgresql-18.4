heap_page_prune_and_freeze

定義

  • 種別: function
  • 定義箇所: src/backend/access/heap/pruneheap.c:349

目的

ヒープページ内の不要なタプル（デッドタプル）の削除（剪定）および古いタプルの凍結（Freezing）を実行し、ページ内の断片化解消と領域再利用を行う。

シグネチャまたは形状

void heap_page_prune_and_freeze(Relation relation, Buffer buffer,
                                GlobalVisState *vistest,
                                int options,
                                struct VacuumCutoffs *cutoffs,
                                PruneFreezeResult *presult,
                                PruneReason reason,
                                OffsetNumber *off_loc,
                                TransactionId *new_relfrozen_xid,
                                MultiXactId *new_relmin_mxid)

入力と出力

  • **入力**:
  • relation, buffer: 対象のリレーションとバッファ。
  • vistest: 可視性判定用状態。
  • options: 処理オプション（HEAP_PAGE_PRUNE_MARK_UNUSED_NOW, HEAP_PAGE_PRUNE_FREEZE 等）。
  • cutoffs: 凍結判定用のトランザクションカットオフ情報。
  • **出力**:
  • presult: 削除されたオフセット等の結果情報。
  • off_loc: 処理中のオフセット番号（エラー表示用）。
  • new_relfrozen_xid, new_relmin_mxid: 更新された凍結XIDおよびマルチXactID。

前提条件と不変条件

  • 対象ページに対する排他ロックを保持していること。
  • PruneState 構造体を用いて、ページ変更前に削除・凍結対象のリストを作成し、変更をクリティカルセクション内で適用する。
  • ページ内のタプル可視性判定（HTSV）は、一貫性を保つため各タプルにつき一度だけ行う。

アルゴリズムまたは振る舞い

1. PruneState を初期化し、オプションに基づき凍結の可否や可視性判定の基準を設定する。 2. ページ内のタプルをオフセットの大きい順（物理的には逆順）に走査し、各タプルの可視性を判定する。 3. ItemId の状態（未使用、デッド、リダイレクト等）に応じて、削除対象や凍結対象を PruneState に記録する。 4. 走査完了後、クリティカルセクション内で heap_page_prune_execute を呼び出し、物理的なページ変更を適用する。 5. 変更内容を WAL に記録する。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/access/heap/heapam_handler.c:1057 および src/backend/access/heap/pruneheap.c:263。
  • **呼び出し先**: 内部で heap_prune_record_... 系の関数群や heap_prune_chain を呼び出す。詳細は提供されたソース excerpt には網羅されていない。

関連データ構造

  • PruneState: 剪定処理中の作業状態を保持する構造体。
  • PruneFreezeResult: 処理結果を格納する構造体。
  • Buffer, Relation: PostgreSQL の標準的なバッファおよびリレーション管理構造。

失敗モードと境界条件

  • ItemId が使用されていない場合やデッド状態の場合のスキップ処理。
  • 凍結オプションが有効な場合、new_relfrozen_xid 等のポインタが必須（アサーションで検証）。
  • ページ内のタプル走査中に発生したエラーは、off_loc を通じて特定可能。

保守担当者向けメモ

  • ページ単位のロックを伴うため、パフォーマンスへの影響に注意すること。
  • HOT チェーンの整合性は page_verify_redirects 等のアサーションで保護されている。
  • ページ全体の凍結判定は、可視性ビット（VM）の更新と密接に関連している。
