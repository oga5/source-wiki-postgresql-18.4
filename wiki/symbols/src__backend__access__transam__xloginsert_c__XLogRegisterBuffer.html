XLogRegisterBuffer

定義

  • 種別: function
  • 定義箇所: src/backend/access/transam/xloginsert.c:241

目的

WALレコード構築プロセスにおいて、変更対象となるバッファを登録し、そのメタデータ（リレーショナルファイルロケータ、フォーク番号、ブロック番号）を一時的な作業領域に保持します。

シグネチャまたは形状

void XLogRegisterBuffer(uint8 block_id, Buffer buffer, uint8 flags)

入力と出力

  • **入力**:
  • block_id: 登録するブロックの識別子（0〜max_registered_buffers未満）。
  • buffer: 変更対象のバッファハンドル。
  • flags: 登録オプション（REGBUF_FORCE_IMAGE, REGBUF_NO_IMAGE, REGBUF_NO_CHANGE, REGBUF_WILL_INIT, REGBUF_STANDARD など）。
  • **出力**: なし（副作用として内部状態 registered_buffers を更新）。

前提条件と不変条件

  • XLogBeginInsert() が既に呼び出されていること。
  • REGBUF_NO_CHANGE フラグが指定されていない限り、対象の buffer は排他ロック（Exclusive Lock）されており、かつダーティ状態であること。
  • REGBUF_FORCE_IMAGE と REGBUF_NO_IMAGE を同時に指定することは不可。
  • 同一のバッファを異なる block_id で二重登録していないこと。

アルゴリズムまたは振る舞い

1. **検証**: begininsert_called フラグおよびバッファのロック状態をアサートで確認。 2. **領域確保**: 指定された block_id が現在の最大値を超える場合、max_registered_block_id を更新。上限を超えた場合はエラーをスロー。 3. **メタデータ抽出**: BufferGetTag を使用してバッファから物理的な位置情報を取得し、registered_buffers[block_id] に格納。 4. **初期化**: 登録されたバッファのフラグ、データチェーンのヘッドポインタ、データ長を初期化。 5. **重複チェック**: デバッグビルド時、既に登録済みの他のブロックと物理的に同一のページではないかを確認。 6. **完了**: in_use フラグを true に設定。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/access/brin/brin.c
  • src/backend/access/brin/brin_pageops.c
  • src/backend/access/brin/brin_revmap.c
  • src/backend/access/gin/ginbtree.c

    • (上記はツールによる直接的な参照の証拠に基づく)

  • **呼び出し先**:
  • BufferGetTag, BufferGetPage, BufferIsExclusiveLocked, BufferIsDirty, RelFileLocatorEquals などのバッファ管理 API。

関連データ構造

  • registered_buffer: 登録された各ブロックのメタデータ（rlocator, forkno, block, page, flags 等）を保持する構造体。
  • registered_buffers: xloginsert.c 内で管理される registered_buffer の配列。

失敗モードと境界条件

  • block_id >= max_registered_buffers: 登録可能な最大数を超えた場合に elog(ERROR) を発生させる。
  • ロック未取得: USE_ASSERT_CHECKING が有効な場合、排他ロックされていないバッファを渡すとアサート失敗により停止する。

保守担当者向けメモ

  • 本関数は WAL 構築の「登録フェーズ」の要です。XLogRecordAssemble がこの情報を元に最終的なレコードを生成するため、ここで設定されるメタデータが正確である必要があります。
  • REGBUF_NO_CHANGE を使用する際は、WAL への書き込みが意図通りに行われるか、access/transam/README のルールと照らし合わせて慎重に判断してください。
