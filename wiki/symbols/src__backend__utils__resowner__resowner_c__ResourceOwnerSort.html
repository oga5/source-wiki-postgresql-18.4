ResourceOwnerSort

定義

  • 種別: function
  • 定義箇所: src/backend/utils/resowner/resowner.c:291

目的

リソースオーナーが保持するリソース要素を、解放優先順位に基づいてソートする。

シグネチャまたは形状

static void ResourceOwnerSort(ResourceOwner owner)

入力と出力

  • **入力**: owner (ResourceOwner 型) - ソート対象のリソースオーナー構造体。
  • **出力**: なし（副作用として owner 内部の配列またはハッシュテーブルがソートされる）。

前提条件と不変条件

  • owner が有効な ResourceOwnerData を指していること。
  • ハッシュテーブルが存在する場合、RESOWNER_HASH_MAX_ITEMS の制約により、固定長配列からハッシュテーブルへ全要素を移動可能な十分な空き容量があること。

アルゴリズムまたは振る舞い

1. **データ集約**:

  • ハッシュテーブルが空の場合、固定長配列 (owner->arr) をソート対象とする。
  • ハッシュテーブルが空でない場合、ハッシュテーブル内の有効な要素を先頭に詰め、固定長配列内の全要素をハッシュテーブルへ移動する。その後、固定長配列を空 (narr = 0) にする。

    • 2. **ソート実行**:

  • 集約された要素群に対し、qsort と比較関数 resource_priority_cmp を用いて、リソースの解放優先順位に従って並び替える。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/utils/resowner/resowner.c:717 (リソース解放処理の一部として呼び出される)。
  • **呼び出し先**: qsort (標準ライブラリ)、resource_priority_cmp (比較関数)。

関連データ構造

  • ResourceOwnerData: リソースオーナーの内部状態を保持する構造体。
  • ResourceElem: リソースを識別する要素構造体。

失敗モードと境界条件

  • owner->nhash == 0 の場合、ハッシュテーブルの圧縮・移動処理はスキップされる。
  • Assert により、ハッシュテーブルへの移動時に容量不足が発生していないか検証される。

保守担当者向けメモ

  • 本関数はリソース解放の準備段階で必須となる。
  • ハッシュテーブルへの移動処理は、owner->hash のメモリレイアウトを再構成するため、呼び出し後の状態変化に注意すること。
