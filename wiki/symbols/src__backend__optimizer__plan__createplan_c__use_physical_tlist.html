use_physical_tlist

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/plan/createplan.c:864

目的

プランナが生成した Path ノードから Plan ノードへ変換する際、物理的なタプル構造をそのまま利用可能か（追加の射影処理を省略可能か）を判定する。

シグネチャまたは形状

static bool use_physical_tlist(PlannerInfo *root, Path *path, int flags)

入力と出力

  • **入力**:
  • root: プランナのグローバル情報。
  • path: 評価対象の Path ノード。
  • flags: ターゲットリスト生成の制約を指定するビットフラグ（CP_EXACT_TLIST, CP_SMALL_TLIST, CP_LABEL_TLIST 等）。
  • **出力**:
  • bool: 物理ターゲットリストが使用可能な場合は true、それ以外は false。

前提条件と不変条件

  • flags に CP_EXACT_TLIST または CP_SMALL_TLIST が含まれている場合、常に false を返す。
  • rel->reloptkind が RELOPT_BASEREL である必要がある。
  • CustomPath は対象外。

アルゴリズムまたは振る舞い

1. **RTE種別の検証**: 対象が RTE_RELATION, SUBQUERY, FUNCTION, TABLEFUNC, VALUES, CTE のいずれかであることを確認する。 2. **制約のチェック**:

  • システム列や行全体（whole-row）の Var が要求されていないこと。
  • PlaceHolderInfo が要求されていないこと。
  • BitmapHeapPath のターゲットリストが空でないこと。

    • 3. **インデックススキャンの検証**: T_IndexOnlyScan の場合、すべてのインデックス列が取得可能（canreturn）であることを確認する。 4. **ラベル付けの検証**: CP_LABEL_TLIST が指定されている場合、ソート/グループ列が単純な Var であり、かつ重複がないことを確認する。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/optimizer/plan/createplan.c 内の各プラン生成ロジック（create_scan_plan 等）から呼び出される。
  • **呼び出し先**: bms_is_empty, bms_nonempty_difference, bms_is_subset, IsA, lfirst などのノード操作・ビットマップ操作関数。

関連データ構造

  • PlannerInfo: プランナの全体状態。
  • Path: 変換元のパス構造体。
  • RelOptInfo: リレーションの最適化情報。
  • PlaceHolderInfo: プレースホルダ式に関する情報。

失敗モードと境界条件

  • 結合（Join）ノードや継承関係にあるリレーションは、物理ターゲットリストを使用できないため false を返す。
  • ソート/グループ列に Var 以外の式が含まれる場合、または重複する Var が含まれる場合は false を返す。

保守担当者向けメモ

  • 本関数は、実行エンジンにおける不要な射影（Projection）を回避するための最適化判断を行う。
  • システム列や PlaceHolder の扱いに関するロジックは setrefs.c での参照解決と密接に関連しており、変更時には注意が必要。
