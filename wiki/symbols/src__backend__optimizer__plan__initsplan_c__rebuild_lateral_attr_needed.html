rebuild_lateral_attr_needed

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/plan/initsplan.c:857

目的

クエリ内の LATERAL 参照に関連する変数依存関係を再構築し、各リレーションで必要とされる属性情報（attr_needed）を最新の状態に更新します。

シグネチャまたは形状

void rebuild_lateral_attr_needed(PlannerInfo *root)

入力と出力

  • **入力**: PlannerInfo *root（クエリ全体のプランナ情報）
  • **出力**: なし（副作用として root 内の各 RelOptInfo の属性必要情報を更新）

前提条件と不変条件

  • root->hasLateralRTEs が真である場合にのみ処理を実行します。
  • find_lateral_references によって事前に brel->lateral_vars が抽出・保存されていることを前提とします。

アルゴリズムまたは振る舞い

1. root->hasLateralRTEs を確認し、偽であれば即座に終了します。 2. root->simple_rel_array を走査し、RELOPT_BASEREL である各リレーション（brel）を特定します。 3. brel->lateral_vars が空（NIL）でないリレーションに対してのみ処理を行います。 4. 各リレーションについて、そのリレーション自身を指すビットマップ（where_needed）を作成します。 5. add_vars_to_attr_needed を呼び出し、lateral_vars に含まれる各変数やプレースホルダ変数が、当該リレーションで必要であることを記録します。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/optimizer/plan/analyzejoins.c:662
  • src/backend/optimizer/plan/analyzejoins.c:2077
  • **呼び出し先**:
  • add_vars_to_attr_needed

関連データ構造

  • PlannerInfo: クエリ全体のプランナ状態を保持。
  • RelOptInfo: 各リレーションの最適化情報を保持。lateral_vars フィールドを使用して依存変数を管理。

失敗モードと境界条件

  • LATERAL 参照を含まないクエリでは、処理はスキップされます。
  • brel が NULL または RELOPT_BASEREL ではない場合、そのリレーションは無視されます。

保守担当者向けメモ

  • 本関数は extract_lateral_references の再実行を避ける設計となっており、事前に抽出された lateral_vars を再利用します。
  • LATERAL 依存関係の追跡は複雑であるため、本関数を修正する際は add_vars_to_attr_needed の仕様と整合性が取れているか確認してください。
