ParsedLex

定義

  • 種別: struct
  • 定義箇所: src/backend/tsearch/ts_parse.c:27

目的

全文検索処理において、解析された語彙（Lexeme）を連結リスト形式で保持するためのデータ構造です。

シグネチャまたは形状

typedef struct ParsedLex
{
    int            type;      /* トークンの型 */
    char          *lemm;      /* 語彙の文字列 */
    int            lenlemm;   /* 文字列の長さ */
    struct ParsedLex *next;   /* リストの次要素へのポインタ */
} ParsedLex;

入力と出力

  • 本構造体はデータ保持用であり、関数ではないため直接的な入出力はありません。
  • palloc を用いてメモリ確保され、LexizeData 構造体内のリスト（head/tail）や一時的な作業用リストとして利用されます。

前提条件と不変条件

  • lemm は lenlemm で指定された長さの文字列を指す必要があります。
  • next はリストの終端では NULL である必要があります。

アルゴリズムまたは振る舞い

  • 辞書処理（LexizeExec）において、トークンを順次処理する際のノードとして機能します。
  • LPLAddTail 関数等により、リストの末尾への追加や、RemoveHead による先頭からの取り出しが行われます。

呼び出し元と呼び出し先

  • 呼び出し元: ts_parse.c 内の LPLAddTail、RemoveHead、setCorrLex 等の関数が本構造体のインスタンスを操作します。
  • 呼び出し先: 本構造体はデータ保持用であり、特定の関数を呼び出すことはありません。

関連データ構造

  • ListParsedLex: ParsedLex のリストを管理する構造体。
  • LexizeData: 辞書処理の状態を保持し、ParsedLex を内部で参照します。

失敗モードと境界条件

  • メモリ不足時: palloc が失敗した場合、PostgreSQL のエラーハンドリングにより処理が中断されます。
  • lenlemm が不正な場合: 文字列操作時にバッファオーバーフロー等のリスクがあるため、呼び出し側での整合性確保が必要です。

保守担当者向けメモ

  • LexizeExec 内で頻繁にメモリ確保・解放が行われるため、メモリリークの発生に注意してください。
  • next ポインタの操作において、リストの循環や不正な参照が発生しないよう、LPLAddTail 等の既存ユーティリティ関数を介した操作を推奨します。
