add_vars_to_attr_needed

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/plan/initsplan.c:364

目的

クエリ内の変数（Var）またはプレースホルダ変数（PlaceHolderVar）が、プランのどの段階で必要とされるかを追跡し、関連するリレーションまたはプレースホルダ情報の必要ビットマップを更新します。

シグネチャまたは形状

void add_vars_to_attr_needed(PlannerInfo *root, List *vars, Relids where_needed)

入力と出力

  • root: プランナのグローバル情報構造体。
  • vars: 追跡対象となる Var または PlaceHolderVar ノードのリスト。
  • where_needed: 変数が必要とされるリレーションの集合（Relids）。
  • 出力: なし（副作用として RelOptInfo または PlaceHolderInfo を更新）。

前提条件と不変条件

  • where_needed は空であってはならない（Assert による検証）。
  • Var ノードの場合、その属性番号（attno）は対象リレーションの有効範囲内である必要がある。

アルゴリズムまたは振る舞い

1. vars リストを反復処理する。 2. 各ノードが Var の場合:

  • find_base_rel で対象リレーションを取得。
  • where_needed が既にそのリレーションに含まれている場合はスキップ。
  • そうでない場合、rel->attr_needed の該当属性ビットマップに where_needed を追加する。

    • 3. 各ノードが PlaceHolderVar の場合:

  • find_placeholder_info で対応する PlaceHolderInfo を取得。
  • phinfo->ph_needed ビットマップに where_needed を追加する。

    • 4. 上記以外のノード型が渡された場合は elog(ERROR) を発生させる。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/optimizer/path/equivclass.c
  • src/backend/optimizer/plan/initsplan.c
  • src/backend/optimizer/util/placeholder.c
  • **呼び出し先**:
  • find_base_rel
  • find_placeholder_info
  • bms_add_members (ビットマップ操作)

関連データ構造

  • PlannerInfo: プランナの全体状態。
  • RelOptInfo: リレーションの最適化情報。
  • PlaceHolderVar / PlaceHolderInfo: プレースホルダ変数の定義と評価情報。
  • Relids: リレーションIDの集合（ビットマップ）。

失敗モードと境界条件

  • 未知のノード型が vars リストに含まれている場合、elog(ERROR) により処理が中断される。
  • where_needed が空の場合、Assert によりデバッグビルドで停止する。

保守担当者向けメモ

  • 本関数は、add_vars_to_targetlist と類似していますが、ターゲットリストの構築ではなく、変数の「必要性（どのリレーションで評価されるべきか）」を記録するために使用されます。
  • LATERAL 参照や複雑なジョイン条件の解析において、変数の依存関係を正しく伝播させるために不可欠な関数です。
