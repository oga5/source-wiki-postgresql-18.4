RecordIOData

定義

  • 種別: struct
  • 定義箇所: src/backend/utils/adt/rowtypes.c:43

目的

複合型（Composite Types）や匿名レコードの入出力処理において、実行時の型情報および各フィールドのI/O関数情報をキャッシュするために使用される構造体です。

シグネチャまたは形状

typedef struct RecordIOData
{
    Oid            record_type;
    int32          record_typmod;
    int            ncolumns;
    ColumnIOData   columns[FLEXIBLE_ARRAY_MEMBER];
} RecordIOData;

入力と出力

  • 本構造体自体は、関数呼び出しの fcinfo->flinfo->fn_extra に格納されるキャッシュデータとして機能します。
  • ncolumns に基づき、可変長配列 columns を通じて各フィールドのI/Oメタデータを提供します。

前提条件と不変条件

  • columns フィールドは FLEXIBLE_ARRAY_MEMBER として定義されており、構造体インスタンスは ncolumns の値に応じたサイズで動的に確保される必要があります。
  • キャッシュの生存期間は fn_extra に依存します。

アルゴリズムまたは振る舞い

  • レコードの入出力処理において、初回実行時に型定義（TupleDesc）から情報を取得し、本構造体にキャッシュします。
  • 以降の呼び出しでは、再計算を避けるためにこのキャッシュを再利用します。
  • 削除された列（attisdropped）は処理対象から除外されます。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • contrib/hstore/hstore_io.c 内の関数群が fn_extra を介して本構造体を参照しています。
  • src/backend/utils/adt/jsonfuncs.c において、可変長構造体として参照されています。
  • **呼び出し先**:
  • 本構造体自体はデータ保持用であり、直接的な関数呼び出し先はありません。ただし、保持する ColumnIOData を通じて各フィールドのI/O関数が呼び出されます。

関連データ構造

  • ColumnIOData: 各列のI/O情報を保持する構造体。
  • TupleDesc: レコードの構造定義。

失敗モードと境界条件

  • ncolumns が実際のタプル構造と一致しない場合、メモリ破壊や不正なアクセスが発生する可能性があります。
  • ALTER TYPE 等により型定義が変更された場合、キャッシュされた RecordIOData が無効になる可能性があるため、利用時には注意が必要です。

保守担当者向けメモ

  • 本構造体は fn_extra を介して共有されるため、スレッドセーフな読み取りを前提としています。
  • レコードの再帰的な構造を扱う際は、スタックオーバーフローを防ぐために check_stack_depth() を併用してください。
