ginInsertCleanup

定義

  • 種別: function
  • 定義箇所: src/backend/access/gin/ginfast.c:779

目的

GINインデックスのペンディングリスト（Pending List）に蓄積されたデータをメインのB-tree構造へ転送し、リストを解放する。

シグネチャまたは形状

void ginInsertCleanup(GinState *ginstate, bool full_clean, bool fill_fsm, bool forceCleanup, IndexBulkDeleteResult *stats)

入力と出力

  • **入力**:
  • ginstate: GINインデックスの状態管理構造体。
  • full_clean: true の場合、リスト全体を空になるまで処理する。
  • fill_fsm: 削除されたページをFSM（Free Space Map）へ返却するかどうか。
  • forceCleanup: 強制実行フラグ。true ならメタページを排他ロックして待機する。
  • stats: 統計情報格納用（現時点では未使用）。
  • **出力**: なし（副作用としてインデックス構造を更新）。

前提条件と不変条件

  • メタページに対してロックを取得し、並行処理を制御する。
  • full_clean が false の場合、処理開始時点のテールページまでを処理対象とする（無限ループ防止）。
  • 処理中、ペンディングリストのページは GinPageIsDeleted でないことを前提とする。

アルゴリズムまたは振る舞い

1. **ロック取得**: forceCleanup に応じてメタページを排他ロックする。 2. **初期化**: メタページからリストのヘッドとテールを取得し、メモリコンテキストを準備する。 3. **データ集約**: ペンディングリストをページ単位でスキャンし、processPendingPage を用いてデータを BuildAccumulator に蓄積する。 4. **転送**: メモリ上の蓄積データが上限（work_mem）に達するか、リストの末尾に到達した時点で、ginEntryInsert を呼び出しメインインデックスへ挿入する。 5. **ページ解放**: 転送完了後、処理済みページをリストから切り離し、必要に応じてFSMへ返却する。 6. **並行挿入への対応**: ページ処理中に挿入されたデータがある場合、再スキャンして処理を継続する。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/access/gin/ginfast.c (挿入処理時)
  • src/backend/access/gin/ginvacuum.c (VACUUM処理時)
  • gin_clean_pending_list (SQL関数)
  • **呼び出し先**:
  • processPendingPage
  • ginBeginBAScan / ginGetBAEntry
  • ginEntryInsert
  • vacuum_delay_point

関連データ構造

  • GinMetaPageData: ペンディングリストのヘッド/テール管理。
  • BuildAccumulator: 転送用データの一時集約。
  • GinState: インデックス操作のコンテキスト。

失敗モードと境界条件

  • **並行処理**: forceCleanup が false の場合、他プロセスが既にクリーンアップ中であれば即座に終了する。
  • **メモリ制限**: work_mem を超えるデータが蓄積された場合、部分的な転送を行いメモリを解放する。
  • **リストの空状態**: メタページのヘッドが InvalidBlockNumber の場合は何もせず終了する。

保守担当者向けメモ

  • 本関数はメタページの排他ロックを保持するが、リストの各ページ処理中は共有ロックに留めることで並行性を確保している。
  • 転送処理（ginEntryInsert）は時間がかかるため、ロックを解放した状態で実行される。
