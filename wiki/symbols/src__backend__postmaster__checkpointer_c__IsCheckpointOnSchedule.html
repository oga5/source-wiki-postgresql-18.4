IsCheckpointOnSchedule

定義

  • 種別: function
  • 定義箇所: src/backend/postmaster/checkpointer.c:841

目的

現在実行中のチェックポイントが、設定された目標時間およびWAL生成量に対して予定通りに進行しているかを判定します。

シグネチャまたは形状

static bool IsCheckpointOnSchedule(double progress)

入力と出力

  • **入力**: progress (0.0 から 1.0 の範囲で、チェックポイントの完了進捗を示す値)
  • **出力**: true (予定通りまたは先行している場合)、false (遅延している場合)

前提条件と不変条件

  • ckpt_active が真であること（チェックポイントがアクティブであること）。
  • CheckPointCompletionTarget、CheckPointSegments、CheckPointTimeout などのグローバル設定値が有効であること。
  • ckpt_start_recptr および ckpt_start_time がチェックポイント開始時に初期化されていること。

アルゴリズムまたは振る舞い

1. **進捗スケーリング**: 引数 progress に CheckPointCompletionTarget を乗算して調整します。 2. **キャッシュ確認**: progress がキャッシュされた ckpt_cached_elapsed より小さい場合は、計算を省略して false を返します。 3. **WAL進捗判定**:

  • 通常時は GetInsertRecPtr()、リカバリ時は GetXLogReplayRecPtr() を使用して現在のWAL位置を取得。
  • 開始位置からの経過WAL量を CheckPointSegments で正規化し、進捗と比較します。遅延していればキャッシュを更新し false を返します。

    • 4. **時間進捗判定**:

  • 現在時刻と ckpt_start_time の差分を CheckPointTimeout で正規化し、進捗と比較します。
  • 遅延していればキャッシュを更新し false を返します。

    • 5. **完了**: 上記のいずれの条件でも遅延が検出されなければ true を返します。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/postmaster/checkpointer.c 内のチェックポイント実行ループ（行788付近）。
  • **呼び出し先**:
  • RecoveryInProgress()
  • GetXLogReplayRecPtr() / GetInsertRecPtr()
  • gettimeofday()

関連データ構造

  • ckpt_active (グローバルフラグ)
  • ckpt_cached_elapsed (キャッシュ値)
  • ckpt_start_recptr (開始時のWAL位置)
  • ckpt_start_time (開始時の時刻)

失敗モードと境界条件

  • **リカバリ中の挙動**: リカバリ中は max_wal_size を超過する可能性があります。これはチェックポイントレコードの発見と開始位置の間にギャップが存在するためです。
  • **計算精度**: WAL位置の比較は RedoRecPtr ではなく開始時の位置に基づいているため、厳密な正確性よりも概算的なペース管理を目的としています。

保守担当者向けメモ

  • 本関数はチェックポイントの書き込み速度を平滑化（スロットリング）するための判断基準として使用されます。
  • ckpt_cached_elapsed は計算コストを抑えるための最適化であり、進捗が戻ることはないという前提で設計されています。
