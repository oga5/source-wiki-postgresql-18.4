ReplicationSlotPersistentData

定義

  • 種別: struct
  • 定義箇所: src/include/replication/slot.h:70

目的

レプリケーションスロットの状態をディスク上に永続化するためのデータ構造です。クラッシュリカバリ後も、スロットが保持すべきWALの範囲や論理デコーディングの進捗を復元するために使用されます。

シグネチャまたは形状

typedef struct ReplicationSlotPersistentData {
    NameData    name;
    Oid         database;
    ReplicationSlotPersistency persistency;
    TransactionId xmin;
    TransactionId catalog_xmin;
    XLogRecPtr  restart_lsn;
    ReplicationSlotInvalidationCause invalidated;
    XLogRecPtr  confirmed_flush;
    XLogRecPtr  two_phase_at;
    bool        two_phase;
    NameData    plugin;
    char        synced;
    bool        failover;
} ReplicationSlotPersistentData;

入力と出力

  • **入力**: スロット作成時や更新時に、ReplicationSlot 構造体の data フィールドを通じて値が設定されます。
  • **出力**: チェックポイント処理時にディスクへ書き出され、サーバー起動時の StartupReplicationSlots によってメモリへ読み込まれます。

前提条件と不変条件

  • xmin および catalog_xmin は、メモリ上の effective_xmin とは異なり、必ずしも最新のディスク書き込み状態を反映しているとは限りません。
  • スロットの属性変更時は、適切なロック（ReplicationSlotControlLock または各スロットの mutex）の取得が必須です。

アルゴリズムまたは振る舞い

  • **永続化**: スロット所有プロセスが ReplicationSlotMarkDirty を呼び出し、チェックポイント時に CheckPointReplicationSlots がこの構造体の内容をディスクへフラッシュします。
  • **リカバリ**: サーバー起動時に、ディスク上のデータから本構造体が再構築され、共有メモリ上のスロット状態が初期化されます。
  • **論理デコーディング**: confirmed_flush はクライアントが受信確認した最新のLSNとして機能し、再接続時の開始位置の安全策として利用されます。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/replication/slot.c 内の各関数（スロットの初期化、更新、チェックポイント処理など）。
  • **呼び出し先**: 本構造体自体はデータ保持用であり、直接的な関数呼び出しは行いません。

関連データ構造

  • ReplicationSlot: メモリ内でのスロット表現。本構造体を data フィールドとして内包します。
  • ReplicationSlotCtlData: 全スロットを管理する共有メモリ上の配列。
  • ReplicationSlotPersistency: スロットの永続性レベルを定義する列挙型。

失敗モードと境界条件

  • **無効化**: invalidated フィールドが RS_INVAL_NONE 以外の場合、スロットは無効状態とみなされます。
  • **整合性**: two_phase が有効な場合、two_phase_at に記録されたLSNが整合性ポイントとして利用されます。

保守担当者向けメモ

  • 本構造体はディスクフォーマットに直結するため、フィールドの追加や変更を行う際は、ディスク上のレイアウト互換性に注意してください。
  • xmin 等のフィールドはメモリ上の動的状態と混同しやすいため、コード修正時は effective_xmin との役割の違いを再確認してください。
