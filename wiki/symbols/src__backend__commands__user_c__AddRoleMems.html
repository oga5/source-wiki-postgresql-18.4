AddRoleMems

定義

  • **種別**: function
  • **定義箇所**: src/backend/commands/user.c:1680

目的

指定されたロールに対して、メンバーとなるロールのリストを追加（メンバーシップの付与）します。

シグネチャまたは形状

static void AddRoleMems(Oid currentUserId, const char *rolename, Oid roleid,
                        List *memberSpecs, List *memberIds,
                        Oid grantorId, GrantRoleOptions *popt)

入力と出力

  • **入力**:
  • currentUserId: 操作を実行するユーザーのOID。
  • rolename: 対象ロール名。
  • roleid: 対象ロールのOID。
  • memberSpecs: 追加するメンバーの RoleSpec リスト。
  • memberIds: 追加するメンバーのOIDリスト。
  • grantorId: 権限付与者（GRANTOR）のOID。
  • popt: ADMIN OPTION 等のオプションを含む GrantRoleOptions 構造体。
  • **出力**: なし（成功時はカタログ更新、失敗時はエラーをスロー）。

前提条件と不変条件

  • list_length(memberSpecs) == list_length(memberIds) であること。
  • メンバーシップの循環（ループ）は禁止されており、チェックされる。
  • pg_database_owner をメンバーに含めることはできない。
  • ADMIN OPTION を付与する場合、付与者自身がその権限を失うような循環的な付与は禁止される。

アルゴリズムまたは振る舞い

1. check_role_grantor を呼び出し、付与者の正当性を検証する。 2. pg_auth_members カタログを RowExclusiveLock で開き、対象ロールに対して ShareUpdateExclusiveLock を取得して競合を防ぐ。 3. メンバーごとに以下のチェックを行う:

  • pg_database_owner ではないこと。
  • メンバーシップのループ（自身をメンバーにする等）が発生しないこと。

    • 4. popt->admin が真の場合、plan_member_revoke を用いて再帰的な権限剥奪の影響をシミュレートし、循環的な ADMIN OPTION 付与が発生しないか検証する。 5. 検証通過後、pg_auth_members カタログを走査し、既存のメンバーシップがあれば更新（Update）、なければ新規挿入（Insert）を行う。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/commands/user.c 内の GrantRole 等のロール管理関数から呼び出される。
  • **呼び出し先**: check_role_grantor, table_open, LockSharedObject, is_member_of_role_nosuper, SearchSysCacheList1, initialize_revoke_actions, plan_member_revoke, SearchSysCache3 等。

関連データ構造

  • pg_auth_members: メンバーシップ情報を保持するシステムカタログ。
  • GrantRoleOptions: 権限付与時のオプション（ADMIN, INHERIT, SET）を保持。
  • RevokeRoleGrantAction: 権限剥奪時のアクションを定義する列挙型。

失敗モードと境界条件

  • ERRCODE_FEATURE_NOT_SUPPORTED: pg_database_owner をメンバーに追加しようとした場合。
  • ERRCODE_INVALID_GRANT_OPERATION: メンバーシップのループが発生した場合、または ADMIN OPTION の循環付与が発生した場合。

保守担当者向けメモ

  • 本関数は pg_auth_members を直接操作するため、カタログの整合性維持が最優先されます。
  • ADMIN OPTION の循環チェックには plan_member_revoke を使用したシミュレーションが含まれており、複雑な依存関係を考慮する必要があります。
  • メンバーシップの変更は LockSharedObject によるロック下で行われ、並行実行による競合を回避しています。
