save_buffer

定義

  • 種別: struct
  • 定義箇所: src/backend/postmaster/syslogger.c:103

目的

Syslogger プロセスにおいて、各バックエンドプロセスから送信されるログメッセージを PID ごとに一時的に蓄積（バッファリング）するためのデータ構造です。

シグネチャまたは形状

typedef struct
{
    int32       pid;            /* 送信元プロセスの PID */
    StringInfoData data;        /* 蓄積されたデータ（StringInfo 形式） */
} save_buffer;

入力と出力

  • 本構造体はデータ保持用であり、関数ではないため直接の入出力はありません。
  • data フィールドを通じて、パイプから読み込まれたログの断片を完全なメッセージとして再構成するために使用されます。

前提条件と不変条件

  • pid はログの送信元プロセスを一意に識別します。
  • data は StringInfoData 型であり、動的に拡張可能な文字列バッファとして機能します。

アルゴリズムまたは振る舞い

  • Syslogger はパイプから受信したログデータが完全なメッセージでない場合、この構造体を使用して pid ごとにバッファリングを行います。
  • 複数のプロセスからログが混在して送られてくる状況下で、PID をキーとして各プロセスのログを分離・保持します。

呼び出し元と呼び出し先

  • 呼び出し元（参照箇所）:
  • src/backend/postmaster/syslogger.c:904
  • src/backend/postmaster/syslogger.c:930
  • src/backend/postmaster/syslogger.c:963
  • src/backend/postmaster/syslogger.c:1054
  • 呼び出し先: 構造体定義のため、特定の関数呼び出し関係はありません。

関連データ構造

  • StringInfoData: lib/stringinfo.h で定義される、可変長文字列を扱うための構造体。
  • PipeProtoHeader: パイプ経由の通信プロトコルヘッダー。

失敗モードと境界条件

  • メモリ不足: palloc による save_buffer の確保時にメモリが不足した場合、PostgreSQL の標準的なエラー処理に従いアボートします。
  • バッファの肥大化: 特定のプロセスがログを送信し続け、かつメッセージが完了しない場合、data バッファがメモリを消費し続ける可能性があります。

保守担当者向けメモ

  • 本構造体は syslogger.c 内のログ再構成ロジックにおいて、PID ごとの状態管理の要となります。
  • palloc を使用して動的に確保されるため、不要になった際のメモリ解放（pfree 等）が適切に行われているか、関連するリスト操作を確認してください。
