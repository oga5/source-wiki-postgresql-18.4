PagetableEntry

定義

  • 種別: struct
  • 定義箇所: src/backend/nodes/tidbitmap.c:91

目的

TIDBitmap 内で、特定のページまたはページ範囲におけるタプルの存在状態を管理するためのハッシュテーブルエントリです。正確なタプル位置の保持（正確なページ）と、ページ単位の存在フラグの保持（損失ありページ）の両方を表現します。

シグネチャまたは形状

typedef struct PagetableEntry
{
    BlockNumber blockno;        /* ハッシュテーブルのキー */
    char        status;         /* ハッシュエントリの状態 */
    bool        ischunk;        /* T = 損失あり(lossy), F = 正確(exact) */
    bool        recheck;        /* 再チェックが必要か */
    bitmapword  words[Max(WORDS_PER_PAGE, WORDS_PER_CHUNK)];
} PagetableEntry;

入力と出力

  • **入力**: blockno をキーとしてハッシュテーブルに格納されます。
  • **出力**: ischunk フラグに応じて、words 配列が個別のタプルオフセット（正確な場合）またはページ範囲の存在ビットマップ（損失ありの場合）として解釈されます。

前提条件と不変条件

  • ischunk が false の場合、words はページ内の個別のタプル位置を正確に示します。
  • ischunk が true の場合、words はページ範囲（チャンク）内のページ存在フラグを示します。
  • blockno はハッシュテーブルの検索キーとして一意である必要があります。

アルゴリズムまたは振る舞い

  • TIDBitmap がメモリ制限に達した際、tbm_lossify 関数によって正確なページ情報が損失あり情報へ変換される過程で、この構造体の ischunk フラグが切り替わります。
  • 集合演算（UNION, INTERSECT）の際、tbm_union_page や tbm_intersect_page を通じて、このエントリの内容が他のビットマップとマージされます。

呼び出し元と呼び出し先

  • **呼び出し元**: tbm_get_pageentry がこの構造体を生成・取得します。また、tbm_union_page や tbm_intersect_page がこの構造体を引数として参照します。
  • **呼び出し先**: 構造体自体はデータ保持用であり、直接関数を呼び出すことはありません。
  • **注記**: 呼び出し関係の詳細は、提供されたソースの範囲内では上記関数による参照に限定されます。

関連データ構造

  • TIDBitmap: 本構造体をハッシュテーブルとして保持するメイン構造体。
  • pagetable_hash: PagetableEntry を格納するハッシュテーブル定義。

失敗モードと境界条件

  • メモリ不足時: tbm_lossify により、正確な情報が損失あり情報へ変換され、詳細なタプル情報が失われる可能性があります。
  • words 配列のサイズ: WORDS_PER_PAGE と WORDS_PER_CHUNK の大きい方に固定されており、これを超えるデータは保持できません。

保守担当者向けメモ

  • ischunk フラグの変更は、ビットマップの精度に直接影響するため、変更時には recheck フラグの整合性に注意してください。
  • src/backend/access/common/tidstore.c に類似の名称が存在しますが、本構造体は tidbitmap.c 専用の管理用エントリです。
