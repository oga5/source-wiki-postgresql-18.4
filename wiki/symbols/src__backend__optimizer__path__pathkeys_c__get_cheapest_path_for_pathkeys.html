get_cheapest_path_for_pathkeys

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/path/pathkeys.c:619

目的

指定されたパスキー（pathkeys）を満たし、かつ外部リレーションの制約（required_outer）を充足するパスの中で、コストが最小の Path を選択する。

シグネチャまたは形状

Path *get_cheapest_path_for_pathkeys(List *paths, List *pathkeys,
                                     Relids required_outer,
                                     CostSelector cost_criterion,
                                     bool require_parallel_safe)

入力と出力

  • **入力**:
  • paths: 評価対象となる Path ノードのリスト。
  • pathkeys: 満たすべき順序を示す PathKey のリスト。
  • required_outer: パスが依存してもよい外部リレーションの集合。
  • cost_criterion: コスト比較の基準（STARTUP_COST または TOTAL_COST）。
  • require_parallel_safe: 並列安全性を必須とするかのフラグ。
  • **出力**: 条件を満たす最小コストの Path へのポインタ。該当するパスがない場合は NULL を返す。

前提条件と不変条件

  • 入力される paths リスト内の各 Path は、適切に初期化されている必要がある。
  • pathkeys は正規化された形式であること。
  • compare_path_costs は、指定された cost_criterion に基づいてコストを比較する。

アルゴリズムまたは振る舞い

1. paths リストを走査し、各 path を評価する。 2. require_parallel_safe が真の場合、path->parallel_safe が偽のパスをスキップする。 3. 既に matched_path が見つかっている場合、現在の path とコストを比較し、現在の path の方が高コストであればスキップする。 4. pathkeys_contained_in(pathkeys, path->pathkeys) により、対象パスが要求された順序を提供可能か確認する。 5. bms_is_subset(PATH_REQ_OUTER(path), required_outer) により、パスの外部依存関係が許容範囲内か確認する。 6. 上記条件をすべて満たす場合、そのパスを matched_path として更新する。 7. リスト走査終了後、最終的な matched_path を返す。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/optimizer/path/allpaths.c
  • src/backend/optimizer/path/joinpath.c
  • src/backend/optimizer/prep/prepunion.c
  • **呼び出し先**:
  • compare_path_costs
  • pathkeys_contained_in
  • bms_is_subset
  • PATH_REQ_OUTER (マクロ)

関連データ構造

  • Path: 実行計画のノード。
  • PathKey: 順序を定義する構造体。
  • Relids: リレーションIDの集合（ビットマップ）。

失敗モードと境界条件

  • paths が空の場合、NULL を返す。
  • 条件を満たすパスが存在しない場合、NULL を返す。
  • require_parallel_safe が真で、条件を満たすパスがすべて並列非安全な場合、NULL を返す。

保守担当者向けメモ

  • コスト比較がパスキー比較よりも安価であるという前提で最適化されているが、この前提の妥当性は将来的に再評価が必要である可能性がある（ソース内の XXX コメント参照）。
