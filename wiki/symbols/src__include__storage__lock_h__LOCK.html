LOCK

定義

  • 種別: struct
  • 定義箇所: src/include/storage/lock.h:309

目的

共有メモリ上で管理されるロックオブジェクトの表現。リレーション、ページ、タプルなどの特定のデータベースオブジェクトに対するロック状態（取得済みモード、待機中プロセス等）を保持し、排他制御を実現する。

シグネチャまたは形状

typedef struct LOCK
{
    LOCKTAG     tag;
    LOCKMASK    grantMask;
    LOCKMASK    waitMask;
    dlist_head  procLocks;
    dclist_head waitProcs;
    int         requested[MAX_LOCKMODES];
    int         nRequested;
    int         granted[MAX_LOCKMODES];
    int         nGranted;
} LOCK;

入力と出力

  • **入力**: ロック取得要求時に LOCKTAG をキーとしてハッシュテーブルから検索・生成される。
  • **出力**: ロック状態の更新（grantMask, requested, granted 等）および待機キュー（waitProcs）へのプロセス追加・削除。

前提条件と不変条件

  • tag はロック対象を一意に識別する。
  • requested 配列の合計値は nRequested と一致し、granted 配列の合計値は nGranted と一致する。
  • procLocks は、このロックに関連付けられたすべての PROCLOCK オブジェクトを保持する。
  • waitProcs は、このロックの取得を待機している PGPROC オブジェクトのリストである。

アルゴリズムまたは振る舞い

1. **競合判定**: LockAcquire が呼び出されると、grantMask を参照して要求されたロックモードとの競合をチェックする。 2. **待機管理**: 競合が発生した場合、呼び出し元プロセスを waitProcs に追加し、待機状態へ遷移させる。 3. **状態更新**: ロックの取得・解放に伴い、grantMask、waitMask、および各モードのカウント（requested, granted）を更新する。 4. **ウェイクアップ**: LockRelease によりロックが解放される際、waitProcs 内のプロセスに対して再開を試みる。

呼び出し元と呼び出し先

  • **呼び出し元**: LockAcquire, LockRelease 等のロックマネージャAPI（src/backend/storage/lmgr/lock.c）。
  • **呼び出し先**:
  • dlist_head, dclist_head（リスト操作）
  • PGPROC（待機プロセス管理）
  • PROCLOCK（プロセスとロックの関連付け）
  • 注: 具体的な関数呼び出しの網羅的な関係性は、提供されたソース範囲内では不完全です。

関連データ構造

  • LOCKTAG: ロック対象の識別子。
  • PROCLOCK: プロセスとロックの関連付けを保持する構造体。
  • PGPROC: 待機中のプロセス情報。
  • LOCKMASK: ロックモードのビットマスク。

失敗モードと境界条件

  • **ハッシュ衝突**: ロックテーブルのパーティション内でハッシュ衝突が発生した場合、適切なチェーン管理が必要となる。
  • **デッドロック**: 複数のプロセスが相互にロックを待機する場合、デッドロック検出器が介入する。
  • **メモリ制限**: 共有メモリ上のロックテーブルが枯渇した場合、ロック取得に失敗する可能性がある。

保守担当者向けメモ

  • LOCKTAG は16バイトに固定されており、フィールド追加時はサイズ制約に注意すること。
  • ロックテーブルは共有メモリ上でパーティション化されているため、アクセス時には LockHashPartition マクロによる保護が必須である。
