SnapBuild

定義

  • 種別: struct
  • 定義箇所: src/include/replication/snapbuild_internal.h:26

目的

論理デコーディングにおけるスナップショット構築プロセスの内部状態を保持・管理する構造体です。トランザクションの追跡、カタログ変更履歴の管理、およびスナップショットのシリアライズをサポートします。

シグネチャまたは形状

struct SnapBuild {
    SnapBuildState state;
    MemoryContext context;
    TransactionId xmin;
    TransactionId xmax;
    XLogRecPtr start_decoding_at;
    XLogRecPtr two_phase_at;
    TransactionId initial_xmin_horizon;
    bool building_full_snapshot;
    bool in_slot_creation;
    Snapshot snapshot;
    XLogRecPtr last_serialized_snapshot;
    ReorderBuffer *reorder;
    TransactionId next_phase_at;
    struct { size_t xcnt; size_t xcnt_space; bool includes_all_transactions; TransactionId *xip; } committed;
    struct { size_t xcnt; TransactionId *xip; } catchange;
};

入力と出力

  • **入力**: WAL解析を通じて更新される xmin/xmax、xl_running_xacts 情報、およびディスクから復元されるシリアライズ済みデータ。
  • **出力**: 論理デコーディングで使用される Snapshot オブジェクト、およびディスク保存用のシリアライズ状態。

前提条件と不変条件

  • catchange 配列は、スナップショット復元時にカタログ変更を正しく追跡するために使用されます。
  • committed 配列は、CONSISTENT状態に達するまでは全トランザクションを記録し、その後はカタログ変更を行ったもののみを記録します。
  • in_slot_creation が真の場合、スナップショットの復元はスキップされます。

アルゴリズムまたは振る舞い

1. **状態管理**: SnapBuildState に基づき、スナップショットが「一貫性のある状態（CONSISTENT）」に達するまでトランザクション情報を蓄積します。 2. **シリアライズ**: SnapBuildOnDisk 構造体と連携し、現在のメモリ状態をディスクへ永続化します。 3. **復元**: SnapBuildRestoreSnapshot() を通じて、ディスク上のデータからメモリ状態を再構築します。 4. **カタログ追跡**: catchange 配列により、シリアライズ後にコミットされたトランザクションのカタログ変更漏れを防ぎます。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/replication/logical/decode.c (論理デコーディングコンテキスト経由)、src/backend/replication/logical/reorderbuffer.c。
  • **呼び出し先**: SnapBuildPurgeOlderTxn, SnapBuildBuildSnapshot, SnapBuildDistributeSnapshotAndInval (いずれも src/backend/replication/logical/snapbuild.c 内)。

関連データ構造

  • SnapBuildOnDisk: ディスク保存用のシリアライズ形式。
  • ReorderBuffer: 変更されたトランザクションの再順序付け管理に使用。
  • Snapshot: 論理デコーディングで利用されるスナップショットの実体。

失敗モードと境界条件

  • catchange 配列の管理が不適切な場合、カタログ変更を見逃し、誤ったスナップショットでカタログを参照するリスクがあります。
  • committed 配列のソート順序や管理ロジックは、プロファイル結果に応じて改善の余地があるとされています。
  • SnapBuildOnDisk のレイアウト変更は、シリアライズデータの互換性に直接影響します。

保守担当者向けメモ

  • 本構造体は論理デコーディングの核心であり、変更には細心の注意が必要です。
  • catchange 配列の管理ロジック（SnapBuildPurgeOlderTxn 等）を理解することが、論理デコーディングの正確性を担保する鍵となります。
  • シリアライズ形式を変更する場合は、バージョン管理およびチェックサム計算ロジックの整合性を再確認してください。
