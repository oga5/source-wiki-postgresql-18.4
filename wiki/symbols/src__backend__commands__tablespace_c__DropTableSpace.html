DropTableSpace

定義

  • 種別: function
  • 定義箇所: src/backend/commands/tablespace.c:394

目的

指定されたテーブルスペースをシステムから削除し、関連する物理ディレクトリおよびメタデータを破棄します。

シグネチャまたは形状

void DropTableSpace(DropTableSpaceStmt *stmt)

入力と出力

  • **入力**: DropTableSpaceStmt *stmt（削除対象のテーブルスペース名および missing_ok フラグを含む構造体）
  • **出力**: なし（成功時はカタログ更新および物理削除を実行、失敗時はエラーをスロー）

前提条件と不変条件

  • 呼び出し元は pg_tablespace カタログに対する RowExclusiveLock を保持している必要があります。
  • 実行ユーザーは対象テーブルスペースの所有者である必要があります。
  • 標準のテーブルスペース（IsPinnedObject が真のもの）は削除できません。
  • 依存オブジェクトが存在する場合、削除は拒否されます。

アルゴリズムまたは振る舞い

1. pg_tablespace カタログから対象の OID を検索します。存在しない場合、missing_ok に応じてエラーまたは通知を返します。 2. 所有権チェックおよび依存関係チェック（checkSharedDependencies）を実行します。 3. CatalogTupleDelete によりカタログからエントリを削除し、関連するコメント、セキュリティラベル、所有者依存関係を削除します。 4. TablespaceCreateLock を排他モードで取得し、物理ディレクトリの削除（destroy_tablespace_directories）を試みます。 5. 削除に失敗した場合、チェックポイントを強制実行し、プロセスシグナルバリアを使用してファイルハンドルを解放させた後、再試行します。それでも失敗した場合はエラーをスローします。 6. 削除操作を WAL（XLOG_TBLSPC_DROP）に記録し、同期コミットを強制してトランザクションを完了します。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/tcop/utility.c（SQL 解析後の DDL 実行時）
  • **呼び出し先**: table_open, table_beginscan_catalog, heap_getnext, object_ownercheck, IsPinnedObject, checkSharedDependencies, InvokeObjectDropHook, CatalogTupleDelete, DeleteSharedComments, DeleteSharedSecurityLabel, deleteSharedDependencyRecordsFor, LWLockAcquire, destroy_tablespace_directories, RequestCheckpoint, LWLockRelease, WaitForProcSignalBarrier, EmitProcSignalBarrier, XLogBeginInsert, XLogRegisterData, XLogInsert, ForceSyncCommit, table_close

関連データ構造

  • DropTableSpaceStmt: 削除対象の情報を保持する構文解析ツリー構造体。
  • pg_tablespace: テーブルスペース情報を管理するシステムカタログ。
  • xl_tblspc_drop_rec: WAL 記録用の構造体。

失敗モードと境界条件

  • **オブジェクト不在**: テーブルスペースが存在しない場合、missing_ok が偽ならエラー。
  • **権限不足**: 所有者でない場合、またはシステム固定のテーブルスペースである場合にエラー。
  • **依存関係**: 他のオブジェクトがテーブルスペースを使用している場合、エラー。
  • **ディレクトリ削除失敗**: 物理ディレクトリが空でない（ファイルが残っている）場合、チェックポイントとバリアによる解放を試みるが、それでも空でなければエラー。

保守担当者向けメモ

  • 本関数は TablespaceCreateLock を使用して、データベース作成処理との競合を回避しています。
  • Windows 環境ではファイルハンドルが残存しやすいため、PROCSIGNAL_BARRIER_SMGRRELEASE を用いた明示的な解放処理が含まれています。
  • 物理削除後の WAL 記録と同期コミットの順序は、クラッシュリカバリ時の整合性を保つために重要です。
