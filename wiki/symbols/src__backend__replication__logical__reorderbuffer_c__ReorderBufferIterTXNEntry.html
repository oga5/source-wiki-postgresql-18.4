ReorderBufferIterTXNEntry

定義

  • 種別: struct
  • 定義箇所: src/backend/replication/logical/reorderbuffer.c:159

目的

論理デコーディングにおいて、複数のトランザクションにまたがる変更レコードを LSN（Log Sequence Number）順にマージ・イテレーションするための各要素を保持する。

シグネチャまたは形状

typedef struct ReorderBufferIterTXNEntry
{
    XLogRecPtr    lsn;
    ReorderBufferChange *change;
    ReorderBufferTXN *txn;
    TXNEntryFile file;
    XLogSegNo    segno;
} ReorderBufferIterTXNEntry;

入力と出力

本構造体はデータ保持用であり、関数ではないため入出力の定義はない。各フィールドはイテレータの状態管理に使用される。

前提条件と不変条件

  • lsn は、対応する change またはディスク上のレコードが指すログ位置を示す。
  • txn がメモリ上に存在しない場合、file および segno を用いてディスク上のデータへアクセスする必要がある。

アルゴリズムまたは振る舞い

バイナリヒープを用いたマージソート処理において、各トランザクションの現在の変更位置を追跡するために使用される。ReorderBufferIterTXNInit 等の処理で、複数のトランザクションから取り出した先頭レコードを本構造体に格納し、ヒープで比較することで LSN 順の出力を実現する。

呼び出し元と呼び出し先

  • 呼び出し元: src/backend/replication/logical/reorderbuffer.c 内のイテレーション処理（ReorderBufferIterTXNInit 等）で動的に確保・参照される。
  • 呼び出し先: 本構造体自体はデータ構造であり、直接関数を呼び出すことはない。

関連データ構造

  • ReorderBufferTXN: トランザクションの全体状態。
  • ReorderBufferChange: トランザクション内の個別の変更内容。
  • ReorderBufferIterTXNState: 本構造体の配列を保持し、イテレーション全体を管理する構造体。
  • TXNEntryFile: ディスクに退避されたトランザクションデータのファイルハンドル。

失敗モードと境界条件

  • メモリ不足時: ReorderBufferIterTXNInit 等で本構造体の配列確保に失敗する可能性がある。
  • ディスク読み取り失敗: file を介したアクセス時に、退避されたデータが破損している場合や読み取りエラーが発生する場合がある。

保守担当者向けメモ

  • 本構造体は ReorderBufferIterTXNState 内で配列として使用される。
  • FLEXIBLE_ARRAY_MEMBER を用いた動的確保が行われる箇所があるため、メモリレイアウトの変更時には注意が必要である。
  • 構造体内の lsn フィールドは、ヒープの比較キーとして使用されるため、常に最新の変更位置と同期している必要がある。
