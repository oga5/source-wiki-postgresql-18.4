RWConflictPoolHeaderData

定義

  • **種別**: struct
  • **定義箇所**: src/include/storage/predicate_internals.h:206

目的

SSI（Serializable Snapshot Isolation）において、トランザクション間の読み書き競合（rw-conflict）を表現する RWConflictData オブジェクトを管理するためのプールヘッダーです。

シグネチャまたは形状

typedef struct RWConflictPoolHeaderData
{
	dlist_head	availableList;
	RWConflict	element;
} RWConflictPoolHeaderData;

入力と出力

  • **入力**: 共有メモリ上の RWConflict オブジェクトのリスト管理。
  • **出力**: availableList を介した、再利用可能な RWConflict 要素の提供。

前提条件と不変条件

  • availableList は lib/ilist.h で定義される dlist_head を使用し、共有メモリ上で競合オブジェクトを追跡します。
  • 本構造体は、SSIサブシステムが共有メモリ上で競合関係を動的に割り当て・解放するために使用されます。

アルゴリズムまたは振る舞い

  • 未使用の RWConflict オブジェクトを availableList に保持し、必要に応じて取り出すことで、競合発生時のメモリ割り当てオーバーヘッドを抑制します。
  • 競合関係が解消された際、オブジェクトは再びこのプールへ戻されます。

呼び出し元と呼び出し先

  • **呼び出し元**: 提供されたソース情報からは、具体的な関数呼び出し元は特定できません。
  • **呼び出し先**: lib/ilist.h のリスト操作関数群を利用します。

関連データ構造

  • RWConflict: 競合関係を表現するデータ構造。
  • RWConflictPoolHeader: 本構造体へのポインタ型。
  • SERIALIZABLEXACT: 競合の当事者となるトランザクション状態。

失敗モードと境界条件

  • プールが枯渇した場合の挙動については、本ヘッダー定義からは判断できません。
  • 共有メモリ上のデータ構造であるため、アクセスには適切なロック（LWLock 等）による保護が前提となります。

保守担当者向けメモ

  • RWConflictPoolHeaderData のサイズは MAXALIGN で調整されており、共有メモリレイアウト変更時にはアライメントに注意してください。
  • element フィールドはプール内の要素の起点として機能しますが、実際のメモリ配置については predicate.c での初期化ロジックを確認してください。
