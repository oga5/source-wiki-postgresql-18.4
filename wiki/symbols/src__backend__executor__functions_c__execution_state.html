execution_state

定義

  • 種別: struct
  • 定義箇所: src/backend/executor/functions.c:68

目的

SQL関数内の各クエリの実行状態を個別に管理し、関数実行エンジンがクエリを順次処理するためのリンク付きリスト構造を提供します。

シグネチャまたは形状

typedef struct execution_state
{
    struct execution_state *next;
    ExecStatus    status;
    bool          setsResult;
    bool          lazyEval;
    PlannedStmt  *stmt;
    QueryDesc    *qd;
} execution_state;

入力と出力

  • **入力**: 関数実行時に SQLFunctionCache を通じて初期化され、各クエリの計画（PlannedStmt）が保持されます。
  • **出力**: QueryDesc を介して実行結果が生成されます。

前提条件と不変条件

  • next ポインタにより、関数内のクエリがリスト形式で連結されます。
  • qd（QueryDesc）は、status が RUN 状態である場合にのみ有効（非NULL）です。

アルゴリズムまたは振る舞い

  • SQL関数実行時、この構造体を用いてクエリの解析・計画・実行を順次行います。
  • lazyEval が真の場合、セットを返す関数において全行を一度に計算せず、呼び出しごとに1行ずつフェッチする挙動を制御します。
  • postquel_start で実行を開始し、postquel_getnext で結果を取得、postquel_end でリソースを解放するライフサイクルを持ちます。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/executor/functions.c 内の関数実行ロジック（fmgr_sql 等）。
  • **呼び出し先**: postquel_start, postquel_getnext, postquel_end がこの構造体を引数として受け取り操作します。

関連データ構造

  • SQLFunctionCache: 関数実行ごとの状態を保持する上位構造体。
  • PlannedStmt: 実行対象のクエリ計画。
  • QueryDesc: 実行中のクエリの記述子。

失敗モードと境界条件

  • エラー発生時には sql_exec_error_callback が呼び出され、実行コンテキストのトレースバックが行われます。
  • 実行終了時には postquel_end を経由してリソースを解放する必要があります。

保守担当者向けメモ

  • 本構造体は関数内の各文を順次実行する「Postquel」スタイルの実行モデルの中核です。
  • lazyEval フラグの扱いは、セットを返す関数のパフォーマンスに直結するため、変更時は注意が必要です。
