spgRedoSplitTuple

定義

  • 種別: function
  • 定義箇所: src/backend/access/spgist/spgxlog.c:450

目的

SP-GiST インデックスのタプル分割操作を WAL リプレイ時に再現する。元のページからプレフィックスを削除し、新しいタプル（プレフィックスおよびポストフィックス）を適切なページへ配置することで、インデックスの論理構造を復元する。

シグネチャまたは形状

static void spgRedoSplitTuple(XLogReaderState *record)

入力と出力

  • **入力**: XLogReaderState *record（リプレイ対象の WAL レコード）。
  • **出力**: なし（副作用としてバッファページを更新）。

前提条件と不変条件

  • WAL レコード内のタプルはアライメントされていないため、アクセス時に memcpy を用いてヘッダーをコピーする必要がある。
  • リンク切れを防ぐため、ポストフィックス（子側）の更新を先に行い、その後に元のページ（親側）を更新する順序が守られる必要がある。

アルゴリズムまたは振る舞い

1. WAL レコードから spgxlogSplitTuple 構造体およびプレフィックス・ポストフィックスのタプルデータを抽出する。 2. **ポストフィックスの処理**:

  • xldata->postfixBlkSame が偽の場合、対象ページを読み込み（新規作成または既存読み込み）、addOrReplaceTuple を用いてポストフィックスを挿入する。

    • 3. **元のページの処理**:

  • xldata->offnumPrefix にある既存タプルを削除する。
  • 新しいプレフィックスを PageAddItem で挿入する。
  • xldata->postfixBlkSame が真の場合は、同じページ内にポストフィックスも追加する。

    • 4. 各操作後、ページに LSN を設定し、バッファをダーティとしてマークする。

呼び出し元と呼び出し先

  • **呼び出し元**: spg_redo（src/backend/access/spgist/spgxlog.c 内のディスパッチャ）。
  • **呼び出し先**: XLogRecGetData, XLogInitBufferForRedo, SpGistInitBuffer, XLogReadBufferForRedo, BufferGetPage, addOrReplaceTuple, PageSetLSN, MarkBufferDirty, UnlockReleaseBuffer, PageIndexTupleDelete, PageAddItem, elog。

関連データ構造

  • spgxlogSplitTuple: 分割操作の詳細（オフセット番号やフラグ）を保持する WAL 構造体。
  • SpGistInnerTupleData: タプルのヘッダー情報。

失敗モードと境界条件

  • PageAddItem が期待したオフセット番号を返さない場合、elog(ERROR) によりリプレイが中断される。
  • XLogReadBufferForRedo 等のバッファ操作関数が失敗した場合の挙動は、それら関数の仕様に依存する。

保守担当者向けメモ

  • 本関数はスタンバイサーバーでの整合性維持を目的としている。
  • postfixBlkSame フラグにより、ポストフィックスが元のページと同じページに配置されるか、別のページに配置されるかを切り替えている。
  • ページ更新の順序（子から親へ）は、スタンバイ読み取り時の整合性を担保するために重要である。
