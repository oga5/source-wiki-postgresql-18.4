RelidByRelfilenumber

定義

  • 種別: function
  • 定義箇所: src/backend/utils/cache/relfilenumbermap.c:140

目的

指定されたテーブルスペースとファイル番号（relfilenumber）から、対応するリレーションの OID（relid）を逆引きします。キャッシュを利用して検索コストを削減し、存在しない場合はカタログ検索を行います。

シグネチャまたは形状

Oid RelidByRelfilenumber(Oid reltablespace, RelFileNumber relfilenumber)

入力と出力

  • **入力**:
  • reltablespace: 検索対象のテーブルスペース OID。MyDatabaseTableSpace の場合は 0 に正規化されます。
  • relfilenumber: 検索対象のファイル番号。
  • **出力**:
  • 対応するリレーションの Oid。見つからない場合は InvalidOid を返します。

前提条件と不変条件

  • RelfilenumberMapHash が初期化されていない場合、内部で InitializeRelfilenumberMap() が呼び出されます。
  • 一時テーブル（RELPERSISTENCE_TEMP）は検索対象から除外されます。
  • 検索結果（InvalidOid を含む）はキャッシュに格納されます。

アルゴリズムまたは振る舞い

1. **キャッシュ検索**: RelfilenumberMapHash を検索し、ヒットすればその relid を即座に返します。 2. **カタログ検索（ミス時）**:

  • reltablespace が GLOBALTABLESPACE_OID の場合: RelationMapFilenumberToOid を使用して解決します。
  • それ以外の場合: pg_class に対して ClassTblspcRelfilenodeIndexId インデックスを用いたスキャンを実行します。
  • カタログ検索でも見つからない場合: RelationMapFilenumberToOid（非共有）を試行します。

    • 3. **キャッシュ更新**: 検索結果をハッシュテーブルに登録し、呼び出し元へ返します。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • contrib/pg_prewarm/autoprewarm.c
  • src/backend/replication/logical/reorderbuffer.c
  • src/backend/utils/adt/dbsize.c
  • **呼び出し先**:
  • InitializeRelfilenumberMap
  • hash_search
  • RelationMapFilenumberToOid
  • table_open / table_close
  • systable_beginscan / systable_getnext / systable_endscan

関連データ構造

  • RelfilenumberMapKey: 検索キー（テーブルスペースとファイル番号）。
  • RelfilenumberMapEntry: キャッシュエントリ（OID を保持）。
  • RelfilenumberMapHash: キャッシュを保持するハッシュテーブル。

失敗モードと境界条件

  • **重複**: カタログ検索中に予期せぬ重複が見つかった場合、elog(ERROR) を発生させます。
  • **キャッシュ破損**: キャッシュ登録時に既にエントリが存在する場合、elog(ERROR) を発生させます。
  • **一時テーブル**: 一時テーブルは検索対象外であり、キャッシュにも反映されません。

保守担当者向けメモ

  • 負のキャッシュ（InvalidOid の保存）を行っているため、存在しないファイル番号に対する繰り返し検索もキャッシュヒットします。
  • pg_class を開く処理がキャッシュ無効化を誘発する可能性があるため、キャッシュへの登録はカタログ検索完了後に行う設計になっています。
