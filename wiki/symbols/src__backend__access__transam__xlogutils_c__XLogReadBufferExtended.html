XLogReadBufferExtended

定義

  • 種別: function
  • 定義箇所: src/backend/access/transam/xlogutils.c:459

目的

WALリプレイ中に、指定されたリレーションのブロックをバッファキャッシュへ読み込みます。必要に応じて物理ファイルの拡張や、存在しないページへの参照の記録を行います。

シグネチャまたは形状

Buffer XLogReadBufferExtended(RelFileLocator rlocator, ForkNumber forknum,
                              BlockNumber blkno, ReadBufferMode mode,
                              Buffer recent_buffer)

入力と出力

  • **入力**:
  • rlocator: 対象リレーションの識別子。
  • forknum: 対象のフォーク番号。
  • blkno: 読み込むブロック番号（P_NEW は不可）。
  • mode: 読み込みモード（ReadBufferMode）。
  • recent_buffer: 直前に使用したバッファ（最適化用）。
  • **出力**:
  • 成功時: 読み込まれたブロックの Buffer。
  • 失敗時: InvalidBuffer（ページが存在しない場合など）。

前提条件と不変条件

  • 本関数はリカバリプロセス中（InRecovery）に呼び出されることを前提としています。
  • blkno は P_NEW 以外である必要があります。
  • リカバリ中であるため、他のバックエンドによる同時書き込みは発生しないと仮定し、PageIsNew のチェックをロックなしで行います。

アルゴリズムまたは振る舞い

1. **高速パス**: recent_buffer が有効かつ mode が RBM_NORMAL の場合、ReadRecentBuffer を試行して成功すれば即座に返却します。 2. **ストレージ準備**: smgropen でリレーションを開き、smgrcreate でファイルが存在しない場合に作成します。 3. **読み込み/拡張**:

  • ページが既存の場合: ReadBufferWithoutRelcache で読み込みます。
  • ページが範囲外の場合: mode が RBM_NORMAL なら無効ページとして記録し InvalidBuffer を返します。それ以外（かつリカバリ中）であれば ExtendBufferedRelTo でファイルを拡張します。

    • 4. **初期化チェック**: mode が RBM_NORMAL の場合、ページが初期化済みか確認し、未初期化であれば無効ページとして記録して InvalidBuffer を返します。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/access/transam/xlogrecovery.c, src/backend/access/transam/xlogutils.c, src/backend/access/hash/hashovfl.c, src/backend/storage/freespace/freespace.c 等から呼び出されます。
  • **呼び出し先**: smgropen, smgrcreate, smgrnblocks, ReadBufferWithoutRelcache, ExtendBufferedRelTo, log_invalid_page, BufferGetPage, PageIsNew, ReleaseBuffer 等。

関連データ構造

  • RelFileLocator: リレーションの物理的識別子。
  • Buffer: バッファキャッシュのハンドル。
  • SMgrRelation: ストレージマネージャの管理構造体。

失敗モードと境界条件

  • blkno が P_NEW の場合: Assert により停止します。
  • ページが存在しない場合: mode に応じて log_invalid_page を呼び出し、InvalidBuffer を返します。
  • ページが未初期化の場合: log_invalid_page を呼び出し、バッファを解放して InvalidBuffer を返します。

保守担当者向けメモ

  • 本関数はリカバリ専用のユーティリティであり、通常のバックエンドプロセスから呼び出すと予期せぬ動作を引き起こす可能性があります。
  • smgrcreate を呼び出すことで、削除予定のリレーションに対する書き込みが発生した場合でも、リカバリ中に整合性を保つ設計となっています。
