xl_btree_reuse_page

定義

  • 種別: struct
  • 定義箇所: src/include/access/nbtxlog.h:186

目的

B-treeインデックスにおいて、ページが再利用（リサイクル）された際にその事実をWAL（Write-Ahead Log）に記録するための構造体です。

シグネチャまたは形状

typedef struct xl_btree_reuse_page
{
    RelFileLocator locator;
    BlockNumber block;
    FullTransactionId snapshotConflictHorizon;
    bool        isCatalogRel;
} xl_btree_reuse_page;

入力と出力

  • **入力**: ページ再利用時に、対象リレーションの識別子（locator）、対象ブロック番号（block）、スナップショット競合の境界（snapshotConflictHorizon）、およびカタログリレーションか否かのフラグ（isCatalogRel）を保持します。
  • **出力**: WALレコードの一部として書き出され、リカバリ時にページ再利用の整合性を保証するために使用されます。

前提条件と不変条件

  • isCatalogRel は、スタンバイ側での論理デコーディング中に発生するリカバリ競合を適切に処理するために使用されます。

アルゴリズムまたは振る舞い

  • ページが実際にリサイクルされるタイミングで、この構造体を含むWALレコードが生成されます。
  • リカバリ処理（btree_redo）において、このレコードを読み込むことで、特定のブロックが再利用されたことをシステムが認識し、整合性を維持します。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/access/nbtree/nbtpage.c（ページリサイクル処理時）
  • **呼び出し先**: src/backend/access/nbtree/nbtxlog.c（REDO処理）、src/backend/access/rmgrdesc/nbtdesc.c（ログデコード処理）

関連データ構造

  • SizeOfBtreeReusePage: 本構造体のサイズを計算するためのマクロ（isCatalogRel フィールドまで）。

失敗モードと境界条件

  • 構造体内の各フィールドが正しく設定されていない場合、リカバリ時にスタンバイ側で論理デコーディングの競合解決が正しく行われない可能性があります。

保守担当者向けメモ

  • 本構造体は、B-treeの物理的なページ管理とWALによるリカバリの整合性を結びつける重要な役割を担っています。変更時は nbtxlog.c のREDOロジックへの影響を確認してください。
