PathCostComparison

定義

  • 種別: enum
  • 定義箇所: src/backend/optimizer/util/pathnode.c:37

目的

クエリプランナにおいて、2つのパス間のコスト比較結果を表現するために使用される列挙型です。

シグネチャまたは形状

typedef enum {
    COSTS_EQUAL,        /* パスのコストは曖昧に等しい */
    COSTS_BETTER1,      /* 1番目のパスが2番目より安価 */
    COSTS_BETTER2,      /* 2番目のパスが1番目より安価 */
    COSTS_DIFFERENT     /* コストに関して支配関係にない */
} PathCostComparison;

入力と出力

  • 本シンボル自体は列挙型であり、関数ではありません。
  • コスト比較関数（compare_path_costs_fuzzily 等）の戻り値として使用され、プランナがパスの優劣を判定する際の論理的な状態を返します。

前提条件と不変条件

  • COSTS_EQUAL は、浮動小数点誤差を考慮した「ファジー比較」の結果として返されます。
  • COSTS_DIFFERENT は、一方のパスが他方をコスト面で完全に支配していない（例：起動コストは低いが総コストは高いなど）状態を示します。

アルゴリズムまたは振る舞い

  • プランナは、この列挙値を用いて add_path 内でパスの選別を行います。
  • 支配関係にあるパス（一方が他方より安価かつ効率的）を特定し、不要なパスを破棄するための判断基準となります。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/optimizer/util/pathnode.c 内のコスト比較ロジック（compare_path_costs_fuzzily など）から返されます。
  • **呼び出し先**: add_path などのパス管理関数が、この値を受け取り、パスリストの更新や削除の制御フローを決定します。

関連データ構造

  • Path: プランナが扱うアクセスパスの基本構造体。
  • RelOptInfo: パスリストを保持するリレーション情報構造体。

失敗モードと境界条件

  • 浮動小数点数の比較において STD_FUZZ_FACTOR を用いた許容誤差範囲内にある場合、厳密な大小関係ではなく COSTS_EQUAL が選択されます。

保守担当者向けメモ

  • 本列挙型は pathnode.c 内のコスト比較ロジックと密結合しています。比較ロジックの拡張や変更を行う際は、この列挙値の定義が網羅的であるか確認してください。
