RunningTransactionsData

定義

  • 種別: struct
  • 定義箇所: src/include/storage/standby.h:86

目的

ホットスタンバイ環境において、プライマリサーバーで実行中のトランザクション状態をスタンバイ側で再現・保持するために使用されます。スタンバイ起動時のスナップショット作成や、リカバリ中のクエリ競合判定の基礎データとなります。

シグネチャまたは形状

typedef struct RunningTransactionsData
{
	int			xcnt;			/* xids[] 内のトランザクションID数 */
	int			subxcnt;		/* xids[] 内のサブトランザクションID数 */
	subxids_array_status subxid_status;
	TransactionId nextXid;		/* プライマリの nextXid */
	TransactionId oldestRunningXid; /* 実行中の最古のXID */
	TransactionId oldestDatabaseRunningXid; /* 現在のデータベース内での最古のXID */
	TransactionId latestCompletedXid;	/* xmax決定用の最新完了XID */
	TransactionId *xids;		/* 実行中の(サブ)トランザクションID配列 */
} RunningTransactionsData;

入力と出力

  • **入力**: プライマリサーバーから送信されるWALレコード（xl_running_xacts等）から情報を読み取り、本構造体に格納します。
  • **出力**: スタンバイ側でのスナップショット生成や、ResolveRecoveryConflictWith... 関数群による競合解決の判断材料として利用されます。

前提条件と不変条件

  • oldestRunningXid は oldestXmin とは異なります。
  • xids 配列には、実行中のトランザクションおよびサブトランザクションのIDが格納されます。
  • subxid_status は、サブトランザクションID配列の状態（オーバーフローの有無など）を保持します。

アルゴリズムまたは振る舞い

1. スタンバイプロセスがWALから実行中トランザクション情報を読み込み、本構造体を初期化します。 2. リカバリ中にクエリが実行される際、この構造体を参照して、プライマリ側の変更と競合するかを判定します。 3. 競合が検出された場合、ResolveRecoveryConflictWith... 関数が呼び出され、待機またはクエリの中断が行われます。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/access/transam/xlog.c および src/backend/storage/ipc/standby.c にて、WAL適用時の状態保持用としてインスタンス化されます。
  • **呼び出し先**: src/backend/storage/ipc/procarray.c にて、スナップショットのオーバーフロー判定やトランザクション管理のロジックから参照されます。

関連データ構造

  • RunningTransactions: RunningTransactionsData * 型のエイリアス。
  • subxids_array_status: サブトランザクションID配列の状態を示す列挙型。
  • xl_running_xacts: WALレコードとして書き出される際の対応構造体。

失敗モードと境界条件

  • subxid_status がオーバーフローを示している場合、サブトランザクションの追跡が不完全になる可能性があります。
  • xids 配列のメモリ管理は、リカバリプロセスのライフサイクルに依存します。

保守担当者向けメモ

  • 本構造体は snapmgr.h で管理される通常のスナップショットとは独立しています。
  • スタンバイ特有のデータ構造であるため、通常のトランザクション管理ロジックと混同しないよう注意してください。
  • max_standby_archive_delay 等のGUCパラメータが、本構造体に基づく競合解決の待機時間に影響を与えます。
