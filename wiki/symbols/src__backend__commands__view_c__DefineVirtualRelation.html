DefineVirtualRelation

定義

  • 種別: function
  • 定義箇所: src/backend/commands/view.c:44

目的

PostgreSQLにおけるビュー（CREATE VIEW / CREATE OR REPLACE VIEW）のリレーション作成および更新を管理する。

シグネチャまたは形状

ObjectAddress DefineVirtualRelation(RangeVar *relation, List *tlist, bool replace, List *options, Query *viewParse)

入力と出力

  • **入力**:
  • relation: ビューの識別子（RangeVar）。
  • tlist: ビューのSELECTリストに基づくターゲットリスト。
  • replace: CREATE OR REPLACE モードか否かを示すフラグ。
  • options: ビューのオプションリスト。
  • viewParse: ビューの定義クエリ（Query）。
  • **出力**:
  • 作成または更新されたビューの ObjectAddress を返す。

前提条件と不変条件

  • replace が真の場合、既存のリレーションが RELKIND_VIEW である必要がある。
  • 既存のビューを置換する場合、新しい列リストは古い列リストの初期プレフィックスと一致しなければならない（checkViewColumns による検証）。
  • テンポラリビューはテンポラリビューとしてのみ置換可能である。

アルゴリズムまたは振る舞い

1. **列定義の生成**: tlist から ColumnDef ノードのリストを作成し、照合順序（collation）の妥当性を検証する。 2. **名前空間の解決**: RangeVarGetAndCheckCreationNamespace を呼び出し、作成先スキーマの権限確認とロックを行う。 3. **置換処理 (replace == true)**:

  • 既存のビューをロックし、CheckTableNotInUse で使用中ではないことを確認する。
  • checkViewColumns で列構成の互換性を検証する。
  • 新しい列が追加されている場合、AlterTableInternal を使用して列を追加する。
  • StoreViewQuery を呼び出し、新しいクエリをルールとして保存する。
  • AlterTableInternal を使用してビューのオプションを更新する。
  • 拡張機能への依存関係を記録する。

    • 4. **新規作成 (replace == false)**:

  • ソースコード上では、この後の処理（else ブロック以降）に続く。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/commands/view.c 内の DefineView から呼び出されることが確認されている。
  • **呼び出し先**: makeColumnDef, type_is_collatable, RangeVarGetAndCheckCreationNamespace, relation_open, CheckTableNotInUse, BuildDescForRelation, checkViewColumns, AlterTableInternal, StoreViewQuery, recordDependencyOnCurrentExtension, relation_close 等。

関連データ構造

  • RangeVar: ビューの識別情報。
  • ColumnDef: ビューの列定義。
  • Query: ビューの定義クエリ。
  • ObjectAddress: 作成されたオブジェクトの識別子。

失敗モードと境界条件

  • 照合順序が決定できない場合、ERRCODE_INDETERMINATE_COLLATION でエラー終了する。
  • 対象がビューではない場合、ERRCODE_WRONG_OBJECT_TYPE でエラー終了する。
  • ビューが使用中の場合、CheckTableNotInUse によりエラー終了する。
  • 列の型や名前の変更など、互換性のない変更を試みた場合、checkViewColumns によりエラー終了する。

保守担当者向けメモ

  • 本関数は CREATE OR REPLACE VIEW の複雑な更新ロジックを内包している。
  • ALTER TABLE のインフラ（AlterTableInternal）を再利用しているため、ビュー特有の制約とテーブル操作の副作用に注意すること。
  • StoreViewQuery の呼び出し順序は、ビューのオプション更新よりも前である必要がある（WITH CHECK OPTION 等の依存関係のため）。
