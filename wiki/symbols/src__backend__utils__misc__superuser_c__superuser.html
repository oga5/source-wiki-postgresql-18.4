superuser

定義

  • 種別: function
  • 定義箇所: src/backend/utils/misc/superuser.c:45

目的

現在のセッションユーザーがスーパーユーザー権限を持っているかどうかを判定します。システムカタログ pg_authid を直接参照する代わりに本関数を使用することで、キャッシュの利用やシングルユーザーモード時の例外処理を透過的に行います。

シグネチャまたは形状

bool superuser(void);

入力と出力

  • 入力: なし（内部で GetUserId() を呼び出し、現在のセッションユーザーIDを取得）。
  • 出力: bool 型。ユーザーがスーパーユーザーであれば true、そうでなければ false を返します。

前提条件と不変条件

  • GetUserId() が有効なセッションユーザーIDを返す環境で呼び出される必要があります。
  • 権限判定はプロセスローカルのキャッシュ（last_roleid 等）に依存しており、システムカタログの変更時にはコールバックを通じて無効化されます。

アルゴリズムまたは振る舞い

1. 現在のセッションユーザーIDを GetUserId() で取得します。 2. 取得したIDを引数として superuser_arg() を呼び出します。 3. superuser_arg() が返す判定結果をそのまま呼び出し元へ返却します。

呼び出し元と呼び出し先

  • 呼び出し先: superuser_arg
  • 呼び出し元（直接的な証拠）:
  • contrib/dblink/dblink.c
  • contrib/pageinspect/brinfuncs.c
  • contrib/basebackup_to_shell/basebackup_to_shell.c

関連データ構造

  • pg_authid: ロール情報を保持するシステムカタログ。
  • last_roleid / last_roleid_is_super: 判定結果を高速化するための静的キャッシュ変数。

失敗モードと境界条件

  • シングルユーザーモード（!IsUnderPostmaster）では、管理者ユーザーが削除された場合でもシステムがロックアウトされないよう、特定のブートストラップIDに対して true を返す例外処理が superuser_arg 内で行われます。

保守担当者向けメモ

  • コードベース全体において、pg_authid.rolsuper を直接参照することは推奨されません。必ず本関数（または superuser_arg）を使用してください。
  • キャッシュはプロセスローカルです。システムカタログの更新は RoleidCallback を通じて検知され、プロセス内のキャッシュがクリアされます。
