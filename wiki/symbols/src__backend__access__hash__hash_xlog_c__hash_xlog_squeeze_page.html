hash_xlog_squeeze_page

定義

  • 種別: function
  • 定義箇所: src/backend/access/hash/hash_xlog.c:626

目的

ハッシュインデックスのオーバーフローページを解放（Squeeze）する操作をWALリプレイ時に再適用し、インデックスの物理的な整合性を復元する。

シグネチャまたは形状

static void hash_xlog_squeeze_page(XLogReaderState *record)

入力と出力

  • **入力**: XLogReaderState *record（リプレイ対象のWALレコード）。
  • **出力**: なし（副作用としてバッファの状態を更新）。

前提条件と不変条件

  • リプレイ対象のWALレコードは xl_hash_squeeze_page 構造体を含む必要がある。
  • 操作の開始時に、プライマリバケットページに対してクリーンアップロックを取得し、スキャンとの競合を防止する。
  • 挿入されたタプル数（ninserted）は、WALレコード内の記録（xldata->ntups）と一致しなければならない。

アルゴリズムまたは振る舞い

1. **ロック取得**: プライマリバケットページを読み込み、クリーンアップロックを確保する。 2. **タプル移動**: writebuf（移動先ページ）に対し、WALレコードから読み取ったタプルを PageAddItem で追加する。 3. **ページ更新**:

  • 移動先ページが解放されるオーバーフローページの直前である場合、hasho_nextblkno を更新する。
  • 解放されたオーバーフローページを _hash_pageinit で初期化し、LH_UNUSED_PAGE フラグを立てる。
  • 解放されたページの前後にあるページ（prevbuf, nextbuf）のリンク情報（hasho_nextblkno, hasho_prevblkno）を修正する。

    • 4. **永続化**: 変更された各バッファに対してLSNを設定し、MarkBufferDirty を呼び出す。 5. **解放**: 取得したすべてのバッファのロックを解除する。

呼び出し元と呼び出し先

  • **呼び出し元**: hash_redo（src/backend/access/hash/hash_xlog.c:1098 にて確認）。
  • **呼び出し先**: XLogReadBufferForRedoExtended, XLogReadBufferForRedo, _hash_pageinit, PageAddItem, MarkBufferDirty, UnlockReleaseBuffer 等。

関連データ構造

  • xl_hash_squeeze_page: WALレコードのデータ構造。
  • HashPageOpaque: ハッシュインデックスページのメタデータ（hasho_nextblkno 等）。
  • Buffer: PostgreSQLのバッファ管理構造体。

失敗モードと境界条件

  • PageAddItem が失敗した場合、elog(ERROR) によりリカバリ処理が中断される。
  • タプルが存在しない場合、is_prim_bucket_same_wrt または is_prev_bucket_same_wrt が真であることをアサーションで検証する。

保守担当者向けメモ

  • 本関数は冪等（idempotent）である必要がある。
  • 通常のインデックス操作とは異なり、リプレイ中は同時アクセスがないため、メタページやビットマップページの更新は本関数内では行わず、呼び出し元または別のリプレイ処理に委ねられている。
