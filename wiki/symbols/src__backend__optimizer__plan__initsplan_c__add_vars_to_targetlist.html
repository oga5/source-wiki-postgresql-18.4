add_vars_to_targetlist

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/plan/initsplan.c:293

目的

クエリの最適化過程において、特定の変数（Var または PlaceHolderVar）がどのリレーションで必要とされるかを追跡し、必要に応じてリレーションのターゲットリスト（reltarget）へ変数を追加します。

シグネチャまたは形状

void add_vars_to_targetlist(PlannerInfo *root, List *vars, Relids where_needed)

入力と出力

  • **入力**:
  • root: プランナのグローバル情報構造体。
  • vars: 追加対象となる Var または PlaceHolderVar ノードのリスト。
  • where_needed: その変数が使用されるリレーションの集合（Relids）。
  • **出力**: なし（副作用として root 内の RelOptInfo や PlaceHolderInfo を更新）。

前提条件と不変条件

  • where_needed は空であってはならない（Assert(!bms_is_empty(where_needed))）。
  • Var の場合、その varno に対応する RelOptInfo が root 内に存在する必要がある。

アルゴリズムまたは振る舞い

1. vars リストを反復処理し、各ノードの型に応じて以下の処理を行う。 2. **Var ノードの場合**:

  • 対応する RelOptInfo を取得し、where_needed が既にそのリレーションの attr_needed に含まれているか確認する。
  • 含まれていない場合、その変数を reltarget->exprs に追加する。この際、varnullingrels を NULL にクリアする（外部結合によるヌル化情報は上位階層で再構築されるため）。
  • rel->attr_needed を更新し、変数がどこで必要かを記録する。

    • 3. **PlaceHolderVar ノードの場合**:

  • find_placeholder_info を用いて対応する PlaceHolderInfo を取得し、ph_needed に where_needed を追加する。

    • 4. 上記以外のノード型が渡された場合はエラー（elog(ERROR)）を発生させる。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/optimizer/path/equivclass.c
  • src/backend/optimizer/plan/initsplan.c 内の複数箇所（deconstruct_jointree 関連処理など）
  • **呼び出し先**:
  • find_base_rel
  • find_placeholder_info
  • copyObject
  • bms_is_subset, bms_add_members, bms_is_empty などのビットマップセット操作関数

関連データ構造

  • PlannerInfo: プランナの全体状態を保持。
  • RelOptInfo: ベースリレーションの情報を保持。reltarget および attr_needed を含む。
  • Var: クエリ内の変数参照。
  • PlaceHolderVar: 外部結合等で生成されるプレースホルダ変数。
  • PlaceHolderInfo: プレースホルダ変数の必要性情報を管理。

失敗モードと境界条件

  • 未知のノード型が vars リストに含まれている場合、elog(ERROR) により処理が中断される。
  • Var の属性番号（varattno）がリレーションの範囲外である場合、Assert によりデバッグビルドで停止する。

保守担当者向けメモ

  • 本関数は varnullingrels をクリアしてターゲットリストに追加する設計となっている。これは、スキャンレベルではヌル化が発生しないという前提に基づいている。
  • reltarget のコストや幅（width）の再計算は本関数内では行われず、後続のフェーズに委ねられている。
