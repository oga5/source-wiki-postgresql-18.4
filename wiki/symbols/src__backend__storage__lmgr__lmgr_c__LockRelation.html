LockRelation

定義

  • 種別: function
  • 定義箇所: src/backend/storage/lmgr/lmgr.c:245

目的

指定されたリレーションに対して指定されたモードでロックを取得し、必要に応じてリレーションキャッシュの整合性を維持するためのインバリデーション処理を実行する。

シグネチャまたは形状

void LockRelation(Relation relation, LOCKMODE lockmode)

入力と出力

  • **入力**:
  • relation: ロック対象のリレーション構造体。
  • lockmode: 取得するロックのモード（LOCKMODE）。
  • **出力**: なし（戻り値は void）。

前提条件と不変条件

  • relation 引数は有効なリレーション構造体を指している必要がある。
  • ロック取得成功後、LOCKACQUIRE_ALREADY_CLEAR 以外の結果が返された場合、キャッシュの整合性を保証するために AcceptInvalidationMessages() が呼び出される。

アルゴリズムまたは振る舞い

1. relation の rd_lockInfo からデータベースIDとリレーションIDを抽出し、SET_LOCKTAG_RELATION マクロを用いて LOCKTAG を構築する。 2. LockAcquireExtended を呼び出し、共有メモリ上のロックテーブルに対してロックを要求する。 3. ロック取得の結果（res）を確認する。 4. res が LOCKACQUIRE_ALREADY_CLEAR でない場合、AcceptInvalidationMessages() を呼び出して保留中のインバリデーションメッセージを処理し、MarkLockClear を用いてロック状態を更新する。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/catalog/dependency.c:1492
  • src/backend/catalog/index.c:1001
  • **呼び出し先**:
  • LockAcquireExtended
  • AcceptInvalidationMessages
  • MarkLockClear

関連データ構造

  • Relation: ロック対象のメタデータ（rd_lockInfo）を保持。
  • LOCKTAG: ロック対象を一意に識別するためのタグ。
  • LOCALLOCK: プロセスローカルなロック管理構造体。

失敗モードと境界条件

  • ロック取得時にデッドロックが発生した場合、LockAcquireExtended の内部ロジックによりエラーが発生する可能性がある。
  • メモリ不足や共有メモリの枯渇など、低レベルなロック管理機能が失敗した場合の挙動は LockAcquireExtended に依存する。

保守担当者向けメモ

  • 本関数は LockRelationOid と同様のインバリデーションロジックを共有している。修正を行う際は両者の整合性に注意すること。
  • ロック取得後の AcceptInvalidationMessages() の呼び出しは、ロック取得中に他セッションによってスキーマ変更が行われた場合に備えた必須の処理である。これを省略すると、古い relcache エントリによる不整合が発生するリスクがある。
