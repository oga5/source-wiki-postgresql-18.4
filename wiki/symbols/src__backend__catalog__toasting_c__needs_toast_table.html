needs_toast_table

定義

  • 種別: function
  • 定義箇所: src/backend/catalog/toasting.c:400

目的

指定されたリレーション（テーブル）に対して、TOASTテーブルを作成する必要があるかどうかを判定する述語関数です。

シグネチャまたは形状

static bool needs_toast_table(Relation rel);

入力と出力

  • **入力**: Relation rel （判定対象のリレーション構造体）
  • **出力**: bool （TOASTテーブルが必要な場合は true、不要な場合は false）

前提条件と不変条件

  • rel は有効なリレーション構造体を指している必要があります。
  • IsBootstrapProcessingMode() が真の場合、一部の制約（共有テーブルやカタログテーブルに関する制限）がバイパスされます。

アルゴリズムまたは振る舞い

以下の順序で判定を行い、いずれかの条件に合致した場合は false を返します。 1. **パーティションテーブル**: relkind が RELKIND_PARTITIONED_TABLE の場合は false を返します。 2. **共有テーブル**: relisshared が真かつブートストラップモードでない場合は false を返します。 3. **カタログテーブル**: IsCatalogRelation(rel) が真かつブートストラップモードでない場合は false を返します。 4. **その他**: 上記のいずれにも該当しない場合、table_relation_needs_toast_table(rel) を呼び出し、テーブルアクセスメソッド（AM）側の判断を返します。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/catalog/toasting.c 内のテーブル生成・管理ロジックから参照されています。
  • **呼び出し先**:
  • table_relation_needs_toast_table(rel) （テーブルアクセスメソッドの判定関数）

関連データ構造

  • Relation: リレーション情報を保持する構造体。
  • Form_pg_class: rel->rd_rel を通じてアクセスされる、テーブルの属性情報。

失敗モードと境界条件

  • initdb 実行中（ブートストラップモード）と通常運用時で、カタログテーブルや共有テーブルに対する判定ロジックが異なります。
  • パーティションテーブルに対しては、一律でTOASTテーブルの作成を抑制します。

保守担当者向けメモ

  • 本関数はTOASTテーブル作成のゲートキーパーです。新しいテーブルタイプやストレージ要件を追加する際は、この判定ロジックが適切か確認してください。
  • table_relation_needs_toast_table への委譲により、AMごとの柔軟な判定が可能になっています。
