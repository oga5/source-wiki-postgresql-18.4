ExecSubqueryScan

定義

  • 種別: function
  • 定義箇所: src/backend/executor/nodeSubqueryscan.c:82

目的

FROM 句に記述されたサブクエリを実行し、その結果をスキャンして上位ノードへ供給する。

シグネチャまたは形状

TupleTableSlot *ExecSubqueryScan(PlanState *pstate)

入力と出力

  • **入力**: PlanState *pstate（実行中のサブクエリノードの状態）
  • **出力**: TupleTableSlot *（サブクエリから取得されたタプル）

前提条件と不変条件

  • pstate は SubqueryScanState 型にキャスト可能であること。
  • ExecScan を介して呼び出される標準的なスキャンインターフェースとして動作する。

アルゴリズムまたは振る舞い

1. pstate を SubqueryScanState にキャストする。 2. ExecScan 関数を呼び出し、スキャン処理を委譲する。 3. ExecScan 内で SubqueryNext を用いて子プランからタプルを取得し、SubqueryRecheck を用いて再チェックを行う。 4. 取得したタプルに対してフィルタリング（qual）およびプロジェクションを適用し、結果を返す。

呼び出し元と呼び出し先

  • **呼び出し元**: 実行エンジン内の上位ノード（ExecProcNode を経由）。
  • **呼び出し先**:
  • ExecScan（スキャン基盤）
  • SubqueryNext（タプル取得の実務ルーチン）
  • SubqueryRecheck（再チェックルーチン）
  • castNode（型変換）

関連データ構造

  • SubqueryScanState: サブクエリ実行時の状態を保持する構造体。
  • PlanState: 実行プランのノード状態。
  • TupleTableSlot: タプルを保持するスロット。

失敗モードと境界条件

  • 子プランがタプルを返さない場合、ExecScan の仕様に従い NULL または空のスロットを返す。
  • 実行中のエラーは PostgreSQL の標準的なエラーハンドリング機構に依存する。

保守担当者向けメモ

  • SubqueryNext は ExecCopySlot() を行わず、子プランのタプルスロットを直接返すことでオーバーヘッドを最小化している。
  • SubPlan と SubqueryScan のコード統合は長年の課題であるが、現状では別々の実装として維持されている。
