processPendingPage

定義

  • 種別: function
  • 定義箇所: src/backend/access/gin/ginfast.c:708

目的

GINインデックスのペンディングリスト内のページをスキャンし、格納されているインデックスエントリを抽出して、BuildAccumulator を介してメインのインデックス構造へ再挿入するための準備を行う。

シグネチャまたは形状

static void processPendingPage(BuildAccumulator *accum, KeyArray *ka, Page page, OffsetNumber startoff)

入力と出力

  • **入力**:
  • accum: 挿入データを集約するための BuildAccumulator 構造体。
  • ka: キーを一時保持するための KeyArray 構造体。
  • page: 処理対象のペンディングリストページ。
  • startoff: ページ内の処理開始オフセット。
  • **出力**: なし（副作用として accum にデータが蓄積される）。

前提条件と不変条件

  • page は有効なGINペンディングリストページであること。
  • ka は呼び出し前に初期化されていること。
  • accum は有効な状態であること。

アルゴリズムまたは振る舞い

1. ka->nvalues を 0 にリセットする。 2. ページ内の各アイテムを startoff から順に走査する。 3. 各アイテム（IndexTuple）のヒープTIDおよび属性番号（attrnum）を確認する。 4. ヒープTIDまたは属性番号が変更された場合、それまでに蓄積されたキーを ginInsertBAEntries で accum に渡してフラッシュし、ka をリセットする。 5. 現在のアイテムからキーとカテゴリを抽出し、addDatum を用いて ka に追加する。 6. ループ終了後、残存しているキーを ginInsertBAEntries で accum に渡す。

関連データ構造

  • BuildAccumulator: 挿入データの集約状態を保持。
  • KeyArray: 抽出されたキーとカテゴリの配列を管理。
  • IndexTuple: GINインデックスの各エントリ。

失敗モードと境界条件

  • ページ内にアイテムが存在しない場合、maxoff のチェックにより処理はスキップされる。
  • ヒープTIDや属性番号の境界で正しく ginInsertBAEntries が呼び出されることが、データの整合性維持に不可欠である。

保守担当者向けメモ

  • 本関数は ginInsertCleanup の一部として機能し、ペンディングリストからメインインデックスへのデータ移行の橋渡しを行う。
  • ginInsertBAEntries は単一のヒープタプルおよび単一カラムに対するキーの集合を一度に処理するため、境界条件の判定ロジックを変更する際は注意が必要である。
