record_in

定義

  • 種別: function
  • 定義箇所: src/backend/utils/adt/rowtypes.c:73

目的

複合型（Composite Types）のテキスト形式入力を解析し、HeapTupleHeader 形式の内部表現へ変換します。

シグネチャまたは形状

Datum record_in(PG_FUNCTION_ARGS)

入力と出力

  • **入力**:
  • PG_GETARG_CSTRING(0): 入力文字列。
  • PG_GETARG_OID(1): 対象となる複合型の OID。
  • PG_GETARG_INT32(2): 型修飾子（typmod）。
  • **出力**: 変換されたタプルを表す HeapTupleHeader（Datum として返却）。

前提条件と不変条件

  • check_stack_depth() により、再帰的なレコード定義に対するスタックオーバーフロー保護が行われます。
  • 匿名レコード（RECORDOID）かつ型修飾子が負の場合、入力はサポートされずエラーとなります。
  • 実行時の型情報（TupleDesc）に基づき、フィールドごとの I/O 関数を fn_extra にキャッシュして再利用します。

アルゴリズムまたは振る舞い

1. lookup_rowtype_tupdesc を呼び出し、対象型の構造を取得します。 2. fcinfo->flinfo->fn_extra を確認し、型定義に対応するキャッシュ（RecordIOData）を初期化または更新します。 3. 入力文字列を ( で開始し、各フィールドを , で区切って順次解析します。 4. フィールド値が引用符で囲まれている場合、エスケープ処理（\" や \\）を適切に解釈します。 5. 削除された列（attisdropped）はスキップし、NULL として扱います。 6. 解析した各フィールドの値を、対応する型の入力関数を用いて変換し、最終的にタプルを構築します。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/pl/plpython/plpy_typeio.c（Python 文字列から複合型への変換）
  • SQL インターフェース経由で複合型の入力時に呼び出されます。
  • **呼び出し先**:
  • lookup_rowtype_tupdesc
  • 各フィールド型の入力関数（キャッシュ経由）
  • check_stack_depth

関連データ構造

  • RecordIOData: フィールドごとの I/O 情報を保持するキャッシュ構造体。
  • ColumnIOData: 各列の型情報や関数ポインタを保持。
  • TupleDesc: 複合型の構造定義。

失敗モードと境界条件

  • **不正な形式**: 括弧の欠如、予期せぬ入力終了、または列数が一致しない場合にエラーを発生させます。
  • **匿名レコード**: 型修飾子が指定されていない匿名レコードの入力は、現在の実装ではサポートされていません。
  • **スタック深度**: 再帰的なレコード定義が深すぎる場合、check_stack_depth によりエラーとなります。

保守担当者向けメモ

  • fn_extra は関数呼び出しのライフサイクルに依存するため、型定義が変更される環境下でのキャッシュの有効性に注意してください。
  • 削除された列（attisdropped）の処理は、物理的な列位置ではなく論理的な順序に基づいている点に留意してください。
