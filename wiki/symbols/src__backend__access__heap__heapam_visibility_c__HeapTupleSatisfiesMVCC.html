HeapTupleSatisfiesMVCC

定義

  • 種別: function
  • 定義箇所: src/backend/access/heap/heapam_visibility.c:959

目的

PostgreSQLのMVCC（多版同時実行制御）ルールに基づき、特定のタプルが指定されたスナップショットから可視であるかを判定します。

シグネチャまたは形状

bool HeapTupleSatisfiesMVCC(HeapTuple htup, Snapshot snapshot, Buffer buffer)

入力と出力

  • **入力**:
  • htup: 判定対象のタプル構造体。
  • snapshot: 可視性判定の基準となるスナップショット。
  • buffer: タプルが格納されているバッファ（ヒントビット更新用）。
  • **出力**:
  • bool: タプルが可視であれば true、不可視であれば false を返します。

前提条件と不変条件

  • 呼び出し元は、snapshot が登録済み（regd_count > 0 または active_count > 0）であることを保証する必要があります。
  • htup->t_self が有効であり、htup->t_tableOid が設定されている必要があります。
  • 対象バッファに対して少なくとも共有ロックを保持している必要があります。

アルゴリズムまたは振る舞い

1. **Xminの検証**: タプルの挿入トランザクション（Xmin）の状態を確認します。

  • 未コミットかつ無効な場合は false を返します。
  • HEAP_MOVED_OFF や HEAP_MOVED_IN フラグがある場合、バイナリアップグレード用の特殊処理を行います。
  • 自トランザクションによる挿入の場合、コマンドID（curcid）を比較して可視性を判定します。
  • 他トランザクションの場合、スナップショットとCLOGを参照し、必要に応じて SetHintBits でヒントビットを更新します。

    • 2. **Xmaxの検証**: タプルの削除/更新トランザクション（Xmax）の状態を確認します。

  • HEAP_XMAX_INVALID または HEAP_XMAX_IS_LOCKED_ONLY の場合は可視とみなします。
  • HEAP_XMAX_IS_MULTI の場合はマルチトランザクションIDを解析し、更新者の状態を判定します。
  • 削除トランザクションがコミット済みであれば false、未コミットまたはアボート済みであれば true を返します。

    • 3. **ヒントビット更新**: 判定過程でトランザクションの確定状態が判明した場合、SetHintBits を呼び出し、タプルヘッダーの t_infomask を更新してバッファをダーティにします。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/access/heap/heapam_visibility.c 内の他の可視性判定関数（例: HeapTupleSatisfiesUpdate 等の内部から呼び出されるケースあり）。
  • src/backend/replication/logical/snapbuild.c。
  • **呼び出し先**:
  • TransactionIdIsCurrentTransactionId, XidInMVCCSnapshot, TransactionIdDidCommit（トランザクション状態確認）。
  • SetHintBits（ヒントビット更新）。
  • HeapTupleHeaderGetXvac, HeapTupleGetUpdateXid 等のアクセサ関数。

関連データ構造

  • HeapTuple: タプル本体。
  • Snapshot: 可視性判定の基準。
  • Buffer: バッファ管理構造体。
  • HeapTupleHeader: タプルヘッダー（t_infomask, Xmin, Xmax を保持）。

失敗モードと境界条件

  • **無効なタプル**: Xmin が無効な場合、即座に false を返します。
  • **レースコンディション**: コミット直後のトランザクション状態判定において、TransactionIdIsInProgress を TransactionIdDidCommit より先にチェックする順序を守ることで競合を回避しています。
  • **バイナリアップグレード**: HEAP_MOVED_OFF / HEAP_MOVED_IN フラグを持つ古い形式のタプルに対する特殊なパスが存在します。

保守担当者向けメモ

  • 本関数は、判定結果をキャッシュするためにヒントビットを更新します。この際、バッファがダーティになるため、呼び出し元は適切なロックを保持している必要があります。
  • XidInMVCCSnapshot のチェックは、TransactionIdDidCommit よりも前に行う必要があります。
