plpgsql_call_handler

定義

  • 種別: function
  • 定義箇所: src/pl/plpgsql/src/pl_handler.c:223

目的

PostgreSQL の手続き型言語 PL/pgSQL における関数実行のメインエントリーポイントです。SQL レベルでの関数呼び出し、トリガー、およびイベントトリガーの実行を仲介し、SPI（Server Programming Interface）接続、コンパイル、実行、リソース解放を統括します。

シグネチャまたは形状

Datum plpgsql_call_handler(PG_FUNCTION_ARGS)

入力と出力

  • **入力**: PG_FUNCTION_ARGS（FunctionCallInfo 構造体）。呼び出し元のコンテキスト情報を含みます。
  • **出力**: Datum 型。実行された関数またはトリガーの戻り値。イベントトリガーの場合は戻り値なし。

前提条件と不変条件

  • SPI_connect_ext により SPI コンテキストが確立されている必要があります。
  • PG_TRY/PG_FINALLY ブロックにより、エラー発生時でも use_count のデクリメントとリソース解放が保証されます。
  • nonatomic な呼び出し（CALL 文など）で requires_procedure_resowner が真の場合、専用の ResourceOwner が作成されます。

アルゴリズムまたは振る舞い

1. **SPI 接続**: fcinfo->context を確認し、アトミック性に応じて SPI_connect_ext を呼び出します。 2. **コンパイル**: plpgsql_compile を呼び出し、対象関数を内部表現（PLpgSQL_function）に変換または取得します。 3. **状態保護**: func->cur_estate を退避し、use_count をインクリメントして実行中の削除を防止します。 4. **実行**: PG_TRY ブロック内で、呼び出し種別（トリガー、イベントトリガー、通常関数）に応じて適切な実行関数（plpgsql_exec_trigger 等）を呼び出します。 5. **クリーンアップ**: PG_FINALLY ブロックにて use_count を戻し、作成された ResourceOwner があれば解放します。 6. **終了**: SPI_finish を呼び出し、SPI 接続を終了して結果を返します。

呼び出し元と呼び出し先

  • **呼び出し元**: PostgreSQL の関数マネージャ（PostgreSQL のコアエンジン）。
  • **呼び出し先**:
  • SPI_connect_ext, SPI_finish
  • plpgsql_compile
  • plpgsql_exec_trigger, plpgsql_exec_event_trigger, plpgsql_exec_function
  • ResourceOwnerCreate, ResourceOwnerDelete, ReleaseAllPlanCacheRefsInOwner

関連データ構造

  • PLpgSQL_function: コンパイルされた関数定義。
  • PLpgSQL_execstate: 実行時の状態管理構造体。
  • ResourceOwner: プロシージャ実行時のリソース管理用。
  • FunctionCallInfo: 関数呼び出しの引数およびコンテキスト情報。

失敗モードと境界条件

  • SPI_finish が失敗した場合、elog(ERROR) により異常終了します。
  • PG_TRY ブロック内で例外が発生した場合、PG_FINALLY によりリソースのクリーンアップが実行された後、例外が上位へ伝播します。
  • nonatomic な実行において ResourceOwner の解放に失敗すると、プロセス存続期間中のメモリリークが発生する可能性があります。

保守担当者向けメモ

  • 本関数はリソース管理の要であり、PG_TRY ブロックの前後にコードを追加する際は、リソースリークを避けるため細心の注意を払ってください。
  • nonatomic な実行時の ResourceOwner 管理は、トランザクションを跨ぐ可能性があるため、特に慎重な実装が求められます。
