flushCachedPage

定義

  • 種別: function
  • 定義箇所: contrib/bloom/blinsert.c:47

目的

インデックス構築中にメモリ上で保持されている Bloom フィルタのページデータを、新規に割り当てたディスク上のバッファへ書き出し、永続化します。

シグネチャまたは形状

static void flushCachedPage(Relation index, BloomBuildState *buildstate)

入力と出力

  • **入力**:
  • index: 対象となるインデックスの Relation 構造体。
  • buildstate: 構築中の状態を保持する BloomBuildState 構造体（ページデータを含む）。
  • **出力**: なし（副作用としてバッファへの書き込みと WAL 記録を行う）。

前提条件と不変条件

  • buildstate->data.data には、BLCKSZ サイズの有効なページイメージが格納されている必要があります。
  • BloomNewBuffer が正常に新しいバッファを割り当てられる状態である必要があります。

アルゴリズムまたは振る舞い

1. BloomNewBuffer(index) を呼び出し、インデックス用の新しいバッファを割り当てます。 2. GenericXLogStart(index) を呼び出し、GenericXLog による WAL 記録を開始します。 3. GenericXLogRegisterBuffer を使用して、割り当てたバッファを GENERIC_XLOG_FULL_IMAGE モードで登録し、ページポインタを取得します。 4. memcpy を用いて、buildstate->data.data の内容をバッファ上のページへコピーします。 5. GenericXLogFinish(state) を呼び出し、変更を WAL にコミットします。 6. UnlockReleaseBuffer(buffer) を呼び出し、バッファのロックを解放して参照を終了します。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • contrib/bloom/blinsert.c 内のインデックス構築処理（blbuild 関連のフロー）から呼び出されます。
  • **呼び出し先**:
  • BloomNewBuffer
  • GenericXLogStart
  • GenericXLogRegisterBuffer
  • GenericXLogFinish
  • UnlockReleaseBuffer

関連データ構造

  • Relation: インデックスのメタデータ管理に使用。
  • BloomBuildState: 構築中のページキャッシュを保持。
  • GenericXLogState: WAL 記録の管理に使用。

失敗モードと境界条件

  • BloomNewBuffer がディスク領域不足やその他の理由で失敗した場合、例外が発生し処理が中断されます。
  • GenericXLog 関連の関数が内部エラーを発生させた場合、トランザクションの整合性が優先され、処理が停止します。

保守担当者向けメモ

  • 本関数は GenericXLog を使用しているため、ページ全体のイメージを WAL に書き出します。パフォーマンス上の理由から、頻繁な呼び出しは避ける設計となっています。
  • BLCKSZ サイズのコピーを行うため、buildstate->data.data のメモリレイアウトがページ構造と一致していることが前提となります。
