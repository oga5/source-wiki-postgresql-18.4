ginxlogSegmentAction

定義

  • 種別: struct
  • 定義箇所: src/include/access/ginxlog.h:77

目的

GINインデックスのセグメントに対する操作を記述し、WAL（Write-Ahead Log）リカバリ時に特定のセグメントへ適用すべき変更内容を保持するために使用されます。

シグネチャまたは形状

typedef struct {
    uint8   segno;  /* 操作対象のセグメント番号 */
    char    type;   /* 操作の種類 */
    /* 続くデータは操作タイプに依存 */
} ginxlogSegmentAction;

入力と出力

  • **入力**: WALレコードの一部として、この構造体およびそれに続く可変長データが書き込まれます。
  • **出力**: リカバリ処理において、この構造体の情報を基にインデックスページの状態が更新されます。

前提条件と不変条件

  • 構造体直後に操作固有のデータが配置される可変長構造です。
  • INSERT および REPLACE アクションの場合、直後に GinPostingList 構造体が続きます。
  • ADDITEMS アクションの場合、直後に uint16 型の追加アイテム数と、それに続く ItemPointer の配列が配置されます。
  • DELETE アクションの場合、追加のデータは存在しません。

アルゴリズムまたは振る舞い

1. WAL再生時、ginxlogSegmentAction を読み取り、segno で対象セグメントを特定します。 2. type フィールドに基づき、後続のデータを解釈します。 3. 該当するアクション（挿入、置換、アイテム追加、削除）を実行し、インデックスの整合性を復元します。

呼び出し元と呼び出し先

  • **呼び出し元**: 提供されたソース情報からは特定の呼び出し元は特定できません。
  • **呼び出し先**: 提供されたソース情報からは特定の呼び出し先は特定できません。

関連データ構造

  • GinPostingList: INSERT / REPLACE アクション時に参照されます。
  • ItemPointer: ADDITEMS アクション時に参照されます。

失敗モードと境界条件

  • 構造体内のフィールド順序やサイズが変更されると、WALのバイナリ互換性が失われ、リカバリが失敗する可能性があります。
  • アライメントによるパディングが意図せず挿入されると、データ読み取り時にオフセットのズレが生じます。

保守担当者向けメモ

  • 本構造体は memcpy で直接読み書きされるため、フィールドの追加や変更を行う際は、WALのバイナリレイアウトへの影響を厳密に評価してください。
  • ginxlog 関連の構造体は、src/backend/access/gin/ginxlog.c のREDOロジックと密接に結合しています。変更時は必ず当該ファイルのREDO処理を確認してください。
