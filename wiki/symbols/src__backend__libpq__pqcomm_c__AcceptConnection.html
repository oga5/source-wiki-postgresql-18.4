AcceptConnection

定義

  • 種別: function
  • 定義箇所: src/backend/libpq/pqcomm.c:793

目的

サーバーソケットに対するクライアントからの接続要求を受け入れ、新しいソケットを確立してクライアントのアドレス情報を取得する。

シグネチャまたは形状

int AcceptConnection(pgsocket server_fd, ClientSocket *client_sock);

入力と出力

  • **入力**:
  • server_fd: 待機中のサーバーソケットのファイル記述子。
  • client_sock: 接続後のソケット情報およびリモートアドレスを格納するための ClientSocket 構造体へのポインタ。
  • **出力**:
  • STATUS_OK (0): 接続成功。
  • STATUS_ERROR (-1): 接続失敗。

前提条件と不変条件

  • client_sock->raddr.salen は、accept システムコールに渡す前に sizeof(client_sock->raddr.addr) で初期化される必要がある。

アルゴリズムまたは振る舞い

1. client_sock->raddr.salen をアドレス構造体のサイズで初期化する。 2. accept() を呼び出し、サーバーソケットから接続を抽出する。 3. accept() が PGINVALID_SOCKET を返した場合:

  • ereport を通じてログにエラーを出力する。
  • 0.1秒間スリープ (pg_usleep) して CPU リソースの過剰消費を防ぐ。
  • STATUS_ERROR を返す。

    • 4. 成功した場合は、client_sock->sock に新しいソケットを格納し、STATUS_OK を返す。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/postmaster/postmaster.c:1702
  • **呼び出し先**: accept() (システムコール), ereport(), pg_usleep()

関連データ構造

  • ClientSocket: 接続されたソケットとリモートアドレス (raddr) を保持する構造体。
  • pgsocket: ソケットのファイル記述子を表す型。

失敗モードと境界条件

  • **ファイル記述子不足**: カーネルのファイルテーブルが枯渇している場合、accept() が失敗する。この際、CPU 負荷を抑えるための待機処理が実行される。
  • **無効なソケット**: server_fd が無効な場合、accept() は失敗する。

保守担当者向けメモ

  • 本関数は postmaster プロセスから呼び出されることを想定している。
  • accept() 失敗時のスリープ処理は、postmaster が読み込み準備完了状態のソケットに対して繰り返し accept を試行し続けることによるビジーウェイトを回避するためのものである。
