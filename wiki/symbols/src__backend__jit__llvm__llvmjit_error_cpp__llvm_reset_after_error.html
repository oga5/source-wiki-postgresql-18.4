llvm_reset_after_error

定義

  • 種別: function
  • 定義箇所: src/backend/jit/llvm/llvmjit_error.cpp:95

目的

LLVMのJITコンパイル処理中に発生したエラーからのリカバリ時、一時的に差し替えられたエラーハンドラを強制的に初期状態へ戻し、状態管理用のカウンタをリセットします。

シグネチャまたは形状

void llvm_reset_after_error(void)

入力と出力

  • 入力: なし
  • 出力: なし（副作用としてグローバルなエラーハンドラ設定と状態変数を更新）

前提条件と不変条件

  • fatal_new_handler_depth が 0 以外の値である場合、エラーハンドラがPostgreSQL専用のものに差し替えられているとみなされます。
  • 実行後、fatal_new_handler_depth は必ず 0 にリセットされます。

アルゴリズムまたは振る舞い

1. fatal_new_handler_depth が 0 でないかを確認します。 2. 0 でない場合、以下の処理を実行してハンドラを復元します。

  • std::set_new_handler(old_new_handler) を呼び出し、C++のメモリ割り当てハンドラを元の状態に戻します。
  • llvm::remove_bad_alloc_error_handler() を呼び出し、LLVMのメモリ不足ハンドラを解除します。
  • llvm::remove_fatal_error_handler() を呼び出し、LLVMの致命的エラーハンドラを解除します。

    • 3. 最後に fatal_new_handler_depth を 0 に設定します。

呼び出し元と呼び出し先

  • 呼び出し元:
  • src/backend/jit/llvm/llvmjit.c:153: cb->reset_after_error に関数ポインタとして代入されています。
  • 呼び出し先:
  • std::set_new_handler
  • llvm::remove_bad_alloc_error_handler
  • llvm::remove_fatal_error_handler

関連データ構造

  • fatal_new_handler_depth: ネストレベルを追跡する静的変数。
  • old_new_handler: 以前の new_handler を保持する変数（本関数内で参照）。

失敗モードと境界条件

  • fatal_new_handler_depth が 0 の場合、ハンドラの復元処理はスキップされ、単に 0 へのリセットのみが行われます。

保守担当者向けメモ

  • 本関数は extern "C" インターフェースとして提供されており、C言語側からJITエラーリカバリのクリーンアップとして利用されます。
  • LLVMの内部状態を直接操作するため、呼び出し順序やネストレベルの整合性に注意してください。
