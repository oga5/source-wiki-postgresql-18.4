setup_test_matches

定義

  • 種別: function
  • 定義箇所: src/test/modules/test_regex/test_regex.c:434

目的

正規表現のコンパイル済みパターンを対象文字列に対して実行し、マッチング結果（位置情報など）を保持するコンテキスト構造体 test_regex_ctx を初期化・構築する。

シグネチャまたは形状

static test_regex_ctx *setup_test_matches(text *orig_str,
                                          regex_t *cpattern,
                                          test_re_flags *re_flags,
                                          Oid collation,
                                          bool use_subpatterns)

入力と出力

  • **入力**:
  • orig_str: 対象となる入力文字列。
  • cpattern: コンパイル済みの正規表現パターン。
  • re_flags: テスト用の実行オプション（glob フラグ等を含む）。
  • collation: 照合順序。
  • use_subpatterns: サブパターンを記録するかどうかのフラグ。
  • **出力**:
  • マッチング結果と状態を格納した test_regex_ctx 構造体へのポインタ。

前提条件と不変条件

  • orig_str は PostgreSQL の text 型であること。
  • cpattern は有効な regex_t オブジェクトであること。
  • re_flags->glob が真の場合、文字列全体に対して繰り返しマッチングを行う。
  • array_idx は常に array_len 以下であること（Assert による検証）。

アルゴリズムまたは振る舞い

1. 入力文字列を pg_wchar 形式に変換し、マッチング用バッファを準備する。 2. use_subpatterns に応じて pmatch（マッチング結果格納用）のサイズを決定する。 3. test_re_execute を呼び出し、パターンマッチングを実行する。

  • glob フラグが有効な場合、マッチングが成功する限り start_search を更新して繰り返し実行する。
  • マッチング位置（rm_so, rm_eo）を matchctx->match_locs に記録し、必要に応じて動的に領域を拡張する。

    • 4. マッチングが一つも得られず、かつ partial および indices フラグが有効な場合は、details.rm_extend から部分マッチ情報を取得する。 5. マルチバイト文字環境（eml > 1）の場合、変換用バッファ conv_buf を確保し、ワイド文字列を保持する。シングルバイト環境ではメモリを解放する。

呼び出し元と呼び出し先

  • **呼び出し元**: src/test/modules/test_regex/test_regex.c 内の test_regex 関数（SRF_IS_FIRSTCALL 時）。
  • **呼び出し先**: test_re_execute（マッチング実行）、palloc/repalloc（メモリ確保）、pg_mb2wchar_with_len（文字変換）、ereport（エラー報告）。

関連データ構造

  • test_regex_ctx: マッチング結果と実行状態を保持するメイン構造体。
  • test_re_flags: 実行オプションを格納。
  • regmatch_t: 正規表現エンジンが返すマッチング位置情報。

失敗モードと境界条件

  • **メモリ不足**: array_len が MaxAllocSize を超える場合、ERRCODE_PROGRAM_LIMIT_EXCEEDED でエラー終了する。
  • **マッチングなし**: nmatches が 0 の場合、partial フラグの有無により挙動が分岐する。
  • **ゼロ長マッチ**: マッチングが空（rm_so == rm_eo）の場合、無限ループを避けるために検索開始位置を 1 文字進める。

保守担当者向けメモ

  • 本関数は SRF の初回呼び出し時に一度だけ実行されることを前提としている。
  • match_locs の動的拡張ロジックは、2^n-1 のサイズ増加ルールを採用しており、メモリ効率と上限チェックを考慮している。
  • マルチバイト文字の変換バッファ確保ロジックは、eml（最大文字長）に基づいた安全なサイズ計算を行っている。
