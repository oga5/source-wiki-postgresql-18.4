prune_lexemes_hashtable

定義

  • 種別: function
  • 定義箇所: src/backend/tsearch/ts_typanalyze.c:452

目的

Lossy Counting アルゴリズムの一環として、ハッシュテーブル内の低頻度な語彙エントリを削除し、メモリ消費を抑制しつつ統計情報の精度を維持する。

シグネチャまたは形状

static void prune_lexemes_hashtable(HTAB *lexemes_tab, int b_current)

入力と出力

  • lexemes_tab (HTAB*): 語彙の頻度を追跡しているハッシュテーブル。
  • b_current (int): 現在のバケット番号（閾値判定に使用）。
  • 戻り値: なし。

前提条件と不変条件

  • lexemes_tab は有効なハッシュテーブルであること。
  • 各エントリは TrackItem 構造体として管理されていること。
  • 削除対象となったエントリのキーに含まれる文字列（lexeme）は、pfree によって解放される。

アルゴリズムまたは振る舞い

1. hash_seq_init を用いてハッシュテーブル全体を走査する。 2. 各エントリについて、frequency + delta <= b_current を満たすか判定する。 3. 条件を満たす場合、そのエントリを hash_search (HASH_REMOVE) で削除し、関連する文字列メモリを pfree で解放する。 4. 削除に失敗した場合は elog(ERROR) を発生させる。

呼び出し元と呼び出し先

  • 呼び出し元: src/backend/tsearch/ts_typanalyze.c 内の compute_tsvector_stats 関数から呼び出される。
  • 呼び出し先: hash_seq_init, hash_seq_search, hash_search, pfree, elog。

関連データ構造

  • HTAB: PostgreSQL のハッシュテーブル構造体。
  • TrackItem: 語彙の頻度（frequency）と誤差（delta）を保持する構造体。

失敗モードと境界条件

  • ハッシュテーブルの走査中に予期せぬ削除失敗が発生した場合、elog(ERROR) によりトランザクションが中断される。
  • メモリ不足時には pfree 以前の段階でエラーが発生する可能性がある。

保守担当者向けメモ

  • 本関数は Lossy Counting アルゴリズムのメモリ管理の要である。削除条件 frequency + delta <= b_current はアルゴリズムの理論的保証に基づいているため、変更時は注意が必要である。
