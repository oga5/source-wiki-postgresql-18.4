ConditionalLockRelation

定義

  • 種別: function
  • 定義箇所: src/backend/storage/lmgr/lmgr.c:277

目的

指定されたリレーションに対して、待機せずに条件付きでロックを取得します。ロックが即座に取得できない場合は待機せず、呼び出し元へ制御を戻します。

シグネチャまたは形状

bool ConditionalLockRelation(Relation relation, LOCKMODE lockmode)

入力と出力

  • **入力**:
  • relation: ロック対象のリレーション記述子。
  • lockmode: 取得を試みるロックモード（LOCKMODE）。
  • **出力**:
  • bool: ロック取得に成功した場合は true、競合により取得できなかった場合は false を返します。

前提条件と不変条件

  • relation 引数は有効なリレーション記述子である必要があります。
  • ロック取得成功後、必要に応じてキャッシュインバリデーションメッセージを処理し、リレーションキャッシュの整合性を維持します。

アルゴリズムまたは振る舞い

1. SET_LOCKTAG_RELATION マクロを使用して、リレーションのデータベースIDとリレーションIDから LOCKTAG を構築します。 2. LockAcquireExtended を呼び出し、ロック取得を試みます（待機しない設定 conditional=true を使用）。 3. 取得結果が LOCKACQUIRE_NOT_AVAIL の場合、即座に false を返します。 4. ロックが取得できた場合、LOCKACQUIRE_ALREADY_CLEAR でなければ AcceptInvalidationMessages() を呼び出してキャッシュを更新し、MarkLockClear() でロック状態をマークします。 5. 成功した場合は true を返します。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/access/heap/vacuumlazy.c:3231 (確認済み)
  • **呼び出し先**:
  • LockAcquireExtended
  • AcceptInvalidationMessages
  • MarkLockClear

関連データ構造

  • Relation: ロック対象の構造体。
  • LOCKTAG: ロック対象を識別するタグ。
  • LOCALLOCK: プロセスローカルなロック管理構造体。
  • LOCKMODE: ロックの強度を指定する列挙型。

失敗モードと境界条件

  • 他のセッションが競合するロックを保持している場合、false を返して終了します。
  • ロック取得処理自体が内部エラー（メモリ不足や共有メモリの枯渇など）に遭遇した場合の挙動は LockAcquireExtended に依存します。

保守担当者向けメモ

  • 本関数は LockRelation と異なり、待機を行わないため、デッドロックの懸念がある箇所や、ロック取得を試行して失敗した際に別の処理へ分岐したい場合に適しています。
  • ロック取得後の AcceptInvalidationMessages() の呼び出しは、スキーマ変更等によるキャッシュの不整合を防ぐために不可欠です。
