expand_partitioned_rtentry

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/util/inherit.c:317

目的

パーティション分割されたテーブルの範囲テーブルエントリ（RTE）を再帰的に展開し、パーティションプルーニングを実行して、クエリ計画に必要な子テーブルの情報を構築します。

シグネチャまたは形状

static void expand_partitioned_rtentry(
    PlannerInfo *root, 
    RelOptInfo *relinfo,
    RangeTblEntry *parentrte,
    Index parentRTindex, 
    Relation parentrel,
    Bitmapset *parent_updatedCols,
    PlanRowMark *top_parentrc, 
    LOCKMODE lockmode
)

入力と出力

  • **入力**:
  • root: プランナのグローバル情報。
  • relinfo: 親テーブルの RelOptInfo。
  • parentrte: 親テーブルの RangeTblEntry。
  • parentrel: 開かれた親テーブルの Relation。
  • parent_updatedCols: 更新対象の列集合。
  • top_parentrc: 行マーク情報。
  • lockmode: テーブルロックモード。
  • **出力**: なし（副作用として root 内の append_rel_list や relinfo の各フィールドを更新）。

前提条件と不変条件

  • parentrte->inh が真であること。
  • parentrel に対応するパーティション記述子（PartitionDesc）が存在すること。

アルゴリズムまたは振る舞い

1. PartitionDirectoryLookup を使用してパーティション記述子を取得します。 2. has_partition_attrs を用いて、更新対象の列がパーティションキーに含まれるか判定し、root->partColsUpdated を更新します。 3. prune_append_rel_partitions を呼び出し、不要なパーティションを除外（プルーニング）します。 4. 生存したパーティションに対して以下を繰り返します：

  • try_table_open で子テーブルを開き、失敗した場合はプルーニング済みとして扱います。
  • expand_single_inheritance_child を呼び出し、子テーブルの RTE、AppendRelInfo、PlanRowMark を生成します。
  • build_simple_rel で子テーブルの RelOptInfo を構築し、relinfo->part_rels に格納します。
  • 子テーブルがさらにパーティション分割されている場合、再帰的に本関数を呼び出します。

呼び出し元と呼び出し先

  • **呼び出し元**: expand_inherited_rtentry（src/backend/optimizer/util/inherit.c）から呼び出されます。また、再帰的に自身を呼び出します。
  • **呼び出し先**: PartitionDirectoryLookup, has_partition_attrs, prune_append_rel_partitions, expand_planner_arrays, try_table_open, expand_single_inheritance_child, build_simple_rel, translate_col_privs, table_close。

関連データ構造

  • PlannerInfo: プランナの全体状態。
  • RelOptInfo: リレーションの最適化情報。
  • RangeTblEntry: 範囲テーブルエントリ。
  • PartitionDesc: パーティションのメタデータ。
  • AppendRelInfo: 継承関係の変換情報。

失敗モードと境界条件

  • パーティションが 0 個の場合、処理を中断して戻ります。
  • try_table_open が NULL を返した場合（パーティションが削除された場合など）、そのパーティションは無視されます。
  • 他セッションのテンポラリリレーションがパーティションとして検出された場合、エラーが発生します。

保守担当者向けメモ

  • 本関数は再帰的であるため、深いパーティション階層を持つテーブルではスタック深さに注意してください（check_stack_depth() が呼び出されています）。
  • relinfo->part_rels は palloc0 で確保されるため、プルーニングされたパーティションのインデックスは NULL になります。
