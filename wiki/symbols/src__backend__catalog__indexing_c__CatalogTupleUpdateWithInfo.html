CatalogTupleUpdateWithInfo

定義

  • 種別: function
  • 定義箇所: src/backend/catalog/indexing.c:336

目的

システムカタログのタプル更新処理において、既存のインデックス状態（CatalogIndexState）を再利用することで、インデックスオープン処理のオーバーヘッドを削減しつつ、ヒープ更新とインデックス更新を統合的に実行する。

シグネチャまたは形状

void CatalogTupleUpdateWithInfo(Relation heapRel, ItemPointer otid, HeapTuple tup, CatalogIndexState indstate)

入力と出力

  • **入力**:
  • heapRel: 更新対象のカタログリレーション記述子。
  • otid: 更新対象タプルの物理位置（ItemPointer）。
  • tup: 更新後の新しいタプルデータ。
  • indstate: 既にオープン済みのインデックス情報（CatalogIndexState）。
  • **出力**: なし（void）。

前提条件と不変条件

  • indstate は、対象リレーションに対して CatalogOpenIndexes を用いて適切に初期化されている必要がある。
  • システムカタログ特有の制約（NOT NULL 等）が CatalogTupleCheckConstraints によって検証される。

アルゴリズムまたは振る舞い

1. CatalogTupleCheckConstraints を呼び出し、更新後のタプルがカタログの制約を満たしているか検証する。 2. simple_heap_update を呼び出し、ヒープ上のタプルを更新する。この際、HOT (Heap Only Tuple) 更新かどうかの判定結果が updateIndexes に格納される。 3. CatalogIndexInsert を呼び出し、indstate に基づいてインデックスを更新する。HOT 更新と判定された場合、インデックス更新はスキップされる。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/commands/analyze.c
  • src/backend/commands/cluster.c
  • src/backend/commands/tsearchcmds.c
  • src/backend/storage/large_object/inv_api.c
  • src/backend/catalog/indexing.c（内部的なラッパー呼び出し）
  • **呼び出し先**:
  • CatalogTupleCheckConstraints
  • simple_heap_update
  • CatalogIndexInsert

関連データ構造

  • Relation: リレーション記述子。
  • HeapTuple: 更新対象のタプル構造体。
  • CatalogIndexState: インデックス更新に必要な情報を保持する不透明な構造体。
  • TU_UpdateIndexes: インデックス更新の要否を制御する列挙型。

失敗モードと境界条件

  • カタログ制約違反が発生した場合、CatalogTupleCheckConstraints によりエラーがスローされる。
  • simple_heap_update が失敗した場合（同時更新による競合など）、例外が発生する。
  • システムカタログでは式インデックス、部分インデックス、除外制約はサポートされておらず、これらが含まれる場合は CatalogIndexInsert 内の Assert で保護されている。

保守担当者向けメモ

  • 複数行の更新を行う際は、CatalogOpenIndexes をループの外で呼び出し、indstate を本関数に渡すことでパフォーマンスを最適化できる。
  • CatalogTupleUpdate と比較して、インデックスの再オープンを避けるための最適化版である。
