doPickSplit

定義

  • **種別**: function
  • **定義箇所**: src/backend/access/spgist/spgdoinsert.c:676

目的

SP-GiSTインデックスにおいて、リーフページの容量が不足した際に、既存のリーフタプルと新規挿入タプルを再分配し、インナータプルを生成してツリー構造を更新する。

シグネチャまたは形状

static bool doPickSplit(Relation index, SpGistState *state,
                        SPPageDesc *current, SPPageDesc *parent,
                        SpGistLeafTuple newLeafTuple,
                        int level, bool isNulls, bool isNew)

入力と出力

  • **入力**:
  • index: 対象のインデックスリレーション。
  • state: SP-GiSTの操作状態。
  • current: 現在のリーフページ情報。
  • parent: 親ページ情報。
  • newLeafTuple: 挿入対象の新しいリーフタプル。
  • level, isNulls, isNew: 分割制御用のメタデータ。
  • **出力**:
  • 戻り値: 処理が成功し、新しいタプルがインストールされた場合に true を返す。

前提条件と不変条件

  • current ページはリーフページである必要がある。
  • ルートページの場合、タプルはリンクされていないためシーケンシャルにスキャンされる。
  • ルート以外の場合、リーフチェーンを辿ってタプルを収集する。
  • 壊れたopclassによる無限ループを防ぐため、呼び出し元で numNoProgressCycles によるガードが推奨される。

アルゴリズムまたは振る舞い

1. **初期化**: ページ内の全リーフタプルを収集し、作業用配列（oldLeafs等）を割り当てる。 2. **タプル収集**:

  • ルートページなら全タプルを、それ以外ならリーフチェーン上の全タプルを抽出する。
  • 抽出したタプルと newLeafTuple を入力として spgPickSplit プロシージャを呼び出す。

    • 3. **分割実行**: ユーザー定義の SPGIST_PICKSPLIT_PROC を実行し、タプルの再分配方法を決定する。 4. **タプル再形成**: 分割結果に基づき、新しいリーフタプルを形成する。 5. **構造更新**: ページ内の既存タプルを削除またはリダイレクト/プレースホルダーへ置換し、新しいインナータプルを挿入する。

呼び出し元と呼び出し先

  • **呼び出し元**: spgdoinsert 関数（src/backend/access/spgist/spgdoinsert.c 内の複数箇所）。
  • **呼び出し先**:
  • index_getprocinfo (SPGIST_PICKSPLIT_PROC)
  • spgFormLeafTuple, spgDeformLeafTuple
  • PageGetItem, PageGetItemId 等のページ操作関数。

関連データ構造

  • SPPageDesc: 挿入中のページ状態を保持。
  • SpGistLeafTuple: リーフタプルの構造。
  • spgPickSplitIn / spgPickSplitOut: 分割処理の入出力インターフェース。
  • spgxlogPickSplit: WAL記録用の構造体。

失敗モードと境界条件

  • **容量不足**: 分割後もページ容量が不足する場合、再帰的な分割や移動が発生する可能性がある。
  • **無効なタプル状態**: SPGIST_LIVE 以外の予期しないタプル状態が検出された場合、elog(ERROR) を発生させる。
  • **無限ループ**: picksplit プロシージャが進捗のない分割を繰り返す場合、システムが停止するリスクがある。

保守担当者向けメモ

  • 構造変更は START_CRIT_SECTION() で保護されており、WAL整合性が保証されている。
  • ロック取得順序によるデッドロックを避けるため、ConditionalLockBuffer 等の利用状況に注意すること。
