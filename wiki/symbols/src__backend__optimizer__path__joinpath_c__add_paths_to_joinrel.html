add_paths_to_joinrel

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/path/joinpath.c:123

目的

2つの結合対象リレーション（outerrel, innerrel）から、指定された結合タイプに基づいた実行可能な結合パス（Path）を生成し、joinrel のパスリストへ追加する。

シグネチャまたは形状

void add_paths_to_joinrel(PlannerInfo *root,
                          RelOptInfo *joinrel,
                          RelOptInfo *outerrel,
                          RelOptInfo *innerrel,
                          JoinType jointype,
                          SpecialJoinInfo *sjinfo,
                          List *restrictlist)

入力と出力

  • **入力**:
  • root: プランナのグローバル情報。
  • joinrel: 結合結果を格納するリレーション情報。
  • outerrel, innerrel: 結合対象の入力リレーション。
  • jointype: 結合の種類（JOIN_INNER, JOIN_SEMI 等）。
  • sjinfo: 特殊な結合（外部結合等）のメタデータ。
  • restrictlist: 結合条件式（RestrictInfo のリスト）。
  • **出力**: なし（副作用として joinrel->pathlist を更新）。

前提条件と不変条件

  • joinrel は既に初期化されている必要がある。
  • jointype が JOIN_UNIQUE_OUTER の場合、内部的に JOIN_INNER として扱われる。
  • param_source_rels は、結合パスが依存すべきパラメータソースリレーションの集合として計算される。

アルゴリズムまたは振る舞い

1. **メタデータ初期化**: JoinPathExtraData を構築し、結合条件や一意性フラグをセットする。 2. **一意性判定**: innerrel_is_unique を呼び出し、内部リレーションが結合に対して一意であるかを判定する（SEMI/ANTI 結合を除く）。 3. **マージ結合句の選択**: select_mergejoin_clauses を呼び出し、マージ結合が可能な条件を抽出する。 4. **コスト補正**: SEMI/ANTI 結合または一意な結合の場合、compute_semi_anti_join_factors でコスト見積もり用の補正係数を計算する。 5. **パラメータ化の制約**: root->join_info_list を走査し、結合順序の制約に基づいた param_source_rels を決定する。 6. **パス生成の試行**:

  • sort_inner_and_outer: 両方のリレーションをソートするマージ結合パスを生成。
  • match_unsorted_outer: ソート済みまたはソート不要な外部リレーションを利用するマージ結合およびネステッドループ結合パスを生成。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/optimizer/path/joinrels.c 内の結合順序決定ロジックから複数回呼び出される。
  • **呼び出し先**:
  • innerrel_is_unique, select_mergejoin_clauses, compute_semi_anti_join_factors
  • sort_inner_and_outer, match_unsorted_outer
  • bms_ 系関数（ビットマップ操作）

関連データ構造

  • JoinPathExtraData: 結合処理中に保持される一時的なメタデータ構造体。
  • RelOptInfo: 結合対象および結果のリレーション情報。
  • SpecialJoinInfo: 外部結合やセミ結合の制約情報。

失敗モードと境界条件

  • mergejoin_allowed が false の場合、マージ結合パスの生成はスキップされる（ただし JOIN_FULL の場合は強制的に許可される）。
  • LATERAL サブクエリが存在する場合、param_source_rels に追加の依存関係が強制的に付与される。

保守担当者向けメモ

  • 本関数は結合パス生成のメインエントリポイントである。
  • JOIN_SEMI や JOIN_ANTI における inner_unique の扱いは、最適化の余地（Memoize の利用など）があるため注意が必要。
  • NOT_USED ブロック以降に、さらなるパス生成ロジック（ハッシュ結合など）が続く構造になっている。
