ExecLockRows

定義

  • 種別: function
  • 定義箇所: src/backend/executor/nodeLockRows.c:37

目的

SQLの SELECT ... FOR UPDATE や FOR SHARE 構文で要求される行レベルのロックを適用し、同時実行制御（EPQ: EvalPlanQual）による再評価を行う実行ノードのメインループ処理。

シグネチャまたは形状

TupleTableSlot *ExecLockRows(PlanState *pstate)

入力と出力

  • **入力**: PlanState *pstate（LockRowsState 構造体としてキャストされる実行ノードの状態）。
  • **出力**: ロックが成功したタプルを格納する TupleTableSlot、または全タプル処理完了時に NULL。

前提条件と不変条件

  • pstate は LockRowsState 型であること。
  • 下位ノード（outerPlan）からタプルが供給されること。
  • lr_arowMarks にロック対象の行マーク情報が保持されていること。

アルゴリズムまたは振る舞い

1. **タプル取得**: ExecProcNode を呼び出し、下位ノードからタプルを取得する。取得できない場合は EvalPlanQualEnd を呼び出し終了する。 2. **行マークの走査**: lr_arowMarks リストを反復し、各リレーションに対してロックを試行する。 3. **子リレーション判定**: erm->rti != erm->prti の場合、tableoid を確認し、現在のタプルが対象リレーションのものか判定する。不一致ならスキップする。 4. **外部テーブル処理**: RELKIND_FOREIGN_TABLE の場合、FDWの RefetchForeignRow を呼び出す。更新が発生した場合は epq_needed を真にする。 5. **ロック試行**: table_tuple_lock を呼び出し、erm->markType に応じたロックモード（Exclusive, Share等）を適用する。

  • TM_WouldBlock: SKIP LOCKED 指定時にロック不可の場合、次のタプルへ遷移。
  • TM_SelfModified: 自トランザクションによる更新済みの場合、そのタプルを無視して次のタプルへ遷移。
  • TM_Ok: ロック成功。

呼び出し元と呼び出し先

  • **呼び出し元**: 実行エンジン（ExecProcNode 経由で lrstate->ps.ExecProcNode に登録）。
  • **呼び出し先**: ExecProcNode（下位ノード）、EvalPlanQualEnd、table_tuple_lock、GetFdwRoutineForRelation、fdwroutine->RefetchForeignRow。

関連データ構造

  • LockRowsState: ノードの実行時状態を保持。
  • ExecAuxRowMark: ロック対象のメタデータ。
  • TupleTableSlot: 処理対象のタプルを保持。
  • EvalPlanQualState: EPQの状態管理。

失敗モードと境界条件

  • ctid や tableoid が NULL の場合、elog(ERROR) を発生させる。
  • 外部テーブルで RefetchForeignRow が定義されていない場合、ereport(ERROR) で失敗する。
  • サポートされていない markType が指定された場合、elog(ERROR) を発生させる。

保守担当者向けメモ

  • 本関数は下位ノードからタプルが返された直後にロックを試みるため、プランツリー上の配置順序が重要である。
  • TM_SelfModified の処理において、現在の実装では「削除されたもの」として扱うことで「ハロウィン問題」を回避している。
  • EPQの再評価ロジックは EvalPlanQual 機構に依存しているため、競合時の挙動変更には注意が必要。
