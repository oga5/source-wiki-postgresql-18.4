finish_heap_swap

定義

  • 種別: function
  • 定義箇所: src/backend/commands/cluster.c:1444

目的

テーブル再構築（CLUSTER や VACUUM FULL）の最終段階として、新旧の物理ファイルを入れ替え、インデックスの再構築、クリーンアップ、および関連するメタデータの更新を行う。

シグネチャまたは形状

void finish_heap_swap(Oid OIDOldHeap, Oid OIDNewHeap,
                      bool is_system_catalog,
                      bool swap_toast_by_content,
                      bool check_constraints,
                      bool is_internal,
                      TransactionId frozenXid,
                      MultiXactId cutoffMulti,
                      char newrelpersistence)

入力と出力

  • **入力**:
  • OIDOldHeap, OIDNewHeap: 入れ替え対象の旧テーブルと新テーブルのOID。
  • is_system_catalog: システムカタログかどうかのフラグ。
  • swap_toast_by_content: TOASTテーブルを内容で入れ替えるか（リンクで入れ替えるか）の指定。
  • check_constraints: 制約チェックを行うか。
  • is_internal: 内部操作かどうか。
  • frozenXid, cutoffMulti: relfrozenxid および relminmxid の更新値。
  • newrelpersistence: 新しいリレーションの永続性設定。
  • **出力**: なし（副作用として物理ファイルおよびカタログを更新）。

前提条件と不変条件

  • 呼び出し元で適切なロック（通常は AccessExclusiveLock）が保持されていること。
  • OIDNewHeap は現在のトランザクション内で作成された一時的なリレーションであること。

アルゴリズムまたは振る舞い

1. **ファイルスワップ**: swap_relation_files を呼び出し、物理ファイルを入れ替える。 2. **キャッシュ無効化**: システムカタログの場合は CacheInvalidateCatalog を実行。 3. **インデックス再構築**: reindex_relation を呼び出し、新しいヒープに対してインデックスを再構築する。 4. **メタデータ更新**: pg_class が対象の場合、relfrozenxid 等を更新する。 5. **一時リレーション削除**: performDeletion を使用して、旧ファイルとなった一時テーブルを削除する。 6. **マッピング削除**: RelationMapRemoveMapping を呼び出し、一時的なリレーションマッピングをクリーンアップする。 7. **TOASTリネーム**: swap_toast_by_content が偽の場合、TOASTテーブルとそのインデックスを適切な命名規則にリネームする。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/commands/cluster.c (複数箇所)
  • src/backend/commands/matview.c
  • src/backend/commands/tablecmds.c
  • **呼び出し先**:
  • swap_relation_files, reindex_relation, performDeletion, RelationMapRemoveMapping, RenameRelationInternal, table_open, table_close 等。

関連データ構造

  • ReindexParams: インデックス再構築時のパラメータ。
  • ObjectAddress: 削除対象のオブジェクト指定に使用。

失敗モードと境界条件

  • pg_class の更新時にキャッシュルックアップが失敗した場合、エラーが発生する。
  • RelationMapRemoveMapping の呼び出しを忘れると、コミット時にリレーションマッパーがエラーを報告する。

保守担当者向けメモ

  • 本関数はファイルスワップの最終工程を担うため、システムカタログの整合性に直結する。
  • pg_class の更新ロジックは、swap_relation_files が直接更新できないケースを補完する特殊な処理である。
  • TOASTテーブルのリネーム処理は、カタログ上の名前と物理的な参照関係の整合性を保つために重要である。
