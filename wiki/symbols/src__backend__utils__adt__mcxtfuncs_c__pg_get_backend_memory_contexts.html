pg_get_backend_memory_contexts

定義

  • 種別: function
  • 定義箇所: src/backend/utils/adt/mcxtfuncs.c:182

目的

現在のバックエンドにおける全メモリコンテキスト（MemoryContext）を階層構造とともに列挙し、その統計情報をSQLの結果セットとして提供します。

シグネチャまたは形状

Datum pg_get_backend_memory_contexts(PG_FUNCTION_ARGS)

入力と出力

  • **入力**: PG_FUNCTION_ARGS（SQL関数呼び出し規約に基づく引数）。
  • **出力**: Datum 型（実際には Tuplestore を介してSRFとして行セットを返却）。

前提条件と不変条件

  • TopMemoryContext を起点として走査を開始します。
  • TopMemoryContext には常に context_id = 1 が割り当てられます。
  • 走査には幅優先探索（BFS）を採用しており、これによりコンテキストIDの割り当てを安定させています。

アルゴリズムまたは振る舞い

1. context_id_lookup 用のハッシュテーブルを作成します。 2. TopMemoryContext をリストの初期値として設定します。 3. リストを反復処理し、各コンテキストに対して以下を実行します。

  • ハッシュテーブルにコンテキストを登録し、一意の context_id を割り当てます。
  • PutMemoryContextsStatsTupleStore を呼び出し、現在のコンテキストの統計情報を結果セットに追加します。
  • 現在のコンテキストの全子ノードをリストの末尾に追加し、次以降の反復で処理します。

    • 4. 最後にハッシュテーブルを破棄します。

呼び出し元と呼び出し先

  • **呼び出し先**:
  • hash_create, hash_search, hash_destroy（ハッシュテーブル管理）
  • InitMaterializedSRF（SRF初期化）
  • PutMemoryContextsStatsTupleStore（統計情報の格納）
  • list_make1, lappend（リスト操作）
  • **呼び出し元**: 提供されたソースコード内には、本関数を直接呼び出している具体的な関数名は記載されていません。

関連データ構造

  • MemoryContext: メモリ管理の基本単位。
  • MemoryContextId: コンテキストとIDを紐付けるための内部構造体。
  • Tuplestore: 結果セットを保持するための構造。
  • HTAB: コンテキストID検索用のハッシュテーブル。

失敗モードと境界条件

  • メモリ不足時には hash_create やリスト操作がエラーを発生させる可能性があります。
  • 循環参照が存在する場合、リストの反復処理が無限ループに陥る可能性があります（ただし、メモリコンテキストの階層構造は通常ツリー状です）。

保守担当者向けメモ

  • BFSを採用しているのは、深さ優先探索と比較してコンテキストIDの変動を抑え、出力を安定させるためです。
  • PutMemoryContextsStatsTupleStore は、この関数で生成された context_id_lookup を利用して、出力の「path」列を構築します。
