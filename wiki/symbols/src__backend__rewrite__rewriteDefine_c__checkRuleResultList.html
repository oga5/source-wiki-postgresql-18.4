checkRuleResultList

定義

  • 種別: function
  • 定義箇所: src/backend/rewrite/rewriteDefine.c:505

目的

ビューの SELECT ルールや RETURNING 句において、指定されたターゲットリストが対象リレーションのスキーマ（列数、列名、データ型、型修飾子）と適合しているかを検証します。

シグネチャまたは形状

static void checkRuleResultList(List *targetList, TupleDesc resultDesc, bool isSelect, bool requireColumnNameMatch)

入力と出力

  • **入力**:
  • targetList: 検証対象の TargetEntry リスト。
  • resultDesc: 対象リレーションの TupleDesc。
  • isSelect: SELECT ルールの検証であるかを示すフラグ。
  • requireColumnNameMatch: 列名の完全一致を要求するかどうか（isSelect が真の場合のみ指定可能）。
  • **出力**: なし。検証失敗時は ereport(ERROR, ...) により処理を中断します。

前提条件と不変条件

  • isSelect が偽の場合、requireColumnNameMatch は必ず偽でなければなりません（Assert による制約）。
  • resjunk（システム内部用列）としてマークされたエントリは検証対象から除外されます。

アルゴリズムまたは振る舞い

1. targetList を走査し、resjunk 以外の各エントリについて、対応するリレーションの列（resultDesc）と以下の項目を順次比較します。

  • **列数**: リストの要素数がリレーションの列数と一致するか確認します。
  • **削除済み列**: 対象リレーションに attisdropped な列が含まれている場合、エラーを発生させます。
  • **列名**: requireColumnNameMatch が真の場合、resname がリレーションの列名と一致するか確認します。
  • **データ型**: exprType を用いて取得した式の型が、リレーションの列型（atttypid）と一致するか確認します。
  • **型修飾子（typmod）**: 両者が「未指定（-1）」でない限り、値が一致することを確認します。

    • 2. 全ての走査終了後、リストの要素数がリレーションの列数と一致しているか最終確認を行います。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/rewrite/rewriteDefine.c 内の DefineQueryRewrite（SELECT ルールおよび RETURNING 句の検証箇所）。
  • **呼び出し先**: exprType, exprTypmod, TupleDescAttr, NameStr, format_type_be, format_type_with_typemod 等のユーティリティ関数。

関連データ構造

  • List / ListCell: ターゲットリストの保持に使用。
  • TargetEntry: クエリのターゲットリスト要素。
  • TupleDesc: リレーションのスキーマ定義。
  • Form_pg_attribute: pg_attribute カタログの行構造。

失敗モードと境界条件

  • **ERRCODE_INVALID_OBJECT_DEFINITION**: 列数不一致、列名不一致、型不一致、または型修飾子の不一致が発生した場合。
  • **ERRCODE_FEATURE_NOT_SUPPORTED**: 削除済み列（attisdropped）を含むリレーションに対してルールを適用しようとした場合。

保守担当者向けメモ

  • 本関数は SELECT ルールと RETURNING 句の両方で再利用されています。
  • 削除済み列のサポートについては、ruleutils.c 等の修正が必要となるため、現時点では明示的に禁止されています。
  • typmod の比較において -1 を「未指定」として許容するロジックは、numeric 型などのデフォルト値との整合性を保つために重要です。
