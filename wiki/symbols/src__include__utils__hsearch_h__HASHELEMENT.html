HASHELEMENT

定義

  • 種別: struct
  • 定義箇所: src/include/utils/hsearch.h:51

目的

動的ハッシュテーブル（dynahash）における各エントリのヘッダー構造体です。ハッシュテーブル内の要素を連結リストとして管理し、検索効率を維持するためのメタデータを保持します。

シグネチャまたは形状

typedef struct HASHELEMENT
{
    struct HASHELEMENT *link;    /* 同一バケット内の次エントリへのリンク */
    uint32        hashvalue;     /* このエントリのハッシュ関数結果 */
} HASHELEMENT;

入力と出力

  • 本構造体自体はデータ保持用であり、関数としての入出力は持ちません。
  • ユーザーデータは、HASHELEMENT 構造体の直後にメモリ配置されることが期待されます。

前提条件と不変条件

  • ユーザーデータは HASHELEMENT の直後に配置される必要があります。
  • ハッシュキーは、ユーザーデータの先頭部分に配置されている必要があります。
  • MAXALIGN(sizeof(HASHELEMENT)) を用いて、ヘッダーとユーザーデータの境界をアライメントする必要があります。

アルゴリズムまたは振る舞い

  • link フィールドにより、ハッシュ衝突が発生した際に同一バケット内のエントリを連結リストとして管理します。
  • hashvalue フィールドには、計算済みのハッシュ値が格納され、再計算を避けるために利用されます。
  • dynahash は、このヘッダーを先頭に持つメモリブロックを確保し、ユーザーデータへのポインタを計算してアクセスします。

呼び出し元と呼び出し先

  • 呼び出し元: src/backend/utils/hash/dynahash.c 内のハッシュテーブル操作関数群（hash_search 等）。
  • 呼び出し先: 直接的な関数呼び出しはありませんが、ELEMENTKEY マクロや ELEMENT_FROM_KEY マクロを通じて、ヘッダーとデータの相互変換に使用されます。

関連データ構造

  • HTAB: ハッシュテーブル全体を管理する構造体。
  • HASHBUCKET: HASHELEMENT * の別名として定義され、バケットの先頭ポインタとして使用されます。

失敗モードと境界条件

  • メモリ配置の不整合: MAXALIGN を使用せずにヘッダーとユーザーデータを計算した場合、メモリアクセス違反やデータ破壊が発生します。
  • ユーザーデータサイズ: entrysize の指定が誤っている場合、隣接するエントリのメモリ領域を破壊する可能性があります。

保守担当者向けメモ

  • 本構造体は dynahash の内部実装と密接に結合しています。構造体のメンバを変更する場合は、dynahash.c 内のオフセット計算マクロ（ELEMENTKEY 等）をすべて更新する必要があります。
