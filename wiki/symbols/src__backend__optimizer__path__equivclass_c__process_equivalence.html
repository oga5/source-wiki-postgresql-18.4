process_equivalence

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/path/equivclass.c:178

目的

クエリ内の等価述語（RestrictInfo）を解析し、既存の等価クラス（EquivalenceClass）へ統合または新規作成を行うことで、プランナが推移律を利用した最適化を行えるようにする。

シグネチャまたは形状

bool process_equivalence(PlannerInfo *root, RestrictInfo **p_restrictinfo, JoinDomain *jdomain)

入力と出力

  • **入力**:
  • root: プランナのグローバル情報。
  • p_restrictinfo: 処理対象の制約情報（RestrictInfo）。
  • jdomain: 定数マッチングのスコープを決定するドメイン。
  • **出力**:
  • 戻り値: 等価クラスへの登録が成功した場合は true、それ以外は false。
  • p_restrictinfo: 処理過程で X=X 型の述語が厳密（strict）な演算子である場合、IS NOT NULL テストに変換され更新される。

前提条件と不変条件

  • restrictinfo->left_ec および right_ec が NULL であること（未処理であること）。
  • セキュリティ上の理由で遅延可能な制約かつ leakproof でない場合は処理を中断する。
  • 揮発性（volatile）な等価クラスとはマッチングを行わない。

アルゴリズムまたは振る舞い

1. **正規化**: canonicalize_ec_expression を用いて入力式を正規化する。 2. **自明な等価性の処理**: X=X のような式の場合、演算子が厳密であれば IS NOT NULL 制約に変換し、false を返す。 3. **マッチング**: root->eq_classes を走査し、以下のいずれかの処理を行う。

  • 両方の式が同一クラスにある場合: 何もしない。
  • 両方の式が異なるクラスにある場合: 両クラスをマージする。
  • 片方のみ見つかった場合: 見つからなかった式を既存クラスに追加する。
  • どちらも見つからない場合: 新規に等価クラスを作成し、両式を追加する。

    • 4. **制約の適用**: 照合には照合順序（collation）と演算子族（opfamilies）の一致を条件とする。

関連データ構造

  • EquivalenceClass: 等価な式の集合。
  • EquivalenceMember: クラス内の個別の式。
  • RestrictInfo: 処理対象となる制約。

失敗モードと境界条件

  • X=X のような式で演算子が非厳密な場合、最適化を行わず false を返す。
  • 揮発性関数を含む等価クラスはマッチング対象外となる。
  • 定数マッチングは JoinDomain が一致する場合のみ行われる。

保守担当者向けメモ

  • 本関数は root->eq_classes を直接操作する。
  • ec_childmembers は現時点では存在しない前提で Assert が記述されている。
  • simplehash による高速化の閾値や、継承テーブルの扱い（ec_childmembers）については equivclass.c の全体設計を参照すること。
