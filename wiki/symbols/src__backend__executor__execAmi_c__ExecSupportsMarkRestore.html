ExecSupportsMarkRestore

定義

  • 種別: function
  • 定義箇所: src/backend/executor/execAmi.c:417

目的

プランニング段階において、指定された Path ノードが「マーク（Mark）」および「リストア（Restore）」操作をサポートしているかを判定する。

シグネチャまたは形状

bool ExecSupportsMarkRestore(Path *pathnode)

入力と出力

  • **入力**: Path *pathnode（判定対象のパスノード）
  • **出力**: bool（サポートしている場合は true、それ以外は false）

前提条件と不変条件

  • pathnode は有効な Path 構造体へのポインタであること。
  • ノードの判定には nodeTag ではなく、プランノードの種類を示す pathtype が使用される。

アルゴリズムまたは振る舞い

pathnode->pathtype に基づき、以下の通り判定を行う：

  • **IndexScan / IndexOnlyScan**: インデックスの amcanmarkpos フラグを参照し、AM がサポートしている場合にのみ true を返す。
  • **Material / Sort**: 常に true を返す。
  • **CustomScan**: CUSTOMPATH_SUPPORT_MARK_RESTORE フラグが設定されている場合に true を返す。
  • **Result**: 子ノードが存在する場合、その子ノードのサポート状況を再帰的に確認する。子ノードを持たない場合は false を返す。
  • **Append / MergeAppend**: サブパスがちょうど1つ存在する場合、その子ノードのサポート状況を再帰的に確認する。それ以外の場合は false を返す。
  • **その他**: 上記以外は false を返す。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/optimizer/path/costsize.c（コスト見積もり時）
  • src/backend/executor/execAmi.c（再帰呼び出し）
  • **呼び出し先**:
  • ExecSupportsMarkRestore（再帰呼び出し）

関連データ構造

  • Path
  • IndexPath
  • CustomPath
  • ProjectionPath
  • AppendPath
  • MergeAppendPath

失敗モードと境界条件

  • 子ノードを持たない Result ノードや、複数のサブパスを持つ Append/MergeAppend ノードは、マーク/リストアをサポートしないものとして扱われる。
  • インデックススキャンの場合、インデックスの物理的な特性（amcanmarkpos）に依存する。

保守担当者向けメモ

  • 新しいプランノードタイプを追加した際、そのノードがマーク/リストアをサポートする場合は、本関数の switch 文にケースを追加する必要がある。
  • Result ノードの判定において、ProjectionPath 等の複数のパスタイプが混在するため、IsA マクロによる型チェックを適切に行うこと。
