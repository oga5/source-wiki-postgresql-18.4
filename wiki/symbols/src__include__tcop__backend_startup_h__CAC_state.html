CAC_state

定義

  • 種別: enum
  • 定義箇所: src/include/tcop/backend_startup.h:33

目的

Postmasterからバックエンドプロセスに対し、現在のサーバー状態に基づいた接続受け入れ可否を伝達するために使用されます。

シグネチャまたは形状

typedef enum CAC_state
{
    CAC_OK,             /* 接続許可 */
    CAC_STARTUP,        /* 起動中 */
    CAC_SHUTDOWN,       /* シャットダウン中 */
    CAC_RECOVERY,       /* リカバリ中 */
    CAC_NOTHOTSTANDBY,  /* ホットスタンバイ不可 */
    CAC_TOOMANY,        /* 接続数上限超過 */
} CAC_state;

入力と出力

  • 入力: Postmasterプロセス内の canAcceptConnections() 関数によって現在のサーバー状態から算出されます。
  • 出力: バックエンドプロセスの初期化処理（BackendInitialize）へ渡され、接続を継続するか拒否するかの判断基準として利用されます。

前提条件と不変条件

  • Postmasterからフォークされた直後のバックエンドプロセスに引き継がれる必要があります。
  • CAC_OK 以外の値が渡された場合、バックエンドは適切なエラー処理を行い、接続を終了させる必要があります。

アルゴリズムまたは振る舞い

1. Postmasterが canAcceptConnections() を呼び出し、現在のサーバー状態を CAC_state 値として取得します。 2. この値がバックエンド起動時のデータ構造に含まれ、バックエンドプロセスへ渡されます。 3. バックエンドプロセスは BackendInitialize() 内でこの値を確認し、接続の可否を判定します。

呼び出し元と呼び出し先

  • 呼び出し元:
  • src/backend/postmaster/postmaster.c: canAcceptConnections() 関数がこの値を生成します。
  • 呼び出し先:
  • src/backend/tcop/backend_startup.c: BackendInitialize() 関数が引数として受け取り、接続処理の制御に使用します。

関連データ構造

  • BackendStartupData: バックエンド起動時に引き継がれる構造体であり、CAC_state を保持します。

失敗モードと境界条件

  • CAC_TOOMANY: 接続数が最大値に達している場合に発生します。
  • CAC_RECOVERY / CAC_SHUTDOWN: サーバーが通常運用外の状態にある場合に発生します。
  • 境界条件: 認証プロセス以前の初期化フェーズで評価されるため、これら以外の状態（認証失敗など）は別のメカニズムで処理されます。

保守担当者向けメモ

  • 本列挙型の値を追加・変更する場合は、src/backend/postmaster/postmaster.c 内の判定ロジックおよび src/backend/tcop/backend_startup.c 内の初期化ロジックの両方を更新する必要があります。
