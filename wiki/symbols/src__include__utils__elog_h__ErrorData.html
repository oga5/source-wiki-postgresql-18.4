ErrorData

定義

  • 種別: struct
  • 定義箇所: src/include/utils/elog.h:432

目的

PostgreSQL のエラー報告プロセスにおいて、発生したエラーに関する詳細情報（メッセージ、コンテキスト、SQLSTATE、発生位置など）を保持・伝播するための構造体です。

シグネチャまたは形状

typedef struct ErrorData {
    int         elevel;
    bool        output_to_server;
    bool        output_to_client;
    bool        hide_stmt;
    bool        hide_ctx;
    const char *filename;
    int         lineno;
    const char *funcname;
    const char *domain;
    const char *context_domain;
    int         sqlerrcode;
    char       *message;
    char       *detail;
    char       *detail_log;
    char       *hint;
    char       *context;
    char       *backtrace;
    const char *message_id;
    char       *schema_name;
    char       *table_name;
    char       *column_name;
    char       *datatype_name;
    char       *constraint_name;
    int         cursorpos;
    int         internalpos;
    char       *internalquery;
    int         saved_errno;
    struct MemoryContextData *assoc_context;
} ErrorData;

入力と出力

  • **入力**: ereport() マクロ呼び出し時に収集される各種エラー属性。
  • **出力**: ログ出力先（stderr, syslog, CSV, JSON 等）への書き出し、またはクライアントへのエラー通知。

前提条件と不変条件

  • ereport() 呼び出しの過程で初期化され、errfinish() によって処理が完了する。
  • PG_TRY ブロック内で使用される場合、関連する非定数文字列を保持するメモリコンテキスト（assoc_context）が適切に管理されている必要がある。

アルゴリズムまたは振る舞い

1. **初期化**: ereport() が呼び出されると、errstart() により ErrorData が生成される。 2. **構築**: errmsg(), errcode() 等の関数が、この構造体の各フィールドに情報を蓄積する。 3. **完了**: errfinish() が呼び出されると、構造体の内容に基づきログ出力が行われる。ERROR 以上のレベルでは、トランザクションの異常終了処理（siglongjmp）が実行される。

呼び出し元と呼び出し先

  • **呼び出し元（直接的な参照の例）**:
  • contrib/postgres_fdw/connection.c: CopyErrorData()
  • src/backend/libpq/pqmq.c: pq_parse_errornotice()
  • src/backend/utils/error/csvlog.c: write_csvlog()
  • src/backend/replication/logical/reorderbuffer.c: CopyErrorData()
  • **呼び出し先**: 構造体自体はデータ保持用であり、直接的な関数呼び出しは行わない。

関連データ構造

  • MemoryContextData: エラーに関連する動的文字列を保持するメモリ領域。
  • StringInfo: ログメッセージ構築に使用される文字列バッファ。

失敗モードと境界条件

  • **再帰的エラー**: エラー処理中にさらにエラーが発生した場合、elog.c 内の安全策（エスケープハッチ）が機能する。
  • **メモリ不足**: エラーメッセージの構築中にメモリ不足が発生した場合の挙動は、MemoryContext の実装に依存する。

保守担当者向けメモ

  • ereport は頻繁に呼び出されるため、メッセージ生成のコストに留意すること。
  • PG_TRY ブロック内で使用するローカル変数は、コンパイラの最適化による不整合を防ぐため volatile 宣言が必須である。
  • PANIC レベルのエラーはプロセスを強制終了させるため、使用には細心の注意を払うこと。
