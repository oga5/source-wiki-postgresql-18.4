GetVisibilityMapPins

定義

  • 種別: function
  • 定義箇所: src/backend/access/heap/hio.c:139

目的

ヒープページが「全可視（all-visible）」である場合に、対応する Visibility Map（VM）のピンを確保し、整合性を維持する。

シグネチャまたは形状

static bool GetVisibilityMapPins(Relation relation, Buffer buffer1, Buffer buffer2,
                                 BlockNumber block1, BlockNumber block2,
                                 Buffer *vmbuffer1, Buffer *vmbuffer2)

入力と出力

  • **入力**:
  • relation: 対象のリレーション。
  • buffer1, buffer2: 処理対象のバッファ。
  • block1, block2: 対応するブロック番号。
  • vmbuffer1, vmbuffer2: VM ピンを格納するバッファポインタ。
  • **出力**:
  • 戻り値 (bool): ロックの解放と再取得が発生した場合は true、それ以外は false。
  • vmbuffer1, vmbuffer2: 確保された VM ピン。

前提条件と不変条件

  • buffer1 は有効である必要がある。
  • buffer2 が有効な場合、デッドロック回避のため block1 <= block2 である必要がある（関数内で自動的に順序を入れ替える）。

アルゴリズムまたは振る舞い

1. **順序調整**: block1 > block2 の場合、引数を入れ替えてロック順序を正規化する。 2. **ピン確認**: PageIsAllVisible を用いてページが全可視か確認し、未ピン留めであればピンが必要と判断する。 3. **ロック解放とI/O**: ピンが必要な場合、デッドロックを防ぐため保持しているバッファロックを一時的に解放する。 4. **ピン取得**: visibilitymap_pin を呼び出し、必要な VM ピンを取得する。 5. **再ロック**: 解放したバッファを排他モードで再ロックする。 6. **再試行**: 2つのバッファを扱う際、片方のみピン留めした後に他方が全可視状態に変化する可能性があるため、必要に応じてループを継続する。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/access/heap/hio.c 内の RelationGetBufferForTuple 等から呼び出される。
  • **呼び出し先**: visibilitymap_pin, visibilitymap_pin_ok, LockBuffer, BufferGetPage 等。

関連データ構造

  • Relation: リレーション記述子。
  • Buffer: バッファマネージャのバッファ識別子。
  • BlockNumber: ブロック番号。

失敗モードと境界条件

  • buffer2 が InvalidBuffer の場合、単一のバッファのみを対象として処理する。
  • ページが全可視でない場合、ピン留め処理はスキップされる。
  • ロックの解放と再取得が発生するため、呼び出し元はバッファの内容が変更されている可能性を考慮する必要がある。

保守担当者向けメモ

  • 本関数は、バッファロックを保持したまま I/O を伴う VM ピン取得を行うための再試行ロジックを含んでいる。
  • ロック順序のルール（ページ番号の昇順）を変更する場合は、本関数の入れ替えロジックも整合させる必要がある。
