StoreViewQuery

定義

  • 種別: function
  • 定義箇所: src/backend/commands/view.c:510

目的

ビューの定義クエリをルールシステムに保存するためのラッパー関数です。ビュー作成または置換処理の最終段階において、指定されたクエリを ON SELECT ルールとして登録します。

シグネチャまたは形状

void StoreViewQuery(Oid viewOid, Query *viewParse, bool replace)

入力と出力

  • viewOid (Oid): 対象となるビューのリレーションOID。
  • viewParse (Query *): ビューの定義として保存する解析済みのクエリツリー。
  • replace (bool): CREATE OR REPLACE VIEW の場合は true、新規作成の場合は false を指定します。
  • 出力: なし（副作用としてルールシステムへ登録）。

前提条件と不変条件

  • viewOid は有効なリレーション（ビュー）を指している必要があります。
  • viewParse は解析済みであり、ビューの定義として妥当なクエリ構造である必要があります。

アルゴリズムまたは振る舞い

1. 受け取った viewOid、viewParse、replace フラグをそのまま DefineViewRules へ渡します。 2. DefineViewRules を呼び出し、ビューに関連付けられたルール（ON SELECT ルール）の作成・更新を実行します。

呼び出し元と呼び出し先

  • 呼び出し元:
  • src/backend/commands/createas.c: CREATE TABLE AS 等の文脈で呼び出されます。
  • src/backend/commands/view.c: DefineView 内でビュー作成・置換処理の一部として呼び出されます。
  • 呼び出し先:
  • DefineViewRules (同ファイル内)

関連データ構造

  • Query: SQL解析後のクエリツリー構造体。
  • Oid: PostgreSQLのオブジェクト識別子。

失敗モードと境界条件

  • ルールシステムへの登録（DefineViewRules）が失敗した場合、トランザクションのロールバックが発生する可能性があります。
  • 既存のルールとの競合や、カタログ操作における権限不足などが失敗の原因となります。

保守担当者向けメモ

  • 本関数は単なるラッパーであり、実質的なロジックは DefineViewRules に集約されています。
  • createas.c からの呼び出し時には、クエリツリーが破壊的に変更される可能性があるため、必要に応じてコピーを渡す実装となっています。
