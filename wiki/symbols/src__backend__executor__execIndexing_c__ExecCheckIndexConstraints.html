ExecCheckIndexConstraints

定義

  • 種別: function
  • 定義箇所: src/backend/executor/execIndexing.c:542

目的

テーブルへのタプル挿入前に、関連する一意性制約および排他制約の違反がないかを検証する。

シグネチャまたは形状

bool ExecCheckIndexConstraints(ResultRelInfo *resultRelInfo, TupleTableSlot *slot,
                               EState *estate, ItemPointer conflictTid,
                               ItemPointer tupleid, List *arbiterIndexes)

入力と出力

  • **入力**:
  • resultRelInfo: 対象リレーションのインデックス情報を含む構造体。
  • slot: 検証対象のタプル。
  • estate: 実行状態。
  • conflictTid: 競合が発生した場合、そのタプルIDが格納される（出力用）。
  • tupleid: 更新対象のタプルID（新規挿入時は無効値）。
  • arbiterIndexes: 検証対象とするインデックスのリスト（指定がない場合は全インデックス）。
  • **出力**:
  • bool: 制約をすべて満たす場合は true、違反がある場合は false を返す。

前提条件と不変条件

  • arbiterIndexes が指定されている場合、対象インデックスは即時制約（indimmediate が真）である必要がある。遅延制約が含まれる場合はエラーとなる。
  • インデックスが ii_ReadyForInserts でない場合は検証対象から除外される。

アルゴリズムまたは振る舞い

1. resultRelInfo からインデックス情報を取得し、各インデックスを順次走査する。 2. インデックスが「一意性制約」または「排他制約」を持ち、かつ挿入準備が完了しているかを確認する。 3. 部分インデックス（ii_Predicate が存在）の場合、ExecQual を用いて述語を評価し、不適合な場合はスキップする。 4. FormIndexDatum を呼び出し、インデックスキー値を生成する。 5. check_exclusion_or_unique_constraint を呼び出し、制約違反をチェックする。

  • 競合が検出された場合、conflictTid に該当タプルIDを格納し、直ちに false を返す。

    • 6. すべての対象インデックスで違反がなければ true を返す。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/executor/execReplication.c
  • src/backend/executor/nodeModifyTable.c
  • **呼び出し先**:
  • check_exclusion_or_unique_constraint
  • ExecPrepareQual / ExecQual
  • FormIndexDatum

関連データ構造

  • ResultRelInfo: インデックス情報（ri_IndexRelationInfo）を保持。
  • IndexInfo: インデックスの制約属性や述語情報を保持。
  • TupleTableSlot: 検証対象タプルのコンテナ。

失敗モードと境界条件

  • **エラー**: 遅延制約（indimmediate が偽）を arbiterIndexes として指定した場合、ERRCODE_OBJECT_NOT_IN_PREREQUISITE_STATE でエラー終了する。
  • **エラー**: arbiterIndexes が指定されているにもかかわらず、該当するインデックスが見つからなかった場合は elog(ERROR) を発生させる。

保守担当者向けメモ

  • 本関数は INSERT ... ON CONFLICT の競合検出において中心的な役割を果たす。
  • 排他制約のチェックは挿入後に行われるため、デッドロック回避のための待機ロジック（CEOUC_WAIT）が check_exclusion_or_unique_constraint 内で利用されている点に留意すること。
