SICleanupQueue

定義

  • 種別: function
  • 定義箇所: src/backend/storage/ipc/sinvaladt.c:576

目的

共有キャッシュ無効化メッセージの循環バッファを整理し、全バックエンドの進捗に基づいて不要になった古いメッセージを解放する。また、処理が極端に遅延しているバックエンドを特定し、シグナル送信やリセット状態への移行を行う。

シグネチャまたは形状

void SICleanupQueue(bool callerHasWriteLock, int minFree)

入力と出力

  • callerHasWriteLock (bool): 呼び出し元が既に SInvalWriteLock を保持しているかを示す。
  • minFree (int): 確保すべき最小空き領域数。
  • 出力なし（副作用として共有メモリ上の SISeg 構造体および各 ProcState を更新）。

前提条件と不変条件

  • SInvalReadLock および SInvalWriteLock を排他モードで取得する（呼び出し元が既に書き込みロックを保持している場合を除く）。
  • SISeg 構造体の maxMsgNum および各プロセスの nextMsgNum が整合性を保っていること。
  • resetState または sendOnly のバックエンドはクリーンアップ対象から除外される。

アルゴリズムまたは振る舞い

1. **ロック取得**: SInvalReadLock を排他取得（書き込みロック未保持時は SInvalWriteLock も取得）。 2. **進捗スキャン**: 全バックエンドの nextMsgNum を走査し、以下の処理を行う。

  • lowbound（maxMsgNum - MAXNUMMESSAGES + minFree）より遅れているプロセスを resetState = true に設定。
  • 全バックエンドの最小 nextMsgNum を算出し、segP->minMsgNum を更新。
  • SIG_THRESHOLD を超えて遅延しているプロセスを特定し、needSig に記録。

    • 3. **カウンタ正規化**: minMsgNum が MSGNUMWRAPAROUND を超えた場合、全メッセージカウンタから定数を減算しオーバーフローを防止。 4. **閾値更新**: 残りのメッセージ数に基づき、次回のクリーンアップ実行タイミング（nextThreshold）を計算。 5. **シグナル送信**: needSig が存在する場合、ロックを解放してから SendProcSignal を実行し、遅延プロセスに PROCSIG_CATCHUP_INTERRUPT を通知。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/storage/ipc/sinval.c（メッセージ挿入時など）
  • src/backend/storage/ipc/sinvaladt.c（内部的なバッファ管理）
  • **呼び出し先**:
  • SendProcSignal（遅延プロセスへの通知）
  • elog（デバッグログ出力）
  • LWLockAcquire / LWLockRelease（ロック制御）

関連データ構造

  • SISeg: 共有メモリ上のメイン構造体。
  • ProcState: 各バックエンドの進捗状態を保持する構造体。
  • SharedInvalidationMessage: キュー内のメッセージ単位。

失敗モードと境界条件

  • **バッファ溢れ**: lowbound を下回るプロセスは resetState となり、強制的にキャッシュ再構築を促される。
  • **カウンタラップアラウンド**: MSGNUMWRAPAROUND に達すると全カウンタがシフトされ、数値のオーバーフローを防ぐ。
  • **ロック競合**: SendProcSignal 実行中はロックを解放し、再取得を行うことでデッドロックを回避している。

保守担当者向けメモ

  • 本関数は SInvalReadLock を排他モードで取得するため、頻繁な呼び出しはパフォーマンスに影響を与える可能性がある。nextThreshold による制御が適切に機能しているか注意すること。
  • SendProcSignal は低速な処理であるため、ロック保持時間を最小化する設計となっている。
