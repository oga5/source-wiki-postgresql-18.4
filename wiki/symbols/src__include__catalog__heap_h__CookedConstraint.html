CookedConstraint

定義

  • 種別: struct
  • 定義箇所: src/include/catalog/heap.h:35

目的

パーサーによって生成された未加工の制約式を、システムカタログへの登録や実行が可能な形式（クック済み）に変換した情報を保持するために使用されます。

シグネチャまたは形状

typedef struct CookedConstraint
{
    ConstrType  contype;        /* 制約タイプ (DEFAULT, CHECK, NOTNULL) */
    Oid         conoid;         /* 作成済みであればOID、未作成ならInvalid */
    char       *name;           /* 制約名、またはNULL */
    AttrNumber  attnum;         /* 対象属性番号 (NOTNULL, DEFAULTのみ) */
    Node       *expr;           /* 変換済みのデフォルト式またはチェック式 */
    bool        is_enforced;    /* 強制されるか (CHECKのみ) */
    bool        skip_validation;/* 検証をスキップするか (CHECKのみ) */
    bool        is_local;       /* ローカル定義か */
    int16       inhcount;       /* 継承回数 */
    bool        is_no_inherit;  /* 継承不可のローカル定義か */
} CookedConstraint;

入力と出力

  • **入力**: テーブル作成や変更時にパーサーから渡される未加工の式が、cookDefault 等の関数を通じて本構造体に変換されます。
  • **出力**: AddRelationNewConstraints 等の関数により、本構造体の情報がシステムカタログ（pg_constraint 等）へ永続化されます。

前提条件と不変条件

  • expr フィールドは、解析済み（transformed）の Node である必要があります。
  • attnum は CONSTR_NOTNULL または CONSTR_DEFAULT の場合にのみ有効です。
  • is_enforced および skip_validation は CONSTR_CHECK の場合にのみ適用されます。

アルゴリズムまたは振る舞い

1. テーブル作成時、未加工の制約式が解析され、CookedConstraint 構造体のリストとして生成されます。 2. AddRelationNewConstraints が呼び出されると、リスト内の各 CookedConstraint が処理されます。 3. 処理の過程で、各構造体には新しく生成されたカタログタプルの OID が格納されます。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/catalog/heap.c 内のテーブル作成および制約追加処理（AddRelationNewConstraints など）。
  • **呼び出し先**: 本構造体自体はデータ保持用であり、直接関数を呼び出すことはありませんが、カタログ操作関数によって参照・更新されます。

関連データ構造

  • RawColumnDefault: 未変換のデフォルト値式を保持する構造体。
  • Node: 解析ツリーの基底構造体。
  • ConstrType: 制約の種類を定義する列挙型。

失敗モードと境界条件

  • conoid が InvalidOid の場合、制約がまだカタログに登録されていないことを示します。
  • 継承された制約の場合、inhcount が適切に管理されないと整合性が失われる可能性があります。

保守担当者向けメモ

  • 本構造体は heap_create_with_catalog を通じたテーブル定義のライフサイクルにおいて中心的な役割を果たします。
  • カタログスキーマの変更に伴い、制約の属性（継承フラグや検証状態など）を追加・変更する場合は、本構造体の定義と heap.c 内の変換ロジックの両方を更新する必要があります。
