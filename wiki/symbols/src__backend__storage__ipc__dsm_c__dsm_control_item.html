dsm_control_item

定義

  • 種別: struct
  • 定義箇所: src/backend/storage/ipc/dsm.c:79

目的

動的共有メモリ（DSM）セグメントのメタデータとライフサイクル状態を、システム全体で共有される制御領域内で管理するために使用されます。

シグネチャまたは形状

typedef struct dsm_control_item
{
    dsm_handle  handle;
    uint32      refcnt;
    size_t      first_page;
    size_t      npages;
    void       *impl_private_pm_handle;
    bool        pinned;
} dsm_control_item;

入力と出力

本構造体は関数ではなくデータ構造であるため、入力および出力の概念は適用されません。各フィールドは DSM セグメントの識別子、参照カウント、メモリ配置情報、およびプラットフォーム固有のハンドルを保持します。

前提条件と不変条件

  • refcnt フィールドは以下の状態を示します：
  • 2 以上: セグメントはアクティブ。
  • 1: セグメントは「死にかけ（moribund）」の状態であり、破棄待ち。
  • 0: セグメントは存在しない（未使用スロット）。
  • impl_private_pm_handle は Windows 環境でのみ使用されます。

アルゴリズムまたは振る舞い

DSM セグメントの作成時や接続時に、制御領域内の dsm_control_item 配列が操作されます。参照カウントの増減を通じて、最後の利用者がデタッチした際にセグメントの破棄をトリガーする仕組みの基盤となります。

呼び出し元と呼び出し先

  • 呼び出し元: dsm.c 内の DSM 管理関数（dsm_create, dsm_attach, dsm_detach 等）が本構造体を含む配列を操作します。
  • 呼び出し先: 本構造体自体はデータ保持用であり、直接関数を呼び出すことはありません。

関連データ構造

  • dsm_control_header: 本構造体を配列として保持する親構造体。
  • dsm_segment: バックエンドローカルなセグメント管理構造体。

失敗モードと境界条件

  • 制御領域内のスロットが枯渇した場合、新しい DSM セグメントの作成が失敗します。
  • refcnt の不整合は、セグメントの早期解放やリークを引き起こす可能性があります。

保守担当者向けメモ

  • dsm_control_item の配列は dsm_control_header 内で FLEXIBLE_ARRAY_MEMBER を用いて定義されています。サイズ計算時には sizeof(dsm_control_item) を用いる必要があります。
  • Windows 固有の impl_private_pm_handle の取り扱いには注意してください。
