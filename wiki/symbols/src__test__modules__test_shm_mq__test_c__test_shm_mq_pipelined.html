test_shm_mq_pipelined

定義

  • 種別: function
  • 定義箇所: src/test/modules/test_shm_mq/test.c:131

目的

PostgreSQLの共有メモリメッセージキュー（shm_mq）のノンブロッキングI/Oインターフェースを用いたパイプライン処理の検証。送信と受信を並行して行い、キューの輻輳やデッドロック耐性をテストする。

シグネチャまたは形状

Datum test_shm_mq_pipelined(PG_FUNCTION_ARGS)

SQL関数として呼び出され、以下の引数を取る： 1. queue_size (int64): キューのサイズ 2. message (text): 送信するメッセージ内容 3. loop_count (int32): 送受信の繰り返し回数 4. nworkers (int32): バックグラウンドワーカー数 5. verify (bool): 受信データの整合性検証を行うか否か

入力と出力

  • **入力**: SQL経由で渡されるパラメータ（上記シグネチャ参照）。
  • **出力**: PG_RETURN_VOID() により値を返さない。エラー発生時は ereport により例外を投げる。

前提条件と不変条件

  • loop_count および nworkers は0以上であること。
  • test_shm_mq_setup が正常に共有メモリセグメントとキューハンドルを初期化できること。
  • shm_mq_send が SHM_MQ_WOULD_BLOCK を返した場合、同一のメッセージ内容で再試行する必要がある。

アルゴリズムまたは振る舞い

1. test_shm_mq_setup を呼び出し、共有メモリとワーカーを準備。 2. ループ内で shm_mq_send と shm_mq_receive をノンブロッキングモードで呼び出す。 3. 送信または受信が成功するたびに進捗フラグを立てる。 4. 進捗がない（wait が真の）場合、WaitLatch を使用して他のプロセスからの通知を待機する。 5. receive_count が loop_count に達した時点で、send_count との整合性を確認しループを終了。 6. 最後に dsm_detach で共有メモリセグメントを解放。

呼び出し元と呼び出し先

  • **呼び出し元**: SQLインターフェース（PG_FUNCTION_INFO_V1 による登録）。
  • **呼び出し先**: test_shm_mq_setup, shm_mq_send, shm_mq_receive, verify_message, WaitLatch, dsm_detach。

関連データ構造

  • dsm_segment: 動的共有メモリセグメント。
  • shm_mq_handle: メッセージキューのハンドル。
  • we_message_queue: 待機イベントID。

失敗モードと境界条件

  • loop_count または nworkers が負の場合: ERRCODE_INVALID_PARAMETER_VALUE でエラー。
  • shm_mq_send/receive が SHM_MQ_DETACHED を返した場合: ERRCODE_OBJECT_NOT_IN_PREREQUISITE_STATE でエラー。
  • ループ終了時に send_count != receive_count の場合: ERRCODE_INTERNAL_ERROR でエラー。

保守担当者向けメモ

  • WaitLatch の待機イベントとして we_message_queue を使用しており、初回呼び出し時に WaitEventExtensionNew で生成される。
  • verify_message は比較処理が重いため、パイプラインテストのパフォーマンスに影響を与える可能性がある。
  • ノンブロッキングI/Oの特性上、shm_mq_send 失敗時の再試行ロジックが重要である。
