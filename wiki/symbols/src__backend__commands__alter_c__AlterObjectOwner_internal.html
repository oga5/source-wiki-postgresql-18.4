AlterObjectOwner_internal

定義

  • 種別: function
  • 定義箇所: src/backend/commands/alter.c:925

目的

データベースオブジェクトの所有者を変更し、関連するカタログ情報の更新および依存関係の再構築を行う。

シグネチャまたは形状

void AlterObjectOwner_internal(Oid classId, Oid objectId, Oid new_ownerId)

入力と出力

  • **入力**:
  • classId: 対象オブジェクトのカタログOID。
  • objectId: 対象オブジェクトのOID。
  • new_ownerId: 新しい所有者のロールOID。
  • **出力**: なし（副作用としてカタログを更新）。

前提条件と不変条件

  • 対象オブジェクトが存在すること。
  • 呼び出し元が適切なロックを取得していること（本関数内で RowExclusiveLock を使用）。
  • 所有者変更が不要（old_ownerId == new_ownerId）な場合、カタログ更新はスキップされる。

アルゴリズムまたは振る舞い

1. **カタログ特定**: classId に基づき対象のシステムカタログを特定する。 2. **タプル取得**: get_catalog_object_by_oid_extended を使用して対象オブジェクトのタプルを取得・ロックする。 3. **権限検証**: スーパーユーザーでない場合、以下のチェックを行う。

  • 現在のユーザーがオブジェクトの所有者であること。
  • 現在のユーザーが new_ownerId にロール変更可能であること。
  • オブジェクトがスキーマに属する場合、new_ownerId がそのスキーマに対する CREATE 権限を持つこと。

    • 4. **カタログ更新**:

  • heap_modify_tuple を用いて所有者属性を更新する。
  • ACL属性が存在する場合、aclnewowner を用いてACLを再計算し更新する。
  • CatalogTupleUpdate でカタログを永続化する。

    • 5. **依存関係更新**: changeDependencyOnOwner を呼び出し、所有者依存関係を更新する。 6. **後処理**: InvokeObjectPostAlterHook を呼び出し、フック処理（イベントトリガー等）を実行してロックを解放する。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/commands/alter.c (エントリポイント経由)
  • src/backend/catalog/pg_shdepend.c
  • **呼び出し先**:
  • table_open, table_close
  • get_catalog_object_by_oid_extended
  • superuser, has_privs_of_role, check_can_set_role, object_aclcheck
  • heap_modify_tuple, CatalogTupleUpdate
  • aclnewowner
  • changeDependencyOnOwner
  • InvokeObjectPostAlterHook

関連データ構造

  • Relation: システムカタログのテーブル構造。
  • HeapTuple: カタログ内のオブジェクト行データ。
  • Acl: アクセス制御リスト。

失敗モードと境界条件

  • **キャッシュルックアップ失敗**: 対象オブジェクトがカタログに見つからない場合、elog(ERROR) を発生させる。
  • **権限エラー**: 所有権がない、またはスキーマ作成権限がない場合、aclcheck_error を発生させる。
  • **Large Object**: classId が LargeObjectRelationId の場合、自動的に LargeObjectMetadataRelationId を対象として処理する。

保守担当者向けメモ

  • 本関数はカタログの属性番号を動的に取得するため、新しいオブジェクト型を追加する際は get_object_attnum_... 系関数が正しく実装されている必要がある。
  • InvokeObjectPostAlterHook に渡されるのは classId であり、内部で使用する catalogId とは異なる場合がある点に注意すること。
