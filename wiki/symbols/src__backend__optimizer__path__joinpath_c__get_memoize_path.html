get_memoize_path

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/path/joinpath.c:674

目的

Nested Loop 結合において、内部リレーションの再スキャンコストを削減するための Memoize パスを生成するかどうかを判定し、生成可能な場合はパスを作成して返します。

シグネチャまたは形状

Path *get_memoize_path(PlannerInfo *root, RelOptInfo *innerrel,
                       RelOptInfo *outerrel, Path *inner_path,
                       Path *outer_path, JoinType jointype,
                       JoinPathExtraData *extra)

入力と出力

  • **入力**:
  • root: プランナのグローバル情報。
  • innerrel, outerrel: 結合対象のリレーション情報。
  • inner_path, outer_path: 検討中の結合パス。
  • jointype: 結合の種類（JOIN_SEMI, JOIN_ANTI 等）。
  • extra: 結合に関する追加メタデータ（inner_unique フラグや制限句リスト等）。
  • **出力**: 条件を満たす場合は Memoize パス（Path *）、それ以外は NULL。

前提条件と不変条件

  • enable_memoize が有効であること。
  • 外部パスの行数が 2 以上であること（1 行のみの場合はキャッシュの恩恵がないため）。
  • 内部パスまたはリレーションに、キャッシュキーとして利用可能なパラメータ化句、Lateral 変数、または PlaceHolderVar が存在すること。
  • 揮発性関数（volatile functions）が内部リレーションのターゲットリストや制限句に含まれていないこと。

アルゴリズムまたは振る舞い

1. **初期チェック**: enable_memoize の確認および外部パスの行数チェックを行う。 2. **キー抽出**: extract_lateral_vars_from_PHVs を使用して、キャッシュキーとなり得る Lateral 変数を特定する。 3. **結合タイプ制約**: SEMI または ANTI 結合の場合、inner_unique が真でない限りパス生成を中止する。 4. **一意性制約**: inner_unique が真の場合、結合条件のすべてがパラメータ化されていることを確認する（完全なキャッシュエントリ完了の保証のため）。 5. **揮発性チェック**: 内部リレーションおよびパラメータ化されたパスの制限句に揮発性関数が含まれていないか検証する。 6. **ハッシュ演算子検証**: paraminfo_get_equal_hashops を呼び出し、パラメータに対するハッシュ演算子の存在を確認する。 7. **パス生成**: すべての条件を満たす場合、create_memoize_path を呼び出してパスを生成し返す。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/optimizer/path/joinpath.c 内の結合パス生成ロジック（add_paths_to_joinrel 等の文脈）。
  • **呼び出し先**:
  • extract_lateral_vars_from_PHVs
  • contain_volatile_functions
  • paraminfo_get_equal_hashops
  • create_memoize_path

関連データ構造

  • Path, RelOptInfo, PlannerInfo, JoinPathExtraData, RestrictInfo

失敗モードと境界条件

  • enable_memoize が無効な場合、即座に NULL を返す。
  • 内部リレーションにキャッシュ可能なキーが存在しない場合、NULL を返す。
  • 揮発性関数が含まれる場合、キャッシュの安全性確保のため NULL を返す。
  • inner_unique な結合において、結合条件の一部がパラメータ化されていない場合、キャッシュの完了判定が困難なため NULL を返す。

保守担当者向けメモ

  • SEMI/ANTI 結合における inner_unique の扱いについては、add_paths_to_joinrel での判定ロジックと密接に関連しているため、変更時は注意が必要。
  • Memoize ノードは Nested Loop のコストを劇的に改善する可能性があるが、キャッシュキーの選択とハッシュ演算子の可用性に強く依存する。
