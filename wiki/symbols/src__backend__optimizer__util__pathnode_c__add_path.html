add_path

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/util/pathnode.c:463

目的

リレーションの候補パスリスト（pathlist）に新しい Path を追加し、コストや制約に基づき既存のパスとの支配関係を判定して、不要なパスの削除や新しいパスの破棄を行うことで、効率的なパスのみを保持する。

シグネチャまたは形状

void add_path(RelOptInfo *parent_rel, Path *new_path)

入力と出力

  • **入力**:
  • parent_rel: 対象となるリレーション情報（RelOptInfo）。
  • new_path: 追加を試みる新しいパス（Path）。
  • **出力**: なし（parent_rel->pathlist を直接更新する）。

前提条件と不変条件

  • CHECK_FOR_INTERRUPTS() が呼び出され、クエリキャンセル要求を監視する。
  • パラメータ化されたパス（param_info を持つ）は、パスキーを持たないものとして扱われる。
  • コスト比較には STD_FUZZ_FACTOR を用いた曖昧な比較（compare_path_costs_fuzzily）が適用される。

アルゴリズムまたは振る舞い

1. parent_rel->pathlist を走査し、各 old_path と new_path を比較する。 2. コスト、パスキー、パラメータ化（required_outer）、行数、並列安全性（parallel_safe）を順次評価する。 3. new_path が old_path を支配している場合、old_path を pathlist から削除し、IndexPath 以外の型であれば pfree でメモリを解放する。 4. old_path が new_path を支配している場合、new_path の追加を中止する。 5. 支配関係がない場合は、両方のパスを保持する。

呼び出し元と呼び出し先

  • **呼び出し元**: contrib/file_fdw および contrib/postgres_fdw 内の各関数（file_fdw.c, postgres_fdw.c 等）。
  • **呼び出し先**: compare_path_costs_fuzzily, compare_pathkeys, bms_subset_compare, foreach_delete_current, pfree。

関連データ構造

  • RelOptInfo: パスリストを保持するリレーション情報。
  • Path: 実行計画の候補となるパス構造体。
  • PathCostComparison, PathKeysComparison, BMS_Comparison: 比較結果を保持する列挙型。

失敗モードと境界条件

  • IndexPath 型のパスは共有される可能性があるため、支配関係で削除対象となっても pfree されない。
  • コストや行数が同等の場合、並列安全性や行数、微小なコスト差を順に評価し、最終的に古いパスを優先する。

保守担当者向けメモ

  • add_path はプランナの頻繁な実行経路であるため、効率的な比較順序（コスト比較を優先）が維持されている。
  • 浮動小数点誤差によるプランの不安定化を防ぐため、厳密な比較ではなく STD_FUZZ_FACTOR を用いたファジー比較が必須である。
