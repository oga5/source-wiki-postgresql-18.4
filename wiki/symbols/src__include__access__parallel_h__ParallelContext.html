ParallelContext

定義

  • 種別: struct
  • 定義箇所: src/include/access/parallel.h:31

目的

PostgreSQLのバックエンドプロセスにおける並列クエリ実行セッション全体を管理するための構造体です。バックグラウンドワーカーのライフサイクル、共有メモリの割り当て、およびプロセス間通信の基盤を提供します。

シグネチャまたは形状

typedef struct ParallelContext
{
	dlist_node	node;
	SubTransactionId subid;
	int			nworkers;		/* 最大起動ワーカー数 */
	int			nworkers_to_launch; /* 実際に起動するワーカー数 */
	int			nworkers_launched;
	char	   *library_name;
	char	   *function_name;
	ErrorContextCallback *error_context_stack;
	shm_toc_estimator estimator;
	dsm_segment *seg;
	void	   *private_memory;
	shm_toc    *toc;
	ParallelWorkerInfo *worker;
	int			nknown_attached_workers;
	bool	   *known_attached_workers;
} ParallelContext;

入力と出力

  • 入力: CreateParallelContext() を通じて生成され、ワーカー数や実行関数名などが設定されます。
  • 出力: 共有メモリセグメント（dsm_segment）および動的共有メモリテーブル（shm_toc）を保持し、リーダープロセスとワーカープロセス間でのデータ共有を可能にします。

前提条件と不変条件

  • トランザクションと密接に結びついており、トランザクション終了時に AtEOXact_Parallel によるクリーンアップが必要です。
  • 適切に破棄（DestroyParallelContext）されない場合、共有メモリのリソースリークが発生します。

アルゴリズムまたは振る舞い

1. **初期化**: CreateParallelContext で構造体が割り当てられ、InitializeParallelDSM で共有メモリが準備されます。 2. **起動**: LaunchParallelWorkers により、設定された数のバックグラウンドワーカーが起動されます。 3. **同期**: リーダープロセスは WaitForParallelWorkersToFinish を使用してワーカーの完了を待機します。 4. **終了**: DestroyParallelContext により共有メモリセグメントが解放され、リソースがクリーンアップされます。

呼び出し元と呼び出し先

  • 呼び出し元（直接的な参照の証拠）:
  • src/backend/access/brin/brin.c
  • src/backend/access/gin/gininsert.c
  • src/backend/access/nbtree/nbtsort.c
  • src/backend/access/transam/parallel.c（ProcessParallelMessage, WaitForParallelWorkersToExit 等）
  • 呼び出し先: 構造体内部で shm_toc や dsm_segment を管理しており、これらに関連するストレージ管理関数を間接的に利用します。

関連データ構造

  • dsm_segment: 共有メモリセグメント。
  • shm_toc: 共有メモリ内のテーブル・オブ・コンテント。
  • ParallelWorkerInfo: 各ワーカーの情報を保持する構造体。
  • dlist_node: コンテキストを管理するための双方向リストノード。

失敗モードと境界条件

  • ワーカーの起動失敗や、共有メモリの割り当て失敗が発生する可能性があります。
  • AtEOXact_Parallel が呼び出されない場合、リソースリークが発生します。

保守担当者向けメモ

  • 本構造体はクエリ実行の並列化専用です。pg_dump の並列処理用構造体とは別物であるため混同しないよう注意してください。
  • IsParallelWorker() マクロを使用して、現在のプロセスがリーダーかワーカーかを判別するコードパスが一般的です。
