ConflictTupleInfo

定義

  • 種別: struct
  • 定義箇所: src/include/replication/conflict.h:66

目的

論理レプリケーションの適用中に発生した競合に関与する、ローカル行のメタデータおよび関連するインデックス情報を保持するために使用されます。

シグネチャまたは形状

typedef struct ConflictTupleInfo
{
    struct TupleTableSlot *slot;
    Oid            indexoid;
    TransactionId  xmin;
    RepOriginId    origin;
    TimestampTz    ts;
} ConflictTupleInfo;

入力と出力

本構造体はデータ保持用であり、関数ではありません。各フィールドには以下の情報が格納されます。

  • slot: 競合したローカル行を保持するタプルスロット。
  • indexoid: 競合が発生したインデックスのOID。
  • xmin: 競合を引き起こした変更のトランザクションID。
  • origin: 変更のレプリケーション元識別子。
  • ts: 競合したローカル行の変更タイムスタンプ。

前提条件と不変条件

  • GetTupleTransactionInfo() を用いて情報を収集する場合、track_commit_timestamp が無効であればタイムスタンプ情報は取得できません。
  • 競合報告時に使用される場合、呼び出し側は渡されるすべてのインデックス情報が有効であることを保証する必要があります。

アルゴリズムまたは振る舞い

論理レプリケーションワーカーがローカルテーブルへの適用中に制約違反等を検知した際、この構造体にメタデータを格納します。その後、この構造体（またはそのリスト）が ReportApplyConflict() に渡され、エラーログの生成および統計情報の更新に使用されます。

呼び出し元と呼び出し先

  • **呼び出し元（直接的な証拠）**:
  • src/backend/executor/execReplication.c
  • src/backend/replication/logical/worker.c
  • src/backend/replication/logical/conflict.c
  • **呼び出し先**:
  • 本構造体自体はデータコンテナであり、特定の関数を呼び出すものではありません。

関連データ構造

  • ConflictType: 競合の種類を定義する列挙型。
  • TupleTableSlot: 競合した行データを保持するために参照されます。

失敗モードと境界条件

  • track_commit_timestamp が無効な環境では、ts フィールドに有効なタイムスタンプが格納されない可能性があります。
  • 競合発生時にインデックスが特定できない場合、indexoid が適切に設定されない可能性があります。

保守担当者向けメモ

  • 本構造体は execnodes.h 等の重いヘッダーのインクルードを避けるため、TupleTableSlot を前方宣言で扱っています。
  • 統計情報（PgStat）の収集コードと密接に関連しているため、フィールドの追加・変更時には統計情報の整合性に注意してください。
