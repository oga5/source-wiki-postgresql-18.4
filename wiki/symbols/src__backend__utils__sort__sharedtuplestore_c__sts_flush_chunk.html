sts_flush_chunk

定義

  • 種別: function
  • 定義箇所: src/backend/utils/sort/sharedtuplestore.c:195

目的

SharedTuplestoreAccessor が保持する現在の書き込み用チャンクバッファを一時ファイルへフラッシュし、バッファを初期化して次の書き込みに備える。

シグネチャまたは形状

static void sts_flush_chunk(SharedTuplestoreAccessor *accessor)

入力と出力

  • **入力**: accessor (SharedTuplestoreAccessor *) - 書き込み対象のファイルハンドルおよびバッファ状態を保持する構造体。
  • **出力**: なし（副作用としてファイル書き込みと構造体の状態更新を行う）。

前提条件と不変条件

  • accessor->write_file が有効な BufFile であること。
  • accessor->write_chunk が STS_CHUNK_PAGES * BLCKSZ のサイズで確保されていること。

アルゴリズムまたは振る舞い

1. STS_CHUNK_PAGES * BLCKSZ 分のデータを BufFileWrite を用いて accessor->write_file に書き出す。 2. memset を使用して accessor->write_chunk 全体をゼロクリアする。 3. accessor->write_pointer をチャンク内のデータ領域先頭（&accessor->write_chunk->data[0]）にリセットする。 4. 共有状態（accessor->sts->participants）内の当該参加者のページカウント（npages）を STS_CHUNK_PAGES 分加算する。

呼び出し元と呼び出し先

  • **呼び出し先**: BufFileWrite（storage/buffile.h）および memset（標準ライブラリ）。

関連データ構造

  • SharedTuplestoreAccessor: 操作の起点となる構造体。
  • SharedTuplestore: 共有メモリ上の制御構造体。
  • BufFile: 一時ファイルの抽象化レイヤー。

失敗モードと境界条件

  • BufFileWrite が失敗した場合、PostgreSQL の標準的なエラー処理（ereport 等）により中断される可能性がある。
  • チャンクサイズ（STS_CHUNK_PAGES * BLCKSZ）が固定であるため、バッファオーバーフローの懸念はないが、ディスク容量不足による書き込み失敗の可能性がある。

保守担当者向けメモ

  • 本関数は書き込み専用の処理であり、ロックは不要である（各バックエンドが専用ファイルを持つため）。
  • ページカウントの更新は共有メモリ上の構造体に対して行われるため、この関数は書き込みプロセスが自身の担当領域を更新する唯一の手段である。
