ExpandedObjectHeader

定義

  • 種別: struct
  • 定義箇所: src/include/utils/expandeddatum.h:100

目的

PostgreSQLにおいて、メモリ上で複雑なデータ型（配列やレコード等）を効率的に操作するための共通ヘッダを提供します。ディスク上のフラットな形式とメモリ上の展開済み形式を橋渡しし、メモリコンテキスト管理と型固有のメソッド実行を抽象化します。

シグネチャまたは形状

struct ExpandedObjectHeader
{
    int32       vl_len_;        /* EOH_HEADER_MAGIC */
    const ExpandedObjectMethods *eoh_methods;
    MemoryContext eoh_context;
    char        eoh_rw_ptr[EXPANDED_POINTER_SIZE];
    char        eoh_ro_ptr[EXPANDED_POINTER_SIZE];
};

入力と出力

  • **入力**: オブジェクト生成時に eoh_methods（操作関数群）と eoh_context（メモリコンテキスト）が登録されます。
  • **出力**: eoh_rw_ptr および eoh_ro_ptr を通じて、読み取り書き込み可能（R/W）または読み取り専用（R/O）の Datum を提供します。

前提条件と不変条件

  • vl_len_ フィールドは常に EOH_HEADER_MAGIC を保持し、通常の varlena オブジェクトと識別可能である必要があります。
  • オブジェクトは eoh_context に依存しており、このコンテキストが破棄されるとオブジェクトも無効化されます。
  • R/Wポインタを介した操作は、メモリ上で直接データを変更するインプレース更新を許可しますが、失敗時の整合性維持は呼び出し側の責任です。

アルゴリズムまたは振る舞い

1. **初期化**: EOH_init_header によりメソッドテーブルとメモリコンテキストが紐付けられます。 2. **操作**: R/Wポインタを持つ関数は、コピーを作成せずにメモリ上のデータを直接操作します。 3. **変換**: MakeExpandedObjectReadOnly を呼び出すことで、R/WポインタをR/Oポインタへ切り替え可能です。 4. **永続化**: EOH_flatten_into を経由して、展開済みデータをディスク保存用のフラットな varlena 形式へ変換します。

呼び出し元と呼び出し先

  • **呼び出し元**: detoast.c, heaptuple.c, execTuples.c, datum.c 等の各実行エンジンおよびデータ操作レイヤー。
  • **呼び出し先**: array_expanded.c（EA_get_flat_size, EA_flatten_into 等の型固有実装）および expandeddatum.c（EOH_init_header 等の共通関数）。

関連データ構造

  • ExpandedObjectMethods: フラット化処理を定義する関数ポインタ構造体。
  • MemoryContext: オブジェクトの生存期間を管理するメモリコンテキスト。
  • varlena: TOASTポインタおよびデータ構造の基底。

失敗モードと境界条件

  • eoh_context が既に解放されている状態でアクセスした場合、メモリ破壊や不正なメモリ参照が発生します。
  • EOH_HEADER_MAGIC が一致しない場合、VARATT_IS_EXPANDED_HEADER マクロによる判定が失敗し、展開済みオブジェクトとして正しく処理されません。

保守担当者向けメモ

  • 本構造体は varatt.h でも定義されていますが、src/include/utils/expandeddatum.h が主定義です。
  • メモリリークを防ぐため、オブジェクトの生存期間と eoh_context のライフサイクルを厳密に一致させてください。
  • R/Wポインタの変更は副作用を伴うため、トランザクションやエラーハンドリングの文脈で整合性が保たれているか確認してください。
