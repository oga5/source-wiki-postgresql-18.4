pgstat_get_entry_ref

定義

  • **種別**: function
  • **定義箇所**: src/backend/utils/activity/pgstat_shmem.c:456

目的

共有メモリ上の統計情報エントリへの参照（PgStat_EntryRef）を取得します。ローカルキャッシュに存在しない場合は共有メモリ上のハッシュテーブルを検索し、必要に応じて新規作成または再初期化を行います。

シグネチャまたは形状

PgStat_EntryRef *pgstat_get_entry_ref(PgStat_Kind kind, Oid dboid, uint64 objid, bool create, bool *created_entry)

入力と出力

  • **入力**:
  • kind: 統計情報の種類（PgStat_Kind）。
  • dboid: データベースOID。
  • objid: オブジェクトID。
  • create: 存在しない場合に新規作成するかどうか。
  • created_entry: 新規作成された場合に true が設定される出力用ポインタ（任意）。
  • **出力**:
  • 成功時: 該当する統計エントリへの参照（PgStat_EntryRef*）。
  • 失敗時: create が false でエントリが存在しない場合、NULL を返す。

前提条件と不変条件

  • 統計システムが初期化済みであること（pgstat_assert_is_up()）。
  • 共有メモリ上のハッシュテーブルが有効であること。
  • システムがシャットダウン状態でないこと。
  • create が false の場合、created_entry は NULL であること。

アルゴリズムまたは振る舞い

1. **GC実行**: 必要に応じて pgstat_gc_entry_refs() を呼び出し、不要になった参照をクリーンアップする。 2. **ローカルキャッシュ検索**: pgstat_get_entry_ref_cached() を呼び出し、ローカルメモリ上のハッシュテーブルにヒットすれば即座に返す。 3. **共有メモリ検索**: dshash_find() で共有メモリ上のハッシュテーブルを検索する。 4. **新規作成**: create が true でエントリが見つからない場合、dshash_find_or_insert() を実行し、pgstat_init_entry() で初期化する。 5. **再利用**: 既に dropped フラグが立っているエントリに対して create が true の場合、pgstat_reinit_entry() で再初期化を行う。 6. **参照取得**: 最終的に見つかったエントリに対し、pgstat_acquire_entry_ref() を呼び出して参照を確立する。

呼び出し元と呼び出し先

  • **呼び出し元**: pgstat.c（各統計取得関数）、pgstat_database.c、pgstat_replslot.c、および pgstat_shmem.c 内のヘルパー関数。
  • **呼び出し先**: pgstat_need_entry_refs_gc, pgstat_gc_entry_refs, pgstat_get_entry_ref_cached, dshash_find, dshash_find_or_insert, pgstat_init_entry, pgstat_acquire_entry_ref, pgstat_release_entry_ref, pgstat_reinit_entry, dsa_get_address 等。

関連データ構造

  • PgStat_EntryRef: バックエンドが保持する統計エントリへの参照構造体。
  • PgStatShared_HashEntry: 共有メモリ上のハッシュテーブルエントリ。
  • PgStat_HashKey: 統計エントリを一意に特定するキー。

失敗モードと境界条件

  • **メモリ不足**: pgstat_init_entry() による新規エントリ確保に失敗した場合、ERROR を発生させる。
  • **削除済みエントリ**: dropped フラグが立っているエントリに対し create が false の場合、参照を解放して NULL を返す。
  • **同時実行**: dshash_find_or_insert() を使用することで、他バックエンドによる同時挿入を安全に処理する。

保守担当者向けメモ

  • 本関数は頻繁に呼び出されるため、ローカルキャッシュによる最適化が重要です。
  • dropped フラグが立っているエントリの再利用ロジックは、OIDの再利用やレプリケーションスロットの挙動に関連するため、変更時は注意が必要です。
