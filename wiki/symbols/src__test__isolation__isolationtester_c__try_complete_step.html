try_complete_step

定義

  • 種別: function
  • 定義箇所: src/test/isolation/isolationtester.c:817

目的

非同期に発行されたSQLステップの完了を待機、またはブロック条件（PermutationStepBlocker）が満たされるまでポーリングを行い、必要に応じてタイムアウト処理やクエリキャンセルを実行する。

シグネチャまたは形状

static bool try_complete_step(TestSpec *testspec, PermutationStep *pstep, int flags)

入力と出力

  • **入力**:
  • testspec: テスト仕様全体を保持する構造体。
  • pstep: 実行対象の順列ステップ。
  • flags: 動作制御フラグ（STEP_RETRY, STEP_NONBLOCK 等）。
  • **出力**:
  • bool: ステップがまだ進行中（待機中）であれば true、完了していれば false を返す。

前提条件と不変条件

  • conns 配列が初期化されており、pstep->step->session に対応する接続が有効であること。
  • flags に STEP_NONBLOCK が含まれる場合、ロック待ち判定用のプリペアドステートメント PREP_WAITING が事前に準備されていること。

アルゴリズムまたは振る舞い

1. **初期待機チェック**: flags に STEP_RETRY が含まれない場合、PSB_ONCE ブロッカーがあれば即座に待機状態として true を返す。 2. **ポーリングループ**: PQisBusy が真の間、select() を用いてソケットを監視する。

  • **タイムアウト/ロック判定**: select() がタイムアウトした場合、STEP_NONBLOCK フラグがあれば PREP_WAITING を実行してロック待ち状態を判定する。
  • **キャンセル処理**: 経過時間が max_step_wait を超えた場合、PQcancelBlocking でクエリをキャンセルする。さらに 2 * max_step_wait を超えた場合はエラー終了する。

    • 3. **入力消費**: PQconsumeInput を呼び出し、通知（NOTICE）や結果データを取り込む。 4. **完了判定**: ループ終了後、step_has_blocker を確認し、ブロッカーが存在する場合は待機状態として true を返す。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • run_permutation (推測: run_permutation 内のステップ実行ループから呼び出される)
  • try_complete_steps
  • **呼び出し先**:
  • PQsocket, PQisBusy, PQconsumeInput, PQexecPrepared, PQcancelCreate, PQcancelBlocking, PQcancelFinish (libpq関数)
  • step_has_blocker

関連データ構造

  • TestSpec: テスト全体の定義。
  • PermutationStep: 実行中のステップとブロッカー情報。
  • IsoConnInfo: セッションごとの接続状態とバックエンドPID。

失敗モードと境界条件

  • **ソケット異常**: PQsocket が無効な場合、エラーを出力して終了する。
  • **ロック判定失敗**: PREP_WAITING クエリが失敗した場合、エラーを出力して終了する。
  • **タイムアウト**: 2 * max_step_wait を超過すると、テスト全体を異常終了させる。
  • **キャンセル失敗**: PQcancelBlocking が失敗した場合、エラーメッセージを出力する。

保守担当者向けメモ

  • select() のタイムアウト値は 10ms に固定されており、ロック待ちの頻繁なチェックを意図している。
  • STEP_NONBLOCK フラグによるロック判定は、バックエンドの pg_isolation_test_session_is_blocked に依存している。
  • step_has_blocker による最終チェックは、クエリ自体は完了していても、テスト仕様上の依存関係で待機が必要なケースを処理している。
