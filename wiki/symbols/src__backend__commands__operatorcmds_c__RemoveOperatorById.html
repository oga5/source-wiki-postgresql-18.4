RemoveOperatorById

定義

  • 種別: function
  • 定義箇所: src/backend/commands/operatorcmds.c:445

目的

指定されたOIDを持つ演算子をシステムカタログ pg_operator から削除し、関連する依存関係（commutator/negator）を更新する。

シグネチャまたは形状

void RemoveOperatorById(Oid operOid)

入力と出力

  • **入力**: operOid (削除対象の演算子のOID)
  • **出力**: なし（副作用としてカタログを更新）

前提条件と不変条件

  • pg_operator テーブルに対する RowExclusiveLock が取得可能であること。
  • 指定された operOid がシステムキャッシュ OPEROID に存在すること。

アルゴリズムまたは振る舞い

1. pg_operator テーブルを RowExclusiveLock で開く。 2. SearchSysCache1 を使用して対象演算子のタプルを取得する。存在しない場合はエラーを投げる。 3. 演算子に commutator (oprcom) または negator (oprnegate) が設定されている場合、OperatorUpd を呼び出してそれらのリンクをリセットする。

  • 削除対象の演算子が自分自身を commutator または negator として参照している場合、キャッシュを再取得して整合性を保つ。

    • 4. CatalogTupleDelete を用いてカタログからタプルを物理削除する。 5. キャッシュを解放し、テーブルを閉じる。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/catalog/dependency.c:1408 (直接の証拠)
  • **呼び出し先**:
  • table_open, table_close
  • SearchSysCache1, ReleaseSysCache
  • OperatorUpd
  • CatalogTupleDelete

関連データ構造

  • pg_operator: 演算子情報を保持するシステムカタログ。
  • Form_pg_operator: pg_operator タプルの構造体。

失敗モードと境界条件

  • SearchSysCache1 が有効なタプルを返さない場合、elog(ERROR) により処理が中断される。
  • 削除対象が他のデータベースオブジェクトから依存されている場合、本関数が呼び出される前に依存関係チェックが完了している必要がある（本関数自体には依存関係の検証ロジックは含まれない）。

保守担当者向けメモ

  • 本関数はカタログの物理削除を担当する。依存関係の整合性維持は呼び出し側の責務である。
  • 自己参照（self-commutator/self-negator）の更新時にキャッシュを再取得するロジックは、削除対象のタプルが更新によって無効化されることを防ぐためのものである。
