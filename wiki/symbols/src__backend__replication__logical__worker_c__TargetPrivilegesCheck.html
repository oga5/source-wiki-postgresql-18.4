TargetPrivilegesCheck

定義

  • 種別: function
  • 定義箇所: src/backend/replication/logical/worker.c:2355

目的

論理レプリケーションの適用処理において、対象リレーションに対する現在のユーザーの権限および行レベルセキュリティ（RLS）設定を検証する。

シグネチャまたは形状

static void TargetPrivilegesCheck(Relation rel, AclMode mode)

入力と出力

  • **入力**:
  • rel: 検証対象のリレーション（Relation 構造体）。
  • mode: 要求されるアクセス権限（AclMode、例: ACL_INSERT, ACL_UPDATE 等）。
  • **出力**: なし（成功時は正常終了）。
  • **副作用**: 権限不足または RLS が有効な場合に ereport(ERROR, ...) を発行し、トランザクションを中断する。

前提条件と不変条件

  • 呼び出し元は、有効なリレーション記述子を保持している必要がある。
  • 現在のユーザーコンテキスト（GetUserId()）が、検証対象の権限チェックに使用される。

アルゴリズムまたは振る舞い

1. RelationGetRelid を使用してリレーションの OID を取得する。 2. pg_class_aclcheck を呼び出し、現在のユーザーが指定された mode の権限を持っているか確認する。権限がない場合は aclcheck_error でエラーを投げる。 3. check_enable_rls を呼び出し、対象リレーションで RLS が有効か判定する。 4. RLS が有効な場合、論理レプリケーションは RLS ポリシーを尊重するインフラストラクチャを欠いているため、ERRCODE_FEATURE_NOT_SUPPORTED エラーを発生させる。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/replication/logical/worker.c 内の各適用ハンドラ（INSERT, UPDATE, DELETE, TRUNCATE 等の処理フロー）。
  • **呼び出し先**:
  • RelationGetRelid
  • pg_class_aclcheck
  • aclcheck_error
  • get_relkind_objtype
  • get_rel_name
  • check_enable_rls
  • ereport
  • GetUserId
  • GetUserNameFromId
  • RelationGetRelationName

関連データ構造

  • Relation: リレーションのメタデータおよび記述子。
  • AclMode: 権限の種類（utils/acl.h で定義）。

失敗モードと境界条件

  • **権限エラー**: ユーザーが対象テーブルに対して必要な権限（INSERT/UPDATE/DELETE/TRUNCATE/SELECT）を持っていない場合、aclcheck_error によりエラーとなる。
  • **RLS 制約**: RLS が有効なテーブルに対してレプリケーションを行おうとした場合、機能非サポートとしてエラーとなる。これは TRUNCATE コマンドに対しても同様に適用される。

保守担当者向けメモ

  • 本関数は論理レプリケーションのセキュリティ境界を定義している。RLS 対応を追加する場合は、単にこのチェックを外すだけでなく、レプリケーションワーカーが RLS ポリシーを適切に評価・適用できる仕組みが必要となる。
  • 現在の設計では、レプリケーションの整合性を保つため、RLS が有効なテーブルへのレプリケーションを一律で禁止している。
