ApplyExtensionUpdates

定義

  • 種別: function
  • 定義箇所: src/backend/commands/extension.c:3554

目的

拡張機能のバージョン更新パスを順次適用し、メタデータの更新、依存関係の再構築、および更新スクリプトの実行を行う。

シグネチャまたは形状

static void ApplyExtensionUpdates(Oid extensionOid,
                                  ExtensionControlFile *pcontrol,
                                  const char *initialVersion,
                                  List *updateVersions,
                                  char *origSchemaName,
                                  bool cascade,
                                  bool is_create)

入力と出力

  • **入力**:
  • extensionOid: 更新対象の拡張機能のOID。
  • pcontrol: 拡張機能の制御ファイル情報。
  • initialVersion: 更新開始時のバージョン名。
  • updateVersions: 適用すべきバージョン遷移のリスト。
  • origSchemaName: インストール時のスキーマ名。
  • cascade: 依存関係の自動解決フラグ。
  • is_create: 作成プロセス中かどうかのフラグ。
  • **出力**: なし（副作用としてシステムカタログ pg_extension および依存関係の更新を行う）。

前提条件と不変条件

  • pg_extension カタログに対する RowExclusiveLock が保持されていること。
  • updateVersions リスト内の各バージョンに対応する補助制御ファイルが存在すること。

アルゴリズムまたは振る舞い

1. updateVersions リストを反復処理し、各バージョンに対して以下を実行する。 2. read_extension_aux_control_file を呼び出し、対象バージョンの制御情報を取得する。 3. pg_extension カタログの該当タプルを更新し、extversion と extrelocatable を新しい値に書き換える。 4. 現在のバージョンに必要な依存関係（requires）を特定し、get_required_extension を用いて不足している拡張機能をインストールする。 5. deleteDependencyRecordsForClass および recordDependencyOn を使用して、旧バージョンの依存関係を削除し、新バージョンの依存関係を再構築する。 6. execute_extension_script を呼び出し、実際の更新用SQLスクリプトを実行する。 7. oldVersionName を更新し、次のバージョンへ進む。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/commands/extension.c 内の拡張機能管理ロジック（AlterExtension 等）から呼び出される。
  • **呼び出し先**: read_extension_aux_control_file, table_open, systable_beginscan, heap_modify_tuple, CatalogTupleUpdate, get_required_extension, deleteDependencyRecordsForClass, recordDependencyOn, InvokeObjectPostAlterHook, execute_extension_script 等。

関連データ構造

  • ExtensionControlFile: 拡張機能のメタデータを保持する構造体。
  • pg_extension: 拡張機能情報を格納するシステムカタログ。

失敗モードと境界条件

  • pg_extension タプルが見つからない場合、elog(ERROR) により処理を中断する。
  • 依存関係の解決に失敗した場合、get_required_extension 経由でエラーが発生する。
  • スクリプト実行中にエラーが発生した場合、トランザクションのロールバックにより変更が破棄される。

保守担当者向けメモ

  • 本関数は更新パス上の各ステップを逐次実行する。execute_extension_script 内で CommandCounterIncrement が行われるため、ループ内でのカタログ操作は一貫性が保たれる。
  • 依存関係の再構築は DEPENDENCY_NORMAL 型で行われる。
