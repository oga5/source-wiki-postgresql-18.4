getLOs

定義

  • 種別: function
  • 定義箇所: src/bin/pg_dump/pg_dump.c:3854

目的

データベース内のラージオブジェクト（LO）のメタデータおよびデータを収集し、ダンプ対象オブジェクト（DumpableObject）として登録する。

シグネチャまたは形状

static void getLOs(Archive *fout)

入力と出力

  • **入力**: Archive *fout（ダンプ処理のコンテキストを保持するアーカイブ構造体）。
  • **出力**: なし（副作用として fout 内のダンプ対象リストに DO_LARGE_OBJECT および DO_LARGE_OBJECT_DATA 型のオブジェクトを追加する）。

前提条件と不変条件

  • fout が有効なデータベース接続を保持していること。
  • pg_largeobject_metadata システムカタログへのアクセス権限があること。
  • MAX_BLOBS_PER_ARCHIVE_ENTRY に基づき、所有者とACLが同一のラージオブジェクトがグループ化される。

アルゴリズムまたは振る舞い

1. pg_largeobject_metadata から OID、所有者、ACL 情報を取得するクエリを実行する。 2. 取得した結果を所有者と ACL でソートし、グループ化する。 3. 各グループに対して以下の処理を行う：

  • LoInfo 構造体を割り当て、メタデータ用 DumpableObject を作成する。
  • グループ内の全 OID を LoInfo に記録し、recordAdditionalCatalogID でカタログ ID を登録する。
  • DUMP_COMPONENT_DATA を設定する（binary_upgrade モード時は除外）。
  • 依存関係解決のため、DO_LARGE_OBJECT_DATA 型のプレースホルダーオブジェクトを作成し、メタデータオブジェクトへの依存関係を定義する。

    • 4. 使用したリソース（PGresult 等）を解放する。

呼び出し元と呼び出し先

  • **呼び出し元**: src/bin/pg_dump/pg_dump.c 内のメイン処理フローから呼び出される（ソースコード上の直接的な呼び出し元は main 相当のフロー内）。
  • **呼び出し先**: ExecuteSqlQuery, PQfnumber, PQntuples, AssignDumpId, pg_malloc, getRoleName, recordAdditionalCatalogID 等のユーティリティ関数。

関連データ構造

  • Archive: ダンプ全体の管理構造体。
  • DumpableObject: ダンプ対象の抽象化。
  • LoInfo: ラージオブジェクト固有の情報を保持する構造体。

失敗モードと境界条件

  • データベース接続エラーやクエリ実行失敗時は ExecuteSqlQuery 経由でエラー終了する可能性がある。
  • binary_upgrade モードの場合、ラージオブジェクトのデータ自体はダンプ対象から除外される。
  • メモリ不足時は pg_malloc が失敗する。

保守担当者向けメモ

  • ラージオブジェクトのグループ化ロジックは、MAX_BLOBS_PER_ARCHIVE_ENTRY の制限に依存している。
  • binary_upgrade モード時の挙動（データはコピーせずメタデータのみを扱う）は、pg_upgrade の仕様と密接に関連しているため変更時は注意が必要。
