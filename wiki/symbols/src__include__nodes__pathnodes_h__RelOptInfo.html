RelOptInfo

定義

  • 種別: struct
  • 定義箇所: src/include/nodes/pathnodes.h:883

目的

クエリプランナが最適化の過程で、ベースリレーションや結合リレーションに関する統計情報、コスト見積もり、および生成されたパス（Path）の候補を保持するために使用する構造体です。

シグネチャまたは形状

typedef struct RelOptInfo {
    NodeTag     type;
    RelOptKind  reloptkind;
    Relids      relids;
    Cardinality rows;
    /* ... 多数のプランニング用フィールド ... */
} RelOptInfo;

入力と出力

本構造体は関数ではなくデータ構造であるため、直接的な入力・出力はありません。プランナの最適化フェーズにおいて、PlannerInfo 内の simple_rel_array 等に格納され、パス生成やコスト計算の入力として参照されます。

前提条件と不変条件

  • RelOptInfo と Path の間には複雑なリンクが存在するため、copyObject() 等の汎用関数によるコピーはサポートされていません。
  • relids は Bitmapset であり、リレーションの集合を表現します。
  • ベースリレーションの場合、relid や indexlist 等の固有情報が設定されますが、結合リレーションではこれらは設定されません。

アルゴリズムまたは振る舞い

  • プランナは、クエリ内の各リレーションに対して RelOptInfo を構築します。
  • pathlist にはスキャンや結合のアルゴリズムを表す Path 構造体が格納されます。
  • 結合探索（動的計画法やGEQO）の過程で、結合リレーション用の RelOptInfo が動的に構築されます。
  • 最終的に、cheapest_total_path 等のフィールドから最適なパスが選択されます。

呼び出し元と呼び出し先

  • 呼び出し元: プランナの最適化ロジック（src/backend/optimizer/ 配下の各モジュール）。
  • 呼び出し先: contrib/file_fdw/file_fdw.c 等の外部データラッパー（FDW）が、fdw_private フィールドを通じて独自の最適化情報を保持するために参照します。
  • 補足: 特定の関数呼び出し関係については、提供されたソース excerpt には記載がありません。

関連データ構造

  • Path: 実行計画の候補。
  • PlannerInfo: クエリ全体の最適化状態を保持するルート構造体。
  • Relids: リレーションの集合を表す Bitmapset。
  • RestrictInfo: 制限句のメタデータ。
  • FdwRoutine: 外部テーブル用のルーチン定義。

失敗モードと境界条件

  • nparts が -1 の場合、パーティション数が未設定であることを示します。
  • fdw_private や fdwroutine は、外部テーブルでない場合は使用されません。
  • パーティション分割されたリレーションにおいて、part_rels 配列は境界情報と対応しますが、非常に肥大化する可能性があります。

保守担当者向けメモ

  • 本構造体はプランナの再帰的な処理で頻繁に参照されます。
  • pg_node_attr マクロが付与されたフィールド（attr_needed, fdw_private 等）は、ノードサポートコード生成器によってシリアライズ対象外などの制御が行われています。
  • 構造体の変更時は、プランナのコスト見積もりロジックへの影響を慎重に評価してください。
