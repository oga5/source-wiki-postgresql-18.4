amcheck_index_mainfork_expected

定義

  • 種別: function
  • 定義箇所: contrib/amcheck/verify_common.c:35

目的

リカバリ実行中（ホットスタンバイモード）において、ログ出力されない（unlogged）インデックスの検証をスキップすべきかどうかを判定する。

シグネチャまたは形状

static bool amcheck_index_mainfork_expected(Relation rel)

入力と出力

  • **入力**: Relation rel（検証対象のインデックスリレーション）
  • **出力**: bool
  • true: 検証を続行可能。
  • false: リカバリ中の unlogged インデックスであるため、検証をスキップする。

前提条件と不変条件

  • 入力される rel は有効なリレーションポインタであること。
  • 本関数は、検証対象がリカバリ中に安全にアクセス可能かを判断するガードレールとして機能する。

アルゴリズムまたは振る舞い

1. リレーションの永続性（relpersistence）を確認する。 2. リレーションが RELPERSISTENCE_UNLOGGED ではない、または現在リカバリ中ではない場合は true を返す。 3. リカバリ中かつ unlogged インデックスである場合、NOTICE レベルのメッセージをログに出力し、検証をスキップするために false を返す。

呼び出し元と呼び出し先

  • **呼び出し元**: contrib/amcheck/verify_common.c 内の検証プロセス（行190付近で呼び出されている）。
  • **呼び出し先**:
  • RecoveryInProgress()（PostgreSQL 内部関数）
  • RelationGetRelationName()（PostgreSQL 内部関数）
  • ereport()（PostgreSQL 内部関数）

関連データ構造

  • Relation: インデックスのメタデータおよび永続性フラグを保持する構造体。

失敗モードと境界条件

  • リカバリ中に unlogged インデックスを検証しようとすると、エラーではなく NOTICE を発行して安全にスキップする。

保守担当者向けメモ

  • 本関数は、ホットスタンバイ環境における不整合を防ぐための重要なガードレールである。
  • RELPERSISTENCE_UNLOGGED 以外の永続性を持つインデックスについては、リカバリ中であっても本関数は検証を許可する。
