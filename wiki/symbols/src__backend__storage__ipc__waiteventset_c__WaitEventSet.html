WaitEventSet

定義

  • 種別: struct
  • 定義箇所: src/backend/storage/ipc/waiteventset.c:117

目的

複数のイベント（ソケット、ラッチ、ポストマスターの終了など）を単一のインターフェースで効率的に待機するための抽象化レイヤーを提供します。OS 固有のイベント通知プリミティブ（epoll, kqueue, poll, WaitForMultipleObjects）をラップし、プラットフォーム間の差異を吸収します。

シグネチャまたは形状

struct WaitEventSet {
    ResourceOwner owner;
    int           nevents;
    int           nevents_space;
    WaitEvent    *events;
    Latch        *latch;
    int           latch_pos;
    bool          exit_on_postmaster_death;
    /* プラットフォーム依存のメンバ (epoll_fd, kqueue_fd, pollfds, handles 等) */
};

入力と出力

  • **入力**: CreateWaitEventSet により生成され、AddWaitEventToSet を通じて監視対象のイベントが登録されます。
  • **出力**: WaitEventSetWait を呼び出すことで、発生したイベント情報が WaitEvent 構造体として返されます。

前提条件と不変条件

  • nevents は常に nevents_space 以下であること。
  • WL_LATCH_SET が指定された場合、latch ポインタと latch_pos が正しく設定されていること。
  • コンパイル時に選択されたプラットフォーム固有のメンバ（epoll_fd 等）が、対応する OS プリミティブと整合していること。

アルゴリズムまたは振る舞い

1. **初期化**: CreateWaitEventSet でリソースを確保し、OS 固有の待機メカニズムを準備します。 2. **登録**: AddWaitEventToSet で監視対象を追加し、OS 固有のシステムコール（epoll_ctl 等）でカーネルに登録します。 3. **待機**: WaitEventSetWait が呼び出されると、内部の待機ブロックが OS プリミティブを呼び出し、イベント発生までプロセスを停止します。 4. **復帰**: イベント発生後、発生したイベントを解析して呼び出し元へ通知します。

呼び出し元と呼び出し先

  • **呼び出し元**: postmaster.c, syslogger.c, walreceiver.c, postgres_fdw.c 等の各バックエンドプロセス。
  • **呼び出し先**: OS 固有のシステムコール（epoll_wait, kevent, poll, WaitForMultipleObjects 等）および latch.c のラッチ管理関数。

関連データ構造

  • WaitEvent: 待機対象のイベント定義を保持する構造体。
  • Latch: プロセス起床のための同期プリミティブ。
  • ResourceOwner: メモリおよびファイル記述子のライフサイクル管理に使用。

失敗モードと境界条件

  • nevents が nevents_space を超える追加は不可。
  • プラットフォーム固有の制限（epoll の FD 上限など）に依存します。
  • WL_EXIT_ON_PM_DEATH が設定されている場合、ポストマスターの死滅を検知すると即座に終了します。

保守担当者向けメモ

  • コンパイル時のフラグ（WAIT_USE_EPOLL 等）により実装が大きく異なります。デバッグ時は対象環境のプリミティブを確認してください。
  • Unix 系環境では、シグナルハンドラとの競合を避けるためにセルフパイプや signalfd が重要な役割を果たします。
  • Windows 版は WaitForMultipleObjects を使用するため、実装が他プラットフォームと大きく異なります。
