ReorderBufferChange

定義

  • **種別**: struct
  • **定義箇所**: src/include/replication/reorderbuffer.h:76

目的

論理デコーディングにおいて、WALから読み取られた個別の変更内容（挿入、更新、削除、メッセージ、内部状態変更など）を保持し、トランザクション単位で管理・処理するためのデータ構造です。

シグネチャまたは形状

typedef struct ReorderBufferChange {
    XLogRecPtr lsn;
    ReorderBufferChangeType action;
    struct ReorderBufferTXN *txn;
    RepOriginId origin_id;
    union {
        struct { ... } tp;          /* INSERT/UPDATE/DELETE */
        struct { ... } truncate;    /* TRUNCATE */
        struct { ... } msg;         /* Message */
        Snapshot snapshot;          /* INTERNAL_SNAPSHOT */
        CommandId command_id;       /* INTERNAL_COMMAND_ID */
        struct { ... } tuplecid;    /* INTERNAL_TUPLECID */
        struct { ... } inval;       /* Invalidation */
    } data;
    dlist_node node;
} ReorderBufferChange;

入力と出力

  • **入力**: WALレコードから抽出された情報（LSN、変更タイプ、関連するタプルデータ、メッセージ内容、無効化メッセージ等）。
  • **出力**: ReorderBufferTXN 内のリストへの格納、または出力プラグインのコールバック関数への引数として渡されます。

前提条件と不変条件

  • data 共用体の有効なメンバは action フィールドの値に依存します。
  • node フィールドにより、トランザクション内の変更リスト（dlist）にリンクされます。
  • 処理対象のトランザクション（txn）が正しく設定されている必要があります。

アルゴリズムまたは振る舞い

1. **保持**: WALから読み取られた変更が ReorderBufferChange としてインスタンス化されます。 2. **分類**: action に基づき、data 共用体の適切な構造体にデータが格納されます。 3. **管理**: dlist_node を介して、所属する ReorderBufferTXN の変更リストに連結されます。 4. **消費**: トランザクションのコミットや準備のタイミングで、出力プラグインのコールバックへ渡され、最終的に解放されます。

呼び出し元と呼び出し先

  • **呼び出し元（直接的な言及）**:
  • src/backend/replication/logical/decode.c: WALデコード処理中にインスタンス化されます。
  • **呼び出し先（利用側）**:
  • contrib/test_decoding/test_decoding.c: 出力プラグインのテスト実装において、変更内容の解析に使用されています。

関連データ構造

  • ReorderBufferTXN: この変更が所属するトランザクション。
  • ReorderBufferChangeType: action の種類を定義する列挙型。
  • dlist_node: トランザクション内の変更リストを管理するための二重連結リストノード。
  • HeapTuple, Snapshot, RelFileLocator: 変更内容の詳細を保持する各データ構造。

失敗モードと境界条件

  • メモリ制限（logical_decoding_work_mem）を超過した場合、この構造体を含むトランザクション全体がディスクへシリアライズ（スプール）される可能性があります。
  • action と data 共用体の不整合は、デコード処理時のデータ破損やクラッシュを招く可能性があります。

保守担当者向けメモ

  • 本構造体は共用体を使用しているため、新しい action タイプを追加する際は data 共用体の定義と、それを利用するすべてのコールバック実装を更新する必要があります。
  • dlist_node を使用しているため、リスト操作時のメモリ管理（確保と解放）には注意が必要です。
