relation_excluded_by_constraints

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/util/plancat.c:1609

目的

リレーションの制約（CHECK制約やNOT NULL制約）とクエリの制限句（RestrictInfo）を比較し、制約条件と矛盾してスキャンが不要であると論理的に証明できる場合に true を返す。

シグネチャまたは形状

bool relation_excluded_by_constraints(PlannerInfo *root, RelOptInfo *rel, RangeTblEntry *rte)

入力と出力

  • **入力**:
  • root: プランナのグローバル情報。
  • rel: 対象リレーションの最適化用情報。
  • rte: 対象リレーションの範囲テーブルエントリ。
  • **出力**:
  • bool: スキャンを排除できる場合は true、それ以外は false。

前提条件と不変条件

  • IS_SIMPLE_REL(rel) が真であること（単純なリレーションのみを対象とする）。
  • rel->baserestrictinfo が空の場合、排除は不可能とみなされ直ちに false を返す。
  • 評価には不変（immutable）な関数のみを使用する。

アルゴリズムまたは振る舞い

1. **定数評価**: baserestrictinfo 内に FALSE または NULL と評価される定数句が含まれているか確認し、あれば即座に true を返す。 2. **モード判定**: constraint_exclusion GUC 設定に基づき、処理を継続するか判断する（OFF なら false、PARTITION なら対象がパーティションメンバーか確認、ON なら常に試行）。 3. **自己矛盾チェック**: baserestrictinfo 内の不変な句同士を比較し、自己矛盾があれば true を返す。 4. **制約取得**: rte が RTE_RELATION の場合、get_relation_constraints を呼び出し、継承やパーティション設定を考慮した制約式リストを取得する。 5. **制約排除判定**: 取得した制約式のうち不変なもののみを抽出し、predicate_refuted_by を用いて baserestrictinfo との矛盾を検証する。矛盾が証明できれば true を返す。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/optimizer/path/allpaths.c (クエリ最適化のパス生成プロセス)
  • **呼び出し先**:
  • get_relation_constraints
  • contain_mutable_functions
  • predicate_refuted_by

関連データ構造

  • PlannerInfo: プランナの全体状態。
  • RelOptInfo: リレーションの最適化用メタデータ。
  • RangeTblEntry: クエリ内のテーブル参照情報。
  • RestrictInfo: 制限句のラッパー構造体。

失敗モードと境界条件

  • **非不変関数**: 制約や制限句に不変でない関数が含まれる場合、誤った排除を防ぐためそれらの句は無視される。
  • **継承**: rte->inh が設定されている場合、継承された制約の扱いに制限がある。
  • **RTE種別**: RTE_RELATION 以外のRTEに対しては、制約排除の試行を中断する。

保守担当者向けメモ

  • attnotnull 制約の継承状態の扱いは将来的に改善の余地がある（ソース内の XXX コメント参照）。
  • predicate_refuted_by は、制約排除のために「強い反証（strong refutation）」を必要とする。
