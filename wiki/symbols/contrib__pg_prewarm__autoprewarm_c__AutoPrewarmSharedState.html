AutoPrewarmSharedState

定義

  • 種別: struct
  • 定義箇所: contrib/pg_prewarm/autoprewarm.c:65

目的

PostgreSQL の自動プリウォーム機能において、バックグラウンドワーカー間で共有バッファのダンプ情報やプリウォーム処理の進捗を同期・管理するための共有メモリ制御構造体です。

シグネチャまたは形状

typedef struct AutoPrewarmSharedState
{
	LWLock		lock;			/* 相互排他制御用ロック */
	pid_t		bgworker_pid;	/* メインバックグラウンドワーカーのPID */
	pid_t		pid_using_dumpfile; /* ダンプファイル使用中のプロセスPID */

	/* データベースごとのワーカーとの通信用 */
	dsm_handle	block_info_handle; /* 動的共有メモリのハンドル */
	Oid			database;          /* 対象データベースのOID */
	int			prewarm_start_idx; /* プリウォーム開始インデックス */
	int			prewarm_stop_idx;  /* プリウォーム終了インデックス */
	int			prewarmed_blocks;  /* プリウォーム済みブロック数 */
} AutoPrewarmSharedState;

入力と出力

  • **入力**: 共有メモリ上の状態更新（lock による保護下での各フィールドへの書き込み）。
  • **出力**: データベースごとのワーカーに対するプリウォーム範囲（prewarm_start_idx, prewarm_stop_idx）の指示および進捗状況の共有。

前提条件と不変条件

  • 本構造体は共有メモリ上に配置される必要があります。
  • lock フィールドを使用して、構造体内のデータへのアクセスを排他制御する必要があります。
  • データベースごとのワーカーは、block_info_handle を介して BlockInfoRecord の配列にアクセスします。

アルゴリズムまたは振る舞い

  • リーダーワーカーが apw_init_shmem 等を通じて本構造体を初期化し、共有メモリを確保します。
  • リーダーワーカーは apw_dump_now を呼び出し、現在のバッファ状態をファイルへ書き出す際に pid_using_dumpfile を管理します。
  • 各データベースごとのワーカーは、本構造体から割り当てられた範囲（prewarm_start_idx から prewarm_stop_idx）を読み込み、read_stream API を用いて共有バッファへデータをロードします。

呼び出し元と呼び出し先

  • **呼び出し元**: autoprewarm.c 内の apw_init_state（初期化時）、apw_dump_now（ダンプ時）、および各ワーカーのメインループ（autoprewarm_database_main 等）。
  • **呼び出し先**: 本構造体自体はデータ保持用であり、直接的な関数呼び出しは行いませんが、LWLock を介して storage/lwlock.h の機能を利用します。

関連データ構造

  • BlockInfoRecord: プリウォーム対象となる各ブロックのメタデータ。block_info_handle を通じて参照されます。

失敗モードと境界条件

  • 共有メモリの確保に失敗した場合、自動プリウォーム機能は起動できません。
  • lock の取得に失敗またはデッドロックが発生した場合、ワーカーの処理が停止する可能性があります。
  • prewarm_start_idx や prewarm_stop_idx が不正な範囲を指定した場合、プリウォーム処理が正しく行われない可能性があります。

保守担当者向けメモ

  • 本構造体は autoprewarm.c 内の静的変数 apw_state を通じて参照されます。
  • 共有メモリ上の構造であるため、フィールドの追加や変更を行う際は、共有メモリレイアウトの互換性に注意してください。
