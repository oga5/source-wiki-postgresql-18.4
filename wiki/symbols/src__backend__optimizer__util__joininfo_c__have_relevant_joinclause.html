have_relevant_joinclause

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/util/joininfo.c:38

目的

二つのリレーション（rel1, rel2）間に、結合条件として利用可能な制約が存在するかを判定します。

シグネチャまたは形状

bool have_relevant_joinclause(PlannerInfo *root, RelOptInfo *rel1, RelOptInfo *rel2)

入力と出力

  • **入力**:
  • root: プランナのグローバル情報。
  • rel1, rel2: 結合対象となるリレーション情報（RelOptInfo）。
  • **出力**:
  • bool: 両リレーション間に結合条件が存在する場合は true、存在しない場合は false を返します。

前提条件と不変条件

  • rel1 および rel2 は有効な RelOptInfo 構造体である必要があります。
  • joininfo リストには RestrictInfo が格納されていることが期待されます。

アルゴリズムまたは振る舞い

1. **効率的な走査**: rel1 と rel2 のうち、joininfo リストの長さが短い方を対象として選択します。 2. **リスト走査**: 選択したリスト内の各 RestrictInfo について、その required_relids が他方のリレーションの relids と重複（bms_overlap）するかを確認します。 3. **EquivalenceClass の確認**: リスト内に条件が見つからなかった場合、かつ両リレーションが has_eclass_joins フラグを持っている場合に限り、have_relevant_eclass_joinclause を呼び出して EquivalenceClass 経由の結合可能性を判定します。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/optimizer/geqo/geqo_eval.c
  • src/backend/optimizer/path/equivclass.c
  • src/backend/optimizer/path/joinrels.c
  • **呼び出し先**:
  • have_relevant_eclass_joinclause (条件付き)
  • bms_overlap (ビットマップセットの比較)
  • list_length (リスト長取得)

関連データ構造

  • RelOptInfo: 結合対象のリレーション情報。
  • RestrictInfo: 結合条件を保持する構造体。
  • EquivalenceClass: 等価クラスによる結合条件の管理構造。

失敗モードと境界条件

  • リストが空の場合、EquivalenceClass の判定に依存します。
  • has_eclass_joins が false の場合、EquivalenceClass のチェックはスキップされます。

保守担当者向けメモ

  • joininfo リストだけでなく EquivalenceClass も考慮する必要があるため、判定ロジックの変更時には両方のデータ構造への影響を考慮してください。
  • bms_overlap による判定は高速ですが、RestrictInfo が保持する required_relids の正確性に依存します。
