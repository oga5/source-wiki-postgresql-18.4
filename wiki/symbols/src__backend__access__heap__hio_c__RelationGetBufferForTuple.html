RelationGetBufferForTuple

定義

  • 種別: function
  • 定義箇所: src/backend/access/heap/hio.c:501

目的

ヒープリレーションへのタプル挿入に適したページ（バッファ）を確保する。必要に応じて空き領域マップ（FSM）を用いた探索や、リレーションの拡張を行う。

シグネチャまたは形状

Buffer RelationGetBufferForTuple(Relation relation, Size len,
                                 Buffer otherBuffer, int options,
                                 BulkInsertState bistate,
                                 Buffer *vmbuffer, Buffer *vmbuffer_other,
                                 int num_pages)

入力と出力

  • **入力**:
  • relation: 対象のリレーション。
  • len: 挿入するタプルのサイズ。
  • otherBuffer: 更新操作などで既にロックされている関連バッファ（ない場合は InvalidBuffer）。
  • options: 挿入オプション（HEAP_INSERT_SKIP_FSM 等）。
  • bistate: 大量挿入時の状態管理構造体。
  • vmbuffer, vmbuffer_other: 可視性マップのピンを保持するためのバッファポインタ。
  • num_pages: 拡張が必要な場合の拡張ページ数。
  • **出力**: 挿入先として確保され、排他ロックされた Buffer。

前提条件と不変条件

  • len が MaxHeapTupleSize を超える場合はエラーとなる。
  • otherBuffer が指定されている場合、bistate は使用できない。
  • 複数のバッファをロックする際は、デッドロックを回避するため常にページ番号の昇順でロックを取得する。

アルゴリズムまたは振る舞い

1. **初期化**: 挿入に必要な空き領域サイズを計算する。 2. **ターゲット探索**: BulkInsertState またはリレーションのキャッシュからターゲットページを特定する。見つからない場合は FSM を使用して探索する。 3. **ロックとピン**:

  • ターゲットページを読み込み、必要に応じて可視性マップ（Visibility Map）をピン留めする。
  • otherBuffer がある場合は、ページ番号の順序に従って両方のバッファを排他ロックする。

    • 4. **再検証**: ロック取得前に確認した可視性マップの状態が変化している可能性があるため、必要に応じて再試行ロジックへ移行する。 5. **拡張**: 適切なページが見つからない場合はリレーションを拡張する。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/access/heap/heapam.c 内の heap_insert 等の挿入処理。
  • **呼び出し先**: ReadBufferBI, ReadBuffer, LockBuffer, GetVisibilityMapPins, visibilitymap_pin, RelationGetTargetBlock, GetPageWithFreeSpace, RelationGetNumberOfBlocks 等。

関連データ構造

  • Relation: リレーション記述子。
  • Buffer: バッファマネージャのバッファ識別子。
  • BulkInsertState: 大量挿入時のバッファキャッシュ状態。

失敗モードと境界条件

  • **サイズ超過**: タプルサイズが MaxHeapTupleSize を超える場合、ereport(ERROR) を発生させる。
  • **FSM 不整合**: FSM の情報が古い場合、ループ内で再試行を行う。
  • **デッドロック**: otherBuffer とターゲットバッファのロック順序が不適切な場合、デッドロックの可能性があるため、ページ番号による順序制御が必須。

保守担当者向けメモ

  • 本関数はバッファロックを保持した状態で終了するため、呼び出し側で適切にロックを解放する必要がある。
  • 可視性マップのピン管理は複雑であり、ロック取得後の再チェックロジックを変更する際は注意が必要。
