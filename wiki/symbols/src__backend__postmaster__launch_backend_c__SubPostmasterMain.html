SubPostmasterMain

定義

  • 種別: function
  • 定義箇所: src/backend/postmaster/launch_backend.c:597

目的

EXEC_BACKEND モード（Windows等）において、親プロセスから起動された子プロセスが、共有メモリの再アタッチや環境変数の復元を行い、本来のプロセスメイン関数へ移行するためのエントリポイントを提供します。

シグネチャまたは形状

void SubPostmasterMain(int argc, char *argv[])

入力と出力

  • **入力**:
  • argc, argv: コマンドライン引数。argv[1] に --forkchild=<kind>、argv[2] に状態ファイルパスが指定される必要があります。
  • **出力**:
  • なし（成功時は各プロセスのメイン関数へ制御が移り、戻ることはありません）。

前提条件と不変条件

  • IsPostmasterEnvironment が true に設定されます。
  • whereToSendOutput が DestNone に初期化されます。
  • argc は 3 である必要があります。
  • argv[1] は --forkchild= で始まる必要があります。
  • 共有メモリへのアタッチを行う場合、親プロセスと同じアドレス空間を確保するため、OS側のメモリ配置のランダム化（ASLR等）が無効化されていることが推奨されます。

アルゴリズムまたは振る舞い

1. **初期化**: InitializeGUCOptions() を呼び出し、エラーログ出力等の基盤を整えます。 2. **プロセス特定**: argv[1] を解析し、child_process_kinds テーブルから対応するプロセスタイプを特定します。 3. **状態復元**: read_backend_variables() を使用して、親プロセスから引き継がれた状態データ（startup_data）を読み込みます。 4. **リソース管理**: ClosePostmasterPorts() で親プロセスのソケットを閉じ、InitPostmasterChild() で子プロセスとしての初期化を行います。 5. **共有メモリ再接続**: プロセスタイプに応じて PGSharedMemoryReAttach() または PGSharedMemoryNoReAttach() を実行します。 6. **環境再構築**: GUC変数の読み込み、データディレクトリの検証、制御ファイルの再読み込み、共有ライブラリのプリロードを行います。 7. **メイン関数実行**: 特定されたプロセスタイプに対応する main_fn を呼び出し、プロセスを移行します。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/main/main.c のメインエントリポイントから直接呼び出されます。
  • **呼び出し先**:
  • read_backend_variables, ClosePostmasterPorts, InitPostmasterChild, PGSharedMemoryReAttach, PGSharedMemoryNoReAttach, read_nondefault_variables, checkDataDir, LocalProcessControlFile, process_shared_preload_libraries, InitShmemAccess 等の内部関数。
  • child_process_kinds[child_type].main_fn（各プロセスのメイン関数）。

関連データ構造

  • child_process_kinds: プロセスタイプごとのメイン関数や共有メモリ要否を保持するテーブル。
  • BackendStartupData: プロセス間で受け渡される接続タイミング等の状態情報。
  • BackendType: プロセスの種類を識別する列挙型。

失敗モードと境界条件

  • argc != 3 または --forkchild 引数が不正な場合: FATAL エラーで終了します。
  • 未知の child_kind が指定された場合: ERROR で終了します。
  • 共有メモリの再アタッチに失敗した場合: 実行環境のメモリ配置の不一致が原因である可能性が高いです。

保守担当者向けメモ

  • 本関数は pg_unreachable() で終了するため、呼び出し元へ戻ることはありません。
  • EXEC_BACKEND モード特有の処理であり、Unix系環境での fork() によるメモリ継承とは異なるシリアライズ/デシリアライズのロジックを含みます。
  • 共有メモリへのアタッチ順序は厳密であり、他のサブシステムが共有メモリに依存する前に完了させる必要があります。
