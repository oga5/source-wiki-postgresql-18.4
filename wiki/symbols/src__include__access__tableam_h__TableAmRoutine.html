TableAmRoutine

定義

  • **種別**: struct
  • **定義箇所**: src/include/access/tableam.h:288

目的

PostgreSQLのテーブルアクセス手法（Table AM）を抽象化するための関数ポインタ群を保持する構造体です。ストレージエンジンに依存しない汎用的なテーブル操作APIを提供し、heap 以外の物理ストレージ実装の追加を可能にします。

シグネチャまたは形状

typedef struct TableAmRoutine {
    NodeTag type;
    const TupleTableSlotOps *(*slot_callbacks) (Relation rel);
    TableScanDesc (*scan_begin) (...);
    void (*scan_end) (TableScanDesc scan);
    void (*scan_rescan) (...);
    bool (*scan_getnextslot) (...);
    void (*scan_set_tidrange) (...);
    bool (*scan_getnextslot_tidrange) (...);
    Size (*parallelscan_estimate) (Relation rel);
    Size (*parallelscan_initialize) (...);
    void (*parallelscan_reinitialize) (...);
    struct IndexFetchTableData *(*index_fetch_begin) (Relation rel);
    void (*index_fetch_reset) (...);
    void (*index_fetch_end) (...);
    bool (*index_fetch_tuple) (...);
    /* ...以降、タプル操作等のコールバックが続く */
} TableAmRoutine;

入力と出力

  • **入力**: Relation 構造体や TableScanDesc などのコンテキスト情報。
  • **出力**: 各操作の結果（bool による成功判定、TupleTableSlot へのタプル格納、Size によるメモリ見積もりなど）。

前提条件と不変条件

  • type フィールドには必ず T_TableAmRoutine が設定されている必要があります。
  • scan_set_tidrange を実装する場合、scan_getnextslot_tidrange も併せて実装する必要があります。
  • 各操作は、呼び出し側が適切なリレーションロックを保持していることを前提とします。

アルゴリズムまたは振る舞い

  • **ディスパッチ**: table_* で始まるラッパー関数が、Relation->rd_tableam に格納された本構造体の関数ポインタを呼び出すことで、具体的なストレージ実装（heapam.c 等）へ処理を委譲します。
  • **スキャン**: scan_begin で開始し、scan_getnextslot でタプルを順次取得します。TID範囲指定スキャンや並列スキャンにも対応しています。
  • **インデックススキャン**: index_fetch_tuple を通じて、インデックスから得られた TID に基づくタプルのフェッチと可視性判定を行います。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/access/table/tableam.c（rel->rd_tableam を経由）、src/backend/utils/cache/relcache.c（初期化時）。
  • **呼び出し先**: 各AMの実装（例: heapam_handler.c 内の heapam_methods）。
  • *注: 特定の関数呼び出しの連鎖については、ソースコード全体を通じた動的な解析が必要です。*

関連データ構造

  • Relation: テーブルのメタデータと AM への参照を保持。
  • TableScanDesc: スキャン状態を保持する基底構造体。
  • TupleTableSlot: タプルデータを保持するスロット。
  • IndexFetchTableData: インデックススキャン時のフェッチ状態を保持。

失敗モードと境界条件

  • scan_set_tidrange は、TID がテーブルの範囲外である場合や、無効なオフセットを指定された場合でも適切に処理する必要があります。

保守担当者向けメモ

  • 本構造体は直接呼び出さず、必ず table_ プレフィックスのついたラッパー関数を経由してください。
  • 新しい AM を追加する際は、全フィールドを適切に埋める必要があります。
  • index_fetch_tuple の call_again 引数は、同一 TID に対して複数のタプルが存在する可能性がある場合に制御に使用されます。
