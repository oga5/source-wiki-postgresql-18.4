ReThrowError

定義

  • **種別**: function
  • **定義箇所**: src/backend/utils/error/elog.c:1958

目的

以前にコピーされたエラーデータ（ErrorData）をエラーコンテキストに再投入し、PG_RE_THROW() を呼び出してエラー処理を再開させる。

シグネチャまたは形状

void ReThrowError(ErrorData *edata)

入力と出力

  • **入力**: ErrorData *edata - 再スローするエラー情報を保持する構造体へのポインタ。
  • **出力**: なし（PG_RE_THROW() により制御が戻らないため）。

前提条件と不変条件

  • edata->elevel が ERROR であること（Assert により検証）。
  • 呼び出し元は、エラー状態を保持する ErrorData を適切に管理していること。

アルゴリズムまたは振る舞い

1. recursion_depth をインクリメントし、メモリコンテキストを ErrorContext に切り替える。 2. get_error_stack_entry() を使用して新しい ErrorData 領域を確保する。 3. 入力された edata を新しい領域にコピーする。 4. message, detail, hint 等の動的に割り当てられた各フィールドに対し、pstrdup() を用いて ErrorContext 上へ個別にコピーを作成する。 5. newedata->assoc_context を ErrorContext に設定する。 6. recursion_depth をデクリメントする。 7. PG_RE_THROW() を呼び出し、エラーハンドラへ制御を移す。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/executor/spi.c
  • src/pl/plpgsql/src/pl_exec.c
  • **呼び出し先**:
  • get_error_stack_entry()
  • MemoryContextSwitchTo()
  • pstrdup()
  • PG_RE_THROW()

関連データ構造

  • ErrorData: エラーの詳細情報を保持する構造体。
  • ErrorContext: エラー処理専用のメモリコンテキスト。

失敗モードと境界条件

  • ErrorContext が枯渇している場合、pstrdup() 等のメモリ割り当てで致命的なエラーが発生する可能性がある。
  • edata が指す構造体が不正な場合、memcpy や pstrdup でメモリ破壊が発生する可能性がある。

保守担当者向けメモ

  • 本関数は PG_RE_THROW() を含むため、呼び出し元には戻らない。
  • エラー処理中の再帰的なエラー発生を防ぐため、recursion_depth の管理が重要である。
  • フィールドのコピー漏れはメモリリークや不正なポインタ参照の原因となるため、ErrorData 構造体に新しいフィールドを追加した際は本関数でのコピー処理も更新すること。
