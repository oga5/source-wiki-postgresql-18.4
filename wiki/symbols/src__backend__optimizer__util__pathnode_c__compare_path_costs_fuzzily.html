compare_path_costs_fuzzily

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/util/pathnode.c:184

目的

2つの Path ノード間において、指定された fuzz_factor を用いてコスト（総コストおよび起動コスト）を比較し、どちらが優れているか、あるいは同等かを判定する。

シグネチャまたは形状

PathCostComparison compare_path_costs_fuzzily(Path *path1, Path *path2, double fuzz_factor)

入力と出力

  • **入力**:
  • path1, path2: 比較対象の Path 構造体へのポインタ。
  • fuzz_factor: コスト比較の許容誤差係数。
  • **出力**:
  • PathCostComparison 型（列挙型）: COSTS_BETTER1 (path1が優位), COSTS_BETTER2 (path2が優位), COSTS_DIFFERENT (優劣判定不能), COSTS_EQUAL (同等) のいずれかを返す。

前提条件と不変条件

  • path1 および path2 は有効な Path 構造体を指している必要がある。
  • disabled_nodes の値が異なる場合、その値が小さい方が常に優先される。

アルゴリズムまたは振る舞い

1. **無効ノード数比較**: disabled_nodes が異なる場合、値が小さい方を優先する。 2. **総コスト比較**: total_cost を fuzz_factor を用いて比較する。

  • 片方の総コストが他方の fuzz_factor 倍を超えて大きい場合、起動コストを考慮する。
  • 起動コストの考慮が必要な場合（CONSIDER_PATH_STARTUP_COST マクロによる判定）で、かつ起動コストも逆転している場合は COSTS_DIFFERENT を返す。
  • そうでなければ、総コストが小さい方を優位とする。

    • 3. **起動コスト比較**: 総コストが同等（fuzz_factor の範囲内）とみなされる場合、起動コストを比較する。

  • 起動コストに有意な差があれば、小さい方を優位とする。

    • 4. **最終判定**: 両方のコストが fuzz_factor の範囲内で同等であれば COSTS_EQUAL を返す。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/optimizer/util/pathnode.c 内の add_path および add_partial_path 等で、パスの支配関係を判定するために呼び出される。
  • **呼び出し先**:
  • マクロ CONSIDER_PATH_STARTUP_COST を使用。

関連データ構造

  • Path: コスト情報（total_cost, startup_cost）および disabled_nodes を保持する構造体。
  • PathCostComparison: 比較結果を保持する列挙型。

失敗モードと境界条件

  • fuzz_factor が 1.0 未満の場合、厳密な比較に近い挙動となる。
  • path1 または path2 が NULL の場合の挙動は定義されていない。

保守担当者向けメモ

  • CONSIDER_PATH_STARTUP_COST マクロは、パスのパラメータ化状態（param_info）に応じて、親リレーションの起動コスト考慮フラグを切り替える。
  • この関数は浮動小数点誤差によるプランの不安定化を防ぐための重要なロジックであり、変更時は add_path 等のパス選別ロジックへの影響を考慮すること。
