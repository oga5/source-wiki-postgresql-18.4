RewriteMappingFile

定義

  • **種別**: struct
  • **定義箇所**: src/backend/access/heap/rewriteheap.c:191

目的

テーブル再構築時における論理デコーディング用のマッピング情報を保持し、古いタプルID（TID）と新しいTIDの対応関係を永続化するための状態を管理します。

シグネチャまたは形状

typedef struct RewriteMappingFile
{
    TransactionId xid;          /* 該当行を参照する可能性のあるトランザクションID */
    int           vfd;          /* マッピングファイルのファイル記述子 */
    off_t         off;          /* 書き込み済みオフセット */
    dclist_head   mappings;     /* メモリ上のマッピングリスト */
    char          path[MAXPGPATH]; /* エラーメッセージ用のファイルパス */
} RewriteMappingFile;

入力と出力

  • **入力**: 再構築処理中に生成されるトランザクションIDおよびマッピングデータ。
  • **出力**: pg_logical/mappings 配下のファイルへの永続化データ。

前提条件と不変条件

  • 論理レプリケーションが有効な場合にのみ使用されます。
  • 構造体内の mappings は dclist_head を用いて管理されます。
  • path は MAXPGPATH のサイズ制限に従います。

アルゴリズムまたは振る舞い

  • 再構築処理中、古いヒープから新しいヒープへのタプル移動に伴うTIDの変更を追跡します。
  • vfd を通じてディスク上のマッピングファイルへ逐次書き込みを行い、クラッシュリカバリ時に論理デコーディングが整合性を維持できるようにします。
  • チェックポイント時に不要なマッピングファイルの削除や同期が行われます。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/access/heap/rewriteheap.c 内のハッシュテーブル操作関数（hash_seq_search 等）および src/backend/replication/logical/reorderbuffer.c 内の論理デコーディング処理。
  • **呼び出し先**: 直接的な関数呼び出し関係はソースから特定できませんが、dclist 関連のリスト操作関数を利用します。

関連データ構造

  • dclist_head: マッピングリストの管理に使用。
  • RewriteMappingDataEntry: マッピングの各エントリを構成するデータ構造。

失敗モードと境界条件

  • **メモリ不足**: 大量の未解決タプルが存在する場合、ハッシュテーブルの肥大化によりメモリ不足が発生する可能性があります。
  • **ファイルシステム制限**: path が MAXPGPATH を超える場合や、ファイル書き込み権限・ディスク容量不足によりマッピングファイルの作成が失敗する可能性があります。

保守担当者向けメモ

  • 本構造体は rewriteheap.c と reorderbuffer.c の両方で参照されています。変更時は論理レプリケーションの整合性に影響を与えるため注意してください。
  • vfd の管理は storage/fd.h の流儀に従います。
