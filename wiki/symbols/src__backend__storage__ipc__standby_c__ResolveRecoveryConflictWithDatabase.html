ResolveRecoveryConflictWithDatabase

定義

  • **種別**: function
  • **定義箇所**: src/backend/storage/ipc/standby.c:568

目的

指定されたデータベースに対して接続中のすべてのバックエンドプロセスを強制的に終了させ、データベース削除等のリカバリ操作に伴う競合を解消する。

シグネチャまたは形状

void ResolveRecoveryConflictWithDatabase(Oid dbid)

入力と出力

  • **入力**: dbid (Oid) - 競合解消対象となるデータベースの識別子。
  • **出力**: なし（副作用として対象バックエンドの終了を待機する）。

前提条件と不変条件

  • 本関数を呼び出す前に、対象データベースに対する AccessExclusiveLock が取得済みであること。
  • 呼び出し元は、データベース削除等の排他的なリカバリ操作を実行中であること。

アルゴリズムまたは振る舞い

1. CountDBBackends(dbid) を用いて、対象データベースに接続中のバックエンドが存在するか確認する。 2. 接続中のバックエンドが存在する場合、CancelDBBackends を呼び出し、PROCSIG_RECOVERY_CONFLICT_DATABASE シグナルを送信して対象プロセスを強制終了させる。 3. pg_usleep(10000) を実行し、10ミリ秒待機する。 4. 接続中のバックエンドがゼロになるまで、上記 2〜3 の手順を繰り返す。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/commands/dbcommands.c:3428
  • **呼び出し先**:
  • CountDBBackends
  • CancelDBBackends
  • pg_usleep

関連データ構造

  • なし（本関数はプロセス管理およびシグナル送信機能に依存する）。

失敗モードと境界条件

  • **即時終了**: 本関数は待機を行わず、対象バックエンドを即座に終了させる設計である。
  • **接続ブロック**: 実行中、新規接続を試みるプロセスは InitPostgres() 内でブロックされ、データベース削除を確認した時点で切断される。

保守担当者向けメモ

  • ResolveRecoveryConflictWithVirtualXIDs と異なり、アイドル状態のセッションも対象として強制終了させる。
  • 負荷の高いシステムで対象バックエンドに過度な負荷をかけないよう、ループ内での待機処理（pg_usleep）が組み込まれている。
