ResourceOwnerAddToHash

定義

  • 種別: function
  • 定義箇所: src/backend/utils/resowner/resowner.c:244

目的

リソースオーナーが管理するハッシュテーブルへ、新しいリソース要素（Datum）を登録する。

シグネチャまたは形状

static void ResourceOwnerAddToHash(ResourceOwner owner, Datum value, const ResourceOwnerDesc *kind)

入力と出力

  • **入力**:
  • owner: リソースを保持する ResourceOwnerData 構造体へのポインタ。
  • value: 登録対象のリソース識別子（Datum）。
  • kind: リソースの種類を定義する ResourceOwnerDesc 構造体。
  • **出力**: なし（副作用として owner 内のハッシュテーブルを更新）。

前提条件と不変条件

  • kind は NULL であってはならない。
  • 呼び出し前に ResourceOwnerEnlarge() が実行され、ハッシュテーブルに空き容量が確保されている必要がある。

アルゴリズムまたは振る舞い

1. hash_resource_elem を用いて value と kind からハッシュ値を計算し、owner->capacity に基づくマスクを適用して初期インデックスを算出する。 2. 線形探索を行い、owner->hash[idx].kind が NULL である空きスロットを特定する。 3. 特定したスロットに value と kind を格納する。 4. owner->nhash（ハッシュテーブル内の要素数）をインクリメントする。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/utils/resowner/resowner.c 内の内部処理（リソースの再配置や配列からハッシュへの移行処理など）。
  • **呼び出し先**:
  • hash_resource_elem（ハッシュ値計算用関数）。

関連データ構造

  • ResourceOwnerData: リソースオーナーの内部状態を保持する構造体。
  • ResourceOwnerDesc: リソースの種類を定義する構造体。

失敗モードと境界条件

  • ハッシュテーブルが満杯の場合、無限ループに陥る可能性があるため、必ず事前に ResourceOwnerEnlarge() で十分な容量を確保しておく必要がある。

保守担当者向けメモ

  • 本関数は static であり、resowner.c 内部でのみ使用される。
  • ハッシュ衝突が発生した場合は、単純な線形探索（idx + 1）で次の空きスロットを探す実装となっている。
