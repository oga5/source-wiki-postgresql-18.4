find_join_rel

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/util/relnode.c:526

目的

指定されたリレーション集合（Relids）に対応する結合リレーションノード（RelOptInfo）を、プランナの管理リストまたはハッシュテーブルから検索する。

シグネチャまたは形状

RelOptInfo *find_join_rel(PlannerInfo *root, Relids relids)

入力と出力

  • **入力**:
  • root: 現在のクエリプランナのコンテキスト情報。
  • relids: 検索対象となるリレーションの集合（ビットマップ）。
  • **出力**:
  • 見つかった場合は対応する RelOptInfo * を返し、存在しない場合は NULL を返す。

前提条件と不変条件

  • root->join_rel_list には、現在構築済みの結合リレーションが格納されている必要がある。
  • 検索効率のため、リストの長さが32を超えると root->join_rel_hash が構築され、以降はハッシュ検索が優先される。

アルゴリズムまたは振る舞い

1. **ハッシュテーブルの遅延構築**: join_rel_list の要素数が32を超えており、かつ join_rel_hash が未作成の場合、build_join_rel_hash を呼び出してハッシュテーブルを生成する。 2. **検索実行**:

  • ハッシュテーブルが存在する場合: hash_search を使用して relids をキーに高速検索を行う。
  • ハッシュテーブルが存在しない場合: join_rel_list を線形走査し、bms_equal を用いて relids が一致するノードを探す。

    • 3. **戻り値**: 一致するノードが見つかればそのポインタを返し、見つからなければ NULL を返す。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • contrib/postgres_fdw/postgres_fdw.c
  • src/backend/optimizer/path/joinrels.c
  • src/backend/optimizer/util/relnode.c
  • src/backend/utils/adt/selfuncs.c
  • **呼び出し先**:
  • build_join_rel_hash（条件合致時）
  • hash_search（ハッシュテーブル存在時）
  • bms_equal（リスト走査時）

関連データ構造

  • RelOptInfo: 検索対象となるリレーションノード。
  • PlannerInfo: 結合リレーションのリストおよびハッシュテーブルを保持するコンテキスト。
  • JoinHashEntry: ハッシュテーブル内で使用されるエントリ構造体。

失敗モードと境界条件

  • 該当する relids を持つリレーションが存在しない場合、関数は NULL を返す（エラーにはならない）。
  • リスト長が32以下の場合は線形探索が行われるため、結合数が多いクエリでは検索コストが増大する可能性がある。

保守担当者向けメモ

  • relids のアドレスを直接渡さず、ローカル変数 hashkey にコピーしてからハッシュ検索を行うことで、レジスタ効率を維持する最適化が施されている。このロジックを変更する際は注意すること。
  • join_rel_hash は動的に作成されるため、デバッグ時にはリストとハッシュの両方のパスを考慮する必要がある。
