verify_control_file

定義

  • 種別: function
  • 定義箇所: src/bin/pg_verifybackup/pg_verifybackup.c:754

目的

バックアップディレクトリ内の制御ファイル（global/pg_control）を読み込み、CRC、バージョン、およびシステム識別子がマニフェストファイルと整合しているかを検証する。

シグネチャまたは形状

static void verify_control_file(const char *controlpath, uint64 manifest_system_identifier)

入力と出力

  • **入力**:
  • controlpath: 制御ファイルへのパス（const char *）。
  • manifest_system_identifier: マニフェストから読み取ったシステム識別子（uint64）。
  • **出力**: なし（検証失敗時は report_fatal_error を呼び出しプロセスを終了する）。

前提条件と不変条件

  • controlpath で指定されたファイルが読み取り可能であること。
  • get_controlfile_by_exact_path が制御ファイルの内容を正しく取得できること。

アルゴリズムまたは振る舞い

1. get_controlfile_by_exact_path を呼び出し、制御ファイルの内容と CRC の整合性を取得する。 2. CRC が不正な場合、致命的エラーとして終了する。 3. 制御ファイルのバージョンが PG_CONTROL_VERSION と一致しない場合、致命的エラーとして終了する。 4. 制御ファイルのシステム識別子が、引数 manifest_system_identifier と一致しない場合、致命的エラーとして終了する。 5. 最後に pfree を呼び出し、割り当てられたメモリを解放する。

呼び出し元と呼び出し先

  • **呼び出し元**: src/bin/pg_verifybackup/pg_verifybackup.c 内の検証ロジック（735行目付近）。
  • **呼び出し先**: get_controlfile_by_exact_path（外部関数）、pg_log_debug、report_fatal_error、pfree。

関連データ構造

  • ControlFileData: 制御ファイルの内容を保持する構造体。

失敗モードと境界条件

  • **CRC 不一致**: ファイル破損または読み取りエラー。
  • **バージョン不一致**: 異なる PostgreSQL バージョンで作成されたバックアップである可能性。
  • **システム識別子不一致**: 異なるデータベースインスタンスのバックアップである可能性。
  • いずれの失敗も report_fatal_error によりプログラムが終了する。

保守担当者向けメモ

  • 本関数は制御ファイルの整合性を厳格にチェックする。将来的に制御ファイルの構造が変更された場合、PG_CONTROL_VERSION のチェックロジックに影響が出る可能性がある。
