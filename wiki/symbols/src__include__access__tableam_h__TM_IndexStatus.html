TM_IndexStatus

定義

  • **種別**: struct
  • **定義箇所**: src/include/access/tableam.h:215

目的

インデックスの削除処理、特にボトムアップ（bottom-up）インデックス削除において、各インデックスエントリの状態を管理するために使用される構造体です。

シグネチャまたは形状

typedef struct TM_IndexStatus
{
    OffsetNumber idxoffnum;     /* インデックスAMページ内のオフセット番号 */
    bool         knowndeletable; /* 削除可能であることが判明しているか */
    bool         promising;      /* 有望な（重複した）インデックスエントリか */
    int16        freespace;      /* 削除時に解放されるインデックス内の領域 */
} TM_IndexStatus;

入力と出力

本構造体は、インデックス削除の判断プロセスにおいて、対象となるインデックスエントリのメタデータ（オフセットや削除の可否、解放可能領域）を保持・伝達する役割を担います。

前提条件と不変条件

  • TM_IndexStatus の配列は、インデックス削除処理のコンテキスト（delstate 等）において、対象となるインデックスエントリ数分だけメモリ確保（palloc）される必要があります。
  • idxoffnum は、対象となるインデックスページ内での有効なオフセットを指している必要があります。

アルゴリズムまたは振る舞い

インデックスの削除処理において、各インデックスエントリの状態を追跡します。特に knowndeletable フラグにより、そのエントリが削除対象として特定済みであるかを管理し、promising フラグを用いて重複エントリの最適化判断を行います。freespace は、削除が実行された場合にインデックスページ内で再利用可能となる領域サイズを記録します。

関連データ構造

  • TM_IndexDelete: インデックス削除操作を管理する構造体。
  • TM_IndexDeleteOp: インデックス削除操作の種別を定義。
  • OffsetNumber: インデックスページ内の位置を特定するための型。

失敗モードと境界条件

  • メモリ確保失敗: palloc による配列確保時にメモリ不足が発生する可能性があります。
  • 配列インデックスの範囲外アクセス: delstate->status へのアクセス時に、インデックスIDが確保された配列サイズを超えないよう注意が必要です。

保守担当者向けメモ

  • 本構造体はボトムアップインデックス削除の最適化に直結しています。フィールドを追加・変更する場合は、nbtree の重複排除（deduplication）ロジックへの影響を考慮してください。
  • freespace の単位はバイト数であり、インデックスAMの実装がこの値を正しく解釈していることを確認してください。
