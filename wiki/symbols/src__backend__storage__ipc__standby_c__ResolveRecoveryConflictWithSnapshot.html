ResolveRecoveryConflictWithSnapshot

定義

  • **種別**: function
  • **定義箇所**: src/backend/storage/ipc/standby.c:467

目的

ホットスタンバイ環境において、プライマリから受信したWALレコードが要求するスナップショットの整合性を確保するため、競合するクエリを実行中のバックエンドを特定し、必要に応じて終了させる。

シグネチャまたは形状

void ResolveRecoveryConflictWithSnapshot(TransactionId snapshotConflictHorizon,
                                         bool isCatalogRel,
                                         RelFileLocator locator)

入力と出力

  • **入力**:
  • snapshotConflictHorizon: 競合の基準となるトランザクションID。
  • isCatalogRel: 対象がシステムカタログリレーションであるかを示すフラグ。
  • locator: 対象リレーションの識別子（データベースOIDを含む）。
  • **出力**: なし（副作用としてバックエンドのキャンセルやレプリケーションスロットの無効化を行う）。

前提条件と不変条件

  • snapshotConflictHorizon が InvalidTransactionId の場合、本関数は何もせず即座に復帰する。
  • snapshotConflictHorizon は TransactionIdIsNormal を満たす必要がある（アサートによる検証）。

アルゴリズムまたは振る舞い

1. **無効値チェック**: snapshotConflictHorizon が有効でない場合は終了。 2. **競合バックエンドの特定**: GetConflictingVirtualXIDs を呼び出し、指定されたホライゾンとデータベースOIDに基づいて競合する仮想トランザクションIDのリストを取得する。 3. **競合解決の実行**: ResolveRecoveryConflictWithVirtualXIDs を呼び出し、PROCSIG_RECOVERY_CONFLICT_SNAPSHOT シグナルを送信して対象バックエンドを停止させる。 4. **カタログリレーションの処理**: wal_level が WAL_LEVEL_LOGICAL かつ isCatalogRel が真の場合、InvalidateObsoleteReplicationSlots を呼び出し、不要になったレプリケーションスロットを無効化する。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • src/backend/access/gist/gistxlog.c
  • src/backend/access/hash/hash_xlog.c
  • src/backend/access/heap/heapam.c
  • src/backend/access/heap/heapam_xlog.c
  • src/backend/access/nbtree/nbtxlog.c
  • src/backend/access/spgist/spgxlog.c
  • **呼び出し先**:

  • ResolveRecoveryConflictWithVirtualXIDs
  • InvalidateObsoleteReplicationSlots

関連データ構造

  • TransactionId: トランザクション識別子。
  • RelFileLocator: リレーションの物理的な場所を特定する構造体。
  • VirtualTransactionId: 仮想トランザクションID。

失敗モードと境界条件

  • **タイムアウト**: 本関数内では WaitExceedsMaxStandbyDelay() による待機チェックは行われない。これは、物理レプリケーションスロットの使用等により、通常この種の競合には到達しないという設計上の前提に基づいている。

保守担当者向けメモ

  • 本関数は32ビットの TransactionId を扱う。64ビット版の ResolveRecoveryConflictWithSnapshotFullXid も存在するため、用途に応じて使い分けること。
  • システムカタログに対する競合が発生した場合、論理レプリケーションスロットの無効化が連鎖的に発生する点に注意が必要。
