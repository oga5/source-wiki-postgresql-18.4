RewriteStateData

定義

  • 種別: struct
  • 定義箇所: src/backend/access/heap/rewriteheap.c:130

目的

テーブル（ヒープ）の物理的な再構築処理（VACUUM FULL や CLUSTER 等）において、再構築中の状態を保持・管理するために使用される構造体です。

シグネチャまたは形状

typedef struct RewriteStateData {
    Relation    rs_old_rel;
    Relation    rs_new_rel;
    BulkWriteState *rs_bulkstate;
    BulkWriteBuffer rs_buffer;
    BlockNumber rs_blockno;
    bool        rs_logical_rewrite;
    TransactionId rs_oldest_xmin;
    TransactionId rs_freeze_xid;
    TransactionId rs_logical_xmin;
    MultiXactId rs_cutoff_multi;
    MemoryContext rs_cxt;
    XLogRecPtr  rs_begin_lsn;
    HTAB       *rs_unresolved_tups;
    HTAB       *rs_old_new_tid_map;
    HTAB       *rs_logical_mappings;
    uint32      rs_num_rewrite_mappings;
} RewriteStateData;

入力と出力

  • **入力**: begin_heap_rewrite 関数内で palloc0 を用いてメモリ確保され、初期化されます。
  • **出力**: 再構築処理の終了時（end_heap_rewrite）に、保持していたハッシュテーブルやメモリコンテキストが解放されます。

前提条件と不変条件

  • 本構造体を使用する呼び出し元は、対象テーブルに対して AccessExclusiveLock を保持している必要があります。
  • rs_cxt は、ハッシュテーブルやタプルデータなどの再構築に関連する動的メモリを管理するための専用コンテキストとして使用されます。

アルゴリズムまたは振る舞い

  • **状態管理**: ヒープ再構築のライフサイクル（開始から終了まで）を追跡します。
  • **更新チェーン解決**: rs_unresolved_tups および rs_old_new_tid_map ハッシュテーブルを使用し、古いヒープと新しいヒープ間での ctid リンクの整合性を維持します。
  • **バルク書き込み**: rs_bulkstate と rs_buffer を介して、ページ単位の効率的な書き込みを制御します。
  • **論理デコーディング**: rs_logical_rewrite が有効な場合、論理レプリケーション用のマッピング情報を rs_logical_mappings に蓄積します。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/access/heap/rewriteheap.c 内の begin_heap_rewrite 関数でインスタンス化されます。
  • **呼び出し先**: 構造体内部の各フィールドは、rewrite_heap_tuple や end_heap_rewrite 等のヒープ再構築関連関数によって参照・更新されます。

関連データ構造

  • BulkWriteState, BulkWriteBuffer: バルク書き込み用。
  • HTAB: 更新チェーン解決および論理マッピング用のハッシュテーブル。
  • RewriteState: RewriteStateData へのポインタ型（src/include/access/rewriteheap.h で定義）。

失敗モードと境界条件

  • **メモリ不足**: 更新チェーンの解決にインメモリのハッシュテーブルを使用するため、巨大なテーブルで未解決の更新が大量に滞留するとメモリ不足が発生する可能性があります。

保守担当者向けメモ

  • 本構造体は不透明な構造体として扱われることが想定されています。
  • 論理レプリケーションが有効な場合、マッピングファイルが pg_logical/mappings に生成されるため、チェックポイント処理（CheckPointLogicalRewriteHeap）との整合性に注意してください。
