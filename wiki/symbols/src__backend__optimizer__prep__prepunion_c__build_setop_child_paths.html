build_setop_child_paths

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/prep/prepunion.c:480

目的

集合演算（UNION, INTERSECT, EXCEPT）の対象となる子サブクエリに対して、実行計画のパスを構築する。特に、集合演算に必要なソート順序を満たすパスや、コスト効率の良いパスを生成・選定する。

シグネチャまたは形状

static void build_setop_child_paths(PlannerInfo *root, RelOptInfo *rel,
                                    bool trivial_tlist, List *child_tlist,
                                    List *interesting_pathkeys, double *pNumGroups)

入力と出力

  • root: 親クエリの PlannerInfo。
  • rel: 処理対象の RelOptInfo（RTE_SUBQUERY である必要がある）。
  • trivial_tlist: 入出力の型が一致し変換不要かを示すフラグ。
  • child_tlist: 子クエリのターゲットリスト。
  • interesting_pathkeys: 集合演算で要求されるソート順序。
  • pNumGroups: （出力）推定グループ数。呼び出し元が指定した場合に格納される。

前提条件と不変条件

  • rel->rtekind は RTE_SUBQUERY でなければならない。
  • final_rel（UPPERREL_FINAL）の consider_parallel フラグを rel に反映させる必要がある。
  • postprocess_setop_rel を呼び出して後処理を完了させる。

アルゴリズムまたは振る舞い

1. **初期化**: set_subquery_size_estimates を呼び出し、サブクエリのサイズを見積もる。 2. **パス生成**: final_rel->pathlist を走査し、以下のパスを生成する。

  • 最安コストのパス（cheapest_total_path）をそのまま追加。
  • interesting_pathkeys が指定されている場合、ソートが必要なパスに対して create_sort_path または create_incremental_sort_path を適用してソート済みパスを生成。

    • 3. **並列処理**: consider_parallel が有効な場合、partial_pathlist から最安のパスを抽出し、add_partial_path を行う。 4. **後処理**: postprocess_setop_rel を呼び出し、リレーションのパスを確定させる。 5. **統計**: pNumGroups が指定されている場合、サブクエリの groupClause 等に基づきグループ数を推定する。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/optimizer/prep/prepunion.c 内の各所（generate_union_paths や generate_nonunion_paths 等から複数回参照）。
  • **呼び出し先**: add_setop_child_rel_equivalences, set_subquery_size_estimates, fetch_upper_rel, add_path, create_subqueryscan_path, create_sort_path, create_incremental_sort_path, add_partial_path, postprocess_setop_rel 等。

関連データ構造

  • RelOptInfo: 処理対象のリレーション情報。
  • PlannerInfo: プランニングのコンテキスト。
  • Path: 生成される実行パス。
  • PathKey: ソート順序の定義。

失敗モードと境界条件

  • rel が RTE_SUBQUERY でない場合、Assert により停止する。
  • final_rel->consider_parallel が偽の場合、部分パス（partial_pathlist）は空でなければならない。
  • pNumGroups の推定において、varno 0 を含むターゲットリストが混入すると誤った結果を招く可能性がある（ソース内のコメントによる警告）。

保守担当者向けメモ

  • 本関数は subquery_planner の実行前にサイズ見積もりを行う必要がある。
  • varno 0 のハックは将来的な設計変更が必要な箇所である。
  • enable_incremental_sort の設定に応じて、ソート手法の選択が分岐する。
