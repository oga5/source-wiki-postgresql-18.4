create_plan_recurse

定義

  • 種別: function
  • 定義箇所: src/backend/optimizer/plan/createplan.c:387

目的

プランナが生成した Path ツリーを、実行エンジンが処理可能な Plan ノードのツリーへと再帰的に変換する。

シグネチャまたは形状

Plan *create_plan_recurse(PlannerInfo *root, Path *best_path, int flags)

入力と出力

  • **入力**:
  • root: クエリ全体のプランニング情報を保持する PlannerInfo 構造体。
  • best_path: 変換対象となる Path ノード。
  • flags: ターゲットリストの生成制約などを指定するビットフラグ（CP_ 系）。
  • **出力**:
  • 変換後の Plan ノードへのポインタ。

前提条件と不変条件

  • 呼び出し前に check_stack_depth() を実行し、スタックオーバーフローを防止する。
  • best_path の pathtype に基づいて適切な変換関数を呼び出す。
  • 未知のノードタイプが渡された場合は elog(ERROR) を発生させる。

アルゴリズムまたは振る舞い

1. スタック深さを確認する。 2. best_path->pathtype を評価し、ノードタイプに応じた変換関数（create_scan_plan, create_join_plan, create_agg_plan 等）へディスパッチする。 3. T_Result や T_Unique など、複数のサブタイプを持つノードについては、IsA() マクロを用いて具体的なパス型を判定し、適切な生成関数を選択する。 4. 生成された Plan ノードを呼び出し元へ返す。

呼び出し元と呼び出し先

  • **呼び出し元**:
  • create_plan (同ファイル内)
  • 各種 create_xxx_plan 関数（再帰的なサブプラン生成時）
  • **呼び出し先**:
  • create_scan_plan, create_join_plan, create_append_plan, create_projection_plan, create_agg_plan 等の各ノード生成関数。

関連データ構造

  • Path: 変換元の最適化パス。
  • Plan: 変換先の実行計画ノード。
  • PlannerInfo: プランニング全体のコンテキスト。

失敗モードと境界条件

  • **スタックオーバーフロー**: 複雑すぎるプランツリーに対しては check_stack_depth() がエラーを投げる。
  • **未知のノードタイプ**: pathtype が定義されていない場合、elog(ERROR) により処理を中断する。

保守担当者向けメモ

  • 本関数は createplan.c の中核となるディスパッチャである。新しい Path ノードタイプを追加した際は、必ず switch 文にケースを追加すること。
  • T_Result や T_Unique のように、単一の pathtype に対して複数のパス構造体が割り当てられているケースがあるため、追加時は IsA() による型チェックを適切に行うこと。
