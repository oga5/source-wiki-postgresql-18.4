ExecMergeAppend

定義

  • 種別: function
  • 定義箇所: src/backend/executor/nodeMergeAppend.c:214

目的

ソート済みの複数の入力サブプランをマージし、単一のソート済みストリームとしてタプルを順次提供する。

シグネチャまたは形状

TupleTableSlot *ExecMergeAppend(PlanState *pstate)

入力と出力

  • **入力**: PlanState *pstate（MergeAppendState 構造体としてキャストされる実行ノードの状態）。
  • **出力**: マージされた順序における次の TupleTableSlot。すべてのサブプランが終了した場合は空のタプルを返す。

前提条件と不変条件

  • 各サブプランは、共通のソートキーに従って既にソート済みであること。
  • ms_initialized が false の場合、初回実行時にサブプランのプルーニングと binaryheap の構築が行われる。
  • binaryheap は、各サブプランから取得したタプルを保持し、最小（または最大）のタプルを効率的に特定するために使用される。

アルゴリズムまたは振る舞い

1. **割り込みチェック**: CHECK_FOR_INTERRUPTS() を実行。 2. **初期化フェーズ (!ms_initialized)**:

  • サブプランが一つも存在しない場合、空のタプルを返して終了。
  • ms_valid_subplans が未設定の場合、ExecFindMatchingSubPlans を呼び出し、実行時プルーニングを実行。
  • 各有効なサブプランから最初のタプルを取得し、binaryheap に挿入。その後 binaryheap_build でヒープを構築。

    • 3. **反復フェーズ**:

  • 前回の呼び出しでタプルを返したサブプランから次のタプルを取得。
  • 新しいタプルが取得できた場合、binaryheap_replace_first でヒープを更新。
  • 取得できなかった場合、binaryheap_remove_first で当該サブプランをヒープから除外。

    • 4. **結果の返却**:

  • ヒープが空であれば終了（空のタプルを返却）。
  • ヒープの先頭にあるサブプランのタプルを result として返却。

呼び出し元と呼び出し先

  • **呼び出し元**: 実行エンジン（PlanState の ExecProcNode フィールド経由）。
  • **呼び出し先**:
  • ExecFindMatchingSubPlans（初期化時）
  • ExecProcNode（各サブプランのタプル取得用）
  • binaryheap 操作関数群（binaryheap_add_unordered, binaryheap_build, binaryheap_first, binaryheap_replace_first, binaryheap_remove_first, binaryheap_empty）

関連データ構造

  • MergeAppendState: ノードの実行状態を保持。
  • TupleTableSlot: タプルを格納するコンテナ。
  • binaryheap: マージ処理に使用される優先度付きキュー。

失敗モードと境界条件

  • **サブプランの全除外**: ms_nplans == 0 の場合、即座に空のタプルを返す。
  • **サブプランの枯渇**: 全サブプランからタプルが取得できなくなった場合、binaryheap が空になり、終了状態へ遷移する。

保守担当者向けメモ

  • heap_compare_slots は SortSupport を使用しており、省略キー（abbreviated key）による最適化は意図的に無効化されている。
  • サブプラン数が多い場合、ヒープ操作のコストが増大するため注意が必要。
  • 呼び出し元と呼び出し先の詳細なコールグラフは、実行時の動的解析により補完可能。
