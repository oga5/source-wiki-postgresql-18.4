RelationInitPhysicalAddr

定義

  • 種別: function
  • 定義箇所: src/backend/utils/cache/relcache.c:1338

目的

リレーション記述子（Relation）の物理ストレージ位置情報（rd_locator）を初期化または更新する。

シグネチャまたは形状

static void RelationInitPhysicalAddr(Relation relation)

入力と出力

  • **入力**: relation (初期化対象のリレーション記述子へのポインタ)
  • **出力**: なし（引数 relation の内部構造体が直接更新される）

前提条件と不変条件

  • relation->rd_rel が有効であること。
  • ストレージを持たないリレーション（RELKIND_HAS_STORAGE が偽）の場合、本関数は何もせず即座に復帰する。

アルゴリズムまたは振る舞い

1. **テーブルスペースの決定**: rd_rel->reltablespace が設定されていればそれを使用し、未設定なら MyDatabaseTableSpace を使用する。 2. **データベースOIDの決定**: グローバルテーブルスペースの場合は InvalidOid、それ以外は MyDatabaseId を設定する。 3. **物理ファイル番号の特定**:

  • relfilenode が存在する場合、それを使用する。ただし、論理デコーディングの履歴スナップショットがアクティブな場合は、カタログから最新の relfilenode を再取得して更新する。
  • relfilenode が存在しない（マップが必要な）場合、RelationMapOidToFilenumber を呼び出して OID から物理ファイル番号を解決する。

    • 4. **並列ワーカーの調整**: 並列ワーカー実行時、物理ファイル番号が変更された場合は、WALスキップの要否に応じて rd_firstRelfilelocatorSubid を再設定する。

呼び出し元と呼び出し先

  • **呼び出し元**: src/backend/utils/cache/relcache.c 内の複数箇所（RelationBuildDesc 等、リレーション構築・再構築プロセス）。
  • **呼び出し先**: RelationMapOidToFilenumber, ScanPgRelation, heap_freetuple, RelationGetRelid, RelationGetRelationName, RelationIsAccessibleInLogicalDecoding, HistoricSnapshotActive, IsTransactionState, IsParallelWorker, RelFileLocatorSkippingWAL。

関連データ構造

  • RelationData (引数 relation の型)
  • RelFileLocator (更新対象の rd_locator フィールド)
  • Form_pg_class (カタログスキャン時に使用)

失敗モードと境界条件

  • pg_class エントリが見つからない場合、elog(ERROR) を発生させる。
  • リレーションマッピングが見つからない場合、elog(ERROR) を発生させる。

保守担当者向けメモ

  • 本関数は relcache.c 内でリレーションの物理位置を確定させる唯一のロジックであるため、類似のロジックを持つ catalog.c や dbsize.c との整合性に注意すること。
  • 論理デコーディング中の relfilenode 再取得処理は、スキーマ変更を伴うリライトとの競合を防ぐための重要なハックである。
