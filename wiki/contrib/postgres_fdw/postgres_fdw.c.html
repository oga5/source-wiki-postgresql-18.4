contrib/postgres_fdw/postgres_fdw.c

概要

postgres_fdw.c は、PostgreSQL の外部データラッパー（FDW）のメインエントリポイントです。リモートの PostgreSQL サーバーに対するクエリのプランニング、実行、およびデータ操作（DML）のライフサイクルを管理します。本ファイルは、PostgreSQL コアのプランナやエグゼキュータから呼び出されるコールバック関数群を実装し、リモートサーバーとの通信やクエリの変換を調整するハブの役割を果たします。

モジュール内での役割

postgres_fdw モジュールにおける「司令塔」です。deparse.c（SQL生成）、connection.c（接続管理）、option.c（設定検証）といった各サブコンポーネントを統合し、プランナが生成したパスを実際の実行プランに変換したり、実行時のタプル変換やエラーハンドリングを制御します。

責務

  • **プランニング**: GetForeignRelSize や GetForeignPaths を通じて、リモートテーブルのコスト見積もりと実行パスの生成を行う。
  • **実行**: BeginForeignScan / IterateForeignScan を介したリモートカーソルの管理とタプル取得。
  • **データ操作**: PlanForeignModify や ExecForeignInsert 等によるリモート DML 操作の実行と、RETURNING 句の処理。
  • **最適化**: リモートサーバーの統計情報を利用したコスト見積もり（estimate_path_cost_size）と、結合・集約のプッシュダウン判定。
  • **リソース管理**: リモート接続のライフサイクル管理と、実行コンテキスト（メモリコンテキスト）の管理。

重要なシンボル

プランニング用構造体

  • PgFdwRelationInfo: プランナが使用するリモートテーブルの最適化情報（コスト、条件、プッシュダウン可否など）。
  • PgFdwPathExtraData: コスト見積もり時に使用する追加情報（LIMIT、ソート情報など）。

実行状態管理構造体

  • PgFdwScanState: ForeignScan ノードの実行状態（接続、カーソル、タプルバッファ）。
  • PgFdwModifyState: ModifyTable ノードの実行状態（INSERT/UPDATE/DELETE 用）。
  • PgFdwDirectModifyState: 直接的なリモート更新（UPDATE/DELETE）の実行状態。
  • PgFdwAnalyzeState: ANALYZE 実行時のサンプリング状態。

コールバック関数群

  • postgres_fdw_handler: FDW の全コールバック関数を登録するエントリポイント。
  • postgresGetForeignRelSize / postgresGetForeignPaths: プランナが呼び出すパス生成の起点。
  • postgresGetForeignPlan: プランナが最終的な ForeignScan ノードを作成する際に呼び出す。
  • postgresIterateForeignScan: エグゼキュータがタプルをフェッチする際に呼び出す。

制御フローとデータフロー

1. **プランニング**: postgresGetForeignRelSize が PgFdwRelationInfo を初期化し、classifyConditions でプッシュダウン可能な条件を分離します。estimate_path_cost_size がリモートの EXPLAIN を実行してコストを算出します。 2. **実行**: postgresBeginForeignScan でリモート接続を確立し、DECLARE CURSOR を発行します。postgresIterateForeignScan が FETCH を繰り返し、make_tuple_from_result_row でリモートのテキスト形式データをローカルのタプル形式に変換します。 3. **DML**: postgresPlanForeignModify がリモート SQL を生成し、execute_foreign_modify がプリペアドステートメントを使用して実行します。

読者向けメモ

  • **メモリ管理**: PgFdwScanState 等の構造体は estate->es_query_cxt に割り当てられ、クエリ終了時に自動解放されます。タプル変換には temp_cxt を使用し、メモリリークを防いでいます。
  • **エラーハンドリング**: リモートエラーは pgfdw_report_error を通じてローカルの PostgreSQL エラーとして報告されます。conversion_error_callback により、変換失敗時にどのカラムでエラーが起きたか特定可能です。
  • **制約**: postgres_fdw はリモートのトランザクション状態をローカルと同期させますが、分散トランザクションの完全な原子性は保証されません。
  • **未実装**: 複雑な結合や集約のプッシュダウンは deparse.c に依存しており、本ファイルはそれらの実行を仲介する役割に徹しています。
