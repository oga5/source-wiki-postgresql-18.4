contrib/postgres_fdw/deparse.c

概要

deparse.c は、postgres_fdw におけるクエリ変換（デパース）の中核を担うファイルです。ローカルの PostgreSQL クエリツリーを解析し、リモートサーバーで実行可能な SQL 文字列に変換します。また、式がリモート実行可能かどうかの安全性判定も行います。

モジュール内での役割

postgres_fdw のプランナとエグゼキュータの間に位置し、ローカルで最適化されたプランをリモートサーバーが理解できる SQL に翻訳します。postgres_fdw.c から呼び出され、リモート実行の可否判定および SQL 生成を完結させます。

責務

  • **安全性判定**: 式がリモートで安全に実行可能か（関数・演算子の実行可能性、照合順序の整合性、可変性の有無）を検証。
  • **SQL 生成**: SELECT, INSERT, UPDATE, DELETE 文の構築。
  • **プッシュダウン支援**: 結合、集約、フィルタ条件、ソート、LIMIT 句の SQL への変換。
  • **パラメータ管理**: ローカルの式をリモートのパラメータ（$1, $2...）にマッピング。

重要なシンボル

コンテキスト構造体

  • foreign_glob_cxt: プランニング中のグローバルな状態（PlannerInfo 等）を保持。
  • deparse_expr_cxt: SQL 生成時のコンテキスト。出力バッファ、パラメータリスト、対象リレーション情報を保持。
  • foreign_loc_cxt: 式の再帰的な走査中に照合順序（Collation）の安全性を追跡。

安全性判定関数

  • classifyConditions: 条件式をリモート実行可能なものとローカル実行が必要なものに分類。
  • is_foreign_expr: 式全体がリモートで安全に実行可能か判定。
  • foreign_expr_walker: 式ツリーを再帰的に走査し、ノードごとの安全性を検証。

SQL 生成関数

  • deparseSelectStmtForRel: SELECT 文全体の構築（FROM, WHERE, GROUP BY, ORDER BY, LIMIT 等）。
  • deparseInsertSql / UpdateSql / DeleteSql: DML 文の構築。
  • deparseExpr: 個別の式ノードを SQL 文字列に変換するディスパッチャ。

制御フローとデータフロー

1. **判定フェーズ**: classifyConditions が is_foreign_expr を呼び出し、各条件式を走査。foreign_expr_walker はノードを再帰的にチェックし、照合順序が外部テーブルの列に由来するかを確認。 2. **生成フェーズ**: deparseSelectStmtForRel が SQL 生成の起点となり、deparseFromExpr で FROM 句を、appendWhereClause で WHERE 句を構築。 3. **再帰的変換**: deparseExpr がノードタグに応じて各 deparse* 関数（deparseFuncExpr, deparseOpExpr 等）を呼び出し、StringInfo バッファに SQL を蓄積。 4. **パラメータ化**: deparseVar や deparseParam が、ローカルの式を params_list に登録し、リモート用のパラメータ番号（$n）に変換。

読者向けメモ

  • **照合順序の制約**: 本モジュールは、照合順序が外部テーブルの列に由来しない式を「安全ではない」と見なします。これはリモートサーバーとの照合順序の不一致を防ぐためです。
  • **メモリ管理**: StringInfo を多用して SQL を構築します。メモリは CurrentMemoryContext で管理されます。
  • **未実装・浅い部分**: 複雑なサブクエリのプッシュダウンや、一部の特殊な型変換については、安全性を優先してプッシュダウンを諦める設計になっています。
  • **注意点**: deparse_type_name はローカルの search_path に依存せず、pg_catalog を明示的に使用するよう設計されています。これはリモートサーバーの環境に依存しないためです。
