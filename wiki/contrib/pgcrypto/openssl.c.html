contrib/pgcrypto/openssl.c

概要

openssl.c は、pgcrypto モジュールが提供する暗号化およびハッシュ機能を OpenSSL ライブラリの EVP (Envelope) API を介して実装するためのラッパーです。PostgreSQL のメモリ管理およびリソース管理機構と OpenSSL のライフサイクルを統合する役割を担います。

モジュール内での役割

pgcrypto のバックエンド実装の一つであり、px.h で定義された抽象インターフェース（PX_MD, PX_Cipher）を OpenSSL の具体的な関数群にマッピングします。他の実装（pgp-mpi-openssl.c など）と連携し、PostgreSQL が提供する暗号化関数の基盤を支えています。

責務

  • **抽象化**: OpenSSL の複雑な EVP API を pgcrypto 共通のインターフェースへ変換する。
  • **リソース管理**: ResourceOwner を使用し、エラー発生時やトランザクション終了時に OpenSSL のコンテキスト（EVP_MD_CTX, EVP_CIPHER_CTX）を確実に解放する。
  • **互換性維持**: 暗号アルゴリズムのエイリアス解決や、OpenSSL のバージョン差異（FIPS モードの判定など）の吸収。
  • **安全性**: Blowfish 等の鍵長制限チェックなど、OpenSSL の挙動を補完する検証を行う。

重要なシンボル

  • OSSLDigest: ハッシュ計算用のコンテキスト構造体。ResourceOwner と紐付けられる。
  • OSSLCipher: 暗号化/復号用のコンテキスト構造体。鍵や IV の状態を保持する。
  • px_find_digest(): 指定された名前のハッシュアルゴリズムを検索し、初期化済みの PX_MD を返す。
  • px_find_cipher(): 指定された名前の暗号アルゴリズムを検索し、初期化済みの PX_Cipher を返す。
  • CheckBuiltinCryptoMode(): FIPS モード等の設定に基づき、組み込み暗号機能の使用可否を検証する。

制御フローとデータフロー

1. **初期化**: px_find_... 関数が呼ばれると、TopMemoryContext 上にコンテキスト構造体（OSSLDigest または OSSLCipher）を確保し、ResourceOwner に登録する。 2. **操作**: update や encrypt/decrypt などのメソッドが呼ばれると、内部で EVP_DigestUpdate や EVP_CipherUpdate 等の OpenSSL 関数を呼び出す。 3. **終了**: free メソッドが呼ばれると、OpenSSL のコンテキストを破棄し、ResourceOwner から登録を解除する。 4. **異常系**: トランザクションがアボートされると、ResourceOwner のコールバック（ResOwnerRelease...）が自動的に呼び出され、メモリリークを防ぐ。

直接依存

  • postgres.h: PostgreSQL のコア定義。
  • openssl/evp.h, openssl/crypto.h: OpenSSL の主要な暗号化・ハッシュ API。
  • px.h: pgcrypto の内部抽象インターフェース。
  • utils/resowner.h: リソース管理（メモリリーク防止）のためのヘッダー。
  • openssl/err.h
  • openssl/rand.h
  • utils/memutils.h

関連コード

  • src/include/common/openssl.h: OpenSSL のバージョン依存定義や共通設定。
  • contrib/pgcrypto/px.c: px_find_digest や px_find_cipher を呼び出す上位層のロジック。
  • contrib/pgcrypto/pgp-mpi-openssl.c: OpenSSL を使用した PGP 関連の MPI（多倍長整数）処理。

読者向けメモ

  • 本ファイルは OpenSSL の EVP API を直接ラップしているため、OpenSSL のバージョンアップに伴う挙動の変化（特に FIPS 関連）に注意が必要です。
  • ResourceOwner の使用は、PostgreSQL のエラーハンドリング（elog(ERROR, ...)）でスタックが巻き戻される際に、OpenSSL のコンテキストがリークするのを防ぐための必須の設計です。新しいアルゴリズムを追加する際は、必ず ResourceOwner への登録・解除フローを維持してください。
