contrib/pg_prewarm/autoprewarm.c

概要

autoprewarm.c は、PostgreSQL の共有バッファ（shared_buffers）の状態を定期的にファイルへダンプし、サーバー再起動時にその内容を自動的に読み戻すためのバックグラウンドワーカーを実装しています。これにより、再起動後のウォームアップ時間を短縮します。

モジュール内での役割

pg_prewarm モジュールにおける「自動化」レイヤーを担います。手動でプリウォームを行う pg_prewarm.c とは異なり、本ファイルはバックグラウンドワーカーとして常駐し、サーバーのライフサイクルに合わせた自動的な状態保存と復元を管理します。

責務

  • **状態の永続化**: autoprewarm.blocks ファイルへ現在のバッファ情報を定期的に書き出す。
  • **自動復元**: サーバー起動時にダンプファイルを読み込み、共有バッファを再構築する。
  • **ワーカー管理**:
  • リーダーワーカーがダンプ情報の読み込みとソートを行う。
  • データベースごとに個別のワーカーを起動し、効率的にプリウォームを実行する。

重要なシンボル

  • AutoPrewarmSharedState: 共有メモリ上の制御構造体。ワーカー間の同期やダンプ情報の受け渡しに使用。
  • BlockInfoRecord: ダンプされる各ブロックのメタデータ（DB, テーブルスペース, ファイル番号, フォーク番号, ブロック番号）。
  • autoprewarm_main: リーダーワーカーのメインループ。
  • autoprewarm_database_main: 各データベースごとのプリウォーム処理を実行するワーカーのメイン関数。
  • apw_dump_now: 現在の共有バッファ状態をファイルに書き出す関数。

制御フローとデータフロー

1. **初期化**: _PG_init で GUC 変数を登録し、有効であればリーダーワーカーを起動。 2. **起動時**: autoprewarm_main が autoprewarm.blocks を読み込み、BlockInfoRecord の配列を動的共有メモリ（DSM）に展開。 3. **プリウォーム**: リーダーワーカーがデータベースごとに autoprewarm_database_main を起動。各ワーカーは read_stream API を使用して、対象ブロックを効率的に読み込む。 4. **定期ダンプ**: リーダーワーカーは autoprewarm_interval ごとに apw_dump_now を呼び出し、現在のバッファ状態を一時ファイル経由でアトミックに更新する。

読者向けメモ

  • **ロックの注意点**: プリウォームはカタログが整合性を保てる状態（ConsistentState）になるまで開始されません。
  • **パフォーマンス**: プリウォーム処理は have_free_buffer() をチェックし、空きバッファが枯渇した場合は処理を中断する安全装置を備えています。
  • **ファイル形式**: ダンプファイルはテキスト形式であり、デバッグや手動編集が容易です。ただし、大規模な shared_buffers ではファイルサイズが大きくなる可能性がある点に注意してください。
