contrib/pg_prewarm/pg_prewarm.c

概要

PostgreSQL のデータファイルを OS キャッシュまたは共有バッファへ事前に読み込む（プリウォーム）ための SQL 関数 pg_prewarm を提供するソースファイルです。

モジュール内での役割

contrib/pg_prewarm モジュールの中心的な実行ロジックを担います。ユーザーが SQL インターフェースを通じて指定したリレーションのブロックを、指定された手法でメモリ上にロードします。

責務

  • **引数の検証**: リレーション、フォーク、ブロック範囲、およびプリウォーム手法（prefetch, read, buffer）の妥当性を確認します。
  • **権限チェック**: 対象リレーションに対する SELECT 権限を検証します（インデックスの場合は親テーブルの権限を確認）。
  • **プリウォーム実行**: 選択された手法に基づき、OS へのヒント送信、直接読み込み、または共有バッファへのロードを実行します。

重要なシンボル

  • pg_prewarm: 外部公開されている SQL 関数。プリウォームのメインエントリポイント。
  • PrewarmType: プリウォーム手法を定義する列挙型。
  • PREFETCH: OS の posix_fadvise 等を利用した非同期読み込み。
  • READ: smgrread を用いた同期的な物理読み込み（OS キャッシュへ格納）。
  • BUFFER: ReadStream を用いた共有バッファへの読み込み。

制御フローとデータフロー

1. **初期化**: SQL 引数を解析し、PrewarmType を決定します。 2. **アクセス制御**: relation_open でリレーションをロックし、pg_class_aclcheck で権限を確認します。 3. **範囲計算**: 指定されたブロック範囲を検証し、NULL の場合はデフォルト値（0 から最終ブロックまで）を適用します。 4. **実行ループ**:

  • PREFETCH: PrefetchBuffer を呼び出し、OS に読み込みを依頼します。
  • READ: smgrread でバッファへ直接読み込みます。
  • BUFFER: read_stream_begin_relation を介して共有バッファへロードします。

    • 5. **終了**: ロックを解放し、処理したブロック数を返します。

読者向けメモ

  • **割り込み**: 各ループ内では CHECK_FOR_INTERRUPTS() が呼び出されており、長時間実行されるプリウォーム処理中でもユーザーによるキャンセルが可能です。
  • **インデックスの扱い**: インデックスをプリウォームする場合、内部的に親テーブルの権限チェックとロックが行われるため、デッドロック回避のための順序（テーブル→インデックス）が考慮されています。
  • **パフォーマンス**: BUFFER モードは共有バッファを消費するため、メモリ圧迫に注意が必要です。PREFETCH や READ は OS キャッシュを利用するため、共有バッファを汚染しません。
