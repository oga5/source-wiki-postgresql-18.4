contrib/pageinspect/hashfuncs.c

概要

PostgreSQL のハッシュインデックス（HASH AM）の内部構造を低レベルで調査するための SQL 関数群を提供します。主にデバッグやインデックスの物理的な状態確認を目的としています。

モジュール内での役割

contrib/pageinspect モジュールの一部として、ハッシュインデックス特有のページレイアウト（メタページ、バケットページ、オーバーフローページ、ビットマップページ）を解析し、ユーザーが SQL 経由で内容を読み取れるようにします。

責務

  • **ページ検証**: verify_hash_page を通じて、入力された bytea データが有効なハッシュインデックスページであるかを検証します。
  • **統計情報の抽出**: 特定のページ内のライブ/デッドタプル数や空き容量を算出します。
  • **メタデータ解析**: ハッシュインデックスのメタページから、バケット数やビットマップ配置などの内部パラメータを抽出します。
  • **アイテム列挙**: ページ内のインデックスアイテム（IndexTuple）を走査し、TID やハッシュキーを返します。

重要なシンボル

  • verify_hash_page: 全ての関数で呼び出されるゲートキーパー。ページヘッダーの整合性とタイプをチェックします。
  • HashPageStat: ページ統計情報を保持する構造体。
  • hash_page_type: ページの種類（メタ、バケット等）を文字列で返します。
  • hash_page_items: SRF（Set Returning Function）として、ページ内の全アイテムを列挙します。
  • hash_bitmap_info: 指定したオーバーフローブロックがどのビットマップページで管理されているかを特定します。

制御フローとデータフロー

1. **入力**: ユーザーが get_raw_page() で取得した bytea データを関数に渡します。 2. **検証**: verify_hash_page がページヘッダーの hasho_page_id と hasho_flag を確認し、不正な場合はエラーを投げます。 3. **解析**:

  • 統計系関数は PageGetMaxOffsetNumber でループし、ItemIdIsDead でタプル状態を判定します。
  • アイテム列挙関数は FuncCallContext を使用して、複数行の結果を段階的に返します。

    • 4. **出力**: heap_form_tuple を用いて、解析結果を PostgreSQL の行形式に変換して返却します。

読者向けメモ

  • **権限**: 全ての関数は superuser() チェックを必須としています。
  • **安全性**: 本モジュールは物理ページを直接読み取るため、インデックスが破損している場合や、誤ったページを指定した場合には予期せぬエラーが発生する可能性があります。
  • **読み方**: まず verify_hash_page を読み、ページ検証の仕組みを理解してください。その後、hash_page_items のような SRF 実装パターンを確認すると、他の pageinspect ファイルの理解も早まります。
