contrib/spi/refint.c

概要

refint.c は、PostgreSQL の SPI (Server Programming Interface) を利用して、トリガーベースの参照整合性制約を実装するための関数群を提供します。標準の外部キー制約が利用できない環境や、カスタムな整合性チェックが必要なケースにおいて、CREATE TRIGGER 文から呼び出されるプロシージャとして機能します。

モジュール内での役割

contrib/spi モジュールの一部として、データベースのデータ整合性を維持するための拡張機能を提供します。他の spi モジュール（autoinc や moddatetime など）と同様に、トリガー関数を通じてデータベースの自動操作を補助します。

責務

  • **主キー制約の検証**: check_primary_key により、挿入または更新されるタプルが参照先のテーブルに存在するかを確認します。
  • **外部キー制約の維持**: check_foreign_key により、削除または更新時に参照整合性を維持するためのアクション（RESTRICT, CASCADE, SET NULL）を実行します。
  • **実行計画のキャッシュ**: SPI_prepare で作成したクエリ計画を EPlan 構造体で管理し、セッション内で再利用することでオーバーヘッドを削減します。

重要なシンボル

  • check_primary_key: 参照先テーブルの存在確認を行うトリガー関数。
  • check_foreign_key: 参照元テーブルに対する連鎖的な削除・更新・制限を行うトリガー関数。
  • EPlan: 実行計画（SPIPlanPtr）と識別子を保持するキャッシュ用構造体。
  • find_plan: キャッシュされた EPlan を検索、または新規作成する内部関数。

制御フローとデータフロー

1. **トリガー起動**: TriggerData を介してトリガー引数（テーブル名、カラム名、アクション種別）を取得します。 2. **計画の検索**: ident 文字列（トリガー名とリレーションIDの組み合わせ）を用いて find_plan を呼び出し、既存の実行計画を検索します。 3. **計画の準備**: 計画が存在しない場合、SPI_prepare を用いて動的に SQL を構築し、TopMemoryContext に保存します。 4. **実行**: SPI_execp を使用して、準備されたクエリを現在のタプル値で実行します。 5. **結果の処理**: 整合性違反が検出された場合は ereport(ERROR) でトランザクションを中断し、正常な場合はタプルを返します。

直接依存

  • postgres.h: 基本的な型定義とマクロ。
  • executor/spi.h: SPI インターフェース。
  • commands/trigger.h: トリガー管理用構造体。
  • utils/memutils.h: メモリコンテキスト管理（TopMemoryContext 等）。
  • utils/rel.h: リレーション情報へのアクセス。
  • ctype.h
  • utils/builtins.h

関連コード

  • contrib/spi/: 本ファイルが属するディレクトリ。他の SPI 拡張の実装例が含まれます。
  • src/backend/executor/spi.c: SPI の基盤実装。本ファイルが利用する SPI_prepare や SPI_execp の詳細を理解する際に参照します。

読者向けメモ

  • **メモリ管理**: SPI_keepplan を使用して、計画を TopMemoryContext に保持しています。これは、トリガー関数終了後も計画を生存させるために不可欠です。
  • **制約**: 本実装は古いスタイルのトリガー関数であり、現代の PostgreSQL では標準の FOREIGN KEY 制約を使用することが強く推奨されます。本コードは学習用、または極めて特殊な要件のための実装例として参照してください。
  • **デバッグ**: DEBUG_QUERY マクロを有効にすると、実行される SQL がログに出力されます。
