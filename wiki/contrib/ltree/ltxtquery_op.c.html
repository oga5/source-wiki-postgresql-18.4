contrib/ltree/ltxtquery_op.c

概要

ltxtquery_op.c は、ltree 型と ltxtquery 型（ブール演算を含む複雑な検索クエリ）を照合するための演算子ロジックを提供します。

モジュール内での役割

contrib/ltree モジュールにおいて、ユーザーが定義した複雑な検索条件（AND, OR, NOT を含む）を ltree パスに対して評価するエンジンとして機能します。

責務

  • **クエリ実行エンジン**: ltree_execute を通じて、ポーランド記法で格納された ltxtquery のツリー構造を再帰的に走査し、論理演算を解決する。
  • **条件照合**: ltree の各ラベルとクエリ内の各要素を比較し、前方一致や大文字小文字の区別などのフラグを考慮したマッチングを行う。
  • **演算子インターフェース**: PostgreSQL の SQL 関数として ltxtq_exec（@ 演算子）および ltxtq_rexec（@ 演算子の引数逆転版）を公開する。

重要なシンボル

  • ltree_execute: 汎用的な再帰的ブール演算評価関数。コールバック関数 chkcond を用いて個別のマッチングロジックを抽象化している。
  • checkcondition_str: ltree_execute から呼び出される具体的な照合関数。ltree の各レベルを走査し、クエリの条件と一致するか判定する。
  • CHKVAL: 照合時に必要な ltree データとクエリのオペランド文字列を保持するコンテキスト構造体。

追加の索引済みシンボル

  • ltxtq_exec
  • ltxtq_rexec

制御フローとデータフロー

1. **エントリポイント**: ltxtq_exec が SQL から呼び出され、ltree と ltxtquery を取得。 2. **コンテキスト構築**: CHKVAL 構造体に評価対象の ltree とクエリのオペランドを格納。 3. **再帰評価**: ltree_execute がクエリの ITEM 配列を走査。

  • VAL 型（値）に到達すると checkcondition_str を呼び出す。
  • !, &, | 演算子に応じて再帰的に左右のノードを評価。

    • 4. **マッチング**: checkcondition_str が ltree の各レベルをループし、ltree_label_match 等を用いて一致を確認。

直接依存

  • postgres.h: PostgreSQL の基本型およびマクロ。
  • ltree.h: ltree および ltxtquery のデータ構造定義と操作マクロ。
  • miscadmin.h: スタックオーバーフロー防止のための check_stack_depth() を提供。
  • ctype.h

関連コード

  • ltxtquery_io.c: ltxtquery 型の入出力および解析を担当。
  • ltree_op.c: 単純な ltree 比較演算子や補助関数。
  • lquery_op.c: lquery 型（パターンマッチング）の評価ロジック。

読者向けメモ

  • ltree_execute は再帰関数であるため、複雑なクエリに対してはスタック消費に注意が必要です。check_stack_depth() が挿入されているのはそのためです。
  • ltxtquery はポーランド記法（後置記法）で格納されているため、ltree_execute のポインタ演算（curitem + 1 や curitem->left）は、この構造を前提としています。
  • LVAR_ 系フラグ（LVAR_SUBLEXEME, LVAR_ANYEND など）がマッチングの挙動を決定するため、これらを理解することが照合ロジック把握の鍵となります。
