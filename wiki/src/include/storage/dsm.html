src/include/storage/dsm.h

概要

dsm.h は、PostgreSQL における動的共有メモリ（Dynamic Shared Memory: DSM）セグメントを管理するための高レベルインターフェースを定義します。低レベルな実装（dsm_impl.h）をラップし、セグメントの作成、アタッチ、デタッチ、およびリソース管理を安全に行うための API を提供します。

モジュール内での役割

src/include/storage モジュールにおいて、プロセス間通信（IPC）の基盤を担います。静的な共有メモリとは異なり、実行時に必要に応じてメモリ領域を確保・解放できるため、並列クエリ実行やバックグラウンドワーカー間のデータ共有に不可欠です。

責務

  • **ライフサイクル管理**: セグメントの作成から、参照カウントに基づく自動的な解放までを管理。
  • **リソース所有権の追跡**: ResourceOwner と連携し、バックエンド終了時に不要なセグメントが残らないよう自動デタッチを行う。
  • **ピン留め機能**: セグメントやマッピングをセッション寿命まで保持するための制御。
  • **コールバック機構**: セグメントデタッチ時に実行されるクリーンアップ処理の登録。

重要なシンボル

  • dsm_segment: セグメントの状態を保持する不透明な構造体。
  • dsm_create(): 指定サイズで新しい DSM セグメントを作成。
  • dsm_attach(): 既存の DSM セグメントを現在のプロセスにマッピング。
  • dsm_detach(): セグメントへのマッピングを解除し、参照カウントを減らす。
  • dsm_handle: セグメントを一意に識別するためのハンドル。

制御フローとデータフロー

1. **初期化**: dsm_shmem_init() により、共有メモリ制御領域が初期化されます。 2. **作成/アタッチ**: dsm_create() または dsm_attach() を呼び出すと、dsm_segment 構造体が生成され、プロセスローカルなリストに登録されます。 3. **利用**: dsm_segment_address() で取得したポインタを介して共有メモリ領域にアクセスします。 4. **終了**: プロセス終了時、dsm_detach_all() が呼ばれ、on_dsm_detach コールバックを通じてリソースが安全に解放されます。

直接依存

  • storage/dsm_impl.h: OS 依存の低レベルな DSM 実装（mmap 等）へのアクセス。

関連コード

  • src/backend/storage/ipc/dsm.c: 本ヘッダーで宣言された関数の実装。
  • src/include/storage/dsm_registry.h: 名前付き DSM セグメントを管理するための高レベルなレジストリ。
  • src/include/storage/shm_toc.h: DSM 内に複数のデータ構造を配置するためのテーブル・オブ・コンテント機能。

読者向けメモ

  • **自動クリーンアップ**: このモジュールで作成されたマッピングは、明示的に dsm_detach しなくても、リソースオーナー（トランザクション等）の終了時に自動的にクリーンアップされます。
  • **ピン留め**: dsm_pin_mapping() を使用すると、リソースオーナーの寿命を超えてマッピングを保持できます。セッション終了まで保持したい場合に利用してください。
  • **エラーハンドリング**: DSM_CREATE_NULL_IF_MAXSEGMENTS フラグを使用すると、セグメント上限に達した際にエラーではなく NULL を返す制御が可能です。
