src/include/storage/pg_sema.h

概要

pg_sema.h は、PostgreSQL におけるセマフォ操作のためのプラットフォーム非依存な抽象化レイヤーを定義するヘッダーファイルです。OS 固有のセマフォ実装を隠蔽し、バックエンド全体で一貫した同期プリミティブを提供します。

モジュール内での役割

src/include/storage モジュールの一部として、プロセス間同期のための低レベルな基盤を提供します。主に共有メモリ上のリソース管理や、プロセス間のシグナル伝達の基礎として利用されます。

責務

  • **抽象化**: プラットフォームごとのセマフォ実装（POSIXセマフォ、System Vセマフォ、Windowsハンドル等）を PGSemaphore 型として統一する。
  • **ライフサイクル管理**: セマフォの初期化、共有メモリサイズの算出、およびリソースの確保を行う。
  • **同期操作**: ロック（P操作）、アンロック（V操作）、および非ブロッキングな試行（TryLock）のインターフェースを定義する。

重要なシンボル

  • PGSemaphore: セマフォを指す抽象ハンドル。Windows環境では HANDLE、それ以外では struct PGSemaphoreData * として定義されます。
  • PGSemaphoreLock(sema): カウントをデクリメントし、0未満になる場合はブロックする（P操作）。
  • PGSemaphoreUnlock(sema): カウントをインクリメントする（V操作）。
  • PGSemaphoreTryLock(sema): ブロックせずにロックを試みる。成功時は true を返す。
  • PGSemaphoreShmemSize(maxSemas): セマフォ管理に必要な共有メモリサイズを計算する。

制御フローとデータフロー

1. **初期化**: ポストマスター起動時に PGReserveSemaphores が呼ばれ、システム全体で必要なセマフォ数が確保されます。 2. **生成**: PGSemaphoreCreate により、個別のセマフォインスタンスが初期化されます。 3. **同期**: 共有リソースへのアクセス前後で PGSemaphoreLock / PGSemaphoreUnlock が呼び出され、排他制御やリソースカウントの調整が行われます。 4. **終了**: プロセス終了時や共有メモリ再初期化時に、必要に応じてリセットや解放が行われます。

直接依存

本ファイルはプラットフォーム非依存を維持するため、他のPostgreSQL内部ヘッダーへの直接的な依存関係は持っていません。ただし、コンパイル時には c.h 等の基本定義が前提となります。

関連コード

  • src/backend/storage/ipc/pg_sema.c: 本ヘッダーで宣言された関数のプラットフォーム固有の実装が含まれます。
  • src/include/storage/latch.h: より高レベルな待機イベント管理（Latch）を提供しており、現代のPostgreSQLではセマフォよりもこちらが優先されるケースが多いです。
  • src/include/storage/lwlock.h: 軽量ロック（LWLock）の実装において、セマフォの概念が内部的に利用されることがあります。

読者向けメモ

  • **実装の隠蔽**: PGSemaphoreData は不透明な構造体として扱われます。プラットフォーム非依存のコードは、その内部構造に決して触れてはなりません。
  • **Windowsの例外**: Windows環境では PGSemaphore がネイティブな HANDLE に直接マッピングされるため、他のプラットフォームとABI互換性がない点に注意してください。
  • **利用の注意**: セマフォは低レベルなプリミティブです。デッドロックのリスクがあるため、可能な限り LWLock や Latch などの高レベルな同期機構の使用を検討してください。
