src/include/storage/md.h

概要

md.h は、PostgreSQL のストレージマネージャ（smgr）層における「磁気ディスク（Magnetic Disk）」ストレージ管理の公開インターフェースを定義するヘッダーファイルです。物理的なディスクの種類を問わず、OS のファイルシステムを介してリレーション（テーブルやインデックス）のデータを読み書きするための基盤を提供します。

モジュール内での役割

storage/smgr サブシステムの一部として、smgr.h で定義された抽象的なストレージ操作を、具体的なファイルシステム操作へマッピングする役割を担います。md.c と対になっており、PostgreSQL のバッファマネージャが要求するブロック単位の I/O を、OS レベルのファイル操作に変換します。

責務

  • **セグメント管理**: OS のファイルサイズ制限（通常 2GB）を超えるリレーションを、複数のセグメントファイルに分割して管理する。
  • **ファイル操作**: リレーションの作成、オープン、クローズ、削除、および拡張（extend/truncate）の実行。
  • **I/O インターフェース**: ブロック単位の読み書き（mdreadv, mdwritev）および非同期 I/O（AIO）のサポート。
  • **同期管理**: チェックポイントやリカバリ時に必要な、ファイル同期（fsync）の登録と実行。

重要なシンボル

  • SMgrRelation: ストレージマネージャが管理するリレーションの構造体。
  • mdextend / mdtruncate: リレーションのサイズを変更する主要関数。
  • mdreadv / mdwritev: 複数のブロックを効率的に読み書きするためのベクトル I/O インターフェース。
  • mdsyncfiletag / mdunlinkfiletag: 同期処理（sync）の際に使用されるファイルタグ操作関数。

制御フローとデータフロー

1. **初期化**: mdinit() により、ストレージ管理に必要なコンテキストが準備されます。 2. **アクセス**: バッファマネージャが mdreadv や mdwritev を呼び出すと、md.c は内部の MdfdVec 配列を参照し、対象のブロックがどのセグメントファイルに属するかを特定します。 3. **セグメント解決**: mdfd() 関数が、指定された BlockNumber に対応するファイル記述子（VFD）を解決します。 4. **I/O 実行**: 解決されたファイル記述子を使用して、OS の read/write システムコール（または AIO）が発行されます。

直接依存

  • storage/aio_types.h: 非同期 I/O 関連の型定義。
  • storage/block.h: ブロック番号やサイズに関する定義。
  • storage/relfilelocator.h: リレーションを識別するための識別子。
  • storage/smgr.h: ストレージマネージャの抽象インターフェース。
  • storage/sync.h: 同期処理（fsync）のための定義。

関連コード

  • src/backend/storage/smgr/md.c: 本ヘッダーで宣言された関数の実装。
  • src/include/storage/smgr.h: md.c が実装する SMgrRelation 構造体の定義元。
  • src/backend/storage/file/fd.c: md.c が使用する仮想ファイル記述子（VFD）の管理層。

読者向けメモ

  • **セグメントの概念**: PostgreSQL は大きなテーブルを RELSEG_SIZE ごとのファイルに分割します。md.c はこの分割を透過的に扱うため、上位層は単一の大きなファイルとしてリレーションを扱えます。
  • **非同期 I/O**: 近年の PostgreSQL では mdstartreadv などを通じて非同期 I/O が強化されています。I/O パフォーマンスを追う際は、これらのベクトル I/O インターフェースが重要になります。
  • **不変条件**: mdtruncate を実行しても、物理ファイルは即座に削除されず、サイズ 0 のまま保持されることがあります。これは他のプロセスがファイル記述子を保持している可能性を考慮した設計です。
