src/include/storage/lwlock.h

概要

lwlock.h は、PostgreSQL における軽量ロック（Lightweight Lock: LWLock）のデータ構造とインターフェースを定義するヘッダーファイルです。LWLock は、共有メモリ上のデータ構造を保護するための低オーバーヘッドな相互排他メカニズムを提供します。

モジュール内での役割

src/include/storage モジュールにおいて、共有メモリ管理やロック管理の基盤を担います。他の複雑なロック機構（lock.h で定義される重量ロックなど）が共有状態を保護するために、この LWLock を利用します。

責務

  • **ロック構造の定義**: キャッシュライン境界を考慮した LWLock 構造体の定義。
  • **排他制御の提供**: 共有モードおよび排他モードでのロック取得・解放インターフェースの提供。
  • **動的トランチ（Tranche）管理**: 拡張機能やコア機能が独自のロックグループ（トランチ）を動的に割り当て、管理するための仕組みを提供。
  • **待機管理**: ロック競合時の待機プロセス管理と、変数更新待ち（LWLockUpdateVar 等）のサポート。

重要なシンボル

  • LWLock: ロックの本体。tranche ID とアトミックな state 変数、待機リストを保持します。
  • LWLockPadded: LWLock をキャッシュラインサイズ（LWLOCK_PADDED_SIZE）にパディングした構造体。キャッシュ競合を最小化します。
  • MainLWLockArray: システム全体で共有される主要な LWLock の配列。
  • LWLockAcquire / LWLockRelease: ロックの取得および解放を行う主要関数。
  • BuiltinTrancheIds: システムで定義済みのロックグループ ID。

追加の索引済みシンボル

制御フローとデータフロー

1. **初期化**: CreateLWLocks により共有メモリ上に MainLWLockArray が確保されます。 2. **取得**: LWLockAcquire はアトミックな compare-and-exchange を試みます。成功すれば即座にロックを獲得し、失敗した場合は waiters リストに自身（PGPROC）を追加して OS レベルで待機します。 3. **解放**: LWLockRelease はアトミックな減算を行い、待機者がいれば適切なプロセスを起床させます。 4. **変数待ち**: LWLockUpdateVar と LWLockWaitForVar を使用することで、ロックを保持したまま特定の共有変数の変更を待機するフローが可能です。

直接依存

  • port/atomics.h: アトミック操作の基盤。
  • storage/lwlocknames.h: ロック名等の定義。
  • storage/proclist_types.h: 待機プロセスリストの型定義。

読者向けメモ

  • **パフォーマンス**: LWLock はスピンロックよりも高機能ですが、重量ロックより軽量です。頻繁にアクセスされる共有データ構造の保護に適しています。
  • **キャッシュ競合**: LWLockPadded を使用しているため、構造体のサイズ変更には注意が必要です。
  • **デバッグ**: LOCK_DEBUG が定義されている場合、ロックの所有者や待機者数が追跡可能になりますが、パフォーマンスに影響するため通常は無効です。
  • **フロントエンド**: 本ヘッダーはフロントエンドコードからはインクルードできません。
