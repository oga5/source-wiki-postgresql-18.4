src/include/storage/aio_internal.h

概要

PostgreSQL の非同期 I/O (AIO) サブシステムの内部実装詳細を定義するヘッダーファイルです。このファイルは、AIO サブシステム内部でのみ使用されるデータ構造、状態遷移、および IO メソッドのインターフェースを規定します。

モジュール内での役割

storage/aio.h が公開 API を提供するのに対し、本ファイルはサブシステム内部の制御構造（PgAioCtl や PgAioBackend）や、I/O 完了までの状態遷移を管理するための詳細な定義を提供します。

責務

  • **状態管理**: PgAioHandleState を用いた I/O ハンドルのライフサイクル管理。
  • **バックエンド制御**: 各バックエンドごとの I/O 状態（ステージング、実行中、完了待ち）の保持。
  • **IO メソッドの抽象化**: IoMethodOps 構造体による、異なる I/O 実装（同期、ワーカー、io_uring 等）のプラグイン可能なインターフェースの定義。
  • **デバッグ支援**: AIO サブシステム専用のデバッグログマクロの提供。

重要なシンボル

  • PgAioHandle: 個別の I/O 操作を追跡する主要な構造体。
  • PgAioBackend: バックエンドごとの I/O キューやステージング領域を管理。
  • PgAioCtl: システム全体で共有される AIO 制御構造体。
  • IoMethodOps: I/O 操作の実行・待機・初期化を抽象化する関数ポインタテーブル。
  • PGAIO_SUBMIT_BATCH_SIZE: 一括投入可能な最大 I/O 数（32）。

追加の索引済みシンボル

  • PgAioHandleState

制御フローとデータフロー

1. **定義**: pgaio_io_start_* により PgAioHandle が HANDED_OUT から DEFINED 状態へ移行。 2. **ステージング**: pgaio_io_stage を経て STAGED 状態となり、PgAioBackend の staged_ios 配列に格納される。 3. **実行**: submit コールバックが呼び出され、SUBMITTED 状態へ移行。 4. **完了**: I/O 完了後、pgaio_io_process_completion が呼ばれ、COMPLETED_IO から COMPLETED_LOCAL へ遷移し、ハンドルが再利用可能（IDLE）になる。

直接依存

  • lib/ilist.h: リンクドリスト管理用。
  • port/pg_iovec.h: iovec 構造体の利用。
  • storage/aio.h: 公開 API 定義。
  • storage/condition_variable.h: I/O 完了待ち合わせ用。

関連コード

  • src/backend/storage/aio/aio.c: AIO サブシステムのメインロジック。
  • src/include/storage/aio_types.h: AIO で使用される列挙型や基本型の定義。
  • src/include/storage/io_worker.h: バックグラウンドワーカーによる I/O 実行関連。

読者向けメモ

  • **状態遷移**: PgAioHandle の状態変更は必ず pgaio_io_update_state() を経由する必要があります。
  • **メモリ効率**: PgAioHandle 内のフィールドは、メモリ使用量を抑えるためにビットフィールドではなく uint8 を多用しています。
  • **デバッグ**: PGAIO_VERBOSE マクロが有効な場合、pgaio_debug_io を通じて詳細な I/O トレースが可能です。パフォーマンスへの影響があるため、本番環境での取り扱いには注意してください。
