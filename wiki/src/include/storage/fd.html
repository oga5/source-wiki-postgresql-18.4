src/include/storage/fd.h

概要

PostgreSQLにおける「仮想ファイル記述子（VFD）」管理システムのインターフェース定義です。OSのファイル記述子（FD）制限を回避し、多数のファイルを効率的に扱うための抽象化レイヤーを提供します。

モジュール内での役割

storage サブシステムにおいて、物理的なファイルアクセスを仲介する最下位の抽象化層です。smgr（ストレージマネージャ）などの上位モジュールが直接OSの open(2) を呼ぶ代わりに、このモジュールを介することで、プロセスごとのFD枯渇を防ぎ、LRUアルゴリズムによるFDの動的な開閉を実現します。

責務

  • **VFD管理**: File 型による仮想的なファイルハンドルの提供と、LRUキャッシュによる実FDの管理。
  • **リソース保護**: AllocateFile や OpenTransientFile を通じた、トランザクション終了時の自動クローズ保証。
  • **プラットフォーム抽象化**: O_DIRECT のようなOS固有フラグの正規化や、Windows/Unix間の挙動差異の吸収。
  • **永続性保証**: fsync 関連のラッパー提供による、データ整合性の維持。

重要なシンボル

  • File: VFDを表す整数型。OSのFDとは直接互換性がありません。
  • PathNameOpenFile: VFDを生成し、ファイルを開くための主要なエントリポイント。
  • AllocateFile / FreeFile: fopen/fclose の代替。サブトランザクションでの自動クローズをサポート。
  • FileRead / FileWrite: iovec を用いた読み書きのラッパー。
  • PG_O_DIRECT: OS依存のダイレクトI/Oフラグを抽象化したマクロ。

追加の索引済みシンボル

  • FileExtendMethod

制御フローとデータフロー

1. **オープン**: PathNameOpenFile が呼ばれると、VFDが割り当てられ、必要に応じてLRUリストから古い実FDがクローズされます。 2. **アクセス**: FileRead/FileWrite を通じてデータが転送されます。内部では FileReadV/FileWriteV が呼び出され、オフセット指定による読み書きが行われます。 3. **クローズ**: FileClose によりVFDが解放されます。一時ファイルの場合は、トランザクション終了時に AtEOXact_Files 等を通じて自動的にクローズ・削除されます。 4. **外部FD**: BasicOpenFile 等で直接OSのFDを取得した場合、AcquireExternalFD を呼び出して、VFD管理システムにそのFDの存在を通知する必要があります。

読者向けメモ

  • **原則**: PostgreSQL内では、標準の open, fopen, opendir を直接使用してはいけません。必ず本ヘッダーで定義されたラッパーを使用してください。
  • **EMFILE対策**: BasicOpenFile 等で直接FDを確保する場合、必ず AcquireExternalFD を呼び出してください。これを怠ると、VFD管理システムがFDの空き数を誤認し、EMFILE エラーが発生するリスクがあります。
  • **移植性**: FileRead/FileWrite はオフセット指定が必須です。ファイルポインタの概念はVFDには存在しないため注意してください。
