src/include/storage/smgr.h

概要

smgr.h は、PostgreSQL のストレージマネージャ（SMgr）層の公開インターフェースを定義するヘッダーファイルです。物理的なリレーションファイル（テーブルやインデックスなど）に対するすべての I/O 操作を抽象化し、上位レイヤーがファイルシステムの実装詳細を意識せずにデータアクセスを行えるようにします。

モジュール内での役割

src/include/storage モジュールにおいて、SMgr はバッファマネージャ（bufmgr.h）と物理ストレージ（md.c 等）の間に位置するスイッチング層です。リレーションのオープン、クローズ、読み書き、同期といった物理的なファイル操作を、SMgrRelation というキャッシュされたハンドルを通じて一元管理します。

責務

  • **リレーションハンドルの管理**: SMgrRelation オブジェクトの生成・破棄・ピン留め（参照カウント管理）のインターフェース提供。
  • **I/O 操作の抽象化**: 読み込み（smgrreadv）、書き込み（smgrwritev）、拡張（smgrextend）などの物理 I/O 操作の定義。
  • **リソースのライフサイクル制御**: トランザクション終了時のリソース解放や、ファイル記述子の強制クローズ（smgrreleaseall）の制御。
  • **キャッシュの保持**: リレーションのサイズ（ブロック数）や挿入ターゲットブロックの情報を保持し、I/O 効率を向上させる。

重要なシンボル

  • SMgrRelationData: リレーションの物理識別子（RelFileLocatorBackend）と、オープン中のファイル記述子、キャッシュされたブロック数などを保持する構造体。
  • SMgrRelation: SMgrRelationData へのポインタ。
  • smgropen(): 指定されたリレーションのハンドルを取得または作成する。
  • smgrreadv / smgrwritev: 複数のブロックを一度に処理するベクトル I/O インターフェース。
  • smgrpin() / smgrunpin(): リレーションハンドルがトランザクション終了時に破棄されないよう保護する。

追加の索引済みシンボル

  • smgrread
  • smgrwrite

制御フローとデータフロー

1. **オープン**: 上位レイヤーが smgropen() を呼び出し、ハッシュテーブルから既存の SMgrRelation を取得するか、新規作成します。 2. **操作**: smgrreadv や smgrwritev が呼び出されると、内部で適切なストレージマネージャ（通常は md.c）へディスパッチされます。 3. **同期**: smgrdosyncall 等により、ダーティなデータが物理ディスクへフラッシュされます。 4. **クローズ**: トランザクション終了時に AtEOXact_SMgr が呼ばれ、ピン留めされていない SMgrRelation が解放されます。

直接依存

  • lib/ilist.h: dlist_node によるリスト管理に使用。
  • storage/aio_types.h: 非同期 I/O 関連の型定義。
  • storage/block.h: BlockNumber や ForkNumber 等のブロック管理用定義。
  • storage/relfilelocator.h: リレーションを特定するための識別子定義。

関連コード

  • src/backend/storage/smgr/smgr.c: 本ヘッダーで宣言された関数の実装およびハッシュテーブル管理。
  • src/backend/storage/smgr/md.c: 実際のファイルシステム I/O を行うストレージマネージャの実装。
  • src/include/storage/bufmgr.h: バッファマネージャ。SMgr を利用してディスク I/O を実行する主要なクライアント。

読者向けメモ

  • SMgrRelation はトランザクション単位で管理されます。ピン留めされていないものはトランザクション終了時に自動的にクローズされるため、長期的な参照が必要な場合は必ず smgrpin() を使用してください。
  • smgr.c の関数は、割り込み（elog(ERROR) 等）によって smgrreleaseall がトリガーされる可能性があるため、リエントラントではないことに注意が必要です。
  • smgrread / smgrwrite は smgrreadv / smgrwritev の単一ブロック用ラッパーです。新規実装時はベクトル I/O の利用が推奨されます。
