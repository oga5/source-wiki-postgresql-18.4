src/include/common/link-canary.h

概要

link-canary.h は、PostgreSQL のビルド環境が「フロントエンド（クライアントライブラリ等）」か「バックエンド（サーバー本体）」かを実行時に判別するための仕組みを提供します。

モジュール内での役割

src/common 配下の関数は、フロントエンドとバックエンドの両方から利用されます。しかし、ELF 形式のプラットフォーム等では、共有ライブラリ（libpq など）がバックエンドにロードされた際、意図せずバックエンド側の同名関数が呼び出される「シンボル衝突」のリスクがあります。本モジュールは、このリンク状態を検証するためのガードレールとして機能します。

責務

  • フロントエンドとバックエンドの実行環境を区別するインターフェースの提供。
  • 共有ライブラリが正しい環境でリンクされているかを検証するための判定ロジックの定義。

重要なシンボル

  • pg_link_canary_is_frontend(void): 呼び出し元がフロントエンドとしてコンパイルされていれば true を、バックエンドであれば false を返す関数。

制御フローとデータフロー

1. **呼び出し**: 共有ライブラリ（libpq 等）が、自身の環境が期待通りであるかを確認するために pg_link_canary_is_frontend() を呼び出します。 2. **判定**: link-canary.c 内で、コンパイル時に定義される FRONTEND マクロの有無に基づき、真偽値が返されます。 3. **検証**: 呼び出し側は戻り値を確認し、期待と異なる環境であればエラー処理や警告を行うことが可能です。

直接依存

  • なし（標準的な C のヘッダーガードのみ）。

関連コード

  • src/common/link-canary.c: 本ヘッダーで宣言された関数の実装が含まれています。FRONTEND マクロによる条件分岐が記述されています。

読者向けメモ

  • 本ファイルは、機能的な処理を行うものではなく、リンク時のシンボル解決に関するデバッグや安全性の確保を目的とした特殊なユーティリティです。
  • 一般的なアプリケーションロジックで使用されることはありません。主にライブラリのロード時や初期化時の整合性チェックに利用されます。
