src/include/nodes/pathnodes.h

概要

pathnodes.h は、PostgreSQL プランナー（最適化エンジン）がクエリの実行計画を探索・構築する過程で使用する内部データ構造を定義するヘッダーファイルです。本ファイルは、クエリの最適化における「探索空間」と「コスト評価」の基盤を担っており、プランナーがクエリをどのように処理するかを決定するための情報を保持します。

モジュール内での役割

src/include/nodes モジュールにおいて、本ファイルはプランナー専用のデータ構造を定義する中心的な役割を果たします。parsenodes.h で定義されたクエリツリーを、plannodes.h で定義される最終的な実行計画へと変換する過程で、中間状態（RelOptInfo, Path など）を管理します。

責務

  • **プランナーのグローバル状態管理**: 計画作成全体で共有される情報の保持。
  • **リレーション情報の管理**: ベースリレーション、結合リレーション、およびそれらの統計情報やコストの保持。
  • **パス探索の管理**: 実行計画の候補となる Path ノードの定義と、それらのコスト・順序情報の管理。
  • **最適化の制約管理**: 結合順序、等価クラス（EquivalenceClass）、およびインデックス利用可能性の追跡。

重要なシンボル

プランナーのコンテキスト

  • PlannerInfo: クエリレベルごとの計画情報を保持するルート構造体。
  • PlannerGlobal: クエリ全体（サブクエリ含む）で共有されるグローバルな計画情報。

リレーションとパス

  • RelOptInfo: リレーション（ベース、結合、上位）ごとの最適化情報を保持。
  • Path: 実行計画の候補となるパスの基底構造体。IndexPath, NestPath, HashPath など多数の派生型が存在。
  • PathTarget: パスが計算する出力列の定義。

最適化の制約と等価性

  • EquivalenceClass: 結合条件やソート順序の等価性を管理する構造体。
  • PathKey: パスのソート順序を表現するノード。
  • RestrictInfo: 制限句（WHERE/JOIN条件）の最適化用メタデータ。
  • SpecialJoinInfo: 外結合やセミ結合の制約情報を保持。

パーティション関連

  • PartitionScheme: パーティションの構造定義。
  • RelOptInfo 内のパーティション関連フィールド（nparts, boundinfo 等）。

制御フローとデータフロー

1. **初期化**: PlannerInfo が生成され、クエリツリーからリレーション情報が抽出されます。 2. **探索**: RelOptInfo を中心に、結合順序やインデックススキャンなどの Path が生成されます。 3. **コスト評価**: Path の startup_cost や total_cost が計算され、cheapest_total_path 等が選定されます。 4. **変換**: 最終的に選択された Path ツリーが create_plan() を通じて plannodes.h の Plan ツリーへ変換されます。

直接依存

  • access/sdir.h: スキャン方向の定義。
  • lib/stringinfo.h: 文字列操作ユーティリティ。
  • nodes/params.h: パラメータ管理。
  • nodes/parsenodes.h: 解析ツリーの定義。
  • storage/block.h: ブロック番号の定義。

関連コード

  • optimizer/path/: Path ノードを生成・操作するロジック。
  • optimizer/plan/: Path から Plan を生成するロジック。
  • optimizer/util/: RelOptInfo や RestrictInfo を操作するユーティリティ。

読者向けメモ

  • **メモリ所有権**: Path や RelOptInfo はプランナーのメモリコンテキスト（PlannerInfo.planner_cxt）で管理されます。これらは copyObject() によるコピーがサポートされていないものが多く、ポインタの扱いに注意が必要です。
  • **設計上の注意**: RelOptInfo や Path は非常に巨大な構造体であり、デバッグ時の nodeToString() 出力は非常に長大になります。一部のフィールドには pg_node_attr で read_write_ignore や write_only 属性が付与されており、ダンプ出力が抑制されています。
  • **未実装・浅い領域**: CustomPath などの拡張用構造体は、外部モジュールが独自に拡張できるよう設計されており、コア側のロジックはフック経由で呼び出されます。
