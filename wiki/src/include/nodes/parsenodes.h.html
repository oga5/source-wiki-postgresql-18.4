src/include/nodes/parsenodes.h

概要

parsenodes.h は、PostgreSQL のパーサーが SQL 文を解析した結果生成する「パースツリー」および「クエリツリー」のデータ構造を定義するヘッダーファイルです。PostgreSQL のクエリ処理パイプラインにおける最初のステップ（解析フェーズ）で生成されるノードの仕様を網羅しています。

モジュール内での役割

src/include/nodes モジュールにおいて、SQL 文の構文解析結果を保持する役割を担います。パーサーが生成した生のパースツリー（RawStmt）から、解析後のクエリツリー（Query）まで、SQL の文法要素をメモリ上の構造体として表現します。

責務

  • **クエリツリーの定義**: 解析後のクエリ（Query）の構造定義。
  • **文法要素の構造化**: SELECT, INSERT, UPDATE, DELETE, MERGE などの DML 文、および CREATE, ALTER, DROP などの DDL 文のノード定義。
  • **解析補助構造**: 式（A_Expr）、定数（A_Const）、関数呼び出し（FuncCall）、範囲テーブル（RangeTblEntry）など、クエリを構成する部品の定義。
  • **エラー位置情報の管理**: 各ノードに location フィールドを持たせ、SQL ソースコード上の位置を追跡可能にする。

重要なシンボル

クエリツリーのルート

  • Query: 解析後のクエリツリーのルート。commandType（SELECT/INSERT 等）や、rtable（範囲テーブル）、targetList（出力列）などを保持。

DML 文のノード

  • SelectStmt, InsertStmt, UpdateStmt, DeleteStmt, MergeStmt: 各 SQL 文の解析結果を保持する構造体。

範囲テーブル（Range Table）

  • RangeTblEntry: FROM 句に現れるテーブル、サブクエリ、結合、関数などを表現。
  • RTEKind: 範囲テーブルエントリの種類（RTE_RELATION, RTE_SUBQUERY, RTE_JOIN 等）。

式と定数

  • A_Expr: 演算子式（A_Expr_Kind で種類を判別）。
  • A_Const: リテラル定数。ValUnion を通じて整数、浮動小数点、文字列等を保持。
  • FuncCall: 関数呼び出し。agg_order や agg_filter など集約関数特有のフィールドも含む。

DDL 文のノード

  • CreateStmt, AlterTableStmt, DropStmt, IndexStmt: 各種 DDL 操作を表現する構造体。

制御フローとデータフロー

1. **解析**: パーサーが SQL を読み込み、RawStmt をルートとするパースツリーを構築します。 2. **変換**: parse_analyze() がパースツリーを走査し、Query 構造体へと変換します。この過程で RangeTblEntry が構築され、式が A_Expr から Expr（primnodes.h で定義）へ変換されます。 3. **計画**: プランナーが Query を受け取り、Plan ツリー（plannodes.h）を生成します。この際、Query 内の rtable や targetList が参照されます。

直接依存

  • nodes/bitmapset.h: ビットマップ集合の利用。
  • nodes/lockoptions.h: ロックモードの定義。
  • nodes/primnodes.h: 式の基本構造（Expr 等）。
  • nodes/value.h: 定数ノードの基盤。
  • partitioning/partdefs.h: パーティション関連の定義。

関連コード

  • src/backend/parser/: 本ファイルで定義されたノードを生成・変換するパーサーの実装。
  • src/backend/optimizer/: Query ツリーを Plan ツリーに変換するプランナー。
  • src/include/nodes/primnodes.h: 解析後の式ノードの定義。
  • src/include/nodes/plannodes.h: 実行計画ノードの定義。

読者向けメモ

  • **メモリ管理**: すべてのノードは NodeTag を先頭に持ち、palloc を使用したメモリコンテキスト管理を前提としています。
  • **位置情報**: ParseLoc はソース文字列のバイトオフセットです。エラー報告に不可欠ですが、解析が進むと無効になる場合があります。
  • **shallow な領域**: 本ファイルは非常に巨大であるため、特定の DDL 文（CreateStmt 等）の詳細は、関連する commands/ 配下のソースコードと併せて読むことを推奨します。
  • **jumble 属性**: pg_node_attr(query_jumble_ignore) などの属性は、pg_stat_statements 等でクエリの正規化（ハッシュ化）を行う際に無視すべきフィールドを指定しています。
