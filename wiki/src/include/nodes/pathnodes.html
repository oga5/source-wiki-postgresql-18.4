src/include/nodes/pathnodes.h

概要

src/include/nodes/pathnodes.h は、PostgreSQL のクエリプランナがクエリ最適化の過程で使用する内部データ構造を定義するヘッダーファイルです。特に、プランナが探索する「パス（Path）」や、リレーションに関する最適化情報（RelOptInfo）の定義が中心となります。

モジュール内での役割

src/include/nodes モジュールにおいて、本ファイルはプランナの「最適化フェーズ」を支える中核的なデータ構造を提供します。parsenodes.h がクエリの構文木を定義するのに対し、本ファイルはそれらを基に生成される実行計画の候補（Path）や、コスト見積もりに必要なリレーションの統計情報を保持します。

責務

  • **プランニング状態の保持**: PlannerInfo および PlannerGlobal を通じて、クエリ全体の最適化状態を管理する。
  • **リレーション情報の定義**: RelOptInfo を通じて、ベースリレーションや結合リレーションのコスト、行数、インデックス情報などを保持する。
  • **パスの定義**: Path 構造体およびその派生型（IndexPath, NestPath, HashPath など）を定義し、スキャンや結合のアルゴリズムを表現する。
  • **最適化の制約管理**: EquivalenceClass や PathKey を用いて、等価性やソート順序に関する情報を管理し、プランナが効率的な計画を選択できるようにする。

重要なシンボル

  • PlannerInfo: プランナのルート構造体。現在のクエリレベルの最適化状態を保持。
  • RelOptInfo: 特定のリレーション（ベースまたは結合）に関する最適化情報。
  • Path: 実行計画の候補を表す抽象的な構造体。
  • RestrictInfo: 制限句（WHERE/JOIN条件）の最適化用メタデータ。
  • EquivalenceClass: 等価な式グループを管理し、マージ結合やソート順序の最適化に使用。

制御フローとデータフロー

1. **初期化**: PlannerInfo が作成され、クエリの構文木（Query）がセットされる。 2. **リレーション探索**: RelOptInfo がベースリレーションごとに作成され、simple_rel_array に格納される。 3. **パス生成**: プランナは RelOptInfo に対して Path を生成し、pathlist に追加する。 4. **結合探索**: 動的計画法や遺伝的アルゴリズム（GEQO）により、JoinPath が生成され、結合リレーションの RelOptInfo が構築される。 5. **最終化**: 最適な Path が選択され、create_plan() を通じて実行可能な Plan ノードツリーへ変換される。

直接依存

  • access/sdir.h: スキャン方向（ScanDirection）の定義。

関連コード

  • src/backend/optimizer/plan/: create_plan() によるパスからプランへの変換。
  • src/backend/optimizer/util/: RelOptInfo や EquivalenceClass の操作関数。

読者向けメモ

  • 本ファイルで定義される構造体は、プランナの再帰的な処理で頻繁に参照されます。特に RelOptInfo と Path の間には循環参照に近い複雑なリンクがあるため、copyObject() 等の汎用関数によるコピーはサポートされていません。
  • pg_node_attr マクロは、PostgreSQL のノードサポートコード生成器（gen_node_support.pl）が使用する属性定義です。これらはノードのシリアライズやデバッグ出力の挙動を制御します。
  • Relids は Bitmapset の別名であり、リレーションの集合を効率的に扱うために多用されます。ビット演算による集合操作に慣れておくことが重要です。
