src/include/nodes/supportnodes.h

概要

本ファイルは、PostgreSQLの「プランナーサポート関数（Planner Support Functions）」のためのAPI定義を提供します。これらは、特定のSQL関数に対して追加のメタデータや最適化のヒントをシステムに提供するための仕組みです。名前は「プランナー」ですが、実行時（Executor）にも利用されます。

モジュール内での役割

src/include/nodes モジュールの一部として、クエリ最適化および実行フェーズで利用されるノード構造体を定義します。プランナーが特定の関数呼び出しに対して「より賢い」判断を下すためのインターフェースを規定する役割を担います。

責務

  • 関数呼び出しの最適化（定数畳み込み等）を支援する構造体の定義。
  • プランナーがコスト見積もりや選択率（Selectivity）を算出する際の補助情報の提供。
  • インデックス利用の可否判断や、ウィンドウ関数の単調性判定など、高度な最適化のためのデータ構造の提供。
  • インプレース（In-place）更新の安全性判定など、メモリ効率化のためのインターフェース定義。

重要なシンボル

  • SupportRequestSimplify: プランナーによる定数畳み込み時に、関数呼び出しを簡略化するためのリクエスト。
  • SupportRequestSelectivity: 関数が返すブール値の選択率見積もりを要求する構造体。
  • SupportRequestCost: 関数の実行コスト（起動コストおよびタプルあたりのコスト）を算出するためのリクエスト。
  • SupportRequestIndexCondition: 関数呼び出しをインデックス検索可能な条件式に変換するためのリクエスト。
  • SupportRequestWFuncMonotonic: ウィンドウ関数の単調性を判定するためのリクエスト。

追加の索引済みシンボル

  • SupportRequestModifyInPlace
  • SupportRequestOptimizeWindowClause
  • SupportRequestRows

制御フローとデータフロー

1. **リクエストの発生**: プランナーまたは実行エンジンが、特定の関数に対して最適化が必要と判断した際、対応する SupportRequest 構造体を生成します。 2. **サポート関数の呼び出し**: 対象関数に紐付けられたサポート関数（SQLレベルで prosupport として登録されたC関数）が、この構造体を引数として呼び出されます。 3. **結果の返却**: サポート関数は、最適化が可能であれば情報を構造体に格納してポインタを返し、不可能であれば NULL を返します。 4. **適用**: 呼び出し元は返された結果に基づき、プランの書き換えやコスト見積もりの調整を行います。

直接依存

  • nodes/plannodes.h: プランノードの定義を利用するためにインクルードされています。

関連コード

  • src/backend/optimizer/util/clauses.c: サポート関数を呼び出し、結果を処理する主要なロジックが含まれます。
  • src/include/nodes/primnodes.h: FuncExpr や OpExpr など、サポート対象となるノードの定義が参照されます。
  • src/include/catalog/pg_proc.h: サポート関数が prosupport 属性として各関数に登録される仕組みに関連します。

読者向けメモ

  • **命名の注意**: 「プランナーサポート関数」という名称ですが、実行時にも呼び出されるため、プランナー専用のロジックと決めつけないよう注意してください。
  • **NULLの扱い**: サポート関数がリクエストを処理できない場合は、エラーを発生させるのではなく NULL を返すことがプロトコルです。これにより、将来的なリクエストタイプの追加に対して後方互換性を保っています。
  • **メモリ管理**: SupportRequest 構造体自体は一時的なものであり、サポート関数内で永続的なメモリ確保を行う必要はありません。
