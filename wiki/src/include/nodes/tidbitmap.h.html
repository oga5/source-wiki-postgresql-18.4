src/include/nodes/tidbitmap.h

概要

tidbitmap.h は、PostgreSQL における TID（Tuple Identifier）の集合を効率的に管理するためのビットマップデータ構造を定義するヘッダーファイルです。主にインデックススキャンやビットマップヒープスキャンにおいて、対象となるタプルの位置情報をメモリ上に保持するために使用されます。

モジュール内での役割

src/include/nodes モジュールの一部として、実行計画や実行時のノード管理を補完します。Bitmapset が整数集合の管理に特化しているのに対し、本モジュールは ItemPointer（ブロック番号とオフセット番号）の管理に特化しており、メモリ消費を抑えるための「損失付き（lossy）」格納方式をサポートしています。

責務

  • **TID 集合の保持**: 大量の TID をメモリ効率よく格納するデータ構造の提供。
  • **損失付き格納（Lossy Storage）**: メモリ不足時に、個別のオフセット情報を破棄し「ページ単位」の管理へ切り替える機能の提供。
  • **イテレータの抽象化**: プライベート（ローカル）および共有（パラレルクエリ用）イテレータを統一的に扱うための API 提供。
  • **集合演算**: 複数のビットマップ間での union や intersect 操作の定義。

重要なシンボル

  • TIDBitmap: TID 集合を保持する不透明な構造体。詳細は tidbitmap.c で定義。
  • TBMIterateResult: イテレータが返す結果構造体。ブロック番号、損失フラグ、再チェック要否を含む。
  • TBMIterator: プライベートと共有の両方をラップする統合イテレータ構造体。
  • tbm_add_tuples: ビットマップへの TID 追加用関数。
  • tbm_iterate: ビットマップを走査するための汎用関数。

追加の索引済みシンボル

  • TBMPrivateIterator
  • TBMSharedIterator
  • tbm_exhausted

制御フローとデータフロー

1. **初期化**: tbm_create でメモリ制限を指定してビットマップを生成。 2. **構築**: tbm_add_tuples や tbm_add_page を用いて TID を追加。メモリ制限を超えると自動的に損失付きモードへ移行。 3. **走査**: tbm_begin_iterate でイテレータを初期化し、tbm_iterate をループ内で呼び出して TBMIterateResult を取得。 4. **終了**: tbm_end_iterate でリソースを解放。

直接依存

  • access/htup_details.h: MaxHeapTuplesPerPage 定数の参照。
  • storage/itemptr.h: ItemPointer および BlockNumber 型の定義。
  • utils/dsa.h: 動的共有メモリ（DSA）領域でのビットマップ管理用。

関連コード

  • src/backend/nodes/tidbitmap.c: 本ヘッダーで宣言された関数の実装および内部データ構造（ハッシュテーブル等）の定義。
  • src/include/nodes/bitmapset.h: 類似のビットセット操作ライブラリ。
  • src/backend/executor/nodeBitmapHeapscan.c: 本モジュールを利用してヒープを走査する実行ノード。

読者向けメモ

  • 本モジュールは「損失付き」状態を許容するため、イテレータから返される結果に対しては、常に recheck フラグを確認し、必要に応じてヒープ上のタプルを再検証する必要があります。
  • TIDBitmap 構造体の内部は隠蔽されているため、直接メンバにアクセスせず、提供されている API を通じて操作してください。
  • パラレルクエリで使用する場合は、dsa_area を介した共有イテレータのライフサイクル管理に注意が必要です。
