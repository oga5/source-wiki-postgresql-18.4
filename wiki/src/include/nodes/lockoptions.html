src/include/nodes/lockoptions.h

概要

本ファイルは、PostgreSQLのクエリ実行およびプランニングにおいて、行レベルのロック（SELECT FOR UPDATE 等）に関連する列挙型定義を集約したヘッダーファイルです。

モジュール内での役割

src/include/nodes ディレクトリの一部として、パーサーやプランナーがロック要件を表現するために使用する共通の型定義を提供します。主に parsenodes.h や plannodes.h で定義されるノード構造体から参照されます。

責務

  • ロックの強度（LockClauseStrength）の定義。
  • ロック競合時の待機ポリシー（LockWaitPolicy）の定義。
  • タプルに対するロックモード（LockTupleMode）の定義。
  • これらの列挙値の順序を規定し、複数のロック指定が競合した際の優先順位付けをサポートする。

重要なシンボル

  • LockClauseStrength: FOR UPDATE などの句の強度。値が大きいほど優先度が高い。
  • LockWaitPolicy: NOWAIT や SKIP LOCKED を制御するポリシー。
  • LockTupleMode: 内部的なタプルロックのモード。LockClauseStrength と密接に関連する。

制御フローとデータフロー

本ファイル自体はデータ構造の定義のみを提供します。

  • **データフロー**: パーサーが SQL 文からロック指定を解析し、これらの列挙型を ParseState や Query 構造体に格納します。その後、プランナーが applyLockingClause 関数を通じて、複数のロック指定を統合する際にこれらの列挙値の順序（数値の大小）を利用して最終的なロック要件を決定します。

直接依存

  • なし（純粋なヘッダーファイルであり、他のヘッダーへの依存はありません）。

関連コード

  • src/include/nodes/parsenodes.h: RowMarkClause 構造体などで本ファイルの定義を利用。
  • src/backend/parser/parse_clause.c: applyLockingClause 関数が定義されており、ここで列挙値の優先順位判定が行われます。
  • src/include/nodes/plannodes.h: 実行計画における PlanRowMark などで利用されます。

読者向けメモ

  • 列挙型の順序は単なるラベルではなく、applyLockingClause における優先順位判定のロジックに直結しています。新しいロックモードを追加する際は、既存の順序が意図した優先順位を維持しているか確認してください。
  • LockClauseStrength は SQL レベルの指定を、LockTupleMode はより低レイヤーのタプルロック処理を表現しているという違いに注意してください。
