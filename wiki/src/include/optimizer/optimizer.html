src/include/optimizer/optimizer.h

概要

optimizer.h は、PostgreSQL のクエリプランナ（オプティマイザ）が提供する外部 API を定義するヘッダーファイルです。プランナ内部の複雑な実装詳細を隠蔽し、プランナ以外のモジュールがクエリ最適化機能を利用するための主要なインターフェースとして機能します。

モジュール内での役割

本ファイルは src/include/optimizer/ ディレクトリにおける「公開ゲートウェイ」です。プランナ内部の構造体（PlannerInfo 等）を不透明な型（opaque typedef）として扱うことで、プランナ内部のヘッダーファイルへの直接依存を最小限に抑え、モジュール間の結合度を低減させています。

責務

  • プランナの主要なエントリポイント（planner() 関数など）の宣言。
  • コスト見積もり、式評価、述語テスト、変数抽出など、プランナが提供するユーティリティ関数の公開。
  • プランナの動作を制御するグローバルなコストパラメータ（GUC）の宣言。
  • プランナ内部で使用される主要なデータ構造の型定義の提供。

重要なシンボル

  • planner(): クエリツリーをプランツリーに変換するプランナのメイン関数。
  • PlannerInfo: プランニングプロセス全体で保持される状態を管理する構造体。
  • clause_selectivity(): 特定の句（clause）の選択率を計算する関数群。
  • eval_const_expressions(): 式内の定数部分を評価・簡略化する関数。
  • pull_var_clause(): 式から変数ノードを抽出するユーティリティ。

追加の索引済みシンボル

  • DebugParallelMode
  • IndexOptInfo
  • SpecialJoinInfo

制御フローとデータフロー

本ファイル自体は関数定義を含まないヘッダーですが、以下のデータフローをサポートします。 1. **プランニング開始**: planner() が Query を受け取り、PlannerInfo を初期化して最適化プロセスを開始します。 2. **コスト見積もり**: clause_selectivity 等が PlannerInfo を参照し、統計情報に基づいてコストを算出します。 3. **式最適化**: eval_const_expressions や canonicalize_qual がクエリツリーを走査し、実行可能な形へ変換します。 4. **変数管理**: pull_var_clause 等が式ツリーを解析し、必要な変数情報を抽出してプラン生成に利用します。

直接依存

  • nodes/parsenodes.h: Query 構造体などの定義を参照するために必要です。

関連コード

  • src/include/optimizer/pathnodes.h: PlannerInfo 等の完全な定義が含まれており、プランナ内部の実装ではこちらが使用されます。
  • src/backend/optimizer/plan/planner.c: planner() 関数の実装本体。
  • src/backend/optimizer/path/costsize.c: コスト計算ロジックの実装。

読者向けメモ

  • **不透明な型**: PlannerInfo 等を扱う際は、ポインタとしてのみ扱い、内部メンバに直接アクセスしないことが推奨されます。
  • **依存関係の回避**: プランナ以外のモジュールを開発する際は、optimizer/ 以下の他のヘッダーではなく、まずはこのファイルをインクルードしてください。
  • **GUC パラメータ**: seq_page_cost などのコストパラメータは PGDLLIMPORT で宣言されており、プランナの挙動を調整する際に重要となります。
