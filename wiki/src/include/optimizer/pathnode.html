src/include/optimizer/pathnode.h

概要

pathnode.h は、PostgreSQL オプティマイザがクエリ実行計画を構築する過程で利用する「パス（Path）」ノードの生成および操作に関するインターフェースを定義するヘッダーファイルです。

モジュール内での役割

オプティマイザの主要な構成要素である pathnode.c および relnode.c の公開 API を提供します。クエリの探索空間において、特定の関係（Relation）に対する実行戦略（スキャン、結合、集約など）を表現する Path 構造体の生成と管理を担います。

責務

  • **Path ノードの生成**: スキャン（SeqScan, IndexScan 等）、結合（NestLoop, HashJoin 等）、および各種演算（Sort, Agg, Limit 等）に対応する Path 構造体のファクトリ関数を提供します。
  • **パスの管理**: RelOptInfo にパスを追加する際のコスト評価や、最適パスの選定（set_cheapest）を管理します。
  • **リレーション情報の管理**: RelOptInfo の構築、検索、およびパラメータ化されたパス（Parameterized Path）の管理機能を提供します。

重要なシンボル

  • Path: クエリ実行計画の候補を表現する基本構造体。
  • RelOptInfo: 特定のクエリ範囲（ベースリレーションや結合リレーション）に関する情報を保持する構造体。
  • add_path() / add_partial_path(): 新しいパスを RelOptInfo に登録し、既存のパスと比較して不要であれば破棄する関数。
  • create_*_path() 系関数: 各種スキャンや結合アルゴリズムに対応するパスノードを生成する関数群。
  • compare_path_costs(): 2つのパスのコストを比較し、どちらが優れているかを判定するユーティリティ。

制御フローとデータフロー

1. **パス生成**: プランナが各リレーションに対して create_*_path 関数を呼び出し、候補となる Path ノードを生成します。 2. **パス登録**: 生成されたパスは add_path に渡されます。ここでコストやパスキー（PathKeys）が評価され、そのリレーションにとって「最良」なパスが保持されます。 3. **リレーション構築**: relnode.c の関数群（build_join_rel 等）を通じて、結合リレーションのメタデータが構築され、パスの探索が進行します。

直接依存

  • nodes/bitmapset.h: パスが依存するパラメータやリレーションの集合を扱うために使用。
  • nodes/pathnodes.h: Path, RelOptInfo, PlannerInfo などの主要なデータ構造定義。

関連コード

  • src/backend/optimizer/util/pathnode.c: 本ヘッダーで宣言されたパス生成関数の実装。
  • src/backend/optimizer/util/relnode.c: リレーション情報の管理と RelOptInfo 操作の実装。
  • src/include/nodes/pathnodes.h: Path 構造体自体の詳細な定義。

読者向けメモ

  • 本ファイルは非常に多くの関数宣言を含みますが、基本的には「特定の操作（例：HashJoin）に対応する Path を作る」という命名規則に従っています。
  • パス生成関数は、多くの場合 PlannerInfo を引数に取り、クエリ全体のコンテキストを参照します。
  • add_path は単なるリストへの追加ではなく、コストベースの最適化（不要なパスの剪定）を行う重要な関数であることに注意してください。
