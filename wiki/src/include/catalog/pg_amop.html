src/include/catalog/pg_amop.h

概要

pg_amop は、インデックス演算子ファミリー（Operator Family）および演算子クラス（Operator Class）に関連付けられた演算子を管理するシステムカタログです。インデックスが特定の演算子をどのように利用するか（検索用か、順序付け用か）を定義します。

モジュール内での役割

src/include/catalog 内において、インデックスの振る舞いを決定づけるメタデータの一部を担います。pg_am（アクセス手法）、pg_opfamily（演算子ファミリー）、pg_operator（演算子定義）を繋ぐハブとして機能し、クエリプランナがインデックスを利用可能か判断するための基礎情報を提供します。

責務

  • インデックス演算子ファミリーと演算子のマッピングの保持。
  • 演算子の目的（検索 s または順序付け o）の定義。
  • インデックスアクセス手法（AM）と演算子の互換性の明示。
  • 検索効率化のための非正規化データ（amopmethod）の保持。

重要なシンボル

  • FormData_pg_amop: カタログの行構造体。
  • AMOP_SEARCH / AMOP_ORDER: amoppurpose 列で使用される、演算子の用途を示す定数。
  • pg_amop_fam_strat_index: (amopfamily, lefttype, righttype, strategy) をキーとする主検索用インデックス。
  • pg_amop_opr_fam_index: 演算子からファミリーを逆引きするためのインデックス。

制御フローとデータフロー

本ファイルはデータ定義（BKI）であり、直接的な実行フローは持ちません。 1. **定義**: CATALOG マクロにより、PostgreSQL のブートストラップ時にシステムカタログとして生成されます。 2. **検索**: プランナやインデックス作成プロセスは、syscache（AMOPSTRATEGY または AMOPOPID）を介してこのカタログを検索し、特定の演算子がインデックスで利用可能かを確認します。 3. **更新**: CREATE OPERATOR CLASS や CREATE OPERATOR FAMILY コマンド実行時に、このテーブルへエントリが追加されます。

読者向けメモ

  • **非正規化**: amopmethod は pg_opfamily からのコピーです。これはカタログ検索のパフォーマンスを向上させるための意図的な設計です。
  • **一意性**: pg_amop_opr_fam_index により、同一ファミリー内で同じ演算子が複数の目的（検索と順序付け）で定義されることは許容されますが、同一目的で重複することは禁止されています。
  • **読み方**: BKI_LOOKUP などの注釈は、Catalog.pm がスキーマを生成する際に使用されます。C言語の構造体定義としてだけでなく、データベーススキーマの設計図として読んでください。
