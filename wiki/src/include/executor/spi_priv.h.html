src/include/executor/spi_priv.h

概要

本ファイルは、PostgreSQL のサーバープログラミングインターフェース（SPI）の内部的なデータ構造と型定義を保持するプライベートヘッダーです。SPI は、C 言語で記述された関数やトリガーから SQL コマンドを実行するための仕組みであり、本ファイルはその実行状態や計画の管理に必要な構造体を定義しています。

モジュール内での役割

src/include/executor/spi.h が公開 API を提供するのに対し、本ファイルは SPI の実装層（src/backend/executor/spi.c 等）が使用する内部的な状態管理用ヘッダーです。SPI のネストされた呼び出しや、メモリコンテキストの管理、計画のキャッシュ状態を追跡するために利用されます。

責務

  • **実行コンテキストの保持**: _SPI_connection 構造体を通じた、SPI 呼び出しごとの実行状態（処理行数、タプルテーブル、メモリコンテキスト等）の管理。
  • **計画のライフサイクル管理**: _SPI_plan 構造体による、保存済み（saved）、未保存（unsaved）、一時的（temporary）な計画の区別と、それに関連するメモリコンテキストの制御。
  • **ネスト管理**: SPI 呼び出しが再帰的またはネストして行われる際の、上位レベルの実行状態の退避と復元。

重要なシンボル

  • _SPI_connection: SPI 呼び出しの現在の状態を保持する構造体。ネストされた呼び出しのスタック管理に使用されます。
  • _SPI_plan: 実行計画を表現する構造体。plancache_list を通じて実際のプランキャッシュと連携します。
  • _SPI_PLAN_MAGIC: _SPI_plan 構造体の整合性を検証するためのマジックナンバー。
  • plancxt: _SPI_plan が使用するメモリコンテキスト。計画の生存期間を決定します。

制御フローとデータフロー

SPI 呼び出しが開始されると、_SPI_connection が作成され、現在のメモリコンテキストやサブトランザクション ID が記録されます。実行計画（_SPI_plan）は、その性質（一時的か永続的か）に応じて適切なメモリコンテキストに配置されます。plancache.c との連携により、計画の無効化イベントが発生した際の再計画やキャッシュの更新が行われます。

直接依存

  • executor/spi.h: 公開されている SPI のインターフェース定義。
  • utils/queryenvironment.h: クエリ実行環境の定義。

関連コード

  • src/backend/executor/spi.c: 本ヘッダーで定義された構造体を利用して SPI の主要機能（SPI_connect, SPI_execute 等）を実装しているファイル。
  • src/backend/utils/cache/plancache.c: _SPI_plan が保持する CachedPlanSource の管理を行うモジュール。

読者向けメモ

  • 本ファイルで定義されている構造体は _（アンダースコア）で始まっており、外部モジュールからの直接的な利用は想定されていません。
  • _SPI_plan のメモリ管理は複雑です。特に plancxt が NULL の場合は「一時的な計画」として扱われ、関数終了時に自動的に解放される設計になっています。
  • SPI のネストレベルを理解するには、_SPI_connection 内の outer_ で始まるメンバ変数がどのようにスタックとして機能しているかを追うのが近道です。
