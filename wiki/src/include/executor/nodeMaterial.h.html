src/include/executor/nodeMaterial.h

概要

nodeMaterial.h は、PostgreSQLのクエリ実行エンジンにおける「マテリアライズ（Materialization）」ノードのインターフェースを定義するヘッダーファイルです。このノードは、サブプランから取得したタプルを一時的にメモリまたはディスク上の Tuplestore に保存し、再スキャンや逆方向スキャンを可能にします。

モジュール内での役割

src/include/executor モジュールの一部として、実行計画の各ノードタイプを定義する一連の node*.h ファイル群の一つです。主に nodeMaterial.c で実装されるマテリアライズ処理の公開 API を提供します。

責務

  • マテリアライズノードの初期化および終了処理の宣言。
  • スキャン位置のマーク（Mark）と復元（Restore）機能のインターフェース提供。
  • ノードの再スキャン（ReScan）操作の定義。

重要なシンボル

  • MaterialState: マテリアライズ処理の実行時状態を保持する構造体。
  • ExecInitMaterial: ノードの初期化を行い、Tuplestore の準備やサブプランの初期化を担当。
  • ExecMaterialMarkPos / ExecMaterialRestrPos: スキャン位置の保存と復元を行うための関数。
  • ExecReScanMaterial: マテリアライズされたデータを保持したまま、スキャンを先頭からやり直すための関数。

制御フローとデータフロー

本ファイルはインターフェース定義のみですが、実際のフローは以下の通りです： 1. **初期化**: ExecInitMaterial が呼ばれ、MaterialState が構築されます。 2. **データ蓄積**: ExecMaterial（nodeMaterial.c）がサブプランからタプルを順次取得し、Tuplestore に格納します。 3. **再利用**: 2回目以降のアクセスや逆方向スキャンが必要な場合、Tuplestore から直接タプルが読み出されます。 4. **終了**: ExecEndMaterial により Tuplestore が解放されます。

直接依存

  • nodes/execnodes.h: MaterialState や Material ノード構造体、EState などの実行時ノード定義に依存しています。

関連コード

  • src/backend/executor/nodeMaterial.c: 本ヘッダーで宣言された関数の実装が含まれます。
  • src/include/executor/tuptable.h: タプル操作の基盤となる TupleTableSlot 等が定義されています。
  • src/include/executor/executor.h: 実行エンジン全体の共通インターフェースが定義されています。

読者向けメモ

  • このノードは、サブプランの結果を一度だけ評価し、その結果をキャッシュしたい場合（例：ソート済みデータの再利用や、サブクエリの複数回参照）に使用されます。
  • Tuplestore の管理は nodeMaterial.c 内で行われます。メモリ消費量（work_mem）が不足した場合は自動的にディスクへ溢れる（spill）設計になっています。
  • ExecMaterialMarkPos 等の関数は、SQLの DECLARE CURSOR や特定の結合アルゴリズムで必要となる「位置の記憶」をサポートするために存在します。
