src/include/libpq/sasl.h

概要

本ファイルは、PostgreSQLバックエンドにおけるSASL（Simple Authentication and Security Layer）認証メカニズムを定義するためのインターフェースを提供します。認証プロセスの抽象化を行い、SCRAMなどの具体的な認証方式をプラグインとして追加可能にするための構造体と定数を定義しています。

モジュール内での役割

src/include/libpq モジュールの一部として、認証サブシステム（主に auth.c）と個別の認証メカニズム実装との間の契約を定義します。フロントエンド側の対応する実装（fe-auth-sasl.h）と対になるバックエンド側の基盤です。

責務

  • SASL認証のステータスコード（成功・継続・失敗）の定義。
  • バックエンド認証メカニズムが実装すべきコールバック関数群（pg_be_sasl_mech）の規定。
  • 認証プロセスにおけるメッセージ交換の最大長制限の管理。
  • CheckSASLAuth 関数を通じた、認証ロジックの実行と状態管理の統合。

重要なシンボル

  • pg_be_sasl_mech: 各認証方式（SCRAM等）が実装すべきインターフェース構造体。init（初期化）、exchange（メッセージ交換）、get_mechanisms（方式リスト取得）を保持します。
  • PG_SASL_EXCHANGE_*: 認証交換の進捗を示すステータスコード。
  • CheckSASLAuth(): 認証メカニズムのコールバックを呼び出し、クライアントとの対話を駆動するメイン関数。

制御フローとデータフロー

1. **初期化**: CheckSASLAuth が呼び出されると、メカニズムの init が呼ばれ、認証セッション用の不透明な状態（void *state）が生成されます。 2. **交換**: クライアントからのメッセージを受け取るたびに exchange が呼び出されます。

  • input としてクライアントのデータが渡され、output にサーバー側の応答が格納されます。
  • 戻り値が PG_SASL_EXCHANGE_CONTINUE の間、このループが繰り返されます。

    • 3. **終了**: PG_SASL_EXCHANGE_SUCCESS または FAILURE が返されると、認証プロセスが完了します。

直接依存

  • lib/stringinfo.h: 認証メッセージのバッファ操作に使用。
  • libpq/libpq-be.h: 接続状態を保持する Port 構造体の定義に使用。

関連コード

  • src/include/libpq/auth.h: 認証フロー全体の制御。
  • src/include/libpq/scram.h: 本インターフェースを利用した具体的な認証方式の実装例。
  • src/interfaces/libpq/fe-auth-sasl.h: クライアント（libpq）側の対となるインターフェース。

読者向けメモ

  • 本ファイルはインターフェース定義のみであり、具体的な認証ロジックは含まれていません。
  • exchange コールバック内で input に埋め込み NULL が含まれる可能性があるため、inputlen を必ず参照してください。
  • ユーザー名の列挙攻撃を防ぐため、shadow_pass が NULL（ユーザーが存在しない）場合でも、認証失敗時と同様の挙動を維持するよう実装する必要があります。
