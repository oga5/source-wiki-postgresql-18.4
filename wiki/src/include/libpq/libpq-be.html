src/include/libpq/libpq-be.h

概要

PostgreSQL バックエンド（サーバー側）におけるクライアント接続管理のための主要なデータ構造とインターフェースを定義するヘッダーファイルです。クライアント認証、通信プロトコル、およびセキュリティ（SSL/TLS, GSSAPI）に関する状態保持を担います。

モジュール内での役割

libpq サブシステムにおいて、バックエンド内部でのみ使用される定義を提供します。クライアント向けに公開される pqcomm.h とは対照的に、サーバープロセス（Postmaster やバックエンドプロセス）が接続状態を追跡するためのプライベートなインターフェースとして機能します。

責務

  • クライアント接続状態（Port 構造体）の定義と管理。
  • 認証プロセス（SCRAM, GSSAPI, SSL）に必要な状態の保持。
  • ネットワークソケットの属性（TCP keepalive 等）の抽象化。
  • SSL/TLS および GSSAPI 通信のためのバックエンド側関数プロトタイプの提供。

重要なシンボル

  • Port: 接続中のクライアントに関する全情報を保持する中心的な構造体。MyProcPort グローバル変数を通じてアクセスされます。
  • ClientConnectionInfo: 並列ワーカーへ引き継ぐべき認証情報を保持する構造体。
  • be_tls_* 系関数: SSL/TLS 通信の抽象化レイヤー。実装は be-secure-openssl.c 等に委譲されます。
  • pg_gssinfo: GSSAPI 認証および暗号化の状態を保持する構造体。

追加の索引済みシンボル

  • ClientSocket
  • SSL_CTX
  • gss_buffer_desc

制御フローとデータフロー

1. **接続確立**: Postmaster が接続を受け入れると、Port 構造体が初期化され、ソケット情報やアドレスが格納されます。 2. **認証**: Port 内の hba や認証用フィールド（SCRAM キー等）を使用して認証が行われます。 3. **通信**: 認証後、ssl_in_use や gss 構造体のフラグに基づき、be_tls_read/write や be_gssapi_read/write を介してデータが送受信されます。 4. **並列処理**: ClientConnectionInfo を介して、認証済みのアイデンティティ情報が並列ワーカープロセスへシリアライズ・伝播されます。

直接依存

  • libpq/pqcomm.h: 基本的な通信プロトコル定義。
  • libpq/hba.h: ホストベース認証の設定構造体。
  • common/scram-common.h: SCRAM 認証用定数。
  • openssl/ssl.h / openssl/err.h: SSL/TLS 実装用（USE_OPENSSL 定義時）。
  • libpq/pg-gssapi.h: GSSAPI 関連の型定義。
  • datatype/timestamp.h
  • netinet/tcp.h
  • ntsecapi.h
  • security.h
  • sys/time.h

関連コード

  • src/backend/libpq/be-secure-openssl.c: be_tls_* 関数の具体的な実装。
  • src/backend/libpq/auth.c: Port 構造体を利用した認証ロジックの実装。
  • src/backend/postmaster/postmaster.c: Port 構造体の初期化と管理の起点。

読者向けメモ

  • **メモリ管理**: Port 構造体およびその配下のデータは、すべて TopMemoryContext で割り当てられる必要があります。
  • **ABI 互換性**: GSSAPI や SSL が無効なビルドであっても、構造体のオフセットを維持するために void * 等のプレースホルダーが使用されています。拡張機能開発者は、これらの内部構造を直接操作する際に注意が必要です。
  • **raw_buf**: SSL ハンドシェイク時に「読み込みすぎた」データを一時保持するためのハック的なバッファです。SSL 接続確立フローを追う際に重要なポイントとなります。
