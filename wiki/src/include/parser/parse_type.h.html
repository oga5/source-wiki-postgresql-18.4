src/include/parser/parse_type.h

概要

parse_type.h は、PostgreSQLのパーサーがSQL文中の型名（TypeName）を解決し、システムカタログ（pg_type）から型情報を取得するためのインターフェースを提供します。

モジュール内での役割

パーサーサブシステムにおいて、構文解析中に現れる型名文字列を、実行時に利用可能な内部表現（OIDやHeapTuple）へ変換する中核的な役割を担います。parse_node.h で定義されるノード構造体と、システムカタログのメタデータを橋渡しします。

責務

  • **型解決**: TypeName 構造体から pg_type のエントリ（Type）を検索する。
  • **型情報の抽出**: OID、型長、値渡しフラグ、照合順序（Collation）などの属性を Type オブジェクトから取得する。
  • **型修飾子（typmod）の処理**: 型名に伴う精度や長さの指定（例: varchar(10)）を解析・検証する。
  • **複雑な型の判定**: 複合型やドメインを識別するためのユーティリティ提供。

重要なシンボル

  • Type: HeapTuple への typedef。pg_type システムカタログの行を指します。
  • LookupTypeName() / LookupTypeNameExtended(): 型名から Type を検索する主要関数。
  • typenameTypeId(): 型名から対応する Oid を直接取得する簡便な関数。
  • ISCOMPLEX(typeid): 型が複合型（または複合型上のドメイン）であるかを判定するマクロ。

制御フローとデータフロー

1. **入力**: パーサーが構文解析中に TypeName ノードを生成します。 2. **検索**: LookupTypeName 等が pg_type システムカタログを検索し、HeapTuple を返します。 3. **変換**: 呼び出し元は、必要に応じて typenameTypeId や typeLen などのアクセサを使用して、型に関する具体的な属性値を取り出します。 4. **解放**: Type（HeapTuple）を取得した呼び出し元は、不要になった時点で ReleaseSysCache を呼び出す責任があります。

直接依存

  • access/htup.h: HeapTuple 型の定義。
  • parser/parse_node.h: ParseState や TypeName 構造体の定義。

関連コード

  • src/backend/parser/parse_type.c: 本ヘッダーで宣言された関数の実装。
  • src/include/catalog/pg_type.h: 型情報の定義元。
  • src/include/utils/lsyscache.h: システムキャッシュ経由での型属性取得に関する補助関数群。

読者向けメモ

  • Type は HeapTuple の別名であるため、メモリ管理（ReleaseSysCache）には注意が必要です。
  • LookupTypeName 系の関数は、型が見つからない場合に missing_ok フラグによってエラーを投げるか NULL を返すかを選択できます。
  • ISCOMPLEX マクロは、単なる複合型だけでなく、複合型をベースにしたドメインも対象とする点に留意してください。
