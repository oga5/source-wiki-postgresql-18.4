src/include/parser/analyze.h

概要

analyze.h は、パーサーが生成した生の構文木（RawStmt）を、クエリ実行可能な形式である Query 構造体へと変換する「解析（Parse Analysis）」フェーズの主要なインターフェースを定義しています。

モジュール内での役割

src/include/parser モジュールにおいて、構文解析（スキャナ/パーサー）と意味解析の橋渡しを担います。parser.h が構文解析の入り口であるのに対し、本ファイルは構文木に対して型チェック、名前解決、およびクエリの書き換え準備を行うための関数群を提供します。

責務

  • **解析の実行**: 生の構文木を Query 構造体に変換するメインエントリーポイントの提供。
  • **変換処理**: INSERT や UPDATE などのステートメント固有の句（TargetList や ReturningClause など）を意味的に正しい形式へ変換。
  • **ロック管理**: クエリ内のロック句（FOR UPDATE 等）の妥当性チェックと適用。
  • **フック提供**: 外部プラグインが解析終了時に介入するためのフック機構の定義。

重要なシンボル

  • parse_analyze_...: RawStmt を Query に変換する一連の関数群。パラメータの解決方法（固定か可変か）に応じて使い分けられます。
  • transformStmt: 構文木を解析し、意味的な変換を行う再帰的なコア関数。
  • post_parse_analyze_hook: 解析完了後に実行される外部プラグイン用のコールバック関数ポインタ。
  • ParseState: 解析中の状態（名前空間、パラメータ、CTE情報など）を保持するコンテキスト構造体。

追加の索引済みシンボル

  • JumbleState

制御フローとデータフロー

1. **入力**: RawStmt（パーサーの出力）が parse_analyze_* 関数に渡されます。 2. **初期化**: ParseState が作成され、クエリ環境が設定されます。 3. **変換**: transformStmt が呼び出され、構文木をトラバースしながら型解決や名前解決を行い、Query 構造体を構築します。 4. **フック**: 解析の最後に post_parse_analyze_hook が設定されていれば呼び出されます。 5. **出力**: 意味的に整合性が取れた Query 構造体が返されます。

直接依存

  • nodes/params.h: パラメータ解決のための定義。
  • nodes/queryjumble.h: クエリのフィンガープリント（Jumble）生成用。
  • parser/parse_node.h: ParseState 構造体など、解析状態管理の基盤。

関連コード

  • src/backend/parser/analyze.c: 本ヘッダーで宣言された関数の実装本体。
  • src/backend/commands/analyze.c: 名前は似ていますが、こちらは統計情報収集（ANALYZE コマンド）を担当する別モジュールです。混同に注意してください。
  • src/include/parser/parsetree.h: 構文木のノード定義。

読者向けメモ

  • 本ファイルは「統計情報の収集」ではなく「SQLの解析」を目的としています。
  • parse_analyze 系関数は、クエリの実行計画作成の前段階として必須のプロセスです。
  • ParseState は解析中に動的に更新されるため、再帰的な解析（parse_sub_analyze）を行う際は、親の ParseState とのスコープ管理に注意が必要です。
