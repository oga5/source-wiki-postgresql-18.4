src/include/port/atomics.h

概要

atomics.h は、PostgreSQL におけるプラットフォームおよびコンパイラ非依存の原子操作（Atomic Operations）およびメモリバリアを提供するための抽象化レイヤーです。マルチスレッド環境やロックレスアルゴリズムの実装において、CPU やコンパイラによる命令の並び替えを制御し、メモリの一貫性を保証します。

モジュール内での役割

src/include/port モジュールにおいて、ハードウェア固有の差異を吸収する最下層のユーティリティとして機能します。上位のロック機構（LWLock, Spinlock）や、高並列データ構造の実装を支える基盤です。

責務

  • **メモリバリアの提供**: コンパイラおよび CPU レベルでの命令順序制御（pg_compiler_barrier, pg_memory_barrier 等）。
  • **原子操作の抽象化**: 32bit/64bit 整数およびフラグに対するアトミックな読み書き、比較交換（CAS）、算術演算の提供。
  • **プラットフォーム適応**: アーキテクチャ（ARM, x86, PPC）やコンパイラ（GCC, MSVC, SunPro）に応じた最適な実装の選択。
  • **フォールバック**: ネイティブな 64bit アトミック操作が利用できない環境に対するスピンロックベースの代替実装の提供。

重要なシンボル

  • pg_memory_barrier(): CPU のメモリ順序を完全に保証するフルバリア。
  • pg_atomic_compare_exchange_u32(): CAS 操作。ロックレスアルゴリズムの核。
  • pg_atomic_uint32 / pg_atomic_uint64: アトミック操作対象の型定義。
  • pg_atomic_test_set_flag(): テスト・アンド・セット（TAS）による簡易的なロック取得。

制御フローとデータフロー

1. **初期化**: コンパイル時にアーキテクチャおよびコンパイラ固有のヘッダー（arch-*.h, generic-*.h）を読み込み、実装を決定します。 2. **抽象化**: atomics.h 内の static inline 関数が、各プラットフォームで定義された _impl 関数を呼び出します。 3. **実行**:

  • ネイティブサポートがある場合、コンパイラの組み込み関数やインラインアセンブラが直接実行されます。
  • 64bit 操作でサポートがない場合、src/backend/port/atomics.c 内のスピンロックを用いた実装へフローが転送されます。

読者向けメモ

  • **アライメント**: pg_atomic_uint32 等の操作には、適切なメモリ配置（4バイトまたは8バイト）が必須です。AssertPointerAlignment でチェックされています。
  • **利用指針**: アトミック操作は非常に低レベルで難解です。可能な限り LWLock や Spinlock などの高レベルなロック機構の使用を優先してください。
  • **バリアの強弱**: pg_read_barrier や pg_write_barrier はフルバリアよりも軽量ですが、プラットフォームによってはフルバリアと等価になる場合があります。
