src/include/access/xlog.h

概要

PostgreSQL の先行書き込みログ（WAL）管理システムの中心的なヘッダーファイルです。WAL の生成、フラッシュ、リカバリ、およびチェックポイント処理に関するインターフェースと、システム全体で共有される WAL 関連の GUC パラメータや状態定義を保持します。

モジュール内での役割

src/include/access モジュールにおいて、WAL サブシステムの「公開 API」として機能します。xlog.c（実装本体）と密接に連携し、他のバックエンドプロセスやストレージ管理コンポーネントが WAL の状態を照会したり、ログの挿入・フラッシュを要求したりするための窓口となります。

責務

  • WAL の動作モード（WalLevel）や同期手法（WalSyncMethod）の定義。
  • チェックポイントの動作を制御するフラグ（CHECKPOINT_*）の提供。
  • WAL の挿入位置（LSN）管理およびフラッシュ制御関数の宣言。
  • リカバリ状態の管理と、バックアップ開始・終了のためのセッションレベル API の提供。
  • チェックポイント統計情報（CheckpointStatsData）の構造定義。

重要なシンボル

  • XLogRecPtr: WAL 内の論理的な位置を示す 64 ビットのポインタ。
  • WalLevel: WAL の詳細度（Minimal, Replica, Logical）を決定する列挙型。
  • XLogInsertRecord(): WAL レコードをバッファに挿入する主要関数。
  • XLogFlush(): 指定した LSN まで WAL をディスクに強制書き込みする関数。
  • RecoveryInProgress(): 現在のシステムがリカバリ中かどうかを判定する関数。
  • CHECKPOINT_*: CreateCheckPoint 等の挙動を制御するビットマスク群。

制御フローとデータフロー

本ファイルは定義が中心ですが、以下のフローを規定しています： 1. **挿入**: XLogInsertRecord を通じて WAL レコードがバッファに書き込まれ、XLogRecPtr が進みます。 2. **フラッシュ**: XLogFlush が呼び出されると、バッファ内のデータが物理ファイルへ書き出されます。 3. **リカバリ**: StartupXLOG がシステム起動時に呼び出され、xlogrecovery.c の機能を利用して WAL を再適用します。 4. **チェックポイント**: CreateCheckPoint が定期的に実行され、ダーティページをディスクにフラッシュし、制御ファイルを更新します。

直接依存

  • access/xlogbackup.h: バックアップ関連の構造体定義。
  • access/xlogdefs.h: XLogRecPtr や TimeLineID などの基本型。
  • datatype/timestamp.h: チェックポイント統計用。
  • lib/stringinfo.h: WAL 記述用。
  • nodes/pg_list.h: バックアップ用テーブルスペースリスト管理。

関連コード

  • src/backend/access/transam/xlog.c: 本ヘッダーの主要な実装。
  • src/include/access/xloginsert.h: WAL レコード構築のための補助関数群。
  • src/include/access/xlogreader.h: WAL を解析・読み取るためのインターフェース。
  • src/include/access/xlogrecovery.h: リカバリ処理に特化した関数群。

読者向けメモ

  • 本ファイルは非常に多くの外部モジュールから参照されます。変更を加える際は、バイナリ互換性や PGDLLIMPORT の影響範囲に注意してください。
  • WAL_DEBUG が定義されている場合、デバッグ用のグローバル変数が追加されます。
  • XLogArchivingActive() などのマクロは、wal_level と XLogArchiveMode の整合性をチェックする重要なガードレールとして機能しています。
