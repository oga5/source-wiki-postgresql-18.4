src/include/utils/hsearch.h

概要

PostgreSQL の動的ハッシュテーブル実装（dynahash）の公開インターフェースを定義するヘッダーファイルです。メモリ内での効率的なデータ検索、挿入、削除を可能にする汎用的なハッシュテーブル機能を提供します。

モジュール内での役割

src/include/utils モジュールにおいて、メモリ管理やデータ構造の基盤として機能します。特に、バックエンド内のキャッシュや一時的なデータ保持において、動的に拡張可能なハッシュテーブルを提供することで、他のサブシステムを支えています。

責務

  • ハッシュテーブルの作成・破棄・管理のための API 定義。
  • ユーザー定義のハッシュ関数、比較関数、メモリ割り当て関数のインターフェース提供。
  • 共有メモリ上でのハッシュテーブル構築のサポート。
  • シーケンシャルスキャン（全要素走査）のための状態管理構造の提供。

重要なシンボル

  • HTAB: ハッシュテーブルを管理する不透明（opaque）な構造体。
  • HASHCTL: ハッシュテーブル作成時に設定を指定するための制御構造体。
  • HASH_SEQ_STATUS: ハッシュテーブルのシーケンシャルスキャン状態を保持する構造体。
  • hash_create() / hash_search(): ハッシュテーブルの主要な操作関数。
  • HASHACTION: HASH_FIND, HASH_ENTER 等、検索時の動作を指定する列挙型。

追加の索引済みシンボル

  • HASHELEMENT
  • HASHHDR
  • Size

制御フローとデータフロー

1. **初期化**: HASHCTL 構造体にフラグ（HASH_ELEM 等）と関数ポインタを設定し、hash_create() を呼び出して HTAB を取得します。 2. **操作**: hash_search() にキーと HASHACTION を渡して要素の検索・挿入・削除を行います。 3. **走査**: hash_seq_init() でスキャンを開始し、hash_seq_search() を繰り返し呼び出すことで全要素を順次取得します。 4. **終了**: hash_destroy() でメモリを解放します。

直接依存

本ファイルは純粋なインターフェース定義であり、他のヘッダーファイルへの依存関係は明示されていません。ただし、Size 型や uint32 型などの標準的な型定義を前提としています。

関連コード

  • src/backend/utils/hash/dynahash.c: 本ヘッダーで宣言された関数の実体。
  • src/include/utils/memutils.h: ハッシュテーブルが使用するメモリコンテキストの管理に関連。

読者向けメモ

  • **不透明な構造体**: HTAB や HASHHDR は内部実装の詳細を隠蔽しているため、呼び出し側はポインタを介して操作する必要があります。
  • **メモリ管理**: HASH_CONTEXT を指定しない場合、デフォルトのメモリコンテキストが使用されます。共有メモリを使用する場合は HASH_SHARED_MEM フラグが必須です。
  • **キーの配置**: HASHELEMENT を使用する場合、ユーザーデータは HASHELEMENT の直後に配置され、ハッシュキーはユーザーデータの先頭にあることが期待されます。
  • **必須フラグ**: HASH_ELEM は現在必須のフラグであり、キーサイズとエントリサイズの指定が求められます。
