src/include/utils/elog.h

概要

PostgreSQL のエラー報告およびログ出力機能のインターフェースを定義するヘッダーファイルです。バックエンド全体で利用されるエラーハンドリングの基盤であり、ログレベルの定義、エラー報告用 API、および例外処理のための PG_TRY マクロ群を提供します。

モジュール内での役割

src/include/utils モジュールにおいて、システム全体の堅牢性を支える中心的な役割を担います。他のモジュールがエラーを報告する際の標準的な手段を提供し、ログ出力先（stderr, syslog, CSV, JSON 等）への抽象化レイヤーとして機能します。

責務

  • **エラーレベルの定義**: DEBUG から PANIC までの重要度を定義し、ログ出力の制御を可能にする。
  • **エラー報告 API の提供**: ereport() および elog() マクロを通じた、構造化されたエラー情報の収集と伝播。
  • **例外処理の提供**: PG_TRY/CATCH/FINALLY マクロによる、トランザクション境界を跨ぐエラーの捕捉とリカバリ。
  • **SQLSTATE の管理**: エラーコードのコンパクトな表現と変換ロジックの提供。

重要なシンボル

  • ereport(elevel, ...): 推奨される新しいエラー報告 API。可変長引数でエラーの詳細（メッセージ、ヒント、SQLSTATE 等）を記述する。
  • elog(elevel, ...): 簡略化された古いエラー報告 API。内部的には ereport を呼び出す。
  • ErrorData: エラー発生時に収集される情報を保持する構造体。
  • PG_TRY() / PG_CATCH(): sigjmp_buf を利用した例外処理ブロック。
  • PG_RE_THROW(): キャッチしたエラーを上位へ再送出する。

追加の索引済みシンボル

  • ErrorContextCallback
  • PGErrorVerbosity

制御フローとデータフロー

1. **報告**: ereport() が呼び出されると、errstart() でエラーの重要度を判定し、必要であれば ErrorData 構造体を初期化する。 2. **構築**: 続く errmsg(), errcode() 等の関数が ErrorData に情報を蓄積する。 3. **完了**: errfinish() が呼び出されると、蓄積された ErrorData がログ出力先へ送られ、ERROR 以上の場合はトランザクションの異常終了処理（siglongjmp）が実行される。 4. **捕捉**: PG_TRY ブロック内の sigsetjmp により、エラー発生時に PG_CATCH ブロックへ制御が戻る。

直接依存

  • setjmp.h: 例外処理（スタックの保存と復帰）に使用。
  • lib/stringinfo.h: ログメッセージ構築用の文字列バッファ。
  • utils/errcodes.h: SQLSTATE エラーコード定義。

関連コード

  • src/backend/utils/error/elog.c: 本ヘッダーで宣言された関数の実装。ログ出力の具体的なロジックや、再帰的なエラー発生時の安全策（エスケープハッチ）が含まれる。
  • src/backend/utils/error/err.c: エラーメッセージの翻訳や SQLSTATE の管理に関連する補助機能。

読者向けメモ

  • **パフォーマンス**: ereport は頻繁に呼び出される可能性があるため、メッセージ生成のコストには注意が必要です。
  • **再帰エラー**: エラー処理中にさらにエラーが発生する「再帰的エラー」への対策が elog.c に実装されています。PANIC レベルのエラーはプロセスを強制終了させるため、注意して使用してください。
  • **volatile 変数**: PG_TRY ブロック内で変更され、PG_CATCH ブロックで使用されるローカル変数は、コンパイラの最適化による不整合を防ぐため volatile 宣言が必須です。
