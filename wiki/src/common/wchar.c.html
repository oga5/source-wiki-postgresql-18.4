src/common/wchar.c

概要

PostgreSQL がサポートする多様なマルチバイト文字エンコーディング（EUC, UTF-8, SJIS, Big5 等）を抽象化し、文字の長さ計算、表示幅計算、および妥当性検証を行うためのコアライブラリです。

モジュール内での役割

src/common 内で、文字列処理の基盤を提供します。バックエンドとフロントエンド（libpq 等）の両方から参照され、エンコーディング固有の差異を pg_wchar_table というディスパッチテーブルを通じて統一的なインターフェースに変換します。

責務

  • **エンコーディング抽象化**: 各エンコーディングの操作関数をテーブル化し、統一的な API を提供。
  • **文字操作**: マルチバイト文字列の長さ（バイト数）および表示幅（カラム数）の算出。
  • **妥当性検証**: 文字列が指定されたエンコーディングとして正当かどうかの検証（セキュリティ上の不正なバイト列の排除）。
  • **変換支援**: pg_wchar（UCS-4）と各エンコーディング間の相互変換。

重要なシンボル

ディスパッチテーブル

  • pg_wchar_table: 各エンコーディングの mblen, dsplen, verifychar, verifystr などの関数ポインタを保持する構造体配列。

公開 API（エンコーディング汎用）

  • pg_encoding_mblen: 文字列のバイト長を返す。
  • pg_encoding_dsplen: 文字列の表示幅を返す。
  • pg_encoding_verifymbchar: 単一文字の妥当性を検証。
  • pg_encoding_verifymbstr: 文字列全体の妥当性を検証し、有効なバイト数を返す。
  • pg_encoding_max_length: 特定エンコーディングの最大文字バイト数を取得。

内部ヘルパー（UTF-8 高速化）

  • utf8_advance: シフトベースの DFA を用いて UTF-8 の状態遷移を計算。
  • pg_utf8_islegal: RFC3629 に基づく UTF-8 の正当性チェック。

制御フローとデータフロー

1. **ディスパッチ**: 公開 API は pg_wchar_table を参照し、encoding ID に応じて適切な関数を呼び出します。 2. **検証フロー**: verifystr 系関数は、文字列を走査し、各文字の verifychar を呼び出します。不正なバイト列に遭遇した時点で処理を中断し、そこまでのオフセットを返します。 3. **UTF-8 高速化**: pg_utf8_verifystr は、SIMD 的なアプローチ（STRIDE_LENGTH）で ASCII 部分をスキップし、マルチバイト部分のみを DFA で処理する最適化を行っています。

読者向けメモ

  • **メモリ所有権**: 本ファイル内の関数はメモリの割り当てを行いません。呼び出し側が十分なバッファを確保する必要があります（特に to バッファ）。
  • **GB18030 の例外**: pg_gb18030_mblen は 2 バイト目を確認する必要があるため、他のエンコーディングよりも注意が必要です。
  • **UTF-8 DFA**: Utf8Transition テーブルは、Per Vognsen 氏のシフトベース DFA 実装を採用しており、パイプライン効率を最大化するように設計されています。
  • **省略された詳細**: 各エンコーディングの具体的な変換ロジック（pg_euc2wchar_with_len 等）は、各エンコーディングの仕様に依存するため、詳細なビット演算の解説は省略しています。
