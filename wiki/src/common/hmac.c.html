src/common/hmac.c

概要

RFC 2104 に準拠した HMAC（Keyed-Hashing for Message Authentication）のフォールバック実装を提供します。OpenSSL 等の外部ライブラリに依存しない環境でも、PostgreSQL がサポートする各種ハッシュアルゴリズム（MD5, SHA1, SHA2系）を用いたメッセージ認証コードの生成を可能にします。

モジュール内での役割

src/common における暗号化ユーティリティの一部であり、cryptohash インターフェースをラップして HMAC 特有の二段階ハッシュ処理（IPAD/OPAD）を隠蔽します。

責務

  • **HMAC コンテキストの管理**: pg_hmac_ctx の生成、初期化、およびメモリ解放。
  • **鍵の正規化**: 鍵長がブロックサイズを超える場合のハッシュ化による短縮処理。
  • **二段階ハッシュの実行**: k_ipad（内部パディング）と k_opad（外部パディング）を用いた HMAC 計算の制御。
  • **エラーハンドリング**: 実行時のメモリ不足や内部ハッシュ関数のエラーを捕捉し、状態として保持。

重要なシンボル

  • pg_hmac_ctx: HMAC 計算の状態を保持する不透明な構造体。
  • pg_hmac_create/init/update/final: HMAC 計算のライフサイクルを管理する主要 API。
  • HMAC_IPAD / HMAC_OPAD: RFC 2104 で定義されたパディング用定数。

制御フローとデータフロー

1. **初期化 (pg_hmac_init)**: 鍵がブロックサイズより長い場合、一度ハッシュ化して短縮します。その後、鍵と HMAC_IPAD を XOR し、内部ハッシュコンテキストを初期化します。 2. **更新 (pg_hmac_update)**: ユーザーから渡されたデータを内部の pg_cryptohash コンテキストへ逐次渡します。 3. **終了 (pg_hmac_final)**:

  • まず内部ハッシュを pg_cryptohash_final で確定させます。
  • 次に、その結果を k_opad と XOR した値と組み合わせて再度ハッシュ化し、最終的な HMAC 値を生成します。

直接依存

  • common/cryptohash.h: 基盤となるハッシュ関数（MD5, SHA等）の抽象化レイヤー。
  • common/hmac.h: 公開インターフェース定義。
  • postgres.h / postgres_fe.h: バックエンド/フロントエンド環境に応じたメモリ管理（palloc/malloc）の切り替え。
  • common/md5.h
  • common/sha1.h
  • common/sha2.h

関連コード

  • src/common/cryptohash.c: 実際のハッシュ計算を行う低レイヤーの実装。
  • src/common/hmac_openssl.c: OpenSSL を利用した HMAC 実装（ビルド構成により選択される）。

読者向けメモ

  • **メモリ管理**: バックエンドでは palloc を使用するため、エラー時はメモリが自動解放されますが、フロントエンドでは malloc を使用するため、呼び出し側で pg_hmac_free を呼び出す責任があります。
  • **セキュリティ**: 本実装はフォールバック用です。可能な限り OpenSSL 等の最適化されたライブラリの使用が推奨されます。
  • **定数**: k_ipad と k_opad は、サポートされる最大ブロックサイズ（PG_SHA512_BLOCK_LENGTH）に合わせて確保されており、実装の簡素化を図っています。
