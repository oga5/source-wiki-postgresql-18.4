src/common/jsonapi.c

概要

jsonapi.c は、PostgreSQL のバックエンドおよびフロントエンド（libpq 等）で共有される、JSON の字句解析および構文解析を行うための汎用ライブラリです。ストリーミング（インクリメンタル）解析と、従来の再帰下降解析の両方をサポートします。

モジュール内での役割

src/common 内の JSON 処理基盤として、JSON データの検証や、特定の構造に対するセマンティックアクション（コールバック）の実行を提供します。環境に応じて StringInfo（バックエンド）または PQExpBuffer（フロントエンド）を抽象化し、メモリ管理を統一しています。

責務

  • **字句解析**: JSON トークンの切り出し、エスケープシーケンスのデコード、Unicode サロゲートペアの処理。
  • **構文解析**:
  • 再帰下降パーサーによる標準的な JSON 解析。
  • テーブル駆動型（Dragon Book 4.3 準拠）の非再帰パーサーによるインクリメンタル解析。
  • **エラー報告**: 解析コンテキストに応じた詳細なエラーメッセージの生成。
  • **メモリ管理**: 解析コンテキストのライフサイクル管理および、環境依存のメモリ割り当ての抽象化。

重要なシンボル

解析コンテキストと状態

  • JsonLexContext: 解析のメイン構造体。入力バッファ、現在のトークン位置、解析レベル、フラグを保持。
  • JsonParserStack: インクリメンタル解析用の予測スタックおよび状態保持用構造体。
  • JsonIncrementalState: 部分的なトークン（チャンク境界を跨ぐデータ）を蓄積するための状態。

公開インターフェース

  • makeJsonLexContextCstringLen: 解析コンテキストの初期化（再帰下降用）。
  • makeJsonLexContextIncremental: インクリメンタル解析用コンテキストの初期化。
  • pg_parse_json: 再帰下降パーサーの実行。
  • pg_parse_json_incremental: インクリメンタルパーサーの実行。
  • freeJsonLexContext: コンテキストに関連するメモリの解放。
  • json_errdetail: 解析エラーの詳細メッセージ生成。

内部ヘルパー

  • json_lex: 次のトークンを取得する字句解析器。
  • parse_object, parse_array, parse_scalar: 再帰下降パーサーの各構成要素。

制御フローとデータフロー

1. **初期化**: makeJsonLexContext... 関数で JsonLexContext を作成し、必要に応じて need_escapes フラグを設定。 2. **解析**:

  • **再帰下降**: pg_parse_json が呼び出され、parse_object 等が再帰的に呼び出される。
  • **インクリメンタル**: pg_parse_json_incremental が呼び出され、td_parser_table を参照して予測スタックを操作し、トークンと非終端記号をマッチングさせる。

    • 3. **セマンティックアクション**: 解析中に JsonSemAction で渡された関数ポインタが呼び出され、ユーザー定義の処理（データ抽出など）が実行される。 4. **エラー処理**: report_parse_error が解析コンテキストと現在のトークンに基づいてエラーコードを返し、json_errdetail が人間が読める形式に変換する。

読者向けメモ

  • **メモリ所有権**: JsonLexContext の flags（JSONLEX_FREE_STRUCT, JSONLEX_FREE_STRVAL）により、メモリ解放の責任範囲が制御されます。freeJsonLexContext を適切に呼び出す必要があります。
  • **インクリメンタル解析**: チャンク境界でトークンが分断される場合、inc_state->partial_token にデータが蓄積されます。このため、インクリメンタル解析中は prev_token_terminator の使用は避けるべきです。
  • **制限事項**: インクリメンタルパーサーのスタックサイズは現在 JSON_TD_MAX_STACK（6400）に固定されています。
  • **省略された詳細**: 本ファイルは字句解析の Unicode 変換ロジックが非常に詳細ですが、ここでは構造的な役割に焦点を当てています。pg_unicode_to_server_noerror 等のバックエンド固有の変換関数は、フロントエンドビルド時には除外されるよう #ifndef FRONTEND でガードされています。
