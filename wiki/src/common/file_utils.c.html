src/common/file_utils.c

概要

ファイルおよびディレクトリ操作のためのユーティリティ関数群を提供します。主にPostgreSQLのフロントエンドツール（pg_basebackup 等）において、データディレクトリの同期、耐久性を考慮したファイルリネーム、および堅牢なI/O操作をサポートします。

モジュール内での役割

src/common モジュールの一部として、プラットフォーム固有のファイルシステム操作を抽象化します。バックエンド（src/backend）とフロントエンド（src/bin）の両方から利用可能な低レベルなファイル操作インターフェースを提供し、特にデータディレクトリの整合性維持に貢献します。

責務

  • **データディレクトリの同期**: sync_pgdata を通じて、fsync または syncfs を用いたデータディレクトリ全体の永続化。
  • **耐久性のあるファイル操作**: durable_rename による、クラッシュ耐性を備えたファイルリネーム処理。
  • **堅牢なI/O**: pg_pwritev_with_retry や pg_pwrite_zeros による、部分書き込みを考慮した安全な書き込み処理。
  • **ディレクトリ走査**: walkdir による、シンボリックリンクや除外パスを考慮した再帰的なディレクトリ処理。

重要なシンボル

  • sync_pgdata: データディレクトリ全体を同期するメイン関数。
  • durable_rename: ファイル移動時に、移動元・移動先・親ディレクトリを確実に fsync する関数。
  • walkdir: ディレクトリツリーを走査し、各エントリに対して指定されたアクション（fsync等）を実行する。
  • pg_pwritev_with_retry: pwritev の部分書き込みを自動的に再試行するラッパー。
  • get_dirent_type: ファイルタイプを判定する（direntの拡張機能またはstatを使用）。

制御フローとデータフロー

1. **同期処理**: sync_pgdata は DataDirSyncMethod に応じて syncfs（ファイルシステム単位）または walkdir（ファイル単位）を選択します。walkdir は再帰的にディレクトリを辿り、各ファイルに対して fsync_fname を呼び出します。 2. **耐久性リネーム**: durable_rename は、まず oldfile を fsync し、次に newfile が存在すれば fsync します。rename 実行後、newfile とその親ディレクトリに対して再度 fsync を行い、メタデータの永続化を保証します。 3. **I/O再試行**: pg_pwritev_with_retry は、compute_remaining_iovec を使用して、部分的に書き込まれた iovec の残量を計算し、すべてのデータが書き込まれるまでループを継続します。

読者向けメモ

  • **FRONTEND マクロ**: 本ファイルの大部分は FRONTEND 定義時にのみコンパイルされます。バックエンドコードから利用可能な関数（get_dirent_type 等）と、フロントエンド専用の関数が混在している点に注意してください。
  • **fsync の重要性**: durable_rename や sync_pgdata は、OSのクラッシュ時にデータが失われないよう、ディレクトリの fsync を含めた厳格な順序で操作を行います。
  • **エラー処理**: fsync_fname などは、ディレクトリの fsync がサポートされていない環境でのエラーを適切に無視するよう設計されています。
