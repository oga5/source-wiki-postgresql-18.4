src/interfaces/ecpg/ecpglib/execute.c

概要

ECPG ランタイムにおける SQL 実行のメインエンジンです。プリプロセッサが生成した C コードと PostgreSQL バックエンド間の橋渡しを行い、SQL 文の構築、パラメータのバインド、実行、および結果のホスト変数への格納を統括します。

モジュール内での役割

ecpglib の中核であり、SQL 実行のライフサイクル（準備・実行・後処理）を管理します。libpq を直接呼び出すのではなく、ECPG 固有の型変換や互換性モードを抽象化したレイヤーとして機能します。

責務

  • **SQL 実行ライフサイクル**: ECPGdo を起点とした SQL 実行フローの制御。
  • **パラメータ構築**: ホスト変数から SQL プレースホルダーへの変換、エスケープ処理、および libpq 用のパラメータ配列構築。
  • **データ型変換**: C 言語の型と PostgreSQL の OID 間のマッピングおよび配列/ベクトルの判定。
  • **結果の格納**: バックエンドから返された PGresult を解析し、アプリケーションのホスト変数へ値をコピー。
  • **環境管理**: ロケール（数値フォーマット）の切り替えと復元、トランザクションの自動開始。

重要なシンボル

実行制御

  • ECPGdo: 外部公開される SQL 実行のメインエントリーポイント。
  • ecpg_do_prologue / ecpg_do_epilogue: ステートメント構造体の初期化と、ロケール復元を含む後処理。
  • ecpg_execute: libpq の PQexecParams や PQexecPrepared を呼び出す実行層。
  • ecpg_process_output: 実行結果を解析し、ホスト変数への格納やエラーハンドリングを行う。

パラメータ・データ処理

  • ecpg_build_params: SQL 文内のプレースホルダーを解析し、ホスト変数をバインド可能な形式に変換。
  • ecpg_store_input: 個別のホスト変数を SQL 文字列またはパラメータ配列へ変換。
  • ecpg_store_result: PGresult からホスト変数へのデータコピーと型変換。
  • ecpg_is_type_an_array: OID に基づき、その型が配列かベクトルかを判定（キャッシュ機構付き）。

制御フローとデータフロー

1. **初期化**: ECPGdo が va_list を受け取り、ecpg_do_prologue で statement 構造体を生成。ロケールを "C" に固定。 2. **パラメータ構築**: ecpg_build_params が inlist を走査。next_insert でプレースホルダー位置を特定し、ecpg_store_input で値をエスケープまたはバイナリ変換。 3. **実行**: ecpg_execute が libpq を介してクエリを送信。 4. **結果処理**: ecpg_process_output が PGresult を解析。ecpg_store_result を通じて outlist の各変数へ値を格納。 5. **終了**: ecpg_do_epilogue でメモリ解放とロケール復元。

読者向けメモ

  • **メモリ管理**: ecpg_alloc / ecpg_free を使用。statement 構造体は ecpg_do_epilogue で確実に解放されます。
  • **ロケール**: 数値変換の整合性を保つため、実行中に LC_NUMERIC を "C" に変更します。HAVE_USELOCALE が定義されている場合はスレッドセーフな uselocale を使用します。
  • **エスケープ**: quote_postgres は PQescapeString を使用し、standard_conforming_strings の設定に応じて E'' 構文を使い分けます。
  • **未実装/制限**: 配列の動的処理や一部の複雑な型変換は、将来的な拡張のために FIXME として残されている箇所があります。特に ECPGt_char_variable のような動的カーソル処理は、バックエンドの機能不足をクライアント側でシミュレートしています。
