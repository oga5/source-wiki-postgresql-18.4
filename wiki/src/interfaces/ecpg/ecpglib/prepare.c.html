src/interfaces/ecpg/ecpglib/prepare.c

概要

ECPG（Embedded SQL for C）におけるプリペアドステートメントの管理、キャッシュ、およびライフサイクルを制御するモジュールです。SQLの PREPARE および DEALLOCATE コマンドの実行と、自動プリペア機能のためのステートメントキャッシュを実装しています。

モジュール内での役割

ecpglib の一部として、アプリケーションが発行する動的SQLの準備と再利用を担います。execute.c などがクエリを実行する際、このモジュールが提供するキャッシュ機構を利用することで、バックエンドへの通信オーバーヘッドを削減します。

責務

  • **ステートメントの準備**: ECPGprepare を通じたSQLのプリペアと、C変数からプレースホルダ（$n）への変換。
  • **ライフサイクル管理**: プリペアドステートメントの登録、検索、および DEALLOCATE による破棄。
  • **自動キャッシュ**: ecpg_auto_prepare によるクエリのハッシュベースのキャッシュ管理（LRUに近いアルゴリズムで古いエントリを置換）。
  • **接続管理**: 接続ごとのプリペアドステートメントリストの維持と、接続終了時のクリーンアップ。

重要なシンボル

  • ECPGprepare: EXEC SQL PREPARE 文のフロントエンド実装。
  • ECPGdeallocate: EXEC SQL DEALLOCATE 文のフロントエンド実装。
  • ecpg_auto_prepare: クエリを自動的にキャッシュし、必要に応じてプリペアを行う内部関数。
  • stmtCacheEntries: キャッシュされたステートメントを保持するハッシュテーブル構造体。
  • replace_variables: SQL内のホスト変数（: や ?）をPostgreSQL形式の $n に変換するパーサ。

制御フローとデータフロー

1. **準備**: ECPGprepare が呼ばれると、replace_variables でクエリを正規化し、PQprepare でバックエンドに送信します。 2. **登録**: 成功したステートメントは prepared_statement 構造体として接続オブジェクト（connection）のリストにリンクされます。 3. **キャッシュ**: ecpg_auto_prepare は HashStmt でクエリをハッシュ化し、stmtCacheEntries を検索します。ヒットすれば既存のIDを返し、ミスすれば新規IDを生成してキャッシュに追加します。 4. **破棄**: deallocate_one が呼び出されると、バックエンドに対して DEALLOCATE コマンドを発行し、メモリを解放してリストから削除します。

読者向けメモ

  • **キャッシュの仕組み**: stmtCacheNBuckets（2039）個のバケットを持ち、各バケットは stmtCacheEntPerBucket（8）個の固定長エントリを持ちます。ハッシュ衝突時は最も実行回数（execs）が少ないエントリが置換対象となります。
  • **自動プリペア**: ECPGの自動プリペア機能は、アプリケーションが明示的に PREPARE を書かなくても、ライブラリ側でクエリをハッシュ化して自動的に PQprepare を発行する仕組みです。
  • **メモリ管理**: ecpg_alloc / ecpg_free を使用してメモリを管理しています。deallocate_one 実行時のメモリリークには注意が必要です。
