src/interfaces/libpq/fe-misc.c

概要

fe-misc.c は、libpq における低レベルな通信 I/O、バッファ管理、およびプラットフォーム抽象化を担うユーティリティモジュールです。PostgreSQL プロトコル v3.0 のメッセージ形式に従ったデータの読み書き、ソケットの非ブロッキング I/O 制御、およびエラーメッセージのフォーマット機能を提供します。

モジュール内での役割

libpq の通信層の最下層に位置し、fe-protocol3.c などの上位層がソケットを直接操作することなく、ストリームとしてデータを扱えるように抽象化を提供します。PGconn 構造体が保持する入出力バッファの管理と、プラットフォーム依存のソケットポーリング（poll/select）を隠蔽します。

責務

  • **データシリアライズ/デシリアライズ**: ネットワークバイトオーダーとホストバイトオーダーの変換、およびプロトコルメッセージの構築。
  • **バッファ管理**: PGconn 内の入出力バッファの動的拡張、再配置、およびメモリ確保の失敗時のエラーハンドリング。
  • **ソケット I/O**: 非ブロッキングモードでのソケット読み書き、およびタイムアウト付きの待機処理。
  • **国際化・ユーティリティ**: エラーメッセージの翻訳（NLS）、マルチバイト文字の長さ計算、環境変数からのエンコーディング取得。

重要なシンボル

メッセージ構築・解析

  • pqPutMsgStart / pqPutMsgEnd: メッセージ構築の開始と終了（長さフィールドの計算を含む）。
  • pqPutc / pqPuts / pqPutInt: メッセージへのデータ書き込み。
  • pqGetc / pqGets / pqGetInt: 受信バッファからのデータ読み出し。
  • pqParseDone: メッセージ解析完了後のバッファポインタ更新。

バッファ管理

  • pqCheckOutBufferSpace / pqCheckInBufferSpace: バッファの動的拡張とメモリ確保。

I/O 制御

  • pqReadData: ソケットからデータを読み込み、inBuffer に追加。
  • pqFlush: 出力バッファ内のデータをソケットへフラッシュ。
  • pqWait / pqWaitTimed: ソケットの読み書き可能状態を待機。

ユーティリティ

  • PQmblen / PQdsplen: マルチバイト文字のバイト長および表示幅の取得。
  • libpq_append_conn_error: PGconn のエラーバッファへの翻訳済みエラーメッセージ追加。

制御フローとデータフロー

1. **送信フロー**: pqPutMsgStart でバッファを確保し、pqPut* 関数群でデータを積む。pqPutMsgEnd でメッセージ長を確定し、バッファサイズが 8KB を超えるか明示的な pqFlush が呼ばれると pqSendSome がソケットへ書き出す。 2. **受信フロー**: pqReadData がソケットから inBuffer へ読み込む。必要に応じて memmove でバッファを左詰めし、空き領域を確保する。上位層は pqGet* 関数群を使用して inBuffer からデータを消費する。 3. **待機フロー**: pqWaitTimed が PQsocketPoll を呼び出し、poll または select を用いてソケットの状態を監視する。タイムアウト時は libpq_append_conn_error でエラーを記録する。

読者向けメモ

  • **メモリ所有権**: PGconn 内の inBuffer および outBuffer は libpq が所有し、realloc で動的に拡張されます。呼び出し側はバッファの生存期間を意識する必要はありません。
  • **非ブロッキング I/O**: libpq は基本的に非ブロッキングモードで動作します。pqSendSome や pqReadData が EAGAIN や EWOULDBLOCK を受け取った場合、呼び出し側は適切に待機（pqWait）して再試行する必要があります。
  • **バッファの左詰め**: pqReadData および pqCheckInBufferSpace は、バッファの断片化を防ぐために memmove を使用してデータを先頭に寄せます。このため、バッファ内のポインタを保持し続けることは危険です。
  • **省略された詳細**: 本ファイルは低レベルな I/O に特化しているため、認証ロジックやクエリ実行のステートマシンについては記述されていません。これらは fe-auth.c や fe-exec.c が担当します。
