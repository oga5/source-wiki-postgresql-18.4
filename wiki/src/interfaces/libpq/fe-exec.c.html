src/interfaces/libpq/fe-exec.c

概要

fe-exec.c は、libpq におけるクエリ実行の心臓部です。アプリケーションからのクエリ送信（同期・非同期・パイプライン）、サーバーからの結果セット（PGresult）の構築、および結果データのメモリ管理を担います。libpq の中で最も複雑なメモリ管理ロジックと、プロトコル状態遷移の多くを包含しています。

モジュール内での役割

libpq のクエリ実行層の中核であり、fe-protocol3.c（プロトコル解析）と fe-misc.c（低レベル I/O）の間に位置します。アプリケーションが PQexec や PQsendQuery を呼び出すと、このファイルがリクエストを構築し、プロトコル層を介して送信し、結果を PGresult 構造体に集約します。

責務

  • **PGresult 管理**: PGresult のライフサイクル管理、属性設定、および効率的なメモリ割り当て。
  • **クエリ送信**: 同期 (PQexec) および非同期 (PQsendQuery) クエリ実行のディスパッチ。
  • **パイプライン管理**: パイプラインモードにおけるコマンドキューの管理と同期ポイントの制御。
  • **データ変換・エスケープ**: SQL インジェクションを防ぐためのリテラルおよび識別子のエスケープ処理。
  • **COPY 制御**: COPY プロトコルにおけるデータ転送の抽象化。

重要なシンボル

PGresult ライフサイクルとメモリ管理

  • PQmakeEmptyPGresult: PGresult の初期化。
  • PQclear: PGresult が保持する全リソース（イベント、データブロック、タプル配列）の解放。
  • pqResultAlloc: PGresult 内部のメモリブロック管理。断片化を防ぐため、大きなブロックを確保し、その中でオフセット管理を行う。
  • PQsetvalue: PGresult にタプルデータを追加する。

クエリ実行 API

  • PQexec / PQexecParams: 同期実行のメインエントリ。
  • PQsendQuery / PQsendQueryParams: 非同期実行のエントリ。
  • PQgetResult: サーバーからの応答を解析し、PGresult を返す。

パイプライン・状態管理

  • pqPipelineProcessQueue: パイプラインモードで次のクエリ処理を開始する。
  • pqCommandQueueAdvance: コマンドキューの進捗管理。
  • pqRowProcessor: サーバーから受信した行データを PGresult に格納する。

制御フローとデータフロー

1. **送信**: PQsendQueryInternal がコマンドを構築し、pqPutMsgStart 等でバッファへ書き込み、pqFlush でソケットへ送出します。 2. **受信**: PQgetResult が parseInput を呼び出し、fe-protocol3.c のパーサーを駆動します。 3. **結果構築**: パーサーが受信した行データに対し pqRowProcessor が呼ばれ、pqResultAlloc を介して PGresult 内のメモリブロックにデータがコピーされます。 4. **状態遷移**: conn->asyncStatus が BUSY から READY へ遷移することで、アプリケーションに結果の準備完了を通知します。

読者向けメモ

  • **メモリ管理**: PGresult は個別の malloc を最小化するため、PGRESULT_DATA_BLOCKSIZE (2048バイト) 単位のブロック割り当てを行っています。このため、PQclear を呼び出さないとメモリリークが確実に発生します。
  • **スレッド安全性**: libpq は接続単位でスレッドセーフですが、同一の PGconn を複数のスレッドから同時に操作することはできません。
  • **浅い領域**: PQescapeString 等のエスケープ関数は、マルチバイト文字の検証を行いますが、エンコーディングの完全なバリデーションはサーバー側の責務であるという前提で実装されています。
  • **パイプラインモード**: パイプラインモードは非常に複雑な状態遷移を伴うため、pqCommandQueueAdvance のロジックを理解する際は、PGconn の asyncStatus と pipelineStatus の組み合わせに注意してください。
