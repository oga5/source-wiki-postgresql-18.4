src/test/modules/test_tidstore/test_tidstore.c

概要

test_tidstore.c は、PostgreSQL の TidStore データ構造の機能と整合性を検証するためのテスト用モジュールです。SQL インターフェースを通じて TidStore の作成、データ挿入、検索、イテレーション、および破棄の各操作をシミュレートします。

モジュール内での役割

本ファイルは src/test/modules/test_tidstore ディレクトリにおける唯一のソースであり、TidStore API の利用例および回帰テストとしての役割を担います。

責務

  • **TidStore のライフサイクル管理**: test_create および test_destroy によるインスタンスの生成と解放。
  • **データ整合性の検証**: do_set_block_offsets で挿入したデータと、check_set_block_offsets で取得・検索した結果を照合し、期待値と一致するかを確認する。
  • **メモリ使用量の監視**: test_is_full を通じて、TidStore のメモリ使用量計測機能が正常に動作しているかを検証する。

重要なシンボル

  • tidstore: テスト対象となる TidStore インスタンスを保持する静的変数。
  • ItemArray: テストの正当性を検証するために、挿入済みデータを保持しておくための構造体。
  • do_set_block_offsets: 指定したブロック番号とオフセットの配列を TidStore に格納する関数。
  • check_set_block_offsets: TidStore 内の全データを走査し、ItemArray と比較して不整合がないか確認する関数。

制御フローとデータフロー

1. **初期化**: test_create が呼び出され、TidStore がメモリ上に確保されます。 2. **データ挿入**: do_set_block_offsets が SQL から呼ばれると、TidStore への書き込みと同時に、検証用配列 items にも同じデータをコピーします。 3. **検証**: check_set_block_offsets が実行されると、以下の手順で整合性を確認します。

  • TidStoreIsMember を用いて、挿入した全 TID が存在するか確認。
  • TidStoreBeginIterate を用いて全データを取得し、ソートした items と比較。

    • 4. **終了**: test_destroy によりメモリが解放されます。

読者向けメモ

  • 本モジュールは単一プロセスでの実行を前提としています。LWLock を使用していますが、これは TidStore の API 仕様を網羅するためのものであり、並行実行のテストを目的としたものではありません。
  • TidStore の内部状態を検証するために、ItemArray という補助的な構造体で「期待される状態」を保持しています。テストコードを読む際は、この配列と TidStore の状態が常に同期されている点に注目してください。
