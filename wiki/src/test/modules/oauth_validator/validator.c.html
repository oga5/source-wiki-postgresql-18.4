src/test/modules/oauth_validator/validator.c

概要

PostgreSQLのサーバーサイドOAuthトークン検証機能をテストするためのモジュールです。libpq/oauth.h で定義されたフックインターフェースを実装し、トークン検証のライフサイクル（起動、検証、終了）をシミュレートします。

モジュール内での役割

本ファイルは、OAuth検証モジュールの標準的な実装例として機能します。他の検証モジュール（magic_validator.c 等）が特定の検証ロジックをテストするのに対し、本ファイルは基本的なコールバックの登録とGUC（設定パラメータ）による動作制御のテンプレートを提供します。

責務

  • **コールバックの提供**: OAuthValidatorCallbacks 構造体を介して、サーバーが必要とする検証用関数を公開する。
  • **設定管理**: oauth_validator.authn_id および oauth_validator.authorize_tokens を定義し、検証結果を動的に変更可能にする。
  • **状態検証**: private_data にマジックナンバー（PRIVATE_COOKIE）を保持し、ライフサイクル全体で状態が正しく維持されているかを検証する。

重要なシンボル

  • validator_callbacks: サーバーに提供するコールバック関数のテーブル。
  • _PG_oauth_validator_module_init: モジュール読み込み時にサーバーから呼び出されるエントリポイント。
  • validate_token: トークン検証のメインロジック。GUCの設定に基づき ValidatorModuleResult を設定する。
  • PRIVATE_COOKIE: 状態の整合性を確認するための定数。

追加の索引済みシンボル

  • validator_shutdown
  • validator_startup

制御フローとデータフロー

1. **初期化**: _PG_init でGUCを登録し、_PG_oauth_validator_module_init でコールバックテーブルをサーバーへ渡す。 2. **起動**: validator_startup が呼ばれ、private_data に PRIVATE_COOKIE を格納する。 3. **検証**: validate_token が呼ばれる際、private_data が期待値と一致するか確認する。その後、GUC authorize_tokens の値に基づいて res->authorized を決定し、authn_id を設定する。 4. **終了**: validator_shutdown で private_data の整合性を最終確認する。

  • utils/guc.h: GUC（設定パラメータ）の定義用。
  • utils/memutils.h: メモリ管理用（pstrdup 等）。

読者向けメモ

  • 本ファイルはテスト用であるため、sversion のチェックや PRIVATE_COOKIE による状態確認など、実運用モジュールでは推奨されないデバッグ用のコードが含まれています。
  • MyProcPort->hba を参照して oauth_issuer や oauth_scope をログ出力しており、HBA設定とOAuth検証の連携を確認する際の参考になります。
