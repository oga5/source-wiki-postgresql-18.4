src/test/modules/oauth_validator/oauth_hook_client.c

概要

本ファイルは、PostgreSQL の libpq における OAuth 認証フック機能の動作を検証するためのテスト用クライアントプログラムです。Perl テストスクリプト（t/002_client.pl）から呼び出され、認証フローの正常系および異常系をシミュレートします。

モジュール内での役割

oauth_validator モジュールにおける「テストドライバ」の役割を担います。libpq が提供する PQsetAuthDataHook を利用して、認証プロセス中に任意のトークン注入や意図的なエラー発生を行い、クライアント側の挙動を検証します。

責務

  • **フックの注入**: PQsetAuthDataHook を介して認証データ処理関数を登録する。
  • **認証フローの制御**: 正常なトークン提供、非同期処理のシミュレーション、および意図的な失敗（misbehavior）の注入。
  • **検証**: 期待されるスコープや URI が libpq から正しく渡されているかをチェックする。
  • **ストレス試験**: 非同期接続時のビジーウェイト（stress-async）によるエッジケースのテスト。

重要なシンボル

  • handle_auth_data: 認証フックのメインエントリポイント。認証要求をインターセプトし、設定に応じてトークンを返すか、エラーを返します。
  • async_cb / misbehave_cb: 非同期認証プロセスを制御するコールバック。hang-forever や不正な状態遷移のテストに使用されます。
  • main: コマンドライン引数を解析し、libpq の接続処理を開始するテストの起点。

追加の索引済みシンボル

  • usage

制御フローとデータフロー

1. **初期化**: main 関数がコマンドライン引数からテストモード（--misbehave 等）を読み込みます。 2. **フック登録**: PQsetAuthDataHook を呼び出し、handle_auth_data を登録します。 3. **接続開始**: PQconnectdb または PQconnectStart（非同期）で接続を開始します。 4. **認証介入**: libpq が認証を必要とすると handle_auth_data が呼び出されます。ここで req->token への代入や、非同期コールバック（async_cb 等）の割り当てが行われます。 5. **検証**: 接続結果を確認し、成功または失敗を標準出力/エラーに出力して終了します。

直接依存

  • postgres_fe.h: フロントエンド用共通定義。
  • libpq-fe.h: libpq の公開 API。
  • getopt_long.h: コマンドライン引数解析用。
  • sys/socket.h: 非同期テストにおけるソケット操作用。

関連コード

  • t/002_client.pl: 本プログラムを起動し、テストケースを自動実行する Perl スクリプト。
  • libpq/oauth.h: libpq 側で定義されている OAuth 関連の構造体や定数。

読者向けメモ

  • 本ファイルは実運用コードではなく、あくまでテストツールです。
  • misbehave_mode を指定することで、libpq が不正なコールバックの戻り値に対してどのように反応するか（堅牢性）をテストできます。
  • 非同期接続のテスト（stress-async）は、PQconnectPoll の呼び出しループを強制することで、非同期処理の競合やタイミング問題を誘発させる設計になっています。
