src/test/modules/spgist_name_ops/spgist_name_ops.c

概要

本ファイルは、PostgreSQLのSP-GiSTインデックスにおける name 型用のテスト用演算子クラス（opclass）の実装です。name 型の値をインデックス化しますが、内部的なストレージ形式として text 型を利用します。

モジュール内での役割

src/test/modules/spgist_name_ops モジュールの中心的な実装ファイルです。コアの spgtextproc.c と同様のロジックを name 型向けに適用し、照合順序（collation）を考慮しないシンプルな文字列比較を提供します。

責務

  • **インデックス構成の定義**: spgist_name_config により、インデックスのプレフィックスやリーフノードの型を定義。
  • **データ圧縮と変換**: spgist_name_compress を通じて name 型を text 型へ変換。
  • **ツリー操作**: spgist_name_choose により、新しい値をインデックスのどこに配置するか（または分割するか）を決定。
  • **検索ロジック**: spgist_name_inner_consistent および spgist_name_leaf_consistent を用いて、インデックスツリーの走査と条件一致判定を実行。

重要なシンボル

  • spgist_name_config: インデックスのメタデータ（型情報など）を設定。
  • spgist_name_choose: 挿入時のノード選択およびタプル分割ロジック。
  • spgist_name_inner_consistent: 内部ノードでの検索条件判定。
  • spgist_name_leaf_consistent: リーフノードでの最終的な値比較。
  • formTextDatum: text 型のデータムを構築する内部ヘルパー。

追加の索引済みシンボル

  • commonPrefix
  • searchChar
  • spgist_name_compress

制御フローとデータフロー

1. **挿入時**: spgist_name_compress で name を text に変換後、spgist_name_choose が呼び出されます。ここでは、既存のプレフィックスとの共通部分を commonPrefix で特定し、必要に応じて spgSplitTuple（分割）や spgAddNode（ノード追加）を指示します。 2. **検索時**: spgist_name_inner_consistent がツリーを再帰的に走査します。各ノードで reconstructedValue を復元し、scankeys と比較して一致する子ノードを特定します。最後に spgist_name_leaf_consistent が実際の値とクエリを比較し、結果を返します。

直接依存

  • postgres.h: PostgreSQLの基本定義。
  • access/spgist.h: SP-GiSTのインターフェース定義。
  • catalog/pg_type.h: TEXTOID などの型ID定義。
  • utils/datum.h: データム操作用。
  • varatt.h: 可変長データ（varlena）操作用。

関連コード

  • src/backend/access/spgist/spgtextproc.c: 本実装のベースとなったコアのテキスト用SP-GiST実装。ロジックの多くが共通しています。

読者向けメモ

  • 本実装は照合順序（Collation）を考慮していません。memcmp を使用したバイナリ比較が基本となります。
  • reconstructedValue は常にロング形式の varlena ヘッダーを持つものとして扱われます。
  • allTheSame フラグが立っている場合の分割処理など、SP-GiST特有の複雑なエッジケースが含まれているため、コアの spgtextproc.c と比較しながら読むことを推奨します。
