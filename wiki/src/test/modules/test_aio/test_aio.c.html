src/test/modules/test_aio/test_aio.c

概要

PostgreSQL の非同期 I/O (AIO) サブシステムの動作を SQL レベルから検証するためのテスト用ヘルパーモジュールです。本番環境では使用されないデバッグ・テスト専用の関数群を提供します。

モジュール内での役割

src/test/modules/test_aio ディレクトリの主要な実装ファイルです。AIO の内部状態を操作したり、意図的にエラーを発生させたりすることで、AIO の堅牢性をテストするためのインターフェースを公開します。

責務

  • **SQL インターフェースの提供**: C で実装された低レベルな AIO 操作（バッファの作成、IO の開始・終了、ハンドル管理）を SQL から呼び出し可能にする。
  • **エラー注入**: InjectionPoint を利用し、短縮読み込み（short read）や再オープン失敗などの異常系シナリオをシミュレートする。
  • **バッファ管理の強制**: 共有バッファやローカルバッファの状態を直接操作し、テストに必要な前提条件を整える。

重要なシンボル

  • InjIoErrorState: エラー注入の状態を保持する共有メモリ構造体。
  • test_aio_shmem_startup: モジュールロード時に注入ポイントを登録するフック関数。
  • read_rel_block_ll: 低レベルな AIO 読み込みを実行する主要なテスト関数。
  • inj_io_short_read / inj_io_reopen: 注入ポイントから呼び出されるエラー発生ロジック。

制御フローとデータフロー

1. **初期化**: _PG_init が共有メモリの確保と InjectionPoint のアタッチをフックします。 2. **操作実行**: SQL から read_rel_block_ll などの関数が呼ばれると、create_toy_buffer でバッファを確保し、pgaio API を介して I/O を発行します。 3. **エラー注入**: I/O 処理の途中で InjectionPoint がトリガーされると、InjIoErrorState のフラグに基づき、結果の改ざんや強制的なエラー発生が行われます。

読者向けメモ

  • **安全性**: 本ファイル内の関数は、共有バッファの状態を直接操作する「ハック」を含みます。本番コードの参考にはしないでください。
  • **デバッグ**: 注入ポイントが有効な場合、ログに詳細が出力されます。テスト失敗時は log_min_messages = LOG に設定して確認してください。
  • **制約**: read_rel_block_ll は PG_IOV_MAX までのブロック数しか扱えません。また、排他ロックを多用するため、テスト実行時は並行性に注意が必要です。
