src/test/modules/worker_spi/worker_spi.c

概要

PostgreSQLのバックグラウンドワーカー（Background Worker）機能を利用したサンプル実装です。データベースへの接続、SPI（Server Programming Interface）を通じたSQL実行、GUC変数による設定管理、およびシグナル処理のベストプラクティスを網羅しています。

モジュール内での役割

src/test/modules/worker_spi モジュールの中心的な実装ファイルです。バックグラウンドワーカーの登録から実行、終了までのライフサイクルを管理し、テストや学習のためのリファレンスとして機能します。

責務

  • **ワーカーのライフサイクル管理**: _PG_init によるワーカーの登録と、worker_spi_main によるプロセス実行。
  • **SPIによるデータ操作**: データベース接続、スキーマ/テーブルの自動生成、および定期的な集計処理。
  • **設定とシグナル**: GUC変数による動的設定変更（SIGHUP）への対応と、Postmaster終了時の安全な停止。
  • **動的起動**: worker_spi_launch 関数による、実行中のバックグラウンドワーカーの動的生成。

重要なシンボル

  • worker_spi_main: ワーカープロセスのメインループ。
  • initialize_worker_spi: 起動時に必要なスキーマとテーブルを構築する初期化関数。
  • _PG_init: モジュールロード時にワーカーを登録するエントリーポイント。
  • worker_spi_launch: SQLから呼び出し可能な、ワーカーを動的に起動する関数。

追加の索引済みシンボル

制御フローとデータフロー

1. **初期化**: _PG_init がワーカーを登録。各ワーカーは worker_spi_main を開始。 2. **接続**: worker_spi_main が BackgroundWorkerInitializeConnection を呼び出し、DBセッションを確立。 3. **セットアップ**: initialize_worker_spi がトランザクション内でテーブル構造を構築。 4. **メインループ**: WaitLatch で指定時間待機し、CHECK_FOR_INTERRUPTS でシグナルを処理。SPI経由で DELETE と UPDATE を組み合わせた集計クエリを定期実行。 5. **終了**: SIGTERM を受信するとループを抜け、プロセスが終了。

読者向けメモ

  • **待機処理**: usleep 等の直接呼び出しは避け、必ず WaitLatch を使用してください。これによりPostmasterの終了を即座に検知できます。
  • **トランザクション**: SPIを使用する際は、StartTransactionCommand と CommitTransactionCommand で明示的に囲む必要があります。
  • **メモリ管理**: worker_spi_main 内でのメモリ割り当ては、プロセス終了時にクリーンアップされるため、長時間実行されるワーカーではメモリリークに注意してください。
  • **デバッグ**: debug_query_string を適切に設定することで、pg_stat_activity ビューからワーカーが実行中のクエリを可視化できます。
