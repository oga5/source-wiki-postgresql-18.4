src/backend/postmaster/syslogger.c

概要

syslogger.c は、PostgreSQL のログ収集プロセス（SysLogger）を実装するモジュールです。Postmaster および全バックエンドプロセスから出力される stderr をパイプ経由で集約し、設定に基づいたローテーション（サイズ・時間）を行いながらファイルへ書き出します。

モジュール内での役割

Postmaster の補助プロセスの一つとして動作します。他のプロセスが直接ファイルへ書き込むのではなく、パイプ経由で本プロセスにログを転送することで、ログファイルの排他制御やローテーションの複雑さを一元化しています。

責務

  • **ログ集約**: パイプ経由で送られてくる各プロセスのログを、PID 単位で再構成（プロトコル解析）して集約。
  • **ログローテーション**: 設定（Log_RotationAge, Log_RotationSize）に基づき、ログファイルを切り替える。
  • **メタデータ管理**: 現在使用中のログファイル名を外部から参照可能なファイル（current_logfiles）に書き出す。
  • **プラットフォーム適応**: Windows と Unix で異なるパイプ処理やスレッド/プロセスモデルを抽象化。

重要なシンボル

プロセス管理・メインループ

  • SysLoggerMain: プロセスのメインエントリポイント。イベントループを回し、ログの読み込みとローテーションを制御。
  • SysLogger_Start: Postmaster が SysLogger を起動する際に呼び出す関数。パイプの作成と初期ファイルオープンを担当。

ログ処理・プロトコル

  • process_pipe_input: パイプから読み込んだデータを解析し、PipeProtoHeader に基づいてメッセージを再構築。
  • write_syslogger_file: 最終的なログファイルへの書き込みを行う。elog.c からも呼び出される。
  • save_buffer: PID ごとに断片化されたログを一時保持するための構造体。buffer_lists で管理。

ローテーション・設定

  • logfile_rotate: ローテーションの実行関数。
  • set_next_rotation_time: 次回の時間ベースのローテーション時刻を計算。
  • update_metainfo_datafile: current_logfiles を更新し、外部ツールが現在のログパスを追跡できるようにする。

制御フローとデータフロー

1. **初期化**: SysLogger_Start がパイプを作成し、Postmaster が stderr をパイプの書き込み端にリダイレクト。 2. **集約**: SysLoggerMain が WaitEventSet を使用してパイプの読み込みを待機。 3. **解析**: process_pipe_input がパイプから PipeProtoHeader を読み取り、PID 単位で save_buffer に蓄積。PIPE_PROTO_IS_LAST フラグが立つと、完全なメッセージとして write_syslogger_file へ渡す。 4. **ローテーション**: SIGHUP（設定変更）またはサイズ/時間制限に達した際、logfile_rotate が新しいファイルハンドルを開き、古いものを閉じる。 5. **終了**: パイプの EOF を検知すると、全バックエンドが終了したと判断し proc_exit を呼び出す。

読者向けメモ

  • **メモリ所有権**: save_buffer 内の StringInfo は palloc されており、メッセージ完了時に pfree されます。
  • **Windows 特有の挙動**: Windows ではパイプの読み込みに専用スレッド (pipeThread) を使用します。これは ReadFile がブロッキングするためで、メインスレッドがシグナルやラッチを処理できるようにするためです。
  • **エラーパス**: ログファイルオープン失敗時は ereport(FATAL) が基本ですが、ローテーション時の失敗は rotation_disabled フラグを立てて継続を試みます。
  • **制約**: syslogPipe は Postmaster から継承されるため、Postmaster 終了後も SysLogger が生き残ることで、全プロセスの「最後の喘ぎ」まで確実にログを回収する設計になっています。
