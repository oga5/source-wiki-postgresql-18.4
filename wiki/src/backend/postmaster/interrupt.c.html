src/backend/postmaster/interrupt.c

概要

PostgreSQLのバックグラウンドプロセス（Postmaster配下の各ワーカ）における、シグナル受信時の共通的な割り込み処理ルーチンを提供します。

モジュール内での役割

Postmasterが管理する各プロセスが、シグナル（SIGHUP, SIGTERM, SIGQUIT等）を安全かつ一貫した方法で処理するためのユーティリティ層です。個別のプロセスが複雑なシグナルハンドラを個別に実装する負担を軽減します。

責務

  • **フラグ管理**: シグナル受信時に sig_atomic_t 型のフラグを立て、メインループでの処理を予約する。
  • **ラッチの通知**: SetLatch を呼び出し、シグナル受信時に待機中のプロセスを即座に起床させる。
  • **定型処理の実行**: 設定再読み込みやシャットダウン要求などの標準的な割り込み処理を、メインループの適切なタイミングで実行する。
  • **緊急終了**: 共有メモリ破損時などを想定した、クリーンアップを行わない即時終了処理の提供。

重要なシンボル

  • ConfigReloadPending: 設定再読み込み（SIGHUP）が要求されたことを示すフラグ。
  • ShutdownRequestPending: プロセスのシャットダウン（SIGTERM等）が要求されたことを示すフラグ。
  • ProcessMainLoopInterrupts(): メインループ内で呼び出される、保留中の割り込みを処理する統合関数。
  • SignalHandlerForCrashExit(): _exit(2) を呼び出し、クリーンアップなしでプロセスを終了させるハンドラ。

制御フローとデータフロー

1. **シグナル受信**: OSからシグナルを受け取ると、SignalHandlerFor... 系関数が実行されます。 2. **フラグ設定**: ハンドラ内で Pending フラグを true にし、SetLatch でプロセスを起床させます。 3. **メインループでの検知**: プロセスはメインループの反復ごとに ProcessMainLoopInterrupts を呼び出します。 4. **処理の実行**: ProcessMainLoopInterrupts 内でフラグを確認し、ProcessConfigFile や proc_exit などの実処理を実行します。

読者向けメモ

  • 本ファイルで定義されるハンドラは、シグナルハンドラ内での非同期安全性を考慮し、最小限の操作（フラグ設定とラッチ通知）のみを行う設計になっています。
  • SignalHandlerForCrashExit は、共有メモリの破損を想定しているため、proc_exit（クリーンアップ処理）を意図的に回避しています。この挙動は、Postmasterによるシステムリセットを誘発するための重要な設計です。
  • 独自のバックグラウンドプロセスを実装する際は、メインループ内で ProcessMainLoopInterrupts を定期的に呼び出すことで、これらの標準的なシグナル処理を容易に組み込めます。
