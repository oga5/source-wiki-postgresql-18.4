src/backend/bootstrap/bootstrap.c

概要

PostgreSQLの初期化プロセス（initdb）において、システムカタログを構築するための「ブートストラップモード」を実装するファイルです。SQLを解釈するのではなく、BKI（Backend Interface）と呼ばれる専用の低レベル言語を解析・実行し、データベースのテンプレートを作成します。

モジュール内での役割

src/backend/bootstrap モジュールの中心であり、システムカタログの初期スキーマ定義とデータ挿入を担います。通常のバックエンドとは異なり、システムインデックスが未構築の状態でも動作するように設計されています。

責務

  • **BKIコマンドの実行**: OPEN, CREATE, INSERT などのBKIコマンドを処理し、システムカタログを構築する。
  • **型情報の管理**: pg_type が構築される前後の過渡期において、データ型の属性（サイズ、アライメント等）を解決する。
  • **インデックスの遅延構築**: カタログ定義時にインデックスを登録し、全てのカタログ定義完了後に一括で構築する。
  • **ブートストラップ環境の初期化**: スタンドアロンプロセスとしてのメモリ管理、信号処理、GUC設定の適用。

重要なシンボル

  • BootstrapModeMain: ブートストラップモードのメインエントリポイント。
  • boot_reldesc: 現在操作中のリレーション記述子。
  • TypInfo: pg_type 構築前に使用されるハードコードされた基本型定義。
  • DefineAttr: カタログの属性定義を行う関数。
  • InsertOneTuple: 構築中のリレーションにタプルを挿入する関数。
  • index_register / build_indices: インデックス構築の二段階処理（登録と実行）を管理。

制御フローとデータフロー

1. **初期化**: BootstrapModeMain が引数を解析し、GUCを設定。InitPostgres を経てブートストラップ環境を構築。 2. **解析**: boot_yyparse（Lex/Yacc）を呼び出し、BKIファイルを読み込む。 3. **カタログ構築**:

  • DefineAttr で属性を定義。
  • InsertOneValue / InsertOneTuple でデータを挿入。
  • gettype が必要に応じて pg_type を読み込み、型情報をキャッシュする。

    • 4. **終了処理**: 全ての定義完了後、RelationMapFinishBootstrap でリレーションマップを書き出し、cleanup を経て終了。

読者向けメモ

  • **ブートストラップの特殊性**: このファイルは「鶏と卵」の問題を解決するために存在します。システムカタログ自体を構築するために、カタログの定義をハードコードしたり、段階的に読み込んだりする複雑なロジックが含まれています。
  • **型情報の解決**: gettype 関数は、pg_type がまだ存在しない初期段階と、存在した後の段階で挙動が変わるため、読み解く際は現在の Typ リストの状態に注意してください。
  • **デバッグ**: DEBUG4 ログが随所に埋め込まれており、initdb 実行時に詳細な構築過程を追跡可能です。
