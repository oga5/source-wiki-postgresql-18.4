src/backend/replication/logical/applyparallelworker.c

概要

論理レプリケーションにおけるストリーミングトランザクションの並列適用をサポートするモジュールです。リーダー適用ワーカー（Leader Apply Worker）が受信した変更データを、動的共有メモリ（DSM）を介して並列適用ワーカー（Parallel Apply Worker）へ転送し、サブスクライバー側で並列に適用することでスループットを向上させます。

モジュール内での役割

logical/worker.c が管理する論理レプリケーション適用プロセスを拡張し、大規模トランザクションのストリーミング時にタスクを分散させるためのインフラを提供します。リーダーとワーカー間の通信、ワーカーのライフサイクル管理、およびデッドロック回避のためのロック制御を担います。

責務

  • **ワーカー管理**: ワーカーの起動、再利用のためのプール管理、および終了処理。
  • **通信インフラ**: DSMを用いたメッセージキュー（shm_mq）の構築と、リーダー・ワーカー間のデータ転送。
  • **デッドロック制御**: 適用順序の維持と、競合するトランザクション間のデッドロックを検出するためのセッションレベルロック（PARALLEL_APPLY_LOCK_STREAM, PARALLEL_APPLY_LOCK_XACT）の管理。
  • **フォールバック処理**: メッセージキューが溢れた際のファイルへのシリアライズ（スプール）と、それらの再読み込み。

重要なシンボル

ワーカー管理と状態

  • ParallelApplyWorkerPool: 起動済みのワーカーを保持するリスト。再利用を促進し、起動コストを削減する。
  • ParallelApplyTxnHash: トランザクションID（XID）とワーカー情報を紐付けるハッシュテーブル。
  • ParallelApplyWorkerInfo: ワーカーのDSMセグメント、キュー、状態を保持する構造体。
  • MyParallelShared: ワーカープロセスから参照される、リーダーと共有される状態領域。

ロック制御

  • pa_lock_stream() / pa_unlock_stream(): ストリーム転送の同期用ロック。
  • pa_lock_transaction() / pa_unlock_transaction(): トランザクション適用順序とデッドロック検出用ロック。

ライフサイクル・通信

  • pa_launch_parallel_worker(): ワーカーの起動とDSMのセットアップ。
  • pa_send_data(): リーダーからワーカーへデータを送信（非ブロッキング、タイムアウト付き）。
  • LogicalParallelApplyLoop(): ワーカー側のメインループ。メッセージ受信と適用ディスパッチを行う。

制御フローとデータフロー

1. **起動**: リーダーが pa_allocate_worker() を呼び出し、DSMを確保して LogicalParallelApplyLoop を持つワーカーを起動。 2. **データ転送**: リーダーは pa_send_data() でメッセージキューへデータを投入。キューが満杯の場合、pa_switch_to_partial_serialize() によりファイルへスプールし、ワーカーは pa_process_spooled_messages_if_required() でそれを読み込む。 3. **同期**: リーダーは pa_xact_finish() でワーカーの完了を待ち、コミット順序を保証する。 4. **デッドロック回避**: リーダーとワーカーは lmgr ロックを相互に取得・解放することで、依存関係のあるトランザクションのデッドロックを PostgreSQL のデッドロック検出器に認識させる。

読者向けメモ

  • **メモリ所有権**: ParallelApplyWorkerInfo は ApplyContext 上で永続的に管理されます。DSMセグメントはワーカー終了時に pa_shutdown で適切にデタッチされます。
  • **デッドロック検出**: 本モジュールは XactLockTableWait を直接使わず、セッションレベルのロックを介して lmgr に依存しています。これは準備済みトランザクション（Prepared Transaction）の扱いを考慮した設計です。
  • **制限事項**: 現在、ワーカープールの上限は max_parallel_apply_workers_per_subscription / 2 とハードコードされています。
  • **省略箇所**: 本ファイルは通信と管理に特化しており、実際のSQL適用ロジック（apply_dispatch）は worker.c 等の外部モジュールに依存しています。
