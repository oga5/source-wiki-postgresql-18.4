src/backend/replication/logical/logical.c

概要

本ファイルは、PostgreSQLの論理デコーディングにおける調整役です。WALから変更データを抽出し、出力プラグインへ渡すまでのライフサイクルを管理します。LogicalDecodingContext を中心に、WAL読み込み、トランザクション再構築（ReorderBuffer）、スナップショット構築（SnapBuild）を統合し、消費者が複雑な内部処理を意識せずに変更ストリームを扱えるように抽象化を提供します。

モジュール内での役割

論理レプリケーションサブシステムにおける「デコーディング・エンジン」のフロントエンドです。reorderbuffer.c（トランザクション管理）や snapbuild.c（カタログ可視性）といった低レイヤーのコンポーネントと、外部プラグインやSQLインターフェース（logicalfuncs.c）の橋渡しを行います。

責務

  • **コンテキスト管理**: LogicalDecodingContext の生成、初期化、および破棄。
  • **プラグイン連携**: 出力プラグインの動的ロードと、各コールバック（begin, change, commit等）のラッパー提供。
  • **エラーハンドリング**: プラグイン実行時のエラーコンテキスト（LSNやコールバック名）のスタック管理。
  • **スロット状態の更新**: confirmed_flush、catalog_xmin、restart_lsn の進捗管理と永続化。
  • **ストリーミング制御**: 大規模トランザクションのストリーミングおよび2フェーズコミットのデコーディング制御。

重要なシンボル

コンテキスト管理

  • LogicalDecodingContext: デコーディングセッションの全状態を保持する構造体。
  • CreateInitDecodingContext: 新規スロット用のコンテキスト作成。
  • CreateDecodingContext: 既存スロット用のコンテキスト作成。
  • FreeDecodingContext: リソース解放とシャットダウン処理。

プラグイン・コールバックラッパー

  • startup_cb_wrapper / shutdown_cb_wrapper: プラグインのライフサイクル管理。
  • begin_cb_wrapper / commit_cb_wrapper: トランザクション境界の処理。
  • change_cb_wrapper / truncate_cb_wrapper: DML/DDL変更の通知。
  • stream_*_cb_wrapper: ストリーミング転送用コールバック群。

スロット進捗管理

  • LogicalConfirmReceivedLocation: 消費者が処理したLSNを記録し、restart_lsn や catalog_xmin を進める。
  • LogicalIncreaseXminForSlot: カタログの安全なxmin horizonを更新。
  • LogicalIncreaseRestartDecodingForSlot: WAL保持のための restart_lsn を更新。

制御フローとデータフロー

1. **初期化**: Create*DecodingContext が ReorderBuffer と SnapBuild を初期化し、プラグインをロードします。 2. **WAL読み込み**: DecodingContextFindStartpoint が XLogReadRecord をループし、LogicalDecodingProcessRecord を通じて SnapBuild が一貫性のある状態になるまでWALを消化します。 3. **デコード実行**: ReorderBuffer がWALレコードを蓄積・整列し、コミット時に *_cb_wrapper を呼び出します。 4. **ラッパー経由の出力**: 各ラッパーは error_context_stack に情報をプッシュし、プラグインのコールバックを実行します。プラグインは OutputPluginPrepareWrite / OutputPluginWrite を介してデータを書き出します。 5. **進捗報告**: 消費者が LogicalConfirmReceivedLocation を呼び出すと、スロットのメタデータが更新され、不要なWALや古いカタログ行が掃除可能になります。

読者向けメモ

  • **メモリ管理**: LogicalDecodingContext は専用の MemoryContext を持ちます。プラグインのコールバック実行時には、このコンテキストがアクティブになるよう注意が必要です。
  • **エラーハンドリング**: プラグイン内でエラーが発生した場合、output_plugin_error_callback がスタックに積まれているため、どのLSNで失敗したかがログに詳細に出力されます。
  • **制約**: fast_forward モードは、変更データを生成せずスロットの進捗のみを進めるための軽量モードです。
  • **省略箇所**: 本ファイルは調整役であるため、WALレコードの具体的な解析ロジック（LogicalDecodingProcessRecord 等）は decode.c に委譲されています。また、並列適用ワーカーの詳細は applyparallelworker.c を参照してください。
