src/backend/storage/freespace/fsmpage.c

概要

fsmpage.c は、PostgreSQL の空き領域マップ（FSM: Free Space Map）における個々のページ内のデータ構造を操作するための低レベル API を提供します。FSM ページは、各スロットの空き容量を保持するバイナリツリー構造として実装されており、本ファイルはこのツリーの検索、更新、および整合性維持を担います。

モジュール内での役割

src/backend/storage/freespace モジュールにおいて、本ファイルは「ページ単位の操作」を抽象化するレイヤーです。上位の freespace.c が FSM 全体の構造やファイル管理を行うのに対し、本ファイルは特定のページバッファを受け取り、その内部のノード値を操作する役割を果たします。

責務

  • **ツリー操作**: ページ内のバイナリツリー構造（親・子ノードのインデックス計算）の管理。
  • **検索最適化**: 指定された空き容量以上のスロットを効率的に探索するアルゴリズムの提供。
  • **整合性維持**: スロット値の更新に伴うツリーの再帰的な更新（伝播）および、破損時のページ再構築。
  • **カプセル化**: ページ内部のノード配置を隠蔽し、スロット単位の読み書きインターフェースを提供。

重要なシンボル

  • fsm_set_avail(): 特定スロットの値を更新し、親ノードへ値を伝播させる。
  • fsm_search_avail(): 指定した空き容量を満たすスロットを、fp_next_slot をヒントに効率的に探索する。
  • fsm_rebuild_page(): ページ内の全ノードを再計算し、ツリーの整合性を強制的に修正する。
  • leftchild() / rightchild() / parentof(): ツリー構造をナビゲートするためのマクロ。

制御フローとデータフロー

1. **検索**: fsm_search_avail は、まずルートノードを確認して探索の可否を判断します。その後、fp_next_slot を起点に右方向へ移動しつつ、親ノードへ登ることで「探索三角形」を拡大し、対数時間で候補を見つけます。 2. **更新**: fsm_set_avail は、リーフノードの値を変更した後、親ノードを辿りながら最大値を更新します。 3. **破損検知**: 探索中に期待値と実際の値が一致しない場合（torn page 等）、fsm_rebuild_page を呼び出してツリーを再構築し、探索をリトライします。

読者向けメモ

  • **ロックの扱い**: fsm_get_avail や fsm_get_max_avail は単一バイトの読み取りであるためロック不要ですが、更新を伴う関数は適切なバッファロックを前提としています。
  • **探索のヒント**: fp_next_slot はあくまでヒントであり、厳密な順序を保証するものではありません。これにより、排他ロックの競合を最小限に抑えています。
  • **再構築**: fsm_rebuild_page はコストが高い操作です。破損が疑われる場合にのみ実行される安全装置として機能します。
