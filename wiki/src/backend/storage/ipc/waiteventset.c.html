src/backend/storage/ipc/waiteventset.c

概要

waiteventset.c は、PostgreSQL における「複数のイベント（ソケット、ラッチ、プロセス終了など）を同時に待機する」ための抽象化レイヤーを提供します。OS 固有の多重化プリミティブ（epoll, kqueue, poll, WaitForMultipleObjects）をラップし、プラットフォーム間で一貫したイベント待機インターフェースを実現します。

モジュール内での役割

src/backend/storage/ipc モジュールにおいて、プロセス間同期（latch.c）と並び、バックエンドプロセスが効率的に待機状態に入るための基盤です。WaitLatch などの旧来の関数よりも、長寿命なイベントセットを保持することで、システムコール回数を削減し、スケーラビリティを向上させます。

責務

  • **イベント多重化**: OS 固有の待機プリミティブを抽象化し、WaitEventSet を通じて統一的な待機処理を提供。
  • **レースコンディションの回避**: poll() 等で発生する「シグナル受信と待機開始の間の競合」を、セルフパイプ（Self-pipe）や signalfd を用いて解決。
  • **リソース管理**: ResourceOwner を介した WaitEventSet のライフサイクル管理（メモリ解放、FD クローズ）。
  • **プラットフォーム抽象化**: Windows と Unix 系の差異を吸収し、特にシグナル処理やソケットイベントの通知方式を隠蔽。

重要なシンボル

ライフサイクル管理

  • CreateWaitEventSet: WaitEventSet 構造体のメモリ確保と初期化。
  • FreeWaitEventSet: リソースの解放と ResourceOwner からの登録解除。
  • InitializeWaitEventSupport: プロセス起動時に必要なセルフパイプや signalfd を準備。

イベント操作

  • AddWaitEventToSet: イベント（ソケット、ラッチ等）をセットに追加。
  • ModifyWaitEvent: 既存イベントのマスクやラッチの関連付けを変更。

待機処理

  • WaitEventSetWait: 指定されたイベントのいずれかが発生するまで待機。
  • WaitEventSetWaitBlock: 各 OS プリミティブ（epoll_wait, kevent 等）を呼び出す内部関数。

内部状態

  • WaitEventSet: 待機対象イベントの配列、OS 固有の FD、リソース所有者情報を保持するメイン構造体。

制御フローとデータフロー

1. **初期化**: InitializeWaitEventSupport がプロセスごとに一度呼ばれ、OS 固有の通知用 FD（セルフパイプ等）を確保します。 2. **登録**: AddWaitEventToSet でイベントを登録し、OS 固有の調整関数（WaitEventAdjustEpoll 等）を介してカーネルへイベントを登録します。 3. **待機**: WaitEventSetWait が呼ばれると、WaitEventSetWaitBlock が OS の待機プリミティブを呼び出します。 4. **起床**:

  • ラッチがセットされると、latch_sigurg_handler がセルフパイプに書き込み、待機中のシステムコールを中断させます。
  • drain() 関数により、セルフパイプや signalfd からデータを読み出し、待機状態をクリアします。

    • 5. **結果返却**: 発生したイベントを WaitEvent 配列に格納して呼び出し元へ返します。

読者向けメモ

  • **メモリ所有権**: WaitEventSet は TopMemoryContext に割り当てられ、ResourceOwner に紐付けられます。プロセス終了時に自動的にクリーンアップされます。
  • **セルフパイプの重要性**: Unix 系で poll() を使用する場合、シグナルハンドラから write() を行うことで poll() を確実に起床させる「セルフパイプ・トリック」が不可欠です。
  • **Windows の挙動**: Windows では WaitForMultipleObjects を使用し、シグナルイベントを配列の先頭に置くことで、他のイベントよりも優先的にシグナルを処理する設計になっています。
  • **省略された詳細**: 各 OS 固有の WaitEventAdjust* 関数群は、カーネルへのイベント登録ロジックの詳細であり、本質的なフローを理解する上では「OS への登録・更新を行う」という抽象的な理解で十分です。
