src/backend/storage/ipc/latch.c

概要

latch.c は、PostgreSQL におけるプロセス間通信（IPC）および同期のための「ラッチ（Latch）」メカニズムを実装するファイルです。ラッチは、プロセスが特定のイベント（シグナルや共有メモリ上のフラグ更新など）を待機してスリープし、イベント発生時に効率的に起床するための軽量な同期プリミティブです。

モジュール内での役割

src/backend/storage/ipc モジュールにおいて、プロセスが待機状態に入る際の標準的なインターフェースを提供します。waiteventset.c と密接に連携し、OS 固有の待機システムコール（epoll, kqueue, select 等）を抽象化して、シグナルハンドラや他プロセスからの通知を確実に受け取れるようにします。

責務

  • **ラッチのライフサイクル管理**: ローカルラッチおよび共有ラッチの初期化、所有権の割り当て（OwnLatch）、解放（DisownLatch）。
  • **待機と通知**: WaitLatch による待機処理と、SetLatch による他プロセス（または自身）への起床通知。
  • **競合の回避**: メモリバリアを用いたフラグ更新の順序保証と、シグナルハンドラ内での安全な通知処理。

重要なシンボル

  • Latch: ラッチの状態を保持する構造体。is_set（状態）、owner_pid（所有者）、is_shared（共有フラグ）を含む。
  • SetLatch(Latch *latch): ラッチをセットし、待機中のプロセスを起床させる。シグナルハンドラから呼び出し可能。
  • WaitLatch(...): ラッチがセットされるか、タイムアウト、またはポストマスターの終了を待機する。
  • ResetLatch(Latch *latch): ラッチをクリアする。待機ループの開始時に呼び出す必要がある。

制御フローとデータフロー

1. **待機側**: ResetLatch で状態をリセットし、WaitLatch を呼び出して待機状態へ移行します。WaitLatch は内部で WaitEventSet を使用し、OS の待機プリミティブに登録されます。 2. **通知側**: SetLatch が呼び出されると、is_set フラグが真になり、必要に応じて kill() 等のシグナル送信や Windows の SetEvent を通じて待機側プロセスを起床させます。 3. **メモリバリア**: SetLatch と ResetLatch の前後で pg_memory_barrier() を使用し、フラグの更新が他プロセスから正しく観測されることを保証します。

読者向けメモ

  • **コーディングパターン**: WaitLatch を使用する際は、必ず ResetLatch をループの適切な位置（通常は WaitLatch の前）に配置してください。これを怠ると、シグナルが ResetLatch の直前に届いた場合にイベントを見逃す競合が発生します。
  • **シグナルハンドラ**: SetLatch はシグナルハンドラから呼び出されることを想定しています。そのため、内部で errno を破壊しないよう注意が払われていますが、呼び出し側でも必要に応じて errno の保存・復元を行うのが安全です。
  • **共有ラッチ**: 共有メモリ上のラッチを扱う際は、初期化順序と所有権の移譲に注意してください。OwnLatch はロックを伴わないため、呼び出し側の排他制御が必要です。
