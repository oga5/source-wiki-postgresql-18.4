src/backend/storage/ipc/shm_mq.c

概要

shm_mq.c は、PostgreSQL のバックエンドプロセス間でメッセージをやり取りするための、単一リーダー・単一ライターの共有メモリメッセージキューを実装しています。主にバックグラウンドワーカーとの通信に使用されます。

モジュール内での役割

src/backend/storage/ipc モジュールの一部として、プロセス間通信（IPC）の抽象化を提供します。共有メモリ（DSM）上に構築され、プロセス間の同期には Latch を利用します。

責務

  • **キューの管理**: 共有メモリ上のリングバッファ構造の初期化と管理。
  • **同期制御**: 送信側・受信側の PGPROC を介した Latch によるブロッキング/ウェイクアップ制御。
  • **データ転送**: メッセージの断片化や再構築を透過的に行い、必要に応じてローカルバッファへのコピーを行う。
  • **ライフサイクル管理**: プロセスの終了やデタッチ時のクリーンアップと、相手側への通知。

重要なシンボル

  • struct shm_mq: 共有メモリ上に配置されるキュー本体。
  • struct shm_mq_handle: 各バックエンドが保持するキューへのアクセス用プライベートハンドル。
  • shm_mq_send / shm_mq_receive: メッセージ送受信の主要インターフェース。
  • shm_mq_attach / shm_mq_detach: キューへの接続と切断。

制御フローとデータフロー

1. **初期化**: shm_mq_create で共有メモリ領域を初期化し、送信側・受信側を shm_mq_set_sender/receiver で登録します。 2. **送受信**:

  • 送信側は shm_mq_send を呼び出し、リングバッファの空き領域にデータを書き込みます。
  • 受信側は shm_mq_receive を呼び出し、バッファからデータを読み取ります。
  • バッファが満杯または空の場合、WaitLatch を使用して相手からの通知を待ちます。

    • 3. **効率化**: 頻繁な SetLatch を避けるため、一定量（リングサイズの1/4）のデータが溜まるまで共有メモリ上のポインタ更新を遅延させる仕組みがあります。 4. **終了**: shm_mq_detach により、相手側に mq_detached フラグを立てて通知し、リソースを解放します。

読者向けメモ

  • **ロックの最小化**: mq_bytes_read や mq_bytes_written はアトミック操作とメモリバリアで保護されており、スピンロックを極力避ける設計になっています。
  • **非同期性**: nowait = true を指定すると、ブロッキングせずに SHM_MQ_WOULD_BLOCK を返すため、イベントループ内での利用が可能です。
  • **デバッグ**: 相手プロセスが予期せず終了した場合、BackgroundWorkerHandle を介して検知する仕組みがあるため、デッドロックを回避できます。
