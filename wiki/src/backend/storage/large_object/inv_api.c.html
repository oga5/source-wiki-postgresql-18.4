src/backend/storage/large_object/inv_api.c

概要

PostgreSQLの「インバージョン（Inversion）」ファイルシステムに基づくラージオブジェクト（LO）を操作するためのユーザーレベルAPIを提供します。pg_largeobject カタログテーブルを直接操作し、ストリームのような読み書きインターフェースを実現します。

モジュール内での役割

src/backend/storage/large_object ディレクトリにおける主要な実装ファイルです。SQLレベルの lo_open や lo_read 等の関数が内部で呼び出すバックエンド側の実体であり、システムカタログとユーザー空間の橋渡しを行います。

責務

  • **LOのライフサイクル管理**: 作成 (inv_create)、オープン (inv_open)、クローズ (inv_close)、削除 (inv_drop)。
  • **データアクセス**: pg_largeobject テーブルに対するページ単位の読み込み (inv_read)、書き込み (inv_write)、切り詰め (inv_truncate)。
  • **権限チェック**: 各操作におけるACL（アクセス制御リスト）の検証。
  • **互換性維持**: 過去の仕様に基づくリソース管理や、pg_largeobject のデータフィールド（TOAST済みデータ）のデコード。

重要なシンボル

  • LargeObjectDesc: オープンされたLOの状態（ID、オフセット、フラグ、スナップショット）を保持する構造体。
  • inv_open: LargeObjectDesc を生成し、権限チェックとスナップショットの初期化を行う。
  • inv_read / inv_write: 指定されたオフセットに基づき、pg_largeobject のページをスキャンしてデータを読み書きする。
  • lo_heap_r / lo_index_r: pg_largeobject テーブルとインデックスへのキャッシュされたリレーション参照。

制御フローとデータフロー

1. **初期化**: open_lo_relation() が呼ばれると、pg_largeobject テーブルとインデックスが現在のトランザクションでオープンされます。 2. **スキャン**: systable_beginscan_ordered を使用し、loid と pageno をキーにして必要なページを順次取得します。 3. **データ処理**:

  • 読み込み時: ページ間の「穴（hole）」をゼロ埋めし、TOASTされたデータを detoast_attr で展開してバッファにコピーします。
  • 書き込み時: 既存ページを更新するか、新しいページを挿入します。必要に応じて CatalogTupleUpdateWithInfo や CatalogTupleInsertWithInfo を使用してカタログを更新します。

    • 4. **終了**: CommandCounterIncrement() を呼び出し、変更を後続の操作から可視化します。

読者向けメモ

  • **メモリ管理**: inv_open で割り当てられた LargeObjectDesc は、呼び出し元が適切なメモリコンテキストで管理する必要があります。
  • **TOAST**: pg_largeobject.data はTOASTされている可能性があるため、getdatafield 内で必ずデトースト処理が行われます。
  • **トランザクション**: lo_heap_r 等のリレーション参照は、トランザクション終了時に close_lo_relation で適切にクローズされるよう設計されています。
  • **ページング**: LOは LOBLKSIZE（通常2KB）単位で管理されます。ファイルシステムと同様に、書き込み時にページが飛んでいる場合は「穴」としてゼロ埋めされます。
