src/backend/storage/file/fd.c

概要

本ファイルは、PostgreSQLにおける「仮想ファイル記述子（VFD）」管理の中核です。OSのファイル記述子（FD）制限を回避するためのLRUキャッシュ機構を提供し、データベース内の物理ファイルI/Oを抽象化します。また、一時ファイルのライフサイクル管理、ディレクトリ操作、およびクラッシュセーフなファイル操作（fsync、durable_rename等）の基盤を担います。

モジュール内での役割

storage/file サブシステムの最下層に位置し、OSの低レベルI/O（open, read, write, close）をラップします。buffile.c や sharedfileset.c などの上位モジュールは、本ファイルが提供する File 型（VFDインデックス）を通じてファイルにアクセスします。

責務

  • **VFD管理**: LRUアルゴリズムを用いたFDの動的開閉と、プロセスごとのFD制限（max_safe_fds）の遵守。
  • **一時ファイル管理**: トランザクション終了時の自動クリーンアップ、一時テーブルスペースへの分散配置、およびサイズ制限の監視。
  • **耐久性保証**: fsync のラッパー提供、および durable_rename / durable_unlink によるクラッシュ後の整合性維持。
  • **リソース追跡**: ResourceOwner を用いた、エラー発生時やトランザクション終了時のファイル記述子リーク防止。

重要なシンボル

VFD管理

  • Vfd: 仮想ファイル記述子の状態を保持する構造体。LRUリストのリンク、ファイル名、現在のFD、リソース所有者情報を含む。
  • VfdCache: VFDを格納する動的配列。File 型のインデックスでアクセスされる。
  • max_safe_fds: プロセスが同時に開ける最大FD数。set_max_safe_fds で初期化される。

ファイル操作インターフェース

  • PathNameOpenFile: 指定パスのファイルをVFDとして開く。
  • OpenTemporaryFile: 一時ファイルを作成し、自動削除とリソース所有権を登録する。
  • FileReadV / FileWriteV: iovec を用いた分散I/O操作。
  • FileSync: pg_fsync を呼び出し、データの永続化を行う。

外部FD管理（Allocate系）

  • AllocateFile / AllocateDir: fopen/opendir のラッパー。ResourceOwner に登録され、自動クローズ対象となる。
  • OpenTransientFile: open のラッパー。短期間のファイル操作用。

制御フローとデータフロー

1. **ファイルアクセス**: FileAccess(File) が呼び出されると、対象VFDがオープン済みか確認する。未オープンなら LruInsert を呼び出し、必要に応じて ReleaseLruFiles で古いFDを閉じてから open を実行する。 2. **LRU更新**: ファイルアクセスが発生するたびに、Delete と Insert を用いて、該当VFDをLRUリストの先頭（Most Recently Used）へ移動させる。 3. **一時ファイル**: OpenTemporaryFile は PathNameOpenFile を呼び出した後、FD_DELETE_AT_CLOSE フラグを立て、ResourceOwner に登録する。これにより、FileClose またはトランザクション終了時に自動的に unlink される。 4. **耐久性**: durable_rename は、操作対象のファイルと親ディレクトリに対して fsync を発行し、OSクラッシュ時にファイルシステムが不整合な状態にならないよう保証する。

直接依存

  • storage/fd.h: 公開インターフェース定義。
  • utils/resowner.h: リソース所有権管理（自動クローズ用）。
  • access/xact.h: トランザクション終了時のクリーンアップフック。
  • storage/aio.h: 非同期I/O連携。
  • 標準ヘッダー: <fcntl.h>, <unistd.h>, <sys/stat.h> 等の低レベルI/O用。

関連コード

  • buffile.c: File 型を基盤としたバッファ付きI/O。
  • sharedfileset.c: プロセス間で共有される一時ファイルの管理。
  • reinit.c: アンログドテーブルのリカバリ処理。

読者向けメモ

  • **メモリ所有権**: Vfd 構造体内の fileName は malloc されており、FreeVfd で解放されます。
  • **エラーパス**: BasicOpenFilePerm は EMFILE/ENFILE を検知すると ReleaseLruFile を呼び出して再試行します。このため、ファイルオープンは「失敗する可能性がある」ことを前提に設計する必要があります。
  • **制約**: File 型は単なる int（VfdCache へのインデックス）であり、OSのFDとは異なります。直接 close() を呼ぶと内部状態が破壊されるため、必ず FileClose を使用してください。
  • **省略箇所**: 本ファイルには pg_fsync のプラットフォーム依存実装（F_FULLFSYNC 等）が含まれますが、これらはOSの耐久性保証モデルに依存するため、移植時には注意が必要です。
