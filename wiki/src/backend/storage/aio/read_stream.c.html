src/backend/storage/aio/read_stream.c

概要

read_stream.c は、PostgreSQL におけるリレーションデータの読み込みを効率化するための「先読み（Look-ahead）」メカニズムを提供します。単一ブロックごとの ReadBuffer() 呼び出しによる I/O 待機やオーバーヘッドを回避し、適応的な先読み距離を用いて複数のブロックをバッチ処理（StartReadBuffers / WaitReadBuffers）します。

モジュール内での役割

src/backend/storage/aio サブシステムの一部として、非同期 I/O（AIO）やバッチ処理を活用するための高レベルな抽象化レイヤーです。バッファマネージャと連携し、シーケンシャルアクセスやデータ駆動型のアクセスパターンにおいて、I/O 効率を最大化する役割を担います。

責務

  • **先読み制御**: キャッシュヒット率に基づき、先読み距離を動的に調整する。
  • **バッチ I/O 管理**: ユーザー提供のコールバックからブロック番号を取得し、可能な限り隣接ブロックをマージして StartReadBuffers を呼び出す。
  • **バッファ管理**: 循環キューを用いて、ピン留めされたバッファと進行中の I/O を追跡・管理する。
  • **リソース解放**: ストリーム終了時のバッファ解放およびメモリ管理。

重要なシンボル

  • ReadStream: ストリームの状態（キュー、先読み距離、進行中の I/O など）を保持するメイン構造体。
  • read_stream_begin_relation: ストリームを開始するための主要なエントリポイント。
  • read_stream_next_buffer: ストリームから次のバッファを取得する。必要に応じて I/O 待機や先読み処理をトリガーする。
  • read_stream_start_pending_read: 蓄積されたブロック要求を実際の I/O 操作へ変換する。
  • read_stream_look_ahead: ストリームの先読み距離を維持するために、コールバックからブロック番号を補充する。

制御フローとデータフロー

1. **初期化**: read_stream_begin_relation でストリームを生成し、コールバック関数を登録する。 2. **バッファ取得**: read_stream_next_buffer が呼ばれると、キュー内のバッファを返却する。 3. **先読み**: read_stream_look_ahead がコールバックを呼び出し、次に必要なブロックを特定する。 4. **I/O 実行**: 連続するブロックは pending_read に蓄積され、一定量に達するか制限に達した時点で StartReadBuffers が発行される。 5. **待機と消費**: WaitReadBuffers により I/O 完了を待ち、バッファを消費者に引き渡す。

読者向けメモ

  • **先読み距離**: キャッシュヒット時は距離が 1 に縮小され、I/O が発生すると指数関数的に増加します。
  • **バッチモード**: READ_STREAM_USE_BATCHING フラグを使用する場合、コールバック関数は pgaio_enter_batchmode() の制約（デッドロック回避など）を遵守する必要があります。
  • **Fast Path**: キャッシュヒット率が高い場合、キュー管理をスキップする最適化パスが有効になります。デバッグ時はこの挙動に注意してください。
