src/backend/storage/aio/method_io_uring.c

概要

Linux の io_uring インターフェースを利用した非同期 I/O (AIO) 実装を提供します。PostgreSQL のバックエンドプロセスごとに io_uring インスタンスを割り当て、高効率な I/O 処理を実現します。

モジュール内での役割

src/backend/storage/aio サブシステムにおける I/O メソッドの実装の一つです。IoMethodOps 構造体を通じて、汎用的な AIO インターフェースと io_uring 特有の低レイヤー処理を橋渡しします。

責務

  • **リソース管理**: バックエンドごとの io_uring コンテキストの初期化と共有メモリ管理。
  • **I/O 投入**: PgAioHandle を io_uring の Submission Queue Entry (SQE) に変換し、カーネルへ発行。
  • **完了処理**: Completion Queue (CQ) からのイベント回収と、対応する PgAioHandle の状態更新。
  • **排他制御**: 複数バックエンドが他者の完了イベントを処理する際の競合防止（completion_lock）。

重要なシンボル

  • pgaio_uring_ops: IoMethodOps 実装。AIO サブシステムがこのメソッドを呼び出すためのエントリポイント。
  • PgAioUringContext: バックエンドごとの io_uring インスタンスとロックを保持する構造体。
  • pgaio_uring_submit: I/O をキューに投入するメイン関数。
  • pgaio_uring_wait_one: 特定の I/O 完了を待機する同期ポイント。
  • pgaio_uring_drain_locked: 完了キューをスキャンし、保留中の I/O を処理するヘルパー。

制御フローとデータフロー

1. **初期化**: pgaio_uring_shmem_init が起動時に全バックエンド分の PgAioUringContext を共有メモリ上に確保します。 2. **投入**: pgaio_uring_submit が PgAioHandle を受け取り、io_uring_get_sqe で取得したスロットに I/O 情報を書き込み、io_uring_submit でカーネルへ通知します。 3. **待機・回収**: pgaio_uring_wait_one は、対象の I/O が完了するまで completion_lock を取得し、io_uring_wait_cqes でカーネルからの通知を待ちます。完了イベントは pgaio_uring_drain_locked を経由して pgaio_io_process_completion に渡されます。

読者向けメモ

  • **ロックの重要性**: completion_lock は、あるバックエンドが忙しい場合に他のバックエンドが代わりに完了処理を行うために必須です。
  • **カーネル機能の probe**: pgaio_uring_check_capabilities は、カーネルが io_uring_queue_init_mem をサポートしているか実行時に判定します。これにより、不要なメモリマッピングを回避しパフォーマンスを最適化しています。
  • **注意点**: io_uring を使用する場合、ulimit -n (ファイルディスクリプタ制限) が十分に高い値である必要があります。
