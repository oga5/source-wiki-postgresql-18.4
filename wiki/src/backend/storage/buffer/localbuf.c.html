src/backend/storage/buffer/localbuf.c

概要

localbuf.c は、PostgreSQL の一時テーブル専用の高速なバッファマネージャを実装しています。一時テーブルは WAL ログ出力やチェックポイントの対象外であるため、共有バッファマネージャ（bufmgr.c）の複雑なロック機構を介さず、各バックエンドプロセスがローカルメモリ上で直接管理します。

モジュール内での役割

src/backend/storage/buffer モジュールにおいて、共有バッファと対になる「プロセスプライベートなバッファ」を管理します。共有バッファが全プロセスで共有されるのに対し、本モジュールは各バックエンドのセッション内でのみ完結するデータアクセスを高速化します。

責務

  • **ローカルバッファのライフサイクル管理**: 一時テーブル用バッファの初期化、割り当て、解放。
  • **ハッシュテーブルによる高速検索**: BufferTag をキーとしたローカルバッファの高速なルックアップ。
  • **クロックスイープアルゴリズム**: バッファ不足時の置換対象選定（GetLocalVictimBuffer）。
  • **I/O 処理の抽象化**: 一時テーブルの読み書きおよび拡張処理の実行。

重要なシンボル

  • LocalBufferDescriptors: ローカルバッファのメタデータ（BufferDesc）を保持する配列。
  • LocalBufHash: バッファのタグからバッファ ID を引くためのハッシュテーブル。
  • LocalRefCount: 各ローカルバッファの参照カウントを保持する配列。
  • LocalBufferAlloc(): 指定されたブロックに対応するローカルバッファを取得または作成する主要関数。
  • GetLocalVictimBuffer(): 置換対象のバッファを選択し、必要に応じてフラッシュする関数。

制御フローとデータフロー

1. **初期化**: 初めて一時テーブルにアクセスした際に InitLocalBuffers() が呼ばれ、メモリ領域とハッシュテーブルが確保されます。 2. **検索**: LocalBufferAlloc() が LocalBufHash を検索し、存在すればピン留めして返します。 3. **割り当て**: 存在しない場合、GetLocalVictimBuffer() がクロックスイープで空きバッファを探します。必要に応じて古いバッファをディスクへフラッシュし、新しいタグをハッシュテーブルに登録します。 4. **I/O**: FlushLocalBuffer() や ExtendBufferedRelLocal() を通じて、smgr（ストレージマネージャ）層と連携し、物理的な読み書きを行います。

読者向けメモ

  • **ロック不要**: ローカルバッファはプロセス専用であるため、共有バッファのような複雑なスピンロックや LWLock は不要です。ただし、アトミック操作による状態更新は行われます。
  • **遅延割り当て**: InitLocalBuffers() はヘッダーのみを初期化し、実際のデータブロックメモリは GetLocalBufferStorage() により必要に応じて動的に確保されます。
  • **並列クエリの制限**: 一時テーブルはプロセスローカルであるため、パラレルワーカーからはアクセスできません。InitLocalBuffers 内で IsParallelWorker() をチェックし、エラーを投げることでこれを強制しています。
  • **デバッグ**: LBDEBUG マクロを有効にすると、バッファ操作の詳細なログが出力されます。
