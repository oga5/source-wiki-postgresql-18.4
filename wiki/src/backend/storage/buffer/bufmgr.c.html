src/backend/storage/buffer/bufmgr.c

概要

bufmgr.c は PostgreSQL のバッファマネージャのメインエントリポイントであり、共有メモリ上のバッファプールとプロセスローカルなバッファの管理を統括します。ディスク上のページをメモリに読み込み、ピン留め（Pin）し、必要に応じてダーティページのフラッシュや置換を行うための中心的なインターフェースを提供します。

モジュール内での役割

バッファ管理サブシステムにおける「司令塔」です。buf_table.c（ハッシュ検索）、freelist.c（置換戦略）、localbuf.c（一時テーブル管理）といった下位モジュールを呼び出し、上位レイヤー（アクセスメソッド等）からの要求を調整します。

責務

  • **ライフサイクル管理**: ReadBuffer 等を通じたページのピン留め・解放、およびトランザクション終了時のリソースクリーンアップ。
  • **I/O 実行**: 同期および非同期 I/O（AIO）の開始と完了待ち、エラーハンドリング。
  • **バッファ状態遷移**: ページの読み込み、ゼロ初期化、ダーティフラグの管理、およびチェックポイント時のフラッシュ制御。
  • **並行性制御**: バッファヘッダーのロック管理、およびピン留めによるページ保護。

重要なシンボル

ライフサイクル・アクセス

  • ReadBuffer() / ReadBufferExtended(): ページをバッファに読み込み、ピン留めして返すメイン関数。
  • ReleaseBuffer(): バッファのピンを解放する。
  • MarkBufferDirty(): ページが変更されたことを記録し、将来のフラッシュ対象とする。
  • LockBuffer() / LockBufferForCleanup(): バッファの内容に対するロック制御。

I/O 制御

  • StartReadBuffers() / WaitReadBuffers(): 非同期 I/O を活用した複数ブロックの読み込み制御。
  • FlushBuffer(): 特定のバッファを物理的にディスクへ書き出す。
  • BufferSync(): チェックポイント時に全ダーティバッファをフラッシュする。

内部管理・状態

  • PrivateRefCountArray / PrivateRefCountHash: バックエンドごとのピン留め状況を追跡する構造。
  • BufferDesc: バッファのメタデータ（タグ、状態フラグ、参照カウント）を保持する共有メモリ上の構造体。

制御フローとデータフロー

1. **要求**: ReadBuffer が呼ばれると、BufferAlloc がハッシュテーブルを検索。 2. **ヒット時**: 既存のバッファをピン留めして終了。 3. **ミス時**: GetVictimBuffer で置換候補を選定し、smgr を介してディスクから読み込む。 4. **I/O**: StartReadBuffers が非同期 I/O を開始し、WaitReadBuffers で完了を待機。この際、PgAioHandle を通じて状態が管理される。 5. **クリーンアップ**: ResourceOwner がトランザクション終了時にピンを自動解放し、リークを防ぐ。

読者向けメモ

  • **メモリ所有権**: バッファのピン留めは ResourceOwner によって管理されます。手動で ReleaseBuffer を呼び出すのが基本ですが、エラー発生時には ResourceOwner が自動的にピンを解放します。
  • **ロックの階層**: バッファヘッダーのロック（スピンロック）と、バッファ内容のロック（LWLock）は別物です。LockBufHdr は短時間のみ保持してください。
  • **非同期 I/O**: 本ファイルは AIO 統合の最前線にあります。StartReadBuffers などの新しい API は、将来的なパフォーマンス向上のために設計されています。
  • **未実装・浅い部分**: FlushRelationBuffers 等の全バッファスキャンは、大規模なバッファプールでは非常に低速です。これらはパフォーマンスクリティカルなパスでは使用されない前提で実装されています。
