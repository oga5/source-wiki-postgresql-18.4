src/backend/libpq/be-secure-openssl.c

概要

PostgreSQL バックエンドにおける OpenSSL を用いた TLS 通信機能の実装です。サーバー側の SSL コンテキストの初期化、証明書と秘密鍵の読み込み、およびクライアントとのセキュアな接続確立とデータ送受信を担います。

モジュール内での役割

src/backend/libpq サブシステムの一部として、be-secure.c が提供する抽象化レイヤーの背後で OpenSSL 固有の処理をカプセル化します。PostgreSQL の通信プロトコル（pqcomm.c）と OpenSSL ライブラリの橋渡しを行います。

責務

  • **SSL コンテキスト管理**: SSL_CTX の生成、プロトコルバージョン制限、暗号スイートの設定、証明書・CRL の読み込み。
  • **セキュアな I/O**: OpenSSL の BIO インターフェースをカスタマイズし、PostgreSQL のソケット処理と統合。
  • **認証と検証**: クライアント証明書の検証、ALPN（Application-Layer Protocol Negotiation）のネゴシエーション。
  • **鍵管理**: DH/ECDH パラメータの読み込みと設定、パスフレーズ保護された秘密鍵のハンドリング。

重要なシンボル

  • be_tls_init: SSL コンテキストを構築し、GUC 設定に基づいて環境をセットアップする。
  • be_tls_open_server: TLS ハンドシェイクを実行し、クライアント証明書情報を抽出する。
  • be_tls_read / be_tls_write: OpenSSL を介したセキュアなデータ送受信。
  • SSL_context: サーバー全体で共有される現在の SSL 設定状態。
  • port_bio_method: PostgreSQL のソケットを OpenSSL の BIO に適合させるためのメソッド定義。

制御フローとデータフロー

1. **初期化**: be_tls_init が呼び出され、証明書ファイルやプロトコル設定を読み込み SSL_CTX を作成します。 2. **接続確立**: be_tls_open_server が SSL_accept を呼び出し、ハンドシェイクを行います。この際、port_bio_read/write を通じて PostgreSQL のソケットからデータを読み書きします。 3. **データ転送**: be_tls_read/write がアプリケーション層からの要求を受け取り、OpenSSL の SSL_read/write を実行します。 4. **終了**: be_tls_close によりセッションを適切にシャットダウンし、リソースを解放します。

読者向けメモ

  • **BIO のカスタマイズ**: OpenSSL 標準の BIO ではなく、PostgreSQL のシグナル処理や非ブロッキング I/O と整合させるために port_bio_read/write が定義されています。
  • **エラー処理**: OpenSSL のエラーキューはスレッドローカルであるため、ERR_clear_error() と ERR_get_error() の呼び出し順序が重要です。
  • **パスフレーズ**: サーバー起動時や設定リロード時に秘密鍵のパスフレーズを安全に扱うため、ssl_passphrase_command を介した外部コマンド実行機能が組み込まれています。
