src/backend/libpq/auth.c

概要

src/backend/libpq/auth.c は、PostgreSQL バックエンドにおけるクライアント認証プロセスの中心的な実装です。pg_hba.conf で定義された認証方式に基づき、フロントエンドとのハンドシェイク、資格情報の検証、および認証結果の確定を統括します。

モジュール内での役割

libpq サブシステムにおいて、通信の物理層（pqcomm.c）とアクセス制御ポリシー（hba.c）の間に位置します。認証方式ごとの具体的なロジック（SCRAM, GSSAPI, LDAP 等）を呼び出し、認証成功時にセッションのアイデンティティを確立する役割を担います。

責務

  • **認証ディスパッチ**: Port 構造体の HBA 設定に基づき、適切な認証関数を呼び出す。
  • **認証プロトコル制御**: sendAuthRequest を介した認証要求の送信と、recv_password_packet によるパスワード応答の受信。
  • **アイデンティティ管理**: 認証成功時に set_authn_id を呼び出し、セッションの認証済み ID を記録する。
  • **エラーハンドリング**: 認証失敗時のログ出力と、セキュリティを考慮したクライアントへのエラー通知。
  • **認証フックの提供**: ClientAuthentication_hook を通じた外部プラグインによる認証プロセスへの介入。

重要なシンボル

認証ディスパッチ

  • ClientAuthentication(Port *port): 認証プロセスのエントリポイント。
  • sendAuthRequest(...): フロントエンドへ認証要求メッセージを送信する。
  • auth_failed(...): 認証失敗時の処理。FATAL エラーを発生させ、プロセスを終了させる。
  • set_authn_id(Port *port, const char *id): 認証済み ID を MyClientConnectionInfo に設定する。

認証方式の実装

  • **パスワード系**: CheckPasswordAuth (平文), CheckPWChallengeAuth (MD5/SCRAM), CheckMD5Auth。
  • **外部認証系**: CheckPAMAuth (PAM), CheckLDAPAuth (LDAP), CheckRADIUSAuth (RADIUS), CheckBSDAuth (BSD)。
  • **OS/ネットワーク系**: ident_inet (Ident), auth_peer (Peer), pg_GSS_recvauth (GSSAPI), pg_SSPI_recvauth (SSPI)。

制御フローとデータフロー

1. **開始**: ClientAuthentication が呼び出され、hba_getauthmethod で認証方式を決定。 2. **検証**: 必要に応じて secure_loaded_verify_locations 等でクライアント証明書を検証。 3. **ディスパッチ**: switch 文で port->hba->auth_method に応じた関数を実行。 4. **メッセージ交換**: 認証関数内で sendAuthRequest でチャレンジを送り、recv_password_packet で応答を待機。 5. **完了**: 認証成功時に set_authn_id で ID を確定し、AUTH_REQ_OK を送信。失敗時は auth_failed で終了。

読者向けメモ

  • **メモリ管理**: set_authn_id は TopMemoryContext に ID をコピーするため、外部ライブラリから渡された文字列も安全に保持されます。
  • **セキュリティ**: auth_failed は、攻撃者に情報を与えすぎないよう、詳細なエラー内容はログにのみ出力し、クライアントには汎用的なメッセージを返します。
  • **未実装・浅い部分**: RADIUS 認証のタイムアウト処理や、一部のプラットフォーム固有の認証（Solaris の PAM 等）は、レガシーな回避策が含まれており、保守性が低くなっています。
  • **注意点**: recv_password_packet は PG_MAX_AUTH_TOKEN_LENGTH (65535バイト) を上限としており、これを超える認証トークンはプロトコル違反として扱われます。
