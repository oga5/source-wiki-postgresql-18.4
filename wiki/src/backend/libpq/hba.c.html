src/backend/libpq/hba.c

概要

hba.c は、PostgreSQL のホストベース認証（HBA）およびユーザーマッピング（Ident）設定ファイルの解析と、接続要求に対する認証ルールの照合を担うモジュールです。pg_hba.conf および pg_ident.conf の読み込み、トークン化、構文解析、および実行時のアクセス制御判定を統括します。

モジュール内での役割

libpq サブシステムにおいて、認証フェーズの「ポリシー決定エンジン」として機能します。auth.c が実際の認証プロトコル（SCRAM, GSSAPI 等）を実行するのに対し、本モジュールは「どの接続元から、どのユーザーが、どのデータベースに対して、どの認証方式を使うべきか」を決定します。

責務

  • **設定ファイルの解析**: pg_hba.conf / pg_ident.conf の読み込み、再帰的なファイルインクルード処理、およびトークン化。
  • **構文検証**: 設定行の構文チェック、認証オプション（LDAP, RADIUS, OAuth 等）の妥当性検証。
  • **アクセス制御判定**: 接続要求（hbaPort）とロード済みのルールリストを照合し、適用すべき認証方式を特定。
  • **ユーザーマッピング**: Ident 認証におけるシステムユーザーとデータベースユーザーの対応付け（正規表現置換を含む）。

重要なシンボル

設定管理・状態保持

  • parsed_hba_lines / parsed_ident_lines: 解析済みのルールを保持するグローバルリスト。
  • tokenize_context: ファイル解析中に使用される一時的なメモリコンテキスト。
  • UserAuthName: UserAuth 列挙型と文字列を対応付けるマッピング配列。

解析・構築

  • load_hba() / load_ident(): 設定ファイルを再読み込みし、成功時にグローバル状態を更新するエントリポイント。
  • parse_hba_line() / parse_ident_line(): トークン化された行を構造体（HbaLine, IdentLine）へ変換。
  • tokenize_auth_file(): ファイルを読み込み、行単位でトークン化する中心的なパーサー。

照合ロジック

  • hba_getauthmethod(): 外部（auth.c）から呼び出される、認証方式決定のメインインターフェース。
  • check_hba(): 接続要求とルールリストを走査し、マッチする HbaLine を特定。
  • check_usermap(): Ident 認証において、ユーザーマッピングの妥当性を検証。

制御フローとデータフロー

1. **初期化/リロード**: load_hba() が open_auth_file() を経由してファイルを読み込み、tokenize_auth_file() で TokenizedAuthLine リストを生成。 2. **構文解析**: parse_hba_line() がトークンを検証し、HbaLine 構造体を構築。エラー時は err_msg を設定して報告。 3. **照合**: hba_getauthmethod() が check_hba() を呼び出し、hbaPort の情報（IP, SSL状態, DB名, ユーザー名）を parsed_hba_lines の各ルールと順次比較。 4. **判定**: マッチしたルールが見つかれば port->hba に格納。見つからない場合は uaImplicitReject を設定。

読者向けメモ

  • **メモリ管理**: load_hba() は、解析成功時にのみグローバルな parsed_hba_context を差し替えます。解析失敗時は古い設定を維持するアトミックな更新設計になっています。
  • **エラー処理**: parse_hba_line() 等の関数は、エラー時に err_msg を設定し、呼び出し元でログ出力を行う設計です。
  • **制約**: hba.c は PostmasterContext に依存しており、主に postmaster プロセスで実行されることを想定しています。
  • **省略箇所**: LDAP や RADIUS の詳細なプロトコルハンドリングは、各ライブラリの API に依存しており、本ファイルでは設定の妥当性検証に留めています。
