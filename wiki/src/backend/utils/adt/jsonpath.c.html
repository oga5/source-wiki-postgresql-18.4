src/backend/utils/adt/jsonpath.c

概要

jsonpath.c は、PostgreSQL の jsonpath データ型の内部バイナリ表現の構築、シリアライズ、デシリアライズ、およびテキスト変換を担う実装ファイルです。jsonpath は SQL/JSON 標準に基づいたクエリ言語であり、本ファイルはパース後の AST（抽象構文木）をメモリ効率の良いフラットなバイナリ形式へ変換し、実行エンジンが走査可能な状態にする役割を持ちます。

モジュール内での役割

src/backend/utils/adt モジュールの一部として、JSONB 関連の高度なクエリ機能を支える基盤です。jsonpath_exec.c がこのバイナリ形式を走査してクエリを実行するのに対し、本ファイルは「型としての定義」と「永続化・変換」に特化しています。

責務

  • **I/O 処理**: SQL 文字列と内部バイナリ表現（JsonPath 構造体）の相互変換。
  • **バイナリレイアウト管理**: AST を再帰的に走査し、4バイト境界でアライメントされた可変長アイテムのシーケンスへ変換。
  • **メタデータ管理**: JsonPath ヘッダー（バージョン、Lax/Strict モード）の管理。
  • **プランナ支援**: jspIsMutable を通じて、クエリ内に含まれる関数が可変（volatile）かどうかを判定し、プランナの最適化を支援。

重要なシンボル

入出力・変換

  • jsonpath_in / jsonpath_out: SQL インターフェース用の入出力関数。
  • jsonPathFromCstring: 文字列をパースし、バイナリ形式へフラット化するメインエントリ。
  • jsonPathToCstring: バイナリ形式を SQL 形式の文字列へ復元。

バイナリ構築・走査

  • flattenJsonPathParseItem: AST をバイナリへ変換する再帰関数。
  • jspInit / jspInitByBuffer: バイナリバッファから JsonPathItem 構造体を初期化し、走査可能にする。
  • jspGetArg / jspGetNext / jspGetLeftArg / jspGetRightArg: バイナリ内のリンクを辿り、引数や次のアイテムを取得するアクセサ群。

プランナサポート

  • jspIsMutable: クエリの可変性を判定するエントリポイント。
  • jspIsMutableWalker: jspIsMutable の内部再帰走査関数。

制御フローとデータフロー

1. **変換フロー**: jsonPathFromCstring が parsejsonpath（別モジュール）で AST を生成し、flattenJsonPathParseItem がそれを StringInfo バッファへ順次書き込みます。この際、アイテム間のリンク（オフセット）は reserveSpaceForItemPointer で予約し、再帰終了後に確定させます。 2. **走査フロー**: jspInitByBuffer がバイナリバッファのオフセットを読み取り、JsonPathItem 構造体を構築します。この構造体は base ポインタと現在の pos を保持し、アクセサ関数がこれを用いて動的に次のアイテムや引数を解決します。 3. **可変性判定フロー**: jspIsMutable がクエリ全体を走査し、タイムゾーン依存の型変換や volatile な関数が含まれるかを判定します。

読者向けメモ

  • **メモリレイアウト**: バイナリ形式は JsonPath ヘッダーの後に、JsonPathItemType、next ポインタ、および型固有のペイロードが続く構造です。INTALIGN マクロにより、各アイテムは 4 バイト境界に配置されます。
  • **メモリ所有権**: JsonPath 構造体は varlena 型であり、PostgreSQL のメモリ管理（palloc）下で管理されます。jsonPathFromCstring で生成されたバッファは、最終的に SET_VARSIZE でサイズが確定されます。
  • **制約**: jspIsMutable などの走査関数は、スタックオーバーフローを防ぐために check_stack_depth() を呼び出しています。
  • **省略箇所**: 本ファイルにはパースロジックそのものは含まれていません（parsejsonpath は外部依存）。また、バイナリの実行ロジックは jsonpath_exec.c に委譲されています。
