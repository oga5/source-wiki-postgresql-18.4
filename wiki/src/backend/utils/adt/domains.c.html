src/backend/utils/adt/domains.c

概要

PostgreSQLのドメイン型（既存のデータ型に制約を付与したもの）に対する入出力処理と制約検証を担うモジュールです。ドメイン型への入力値が、定義された NOT NULL や CHECK 制約を満たしているかを検証するロジックを提供します。

モジュール内での役割

src/backend/utils/adt 内において、型システムと実行エンジンを繋ぐ役割を果たします。ドメイン型は基底型のI/O関数を再利用しますが、本ファイルはそれに加えて「制約チェック」という追加レイヤーを挿入することで、ドメインの整合性を保証します。

責務

  • **I/O関数のラップ**: 基底型の入力関数（domain_in, domain_recv）を呼び出した後、制約検証を自動的に実行する。
  • **制約のキャッシュ**: カタログアクセスを最小化するため、DomainIOData 構造体を用いて制約情報や実行コンテキストをキャッシュする。
  • **制約検証の実行**: CHECK 制約を ExprContext 上で評価し、違反時には適切なエラー情報を生成する。

重要なシンボル

  • DomainIOData: ドメインのI/O状態を保持する構造体。基底型の関数情報、制約リスト、評価用コンテキストを格納。
  • domain_in / domain_recv: ドメイン型の標準的な入力関数。SQLレベルでの型変換時に呼び出される。
  • domain_check_input: DomainIOData を用いて、値が制約を満たすか検証する内部関数。
  • domain_check: 既存のデータに対してドメイン制約を再検証するための公開API。

制御フローとデータフロー

1. **初期化**: domain_state_setup が呼び出され、TypeCache から基底型情報と制約情報を取得し、DomainIOData を構築する。 2. **変換**: domain_in 等が基底型の入力関数を実行し、生の Datum を生成する。 3. **検証**: domain_check_input が DomainConstraintRef を通じて制約リストを走査する。

  • NOT NULL 制約: 値が NULL かどうかを判定。
  • CHECK 制約: ExprContext を作成し、ExecCheck で式を評価。

    • 4. **終了**: 検証失敗時はエラーを報告し、成功時は値を返す。ReScanExprContext でメモリをクリーンアップする。

関連コード

  • src/backend/utils/cache/typcache.c: ドメイン制約のキャッシュ管理の実体。
  • src/backend/executor/execExpr.c: CHECK 制約の評価ロジックの基盤。
  • src/backend/utils/adt/domains.c の各関数は、SQLの CREATE DOMAIN で定義された制約を評価するために、実行時の式評価エンジンと密接に連携します。

読者向けメモ

  • **キャッシュの生存期間**: DomainIOData は FmgrInfo の fn_extra に保持されます。これにより、同一の関数呼び出しが続く間はカタログ検索をスキップできます。
  • **エラーハンドリング**: escontext を介したエラー報告に対応しており、domain_check_safe を使用することで、例外を投げずに検証結果をブール値で受け取ることが可能です。
  • **メモリ管理**: CHECK 制約の評価には ExprContext を使用するため、メモリリークを防ぐための ReScanExprContext の呼び出しが重要です。
