src/backend/utils/adt/ri_triggers.c

概要

参照整合性（Referential Integrity: RI）制約を強制するための汎用トリガープロシージャを実装するモジュールです。外部キー（FK）制約の挿入・更新・削除イベントに対し、SPI（Server Programming Interface）を用いて制約チェックやカスケード操作（CASCADE, SET NULL, SET DEFAULT, RESTRICT, NO ACTION）を実行します。

モジュール内での役割

PostgreSQL のトリガーシステムと SQL レベルの制約定義を繋ぐブリッジです。pg_constraint カタログから制約情報を抽出し、実行時に必要なクエリを動的に生成・キャッシュして実行します。

責務

  • **制約チェック**: 外部キーの整合性検証（INSERT/UPDATE 時）。
  • **参照アクション**: 削除・更新時のカスケード処理（RESTRICT, CASCADE, SET NULL/DEFAULT）。
  • **クエリ最適化**: 実行計画（SPI Plan）のキャッシュと再利用。
  • **メタデータ管理**: pg_constraint からの制約情報の抽出とキャッシュ。
  • **検証**: ALTER TABLE ADD FOREIGN KEY 時の初期データ検証。

重要なシンボル

制約情報管理

  • RI_ConstraintInfo: 外部キー制約の定義（列番号、演算子、アクション種別等）を保持するキャッシュ構造体。
  • ri_constraint_cache: 制約情報を保持するハッシュテーブル。
  • ri_LoadConstraintInfo(): pg_constraint から情報を読み込みキャッシュを構築する。

クエリ実行とキャッシュ

  • RI_QueryKey: キャッシュされた SPI 計画を識別するためのキー。
  • ri_query_cache: 実行計画を保持するハッシュテーブル。
  • ri_PlanCheck(): 制約チェック用のクエリを準備し、キャッシュに保存する。
  • ri_PerformCheck(): キャッシュされた計画を用いて実際に SPI クエリを実行する。

トリガー関数（エントリポイント）

  • RI_FKey_check_ins, RI_FKey_check_upd: FK 側の挿入・更新チェック。
  • RI_FKey_cascade_del, RI_FKey_cascade_upd: PK 側の削除・更新に伴うカスケード処理。
  • RI_FKey_restrict_del, RI_FKey_noaction_del 等: 削除・更新時の制限処理。

制御フローとデータフロー

1. **トリガー起動**: トリガーマネージャが RI_FKey_... 関数を呼び出す。 2. **情報取得**: ri_FetchConstraintInfo がキャッシュから制約定義を取得（なければロード）。 3. **クエリ構築**: ri_BuildQueryKey でクエリ種別を特定し、キャッシュに計画が存在するか確認。存在しない場合は ri_PlanCheck でクエリ文字列を生成し、SPI 計画を準備。 4. **実行**: ri_PerformCheck が SPI を介して対象テーブルの行を操作・検証。 5. **エラー報告**: 制約違反時は ri_ReportViolation がエラーメッセージを構築し、トランザクションをアボートする。

直接依存

  • executor/spi.h: SQL クエリ実行用インターフェース。
  • utils/syscache.h: pg_constraint 等のシステムカタログキャッシュ。
  • access/table.h, access/tableam.h: テーブルアクセス層。
  • lib/ilist.h: キャッシュ管理用のリスト構造。

関連コード

  • commands/trigger.c: トリガーの登録と実行管理。
  • catalog/pg_constraint.h: 外部キー制約のカタログ定義。
  • executor/spi.c: SPI の実装詳細。

読者向けメモ

  • **メモリ管理**: キャッシュ（ri_constraint_cache 等）はバックエンドの生存期間中保持されます。SPI 計画は SPI_keepplan() で保存され、メモリリークを防ぐため再利用されます。
  • **制約の継承**: パーティションテーブルの FK 制約において、共通の祖先を持つ制約はクエリ計画を共有する最適化が行われています。
  • **制約の浅い部分**: MATCH PARTIAL は現在未実装です。
  • **セキュリティ**: クエリ実行時は SetUserIdAndSecContext を使用し、制約定義テーブルの所有者の権限で実行されます。
  • **注意**: ri_constraint_cache のエントリは InvalidateConstraintCacheCallBack によって無効化されますが、メモリ解放は行われません。これはアクティブな参照を安全に扱うためです。
