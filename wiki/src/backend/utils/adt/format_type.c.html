src/backend/utils/adt/format_type.c

概要

PostgreSQLのデータ型OIDおよび型修飾子（typmod）を、人間が読みやすい標準的なSQL形式の文字列に変換する機能を提供します。主にシステムカタログの情報を基に、型名や修飾子を適切にフォーマットします。

モジュール内での役割

src/backend/utils/adt モジュールの一部として、型情報の表示やダンプ、エラーメッセージ生成における「型名の正規化」を担います。SQLパーサーが解釈可能な形式で型名を出力することで、pg_dump 等のツールがスキーマを正確に再構築できるように支援します。

責務

  • **型名のフォーマット**: OIDから型名を解決し、必要に応じてスキーマ修飾（schema.type）や引用符付けを行う。
  • **型修飾子の処理**: typmod を含む型（varchar(n) など）に対し、型固有の出力関数（typmodout）を呼び出して修飾子を付与する。
  • **組み込み型の特別扱い**: SQL標準に準拠した型名（boolean, integer 等）への変換。
  • **配列型の分解**: 配列型（int[] など）を基底型名に [] を付与した形式に変換する。

重要なシンボル

  • format_type: SQLから呼び出されるメイン関数。format_type_extended のラッパー。
  • format_type_extended: 変換ロジックの核心。フラグ制御により柔軟なフォーマットが可能。
  • printTypmod: 型名に (n) などの修飾子を付与するヘルパー関数。
  • type_maximum_size: 型と typmod から最大サイズを算出する。format_type と型情報の知識を共有している。

制御フローとデータフロー

1. **入力**: OIDと typmod を受け取る。 2. **キャッシュ検索**: SearchSysCache1(TYPEOID, ...) を使用して pg_type カタログから型情報を取得。 3. **配列判定**: IsTrueArrayType で配列か確認し、配列であれば基底型へポインタを切り替える。 4. **変換**:

  • 組み込み型であれば switch 文でハードコードされた名称を選択。
  • それ以外は quote_qualified_identifier でスキーマ修飾と引用符付けを行う。

    • 5. **修飾子付与**: with_typemod が有効な場合、printTypmod を呼び出し、型固有の typmodout 関数で文字列を生成。 6. **出力**: 最終的な文字列を palloc して返却。

読者向けメモ

  • typmod が -1 の場合と NULL（未指定）の場合で出力結果が異なるケースがある点に注意してください。これはパーサーの解釈を維持するための仕様です。
  • format_type_extended の flags 引数は、エラーハンドリング（??? を返すか、エラーにするか）を制御する重要なインターフェースです。
  • type_maximum_size は format_type と密接に関連していますが、型定義の知識をハードコードしている箇所があるため、新しい型を追加する際は両方の修正が必要になる場合があります。
