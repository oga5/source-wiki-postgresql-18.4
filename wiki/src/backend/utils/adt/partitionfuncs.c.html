src/backend/utils/adt/partitionfuncs.c

概要

パーティションテーブルの階層構造を調査・取得するための SQL 呼び出し可能な関数群を提供します。主にシステムカタログを走査し、パーティションツリーの構成要素や親子関係を特定します。

モジュール内での役割

src/backend/utils/adt モジュールの一部として、ユーザーが SQL レベルでパーティションのメタデータにアクセスするためのインターフェースを実装しています。

責務

  • パーティションツリーの全メンバーを列挙する（pg_partition_tree）。
  • 指定されたリレーションのルート（最上位親）を特定する（pg_partition_root）。
  • 指定されたリレーションの全祖先を列挙する（pg_partition_ancestors）。
  • 入力された OID がパーティションツリーの一部として有効かどうかの検証。

重要なシンボル

  • pg_partition_tree: パーティションツリーの全階層をテーブル形式で返す SRF（Set Returning Function）。
  • pg_partition_root: 指定リレーションのルート OID を返す関数。
  • pg_partition_ancestors: 指定リレーションからルートまでの祖先リストを返す SRF。
  • check_rel_can_be_partition: 内部ヘルパー。リレーションがパーティションツリーに含まれ得るか（存在確認および relkind 確認）を判定。

制御フローとデータフロー

  • **SRF の実装**: pg_partition_tree および pg_partition_ancestors は FuncCallContext を使用して状態を保持します。初回呼び出し時に find_all_inheritors や get_partition_ancestors を用いてリストを取得し、以降の呼び出しでそのリストを順次消費します。
  • **検証フロー**: 各関数はまず check_rel_can_be_partition を呼び出します。ここで SearchSysCacheExists1 を用いてリレーションの存在を確認し、get_rel_relkind でパーティション化可能な型かを確認します。
  • **データ生成**: pg_partition_tree は heap_form_tuple を使用して、リレーションID、親ID、リーフフラグ、階層レベルを含む行を構築します。

直接依存

  • catalog/partition.h: パーティション関連のカタログ操作関数（get_partition_ancestors 等）。
  • catalog/pg_class.h, catalog/pg_inherits.h: リレーションおよび継承関係の定義。
  • utils/lsyscache.h: システムキャッシュ経由でのリレーション属性取得。
  • funcapi.h: SRF 実装のためのマクロと構造体。
  • access/htup_details.h
  • postgres.h
  • utils/fmgrprotos.h
  • utils/syscache.h

関連コード

  • src/backend/catalog/partition.c: パーティションの定義や管理を行うコアロジック。
  • src/backend/nodes/list.c: リスト操作（lcons_oid, list_nth_oid 等）。
  • src/backend/access/common/heaptuple.c: heap_form_tuple による行構築。

読者向けメモ

  • 本ファイル内の関数は、主に pg_partition_tree などのシステムビューのバックエンドとして機能します。
  • SRF_IS_FIRSTCALL を使用したメモリ管理パターン（multi_call_memory_ctx への切り替え）は、PostgreSQL の SRF 実装における標準的な手法です。
  • check_rel_can_be_partition はエラーを投げず false を返す設計であるため、呼び出し側で適切にハンドリングする必要があります。
