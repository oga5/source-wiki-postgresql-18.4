src/backend/utils/adt/ruleutils.c

概要

ruleutils.c は、PostgreSQL の内部表現（Query ツリー、Plan ツリー、Expr ツリーなど）を、人間が読み書き可能な SQL ソースコード形式に逆コンパイル（deparse）するための中心的なモジュールです。pg_get_viewdef や pg_get_ruledef といったシステムカタログの定義を表示する SQL 関数のバックエンド実装を担います。

モジュール内での役割

src/backend/utils/adt 内に位置し、主にメタデータ管理やシステムカタログの可視化を支援します。実行エンジンやプランナが生成した複雑なノード構造を、SQL 文として再構築することで、pg_dump や EXPLAIN、ビューの定義確認といった機能を支えています。

責務

  • **逆コンパイル（Deparsing）**: 内部ノード（Query, Plan, Expr）から SQL 文字列への変換。
  • **名前解決**: 逆コンパイル時に、RTE（Range Table Entry）のエイリアスや列名を一意かつ整合的に決定する。
  • **整形（Pretty Printing）**: 読みやすい SQL を生成するためのインデントや改行の制御。
  • **システムカタログの逆生成**: トリガー、インデックス、制約、パーティション定義の SQL 文生成。

重要なシンボル

コンテキスト管理

  • deparse_context: 逆コンパイル中の状態（出力バッファ、名前空間、整形フラグ）を保持する構造体。
  • deparse_namespace: クエリレベルごとの名前空間（RTE リスト、列名マッピング、プランツリー情報）を管理。
  • deparse_columns: 各 RTE における列名のエイリアス割り当てと競合解決を管理する構造体。

公開 API（SQL 関数経由）

  • pg_get_ruledef, pg_get_viewdef: ルールやビューの定義を SQL 文字列として取得。
  • pg_get_triggerdef: トリガー定義の生成。
  • pg_get_indexdef: インデックス定義の生成。
  • pg_get_constraintdef: 制約定義の生成。
  • pg_get_expr: 式ツリーの逆コンパイル。

制御フローとデータフロー

1. **初期化**: deparse_context を初期化し、対象となる Query や Plan に基づいて deparse_namespace を構築します。 2. **名前解決**: set_rtable_names および set_using_names を呼び出し、クエリ内のすべての RTE に対して一意なエイリアスと列名を割り当てます。 3. **再帰的走査**: get_rule_expr を起点として、ノードツリーを再帰的に走査します。 4. **文字列構築**: StringInfo バッファに対して、各ノードの型に応じた SQL 文字列を順次 append していきます。 5. **整形**: appendContextKeyword 等のヘルパー関数が、pretty-print フラグに応じて改行やインデントを挿入します。

読者向けメモ

  • **メモリ管理**: 逆コンパイル処理は非常にメモリを消費しやすいため、StringInfo を用いて効率的に文字列を構築しています。
  • **複雑性**: 本ファイルが巨大な理由は、SQL のあらゆる構文（JOIN, JSON, Window 関数, パーティション等）を網羅的に逆コンパイルする必要があるためです。
  • **注意点**: deparse_columns の列名割り当てロジックは、ビューの列名変更や削除に対応するために非常に複雑です。特に JOIN USING の処理は、名前の競合を避けるための再帰的な名前選択が行われており、このファイルの難所です。
  • **制限事項**: 一部のユーティリティコマンドや、非常に古い形式のルール定義については、完全な逆コンパイルがサポートされていない場合があります。
