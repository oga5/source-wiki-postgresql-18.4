src/backend/utils/error/elog.c

概要

PostgreSQL のエラー報告およびログ出力のメインパイプラインを担う中核ファイルです。ereport() および elog() マクロを介したエラー情報の収集、構造化、および出力先（サーバーログ、クライアント、syslog、Windows イベントログ等）へのルーティングをオーケストレートします。

モジュール内での役割

エラー処理サブシステムにおける「オーケストレーター」です。elog.h で定義されたインターフェースを実装し、csvlog.c や jsonlog.c などの出力プラグイン、および assert.c と連携して、システム全体の堅牢性を維持します。

責務

  • **エラーライフサイクル管理**: errstart() から errfinish() までのエラー報告サイクルの制御。
  • **再入可能性の確保**: 再帰的なエラー発生（メモリ枯渇時など）を検出し、ErrorContext のリセットやスタック管理を通じて無限ループを防止。
  • **ログの構造化とルーティング**: ログプレフィックスの生成、SQLSTATE の変換、および設定に応じた適切な出力先へのデータ転送。
  • **例外処理の橋渡し**: ERROR 以上の深刻度における siglongjmp を用いたトランザクションアボート処理のトリガー。

重要なシンボル

エラー状態管理

  • errordata_stack: 再入可能なエラー報告をサポートする ErrorData の静的スタック（深度 5）。
  • errordata_stack_depth: 現在のスタック深度。
  • recursion_depth: エラー処理中の再帰を追跡し、無限ループを検知。

エラー報告 API

  • errstart(): エラー報告サイクルの開始。深刻度の昇格や出力要否の判定を行う。
  • errfinish(): 報告サイクルの終了。出力の実行とスタックのポップを行う。
  • errmsg(), errdetail(), errhint(): ErrorData に情報を追加するミューテータ。

出力・ルーティング

  • EmitErrorReport(): 構築された ErrorData を各出力先へディスパッチする中心関数。
  • send_message_to_server_log(): サーバーログへの書き込み処理。
  • send_message_to_frontend(): クライアントへのエラーメッセージ送信（FE/BE プロトコル）。

グローバル変数・フック

  • error_context_stack: エラー発生時に呼び出されるコンテキストコールバックのリスト。
  • emit_log_hook: ログ出力前にメッセージをインターセプトするフック。

制御フローとデータフロー

1. **開始**: ereport() マクロが errstart() を呼び出し、ErrorData をスタックに確保。 2. **構築**: errmsg() 等の関数が ErrorData の各フィールドを埋める。この際、EVALUATE_MESSAGE マクロが StringInfo を用いてフォーマット済み文字列を生成。 3. **終了**: errfinish() が呼び出され、EmitErrorReport() を通じて出力処理へ移行。 4. **出力**: EmitErrorReport() は Log_destination に基づき、csvlog.c / jsonlog.c / syslog / stderr へデータをルーティング。 5. **例外**: ERROR の場合、pg_re_throw() を経由して siglongjmp を実行し、呼び出し元へ制御を戻す。

読者向けメモ

  • **メモリ所有権**: エラー報告中に生成される文字列は、原則として ErrorContext に割り当てられます。CopyErrorData() を使用して別のコンテキストへコピーする場合、呼び出し側で FreeErrorData() を呼び出す必要があります。
  • **再帰の回避**: in_error_recursion_trouble() は、エラー処理中にエラーが発生した際の無限ループを防ぐ重要なガードです。この状態では、翻訳（gettext）をスキップするなどの安全策が講じられます。
  • **制約**: errordata_stack は深度 5 に固定されています。これを超える再帰は PANIC を引き起こし、プロセスを終了させます。
  • **浅い領域**: write_syslog や write_eventlog などのプラットフォーム固有の出力実装は、OS のログ機能に依存しており、エラー処理の「末端」として機能します。
