src/backend/utils/misc/rls.c

概要

PostgreSQLの行レベルセキュリティ（RLS: Row Level Security）が特定のクエリやテーブルに対して有効であるかを判定するためのユーティリティ関数群を提供します。

モジュール内での役割

src/backend/utils/misc ディレクトリ内に位置し、RLSの適用可否を判断する中心的なロジックを担います。プランナや実行エンジンが、クエリ実行時にRLSポリシーを適用すべきか、あるいは環境の変化によってプランの再評価が必要かを判断する際の判断基準を提供します。

責務

  • テーブルのメタデータ（pg_class）に基づき、RLSが有効化されているかを判定する。
  • ユーザーの権限（BYPASSRLS属性や所有者権限）を考慮し、RLSを適用すべきか、あるいはバイパスすべきかを決定する。
  • RLS設定が環境（ユーザーIDやGUC設定）に依存する場合、プランキャッシュの無効化が必要であることを示すステータスを返す。
  • SQLから直接RLS状態を確認するための関数（row_security_active等）を提供する。

重要なシンボル

  • check_enable_rls: RLSの適用状態を判定する中核関数。RLS_NONE, RLS_NONE_ENV, RLS_ENABLED のいずれかを返す。
  • CheckEnableRlsResult: 判定結果を保持する列挙型。プランキャッシュの無効化戦略を決定するために使用される。
  • row_security_active: SQLからテーブルのOIDを指定してRLS有効性を確認する関数。
  • row_security_active_name: SQLからテーブル名を指定してRLS有効性を確認する関数。

制御フローとデータフロー

1. **入力**: テーブルのOID、チェック対象のユーザーID、エラー抑制フラグを受け取る。 2. **メタデータ取得**: syscache（RELOID）を使用して pg_class から relrowsecurity および relforcerowsecurity フラグを取得する。 3. **判定ロジック**:

  • BYPASSRLS 権限を持つユーザーは常にバイパス（RLS_NONE_ENV）。
  • テーブル所有者は、FORCE ROW SECURITY が設定されていない限りバイパス（RLS_NONE_ENV）。
  • 上記以外でRLSが有効な場合、GUC変数 row_security の状態を確認し、必要に応じてエラーをスローするか RLS_ENABLED を返す。

    • 4. **出力**: 判定結果に基づき、プランナがクエリプランを最適化・キャッシュするためのステータスを返す。

読者向けメモ

  • RLS_NONE_ENV は「現在はRLSが適用されないが、ユーザーや環境が変われば適用される可能性がある」ことを意味し、プランキャッシュの無効化トリガーとして機能します。
  • InNoForceRLSOperation() は、参照整合性チェックなどの特殊な操作中にRLSを強制適用しないための仕組みです。
  • 本ファイルはRLSの「適用判定」のみを扱い、実際のポリシー評価やフィルタリング処理は実行エンジン側の別の箇所で行われます。
