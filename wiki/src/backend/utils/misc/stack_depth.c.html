src/backend/utils/misc/stack_depth.c

概要

PostgreSQL プロセスにおけるスタックオーバーフローを未然に防ぐための監視・制限機能を提供します。ハードウェアレベルのセグメンテーション違反（SIGSEGV）が発生する前に、PostgreSQL のエラーとして安全に処理を中断させることを目的としています。

モジュール内での役割

src/backend/utils/misc ディレクトリの一部として、バックエンドの実行環境を保護するユーティリティを提供します。特に、再帰的な関数呼び出しが多用されるクエリ実行エンジンにおいて、スタック消費量を監視するガードレールとして機能します。

責務

  • **スタック基準点の管理**: プロセス開始時にスタックの基準位置を記録し、必要に応じて一時的な変更（マルチスレッド環境等）をサポートする。
  • **スタック深さの監視**: 現在のスタックポインタと基準点の差分を計算し、設定値（max_stack_depth）を超過していないか判定する。
  • **GUC 連携**: 設定パラメータ max_stack_depth の妥当性チェック（OS のリソース制限との整合性確認）と更新を行う。

重要なシンボル

  • max_stack_depth: ユーザーが設定可能なスタック上限（KB単位）。
  • stack_base_ptr: スタック深さ計算の起点となるポインタ。
  • check_stack_depth(): スタック超過時に ERROR を投げる公開関数。
  • stack_is_too_deep(): スタック超過状態を真偽値で返す判定関数。
  • get_stack_depth_rlimit(): OS の getrlimit(RLIMIT_STACK, ...) をラップし、プラットフォームの物理的な制限を取得する。

制御フローとデータフロー

1. **初期化**: プロセス起動時に set_stack_base() が呼ばれ、現在のスタックアドレスが stack_base_ptr に格納されます。 2. **監視**: 再帰的な処理の入り口で check_stack_depth() が呼び出されます。 3. **計算**: stack_is_too_deep() 内で、現在のローカル変数のアドレスと stack_base_ptr の差分を計算し、max_stack_depth_bytes と比較します。 4. **検証**: GUC 設定変更時、check_max_stack_depth() が OS のリソース制限（ulimit -s 等）と比較し、設定値が物理限界を超えていないか検証します。

関連コード

読者向けメモ

  • **スタックの方向**: スタックがアドレスの増加方向・減少方向どちらに伸びる環境でも動作するように、差分の絶対値をとる設計になっています。
  • **スレッド対応**: restore_stack_base は、PL/Java のようなバックエンド内で別スレッドを扱う特殊なケースのために存在します。通常のバックエンド処理では意識する必要はありません。
  • **パフォーマンス**: stack_is_too_deep は頻繁に呼ばれる可能性があるため、計算は最小限のポインタ演算のみで行われるよう最適化されています。
