src/backend/utils/activity/pgstat_shmem.c

概要

PostgreSQLの統計情報（Cumulative Statistics）を共有メモリ上で管理するための基盤機能を提供します。動的共有メモリ（DSA）とハッシュテーブル（dshash）を組み合わせ、バックエンド間で統計情報を共有・同期するための仕組みを実装しています。

モジュール内での役割

pgstat.c が提供する統計システムの上位インターフェースに対し、低レイヤーのメモリ管理と同期制御を提供します。各バックエンドが共有メモリ上の統計情報へ効率的にアクセスするためのローカルキャッシュ（pgStatEntryRefHash）を管理し、共有メモリのライフサイクルを制御します。

責務

  • **共有メモリの初期化**: StatsShmemInit により、Postmaster起動時に統計用共有メモリ領域（DSAおよびハッシュテーブル）を確保・初期化する。
  • **参照管理**: 各バックエンドが保持する共有統計情報への参照（PgStat_EntryRef）を管理し、不要になった統計情報の解放を調整する。
  • **GCメカニズム**: 統計情報が削除された際、他のバックエンドが保持する古い参照を無効化するためのガベージコレクション（GC）機能を提供。
  • **ロック制御**: 統計情報へのアクセスを保護するための共有/排他ロックの抽象化。

重要なシンボル

  • StatsShmemInit: 統計システム全体の共有メモリ領域を確保する。
  • pgstat_get_entry_ref: 統計エントリへの参照を取得する。存在しない場合は作成する。
  • pgstat_lock_entry / pgstat_unlock_entry: 統計エントリに対するLWLockの操作。
  • pgStatLocal.shmem: 現在のバックエンドが参照する共有統計制御構造体。
  • pgStatEntryRefHash: 各バックエンドが保持する、共有統計エントリへのローカルキャッシュ。

制御フローとデータフロー

1. **初期化**: StatsShmemInit が共有メモリを確保し、dshash を構築する。 2. **アクセス**: pgstat_get_entry_ref が呼ばれると、まずローカルの pgStatEntryRefHash を確認。キャッシュミス時は共有メモリ上の dshash を検索し、必要に応じてエントリを生成（pgstat_init_entry）する。 3. **同期**: 統計情報の削除時、dropped フラグを立て、参照カウントが0になった時点で dsa_free を実行する。 4. **GC**: 他のバックエンドが古い参照を保持している場合、pgstat_request_entry_refs_gc を通じてGCを要求し、各バックエンドが pgstat_gc_entry_refs でローカルキャッシュをクリーンアップする。

読者向けメモ

  • 本ファイルは「共有メモリ上の統計情報のライフサイクル」を管理する複雑な箇所です。特に、refcount と generation を用いた楽観的なロック制御と、バックエンド間での削除同期（GC）の仕組みを理解することが重要です。
  • pgstat_get_entry_ref は頻繁に呼ばれるため、ローカルキャッシュによる最適化が効いている点に注目してください。
  • 統計情報の削除は即時ではなく、全バックエンドが参照を解放するまで遅延される設計になっています。
