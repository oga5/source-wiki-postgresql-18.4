src/backend/commands/explain.c

概要

explain.c は、SQL の EXPLAIN コマンドを実行し、クエリの実行計画を解析・可視化するための中心的なモジュールです。プランナが生成した PlannedStmt を受け取り、実行計画ツリーを走査して、コスト、行数、実行時間、バッファ使用量、JIT 統計などの詳細情報を指定されたフォーマット（TEXT, XML, JSON, YAML）で出力します。

モジュール内での役割

commands モジュールの一部として、クエリ実行の可視化という「観測」の責務を担います。executor サブシステムと密接に連携し、EXPLAIN ANALYZE 実行時には実際にクエリを実行して統計情報を収集します。

責務

  • **コマンドディスパッチ**: ExplainQuery を起点に、クエリの書き換え、プランニング、実行、結果の整形を制御。
  • **計画ツリーの走査**: ExplainNode を中心とした再帰的な関数群により、計画ツリーの各ノード（スキャン、結合、集約等）の情報を抽出。
  • **統計情報の収集と整形**: EXPLAIN ANALYZE 時の実行時間、バッファ使用量、WAL 使用量、JIT 統計の集計と出力。
  • **プラグイン拡張**: フック（ExplainOneQuery_hook 等）を提供し、外部モジュールが計画出力に独自の情報を追加できるようにする。

重要なシンボル

コマンド実行・制御

  • ExplainQuery: EXPLAIN コマンドのメインエントリーポイント。
  • ExplainOneQuery: 単一クエリの計画生成と出力を管理。
  • ExplainOnePlan: 計画済みクエリの実行（ANALYZE 時）と統計出力の制御。
  • ExplainOneUtility: CREATE TABLE AS や DECLARE CURSOR などのユーティリティ文に対する計画生成。

計画ツリー走査・出力

  • ExplainNode: 計画ツリーの各ノードを再帰的に処理する中心関数。
  • ExplainPrintPlan: 計画ツリー全体のテキストダンプを生成。
  • ExplainPrintTriggers: トリガーの実行統計を出力。
  • ExplainPrintJITSummary: JIT コンパイルの統計情報を出力。

フック（拡張用）

  • ExplainOneQuery_hook: ExplainOneQuery の挙動を上書きするプラグイン用フック。
  • explain_per_plan_hook / explain_per_node_hook: 計画全体または各ノードにカスタム情報を追加するフック。

制御フローとデータフロー

1. **初期化**: ExplainQuery が ExplainState を作成し、オプションを解析。 2. **計画生成**: standard_ExplainOneQuery が pg_plan_query を呼び出し、PlannedStmt を取得。 3. **実行（ANALYZE時）**: ExplainOnePlan が ExecutorStart/Run/Finish/End を呼び出し、QueryDesc を通じて統計情報を収集。 4. **出力生成**: ExplainNode が再帰的にツリーを巡回し、各ノードの情報を ExplainState のバッファ（es->str）に書き込む。 5. **後処理**: ExplainEndOutput でフォーマットを閉じ、結果をクライアントへ送信。

読者向けメモ

  • **メモリ管理**: ExplainState は ExplainQuery のライフサイクルを通じて生存します。ANALYZE 実行時のメモリ消費計測には、専用の MemoryContext が作成されます。
  • **再帰的走査**: ExplainNode は PlanState ツリーを再帰的に走査します。ancestors リストは、サブプランやパラメータの解決のために親ノードの情報を保持します。
  • **未実装・浅い部分**: ExplainOneUtility は一部のユーティリティ文のみをサポートしており、計画を持たない文については限定的な出力となります。
  • **インデント管理**: TEXT フォーマットでは es->indent を手動で管理しますが、他のフォーマットでは ExplainOpenGroup/ExplainCloseGroup を使用して論理構造を定義します。
