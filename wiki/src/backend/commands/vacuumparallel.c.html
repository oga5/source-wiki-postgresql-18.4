src/backend/commands/vacuumparallel.c

概要

vacuumparallel.c は、PostgreSQL の VACUUM 処理において、インデックスのバルク削除およびクリーンアップを並列実行するためのサポートルーチンを提供します。複数のワーカープロセスを起動し、共有メモリ（DSM）を介してインデックス統計や削除対象のタプル情報（TidStore）を共有することで、大規模テーブルのメンテナンス時間を短縮します。

モジュール内での役割

src/backend/commands/vacuum.c（Lazy Vacuum のメインロジック）から呼び出される補助モジュールです。インデックス処理のフェーズにおいて、単一プロセスではなく並列実行が可能な場合に、その実行環境の構築と制御を担います。

責務

  • **並列実行環境の管理**: ParallelVacuumState の初期化、ワーカープロセスの起動、および終了処理。
  • **共有リソースの調整**: maintenance_work_mem のワーカー間分配や、共有コストベースの遅延制御（VacuumSharedCostBalance）の管理。
  • **インデックス処理の分散**: インデックスの並列安全性を判定し、リーダーとワーカー間でタスクを動的に割り当て。
  • **統計情報の集約**: 各ワーカーが処理したインデックスの統計結果（IndexBulkDeleteResult）を DSM から回収。

重要なシンボル

  • ParallelVacuumState: 並列 VACUUM の状態を保持するメイン構造体。
  • PVShared: DSM 上に配置される、全ワーカーで共有される制御情報。
  • parallel_vacuum_init: 並列実行の準備を行い、コンテキストを作成する。
  • parallel_vacuum_bulkdel_all_indexes: インデックスのバルク削除フェーズを開始する。
  • parallel_vacuum_main: 並列ワーカープロセスのエントリポイント。

制御フローとデータフロー

1. **初期化**: parallel_vacuum_init が ParallelContext を作成し、DSM 上に PVShared や TidStore を確保します。 2. **タスク割り当て**: リーダープロセスが parallel_vacuum_process_all_indexes を呼び出し、各インデックスのステータスを更新してワーカーを起動します。 3. **並列実行**: リーダーとワーカーは parallel_vacuum_process_safe_indexes を通じて、アトミックなインデックスカウンタ（shared->idx）をインクリメントしながら、未処理のインデックスを順次取得・処理します。 4. **同期と終了**: 全ワーカーの完了を待ち、リーダーが統計情報を回収して parallel_vacuum_end で DSM を解放します。

読者向けメモ

  • **並列安全性**: すべてのインデックスが並列処理に適しているわけではありません。amparallelvacuumoptions を確認し、min_parallel_index_scan_size 以下の小さなインデックスはリーダーが単独で処理する設計になっています。
  • **エラーハンドリング**: 並列ワーカー内でエラーが発生した場合、parallel_vacuum_error_callback が呼び出され、どのインデックス処理中に失敗したかが報告されます。
  • **メモリ管理**: ワーカーは maintenance_work_mem を共有するため、ワーカー数に応じて自動的に分割されます。この設定は PVShared を通じて各ワーカーに伝達されます。
