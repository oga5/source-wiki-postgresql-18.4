src/backend/commands/alter.c

概要

PostgreSQLにおける、汎用的なオブジェクト変更操作（ALTER）のドライバ層を担うファイルです。特定のオブジェクト型に依存しない共通的な変更処理（名前変更、スキーマ移動、所有者変更、依存関係の更新）を実装しています。

モジュール内での役割

src/backend/commands モジュールにおいて、各オブジェクト固有のコマンド（tablecmds.c, typecmds.c など）を補完する共通基盤として機能します。個別のコマンド実装が複雑化するのを防ぎ、カタログ操作の定型処理を抽象化します。

責務

  • **汎用的な ALTER ステートメントの実行**: RENAME, SET SCHEMA, OWNER TO, DEPENDS ON の各ステートメントを適切な内部関数へディスパッチする。
  • **カタログ更新の抽象化**: システムカタログのタプルを読み込み、権限チェックを行い、更新を適用する一連の定型作業（AlterObjectRename_internal 等）の提供。
  • **依存関係と権限の管理**: スキーマ移動時の依存関係更新や、所有者変更に伴うACL（アクセス制御リスト）の再計算。

重要なシンボル

  • ExecRenameStmt: RENAME ステートメントのメインエントリポイント。
  • ExecAlterObjectSchemaStmt: SET SCHEMA ステートメントのメインエントリポイント。
  • ExecAlterOwnerStmt: OWNER TO ステートメントのメインエントリポイント。
  • AlterObjectRename_internal: 汎用的な名前変更ロジック。
  • AlterObjectNamespace_internal: 汎用的なスキーマ移動ロジック。
  • AlterObjectOwner_internal: 汎用的な所有者変更ロジック。

制御フローとデータフロー

1. **ディスパッチ**: Exec...Stmt 関数がパーサーから渡された Stmt ノードを受け取り、オブジェクト型に応じて処理を分岐します。 2. **準備**: get_object_address を使用して対象オブジェクトのOIDとカタログ情報を特定し、ロックを取得します。 3. **検証**: superuser() チェックや has_privs_of_role を用いた所有権・権限チェックを行います。 4. **更新**: SearchSysCacheCopy でタプルを取得し、heap_modify_tuple で変更を適用後、CatalogTupleUpdate でカタログを更新します。 5. **フック**: InvokeObjectPostAlterHook を呼び出し、イベントトリガーやキャッシュ無効化などの後処理を実行します。

直接依存

  • catalog/: システムカタログ定義（pg_proc, pg_namespace 等）および依存関係管理。
  • access/: table.h や htup_details.h を介した低レベルなタプル操作。
  • utils/: syscache.h（カタログキャッシュ）、acl.h（権限チェック）、lsyscache.h（OIDからの情報取得）。

読者向けメモ

  • 本ファイルは「単純なカタログ更新」を対象としています。テーブルやインデックスのように、物理的なストレージや複雑な依存関係を伴うオブジェクトは、各専用のコマンドファイル（tablecmds.c 等）で処理されるため、本ファイルには含まれません。
  • AlterObject..._internal 関数群は、カタログの属性番号（Anum_...）を動的に取得して処理を行うため、新しいオブジェクト型を追加する際は、get_object_attnum_... 系関数が正しく実装されている必要があります。
