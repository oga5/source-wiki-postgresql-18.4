src/backend/commands/explain_dr.c

概要

explain_dr.c は、EXPLAIN (SERIALIZE) オプション実行時に使用される特殊な DestReceiver を実装するモジュールです。クエリ結果をクライアントへ送信する代わりに、シリアライズ処理（データ型の変換やTOAST解除など）のオーバーヘッドを計測・収集する役割を担います。

モジュール内での役割

src/backend/commands サブシステムの一部として、EXPLAIN コマンドの計測機能を拡張します。通常の DestReceiver がネットワーク経由でのデータ転送を目的とするのに対し、本モジュールは「シリアライズにかかるコスト」を定量化するためのサンドボックスとして機能します。

責務

  • **シリアライズのシミュレーション**: printtup と同様のロジックでデータを構築し、実際のネットワーク送信をスキップする。
  • **リソース計測**: シリアライズ処理中の CPU 時間、バッファアクセス数、および生成されたバイト数を集計する。
  • **メモリ管理**: 行ごとの一時的なメモリコンテキストを管理し、大規模な結果セットでもメモリリークを防ぐ。

重要なシンボル

  • SerializeDestReceiver: 計測状態を保持する構造体。DestReceiver を継承し、フォーマット情報や計測メトリクスを保持する。
  • CreateExplainSerializeDestReceiver: EXPLAIN 実行時に計測用レシーバーを生成するファクトリ関数。
  • serializeAnalyzeReceive: 各タプルを受け取り、シリアライズ処理と計測を行うメインループ。
  • GetSerializationMetrics: 収集された計測結果を ExplainState に引き渡すためのインターフェース。

追加の索引済みシンボル

  • serializeAnalyzeDestroy
  • serializeAnalyzeShutdown

制御フローとデータフロー

1. **初期化**: CreateExplainSerializeDestReceiver でレシーバーが作成され、serializeAnalyzeStartup でメモリコンテキストとバッファが初期化される。 2. **データ処理**: serializeAnalyzeReceive が呼び出されるたびに、以下の処理が実行される：

  • serialize_prepare_info で出力関数（typoutput / typsend）をキャッシュ。
  • slot_getallattrs でタプルをデコード。
  • pq_send... 系関数で StringInfo バッファにデータを構築（送信はしない）。
  • 経過時間とバッファ使用量を計測し、metrics に加算。

    • 3. **終了**: serializeAnalyzeShutdown で確保したメモリとリソースを解放する。

読者向けメモ

  • 本モジュールは「実際にデータを送らない」という点が重要です。pq_endmessage_reuse を呼び出さず、バッファの長さ（len）だけをカウントすることで、ネットワーク負荷を除外した純粋なシリアライズコストを測定しています。
  • CREATE TABLE AS など、他の DestReceiver が使われるケースでは GetSerializationMetrics がゼロを返すよう設計されており、安全に呼び出せるようになっています。
