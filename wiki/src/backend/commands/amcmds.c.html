src/backend/commands/amcmds.c

概要

PostgreSQL の「アクセスメソッド（AM）」を管理するための SQL コマンド処理を実装するモジュールです。主に CREATE ACCESS METHOD 文の実行と、システムカタログ pg_am へのアクセスを担います。

モジュール内での役割

src/backend/commands ディレクトリの一部として、データベースの拡張性に関わるメタデータ操作を提供します。インデックスやテーブルのストレージエンジンを定義する際のバックエンド処理を抽象化し、他のコマンド処理から利用可能な API を提供します。

責務

  • **アクセスメソッドの登録**: CREATE ACCESS METHOD 文を受け取り、pg_am カタログへの挿入と依存関係の記録を行う。
  • **バリデーション**: ハンドラ関数のシグネチャ（引数と戻り値の型）が、指定された AM タイプ（INDEX/TABLE）に適合しているか検証する。
  • **検索・解決**: 名前や OID を用いたアクセスメソッド情報の取得と、型チェックの提供。

重要なシンボル

  • CreateAccessMethod: CREATE ACCESS METHOD 文のメインエントリポイント。
  • get_index_am_oid / get_table_am_oid: 名前から特定の型の AM OID を取得するヘルパー。
  • lookup_am_handler_func: ハンドラ関数の存在確認と型チェックを行う内部関数。

追加の索引済みシンボル

  • get_am_name
  • get_am_oid
  • get_am_type_oid
  • get_am_type_string

制御フローとデータフロー

1. **登録フロー**:

  • CreateAccessMethod が pg_am をロックし、権限（スーパーユーザー）と名前の重複を確認。
  • lookup_am_handler_func でハンドラ関数の OID を取得し、戻り値の型が AM タイプと一致するか検証。
  • heap_form_tuple で新しい行を作成し、CatalogTupleInsert でカタログへ書き込む。
  • recordDependencyOn を呼び出し、ハンドラ関数および現在の拡張（Extension）との依存関係を登録。

    • 2. **検索フロー**:

  • get_am_type_oid が共通の検索ロジックを提供。syscache を使用して pg_am から情報を取得し、必要に応じて型チェックを行う。

読者向けメモ

  • 本ファイルは「定義（DDL）」を扱うものであり、アクセスメソッドの「実行時動作」は含まれていません。
  • ハンドラ関数は internal 型を引数に取り、特定の型（index_am_handler や table_am_handler）を返す必要があるという制約が lookup_am_handler_func で厳格にチェックされています。
  • 新しいアクセスメソッドを追加する際は、必ず pg_am への登録と適切な依存関係の記録が必須となります。
