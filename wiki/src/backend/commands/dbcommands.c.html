src/backend/commands/dbcommands.c

概要

データベースの作成（CREATE DATABASE）、削除（DROP DATABASE）、名前変更（ALTER DATABASE RENAME）、およびテーブルスペースの移動（ALTER DATABASE SET TABLESPACE）を管理する中核モジュールです。物理的なディレクトリ操作とシステムカタログ（pg_database）の更新を同期させ、WAL（Write-Ahead Logging）による整合性保証を行います。

モジュール内での役割

src/backend/commands モジュールの一部として、データベースレベルのライフサイクルを制御します。テーブルやインデックスといった個別のオブジェクト管理とは異なり、データベース全体を一つの物理的単位として扱い、ストレージマネージャ（smgr）やトランザクション管理と密接に連携します。

責務

  • **データベースのライフサイクル管理**: データベースの作成、削除、名前変更の実行。
  • **物理ストレージの制御**: データベースディレクトリの作成・削除、PG_VERSION ファイルの管理、テーブルスペース間のデータ移動。
  • **整合性の維持**: データベース作成時のテンプレートからのコピー、削除時のバッファキャッシュおよび保留中の同期要求のクリーンアップ。
  • **WAL 記録**: データベース操作の WAL ログ生成と、リカバリ時の再実行（dbase_redo）。
  • **権限と競合の検証**: CREATEDB 権限の確認、接続中のバックエンドの排除、名前や OID の競合チェック。

重要なシンボル

データベース操作のメイン関数

  • createdb: CREATE DATABASE コマンドの実行。戦略（WALログまたはファイルコピー）を選択し、カタログ更新と物理コピーを行う。
  • dropdb: DROP DATABASE コマンドの実行。接続の終了、カタログ削除、物理ディレクトリの削除を行う。
  • RenameDatabase: データベース名の変更。
  • movedb: データベースのデフォルトテーブルスペースの変更。

内部戦略と補助構造体

  • CreateDBStrategy: データベース作成戦略（CREATEDB_WAL_LOG vs CREATEDB_FILE_COPY）。
  • CreateDBRelInfo: データベースコピー時に処理対象となるリレーションのメタデータ。
  • createdb_failure_params / movedb_failure_params: エラー発生時のクリーンアップ用パラメータ。

データベース情報取得・検証

  • get_db_info: 指定されたデータベースの情報を取得し、ロックを獲得する。
  • check_encoding_locale_matches: データベース作成時のエンコーディングとロケールの互換性チェック。
  • check_db_file_conflict: 新規作成するデータベースの OID が既存のディレクトリと衝突しないか確認。

制御フローとデータフロー

1. **ロックの獲得**: get_db_info を通じて対象データベースに AccessExclusiveLock を取得し、並行操作を防止。 2. **検証**: 権限チェック、接続中のバックエンドの確認（CountOtherDBBackends）、名前の重複チェック。 3. **カタログ更新**: pg_database へのタプル挿入または更新。 4. **物理操作**:

  • CreateDatabaseUsingWalLog: ブロック単位でコピーし WAL に記録。
  • CreateDatabaseUsingFileCopy: ファイルシステムレベルでコピーし、チェックポイントを強制して WAL を記録。

    • 5. **エラーハンドリング**: PG_ENSURE_ERROR_CLEANUP を使用し、失敗時に作成途中のディレクトリを削除するコールバックを登録。 6. **コミット**: ForceSyncCommit により、物理的な変更とカタログ更新の同期を保証。

読者向けメモ

  • **メモリ所有権**: get_db_info で取得する文字列（dbCollate 等）は palloc されており、呼び出し元で管理が必要です。
  • **ロックの重要性**: データベースの作成・削除・移動は AccessExclusiveLock を使用します。これは pg_database の行ロックよりも粒度が細かく、並行性を高めるための設計です。
  • **リカバリの注意点**: dbase_redo はリカバリ時に呼び出されます。物理ディレクトリが欠損している場合、リカバリを継続するためにディレクトリを再作成するロジックが含まれています。
  • **省略された詳細**: 本ファイルは非常に大規模なため、個別の ALTER DATABASE オプション（SET 等）の細かな構文解析ロジックは省略していますが、これらは AlterDatabase 関数内で DefElem を走査することで処理されます。
