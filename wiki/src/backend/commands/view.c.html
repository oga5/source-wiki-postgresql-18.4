src/backend/commands/view.c

概要

PostgreSQLにおけるビュー（CREATE VIEW / CREATE OR REPLACE VIEW）の定義および更新を管理するモジュールです。ビューの実体であるリレーションの作成と、そのビューを定義するクエリをルールシステム（ON SELECTルール）として登録する処理を担います。

モジュール内での役割

src/backend/commands サブシステムの一部として、DDLコマンドの実行を担います。パーサーが生成した ViewStmt を受け取り、カタログ操作やリライトルールの生成を通じて、ビューをシステムカタログに永続化します。

責務

  • **ビューの作成と更新**: DefineView を起点として、ビューのメタデータ作成とクエリの保存を行います。
  • **整合性チェック**: CREATE OR REPLACE VIEW において、既存のビュー定義と新しい定義の列構成（型、名前、照合順序）に互換性があるかを検証します。
  • **ルール登録**: ビューの定義クエリを ON SELECT ルールとしてルールシステムに登録します。
  • **バリデーション**: WITH CHECK OPTION の妥当性確認や、サポートされていない構文（SELECT INTOなど）の排除を行います。

重要なシンボル

  • DefineView: ビュー作成コマンドのメインエントリーポイント。
  • DefineVirtualRelation: ビューのリレーションを作成し、必要に応じて既存ビューの置換処理を行う内部関数。
  • StoreViewQuery: ビューの定義クエリをルールとして保存するラッパー。
  • checkViewColumns: 既存ビューと新定義の列定義を比較し、変更不可な属性（型や名前）の変更を阻止する。

追加の索引済みシンボル

  • DefineViewRules

制御フローとデータフロー

1. **解析**: DefineView が parse_analyze_fixedparams を呼び出し、生のSQLを Query 構造体に変換します。 2. **検証**: WITH CHECK OPTION の整合性や、エイリアスの適用、永続性のチェックを行います。 3. **リレーション作成**: DefineVirtualRelation が呼び出されます。

  • replace が真の場合、既存のビューをロックし、checkViewColumns で互換性を確認します。
  • 列が増える場合は AlterTableInternal を使用して列を追加します。
  • replace が偽の場合、DefineRelation で新規リレーションを作成します。

    • 4. **ルール保存**: StoreViewQuery を経由して DefineQueryRewrite が呼び出され、ON SELECT ルールとしてクエリが登録されます。

読者向けメモ

  • ビューは実体を持たないため、RELKIND_VIEW として作成されます。
  • CREATE OR REPLACE VIEW は、既存のビューを完全に作り直すのではなく、列の追加やルールの更新を行うことで「置換」を実現しています。
  • ビューの列定義（型や照合順序）の変更は、他のビューやルールとの依存関係を壊す可能性があるため、厳格に禁止されています。
