src/backend/commands/createas.c

概要

CREATE TABLE AS (CTAS) および CREATE MATERIALIZED VIEW コマンドの実行を担うモジュールです。クエリの実行結果を通常のクライアントへの返送ではなく、新規作成したテーブルへ直接流し込むための専用の DestReceiver を提供します。

モジュール内での役割

src/backend/commands 内において、クエリ結果を永続化するコマンドの実行エンジンとして機能します。tablecmds.c がテーブル定義の物理的な作成を担うのに対し、本ファイルは「クエリ実行」と「結果の挿入」を統合する高レベルな制御フローを管理します。

責務

  • **ターゲットテーブルの作成**: DefineRelation を呼び出し、CTAS 用のスキーマを構築する。
  • **DestReceiver の実装**: クエリ実行エンジンから流れてくるタプルを、新規作成したテーブルへ挿入するインターフェース（intorel_* 関数群）の提供。
  • **実行制御**: WITH NO DATA オプションの有無に応じた実行パスの分岐（プランナのスキップやマテリアライズドビューの更新処理）。

重要なシンボル

  • ExecCreateTableAs: CTAS コマンド実行のメインエントリポイント。
  • DR_intorel: クエリ結果をテーブルに書き込むための DestReceiver 構造体。
  • create_ctas_internal: DefineRelation をラップし、TOAST テーブル作成やマテリアライズドビューのメタデータ保存を行うユーティリティ。
  • intorel_receive: DestReceiver の一部として、各タプルをテーブルに挿入する関数。

制御フローとデータフロー

1. **初期化**: ExecCreateTableAs が呼び出され、CreateIntoRelDestReceiver で DR_intorel を初期化。 2. **定義作成**: WITH NO DATA の場合は create_ctas_nodata を経由し、それ以外は intorel_startup 内で DefineRelation を呼び出しテーブルを物理作成。 3. **実行**:

  • WITH NO DATA でない場合、ExecutorStart / Run を呼び出し、クエリ結果を intorel_receive に流し込む。
  • intorel_receive は table_tuple_insert を使用してデータを格納。

    • 4. **終了**: intorel_shutdown でバルク挿入状態をクリーンアップし、テーブルを閉じる。

読者向けメモ

  • **レガシーな設計**: CTAS は歴史的に SELECT の亜種として実装されており、QueryCompletion への行数報告など、その名残がコードの随所に見られます。
  • **WITH NO DATA**: このオプションが指定された場合、プランナやエグゼキュータを完全にバイパスしてテーブル定義のみを作成します。このパスはダンプ/リストア時の依存関係問題を回避するために重要です。
  • **RLS の制限**: 現時点では CTAS で作成されるテーブルに対して行レベルセキュリティ (RLS) はサポートされておらず、intorel_startup で明示的にエラーチェックを行っています。
