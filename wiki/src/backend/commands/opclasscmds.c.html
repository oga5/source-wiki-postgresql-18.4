src/backend/commands/opclasscmds.c

概要

PostgreSQLのインデックス・アクセスメソッド（AM）に関連する「演算子クラス（Operator Class）」および「演算子ファミリー（Operator Family）」の定義・変更を管理するコマンド処理モジュールです。

モジュール内での役割

src/backend/commands ディレクトリの一部として、DDLコマンド（CREATE OPERATOR CLASS, CREATE OPERATOR FAMILY, ALTER OPERATOR FAMILY）の実行エンジンを提供します。カタログ（pg_opclass, pg_opfamily, pg_amop, pg_amproc）への書き込みと、それらに付随する依存関係の構築を担います。

責務

  • **カタログ操作**: 演算子クラス/ファミリーの新規作成および削除時のカタログ更新。
  • **依存関係管理**: 演算子やサポート関数と、それらを利用するクラス/ファミリー間の依存関係（pg_depend）の構築。
  • **検証**: 演算子やサポート関数のシグネチャが、対象AMの期待する仕様（引数の数、戻り値の型など）に適合しているかのチェック。
  • **キャッシュ検索**: 名前からOIDを解決するためのシステムキャッシュ検索機能の提供。

重要なシンボル

  • DefineOpClass: CREATE OPERATOR CLASS コマンドのメインエントリポイント。
  • DefineOpFamily: CREATE OPERATOR FAMILY コマンドのメインエントリポイント。
  • AlterOpFamily: ALTER OPERATOR FAMILY のエントリポイント。ADD/DROPの振り分けを行う。
  • OpFamilyCacheLookup / OpClassCacheLookup: 名前からシステムカタログのタプルを取得する内部ヘルパー。
  • assignOperTypes / assignProcTypes: 演算子や関数がインデックスAMの要件を満たすか検証し、型情報を割り当てる。

制御フローとデータフロー

1. **解析**: SQL文から名前や引数リストを抽出し、QualifiedNameGetCreationNamespace 等で名前空間を特定。 2. **検証**: GetIndexAmRoutineByAmId でAMのルーチンを取得し、指定された演算子や関数がそのAMの戦略（Strategy）やサポート番号（Support Number）の範囲内か確認。 3. **構築**: OpFamilyMember 構造体を使用して、追加/削除対象のメンバー情報をリスト化。 4. **永続化**: storeOperators / storeProcedures がカタログ（pg_amop / pg_amproc）へ挿入し、recordDependencyOn で依存関係を記録。 5. **通知**: EventTrigger への通知を行い、DDLイベントをトリガーする。

読者向けメモ

  • **権限**: 現在、演算子クラス/ファミリーの作成・変更にはスーパーユーザー権限が必要です。これは、AMの期待する整合性をユーザーレベルで検証することが困難であるためです。
  • **依存関係**: ALTER OPERATOR FAMILY ... ADD は「ソフト」な依存関係を作成しますが、CREATE OPERATOR CLASS 内での定義は「ハード」な依存関係を作成します。この違いは ref_is_hard フラグで制御されています。
  • **検証ロジック**: assignProcTypes 内の switch 文や if 文は、各AM（B-tree, Hash等）固有のサポート関数シグネチャを強制する重要な箇所です。新しいAMを追加する際はここが参照されることが多いです。
