src/backend/commands/trigger.c

概要

PostgreSQL のトリガー（Trigger）サブシステムの中核を担う実装ファイルです。DDL コマンド（CREATE TRIGGER, ALTER TRIGGER 等）によるトリガー定義の管理、および実行時におけるトリガーの起動制御（BEFORE/AFTER/INSTEAD OF）を統括します。

モジュール内での役割

src/backend/commands モジュールの一部として、テーブルやビューに対するイベント駆動型の処理を定義します。エグゼキュータ（executor/）がデータ操作を行う際に呼び出されるトリガー実行関数を提供し、システムカタログ（pg_trigger）とリレーションキャッシュ（relcache）の整合性を維持します。

責務

  • **DDL 管理**: トリガーの作成、削除、名前変更、有効/無効化の実行。
  • **実行時ディスパッチ**: SQL 操作（INSERT/UPDATE/DELETE/TRUNCATE）に応じたトリガー関数の起動。
  • **状態管理**: AFTER トリガーの遅延実行（Deferred）管理と、トランザクション内でのイベントキューイング。
  • **遷移テーブル管理**: REFERENCING 句で指定された遷移テーブル（Transition Tables）の構築と保持。
  • **整合性チェック**: パーティションテーブルや継承関係におけるトリガーの伝播と制約の検証。

重要なシンボル

トリガー定義・管理

  • CreateTrigger: トリガー作成のメインエントリポイント。カタログ更新と依存関係の記録を行う。
  • RemoveTriggerById: トリガーの削除。
  • renametrig: トリガーの名前変更。
  • EnableDisableTrigger: トリガーの有効/無効状態の変更。

実行時ディスパッチ

  • ExecBRInsertTriggers / ExecARInsertTriggers 等: 各操作（INSERT/UPDATE/DELETE/TRUNCATE）の BEFORE/AFTER 行レベルトリガーを実行するエグゼキュータ用関数。
  • ExecCallTriggerFunc: トリガー関数を呼び出すための低レベルラッパー。FmgrInfo を使用して関数呼び出しを最適化する。

内部状態・キャッシュ

  • RelationBuildTriggers: リレーションキャッシュ構築時に pg_trigger からトリガー情報を読み込み、TriggerDesc を生成する。
  • afterTriggers: トランザクション内で遅延実行されるトリガーイベントを保持する静的構造体。
  • MyTriggerDepth: 現在のトリガー実行のネスト深さを追跡する変数。

制御フローとデータフロー

1. **DDL フロー**: CreateTrigger が pg_trigger を更新し、CommandCounterIncrement() を経てリレーションキャッシュを無効化（CacheInvalidateRelcache）することで、全バックエンドに定義変更を伝播させます。 2. **実行フロー**: エグゼキュータが ExecBR... 系の関数を呼び出すと、TriggerEnabled で実行条件（GUC SessionReplicationRole や WHEN 句）を評価し、ExecCallTriggerFunc で関数を起動します。 3. **遅延実行フロー**: AFTER トリガーは AfterTriggerSaveEvent でキューイングされます。トランザクション終了時や SET CONSTRAINTS 実行時に AfterTriggerFireDeferred が呼ばれ、キューに溜まったイベントが afterTriggerInvokeEvents によって順次処理されます。

読者向けメモ

  • **メモリ管理**: TriggerDesc はリレーションキャッシュの一部として CacheMemoryContext に保持されます。一方、実行時のトリガーイベントや遷移テーブルは CurTransactionContext や AfterTriggerEvents コンテキストで管理され、トランザクション終了時に自動解放されます。
  • **制約事項**: 外部テーブル（Foreign Table）に対する遅延トリガーはサポートされていません。
  • **パーティション**: パーティションテーブルに対するトリガー操作は、再帰的に全パーティションへ適用される設計です。
  • **浅い領域**: AfterTriggerExecute 内の複雑なメモリコンテキスト切り替えや、AfterTriggerEventChunk によるイベントリストのチャンク管理は、メモリ効率を最大化するための最適化であり、デバッグ時は注意が必要です。
