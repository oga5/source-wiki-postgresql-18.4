src/backend/commands/variable.c

概要

本ファイルは、PostgreSQLのGUC（Grand Unified Configuration）パラメータのうち、複雑な検証ロジックや状態遷移を伴う「特殊な変数」のハンドリングを実装しています。主に SET コマンドによって実行される設定変更のバリデーションと適用を担当します。

モジュール内での役割

src/backend/commands モジュールの一部として、SQLレベルでの設定変更要求をシステム内部の状態へ反映させる仲介役です。単純な数値や文字列の代入ではなく、システムカタログの参照、トランザクション状態の確認、あるいは外部リソース（タイムゾーンファイル等）の読み込みが必要なパラメータを扱います。

責務

  • **バリデーション (check_hook)**: 設定値が構文的に正しいか、現在のトランザクション状態や権限で変更可能かを検証します。
  • **適用 (assign_hook)**: 検証済みの値を内部変数へ反映させ、必要に応じて関連するキャッシュのクリアや統計情報の更新を行います。
  • **表示 (show_hook)**: 特殊なフォーマット（8進数表記など）が必要な変数の出力形式を制御します。

重要なシンボル

  • check_datestyle / assign_datestyle: 日付形式の複雑な構文解析と状態反映。
  • check_timezone / assign_timezone: タイムゾーン名の解決とセッション状態への適用。
  • check_transaction_read_only / check_transaction_isolation: トランザクション開始後の変更制限など、SQL標準に準拠した制約チェック。
  • check_session_authorization / check_role: ユーザー権限の検証とセッションIDの切り替え。
  • check_client_encoding: クライアントエンコーディングの検証と変換プロセスの準備。

制御フローとデータフロー

1. **GUCサブシステムからの呼び出し**: ユーザーが SET を実行すると、GUCフレームワークが対応する check_hook を呼び出します。 2. **検証**: check_hook 内で pstrdup 等を用いて値を解析し、エラーがあれば GUC_check_errdetail で報告して false を返します。 3. **中間データの受け渡し**: 検証に成功した場合、extra 引数を通じて適用に必要な構造体や値を assign_hook へ渡します。 4. **適用**: assign_hook が呼び出され、グローバル変数やセッション状態を更新します。

読者向けメモ

  • **トランザクション制約**: check_transaction_* 系関数は、トランザクションの進行状況（FirstSnapshotSet など）を厳密にチェックしています。これらは並列ワーカーの初期化時など、特殊なコンテキストでの挙動に注意が必要です。
  • **メモリ管理**: check_hook 内で extra に割り当てたメモリは、GUCサブシステムが管理します。guc_malloc を使用し、メモリリークを防ぐ設計になっています。
  • **エラーハンドリング**: check_hook 内でのエラー報告は、ユーザーに分かりやすいメッセージを出すために GUC_check_errmsg 等の専用マクロを使用してください。
