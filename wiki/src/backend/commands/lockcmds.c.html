src/backend/commands/lockcmds.c

概要

lockcmds.c は、PostgreSQL の LOCK TABLE SQL コマンドを実行するためのバックエンドロジックを実装しています。指定されたリレーションに対して明示的なロックを獲得し、継承関係にあるテーブルやビューの定義に含まれる依存リレーションに対しても再帰的にロックを適用する役割を担います。

モジュール内での役割

src/backend/commands モジュールの一部として、DDL やコマンド実行系とストレージマネージャ（ロック管理）の橋渡しを行います。主に storage/lmgr.h の低レベルなロック関数を呼び出し、SQL レベルのセマンティクス（継承やビューの再帰的ロック）を適用します。

責務

  • **コマンド実行**: LockTableCommand を通じて、ユーザーが指定したリレーションのロック処理を開始する。
  • **権限チェック**: LockTableAclCheck を使用し、ロック対象のリレーションに対してユーザーが適切な権限（UPDATE, DELETE 等）を持っているか検証する。
  • **再帰的ロック**:
  • 継承テーブルに対しては LockTableRecurse で子テーブルを探索しロックする。
  • ビューに対しては LockViewRecurse でビュー定義を解析し、参照されているテーブルやビューを再帰的にロックする。

重要なシンボル

  • LockTableCommand: LOCK TABLE 文のメインエントリーポイント。
  • LockTableRecurse: 継承ツリーを辿り、すべての子テーブルにロックを適用する。
  • LockViewRecurse: ビューのクエリツリーを走査し、依存するリレーションを特定してロックする。
  • LockTableAclCheck: ロックモードに応じた権限チェックを行う。

追加の索引済みシンボル

  • LockViewRecurse_context
  • LockViewRecurse_walker
  • RangeVarCallbackForLockTable

制御フローとデータフロー

1. **解析**: LockTableCommand が LockStmt を受け取り、対象リレーションを一つずつ処理する。 2. **検証**: RangeVarCallbackForLockTable が呼び出され、リレーションの存在確認、型チェック（テーブル/ビューのみ許可）、および権限チェックを行う。 3. **再帰処理**:

  • 継承テーブルの場合: find_all_inheritors で全子テーブルを取得し、順次ロックする。
  • ビューの場合: LockViewRecurse がビューのクエリ定義を読み込み、LockViewRecurse_walker（query_tree_walker を利用）がクエリ内の RangeTblEntry を再帰的に探索してロックを適用する。

    • 4. **ロック適用**: LockRelationOid または ConditionalLockRelationOid（NOWAIT 指定時）を使用してロックを獲得する。

読者向けメモ

  • **ビューのロック**: ビューをロックする場合、そのビューが参照しているすべてのテーブル・ビューもロックされます。これは security_invoker プロパティに応じて、ビュー所有者または現在のユーザーの権限でチェックされます。
  • **NOWAIT の挙動**: NOWAIT が指定されている場合、ロックが即座に取得できないとエラーを発生させます。
  • **再帰の停止**: ビューの再帰処理中、循環参照を防ぐために ancestor_views リストで既にロック済みのビューを追跡しています。
