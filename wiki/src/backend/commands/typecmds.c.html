src/backend/commands/typecmds.c

概要

typecmds.c は、PostgreSQL におけるデータ型（BASE, DOMAIN, ENUM, RANGE, MULTIRANGE, COMPOSITE）の定義、変更、および削除を行う DDL コマンドの実行エンジンです。システムカタログ pg_type を中心としたメタデータ管理と、型に関連するサポート関数（I/O、分析、サブスクリプト等）の整合性を保証します。

モジュール内での役割

src/backend/commands モジュールの一部として、SQL パーサーが生成した構文木（CreateTypeStmt 等）を解釈し、カタログ操作関数を呼び出します。tablecmds.c がテーブル構造を扱うのに対し、本ファイルは型システムそのもののライフサイクルを管理します。

責務

  • **型定義のディスパッチ**: DefineType, DefineDomain, DefineEnum, DefineRange 等による型作成の統括。
  • **サポート関数の検証**: 型の I/O 関数、分析関数、サブスクリプト関数等のシグネチャと権限の検証。
  • **依存関係の管理**: 型の削除時や変更時に、関連する配列型、マルチレンジ型、および依存する制約との整合性を維持。
  • **ALTER 操作の再帰的適用**: ALTER TYPE 実行時に、その型に依存するドメインや配列型へ変更を伝播させる。

重要なシンボル

型定義・作成

  • DefineType: 基本型（Base Type）の作成。配列型の自動生成も行う。
  • DefineDomain: ドメイン型の作成。制約（CHECK, NOT NULL）の処理を含む。
  • DefineEnum: 列挙型の作成。pg_enum への値登録を行う。
  • DefineRange: 範囲型の作成。マルチレンジ型や関連するコンストラクタ関数の生成も行う。

型変更・削除

  • RemoveTypeById: 型の削除。pg_enum や pg_range 等の関連カタログのクリーンアップも実行。
  • AlterType: 型の属性（STORAGE, I/O 関数等）の変更。
  • AlterTypeRecurse: 型変更をドメインや配列型へ再帰的に適用する。
  • AlterDomainDefault / AlterDomainNotNull: ドメインの制約変更。

サポート関数検索

  • findTypeInputFunction / findTypeOutputFunction: I/O 関数のシグネチャ検証と OID 取得。
  • findTypeAnalyzeFunction / findTypeSubscriptingFunction: 分析およびサブスクリプト関数の検証。

制御フローとデータフロー

1. **コマンド受信**: ProcessUtility から各 Define* 関数が呼び出される。 2. **検証**: QualifiedNameGetCreationNamespace で名前空間を特定し、superuser() や object_ownercheck で権限を確認。 3. **カタログ更新**: TypeCreate を呼び出し、pg_type にレコードを挿入。必要に応じて pg_enum や pg_range を更新。 4. **依存関係の構築**: recordDependencyOn を使用し、型と関数、型とスキーマ間の依存関係を記録。 5. **再帰的更新**: AlterType 等では AlterTypeRecurse を呼び出し、pg_type の typbasetype をキーにスキャンを行い、依存するドメインを特定して変更を伝播させる。

読者向けメモ

  • **メモリ管理**: RelToCheck 構造体や AlterTypeRecurseParams はスタックまたは palloc で確保されます。再帰呼び出しが深くなる可能性があるため、check_stack_depth() が随所で呼ばれています。
  • **制約の検証**: validateDomainCheckConstraint は、ドメインに制約を追加する際、既存の全テーブルデータをスキャンして検証します。これは長時間実行される可能性があるため注意が必要です。
  • **未実装・浅い部分**: get_rels_with_domain は、複合型や配列の中に埋め込まれたドメインのチェックが不完全である旨のコメントがあり、現状ではエラーを投げる仕様になっています。
  • **バイナリアップグレード**: binary_upgrade_next_* 変数は pg_upgrade 用であり、通常運用では使用されません。
