src/backend/commands/tablecmds.c

概要

tablecmds.c は、PostgreSQL におけるテーブル、インデックス、シーケンス、ビュー、マテリアライズドビュー、外部テーブルの構造変更（DDL）を司る中核的なソースファイルです。CREATE, ALTER, DROP, TRUNCATE コマンドの実行ロジックを保持し、システムカタログの更新、物理ストレージの再構築、依存関係の管理、およびパーティショニングに関連する複雑な状態遷移を制御します。

モジュール内での役割

src/backend/commands サブシステムにおいて、DDL コマンドの実行エンジンとして機能します。パーサーが生成した構文木を受け取り、カタログ操作（catalog/）やストレージ管理（storage/）と連携して、データベーススキーマの変更をトランザクションの整合性を保ちながら実行します。

責務

  • **DDL コマンドのディスパッチと実行**: DefineRelation や AlterTable を中心としたコマンドのライフサイクル管理。
  • **スキーマ変更の多段階実行**: ALTER TABLE におけるフェーズ 1（準備・検証）、フェーズ 2（カタログ更新）、フェーズ 3（データスキャン・再構築）の制御。
  • **継承とパーティショニングの管理**: 継承関係のカタログ更新、パーティションのバインド管理、インデックスやトリガーのクローン作成。
  • **制約の管理**: CHECK, NOT NULL, FOREIGN KEY 制約の追加・削除・検証。
  • **物理ストレージの再構築**: TRUNCATE や ALTER TABLE によるテーブル再構築時のファイル操作。

重要なシンボル

ALTER TABLE 制御構造

  • AlteredTableInfo: ALTER TABLE 実行中にテーブルごとの変更状態を保持する構造体。
  • AlterTablePass: コマンド実行順序を制御するフェーズ（DROP, ADD_COL, ADD_CONSTR 等）。
  • AT_NUM_PASSES: 実行フェーズの総数。

制約・列管理

  • NewConstraint: フェーズ 3 で検証が必要な制約情報を保持。
  • NewColumnValue: 型変更やデフォルト値計算が必要な列の情報を保持。

ユーティリティ・コールバック

  • DefineRelation: CREATE TABLE のメインエントリポイント。
  • RemoveRelations: DROP コマンドの共通処理。
  • ExecuteTruncate: TRUNCATE コマンドの実行。
  • RangeVarCallbackFor...: RangeVar 解決時に権限やロックを検証するコールバック群。

制御フローとデータフロー

1. **フェーズ 1 (Prep)**: ATController が ATPrepCmd を呼び出し、サブコマンドを AlteredTableInfo の subcmds 配列（AlterTablePass ごとに分類）にキューイングします。 2. **フェーズ 2 (Catalogs)**: ATRewriteCatalogs が各パスを順に実行し、ATExecCmd を通じてカタログを更新します。 3. **フェーズ 3 (Rewrite)**: ATRewriteTables が必要に応じてテーブルをスキャンまたは再構築します。制約の検証やデフォルト値の計算はここで実行されます。 4. **依存関係**: find_composite_type_dependencies や RememberAllDependentForRebuilding を使用して、変更対象に依存するオブジェクトを特定し、再構築を予約します。

読者向けメモ

  • **複雑性**: 本ファイルは非常に巨大であり、継承、パーティショニング、外部テーブル、型変更など、多くの機能が混在しています。特定の機能（例: ALTER TABLE）を追う場合は、ATController からの呼び出し階層を追うのが推奨されます。
  • **メモリ管理**: AlteredTableInfo は CacheMemoryContext 等の適切なメモリコンテキストで管理される必要があります。
  • **ロック**: AccessExclusiveLock が多用されます。デッドロックを避けるため、親テーブルから子テーブル、あるいはテーブルからインデックスという順序でロックを取得する設計になっています。
  • **省略**: 本ファイルには TRUNCATE の詳細な物理削除ロジックや、パーティションのバインド検証ロジックが含まれますが、これらは非常に詳細なため、必要に応じて個別のヘルパー関数を参照してください。
