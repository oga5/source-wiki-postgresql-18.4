src/backend/commands/dropcmds.c

概要

dropcmds.c は、SQL の DROP 文（DROP FUNCTION, DROP TYPE 等）の実行を統括するモジュールです。個別のオブジェクト削除ロジックを呼び出す前に、対象オブジェクトの解決、権限チェック、および依存関係の事前確認を一元的に行います。

モジュール内での役割

src/backend/commands サブシステムにおいて、汎用的な削除処理の入り口を提供します。特定のオブジェクト型（テーブルやインデックスなど）は個別のファイルで処理されますが、本ファイルはそれら以外の多様なオブジェクトに対する共通の削除フローを定義しています。

責務

  • **オブジェクトの解決**: get_object_address を使用して、SQL 文で指定されたオブジェクトの OID を特定します。
  • **一括削除の準備**: 複数のオブジェクトを一度に削除するため、ObjectAddresses リストに収集します。
  • **権限チェック**: 削除対象に対する所有権やスキーマ権限を検証します。
  • **IF EXISTS のハンドリング**: オブジェクトが存在しない場合に、適切な NOTICE メッセージを生成して処理をスキップするロジックを管理します。

重要なシンボル

  • RemoveObjects(DropStmt *stmt): 本ファイルのメインエントリポイント。DROP 文のリストを走査し、削除対象を収集・検証します。
  • does_not_exist_skipping(...): IF EXISTS 指定時に、オブジェクトが見つからない理由（スキーマ不在、型不在など）を特定し、適切な警告メッセージを構築するヘルパー群。

制御フローとデータフロー

1. RemoveObjects が DropStmt を受け取り、空の ObjectAddresses リストを作成します。 2. stmt->objects をループし、get_object_address で各オブジェクトの OID を取得します。 3. オブジェクトが存在しない場合、does_not_exist_skipping を呼び出し、missing_ok フラグに従って処理を継続します。 4. 存在する場合は、object_ownercheck 等で権限を検証し、add_exact_object_address でリストに追加します。 5. ループ終了後、performMultipleDeletions を呼び出し、収集したすべてのオブジェクトを依存関係順に削除します。

読者向けメモ

  • 本ファイルは「すべての」DROP を扱うわけではありません。elog(ERROR, "unsupported object type: ...") が含まれるケースは、専用のコマンドハンドラが存在することを意味します。
  • IF EXISTS のエラーメッセージ生成ロジックは非常に詳細に作り込まれています。これは、ユーザーに対して「なぜ削除できなかったのか（スキーマが悪いのか、型が悪いのか）」を明確に伝えるための設計です。
  • performMultipleDeletions を使用することで、DROP RESTRICT によるエラーを最小限に抑え、一括削除時の整合性を保っています。
