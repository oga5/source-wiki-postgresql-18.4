src/backend/commands/constraint.c

概要

本ファイルは、PostgreSQLにおける遅延制約（Deferrable Constraints）の検証ロジックを実装するモジュールです。主に、トランザクションのコミット時や SET CONSTRAINTS コマンド実行時に、一意性制約や排他制約の違反を再確認するためのトリガー関数を提供します。

モジュール内での役割

src/backend/commands ディレクトリの一部として、テーブル定義や制約管理に関連するコマンドの実行をサポートします。特に、制約の検証という「実行時」の整合性維持を担う重要な役割を果たします。

責務

  • **遅延制約の再検証**: unique_key_recheck 関数を通じて、挿入または更新された行が制約に違反していないかを事後的に確認する。
  • **排他制約のサポート**: 一意性制約だけでなく、複雑な排他制約（Exclusion Constraints）のチェックも同一のトリガーメカニズムで処理する。
  • **トリガープロトコルの遵守**: AFTER ROW トリガーとして正しく呼び出されているかを検証し、実行コンテキストを適切に管理する。

重要なシンボル

  • unique_key_recheck: 本ファイルの核となる関数。遅延制約の検証を行うトリガー関数として登録されます。

制御フローとデータフロー

1. **トリガー起動**: AFTER ROW トリガーとして呼び出され、TriggerData から対象のテーブルと行（TID）を取得します。 2. **行の生存確認**: table_index_fetch_tuple を使用し、対象行が現在も有効かを確認します。既に削除されている場合はチェックをスキップします。 3. **インデックス情報の構築**: BuildIndexInfo を用いて、制約対象のインデックス情報を取得します。 4. **値の抽出**: FormIndexDatum を呼び出し、インデックスキーとなる値を抽出します。 5. **制約検証**:

  • 一意性制約の場合: index_insert を UNIQUE_CHECK_EXISTING モードで呼び出し、既存のインデックスエントリと衝突しないか確認します。
  • 排他制約の場合: check_exclusion_constraint を呼び出し、より複雑な条件での競合をチェックします。

直接依存

  • commands/trigger.h: トリガー関連のデータ構造と定義。
  • executor/executor.h: 式の評価や実行状態（EState）の管理。
  • postgres.h

関連コード

  • src/backend/access/nbtree/nbtinsert.c: index_insert の実体。
  • src/backend/executor/execIndexing.c: FormIndexDatum 等のインデックス関連ユーティリティ。

読者向けメモ

  • 本関数は AFTER ROW トリガーとして設計されているため、他のトリガー関数と同様に TriggerData 構造体の解釈が重要です。
  • unique_key_recheck という名前ですが、歴史的経緯により排他制約のチェックも含まれています。
  • HOT（Heap Only Tuple）更新が発生した場合、トリガーが起動しないケースがあるため、インデックスAM側でのチェックと本関数による再チェックの役割分担を理解することが重要です。
