src/backend/commands/portalcmds.c

概要

本ファイルは、PostgreSQLにおける「ポータル（Portal）」に対するSQLコマンド（DECLARE CURSOR, FETCH, MOVE, CLOSE）の実行ロジックを実装しています。ポータルは、クエリの実行状態を保持し、クライアントに対して結果を段階的に提供するための抽象化レイヤーです。

モジュール内での役割

src/backend/commands モジュールの一部として、SQLレベルのカーソル操作を処理します。低レベルなポータルメモリ管理（portalmem.c）や、FE/BEプロトコルレベルのポータル操作（pquery.c）と連携し、クエリ実行エンジンとクライアント間の橋渡しを行います。

責務

  • **カーソルの定義と初期化**: DECLARE CURSOR を受け取り、クエリの解析・書き換え・プラン作成を経てポータルオブジェクトを生成する。
  • **カーソルの操作**: FETCH や MOVE コマンドに基づき、ポータル内のクエリ実行を進め、結果を適切な宛先（DestReceiver）へ転送する。
  • **ライフサイクル管理**: カーソルのクローズ処理および、トランザクション終了後も保持される「ホールド可能カーソル（Holdable Portal）」の永続化処理。

重要なシンボル

  • PerformCursorOpen: DECLARE CURSOR を実行し、プラン作成とポータルの初期化を行う。
  • PerformPortalFetch: FETCH / MOVE コマンドを処理し、ポータルからタプルを取得する。
  • PerformPortalClose: 指定されたポータルを破棄する。
  • PortalCleanup: ポータル破棄時に実行されるクリーンアップフック。実行中のエグゼキュータを安全に停止する。
  • PersistHoldablePortal: トランザクション終了時に、結果セットを Tuplestore に移してカーソルを永続化する。

制御フローとデータフロー

1. **カーソルオープン**: PerformCursorOpen が pg_plan_query を呼び出し、プランを生成。CreatePortal でメモリを確保し、PortalStart で実行準備を完了させる。 2. **データ取得**: PerformPortalFetch が PortalRunFetch を経由してエグゼキュータを駆動。結果は DestReceiver を通じてクライアントへ送られる。 3. **永続化**: PersistHoldablePortal が呼び出されると、エグゼキュータの出力を Tuplestore に切り替え、全結果をメモリ/ディスクに書き出す。その後、エグゼキュータを終了させ、以降のアクセスは Tuplestore から行われるようにする。

読者向けメモ

  • ポータルは「名前付き」と「名前なし（""）」が存在しますが、本ファイル内の関数は空文字列のカーソル名を明示的に拒否します。
  • PersistHoldablePortal は非常に重要な関数です。トランザクション終了時にクエリ結果を Tuplestore へ退避させるため、メモリ消費量やディスクI/Oに影響を与える可能性があります。
  • PortalCleanup はエラー発生時にも呼ばれるため、PORTAL_FAILED 状態のチェックなど、二重エラーを防ぐための堅牢な実装になっています。
