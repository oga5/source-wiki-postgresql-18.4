src/backend/executor/functions.c

概要

SQL 言語で記述された関数（SQL-language functions）の実行を担うモジュールです。プランナが生成したクエリ計画を順次実行し、結果を収集・変換して呼び出し元へ返します。単一の関数内に複数の SQL 文が含まれる場合や、集合を返す関数（SRF）の実行制御を管理します。

モジュール内での役割

src/backend/executor サブシステムの一部として、fmgr（関数マネージャ）から呼び出される SQL 関数の実行エンジンとして機能します。plancache.c を利用してクエリ計画をキャッシュし、executor の各ノードを再利用することで、SQL 関数のオーバーヘッドを最小化します。

責務

  • **関数実行のライフサイクル管理**: SQLFunctionCache を用いた実行状態の保持と、関数呼び出しごとのコンテキスト管理。
  • **クエリのコンパイルと実行**: SQL 文の解析・書き換え・計画作成、および ExecutorStart/Run/End を用いた実行制御。
  • **結果の収集と変換**: DestReceiver（DR_sqlfunction）を介した結果タプルの収集と、関数定義の戻り値型への適合（JunkFilter による型変換）。
  • **パラメータ解決**: SQL 関数内のパラメータ（$1 等）を、呼び出し時の引数で置換する仕組みの提供。

重要なシンボル

実行状態管理

  • SQLFunctionCache: 関数実行ごとの状態（キャッシュされた計画、パラメータ、タプルストア、メモリコンテキスト）を保持する構造体。
  • execution_state: 関数内の各クエリの実行状態（開始前、実行中、完了）を管理する構造体。
  • ExecStatus: F_EXEC_START, F_EXEC_RUN, F_EXEC_DONE の状態遷移を定義。

キャッシュとコンパイル

  • SQLFunctionHashEntry: 関数定義（pg_proc）から生成される長寿命なキャッシュ構造体。CachedPlanSource を保持。
  • sql_compile_callback: 関数定義が変更された際にキャッシュを再構築するためのコールバック。

実行制御・受信

  • fmgr_sql: SQL 関数のエントリポイント。FmgrInfo からキャッシュを取得し、クエリ実行ループを駆動する。
  • DR_sqlfunction: クエリ結果を収集するための専用 DestReceiver。Tuplestore への格納や JunkFilter によるフィルタリングを行う。

制御フローとデータフロー

1. **初期化**: fmgr_sql が init_sql_fcache を呼び出し、SQLFunctionCache を準備。 2. **クエリ準備**: init_execution_state が CachedPlan を取得し、execution_state リストを構築。 3. **実行ループ**: postquel_start で ExecutorStart を呼び出し、postquel_getnext で ExecutorRun を実行。 4. **結果収集**: DR_sqlfunction が sqlfunction_receive を通じて結果を Tuplestore または JunkFilter のスロットに格納。 5. **終了処理**: postquel_end で ExecutorFinish/End を呼び出し、リソースを解放。

読者向けメモ

  • **メモリ管理**: SQLFunctionCache は FmgrInfo の fn_mcxt に配置され、関数実行の寿命を超えて保持されます。一方、実行中のクエリは subcontext を使用して適宜解放されます。
  • **エラーハンドリング**: error_context_stack を利用して、SQL 関数内のどの文でエラーが発生したかをトレースバックに含める工夫がなされています。
  • **制約**: SQL 関数内ではトランザクション制御文（COMMIT 等）や一部のユーティリティコマンドは禁止されています。
  • **未実装/浅い部分**: SQLFunctionHashEntry のチェーン構造は現在ほとんど使われておらず、将来的なリファクタリングの余地があります。また、lazyEval（遅延評価）は SRF（集合を返す関数）の最適化として重要ですが、複雑な状態遷移を伴うため注意が必要です。
