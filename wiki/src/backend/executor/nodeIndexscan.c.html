src/backend/executor/nodeIndexscan.c

概要

PostgreSQLの実行エンジンにおいて、インデックスを用いたテーブルスキャン（IndexScan）を実行するためのノード実装です。インデックスからタプルを順次取得し、必要に応じてフィルタリングや順序の再評価を行います。

モジュール内での役割

src/backend/executor モジュールの一部として、プランツリーの実行フェーズにおける「インデックススキャン」の責務を担います。nodeIndexonlyscan.c や nodeBitmapIndexscan.c とは、インデックスアクセスという共通の基盤を持ちつつ、ヒープタプルのフェッチと順序保証に特化しています。

責務

  • インデックススキャンの初期化、実行、終了のライフサイクル管理。
  • インデックスの ORDER BY 句に対応するための再ソート（pairingheap を使用）。
  • ランタイムキー（実行時に値が決定する条件）の評価とインデックスへの適用。
  • 並列クエリ実行時のDSM（共有メモリ）管理と統計情報の集約。

重要なシンボル

  • ExecIndexScan: スキャン実行のメインエントリーポイント。
  • IndexNext: 通常のインデックススキャンによるタプル取得。
  • IndexNextWithReorder: ORDER BY 句がある場合の再ソート処理。
  • ExecInitIndexScan: ノードの初期化とスキャンキーの構築。
  • ReorderTuple: 再ソート用キュー（pairingheap）に格納されるタプル構造体。

制御フローとデータフロー

1. **初期化**: ExecInitIndexScan が呼び出され、インデックスの ScanKey を構築します。ランタイムキーがある場合は ExprContext を準備します。 2. **実行**: ExecIndexScan が IndexNext または IndexNextWithReorder を呼び出します。 3. **データ取得**: index_getnext_slot を通じてインデックスからタプルを取得します。 4. **再評価**: インデックスが損失性（lossy）である場合や、ORDER BY の再チェックが必要な場合は、ExecQualAndReset や EvalOrderByExpressions で条件を再検証します。 5. **再ソート**: ORDER BY が必要な場合、タプルを pairingheap にプッシュし、順序が正しいものを順次ポップして返します。

読者向けメモ

  • **再ソートの仕組み**: IndexNextWithReorder は、インデックスが返す順序が不完全な場合や、ORDER BY 句の再評価が必要な場合に、pairingheap を用いて正しい順序を保証します。
  • **ランタイムキー**: ExecIndexEvalRuntimeKeys は、クエリ実行中に値が変化する条件（例: WHERE col = $1）を処理します。ExecReScanIndexScan で再計算されます。
  • **並列処理**: ExecIndexScanEstimate などの関数群は、並列ワーカー間でのスキャン状態共有を管理します。並列実行時は ParallelIndexScanDesc を介して状態が同期されます。
