src/backend/executor/nodeAgg.c

概要

nodeAgg.c は、PostgreSQL の集約（Aggregate）演算を実行するプランノードの実装です。SQL の GROUP BY や集約関数（SUM, COUNT, AVG 等）の計算を担い、ソートベースの集約とハッシュベースの集約の両方をサポートします。また、メモリ制限時のディスク退避（Spilling）や、並列クエリ実行時の統計情報共有といった高度な機能も内包しています。

モジュール内での役割

src/backend/executor サブシステムの一部として、プランナが生成した Agg ノードを実行します。式評価エンジン（execExpr.c）と密接に連携し、集約関数の遷移関数（Transition Function）や最終関数（Final Function）を呼び出すことで、タプルストリームから集約結果を生成します。

責務

  • **集約戦略の実行**:
  • AGG_SORTED: ソート済み入力に対するグループ化。
  • AGG_HASHED: ハッシュテーブルを用いたグループ化。
  • AGG_MIXED: ソートとハッシュを組み合わせた複雑なグループ化。
  • **メモリ管理**: 複数のグループ化セット（Grouping Sets）に対応したメモリコンテキストの切り替えと、ハッシュテーブルのメモリ制限管理。
  • **ディスク退避（Spilling）**: ハッシュテーブルが work_mem を超過した場合の、パーティション分割によるディスクへの退避と再帰的な処理。
  • **集約関数のライフサイクル**: 遷移関数の呼び出し、最終関数の適用、および AggCheckCallContext 等を通じた集約関数への実行コンテキストの提供。

重要なシンボル

状態管理

  • AggState: 集約実行のルート構造体。実行フェーズ、ハッシュテーブル、メモリコンテキストを保持。
  • AggStatePerTrans: 集約の遷移状態（Transition State）情報。遷移関数やシリアライズ関数の情報を保持。
  • AggStatePerGroup: 各グループの現在の遷移値（transValue）を保持する構造体。

ハッシュ集約関連

  • HashAggSpill: ディスク退避時のパーティション情報を管理。
  • HashAggBatch: 再処理が必要なバッチ処理単位。
  • lookup_hash_entries: 入力タプルをハッシュテーブルにマッピングし、必要に応じて退避処理をトリガーする。

集約関数 API

  • AggCheckCallContext: 関数が集約として呼ばれているか確認し、メモリコンテキストを取得。
  • AggGetAggref: 実行中の集約式（Aggref）を取得。
  • AggRegisterCallback: 集約処理終了時のクリーンアップコールバックを登録。

制御フローとデータフロー

1. **初期化**: ExecInitAgg がプランツリーを走査し、集約関数ごとの遷移関数や最終関数の情報を FmgrInfo としてキャッシュする。 2. **実行**: ExecAgg が戦略に応じて agg_retrieve_direct（ソート）または agg_retrieve_hash_table（ハッシュ）を呼び出す。 3. **遷移処理**: advance_aggregates が式評価エンジンを呼び出し、入力タプルを遷移関数に渡す。 4. **ハッシュ退避**: hash_agg_check_limits がメモリ使用量を監視し、超過時に hash_agg_enter_spill_mode を呼び出してディスク退避を開始する。 5. **最終化**: finalize_aggregates が最終関数を呼び出し、結果を射影（Projection）用のスロットに格納する。

読者向けメモ

  • **メモリコンテキスト**: 集約処理は非常にメモリを消費しやすいため、aggcontexts（グループ化セットごと）や hashcontext（ハッシュテーブル用）といった複数のコンテキストを使い分けています。これらは ReScan やグループ境界でリセットされます。
  • **Spilling の複雑性**: ハッシュ集約のディスク退避は再帰的です。hash_choose_num_partitions によるパーティション数の決定ロジックは、メモリ効率と I/O コストのバランスを取るための重要なヒューリスティックです。
  • **省略された詳細**: 本ファイルは非常に巨大であるため、個別の集約関数（SUM, AVG 等）の具体的な実装ロジックは pg_proc に依存しており、ここでは抽象的な呼び出しインターフェースのみを扱っています。また、JIT コンパイルされた式評価の詳細は execExpr.c に委譲されています。
