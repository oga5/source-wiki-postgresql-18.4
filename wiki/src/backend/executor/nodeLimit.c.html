src/backend/executor/nodeLimit.c

概要

SQL の LIMIT および OFFSET 句を処理するための実行ノードです。下位ノード（サブプラン）から返されるタプルのストリームに対し、指定された範囲のタプルのみを抽出・返却します。

モジュール内での役割

src/backend/executor におけるクエリ実行エンジンの一部として、プランツリーの特定のノード（Limit）のライフサイクル管理と実行を担当します。

責務

  • **フィルタリング**: OFFSET 分のタプルをスキップし、LIMIT 分のタプルを通過させる。
  • **状態管理**: LIMIT_INITIAL から LIMIT_INWINDOW などの状態遷移を管理し、順方向および逆方向のスキャンをサポートする。
  • **WITH TIES 対応**: WITH TIES 指定時に、最後のタプルと等価な後続タプルを検出・返却する。
  • **パラメータ評価**: クエリ実行時に動的に決定される LIMIT/OFFSET 値の評価。

重要なシンボル

  • ExecLimit: ノードのメイン実行関数。状態マシンとして動作し、サブプランからタプルを取得する。
  • recompute_limits: LIMIT/OFFSET 式を評価し、内部状態をリセットする。
  • compute_tuples_needed: 下位ノードに伝えるべき必要なタプル数を計算する（最適化用）。
  • ExecInitLimit: ノードの初期化と式評価の準備を行う。

追加の索引済みシンボル

  • ExecEndLimit
  • ExecReScanLimit

制御フローとデータフロー

1. **初期化**: ExecInitLimit が呼ばれ、式（limitOffset, limitCount）のコンパイルとサブプランの初期化が行われる。 2. **実行**: ExecLimit が呼ばれると、現在の lstate に基づいてサブプランから ExecProcNode でタプルを取得する。 3. **フィルタリング**: position カウンタを更新し、offset を超えるまでスキップ、その後 count 分だけタプルを返す。 4. **再スキャン**: ExecReScanLimit が呼ばれると、recompute_limits を通じてパラメータを再評価し、サブプランをリセットする。

直接依存

  • postgres.h: 基本的な PostgreSQL の型とマクロ。
  • executor/executor.h: 実行エンジン共通のインターフェース。
  • executor/nodeLimit.h: LimitState 構造体定義と関数プロトタイプ。
  • miscadmin.h: CHECK_FOR_INTERRUPTS 等の管理機能。

関連コード

  • src/include/nodes/execnodes.h: LimitState 構造体の詳細定義。
  • src/backend/executor/execExpr.c: ExecInitExpr 等の式評価エンジン。
  • src/backend/executor/execProcnode.c: ExecProcNode のディスパッチロジック。

読者向けメモ

  • 本ノードは ExecQual や ExecProject を直接呼び出しませんが、LIMIT/OFFSET 式の評価のために ExprContext を保持しています。
  • WITH TIES が指定されている場合、last_slot を使用して境界値の比較を行うため、メモリ消費や処理フローが通常より複雑になります。
  • ExecSetTupleBound を通じて、下位ノード（特に Sort ノード）に対して必要なタプル数を通知し、実行効率を向上させる仕組みがあります。
