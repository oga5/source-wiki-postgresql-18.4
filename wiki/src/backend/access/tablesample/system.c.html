src/backend/access/tablesample/system.c

概要

system.c は、PostgreSQL の TABLESAMPLE SYSTEM メソッドを実装するモジュールです。このメソッドは、テーブル全体をブロック単位でサンプリングし、選択されたブロック内の全タプルを返します。

モジュール内での役割

tablesample サブシステムの一部として、TsmRoutine インターフェースを実装します。クエリ実行時に SampleScan ノードから呼び出され、どのブロックを読み込むべきかを決定するロジックを提供します。

責務

  • **サンプリングの決定**: ブロック番号とシード値からハッシュ値を計算し、指定された割合に基づいてブロックを選択する。
  • **再現性の保証**: ハッシュ関数を用いることで、物理的なテーブルの変更やスキャン順序に依存しない、一貫したサンプリング結果を提供する。
  • **統計情報の提供**: プランナに対して、サンプリング後の推定ページ数とタプル数を算出する。

重要なシンボル

  • tsm_system_handler: TsmRoutine 構造体を初期化し、メソッドの各関数ポインタを登録するエントリポイント。
  • SystemSamplerData: スキャン中の状態（現在のブロック位置、シード値、カットオフ値）を保持するプライベート構造体。
  • system_nextsampleblock: ハッシュ計算を行い、次に読み込むべきブロック番号を決定する。

追加の索引済みシンボル

制御フローとデータフロー

1. **初期化**: system_beginsamplescan が実行され、サンプリング率からハッシュのカットオフ値を算出する。 2. **ブロック選択**: system_nextsampleblock がループ内でハッシュ値を計算し、hash < cutoff となるブロックが見つかるまで進める。 3. **タプル抽出**: system_nextsampletuple が選択されたブロック内の全タプルを順次返す。ブロック内の全タプルを読み終えると InvalidOffsetNumber を返し、次のブロック選択へ移行させる。

  • optimizer/optimizer.h: プランナによる統計情報推定のためのユーティリティ。

読者向けメモ

  • **ハッシュの性質**: hash_any を使用しているため、プラットフォーム間で一貫した結果が得られます。これは回帰テストにおいて重要です。
  • **パフォーマンス**: system メソッドはブロック単位で読み込むため、bernoulli メソッドよりも I/O 効率が良い傾向がありますが、データの偏りがある場合はサンプリングの代表性が低くなる可能性があります。
  • **状態管理**: SystemSamplerData は palloc0 で確保され、SampleScanState に格納されます。メモリリークを防ぐため、実行終了時のクリーンアップには注意が必要です。
