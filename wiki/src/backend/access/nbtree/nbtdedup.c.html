src/backend/access/nbtree/nbtdedup.c

概要

PostgreSQLのB-treeインデックスにおける重複排除（Deduplication）およびボトムアップ削除（Bottom-up Deletion）を実装するモジュールです。インデックスページの空き容量を確保し、不要なページ分割を抑制することを目的としています。

モジュール内での役割

src/backend/access/nbtree サブシステムの一部として、インデックスの物理的な再構成を担います。nbtinsert.c から呼び出され、ページが満杯になった際の救済措置として機能します。

責務

  • **重複排除**: ページ内の重複するインデックスエントリを「Posting List」形式のタプルに統合し、ページ内の空き容量を増やす。
  • **ボトムアップ削除**: MVCCの更新によって不要になったインデックスエントリを特定し、tableam と連携して物理的に削除する。
  • **Posting Listの管理**: 複数のTIDを保持するタプルの生成、更新、および分割時のTID入れ替え処理。
  • **戦略的制御**: ページ分割を遅延させるための「Single Value」戦略の適用。

重要なシンボル

  • _bt_dedup_pass: 重複排除のメインエントリーポイント。ページをスキャンし、Posting Listを構築してページを再構成する。
  • _bt_bottomupdel_pass: ボトムアップ削除のメインエントリーポイント。削除可能なTIDを特定し、_bt_delitems_delete_check を介して削除を実行する。
  • _bt_form_posting: Posting List形式のタプルを生成する。
  • _bt_update_posting: 既存のPosting Listから特定のTIDを削除して更新する。
  • _bt_swap_posting: Posting Listの分割時に、新しいタプルと既存のTIDを入れ替える。

制御フローとデータフロー

1. **初期化**: BTDedupState を確保し、スキャン対象のページ情報を保持する。 2. **スキャン**: ページ内の各タプルを順次処理し、キーが一致するものを _bt_dedup_save_htid で一時的なPosting Listに蓄積する。 3. **終了処理**:

  • 重複排除の場合: _bt_dedup_finish_pending でPosting Listタプルを生成し、一時ページ（newpage）に書き込む。
  • ボトムアップ削除の場合: _bt_bottomupdel_finish_pending で削除対象のTIDを TM_IndexDeleteOp に登録し、最終的に tableam を呼び出す。

    • 4. **コミット**: 成功した場合、START_CRIT_SECTION を経てページを更新し、WALを記録する。

読者向けメモ

  • **Posting Listの制約**: Posting List内のTIDは常に昇順で保持される必要があります。
  • **アドバイザリな戻り値**: _bt_bottomupdel_pass の戻り値は、呼び出し元に対する「ページ分割を避けるためのヒント」であり、必ずしも削除が成功したことを保証するものではありません。
  • **メモリ管理**: palloc を多用するため、関数終了時の pfree 漏れに注意してください。特にエラー発生時のクリーンアップパスが重要です。
