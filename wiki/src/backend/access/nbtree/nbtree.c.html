src/backend/access/nbtree/nbtree.c

概要

PostgreSQL の B-tree インデックスアクセスメソッドの公開インターフェースを実装するファイルです。Lehman & Yao のアルゴリズムに基づいたインデックスの構築、検索、挿入、およびメンテナンス（VACUUM）の主要なエントリポイントを提供します。

モジュール内での役割

nbtree モジュールの中心的なハブとして機能します。インデックスのライフサイクル管理（スキャン開始から終了まで）や、並列スキャン時の状態管理、VACUUM 処理のオーケストレーションを担います。具体的なページ操作や比較ロジックは他の nbt*.c ファイルに委譲されています。

責務

  • **AM API の提供**: bthandler を通じて、PostgreSQL のインデックスアクセスメソッドとして必要な関数ポインタ群を登録する。
  • **スキャン管理**: btbeginscan から btendscan までのスキャンライフサイクルを管理し、並列スキャン時の同期制御を行う。
  • **VACUUM 処理**: インデックスの物理的なクリーンアップ（btbulkdelete, btvacuumcleanup）を主導し、不要なタプルの削除や空きページの再利用を管理する。
  • **並列処理の同期**: BTParallelScanDescData を用いて、並列クエリ実行時のページスキャン順序と状態を制御する。

重要なシンボル

  • bthandler: インデックスの全機能（挿入、検索、VACUUM 等）を定義する IndexAmRoutine を返す。
  • BTParallelScanDescData: 並列スキャン時の共有状態（現在位置、ロック、条件変数）を保持する構造体。
  • btgettuple / btgetbitmap: スキャン実行時のメインループ。_bt_first や _bt_next を呼び出し、必要に応じて ScalarArrayOp 等のプリミティブスキャンを繰り返す。
  • btvacuumscan: VACUUM の実体。インデックス全体を走査し、不要なタプルの削除とページ削除を行う。

制御フローとデータフロー

1. **スキャン**: btgettuple が呼び出されると、BTScanOpaque を通じて現在のスキャン位置を維持し、ページ境界を跨ぐ際に _bt_next を使用して兄弟ページへ移動します。 2. **並列スキャン**: _bt_parallel_seize でロックを取得し、共有状態を更新してページを確保します。処理完了後は _bt_parallel_release で他ワーカーに制御を戻します。 3. **VACUUM**: btbulkdelete が btvacuumscan を呼び出し、インデックスを物理的に走査します。btvacuumpage が各ページをロックし、不要なタプルを削除（_bt_delitems_vacuum）またはページ自体を削除（_bt_pagedel）します。

読者向けメモ

  • 本ファイルは「公開インターフェース」に特化しています。内部的なアルゴリズムの詳細（特にページ分割や検索の再試行ロジック）を追う場合は、nbtsearch.c や nbtinsert.c を参照してください。
  • 並列スキャン時の LWLock の取り扱いには注意が必要です。_bt_parallel_seize と _bt_parallel_release のペアが正しく呼び出されているかを確認することが、並列実行時のデバッグの鍵となります。
  • VACUUM 処理は非常に複雑です。特に btvacuumpage 内の「バックトラック」処理は、インデックスのページ分割と VACUUM が競合した際の整合性を保つための重要な機構です。
