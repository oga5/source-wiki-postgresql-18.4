src/backend/access/nbtree/nbtxlog.c

概要

PostgreSQL B-tree インデックスの WAL（Write-Ahead Log）リプレイ処理を担うモジュールです。クラッシュリカバリやスタンバイサーバーでのデータ同期時に、WAL レコードを読み取り、インデックスページの状態を正確に再構築します。

モジュール内での役割

nbtree サブシステムの一部として、インデックスの物理的な変更（挿入、分割、削除、メタページ更新など）を WAL から再現する役割を持ちます。インデックスの論理的な操作（検索や挿入）を行う他のファイルとは異なり、このファイルは「物理的なページ状態の復元」に特化しています。

責務

  • WAL レコードの種類に応じたリプレイ関数のディスパッチ。
  • ページ分割（Split）や挿入（Insert）に伴う物理的なページ再構築。
  • メタページの整合性維持と更新。
  • ホットスタンバイ環境におけるリカバリ競合の解決。
  • ページ整合性チェックのためのマスク処理（btree_mask）。

重要なシンボル

  • btree_redo: WAL レコードを解析し、適切な処理関数へ振り分けるメインエントリポイント。
  • _bt_restore_page: ページ内のタプルを順序通りに再配置する汎用ヘルパー。
  • _bt_restore_meta: メタページを WAL レコードの内容で上書き・初期化する関数。
  • btree_mask: リプレイ後のページに対して、WAL に記録されないヒントビットやフラグをマスクし、整合性チェックを可能にする関数。

制御フローとデータフロー

1. **ディスパッチ**: btree_redo が XLogRecGetInfo でレコードタイプを判定。 2. **メモリ管理**: opCtx（メモリコンテキスト）を使用して、リプレイ中に必要な一時的なメモリを確保・解放。 3. **ページ操作**: XLogReadBufferForRedo を使用して対象ページを読み込み、必要に応じて PageAddItem や PageIndexMultiDelete でページ内容を更新。 4. **メタデータ更新**: 必要な場合、_bt_restore_meta を呼び出してルートやレベル情報を更新。 5. **コミット**: PageSetLSN で LSN を更新し、MarkBufferDirty でバッファをダーティとしてマーク。

読者向けメモ

  • **ロックの簡略化**: 通常のインデックス操作では複数のページを同時にロックしますが、リカバリ時は並行更新が発生しないため、ページごとのロックで十分です。このため、コード内ではロックの取得・解放が簡略化されています。
  • **物理的順序の維持**: _bt_restore_page は、ページ内のタプルを元の物理順序で再構築します。これは WAL の一貫性チェックにおいて重要です。
  • **マスク処理**: btree_mask は、リプレイ後のページと元のページを比較する際に、無視すべきビット（ヒントビットやフラグ）をクリアするために使用されます。デバッグやテストでページ内容を検証する際に重要となります。
