src/backend/access/nbtree/nbtutils.c

概要

B-tree インデックスアクセスメソッドの共通ユーティリティ関数群を保持するファイルです。主にスキャンキーの構築、配列型スキャンキー（SAOP: Scalar Array Operators）の高度な処理、およびインデックス構築時のサフィックス切り詰め（Suffix Truncation）ロジックを担います。

モジュール内での役割

nbtree サブシステムにおいて、インデックスの物理的な検索・挿入ロジックと、演算子クラス（OpClass）の比較ロジックを橋渡しする基盤層です。特に、PostgreSQL 12以降で導入された重複排除やスキップスキャン（Skip Scan）に関連する複雑な状態管理を集中させています。

責務

  • **スキャンキー管理**: インデックス検索用の BTScanInsert 構造体の初期化と、スキャン中のキー状態の更新。
  • **配列スキャン最適化**: ANY 演算子等による配列検索において、インデックスの順序性を利用して効率的に検索範囲を絞り込むためのロジック。
  • **サフィックス切り詰め**: インデックスページ分割時に、不要な属性を削除してタプルサイズを圧縮するロジック。
  • **VACUUM 連携**: インデックスの VACUUM サイクル管理と、共有メモリ上の状態保持。

重要なシンボル

スキャンキー・状態管理

  • _bt_mkscankey: インデックス検索用の BTScanInsert を構築。
  • _bt_start_array_keys: スキャン開始時に配列キーの状態を初期化。
  • _bt_advance_array_keys: スキャン中に配列キーを次の要素へ進める（スキップスキャンの核）。

配列検索・比較

  • _bt_binsrch_array_skey: 配列要素に対する二分探索。
  • _bt_compare_array_skey: 配列要素とタプル属性の比較。
  • _bt_binsrch_skiparray_skey: スキップ配列（範囲指定）に対する検索。

インデックスメンテナンス・最適化

  • _bt_truncate: ページ分割時にタプルを切り詰める。
  • _bt_allequalimage: 重複排除が可能か判定する。
  • _bt_vacuum_cycleid / _bt_start_vacuum: VACUUM 実行時のサイクル ID 管理。

制御フローとデータフロー

1. **スキャンキー構築**: _bt_mkscankey がインデックスのメタデータに基づき ScanKey を初期化し、検索の準備を整えます。 2. **配列キーの進行**: _bt_checkkeys が各タプルを検証する際、必要に応じて _bt_advance_array_keys を呼び出します。これが _bt_array_increment / _bt_array_decrement を経由して、配列の現在位置を更新します。 3. **スキップスキャン**: _bt_binsrch_skiparray_skey は、インデックスの順序性を利用して、次の有効なキー範囲を「スキップ」して特定します。 4. **切り詰め**: _bt_truncate は _bt_keep_natts を呼び出し、lastleft と firstright のタプルを比較して、保持すべき最小限の属性数を決定します。

関連コード

  • nbtsearch.c: 本ファイルのユーティリティを利用してツリーを降下する。
  • nbtinsert.c: 挿入時に _bt_truncate を利用してページサイズを最適化する。
  • nbtsplitloc.c: 分割位置決定時に _bt_keep_natts_fast を利用する。

読者向けメモ

  • **メモリ所有権**: _bt_mkscankey で作成される構造体は palloc されており、呼び出し側が適切に解放する必要があります。
  • **スキップスキャンの複雑性**: 本ファイル内の配列キー関連ロジックは、インデックスの順序性を前提とした最適化が多用されています。_bt_advance_array_keys は非常に複雑なステートマシンとなっており、デバッグ時は so->needPrimScan フラグの状態遷移に注意してください。
  • **切り詰め**: _bt_truncate は、インデックスの 1/3 ページ制限を守るための重要なロジックです。DEBUG_NO_TRUNCATE マクロを定義することで、切り詰めを無効化してテスト可能です。
