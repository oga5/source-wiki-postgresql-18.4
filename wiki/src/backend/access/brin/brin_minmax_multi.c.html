src/backend/access/brin/brin_minmax_multi.c

概要

本ファイルは、BRIN インデックスにおける「Multi Min/Max」戦略の実装です。従来の単一範囲（Min/Max）では、外れ値の混入により範囲が過度に拡大し、インデックスの選択性が低下する問題がありました。本モジュールは、複数の小さな間隔（Interval）を保持することで、外れ値の影響を局所化し、スキャン効率を維持します。

モジュール内での役割

BRIN アクセスメソッドの演算戦略の一つとして機能します。brin.c 等のコアモジュールから呼び出される演算関数（consistent, add_value, union 等）を提供し、インデックスの要約データ（Ranges）の構築、シリアライズ、およびマージを担います。

責務

  • **要約の管理**: ページ範囲内の値を複数の間隔または単一の値として保持し、values_per_range オプションに基づき管理する。
  • **動的圧縮**: 保持する値が上限を超えた場合、距離関数（distance）を用いて最も近い間隔同士を貪欲にマージし、サイズを縮小する。
  • **シリアライズ**: インメモリの Ranges 構造体を、ディスク保存用の SerializedRanges（bytea 形式）へ変換する。
  • **演算戦略の提供**: 距離計算（distance）や比較演算を各データ型向けに提供する。

重要なシンボル

データ構造

  • Ranges: インメモリでの要約表現。固定長ヘッダと可変長の values 配列を持つ。
  • SerializedRanges: ディスク上の永続化形式。bytea として格納される。
  • ExpandedRange: マージ処理時に使用する、min/max を明示的に持つ中間表現。
  • MinmaxMultiOpaque: インデックスのメタデータキャッシュ。

主要関数

  • brin_minmax_multi_add_value: 新しい値を要約に追加し、必要に応じて圧縮を行う。
  • brin_minmax_multi_consistent: スキャンキーと要約を比較し、ページ範囲のスキップ可否を判定する。
  • brin_minmax_multi_union: 2つの要約をマージする。
  • reduce_expanded_ranges: 距離関数に基づき、要約のサイズを目標値まで縮小する。

制御フローとデータフロー

1. **追加**: add_value が呼ばれると、まず ensure_free_space_in_buffer でバッファの空きを確認する。 2. **圧縮**: バッファが溢れる場合、build_expanded_ranges で全範囲を中間形式に展開し、build_distances で間隔間の距離を計算。reduce_expanded_ranges で距離が近い順にマージする。 3. **シリアライズ**: brin_range_serialize により、Ranges を SerializedRanges へ変換し、bytea として格納する。 4. **クエリ**: consistent 関数が SerializedRanges をデシリアライズし、各間隔に対してスキャンキーの包含関係をチェックする。

直接依存

  • access/brin.h, access/brin_internal.h: BRIN コア定義。
  • utils/datum.h, utils/fmgr.h: PostgreSQL のデータ型操作および関数呼び出し基盤。
  • 各種 utils/ ヘッダ: uuid, inet, timestamp 等の型固有の距離計算用。

関連コード

  • brin_minmax.c: 単一範囲の Min/Max 戦略。本モジュールの基礎となる。
  • brin_tuple.c: BRIN インデックスのタプル操作。
  • brin_revmap.c: インデックスページとヒープページの対応管理。

読者向けメモ

  • **メモリ管理**: 圧縮処理（reduce_expanded_ranges 等）はメモリ消費が激しいため、一時的な MemoryContext を作成してリークを防いでいます。
  • **圧縮の性質**: 圧縮は貪欲法（Greedy）で行われます。距離関数は近似値でも許容されますが、正確であるほどインデックスの品質が向上します。
  • **制約**: values_per_range は最大 256 まで設定可能です。デフォルトは 32 です。
  • **未実装/浅い部分**: 距離計算のロジックは型ごとに手動で実装されており、一部（UUID や Inet 等）は近似的な計算を行っています。これは完全な精度よりも計算コストを優先しているためです。
