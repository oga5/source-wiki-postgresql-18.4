src/backend/access/transam/xlogreader.c

概要

WAL（Write-Ahead Log）レコードを読み込み、解析（デコード）するための汎用的な基盤機能を提供します。本ファイルはフロントエンド（pg_waldump等）とバックエンドの両方でコンパイルされるため、ereport等のサーバー固有機能に依存せず、コールバックベースの設計を採用しています。

モジュール内での役割

src/backend/access/transam モジュールにおいて、WALの物理的な読み込みと論理的なレコード構造への変換を抽象化するレイヤーです。リカバリ処理（xlogrecovery.c）やレプリケーション、WAL解析ツールがWALの内容を解釈するための唯一のインターフェースとなります。

責務

  • **WALストリームの抽象化**: ページ単位の読み込みをコールバック（page_read）経由で抽象化し、セグメント境界やページ境界を跨ぐレコードの再構築を管理。
  • **レコードのデコード**: 物理的なWALバイト列を、ブロック参照やメインデータを含む論理的な DecodedXLogRecord 構造体に変換。
  • **検証**: ページヘッダーの整合性チェック、レコードヘッダーの検証、およびCRCチェックによるデータの正当性保証。
  • **メモリ管理**: デコード用バッファの循環管理および、巨大レコードに対する動的なメモリ割り当て。

重要なシンボル

状態管理

  • XLogReaderState: リーダーの現在の読み込み位置、バッファ、コールバック関数、エラー状態を保持するメイン構造体。
  • XLogReaderAllocate / XLogReaderFree: リーダーのライフサイクル管理。

読み込み・解析

  • XLogReadRecord: 次の有効なWALレコードを読み込み、デコードして返す。
  • XLogReadAhead: 非同期的な先読みを行い、デコード済みレコードをキューに蓄積する。
  • XLogFindNextRecord: 指定したLSN以降の最初の有効なレコードを探す。

デコード・検証

  • DecodeXLogRecord: 物理的なレコードを論理構造へ変換するコア関数。
  • ValidXLogRecord: CRCチェックを行い、レコードの破損を検出する。
  • XLogReaderValidatePageHeader: WALページヘッダーのマジックナンバーやシステムIDを検証する。

制御フローとデータフロー

1. **初期化**: XLogReaderAllocate でバッファとコールバックをセットアップ。 2. **ページ読み込み**: ReadPageInternal が page_read コールバックを呼び出し、必要なWALページを readBuf にロード。 3. **レコード再構築**: レコードがページを跨ぐ場合、readRecordBuf を使用して断片を連結。 4. **デコード**: DecodeXLogRecord がヘッダーを解析し、ブロック参照やメインデータを DecodedXLogRecord 構造体に展開。 5. **キューイング**: デコード済みレコードは decode_queue にリンクされ、XLogNextRecord を通じて消費される。

直接依存

  • access/xlogreader.h: インターフェース定義。
  • access/xlogrecord.h: WALレコードの物理フォーマット定義。
  • storage/bufmgr.h / common/logging.h: 環境に応じたI/Oおよびログ出力。
  • lz4.h / zstd.h / pg_lzcompress.h: ページイメージの圧縮・解凍用。

関連コード

  • xlog.c: WALの書き込みと管理の司令塔。
  • xlogrecovery.c: 本リーダーを使用してWALをリプレイするリカバリ処理。
  • xloginsert.c: WALレコードを生成する側。

読者向けメモ

  • **メモリ所有権**: XLogReaderState は readBuf や decode_buffer を所有します。XLogReaderFree を呼び出すまでメモリは解放されません。
  • **エラーハンドリング**: report_invalid_record はエラーメッセージをバッファに書き込み、errormsg_deferred フラグを立てます。呼び出し側はこれをチェックする必要があります。
  • **制約**: ValidXLogRecordHeader はレコードのヘッダーが完全に読み込まれていることを前提とします。ページ境界を跨ぐ場合は、再構築後に検証が行われます。
  • **省略された詳細**: 本ファイルは汎用性を重視しているため、特定のWALリソースマネージャ（RMGR）の具体的なリプレイロジックは含まず、あくまで「レコードの抽出」に特化しています。
