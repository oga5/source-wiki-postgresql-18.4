src/backend/access/heap/heapam_handler.c

概要

本ファイルは、PostgreSQLのテーブルアクセスメソッド（AM）インターフェースである TableAmRoutine を、ヒープストレージの実装（heapam.c 等）に接続するためのブリッジ層です。実行エンジンからの抽象的なテーブル操作要求を、ヒープ特有の物理操作へと変換する役割を担います。

モジュール内での役割

src/backend/access/heap モジュールにおける「インターフェース層」です。上位の実行エンジンやコマンド処理系は、このファイルで定義された heapam_methods 構造体を通じてヒープテーブルを操作します。これにより、テーブルの物理格納形式がヒープであることを意識せずに、統一的なAPIでデータアクセスが可能になります。

責務

  • **AMインターフェースの提供**: TableAmRoutine 構造体への関数ポインタの割り当て。
  • **インデックススキャン連携**: インデックスから取得したTIDに基づくヒープタプルのフェッチとHOTチェーンの追跡。
  • **DML操作の仲介**: 挿入、更新、削除、ロック操作を heapam.c の低レベル関数へ委譲。
  • **メンテナンス・DDL処理**: VACUUM、CLUSTER、テーブル切り詰め、物理コピー等の管理操作の実行。
  • **統計情報とプランナ支援**: テーブルサイズ見積もりやTOAST管理の判断。

重要なシンボル

インターフェース定義

  • heapam_methods: TableAmRoutine 型の静的構造体。ヒープAMの全機能がここに集約されます。
  • GetHeapamTableAmRoutine() / heap_tableam_handler(): AMハンドラを取得するためのエントリポイント。

インデックススキャン関連

  • heapam_index_fetch_tuple: インデックススキャン中にヒープタプルをフェッチし、HOTチェーンを辿る。
  • heapam_index_build_range_scan: インデックス構築時のヒープ走査。

DML操作

  • heapam_tuple_insert / heapam_tuple_update / heapam_tuple_delete: 実行エンジンからの要求を heap_insert 等に変換。
  • heapam_tuple_lock: 行ロックの取得と、更新チェーンの追跡（TM_Result を返す）。

メンテナンス・DDL

  • heapam_relation_copy_for_cluster: CLUSTER コマンド実行時の物理再構築。
  • heapam_relation_copy_data: テーブルの物理ファイルを直接コピーする低レベル操作。

制御フローとデータフロー

1. **スキャンフロー**: heapam_scan_analyze_next_block がページを読み込み、heapam_scan_analyze_next_tuple がページ内の各タプルを走査して可視性を判定します。 2. **インデックスフェッチ**: heapam_index_fetch_tuple は、xs_cbuf（現在のバッファ）を維持しつつ、heap_hot_search_buffer を呼び出してHOTチェーンを辿り、可視なタプルを TupleTableSlot に格納します。 3. **更新フロー**: heapam_tuple_update は、古いタプルを削除し新しいタプルを挿入する一連の処理を heap_update に委譲し、結果のTIDをスロットに反映します。 4. **CLUSTERフロー**: heapam_relation_copy_for_cluster は、begin_heap_rewrite で書き換えコンテキストを初期化し、走査したタプルを reform_and_rewrite_tuple で再構築して新しいリレーションへ書き込みます。

読者向けメモ

  • **メモリ管理**: TupleTableSlot を介したタプルの受け渡しが基本です。ExecFetchSlotHeapTuple 等でスロットからタプルを取り出す際、shouldFree フラグによるメモリ解放の責任分担に注意してください。
  • **ロックの粒度**: LockBuffer を用いたバッファレベルの共有ロック（BUFFER_LOCK_SHARE）が頻繁に使用されます。デッドロックを避けるため、ロック取得順序と解放タイミングは heapam.c の規約に従う必要があります。
  • **HOTチェーン**: インデックススキャン時、HOTチェーンの追跡は heap_hot_search_buffer が行います。本ファイルは、チェーンの途中でバッファを切り替える必要があるかどうかの制御（call_again）を管理します。
  • **省略された詳細**: 複雑なWAL生成ロジックや、ページ内の具体的なビットマップ操作は heapam.c や hio.c に委譲されており、本ファイルには含まれていません。
