src/backend/access/hash/hashpage.c

概要

PostgreSQL ハッシュインデックスの物理ページ管理およびバケット構造の操作を担う中心的なファイルです。メタページ、バケットページ、およびオーバーフローページの初期化、読み込み、ロック管理、およびインデックスの拡張（バケット分割）ロジックを実装しています。

モジュール内での役割

src/backend/access/hash サブシステムにおいて、物理ストレージ（smgr）と論理的なハッシュ構造の橋渡しを行います。バッファマネージャを介したページアクセスを抽象化し、ハッシュインデックス特有のページレイアウト（HashPageOpaque）の整合性を保証します。

責務

  • **バッファ管理**: ページ読み込み、初期化、ロック取得（_hash_getbuf 等）、および解放の標準化。
  • **メタページ管理**: インデックスのメタデータ（バケット数、マスク、スペアページ情報）の初期化とキャッシュ。
  • **バケット分割（Split）**: インデックス拡張時のバケット分割処理、および中断された分割の再開（_hash_expandtable）。
  • **物理構造の初期化**: インデックス作成時のメタページ、バケット、ビットマップページの初期配置。

重要なシンボル

バッファアクセス・管理

  • _hash_getbuf: 指定ブロックの読み込みとロック取得。
  • _hash_getinitbuf: 新規ページをゼロ初期化して取得。
  • _hash_getnewbuf: インデックス末尾を拡張して新規ページを取得。
  • _hash_relbuf / _hash_dropbuf: ロックおよびピンの解放。

インデックス構造操作

  • _hash_init: インデックス全体の初期化（メタページ、初期バケット、ビットマップ）。
  • _hash_expandtable: バケット分割のトリガーと実行。
  • _hash_splitbucket: バケット内のタプルを新旧バケットへ再分配。
  • _hash_finish_split: 中断された分割処理の再開。

キャッシュ・ユーティリティ

  • _hash_getcachedmetap: リレーションキャッシュ（rd_amcache）を利用したメタページデータの取得。
  • _hash_getbucketbuf_from_hashkey: ハッシュキーから対象バケットのバッファを特定。

追加の索引済みシンボル

  • log_split_page

制御フローとデータフロー

1. **バケット特定**: _hash_getbucketbuf_from_hashkey がキャッシュされたメタデータを用いて対象バケットを特定します。 2. **分割実行**: _hash_expandtable が排他ロックを取得し、必要に応じて _hash_alloc_buckets で物理領域を確保した後、_hash_splitbucket を呼び出します。 3. **タプル再分配**: _hash_splitbucket は旧バケットの全ページをスキャンし、ハッシュキーに基づいてタプルを新バケットへ移動させます。移動中は INDEX_MOVED_BY_SPLIT_MASK を付与し、分割完了後にフラグをクリアします。 4. **WAL 記録**: ページ変更は XLogRegisterBuffer を介して WAL に記録され、クラッシュリカバリ時の整合性を確保します。

読者向けメモ

  • **メモリ所有権**: _hash_getcachedmetap が返す HashMetaPage はリレーションキャッシュ（rd_indexcxt）に所有されており、呼び出し側は解放してはいけません。
  • **ロック順序**: デッドロックを避けるため、バケットのロックは常にバケット番号の昇順で行う必要があります。
  • **分割の安全性**: 分割処理は「クリティカルセクション」内で実行され、失敗時はエラーとして処理されます。中断された分割は、次回アクセス時に _hash_finish_split によって自動的に再開されます。
  • **省略された詳細**: 本ファイルはページレベルの操作に特化しており、タプルの挿入ロジック（_hash_pgaddmultitup 等）や、ビットマップの具体的なビット操作ロジックは他ファイル（hashutil.c や hashovfl.c）に委譲されています。
