src/backend/access/hash/hash.c

概要

PostgreSQL のハッシュインデックスアクセスメソッド（AM）の公開インターフェースを定義するファイルです。インデックスの作成、挿入、スキャン、およびメンテナンス（VACUUM）といった主要な操作の入り口を提供します。

モジュール内での役割

src/backend/access/hash モジュールにおける「フロントエンド」として機能します。具体的なハッシュ計算やページ操作などの低レイヤー処理は他のファイル（hashfunc.c, hashpage.c 等）に委譲し、本ファイルは PostgreSQL の IndexAmRoutine 構造体を通じてシステムとハッシュインデックスを接続します。

責務

  • **AM インターフェースの提供**: hashhandler 関数により、PostgreSQL のインデックス AM として必要なコールバック関数を登録する。
  • **インデックス構築**: hashbuild を通じた初期構築と、必要に応じたソート処理（_h_spool）の制御。
  • **スキャン管理**: スキャンの開始、再開、終了、およびタプルの取得（hashgettuple）の管理。
  • **メンテナンス**: hashbulkdelete を通じた不要タプルの削除と、VACUUM 後の統計情報更新。

重要なシンボル

  • hashhandler: インデックス AM の定義を返すエントリポイント。
  • HashBuildState: インデックス構築時の状態（スプール状況やタプル数）を保持する構造体。
  • hashbuild: インデックス作成のメインルーチン。
  • hashbulkdelete: VACUUM 処理のメインルーチン。
  • hashbucketcleanup: 特定のバケット内の不要タプルを削除するヘルパー関数。

制御フローとデータフロー

1. **構築**: hashbuild がテーブルをスキャンし、hashbuildCallback を経由してインデックスを構築します。メモリ不足時は _h_spool でソートしてから挿入します。 2. **挿入**: hashinsert がキーをハッシュ値に変換し、_hash_doinsert を呼び出して適切なバケットへ配置します。 3. **スキャン**: hashbeginscan で状態を初期化し、hashgettuple が _hash_first / _hash_next を呼び出してバケットチェーンを辿ります。 4. **削除**: hashbulkdelete が全バケットを走査し、hashbucketcleanup が各ページ内の不要タプルを削除します。この際、スキャンとの競合を防ぐためにロックの連鎖（Lock Chaining）が行われます。

読者向けメモ

  • 本ファイルはインターフェース層であり、具体的なアルゴリズムの詳細は _hash_ で始まる内部関数に隠蔽されています。
  • ハッシュインデックスは「損失あり（Lossy）」のインデックスです。hashgettuple 内で scan->xs_recheck = true が設定されている点に注意してください。
  • hashbucketcleanup でのロック処理は、並行するスキャンとの整合性を保つために非常に複雑なロック連鎖を行っています。デッドロックを避けるための設計意図を理解するには、この関数のコメントが重要です。
