src/backend/access/hash/hashvalidate.c

概要

PostgreSQL のハッシュインデックス（Hash AM）における演算子クラス（opclass）および演算子ファミリー（opfamily）の整合性を検証するためのモジュールです。

モジュール内での役割

src/backend/access/hash サブシステムの一部として、ユーザーが定義した演算子クラスがハッシュインデックスの要件を満たしているかを検証し、カタログ登録時の不整合を未然に防ぐ役割を担います。

責務

  • **整合性検証**: 演算子ファミリー内のサポート関数（ハッシュ関数）と演算子のシグネチャ、および戦略番号がハッシュインデックスの仕様に適合しているかを確認します。
  • **依存関係の調整**: 演算子や関数を演算子ファミリーに追加する際、それらが演算子クラスに強く依存すべきか、ファミリーに緩く依存すべきかを決定します。

重要なシンボル

  • hashvalidate(Oid opclassoid): 演算子クラスの定義が正しいかを検証するメイン関数。amvalidate API を実装しています。
  • hashadjustmembers(...): 演算子や関数をカタログに追加する際、依存関係（hard/soft）を適切に設定するための補助関数。

追加の索引済みシンボル

  • hashadjustmembers
  • hashvalidate

制御フローとデータフロー

1. **検証 (hashvalidate)**:

  • システムカタログ（pg_amproc, pg_amop）からファミリー内の全メンバーを取得。
  • サポート関数のシグネチャ（HASHSTANDARD_PROC 等）を検証し、ハッシュ可能な型リストを作成。
  • 演算子の戦略番号とシグネチャを検証し、ハッシュ関数との整合性を確認。
  • identify_opfamily_groups を使用して、演算子と関数のセットが不足していないかをチェック。

    • 2. **調整 (hashadjustmembers)**:

  • 対象がクロスタイプ（左辺と右辺の型が異なる）か、あるいはオプションのサポート関数であるかを判定。
  • 型が一致する場合は演算子クラスへの「ハード依存」を、それ以外は演算子ファミリーへの「ソフト依存」を割り当てます。

読者向けメモ

  • このファイルは CREATE OPERATOR CLASS 実行時に呼び出されます。
  • hashvalidate 内のチェックは冗長に見える部分がありますが、これは amvalidate API の設計上、ファミリー全体を検証するコストが許容されているためです。
  • クロスタイプ演算子の扱いは複雑であり、hashadjustmembers での依存関係の決定ロジックは、ダンプ/リストア時の整合性に直結するため注意が必要です。
