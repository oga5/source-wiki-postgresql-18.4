src/backend/access/gin/ginlogic.c

概要

GIN（Generalized Inverted Index）インデックスにおける、検索条件の整合性チェック（Consistent Check）を抽象化するモジュールです。演算子クラス（opclass）が提供する論理関数（Boolean/Ternary）の差異を吸収し、スキャン実行エンジンに対して統一的なインターフェースを提供します。

モジュール内での役割

GINスキャン実行時の「インデックスエントリが検索条件に合致するか」を判定するレイヤーです。opclassがどちらか一方の論理関数しか実装していない場合でも、もう一方の関数をシミュレートして提供する「アダプター」の役割を果たします。

責務

  • **インターフェースの統一**: bool 型の整合性関数と GinTernaryValue 型の三値論理関数の相互変換。
  • **シム（Shim）の実装**:
  • 三値論理から二値論理への変換（GIN_MAYBE を recheck フラグ付きの TRUE に変換）。
  • 二値論理から三値論理への変換（MAYBE 入力を全組み合わせで試行し、結果が収束するか判定）。
  • **初期化**: スキャン開始時に、利用可能な関数を判定し GinScanKey に関数ポインタを割り当てる。

重要なシンボル

  • ginInitConsistentFunction: スキャンキーに対して適切な整合性チェック関数をセットアップするエントリポイント。
  • shimTriConsistentFn: 二値論理関数から三値論理を導出する関数。MAX_MAYBE_ENTRIES（4）までの MAYBE 入力を許容します。
  • GinScanKey: 整合性チェックに必要な状態（エントリの合致状況、演算子情報など）を保持する構造体。

制御フローとデータフロー

1. **初期化**: ginInitConsistentFunction が GinState を参照し、opclassが提供する関数OIDを確認します。 2. **関数割り当て**:

  • 実装が存在すれば direct... 系関数を、なければ shim... 系関数を GinScanKey に登録します。

    • 3. **実行**: スキャン中、エンジンは key->boolConsistentFn または key->triConsistentFn を呼び出します。 4. **シムの動作**: shimTriConsistentFn は key->entryRes を一時的に書き換え、全組み合わせを試行した後に元の状態へ復元します。

直接依存

  • postgres.h: 基本的な型定義とマクロ。
  • access/gin_private.h: GinScanKey, GinState, GinTernaryValue 等のGIN内部データ構造。

関連コード

  • src/backend/access/gin/ginget.c: 本モジュールで定義された関数を呼び出し、インデックススキャンを実行するメインロジック。
  • src/backend/access/gin/ginvalidate.c: opclassが整合性関数を正しく実装しているかを検証するコード。

読者向けメモ

  • **パフォーマンス**: shimTriConsistentFn は MAYBE 入力に対して指数関数的な計算量（最大 $2^4$ 回の関数呼び出し）を要します。複雑なクエリで MAYBE が多発する場合、opclass側で三値論理関数を実装することが推奨されます。
  • **スレッド安全性**: shimTriConsistentFn は key->entryRes を一時的に変更します。現状はシングルスレッド前提ですが、将来的な並列スキャン実装時には注意が必要です。
  • **EVERYTHINGモード**: GIN_SEARCH_MODE_EVERYTHING の場合は、常に合致とみなすダミー関数が割り当てられます。
