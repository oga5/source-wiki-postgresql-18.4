src/backend/access/gin/ginget.c

概要

GIN（Generalized Inverted Index）索引の検索処理において、インデックスからタプル（TID）を取得するためのコアロジックを実装するファイルです。主に gingetbitmap 関数を通じて、検索条件に合致するヒープタプルの TID をビットマップ形式で収集します。

モジュール内での役割

src/backend/access/gin サブシステムにおける「検索エンジン」の役割を担います。ginscan.c で初期化されたスキャン状態に基づき、B-tree 構造のインデックスページおよび未処理の挿入データ（Pending List）を走査します。

責務

  • **Pending List の走査**: fastupdate 機能によりメインインデックスに未反映のデータを検索し、結果をビットマップに統合する。
  • **メインインデックスの走査**: B-tree 構造を辿り、各キーの出現位置（Posting Tree または Posting List）から TID を抽出する。
  • **検索キーの最適化**: 頻度に基づいて検索キーを「必須（Required）」と「追加（Additional）」に分類し、効率的な走査を実現する。
  • **一致判定**: ユーザー定義の consistent 関数を呼び出し、TID が検索条件を満たすかを判定する。

重要なシンボル

  • gingetbitmap: GIN 検索のメインエントリーポイント。
  • scanPendingInsert: Pending List を走査し、ビットマップに結果を追加する。
  • startScan: 各検索キーの走査を開始し、頻度に基づいた最適化を行う。
  • keyGetItem: 特定の検索キーに対して、次に合致する TID を取得する。
  • scanGetItem: 全ての検索キーを統合し、最終的な合致 TID を特定する。

制御フローとデータフロー

1. **初期化**: gingetbitmap が ginNewScanKey を呼び出し、スキャンキーを構築。 2. **Pending List 処理**: scanPendingInsert がメタページから Pending List を辿り、合致する TID をビットマップに追加。 3. **メイン走査**: startScan で各キーの出現頻度を推定し、scanGetItem をループ実行。 4. **キー統合**: scanGetItem は各キーの keyGetItem を呼び出し、TID が全ての条件を満たすか（AND 条件）を判定。 5. **結果収集**: 最終的に合致した TID またはページ単位のビットマップを tbm_add_tuples / tbm_add_page で出力。

読者向けメモ

  • **デッドロック回避**: Posting Tree を走査する際、インデックスページをアンロックしてデッドロックを防ぐ実装が随所にあります。再走査が必要なケース（collectMatchBitmap 等）があるため、複雑な状態遷移を伴います。
  • **Lossy Page**: GIN はページ単位の「Lossy」な結果を返すことがあります。これはビットマップ上でページ全体を走査対象とすることを意味し、recheck フラグと併せて理解する必要があります。
  • **最適化**: startScanKey で行われる頻度ベースのキー分類は、クエリ性能を左右する重要な最適化です。頻度の低いキーを先に処理することで、不要な走査をスキップしています。
