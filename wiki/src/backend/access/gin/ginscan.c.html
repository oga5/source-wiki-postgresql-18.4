src/backend/access/gin/ginscan.c

概要

GIN（Generalized Inverted Index）インデックススキャンにおけるスキャン状態の管理と、クエリ条件の解析・初期化を担うモジュールです。インデックススキャン開始時のメモリ確保から、クエリ条件の分解、スキャン終了時のリソース解放までを統括します。

モジュール内での役割

src/backend/access/gin サブシステムにおいて、IndexScanDesc の opaque 領域を管理するインターフェースを提供します。ginget.c がインデックスを走査する際に参照する GinScanKey 構造体を構築し、検索条件を GIN が処理可能な形式に変換する役割を持ちます。

責務

  • **スキャン状態のライフサイクル管理**: ginbeginscan でのメモリコンテキスト作成、ginendscan でのクリーンアップ。
  • **クエリの正規化**: extractQueryFn を呼び出し、ユーザーのクエリをインデックス検索可能なキー（GinScanEntry）に変換。
  • **検索モードの最適化**: GIN_SEARCH_MODE_ALL や EVERYTHING などの特殊な検索モードを適切に配置し、インデックス全体のスキャンや NULL 検索をサポート。
  • **重複排除**: 同一の検索キーに対する GinScanEntry の重複作成を抑制し、メモリ効率を向上。

重要なシンボル

  • ginbeginscan: スキャン状態（GinScanOpaque）を初期化し、メモリコンテキストを準備する。
  • ginNewScanKey: ユーザーの ScanKey を解析し、GIN 内部の GinScanKey 構造体へ変換する主要関数。
  • ginFillScanKey: 個別のインデックス属性に対する検索キーを構築し、GinScanEntry を生成する。
  • GinScanOpaqueData: GIN スキャン中の状態を保持する構造体。キー配列やメモリコンテキストを格納。

制御フローとデータフロー

1. **初期化**: ginbeginscan が呼ばれ、GinScanOpaque が生成される。 2. **解析**: ginNewScanKey が extractQueryFn を実行。得られた Datum を ginFillScanKey に渡し、GinScanEntry のリストを作成。 3. **最適化**: GIN_SEARCH_MODE_ALL 等の特殊モードに対し、必要に応じて「隠しエントリ（Hidden Entry）」を追加し、スキャン順序を調整。 4. **実行**: ginget.c がこの構造体を参照してインデックスを走査。 5. **終了**: ginendscan により、確保されたメモリコンテキストとバッファが解放される。

直接依存

  • access/gin_private.h: GIN 内部構造体（GinScanOpaque, GinScanKey 等）の定義。
  • access/relscan.h: IndexScanDesc 等の標準的なインデックススキャン定義。
  • utils/memutils.h: メモリコンテキスト管理。
  • pgstat.h: インデックススキャン統計情報の更新。
  • postgres.h
  • utils/rel.h

関連コード

  • src/backend/access/gin/ginget.c: 本ファイルで構築された GinScanKey を実際に使用してインデックスを走査する。
  • src/backend/access/gin/ginutil.c: initGinState や ginCompareEntries など、スキャン準備に必要なユーティリティ関数群。

読者向けメモ

  • **メモリ管理**: GinScanOpaque 内には tempCtx と keyCtx の2つのメモリコンテキストが存在します。keyCtx はスキャンキーの生存期間を管理し、rescan 時にリセットされます。
  • **重複排除の制限**: ginFillScanEntry にはパフォーマンス上の理由から、100個以上のエントリがある場合に重複排除を停止するロジックが含まれています。
  • **古いインデックス**: GIN インデックスのバージョンが古い場合、ginNewScanKey 内で全インデックススキャンや NULL 検索が拒否される仕様になっています。REINDEX が推奨されるケースです。
