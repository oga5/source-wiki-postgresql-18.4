src/backend/access/index/indexam.c

概要

indexam.c は、PostgreSQL のインデックスアクセス手法（AM）に対する汎用的なインターフェース層です。特定のインデックス型（B-tree, GIN, GiST 等）の実装詳細を隠蔽し、実行エンジンがインデックスを操作するための統一された API を提供します。

モジュール内での役割

src/backend/access/index モジュールの中核であり、上位層（エグゼキュータ等）と下位層（各インデックス AM の実装）を仲介します。genam.c がインデックススキャンの初期化や構築といった補助的な処理を担うのに対し、本ファイルはスキャンの実行、挿入、保守といったライフサイクル全般を管理します。

責務

  • **インデックス操作の抽象化**: aminsert, amgettuple, ambulkdelete 等の AM API を呼び出すためのラッパーを提供。
  • **整合性チェック**: RELATION_CHECKS や SCAN_CHECKS マクロを用い、不正な状態でのアクセスを防止。
  • **リソース管理**: インデックスのオープン/クローズ、スキャン中の参照カウント管理、スキャン終了時のメモリ解放。
  • **サポート関数管理**: pg_amproc に登録されたサポート関数の検索とキャッシュ（index_getprocinfo）。

重要なシンボル

  • index_open / index_close: インデックスリレーションのライフサイクル管理。
  • index_insert: インデックスへのタプル挿入。
  • index_beginscan / index_endscan: スキャンの開始と終了。
  • index_getnext_tid: スキャンにおける次のタプル識別子（TID）の取得。
  • index_fetch_heap: インデックスが指すヒープタプルの取得。
  • index_getprocinfo: インデックスの演算子クラスに関連付けられたサポート関数の取得。

制御フローとデータフロー

1. **スキャン開始**: index_beginscan が呼び出されると、index_beginscan_internal を経由して対象 AM の ambeginscan が実行され、IndexScanDesc が生成されます。 2. **タプル取得**: index_getnext_slot がループ内で index_getnext_tid（AM 固有の検索）と index_fetch_heap（ヒープからの実データ取得）を交互に呼び出し、結果をスロットに格納します。 3. **保守処理**: index_bulk_delete や index_vacuum_cleanup は、VACUUM 処理中に AM 固有のクリーンアップロジックを起動します。

読者向けメモ

  • **マクロの活用**: CHECK_REL_PROCEDURE や CHECK_SCAN_PROCEDURE は、特定の AM が関数を実装していない場合にエラーを投げる重要なガードです。
  • **HOT チェーン**: index_fetch_heap は HOT（Heap Only Tuple）チェーンを考慮しており、MVCC スナップショット下での整合性を維持するよう設計されています。
  • **ロックの注意点**: index_close はロックを解放しますが、トランザクション終了までロックを保持し続ける設計が一般的であるため、呼び出しタイミングには注意が必要です。
