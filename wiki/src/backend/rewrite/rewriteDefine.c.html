src/backend/rewrite/rewriteDefine.c

概要

本ファイルは、PostgreSQLにおける「書き換えルール（Rewrite Rule）」の定義、更新、および管理を行うためのバックエンドルーチンを提供します。ユーザーが発行する CREATE RULE や ALTER RULE コマンドの実行を担い、システムカタログ pg_rewrite への永続化と整合性チェックを統括します。

モジュール内での役割

src/backend/rewrite モジュールにおいて、ルール定義のライフサイクル管理を担当します。ルールが適用される対象リレーション（テーブルやビュー）の整合性を保ちつつ、カタログ操作とリレーションキャッシュ（relcache）の無効化を調整する役割を担います。

責務

  • **ルール定義の永続化**: CREATE RULE の解析結果を pg_rewrite に挿入または更新する。
  • **整合性検証**: ルール定義時に、対象リレーションの型（ビュー、テーブル等）や、ルールアクション（SELECT, INSERT 等）がルールとして妥当かを検証する。
  • **依存関係の管理**: ルールと対象リレーション、およびルール内で参照されるオブジェクト間の依存関係を pg_depend に記録する。
  • **実行状態の制御**: ルールの有効/無効化（EnableDisableRule）や名前変更（RenameRewriteRule）を処理する。

重要なシンボル

  • DefineQueryRewrite: ルール作成のメインエントリポイント。解析済みのクエリを受け取り、カタログ登録と整合性チェックを行う。
  • InsertRule: pg_rewrite への実際の挿入・更新を行う低レベル関数。
  • checkRuleResultList: ビューの SELECT ルールや RETURNING 句が対象リレーションのスキーマと一致するかを検証する。
  • setRuleCheckAsUser: ルール内のクエリツリーを走査し、権限チェック用のユーザーIDを伝播させる。

制御フローとデータフロー

1. **解析と検証**: DefineRule がパーサーを呼び出し、DefineQueryRewrite へ処理を委譲します。 2. **ロックと権限確認**: 対象リレーションに AccessExclusiveLock を取得し、所有者権限やシステムカタログへの変更可否をチェックします。 3. **ルール妥当性チェック**: checkRuleResultList を使用して、ターゲットリストの型や列名がリレーションと適合するかを確認します。 4. **カタログ更新**: InsertRule が pg_rewrite を更新し、recordDependencyOn で依存関係を構築します。 5. **キャッシュ無効化**: SetRelationRuleStatus や CacheInvalidateRelcache を呼び出し、全バックエンドに対してルール変更を通知します。

読者向けメモ

  • **ロックの重要性**: ルール定義はカタログを書き換えるため、対象リレーションに対して厳格な AccessExclusiveLock を必要とします。これは同時実行中のクエリとの競合を防ぐためです。
  • **ON SELECT ルールの特殊性**: ON SELECT ルールはビューの定義そのものであり、他のルールとは異なる厳格な制約（名前が _RETURN であること、単一の SELECT アクションであること等）が課されます。
  • **キャッシュの無効化**: ルールを変更した後は、必ず CacheInvalidateRelcache を呼び出す必要があります。これを怠ると、他のバックエンドが古いルール定義を参照し続け、システムが不安定になります。
