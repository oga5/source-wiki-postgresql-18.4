src/backend/port/sysv_shmem.c

概要

System V 共有メモリ機能を利用して、PostgreSQL の共有メモリセグメントを管理する実装です。現代の PostgreSQL では、主要な共有メモリ領域は mmap() で確保し、System V 共有メモリはデータディレクトリのロックおよびプロセス間インターロック（生存確認）のための「シム（薄い層）」として利用するのが標準的です。

モジュール内での役割

src/backend/port モジュールの一部として、OS 固有の共有メモリ API を抽象化します。PostgreSQL の共有メモリサブシステム（storage/shmem.h）が要求するインターフェースを、System V 準拠のシステム上で実現する役割を担います。

責務

  • **共有メモリの作成とアタッチ**: shmget および shmat を用いたセグメントの確保。
  • **インターロックの提供**: 既存の共有メモリセグメントが、現在のデータディレクトリに関連するものか、また他プロセスがアタッチ中かを判定。
  • **ライフサイクル管理**: on_shmem_exit を用いた、プロセス終了時のセグメントデタッチおよび削除処理。
  • **Huge Pages のサポート**: mmap を使用する場合の Huge Page 設定とフラグ管理。

重要なシンボル

  • PGSharedMemoryCreate: 共有メモリの初期化と作成を行うメインエントリポイント。
  • PGSharedMemoryIsInUse: 既存のセグメントが他プロセスによって使用中かを確認する。
  • InternalIpcMemoryCreate: shmget を呼び出し、セグメントを生成・アタッチする内部関数。
  • IpcMemoryState: セグメントの状態（アタッチ済み、外部のもの、未使用など）を分類する列挙型。
  • UsedShmemSegAddr / UsedShmemSegID: 現在のプロセスが使用しているセグメントの識別子とアドレス。

制御フローとデータフロー

1. **初期化**: PGSharedMemoryCreate が呼ばれると、データディレクトリの inode/device 番号をキーとしてセグメントを検索します。 2. **作成**: InternalIpcMemoryCreate が shmget を試行します。成功すれば on_shmem_exit にデタッチ/削除用のコールバックを登録します。 3. **インターロック**: PGSharedMemoryAttach が shmctl(IPC_STAT) を呼び出し、セグメントのメタデータ（shm_nattch）を確認して、他プロセスがアタッチしているかを判定します。 4. **終了**: プロセス終了時、登録されたコールバックが shmdt や shmctl(IPC_RMID) を実行し、リソースを解放します。

読者向けメモ

  • **EXEC_BACKEND**: Windows や一部の環境で有効なこのフラグが立っている場合、fork() によるメモリ継承ができないため、PGSharedMemoryReAttach を用いた再アタッチ処理が必要になります。
  • **匿名共有メモリ**: 現在の PostgreSQL は、System V 共有メモリを「ロック用」として使い、実際のデータ領域は mmap で確保するハイブリッド構成が一般的です。このため、AnonymousShmem 関連のコードが随所に存在します。
  • **エラー処理**: shmget 失敗時の EINVAL や EEXIST は、単なるエラーではなく「既存セグメントの再利用」を判断するための重要なシグナルとして扱われています。
