src/backend/port/win32/signal.c

概要

Windows環境におけるPOSIXシグナル機能のエミュレーション層です。Windowsにはネイティブなシグナル機構が存在しないため、名前付きパイプとイベントオブジェクトを用いて、Unixライクなシグナル処理（kill、sigaction、sigprocmask等）を実現します。

モジュール内での役割

src/backend/port/win32 モジュールの一部として、PostgreSQLバックエンドがWindows上でシグナルを適切に受信・処理するための基盤を提供します。他のプラットフォーム依存コードと連携し、プロセス間通信を介したシグナル伝達を抽象化します。

責務

  • **シグナルリスナーの運用**: 名前付きパイプを監視し、外部からのシグナル要求（kill相当）を待機する専用スレッドの管理。
  • **シグナルキューイング**: 受信したシグナルをスレッドセーフにキューへ格納し、メインスレッドに通知する。
  • **シグナルディスパッチ**: メインスレッドにおいて、キューイングされたシグナルを適切なハンドラ関数へ振り分ける。
  • **POSIX互換APIの提供**: pqsigaction や pqsigprocmask を通じて、Unix環境と同一のシグナル制御インターフェースを公開する。

重要なシンボル

  • pg_signal_queue: 受信済みシグナルのビットマスク。メインスレッドとシグナルスレッド間で共有される。
  • pgwin32_signal_event: シグナルの到着をメインスレッドに通知するためのWindowsイベントオブジェクト。
  • pgwin32_signal_initialize(): シグナル処理スレッドの起動と初期化を行う。
  • pgwin32_dispatch_queued_signals(): キュー内のシグナルを処理するメインループ。
  • pg_signal_thread(): 名前付きパイプを介してシグナル要求を待ち受けるバックグラウンドスレッド。

制御フローとデータフロー

1. **受信**: 外部プロセスが名前付きパイプにシグナル番号を書き込むと、pg_signal_thread がそれを検知する。 2. **キューイング**: pg_queue_signal が pg_signal_queue を更新し、pgwin32_signal_event をセットする。 3. **通知**: メインスレッドは pg_usleep や CHECK_FOR_INTERRUPTS 等の待機ポイントでイベントを検知する。 4. **実行**: pgwin32_dispatch_queued_signals が呼び出され、マスクされていないシグナルハンドラが実行される。

直接依存

  • postgres.h: 基本的な型定義とエラー報告機能。
  • libpq/pqsignal.h: シグナル関連の定数および関数宣言。
  • Windows API (windows.h 相当): CreateNamedPipe, CreateEvent, CreateThread 等。

関連コード

  • src/include/libpq/pqsignal.h: シグナルハンドラの型定義や定数。
  • src/port/kill.c (外部): Windows上で kill() を呼び出した際に、このモジュールのパイプへ書き込む実装が含まれる。

読者向けメモ

  • **スレッド安全性**: pg_signal_queue はシグナルスレッドとメインスレッドの両方からアクセスされるため、pg_signal_crit_sec による排他制御が必須です。
  • **デッドロック注意**: シグナルハンドラ内でさらにシグナルをブロックする処理が含まれるため、ハンドラ実行中のロック解除と再取得のフローを追う際は注意してください。
  • **パイプの役割**: 名前付きパイプは「シグナル送信者」と「受信者」の同期を保証するために使用されており、単なる通知以上の役割（送信側への完了通知）を担っています。
