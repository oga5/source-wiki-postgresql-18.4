src/backend/port/sysv_sema.c

概要

System V セマフォ API を使用して、PostgreSQL のセマフォ抽象化（PGSemaphore）を実装するソースファイルです。主に Unix 系システムにおいて、プロセス間同期のための低レベルなセマフォ管理を提供します。

モジュール内での役割

src/backend/port ディレクトリ内で、OS 固有のセマフォ実装の一つとして機能します。PostgreSQL はビルド時に適切なセマフォ実装（SysV, POSIX, Win32）を選択しますが、本ファイルはそのうちの「SysV」版を担当します。

責務

  • **セマフォのライフサイクル管理**: カーネルリソースとしてのセマフォセットの作成、初期化、および終了時の破棄。
  • **リソースの再利用**: クラッシュしたプロセスが残したセマフォセットを検出し、安全に再利用またはクリーンアップする。
  • **抽象化レイヤーの提供**: PGSemaphoreLock や PGSemaphoreUnlock などの共通インターフェースを、semop システムコールを用いて実装する。

重要なシンボル

  • PGSemaphoreData: セマフォセット ID とセット内のインデックスを保持する構造体。
  • SEMAS_PER_SET: 1 つのセマフォセット内に確保するセマフォの数（識別用 1 つを含む）。
  • PGReserveSemaphores: セマフォサブシステムを初期化し、共有メモリ領域を確保する関数。
  • PGSemaphoreCreate: 新しいセマフォを割り当て、必要に応じて新しいセマフォセットを作成する関数。

制御フローとデータフロー

1. **初期化**: PGReserveSemaphores がデータディレクトリの inode をキーのシードとして使用し、共有メモリを確保します。 2. **セマフォ作成**: PGSemaphoreCreate が呼ばれると、現在のセットが満杯か確認し、必要であれば InternalIpcSemaphoreCreate で新しい SysV セマフォセットをカーネルに要求します。 3. **再利用**: IpcSemaphoreCreate は、既存のキーでセマフォセットを作成しようとし、衝突した場合は semctl で所有者 PID を確認し、プロセスが生存していなければクリーンアップを試みます。 4. **操作**: PGSemaphoreLock/Unlock は semop を呼び出し、シグナル割り込み（EINTR）を考慮したループ処理を行います。

読者向けメモ

  • **カーネルパラメータ**: 本実装は SEMMSL（セットあたりのセマフォ数）や SEMMNI（システム全体のセマフォセット数）といったカーネル制限に強く依存します。エラー発生時は ereport のヒントを確認してください。
  • **シグナル処理**: semop はシグナルによって中断される可能性があるため、do-while ループで EINTR を明示的にハンドリングしています。
  • **識別用セマフォ**: 各セマフォセットの末尾に「識別用」のセマフォを確保し、PGSemaMagic 値を書き込むことで、PostgreSQL が作成したセマフォセットを他アプリケーションと区別しています。
