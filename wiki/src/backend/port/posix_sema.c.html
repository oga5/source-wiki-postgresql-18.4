src/backend/port/posix_sema.c

概要

POSIX セマフォ（sem_t）を利用して、PostgreSQL のプロセス間同期用セマフォ（PGSemaphore）を実装するファイルです。名前付きセマフォ（sem_open）と名前なしセマフォ（sem_init）の両方をサポートします。

モジュール内での役割

src/backend/port 内のセマフォ実装の一つです。プラットフォームが POSIX セマフォをサポートしている場合に選択され、PostgreSQL の共有メモリ管理およびプロセス間通信（IPC）の基盤を提供します。

責務

  • **セマフォのライフサイクル管理**: PGSemaphoreCreate による初期化と、on_shmem_exit を利用した終了時のクリーンアップ。
  • **抽象化レイヤーの提供**: PGSemaphoreLock, PGSemaphoreUnlock, PGSemaphoreTryLock などの共通インターフェースを POSIX API にマッピング。
  • **リソースの保護**: 名前付きセマフォ使用時の名前衝突回避や、名前なしセマフォ使用時の共有メモリ配置の管理。

重要なシンボル

  • PGSemaphore: sem_t * へのポインタとして定義される抽象型。
  • PG_SEM_REF(x): PGSemaphoreData 構造体から内部の sem_t を取り出すマクロ。
  • PGReserveSemaphores: セマフォサブシステムを初期化し、必要なリソースを確保する関数。
  • USE_NAMED_POSIX_SEMAPHORES: 名前付きセマフォを使用するかどうかを決定するコンパイル時フラグ。

制御フローとデータフロー

1. **初期化**: PGReserveSemaphores が呼ばれ、データディレクトリの inode をシードとしてセマフォキーを生成します。 2. **確保**: PGSemaphoreCreate が呼ばれると、名前なしの場合は共有メモリ上の配列から、名前付きの場合は sem_open でセマフォを確保します。 3. **操作**: PGSemaphoreLock 等の関数が sem_wait や sem_post を呼び出します。この際、EINTR（シグナル割り込み）が発生した場合はループで再試行する堅牢な設計になっています。 4. **解放**: ReleaseSemaphores がプロセス終了時に呼び出され、sem_close または sem_destroy でリソースを解放します。

読者向けメモ

  • **EXEC_BACKEND の制約**: 名前付きセマフォを使用する場合、EXEC_BACKEND（Windows や一部の Unix 環境）とは共存できません。これは sem_open が共有メモリ外に割り当てられるためです。
  • **リセットの挙動**: PGSemaphoreReset は POSIX 標準に直接的なリセット関数がないため、sem_trywait をループさせてカウントを 0 にする実装になっています。
  • **デバッグ**: sem_unlink は作成直後に実行され、クラッシュ時にセマフォがシステムに残らないよう工夫されています。
