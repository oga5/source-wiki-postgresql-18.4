src/backend/regex/regerror.c

概要

正規表現エンジンで発生したエラーコードを、人間が読み取り可能なメッセージやシンボル名に変換するためのユーティリティファイルです。

モジュール内での役割

src/backend/regex サブシステムにおいて、エラーハンドリングの抽象化層を担います。他のコンポーネントが内部的な数値エラーコードを外部（ユーザーやログ）へ提示する際の標準的なインターフェースを提供します。

責務

  • エラーコードとメッセージの対応付け（マッピング）の管理。
  • エラーコードから名前（REG_...）への変換。
  • 名前からエラーコードへの逆変換。
  • 指定されたバッファサイズに応じた安全な文字列コピーと切り詰め処理。

重要なシンボル

  • pg_regerror: 本ファイルのメイン関数。エラーコードの変換とメッセージ取得を行う。
  • rerrs: エラーコード、名前、説明文を保持する静的な構造体配列。regerrs.h をインクルードして定義される。
  • REG_ATOI / REG_ITOA: pg_regerror の特殊な動作モードを指定する定数（名前と数値の相互変換用）。

追加の索引済みシンボル

  • rerr

制御フローとデータフロー

1. **入力判定**: errcode に基づき、通常のメッセージ取得か、REG_ATOI/REG_ITOA による変換モードかを switch 文で分岐します。 2. **検索**: rerrs 配列を線形探索し、該当するコードまたは名前を探します。 3. **メッセージ生成**: 該当が見つからない場合は、unk テンプレートを使用して未知のエラーであることを示す文字列を生成します。 4. **バッファ出力**: errbuf_size を確認し、バッファが十分であればコピー、不足していれば切り詰めて書き込みます。 5. **戻り値**: 必要なバッファサイズ（終端文字を含む）を返します。

直接依存

  • regex/regguts.h: 正規表現エンジンの内部定義と共通ヘッダー。
  • regex/regerrs.h: エラーコードの定義テーブル。

関連コード

  • regex.h (システム標準): REG_ATOI などの定数が定義されている場所。
  • regcomp.c / regexec.c: エラーが発生する主要なコンポーネント。これらが pg_regerror を呼び出すことでエラーを可視化します。

読者向けメモ

  • 本ファイルは POSIX の regerror に準拠した実装ですが、PostgreSQL 固有の拡張や調整が含まれています。
  • rerrs 配列の末尾には番兵（{-1, "", "oops"}）が配置されており、探索ループの終了条件として利用されています。
  • errbuf_size が 0 の場合でも、必要なバッファサイズを計算して返す設計になっているため、必要なメモリ量を事前に知るために利用可能です。
