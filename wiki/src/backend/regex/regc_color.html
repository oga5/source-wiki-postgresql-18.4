src/backend/regex/regc_color.c

概要

正規表現エンジンにおける「文字の彩色（Coloring）」を管理するモジュールです。このファイルは regcomp.c に #include される形で利用されます。文字や文字クラスを抽象的な「色（color）」としてマッピングし、NFA（非決定性有限オートマトン）の遷移を効率化します。

モジュール内での役割

src/backend/regex サブシステムにおいて、文字コードの集合を NFA のエッジとして扱うための抽象化レイヤーです。個々の文字を直接扱うのではなく、同じ振る舞いをする文字群を同一の「色」に割り当てることで、NFA の状態数とエッジ数を削減します。

責務

  • **カラーマップの管理**: 文字コードから色へのマッピング（locolormap および hicolormap）の生成と保持。
  • **動的な色割り当て**: 新しい文字や範囲が追加された際の色の動的生成とリソース管理。
  • **NFA への統合**: 文字クラス（cvec）や範囲指定を NFA のエッジとして変換・登録する。
  • **最適化**: 冗長なエッジの排除や、サブカラー（subcolor）の統合による NFA の簡素化。

重要なシンボル

  • colormap: 文字と色の対応を保持する主要な構造体。
  • initcm / freecm: カラーマップの初期化とメモリ解放。
  • newcolor: 新しい色を割り当てる。必要に応じてカラーマップの領域を拡張する。
  • subcolor / subcolorhi: 特定の文字や範囲に対して、既存の色を分割して新しいサブカラーを割り当てる。
  • subcolorcvec: cvec（文字ベクトル）を走査し、対応する NFA エッジを生成する。

制御フローとデータフロー

1. **初期化**: initcm で WHITE（デフォルト色）を基点としたマップを作成。 2. **彩色**: 正規表現のコンパイル中、文字や範囲が指定されるたびに subcolor や subcolorhi が呼ばれ、必要に応じて色が分割・再割り当てされる。 3. **NFA 構築**: subcolorcvec が文字クラスを走査し、newarc を通じて NFA にエッジを追加する。 4. **最適化**: okcolors により、不要になったサブカラーを親カラーに統合し、NFA のエッジを整理する。

直接依存

  • regguts.h: 内部構造体（colormap, colordesc, vars 等）の定義。
  • regc_nfa.c: newarc や findarc などの NFA 操作関数。
  • regc_lex.c: 字句解析に関連する補助機能。

関連コード

  • regcomp.c: 本ファイルをインクルードし、コンパイルのメインフローを制御する。
  • regc_cvec.c: 文字集合（cvec）の操作ロジック。
  • rege_dfa.c: 彩色された色を元に DFA 実行時の遷移を行う。

読者向けメモ

  • **「色」の概念**: ここでの「色」は、文字コードの等価類を指します。同じ色を持つ文字は、正規表現のどの位置でも同じ遷移先を持つことが保証されます。
  • **メモリ管理**: newcolor や newhicolorrow は、必要に応じて REALLOC を行いカラーマップを拡張します。この際、ポインタが移動する可能性があるため注意が必要です。
  • **複雑性**: subcoloronerange や subcoloronechr は、既存の範囲を分割して新しい色を割り当てるため、カラーマップの範囲リスト（cmranges）の操作が複雑です。デバッグ時は dumpcolors 関数が非常に有用です。
