src/backend/regex/regc_lex.c

概要

正規表現コンパイラの字句解析フェーズを担当するソースファイルです。regcomp.c に #include される形で利用され、入力文字列をトークン単位に分解して構文解析器へ供給します。

モジュール内での役割

src/backend/regex サブシステムにおいて、正規表現の構文解析（regcomp.c）と、入力文字列の物理的な読み取りの橋渡しを行います。正規表現のフレーバー（ERE, BRE, ARE）に応じたコンテキスト管理と、エスケープシーケンスの解釈を専門とします。

責務

  • **字句解析の実行**: 入力文字列を走査し、正規表現のトークン（PLAIN, BACKREF, CCLASSS など）を生成する。
  • **コンテキスト管理**: 現在の解析状態（L_ERE, L_BRE, L_BRACK 等）を保持し、文脈に応じたトークン解釈を行う。
  • **エスケープ処理**: \ で始まる特殊文字や、ARE（Advanced Regular Expressions）特有の拡張構文のパース。
  • **オプションの初期化**: prefixes 関数による、正規表現の先頭に付与されるフラグ（*** や (?...)）の解析とフラグ設定。

重要なシンボル

  • next(struct vars *v): メインの字句解析関数。次のトークンを特定し、v->nexttype にセットする。
  • lexescape(struct vars *v): \ に続くエスケープシーケンスを解析する。
  • lexcon: 現在の解析コンテキストを保持する変数。
  • SET(c) / RET(c): トークンタイプをセットし、解析を継続するためのマクロ群。
  • L_ERE / L_BRE / L_Q: 解析コンテキストを定義する定数。

制御フローとデータフロー

1. lexstart() が呼び出され、prefixes() で正規表現の先頭オプションを処理し、初期コンテキストを決定する。 2. next() がループ内で呼び出され、現在の lexcon に基づいて入力文字列を消費する。 3. brenext()（BRE用）や lexescape()（エスケープ用）へ処理を委譲し、トークンを特定する。 4. 特定されたトークンは v->nexttype および v->nextvalue に格納され、呼び出し元（regcomp.c）がこれを参照して構文木を構築する。

直接依存

  • regcomp.c: 本ファイルが #include される親ファイル。
  • regex/regguts.h: vars 構造体や正規表現関連の定数定義。

関連コード

  • regcomp.c: 字句解析器が生成したトークンを消費し、NFA（非決定性有限オートマトン）を構築する。
  • regc_nfa.c: 構築されたトークンからNFAの状態遷移を生成する際に利用される。

読者向けメモ

  • 本ファイルは独立したソースファイルとしてコンパイルされるのではなく、regcomp.c の一部として展開されるため、v（struct vars *）というポインタが暗黙的に共有されている点に注意してください。
  • lexcon の状態遷移は非常に重要です。特に L_BRACK（ブラケット式内）や L_EBND（範囲指定内）では、通常の正規表現とは異なる文字の扱いになるため、この状態管理がバグの温床になりやすい箇所です。
  • ARE（Advanced Regular Expressions）の拡張機能（(?...) など）を追う場合は、next() 内の CHR('(') のケースから読み始めると理解が早いです。
