src/backend/catalog/pg_enum.c

概要

pg_enum.c は、PostgreSQL の列挙型（ENUM）の定義を保持するシステムカタログ pg_enum を操作するためのルーチン群を提供します。列挙型の作成、ラベルの追加、名前変更、およびトランザクション内での整合性維持を担います。

モジュール内での役割

src/backend/catalog モジュールの一部として、データ定義言語（DDL）の実行時にシステムカタログを更新する役割を持ちます。特に pg_type と密接に関連し、列挙型の内部表現を管理します。

責務

  • **列挙型定義の管理**: CREATE TYPE AS ENUM による初期値の挿入。
  • **ラベル操作**: ALTER TYPE ... ADD VALUE によるラベルの追加（順序指定含む）および名前変更。
  • **トランザクション整合性**: 未コミットの列挙型・ラベルを追跡し、インデックス破損を防ぐための安全策を提供。
  • **順序の再計算**: ラベル間の隙間がなくなった際の enumsortorder の再割り当て。

重要なシンボル

  • EnumValuesCreate: 列挙型の初期値を pg_enum に挿入する。
  • AddEnumLabel: 新しいラベルを既存の列挙型に追加する。
  • RenumberEnumType: 順序値の精度不足時に既存のラベルを再採番する。
  • uncommitted_enum_types / values: トランザクション内で作成された未コミットの列挙型・値を追跡するハッシュテーブル。

制御フローとデータフロー

1. **追加処理**: AddEnumLabel が呼ばれると、まず LockDatabaseObject で対象型を排他ロックし、競合を防ぎます。 2. **順序決定**: 既存のラベルを SearchSysCacheList1 で取得し、qsort でソート。新しいラベルの挿入位置に基づき enumsortorder を計算します。 3. **再採番**: float4 の精度限界で挿入できない場合、RenumberEnumType を呼び出し、全ラベルの順序を 1 から順に再割り当てして再試行します。 4. **永続化**: CatalogTupleInsert を使用してシステムカタログを更新し、必要に応じて CommandCounterIncrement で可視性を制御します。

読者向けメモ

  • **順序の仕組み**: enumsortorder は float4 型です。これはラベルの挿入を容易にするための設計ですが、精度限界に達すると RenumberEnumType が走るため、カタログ更新のコストが発生します。
  • **バイナリアップグレード**: pg_upgrade 実行時は IsBinaryUpgrade フラグにより、OID の割り当てが制御されます。このモードでは BEFORE/AFTER を指定したラベル追加は禁止されています。
  • **未コミットの扱い**: トランザクションがロールバックされた際にインデックスが破損しないよう、未コミットの列挙型値はハッシュテーブルで追跡され、安全性が確認されるまで使用が制限されます。
