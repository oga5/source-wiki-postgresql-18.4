src/backend/catalog/pg_operator.c

概要

pg_operator.c は、PostgreSQL のシステムカタログ pg_operator に対する操作を管理するモジュールです。演算子の定義、検索、および「シェル（未定義の演算子プレースホルダー）」の作成と解決を担います。

モジュール内での役割

src/backend/catalog ディレクトリにおいて、演算子というデータベースオブジェクトのライフサイクル（作成・更新・依存関係管理）を定義する中心的な役割を果たします。pg_operator.h で定義されたカタログ構造を実体として操作します。

責務

  • **演算子の作成と更新**: CREATE OPERATOR 文のバックエンド処理を実装し、カタログへの挿入や更新を行います。
  • **シェル演算子の管理**: 相互参照（交換子や否定子）が必要な場合、未定義の演算子を「シェル」として先行作成し、後に実体化する仕組みを提供します。
  • **妥当性検証**: 演算子名が構文的に正しいか、また指定された属性（Commutator, Negator 等）が型定義と矛盾しないかを検証します。
  • **依存関係の構築**: 演算子が依存する型、名前空間、関数、所有者との関係を pg_depend に記録します。

重要なシンボル

  • OperatorCreate: 演算子作成のメインエントリポイント。
  • OperatorShellMake: 演算子のシェルを作成する内部関数。
  • OperatorLookup: 既存の演算子を名前と型から検索する。
  • OperatorUpd: 演算子の交換子（Commutator）や否定子（Negator）のリンクを更新する。
  • makeOperatorDependencies: 演算子に関連するオブジェクト間の依存関係を登録する。

制御フローとデータフロー

1. **作成要求**: OperatorCreate が呼び出されると、まず validOperatorName で名前を検証し、OperatorValidateParams で属性の整合性を確認します。 2. **シェル解決**: 関連する交換子や否定子が未定義の場合、get_other_operator を経由して OperatorShellMake を呼び出し、仮の OID を確保します。 3. **カタログ更新**: table_open で pg_operator を開き、heap_form_tuple または heap_modify_tuple を使用してカタログを更新します。 4. **依存関係登録**: makeOperatorDependencies を呼び出し、型や関数への依存関係を pg_depend に書き込みます。 5. **リンク更新**: OperatorUpd を呼び出し、相互参照する演算子側のフィールドを更新して整合性を保ちます。

読者向けメモ

  • **シェル演算子の概念**: PostgreSQL では、演算子 A が演算子 B を参照し、B が A を参照する場合、どちらかを先に作成する必要があります。この際、中身が空の「シェル」を作成して OID を確定させる手法が取られています。
  • **整合性チェック**: OperatorUpd は、既に別の演算子とリンクされているフィールドを上書きしようとするとエラーを発生させます。これはカタログの破損を防ぐための重要なガードレールです。
  • **依存関係の特殊性**: 演算子は oprcom や oprnegate に依存しているように見えますが、これらは pg_depend には記録されません。これらは RemoveOperatorById 等で手動でリンクを解除する設計になっています。
