src/backend/catalog/namespace.c

概要

namespace.c は、PostgreSQL におけるオブジェクトの名前解決と可視性（Visibility）を制御する中核モジュールです。search_path に基づく名前空間の探索順序の管理、一時テーブル（TEMP）用名前空間のライフサイクル管理、および各オブジェクト型（テーブル、型、関数、演算子等）が現在の検索パスで可視であるかの判定機能を提供します。

モジュール内での役割

src/backend/catalog サブシステムの一部として、pg_namespace カタログを直接操作する pg_namespace.c とは異なり、本ファイルは「カタログ定義をどのように探索するか」という論理的な検索パスの抽象化を担当します。

責務

  • **検索パス管理**: GUC search_path の解析、キャッシュ、およびセッションごとの有効な検索パスの再計算。
  • **名前解決**: 非修飾名（Unqualified Name）からオブジェクト OID への変換。
  • **可視性判定**: 指定された OID が現在の検索パスにおいて「最初に見つかるもの」であるかの判定。
  • **一時名前空間管理**: セッションごとの pg_temp_N 名前空間の初期化、クリーンアップ、およびライフサイクル管理。

重要なシンボル

検索パス状態管理

  • activeSearchPath: 現在有効な名前空間 OID のリスト。
  • activeCreationNamespace: オブジェクト作成時のデフォルトターゲット名前空間。
  • activePathGeneration: 検索パスの変更を追跡するためのカウンタ。
  • baseSearchPath: search_path GUC から導出されたベースリスト。

一時名前空間

  • myTempNamespace: 現在のセッションの一時テーブル用名前空間 OID。
  • myTempToastNamespace: 一時テーブルの TOAST 用名前空間 OID。
  • AccessTempTableNamespace(): 一時名前空間の遅延初期化を行う。

名前解決・可視性 API

  • RangeVarGetRelidExtended(): RangeVar からリレーション OID を解決するメイン関数。
  • RelnameGetRelid(): 非修飾リレーション名の解決。
  • *IsVisible(): 各オブジェクト型（RelationIsVisible, TypeIsVisible 等）の可視性判定関数。
  • FuncnameGetCandidates(): 関数呼び出し時の候補リスト取得（オーバーロード解決用）。

制御フローとデータフロー

1. **パス再計算**: recomputeNamespacePath() が baseSearchPathValid をチェックし、無効であれば cachedNamespacePath() を呼び出す。 2. **キャッシュ**: SearchPathCache（simplehash）を用いて、search_path 文字列と roleid の組み合わせから計算済みパスを高速に取得する。 3. **名前解決**: RelnameGetRelid 等は activeSearchPath をイテレートし、各名前空間で get_relname_relid 等のカタログ検索を実行する。 4. **無効化**: InvalidationCallback が SysCache の変更を検知し、baseSearchPathValid を false に設定することで、次回の検索時にパスの再計算を強制する。

読者向けメモ

  • **メモリ管理**: 検索パス関連のリストは TopMemoryContext に保持されます。
  • **一時テーブルの特殊性**: pg_temp は検索パスの先頭に暗黙的に挿入されますが、これはセッションごとに動的に生成されるため、AccessTempTableNamespace による遅延初期化が重要です。
  • **並列処理**: 並列ワーカーはリーダーの myTempNamespace を引き継ぎます（SetTempNamespaceState）。
  • **パフォーマンス**: recomputeNamespacePath は頻繁に呼ばれるため、activePathGeneration による変更検知と SearchPathCache による計算結果の再利用が最適化の鍵となっています。
  • **省略された詳細**: 本ファイルは非常に多くのオブジェクト型（TSParser, TSDictionary 等）の可視性判定を含みますが、これらはすべて *IsVisibleExt パターンに従う定型的な実装であるため、個別のロジックは省略しています。
