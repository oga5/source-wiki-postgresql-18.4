src/backend/catalog/pg_constraint.c

概要

pg_constraint.c は、PostgreSQL のシステムカタログ pg_constraint に対する操作を抽象化・提供するモジュールです。制約（CHECK, FOREIGN KEY, UNIQUE, PRIMARY KEY, EXCLUSION, NOT NULL）の作成、検索、更新、削除、および名前解決を担います。

モジュール内での役割

catalog/ サブシステムの一部として、DDL（ALTER TABLE, CREATE TABLE 等）実行時に制約のメタデータを永続化し、依存関係を管理します。pg_constraint.h で定義されたスキーマに基づき、カタログタプルの物理的な読み書きと、システムキャッシュ（SysCache）を介した高速な検索機能を提供します。

責務

  • **制約のライフサイクル管理**: 制約の新規作成（CreateConstraintEntry）、削除（RemoveConstraintById）、名前変更（RenameConstraintById）。
  • **依存関係の構築**: 制約と対象リレーション、ドメイン、インデックス、演算子間の依存関係（DEPENDENCY_AUTO, DEPENDENCY_NORMAL）の記録。
  • **名前解決と検証**: 制約名の重複チェック（ConstraintNameIsUsed）および自動生成（ChooseConstraintName）。
  • **NOT NULL 制約の特殊処理**: NOT NULL 制約の継承調整（AdjustNotNullInheritance）や、列単位での検索・抽出。
  • **外部キーのメタデータ抽出**: DeconstructFkConstraintRow による外部キー定義の分解と、演算子情報の取得。

重要なシンボル

制約の作成・操作

  • CreateConstraintEntry: 制約のカタログエントリを作成し、依存関係を記録するメイン関数。
  • RemoveConstraintById: 制約を削除し、関連する pg_class の relchecks カウントを更新する。
  • RenameConstraintById: 制約名を変更し、Post-Alter フックを呼び出す。

検索・名前解決

  • ConstraintNameIsUsed / ConstraintNameExists: 指定された名前が既存の制約と衝突するかを判定。
  • ChooseConstraintName: 制約名が衝突しないよう、サフィックスを付与して一意な名前を生成。
  • get_relation_constraint_oid / get_domain_constraint_oid: 名前から制約の OID を取得。

NOT NULL 制約関連

  • findNotNullConstraintAttnum: 指定列の NOT NULL 制約タプルを検索。
  • extractNotNullColumn: NOT NULL 制約の conkey から列番号を抽出。
  • AdjustNotNullInheritance: 継承関係にある NOT NULL 制約の整合性チェックと更新。

外部キー・機能的依存性

  • DeconstructFkConstraintRow: 外部キーの pg_constraint タプルから列番号や演算子 OID を分解。
  • check_functional_grouping: 主キー制約を用いて、クエリのグループ化における関数的依存性を検証。

制御フローとデータフロー

1. **作成フロー**: CreateConstraintEntry が呼び出されると、C 配列を Postgres 配列（ArrayType）に変換し、heap_form_tuple でタプルを構築。CatalogTupleInsert で挿入後、record_object_address_dependencies を用いて pg_depend に依存関係を登録する。 2. **検索フロー**: systable_beginscan を使用し、ConstraintRelidTypidNameIndexId などのインデックス経由で効率的にタプルをスキャンする。 3. **NOT NULL 抽出**: extractNotNullColumn は SysCacheGetAttrNotNull を使用して conkey 属性を直接取得し、DatumGetArrayTypeP でデトーストして列番号を返す。

読者向けメモ

  • **メモリ管理**: CreateConstraintEntry 内で作成される ObjectAddresses リストは、処理終了後に free_object_addresses で解放されます。
  • **ロック**: カタログ操作時は RowExclusiveLock を使用します。ConstraintNameIsUsed 等の検索関数は AccessShareLock を使用しますが、呼び出し側で適切なロック（通常は対象テーブルの排他ロック）を保持していることが前提です。
  • **制約の継承**: coninhcount や conislocal は、パーティションテーブルや継承テーブルにおける制約のライフサイクルを制御する重要なフラグです。
  • **省略された詳細**: 本ファイルはカタログ操作に特化しており、制約の「実行時検証（トリガーやインデックスの作成）」ロジックは commands/ 以下の各ハンドラに委譲されています。
