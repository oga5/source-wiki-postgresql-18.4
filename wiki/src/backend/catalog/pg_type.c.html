src/backend/catalog/pg_type.c

概要

本ファイルは、PostgreSQLのシステムカタログ pg_type に対する操作を抽象化・提供するモジュールです。データ型の作成、依存関係の構築、および型名の変更（配列型やマルチレンジ型を含む）を管理します。

モジュール内での役割

src/backend/catalog サブシステムの一部として、SQLレベルの CREATE TYPE や ALTER TYPE コマンドのバックエンド処理を担います。pg_type.h で定義されたスキーマに基づき、カタログへの行挿入や更新を安全に行うためのインターフェースを提供します。

責務

  • **型定義の管理**: TypeShellMake によるシェル型（未定義のプレースホルダー）の作成と、TypeCreate による完全な型定義への昇格。
  • **依存関係の構築**: GenerateTypeDependencies を通じ、型が依存する関数（I/O関数等）、所有者、名前空間、および拡張機能との関係をカタログに記録。
  • **名前空間の整合性**: 型名の変更時に、関連する配列型やマルチレンジ型の名前を自動的に調整・追従させる。

重要なシンボル

  • TypeShellMake: 新しい型をカタログに登録する際、まず「シェル」として挿入し、後続の定義を待機させるための関数。
  • TypeCreate: 型の属性（サイズ、アライメント、I/O関数等）を検証し、pg_type に永続化するメイン関数。
  • GenerateTypeDependencies: 型の作成・変更時に、システム内の他のオブジェクトとの依存関係を再構築する。
  • RenameTypeInternal: 型の名前を変更し、必要に応じて関連する配列型の名前も再帰的に変更する。
  • binary_upgrade_next_pg_type_oid: pg_upgrade 実行時に、特定のOIDを強制するために使用されるグローバル変数。

制御フローとデータフロー

1. **作成フロー**: TypeCreate が呼び出されると、まず型のサイズやアライメントが妥当か検証されます。次に pg_type をオープンし、既存のシェル型が存在すれば更新、なければ新規挿入を行います。最後に GenerateTypeDependencies が呼ばれ、依存関係が記録されます。 2. **依存関係フロー**: GenerateTypeDependencies は、pg_type の行から I/O 関数やデフォルト値の式を抽出し、dependency.c の機能を利用して pg_depend 等のカタログにレコードを書き込みます。 3. **名前変更フロー**: RenameTypeInternal は、対象の型名を変更した後、typarray フィールドを参照して関連する配列型を特定し、同様に名前を変更します。

読者向けメモ

  • **シェル型**: PostgreSQLでは、循環参照を解決するために、まず名前だけを登録した「シェル型」を作成し、後から詳細を埋める手法が多用されます。TypeShellMake はこのための重要なステップです。
  • **依存関係の再構築**: rebuild フラグが true の場合、既存の依存関係を一度削除してから再構築します。これは ALTER TYPE 等で型定義が変更された際に、古い依存関係をクリーンアップするために不可欠です。
  • **配列型の命名**: 配列型の名前は慣習的に _ を先頭に付与しますが、衝突時は _1, _2 と連番が付与されます。このロジックは makeArrayTypeName に集約されています。
