src/backend/parser/parse_coerce.c

概要

parse_coerce.c は、PostgreSQL の意味解析フェーズにおいて、データ型の変換（Coercion）と型制約（Typmod）の適用を管理するモジュールです。SQL の CAST 構文や、関数呼び出し・演算子適用時の暗黙的な型変換、代入時の型整合性チェックなどを一手に引き受けます。

モジュール内での役割

src/backend/parser サブシステムの一部として、parse_expr.c や analyze.c から呼び出されます。システムカタログ（pg_cast, pg_type）を参照し、型変換が可能かどうかの判定から、実行時に必要な変換ノード（FuncExpr, RelabelType, CoerceToDomain 等）の構築までを担います。

責務

  • **型変換の判定**: 2つの型間で変換が可能か（can_coerce_type）、どのような変換パスが必要か（find_coercion_pathway）を決定する。
  • **変換ノードの生成**: 構文木に型変換のためのノードを挿入する。
  • **ドメイン制約の適用**: ドメイン型への変換時に、制約チェックを行うノードを構築する。
  • **ポリモーフィック型の解決**: ANYELEMENT や ANYCOMPATIBLE などのポリモーフィック型関数の引数・戻り値の型を、実際の入力に基づいて解決する。
  • **共通型の選択**: CASE 文や UNION などで複数の式から共通の型を導出する。

重要なシンボル

エントリーポイント（型変換）

  • coerce_to_target_type: 型変換の汎用的な入り口。呼び出し元がエラーハンドリングを制御できるよう、失敗時に NULL を返す。
  • coerce_type: 変換の核心。式ノードに対して変換ノードを付与する。
  • can_coerce_type: 変換が可能かどうかの事前チェック。

ポリモーフィック型解決

  • enforce_generic_type_consistency: 関数シグネチャのポリモーフィック型を実際の型に解決し、矛盾がないか検証する。
  • check_generic_type_consistency: 実際の引数型がポリモーフィックな宣言と整合しているかを確認する。

共通型導出

  • select_common_type: 式リストから共通のスーパータイプを導出する。
  • coerce_to_common_type: 式を導出された共通型へ変換する。

内部変換パス

  • find_coercion_pathway: pg_cast を検索し、変換手法（関数呼び出し、バイナリ互換、I/O変換など）を特定する。
  • build_coercion_expression: 決定されたパスに基づいて、具体的な変換ノードを生成する。

制御フローとデータフロー

1. **変換パスの特定**: find_coercion_pathway が pg_cast を参照し、CoercionPathType（関数、リラベル、配列変換、I/O変換）を返す。 2. **ノード構築**: build_coercion_expression が、特定されたパスに従って FuncExpr や ArrayCoerceExpr などのノードを生成する。 3. **ドメイン処理**: ターゲットがドメイン型の場合、coerce_to_domain が呼び出され、制約チェックのためのノードがラップされる。 4. **ポリモーフィック解決**: enforce_generic_type_consistency が引数リストを走査し、ANYELEMENT 等の型を具体的な OID に置換する。

直接依存

  • catalog/pg_cast.h, pg_type.h: 型変換ルールと型定義の参照。
  • utils/lsyscache.h, typcache.h: 型情報やキャッシュの取得。
  • nodes/makefuncs.h: 変換ノードの生成補助。

関連コード

  • src/include/parser/parse_coerce.h: 公開インターフェース定義。
  • src/backend/parser/parse_expr.c: 式解析のメインロジック。
  • src/backend/utils/adt/ruleutils.c: 変換ノードの表示制御（hide_coercion_node との関連）。

読者向けメモ

  • **メモリ管理**: 本モジュールは既存の式ノードを破壊せず、新しいノードをラップして返す設計です。
  • **hide_coercion_node**: 複数の変換ステップを重ねる際、中間ノードを COERCE_IMPLICIT_CAST に設定することで、EXPLAIN 等の出力で冗長な変換を表示させない工夫がなされています。
  • **未実装部分**: RECORDARRAYOID と複合型配列の変換など、一部の複雑なケースは将来的な拡張のためにコメントアウトされています。
  • **注意点**: coerce_type は pstate が NULL でも動作しますが、Param の型解決など一部の機能は pstate に依存します。
