src/backend/parser/parse_func.c

概要

SQL 解析フェーズにおいて、関数呼び出し、プロシージャ呼び出し、および集約関数・ウィンドウ関数の解決を担う中核モジュールです。SQL 内の関数名と引数リストをシステムカタログ（pg_proc 等）と照合し、適切な関数 OID の特定、型変換の挿入、および構文上の制約チェックを行います。

モジュール内での役割

src/backend/parser サブシステムの一部として、parse_expr.c から呼び出される関数解決の専門家です。名前解決の結果に基づき、FuncExpr（通常関数）、Aggref（集約）、WindowFunc（ウィンドウ関数）といったノードを生成し、プランナが処理可能な形式へ変換します。

責務

  • **関数・プロシージャ解決**: 名前と引数型に基づき、オーバーロードされた関数から最適な候補を選択する。
  • **型整合性の確保**: 引数の型を関数の宣言型に合わせるための暗黙的キャスト（make_fn_arguments）の挿入。
  • **構文制約の検証**: WITHIN GROUP、FILTER、OVER 句の配置が関数種別に対して妥当かどうかのチェック。
  • **特殊な構文の解釈**: tab.col と col(tab) の等価性（列プロジェクション）の解決。
  • **SRF（Set-Returning Function）の配置制限**: SRF が許可されていない場所（WHERE 句など）での使用を禁止する。

重要なシンボル

関数解決のメインエントリ

  • ParseFuncOrColumn: 本ファイルのメイン関数。関数呼び出し、列参照、型キャストのいずれであるかを判別し、適切なノードを生成する。
  • func_get_detail: システムカタログを検索し、関数候補のリストから最適なものを特定する。

候補選択とマッチング

  • func_match_argtypes: 候補リストから、引数型が一致（またはキャスト可能）なものだけを抽出する。
  • func_select_candidate: 複数の候補がある場合に、型優先度やカテゴリに基づき「最も適切な」関数を選択する。

ユーティリティ・検証

  • make_fn_arguments: 引数リストに対して必要な型変換ノードを挿入する。
  • unify_hypothetical_args: 仮説的集約（Hypothetical-set aggregate）の引数型を統一する。
  • check_srf_call_placement: SRF が現在の構文コンテキストで許可されているか検証する。

制御フローとデータフロー

1. **初期化**: ParseFuncOrColumn が引数リストを走査し、型情報を抽出。 2. **候補検索**: func_get_detail が FuncnameGetCandidates を呼び出し、カタログから候補を取得。 3. **解決**:

  • 候補が単一ならそれを採用。
  • 候補が複数なら func_select_candidate で絞り込み。
  • 候補なしの場合、列プロジェクション（ParseComplexProjection）や型キャストの可能性を検討。

    • 4. **変換**: make_fn_arguments が引数ノードにキャストを適用。 5. **ノード生成**: 最終的に FuncExpr, Aggref, WindowFunc のいずれかを生成して返す。

読者向けメモ

  • **メモリ管理**: 本モジュールで生成されるノードは ParseState のメモリコンテキストに割り当てられます。
  • **エラーハンドリング**: ereport(ERROR, ...) を多用し、解析エラーを即座に報告します。parser_errposition を用いて SQL 内の正確な位置を特定するよう設計されています。
  • **未実装・制限**:
  • 集約関数における名前付き引数のサポートは現在制限されています。
  • ParseComplexProjection は、複雑な型（RECORD 等）の解決において expandRecordVariable に依存しており、ここが解析のボトルネックになる場合があります。
  • **設計思想**: PostgreSQL の関数解決は、単なる名前一致ではなく、型優先度やカテゴリに基づく複雑なヒューリスティックを用いており、func_select_candidate がその心臓部です。
