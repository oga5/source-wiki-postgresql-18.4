src/backend/parser/parse_relation.c

概要

parse_relation.c は、PostgreSQL の意味解析フェーズにおいて、クエリ内のリレーション（テーブル、ビュー、CTE、サブクエリ、関数など）の名前解決と、それらのリレーションを表現する RangeTblEntry (RTE) の構築・管理を担う中核モジュールです。

モジュール内での役割

src/backend/parser サブシステムにおいて、SQL の FROM 句や JOIN 句に現れるデータソースをシステムカタログや ParseState と照合し、プランナが利用可能な形式（RangeTblEntry）へ変換する役割を持ちます。

責務

  • **名前解決**: SQL 内の識別子（テーブル名、エイリアス）を ParseNamespaceItem を通じて解決する。
  • **RTE 構築**: 各種データソース（テーブル、関数、サブクエリ、CTE、VALUES、JOIN）に対応する RangeTblEntry を生成し、ParseState に登録する。
  • **可視性管理**: LATERAL 句やスコープに基づいたリレーションの可視性チェック。
  • **エラー診断**: 存在しないテーブルやカラムに対する、ヒント付きの親切なエラーメッセージ生成（曖昧な名前の解決やスペルミスの推測）。
  • **権限管理**: リレーションに対する SELECT 権限の追跡と RTEPermissionInfo の管理。

重要なシンボル

名前解決・検索

  • refnameNamespaceItem: 名前空間からリレーションを検索するメイン関数。
  • scanNameSpaceForCTE / scanNameSpaceForENR: CTE や一時的な名前付きリレーションの検索。
  • colNameToVar: カラム名を Var ノードに解決する。
  • searchRangeTableForRel / searchRangeTableForCol: エラーメッセージ生成用のヒューリスティック検索。

RTE 構築（ファクトリ群）

  • addRangeTableEntry: 通常のテーブル用。
  • addRangeTableEntryForSubquery: サブクエリ用。
  • addRangeTableEntryForFunction: 関数用。
  • addRangeTableEntryForJoin: JOIN 構造用。
  • addRangeTableEntryForCTE: WITH 句の CTE 用。

診断・ユーティリティ

  • FuzzyAttrMatchState: カラム名のスペルミス推測用状態構造体。
  • markNullableIfNeeded: 外部結合による NULL 許容状態のマーク。
  • markVarForSelectPriv: カラムへのアクセス権限要求の記録。

制御フローとデータフロー

1. **解析開始**: analyze.c 等から addRangeTableEntry... 系関数が呼ばれ、RTE が生成される。 2. **名前空間登録**: addNSItemToQuery により、生成された ParseNamespaceItem が ParseState の p_namespace リストに追加される。 3. **カラム解決**: 式の解析中に colNameToVar が呼ばれると、p_namespace を走査し、scanNSItemForColumn を経由して Var ノードを生成する。 4. **エラー発生時**: errorMissingRTE や errorMissingColumn が呼ばれ、searchRangeTableForRel/Col を用いて、スペルミスや LATERAL 不足などのヒントを生成する。

読者向けメモ

  • **メモリ管理**: ParseState に紐付くメモリコンテキスト上で palloc されるため、個別の pfree は不要です。
  • **不変条件**: p_namespace 内のアイテムは、LATERAL 句の有無やスコープによって可視性が動的に変化します。
  • **エラー診断の深さ**: 本ファイルは単なる解決器ではなく、ユーザーフレンドリーなエラーメッセージ生成のために Levenshtein 距離を用いたスペルチェックなど、非常に詳細な診断ロジックを含んでいます。
  • **省略された詳細**: isQueryUsingTempRelation などの再帰的なクエリツリー走査関数は、主に DDL や特定の制約チェックのために存在しており、通常の名前解決フローとは独立しています。
