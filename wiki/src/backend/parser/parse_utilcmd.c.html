src/backend/parser/parse_utilcmd.c

概要

parse_utilcmd.c は、CREATE TABLE、ALTER TABLE、CREATE INDEX、CREATE STATISTICS、CREATE RULE などの DDL（データ定義言語）ユーティリティコマンドに対する意味解析（Parse Analysis）を担当します。

かつては analyze.c で処理されていましたが、クエリツリーのキャッシュやロック管理の安全性、および実行時の状態依存性を考慮し、現在は各コマンドの実行開始時に呼び出される設計となっています。

モジュール内での役割

src/backend/parser モジュールにおいて、DDL 固有の複雑な変換ロジックを分離したサブシステムです。生の解析ツリー（Raw Parse Tree）を、システムカタログを参照して型解決や制約の展開を行った「実行可能なユーティリティコマンドのリスト」へ変換します。

責務

  • **DDL 変換**: CREATE TABLE 等の構文を、実行時に必要な複数のコマンド（DefineRelation, DefineIndex 等）のシーケンスに分解する。
  • **制約の統合**: 列定義に散らばる制約（CHECK, NOT NULL, FK, UNIQUE/PK）を収集・正規化し、適切な順序で実行されるよう整理する。
  • **LIKE 句の展開**: CREATE TABLE ... (LIKE ...) の解析。列定義のコピーや、インデックス・統計情報のクローン生成を行う。
  • **シリアル/ID 列の管理**: SERIAL や IDENTITY 列に関連するシーケンスの自動生成と所有権の設定。
  • **パーティション解析**: パーティション境界（Partition Bound）の型変換と検証。

重要なシンボル

コンテキスト管理

  • CreateStmtContext: CREATE TABLE 解析中に状態を保持する構造体。列定義、制約リスト、実行順序（before/after）を管理する。
  • CreateSchemaStmtContext: CREATE SCHEMA 内の要素を分類・整理するための構造体。

主要な変換関数

  • transformCreateStmt: CREATE TABLE のメインエントリー。列と制約を分離し、実行順序を決定する。
  • transformAlterTableStmt: ALTER TABLE の各サブコマンドを解析し、必要な追加コマンドを生成する。
  • transformIndexStmt / transformStatsStmt: インデックスや統計情報の式を解析し、プランナが扱える形式に変換する。
  • transformRuleStmt: ルール定義の解析と、OLD/NEW 擬似リレーションの解決。

補助・クローン生成

  • generateSerialExtraStmts: シリアル列のためのシーケンス生成と所有権設定。
  • generateClonedIndexStmt: 既存インデックスから新しいインデックス定義を生成する。
  • transformColumnDefinition: 列定義の型解決と制約の抽出。

制御フローとデータフロー

1. **初期化**: ParseState を作成し、ターゲットとなるリレーションのロックと名前空間の検証を行う。 2. **要素の分離**: transformCreateStmt が tableElts を走査し、列定義と制約を CreateStmtContext に蓄積する。 3. **制約の解決**:

  • transformIndexConstraints: インデックスを生成する制約（PK, UNIQUE等）を処理。
  • transformFKConstraints: 外部キー制約を処理。
  • transformCheckConstraints: CHECK 制約を処理。

    • 4. **順序付け**: blist（実行前）、stmt（本体）、alist（実行後）の順にコマンドリストを連結して返す。 5. **LIKE 展開**: expandTableLikeClause が、テーブル作成後に確定する列番号を必要とするインデックスや統計情報のクローン生成を遅延実行する。

読者向けメモ

  • **メモリ所有権**: CreateStmtContext 内のリストは CurrentMemoryContext 上で構築されます。
  • **実行順序**: CREATE TABLE では、制約の依存関係（例: PK インデックス作成後に FK を作成する）を維持するため、blist と alist を用いた順序制御が極めて重要です。
  • **制約の重複**: transformIndexConstraints は、ユーザーが指定した重複する制約（例: UNIQUE PRIMARY KEY）をマージするロジックを含んでいます。
  • **未実装/浅い部分**: CREATE SCHEMA の解析ロジックは、オブジェクト間の相互参照（forward references）の解決において一部簡略化されています。
