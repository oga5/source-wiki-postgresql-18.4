src/backend/parser/parse_type.c

概要

PostgreSQLのパーサーにおいて、SQL文中の型名（TypeName）をシステムカタログ上の型情報（pg_type）へ解決・変換するためのユーティリティ群を提供します。

モジュール内での役割

src/backend/parser サブシステムの一部として、構文解析フェーズで出現する型指定を、実行エンジンが扱える OID や型修飾子（typmod）へ変換するブリッジの役割を果たします。

責務

  • **型解決**: TypeName 構造体から pg_type のシステムキャッシュエントリを検索・取得する。
  • **型修飾子の検証**: typmodin 関数を呼び出し、指定された型修飾子が妥当か検証・変換する。
  • **メタデータ取得**: 型の OID から、長さ、値渡しフラグ、照合順序（Collation）などの属性を抽出する。
  • **文字列変換**: 型名や型リストをエラーメッセージ等で利用可能な文字列形式に変換する。

重要なシンボル

  • LookupTypeName: TypeName から Type（HeapTuple）を検索する主要なエントリポイント。
  • typenameType: 型の存在確認とシェル型（未定義型）のチェックを行い、有効な Type を返す。
  • typenameTypeId: 型名から直接 OID を取得する簡便な関数。
  • typenameTypeMod: 型修飾子（例: varchar(32) の 32）を解析・検証する内部関数。
  • GetColumnDefCollation: 列定義時の照合順序を決定する。

制御フローとデータフロー

1. **検索**: LookupTypeNameExtended が TypeName の名前リストを解析し、pg_type システムキャッシュを検索します。 2. **型修飾子処理**: 型が見つかると typenameTypeMod が呼び出され、構文解析された typmods リストを typmodin 関数に渡して内部表現へ変換します。 3. **検証**: typenameType などのラッパー関数が、取得したタプルが「シェル型（定義未完了）」でないことを保証します。 4. **終了処理**: 呼び出し元は、取得した Type（HeapTuple）に対して必ず ReleaseSysCache を実行する必要があります。

読者向けメモ

  • **メモリ管理**: 本ファイル内の関数が返す Type は HeapTuple であり、呼び出し側での ReleaseSysCache が必須です。これを忘れるとキャッシュリークを引き起こします。
  • **シェル型**: typisdefined フラグを確認せずに型情報を利用すると、未定義の型を誤って処理する可能性があるため注意してください。
  • **エラー位置**: pstate を引数に取る関数は、エラー発生時に parser_errposition を用いて SQL 文中の正確な位置を報告します。デバッグ時にはこの仕組みが非常に有用です。
