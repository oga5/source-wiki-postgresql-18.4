src/backend/parser/analyze.c

概要

analyze.c は、PostgreSQL のパーサーモジュールにおける「意味解析（Semantic Analysis）」のメインドライバです。parser.c が生成した「生の解析ツリー（RawStmt）」を受け取り、システムカタログを参照して型解決、名前解決、権限チェックを行い、プランナが処理可能な「クエリツリー（Query）」へ変換します。

モジュール内での役割

本ファイルは、SQL 文の種類ごとに変換処理を振り分けるハブの役割を果たします。各句（FROM, WHERE, SELECT 等）の具体的な変換ロジックは parse_*.c ファイル群に委譲されており、本ファイルはそれらを統合して Query 構造体を構築します。

責務

  • **解析の統括**: ParseState を初期化し、SQL 文の種類に応じて適切な変換関数を呼び出す。
  • **文の変換**: INSERT, UPDATE, DELETE, SELECT, MERGE 等の各 SQL 文を Query ノードへ変換。
  • **特殊処理**: SELECT INTO の CREATE TABLE AS への変換、ON CONFLICT 句の解析、RETURNING 句の処理。
  • **フック管理**: プラグインが解析終了時に介入するための post_parse_analyze_hook の提供。
  • **メタデータ付与**: クエリIDの生成（JumbleQuery）や、解析に必要なスナップショットの要否判定。

重要なシンボル

解析エントリーポイント

  • parse_analyze_fixedparams: パラメータ型が既知の場合の解析。
  • parse_analyze_varparams: パラメータ型をコンテキストから推論する場合の解析。
  • parse_analyze_withcb: カスタムコールバックを用いた解析。
  • parse_sub_analyze: サブクエリや CTE 内部の再帰的な解析。

文変換ファミリー

  • transformStmt: 文の種類を判定し、適切な変換関数へディスパッチするメイン関数。
  • transformInsertStmt, transformDeleteStmt, transformUpdateStmt: DML 文の変換。
  • transformSelectStmt, transformValuesClause, transformSetOperationStmt: SELECT 関連の変換。

ユーティリティ・検証

  • stmt_requires_parse_analysis: 解析が必要な文か判定。
  • analyze_requires_snapshot: スナップショットが必要な文か判定。
  • transformReturningClause: RETURNING 句の解析と名前空間管理。
  • transformOnConflictClause: INSERT ... ON CONFLICT の解析。

制御フローとデータフロー

1. **初期化**: make_parsestate() で ParseState を作成し、ソーステキストやパラメータ情報を保持。 2. **ディスパッチ**: transformStmt() が nodeTag を見て、対応する transform*Stmt 関数を呼び出す。 3. **再帰的変換**: 各 transform*Stmt 関数は、parse_relation.c や parse_expr.c の関数を呼び出し、FROM 句、WHERE 句、ターゲットリストを順次変換する。 4. **名前空間管理**: ParseState 内の p_namespace を操作し、可視リレーションを制御。 5. **最終化**: assign_query_collations() で照合順序を確定し、必要に応じて parseCheckAggregates() で集約関数の配置を検証。 6. **解放**: free_parsestate() でメモリを解放。

読者向けメモ

  • **メモリ管理**: ParseState は CurrentMemoryContext 上で確保され、free_parsestate で解放されます。解析中に生成されるノードは、クエリツリーの一部として永続化されるため、メモリコンテキストの扱いに注意が必要です。
  • **エラーパス**: ereport(ERROR, ...) が発生すると、PostgreSQL のエラーハンドリング機構によりスタックが巻き戻されます。
  • **制約**: transformStmt は CMD_UTILITY を返す場合、元の構文木をそのまま保持します。これはプランナがユーティリティコマンドを直接実行するためです。
  • **省略箇所**: DEBUG_NODE_TESTS_ENABLED が有効な場合のデバッグ用コードは、本質的な解析フローではないため詳細な解説を省略しています。
