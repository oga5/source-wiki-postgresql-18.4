src/backend/optimizer/util/appendinfo.c

概要

本ファイルは、PostgreSQLのクエリ最適化プロセスにおいて、継承テーブルやパーティションテーブルの「親リレーション」と「子リレーション」間で式や属性の変換を行うためのルーチンを提供します。

モジュール内での役割

src/backend/optimizer/util モジュールの一部として、主にプランナが継承関係を解決する際に使用されます。特に、親テーブルに対して定義されたクエリ（式やターゲットリスト）を、個々の物理的な子テーブル（リーフリレーション）向けに書き換える役割を担います。

責務

  • **属性変換**: 親テーブルの Var ノードを、対応する子テーブルの Var ノードへ変換する。
  • **メタデータ管理**: AppendRelInfo 構造体を生成し、親と子の列マッピング（逆変換用配列を含む）を保持する。
  • **クエリツリーの調整**: RestrictInfo や PlaceHolderVar などのプランナノードに含まれるリレーションIDセットを、親から子へ適切に置換する。
  • **行識別子管理**: UPDATE/DELETE/MERGE 実行時に必要な行識別子（ctid や wholerow 等）を各子リレーションに伝播させる。

重要なシンボル

  • AppendRelInfo: 親子関係の変換情報を保持する主要な構造体。
  • make_append_rel_info: 親子関係のメタデータを作成するコンストラクタ。
  • adjust_appendrel_attrs: 式ツリー内の Var を親から子へ変換するメインのエントリポイント。
  • adjust_appendrel_attrs_mutator: expression_tree_mutator を用いて再帰的に式を変換する内部関数。
  • add_row_identity_var: 行更新に必要な識別子変数を登録する。

制御フローとデータフロー

1. **初期化**: make_append_rel_info が親子の Relation を受け取り、列名や型を照合して translated_vars リストを構築する。 2. **変換実行**: adjust_appendrel_attrs が呼び出されると、adjust_appendrel_attrs_context を介して式ツリーを走査する。 3. **ノード置換**: adjust_appendrel_attrs_mutator が Var ノードを検出し、AppendRelInfo に基づいて子テーブルの属性番号へ書き換える。 4. **再帰処理**: RestrictInfo や PlaceHolderVar などの複雑なノードに対しては、関連する Relids（ビットマップセット）を adjust_child_relids で更新する。

読者向けメモ

  • **変換のタイミング**: この変換は、サブリンクがサブプランに変換された後に行われるため、Query ノード自体を再帰的に処理する必要はありません。
  • **ROWID_VAR**: ROWID_VAR は特殊なプレースホルダーです。adjust_appendrel_attrs_mutator 内で、対象がリーフリレーションである場合にのみ具体的な変数へ置換されます。
  • **エラーハンドリング**: 属性の不一致（型や照合順序の不一致）は elog や ereport で厳格にチェックされます。これは継承テーブルの定義変更がプランナの想定と乖離していないことを保証するためです。
