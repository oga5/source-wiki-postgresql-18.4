src/backend/optimizer/util/paramassign.c

概要

本ファイルは、PostgreSQLのクエリ最適化フェーズにおいて、実行時に使用される PARAM_EXEC スロットの割り当てと管理を行う関数群を提供します。主にサブクエリやネストループ結合において、上位レベルから下位レベルへ値を渡すためのパラメータIDを生成・追跡します。

モジュール内での役割

optimizer/util モジュールの一部として、プラン作成（createplan.c 等）の過程で必要となる実行時パラメータのインフラを提供します。プランナーが保持する PlannerInfo 構造体内のパラメータ管理用リストを操作し、実行エンジンが正しく値を参照できるようにします。

責務

  • **PARAM_EXEC スロットの管理**: root->glob->paramExecTypes を通じて、実行時に必要なパラメータの型情報を保持する。
  • **パラメータの重複排除**: 同一の Var や PlaceHolderVar が複数回参照される場合、可能な限り既存の PARAM_EXEC ID を再利用する。
  • **ネストループパラメータの追跡**: root->curOuterParams を管理し、ネストループ結合の右側で必要となる上位レベルの値を特定・供給する。
  • **特殊パラメータの割り当て**: 値の受け渡しではなく、実行時のシグナリング（再帰的結合やEvalPlanQualなど）に使用されるIDを確保する。

重要なシンボル

  • replace_outer_var / replace_outer_placeholdervar: 上位レベルの変数を PARAM_EXEC に置換し、必要に応じて plan_params に登録する。
  • replace_nestloop_param_var: ネストループ結合用に変数をパラメータ化し、curOuterParams に登録する。
  • identify_current_nestloop_params: ネストループの左側で供給可能なパラメータを特定し、リストから抽出する。
  • generate_new_exec_param: 新規の PARAM_EXEC ID を発行し、型情報を登録する。

制御フローとデータフロー

1. **置換要求**: プランナーが下位ノードを構築する際、上位レベルの変数（Var や PlaceHolderVar）に遭遇すると、replace_outer_* 系関数を呼び出します。 2. **ID 割り当て**: 既存の plan_params リストを検索し、一致するものがあればその ID を返します。なければ新規に PARAM_EXEC ID を発行し、root->glob->paramExecTypes に型情報を追加します。 3. **ネストループ処理**: ネストループ結合の構築時、identify_current_nestloop_params が curOuterParams をスキャンし、現在の結合ノードで解決可能なパラメータを抽出してプランノードに付与します。 4. **永続化**: paramExecTypes はプラン全体で共有され、一度割り当てられた ID は削除されません。

読者向けメモ

  • **重複排除の限界**: plan_params や curOuterParams は、サブクエリやネストループの処理が完了するとリセットされます。そのため、異なるサブクエリ間で同じ変数を参照しても、それぞれ個別の PARAM_EXEC ID が割り当てられることがあります。
  • **nullingrels の扱い**: identify_current_nestloop_params 内での nullingrels の更新は、外結合の最適化に伴う複雑な調整であり、このファイルにおける最も繊細なロジックの一つです。
  • **デバッグ**: PARAM_EXEC ID の不整合は実行時のクラッシュや誤った結果を招くため、root->glob->paramExecTypes の内容と実際の Param ノードの ID が一致しているか確認することが重要です。
