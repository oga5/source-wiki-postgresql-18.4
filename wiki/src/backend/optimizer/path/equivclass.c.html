src/backend/optimizer/path/equivclass.c

概要

equivclass.c は、PostgreSQL プランナにおける「等価クラス（EquivalenceClass: EC）」の管理と推論を行う中核モジュールです。クエリ内の A = B や B = C といった等価関係を推論し、推移律（A = C）を適用することで、冗長な条件の排除や、結合順序の最適化、インデックススキャンの効率化を実現します。

モジュール内での役割

src/backend/optimizer/path サブシステムの一部として、プランナの初期フェーズでクエリ内の等価条件を収集・正規化します。ここで構築された EC は、pathkeys.c によるソート順序の管理や、joinpath.c における結合条件の生成、indxpath.c におけるインデックス利用の判断に利用されます。

責務

  • **等価関係の統合**: UNION-FIND アルゴリズム的に等価クラスをマージし、正規化された集合を維持する。
  • **推論と導出**: 等価関係から導出可能な制限条件（Restriction Clause）や結合条件（Join Clause）を生成する。
  • **正規化**: 式の型や照合順序（Collation）を正規化し、異なる表現の等価性を判定可能にする。
  • **子リレーション対応**: AppendRel（継承や UNION ALL）における子リレーションへの等価関係の伝播。

重要なシンボル

データ構造

  • EquivalenceClass: 等価な式の集合。ec_members（式）、ec_sources（元となった条件）、ec_derives_list（推論された条件）を保持。
  • EquivalenceMember: EC 内の個別の式。定数や特定のリレーションの式を表現。
  • ECDerivesKey / ECDerivesEntry: 導出された条件を高速検索するためのハッシュテーブル構造。

主要関数

  • process_equivalence: 新たな等価条件を EC に登録・マージするメインエントリ。
  • generate_base_implied_equalities: ベースリレーションに対する制限条件を EC から生成。
  • generate_join_implied_equalities: 結合ノードで必要な結合条件を EC から生成。
  • get_eclass_for_sort_expr: ソート順序（PathKey）に対応する EC を検索・生成。
  • canonicalize_ec_expression: 式を EC 比較に適した型・照合順序に変換。

制御フローとデータフロー

1. **登録**: process_equivalence がクエリ内の WHERE 句等を走査し、等価関係を eq_classes リストに蓄積。 2. **正規化**: canonicalize_ec_expression により、異なる型や照合順序を持つ式を比較可能な形式に統一。 3. **導出**: generate_base_implied_equalities が、EC 内のメンバー間を繋ぐ等価条件を RestrictInfo として生成し、プランナの条件リストへ注入。 4. **結合時**: generate_join_implied_equalities が、結合対象のリレーションに応じて必要な結合条件を動的に生成。 5. **検索**: ec_search_clause_for_ems 等が、既存の導出条件をハッシュテーブルまたはリストから検索し、重複生成を回避。

読者向けメモ

  • **メモリ管理**: EC は PlannerInfo->planner_cxt 上に構築されます。GEQO 等の探索フェーズでは注意が必要です。
  • **不変条件**: ec_merging_done フラグが立つと、EC のマージは禁止されます。これ以降は get_eclass_for_sort_expr 等による追加のみが許可されます。
  • **ハッシュテーブル**: 導出条件が EC_DERIVES_HASH_THRESHOLD (32) を超えると、線形リストからハッシュテーブルへ切り替わり、検索効率を維持します。
  • **制約**: ec_broken フラグが立った EC は、推論に失敗したことを示します。この場合、元の ec_sources を再利用するフォールバック処理が行われます。
