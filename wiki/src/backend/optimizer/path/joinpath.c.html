src/backend/optimizer/path/joinpath.c

概要

joinpath.c は、PostgreSQL オプティマイザにおいて、2 つの結合対象リレーション（RelOptInfo）から可能な結合パス（Path）を生成・評価する中心的なモジュールです。

モジュール内での役割

src/backend/optimizer/path サブシステムの一部として、add_paths_to_joinrel を通じて結合リレーションのパスリストを構築します。joinrels.c が結合順序を決定するのに対し、本ファイルは決定された結合順序に対して、具体的にどのようなアルゴリズム（Nestloop, Mergejoin, Hashjoin）で結合を実行するのが最適かを検討します。

責務

  • **結合パスの生成**: Nestloop, Mergejoin, Hashjoin の各アルゴリズムに基づいたパスを生成し、コスト評価を行う。
  • **パスの選別**: add_path / add_partial_path を呼び出し、コストやパラメータ化の観点で劣るパスを排除する。
  • **特殊な結合の処理**: JOIN_SEMI, JOIN_ANTI, JOIN_UNIQUE_INNER/OUTER などの特殊な結合タイプに対するコスト補正やパス生成。
  • **並列結合のサポート**: パラレルクエリ実行のための partial_path 生成。
  • **拡張性**: set_join_pathlist_hook を提供し、外部拡張が結合パス生成プロセスに介入できるようにする。

重要なシンボル

  • add_paths_to_joinrel: 本ファイルのメインエントリポイント。結合リレーションに対するすべてのパス生成ロジックを統括する。
  • try_nestloop_path / try_mergejoin_path / try_hashjoin_path: 各結合アルゴリズムのパスを生成し、add_path を試みるヘルパー関数。
  • set_join_pathlist_hook: 外部モジュールが結合パス生成をカスタマイズするためのフック。
  • JoinPathExtraData: 結合処理中に必要なメタデータ（SpecialJoinInfo や restrictlist など）を保持する構造体。

制御フローとデータフロー

1. **初期化**: add_paths_to_joinrel が呼び出され、JoinPathExtraData を初期化。 2. **事前準備**: 結合タイプに応じた一意性（inner_unique）の判定や、マージ結合用の句の選択を行う。 3. **パス生成の試行**:

  • sort_inner_and_outer / match_unsorted_outer: Mergejoin のパスを生成。
  • hash_inner_and_outer: Hashjoin のパスを生成。
  • try_nestloop_path: Nestloop のパスを生成。

    • 4. **拡張**: FDW（外部データラッパー）のパス生成や、フックによる外部パスの追加を許可。 5. **評価**: 各関数内で add_path を呼び出し、既存のパスとのコスト比較を経て joinrel->pathlist が更新される。

読者向けメモ

  • 本ファイルは非常に複雑な条件分岐（特に JOIN_FULL や JOIN_UNIQUE 関連）を含みます。まずは add_paths_to_joinrel の全体的な流れを追い、その後に各結合アルゴリズム（Nestloop/Merge/Hash）の try_ 関数を読むのが効率的です。
  • PATH_PARAM_BY_REL などのマクロは、パスが特定のリレーションに依存しているか（パラメータ化されているか）を判定する重要なユーティリティです。
  • Memoize パスの生成ロジック（get_memoize_path）は、Nested Loop のコストを劇的に改善する可能性があるため、最近の PostgreSQL の最適化において重要度が増しています。
