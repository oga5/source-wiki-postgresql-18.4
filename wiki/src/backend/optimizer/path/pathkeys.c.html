src/backend/optimizer/path/pathkeys.c

概要

pathkeys.c は、PostgreSQL プランナにおいて「ソート順序」を抽象化した PathKey 構造体の生成、管理、および比較を行うユーティリティモジュールです。クエリの実行計画において、どのリレーションがどのような順序で出力されるかを正規化された形式で保持し、マージ結合の適合性判定やソートの省略判断、グループ化の最適化に利用されます。

モジュール内での役割

optimizer/path サブシステムの一部として、EquivalenceClass (EC) と密接に連携します。EC が「等価な式の集合」を管理するのに対し、本モジュールは「その集合に基づく順序」を管理します。プランナがパスを生成する際、そのパスが特定の順序要件を満たしているかを判定する中心的な役割を担います。

責務

  • **PathKey の正規化**: 重複や冗長な順序定義を排除し、ポインタ比較可能な「正規化された」パスキーを生成・管理する。
  • **順序の適合性判定**: 2つのパスキーリストを比較し、包含関係や等価性を判定する。
  • **最適化の判断**: マージ結合の条件適合性や、グループ化・DISTINCT・ORDER BY 処理におけるソートの必要性を評価する。
  • **パスキーの切り詰め**: 後続の処理で不要な順序情報を削除し、プランナの探索空間を効率化する。

重要なシンボル

パスキー生成・管理

  • make_canonical_pathkey: 指定された EC と演算子族から正規化された PathKey を生成・取得する。
  • append_pathkeys: 冗長性を排除しつつ、パスキーリストを結合する。
  • make_pathkey_from_sortinfo / make_pathkey_from_sortop: ソート情報から PathKey を構築する主要なファクトリ関数。

比較・適合性判定

  • compare_pathkeys: 2つのパスキーリストを比較し、等価性や優劣を判定する。
  • pathkeys_contained_in: 順序の包含関係を判定する（ソート順序が要件を満たすかの確認）。
  • pathkey_is_redundant: 定数を含む EC や重複する EC を持つパスキーを冗長と判定する。

最適化・切り詰め

  • truncate_useless_pathkeys: マージ結合、ORDER BY、GROUP BY 等の観点から不要なパスキーを削除する。
  • get_useful_group_keys_orderings: グループ化処理において、入力パスの順序を活かせる代替順序を生成する。
  • find_mergeclauses_for_outer_pathkeys: 特定の順序に適合するマージ結合句を抽出する。

制御フローとデータフロー

1. **正規化**: make_canonical_pathkey は root->canon_pathkeys を参照し、既存のインスタンスがあればそれを再利用します。これにより、パスキーの比較は O(1) のポインタ比較で完結します。 2. **冗長性排除**: pathkey_is_redundant は、EC が定数に束縛されている場合や、リスト内に既に同一 EC が存在する場合に true を返します。 3. **パスの選択**: get_cheapest_path_for_pathkeys は、指定された順序要件を満たすパスのうち、コストが最小のものを探索します。 4. **切り詰め**: truncate_useless_pathkeys は、複数の pathkeys_useful_for_* 関数を呼び出し、後続の処理で利用価値のある順序の長さ（プレフィックス）を計算してリストを短縮します。

直接依存

  • EquivalenceClass (equivclass.c): パスキーの基盤となる等価関係の定義。
  • PlannerInfo: クエリ全体のプランニング状態（canon_pathkeys, query_pathkeys 等）。
  • optimizer/pathnode.h, optimizer/paths.h: パス生成の基本構造体。
  • utils/lsyscache.h: 演算子族や型情報の取得。

関連コード

  • equivclass.c: PathKey の構成要素である EquivalenceClass の生成とマージ。
  • costsize.c: パスのコスト見積もり。
  • joinpath.c: マージ結合のパス生成において、本モジュールの関数を多用する。

読者向けメモ

  • **メモリ管理**: make_canonical_pathkey は root->planner_cxt 上でメモリを確保します。GEQO 等の特殊なプランニングコンテキストでも安全に動作するよう設計されています。
  • **不変条件**: ec_merging_done が true になるまでは、正規化されたパスキーを作成してはなりません。
  • **冗長性の定義**: 本モジュールでは「定数と等価な列」や「既にソート済みの列と等価な列」を冗長とみなします。これは、ソートコストを削減するための重要な最適化です。
  • **省略された詳細**: 本ファイルは PathKey の構築と判定に特化しており、実際のソートアルゴリズムやコスト計算の詳細は costsize.c 等に委譲されています。
