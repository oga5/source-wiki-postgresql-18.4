src/backend/optimizer/path/pathkeys.c

概要

pathkeys.c は、PostgreSQL のクエリプランナにおいて「パスキー（PathKey）」の構築、管理、および比較を行うためのユーティリティ群を提供します。パスキーは、クエリの実行計画（Path）が保持するデータのソート順序を表現するデータ構造であり、マージ結合の可否判定や、ORDER BY、GROUP BY、DISTINCT などの最適化において極めて重要な役割を果たします。

モジュール内での役割

src/backend/optimizer/path モジュールにおいて、本ファイルは「順序の抽象化」を担当します。他のファイル（joinpath.c や indxpath.c など）が特定のアルゴリズムでパスを生成する際、そのパスがどのような順序でデータを出力するかをパスキーとして表現し、プランナが後続の処理でその順序を利用できるようにします。

責務

  • **パスキーの正規化**: EquivalenceClass（等価クラス）に基づき、意味的に同一な順序を同一のパスキーとして管理する。
  • **冗長性の排除**: 既に存在する順序と重複するパスキーを検出し、計画の探索空間を効率化する。
  • **有用性の判定**: 特定のパスキーが、マージ結合やクエリの最終出力順序として「有用」かどうかを判定し、不要なパスキーを切り捨てる（truncate_useless_pathkeys）。
  • **変換**: インデックスやパーティションの定義から、そのスキャンが提供する順序をパスキーとして導出する。

重要なシンボル

  • PathKey: 順序を表現する主要な構造体。EquivalenceClass への参照を保持する。
  • make_canonical_pathkey: 正規化されたパスキーを生成・取得する。
  • pathkey_is_redundant: 与えられたパスキーが既存のリストに対して冗長かどうかを判定する。
  • truncate_useless_pathkeys: クエリの実行計画において、これ以上利用価値のないパスキーをリストから削除する。
  • compare_pathkeys: 2 つのパスキーリストを比較し、包含関係や優劣を判定する。

制御フローとデータフロー

1. **構築**: プランナがインデックススキャンやソートノードを検討する際、make_pathkey_from_sortinfo 等を呼び出し、EquivalenceClass を介して正規化された PathKey を作成します。 2. **検証**: append_pathkeys や pathkey_is_redundant を通じて、リストに追加する前に冗長性をチェックします。 3. **最適化**: truncate_useless_pathkeys が、マージ結合の可能性や ORDER BY への適合性を評価し、不要なパスキーを削ぎ落とすことで、プランナが保持するパスの数を抑制します。 4. **比較**: 結合処理において、compare_pathkeys や pathkeys_contained_in を用いて、入力パスの順序が結合条件を満たしているかを確認します。

読者向けメモ

  • **正規化の重要性**: 本ファイルで扱うパスキーはすべて「正規化（Canonical）」されていることが前提です。これにより、複雑な式比較をポインタ比較（==）に置き換えて高速に処理しています。
  • **冗長性の定義**: 単なる重複だけでなく、定数との等価性や、既に存在する順序による包含関係も冗長とみなされます。
  • **デバッグ**: pathkey_is_redundant のロジックを理解することが、プランナがなぜ特定のソート順序を無視するのかを解明する鍵となります。
