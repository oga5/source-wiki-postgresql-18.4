src/backend/optimizer/plan/setrefs.c

概要

setrefs.c は、プランナの最終フェーズを担当するモジュールです。論理的なプランツリーを、実行エンジン（Executor）が直接解釈可能な物理的な形式へ変換します。このファイルは、プランニングの「後処理」の核心であり、変数参照の解決、演算子の関数解決、プラン依存関係の追跡などを行います。

モジュール内での役割

src/backend/optimizer/plan サブシステムにおいて、createplan.c が Path から Plan ノードを生成した直後に呼び出されます。プランツリー全体を走査し、実行時に必要なメタデータを付与・正規化することで、プランナからエグゼキュータへの橋渡しを行います。

責務

  • **参照解決**: Var ノードの varno を、フラット化されたレンジテーブル（finalrtable）上のインデックスへ書き換える。
  • **プランの正規化**: 不要な SubqueryScan や Append ノードの削除、AlternativeSubPlan の最適化。
  • **依存関係の追跡**: プランが依存するリレーション、関数、ドメインを抽出し、プランキャッシュの無効化メカニズム（PlanInvalItem）を構築する。
  • **演算子解決**: OpExpr 等の演算子に対し、実行時に呼び出す関数（regproc）を特定する。

重要なシンボル

プラン変換・走査

  • set_plan_references: 本ファイルのメインエントリーポイント。プランツリー全体を再帰的に走査し、参照解決と最適化を行う。
  • set_plan_refs: 各プランノードの型に応じた具体的な変換処理をディスパッチする。
  • fix_scan_expr / fix_join_expr / fix_upper_expr: 式ツリー内の Var を、スキャン、結合、上位ノードの出力形式に合わせて書き換えるためのミューテータ群。

データ構造とインデックス

  • indexed_tlist: ターゲットリストを高速検索するためのインデックス構造体。build_tlist_index で生成される。
  • tlist_vinfo: indexed_tlist 内で Var の位置情報を保持する。
  • NullingRelsMatch: 外部結合等で発生する varnullingrels の一致条件（完全一致、サブセット、スーパーセット）を定義する列挙型。

依存関係管理

  • record_plan_function_dependency: 特定の関数への依存を PlannerGlobal に記録する。
  • extract_query_dependencies: プランニング前のクエリから依存関係を抽出するユーティリティ。

制御フローとデータフロー

1. **初期化**: set_plan_references が呼び出され、add_rtes_to_flat_rtable によりクエリ内の全レンジテーブルエントリをフラット化する。 2. **再帰的変換**: set_plan_refs がプランツリーをトップダウンで走査。

  • 各ノードの targetlist や qual は、fix_scan_expr 等のミューテータを介して、現在のプランレベルのコンテキスト（rtoffset）に合わせて書き換えられる。
  • 結合ノードでは fix_join_expr が outer_itlist / inner_itlist を使用して Var を OUTER_VAR / INNER_VAR に変換する。

    • 3. **最適化**: trivial_subqueryscan 等の判定により、不要なノードを clean_up_removed_plan_level で除去する。 4. **最終化**: 依存関係（relationOids, invalItems）が PlannerGlobal に集約され、実行可能な PlannedStmt の準備が整う。

読者向けメモ

  • **メモリ管理**: set_plan_references はプランノードをインプレースで修正しますが、式ツリー（targetlist 等）は expression_tree_mutator を用いてコピー・修正されます。これはプランノードが単一の参照であることを前提としています。
  • **インデックスのオフセット**: rtoffset は、サブクエリのレンジテーブルをフラットなレンジテーブルにマージする際に、インデックスの衝突を避けるために使用されます。
  • **省略された詳細**: fix_scan_expr_walker 等の補助関数は、特定の式ノードに対する詳細な走査ロジックを含みますが、基本的には expression_tree_walker のラッパーです。
  • **注意点**: AlternativeSubPlan の解決は、プランの実行回数（num_exec）の推定値に基づいて行われます。この推定はヒューリスティックであり、厳密なコスト計算ではないことに注意してください。
