src/backend/optimizer/plan/createplan.c

概要

createplan.c は、PostgreSQL のプランナにおいて、コストベースの最適化によって選択された Path ツリーを、実行エンジン（Executor）が処理可能な Plan ノードツリーへ変換する役割を担います。プランニングプロセスの最終段階を統括する中心的なファイルです。

モジュール内での役割

src/backend/optimizer/plan サブシステムにおいて、論理的なパス探索（planmain.c 等）と、最終的な変数参照解決（setrefs.c）の間に位置します。Path 構造体に含まれるメタデータを物理的な実行ノードの属性にマッピングし、再帰的にツリーを構築します。

責務

  • **再帰的変換**: create_plan_recurse を中心に、Path ツリーを Plan ツリーへ再帰的に変換する。
  • **ターゲットリスト（tlist）の構築**: 各プランノードが返す列を決定し、必要に応じて射影（Projection）ノードを挿入する。
  • **制約の物理化**: RestrictInfo を実行可能なクエリ条件（qual）へ変換し、適切なプランノードへ配置する。
  • **特殊ノードの生成**: 結合（Join）、集約（Agg）、ソート（Sort）、スキャン（Scan）など、各物理操作に対応するノードを生成する。
  • **パラメータの解決**: 相関サブクエリやネストループ結合で使用されるパラメータ（NestLoopParam）の割り当てと解決。

重要なシンボル

エントリーポイント

  • create_plan: プランニングの物理変換フェーズのメインエントリーポイント。

再帰変換エンジン

  • create_plan_recurse: Path のノードタイプに応じて適切な create_xxx_plan 関数へディスパッチする。

プランノード生成ファミリ

  • create_scan_plan: 各種スキャン（Seq, Index, Bitmap 等）の生成を統括。
  • create_join_plan: 結合（NestLoop, MergeJoin, HashJoin）の生成を統括。
  • create_append_plan / create_merge_append_plan: 集合操作やパーティションスキャンのためのノード生成。
  • make_xxx (多数): 各物理プランノード（SeqScan, HashJoin, Sort 等）の構造体を初期化する低レベル関数群。

ユーティリティ

  • build_path_tlist: PathTarget から実行用のターゲットリストを構築する。
  • replace_nestloop_params: 式に含まれる Var をネストループ結合用の Param に置換する。
  • order_qual_clauses: クエリ条件をコスト順に並び替え、実行効率を最適化する。

制御フローとデータフロー

1. **初期化**: create_plan が PlannerInfo を初期化し、再帰を開始する。 2. **ディスパッチ**: create_plan_recurse が Path の pathtype を見て、対応する create_xxx_plan を呼び出す。 3. **射影とtlist**: 各ノード生成関数は build_path_tlist を呼び出し、必要な列を特定する。必要に応じて inject_projection_plan で Result ノードを挿入し、射影を行う。 4. **制約の適用**: order_qual_clauses で条件を並び替え、extract_actual_clauses で RestrictInfo を実行可能な式リストに変換する。 5. **パラメータ置換**: replace_nestloop_params がツリーを走査し、外側のクエリレベルから渡されるパラメータを解決する。

読者向けメモ

  • **メモリ管理**: プランニング中に生成されるノードは PlannerInfo の planner_cxt メモリコンテキストに割り当てられます。
  • **不変条件**: create_plan 完了後、setrefs.c が呼ばれるまでは、Plan ノード内の Var はパーサーレベルのインデックスを保持しています。
  • **射影の最適化**: use_physical_tlist は、不要な射影ノードを避けるための重要な最適化です。特定の条件下で、物理的なテーブル構造と一致するターゲットリストを直接使用します。
  • **未実装/浅い部分**: NOT_USED マクロで囲まれた古いコードや、特定の拡張機能（FDW/CustomScan）のプラン生成は、各プラグインの責務に依存します。
