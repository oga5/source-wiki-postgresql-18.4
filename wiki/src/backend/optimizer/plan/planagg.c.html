src/backend/optimizer/plan/planagg.c

概要

planagg.c は、PostgreSQL のオプティマイザにおいて MIN および MAX 集約関数を最適化するためのモジュールです。インデックスを利用可能な場合に、全行スキャンを伴う集約処理を、LIMIT 1 を伴うサブクエリ（インデックススキャン）へ置換することで、クエリ実行速度を劇的に向上させます。

モジュール内での役割

src/backend/optimizer/plan サブシステムの一部として、grouping_planner が query_planner を呼び出す直前の前処理フェーズで機能します。標準的な集約パスと競合する MinMaxAggPath を生成し、プランナがより効率的な実行計画を選択できるようにします。

責務

  • **最適化可能性の判定**: クエリ内の集約関数が MIN/MAX のみで構成され、かつインデックススキャンへ置換可能か検証します。
  • **サブクエリの構築**: インデックスを利用して最小値・最大値を効率的に取得するためのサブクエリ（SELECT col FROM tab WHERE col IS NOT NULL ORDER BY col ASC/DESC LIMIT 1）を生成します。
  • **パスの生成**: 生成したサブクエリを MinMaxAggPath としてパッケージ化し、UPPERREL_GROUP_AGG に登録します。

重要なシンボル

  • preprocess_minmax_aggregates: 本モジュールのエントリポイント。集約関数の最適化可否を判断し、パス生成を統括します。
  • can_minmax_aggs: クエリ内の集約関数を走査し、すべてが MIN/MAX であり、かつインデックス利用可能かを確認します。
  • build_minmax_path: 特定の集約関数に対して、インデックスを利用したサブクエリパスを構築します。
  • MinMaxAggInfo: 最適化対象の集約関数に関するメタデータ（対象式、ソート演算子、生成されたパス等）を保持する構造体。

追加の索引済みシンボル

制御フローとデータフロー

1. **検証**: preprocess_minmax_aggregates がクエリ構造（単一テーブル、GROUP BY や WINDOW 関数の不在など）をチェックします。 2. **解析**: can_minmax_aggs が root->agginfos を辿り、最適化対象の集約関数を抽出します。 3. **パス構築**: build_minmax_path が現在の PlannerInfo をクローンし、サブクエリ用の PlannerInfo を作成します。ここで query_planner を再帰的に呼び出し、LIMIT 1 を適用した最適なインデックススキャンパスを探索します。 4. **登録**: 成功した場合、MinMaxAggPath を作成し、UPPERREL_GROUP_AGG に追加します。

読者向けメモ

  • **制約**: 本最適化は、クエリが単一のテーブルを参照している場合にのみ適用されます。結合や複雑な GROUP BY が含まれる場合はスキップされます。
  • **副作用**: build_minmax_path 内で PlannerInfo をクローンしてサブクエリを構築するため、メモリ消費や複雑な状態管理が発生します。
  • **デバッグ**: MinMaxAggPath が生成されない場合は、can_minmax_aggs 内の条件（aggfilter の有無や mutable 関数の存在など）が原因であることが多いです。
