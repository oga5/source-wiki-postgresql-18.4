src/backend/optimizer/prep/prepunion.c

概要

prepunion.c は、SQL の集合演算（UNION, INTERSECT, EXCEPT）をプランニングするための中心的なモジュールです。クエリ内の集合演算ツリーを走査し、各ノードに対して適切な実行パス（Append, MergeAppend, SetOp, RecursiveUnion など）を生成します。

モジュール内での役割

src/backend/optimizer/prep モジュールの一部として、クエリの論理的な集合演算構造を物理的な実行プランへと変換する役割を担います。planner() が呼び出す grouping_planner からの依頼を受け、集合演算のサブクエリを個別にプランニングし、それらを結合するパスを構築します。

責務

  • **集合演算の再帰的プランニング**: SetOperationStmt ツリーを再帰的に辿り、各階層で最適なパスを決定する。
  • **パスの生成**: 集合演算の種類に応じて、Append（UNION ALL）、SetOp（INTERSECT/EXCEPT）、RecursiveUnion（再帰的 UNION）のパスを生成する。
  • **ターゲットリストの管理**: 集合演算の各入力間でデータ型や照合順序（Collation）を整合させるためのターゲットリスト（tlist）を生成・調整する。
  • **子ノードの最適化**: 集合演算の子となるサブクエリに対して、親の要求に応じたソート順やグループ化の情報を伝播させる。

重要なシンボル

  • plan_set_operations: 本ファイルのメインエントリポイント。集合演算ツリーのプランニングを開始する。
  • recurse_set_operations: 集合演算ツリーを再帰的に処理する中核関数。
  • generate_union_paths: UNION / UNION ALL のパス生成を担当。
  • generate_nonunion_paths: INTERSECT / EXCEPT のパス生成を担当。
  • build_setop_child_paths: 子サブクエリに対して、集合演算に有用なパス（ソート済みパスなど）を構築する。

制御フローとデータフロー

1. plan_set_operations が root->parse->setOperations を取得し、再帰処理を開始。 2. recurse_set_operations がツリーを辿り、末端の RangeTblRef（サブクエリ）に到達すると subquery_planner を呼び出して子クエリをプランニング。 3. 各集合演算ノードで generate_union_paths または generate_nonunion_paths を呼び出し、子ノードから返されたパスを結合。 4. 最終的に RelOptInfo（UPPERREL_SETOP）にパスが追加され、postprocess_setop_rel でコスト計算と最適パスの選択が行われる。

読者向けメモ

  • 本ファイルは「集合演算」という名前ですが、再帰的 UNION もここで処理されます。
  • varno 0 を持つ Var ノードが生成されることがありますが、これは集合演算の出力列を指すための特殊な慣習です。
  • trivial_tlist フラグは、入力と出力の型が完全に一致しており、追加の変換処理が不要であることを示す重要な最適化ヒントです。
  • 複雑な集合演算のネストはスタックオーバーフローを引き起こす可能性があるため、check_stack_depth() が随所で呼び出されています。
