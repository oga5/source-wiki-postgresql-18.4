src/bin/scripts/vacuumdb.c

概要

vacuumdb は、PostgreSQL データベースのメンテナンス（VACUUM および ANALYZE）を自動化するためのコマンドラインユーティリティです。データベース全体、特定のデータベース、または特定のテーブルやスキーマに対して、並列処理や統計情報の段階的更新などの高度なオプションを提供します。

モジュール内での役割

src/bin/scripts モジュールの一部として、サーバーサイドの VACUUM SQL コマンドをクライアント側から制御する役割を担います。common.c で定義された共通の接続管理やオプション解析機能を活用し、ユーザーの意図を適切な SQL 文に変換して実行します。

責務

  • ユーザー指定のオプション（vacuumingOptions）に基づいた VACUUM/ANALYZE コマンドの構築。
  • ターゲットとなるオブジェクト（テーブル等）のカタログからの動的取得。
  • 並列実行（--jobs）の管理と、サーバーバージョンに応じた機能のバリデーション。
  • 統計情報の段階的更新（--analyze-in-stages）の制御。

重要なシンボル

  • vacuumingOptions: ユーザーが指定した実行オプションを保持する構造体。
  • VacObjFilter: 処理対象を絞り込むためのフィルタ種別（DB、テーブル、スキーマ等）。
  • vacuum_one_database(): 特定のデータベースに対して VACUUM を実行する中心的な関数。
  • retrieve_objects(): カタログをクエリし、処理対象のテーブルリストを生成する関数。
  • prepare_vacuum_command(): サーバーバージョンを考慮して適切な SQL 文を生成する関数。

制御フローとデータフロー

1. **初期化**: main 関数でオプションを解析し、vacuumingOptions を設定。 2. **フィルタ検証**: check_objfilter() で矛盾するオプション指定がないか確認。 3. **対象取得**: retrieve_objects() がカタログをクエリし、対象テーブルのリストを作成。 4. **実行**:

  • 単一 DB の場合: vacuum_one_database() を呼び出し、必要に応じて並列スロット（ParallelSlot）を使用して実行。
  • 全 DB の場合: vacuum_all_databases() が各 DB に対してループ処理を実行。

    • 5. **コマンド生成**: prepare_vacuum_command() が SQL を構築し、run_vacuum_command() が PQsendQuery を通じてサーバーへ送信。

読者向けメモ

  • **バージョン互換性**: 各オプション（--skip-locked や --parallel 等）はサーバーバージョンに依存しており、vacuum_one_database 内で PQserverVersion を用いた厳格なチェックが行われています。
  • **並列処理**: ParallelSlots を使用して複数の接続を管理しています。concurrentCons が指定された場合、テーブルリストをキューのように消費します。
  • **セキュリティ**: カタログクエリは ALWAYS_SECURE_SEARCH_PATH_SQL を使用し、SQL インジェクションや検索パスの乗っ取りを防ぐ設計になっています。
