src/bin/pg_amcheck/pg_amcheck.c

概要

pg_amcheck は、PostgreSQL のヒープテーブルおよび B-tree 索引の破損を検出するコマンドラインツールです。contrib/amcheck 拡張機能が提供する検証関数を、指定されたデータベース・スキーマ・リレーションに対して並列実行します。本ファイルは、プログラムの引数解析、対象リレーションの特定（SQL クエリ生成）、並列実行の制御、および結果の集約を単一で担います。

モジュール内での役割

本ツールは fe_utils ライブラリに強く依存しており、PostgreSQL フロントエンドの標準的な並列処理パターン（parallel_slot.h）を実装しています。システムカタログをクエリして検証対象を動的に決定し、各リレーションに対して適切な検証関数（verify_heapam や bt_index_check）を呼び出す SQL を構築します。

責務

  • **引数解析と設定**: ユーザー指定のパターン（データベース、スキーマ、リレーション）を解析し、AmcheckOptions に格納。
  • **対象の特定**: SQL クエリ（CTE を多用）を構築し、検証対象となるリレーションのリストをデータベースごとに生成。
  • **並列実行管理**: ParallelSlot を用いて、複数のデータベース接続を管理し、検証 SQL を非同期に発行。
  • **結果処理**: 検証結果（破損メッセージやエラー）をハンドラ関数で受け取り、標準出力へ整形して出力。

重要なシンボル

データ構造

  • AmcheckOptions: 実行オプションとパターンリストを保持するグローバル設定構造体。
  • PatternInfo: ユーザー指定の検索パターン（正規表現）とマッチング状態を管理。
  • RelationInfo: 検証対象リレーションのメタデータ（OID、型、所属 DB 情報）を保持。
  • DatabaseInfo: 接続先データベースの接続情報と amcheck スキーマ情報を保持。

主要関数

  • main: プログラムのライフサイクル管理、接続確立、並列実行ループの制御。
  • compile_database_list / compile_relation_list_one_db: システムカタログをクエリし、検証対象のリストを構築する。
  • prepare_heap_command / prepare_btree_command: 各リレーションに応じた検証 SQL を構築。
  • verify_heap_slot_handler / verify_btree_slot_handler: 非同期実行の結果を受け取り、破損情報を出力するコールバック。

制御フローとデータフロー

1. **初期化**: getopt_long で引数を解析し、PatternInfoArray を構築。 2. **対象特定**: compile_database_list が接続可能なデータベースを特定し、続いて compile_relation_list_one_db が各 DB 内の対象リレーションを特定。 3. **並列実行**: ParallelSlotsSetup でスロットを初期化。メインループで ParallelSlotsGetIdle を呼び出し、空きスロットに prepare_*_command で生成した SQL を PQsendQuery で発行。 4. **結果処理**: ParallelSlot のハンドラが PGresult を解析。破損が検出された場合、all_checks_pass を false に更新し、詳細を標準出力へ書き出す。 5. **終了**: 全スロットの完了を待ち、破損があれば終了コード 2 を、エラーがあれば 1 を返却。

直接依存

  • fe_utils/: 並列処理（parallel_slot.h）、SQL 実行（query_utils.h）、引数解析（option_utils.h）等のユーティリティ。
  • catalog/: pg_am_d.h, pg_class_d.h 等のシステムカタログ定義。
  • libpq: データベース接続およびクエリ実行。

関連コード

  • contrib/amcheck: 本ツールが呼び出すサーバー側の検証関数（verify_heapam, bt_index_check 等）を定義。
  • src/bin/scripts/: 他の PostgreSQL 管理ツールにおける並列処理や引数解析の参考実装。

読者向けメモ

  • **メモリ管理**: RelationInfo や PatternInfo 内の文字列は pstrdup で確保され、ハンドラ関数内で FREE_AND_SET_NULL マクロを用いて適切に解放されます。
  • **並列実行の制約**: ParallelSlots は接続を再利用しますが、データベースを跨ぐ場合は接続の切り替えが発生します。
  • **エラーハンドリング**: should_processing_continue は、サーバーからのエラーが致命的（FATAL/PANIC）かどうかを判定し、処理を継続すべきかを決定します。
  • **省略された詳細**: 本ファイルは SQL クエリの構築に非常に多くの行数を割いていますが、これは複雑なパターンマッチングと除外条件を SQL 側で解決するためです。各 CTE の詳細なロジックは、compile_relation_list_one_db 内の PQExpBuffer 構築部分を参照してください。
