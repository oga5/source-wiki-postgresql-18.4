src/bin/psql/tab-complete.in.c

概要

tab-complete.in.c は、psql の対話型シェルにおけるタブ補完機能を実装するソースファイルです。ユーザーが入力した SQL コマンドやメタコマンドの文脈を解析し、データベースのカタログ情報（テーブル名、列名、関数名など）や SQL キーワードを動的に提案します。

このファイルは非常に巨大ですが、これは SQL の広範な構文を網羅的に補完するためです。また、gen_tabcomplete.pl スクリプトによって前処理されることを前提としており、複雑な if-else チェーンを switch 文に変換することでコンパイル時の負荷を軽減する仕組みを持っています。

モジュール内での役割

src/bin/psql モジュールにおいて、ユーザーの入力体験（UX）を向上させるためのフロントエンド層に位置します。input.c や mainloop.c と密接に連携し、readline ライブラリの補完フックとして動作します。

責務

  • **文脈解析**: 現在の入力行を単語単位に分割し、直前の単語に基づいて補完候補を決定する。
  • **クエリ生成**: 補完候補を取得するために、システムカタログ（pg_class, pg_proc 等）に対する SQL クエリを動的に構築・実行する。
  • **補完ディスパッチ**: SQL コマンド、メタコマンド、ファイル名、変数名など、入力内容に応じた適切な補完ロジックの選択。
  • **クエリ結果の整形**: データベースから取得した候補を、SQL 識別子の引用符ルールに従って整形する。

重要なシンボル

データ構造

  • VersionedQuery: サーバーのバージョンに応じてクエリを切り替えるための構造体。
  • SchemaQuery: スキーマ修飾されたオブジェクト名を取得するためのクエリ構成要素を保持する構造体。
  • pgsql_thing_t: CREATE/DROP コマンドの後に続くオブジェクトの種類を定義する構造体。
  • TCPattern: gen_tabcomplete.pl が生成する、補完ルールを定義する構造体。

補完ディスパッチ用マクロ

  • COMPLETE_WITH_QUERY 系: SQL クエリ結果を補完候補とする。
  • COMPLETE_WITH_SCHEMA_QUERY 系: スキーマ情報を考慮したオブジェクト名補完を行う。
  • COMPLETE_WITH_LIST 系: 静的なキーワードリストを補完候補とする。

コールバック関数

  • psql_completion: readline から呼び出されるメインの補完エントリポイント。
  • match_previous_words: 直前の単語に基づいて補完ルールをマッチングする。

制御フローとデータフロー

1. **トリガー**: ユーザーが TAB キーを押すと、readline が psql_completion を呼び出す。 2. **解析**: get_previous_words が現在の入力行を単語リストに分解する。 3. **マッチング**: match_previous_words が現在の単語リストと定義済みのパターン（tcpatterns）を照合する。 4. **候補取得**: マッチしたパターンに応じて、_complete_from_query がデータベースへクエリを送信し、結果を取得する。 5. **整形**: requote_identifier 等を用いて、SQL 識別子として適切な形式に変換し、readline に返す。

読者向けメモ

  • **メモリ管理**: _complete_from_query 等で生成される文字列は readline が解放するため、malloc されたメモリの所有権には注意が必要です。
  • **副作用**: タブ補完はデータベースへのクエリ実行を伴います。補完候補の取得中にサーバーが重いクエリを処理すると、補完が遅延する可能性があります。
  • **設計の制約**: readline の仕様上、補完関数に状態を渡すことが困難なため、多くのグローバル変数（completion_charp 等）が使用されています。これはスレッドセーフではないため、将来的な並列化には注意が必要です。
  • **未実装部分**: 複雑な SQL 構文（サブクエリや複雑な結合）の完全な解析は行っておらず、あくまで「よく使われるパターン」の補完に特化しています。
