src/bin/pg_upgrade/parallel.c

概要

parallel.c は、pg_upgrade における並列処理（マルチプロセス/マルチスレッド）を管理するモジュールです。データベースのアップグレード処理において、コマンド実行やデータ転送を複数のジョブで並行して行うための抽象化レイヤーを提供します。

モジュール内での役割

pg_upgrade のメイン処理から呼び出され、OS 依存の並列化メカニズム（Unix 系では fork、Windows では _beginthreadex）を隠蔽します。これにより、上位のロジックは並列実行の複雑さを意識せずに、ジョブの投入と完了待ちを行えます。

責務

  • **並列ジョブの管理**: user_opts.jobs に基づき、実行中のジョブ数を追跡し、上限に達した場合は完了待ち（reap）を行います。
  • **OS 抽象化**: Unix と Windows で異なるプロセス/スレッド管理手法を吸収し、統一的な API を提供します。
  • **リソースのクリーンアップ**: 終了した子プロセスやスレッドのハンドルを適切に回収し、リソースリークを防ぎます。

重要なシンボル

  • parallel_exec_prog: 外部コマンドを並列実行するための関数。
  • parallel_transfer_all_new_dbs: テーブルスペースのデータ転送を並列化するための関数。
  • reap_child: 完了した子プロセス/スレッドを回収し、状態を更新する関数。
  • parallel_jobs: 現在実行中の並列ジョブ数を保持するグローバル変数。

追加の索引済みシンボル

  • exec_thread_arg
  • transfer_thread_arg
  • win32_exec_prog
  • win32_transfer_all_new_dbs

制御フローとデータフロー

1. **ジョブ投入**: parallel_exec_prog 等が呼ばれると、まず reap_child(false) を呼び出し、完了済みのジョブがあれば回収します。 2. **制限チェック**: 現在の parallel_jobs が user_opts.jobs に達していれば、reap_child(true) を呼び出し、いずれかのジョブが完了するまでブロックします。 3. **実行**:

  • Unix 系: fork() を行い、子プロセスで exec_prog 等を実行します。
  • Windows 系: _beginthreadex を使用し、引数構造体（exec_thread_arg 等）を介して関数を起動します。

    • 4. **回収**: reap_child は、終了したジョブのステータスを確認し、異常終了時は pg_fatal で停止します。Windows ではハンドルを閉じ、配列を詰め直して次のジョブに備えます。

直接依存

  • postgres_fe.h: PostgreSQL フロントエンド用共通ヘッダー。
  • sys/wait.h: プロセス終了待機用（Unix）。
  • io.h: Windows 用のファイル/スレッド関連。
  • pg_upgrade.h: モジュール共通定義。

関連コード

  • src/bin/pg_upgrade/exec.c: 並列化の対象となる単一ジョブの実行ロジックが含まれています。
  • src/bin/pg_dump/parallel.h: 概念的に類似した並列処理の設計ですが、pg_upgrade とは実装が異なります。

読者向けメモ

  • **Windows 特有の制約**: Windows では WaitForMultipleObjects の制限により、最大ジョブ数は MAXIMUM_WAIT_OBJECTS（通常 64）に制限されます。
  • **エラーハンドリング**: 並列ジョブのいずれかが失敗した場合、即座に pg_fatal が呼ばれ、プロセス全体が停止する設計になっています。
  • **スレッド安全性**: Windows 実装では、スレッド間で引数構造体を安全に受け渡すために、配列のポインタ操作とスロット管理が行われています。コードを読む際は、#ifdef WIN32 で分岐している箇所に注目してください。
