src/bin/pg_upgrade/check.c

概要

check.c は、pg_upgrade におけるアップグレード前後のクラスタ互換性検証の司令塔です。新旧クラスタのメタデータ、データ型、システムカタログ、およびレプリケーション設定が、移行先バージョンで正しく動作するかを網羅的にチェックします。

モジュール内での役割

pg_upgrade のメインフローにおいて、データ移行（relfilenumber.c）やスキーマダンプ（dump.c）を実行する前の「ゲートキーパー」として機能します。検証に失敗した場合、アップグレードを即座に中断し、ユーザーに修正手順を提示します。

責務

  • **データ型互換性検証**: バージョン間で内部表現が変更されたデータ型（jsonb, aclitem 等）の使用状況を調査。
  • **システムカタログ整合性チェック**: 廃止された機能（WITH OIDS、postfix 演算子等）や、移行不可なオブジェクトの検出。
  • **レプリケーション状態の検証**: 論理レプリケーションスロットやサブスクリプションの移行可能性を判定。
  • **環境整合性チェック**: 接続設定、インストールユーザーの権限、テーブルスペースのパス競合などを検証。
  • **レポート生成**: 検出された問題をファイルに出力し、ユーザーへ修正を促す。

重要なシンボル

データ型検証用構造体

  • DataTypesUsageChecks: データ型ごとのチェックルール（SQLクエリ、エラーメッセージ、適用バージョン閾値）を定義した構造体。
  • data_types_usage_checks: 上記構造体の静的配列。各データ型の互換性チェックロジックがここに集約されています。

検証実行関数

  • check_and_dump_old_cluster: 旧クラスタに対する一連の検証を実行し、ダンプ準備を行うメイン関数。
  • check_new_cluster: 新クラスタの空き状態やパスの競合を検証する関数。
  • check_cluster_versions: メジャーバージョンの互換性（ダウングレード禁止等）を検証。

ユーティリティ・コールバック

  • data_type_check_query: DataTypesUsageChecks から動的に SQL を生成するヘルパー。
  • process_data_type_check: 検証クエリの結果を解析し、エラーがあればレポートファイルへ書き出すコールバック。

制御フローとデータフロー

1. **タスクの登録**: check_for_data_types_usage 等が upgrade_task フレームワークを使用して、各データベースに対する検証クエリを登録します。 2. **並列実行**: upgrade_task_run が各データベースに接続し、登録された SQL を実行します。 3. **結果の集約**: 各検証関数（process_...）が結果を PQExpBuffer やファイルに書き込み、エラーフラグを立てます。 4. **中断判定**: 検証終了後、エラーフラグが立っていれば pg_fatal を呼び出し、プロセスを終了させます。

直接依存

  • pg_upgrade.h: クラスタ情報構造体（ClusterInfo）やタスク管理基盤の定義。
  • catalog/pg_authid_d.h, catalog/pg_class_d.h: システムカタログの定数。
  • fe_utils/string_utils.h: 文字列操作ユーティリティ。

関連コード

  • task.c: upgrade_task フレームワークの実装。検証クエリの並列実行を支える基盤。
  • info.c: ClusterInfo や DbInfo 等のメタデータ収集ロジック。
  • controldata.c: pg_control ファイルの検証ロジック。

読者向けメモ

  • **メモリ管理**: upgrade_task で使用されるクエリ文字列や状態構造体は、タスク終了時に upgrade_task_free で解放されます。
  • **エラーパス**: 多くのチェック関数は、エラーを即座に終了させるのではなく、レポートファイルを作成して最後にまとめて報告する設計になっています。
  • **設計の意図**: data_types_usage_checks 配列は、新しいデータ型の互換性問題が発生した際に、SQL クエリを追加するだけで容易に拡張できるように設計されています。
  • **省略された詳細**: 本ファイル内の各 check_for_... 関数は、特定の PostgreSQL バージョン固有の仕様に深く依存しています。これらは pg_upgrade のバージョンアップに伴い、適宜 threshold_version を調整する必要があります。
