src/port/pg_numa.c

概要

pg_numa.c は、PostgreSQL における NUMA (Non-Uniform Memory Access) アーキテクチャ対応の移植性レイヤーを提供します。主に Linux 環境において、メモリページがどの NUMA ノードに配置されているかを効率的に問い合わせるためのインターフェースを定義します。

モジュール内での役割

src/port ディレクトリの一部として、プラットフォーム固有のシステムコールを抽象化し、PostgreSQL のコアエンジンが OS の詳細を意識せずに NUMA 情報を扱えるようにします。現在は Linux の libnuma に依存していますが、将来的な他 OS への拡張を考慮した設計になっています。

責務

  • **NUMA サブシステムの初期化**: libnuma が利用可能か、および現在の環境で NUMA 制御が許可されているかを判定します。
  • **メモリ配置の問い合わせ**: move_pages(2) システムコールをラップし、指定されたメモリページがどのノードにあるかをバッチ処理で取得します。
  • **カーネルバグの回避**: 特定のカーネルバージョンにおける do_pages_stat() のバグを回避するため、問い合わせをチャンク単位に分割します。

重要なシンボル

  • pg_numa_init(): NUMA 機能の利用可否をチェックします。
  • pg_numa_query_pages(): メモリページ配列のノード位置を問い合わせる主要関数です。
  • pg_numa_get_max_node(): システム上の最大ノード番号を返します。
  • NUMA_QUERY_CHUNK_SIZE: カーネルバグ回避のための処理単位（32bit環境では16、それ以外は1024）。

追加の索引済みシンボル

  • pg_numa_get_max_node
  • pg_numa_init
  • pg_numa_query_pages

制御フローとデータフロー

1. **初期化**: pg_numa_init が get_mempolicy を呼び出し、権限チェックと numa_available を通じて利用可否を返します。 2. **問い合わせ**: pg_numa_query_pages が呼び出されると、指定されたページ配列を NUMA_QUERY_CHUNK_SIZE ごとに分割してループ処理します。 3. **システムコール**: 各チャンクに対して numa_move_pages を実行し、結果を status 配列に格納します。 4. **割り込み**: バックエンドプロセスの場合、チャンク処理の合間に CHECK_FOR_INTERRUPTS() を実行し、クエリのキャンセル要求に応答します。

直接依存

  • c.h: PostgreSQL の基本型定義。
  • unistd.h: 標準システムコール定義。
  • miscadmin.h: CHECK_FOR_INTERRUPTS 等のバックエンド管理用マクロ。
  • numa.h / numaif.h: Linux libnuma ライブラリ（USE_LIBNUMA 定義時）。
  • port/pg_numa.h

関連コード

  • src/include/port/pg_numa.h: 公開ヘッダー。pg_numa_touch_mem_if_required インライン関数が含まれており、ページフォールトを強制して正確なノード情報を取得するために使用されます。

読者向けメモ

  • **Linux 依存**: 現在、このファイルの実装は libnuma に強く依存しています。USE_LIBNUMA が定義されていない環境では、すべての関数がダミー（または無効）として動作します。
  • **パフォーマンス**: numa_move_pages を使用しているのは、get_mempolicy よりもバッチ処理によるオーバーヘッドが小さいためです。
  • **カーネルバグ**: NUMA_QUERY_CHUNK_SIZE の制限は、古いカーネルにおける do_pages_stat の不具合を回避するための重要なワークアラウンドです。この値を変更する際は、カーネルの制限事項を再確認してください。
