src/port/dirmod.c

概要

dirmod.c は、Windows 環境におけるディレクトリおよびファイル操作の互換性レイヤーを提供します。POSIX 標準のファイル操作関数が Windows のファイルシステム特性（特に共有違反やジャンクションポイント）と衝突する問題を解決し、PostgreSQL がクロスプラットフォームで安定して動作するように設計されています。

モジュール内での役割

src/port サブシステムの一部として、OS 固有の差異を吸収する役割を担います。主に Windows 上での rename や unlink の挙動をラップし、他のモジュールがプラットフォームを意識せずにファイル操作を行えるようにします。

責務

  • **ファイル操作の堅牢化**: rename や unlink 実行時に発生する共有違反（Sharing Violation）に対して、短時間の再試行（リトライ）ロジックを提供します。
  • **Windows ジャンクションポイントの管理**: Windows におけるシンボリックリンクの代替としてジャンクションポイントを操作する pgsymlink および pgreadlink を実装します。
  • **エラー変換**: Windows API のエラーコードを POSIX 準拠の errno にマッピングします。

重要なシンボル

  • pgrename: Windows の MoveFileEx を使用し、共有違反時にリトライを行う rename のラッパー。
  • pgunlink: unlink または rmdir を適切に選択し、リトライを行う関数。
  • pgsymlink: ジャンクションポイントを作成するための関数。
  • pgreadlink: ジャンクションポイントのターゲットパスを読み取る関数。
  • REPARSE_JUNCTION_DATA_BUFFER: ジャンクションポイントのメタデータを扱うための構造体。

追加の索引済みシンボル

  • lstat_error_was_status_delete_pending

制御フローとデータフロー

1. **リトライループ**: pgrename や pgunlink は、Windows 特有の ERROR_SHARING_VIOLATION や ERROR_LOCK_VIOLATION を検知すると、pg_usleep を用いて最大 10 秒間リトライを試みます。 2. **ジャンクション操作**: pgsymlink は、ディレクトリを作成した後に DeviceIoControl を呼び出し、FSCTL_SET_REPARSE_POINT を用いてリパースポイントを設定します。 3. **パス変換**: pgreadlink は、取得したパスから Windows 内部形式（\??\ プレフィックス）を取り除き、ユーザーが期待する形式へ変換して返します。

直接依存

  • unistd.h, sys/stat.h: POSIX 互換インターフェース。
  • windows.h, winioctl.h: Windows API への直接アクセス。
  • port/win32ntdll.h: NTDLL レベルのステータスチェック（STATUS_DELETE_PENDING 等）。
  • postgres.h
  • postgres_fe.h
  • w32api/winioctl.h

関連コード

  • src/port/win32stat.c: Windows における stat 関連の互換性実装。
  • src/port/path.c: パス操作全般のユーティリティ。
  • src/include/port.h: pgrename 等のプロトタイプ宣言が含まれるヘッダー。

読者向けメモ

  • 本ファイルは主に Windows 環境（WIN32）で有効化されます。Cygwin 環境では一部の挙動が異なるため、条件付きコンパイル（#if defined(WIN32) && !defined(__CYGWIN__)）に注意してください。
  • pgunlink 内の lstat_error_was_status_delete_pending は、ファイルが削除保留状態にある場合に ENOENT を正しく解釈するための重要なハックです。
  • ジャンクションポイント操作は低レベルな DeviceIoControl を使用しているため、Windows SDK のバージョン差異による影響を受けやすい箇所です。
