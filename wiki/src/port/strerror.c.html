src/port/strerror.c

概要

プラットフォーム間で差異のある strerror() および strerror_r() の挙動を抽象化し、PostgreSQL が一貫したエラーメッセージを取得できるようにするためのラッパーモジュールです。

モジュール内での役割

src/port ディレクトリの一部として、OS 固有の不整合を吸収する移植性レイヤーを提供します。特に、スレッドセーフではない標準ライブラリ関数を安全にラップし、Windows 特有のソケットエラーコードへの対応も行います。

責務

  • **移植性の確保**: POSIX API と GNU API の差異を吸収し、strerror_r が存在しない環境へのフォールバックを提供します。
  • **エラーメッセージの補完**: OS が空文字列や不明な値を返す場合に、エラーシンボル（ENOENT 等）や数値コードを用いたフォールバックメッセージを生成します。
  • **Windows 固有対応**: Winsock エラーコードを netmsg.dll を用いて人間が読める形式に変換します。

重要なシンボル

  • pg_strerror(int errnum): 静的バッファを使用してエラーメッセージを取得する簡便なインターフェース。
  • pg_strerror_r(int errnum, char *buf, size_t buflen): 呼び出し元がバッファを提供するスレッドセーフなメイン関数。
  • gnuish_strerror_r(...): プラットフォームの API 差異を吸収する内部ヘルパー。
  • get_errno_symbol(int errnum): 数値の errno を対応するシンボル名（文字列）に変換するマッピング関数。

制御フローとデータフロー

1. pg_strerror_r が呼び出されると、まず Windows 環境であれば Winsock エラーか判定し、該当すれば専用関数で処理します。 2. 次に gnuish_strerror_r を呼び出し、OS 標準の strerror_r（または strerror）からメッセージを取得します。 3. 取得した文字列が空、または無効な場合、get_errno_symbol を試行してシンボル名を取得します。 4. それでも解決しない場合、snprintf を用いて「operating system error %d」という形式の文字列を生成し、バッファに書き込みます。

読者向けメモ

  • このファイルは、標準ライブラリの strerror がスレッドセーフでない可能性を考慮し、可能な限り strerror_r を優先して使用するように設計されています。
  • get_errno_symbol は網羅的ではありませんが、主要なエラーコードをカバーしており、デバッグ時に数値のみが表示される状況を回避するのに役立ちます。
  • Windows 環境で netmsg.dll を動的にロードする処理は、ソケットエラーの可読性を高めるための重要な工夫です。
