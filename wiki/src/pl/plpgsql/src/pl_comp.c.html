src/pl/plpgsql/src/pl_comp.c

概要

PL/pgSQL のコンパイラ実装です。SQL 関数や DO ブロックのソースコードを解析し、実行エンジンが利用する抽象構文木（AST）およびデータ構造（PLpgSQL_function）を構築します。本ファイルは、構文解析器（pl_scanner.c）と実行エンジン（pl_exec.c）の橋渡しを行い、名前解決、型チェック、およびメモリ管理を統括します。

モジュール内での役割

PL/pgSQL サブシステムにおける「コンパイルフェーズ」を担当します。pl_handler.c から呼び出され、システムカタログから取得したソースコードを、実行可能な内部表現へと変換します。

責務

  • **関数コンパイル**: CREATE FUNCTION 等のソースを解析し、PLpgSQL_function 構造体を生成。
  • **名前空間管理**: 変数、レコード、パラメータのスコープ解決および名前解決。
  • **型システム統合**: PostgreSQL の型システムと PL/pgSQL の変数型のマッピングおよび検証。
  • **エラーハンドリング**: コンパイル時の構文エラーに対するコンテキスト情報の付与。
  • **リソース管理**: コンパイル中に使用するメモリコンテキストの管理と、成功時の永続化。

重要なシンボル

コンパイルエントリーポイント

  • plpgsql_compile: 関数コンパイルのメインインターフェース。キャッシュ済みの場合は即座に返却。
  • plpgsql_compile_callback: コンパイルの主処理。AST 構築、名前空間初期化、型解決を行う。
  • plpgsql_compile_inline: DO ブロック（匿名コードブロック）用のコンパイル処理。

名前解決・パーサーフック

  • plpgsql_parser_setup: SQL 式内の変数参照を解決するためのパーサーフック設定。
  • resolve_column_ref: SQL 式内の識別子を PL/pgSQL 変数またはレコードフィールドとして解決。
  • plpgsql_parse_word / plpgsql_parse_dblword: 字句解析器が識別子を検出した際の変数ルックアップ。

データ構造構築

  • plpgsql_build_variable: 変数（PLpgSQL_var）またはレコード（PLpgSQL_rec）の生成と登録。
  • plpgsql_build_datatype: PostgreSQL の型情報から PL/pgSQL の型構造体を構築。
  • build_row_from_vars: 複数の変数を束ねる行（Row）構造体の生成。

制御フローとデータフロー

1. **初期化**: plpgsql_compile が func_cxt（メモリコンテキスト）を作成し、plpgsql_ns_init で名前空間を初期化。 2. **解析**: plpgsql_yyparse を呼び出し、字句解析器と連携して AST を生成。 3. **変数登録**: plpgsql_start_datums でデータム配列を準備し、解析過程で plpgsql_build_variable 等を用いて変数を登録。 4. **解決**: SQL 式が含まれる場合、パーサーフック（plpgsql_pre_column_ref 等）を通じて、SQL 内の識別子を plpgsql_ns_lookup で変数にバインド。 5. **完了**: plpgsql_finish_datums でデータムを関数構造体にコピーし、メモリコンテキストを CacheMemoryContext に昇格させて永続化。

直接依存

  • plpgsql.h: コアデータ構造の定義。
  • parser/parse_node.h: SQL パーサーのノード定義。
  • utils/memutils.h: メモリコンテキスト管理。
  • catalog/: システムカタログ（pg_proc, pg_type）へのアクセス。

関連コード

  • pl_scanner.c: 字句解析器。本ファイルの plpgsql_parse_* 系関数を呼び出す。
  • pl_funcs.c: 名前空間管理（plpgsql_ns_*）や AST 走査ユーティリティ。
  • pl_exec.c: コンパイルされた PLpgSQL_function を実行するエンジン。

読者向けメモ

  • **メモリ管理**: コンパイル中のメモリは plpgsql_compile_tmp_cxt で管理され、失敗時には自動的にクリーンアップされます。成功時のみ MemoryContextSetParent で永続化される設計です。
  • **再入不可**: 本ファイル内のコンパイルロジックは再入可能（re-entrant）ではありません。コンパイル中に別の PL/pgSQL 関数が呼び出されることは想定されていません。
  • **型解決**: ポリモーフィック型（ANYELEMENT 等）の解決は、コンパイル時に呼び出し元の情報に基づいて行われます。
  • **省略された詳細**: plerrcodes.h に依存する例外ラベルの全リストや、個別のステートメント（IF, LOOP 等）の具体的な AST 構築ロジックは、pl_gram.y（パーサー定義）側に委譲されているため、本ファイルでは主に「変数と型の管理」に焦点を当てています。
