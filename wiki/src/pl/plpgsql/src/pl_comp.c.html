src/pl/plpgsql/src/pl_comp.c

概要

PL/pgSQL 言語のコンパイラ部分を実装するファイルです。PostgreSQL の関数定義（CREATE FUNCTION）や匿名コードブロック（DO）のソーステキストを解析し、実行可能な PLpgSQL_function 構造体（実行ツリー）を構築します。

モジュール内での役割

PL/pgSQL サブシステムにおいて、ソースコードを解析して実行可能な中間表現に変換するフロントエンドの役割を担います。pl_handler.c から呼び出され、解析結果を pl_exec.c が実行時に利用します。

責務

  • **関数コンパイル**: SQL テキストを解析し、変数、パラメータ、ステートメントのツリーを生成する。
  • **名前空間管理**: 変数やパラメータのスコープを管理し、名前解決を行う。
  • **データ型構築**: PL/pgSQL で使用するデータ型情報を PostgreSQL のシステムカタログから構築する。
  • **パーサーフック**: SQL 式内の変数参照を解決するためのパーサーコールバックを提供し、動的パラメータを処理する。

重要なシンボル

  • plpgsql_compile: 関数コンパイルの主要なエントリーポイント。
  • plpgsql_compile_inline: DO ブロック用のコンパイル関数。
  • plpgsql_compile_callback: 実際のコンパイル処理を行う内部関数。
  • plpgsql_Datums: コンパイル中に生成されたすべての変数・レコードを保持する配列。
  • plpgsql_parser_setup: SQL 式の解析時に変数参照を解決するためのフック設定。

制御フローとデータフロー

1. **初期化**: plpgsql_compile が cached_function_compile を経由して plpgsql_compile_callback を呼び出します。 2. **コンテキスト作成**: 関数専用のメモリコンテキストを作成し、解析中のメモリ割り当てを管理します。 3. **解析**: plpgsql_yyparse を呼び出し、スキャナからトークンを読み取ってステートメントツリーを構築します。 4. **名前解決**: plpgsql_pre_column_ref 等のフックを通じて、SQL 式内の識別子が PL/pgSQL 変数かテーブル列かを判定し、Param ノードに変換します。 5. **完了**: 構築された PLpgSQL_function を CacheMemoryContext に移動し、永続化します。

読者向けメモ

  • **メモリ管理**: コンパイル中のメモリは plpgsql_compile_tmp_cxt で管理され、成功時にのみ CacheMemoryContext へ昇格されます。このため、コンパイル失敗時のメモリリークが抑制されています。
  • **再入不可**: このファイル内のコードは再入可能ではありません。コンパイル中に別の PL/pgSQL 関数が呼び出されることは想定されていません。
  • **名前解決**: resolve_column_ref は、SQL 式内の識別子が変数か列かを解決する重要な箇所です。複雑な名前解決ルール（A.B.C など）がここで実装されています。
