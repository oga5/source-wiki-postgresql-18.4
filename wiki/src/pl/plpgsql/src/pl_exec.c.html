src/pl/plpgsql/src/pl_exec.c

概要

PL/pgSQL の実行エンジンを実装する中心的なファイルです。コンパイル済みの PLpgSQL_function を受け取り、PLpgSQL_execstate を用いてステートメントを再帰的に実行します。SPI（Server Programming Interface）を介した SQL 実行、変数管理、例外処理、およびトリガー/イベントトリガーの実行を担います。

モジュール内での役割

PL/pgSQL サブシステムの実行フェーズを統括します。pl_comp.c で生成された抽象構文木（AST）を解釈し、PostgreSQL の実行コンテキストと PL/pgSQL の変数状態を橋渡しします。

責務

  • **実行制御**: exec_stmts を中心としたステートメントの逐次実行と制御フロー（ループ、条件分岐、例外処理）の管理。
  • **変数管理**: PLpgSQL_datum のコピー、代入、および型変換（キャスト）。
  • **SPI 連携**: SQL クエリの準備、実行、結果の取得、およびカーソル操作。
  • **メモリ管理**: 実行状態ごとのメモリコンテキスト（関数、ステートメント、式評価）の分離とクリーンアップ。
  • **エラーハンドリング**: ErrorContextCallback を用いたスタックトレースの生成と例外ブロックの制御。

重要なシンボル

実行状態とコンテキスト

  • PLpgSQL_execstate: 関数の実行時状態を保持するメイン構造体。
  • SimpleEcontextStackEntry: 式評価用 ExprContext のスタック管理用構造体。
  • shared_simple_eval_estate: トランザクション内で共有される式評価用 EState。

キャッシュ管理

  • cast_expr_hash: セッション全体で共有されるキャスト式ツリーのキャッシュ。
  • shared_cast_hash: キャスト評価状態のキャッシュ。

エントリーポイント

  • plpgsql_exec_function: 通常の関数および DO ブロックの実行開始点。
  • plpgsql_exec_trigger: トリガー実行の開始点。
  • plpgsql_exec_event_trigger: イベントトリガー実行の開始点。

制御フローとデータフロー

1. **初期化**: plpgsql_estate_setup がメモリコンテキストと SPI 接続を準備。copy_plpgsql_datums が関数定義から実行用変数領域を作成。 2. **実行**: exec_toplevel_block から再帰的に exec_stmts を呼び出し、各ステートメント型に応じた exec_stmt_* 関数へディスパッチ。 3. **式評価**: exec_eval_expr が SPI を介してクエリを実行するか、exec_eval_simple_expr が最適化されたパスで式を評価。 4. **終了**: plpgsql_destroy_econtext と exec_eval_cleanup により、一時的なメモリコンテキストを解放。

読者向けメモ

  • **メモリ所有権**: estate->datum_context は関数生存期間、stmt_mcontext はステートメント生存期間、eval_econtext は式評価生存期間と、メモリ寿命が厳密に管理されています。exec_eval_cleanup の呼び出し漏れはメモリリークに直結します。
  • **最適化**: 「単純な式（Simple Expression）」は SPI をバイパスして直接実行されます。このキャッシュはトランザクション境界でリセットされるため、COMMIT/ROLLBACK を含む関数では再構築が行われます。
  • **未実装・浅い領域**: EXECUTE 文の SELECT INTO は現在未実装です。また、LOOP_RC_PROCESSING マクロは制御フローの複雑さを隠蔽していますが、デバッグ時にはスタックトレースを追う際に注意が必要です。
