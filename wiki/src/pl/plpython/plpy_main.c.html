src/pl/plpython/plpy_main.c

概要

PL/Python 言語ハンドラのメインエントリポイントです。PostgreSQL と Python インタプリタの橋渡しを行い、関数のバリデーション、実行ハンドラ、および匿名コードブロック（DO 文）の実行を管理します。

モジュール内での役割

plpy_main.c は PL/Python モジュールの中心的な制御層です。他のファイル（plpy_exec.c や plpy_procedure.c など）が提供する具体的な実行ロジックや手続き管理機能を呼び出し、PostgreSQL の関数呼び出し規約（PG_FUNCTION_INFO_V1）に従って統合します。

責務

  • **初期化管理**: Python インタプリタの起動と、セッション内での複数バージョン競合の検出。
  • **実行コンテキストの管理**: ネストされた Python 呼び出しを追跡するためのスタック管理（PLyExecutionContext）。
  • **ハンドラ提供**: 関数呼び出し、トリガー実行、および DO ブロック実行の各エントリポイントの提供。
  • **エラー報告**: PostgreSQL のエラースタックに Python 実行時のコンテキスト情報を付与するコールバックの提供。

重要なシンボル

  • plpython3_call_handler: SQL 関数実行時のメインエントリポイント。
  • PLy_initialize: Python インタプリタの初回起動と環境設定を行う内部関数。
  • PLyExecutionContext: 実行中の手続きやメモリコンテキストを保持するスタック構造体。
  • PLy_push_execution_context / PLy_pop_execution_context: 実行コンテキストのスタック操作関数。

制御フローとデータフロー

1. **初期化**: _PG_init でプロセスレベルの競合チェックを行い、PLy_initialize で Python インタプリタを初期化します。 2. **呼び出し**: plpython3_call_handler が呼ばれると、SPI_connect_ext で SPI を確立し、PLy_push_execution_context で新しいコンテキストをスタックに積みます。 3. **実行**: PG_TRY ブロック内で PLy_procedure_get により手続きをロードし、PLy_exec_function または PLy_exec_trigger を実行します。 4. **終了**: PG_CATCH または正常終了時に PLy_pop_execution_context を呼び出し、コンテキストを破棄してメモリを解放します。

読者向けメモ

  • **メモリ管理**: PLyExecutionContext は TopTransactionContext または PortalContext を使用して割り当てられます。scratch_ctx は必要に応じて遅延生成されるため、メモリリークを防ぐために PLy_pop_execution_context での削除が必須です。
  • **競合検出**: plpython_version_bitmask を用いたチェックは、同一セッション内で異なる Python バージョンが混在することによるクラッシュを防ぐための重要な安全装置です。
  • **設計上の注意**: SPI_finish() が plpy_exec.c 側で呼ばれる設計となっており、ハンドラ側との責務分担がやや複雑である点に注意してください。
