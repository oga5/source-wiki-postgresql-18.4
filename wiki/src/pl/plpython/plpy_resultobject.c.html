src/pl/plpython/plpy_resultobject.c

概要

PostgreSQLのSPI（Server Programming Interface）を通じて実行されたクエリの結果を保持・操作するための、Python拡張型 PLyResult を定義するソースファイルです。

モジュール内での役割

plpy モジュールにおいて、SQL実行結果をPythonオブジェクトとしてラップする役割を担います。ユーザーがPL/Python関数内でクエリを実行した際に返されるオブジェクトの基盤となります。

責務

  • PLyResult 型の定義とPython環境への登録。
  • クエリ結果（行データ、ステータス、メタデータ）を保持するオブジェクトの生成と破棄。
  • Python側からのアクセス（インデックス参照、属性取得、文字列変換）を可能にするメソッドの実装。
  • TupleDesc を介した列名や型情報の抽出機能の提供。

重要なシンボル

  • PLyResultObject: クエリ結果を保持するC構造体。
  • PLy_result_init_type(): Python型システムに PLyResult を登録する初期化関数。
  • PLy_result_new(): 新しい PLyResult インスタンスを生成するファクトリ関数。
  • PLyResult_spec: Pythonの型定義仕様（スロットやメソッドの定義を含む）。

制御フローとデータフロー

1. **初期化**: PLy_result_init_type が呼び出され、PyType_FromSpec を通じてPythonの型オブジェクトが作成されます。 2. **生成**: クエリ実行時に PLy_result_new が呼ばれ、PLyResultObject がメモリ確保されます。この際、rows（リスト）や status が初期化されます。 3. **操作**: Python側から result[i] や result.colnames() が呼び出されると、対応するC関数（PLy_result_subscript や PLy_result_colnames）が実行され、内部の tupdesc や rows リストからデータが抽出されます。 4. **破棄**: PLy_result_dealloc が呼び出され、保持しているPythonオブジェクトの参照カウントを減らし、TupleDesc を解放します。

直接依存

  • postgres.h: PostgreSQLの基本定義。
  • plpy_elog.h: エラー報告用インターフェース。
  • plpy_resultobject.h: 構造体定義と外部公開関数。
  • plpy_util.h: ユーティリティ関数（Unicode変換など）。

関連コード

  • src/pl/plpython/plpy_exec.c: クエリ実行のメインロジック。ここで PLy_result_new が呼び出され、結果が格納されます。
  • src/pl/plpython/plpy_typeio.c: PostgreSQLの型とPythonの型を変換するロジック。

読者向けメモ

  • 本ファイルはPythonのC API（特に PyType_Spec と PyType_Slot）を多用しています。Python 3.8以降の参照カウント管理の変更に伴う条件付きコンパイル（PY_VERSION_HEX）が含まれている点に注意してください。
  • tupdesc はクエリ結果のメタデータ（列名や型）を保持しますが、すべてのクエリ結果に存在するわけではないため、アクセス時には必ずNULLチェックが行われています。
