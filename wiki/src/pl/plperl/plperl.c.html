src/pl/plperl/plperl.c

概要

plperl.c は、PostgreSQL の手続き言語である **PL/Perl**（信頼された plperl および信頼されない plperlu）のコア実装ファイルです。このファイルが巨大である理由は、PostgreSQL の拡張言語ハンドラとしてのライフサイクル管理、Perl インタプリタのサンドボックス化とセキュリティ境界の維持、PostgreSQL と Perl 間の複雑なデータ型双方向変換、および SPI（Server Programming Interface）を介したデータベースアクセスの調停処理がすべてここに集約されているためです。

モジュール内での役割

src/pl/plperl ディレクトリにおいて、本ファイルは手続き言語のメインエントリポイントとして機能します。

  • plperl.h で定義されたインライン変換関数（sv2cstr, cstr2sv など）を実質的に駆動します。
  • 同ディレクトリ内の .xs ファイル（SPI.xs や Util.xs）から呼び出される C レベルのバックエンド API を提供します。
  • PostgreSQL の関数マネージャ（fmgr）に対して、呼び出しハンドラ、インラインハンドラ、およびバリデータ関数を公開します。

責務

インタプリタ管理とセキュリティ (Interpreter Lifecycle & Security)

  • **インタプリタの分離**: 信頼された（trusted）関数に対しては、実行時の SQL ユーザーID（GetUserId()）ごとに独立した Perl インタプリタを割り当ててキャッシュします。信頼されない（untrusted）関数に対しては、システム全体で単一の共有インタプリタ（ユーザーID 0）を使用します。これにより、権限昇格攻撃を防ぎます。
  • **サンドボックス化**: 信頼されたインタプリタの初期化時（plperl_trusted_init）に、安全でないオペコード（ファイル入出力やシステムコールなど）を plperl_opmask を用いてマスクし、require や dofile を安全な代替関数（pp_require_safe）に差し替えます。また、DynaLoader などの名前空間を削除して外部共有ライブラリのロードを禁止します。
  • **遅延初期化**: plperl.on_init GUC を用いた事前ロードを可能にするため、初期インタプリタ（plperl_held_interp）を「保持」状態（信頼/非信頼が未決定の状態）で作成し、最初の関数実行時に完全な初期化を行います。

関数のコンパイルとキャッシュ (Function Compilation & Caching)

  • **メタデータキャッシュ**: コンパイルされた関数の情報を plperl_proc_desc 構造体にキャッシュし、plperl_proc_hash で管理します。
  • **キャッシュの検証**: CREATE OR REPLACE FUNCTION などによって関数定義が変更された場合、システムカタログ（pg_proc）の xmin および TID を比較してキャッシュの有効性を検証します（validate_plperl_function）。
  • **サブルーチンの生成**: PostgreSQL::InServer::mkfunc を Perl 空間で実行し、SQL 関数のソーステキストを無名サブルーチンとしてコンパイルします。

データ型変換 (Data Type Conversion)

  • **PostgreSQL → Perl**:
  • スカラー値を文字列または適切な Perl 表現に変換。
  • 複合型（行データ）を Perl のハッシュ参照（HV）に変換（plperl_hash_from_tuple）。
  • 配列を Perl の配列参照（AV）に変換（plperl_ref_from_pg_array）。多次元配列は再帰的に処理されます。
  • **Perl → PostgreSQL**:
  • Perl の SV/AV/HV から PostgreSQL の Datum への変換（plperl_sv_to_datum）。
  • ドメイン制約のチェック（domain_check）や、SQL/JSON 変換（Transform）の適用。

実行制御とコンテキスト管理 (Execution Dispatch & Contexts)

  • 通常の関数、トリガー、イベントトリガー、および匿名コードブロック（DO ステートメント）の実行をハンドリングします。
  • 実行中のコールスタック状態を plperl_call_data 構造体で追跡し、エラー発生時のコンテキスト情報を提供します。

SPI 連携 (SPI & Transaction Integration)

  • Perl 空間から呼び出される SQL 実行（spi_exec_query など）の実体となる C 関数群（plperl_spi_exec など）を提供します。
  • SPI 実行をサブトランザクションで包むことで、Perl 内部でのエラー捕捉（eval）と PostgreSQL のトランザクション整合性を両立させます。

重要なシンボル

ライフサイクル・コンテキスト管理

  • void _PG_init(void): モジュールロード時の初期化。GUC の登録、ハッシュテーブルの作成、および初期「保持」インタプリタの作成を行います。
  • static PerlInterpreter *plperl_init_interp(void): 新しい Perl インタプリタを割り当て、基本初期化（PLC_PERLBOOT の実行など）を行います。
  • static void select_perl_context(bool trusted): 実行コンテキストに応じたインタプリタを選択・有効化し、必要に応じて初期化（plperl_trusted_init / plperl_untrusted_init）を行います。
  • static void activate_interpreter(plperl_interp_desc *interp_desc): アクティブな Perl インタプリタを切り替えます。

SQL 関数インターフェース (エントリポイント)

  • Datum plperl_call_handler(PG_FUNCTION_ARGS): 通常の PL/Perl 関数およびトリガーの実行ハンドラ。
  • Datum plperl_inline_handler(PG_FUNCTION_ARGS): DO ブロック（匿名コード）の実行ハンドラ。
  • Datum plperl_validator(PG_FUNCTION_ARGS): CREATE FUNCTION 時の構文・定義検証ハンドラ。

コンパイル・キャッシュ

  • struct plperl_proc_desc: コンパイルされた関数のメタデータ、Perl サブルーチンへの参照（SV *reference）、引数/戻り値の変換情報を保持する構造体。
  • static plperl_proc_desc *compile_plperl_function(...): pg_proc からソースを読み込み、Perl サブルーチンを生成してキャッシュに登録します。
  • static void plperl_create_sub(plperl_proc_desc *prodesc, const char *s, Oid fn_oid): Perl 側のコンパイラを呼び出してサブルーチンを生成します。

データ変換

  • static Datum plperl_sv_to_datum(SV *sv, Oid typid, int32 typmod, ...): Perl の SV を PostgreSQL の Datum に変換する中心的な関数。
  • static SV *plperl_hash_from_tuple(HeapTuple tuple, TupleDesc tupdesc, bool include_generated): PostgreSQL のタプルを Perl のハッシュ参照に変換します。
  • static SV *plperl_ref_from_pg_array(Datum arg, Oid typid): PostgreSQL の配列を Perl の配列参照に変換します。

SPI 連携

  • HV *plperl_spi_exec(char *query, int limit): Perl から渡されたクエリを SPI で実行します。
  • SV *plperl_spi_prepare(char *query, int argc, SV **argv): プリペアドステートメントを作成し、インタプリタ固有の query_hash に登録します。

制御フローとデータフロー

1. 関数実行の制御フロー

通常関数の実行時、制御は以下の流れで遷移します。

[SQL 実行エンジン]
       │
       ▼
plperl_call_handler(fcinfo)
       │
       ├─► compile_plperl_function() ──► (キャッシュにあれば再利用)
       │         │
       │         ▼ (キャッシュミス時)
       │   select_perl_context() ──► [Perl インタプリタの選択/初期化]
       │         │
       │         ▼
       │   plperl_create_sub() ──► [Perl 空間で sub をコンパイル]
       │
       ├─► activate_interpreter() ──► [Perl コンテキストの切り替え]
       │
       ├─► (引数の変換: Datum -> SV) ──► plperl_hash_from_datum / plperl_ref_from_pg_array
       │
       ├─► plperl_call_perl_func() ──► [perl_run / call_sv() で実行]
       │
       ├─► (戻り値の変換: SV -> Datum) ──► plperl_sv_to_datum()
       │
       ▼
[PG_FINALLY] ──► activate_interpreter(oldinterp) & 参照カウントのデクリメント

2. SPI 実行のデータフローとエラーパス

Perl コード内から SQL を実行する際、PostgreSQL のエラーハンドリング（sigsetjmp）と Perl の例外（croak）を安全に仲介するため、以下のフローを辿ります。

1. **SPI 呼び出し**: Perl 側から plperl_spi_exec が呼び出されます。 2. **安全性の検証**: check_spi_usage_allowed() が走り、END ブロック内や関数コンパイル中の不正な SPI 使用を拒否します。 3. **サブトランザクションの開始**: BeginInternalSubTransaction(NULL) を呼び出し、現在のメモリコンテキストを保存します。 4. **PG_TRY ブロック**:

  • SPI_execute() を実行。
  • 成功した場合、plperl_spi_execute_fetch_result() が結果タプルを Perl のハッシュ配列（HV の AV）に変換します。
  • ReleaseCurrentSubTransaction() でサブトランザクションをコミットします。

    • 5. **PG_CATCH ブロック**:

  • エラーが発生した場合、CopyErrorData() でエラー情報を退避し、FlushErrorState() で PostgreSQL のエラー状態をクリアします。
  • RollbackAndReleaseCurrentSubTransaction() でサブトランザクションをロールバックします。
  • 退避したエラーメッセージを croak_cstr() に渡し、Perl 側の例外としてスロー（die）させます。

読者向けメモ

メモリ所有権とライフタイム

  • **関数記述子 (plperl_proc_desc)**:
  • 各関数記述子は専用のメモリコンテキスト fn_cxt（TopMemoryContext の子）を持ちます。
  • 記述子のライフタイムは fn_refcount で管理されます。参照カウントが 0 になると、free_plperl_function が呼び出され、fn_cxt ごとすべての関連データ（Perl サブルーチン参照 reference を含む）が削除されます。
  • **一時コンテキスト**:
  • セット戻り値関数（SRF）の処理（plperl_return_next_internal）では、行ごとの変換処理で発生するメモリリークを防ぐため、専用の一時コンテキスト current_call_data->tmp_cxt を作成し、1行出力するごとにリセット（MemoryContextReset）します。

インタプリタの不変条件 (Invariants)

  • 信頼された（trusted）PL/Perl インタプリタは、SQL の実効ユーザーID（GetUserId()）ごとに個別に作成され、ハッシュテーブル plperl_interp_hash にキャッシュされます。これにより、一般ユーザーが他人の権限で動作する PL/Perl 関数にコードを注入することを防ぎます。
  • 信頼されない（untrusted）PL/PerlU インタプリタは、ユーザーID 0 (InvalidOid) として1つだけ作成され、システム全体で共有されます。

エッジケースと制限

  • **マルチインタプリタ非対応環境**:
  • Perl ビルドが MULTIPLICITY をサポートしていない場合、複数のインタプリタを割り当てることができません。この環境で異なるユーザーIDによる PL/Perl 関数の並行実行や、PL/Perl と PL/PerlU の混在実行を行うと、"cannot allocate multiple Perl interpreters..." エラーが発生します。
  • **Windows 環境におけるロケール処理**:
  • Perl 起動時に setlocale が勝手に呼び出されて PostgreSQL のロケールが破壊されるのを防ぐため、Windows かつ古い Perl バージョンでは、起動前後にロケールを退避・復元する特殊な処理（PLPERL_RESTORE_LOCALE）が埋め込まれています。

意図的な省略

  • .xs ファイル（SPI.xs, Util.xs）の具体的な Perl-C バインディングの生成ロジックは、このファイルではなくそれぞれの XS ファイルに閉じているため、本ページでは詳細を省略しています。
