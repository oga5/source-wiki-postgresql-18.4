src/backend/optimizer/geqo

システム内での役割

src/backend/optimizer/geqo は、PostgreSQLのクエリ最適化において、結合順序の決定というNP困難な問題を遺伝的アルゴリズム（GA）を用いて近似的に解くためのサブシステムです。このモジュールは、ディレクトリ境界に基づき、GAのライフサイクル管理、個体群管理、および多様な交叉・突然変異アルゴリズムを統合しています。

責務

  • **GAのオーケストレーション**: クエリプランナから呼び出され、個体群の初期化から世代交代、最終的な最適解の抽出までを管理する。
  • **個体群の維持と評価**: 結合順序の候補（染色体）を保持し、プランナのコスト評価関数を用いて各個体の適応度を算出する。
  • **探索の多様性確保**: 交叉（Crossover）および突然変異（Mutation）演算子を提供し、探索空間における局所解への陥没を防ぐ。

このモジュールのファイル

  • geqo_main.c: GAのメインエントリーポイント。
  • geqo_pool.c: 個体群（Pool）の管理とソート。
  • geqo_eval.c: 結合順序のコスト評価。
  • geqo_selection.c: 親個体の選択アルゴリズム。
  • geqo_recombination.c: 交叉の初期化と共通ユーティリティ。
  • geqo_copy.c: 染色体データの複製。
  • geqo_mutation.c: 突然変異操作。
  • geqo_random.c: 乱数生成器。
  • geqo_misc.c: デバッグおよび統計出力。
  • 交叉アルゴリズム実装: geqo_erx.c, geqo_cx.c, geqo_pmx.c, geqo_ox1.c, geqo_ox2.c, geqo_px.c

主要なデータ構造

  • Chromosome: 結合順序（遺伝子配列）と適応度（コスト）を保持する個体データ。
  • Pool: 染色体の集合を管理する構造体。
  • Clump: 結合済みのリレーション集合を管理し、プラン構築を補助する。
  • Edge: エッジ組み換え交叉（ERX）で使用される接続関係データ。

主なフロー

1. **初期化**: geqo() が呼ばれ、乱数シードとGAパラメータ（個体群サイズ等）を決定。 2. **個体群生成**: random_init_pool によりランダムな結合順序を生成し、geqo_eval でコストを評価。 3. **進化ループ**: 世代交代を繰り返し、以下の処理を実行：

  • geqo_selection で親個体を選択。
  • 交叉アルゴリズム（ERX, CX等）で子個体を生成。
  • geqo_eval で子個体のコストを評価。
  • spread_chromo でプールを更新。

    • 4. **終了**: 最良の個体からクエリ木を構築し、プランナへ返却。

外部依存

  • postgres.h: 基本定義およびメモリ管理。
  • optimizer/: joininfo.h, pathnode.h, paths.h 等のプランナ関連ヘッダ。
  • utils/: memutils.h（メモリコンテキスト管理）、pg_prng.h（乱数生成）。

内部構成

本モジュールは、特定の交叉アルゴリズムをコンパイル時のマクロ定義（ERX, CX, PMX 等）によって切り替える構成をとっています。子モジュールは存在せず、ディレクトリ内の各ファイルがフラットに連携します。

推奨読書順

1. geqo_main.c (全体像の把握) 2. geqo_pool.c (個体群管理の理解) 3. geqo_eval.c (コスト評価の仕組み) 4. geqo_selection.c (選択戦略) 5. geqo_erx.c (代表的な交叉アルゴリズム) 6. geqo_recombination.c / geqo_mutation.c (補助演算) 7. geqo_random.c / geqo_copy.c (ユーティリティ)
