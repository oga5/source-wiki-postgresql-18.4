src/include/executor

システム内での役割

src/include/executor は、PostgreSQL のクエリ実行エンジン（Executor）のコアとなるヘッダーファイル群を格納するディレクトリです。プランナーが生成した実行計画（PlannedStmt）を解釈し、タプルの取得、式評価、結合、集約、並列処理などの実行ライフサイクルを管理するインターフェースを提供します。

責務

  • **実行エンジンの抽象化**: プランノード（スキャン、結合、集約等）ごとの実行状態（State 構造体）と操作インターフェースの定義。
  • **式評価のインフラ**: SQL 式を効率的に評価するための命令セットと実行ステップの定義。
  • **データ交換の基盤**: プランノード間で受け渡されるタプルの抽象化（TupleTableSlot）と、並列処理時のプロセス間通信（shm_mq）の提供。
  • **拡張性の提供**: FDW（外部データラッパー）や SPI（サーバープログラミングインターフェース）を通じた、外部からの実行エンジン利用のサポート。

このモジュールのファイル

本モジュールには、実行エンジンを構成する 60 個のヘッダーファイルが含まれます。

  • **主要ノード定義**: nodeSeqscan.h, nodeHashjoin.h, nodeAgg.h, nodeIndexscan.h 等。
  • **実行基盤**: executor.h, execExpr.h, execScan.h, execParallel.h, tuptable.h。
  • **外部インターフェース**: spi.h, spi_priv.h, tablefunc.h。
  • **その他**: execdebug.h, instrument.h, execPartition.h 等。

主要なデータ構造

  • QueryDesc: クエリ実行の全体状態を保持する最上位構造体。
  • TupleTableSlot: プランノード間で受け渡されるタプルの抽象コンテナ。
  • ExprEvalStep / ExprEvalOp: 式評価のためのフラットな命令セット構造。
  • HashJoinTable: ハッシュ結合時のメモリ内ハッシュテーブル管理構造。
  • ParallelExecutorInfo: 並列クエリ実行時のワーカー間共有状態。
  • Instrumentation: 実行時間やバッファ使用量などの統計情報保持構造。

主なフロー

1. **初期化**: ExecutorStart が QueryDesc を受け取り、プランツリーに対応する PlanState ツリーを構築。 2. **実行**: ExecutorRun が ExecProcNode を再帰的に呼び出し、タプルを末端ノードから上位ノードへ伝播。 3. **式評価**: ExecEvalExpr が各ノード内で必要な計算やフィルタリングを実行。 4. **終了**: ExecutorEnd がメモリコンテキストの解放や統計情報の集計を行い、リソースをクリーンアップ。

外部依存

  • nodes/execnodes.h: 実行時ノード構造体の定義。
  • nodes/parsenodes.h / nodes/plannodes.h: 解析および計画ノードの定義。
  • access/parallel.h: 並列クエリ実行の基盤。
  • tcop/dest.h: クエリ結果の出力先管理。
  • utils/memutils.h: メモリコンテキスト管理。

内部構成

本モジュールはディレクトリ境界によって定義されており、以下のカテゴリに大別されます。

  • **実行ノード**: 各種スキャン、結合、集約ノードの定義。
  • **式評価エンジン**: execExpr.h を中心とした式評価インフラ。
  • **SPI**: サーバー内部からの SQL 実行インターフェース。
  • **並列処理**: execParallel.h や tqueue.h を含む並列クエリ実行基盤。

推奨読書順

1. executor.h (全体像の把握) 2. execdesc.h (クエリ実行のパッケージ構造) 3. tuptable.h (データ交換の基本単位) 4. execExpr.h (式評価の仕組み) 5. execScan.h (スキャンノードの共通処理) 6. nodeHashjoin.h / nodeAgg.h (主要な演算ノードの例) 7. execParallel.h (並列実行の仕組み) 8. spi.h (外部インターフェース)
