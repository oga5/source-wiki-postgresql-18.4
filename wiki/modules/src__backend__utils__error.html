src/backend/utils/error

システム内での役割

PostgreSQL におけるエラー報告、ログ出力、およびアサーション失敗時の処理を統合的に管理するサブシステムです。システム全体の堅牢性を維持し、エラー発生時の情報収集と適切なルーティングを担います。

責務

  • **エラーライフサイクル管理**: ereport() を起点としたエラー情報の構築、スタック管理、および例外処理（トランザクションアボート）のトリガー。
  • **構造化ログ出力**: ログのフォーマット（CSV/JSON）に応じたシリアライズと、syslogger プロセスへの転送。
  • **低レイヤーな障害通知**: システムの整合性が疑われる際のアサーション失敗処理と、デバッグ情報の出力。

このモジュールのファイル

  • src/backend/utils/error/elog.c: エラー処理のオーケストレーター。
  • src/backend/utils/error/csvlog.c: CSV 形式のログ出力実装。
  • src/backend/utils/error/jsonlog.c: JSON 形式のログ出力実装。
  • src/backend/utils/error/assert.c: アサーション失敗時の最小限の処理。

主要なデータ構造

  • ErrorData: エラー報告の各要素（メッセージ、詳細、ヒント、SQLSTATE 等）を保持する構造体。
  • errordata_stack: 再入可能なエラー報告をサポートする静的スタック（深度 5）。
  • StringInfo: ログ出力時に使用される可変長文字列バッファ。

主なフロー

1. **エラー発生**: ereport() マクロが errstart() を呼び出し、ErrorData をスタックに確保。 2. **情報構築**: errmsg() 等の関数でエラー詳細を ErrorData に格納。 3. **終了とルーティング**: errfinish() が EmitErrorReport() を呼び出し、設定に応じて csvlog.c や jsonlog.c を経由してログを出力。 4. **例外処理**: ERROR 以上の深刻度の場合、pg_re_throw() を経由して siglongjmp を実行し、トランザクションをアボート。

外部依存

  • postgres.h (基本型)
  • lib/stringinfo.h (バッファ管理)
  • postmaster/syslogger.h (ログパイプライン)
  • utils/memutils.h (メモリコンテキスト)
  • libpq/pqformat.h (クライアント通信)

内部構成

本モジュールは、elog.c を中心としたエラー報告パイプラインと、出力形式を定義するプラグイン群（csvlog.c, jsonlog.c）、および独立性の高い assert.c で構成されます。

推奨読書順

1. src/backend/utils/error/elog.c: 全体の制御フローを理解する。 2. src/backend/utils/error/jsonlog.c: 構造化ログの出力実装を確認する。 3. src/backend/utils/error/csvlog.c: ログ出力の別形式実装を確認する。 4. src/backend/utils/error/assert.c: 異常系における最小限の処理を確認する。
