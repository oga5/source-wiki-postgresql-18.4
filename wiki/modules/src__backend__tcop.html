src/backend/tcop

システム内での役割

src/backend/tcop（Traffic Cop）は、PostgreSQL バックエンドプロセスの中心的な制御層です。クライアントからの通信プロトコルを解釈し、クエリのライフサイクル（解析・計画・実行）を管理する「交通整理」の役割を担います。

責務

  • **プロセス制御**: バックエンド起動時の初期化とメインループの管理。
  • **プロトコル処理**: Simple/Extended Query プロトコルおよび Fastpath（関数直接実行）の解釈とディスパッチ。
  • **実行管理**: 計画済みクエリ（PlannedStmt）の実行戦略決定と、ポータル（Portal）を通じた実行制御。
  • **ユーティリティ実行**: DDL やトランザクション制御などの非 DML コマンドの実行とイベントトリガーの管理。
  • **出力抽象化**: クエリ結果の宛先（クライアント、メモリ等）の制御と完了タグの生成。

このモジュールのファイル

  • backend_startup.c: プロセス起動と接続初期化。
  • postgres.c: メインループとプロトコル処理の統括。
  • pquery.c: ポータルの管理とクエリ実行戦略の決定。
  • utility.c: ユーティリティコマンドの実行とディスパッチ。
  • dest.c: クエリ結果の出力先抽象化（DestReceiver）。
  • cmdtag.c: コマンド完了タグの生成と管理。
  • fastpath.c: SQL を介さない関数直接実行（PQfn）の実装。

主要なデータ構造

  • PortalData / Portal: クエリ実行状態を保持するコンテナ。
  • DestReceiver: クエリ結果の出力先を抽象化するインターフェース。
  • CommandTagBehavior: コマンドごとの振る舞い（行数表示、トリガー可否等）を定義。
  • fp_info: Fastpath 実行時の関数メタデータ。

主なフロー

1. **起動**: BackendMain が接続を確立し、PostgresMain へ移行。 2. **受信**: ReadCommand がクライアントからのメッセージを待機。 3. **ディスパッチ**: メッセージタイプに応じて exec_simple_query や HandleFunctionRequest 等へ分岐。 4. **実行**: PortalStart で戦略を決定し、PortalRun で Executor を呼び出す。 5. **完了**: EndCommand で結果を配送し、ReadyForQuery で次のコマンドを待機。エラー時は sigsetjmp を用いてリカバリ。

外部依存

  • libpq/: 通信プロトコル処理。
  • executor/, optimizer/, parser/: クエリ処理パイプライン。
  • access/xact.h: トランザクション管理。
  • utils/: メモリ管理、スナップショット、システムキャッシュ。

内部構成

本モジュールはディレクトリ境界により定義されており、サブモジュールは持ちません。各ファイルは tcopprot.h を通じて相互にインターフェースを公開しています。

推奨読書順

1. postgres.c（全体像とメインループ） 2. pquery.c（クエリ実行のライフサイクル） 3. utility.c（DDL 等の特殊コマンド処理） 4. dest.c（結果配送の仕組み） 5. backend_startup.c（初期化プロセス） 6. cmdtag.c（完了通知の仕組み） 7. fastpath.c（特殊な関数実行パス）
