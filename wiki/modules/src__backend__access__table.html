src/backend/access/table

システム内での役割

本モジュールは、PostgreSQLにおけるテーブルアクセスメソッド（Table AM）の抽象化レイヤーです。実行エンジンやプランナが特定のストレージ実装（heap等）を意識することなく、統一されたインターフェースを通じてテーブル操作を行えるようにします。

※本ページはディレクトリ境界に基づき構成されています。

責務

  • **抽象化**: テーブル操作の共通インターフェース（Table AM API）の提供と管理。
  • **アクセス制御**: リレーションのオープン・クローズ時におけるテーブル種別の検証。
  • **共通ロジックの集約**: 並列スキャン制御、TOAST処理の判断ロジック、スロット管理などの汎用的な処理の実装。
  • **プラグイン基盤**: 新規アクセスメソッドの動的ロードとAPI契約（必須関数の実装）の検証。

このモジュールのファイル

  • table.c: テーブルアクセスの高レベルなエントリポイント（オープン/クローズ）。
  • tableam.c: AM共通の実行ロジックおよび並列スキャン制御。
  • tableamapi.c: AMハンドラのロードとAPI検証。
  • toast_helper.c: タプル操作時のTOAST（圧縮・外部化）判断ロジック。

主要なデータ構造

  • TableAmRoutine: 各アクセスメソッドが実装すべき関数ポインタ群を保持する構造体。
  • ToastTupleContext: TOAST処理中のタプル状態を管理する構造体。
  • TupleTableSlot: テーブル属性に基づいたデータ保持用スロット。

主なフロー

1. **リレーションのオープン**: table_open がリレーションを検証し、テーブルとしてアクセス可能かを確認する。 2. **AMのロード**: GetTableAmRoutine がカタログからハンドラを取得し、必要なAPIが揃っているかを検証する。 3. **データ操作**: tableam.c のラッパー関数が relation->rd_tableam を介して各AM固有のメソッドをディスパッチする。 4. **TOAST処理**: 更新時に toast_helper.c が属性ごとの圧縮・外部化を判定し、必要に応じて物理的なストレージ操作を呼び出す。

外部依存

  • postgres.h (基本定義)
  • access/tableam.h (AMインターフェース定義)
  • access/relation.h, utils/rel.h (リレーション管理)
  • storage/bufmgr.h, storage/smgr.h (ストレージ管理)
  • access/detoast.h, varatt.h (TOAST関連)

内部構成

本モジュールは、テーブルアクセスの「入り口（table.c）」、「API管理（tableamapi.c）」、「共通実行ロジック（tableam.c）」、「補助ロジック（toast_helper.c）」の4層で構成されています。

推奨読書順

1. tableam.c: AMのディスパッチと並列スキャンの仕組みを理解する。 2. table.c: テーブルアクセスの入り口となるインターフェースを確認する。 3. toast_helper.c: タプル操作におけるデータ圧縮・外部化のロジックを把握する。 4. tableamapi.c: 新規AMを統合するためのAPI契約と検証プロセスを確認する。
