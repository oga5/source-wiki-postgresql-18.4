src/backend/storage/buffer

システム内での役割

本モジュールは PostgreSQL のバッファ管理サブシステムの中核であり、ディスク上のデータページをメモリ（共有バッファプールおよびプロセスローカルバッファ）へ読み込み、管理するための基盤を提供します。データベースの I/O 性能を左右する最重要コンポーネントの一つです。

責務

  • **バッファライフサイクル管理**: ページのピン留め（Pin）、解放、ダーティフラグの管理、およびトランザクション終了時のクリーンアップ。
  • **I/O 制御**: 同期および非同期 I/O（AIO）の実行と、ストレージマネージャ（smgr）との連携。
  • **バッファ置換戦略**: クロック掃引法を用いた共有バッファの再利用と、バルク操作時のリングバッファ戦略の提供。
  • **マッピング管理**: ディスク上のページ（BufferTag）とメモリ上のバッファ ID を対応付けるハッシュテーブルの維持。
  • **メモリ初期化**: システム起動時の共有メモリ領域の確保とデータ構造の初期化。

このモジュールのファイル

  • bufmgr.c: バッファ管理のメインエントリポイント。
  • localbuf.c: 一時テーブル専用のプロセスローカルバッファ管理。
  • freelist.c: バッファ置換アルゴリズムとアクセス戦略の実装。
  • buf_table.c: ページタグとバッファ ID を結びつけるハッシュテーブル管理。
  • buf_init.c: 共有メモリの初期化とバッファプールのセットアップ。

主要なデータ構造

  • BufferDesc: バッファのメタデータ（タグ、状態、参照カウント）を保持する共有メモリ構造体。
  • BufferLookupEnt: ハッシュテーブルのエントリ。
  • BufferStrategyControl: 共有メモリ上の置換戦略のグローバル状態。
  • BufferAccessStrategyData: スキャン等の操作ごとのリングバッファ戦略。
  • LocalBufferDescriptors / LocalBufHash: 一時テーブル用バッファの管理構造。

主なフロー

1. **要求**: ReadBuffer が呼び出されると、BufTableLookup でハッシュテーブルを検索。 2. **ヒット**: 該当バッファをピン留めして返却。 3. **ミス**: StrategyGetBuffer で置換候補を選定し、smgr を介してディスクから読み込み、ハッシュテーブルに登録。 4. **I/O**: 非同期 I/O を活用し、完了を待機してページを確定。 5. **クリーンアップ**: トランザクション終了時に ResourceOwner がピンを自動解放。

外部依存

  • storage/buf_internals.h, storage/bufmgr.h: バッファ管理の定義。
  • storage/aio.h: 非同期 I/O フレームワーク。
  • storage/smgr.h: 物理ストレージ操作。
  • utils/resowner.h: リソース管理。
  • port/atomics.h: 高速な状態更新用。

内部構成

本モジュールは、共有バッファ管理（bufmgr.c, freelist.c, buf_table.c, buf_init.c）と、プロセスローカルな一時テーブル管理（localbuf.c）の二階層で構成されています。

推奨読書順

1. bufmgr.c: 全体の司令塔としての役割を理解する。 2. localbuf.c: 共有バッファとの対比として一時テーブル管理を理解する。 3. freelist.c: バッファ置換の意思決定ロジックを把握する。 4. buf_table.c: 高速検索の仕組みを確認する。 5. buf_init.c: システム起動時のメモリレイアウトを確認する。

未解決の問い

  • 非同期 I/O（AIO）の統合が、将来的にバッファ管理のロック競合に与える影響の精緻な評価。
  • 大規模バッファプールにおける BufferSync 等の全スキャン処理のパフォーマンス改善余地。
