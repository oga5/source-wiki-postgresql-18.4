src/backend/storage/file

システム内での役割

本モジュールは、PostgreSQLにおけるファイルシステム操作の抽象化と管理を行う中核レイヤーです。OSのファイル記述子（FD）制限を回避するための仮想ファイル記述子（VFD）管理から、一時ファイルのライフサイクル管理、ディレクトリの再帰的コピーまで、データベースの物理的なファイルI/Oを安全かつ効率的に制御します。

責務

  • **FD管理**: LRUアルゴリズムを用いたVFDによるファイル記述子の効率的な再利用と枯渇防止。
  • **一時ファイル管理**: トランザクションやプロセスを跨ぐ一時ファイルの作成・共有・自動削除の制御。
  • **データ整合性**: fsync を活用したクラッシュセーフなファイル操作と、アンログドテーブルのリカバリ処理。
  • **高レベルI/O**: 巨大ファイルの透過的な分割管理や、ディレクトリ単位の再帰的コピー機能の提供。

このモジュールのファイル

  • fd.c: VFD管理と低レベルファイルI/Oの基盤。
  • buffile.c: 巨大ファイル対応のバッファ付きI/Oインターフェース。
  • fileset.c: 一時ファイルの名前空間とテーブルスペース分散管理。
  • sharedfileset.c: プロセス間で共有可能な一時ファイルセットの管理。
  • copydir.c: ディレクトリの再帰的コピーと同期ユーティリティ。
  • reinit.c: アンログドテーブルのリカバリと初期化処理。

主要なデータ構造

  • Vfd: 仮想ファイル記述子を保持するキャッシュ構造体。
  • BufFile: 物理ファイルを論理的に連結・バッファリングする構造体。
  • FileSet / SharedFileSet: 一時ファイルの集合を管理する名前空間構造体。
  • unlogged_relation_entry: リカバリ対象のリレーションを追跡するハッシュテーブル用構造体。

主なフロー

1. **ファイルオープン**: PathNameOpenFile 等が fd.c を経由し、必要に応じてLRUキャッシュからFDを解放してOSのファイルを開く。 2. **バッファ付きアクセス**: BufFile を介した読み書き時、内部で fd.c を呼び出し、1GB単位の物理ファイル分割を透過的に処理する。 3. **共有管理**: SharedFileSet がDSMと連携し、参照カウンタを用いて最後のプロセスが終了した際に一時ファイルを自動クリーンアップする。 4. **リカバリ**: 起動時に ResetUnloggedRelations がディレクトリを走査し、_init フォークを基にメインデータを再構築する。

外部依存

  • storage/fd.h, storage/buffile.h, storage/fileset.h 等の公開ヘッダー。
  • utils/resowner.h (リソース所有権管理)。
  • commands/tablespace.h (テーブルスペースのパス解決)。
  • storage/dsm.h (共有メモリ連携)。
  • 標準ライブラリ: <fcntl.h>, <unistd.h>, <sys/stat.h> 等。

内部構成

本モジュールは、fd.c を最下層の抽象化レイヤーとし、その上に一時ファイル管理（fileset.c, sharedfileset.c）やバッファリング（buffile.c）、特殊なファイル操作（copydir.c, reinit.c）が構築される階層構造を持っています。

推奨読書順

1. fd.c（全操作の基盤となるVFD管理を理解する） 2. buffile.c（高レベルなファイルI/Oの仕組みを把握する） 3. fileset.c（一時ファイルの管理単位を理解する） 4. sharedfileset.c（プロセス間共有の仕組みを確認する） 5. copydir.c（ディレクトリ操作の堅牢性を確認する） 6. reinit.c（リカバリ時の特殊処理を確認する）

未解決の問い

  • 各種ファイル操作における fsync の呼び出し頻度とパフォーマンスのトレードオフの最適化。
  • OS固有の高速クローン機能（copy_file_range等）のサポート拡充と、プラットフォーム間での整合性保証の精緻化。
