contrib/postgres_fdw

システム内での役割

postgres_fdw は、PostgreSQL の外部データラッパー（FDW）であり、リモートの PostgreSQL サーバー上のテーブルをローカルテーブルのように透過的にアクセス可能にするモジュールです。プランナとエグゼキュータのインターフェースを実装し、クエリのプッシュダウンやリモート接続の管理を行います。

責務

  • **クエリの最適化とプッシュダウン**: ローカルのクエリツリーを解析し、リモートで実行可能な SQL に変換（デパース）することで、ネットワーク転送量を削減する。
  • **リモート接続のライフサイクル管理**: 接続のキャッシュ、トランザクションの同期、およびセッションパラメータの維持。
  • **データ操作の実行**: SELECT だけでなく、INSERT, UPDATE, DELETE などの DML 操作をリモートサーバーへ委譲する。
  • **実行可能性の判定**: 関数や演算子がリモートサーバーで安全に実行可能か（shippable か）を検証する。

このモジュールのファイル

  • postgres_fdw.c: モジュールのエントリポイント。プランニング・実行の司令塔。
  • postgres_fdw.h: 共有データ構造（PgFdwRelationInfo 等）と関数宣言。
  • deparse.c: クエリツリーからリモート SQL への変換ロジック。
  • connection.c: リモート接続のキャッシュとトランザクション同期。
  • option.c: サーバー・テーブルオプションの検証と GUC 管理。
  • shippable.c: オブジェクトのプッシュダウン可否判定とキャッシュ。

主要なデータ構造

  • PgFdwRelationInfo: プランナが使用するリモートテーブルのコストやプッシュダウン情報を保持する構造体。
  • ConnCacheEntry: リモート接続の状態、トランザクション深度、準備済みステートメントを管理するキャッシュエントリ。
  • deparse_expr_cxt: SQL 生成時のコンテキスト（出力バッファ、パラメータリスト）。
  • ShippableCacheKey: オブジェクトの実行可能性判定結果を保持するキャッシュキー。

主なフロー

1. **プランニング**: postgresGetForeignRelSize がコストを見積もり、deparse.c を用いてリモート実行可能な条件を特定する。 2. **接続**: GetConnection が libpq を使用してリモート接続を確立し、ローカルのトランザクション状態を同期する。 3. **実行**: postgresIterateForeignScan がリモートカーソルからタプルを取得し、ローカル形式へ変換する。 4. **DML**: postgresPlanForeignModify がリモート更新 SQL を生成し、プリペアドステートメントとして実行する。

外部依存

  • libpq: リモートサーバーとの通信用ライブラリ。
  • nodes/pathnodes.h, optimizer/pathnode.h: PostgreSQL プランナのノード定義。
  • access/reloptions.h: オプション処理の基盤。
  • catalog/pg_user_mapping.h: ユーザーマッピング情報へのアクセス。

内部構成

本モジュールはディレクトリ境界に基づいて構成されており、単一の拡張として提供されます。各ファイルは postgres_fdw.h を通じて密接に連携しています。

推奨読書順

1. postgres_fdw.h (全体像と主要構造体の把握) 2. postgres_fdw.c (エントリポイントとライフサイクル) 3. deparse.c (SQL 生成ロジック) 4. connection.c (接続管理とトランザクション) 5. option.c (設定検証) 6. shippable.c (プッシュダウン判定)

未解決の問い

  • 複雑なサブクエリや特定の型変換におけるプッシュダウンの制限をどこまで緩和可能か。
  • 接続の並列実行や分散トランザクション（2PC）のサポート範囲の拡大可能性。
