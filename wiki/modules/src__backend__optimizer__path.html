src/backend/optimizer/path

システム内での役割

src/backend/optimizer/path は、PostgreSQL プランナにおいて、論理的なクエリツリーを物理的な実行計画（Path）へ変換する中核モジュールです。クエリの各リレーションに対して、スキャンや結合の実行手法を網羅的に生成し、コストモデルに基づいて最適な計画を選択する役割を担います。

責務

  • **パス生成の統括**: ベースリレーションのスキャンパスおよび結合リレーションの結合パスの生成。
  • **コストと選択率の評価**: 実行パスのコスト（startup_cost, total_cost）および行数（rows）の見積もり。
  • **等価関係の管理**: EquivalenceClass を用いた等価条件の推論と正規化。
  • **順序の抽象化**: PathKey によるソート順序の管理と冗長性の排除。
  • **最適化の探索**: 動的計画法による結合順序の決定と、インデックス利用の最適化。

このモジュールのファイル

  • allpaths.c: パス生成のメインエントリポイント。
  • clausesel.c: 選択率（Selectivity）の計算。
  • costsize.c: コスト見積もりとカーディナリティ算出。
  • equivclass.c: 等価クラスの管理と推論。
  • indxpath.c: インデックススキャンパスの生成。
  • joinpath.c: 結合アルゴリズム（Nestloop, Hash, Merge）のパス生成。
  • joinrels.c: 結合順序の探索とリレーション構築。
  • pathkeys.c: ソート順序の管理と正規化。
  • tidpath.c: TID スキャンパスの生成。

主要なデータ構造

  • Path / RelOptInfo: 実行パスとリレーションの基本構造。
  • EquivalenceClass / EquivalenceMember: 等価な式の集合とメンバー。
  • PathKey: 順序を表現する正規化された構造体。
  • RestrictInfo: 制限条件や結合条件のメタデータ。
  • IndexClauseSet: インデックス適合条件のコンテナ。
  • RangeQueryClause: 範囲クエリの境界管理用構造体。

主なフロー

1. **初期化**: allpaths.c がリレーションごとのパス生成を開始。 2. **等価関係の構築**: equivclass.c がクエリ内の等価条件を正規化。 3. **パス生成**: 各リレーションに対し、indxpath.c や tidpath.c でスキャンパスを生成。 4. **結合探索**: joinrels.c が動的計画法で結合順序を決定し、joinpath.c が各結合手法を評価。 5. **コスト評価**: costsize.c と clausesel.c が各パスのコストと選択率を算出。 6. **最終化**: pathkeys.c を用いて順序を最適化し、コスト最小のパスを特定。

外部依存

  • optimizer/pathnode.h, optimizer/paths.h: パス生成の基本インターフェース。
  • optimizer/cost.h: コスト計算モデル。
  • optimizer/restrictinfo.h: 条件式の管理。
  • utils/lsyscache.h, utils/selfuncs.h: システムカタログおよび統計情報。
  • nodes/nodeFuncs.h: ノードツリーの操作。

内部構成

本モジュールはディレクトリ境界に基づいた論理的なサブシステムであり、個別のファイルが特定の最適化フェーズ（コスト、結合、インデックス等）を分担するフラットな構成をとっています。

推奨読書順

1. allpaths.c (全体像の把握) 2. costsize.c (コストモデルの理解) 3. equivclass.c (等価関係の推論) 4. pathkeys.c (順序の抽象化) 5. indxpath.c (スキャンパス生成) 6. joinrels.c (結合探索の制御) 7. joinpath.c (結合パス生成) 8. clausesel.c (選択率計算) 9. tidpath.c (特殊スキャン)
