src/backend/parser

システム内での役割

src/backend/parser は、PostgreSQL における SQL 解析の中核を担うモジュールです。SQL 文字列を字句解析・構文解析して「生の解析ツリー」を生成し、さらにシステムカタログを参照して型解決や名前解決を行う「意味解析」を経て、プランナが処理可能な「クエリツリー」へと変換します。

責務

  • **構文解析**: SQL 文字列をトークン化し、文法規則に基づいて構文木を構築する。
  • **意味解析**: 構文木に対して型チェック、演算子・関数解決、権限チェック、名前空間の管理を行う。
  • **DDL 解析**: CREATE TABLE 等のユーティリティコマンドの変換と制約検証。
  • **エラー診断**: 解析エラー時に、SQL 文中の正確な位置とヒントを含む詳細な診断情報を生成する。

このモジュールのファイル

  • analyze.c: 意味解析のメインドライバ。
  • parser.c: 構文解析のメインエントリーポイント。
  • gramparse.h: 字句・構文解析器間の共有定義。
  • parse_*.c: 各種 SQL 句（式、リレーション、集約、型変換、DDL 等）の解析ロジック。
  • scansup.c: 識別子の正規化等の字句解析補助。

主要なデータ構造

  • ParseState: 解析セッション全体で共有される状態管理構造体。
  • Query: 意味解析後の実行可能なクエリツリー。
  • RangeTblEntry (RTE): FROM 句等に現れるデータソースの定義。
  • base_yy_extra_type: 字句・構文解析器間で共有される解析状態。
  • CollateStrength: 照合順序の解決状態を管理する列挙型。

主なフロー

1. **Raw Parsing**: parser.c が SQL 文字列を読み込み、Flex/Bison を用いて「生の解析ツリー」を生成する。 2. **Semantic Analysis**: analyze.c が解析ツリーを受け取り、ParseState を初期化する。 3. **Transformation**: 各 parse_*.c が再帰的に呼び出され、式やリレーションを解決して Query ノードを構築する。 4. **Post-processing**: parse_collate.c 等が照合順序の割り当てや集約関数の検証を行い、クエリツリーを確定させる。

外部依存

  • catalog/: システムカタログ（pg_type, pg_proc 等）の参照。
  • nodes/: 構文木およびクエリツリーのノード定義。
  • utils/: システムキャッシュ（lsyscache）やメモリ管理、マルチバイト文字処理。

内部構成

本モジュールは、解析の各フェーズを担う独立したファイル群で構成されています。analyze.c がハブとなり、各 parse_*.c が特定の SQL 構文要素の変換を専門的に処理する疎結合な設計です。

推奨読書順

1. src/backend/parser/parser.c (解析の入り口) 2. src/backend/parser/analyze.c (意味解析の全体像) 3. src/backend/parser/parse_relation.c (名前解決の基礎) 4. src/backend/parser/parse_expr.c (式解析の基礎) 5. src/backend/parser/parse_coerce.c (型変換の仕組み) 6. src/backend/parser/parse_clause.c (各句の解析)
