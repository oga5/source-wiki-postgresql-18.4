src/backend/optimizer/plan

システム内での役割

src/backend/optimizer/plan は、PostgreSQL のプランナにおいて、論理的なクエリツリー（Query）を、実行エンジン（Executor）が処理可能な物理的な実行計画（PlannedStmt）へ変換する中核モジュールです。

責務

  • **プランニングのオーケストレーション**: クエリの解析から物理プランの生成、最終的な変数参照解決までの一連のプロセスを統括する。
  • **論理から物理への変換**: コストベースで選択された Path ツリーを、実行可能な Plan ノードツリーへ再帰的に変換する。
  • **最適化の適用**: 結合の簡略化、不要な結合の排除、MIN/MAX 集約のインデックス最適化、サブクエリの物理化などを行う。
  • **実行時情報の付与**: 変数参照のインデックス化、演算子の関数解決、プランキャッシュのための依存関係追跡を行う。

このモジュールのファイル

  • planner.c: プランニングのメインエントリーポイント。
  • planmain.c: 結合プランニングのエンジン。
  • createplan.c: Path から Plan への変換。
  • setrefs.c: 最終的な変数参照解決とプランの正規化。
  • initsplan.c: ジョインツリーの分解と制約の初期化。
  • analyzejoins.c: 不要な結合の排除と簡略化。
  • planagg.c: MIN/MAX 集約の最適化。
  • subselect.c: サブクエリおよび CTE の物理プランへの変換。

主要なデータ構造

  • PlannerInfo: プランニングプロセス全体で共有されるグローバル状態。
  • Path / Plan: 物理変換の前後のノード構造体。
  • RestrictInfo: 結合やスキャンにおける制約条件。
  • SpecialJoinInfo: 外部結合等の結合順序制約。
  • MinMaxAggInfo: MIN/MAX 最適化のためのメタデータ。

主なフロー

1. **初期化**: planner() がクエリを受け取り、PlannerInfo を初期化。 2. **論理最適化**: initsplan.c でジョインツリーを分解し、analyzejoins.c で不要な結合を排除。 3. **パス生成**: planmain.c を中心に、コストベースで最適な Path ツリーを探索。 4. **物理変換**: createplan.c が Path を Plan ノードへ再帰的に変換。 5. **最終化**: setrefs.c が変数参照を物理インデックスへ書き換え、PlannedStmt を生成。

外部依存

  • nodes/: ノード定義および操作関数。
  • optimizer/: コスト計算、パス生成、条件式操作の基盤。
  • catalog/: システムカタログ情報へのアクセス。
  • utils/: キャッシュ管理およびメモリ管理。

内部構成

本モジュールは、プランニングの各フェーズ（初期化、論理最適化、物理変換、後処理）を担うファイル群で構成されるフラットな構造です。

推奨読書順

1. planner.c (全体像の把握) 2. planmain.c (結合プランニングの理解) 3. createplan.c (物理変換の仕組み) 4. setrefs.c (最終化処理) 5. initsplan.c (初期化と制約) 6. subselect.c (サブクエリの扱い) 7. analyzejoins.c (結合最適化) 8. planagg.c (集約最適化)
