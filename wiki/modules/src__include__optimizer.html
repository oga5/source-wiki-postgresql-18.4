src/include/optimizer

システム内での役割

src/include/optimizer は、PostgreSQL のクエリプランナ（オプティマイザ）の主要なインターフェースおよび内部データ構造を定義するディレクトリです。SQL クエリを効率的な実行計画（Plan）に変換するための、コスト見積もり、パス探索、遺伝的アルゴリズム（GEQO）、および式操作の基盤を提供します。

責務

  • クエリプランナの外部 API（planner() 等）の公開。
  • 物理プラン生成のためのコストモデルと統計情報の管理。
  • 複雑な結合クエリに対するヒューリスティックな探索（GEQO）の提供。
  • クエリツリーからプランツリーへの変換過程における式・述語・ターゲットリストの操作。

このモジュールのファイル

本モジュールは、プランナの各サブシステムに対応する以下の27個のヘッダーファイルで構成されています。

  • **メイン・制御**: optimizer.h, planner.h, planmain.h
  • **コスト・統計**: cost.h, plancat.h
  • **パス生成**: paths.h, pathnode.h, appendinfo.h, inherit.h, joininfo.h, placeholder.h, prep.h, subselect.h, paramassign.h
  • **式・述語操作**: clauses.h, restrictinfo.h, tlist.h, orclauses.h
  • **GEQO (遺伝的アルゴリズム)**: geqo.h, geqo_copy.h, geqo_gene.h, geqo_misc.h, geqo_mutation.h, geqo_pool.h, geqo_random.h, geqo_recombination.h, geqo_selection.h

主要なデータ構造

  • PlannerInfo: プランニングプロセス全体の状態を保持する中心的な構造体。
  • RelOptInfo: 特定のリレーションに対するパスや統計情報を保持する構造体。
  • Path / PathTarget: 実行計画の候補となるアクセスパスと射影情報。
  • RestrictInfo: 述語に関するメタデータ（移動可能性等）を付与したノード。
  • GeqoPrivateData: GEQO 探索中の状態を保持する構造体。
  • Chromosome / Pool: GEQO における個体と個体群の管理構造。

主なフロー

1. **前処理**: prep.h 等を用い、サブクエリの引き上げやクエリの正規化を行う。 2. **パス探索**: paths.h を中心に、各リレーションに対する最適なアクセスパスを RelOptInfo に蓄積。結合順序の決定には動的計画法または geqo.h を使用。 3. **コスト評価**: cost.h と plancat.h を参照し、統計情報に基づいたコスト見積もりを実施。 4. **プラン構築**: planmain.h を通じて、選択された Path を物理的な Plan ノードへ変換。 5. **最終化**: setrefs.c 等により変数参照を解決し、実行可能な PlannedStmt を生成。

外部依存

  • nodes/: pathnodes.h, plannodes.h, parsenodes.h, nodes.h, bitmapset.h
  • utils/: relcache.h
  • common/: pg_prng.h

内部構成

本ディレクトリは、プランナの機能ごとにヘッダーが分割されたフラットな構成です。各ヘッダーは対応する src/backend/optimizer/ 以下の実装ファイルと密接に結合しています。

推奨読書順

1. optimizer.h (公開 API の理解) 2. planner.h (プランナの全体構造) 3. paths.h (パス生成のハブ) 4. cost.h (コストモデルの理解) 5. planmain.h (プラン構築の各ステップ) 6. restrictinfo.h / clauses.h (述語処理) 7. geqo.h (複雑な結合の最適化)
