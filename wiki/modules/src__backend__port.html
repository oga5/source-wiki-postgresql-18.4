src/backend/port

システム内での役割

src/backend/port は、PostgreSQL が動作するプラットフォーム（OS/アーキテクチャ）ごとの差異を吸収し、共通のインターフェースを提供する抽象化レイヤーです。主にプロセス間通信（IPC）や低レベルな同期プリミティブの OS 固有実装を保持します。

責務

  • **プラットフォーム抽象化**: セマフォ、共有メモリ、アトミック操作といった OS 依存機能の共通 API 化。
  • **リソース管理**: プロセス終了時の IPC リソース（セマフォや共有メモリセグメント）のクリーンアップと再利用。
  • **フォールバック提供**: ハードウェアがネイティブサポートしないアトミック操作のソフトウェアシミュレーション。

このモジュールのファイル

  • src/backend/port/atomics.c: 64bit アトミック操作のフォールバック実装。
  • src/backend/port/posix_sema.c: POSIX セマフォによる同期実装。
  • src/backend/port/sysv_sema.c: System V セマフォによる同期実装。
  • src/backend/port/sysv_shmem.c: System V 共有メモリによる管理実装。
  • src/backend/port/win32_sema.c: Windows セマフォによる同期実装。
  • src/backend/port/win32_shmem.c: Windows 共有メモリによる管理実装。

主要なデータ構造

  • **セマフォ関連**: PGSemaphoreData (共通抽象型), SemTPadded
  • **共有メモリ関連**: IpcMemoryState (状態管理), IpcMemoryKey, IpcMemoryId
  • **同期関連**: IpcSemaphoreKey, IpcSemaphoreId

主なフロー

1. **初期化**: PGReserveSemaphores や PGSharedMemoryCreate が呼び出され、OS 固有の API を用いてリソースを確保します。 2. **操作**: PGSemaphoreLock/Unlock 等の共通関数が、ビルド時に選択された実装（SysV/POSIX/Win32）へ処理を委譲します。 3. **終了**: プロセス終了時に on_shmem_exit 等のフックを通じて、確保したリソースの解放やデタッチを自動的に行います。

外部依存

  • postgres.h, miscadmin.h (基本定義)
  • storage/pg_sema.h, storage/shmem.h, storage/pg_shmem.h (IPC 共通インターフェース)
  • port/atomics.h (アトミック操作定義)
  • sys/ipc.h, sys/sem.h, sys/shm.h (Unix 系 API)
  • windows.h (Win32 API)

内部構成

本モジュールはディレクトリ境界によって定義されており、各ファイルはプラットフォームごとの実装を切り替えるための独立したコンポーネントとして構成されています。

推奨読書順

1. src/backend/port/sysv_sema.c (セマフォの基本設計) 2. src/backend/port/sysv_shmem.c (共有メモリのライフサイクル) 3. src/backend/port/posix_sema.c (POSIX 準拠の抽象化) 4. src/backend/port/win32_shmem.c (Windows 特有のアドレス空間保護) 5. src/backend/port/win32_sema.c (Windows のシグナル統合) 6. src/backend/port/atomics.c (ハードウェア抽象化の補完)

未解決の問い

  • 各プラットフォーム実装におけるリソースクリーンアップの堅牢性（特にクラッシュリカバリ時）の差異を、より統一的なテストスイートで検証可能か。
  • EXEC_BACKEND 環境における共有メモリ再アタッチの複雑性を、将来的にどの程度簡素化できるか。
